# 系统设计面试题 – Gmail 电子邮件系统

## 题目背景
Gmail 是 Google 提供的免费电子邮件服务，面向全球数十亿用户。用户可以通过网页、移动端或 POP/IMAP 客户端收发邮件、搜索历史、管理标签以及使用强大的垃圾邮件过滤。

## 面试场景设定
> **面试官**：  
> “请你设计一个类似 Gmail 的电子邮件系统，满足大规模用户的收发、存储和检索需求。我们先从核心功能和关键指标入手，你可以先描述整体架构思路，然后我们会针对细节进行深入探讨。”

## 功能性需求
1. **发送/接收邮件**  
   - 支持 SMTP、IMAP/POP3 协议，单封邮件大小上限 25 MB（含附件）。  
2. **邮件搜索**  
   - 支持关键字、发件人、时间范围、标签等组合查询，返回结果时延 ≤ 200 ms。  
3. **标签与线程**  
   - 用户可以自定义标签、将邮件自动归档到对应标签；邮件以会话（Thread）形式展示。  
4. **附件存储**  
   - 附件以对象存储形式持久化，支持断点续传、病毒扫描。  
5. **垃圾邮件过滤**  
   - 基于机器学习的过滤模型，实时判断并将垃圾邮件移动到专用文件夹。  
6. **多端同步**  
   - 邮件状态（已读/已删/已标星）在 Web、iOS、Android 等客户端实时同步。

## 非功能性需求（带数字估算）
| 指标 | 目标值 | 说明 |
|------|--------|------|
| **活跃用户数（DAU）** | 1.5 亿 | 全球范围的日活跃 Gmail 用户 |
| **写入 QPS（Send）** | 15 k req/s | 峰值时每秒发送邮件请求数 |
| **读取 QPS（Read/Sync）** | 50 k req/s | 包括邮件列表、单封邮件、同步等读取请求 |
| **平均响应时延** | ≤ 200 ms（99%） | 对外 API（发送、读取、搜索）的端到端时延 |
| **系统可用性** | 99.99%（年均） | 包括邮件投递、检索和同步的可用性 |
| **存储规模** | 15 PB（对象存储） | 估算 1.5 亿用户 × 10 GB（平均使用） = 1.5 EB，考虑冷热分层后实际对象存储约 15 PB |

## 系统边界
**本题范围内**（需要设计）  
- 邮件的发送、接收、存储、索引、搜索、同步与垃圾邮件过滤。  
- 后端服务的高可用、水平扩展、数据持久化与容灾。  
- 监控、日志、限流等运维支撑功能。

**本题范围外**（不必考虑）  
- UI/前端页面、移动端 SDK 的实现细节。  
- 与 Google Drive、Google Calendar、Google Meet 等其他 Google 服务的深度集成。  
- AI 自动回复、智能撰写等高级功能。  
- 国际化、法律合规（GDPR）细节（可在讨论中简要提及，但不需实现）。  

## 提示与追问
1. **存储与检索**  
   - “如果每封邮件平均 100 KB，如何设计存储层才能兼顾高写入吞吐和低延迟搜索？”  
2. **附件去重与冷热分层**  
   - “考虑到大量用户可能上传相同的图片或文档，你会如何在对象存储层实现去重和冷热分层？”  
3. **一致性模型**  
   - “在多端同步场景下，你会选择怎样的一致性策略来保证已读/已删状态的最终一致？”  

> 请围绕上述需求和指标，先给出系统的高层结构图及主要组件的职责划分，然后根据面试官的追问展开细节设计。祝你面试顺利！

# 题解

## 解题思路总览
> **目标**：帮助零经验的同学，从“要解决什么问题”出发，逐层搭建出一个能够支撑 **1.5 亿日活、写入 15 k QPS、读取 50 k QPS** 的 Gmail‑类邮件系统。  
> **方法**：  
> 1. **先把需求拆解成最小可运行（MVP）**，只保留核心功能——发送、收取、持久化、查询。  
> 2. **一步步加复杂度**：在 MVP 基础上加入高可用、扩容、搜索、附件去重、垃圾邮件过滤、跨端同步等。  
> 3. **每一次技术选型都要回答**：*为什么要这么做？不这么做会出现什么痛点？*  
> 4. **用图、表、伪代码把抽象概念具象化**，帮助新手“看到全局、懂得细节”。  

下面按照 **需求 → 高层架构 → 数据模型 → API → 关键组件 → 高可用/扩展 → 面试追问** 的顺序展开，层层递进。

## 第一步：理解需求与规模估算

| 需求 | 解释 | 关键指标 |
|------|------|----------|
| **发送/接收** | 支持 SMTP、IMAP/POP3，单封 ≤ 25 MB | 写入 QPS 15 k |
| **搜索** | 关键字/标签/时间/线程组合查询，时延 ≤ 200 ms | 读取 QPS 50 k |
| **标签/线程** | 用户自定义标签、邮件会话展示 | 业务层关联查询 |
| **附件存储** | 对象存储，断点续传，病毒扫描 | 存储规模约 15 PB |
| **垃圾邮件过滤** | ML 模型实时判别 | 写入路径的额外 CPU/IO |
| **多端同步** | 已读/已删/星标状态即时同步 | 强一致性需求 |
| **非功能** | DAU 1.5 亿、99.99% 可用、200 ms 99% 延迟 | 规模、容灾、监控 |

### 1.1 业务量估算（粗略）

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日发送邮件** | 15 k req/s × 86 400 s ≈ 1.3 × 10⁹ 封 | 1.3 B |
| **每日接收邮件** | 同上（收发基本平衡） | 1.3 B |
| **平均邮件大小** | 100 KB（正文） + 附件 0.5 MB（假设 20% 有附件） | ≈ 150 KB |
| **每日写入存储** | 1.3 B × 150 KB ≈ 195 TB | 近 200 TB |
| **全年存储** | 200 TB × 365 ≈ 73 PB | 冷热分层后约 15 PB（热点 10%） |

> **结论**：系统必须 **水平扩展**、**分层存储**、**异步写入**，并且 **搜索索引** 不能放在同一套写库里。

### 1.2 关键约束

| 约束 | 含义 |
|------|------|
| **时延 ≤ 200 ms** | 前端交互（打开邮件列表、搜索）必须在 200 ms 内返回，要求查询路径尽可能短、缓存命中率高。 |
| **99.99% 可用** | 年宕机时间 < 52 分钟，单点故障必须被冗余（多活）覆盖。 |
| **单封 25 MB** | 限制网络带宽与对象存储写入大小，需要 **分块上传** 与 **流式转发**。 |

## 第二步：高层架构设计

### 2.1 最小可用系统（MVP）

```
+-------------------+       +-------------------+
|   客户端 (Web/APP) | <---> |   API Gateway     |
+-------------------+       +-------------------+
                               |
                 +----------------------------+
                 |   Mail Service (SMTP/IMAP) |
                 +----------------------------+
                               |
            +------------------+------------------+
            |                                     |
   +----------------+                     +----------------+
   |  Mail Store    |                     |  Index Service |
   +----------------+                     +----------------+
            |                                     |
   +----------------+                     +----------------+
   | Object Storage|                     |  Search Index  |
   +----------------+                     +----------------+
```

- **API Gateway**：统一入口，做鉴权、限流、日志。  
- **Mail Service**：实现 SMTP/IMAP/POP3 协议，负责接收/投递邮件。  
- **Mail Store**：关系型/NoSQL 数据库，持久化邮件元数据（发件人、收件人、时间、标签、线程ID）。  
- **Object Storage**：大文件（附件）持久化，使用分块、MD5 去重。  
- **Index Service**：基于倒排索引的搜索引擎（如 Elasticsearch），同步邮件元数据供搜索使用。

> **如果直接把所有数据放进关系库**，写入吞吐会被磁盘 I/O 限制，搜索也会极慢；如果只用对象存储，无法快速定位邮件元信息。MVP 将两者分离，满足 **写入** 与 **查询** 的不同特性。

### 2.2 完整分布式系统（加入高可用、扩展、业务特性）

```
+--------------------------------------------------------------+
|                         CDN / Edge                          |
+--------------------------------------------------------------+
          |  TLS/HTTPS (REST+WebSocket)   |
+--------------------------------------------------------------+
|                         API Gateway                         |
+--------------------------------------------------------------+
|   Auth Service   |   Rate Limiter   |   Metrics/Logging    |
+--------------------------------------------------------------+
|                         Service Mesh                         |
+--------------------------------------------------------------+
|          |               |               |                 |
|  Mail Ingest Service   |   Sync Service   |   Search Service |
|   (SMTP/IMAP/POP3)      | (State Sync)    | (Full‑text)      |
+--------------------------+-----------------+-----------------+
|          |               |               |                 |
|  Mail DB Cluster (Cassandra/Spanner)  |  Index Cluster   |
|  (meta + thread)                      | (Elasticsearch) |
+---------------------------------------+-----------------+
|                     Object Store (GCS‑like)                |
|   +----------------------+   +--------------------------+ |
|   |  Chunk Storage       |   |  Metadata DB (MySQL)      | |
|   +----------------------+   +--------------------------+ |
+--------------------------------------------------------------+
|                     Background Workers                     |
|  - Virus Scan      - Attachment Deduplication - ML Spam   |
+--------------------------------------------------------------+
|                     Monitoring & Alerting                  |
+--------------------------------------------------------------+
```

- **CDN / Edge**：对静态资源（网页、JS、附件下载）做就近加速。  
- **Service Mesh**（如 Istio）提供流量控制、可观测性、熔断。  
- **Mail Ingest Service**：拆分为 **SMTP Frontend**、**IMAP/POP3 Frontend**，各自做流控、限速、协议转码。  
- **Mail DB Cluster**：选用 **强一致性**的分布式数据库（Google Spanner / CockroachDB）或 **AP** 的宽列存储（Cassandra）+ **事务层**（外部事务服务）实现**最终一致**的状态同步。  
- **Sync Service**：处理 **已读/已删/星标** 状态的多端同步，使用 **事件流**（Kafka）+ **幂等写** 保证 **最终一致**。  
- **Search Service**：写入 **Change Data Capture**（CDC）流，实时更新倒排索引。  
- **Background Workers**：病毒扫描、附件去重、垃圾邮件 ML 模型推理，均是 **异步**，不阻塞用户请求。  
- **监控**：Prometheus + Grafana、SLO/SLI 报警，确保 99.99% 可用。

## 第三步：数据库设计

### 3.1 邮件元数据模型（核心表）

| 表名 | 主键 | 重要字段 | 说明 |
|------|------|----------|------|
| **User** | `user_id (UUID)` | `email`, `password_hash`, `quota` | 用户基本信息 |
| **Mail** | `mail_id (UUID)` | `thread_id`, `sender_id`, `sent_ts`, `size`, `has_attachments`, `spam_score` | 单封邮件的元数据 |
| **Mail_Recipient** | `mail_id + recipient_id` (复合 PK) | `mailbox` (INBOX/ SENT / SPAM), `label_ids[]`, `is_read`, `is_starred`, `is_deleted`, `receive_ts` | 多对多关系，支持每个收件人独立状态 |
| **Thread** | `thread_id (UUID)` | `subject`, `first_mail_ts`, `last_mail_ts`, `mail_count` | 会话聚合 |
| **Label** | `label_id (UUID)` | `user_id`, `name`, `color` | 用户自定义标签 |
| **Attachment** | `attachment_id (UUID)` | `mail_id`, `object_key`, `size`, `md5`, `content_type` | 关联对象存储的块信息 |
| **Mail_Index** (ES) | `mail_id` | `subject`, `body_text`, `sender_email`, `recipients`, `labels`, `thread_id`, `sent_ts` | 用于全文检索的倒排索引 |

> **为什么拆成 `Mail` + `Mail_Recipient`？**  
> - Gmail 里 **每个收件人都有自己的已读/已删状态**，如果把这些状态写进 `Mail` 表会导致冲突并且难以扩展。  
> - 采用 **宽表**（`Mail_Recipient`）可以让 **状态更新只涉及单行记录**，写入 QPS 友好。

### 3.2 选型讨论

| 需求 | 关系型 (MySQL) | NoSQL (Cassandra) | NewSQL (Spanner) |
|------|----------------|-------------------|------------------|
| 强事务（发送邮件原子写入多表） | ✅（两阶段提交） | ❌（只能轻量事务） | ✅（分布式事务） |
| 高写入吞吐（10 k QPS） | 可通过分库分表实现 | 天然水平扩展 | 可水平扩展，成本更高 |
| 跨数据中心复制 | 主从延迟 5‑10s | 通过 **Quorum** 实现 1‑2s | 全球同步 < 1s |
| 成本 | 中等 | 低 | 高 |

**推荐**：**Spanner**（或 CockroachDB）作为 **Mail DB**，因为邮件系统对 **事务一致性**（发送邮件必须同时写入 `Mail`、`Mail_Recipient`、`Thread`）要求高。若受预算限制，可采用 **MySQL 读写分离 + Sharding** + **异步双写**（写入 CDC 到搜索、备份）。

### 3.3 附件对象存储

- **对象键**：`{user_id}/{mail_id}/{attachment_md5}`（MD5 作为去重标识）。  
- **分块上传**：客户端先把大文件分成 5 MB 块，逐块上传到 **Chunk Storage**，返回块的 **ETag**（MD5），服务器在收到全部块后 **合并**。  
- **去重**：上传前计算整体 MD5，查询 `Attachment` 表是否已存在同 MD5，若存在则直接指向已有对象键（**指针**），避免重复存储。  
- **冷热分层**：  
  - **热层**（SSD、低延迟）保存最近 30 天的附件。  
  - **冷层**（对象归档）使用低成本磁带/冷存储，配合 **生命周期策略**（30 天后自动降温）。  
- **病毒扫描**：上传完成后，**后台 Worker** 拉取对象键，使用 ClamAV/商业 AV 引擎扫描，若发现病毒则标记 `is_malicious`，并在 UI 中隐藏。

## 第四步：核心 API 设计

> **原则**：RESTful + gRPC 双协议（REST 用于浏览器/移动端，gRPC 用于内部高频通信），统一使用 **JSON** 或 **ProtoBuf**。

| 功能 | HTTP 方法 | 路径 | 请求体 | 响应体 | 关键校验 |
|------|-----------|------|--------|--------|----------|
| **发送邮件** | POST | `/api/v1/mails` | `{sender_id, to[], cc[], bcc[], subject, body, attachments[]}` | `{mail_id, thread_id}` | 1️⃣ 限流 2️⃣ 附件大小检查 3️⃣ Spam Score 计算 |
| **获取邮件列表** | GET | `/api/v1/users/{uid}/mailboxes/{mailbox}?page=1&size=50&label=starred` | - | `{mails:[{mail_id, subject, snippet, sender, ts, is_read, labels}], next_page}` | 只返回 **摘要**（snippet）避免全量拉取 |
| **获取单封邮件** | GET | `/api/v1/mails/{mail_id}` | - | `{mail_id, thread_id, subject, body, attachments[], labels, is_read}` | 权限校验：只能访问自己收件箱/已发送 |
| **标记已读/星标/删除** | PATCH | `/api/v1/mails/{mail_id}/status` | `{is_read?, is_starred?, is_deleted?}` | `{success:true}` | 幂等写，写入 `Mail_Recipient` |
| **搜索** | POST | `/api/v1/search` | `{query:"from:alice label:starred", page, size, sort}` | 同邮件列表结构 | 将请求转发给 **Search Service**，返回 **mail_id** + **snippet** |
| **同步状态** | POST (WebSocket) | `/ws/sync` | `{user_id, last_sync_ts}` | 实时推送 `{mail_id, field, new_value}` | 基于 **Kafka** 的 **Change Feed**，客户端保持长连接 |

### 4.1 幂等性 & 重试

- **发送邮件**：使用 **客户端生成的 UUID**（`client_mail_id`）作为幂等键，服务端在 **Mail DB** 上做 **唯一索引**，重复请求返回已存在的 `mail_id`。  
- **状态更新**：`Mail_Recipient` 表的 **主键**（`mail_id+recipient_id`）天然幂等，`PATCH` 只覆盖字段。

### 4.2 限流 & 防刷

- **API Gateway**：基于 **IP+User** 的 Token Bucket（写 15 k QPS、读 50 k QPS）。  
- **SMTP**：每个发件域（或 IP）单独限速，防止 **spam bot**。  
- **异常监控**：当某用户 QPS 超过阈值，触发 **CAPTCHA** 或 **临时封禁**。

## 第五步：详细组件设计

### 5.1 Mail Ingest Service（SMTP/IMAP）

1. **入口层**（NGINX + TLS）  
   - 负责协议升级、DoS 防护。  
2. **协议解析**（基于 **Apache James** 或自研）  
   - SMTP: `MAIL FROM`, `RCPT TO`, `DATA` 分块读取，实时写入 **Chunk Storage**。  
   - IMAP: 支持 `SELECT`, `FETCH`, `SEARCH`，读取 **Mail DB** + **Object Store**。  
3. **写入流水线**  
   - **Step 1**：写入 **Mail DB**（元数据） → 返回 `mail_id`。  
   - **Step 2**：将邮件正文（text）和附件块的 **MD5** 发送到 **Kafka**（`mail-events` topic）。  
   - **Step 3**：后台 **Indexer** 监听 Kafka，调用 **Search Service** 更新倒排索引。  
   - **Step 4**：**Spam Filter** 消费同一事件，跑 ML 模型，若判为垃圾邮件则写 `spam_score` 并触发 **MoveToSpam** 操作。  

> **不采用直接同步写入搜索** 的原因：同步写会拖慢用户请求的响应时间，且搜索引擎的写入模型（分片、刷新）不适合实时 15 k QPS。使用 **异步 CDC** 能保持 **写入吞吐** 与 **搜索时延** 的平衡。

### 5.2 Sync Service（已读/已删/星标）

- **事件流**：`MailStatusChange`（Kafka） → 每个用户的 **状态机器**。  
- **幂等处理**：在消费者端使用 **Redis** 做去重（`msg_id`），确保同一状态更新只写一次。  
- **推送**：WebSocket 服务器订阅用户对应的 Kafka 分区，实时推送状态变更。  

### 5.3 Search Service（Elasticsearch）

| 功能 | 实现细节 |
|------|----------|
| **倒排索引** | 对 `subject`、`body_text`、`sender_email`、`recipients` 建立分词索引；使用 **IK Analyzer**（中文）+ **Standard Analyzer**（英文）。 |
| **高亮摘要** | `highlight` 功能返回匹配关键字的 snippet，直接在 UI 展示。 |
| **分片与副本** | 依据 **用户 hash**（user_id % N）分片，保证同一用户的邮件基本落在同一分片，降低跨分片查询成本。 |
| **刷新策略** | 每 1 s 刷新一次（`refresh_interval`），在 **写入延迟 1‑2 s** 与 **搜索实时性** 之间做平衡。 |
| **安全隔离** | 使用 **Search Guard** 或 **X‑Pack**，基于 `user_id` 做行级过滤，防止跨用户数据泄露。 |

### 5.4 Attachment Service

1. **上传入口**：REST `/attachments/upload`，支持 **分块**（`multipart/related`），返回每块的 `etag`。  
2. **合并**：所有块上传完成后，客户端调用 **完成** 接口，服务器校验 **整体 MD5**，写入 `Attachment` 表。  
3. **去重流程**：  
   - 先查询 `Attachment` 表是否已有相同 MD5。  
   - 若有，直接返回已有的 `object_key`。  
   - 若无，写入 **Chunk Storage**，生成新对象键。  
4. **下载**：`GET /attachments/{attachment_id}` → 读取 **Object Store**，支持 **Range** 断点续传。  

### 5.5 垃圾邮件过滤

- **模型**：基于 **XGBoost / Deep Learning**，输入特征包括：发件人信誉、链接特征、正文 TF‑IDF、历史用户标记。  
- **部署**：模型以 **TensorFlow Serving** 或 **ONNX Runtime** 形式提供 **REST/gRPC** 接口。  
- **实时判定**：`Mail Ingest Service` 在写入 `Mail DB` 前调用 **Spam Service**，返回 **spam_score**。  
- **后处理**：若 `spam_score > THRESHOLD`，将邮件放入 **Spam** 目录（`mailbox = SPAM`），并记录 **用户反馈**（标记为非垃圾），用于模型离线训练。

| 维度 | 工具 |
|------|------|
| **指标** | Prometheus（QPS、Latency、ErrorRate） |
| **日志** | Fluentd → Elasticsearch + Kibana |
| **追踪** | OpenTelemetry → Jaeger（跨服务链路） |

## 第六步：扩展性与高可用设计

### 6.1 水平扩展

| 层级 | 扩展方式 | 关键点 |
|------|----------|--------|
| **API Gateway** | 增加实例 + L4（Anycast）负载均衡 | DNS TTL 短、健康检查 |
| **Mail Ingest** | 按协议分组（SMTP/IMAP）独立伸缩 | 采用 **Stateless**，会话信息保存在 Redis |
| **Mail DB** | 分片（user_id hash） + 多副本 | 使用 **Spanner** 自动分区，或手动 Sharding |
| **Search** | 动态增加 ES 节点，自动重分片 | 监控 **segment** 大小，触发 **reindex** |
| **Object Store** | 多 AZ 多 Region 复制（CRR） | 读取就近、写入多活 |

### 6.2 容灾与灾备

1. **多 Region 部署**（例如美国、欧洲、亚洲）  
   - **数据同步**：使用 **Spanner** 的跨 Region 同步或 **Cassandra** 的 **NetworkTopologyStrategy**。  
   - **流量路由**：基于 **Geo DNS** 将用户请求路由到最近 Region。  
2. **灾难恢复（DR）**  
   - **热备**：每个 Region 保持 **80% 读写能力**，当主 Region 故障时，快速切换（几秒内 DNS TTL 切换）。  
   - **冷备**：对象存储的 **跨 Region 归档**（Glacier‑like），每 24 h 同步一次。  
3. **故障隔离**  
   - **Circuit Breaker**（Envoy）防止单节点故障蔓延。  
   - **限流** 对内部服务（如 Spam Service）进行 **背压**，防止雪崩。  

### 6.3 一致性模型

| 场景 | 需求 | 选用模型 | 解释 |
|------|------|----------|------|
| **发送邮件（元数据写入）** | 必须原子 | **强一致（Spanner）** 或 **两阶段提交** | 防止邮件只写入一半导致丢失。 |
| **已读/已删状态同步** | 最终一致即可 | **事件驱动 + 幂等写** | 读端可接受几秒延迟，保证高可用。 |
| **标签/星标** | 及时展示 | **读写分离**（写到主库，读从缓存） | 使用 **Redis** 读缓存，写成功后立即 **publish** 事件刷新缓存。 |

## 第七步：常见面试追问与回答

### 7.1 “如果每封邮件平均 100 KB，如何设计存储层才能兼顾高写入吞吐和低延迟搜索？”
**回答要点**  
1. **写入路径**：  
   - **Mail DB** 只存 **元数据**（≈ 1 KB），采用 **分布式 NewSQL**，提供 **强事务**，写入延迟 < 5 ms。  
   - **正文 & 附件** 使用 **对象存储**（分块），写入通过 **异步任务**（Kafka）完成，避免阻塞。  
2. **搜索路径**：  
   - **全文索引** 只针对 **正文文本**（抽取后存入 ES），**不直接查询对象存储**。  
   - **CDC** 将 `Mail` 表的 `subject`、`body_text` 复制到 **Search Service**，实现 **近实时**（1‑2 s）搜索。  
3. **冷热分层**：  
   - 最近 30 天的邮件放在 **SSD**，老邮件转到 **HDD/归档**，搜索时先查询 **热层**，若未命中再查询 **冷层**（异步预热）。  
4. **优势**：  
   - **写入** 只涉及小记录，DB 能支撑 15 k QPS。  
   - **搜索** 只查询倒排索引，返回 200 ms 以内。  
5. **如果不拆分**（把正文也放进 DB），会导致 **磁盘 IO** 成为瓶颈，写入吞吐下降，同时搜索时需要全表扫描，时延爆炸。

### 7.2 “考虑到大量用户可能上传相同的图片或文档，你会如何在对象存储层实现去重和冷热分层？”
**回答要点**  
1. **去重**  
   - 客户端或服务器在 **上传前计算文件的 SHA‑256/MD5**。  
   - 将 **hash** 作为 **对象键**（`{hash}`），在 **Attachment** 表中维护 **引用计数**。  
   - 若已存在相同 hash，直接返回已有对象键并 **increment refcount**，不再写入对象。  
   - **GC**：定时任务检查 **refcount=0** 的对象，删除以释放空间。  
2. **冷热分层**  
   - **Lifecycle Policy**：对象创建后 30 天自动迁移到 **Cold Storage**（如 Nearline / Glacier）。  
   - **访问热点**：读取时若对象在冷层，自动 **恢复**（异步）并返回给用户，同时把对象回迁到热层。  
   - **元数据表**（`Attachment`）记录当前 **storage_class**，供业务判断是否需要提前恢复。  
3. **实现方式**：使用 **分布式对象存储系统**（如 **Ceph RGW**、**MinIO**）提供的 **Bucket Policy** + **Lifecycle Rules**，或者直接在 **云服务**（Google Cloud Storage）上配置。

### 7.3 “在多端同步场景下，你会选择怎样的一致性策略来保证已读/已删状态的最终一致？”
**回答要点**  
1. **业务容忍**：已读/已删是 **用户体验层面的状态**，几秒的延迟对功能没有致命影响，**最终一致**足够。  
2. **实现方式**：  
   - 客户端将状态变更写入 **Mail_Recipient**（写库），并 **produce** 到 **Kafka** 的 `MailStatusChange` topic。  
   - 所有设备（Web、iOS、Android） **subscribe** 对应用户的分区，实时收到 **push**（WebSocket）或 **FCM/APNS** 通知。  
   - 每条消息带有 **唯一 MsgID**，消费者使用 **Redis** 去重，保证幂等。  
3. **冲突解决**：  
   - **基于时间戳**（`last_update_ts`）的 **Last‑Write‑Wins**（LWW）策略。  
   - 若出现并发写（两个设备几乎同时标记已读），采用 **服务器时间** 决定最终状态。  
4. **为何不使用强一致**？  
   - 强一致需要 **同步写** 到所有副本，跨 Region 时延可能 > 200 ms，导致 UI 卡顿。  
   - 异步模型可以让前端 **即时返回成功**，后端在后台完成复制，提升吞吐与可用性。  

### 7.4 其他常见追问

| 追问 | 思考点 | 参考答案 |
|------|--------|----------|
| **邮件撤回** | 需要在收件箱中删除已投递的邮件，需要强一致与跨 Region 同步。 | 使用 **二阶段提交**，先写 “撤回标记”，同步到所有 Region，随后在 UI 隐藏。 |
| **如何实现分页的高效读取** | 大量邮件列表的翻页容易出现 **深分页** 性能问题。 | 使用 **基于 cursor（mail_id）** 的 **keyset pagination**，而不是 offset；前端保存上一页的 **last_mail_id**。 |
| **怎样防止垃圾邮件机器学习模型被攻击** | 对手可能制造对抗样本。 | 采用 **模型监控**（漂移检测），定期 **离线训练**，并在模型前加 **规则过滤**（白名单/黑名单）。 |
| **邮件加密（端到端）** | 需求不在本题范围，但可以点到即止。 | 若要实现，邮件在客户端加密后发送，服务器只存 **加密体**，搜索只能基于 **元数据**（subject 明文），需要 **搜索加密** 方案。 |

## 心得与反思

### 1. 本题最难的 1‑2 个设计决策及思考过程
| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **邮件存储与搜索的分离** | 必须兼顾 **写入吞吐**（15 k QPS）和 **搜索时延**（≤ 200 ms），但两者对底层存储的需求相差巨大。 | 1️⃣ 先列出写入路径（元数据+对象）和查询路径（全文索引）。<br>2️⃣ 评估如果把正文直接放 DB，写入成本和索引成本都会暴涨。<br>3️⃣ 采用 **异步 CDC + Elasticsearch**，在保证 **写入不受搜索阻塞** 的同时，满足 **近实时搜索**。 |
| **多端状态同步的一致性模型** | 需要在 **高可用** 与 **用户体验** 之间找到平衡点，既不能让每次已读都走跨 Region 强一致，也不能出现长期状态不一致。 | 1️⃣ 分析业务容忍度：已读/星标的延迟可接受几秒。<br>2️⃣ 选用 **事件驱动 + 最终一致**，用 Kafka + WebSocket 实现实时推送。<br>3️⃣ 设计 **幂等、LWW** 规则防止冲突。|

### 2. 新手最容易犯的错误（至少2个）

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有数据都放在单体关系库** | 既会导致写入热点，又会让搜索变慢，难以水平扩展。 | 按 **功能**（元数据、全文、附件）拆分存储，使用 **分布式 DB + 搜索引擎 + 对象存储**。 |
| **忽视幂等性和重复请求的处理** | 邮件发送、状态更新容易出现网络重试，若没有幂等键会产生重复邮件或状态冲突。 | 为每个发送请求生成 **client_mail_id**，在 DB 上做唯一约束；对状态更新使用 **主键唯一** 幂等写。 |
| **只考虑单机性能，忘记跨 Region 延迟** | 只在一个数据中心做高可用，无法支撑全球用户。 | 设计 **多 Region 多活**，使用 **跨 Region 同步**（Spanner/跨 Region Kafka）和 **Geo DNS**。 |
| **搜索时直接遍历 DB** | 200 ms 内返回搜索结果几乎不可能。 | 构建 **倒排索引**（Elasticsearch），并使用 **CDC** 实时同步。 |
