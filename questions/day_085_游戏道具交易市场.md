# 第 85 天：设计 游戏道具交易市场

> 生成日期：2026-03-02

---

## 1. 题目背景  
游戏道具交易市场是一个面向数十款线上游戏的统一交易平台，玩家可以在平台上发布、浏览、购买或出售游戏内虚拟道具（装备、皮肤、消耗品等），平台负责撮合、结算并提供安全保障。

## 2. 面试场景设定  
> **面试官**：  
> “现在请你从零开始设计一个高并发、可靠的游戏道具交易市场。我们先从整体思路说起，随后会深入探讨各个关键点。你可以先介绍一下系统的主要模块和数据流吗？”  

## 3. 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| **F1** | **用户注册/登录**：支持邮箱、手机号、第三方（如Steam、QQ）登录，登录后获取唯一的玩家ID。 |
| **F2** | **道具上架**：玩家可以将自己拥有的道具挂到市场，设定单价或选择拍卖模式，系统校验拥有权并冻结道具。 |
| **F3** | **浏览/搜索**：提供分类、关键词、筛选（价格区间、稀有度、游戏）等功能，支持高并发的商品列表查询。 |
| **F4** | **下单/撮合**：买家下单后，系统立即完成撮合（即时购买）或进入拍卖竞价，完成后进行资金与道具的原子转移。 |
| **F5** | **结算与提现**：平台统一结算，支持多币种（游戏币、人民币等），并提供提现到第三方支付渠道。 |
| **F6** | **交易安全与纠纷处理**：提供交易记录、交易状态查询、申诉入口，后台可对异常交易进行人工干预。 |

## 4. 非功能性需求  

| 编号 | 需求 | 估算指标 |
|------|------|----------|
| **N1** | **DAU（每日活跃用户）** | 300 万人（峰值） |
| **N2** | **QPS（查询/浏览）** | 30,000 QPS（列表、搜索） |
| **N3** | **TPS（交易撮合）** | 2,000 TPS（包含即时购买和拍卖出价） |
| **N4** | **平均响应时间** | 95% 请求 < 200 ms，99% < 500 ms |
| **N5** | **可用性** | 99.9% 月可用率（每月累计不可用时间 ≤ 43.2 分钟） |
| **N6** | **存储规模** | 交易日志 + 道具元数据 ≈ 200 TB（预计 3 年） |

> **备注**：以上指标基于假设平台同时服务 10 款热门游戏，每款游戏平均 30 万日活，峰值流量约为平峰的 3 倍。

## 5. 系统边界  

**本题范围内需要考虑的内容**  
- 用户身份验证、道具库存管理、交易撮合、支付结算、搜索/浏览、数据持久化与备份。  
- 高并发读写、数据一致性（如双写、事务）、水平扩展、容错与灾备。  
- 基础监控、报警与日志审计。

**本题范围外（不必实现）**  
- 游戏本体的道具生成、游戏服务器与平台的深度对接（仅假设已有 API）。  
- 复杂的推荐系统、社交功能（好友、聊天）。  
- 第三方支付的完整对账系统（只需抽象支付/提现接口）。  
- 法律合规（如跨境支付、未成年人监管）细节。  

## 6. 提示与追问  

1. **数据一致性**  
   - “如果玩家在下单前突然退出游戏，导致道具被冻结但未完成交易，你会如何保证库存的最终一致性？”  

2. **高并发搜索**  
   - “面对 30k QPS 的搜索请求，你会选用什么存储/索引方案？如何实现热点商品的缓存与失效？”  

3. **交易撮合与防刷**  
   - “在拍卖模式下，如何设计防止刷单/机器人抢拍的机制，同时保证撮合的低延迟？”

# 题解

## 解题思路总览  

1. **从业务角度拆解需求**：先把功能点（注册、上架、搜索、下单、结算、纠纷）和非功能点（并发、时延、可用性、容量）逐一列清楚。  
2. **估算规模**：把 DAU、QPS、TPS、存储量等转化为具体的资源需求（CPU、磁盘、网络、内存）。  
3. **画出最小可运行（MVP）原型**：只用几块服务器、单体应用、单库，验证业务流程。  
4. **逐层演进**：在 MVP 基础上，按“读/写分离 → 缓存 → 消息队列 → 服务拆分 → 多活容灾”逐步加入组件，每一步都解释**为什么**以及**不这么做会出现什么问题**。  
5. **落地到每个关键点**：  
   - **数据库选型 & 表结构**（事务性强的关系库 + 高速查询的搜索引擎）  
   - **核心 API**（RESTful + gRPC）  
   - **交易撮合**（乐观锁 + 事务 + 消息队列）  
   - **搜索**（Elasticsearch + 缓存）  
   - **安全/防刷**（验证码、限流、风控）  
6. **补齐运维需求**：监控、日志、告警、灰度发布、灾备。  
7. **准备面试追问**：从一致性、热点缓存、拍卖防刷等角度给出思路与实现细节。  

> **提示**：面试时先给出“大局观”，再根据面试官的兴趣点逐层展开。下面的章节按**从最小可用系统 → 完整高可用系统**的顺序，详细阐述每一步的设计与思考过程。

## 第一步：理解需求与规模估算  

| 维度 | 需求 | 关键指标 | 计算方式/说明 |
|------|------|----------|---------------|
| **用户** | 注册/登录、浏览、下单 | DAU ≈ 300 万峰值 | 10 款游戏 × 30 万日活 ≈ 300 万。峰值假设为平峰的 3 倍 → 峰值 DAU ≈ 900 万（用于容量上限预估） |
| **浏览** | 商品列表、搜索 | QPS ≈ 30 000 | 300 万活跃 * 10 次/分钟 ≈ 30 万 QPM ≈ 30 k QPS |
| **交易** | 即时购买 + 拍卖出价 | TPS ≈ 2 000 | 交易占活跃用户的 0.1%（300 万 × 0.001）≈ 3 k/分钟 ≈ 50 TPS，峰值取 40 倍 → 2 k TPS |
| **时延** | 95% < 200 ms, 99% < 500 ms | - | 需要缓存、异步化、热点分片等手段 |
| **可用性** | 99.9% 月可用率 | ≤ 43.2 min/月 | 设计容错、快速故障切换 |
| **存储** | 交易日志 + 道具元数据 | 200 TB / 3 年 | 估算：<br>① 交易日志（每笔 1 KB）× 2 k TPS × 3600 s × 24 h × 365 d ≈ 56 TB/年 <br>② 道具元数据（每件 0.5 KB）× 10 M 交易品种 × 3 年 ≈ 14 TB <br>③ 预留冗余 & 备份 → 约 200 TB |
| **多币种** | 人民币、游戏币 | 汇率、账本分离 | 账本采用双写（业务库 + 财务库）或事件溯源方式 |

**业务流程简述**（后面会展开）  

1. **注册/登录** → 获得 `player_id`（全局唯一）。  
2. **道具上架** → 校验玩家拥有该道具 → 冻结道具 → 生成 `listing_id`。  
3. **浏览/搜索** → 读取搜索引擎或缓存返回列表。  
4. **下单/撮合** → 检查库存/余额 → 扣款 + 冻结道具 → 生成订单 → 通过事务或分布式事务完成 “钱 + 物” 原子转移。  
5. **结算/提现** → 按日/周批量结算 → 调用第三方支付。  
6. **纠纷** → 人工或自动审计，回滚相关记录。

## 第二步：高层架构设计  

### 2.1 最小可用系统（MVP）  

```
+-------------------+      +-------------------+
|   前端 (Web/App)  | <--->|   API Gateway     |
+-------------------+      +-------------------+
                               |
                               v
                     +-------------------+
                     |   单体业务服务    |
                     | (Spring Boot)    |
                     +-------------------+
                               |
                               v
                     +-------------------+
                     |   MySQL 主库      |
                     +-------------------+
```

- **为什么这样？**  
  - 只用一套技术栈，快速验证业务流程（注册 → 上架 → 下单）。  
  - MySQL 提供强事务，满足“钱+物”原子性的最基本需求。  
- **不这么做的风险**  
  - 单点故障、无法支撑 30k QPS、扩展困难。  

### 2.2 逐步演进的目标架构  

```
                                 +-------------------+
                                 |   CDN / WAF       |
                                 +-------------------+
                                          |
+-------------------+      +-------------------+      +-------------------+
|   前端 (Web/App)  |<---->|   API Gateway     |<---->|   统一鉴权中心    |
+-------------------+      +-------------------+      +-------------------+
            |                         |                         |
            v                         v                         v
   +----------------+        +----------------+        +----------------+
   |   商品搜索服务 |<------>|   订单/撮合服务|<------>|   用户/资产服务|
   +----------------+        +----------------+        +----------------+
            |                         |                         |
            v                         v                         v
   +----------------+        +----------------+        +----------------+
   |   Elasticsearch|        |   Kafka/Queue  |        |   MySQL (业务) |
   +----------------+        +----------------+        +----------------+
            |                         |                         |
            v                         v                         v
   +----------------+        +----------------+        +----------------+
   |   Redis Cache  |        |   结算服务     |        |   财务库 (PostgreSQL)|
   +----------------+        +----------------+        +----------------+
```

**关键模块说明**  

| 模块 | 主要职责 | 选型理由 |
|------|----------|----------|
| **API Gateway** | 统一入口、限流、路由、灰度、日志 | Kong / Spring Cloud Gateway，轻量且支持插件 |
| **统一鉴权中心** | 登录态校验、JWT 签发、OAuth2 | 采用 JWT + Redis 黑名单，避免每次请求 DB |
| **用户/资产服务** | 注册、登录、玩家信息、钱包、道具库存 | 业务库（MySQL） + Redis 读缓存，事务保证一致性 |
| **商品搜索服务** | 商品列表、关键词、过滤、排序 | Elasticsearch 高速全文检索 + Redis 热点缓存 |
| **订单/撮合服务** | 下单、出价、拍卖、撮合、事务控制 | 主动式 (乐观锁) + 消息队列（Kafka）实现异步结算 |
| **结算服务** | 批量结算、提现、对账 | 采用事件驱动，财务库采用 PostgreSQL 强事务 |
| **缓存层** | 读取热点数据、限流计数 | Redis 读/写分离，使用 LRU + TTL |
| **消息队列** | 解耦高并发写、保证最终一致性 | Kafka 持久化、可回放、分区扩展性 |
| **监控/日志** | Prometheus + Grafana、ELK、链路追踪 (Jaeger) | 可观测性是高可用的前提 |

### 2.3 架构演进的原则  

1. **先保证业务正确性**（事务、幂等），后提升性能（缓存、异步）。  
2. **读写分离**：浏览/搜索是读多写少，使用搜索引擎 + 缓存。交易是写多读少，使用消息队列缓冲。  
3. **最小化跨服务调用**：把高度耦合的“钱+物”转移放在同一服务（订单服务）内完成事务，其他服务仅负责查询。  
4. **故障隔离**：每个业务域独立部署，故障不跨域蔓延。

## 第三步：数据库设计  

### 3.1 关系型数据库（业务库）  

| 表名 | 说明 | 主键 | 关键索引 | 备注 |
|------|------|------|----------|------|
| **users** | 玩家基本信息 | `user_id` (BIGINT) | `email`, `mobile`, `third_party_id` | 登录信息、状态 |
| **wallets** | 玩家钱包（多币种） | `wallet_id` (BIGINT) | `user_id`, `currency` | 冗余 `balance`，乐观锁 `version` |
| **items** | 玩家拥有的道具（冻结/解冻） | `item_id` (BIGINT) | `user_id`, `game_id`, `status` | `status`: `NORMAL`, `FROZEN`, `SOLD` |
| **listings** | 上架商品信息 | `listing_id` (BIGINT) | `game_id`, `item_id`, `status`, `price` | `status`: `ACTIVE`, `SOLD_OUT`, `CANCELLED` |
| **orders** | 交易订单（即时购买） | `order_id` (BIGINT) | `buyer_id`, `listing_id`, `status` | `status`: `PENDING`, `SUCCESS`, `FAILED` |
| **auctions** | 拍卖信息 | `auction_id` (BIGINT) | `listing_id`, `status` | `status`: `RUNNING`, `FINISHED`, `CANCELLED` |
| **bids** | 拍卖出价 | `bid_id` (BIGINT) | `auction_id`, `buyer_id`, `price` | 乐观锁防止并发超价 |
| **transactions** | 资金流水（审计） | `tx_id` (BIGINT) | `user_id`, `order_id`, `type` | `type`: `DEBIT`, `CREDIT` |
| **withdrawals** | 提现申请 | `withdraw_id` (BIGINT) | `user_id`, `status` | `status`: `PENDING`, `SUCCESS`, `FAIL` |

**事务模型**（买家下单）  

```
START TRANSACTION
  -- 1. 检查并锁定买家钱包（SELECT ... FOR UPDATE）
  -- 2. 检查并锁定卖家道具（SELECT ... FOR UPDATE）
  -- 3. 扣除买家余额
  -- 4. 冻结卖家道具 -> status = SOLD
  -- 5. 写入 orders、transactions
COMMIT
```

> **为什么使用行锁（FOR UPDATE）而不是乐观锁？**  
> - 交易 TPS 只有 2k，行锁冲突概率低，能够保证“一次交易只会成功一次”。  
> - 乐观锁适用于冲突概率极低的场景（如浏览），这里需要强一致性。  

### 3.2 搜索引擎（Elasticsearch）  

| 索引 | 字段 | 类型 | 用途 |
|------|------|------|------|
| `items_index` | `listing_id` | keyword | 主键 |
| | `game_id` | keyword | 游戏过滤 |
| | `title` | text (ik_max_word) | 关键词搜索 |
| | `price` | double | 范围过滤 |
| | `rarity` | keyword | 稀有度过滤 |
| | `status` | keyword | 只返回 `ACTIVE` |
| | `updated_at` | date | 排序（最新） |

- **同步机制**：业务库写成功后，发布 **Kafka** `ItemUpdated` 事件，搜索服务消费后 **更新 ES**。  
- **为何采用近实时（<1s）而非同步？**  
  - 防止业务库写成功但搜索未及时更新导致买家看到已售出商品。通过 **Kafka** 保证 **至少一次** 投递，搜索服务使用 **幂等 upsert**（`_id = listing_id`）避免重复。  

### 3.3 财务库（PostgreSQL）  

- **专门用于审计、对账**，与业务库分离，避免业务库锁竞争。  
- 表结构与 `transactions` 类似，但多了 `audit_status`、`settlement_batch_id`。  

### 3.4 分库分表  

- **用户表**：按 `user_id` 哈希分库（4~8 库）  
- **商品表**：按 `game_id` 分库（10 库），每库内部再按 `listing_id` 范围分表（每表 1~2 亿行）  
- **原因**：单库写入 2k TPS，单表容量 200 TB 超出单机磁盘上限，分库分表可以水平扩展。

## 第四步：核心 API 设计  

> **原则**：RESTful + gRPC（内部高性能），所有请求返回统一的 **Result** 对象，错误码统一。  

### 4.1 通用返回结构  

```json
{
  "code": 0,               // 0 成功，非0 为业务错误
  "msg": "OK",
  "data": { ... }          // 业务数据
}
```

### 4.2 用户/鉴权 API  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/auth/register` | 邮箱/手机号/第三方注册 | `{type:"email", email:"a@b.com", pwd:"xxx"}` | `{userId, token}` |
| `POST` | `/api/v1/auth/login` | 登录 | `{type:"email", email:"a@b.com", pwd:"xxx"}` | `{userId, token}` |
| `POST` | `/api/v1/auth/refresh` | 刷新 JWT | `{refreshToken}` | `{newToken}` |
| `GET`  | `/api/v1/users/{userId}` | 查询玩家信息（需 token） | - | `{userId, nickname, avatar, ...}` |

- **安全**：密码使用 **bcrypt**，JWT 有效期 15 min，Refresh Token 30 days，存 Redis 黑名单防止注销后 token 继续使用。  

### 4.3 商品上架 API  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/listings` | 创建上架（即时或拍卖） | `{itemId, price, mode:"FIXED|AUCTION", startTime, endTime}` | `{listingId, status:"ACTIVE"}` |
| `GET`  | `/api/v1/listings/{listingId}` | 查询上架详情 | - | `{listingId, itemInfo, price, mode, status}` |
| `DELETE`| `/api/v1/listings/{listingId}` | 取消上架（仅在未成交前） | - | `{code, msg}` |

- **业务流程**：API → 调用 **ItemService** → 检查 `items.status = NORMAL` → `UPDATE items SET status='FROZEN'` → 写入 `listings` → 发送 `ItemListed` Kafka 事件。  

### 4.4 搜索/浏览 API  

| 方法 | 路径 | 描述 | 参数 | 响应 |
|------|------|------|------|------|
| `GET` | `/api/v1/search` | 关键字、过滤、分页 | `q, gameId, rarity, minPrice, maxPrice, sort, page, size` | `{total, items:[...]}` |
| `GET` | `/api/v1/categories/{gameId}` | 按游戏返回热门分类 | - | `{categories:[...]}` |

- **实现细节**：API 先查询 **Redis HotCache**（`TOP10_GAME_{gameId}`），若未命中再查询 ES，结果写回缓存（TTL 30 s）。  

### 4.5 订单/撮合 API  

#### 4.5.1 即时购买  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/orders` | 立即购买 | `{listingId, buyerId}` | `{orderId, status:"PENDING"}` |

- **内部实现**（伪代码）：

```java
// OrderService.createOrder()
@Transactional
public Order createOrder(Long buyerId, Long listingId) {
    // 1. 读取并锁定买家钱包
    Wallet buyer = walletDao.selectForUpdate(buyerId);
    // 2. 读取并锁定卖家道具
    Listing listing = listingDao.selectForUpdate(listingId);
    if (listing.getStatus() != ACTIVE) throw new BusinessException("已下架");
    // 3. 检查余额
    if (buyer.getBalance() < listing.getPrice()) throw new InsufficientBalance();
    // 4. 扣款
    buyer.setBalance(buyer.getBalance() - listing.getPrice());
    walletDao.update(buyer);
    // 5. 冻结/转移道具
    Item item = itemDao.selectForUpdate(listing.getItemId());
    item.setOwnerId(buyerId);
    item.setStatus(NORMAL);
    itemDao.update(item);
    // 6. 生成订单、流水
    Order order = new Order(...);
    orderDao.insert(order);
    transactionDao.insert(new Tx(buyerId, -listing.getPrice(), order.getId()));
    transactionDao.insert(new Tx(sellerId, +listing.getPrice(), order.getId()));
    // 7. 发送消息给结算服务（异步对账）
    kafkaTemplate.send("order_success", order);
    return order;
}
```

- **幂等**：若买家重复提交，`orderDao` 通过唯一键 `buyerId+listingId` 防止重复创建；业务层返回已有订单状态。  

#### 4.5.2 拍卖出价  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/auctions/{auctionId}/bids` | 出价 | `{buyerId, price}` | `{bidId, status}` |
| `GET`  | `/api/v1/auctions/{auctionId}` | 查看拍卖状态 | - | `{auctionId, highestBid, endTime, status}` |

- **实现要点**：  
  - **乐观锁**：`bids` 表使用 `version`，每次出价 `UPDATE bids SET price=?, version=version+1 WHERE auctionId=? AND version=?`。  
  - **防刷**：每个用户在同一拍卖的出价间隔至少 3 s（在业务层做限流），并使用 **验证码**（滑动验证）+ **行为风控**（IP、设备指纹）。  
  - **撮合**：拍卖结束时，**Scheduler**（Kafka Streams 或 Quartz）读取 `auction_end` 事件，执行最终结算与道具转移，同即时购买的事务逻辑保持一致。  

### 4.6 结算/提现 API  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/withdrawals` | 提现申请 | `{userId, amount, currency, targetAccount}` | `{withdrawId, status:"PENDING"}` |
| `GET`  | `/api/v1/withdrawals/{id}` | 查询提现进度 | - | `{status, txId, failReason}` |

- **结算批次**：每天凌晨，结算服务读取 `order_success`、`auction_success` 事件，聚合同一用户同一币种的收入，生成 **批次**，调用第三方支付 SDK（异步回调），成功后更新 `withdrawals.status = SUCCESS` 并写入财务库。

## 第五步：详细组件设计  

### 5.1 API Gateway & 统一鉴权  

- **技术选型**：Kong（插件生态丰富）或 Spring Cloud Gateway（Java 生态兼容）。  
- **功能**：  
  - **路由**：根据 URL 前缀转发至对应微服务。  
  - **限流**：全局 QPS 30k，使用 **Leaky Bucket**；对登录/下单做 **IP+用户** 双重限流。  
  - **安全**：JWT 校验插件 + 黑名单 Redis。  
  - **日志**：统一请求/响应日志上报至 ELK。  

### 5.2 缓存层（Redis）  

| 场景 | 缓存键 | TTL | 失效策略 |
|------|--------|-----|----------|
| **热点商品列表** | `HOT_LIST_GAME_{gameId}` | 30 s | 定时刷新（每 30 s） |
| **用户钱包** | `WALLET_{userId}` | 5 min | 写穿透：更新后立即写回 |
| **登录态** | `TOKEN_BLACKLIST_{jti}` | 剩余有效期 | 自动过期 |
| **防刷计数** | `BID_COUNT_{userId}_{auctionId}` | 3 s | 原子 INCR + EXPIRE |

- **为什么使用 Redis 而不是 Memcached？**  
  - Redis 支持 **持久化**、**事务**（Lua 脚本）以及 **位图/HyperLogLog**，更适合实现限流、风控等复杂业务。  

### 5.3 消息队列（Kafka）  

| Topic | 生产者 | 消费者 | 关键属性 |
|-------|--------|--------|----------|
| `ItemListed` | Listing Service | Search Service | 分区：`gameId` |
| `OrderSuccess` | Order Service | Settlement Service, Audit Service | 分区：`buyerId` |
| `AuctionEnded` | Scheduler | Order Service (final settle) | 分区：`auctionId` |
| `WithdrawalReq` | Withdrawal Service | Payment Gateway | 分区：`currency` |

- **为何选 Kafka 而不是 RabbitMQ？**  
  - **高吞吐**（TPS > 10k），**持久化**保证不丢失，**分区**天然支持水平扩展，**消费组**可以实现 **水平消费**（如多实例结算）。  

### 5.4 搜索服务（Elasticsearch）  

- **集群规模**：  
  - **节点数**：3 主节点 + 2 只读副本 → 5 节点。  
  - **Shard**：每个 `items_index` 按 `game_id` 设定 5 个 primary shard（共 50 shards），每个 shard 30 GB 左右，满足 200 TB 存储需求。  
- **热点缓存**：使用 **Elastic Cache**（自带的 **node query cache**） + **Redis** 二级缓存。  
- **失效机制**：商品下架或成交后，业务服务发送 `ItemSold` 事件，搜索服务立即 **delete** 该文档，避免买家看到已售商品。  

### 5.5 订单/撮合服务  

- **部署方式**：Spring Boot 微服务，水平伸缩（K8s Deployment + HPA），每实例 2 CPU / 4 GB。  
- **事务实现**：  
  - **即时购买**：单库事务（MySQL）+ **XA**（如果跨库）。  
  - **拍卖结算**：在拍卖结束的 Scheduler 中开启 **分布式事务**（使用 **Seata**）或 **两阶段提交**（先锁定资金/道具，再提交）。  
- **幂等性**：所有写操作在数据库层都有唯一约束（如 `buyerId+listingId`），业务层返回已存在记录的状态。  

### 5.6 结算服务  

- **批处理框架**：使用 **Apache Flink** 或 **Spark Structured Streaming** 读取 `OrderSuccess`、`AuctionEnded` 流，聚合计算每日/每小时结算报表。  
- **对账**：定时把财务库数据与第三方支付对账文件比对，异常记录至 **Audit Service**。  

### 5.7 监控、日志、告警  

| 组件 | 功能 | 选型 |
|------|------|------|
| **Metrics** | QPS、TPS、延迟、错误率 | Prometheus + Grafana |
| **Tracing** | 跨服务调用链路 | OpenTelemetry + Jaeger |
| **Logging** | 业务日志 + 错误日志 | ELK（Filebeat → Logstash → Kibana） |
| **Alert** | 机器故障、业务异常阈值 | Alertmanager + PagerDuty (或企业微信) |
| **Health Check** | k8s liveness/readiness | Spring Actuator |

- **为什么要全链路追踪？**  
  - 在高并发系统中，一个请求会跨 5~8 个微服务，单点日志难以定位根因。Tracing 能快速定位 **慢点**（如搜索超时、订单事务锁等待）。

## 第六步：扩展性与高可用设计  

### 6.1 读写分离  

- **读取**（浏览、搜索）走 **CDN → API GW → 缓存 → ES**，几乎不触及业务库。  
- **写入**（上架、下单、拍卖出价）走 **API GW → Order/Listing Service → MySQL**，并通过 **Kafka** 异步同步至 **ES**、**财务库**。  

### 6.2 水平扩展  

| 维度 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **Web层** | 增加 API GW 实例 + 负载均衡 (LVS/Nginx) | QPS 30k → 60k 只需双倍实例 |
| **业务服务** | K8s HPA 根据 CPU/Latency 自动伸缩 | TPS 2k → 4k |
| **数据库** | 分库分表 + 读写分离（主从） | 读库 5×复制，提高查询并发 |
| **搜索** | 增加 ES 节点、调大 shard 数 | 30k QPS 持续低延迟 |
| **Kafka** | 增加 broker，提升 partitions | 2k TPS + 30k QPS 消息流 |

### 6.3 容错与灾备  

| 场景 | 方案 |
|------|------|
| **单机故障** | 多实例 + 自动重启（K8s） |
| **节点网络分区** | **Circuit Breaker** (Hystrix/Resilience4j) 保护下游服务 |
| **数据库主库宕机** | 主从自动切换 (MHA) + **GTID** 同步 |
| **跨地域灾备** | 同步备份至另一云区域，使用 **Active‑Active**（双活）部署 API GW 与业务服务，采用 **DNS 轮询 + 健康检查** |
| **数据恢复** | 交易日志 + **增量备份**（每 5 min） + **PITR**（Point‑In‑Time Recovery） |
| **支付回滚** | 采用 **补偿事务**（Saga）在提现失败时自动回滚钱包余额 |

### 6.4 防刷与风控  

1. **验证码**：登录、下单、出价前强制滑动或图形验证码（使用 **GeeTest**）。  
2. **行为分析**：基于 **IP、DeviceId、UA、历史行为** 打分，超过阈值直接阻断或加入 **人工审计队列**。  
3. **频率限制**：  
   - 同一用户同一商品 1 s 内只能下单一次（Redis `SETNX` + EXPIRE）。  
   - 同一 IP 每分钟最多 100 次搜索请求（Leaky Bucket）。  
4. **机器人检测**：结合 **浏览器指纹**、**鼠标轨迹**，可使用开源 **BotDetect**。  

### 6.5 数据一致性策略  

| 场景 | 强一致性 | 最终一致性 | 说明 |
|------|----------|------------|------|
| **资金/道具转移** | 事务（行锁） | - | 必须强一致，避免“钱扣了道具没到账”。 |
| **搜索索引** | - | Kafka → ES 异步 | 业务成功后即可返回，搜索稍后同步。 |
| **用户钱包缓存** | - | 缓存失效 + 双写 | 写库成功后立即更新 Redis，若写库失败不更新。 |
| **交易日志** | - | 追加写 Kafka + MySQL 事务日志 | 日志可稍后同步到审计库。 |

## 第七步：常见面试追问与回答  

### Q1. **玩家在下单前退出，导致道具被冻结但未完成交易，如何保证库存最终一致性？**  

**回答思路**  

1. **使用分布式锁 + 超时释放**：  
   - 当玩家点击“下单”，系统先在 Redis 中 **SETNX** `LOCK_ITEM_{itemId}`，并设置 **TTL（30 s）**。  
   - 若在 TTL 内完成事务，正常扣款、转移道具；若超时未完成，则 **自动释放锁**，并把 `item.status` 从 `FROZEN` 改回 `NORMAL`（通过后台**补偿任务**）。  

2. **补偿任务（Saga）**：  
   - 每笔 “冻结道具” 产生一条 **PendingOrder** 记录，状态 `INIT`。  
   - 订单服务在 30 s 内必须将状态更新为 `SUCCESS` 或 `FAILED`。  
   - 若超过 30 s 未更新，**Scheduler** 执行补偿：解冻道具、标记订单为 `TIMEOUT`，并发送通知给玩家。  

3. **幂等性保证**：  
   - 冻结操作和解冻操作都有唯一键 `itemId`，重复执行不影响最终状态。  

**不这么做的后果**  
- 道具一直处于 `FROZEN`，玩家无法再次上架或使用，导致用户体验极差，甚至出现“道具丢失”。  

### Q2. **30k QPS 的搜索请求，选用什么存储/索引方案？如何实现热点商品的缓存与失效？**  

**回答思路**  

1. **存储/索引**：  
   - **Elasticsearch**：倒排索引 + 分布式查询，天然支持 **全文检索**、**聚合过滤**。  
   - **分片**：按照 `game_id` + `listing_id` hash 分片，确保热点商品不会集中在单个 shard。  

2. **缓存层**：  
   - **一级缓存**：ES 自带 **node query cache**（针对相同查询的结果）。  
   - **二级缓存**：Redis `HOT_LIST_GAME_{gameId}`，缓存 **前 100 条** 热门商品（基于最近 5 min PV 排序）。  
   - **TTL**：30 s，保证新上架商品能及时上榜。  

3. **失效策略**：  
   - **主动失效**：商品状态变更（下架、成交）时，业务服务发布 `ItemSold` 事件，搜索服务 **删除对应文档**，并 **清除 Redis 热点键**。  
   - **定时刷新**：每分钟统计一次 PV，重新生成热点列表（使用 **Redis Sorted Set**）。  

4. **高并发支撑**：  
   - **查询路由**：API GW 将搜索请求直接转发到 **搜索服务**（无业务层），减少网络跳数。  
   - **并发控制**：对搜索服务使用 **ThreadPool**，并在 ES 集群上开启 **search.max_concurrent_shard_requests** 限制单查询占用的 shard 数。  

**不采用上述方案的风险**  
- 使用关系库做搜索会导致 **全表扫描**，在 30k QPS 时 CPU、IO 将被瞬间耗尽，响应时间远超 500 ms。  

### Q3. **拍卖模式下，如何防止刷单/机器人抢拍，同时保证撮合低延迟？**  

**回答思路**  

1. **身份验证 + 行为风控**  
   - 登录后绑定 **设备指纹**，每次出价必须携带 **JWT + X-Device-ID**。  
   - 使用 **滑动验证码**（仅在首次出价或异常行为时触发）。  

2. **限流 & 配额**  
   - **每用户每秒最多 1 次出价**（Redis `INCR` + `EXPIRE 1`），超过直接返回 `429 Too Many Requests`。  
   - **每用户每场拍卖最高出价次数** 设为 10 次，防止无限刷价。  

3. **排队撮合**  
   - **出价请求**先写入 **Kafka**（`auction_bid_{auctionId}`）分区，保证 **顺序消费**。  
   - **撮合服务**消费该分区，使用 **乐观锁**检查当前最高价是否被同一用户抢先。  

4. **延迟补偿**  
   - 为防止网络抖动导致的 **“抢先”**，在撮合时加入 **30 ms 随机延迟窗口**（类似拍卖的 “防抢”），所有出价在窗口结束后统一排序。  

5. **结果返回**  
   - **即时返回**：出价成功返回 `accepted`，但实际是否成为最高价在 **撮合结束后** 通过 **WebSocket** 推送给客户端。  

6. **异常检测**  
   - 若同一 IP、同一设备在极短时间内出价次数异常，自动加入 **黑名单**（Redis），后续请求直接拒绝。  

**不做风控的后果**  
- 机器人可以瞬间出价抢占，导致真实玩家体验极差，平台信誉受损；同时大量无效请求会压垮撮合服务。  

### Q4. **如果业务需要在 5 分钟内把交易日志恢复到 1 秒前的状态，如何设计备份与恢复？**  

**回答思路**  

1. **双写日志**：在订单事务提交时，同步写入 **MySQL 事务日志**（表 `order_log`）和 **Kafka**（topic `order_log`）。  
2. **增量备份**：每 5 min 使用 **MySQL binlog** + **Kafka offset** 做一次增量快照，存到对象存储（OSS / S3）。  
3. **恢复流程**：  
   - 先恢复最近一次完整备份（如每日全量备份）。  
   - 再读取最近的增量快照并 **回放** 到 **MySQL**（使用 **MySQL Shell** 或 **Canal**）。  
   - 若需要回到 **1 秒前**，直接读取 **Kafka** 中最近 1 秒的日志（Kafka 保留 24 h），使用 **事务补偿** 把状态回滚。  

**不做增量备份的风险**  
- 完全恢复只能回到上一次全量备份（可能相差数小时），导致业务数据不一致，难以满足监管要求。

## 心得与反思  

### 1. 本题最难的设计决策  

| 决策 | 困难点 | 思考过程 |
|------|--------|----------|
| **事务一致性 vs 高并发** | 交易需要 **强一致性**（钱+物原子转移），但高并发写入会导致锁竞争、性能下降。 | - 先评估业务峰值（2k TPS）是否能接受行锁。 <br> - 通过 **分库分表** 降低单表并发。 <br> - 采用 **乐观锁 + 重试** 作为备选方案，在冲突率极低时提升并发。 |
| **搜索热点与实时性** | 玩家对商品的实时性要求高，搜索 QPS 30k，且商品状态频繁变化（下单、撤单）。 | - 采用 **Elasticsearch** 做近实时索引，配合 **Kafka** 异步同步，保证 **≤1 s** 延迟。 <br> - 为防止“已售商品仍在列表”，引入 **双写 + 幂等删除**，并在业务层做 **缓存失效**。 |

### 2. 新手最容易犯的错误  

1. **把所有业务都塞进单体数据库**  
   - 结果：单库写入瓶颈、扩展困难、灾备成本高。  
   - 建议：**先分库分表**，再根据访问热点拆分服务。  

2. **忽视幂等性与防重放**  
   - 在高并发下，客户端可能因网络抖动多次提交同一请求，导致 **双扣款**、**双上架**等问题。  
   - 解决：**业务唯一键**（如 `buyerId+listingId`），**幂等 token**，**幂等写入**（`INSERT ... ON DUPLICATE KEY UPDATE`）。  

### 3. 学习建议与可延伸方向  

| 方向 | 推荐学习资源 | 关键点 |
|------|--------------|--------|
| **分布式事务** | 《Designing Data-Intensive Applications》章节、Seata 官方文档 | 两阶段提交、Saga、事务补偿 |
| **高性能搜索** | Elasticsearch 官方教程、《Elasticsearch 实战》 | 倒排索引、分片路由、冷热分层 |
| **消息队列与可靠消费** | Kafka 官方文档、Confluent 课程 | Exactly‑once、消费组、幂等生产 |
| **微服务治理** | Spring Cloud Alibaba、Istio 官方 | 服务注册、熔断、限流、灰度发布 |
| **监控与可观测性** | Prometheus + Grafana 实战、OpenTelemetry 入门 | 指标、追踪、日志聚合 |
| **防刷与风控** | 《Real-Time Fraud Detection》、Google reCAPTCHA 文档 | 行为评分、设备指纹、验证码集成 |

> **温馨提示**：在面试时，**先给出整体思路**（模块划分、数据流），**再逐层展开**（数据库、缓存、消息、容错），**最后回答追问**。保持条理清晰、用时控制在 10‑15 分钟，细节可以在面试官追问时补充。
