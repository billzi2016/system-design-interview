# 第 89 天：设计 推荐系统

> 生成日期：2026-02-26

---

## 题目背景  
推荐系统为一个面向海量用户的内容平台（如短视频、资讯或电商）提供个性化推荐。系统需要实时捕获用户行为并基于机器学习模型输出用户感兴趣的物品列表，以提升用户活跃度和转化率。

## 面试场景设定  
> **面试官**： “我们现在要设计一个每天活跃用户约 5000 万、每秒产生上千条行为日志的实时推荐系统。请你从零开始，说明整体架构、关键技术选型以及如何保证高可用和低延迟。先把核心的功能块和数据流给我梳理一下。”

## 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| 1 | **用户行为采集**：实时收集用户的点击、浏览、点赞、搜索、购买等行为，并保证不丢失。 |
| 2 | **特征计算 & 存储**：对用户、物品、上下文等进行特征抽取、聚合（如最近 7 天活跃度、兴趣向量），并持久化用于模型输入。 |
| 3 | **召回（Recall）** | 基于召回策略（协同过滤、内容相似、热点）快速生成候选集合（10k–100k 条）。 |
| 4 | **排序（Ranking）**：使用离线/在线混合模型（GBDT、DeepCTR、RL）对候选集合进行打分，返回前 N（如 20）条给前端。 |
| 5 | **实时反馈回流**：将用户对推荐结果的点击、曝光等反馈实时写回，用于模型在线学习或特征更新。 |
| 6 | **监控与AB测试**：提供推荐质量（CTR、CVR、GMV）和系统健康（错误率、延迟）监控，并支持实验分流。 |

## 非功能性需求  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| DAU（日活跃用户） | 5,000 万 | 高并发用户基数 |
| QPS（行为上报） | 2,000 qps (峰值 5,000 qps) | 包括点击、曝光等多种事件 |
| 推荐延迟 | ≤ 100 ms（95% 请求） | 前端感知的响应时间 |
| 系统可用性 | 99.9%（月均累计宕机 ≤ 43 分钟） | SLA 要求 |
| 存储规模 | 行为日志 30 TB/天，特征库 5 TB | 包括原始日志、聚合特征、模型快照等 |
| 吞吐量（召回） | ≤ 10 ms 生成 10k 候选 | 高效召回层的性能要求 |

## 系统边界  

**本题范围内**（需要设计）：  
- 行为采集、实时流处理、特征计算、召回、排序、实时反馈回流。  
- 数据存储（日志、特征库、模型），以及监控/AB测试框架。  
- 高可用、扩容、容错、延迟控制等关键技术点。

**不考虑**（可在答案中略过）：  
- 推荐模型的离线训练细节（如大规模分布式训练框架）。  
- 前端 UI/UX 设计与展示层实现。  
- 法规合规、隐私脱敏等合规性实现（只需提及但不需细化）。  
- 跨地域多数据中心的灾备（除非主动提及高可用方案）。  

## 提示与追问  

1. **数据倾斜处理**：如果某些热门物品的点击量远高于其他，如何在召回和排序阶段避免热点导致的性能瓶颈？  
2. **模型在线更新**：在保证低延迟的前提下，如何实现模型的增量学习或在线微调？需要哪些数据管道和回滚机制？  
3. **容量规划**：假设业务在一年内增长 2 倍，存储和计算资源应如何预留？请说明扩容策略（水平/垂直）以及对已有系统的影响。

# 题解

## 解题思路总览  

1. **先把业务拆成最小可运行的闭环**：  
   - 行为采集 → 实时流处理 → 召回 → 排序 → 返回结果。  
   - 只要这 5 步能在 100 ms 内完成，就满足最基本的需求。  

2. **在最小闭环上逐步加上非功能需求**：  
   - 数据持久化、特征库、日志备份、监控、AB‑Test。  
   - 再加入高可用（多副本、容错）和弹性扩容（水平扩展）。  

3. **每加入一个新功能或改进，都要回答**：  
   - **为什么要这么做**（业务、性能或可靠性原因）。  
   - **如果不这么做会出现什么问题**（丢数据、延迟爆炸、单点故障等）。  

下面按照 **从需求 → 高层架构 → 细节实现 → 可用性 & 扩展** 的顺序，层层递进地展开。

## 第一步：理解需求与规模估算  

| 需求 | 关键指标 | 业务意义 |
|------|----------|----------|
| 用户行为采集 | 2 k‑5 k QPS，日志 30 TB/天 | 必须**不丢失**，后续特征、模型都依赖它 |
| 特征计算 & 存储 | 7 天窗口聚合，特征库 5 TB | 实时特征是排序模型的输入 |
| 召回 | ≤10 ms 生成 10k‑100k 候选 | 必须在毫秒级返回候选集合，后面排序才能在 100 ms 内结束 |
| 排序 | 95% 延迟 ≤100 ms，返回前 N (≈20) | 前端体验关键 |
| 实时反馈回流 | 实时写回，支持在线学习 | 闭环，使模型随用户兴趣快速迭代 |
| 监控 & AB‑Test | CTR、错误率、延迟等 | 保障质量、支持实验迭代 |

### 规模估算（基于 5,000 万 DAU）

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日活跃用户** | 5,000 万 | - |
| **每秒行为上报** | 2,000 qps (峰值 5,000 qps) | 约 4.3 亿 条/天 |
| **日志体积** | 100 KB/条（含原始字段） | 30 TB/天 |
| **特征库** | 1 KB/用户 × 5,000 万 ≈ 50 GB，+ 物品特征 4 TB = 5 TB | - |
| **召回吞吐** | 10 k 条候选/请求 × 2 k qps = 20 M 条/秒 | 需要高效的向量检索或倒排索引 |
| **排序计算** | 20 条返回 × 2 k qps = 40 k 次模型推理/秒 | 可用在线推理服务（GPU/CPU） |

> **小结**：日志量大、QPS 高、延迟要求严苛，必须采用 **流式处理 + 分布式缓存 + 多副本存储** 的组合。

## 第二步：高层架构设计  

下面先给出 **最小可用（MVP）** 架构，随后在每一层标注“后续可以怎样演进”。  

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  前端/APP    │────►│  行为采集 SDK│────►│  Kafka (日志)│
└─────────────┘      └─────────────┘      └─────┬───────┘
                                                │
                ┌─────────────────────────────────┼─────────────────────────────────┐
                │                                 │                                 │
          ┌─────▼─────┐                     ┌─────▼─────┐                     ┌─────▼─────┐
          │ Flink/Flink│  实时特征聚合 & 召回   │ 召回服务  │   候选集合   │ 排序服务 │  返回20条  │
          └─────┬─────┘                     └─────┬─────┘                     └─────┬─────┘
                │                                 │                                 │
          ┌─────▼─────┐                     ┌─────▼─────┐                     ┌─────▼─────┐
          │  Redis / │   实时特征查询/缓存   │  Elasticsearch│  向量检索/倒排  │  TensorFlow │  在线模型   │
          │  TiKV    │                     │  (或 Milvus) │  (或 Faiss)   │  Serving   │  推理      │
          └─────┬─────┘                     └─────┬─────┘                     └─────┬─────┘
                │                                 │                                 │
          ┌─────▼─────┐                     ┌─────▼─────┐                     ┌─────▼─────┐
          │  MySQL / │   持久化用户特征    │  HBase /  │  长期特征、模型 │  Prometheus│ 监控/告警 │
          │  Postgre │                     │  ClickHouse│  存储          │            │           │
          └───────────┘                     └───────────┘                     └───────────┘
```

### 关键技术选型解释  

| 模块 | 备选技术 | 选型理由 | 不选的风险 |
|------|----------|----------|------------|
| **行为采集** | SDK → HTTP/HTTPS + Kafka Producer | Kafka 天然 **高吞吐、持久、零丢失**，且可在生产者端做幂等写入 | 直接写入 DB 会产生写放大、延迟 |
| **实时流处理** | Flink / Spark Structured Streaming | Flink 支持 **Exactly‑once** 语义、低延迟（< 5 ms），丰富的窗口聚合算子 | Spark 延迟稍高，且在超高 QPS 场景需要更多调优 |
| **特征存储** | Redis (热点) + TiKV / HBase (冷) | Redis 超快读写，适合 **实时召回/排序** 使用；TiKV/HBase 支持 **水平扩展、持久化** | 只用 Redis 会因容量受限导致 OOM |
| **召回** | Elasticsearch (倒排) / Milvus / Faiss (向量) | ES 处理 **基于标签、热门、协同过滤**；Milvus/Faiss 处理 **Embedding 相似度**，都支持 **分布式** | 单一检索方案会在热点物品上产生**热点热点**，导致节点压垮 |
| **排序** | TensorFlow Serving / Pytorch Serve / XGBoost Model Server | 提供 **统一的 RPC 接口**、**模型热更新**，支持 **GPU/CPU 自动弹性伸缩** | 自研推理服务易出现 **内存泄漏、版本不兼容** |
| **监控** | Prometheus + Grafana + Alertmanager | 开源、易集成、支持 **服务级别指标（SLO）** 监控 | 若只靠日志，难以做到**秒级告警** |

## 第三步：数据库设计  

### 1. 行为日志（原始）  

| 表/Topic | 主键 | 关键字段 | 存储介质 |
|----------|------|----------|----------|
| `behavior_raw` (Kafka Topic) | `event_id` (UUID) | `user_id, item_id, event_type, timestamp, context_json` | Kafka + HDFS (Parquet) |
| `behavior_day_partition` (ClickHouse) | `date + event_id` | 同上 | ClickHouse（列式，支持低成本全表扫描） |

- **为什么使用 Kafka + ClickHouse**：Kafka 保证 **实时不丢失**，ClickHouse 为后续 **离线特征计算** 提供低成本的 **大规模查询**。  
- **不使用 MySQL**：单表写入量会远超其水平扩展能力。

### 2. 实时特征库  

| Key | Value | TTL | 存储 |
|-----|-------|-----|------|
| `user:{user_id}:profile` | JSON（兴趣向量、最近 7 天活跃度、曝光计数） | 30 min | Redis (Cluster) |
| `item:{item_id}:profile` | JSON（品类向量、热度、CTR） | 1 h | TiKV / HBase |

- **TTL 设计**：特征随时间衰减，避免缓存雪崩。  
- **热点缓存**：对活跃用户/热门物品使用 **热点缓存**，其余从 TiKV 拉取。

### 3. 召回索引  

| 索引类型 | 技术实现 | 关键字段 |
|----------|----------|----------|
| **倒排索引**（基于标签、品类） | Elasticsearch | `item_id, category, tags, hot_score` |
| **向量索引**（Embedding） | Milvus / Faiss | `item_id, embedding_vector` |
| **热点热点过滤** | Redis Sorted Set | `hot_item_score`（实时更新） |

- **为什么双索引**：倒排快速过滤粗排，向量相似度提供 **细粒度个性化**。  
- **热点过滤**：在召回阶段先剔除 **极端热门**（如“热门榜单”）的前 0.1%，防止 **热点放大**。

### 4. 排序模型元数据  

| 表 | 主键 | 字段 |
|----|------|------|
| `model_meta` | `model_id` | `version, create_time, status, description, rollout_ratio` |
| `model_params` | `model_id` + `shard_id` | **二进制文件**（存放在对象存储 OSS） |

- **模型滚动发布**：`rollout_ratio` 用于 AB‑Test（例如 10% 流量走新模型）。

## 第四步：核心 API 设计  

> 为了让新手同学更易理解，下面只列出 **业务层**（对外）和 **内部服务**（RPC）两类关键接口。

### 1. 对外推荐 API（HTTP/JSON）  

| 方法 | 路径 | 请求 | 响应 | 说明 |
|------|------|------|------|------|
| `GET` | `/recommend` | `{ "user_id": "12345", "scene": "feed", "page_size": 20, "timestamp": 1696000000 }` | `{ "request_id":"abcd", "items":[ {"item_id":"i123","score":0.93}, … ], "latency_ms": 42 }` | 前端调用，**单请求**返回 **N 条** 排序后结果。 |
| `POST` | `/event` | `{ "event_id":"e567", "user_id":"12345", "item_id":"i123", "event_type":"click", "timestamp":... }` | `{ "status":"ok" }` | 行为上报（可批量），SDK 使用。 |

- **幂等**：`event_id` 为全局唯一，服务端去重。  
- **超时**：推荐接口在 **100 ms** 内返回，否则返回 **fallback**（如热门列表）。

### 2. 内部 RPC（gRPC）  

#### 2.1 实时特征服务  

```proto
service FeatureService {
  // 根据 user_id 查询实时特征
  rpc GetUserFeature (UserFeatureReq) returns (UserFeatureResp);
  // 根据 item_id 查询实时特征
  rpc GetItemFeature (ItemFeatureReq) returns (ItemFeatureResp);
}
message UserFeatureReq { string user_id = 1; }
message UserFeatureResp { bytes feature_json = 1; }
```

- **为什么用 gRPC**：二进制协议、低延迟、自动代码生成，适合 **内部高频调用**（每次召回/排序都要查询）。

#### 2.2 召回服务  

```proto
service RecallService {
  rpc Recall (RecallRequest) returns (RecallResponse);
}
message RecallRequest {
  string user_id = 1;
  repeated string candidate_item_ids = 2; // 可选：预过滤
  int32 top_k = 3; // 例如 10000
}
message RecallResponse {
  repeated string item_id = 1;
}
```

- **返回 10k‑100k** ID，后面排序服务再拉取特征。

#### 2.3 排序服务  

```proto
service RankingService {
  rpc Rank (RankRequest) returns (RankResponse);
}
message RankRequest {
  string user_id = 1;
  repeated string candidate_item_id = 2;
  int32 top_n = 3; // 20
}
message RankResponse {
  repeated ScoredItem items = 1;
}
message ScoredItem {
  string item_id = 1;
  float score = 2;
}
```

- **模型热更新**：服务内部读取 `model_meta`，支持 **无缝切换**。

## 第五步：详细组件设计  

下面从 **行为采集 → 实时流处理 → 召回 → 排序 → 反馈回流** 逐层展开关键实现细节。

### 5.1 行为采集  

1. **SDK**（iOS/Android/Web）在用户交互后立即构造 `event`，使用 **异步 HTTP** 或 **gRPC** 调用 `/event`。  
2. **幂等**：每条事件生成 **UUID**（client 端）并写入 **Kafka Producer**，开启 **Exactly‑once**（transactional）模式。  
3. **Kafka 参数**：  
   - `topic = behavior_raw`，`partition = hash(user_id)`（保证同一用户的日志在同一分区，便于后续会话聚合）。  
   - `replication_factor = 3`（保证磁盘故障不丢失）。  
   - `retention = 3 days`（供调试）。  

> **不这样做的后果**：如果直接写入 MySQL，写入吞吐会因为锁竞争而瞬间崩溃；如果不做幂等，网络抖动会导致 **重复计数**，进而影响特征统计。

### 5.2 实时流处理（Flink）

| 步骤 | 业务逻辑 | Flink 算子 | 输出 |
|------|----------|------------|------|
| **脱敏 & 校验** | 检查必填字段、过滤非法事件 | `filter` + `map` | 合法流 |
| **会话窗口聚合** | 统计每用户最近 7 天活跃度、点击次数、曝光计数 | `KeyBy(user_id) → EventTimeSessionWindow(7d)` | `UserFeatureUpdate` |
| **特征写入** | 将聚合结果写入 Redis（热点）和 TiKV（全量） | `process` → `RedisSink` + `TiKVSink` | 实时特征库 |
| **召回触发** | 对每个 **曝光** 事件，立即触发召回（减少冷启动） | `side-output` → `RecallSink` | 候选列表 |
| **状态后端** | 使用 **RocksDB** 持久化状态，checkpoint 每 5 s | `stateBackend = RocksDBStateBackend` | 容错恢复 |

- **Exactly‑once**：Flink 与 Kafka 通过 **两阶段提交**（Two‑Phase Commit）保证。  
- **容错**：Job 采用 **Active‑Standby** 模式，若主任务挂掉，备份自动接管。

### 5.3 召回层  

#### 5.3.1 粗排（倒排+热点过滤）  

1. **热点过滤**：从 Redis `hot_item_set` 取前 0.1% 热门 item_id，直接加入 **热点候选**，并在后面做 **降权**。  
2. **倒排检索**：基于用户兴趣标签（如“体育、科技”）在 Elasticsearch 查询匹配的物品 ID，返回 **前 50k**。  
3. **合并**：将热点、倒排、向量检索结果合并，去重后形成 **10k‑100k** 候选集合。  

#### 5.3.2 精排向量相似度（Milvus）

- 将用户兴趣向量（实时更新）作为查询向量，**近邻搜索** Top 20k。  
- 使用 **IVF‑PQ** 或 **HNSW** 索引，查询耗时 **< 5 ms**。  

#### 5.3.3 防热点设计  

- **分片**：向量索引按 **item_id hash** 分片，热点 item 分布到不同机器上。  
- **限流**：对同一热门 item 的召回次数做 **令牌桶** 限制（例如每秒最多 10 k 次），防止单节点被压垮。  

> **不做热点过滤的风险**：热点物品会在所有召回请求里出现，导致 **倒排索引查询热点子集** 成为瓶颈，CPU/IO 利用率飙升。

### 5.4 排序层  

1. **特征聚合**：召回返回的每个 item_id，调用 **FeatureService** 同时拉取用户特征、物品特征（并行化，Batch 100）。  
2. **模型推理**：使用 **TensorFlow Serving**，模型输入为 **[user_feature, item_feature, context]**，输出 **CTR 预测**（或多任务分数）。  
3. **业务规则**：在模型分数上再乘以 **业务系数**（如广告位权重、库存限制），得到最终排序分数。  
4. **Top‑N**：采用 **Heap**（O(k log n)）快速取前 20。  

- **并发**：排序服务部署 **水平伸缩**（K8s HPA）并且 **CPU/GPU 自动弹性**。  
- **模型热更新**：模型文件放在对象存储（OSS），Serving 监听元数据表 `model_meta`，当 `status=online` 且 `rollout_ratio` 变化时自动拉取新模型。

### 5.5 实时反馈回流  

| 步骤 | 描述 | 技术 |
|------|------|------|
| **前端曝光/点击** | 前端在渲染后立即上报曝光，点击后上报点击 | `/event` 接口（批量） |
| **流式写回** | Flink 读取这些事件，更新 **用户‑item 交互表**（ClickHouse）和 **特征库**（Redis） | Flink → ClickHouseSink / RedisSink |
| **在线学习** | 将最新的点击/曝光数据流向 **Parameter Server**（如 PS‑Lite）或 **Online Gradient Boosting**，每 5 min 生成 **增量模型** | Flink → Online Trainer → Model Registry |
| **回滚机制** | 新模型上线前做 **灰度**（rollout_ratio），监控关键指标（CTR、错误率）若出现回退阈值则自动切回旧模型 | Prometheus Alert → 自动触发模型降级脚本 |

> **不做实时回流的后果**：模型只能每晚离线训练，导致推荐总是 **滞后**，用户兴趣变化捕捉不到，体验下降。

### 5.6 监控 & AB‑Test  

- **指标收集**：Prometheus 抓取每个服务的 **latency、qps、error_rate、cpu/mem**。  
- **业务指标**：在日志中嵌入 **experiment_id**，通过 ClickHouse 实时统计 **CTR、CVR、GMV** 按实验分组。  
- **告警阈值**：  
  - 推荐接口 95% 延迟 > 90 ms → **Warning**  
  - 错误率 > 0.5% → **Critical**  
- **实验平台**：在入口层（API Gateway）根据 **user_id hash** 将流量划分到不同 **model_version**，支持 **A/B/N** 实验。

## 第六步：扩展性与高可用设计  

### 6.1 水平扩容策略  

| 组件 | 扩容方式 | 关键配置 |
|------|----------|----------|
| Kafka | 增加 **partition**（每 2 k qps 建议 12 k 分区） | `num.partitions`、`replication.factor=3` |
| Flink | **TaskManager** 增加，重新平衡 **slot** 分配 | `parallelism = #TaskManagers * slots_per_TM` |
| Redis / TiKV | **Cluster** 模式，自动分片 | `hash slots` |
| Elasticsearch / Milvus | **Shard** 增加，节点加入 **集群** | `number_of_shards`、`replica` |
| 排序服务 | **K8s HPA** 基于 CPU/Latency 自动伸缩 | `targetCPUUtilizationPercentage` |
| 监控 | Prometheus **水平分片**（Thanos） | `store` 集群 |

- **为什么要水平**：所有核心业务的 **峰值 QPS** 与 **数据量** 都在 TB 级别，单机垂直扩容成本高且有天花板。

### 6.2 高可用（HA）设计  

| 失效场景 | 方案 | 说明 |
|----------|------|------|
| Kafka Broker 宕机 | **副本**（3 副本）+ **Leader 自动迁移** | 客户端自动切换至新 Leader |
| Flink JobManager 挂掉 | **HA 模式**（ZooKeeper） | 自动选举新的 Active JobManager |
| Redis 节点失效 | **Cluster** + **Replica**（至少 1 副本） | 自动故障转移，客户端支持 **ASK** 重定向 |
| Elasticsearch 节点失效 | **副本 Shard**（至少 1） | 查询自动路由 |
| 排序模型服务异常 | **灰度回滚**（rollout_ratio=0）+ **Circuit Breaker** | 立即切回旧模型，避免全链路错误 |
| 网络分区 | **跨 AZ 多可用区部署**，使用 **负载均衡器**（L4）进行健康检查 | 单 AZ 故障不影响整体 QPS |

> **不做 HA**：任意单点故障都会导致 **行为日志丢失** 或 **推荐不可用**，直接违背 99.9% SLA。

### 6.3 数据倾斜处理（追问 1）  

1. **热点物品的写入倾斜**  
   - **写入层**：在写入特征库时，对热点 `item_id` 采用 **分布式计数器**（Redis `INCRBY`）而不是单点 DB。  
   - **召回层**：使用 **热点过滤 + 随机抽样**：把热点物品放进 **独立的热点集合**，在召回时对其进行 **抽样**（如每次只取 5%），其余放在普通倒排中。  

2. **热点物品的计算倾斜**  
   - **向量检索**：在 Milvus 中 **预先分配热点 shard**，每个热点 shard 设定 **更高的并发阈值**。  
   - **排序层**：对同一热点 item 的特征查询使用 **本地缓存**（热点缓存），避免频繁访问后端存储。  

### 6.4 在线模型更新（追问 2）  

| 步骤 | 实现方式 | 关键点 |
|------|----------|--------|
| **增量特征** | Flink 实时聚合后写入 **Redis**，模型读取最新特征 | 确保 **Exactly‑once** |
| **增量学习** | 使用 **Online Gradient Boosting**（如 Vowpal Wabbit）或 **Streaming DeepCTR**，输入增量特征，产生 **增量模型参数** | 每 5 min 生成一次模型快照 |
| **模型发布** | 将新模型写入 OSS，更新 `model_meta` 表的 `status=online`、`rollout_ratio=10%` | 支持 **灰度**、**AB‑Test** |
| **回滚** | 监控业务指标（CTR）若下降超过阈值（如 5%），自动将 `rollout_ratio` 设为 0 并切回旧模型 | 需要 **自动化脚本** 与 **告警** 联动 |

- **低延迟保证**：模型更新不直接影响正在进行的请求，因为 **TensorFlow Serving** 会在后台加载新模型，旧请求继续使用旧模型，切换对用户透明。

### 6.5 容量规划与纵向/横向扩展（追问 3）  

| 资源 | 当前需求 | 2 年后（2×增长） | 扩容方案 |
|------|----------|------------------|----------|
| **日志存储** | 30 TB/天 | 60 TB/天 | **水平扩容**：Kafka 增加 Broker + 分区；HDFS/OSS 增加节点/容量。 |
| **特征库** | 5 TB | 10 TB | **TiKV 扩容**：增加 RegionServer；Redis 通过 **Cluster** 增加节点。 |
| **召回索引** | 10 TB 向量 + 5 TB 倒排 | 20 TB 向量 + 10 TB 倒排 | **分片**：Milvus 按 2 倍节点数重新分片；Elasticsearch 增加 Shard 数量。 |
| **计算** | Flink 10 TaskManagers | 20 TaskManagers | **弹性伸缩**：K8s 自动调度或手动添加节点。 |
| **网络** | 10 Gbps | 20 Gbps | **升级到 40 Gbps**，或采用 **多链路聚合（LACP）**。 |

- **横向扩容** 是首选，因 **数据量呈指数级增长**，单机性能提升有限。  
- **垂直扩容**（提升单节点 CPU、内存）可以在短期（几周）内快速缓解峰值，但长期必须 **加节点**，否则成本呈指数上升。  
- **对已有系统的影响**：在 Kafka、Flink、Elasticsearch 增加节点时，需要 **重新平衡分区/分片**，这会产生短暂的 **IO 峰值**，可以在业务低峰期进行。

## 第七步：常见面试追问与回答  

### Q1. 数据倾斜怎么处理？（已在 6.3 简要说明）  

**完整回答**  
- **识别倾斜**：通过监控 `item_id` 的 QPS，使用 **Top‑K** 查询，若某些 `item_id` 的 QPS 超过整体平均的 10 倍，则视为热点。  
- **写入层**：对热点使用 **分布式计数器**（Redis `INCRBY`）或 **分区键**（在 Kafka 中使用 `hash(item_id) % N`），确保写入均匀。  
- **召回层**：  
  1. **热点过滤**：把热点物品放入 **独立的热点集合**，在召回时对其进行 **抽样**（如 5%），并在排序层进行 **降权**。  
  2. **分片**：向量索引按照 `item_id % shard_num` 分片，热点自动分布到多个机器。  
  3. **限流**：对热点的召回请求做 **令牌桶** 限制，防止单节点被压垮。  
- **排序层**：热点物品特征缓存到 **本地 LRU**，降低对后端的读请求。  

> **不做这些** 会导致单节点 CPU/IO 飙升、GC 频繁、甚至节点崩溃，整体推荐服务不可用。

### Q2. 如何实现模型的在线更新并保证回滚？（已在 6.4）  

**要点**  
- **增量特征**：实时流处理将最新点击/曝光写入特征库。  
- **在线学习框架**：如 Vowpal Wabbit、Streaming XGBoost，接收增量特征流，输出 **增量模型**（每 5 min 打包一次）。  
- **模型发布**：模型快照存入对象存储，`model_meta` 表记录 `version、status、rollout_ratio`。  
- **灰度发布**：API 网关根据 `rollout_ratio` 将流量分配到新模型；监控 CTR、错误率。  
- **自动回滚**：若业务指标跌破阈值，自动把 `rollout_ratio` 设为 0 并切回旧模型；回滚脚本使用 **K8s Job** 完成。  

### Q3. 业务增长 2 倍后，如何规划容量？  

**回答要点**  
- **日志**：Kafka 分区数翻倍，Broker 增加 2 倍，HDFS/OSS 扩容。  
- **特征库**：TiKV/Redis 增加节点，保持 **5% 以上的磁盘剩余空间**。  
- **召回**：向量库和倒排索引按节点数等比例扩容，重新平衡分片。  
- **计算**：Flink TaskManager 数量加倍，确保 **并行度 >= QPS / (每个 TaskManager 处理能力)**。  
- **网络**：升级到 40 Gbps 或使用 **多链路聚合**，避免网络瓶颈。  
- **扩容方式**：优先 **水平扩容**（加节点），因为 **数据量是线性增长**，单机性能提升有限。  
- **影响**：扩容会触发 **分区/分片迁移**，需在业务低峰期执行，并做好 **监控告警**。

### Q4. 为什么不直接在推荐 API 里做所有特征计算？  

- **原因**：特征计算需要 **跨事件的聚合**（如最近 7 天点击次数），在每次请求里重新计算会导致 **巨大的 CPU、IO 开销**，且 **延迟难以控制**。  
- **采用实时流处理**：把聚合工作提前做在 Flink 中，得到 **预计算特征**，请求时只做 **读** 操作，才能满足 **≤ 100 ms** 的 SLA。  

### Q5. 如果某个组件（如 Redis）宕机，整体系统还能正常工作吗？  

- **设计**：Redis 采用 **Cluster + Replica**，客户端支持 **ASK/TRY_AGAIN** 重试。  
- **降级**：若热点缓存全部失效，召回层会直接从 TiKV 拉取特征，延迟可能上升到 **20 ms**，仍在 100 ms SLA 内。  
- **监控**：Redis 宕机会触发告警，自动扩容或故障转移。

## 心得与反思  

### 1. 本题最难的设计决策  

| 决策 | 思考过程 | 最终结论 |
|------|----------|----------|
| **是否在召回层使用双索引（倒排 + 向量）** | - 只用倒排：对热点物品仍然会产生大量候选，难以满足 10 k 条的精度。<br>- 只用向量：对冷门物品召回不足，且向量检索对硬件要求高。 | **双索引**：倒排负责 **过滤**（品类、标签），向量负责 **细粒度相似度**。两者组合可以在 **毫秒级** 完成 10k‑100k 候选，兼顾覆盖率与精度。 |
| **模型在线更新的频率与安全性** | - 高频更新（每分钟）能快速捕捉兴趣变化，但风险是 **模型不稳定**，可能出现 CTR 暴跌。<br>- 低频更新（每日）安全，但延迟大。 | 采用 **5 min 增量模型 + 灰度发布**：兼顾 **快速学习** 与 **安全回滚**。在实验环境验证后再提升频率。 |

### 2. 新手最容易犯的错误  

1. **把所有业务都塞进单个服务**  
   - 结果：代码耦合、部署困难、单点故障。  
   - 正确做法：**功能拆分**（行为采集、流处理、召回、排序、监控），每块独立部署、独立扩容。  

2. **忽视 **Exactly‑once** 语义导致数据丢失或重复**  
   - 行为日志是后续所有特征的根基，一旦丢失或重复，模型训练会出现偏差。  
   - 正确做法：在 **Kafka + Flink** 中开启 **事务**、使用 **唯一事件 ID** 幂等写入。  

### 3. 学习建议和可延伸方向  

| 方向 | 推荐学习路径 |
|------|--------------|
| **流式计算** | 1️⃣ 学习 Apache Flink 基础（DataStream、EventTime、状态管理）<br>2️⃣ 实战：搭建一个“实时 UV 统计”项目 |
| **向量检索** | 了解 Milvus / Faiss 的原理，动手在本地跑一次 ANN（Approximate Nearest Neighbor）实验 |
| **分布式缓存** | 深入 Redis Cluster、TiKV（或 RocksDB）的一致性与故障恢复机制 |
| **模型服务化** | 学习 TensorFlow Serving / TorchServe，练习模型热更新、A/B‑Test |
| **高可用设计** | 阅读《Designing Data‑Intensive Applications》，掌握 CAP、事务、幂等、容灾策略 |
| **监控/Observability** | Prometheus + Grafana 实战，配合 OpenTelemetry 进行全链路追踪 |

> **实战技巧**：在学习每一块技术时，都**把它放进完整的推荐链路**里去思考：数据从哪里来、怎么流转、会不会丢、延迟在哪里、如何扩容。这样才能把碎片化的知识串联成系统化的设计能力。
