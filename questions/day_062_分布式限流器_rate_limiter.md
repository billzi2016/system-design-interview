# 分布式限流器（Rate Limiter）系统设计面试题

## 1️⃣ 题目背景
分布式限流器用于在高并发环境下对 API、业务操作或资源访问进行流量控制，防止单个用户或整体流量超过系统容忍阈值，从而保障服务的稳定性与公平性。

## 2️⃣ 面试场景设定
> **面试官**：  
> “我们现在需要在公司内部搭建一个通用的分布式限流平台，所有业务线的微服务都要通过它来做流量控制。请你从零开始设计这个系统，要求能够支撑我们现有的业务规模，并且具备良好的扩展性和高可用性。请先阐述你的整体思路，然后逐步展开关键模块的设计。”

## 3️⃣ 功能性需求
| 编号 | 需求描述 |
|------|----------|
| **F1** | **多维度限流**：支持按全局、用户（UID/IP）、接口（URI/Method）以及自定义标签（如租户、业务线）进行限流。 |
| **F2** | **动态配置**：限流阈值可以在运行时通过管理后台或 API 动态下发，无需重启业务服务。 |
| **F3** | **多种限流算法**：提供 **漏桶（Leaky Bucket）**、**令牌桶（Token Bucket）**、**固定窗口计数**、**滑动窗口计数** 四种实现，业务方可自行选择。 |
| **F4** | **统一 SDK / 中间件**：提供多语言（Java、Go、Python）客户端库，业务只需在入口处调用 `tryAcquire(key)` 即可得到是否放行的结果。 |
| **F5** | **监控与告警**：实时统计每个维度的请求量、被限流次数、突发流量趋势，支持 Prometheus 导出以及阈值告警。 |
| **F6** | **灰度/白名单**：能够对特定用户或 IP 设置免限流/更高阈值，实现灰度发布或 VIP 客户特权。 |

## 4️⃣ 非功能性需求
| 编号 | 指标 | 估算值 | 说明 |
|------|------|--------|------|
| **N1** | **DAU（活跃业务数）** | 10,000+ 微服务实例 | 系统需为数万实例提供统一限流服务。 |
| **N2** | **QPS（请求峰值）** | 200,000 QPS（整体）<br>≈ 20 QPS/实例 | 高峰期整体流量约 20 万次/秒，需要水平扩展。 |
| **N3** | **单次限流判断延迟** | ≤ 2 ms（99.9% 请求） | 限流判断必须在业务调用链的毫秒级完成，避免增加显著延迟。 |
| **N4** | **可用性** | 99.99%（月度 SLO） | 包括单点故障自动容错、数据同步一致性等。 |
| **N5** | **持久化/状态存储量** | ≤ 5 GB（全局限流计数+配置） | 计数数据采用轻量级 KV 存储，保留 1‑2 小时的窗口数据。 |

## 5️⃣ 系统边界
- **本题范围内**  
  - 限流核心算法实现、分布式一致性方案、动态配置分发、监控指标收集与导出。  
  - 高可用架构（多副本、故障转移）、水平扩容与负载均衡。  
  - 对外提供统一 SDK / 中间件以及限流 API。  

- **不在本题范围**  
  - 业务侧的具体限流策略制定（如具体阈值如何计算）。  
  - 完整的 IAM / 权限系统（只需要白名单/灰度功能的基本实现）。  
  - 跨地域（多数据中心）同步的细粒度时钟一致性（可简化为同一可用区内部的同步）。  
  - 完整的计费/配额系统（仅提供限流功能）。  

## 6️⃣ 提示与追问
1. **数据一致性**：在分布式环境下，如何保证不同节点对同一限流键的计数结果保持一致？（如使用 Redis 的 Lua 脚本、基于 Paxos/Raft 的一致性 KV 等）
2. **热点键处理**：如果某个 API 被大量请求，导致对应计数键成为热点，系统该如何削峰或做分片？（如采用二级计数、局部缓存、热点键分散等方案）
3. **扩容平滑**：当 QPS 突然提升，需要新增限流节点，如何做到请求不丢失、计数不跳变？（如使用一致性哈希、动态分片迁移、流量预热等）

# 题解

## 解题思路总览
> **目标**：从“只要能判断一次请求是否被放行” 的最小可行系统（MVP），一步步演进到满足 **10k+ 微服务实例、200k QPS、99.99% 可用性、≤2 ms 延迟** 的完整分布式限流平台。  
> **思路**：  
> 1. **先把需求拆解成数据模型、业务流程、性能指标**；  
> 2. **画出最简版架构**（单机 Redis + SDK），验证算法可行性；  
> 3. **逐层添加可靠性/扩展性**（多副本、分片、缓存、热键处理、配置中心、监控告警）；  
> 4. **每次改动都要解释“为什么要这么做”，以及“不这么做会有什么痛点”。  

下面按章节展开，力求 **由浅入深、一步步可落地**，帮助没有系统设计经验的同学建立完整的思考框架。

## 第一步：理解需求与规模估算

| 需求编号 | 关键点 | 对系统的直接影响 |
|----------|--------|-------------------|
| **F1**   | 多维度限流（全局、UID/IP、URI/Method、标签） | 需要 **键（key）生成规则** 能够灵活组合维度；键的数量可能大，必须考虑 **热点** 与 **存储上限**。 |
| **F2**   | 动态配置 | 配置必须 **实时下发**、**持久化**，且对业务 **零感知**。 |
| **F3**   | 四种限流算法 | 业务侧可以自行选择，需要 **统一抽象层**；不同算法对 **状态存储** 的要求不同（如令牌桶需要计时器）。 |
| **F4**   | 多语言 SDK | SDK 必须 **轻量、低延迟**，内部实现可选 **本地缓存 + 远程调用** 两种模式。 |
| **F5**   | 监控/告警 | 每个维度、每种算法都要 **实时统计**，导出 **Prometheus** 指标。 |
| **F6**   | 灰度/白名单 | 需要 **优先匹配** 白名单/高阈值规则，且能 **热更新**。 |

### 非功能性需求拆解

| 编号 | 需求 | 计算方式 | 隐含的系统设计挑战 |
|------|------|----------|--------------------|
| **N1** | 10k+ 微服务实例 | 每实例在启动时会加载限流 SDK，SDK 需要 **发现** 限流后端节点并保持长连接。 | **服务发现**、**连接数上限**、**客户端负载均衡**。 |
| **N2** | 200k QPS 峰值 | ≈ 20 QPS/实例，实际峰值可能集中在少数热点接口。 | **热点键**、**写放大**、**背压**。 |
| **N3** | ≤2 ms 判定延迟 | 限流判断路径应尽量在 **内存/本地缓存** 完成，网络往返时间 < 1 ms。 | **本地缓存失效**、**网络抖动**、**热点路由**。 |
| **N4** | 99.99% 可用性 | 单月不可用 ≤ 43 分钟。 | **单点故障**、**故障转移时延**、**数据同步一致性**。 |
| **N5** | ≤5 GB 存储 | 只保留 1‑2 小时窗口计数。 | **KV 选型**、**压缩/过期策略**、**热键分片**。 |

> **小结**：核心难点在于 **高并发写入 + 低延迟读取**，以及 **配置/状态的强一致/弱一致取舍**。接下来先搭建最小可用系统，验证思路。

## 第二步：高层架构设计

### 1️⃣ 最小可行系统（MVP）

```
+-------------------+        +-------------------+
|   微服务实例 A   | -----> |  限流服务（Redis） |
+-------------------+        +-------------------+
       SDK (Java)                单机
```

- **SDK**：调用 `tryAcquire(key)` → **Lua 脚本** 直接在 Redis 里完成计数/令牌检查 → 返回 true/false。  
- **Redis**：选用 **单机 Redis**（或单节点集群），因为它提供 **原子 Lua 脚本**，天然满足 **计数原子性**。  
- **优点**：实现最快、最直接，能帮助我们验证 **算法实现**、**键设计**、**监控**。  
- **缺点**：单点故障、容量受限、热点键会压垮单机。

### 2️⃣ 逐步演进的分布式架构

```
+-------------------+      +-------------------+      +-------------------+
|   微服务实例 N   | ---> |   限流网关 (LB)   | ---> |  分片限流节点集群 |
+-------------------+      +-------------------+      +-------------------+
          |                     |                     |
          |   SDK (本地缓存)    |   RPC / HTTP        |   Redis / RocksDB
          |                     |                     |
          +---------------------+---------------------+
```

#### 关键组件

| 组件 | 作用 | 关键技术点 |
|------|------|------------|
| **限流网关（Load Balancer）** | 为 SDK 提供统一入口，做 **流量分发**、**健康检查**、**请求路由**（基于一致性哈希） | Nginx/Envoy + Consul/Eureka |
| **分片限流节点** | 每个节点负责一部分 **键空间**（使用一致性哈希或分段路由），内部使用 **Redis Cluster** / **RocksDB + Raft** 保存计数 | 高并发写、持久化、水平扩容 |
| **配置中心** | 动态下发阈值、白名单、算法选择 | Apollo / Nacos / Etcd |
| **监控/告警模块** | 收集限流指标 → Prometheus → Alertmanager | 通过 Sidecar / Exporter 实现 |
| **SDK** | 业务侧调用入口，支持 **本地缓存+远程降级** | 支持 `tryAcquire(key)`、`updateConfig()`、热更新、灰度策略 |

> **为什么要这么拆？**  
> - **网关 + 分片节点**：把 **写流量**（计数）分散到多个节点，避免单点热点。  
> - **一致性哈希**：保证 **键迁移** 时只影响少数节点，实现 **平滑扩容**。  
> - **配置中心**：满足 **F2** 的动态下发，且能统一管理 **白名单/灰度**。  
> - **监控**：满足 **F5**，并帮助我们在出现热点时快速定位。

## 第三步：数据库设计

### 3.1 选型原则
| 需求 | 关键指标 | 推荐实现 |
|------|----------|----------|
| **原子计数**（每次请求计数+判断） | 低延迟、强原子性 | **Redis**（Lua 脚本） 或 **RocksDB + Raft**（自实现原子操作） |
| **持久化**（1‑2 小时窗口） | 数据量 ≤ 5 GB、写放大低 | **Redis 持久化（AOF+RDB）** 或 **RocksDB**（日志压缩） |
| **水平扩展** | 200k QPS | **Redis Cluster**（分片）或 **自建分布式 KV**（基于 Raft） |
| **热点键分散** | 防止单节点写放大 | **二级计数** + **局部缓存**（见后） |

> **常见方案对比**（简化版）  

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 单机 Redis + Lua | 实现最简单、原子性强、延迟低 | 单点故障、容量受限、热点压垮 | MVP、测试 |
| Redis Cluster | 自动分片、水平扩容、故障转移 | 跨节点事务受限，需要键设计保证同一键在同一槽 | 生产、热点分散 |
| RocksDB + Raft (TiKV) | 强一致性、持久化可靠、可自定义压缩 | 实现复杂、写延迟略高 | 对持久化要求极高的场景 |
| MySQL / PostgreSQL | 熟悉、事务完整 | 写吞吐不足、延迟高 | 仅作配置存储，不做计数 |

### 3.2 数据模型

#### 3.2.1 限流键（Key）设计
```
<algorithm>::<dim1>::<dim2>::...::<dimN>
```
- **algorithm**：`leaky`、`token`、`fixed`、`slide`  
- **维度**（可选组合）  
  - `global`（全局）  
  - `uid:{userId}` / `ip:{ip}`  
  - `uri:{method}:{path}`  
  - `tag:{tenant}:{bizLine}`  

**示例**  
- `token::uid:12345` → 对用户 12345 使用令牌桶  
- `slide::uri:POST:/order::tag:tenantA:sales` → 对租户 A 销售线的 POST /order 接口使用滑动窗口  

> **为什么采用字符串拼接？**  
> - 直观、易于调试、无需额外映射表。  
> - 与 **Redis** 的键空间天然匹配，支持 **通配符查询**（如 `scan slide::*`）用于监控。  
> - **不采用哈希** 是因为我们需要在监控/灰度时**直接看到维度信息**。

#### 3.2.2 配置表（持久化在配置中心）

| 字段 | 类型 | 说明 |
|------|------|------|
| `rule_id` | STRING (PK) | 唯一标识 |
| `algorithm` | ENUM | `leaky`/`token`/`fixed`/`slide` |
| `key_pattern` | STRING | 支持占位符，例如 `uid:{uid}`、`uri:{method}:{path}` |
| `limit` | BIGINT | QPS、TPS、请求数上限等 |
| `capacity` | BIGINT (令牌桶) | 桶容量 |
| `refill_rate` | DOUBLE (令牌桶) | 每秒填充令牌数 |
| `window_size` | INT (秒) (滑动/固定) | 窗口长度 |
| `white_list` | SET<String> | 免限流标识（IP、UID） |
| `gray_list` | MAP<String, BIGINT> | 灰度用户自定义阈值 |
| `status` | ENUM | `ENABLED`/`DISABLED` |
| `gmt_modified` | TIMESTAMP | 最后修改时间 |

> **为什么把阈值等配置放在独立中心？**  
> - 动态下发（F2）需要 **统一入口**，而计数数据只保存在限流节点。  
> - 配置变更不需要 **同步计数**，只影响后续请求的判断逻辑，降低一致性要求。

#### 3.2.3 计数表（存于 Redis）

| Key（由 SDK 生成） | Value | TTL |
|--------------------|-------|-----|
| `slide::uri:GET:/profile::uid:12345` | `{"count": 128, "window_start": 1695648000}` | `window_size`（如 60s） |
| `token::uid:12345` | `{"tokens": 5, "last_refill": 1695648035}` | `capacity`（持久） |

> **TTL**：保证 **N5** 中“保留 1‑2 小时窗口”。对 **固定窗口** 直接使用 `expire`，对 **滑动窗口** 保存滑动计数结构（如环形计数器）。

## 第四步：核心 API 设计

### 4.1 对外 SDK 接口（语言无关）

```java
public interface RateLimiter {
    /**
     * 尝试获取一次许可
     * @param request   包含业务维度信息（uid、ip、uri、method、tags）
     * @return true  放行； false 被限流
     */
    boolean tryAcquire(RateLimitRequest request);

    /**
     * 主动刷新本地缓存（如配置变更后调用）
     */
    void reloadConfig();
}
```

#### RateLimitRequest 示例（Java）

```java
public class RateLimitRequest {
    private String uid;            // 可选
    private String ip;             // 可选
    private String method;         // 必填
    private String uri;            // 必填
    private Map<String,String> tags; // 如 tenant=tenantA, bizLine=sales
    // 其它扩展字段…
}
```

> **为什么只提供 `tryAcquire`？**  
> - 大多数业务只关心“是否放行”。  
> - 统一入口便于 **统一限流策略**（如白名单优先），且 **易于监控**（每一次调用都可记录）。

### 4.2 限流服务内部 RPC 接口（RESTful 示例）

| 方法 | 路径 | 参数 | 返回 |
|------|------|------|------|
| `POST` | `/acquire` | `{key, algorithm, limit, ...}` | `{allowed:true/false, remaining:int, retryAfterMs:int}` |
| `GET` | `/config/{ruleId}` | — | 配置 JSON |
| `POST` | `/config/update` | 配置列表 | `200 OK` |
| `GET` | `/metrics` | — | Prometheus 文本格式 |

> **实现细节**：`/acquire` 内部使用 **Lua 脚本**（Redis）或 **Raft 命令**（RocksDB）保证 **原子判断+计数**。

### 4.3 监控指标（Prometheus Exporter）

| Metric 名称 | 类型 | 说明 |
|-------------|------|------|
| `ratelimiter_requests_total{key, algorithm, result}` | Counter | 请求总量，`result` 为 `allowed`/`rejected` |
| `ratelimiter_current_qps{key}` | Gauge | 最近 1 秒 QPS（滑动窗口实现） |
| `ratelimiter_token_bucket_remaining{key}` | Gauge | 令牌桶剩余令牌数 |
| `ratelimiter_hot_key_seconds{key}` | Gauge | 连续热点键累计时间（用于告警） |

> **为什么要把监控放在限流节点而不是 SDK？**  
> - SDK 只负责调用，无法保证全局视角；节点可以 **统一聚合**，并且 **不会因网络抖动导致监控失真**。

## 第五步：详细组件设计

### 5.1 SDK（本地缓存 + 远程降级）

1. **初始化**：  
   - 启动时向 **配置中心** 拉取全部规则（或按需拉取）。  
   - 与 **限流网关** 建立长连接（HTTP/2 或 gRPC），并获取 **路由表**（一致性哈希环）。  

2. **请求流程**  
   ```
   request → buildKey() → localCache.lookup(key) → 
   if cacheHit && tokenEnough → allow
   else → RPC to assigned node → Lua script → result
   → update localCache (optional)
   ```  

3. **本地缓存策略**  
   - **热点键**（如同一 API 的 UID）在本地保存最近 N 条计数（TTL 1s），用 **读写锁** 保护。  
   - **缓存失效**：TTL 到期或收到配置中心的 **Invalidate** 消息。  

4. **降级方案**  
   - 当网络异常、节点不可用时，**默认放行** 或 **默认拒绝**（可配置），并记录 **fallback_metric** 供运维监控。

> **为什么要加本地缓存？**  
> - 减少网络往返次数，满足 **N3**（≤2 ms）的大部分请求。  
> - 对热点键的 **读请求**（多数是 “计数是否已满”）可以在本地快速判断。

### 5.2 限流网关（Load Balancer）

| 功能 | 方案 | 说明 |
|------|------|------|
| **流量路由** | **一致性哈希**（基于 key hash） | 同一 key 永远落到同一限流节点，避免计数不一致。 |
| **健康检查** | 心跳 + 成功率阈值 | 自动剔除异常节点，通知 SDK 更新路由表。 |
| **流量控制** | 限流网关本身也可以做 **粗粒度**（全局）限流，防止突发流量冲击后端节点。 | 采用 **令牌桶**，阈值为整体 QPS 的 1.2 倍。 |
| **协议** | **gRPC**（双向流）或 **HTTP/2** | 支持长连接，降低握手开销。 |

> **为什么不直接让 SDK 计算哈希？**  
> - 当节点扩容/缩容时，需要统一 **元数据同步**（哈希环）并保证 **所有客户端** 同步更新。网关统一调度更容易实现 **一致性**。

### 5.3 分片限流节点（核心）

#### 5.3.1 计数实现（以 Redis 为例）

```lua
-- token bucket Lua 脚本（伪代码）
local key = KEYS[1]
local now = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local refillRate = tonumber(ARGV[3])
local tokensNeeded = tonumber(ARGV[4])

local data = redis.call('HMGET', key, 'tokens', 'last')
local tokens = tonumber(data[1]) or capacity
local last = tonumber(data[2]) or now

-- 计算应填充的令牌数
local elapsed = now - last
tokens = math.min(capacity, tokens + elapsed * refillRate)

if tokens < tokensNeeded then
    return {0, tokens}
else
    tokens = tokens - tokensNeeded
    redis.call('HMSET', key, 'tokens', tokens, 'last', now)
    redis.call('EXPIRE', key, math.ceil(capacity / refillRate))
    return {1, tokens}
end
```

- **原子性**：Lua 脚本在 Redis 中一次性执行，避免并发冲突。  
- **TTL**：确保旧键自动清理，满足 **N5**。  

#### 5.3.2 热点键二级计数（分散写入）

1. **第一层**：对热点 key（如 `uri:GET:/product`）使用 **散列前缀**，如 `hash(key) % 10` → `slide::uri:GET:/product::shard0`、`shard1` …  
2. **第二层**：在聚合层（定时任务）把 10 个子计数合并为整体计数，用于 **限流判断**（只在聚合时读取）。  

> **优点**：写入分摊到 10 个键，降低单键写入热点。  
> **缺点**：聚合有延迟（毫秒级），但对 **滑动窗口** 已经足够。  

#### 5.3.3 数据持久化 & 失效恢复

- **AOF + RDB**：Redis 持久化每秒一次（AOF）+ 每 5 分钟一次快照（RDB），在节点重启后可快速恢复计数。  
- **节点故障转移**：采用 **Redis Sentinel** 或 **Cluster 主从复制**，故障后自动提升从节点为主，确保 **写可用**。  

### 5.4 配置中心

- **存储**：Etcd / Nacos（强一致性 KV）  
- **下发方式**：  
  - **轮询**：SDK 每 30 秒拉取最新配置（简单）。  
  - **推送**：配置中心向 SDK 推送 **WebSocket / gRPC Stream**（实时）。  
- **灰度/白名单实现**：在配置里维护 **白名单集合**，SDK 在 `tryAcquire` 前先匹配，如果命中直接 `allow`。  

### 5.5 监控与告警

1. **节点内部**：每次计数成功/失败后更新本地 **Prometheus Counter**。  
2. **Exporter**：在每个节点跑一个 **/metrics** HTTP 端口，Prometheus 抓取。  
3. **聚合仪表盘**：Grafana 按维度（全局、UID、URI、标签）展示 QPS、限流率、热点键。  
4. **告警规则**：  
   - `ratelimiter_requests_total{result="rejected"} / ratelimiter_requests_total{result="allowed"} > 0.2` → 触发限流率告警。  
   - `ratelimiter_hot_key_seconds > 30` → 热点键告警，建议分片或调高阈值。

## 第六步：扩展性与高可用设计

### 6.1 水平扩容（平滑迁移）

| 步骤 | 操作 | 影响 |
|------|------|------|
| 1️⃣ **增加节点** | 在 Consul/Etcd 中注册新节点，网关感知后加入哈希环 | 只影响 **新加入的哈希区间** |
| 2️⃣ **迁移键** | 对受影响的键，旧节点在收到请求时返回 **“Redirect”**（带新节点地址） | 客户端短暂重试一次，保证不丢失计数 |
| 3️⃣ **热键预热** | 在新节点启动时，加载 **热点键的历史计数**（从旧节点通过 RPC 拉取） | 防止突发时计数跳变 |
| 4️⃣ **下线旧节点** | 完成迁移后，旧节点退出哈希环，关闭连接 | 完全无缝切换 |

> **为什么使用 “Redirect” 而不是直接在客户端切换？**  
> - 客户端不必感知哈希变化，只要遵循 **一次重试** 即可。  
> - 可以在 **网关层** 实现统一的 **重定向**，避免每个 SDK 实现重复逻辑。

### 6.2 故障容错

| 故障类型 | 处理方式 |
|----------|----------|
| **限流节点宕机** | 主从复制 + Sentinel 自动故障转移；故障期间请求会被 **网关** 转发至 **备份节点**（读写双向），计数会在恢复后 **回放**（通过持久化日志） |
| **网关失效** | 多实例部署在不同机器，使用 **LVS/NGINX** 进行四层负载均衡；客户端开启 **多网关地址**（轮询） |
| **配置中心不可用** | SDK 本地缓存配置（TTL 5 分钟），在失联期间继续使用旧配置；告警触发运维介入 |
| **网络抖动** | SDK 超时后采用 **快速失败**（返回 `allow` 或 `reject`），并记录 `fallback_metric`，供监控分析 |

### 6.3 性能优化

| 场景 | 优化手段 |
|------|----------|
| **热点键写放大** | 二级计数 + **局部缓存**（SDK 本地） |
| **网络往返** | SDK 本地缓存、**gRPC 长连接**、**压缩**（proto） |
| **计数持久化延迟** | **异步 AOF**（每秒一次），对实时判断不产生影响 |
| **监控开销** | **采样**（如 1% 请求才上报）或 **Pushgateway** 缓冲 |

### 6.4 安全与权限

- **限流请求** 使用 **TLS 双向认证**，确保只有内部微服务能调用。  
- **配置中心** 通过 **RBAC** 限制只能内部运维/平台管理员修改阈值。  
- **白名单/灰度** 规则存储在 **加密 KV**，防止泄露业务机密。

## 第七步：常见面试追问与回答

### Q1. 数据一致性如何保证？  
**答**：  
- **计数阶段**：使用 **Redis Lua 脚本** 或 **Raft 事务**，在单节点/单分片内部保证 **原子性**。  
- **跨分片**：同一键始终落在同一分片（**一致性哈希**），因此不存在跨分片竞争。  
- **容灾切换**：采用 **主从复制 + Sentinel**，在故障切换期间从节点会 **同步 AOF 日志**，保证切换后计数不回退。  
- **弱一致**：在极端故障（网络分区）下，允许短暂 **计数不一致**（如两台节点都认为有令牌），但通过 **限流阈值留有安全余量**（如 10%），把风险控制在可接受范围。

### Q2. 热点键会导致单节点写放大，怎么削峰？  
**答**：  
1. **二级计数**：把热点键拆分为 `shard0~shardN`，每个子键写入 1/N 的流量。  
2. **本地缓存**：在 SDK 端缓存最近 1‑2 秒的计数，只有计数接近阈值才请求远程。  
3. **动态分片**：监控发现热点后，**自动开启更多 shard**（如从 10 增至 100），并在后台完成键迁移。  
4. **限流网关粗粒度限流**：在网关层对热点 API 施加 **全局速率限制**（如 5k QPS），防止瞬时冲击。

### Q3. 如何做到无缝扩容？  
**答**：  
- 使用 **一致性哈希** 确定键分配，新增节点只负责 **哈希环中新增的区间**。  
- 对于迁移中的键，旧节点返回 **Redirect**，客户端自动重试一次到新节点。  
- 迁移期间 **计数状态** 通过 **后台同步**（从旧节点拉取最近计数）写入新节点，避免跳变。  
- 扩容过程可以在 **低峰期** 完成，且对业务 **0 影响**（因为旧节点仍然可以处理请求）。

### Q4. 为什么不直接在业务微服务内部实现限流？  
**答**：  
- **统一治理**：所有业务共用一套规则，便于运营、审计、灰度。  
- **资源节约**：每个实例自行计数会导致 **内存/CPU** 大幅浪费，尤其是热点键。  
- **集中监控**：限流平台天然提供 **全局视图**，单实例难以实现。  
- **动态配置**：平台可以实时下发新规则，而不需要 **滚动重启** 微服务。

### Q5. 若业务需要 **分布式事务**（如下单+扣库存）如何与限流配合？  
**答**：  
- 限流只负责 **请求入口的预判**，不参与业务事务。  
- 业务层可在事务前调用 `tryAcquire`，若返回 false，则直接回滚，不进入事务。  
- 若需要 **强一致**（如保证同一用户同一秒只能下单一次），则在事务中使用 **唯一键（业务 ID）** 结合 **Redis SETNX** 实现幂等，限流只起到流量削峰的作用。

## 心得与反思

### 1️⃣ 本题最难的设计决策

| 决策 | 思考过程 |
|------|----------|
| **① 计数存储的强一致 vs. 可用性** | 需要在 **原子计数** 与 **高可用** 之间平衡。最初倾向使用 **强一致的 Raft KV**，但实现复杂且写延迟 > 2 ms。最终选择 **Redis + Lua**（单节点强一致）配合 **主从复制 + Sentinel**，在大多数业务场景下提供足够的强一致，同时在故障时通过 **快速故障转移** 维持可用性。 |
| **② 热点键的处理方案** | 热点键会导致单节点写入瓶颈。考虑了 **分布式锁**、**写入队列**、**二级计数** 三种方案。锁会增加延迟，队列会导致背压。二级计数虽然增加聚合复杂度，但可以在 **毫秒级** 内完成写入，且聚合任务可以异步执行，最终选定二级计数 + 本地缓存。 |

### 2️⃣ 新手最容易犯的错误（至少 2 条）

| 错误 | 说明 | 如何避免 |
|------|------|----------|
| **把所有维度都放在单个键**，导致键空间指数级增长 | 例如 `global:user:uid:ip:uri:method:tag` 组合会产生上千万键，存储和扫描成本极高。 | 采用 **可选维度** 组合（仅使用业务需要的维度），并用 **配置中心** 控制哪些维度参与限流。 |
| **在限流判断后才写入计数**，导致并发竞争 | 先判断再计数会出现“超卖”问题（两个请求同时判断通过，计数后超阈值）。 | 必须 **原子判断+计数**（Lua 脚本、事务），确保“一次请求只能产生一次计数”。 |
| **忽视网络抖动对延迟的影响** | 直接调用远程限流服务，导致大多数请求超时 >2 ms。 | 在 SDK 中加入 **本地缓存** 与 **降级策略**，并使用 **长连接、压缩** 减少往返时间。 |

### 3️⃣ 学习建议和可延伸的方向

1. **深入掌握分布式一致性算法**：了解 **Raft、Paxos、ZAB**，并在实践中使用 **Etcd、Consul** 等实现。  
2. **熟悉高性能 KV（Redis、TiKV）**：学习 **Lua 脚本**、**事务**、**持久化机制**，能够自行实现原子计数。  
3. **系统监控与可观测性**：学习 **Prometheus、Grafana、OpenTelemetry**，把监控嵌入每个模块，形成闭环。  
4. **负载均衡与服务发现**：实践 **Consul/Eureka + Envoy/Nginx**，理解一致性哈希、流量调度策略。  
5. **实验级项目**：可以先用 **单机 Redis + Spring Boot** 实现一个简易限流 SDK，逐步加入 **分片、配置中心、监控**，体会每一步的瓶颈与改进。  

> **一句话总结**：设计分布式限流器的核心是 **“如何在高并发下保持计数的原子性、同时让业务感知不到网络延迟”。** 通过 **本地缓存 + 原子脚本 + 分片路由 + 动态配置**，我们既满足了业务功能，又实现了可扩展、高可用的系统。祝你面试顺利，设计出自己的限流平台！
