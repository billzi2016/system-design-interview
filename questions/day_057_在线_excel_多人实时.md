## 在线 Excel（多人实时）系统设计面试题

### 1. 题目背景
在线 Excel 是一款基于浏览器的电子表格编辑工具，支持多人同时打开同一工作簿并实时协同编辑。用户可以在任意设备上完成数据录入、公式计算、图表展示等工作，且编辑内容会即时同步给其他在线协作者。

### 2. 面试场景设定
> **面试官**：  
> “假设我们要实现一款类似 Google Sheets 的在线 Excel 产品，要求支持多人实时协同编辑、公式自动计算以及跨设备访问。请你从系统层面设计整个架构，重点考虑高并发、低延迟和数据一致性。我们先从核心功能和非功能需求开始讨论，好吗？”

### 3. 功能性需求
| 编号 | 功能描述 |
|------|----------|
| **F1** | **实时协同编辑**：多个用户可以同时在同一工作表的不同单元格进行编辑，编辑操作在 200 ms 内同步到其他在线用户的页面。 |
| **F2** | **公式与计算引擎**：支持常用 Excel 公式（如 SUM、VLOOKUP、IF 等），当相关单元格变化时自动重新计算并实时展示结果。 |
| **F3** | **版本历史与回滚**：系统自动保存编辑历史（每秒一次快照），用户可以查看、比较和回滚到任意历史版本。 |
| **F4** | **权限管理**：工作簿拥有所有者、编辑者、只读者三种角色，支持邀请链接和企业 SSO（OAuth）登录。 |
| **F5** | **跨平台访问**：支持 Web（现代浏览器）和移动端（iOS/Android）客户端，数据在不同终端之间保持一致。 |
| **F6** | **文件导入/导出**：用户可以上传本地 `.xlsx` 文件并在编辑后导出为 `.xlsx`、`.csv` 等格式。 |

### 4. 非功能性需求
| 编号 | 指标 | 估算值 | 备注 |
|------|------|--------|------|
| **N1** | **日活跃用户（DAU）** | 300 万 | 假设主要面向企业和教育用户 |
| **N2** | **峰值并发编辑会话（QPS）** | 50 k QPS（编辑操作） | 峰值出现在工作时间段（UTC+8 9:00‑12:00） |
| **N3** | **编辑操作端到端延迟** | ≤ 200 ms | 包括网络、服务器处理、同步广播 |
| **N4** | **可用性（SLA）** | 99.9% 年可用性 | 包括系统故障恢复时间 ≤ 5 min |
| **N5** | **存储容量** | 100 TB（工作簿原始文件 + 增量快照） | 假设平均文件大小 5 MB，历史快照保留 30 天 |

### 5. 系统边界
- **本题范围内**  
  - 实时协同编辑协议（如 OT/CRDT）设计与实现  
  - 公式计算引擎的分布式调度与缓存  
  - 文档存储、版本控制与快照机制  
  - 权限校验、用户会话管理与负载均衡  
  - 基础监控、日志与故障恢复方案  

- **不在本题范围**  
  - UI/前端细节实现（仅关注 API 与后端交互）  
  - 第三方插件生态（如自定义函数、宏）  
  - 高级图表渲染与可视化（仅提供数据接口）  
  - 跨地域灾备（仅在单个大区内实现高可用）  

### 6. 提示与追问
1. **冲突解决机制**：如果两个用户同时编辑同一个单元格，你会采用哪种算法（OT、CRDT 或其他）？如何保证最终一致性并兼顾低延迟？  
2. **公式计算的扩展性**：面对包含数千行、上万列的大型工作表，如何设计计算引擎的调度与缓存，以避免热点单元格导致的性能瓶颈？  
3. **持久化与快照策略**：在保证版本历史查询性能的前提下，如何平衡快照频率、增量日志存储成本与恢复速度？

## 第一步：理解需求与规模估算  

### 1. 功能需求拆解  

| 编号 | 业务含义 | 对应技术子系统 |
|------|----------|----------------|
| **F1** 实时协同编辑 | 多人同时编辑、低延迟同步 | **协同服务**（OT/CRDT） + **消息推送**（WebSocket） |
| **F2** 公式与计算引擎 | 依赖单元格变化的计算、缓存 | **公式引擎**（分布式任务调度） |
| **F3** 版本历史与回滚 | 定时快照 + 增量日志 | **版本服务** + **对象存储** |
| **F4** 权限管理 | 角色、邀请、SSO | **权限中心** + **认证网关** |
| **F5** 跨平台访问 | Web / iOS / Android 同步 | **统一 API**（REST + WS） |
| **F6** 文件导入/导出 | 解析/生成 `.xlsx`、`.csv` | **文件服务**（转换、存储） |

> **先把每个功能对应到系统的哪块**，后面设计时就能明确职责、避免“功能交叉”导致的耦合。  

### 2. 非功能需求量化  

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 300 万 | 预计同时在线 ≈ 5% → 15 万并发用户 |
| **峰值编辑 QPS** | 50k QPS（编辑操作） | 每个编辑包含 **发送 → 处理 → 广播** 三步 |
| **单操作延迟** | ≤ 200 ms | 包括网络 RTT（≈ 30 ms） + 服务器处理（≈ 150 ms） |
| **存储** | 100 TB（5 MB × 300万 × 30 天） | 需要对象存储 + 增量日志压缩 |
| **可用性** | 99.9% = 8.76 h 年停机 | 故障恢复 ≤ 5 min → 主备热备 + 自动故障转移 |

### 3. 初始容量估算（MVP）  

| 维度 | 估算值 | 备注 |
|------|--------|------|
| **CPU** | 1 核心 ≈ 1500 ops/s 编辑 | 50k QPS / 30 ≈ 1.6k 并发编辑流 |
| **内存** | 8 GB（缓存热点单元格、公式结果） | 采用 LRU + 近实时失效 |
| **网络** | 10 Gbps 入口 | 50k × (payload ≈ 1 KB) ≈ 400 Mbps，留余量 2 倍 |
| **磁盘** | 2 TB SSD（元数据 + 最近 1 天快照） | 30 天冷数据归档至对象存储 |

> **MVP** 目标：单机或少量实例即可支撑 10 k QPS，后续再水平扩容。

## 第二步：高层架构设计  

### 1. 结构概览  

```
+-------------------+      +-------------------+      +-------------------+
|   客户端 (Web/APP) | <--->|   API Gateway     | <--->|   负载均衡(LB)    |
+-------------------+      +-------------------+      +-------------------+
                                  |   |   |
            +---------------------+   +---------------------+
            |                                   |
   +-------------------+               +-------------------+
   |  协同服务 (OT/CRDT) |               |  计算服务 (Formula) |
   +-------------------+               +-------------------+
            |                                   |
   +-------------------+               +-------------------+
   |  版本 & 快照服务  |               |  文件转换服务   |
   +-------------------+               +-------------------+
            |                                   |
   +-------------------+               +-------------------+
   |   持久化层 (DB+OSS) |               |   缓存层 (Redis) |
   +-------------------+               +-------------------+
            |                                   |
   +-------------------+               +-------------------+
   |   监控 & 日志系统   |               |   认证/权限中心   |
   +-------------------+               +-------------------+
```

**说明**  

| 组件 | 负责什么 | 关键技术 |
|------|----------|----------|
| **API Gateway** | 鉴权、流量控制、协议转换（HTTP ↔ WebSocket） | Kong / Envoy + JWT |
| **负载均衡** | 将请求分发到后端实例，提供 Session 粘性（WebSocket） | Nginx/SLB + IP‑hash |
| **协同服务** | 处理编辑操作、冲突解决、广播增量更新 | **CRDT**（基于 LWW‑Cell） + **Kafka**（内部事件流） |
| **计算服务** | 解析公式、调度计算任务、缓存结果 | Spark‑style DAG 调度或 **Flink** 轻量版 + **Redis** |
| **版本 & 快照服务** | 定时生成快照、增量日志、历史查询 | **对象存储 (OSS/S3)** + **MongoDB**（元数据） |
| **文件转换服务** | 解析/生成 `.xlsx`、`.csv` | Apache POI / LibreOffice headless |
| **持久化层** | 核心数据（单元格值、编辑日志） | **CockroachDB**（强一致性） + **Redis**（热点缓存） |
| **监控/日志** | 指标、链路追踪、告警 | Prometheus + Grafana + ELK |
| **认证/权限中心** | SSO、角色校验、邀请链接 | OAuth2.0 + Keycloak |

### 2. 为什么要这样划分？

| 可能的错误划分 | 结果 | 正确划分的好处 |
|----------------|------|----------------|
| 把协同、公式、存储放在同一个单体服务 | 代码臃肿、难水平扩容、单点故障 | **职责分离**，每块可以独立伸缩、独立容错 |
| 只使用 HTTP 轮询同步编辑 | 延迟 > 1 s、带宽浪费 | **WebSocket** 实现 **双向推送**，满足 200 ms 延迟 |
| 不做缓存直接查询 DB | 高 QPS 时 DB 压力爆炸 | **热点缓存**（最近编辑的单元格）显著降低 DB 读写 |
| 只保存增量日志不做快照 | 回滚需要遍历完整日志，时间线性增长 | **定时快照**（每秒一次）将恢复时间固定在 O(快照大小) |

### 3. 数据流（编辑场景）  

1. **客户端** → **Gateway**：WebSocket 发送 `EditCell` 消息（文档 ID、单元格坐标、操作时间戳、值）。  
2. **Gateway** → **协同服务**：鉴权后转发，协同服务基于 **CRDT** 合并冲突，生成 **操作序列号 (OpSeq)**。  
3. **协同服务** → **计算服务**（如果该单元格是公式结果或被公式依赖）：推送 `CellChanged` 事件。  
4. **计算服务** → **缓存/持久化层**：更新公式结果并写入 Redis + CockroachDB。  
5. **协同服务** → **消息队列 (Kafka)**：把已序列化的操作写入日志，用于 **版本快照** 与 **审计**。  
6. **协同服务** → **Gateway** → **所有在线客户端**：广播 **增量更新**（CellID + 新值 + OpSeq）。

## 第三步：数据库设计  

### 1. 选型原则  

| 需求 | 关系型 | 文档型 | KV型 | 说明 |
|------|--------|--------|------|------|
| **强一致性事务**（单元格写入 + 版本号） | ✔︎ | ✘ | ✘ | 必须防止同一单元格出现“写丢失”。 |
| **高并发点写**（热点单元格） | ✘ | ✔︎ | ✔︎ | 文档/KV 天然支持水平分片。 |
| **复杂查询**（历史版本、过滤） | ✔︎ | ✔︎ | ✘ | 需要二级索引。 |
| **大对象存储**（原始 `.xlsx`、快照） | ✘ | ✘ | ✘ | 使用对象存储 (OSS)。 |

**综合**：  
- **元数据 + 编辑日志** → **CockroachDB**（分布式 SQL，兼容 PostgreSQL，提供强一致性 & 自动分片）。  
- **热点单元格缓存** → **Redis Cluster**（TTL、LRU）。  
- **快照/原始文件** → **对象存储 (OSS/S3)**，元信息保存在 CockroachDB。  
- **事件流** → **Kafka**（持久化、顺序消费）。  

### 2. 数据模型（简化）  

#### 2.1 工作簿（Workbook）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 唯一标识 |
| `owner_id` | UUID | 所有者用户 |
| `name` | VARCHAR(255) | 文件名 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 最近编辑时间 |
| `size` | BIGINT | 当前文件大小（字节） |
| `metadata` | JSONB | 其他属性（标签、语言） |

#### 2.2 工作表（Sheet）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 工作表唯一 ID |
| `workbook_id` | UUID | 所属工作簿 |
| `name` | VARCHAR(128) | 工作表名 |
| `row_count` | INT | 行数 |
| `col_count` | INT | 列数 |
| `grid_version` | BIGINT | 整张表的全局版本号（用于快速判断是否需要同步） |

#### 2.3 单元格（Cell）  

> **只存有值的单元格**（空白不落库，节约空间）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `sheet_id` | UUID | 所属工作表 |
| `row` | INT | 行号（0‑based） |
| `col` | INT | 列号（0‑based） |
| `value` | TEXT | 原始字符串（数值/公式） |
| `type` | ENUM('RAW','FORMULA') | 是否为公式 |
| `last_edit_ts` | TIMESTAMP | 最近编辑时间 |
| `op_seq` | BIGINT | 全局操作序列号（保证写入顺序） |
| **PK** | (`sheet_id`, `row`, `col`) | 复合主键，天然分区键（可基于 `sheet_id`） |

#### 2.4 编辑日志（EditLog）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `log_id` | BIGINT (auto‑increment) | 全局唯一序列号 |
| `sheet_id` | UUID | 关联工作表 |
| `op_seq` | BIGINT | 与 Cell 中保持一致 |
| `row` | INT | 单元格行 |
| `col` | INT | 单元格列 |
| `old_value` | TEXT | 旧值（便于回滚） |
| `new_value` | TEXT | 新值 |
| `user_id` | UUID | 发起编辑的用户 |
| `ts` | TIMESTAMP | 编辑时间 |
| **索引** | (`sheet_id`, `op_seq`) | 快速拉取增量日志 |

#### 2.5 快照（Snapshot）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `snapshot_id` | UUID | 唯一 ID |
| `sheet_id` | UUID | 对应工作表 |
| `version` | BIGINT | 对应 `grid_version` |
| `object_key` | VARCHAR(512) | OSS 中的对象路径 |
| `created_at` | TIMESTAMP | 快照生成时间 |
| `ttl` | TIMESTAMP | 过期时间（30 天） |

### 3. 读写路径  

| 场景 | 读取路径 | 写入路径 |
|------|----------|----------|
| **单元格查询**（前端渲染） | 1️⃣ 先读 **Redis**（`GET sheet:ID:row:col`） → 未命中 → 读 **CockroachDB** → 写回 **Redis**（TTL 5 min） | **协同服务** → 生成 **OpSeq** → **事务**写入 **Cell** + **EditLog** → **发布至 Kafka** → **更新 Redis** |
| **公式计算** | **计算服务** 读取 **依赖单元格**（先 Redis，后 DB） | 计算完成后写 **Cell**（type=RAW，value=结果） + **EditLog** |
| **历史快照** | **对象存储**（OSS）下载快照文件 → **解压** → **映射**至内存 | **版本服务** 每秒触发 **全表快照** → **写入 OSS** + **元数据入 DB** |
| **权限校验** | **权限中心**（Redis cache 角色） | **用户管理** → 写 **RBAC** 表（CockroachDB） |

> **为什么采用双写（DB + Kafka）**：DB 保证 **强一致**，Kafka 提供 **持久化顺序日志** 用于 **回放、审计、快照**，两者互为补充。

## 第四步：核心 API 设计  

> **原则**：REST 用于 **CRUD/查询**，WebSocket 用于 **实时双向**。所有 API 必须 **幂等**（重复请求不产生副作用），并返回统一的 **错误码**。  

### 1. 鉴权与会话  

| 方法 | 路径 | 描述 | 返回 |
|------|------|------|------|
| `POST /auth/login` | JSON `{username,password}` | SSO / OAuth 登录，返回 **JWT** | `{access_token, refresh_token}` |
| `POST /auth/refresh` | JSON `{refresh_token}` | 刷新 Access Token | `{access_token}` |
| `GET /auth/me` | Header `Authorization: Bearer xxx` | 获取当前用户信息 | `{user_id, name, roles}` |

### 2. 文档生命周期  

| 方法 | 路径 | 描述 | 参数 | 返回 |
|------|------|------|------|------|
| `POST /workbooks` | JSON `{name, file (base64)}` | 创建新工作簿（可上传 .xlsx） | - | `{workbook_id}` |
| `GET /workbooks/{id}` | - | 获取工作簿元数据 + 权限 | - | `{id, name, owner, …}` |
| `DELETE /workbooks/{id}` | - | 删除工作簿（软删） | - | `204` |
| `GET /workbooks/{id}/snapshots` | - | 列出最近 N 天快照 | `?limit=` | `[{snapshot_id, version, ts}]` |

### 3. 实时协同（WebSocket）  

**连接地址**：`wss://api.example.com/ws?token=ACCESS_TOKEN&workbook_id=XXX`  

| 消息类型 | 客户端 → 服务端 | 字段 | 说明 |
|----------|----------------|------|------|
| `JOIN` | `{type:"JOIN", sheet_id:"...", last_version:123}` | `sheet_id`、`last_version` | 加入编辑会话，服务端返回缺失的增量日志 |
| `EDIT_CELL` | `{type:"EDIT_CELL", row:10, col:5, value:"=SUM(A1:A5)", ts:163...}` | `row/col/value/ts` | 单元格编辑请求 |
| `HEARTBEAT` | `{type:"HEARTBEAT"}` | - | 心跳，保持长连接 |

| 消息类型 | 服务端 → 客户端 | 字段 | 说明 |
|----------|----------------|------|------|
| `ACK_EDIT` | `{type:"ACK_EDIT", op_seq:987654}` | `op_seq` | 编辑已落库并可被其他人看到 |
| `CELL_UPDATE` | `{type:"CELL_UPDATE", row:10, col:5, value:"15", op_seq:987654}` | - | 其他人编辑后推送的增量 |
| `FORMULA_RESULT` | `{type:"FORMULA_RESULT", row:12, col:3, value:"42"}` | - | 计算引擎返回的公式结果 |
| `ERROR` | `{type:"ERROR", code:..., msg:"..."}` | - | 操作失败提示 |

> **为什么使用这种消息结构**：  
> - **type** 明确分发；  
> - **op_seq** 保证 **全局顺序**，客户端可根据 **op_seq** 判断是否丢失消息并请求补偿；  
> - **ACK** 让前端可以 **乐观渲染**（先展示本地编辑），随后收到 ACK 再决定是否回滚。

## 第五步：详细组件设计  

下面把 **每个核心服务** 拆成子模块，解释内部算法、缓存、扩展点。  

### 1. API Gateway & 负载均衡  

| 子模块 | 作用 | 关键技术 |
|--------|------|----------|
| **鉴权插件** | 验证 JWT、提取 `user_id`，写入请求上下文 | Kong JWT 插件 / Envoy RBAC |
| **协议转换** | HTTP ↔ WebSocket 统一入口 | Nginx + `stream` 模块 |
| **流量控制** | QPS 限流、IP 黑名单 | Rate‑limit 插件 |
| **日志采集** | 记录每条请求的响应时间、状态码 | OpenTelemetry + Jaeger |

> **为什么在网关做鉴权**：可以把 **安全** 从业务服务中抽离，统一管理，减少业务代码的复杂度。  

### 2. 协同服务（核心）  

#### 2.1 冲突解决算法  

- **选型**：**CRDT（LWW‑Cell）**（Last‑Write‑Wins） + **Operation‑Based**（OP）  
  - **优点**：天然 **无中心调度**，每个客户端只需要把本地操作发送给服务器，服务器按接收顺序广播；即使出现网络分区，最终收敛仍然保证。  
  - **缺点**：**LWW** 只能保留最新写入，无法保留细粒度的编辑历史（但我们已有 **EditLog** 记录完整历史）。  

- **实现细节**  
  1. **操作结构**（发送给服务器）  
     ```json
     {
       "cell_id": "sheetId:row:col",
       "value": "42",
       "ts": 1695601234567,   // 客户端本地毫秒时间戳
       "client_id": "uuid"
     }
     ```  
  2. **服务器端**  
     - 将 `ts` 与 **服务器时间** 取最大，生成 **全局 OpSeq**（单调递增的 BIGINT）  
     - **CRDT 合并**：如果同一 `cell_id` 已有更大的 `ts`，丢弃本次操作；否则写入 DB。  
  3. **广播**：把 **确定后的 OpSeq** + `cell_id` + `value` 发给 **Kafka** → **协同服务消费者** → **WebSocket** 推送。  

- **为何不选 OT**  
  - OT 需要 **中心变换函数**，对 **高并发** 场景会产生 **序列化瓶颈**。  
  - CRDT 的 **无冲突合并** 更适合 **大规模** 的编辑会话（数千用户同时在线）。  

#### 2.2 关键数据结构  

| 结构 | 说明 | 存储位置 |
|------|------|----------|
| **CellState** | `{value, ts, client_id}` | Redis（热点）+ CockroachDB（持久） |
| **Dependency Graph** | 有向图 `A1 → B2`（B2 依赖 A1） | KV（Redis）+ 持久化至 DB（周期同步） |
| **OpLog Queue** | Kafka topic `sheet-ops-{sheet_id}` | Kafka（持久） |

#### 2.3 可靠性  

- **幂等写入**：利用 **PK (`sheet_id`,`row`,`col`)** + **`op_seq`**，若收到重复 `op_seq` 直接返回 `ACK`。  
- **消息重试**：WebSocket 客户端未收到 `ACK`，在 **5 s** 内自动重发。服务器侧使用 **Kafka Consumer Group**，保证 **至少一次** 投递；业务层通过 **`op_seq`** 去重实现 **Exactly‑once**。  

### 3. 公式计算服务  

#### 3.1 计算模型  

- **依赖图**：每个公式单元格在 **解析阶段**（使用 Excel 语法解析器）生成 **AST**，并抽取 **引用的单元格集合**。这些引用形成 **有向无环图 (DAG)**（Excel 本身不允许循环引用，若出现则返回错误）。  

- **调度**：  
  - **增量计算**：当某个 **源单元格** 改变时，使用 **BFS** 在依赖图中向下遍历，只重新计算受影响的子节点。  
  - **并行执行**：同层级的公式互不依赖，可分配到 **线程池**（或 **Flink Task Slots**）并行计算。  

#### 3.2 缓存策略  

| 层级 | 内容 | 失效机制 |
|------|------|----------|
| **Result Cache (Redis)** | `formula_result:{sheet_id}:{row}:{col}` → 计算结果 | **TTL 5 min** 或 **依赖单元格变更时主动删** |
| **AST Cache** | `formula_ast:{sheet_id}:{row}:{col}` → 解析后的树结构 | **永久**（只在公式变更时更新） |
| **Dependency Cache** | `deps:{sheet_id}:{cell}` → [dependents] | **即时更新**（编辑时同步） |

#### 3.3 扩展性  

- **大表**（上万列、数千行）时，**依赖图**可能非常稀疏。我们采用 **邻接表** 存储在 **KV**，只在 **受影响的子树** 上遍历，时间复杂度 `O(k)`（k 为受影响节点数），而不是全表扫描。  
- **热点公式**（如 `SUM(A:A)`）会被多次计算。我们把 **聚合结果** 预先缓存为 **中间状态**（如列聚合缓存），每次单元格更新仅增量调整。  

#### 3.4 计算服务接口  

- **内部 RPC**（gRPC）用于协同服务的 `CellChanged` 回调：  
  ```proto
  service FormulaEngine {
    rpc NotifyCellChange(CellChange) returns (Ack);
  }
  message CellChange {
    string sheet_id = 1;
    int32 row = 2;
    int32 col = 3;
    string new_value = 4; // 可能是公式或原始值
  }
  ```  

- **外部 REST** 用于调试、即时计算（见上文 API 表）。  

### 4. 版本 & 快照服务  

#### 4.1 快照策略  

| 频率 | 场景 | 成本 | 恢复时间 |
|------|------|------|----------|
| **每秒** | 高频编辑（大多数业务） | 增量日志 + 每秒全表快照（压缩）≈ 200 MB/秒 | 读取最近快照 + 追溯增量日志 → < 1 s |
| **每分钟** | 低活跃文档（只读） | 降低存储 5× | 需回放更多日志（最多 60 s） |

- **实现**：使用 **CockroachDB** 的 **CHANGEFEED** 导出 **增量日志**，每秒将当前 **Cell** 表导出为 **Parquet**（列式压缩）写入 **OSS**，对象名 `snapshots/{sheet_id}/{timestamp}.parquet`。  

#### 4.2 回滚流程  

1. 客户端发起 `restore` 请求 → **版本服务** 创建 **回滚任务**（异步）。  
2. 读取目标快照文件 → **批量写入**至 **CockroachDB**（使用 **COPY FROM**），覆盖当前 `Cell` 表。  
3. 同时生成 **新的 EditLog**（标记为 `ROLLBACK`），广播 **CellUpdate** 给所有在线用户。  

> **为什么采用 “写入覆盖 + 重新广播”**：保持 **操作日志的完整性**，方便审计；并且所有客户端只需要统一接受一次全表更新，避免逐条冲突。  

### 5. 文件导入/导出服务  

| 步骤 | 说明 | 技术 |
|------|------|------|
| **上传** | 前端上传 `.xlsx`（分片） → OSS → 返回 `object_key` | 分片上传（S3 multipart） |
| **解析** | 后端异步读取对象，使用 **Apache POI** 转为内部 **Cell** 表 | Java + POI |
| **保存** | 将解析结果写入 **Cell** 表 + 生成 **快照** | 批量写入 |
| **导出** | 将 **Cell** 表导出为 **Parquet** → POI 生成 `.xlsx`/`.csv` | 同步/异步下载链接 |

> **为什么用异步**：一次完整文件的解析/导出可能耗时数秒，直接阻塞用户会导致超时。通过 **任务队列**（Kafka + Worker）生成 **下载链接**，用户稍后下载。  

### 6. 权限中心  

| 实体 | 权限模型 |
|------|----------|
| **Workbook** | `owner`（最高）<br>`editor`（可编辑）<br>`viewer`（只读） |
| **Invitation** | 链接包含 **token**，访问时校验 **有效期** 与 **角色** |
| **SSO** | OAuth2.0 Authorization Code Flow，返回 **ID Token** 与 **Access Token**，映射到内部 `user_id` |

- **实现**：使用 **Keycloak**（开源）或自研 **OAuth2 Server**，所有业务服务通过 **JWT** 中的 `role` 判断访问权限。  

### 7. 监控、日志、容错  

| 维度 | 指标 | 收集方式 |
|------|------|----------|
| **业务** | QPS、编辑成功率、公式计算时延、快照生成时长 | Prometheus Exporter（每个服务） |
| **系统** | CPU、内存、网络、磁盘 I/O | Node Exporter |
| **链路** | 请求追踪、错误率 | OpenTelemetry + Jaeger |
| **告警** | 延迟 > 150 ms、错误率 > 1% | Alertmanager → Slack/DingTalk |

- **熔断**：在 **Gateway** 对计算服务设定 **5xx** 错误阈值，触发 **熔断** 防止雪崩。  
- **降级**：当公式引擎不可用时返回 **`#CALC_ERROR`**，但仍保证编辑同步。

## 第六步：扩展性与高可用设计  

### 1. 水平扩展  

| 组件 | 扩容方式 | 关键点 |
|------|----------|--------|
| **Gateway** | 增加实例 + DNS 轮询 | 统一 TLS、共享 JWT 公钥 |
| **协同服务** | **无状态**（CRDT 合并） → 多实例，使用 **Kafka** 分区保证同一 sheet 的操作只能落到同一分区 | **分区键 = sheet_id** |
| **计算服务** | 按 **依赖图子任务** 动态调度 → 添加 worker 节点 | 使用 **Flink TaskManager** 或 **Celery** |
| **DB** | **CockroachDB** 自动分片 & 多副本 | 读写分离：读请求走 **read‑replica**，写走 **leader** |
| **Redis** | **Cluster**，键空间均匀分布 | `sheet:{id}:row:{r}:col:{c}` 作为 slot 计算 |
| **Kafka** | 增加 **Broker**，扩大 **partition** 数量 | 维护 **消费者组平衡**，防止热点 partition（热点 sheet）可做 **二级分区**（`sheetId-分段`） |

### 2. 容错与故障恢复  

| 场景 | 处理方式 |
|------|----------|
| **单实例崩溃**（协同/计算） | **自动重启**（K8s Deployment） + **Pod 重建**，状态保存在 **Kafka**（已提交）和 **Redis**（副本） |
| **DB 主节点失效** | **CockroachDB** 自动故障转移，写请求切换到新 leader，客户端不感知 |
| **Redis 故障** | **Cluster** 多副本（3 副本）+ **异步持久化**（RDB+AOF），失效后重新同步 |
| **Kafka Broker 故障** | **副本因子 = 3**，分区 leader 自动切换 |
| **网络分区** | CRDT 仍可本地执行，分区恢复后通过 **op_seq** 自动合并，无冲突（LWW） |

### 3. 性能优化技巧  

1. **热点单元格写入分流**  
   - 将 **极热的计数类单元格**（如 `=SUM(A:A)`）的 **聚合结果** 单独存入 **计数服务**（Redis Atomic Increment），避免每次编辑都遍历全列。  

2. **批量写入**  
   - 客户端在 **短时间内（< 50 ms）** 连续编辑多格时，协同服务先 **缓冲**（batch window 10 ms）再一次性写入 DB，减少事务开销。  

3. **压缩增量日志**  
   - 使用 **Snappy** 对 Kafka 消息进行压缩，降低网络带宽。  

4. **快照分层存储**  
   - 最近 24 h 使用 **热存储**（SSD）保存快照，30 d 以后转入 **对象存储**（冷热分层），降低成本。  

5. **公式依赖缓存失效**  
   - 当单元格 `A1` 被编辑，直接读取其 **dependents list**（Redis），只触发这些公式的重新计算，避免全表遍历。

## 第七步：常见面试追问与回答  

### Q1️⃣ 冲突解决机制（OT vs CRDT）  

| 问题 | 回答要点 |
|------|----------|
| **为什么不直接用 OT？** | OT 需要 **中心化的变换函数**，在高并发下会产生 **序列化瓶颈**；实现复杂，需要对每种操作定义转化函数。 |
| **CRDT 如何保证最终一致性？** | 每个单元格使用 **LWW‑Cell**（基于时间戳的最后写入赢），所有副本只要 **按照时间戳取最大值** 即可；配合 **全局递增 OpSeq**，可以在出现网络分区后自动收敛。 |
| **会不会出现 “写丢失”？** | 业务层仍然记录 **完整 EditLog**（Kafka），即使 LWW 丢弃了旧值，日志里仍保留历史，可用于审计和回滚。 |
| **延迟如何控制在 200 ms 以内？** | - **本地乐观渲染**（先展示本地编辑）<br>- **单向 UDP‑like WebSocket**（无额外握手）<br>- **协同服务无状态**，只做一次 **CRDT 合并 + 写库**，平均处理 < 30 ms，剩余时间为网络 RTT。 |

### Q2️⃣ 公式计算的扩展性  

| 问题 | 回答要点 |
|------|----------|
| **大表（上万列）时如何避免全表重新计算？** | - **依赖图**只在 **受影响子树** 上遍历，复杂度 `O(k)`（k 为受影响节点数）。<br>- **列聚合缓存**（如 `SUM(A:A)`）提前维护，单元格变更只做增量更新。 |
| **热点公式（频繁读取）如何缓存？** | - **Result Cache**（Redis）TTL 5 min，且 **依赖变更时主动失效**。<br>- 对 **相同表达式**（如 `=NOW()`）使用 **全局缓存**，避免重复计算。 |
| **如何保证计算服务不成为瓶颈？** | - **计算任务解耦**：协同服务只负责 **通知**，真正的计算在 **独立 Worker**（水平可扩）。<br>- **批量调度**：一次 `CellChanged` 触发的所有受影响公式一次性提交给调度器，利用 **并行线程池**。 |
| **公式错误或循环引用怎么办？** | 解析阶段检测 **环路**，返回 `#REF!` 或 `#CYCLE!`；计算服务捕获异常后写入错误标记，前端直接展示。 |

### Q3️⃣ 持久化快照与增量日志策略  

| 问题 | 回答要点 |
|------|----------|
| **每秒全表快照成本高吗？** | - **增量日志**（EditLog）本身已记录所有变更。<br>- **每秒快照**采用 **列式压缩 Parquet**，压缩比约 10:1，单秒快照约 20 MB（10 GB/天），在 SSD 上成本可接受。 |
| **如果只保留增量日志会怎样？** | - 回滚到 `t` 需要 **从 0 时刻** 重新回放所有日志，时间随 `t` 线性增长，30 天后恢复时间不可接受（几小时）。 |
| **快照与日志的存储位置为何分离？** | - **快照**大文件适合 **对象存储**（高吞吐、低成本）。<br>- **增量日志**需要 **顺序读写**，适合 **Kafka** + **DB**（可查询）。 |
| **快照失效后如何保证数据安全？** | - **快照**保留 30 天（符合业务需求），**增量日志**保留同样期限，**双份备份**（OSS 多 AZ + DB 多副本）。 |
| **恢复时的步骤？** | 1️⃣ 读取最近的快照<br>2️⃣ 按时间顺序拉取对应区间的增量日志（Kafka Consumer）<br>3️⃣ 逐条写入 DB（批量提交）<br>4️⃣ 完成后发送 **恢复成功**事件给前端。 |

## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 思考过程 | 最终方案 |
|------|----------|----------|
| **冲突解决算法（OT vs CRDT）** | - 需求要 **200 ms** 延迟，且 **并发编辑** 可能高达 50k QPS。<br>- OT 需要全局变换函数、序列化点，容易出现 **瓶颈**。<br>- CRDT 天然 **无中心**，但 LWW 会导致“写丢失”。 | 采用 **CRDT（LWW‑Cell） + 完整 EditLog**，兼顾低延迟和审计回滚。 |
| **公式计算的可扩展性** | - 公式依赖可能形成深度 DAG，单表全量计算不可能。<br>- 必须支持 **增量计算**，并且在热点公式上保持 **毫秒级** 响应。 | 设计 **依赖图 + 增量调度 + 多级缓存**（AST、结果、聚合），把计算解耦为 **Worker**，实现水平扩展。 |

### 2️⃣ 新手最容易犯的错误  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把协同、公式、存储全部放在单体服务** | 代码难维护，水平扩容受限，单点故障。 | **职责分离**：协同、计算、存储各自独立服务，使用轻量协议（gRPC/WebSocket）通信。 |
| **只用数据库存储所有编辑日志** | 高 QPS 时 DB 写入成为瓶颈，且恢复慢。 | **写入 Kafka（增量日志）+ 定时快照**，利用 **分布式日志** 的高吞吐。 |
| **把实时同步依赖轮询** | 延迟大于 1 s，无法满足 200 ms SLA。 | **WebSocket / Server‑Sent Events** 推送增量，保持双向低延迟通道。 |
| **忽视缓存失效** | 公式结果或单元格值缓存不失效导致脏数据。 | **依赖图触发失效**，编辑单元格时主动删除受影响缓存。 |

> **练习方法**：

> 3. 最后加入 **公式计算**（用 Python + Pandas 简易实现），并尝试 **增量计算**。
