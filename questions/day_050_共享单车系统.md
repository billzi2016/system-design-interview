# 第 50 天：设计 共享单车系统

> 生成日期：2026-04-06

---

## 题目背景
共享单车系统是一套面向城市居民和游客的出行服务平台，用户可以通过手机 App（或小程序）实时查找、预定、解锁并完成骑行，还能在骑行结束后进行费用结算和评价。系统需要在高并发、海量地理位置数据和实时计费的场景下保持高可用与低延迟。

## 面试场景设定
> **面试官**：  
> “现在请你从零开始设计一个全国范围的共享单车平台。我们先从整体架构出发，你会怎么划分系统的主要模块？接下来我们会逐步深入到核心业务流程、数据模型以及高并发场景的处理上。”

## 功能性需求
| 编号 | 功能描述 |
|------|----------|
| 1 | **用户注册/登录**：支持手机号、微信/支付宝等第三方登录，完成身份验证与 token 发放。 |
| 2 | **单车实时定位与搜索**：展示用户当前位置附近的可用单车，支持地图渲染、距离排序、过滤（如车型、电池电量）。 |
| 3 | **解锁/骑行计费**：用户扫码或 NFC 解锁单车，系统记录起点时间/位置，骑行结束后自动计算费用（计时/计距或套餐），并完成扣款。 |
| 4 | **骑行结束与归还**：用户在合法停车点或任意位置结束骑行，系统完成锁车、上传结束坐标、生成行程单。 |
| 5 | **订单查询与账单**：用户可以查看历史订单、行程轨迹、费用明细，并支持发票申请、余额充值、优惠券使用等。 |
| 6 | **运维管理（内部功能）**：后台提供单车运维人员定位、故障上报、调度派单、批量导入/下线单车等接口。 |

## 非功能性需求
| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 5,000,000 | 假设覆盖 30 大中城市的活跃用户规模。 |
| **QPS（峰值请求）** | 120,000 请求/秒 | 包括地图搜索、解锁、计费上报等高并发入口（峰值 18:00‑20:00）。 |
| **平均响应时延** | ≤ 150 ms（搜索/解锁）<br>≤ 300 ms（计费上报） | 前端交互感知阈值，要求后端整体 latency ≤ 100 ms。 |
| **可用性** | ≥ 99.95%（全年累计） | 单点故障容忍度要求，单机故障不影响业务。 |
| **存储量** | 约 200 TB（行程轨迹、日志、用户画像）<br>其中实时计费数据 30 TB，历史订单 150 TB | 依据 5 M DAU、平均每日 2 条订单、每单 5 KB 轨迹数据估算。 |

## 系统边界
**本题范围内需要考虑的功能**  
- 用户侧全部前端交互（注册、搜索、解锁、结束、账单查询）。  
- 后端核心业务流程：定位服务、解锁/锁车、计费、订单持久化、运维调度。  
- 高可用架构设计：负载均衡、服务降级、缓存、异步日志/计费。  
- 数据模型与分库分表、冷热数据分离。  

**本题范围外（不需要设计）**  
- 具体的地图渲染 SDK、前端 UI 细节。  
- 支付渠道的完整对接（仅需说明扣款接口抽象）。  
- 线下单车硬件的详细协议（只需假设解锁/上报 API）。  
- 法规合规、隐私加密细节（仅需点出需考虑）。  
- 市场营销、用户增长、推荐系统等业务层面。

## 提示与追问
1. **高并发定位与搜索**  
   - “如果某个热点区域（如地铁站）在早高峰出现 30 万并发定位请求，如何保证搜索 latency 仍在 150 ms 以内？”  

2. **计费一致性与防作弊**  
   - “在网络不稳定或用户恶意篡改 GPS 的情况下，如何保证计费的准确性和防止逃费？”  

3. **运维调度与弹性伸缩**  
   - “单车故障上报后，如何快速将信息同步到用户侧并触发运维人员的派单？在故障激增时系统如何自适应扩容？”

# 题解

## 解题思路总览  

> **目标**：从 **“最小可用系统（MVP）”** 出发，逐层加入 **高可用、伸缩、容灾** 的设计，最终能够满足 **5 M DAU、120 k QPS** 的全国共享单车平台。  
> **方法**：  
> 1. **需求拆解** → 业务、性能、可靠性。  
> 2. **规模预估** → 计算每秒请求数、存储量、热点。  
> 3. **划分子系统**（用户、定位、骑行、计费、订单、运维、监控）。  
> 4. **选择技术栈**（REST + gRPC、缓存、消息队列、分布式数据库等）。  
> 5. **逐层细化**：从单体到微服务、从单库到分库分表、从单机缓存到全局缓存、从单点负载均衡到多活。  
> 6. **验证关键场景**：热点定位、计费一致性、防作弊、运维调度。  

> **思路**：先保证 **功能完整**（用户能注册、搜索、解锁、计费、归还、查询），再保证 **性能指标**（时延、QPS），最后保证 **可用性**（容错、灾备）。每一步都要解释 “为什么要这么做”，以及 “不这么做会有什么风险”。

## 第一步：理解需求与规模估算  

| 维度 | 需求 | 关键指标 | 估算方法 |
|------|------|----------|----------|
| **用户** | 全国 30 城，活跃用户 5 M/日 | DAU = 5 M | 直接给出 |
| **并发** | 高峰 18:00‑20:00，搜索、解锁、计费等请求集中 | QPS ≈ 120 k | 5 M ÷ 24 h ≈ 57 k RPS，峰值约 2 倍 → 120 k |
| **搜索** | 每次搜索返回 20‑30 辆车，单次请求体 ≈ 2 KB | 每秒流量 ≈ 240 MB | 120 k × 2 KB |
| **计费** | 每次骑行结束上报一次，假设每用户平均 2 条订单/天 | 10 M 计费上报/天 ≈ 115 TPS | 10 M ÷ 86400 ≈ 115 |
| **存储** | 行程轨迹 5 KB/单，历史订单 150 TB，实时计费 30 TB | 年增长 ≈ 200 TB | 按需求给出 |
| **热点** | 地铁站、商务中心等集中定位请求，可能 30 万并发定位 | **定位请求** 必须在 150 ms 内返回 | 需要分布式缓存、分片、就近计算 |
| **可用性** | 99.95% → 年宕机 ≤ 4.38 h，单点故障不可接受 | 多活、灰度、自动恢复 | 采用多 AZ、主动健康检查 |

**核心业务流程**（简化版）  

1. **注册/登录** → 生成 JWT/Session Token。  
2. **定位/搜索** → 前端上传经纬度 → 位置服务返回最近可用车辆。  
3. **解锁** → 前端/硬件交互 → 后端生成骑行记录，锁车状态变为 *使用中*。  
4. **骑行计费** → 设备实时或定时上报位置/里程 → 计费中心累计费用。  
5. **结束/归还** → 锁车、上报结束坐标 → 生成订单、扣费。  
6. **订单查询** → 读取历史订单/轨迹。  
7. **运维** → 车辆故障上报 → 调度中心派单。

## 第二步：高层架构设计  

### 1. 系统边界划分  

```
+-------------------+       +-------------------+       +-------------------+
|   移动端/小程序   | <---> |   API Gateway     | <---> |   微服务集群      |
+-------------------+       +-------------------+       +-------------------+
                                      |   |   |
      +-------------------------------+   +-------------------------------+
      |                                                               |
+-------------+   +------------+   +------------+   +-----------------+
| Auth Service|   | Location   |   | Ride       |   | Order Service   |
+-------------+   +------------+   +------------+   +-----------------+
      |                 |                |                |
+-------------+   +------------+   +------------+   +-----------------+
| User DB    |   | GeoCache   |   | Ride DB   |   | Order DB        |
+-------------+   +------------+   +------------+   +-----------------+
      |                 |                |                |
      |                 |                |                |
      |            +----------+      +----------+    +-----------+
      |            |  MQ (Kafka) |    |  MQ (Kafka) |  |  ES / HBase |
      |            +----------+      +----------+    +-----------+
      |                 |                |                |
      |            +----------+      +----------+    +-----------+
      |            |  Billing |      |  Dispatch |    |  Analytics|
      |            +----------+      +----------+    +-----------+
```

### 2. 关键组件解释  

| 组件 | 作用 | 选型建议 | 为什么这样选 |
|------|------|----------|--------------|
| **API Gateway** | 统一入口、鉴权、流量控制、灰度发布 | Kong / Nginx + Lua / Amazon API GW | 统一管理 **QPS**，实现 **限流、熔断、日志**，避免服务直接暴露 |
| **Auth Service** | 登录、Token 发放、第三方 OAuth | Spring Boot + JWT / OIDC | 轻量、无状态，便于水平扩展 |
| **Location Service** | 最近车辆查询、坐标纠偏 | **GeoHash + Redis + Elasticsearch** | GeoHash 支持 **空间分区**，Redis 提供 **毫秒级** 读，ES 支持 **范围查询** 与 **排序** |
| **Ride Service** | 解锁、骑行状态管理、实时计费上报 | gRPC + Go / Rust + Kafka | gRPC 高效、低延迟，Kafka 负责 **异步持久化** 与 **流式计费** |
| **Order Service** | 订单生成、查询、账单 | 微服务 + MySQL（分库分表） + Elasticsearch（全文/过滤） | 订单是 **强一致** 业务，需要事务；ES 提供快速检索 |
| **Dispatch Service**（运维） | 故障上报、派单、调度算法 | Spring Cloud + RabbitMQ + GIS 计算引擎 | 消息队列保证 **可靠投递**，GIS 引擎做 **最近运维人员匹配** |
| **缓存层** | GeoCache、用户 Session、热点车辆 | Redis Cluster（主从） | 读写分离、自动分片，满足 **毫秒级** 读取 |
| **消息队列** | 解耦、削峰、日志/计费流 | Kafka（分区+副本） | 高吞吐、持久化、顺序性（同一车辆同一分区） |
| **计费子系统** | 实时计费、对账、风控 | Flink / Spark Streaming + MySQL | 流处理保证 **实时** 与 **精准** 计费 |
| **存储** | 关系数据、时序轨迹、日志 | MySQL（分库分表）+ HBase/ClickHouse（时序）+ OSS（对象） | 不同数据特性选不同存储，降低单库压力 |
| **监控/告警** | 链路追踪、指标收集 | Prometheus + Grafana + Jaeger | 及时发现 **热点、异常、故障** |
| **CDN** | 静态资源、地图瓦片 | 阿里云 CDN / 腾讯云 CDN | 减轻后端流量，提升用户感知速度 |

### 3. 部署拓扑（多可用区）  

- **每个可用区** 部署完整的微服务集群（API GW、Auth、Location、Ride、Order、Dispatch）以及 **独立的 Redis、MySQL** 实例。  
- **跨 AZ** 使用 **DNS 轮询 + 健康检查**，流量自动切换。  
- **数据同步**：MySQL 主从（异步） + Binlog 同步到 **TiDB/PolarDB** 做读写分离；Redis 主从 + Sentinel；Kafka 跨 AZ MirrorMaker。

## 第三步：数据库设计  

### 1. 数据模型概览  

| 实体 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| **User** | user_id (BIGINT) | phone, third_party_id, token, balance, status | 用户基本信息，**只读**，分库（按 region） |
| **Bike** | bike_id (BIGINT) | imei, model, battery, status, geo_hash, last_update_ts | 车辆状态，热点字段 **geo_hash** 用于定位 |
| **Ride** | ride_id (BIGINT) | user_id, bike_id, start_ts, end_ts, start_loc, end_loc, status | 骑行记录，**强一致**，分库分表（user_id%N） |
| **Order** | order_id (BIGINT) | ride_id, user_id, amount, discount, pay_status, created_ts | 订单，关联 Ride，分库（order_id%M） |
| **LocationLog** | id (BIGINT) | bike_id, ts, lat, lng, speed | 实时轨迹，写入 **HBase / ClickHouse**（时序） |
| **FaultReport** | report_id (BIGINT) | bike_id, user_id, ts, type, desc, status | 故障上报，供运维调度使用 |
| **DispatchTask** | task_id (BIGINT) | report_id, worker_id, assign_ts, status | 运维派单记录 |

### 2. 分库分表策略  

- **User、Bike** 按 **省/城市** 分库（约 30 库），每库 1 TB。  
- **Ride、Order** 按 **user_id** 取模分表（如 64 表），再按 **时间**（年/季度）归档。  
- **LocationLog** 采用 **时序数据库**，按 **天** 或 **小时** 分区，自动过期（如 30 天）后转冷存。  

### 3. 冷热数据分离  

| 数据 | 热度 | 存储 | 读写特性 |
|------|------|------|----------|
| 实时定位/车辆状态 | ★★★★★ | Redis（GeoHash）+ MySQL（主库） | 高频读、低延迟 |
| 当前骑行/计费 | ★★★★ | MySQL（InnoDB）+ Kafka（缓冲） | 事务 + 流式处理 |
| 历史订单/账单 | ★★ | MySQL 分表 + Elasticsearch | 只读、全文检索 |
| 行程轨迹 | ★ | HBase / ClickHouse | 大批量写、查询分析 |
| 日志/审计 | ★ | OSS + ELK | 长期存储、离线分析 |

### 4. 关键索引  

- **Bike**：`INDEX (geo_hash)`（用于范围查询）  
- **Ride**：`INDEX (user_id, start_ts)`、`INDEX (bike_id, status)`  
- **Order**：`INDEX (user_id, created_ts)`、`INDEX (pay_status)`  
- **LocationLog**：`PRIMARY KEY (bike_id, ts)`（时序）

## 第四步：核心 API 设计  

> **原则**：REST 用于用户交互（易调试），gRPC 用于内部高频业务（低延迟）。所有请求均带 **X-Request-ID**、**User-Token**。

### 1. 鉴权相关  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| POST | `/api/v1/auth/register` | `{phone, sms_code, password}` | `{user_id, token}` | 手机号注册，发送短信验证码 |
| POST | `/api/v1/auth/login` | `{phone, password}` 或 `{wechat_code}` | `{user_id, token, expires}` | 支持密码或第三方登录 |
| POST | `/api/v1/auth/refresh` | `{refresh_token}` | `{token, expires}` | JWT 刷新 |

### 2. 定位 & 搜索  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| GET | `/api/v1/bikes/nearby` | `?lat=xx&lng=yy&radius=500&model=all&min_battery=20` | `{bikes:[{bike_id, lat, lng, model, battery}]}` | 使用 **GeoHash + Redis**，返回 **距离排序** |
| POST | `/api/v1/bikes/trace` | `{bike_id, lat, lng, ts}` | `200 OK` | 车辆上报实时位置（后台 Kafka） |

### 3. 解锁 / 开始骑行  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| POST | `/api/v1/rides/start` | `{bike_id, user_token, lat, lng}` | `{ride_id, unlock_token, expire_at}` | **幂等**（同一 bike+user 只创建一次），写入 **Ride** 表，状态 `IN_PROGRESS` |
| POST | `/api/v1/rides/heartbeat` | `{ride_id, lat, lng, ts, speed}` | `200 OK` | 客户端每 5 s 上报一次，写入 **LocationLog**，计费子系统消费 |
| POST | `/api/v1/rides/end` | `{ride_id, lat, lng, lock_status}` | `{order_id, amount, pay_status}` | 结束骑行，更新 **Ride** 状态、生成 **Order**，扣费 |

### 4. 计费 & 对账  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| POST | `/api/v1/billing/report` | `{ride_id, fee, detail}` | `200 OK` | 计费子系统向 **Billing** 服务推送，最终落库到 **Order** |
| GET | `/api/v1/billing/status/{order_id}` | – | `{pay_status, amount}` | 查询支付状态（支付渠道回调后更新） |

### 5. 订单查询  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| GET | `/api/v1/orders` | `?user_id=xx&page=1&size=20` | `{orders:[...]}` | 通过 **Elasticsearch** 快速检索 |
| GET | `/api/v1/orders/{order_id}` | – | `{order_detail, ride_detail, trajectory}` | 包含轨迹（从 HBase 拉取） |

### 6. 运维接口  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| POST | `/api/v1/faults/report` | `{bike_id, user_id, type, desc, lat, lng}` | `{report_id, assign_status}` | 前端/硬件上报故障，写入 **FaultReport** 并推送 MQ |
| POST | `/api/v1/dispatch/assign` | `{report_id, worker_id}` | `200 OK` | 调度中心派单，更新 **DispatchTask** |
| GET | `/api/v1/dispatch/tasks?worker_id=xx` | – | `{tasks:[...]}` | 运维人员查询待处理任务 |

### 7. 幂等性 & 防重放  

- **请求 ID**（`X-Request-ID`）在网关层做 **幂等键**，存入 Redis，30 min TTL。  
- **解锁/结束** 接口采用 **业务幂等**（`ride_id` + `status`），若已完成直接返回成功。

## 第五步：详细组件设计  

### 1. **定位服务（Location Service）**  

1. **GeoHash 分区**：把全国划分为 **12 位** GeoHash（≈ 10 m 精度），每个 GeoHash 对应一个 **Redis ZSET**，成员为 `bike_id:timestamp`，分值为 **距离**（或直接使用 GEOADD）。  
2. **热点缓存**：  
   - **热点区域**（如地铁站）使用 **本地缓存 + LRU**，在每个机器的 **Redis 节点** 上保持热点 GeoHash 的前 500 辆车。  
   - **写入流程**：车辆上报位置 → API Gateway → Kafka → **LocationConsumer** → 更新 **Redis Geo**（异步） + **MySQL**（持久化）。  
3. **查询流程**：  
   - 前端传 `lat,lng,radius` → API GW 计算覆盖的 GeoHash 列表（最多 9 块） → 并行向 **Redis** 发起 `GEORADIUS` → 合并、过滤、排序 → 返回前端。  
4. **容错**：若 Redis 某节点失效，自动路由到 **备份节点**（主从），并在后台 **重建** 失效分区（通过 MySQL 全量同步）。  

### 2. **骑行服务（Ride Service）**  

- **状态机**：`INIT → IN_PROGRESS → FINISHED → CANCELLED`，每一步都有唯一的业务事件。  
- **解锁流程**：  
  1. 前端扫码得到 `bike_id`，调用 `/rides/start`。  
  2. Ride Service 读取 **Redis** 中的车辆状态，检查是否 `AVAILABLE`。  
  3. 使用 **分布式锁**（RedLock）锁定 `bike_id`，防止并发解锁。  
  4. 写入 **Ride** 表（事务），更新 **Bike** 状态为 `IN_USE`，返回 **unlock_token**（一次性、5 min 失效）。  
- **计费上报**：  
  - 客户端每 5 s 上报位置 → Kafka `ride-position` topic。  
  - **Flink** 作业消费，基于 **轨迹点** 计算累计里程、时长，使用 **窗口**（30 s）输出 **partial_fee**，写入 **MySQL billing_tmp** 表。  
- **结束骑行**：  
  1. 前端调用 `/rides/end`，携带 `unlock_token` 与最终坐标。  
  2. Ride Service 校验 token、状态，写入 **Ride** 结束时间、坐标。  
  3. 触发 **Billing**（同步或异步）生成 **Order**，调用 **支付网关**（抽象）。  
  4. 更新 **Bike** 状态为 `AVAILABLE`，释放分布式锁。  

### 3. **计费子系统**  

- **实时计费**：采用 **Flink**，因为它提供 **exactly‑once** 语义（使用 Kafka 的事务）。  
- **计费规则**：  
  - 基础费 + 每分钟/每公里费用 + 时段/套餐折扣。  
  - 规则存放在 **MySQL** 的 `pricing_rules` 表，Flink 在启动时加载，并在 **秒级** 缓存（使用 **RocksDB State Backend**）。  
- **防作弊**：  
  - **GPS 纠偏**：对比车辆实际轨迹与道路网（OSM）匹配，异常跳跃（> 150 km/h）直接标记为 **异常**。  
  - **双向校验**：客户端、硬件、服务器三端时间戳必须在 ±5 s 范围内，否则触发 **风控**。  
  - **上报频率**：若客户端降频（>30 s）系统自动 **结束** 骑行并标记为 **异常**。  

### 4. **订单系统**  

- **事务**：`Ride → Order` 必须 **原子**，使用 **Two‑Phase Commit**（MySQL XA）或 **Saga**（补偿事务）实现。  
- **索引与查询**：订单检索常用 `user_id+date`，在 **ES** 建立倒排索引，支持 **分页+过滤**。  
- **对账**：每日凌晨跑 **对账任务**（MySQL + 支付平台对账表），产生 **对账报告**，异常自动工单。  

### 5. **运维调度（Dispatch Service）**  

- **故障上报流程**：  
  1. 前端/硬件发送 `fault/report` → 写入 **FaultReport** 表（MySQL） + 推送 **Kafka** `fault-event`。  
  2. **DispatchWorker** 消费 `fault-event`，调用 **GIS Service**（如 GeoServer）计算最近的可用运维人员（根据经纬度、当前负载）。  
  3. 生成 **DispatchTask**，写入 **MySQL**，并推送 **WebSocket/FCM** 给对应运维端。  
- **弹性伸缩**：  
  - 当 `fault-event` 队列积压 > 阈值，自动 **扩容** DispatchWorker（K8s HPA）。  
  - 同时 **限流** 故障上报（每辆车每 5 min 最多一次），防止攻击。  

### 6. **监控 & 灰度**  

| 指标 | 监控方式 | 报警阈值 |
|------|----------|----------|
| API latency (p95) | Prometheus + Grafana | > 150 ms (search) / > 300 ms (billing) |
| QPS per service | Prometheus | > 80% of预设上限 |
| Redis hit rate | Redis INFO + Grafana | < 95% |
| Kafka consumer lag | Burrow / Kowl | > 5 min |
| Ride abnormal rate | Flink metrics | > 0.5% |
| DB deadlock | MySQL performance_schema | > 5/min |

- **灰度发布**：使用 **Canary**（K8s Deployment） + **Istio** 路由 5% 流量到新版本，监控异常后快速回滚。

## 第六步：扩展性与高可用设计  

### 1. **热点定位的解决方案**  

1. **GeoHash + 本地缓存**：热点区域（如 10 km²）单独部署 **Redis Cluster** 节点，使用 **读写分离**，把热点 GeoHash 数据复制到 **多副本**。  
2. **分布式读写**：  
   - **写**：车辆上报走 **Kafka** → **LocationConsumer** → 同时写入 **Redis**（主节点）和 **MySQL**（持久化）。  
   - **读**：API GW 直接查询 **Redis**（单机毫秒），若命中率 < 95% 再查询 **MySQL**（备份）。  
3. **缓存预热**：每日凌晨根据历史热点（热力图）预加载对应 GeoHash 到 Redis。  

### 2. **计费一致性与防作弊**  

| 场景 | 方案 |
|------|------|
| 网络抖动导致位置上报丢失 | **Kafka** 持久化 + **Flink** 窗口容错（使用 **Watermark**） |
| 客户端伪造 GPS | **服务器端轨迹匹配**（Map Matching） + **速度阈值**（> 120 km/h） |
| 多端同时解锁同车 | **分布式锁**（RedLock） + **唯一解锁 token**（一次性） |
| 计费结果与支付不一致 | **事务补偿**（Saga） + **每日对账**，异常生成 **人工工单** |

### 3. **故障容错与灾备**  

- **多可用区（AZ）**：每个 AZ 部署完整业务栈，使用 **DNS 轮询** + **健康检查** 实现自动故障转移。  
- **数据复制**：  
  - MySQL 主从（同步）+ **TiDB** 做跨 AZ 多活（读写分离）。  
  - Redis Sentinel + **Cluster**，跨 AZ **异步复制**。  
  - Kafka 跨 AZ **MirrorMaker**，保证消息不丢失。  
- **自动恢复**：  
  - **K8s** + **StatefulSet** 自动重启 Pod。  
  - **Chaos Engineering**（如 Gremlin）定期演练单点故障。  

### 4. **弹性伸缩**  

| 资源 | 触发指标 | 扩容方式 |
|------|----------|----------|
| API GW | QPS > 90% | Horizontal Pod Autoscaler (CPU/RequestRate) |
| Redis | CPU > 80% 或 主从同步延迟 > 30 ms | 添加节点、自动分片 |
| Kafka | Consumer Lag > 1 min | 扩容 Partition 数、增加 Broker |
| Ride Service | RideHeartbeat 延迟 > 5 s | 增加实例、使用 **Service Mesh** 限流 |
| Dispatch Worker | fault-event 队列积压 > 10 k | K8s HPA + 消费者并行度提升 |

### 5. **安全与合规**  

- **HTTPS** + **TLS 双向认证**（设备与服务）。  
- **敏感信息**（手机号、支付信息）加密存储（AES‑256）。  
- **GDPR/国内个人信息保护**：提供 **数据脱敏** 与 **用户删除** 接口。  
- **审计日志**：所有关键操作写入 **ELK**，保留 1 年。

## 第七步：常见面试追问与回答  

### Q1. 高并发定位请求（30 万并发）如何保持 150 ms 延迟？  

**答**：  
1. **分布式缓存**：使用 **Redis GEO**，把全国划分为 **GeoHash** 分区，热点区域单独部署 **专属 Redis 节点**，读缓存直接返回，平均 **RT ≤ 2 ms**。  
2. **读写分离 + 本地缓存**：在每台 API GW 机器上部署 **本地 LRU**（最近 5000 条热点），命中率在热点时可达 **95%**。  
3. **并行查询**：一次搜索可能覆盖 9 个 GeoHash，使用 **并行 pipelined** 请求 Redis，整体 RTT ≈ 30 ms。  
4. **限流 + 限速**：对同一用户/设备每秒最多 5 次搜索，防止恶意刷。  
5. **容量预估**：假设每台 Redis 节点 QPS 30 k，30 万并发可横向扩展到 **10 台**，并使用 **分片**（hash tag）保证均衡。  

### Q2. 计费一致性如何保证，尤其在网络不稳定时？  

**答**：  
- **事件驱动**：骑行期间所有位置、时间点都写入 **Kafka**，Kafka 提供 **持久化 + 顺序**，即使网络中断，客户端会在恢复后 **重发**，Kafka 自动去重。  
- **Flink Exactly‑once**：计费流处理使用 **checkpoint** 与 **transactional sink**，保证 **累计费用** 不会出现重复或遗漏。  
- **最终一致性**：Ride 结束时，系统会 **拉取** Kafka 中该 ride 所有已消费的轨迹，重新计算一次费用（补偿），确保 **实际费用 = 计费系统费用**。  
- **防作弊**：GPS 速度阈值、轨迹匹配、时间戳校验三层防护，异常直接标记为 **手工核算**。  

### Q3. 运维故障上报后如何快速同步到用户侧并触发派单？  

**答**：  
1. **上报** → 写入 **FaultReport** 表 + **Kafka fault-event**。  
2. **实时消费者**（DispatchWorker）读取事件，使用 **GIS Service** 计算最近的运维人员，生成 **DispatchTask**。  
3. **Push**：通过 **WebSocket** 或 **极光推送** 将任务实时推送给运维端 App。  
4. **用户侧**：故障状态会写入 **Redis**（bike_status），搜索接口会直接过滤掉该车，用户搜索时立刻感知。  
5. **弹性**：若故障激增，Kafka 分区自动扩容，DispatchWorker **水平扩容**，并且 **限流** 防止单车频繁上报导致雪崩。  

### Q4. 如何做到跨地区（全国）数据一致性且不影响性能？  

**答**：  
- **业务分库**：按 **省/城市** 分库，业务请求基本落在本地 DB，网络跨区仅在 **全局查询**（如全局排行榜）时使用 **跨库聚合**（使用 **TiDB** 或 **DataX** 同步到全局 OLAP）。  
- **异步复制**：对非强一致业务（如历史轨迹、日志）使用 **异步 Binlog 同步** 到 **ClickHouse/HBase**，延迟在秒级，可接受。  
- **读写分离**：主库负责写，**只读副本**（跨 AZ）负责查询，避免写放大。  

### Q5. 为什么不直接把所有车辆信息放在单一关系库？  

**答**：  
- **写入热点**：定位上报频率极高（每辆车每 5 s），单库会产生 **I/O 瓶颈**，且 **锁竞争** 高。  
- **查询性能**：搜索需要 **空间范围查询**，关系库（MySQL）虽支持 `spatial index`，但在 **千万级** 数据下查询慢且难以水平扩展。  
- **可用性**：单点故障会导致全部定位不可用，违背 **99.95%** 可用性要求。  
- **扩展性**：使用 **Redis GEO**、**Elasticsearch** 可水平扩展，支持 **热点** 与 **分片**。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **定位服务的高并发与低时延** | 需要在 **秒级** 返回 **千级** 车辆列表，同时要处理 **热点**（地铁站）并保持 **数据一致性**。 | - 评估每秒请求数 → 计算 Redis 节点 QPS。<br>- 研究 **GeoHash** + **Redis GEO** 的空间索引特性。<br>- 决定 **热点分片** 与 **本地缓存**（避免跨网）<br>- 设计 **写入链路**（Kafka → Consumer → Redis）以解耦写入压力。 |
| **计费的一致性与防作弊** | 计费涉及 **实时性**、**准确性**、**防篡改**，且要在网络不稳定时仍保证正确。 | - 选用 **事件驱动 + 流处理**（Kafka + Flink）实现 **Exactly‑once**。<br>- 引入 **GPS 轨迹匹配** 与 **速度阈值** 进行风控。<br>- 设计 **补偿事务**（Saga）保证 Ride‑Order 原子性。<br>- 通过 **双向时间戳校验** 防止伪造。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **忽视业务拆分，直接做单体**  
   - **后果**：单体难以横向扩容，热点模块（定位、计费）会成为 **性能瓶颈**，并且一次部署升级会影响全部功能。  
   - **建议**：先划分 **核心微服务**（Auth、Location、Ride、Order、Dispatch），每个服务只负责单一职责，接口使用 **REST/gRPC**。  

2. **只关注 QPS，忘记 **幂等性** 与 **容错**  
   - **后果**：高并发下重复请求会导致 **双扣款、车辆状态冲突**，用户体验极差。  
   - **建议**：在所有写操作（解锁、结束、支付）设计 **幂等键**（如 `X-Request-ID`、业务唯一号），在业务层实现 **幂等检查**（Redis TTL、数据库唯一索引）。  

### 3. 学习建议和可延伸的方向  

| 方向 | 推荐学习路径 |
|------|--------------|
| **分布式缓存（Redis）** | 《Redis 实战》 → 官方文档的 **Cluster、Geo** 章节 → 实战项目（热点缓存） |
| **空间索引 & GIS** | 学习 **GeoHash、QuadTree、R-Tree** → Elasticsearch 地理查询 → 开源 GIS 库（GeoServer、PostGIS） |
| **流处理 & 实时计费** | Apache Flink 官方教程 → Kafka Streams → 实战案例（实时广告计费） |
| **微服务治理** | Spring Cloud / Dubbo → Istio Service Mesh → 了解 **限流、熔断、灰度发布** |
| **高可用 & 灾备** | 《Site Reliability Engineering》 → 了解 **多活、跨 AZ 同步** → 实践 K8s HA、Prometheus + Alertmanager |
| **安全 & 防作弊** | OWASP Top 10 → GPS 防篡改技术 → 双向 TLS、签名机制 |

> **小结**：系统设计不是一次性完成的“图”，而是 **不断迭代** 的过程。先把 **核心业务** 搞通，确保 **功能完整**、**数据一致**，再围绕 **性能瓶颈** 加 **缓存、分片、流处理**，最后做 **容错、监控、自动化**，这样才能在面试中展示 **思路清晰、技术扎实、风险意识强** 的全栈能力。祝你在面试中发挥出色！ 🚴‍♀️🚀
