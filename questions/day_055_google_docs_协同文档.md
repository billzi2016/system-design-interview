# 第 55 天：设计 Google Docs 协同文档

> 生成日期：2026-04-01

---

## 题目背景  
Google Docs 是一款基于浏览器的在线文档编辑与协作平台，用户可以实时创建、编辑、共享文档，并支持多人同步编辑、评论、版本回滚等功能。

## 面试场景设定  
> **面试官**：我们现在要设计一个类似 Google Docs 的协同文档系统，要求支持大规模用户的实时多人编辑，请你从高层架构出发，说明系统的主要组成、数据流向以及关键技术点。

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **文档创建、保存、删除**：用户可以新建文档，编辑后自动保存到云端，支持删除和恢复。 |
| 2 | **实时协同编辑**：多名用户在同一文档上同时编辑，所有人能够在 **≤200 ms** 内看到其他人的光标位置、插入/删除等操作。 |
| 3 | **评论与讨论**：用户可以对文档中的任意段落添加评论、回复，并能 @ 提及其他成员。 |
| 4 | **版本历史 & 回滚**：系统自动记录每次提交的快照，用户可以查看历史版本并回滚到任意版本。 |
| 5 | **权限控制**：文档所有者可以设置查看、编辑、评论等不同权限，并支持对外分享的链接（可设密码或到期时间）。 |
| 6 | **离线编辑**：用户在网络中断时仍能编辑文档，恢复网络后自动同步冲突并合并。 |

## 非功能性需求  

| 指标 | 目标值 | 备注 |
|------|--------|------|
| **DAU（每日活跃用户）** | 2 × 10⁶ | 假设面向企业用户的中大型业务 |
| **并发编辑用户数** | 峰值 5 × 10⁴ 同时编辑同一文档 | 需支撑大型团队协作 |
| **QPS（编辑操作）** | 2 × 10⁴ 读/写请求 / 秒 | 包括光标移动、字符插入/删除、评论等 |
| **端到端延迟** | ≤ 200 ms（95% 请求） | 实时协同编辑的可感知延迟 |
| **可用性** | 99.99%（月度累计） | 每月累计不可用时间 ≤ 4.38 分钟 |
| **存储容量** | 10 PB 文档数据 + 2 PB 版本历史 | 按 100 GB/用户 计 |

## 系统边界  

**本题范围（需要设计）**  
- 文档的实时协同编辑协议（如 OT/CRDT）以及冲突解决机制  
- 前端/后端的通信层、会话管理、负载均衡  
- 文档存储、版本历史、元数据索引  
- 权限校验、分享链接的生成与校验  
- 离线编辑的本地缓存与同步流程  

**不在本题范围（可忽略）**  
- 文档渲染的富文本编辑器实现细节（如 UI 框架）  
- 文字排版、图像/表格等高级编辑功能的实现  
- 第三方插件、API 扩展、企业单点登录（SSO）  
- 跨地域灾备与数据迁移策略（可在讨论中提及但不要求实现）  

## 提示与追问  

1. **冲突解决方案**：如果选择 OT（Operational Transformation）还是 CRDT（Conflict‑free Replicated Data Type），各自的优缺点是什么？在高并发场景下如何保证一致性和性能？  
2. **扩展性设计**：当某篇文档的并发编辑人数从 5 × 10⁴ 突升到 10⁵ 时，系统的热点会出现在哪里？你会怎样进行水平扩展或分片？  
3. **离线同步细节**：用户离线期间产生了多次编辑，恢复网络后如何检测冲突、合并变更并保持版本顺序？请描述具体的数据结构与流程。

# 题解

## 解题思路总览  

1. **先把需求拆开**：把功能需求和非功能需求分别列出，估算流量与存储，明确系统的 **核心瓶颈**（实时协同编辑、强一致性、低延迟）。  
2. **从最小可用系统（MVP）出发**：只实现「文档 CRUD + 基础实时编辑 + 权限校验」的单体服务，帮助面试官快速看到你能把需求落地。  
3. **逐层拆解**：在 MVP 基础上逐步加入 **分布式会话、负载均衡、持久化、版本历史、离线同步** 等高级特性，每加一个特性都要解释**为什么需要**以及**不这么做会有什么问题**。  
4. **技术选型要有依据**：如 OT vs CRDT、WebSocket vs gRPC、关系型 DB vs 文档 DB、对象存储 vs 本地磁盘。  
5. **围绕「高可用、可扩展、低延迟」** 设计：把热点（文档编辑会话）拆分、使用缓存、采用分片、读写分离、异步持久化等手段。  
6. **最后准备面试追问**：冲突解决、热点扩容、离线同步细节等，让面试官看到你对系统细节的深度了解。  

下面按 **由浅入深、一步步搭建** 的顺序展开。

## 第一步：理解需求与规模估算  

### 1. 功能需求拆解  

| 编号 | 功能 | 关键点 | 对系统的影响 |
|------|------|--------|--------------|
| 1 | 文档创建、保存、删除 | 自动保存、软删/硬删 | 持久化、元数据管理 |
| 2 | 实时协同编辑 | ≤200 ms 可见光标、编辑操作 | 实时通信、冲突解决、一致性 |
| 3 | 评论与讨论 | @提及、回复、通知 | 业务子系统（评论服务） |
| 4 | 版本历史 & 回滚 | 快照、增量、恢复 | 大容量存储、异步写 |
| 5 | 权限控制 | 多粒度（查看/编辑/评论）+ 分享链接 | ACL、鉴权、审计 |
| 6 | 离线编辑 | 本地缓存、网络恢复后合并 | 本地数据结构、冲突检测 |

> **注意**：在面试里，先把这些需求用 **业务流程图**（或文字描述）串起来，让面试官看到你对系统整体的认知。  

### 2. 非功能需求量化  

| 指标 | 目标值 | 计算方式 | 说明 |
|------|--------|----------|------|
| DAU | 2 × 10⁶ | 假设 10% 活跃文档 | 需要水平扩展的入口层 |
| 并发编辑用户数（同一文档） | 5 × 10⁴ | 峰值团队协作 | 文档会话是热点，需要拆分 |
| QPS（编辑操作） | 2 × 10⁴ | 包括光标、插入、删除、评论 | 每秒 20k 操作 ≈ 5k 业务请求 + 15k 实时消息 |
| 延迟 ≤200 ms（95%） | | 端到端（浏览器 → 服务器 → 其他客户端） | 需要低延迟网络、就近调度 |
| 可用性 99.99% | ≤4.38 min/月 |  | 主备容错、快速故障转移 |
| 存储 10 PB 文档 + 2 PB 版本 | 100 GB/用户 | 1 TB ≈ 10⁶ 文档 | 需要对象存储 + 分层冷热存储 |

### 3. 粗略流量模型（帮助后面容量规划）  

| 项目 | 估算值 | 计算公式 |
|------|--------|----------|
| 每日编辑请求数 | 2 × 10⁴ × 86 400 ≈ 1.73 × 10⁹ 次 | QPS × 秒/天 |
| 平均每文档大小 | 200 KB（纯文本） | 假设 |
| 每日写入存储 | 1.73 × 10⁹ × 200 KB ≈ 340 TB | 仅实时编辑的增量 |
| 版本快照（每分钟一次） | 2 × 10⁶ × 60 ≈ 1.2 × 10⁸ 份 | 需要压缩、增量存储 |

> **为什么要算这些？**  
> - 容量估算决定 **存储选型**（对象存储+分层）  
> - QPS 决定 **负载均衡、会话服务实例数**  
> - 延迟要求决定 **协议选择（WebSocket）** 与 **就近调度**

## 第二步：高层架构设计  

### 1. MVP（最小可用系统）结构  

```
+-------------------+        +-------------------+
|   前端 (Web/SPA)  | <--WS--> |   API Gateway    |
+-------------------+        +-------------------+
                                 |
                                 v
                         +-----------------+
                         |   文档服务 (单体) |
                         +-----------------+
                                 |
                +----------------+-----------------+
                |                |                 |
                v                v                 v
        +------------+   +------------+   +------------+
        |  元数据 DB |   |  文档存储  |   |  版本存储  |
        +------------+   +------------+   +------------+
```

- **前端** 使用 **WebSocket**（或基于 HTTP/2 的 gRPC-Web）保持长连接。  
- **API Gateway** 负责路由、限流、鉴权。  
- **文档服务** 包含 **OT/CRDT 引擎**、**权限校验**、**即时持久化**。  
- **元数据 DB**（关系型）存放文档属性、ACL、共享链接。  
- **文档存储**（对象存储）保存最新的完整文档。  
- **版本存储**（对象存储 + 差分压缩）保存快照。  

> **不这样做的后果**：所有请求都打到同一台机器，单点故障、容量瓶颈、延迟无法满足 200 ms。  

### 2. 逐步演进到分布式架构  

#### 2.1. 会话层（协同编辑服务）  

- **Session Service**（水平扩展的无状态服务）  
  - 负责 **WebSocket 连接管理**、**分配文档会话 ID**、**心跳检测**。  
  - 使用 **Consistent Hash** 将同一文档的所有连接分配到同一 **协同编辑实例**（后端称为 **Shard**）。  

#### 2.2. 协同编辑核心（OT/CRDT）  

- **编辑引擎 Service**（有状态，按文档分片）  
  - 每个实例维护 **文档的操作日志（Ops Log）**、**当前状态向量**。  
  - **持久化**：将 Ops Log 通过 **Kafka**（或 Pulsar）写入 **持久化层**（如 HBase、Cassandra）供回放和审计。  

#### 2.3. 持久化层  

| 组件 | 角色 | 选型建议 |
|------|------|----------|
| **元数据 DB** | 文档属性、ACL、分享链接 | PostgreSQL + read‑replica |
| **实时操作日志** | OT/CRDT 操作序列 | Kafka + Compact（保留最新状态） |
| **文档快照存储** | 完整文档/版本快照 | 对象存储（OSS/MinIO） + 分层冷热 |
| **增量版本库** | 差分压缩、回滚 | Apache Parquet + HBase（列式） |

#### 2.4. 权限/分享服务  

- **Auth Service**（OAuth2 + JWT）负责登录、Token 发放。  
- **ACL Service**（RBAC）查询 **元数据 DB**，返回 **用户对文档的权限**。  
- **Share Link Service** 生成一次性 **签名 URL**（带 token、过期时间、可选密码），存入 **Redis** 做短期缓存。  

#### 2.5. 评论/讨论子系统  

- **Comment Service**（RESTful）使用 **MongoDB** 或 **Cassandra** 存储 **层次化评论**，支持 **实时推送**（同样走 WebSocket）。  

#### 2.6. 离线同步模块  

- 前端使用 **IndexedDB** 保存本地 **Ops Buffer**。  
- 网络恢复后，客户端把 **未同步 Ops** 按 **Lamport 时间戳** 发给 **编辑引擎**，引擎做 **冲突检测**（OT/CRDT）后返回 **合并结果**。  

#### 2.7. 监控/告警  

- **Prometheus + Grafana** 采集 **QPS、延迟、连接数、错误率**。  
- **Jaeger**（或 Zipkin）追踪跨服务调用链，帮助定位 200 ms 延迟瓶颈。  

> **为什么要把这些组件拆开？**  
> - **会话层** 能做到 **水平扩展**，避免单实例成为热点。  
> - **编辑引擎** 需要 **强一致性**，放在 **同一分片**，减少跨机器的同步开销。  
> - **持久化层** 与 **实时层** 分离，写入快、查询慢的场景分开处理，提高整体吞吐。

## 第三步：数据库设计  

### 1. 元数据表（关系型）  

| 表名 | 主键 | 关键字段 | 备注 |
|------|------|----------|------|
| `users` | `user_id` | email, name, hashed_pwd, created_at | 基础用户信息 |
| `documents` | `doc_id` | owner_id, title, created_at, updated_at, is_deleted | 文档基本信息 |
| `document_acl` | `(doc_id, user_id)` | role (viewer/editor/commenter), expiry_ts | 多对多权限表 |
| `share_links` | `link_id` | doc_id, creator_id, token, expires_at, password_hash, created_at | 短链 + 校验信息 |
| `doc_versions` | `(doc_id, version_number)` | snapshot_key (OSS key), diff_key, created_at, created_by | 版本历史元数据 |

> **索引建议**：  
> - `documents.owner_id`（查询用户所有文档）  
> - `document_acl.user_id`（快速判断用户权限）  
> - `share_links.token`（唯一索引）  

### 2. 实时操作日志（Kafka Topic）  

- **Topic 名称**：`doc-ops-{doc_id}`（分区数 = 1），保证同一文档的操作顺序。  
- **消息结构（JSON/ProtoBuf）**  

```proto
message DocOp {
  string doc_id = 1;
  string user_id = 2;
  uint64 lamport_ts = 3;   // 单调递增
  OpType type = 4;         // INSERT, DELETE, CURSOR_MOVE, COMMENT
  uint32 position = 5;     // 对字符流的偏移
  string content = 6;      // 插入文本（DELETE 时为空）
  string op_id = 7;        // 唯一标识（幂等）
}
```

- **压缩**：Kafka 开启 **Snappy**，开启 **log.compaction**，只保留最新状态的键。  

### 3. 文档快照（对象存储）  

- **Key 规则**：`docs/{doc_id}/snapshots/{version_number}.gz`  
- **内容**：完整的 **UTF‑8 文本**，gzip 压缩。  
- **元数据**（存入 `doc_versions`）记录 **SHA256**、**大小**、**创建者**。  

### 4. 增量版本（差分）  

- 使用 **Rope** 或 **Delta encoding** 生成 **diff**（如 `bsdiff`），存储在 `diff_key`。  
- 读取时 **先读最近的完整快照 + 之后的 diff** 进行合并。  

### 5. 评论存储（文档型）  

| 集合 | 示例文档结构 |
|------|--------------|
| `comments` | `{ "doc_id":"d123", "comment_id":"c456", "author_id":"u1", "parent_id":null, "content":"好点子", "created_at":..., "mentions":["u2"] }` |

- **索引**：`doc_id`、`author_id`、`created_at`（用于分页）

## 第四步：核心 API 设计  

> **原则**：RESTful 用于 CRUD、权限管理；WebSocket 用于实时编辑与推送。  

### 1. 鉴权 & 会话  

| 方法 | 路径 | 描述 | 返回 |
|------|------|------|------|
| `POST /api/v1/login` | 登录，返回 JWT | `{ token, expires_in }` |
| `POST /api/v1/logout` | 失效 JWT | `204 No Content` |
| `GET /api/v1/me` | 获取当前用户信息 | `{ user_id, email, name }` |

### 2. 文档管理（REST）  

| 方法 | 路径 | 描述 | 关键参数 |
|------|------|------|----------|
| `POST /api/v1/docs` | 创建文档 | `title`, `content`（可选） | `owner_id` 来自 JWT |
| `GET /api/v1/docs/{doc_id}` | 获取文档元数据 + 最新快照 URL | `If-None-Match` 用于缓存 |  |
| `PATCH /api/v1/docs/{doc_id}` | 更新标题、元数据（非内容） | `title?`、`is_deleted?` | ACL 校验 |
| `DELETE /api/v1/docs/{doc_id}` | 软删（放入回收站） |  |  |
| `POST /api/v1/docs/{doc_id}/restore` | 恢复软删文档 |  |  |

### 3. 权限 & 分享  

| 方法 | 路径 | 描述 |
|------|------|------|
| `PUT /api/v1/docs/{doc_id}/acl` | 批量设置 ACL（owner） |
| `POST /api/v1/docs/{doc_id}/share` | 生成分享链接，参数 `expire_seconds`, `password?` |
| `GET /share/{token}` | 通过短链打开文档（返回登录页面或直接进入编辑） |

### 4. 版本历史  

| 方法 | 路径 | 描述 |
|------|------|------|
| `GET /api/v1/docs/{doc_id}/versions` | 列表，分页 |
| `GET /api/v1/docs/{doc_id}/versions/{ver}` | 下载指定快照（或 diff） |
| `POST /api/v1/docs/{doc_id}/versions/{ver}/restore` | 回滚到指定版本 |

### 5. 评论（REST）  

| 方法 | 路径 | 描述 |
|------|------|------|
| `POST /api/v1/docs/{doc_id}/comments` | 新增评论，字段 `parent_id`, `content`, `mentions[]` |
| `GET /api/v1/docs/{doc_id}/comments` | 分页获取 |
| `PATCH /api/v1/comments/{comment_id}` | 编辑（仅作者） |
| `DELETE /api/v1/comments/{comment_id}` | 删除（软删） |

### 6. 实时编辑（WebSocket）  

- **连接 URL**：`wss://realtime.example.com/ws?doc_id={doc_id}&token={jwt}`  
- **消息类型**（JSON）  

```json
{
  "type": "op",          // op / cursor / ack / ping
  "payload": { ... }
}
```

- **op 示例**  

```json
{
  "type":"op",
  "payload":{
    "op_id":"uuid",
    "user_id":"u123",
    "lamport_ts": 987654321,
    "op_type":"INSERT",
    "position": 45,
    "content":"hello"
  }
}
```

- **服务器回执**  

```json
{
  "type":"ack",
  "payload":{"op_id":"uuid","server_ts":123456}
}
```

- **广播**：服务器把同一文档的所有已连接客户端 **广播** 该 op（除发送者外），客户端使用 **OT/CRDT** 进行本地变换后渲染。

## 第五步：详细组件设计  

### 1. 实时协同编辑协议（OT vs CRDT）  

| 维度 | OT | CRDT |
|------|----|------|
| **原理** | 对操作序列进行 **转化**（Transformation），确保所有副本执行相同的最终序列 | 每个副本维护 **可合并的状态向量**，操作天然 **幂等且可交换** |
| **实现难度** | 需要 **中心化的转化服务器**，处理变换函数；对 **并发冲突** 较敏感 | 分布式 **无中心**，每个节点都能自行合并，代码更复杂但理论上更健壮 |
| **性能** | 转化过程 O(N)（N 为并发操作数），在 **高并发** 场景会产生瓶颈 | 合并成本 O(1)（基于局部状态），更适合 **5×10⁴+** 并发 |
| **历史兼容** | 已被 Google Docs、Microsoft Office Online 使用多年，生态成熟 | 被 **Automerge、Yjs** 等现代编辑器采用，社区在增长 |
| **网络需求** | 必须 **顺序可靠**（TCP）且 **低延迟**，否则转化冲突增多 | 对 **消息乱序** 有容忍度，天然支持离线编辑 |
| **适用场景** | 小规模协作、对已有 OT 引擎有经验的团队 | 大规模协作、需要离线/跨设备同步的场景 |

**本系统的取舍**  
- **并发人数 5 × 10⁴** 以上，**CRDT** 更具伸缩性，且天然支持离线编辑。  
- 若团队对 OT 更熟悉且想快速交付 MVP，可先实现 **OT**（如 **Google Wave** 的实现），后期再迁移到 **CRDT**（如 **Yjs**）。  

> **面试官追问**：“如果用 OT，如何避免单点瓶颈？”  
> - 采用 **分区 + 多实例**：同一文档的所有操作必须落到同一 **转化节点**（使用 **Consistent Hash**），该节点内部使用 **单线程事件循环**（避免锁竞争），并把转化结果写入 **Kafka**，其余实例只负责转发。  

### 2. 会话管理（WebSocket + Session Service）  

1. **入口**：用户先通过 **API Gateway** 完成 JWT 鉴权，得到 **access token**。  
2. **建立连接**：浏览器使用 `wss://realtime.example.com/ws?doc_id=xxx&token=jwt`。  
3. **Session Service**  
   - **验证 JWT**（使用公钥/私钥）并抽取 `user_id`。  
   - **查找或创建文档会话**：  
     - 使用 **Consistent Hash** (`hash(doc_id) % N`) 将 `doc_id` 映射到 **编辑实例**（EditNode）。  
     - 将 WebSocket **加入该实例的连接池**，并把 **user_id → connection** 存入 **Redis（Hash）** 以便快速定位。  
   - **心跳**：每 30 s 客户端发送 ping，服务端更新 Redis TTL（防止僵尸连接）。  

4. **故障转移**：若某个 EditNode 宕机，Session Service 根据 **Hash 再次映射** 把用户重新路由到新的实例，客户端会自动 **reconnect**（WebSocket 自动重连）。  

### 3. 编辑引擎（CRDT 示例）  

- **内部数据结构**：**RGA（Replicated Growable Array）** 或 **Sequence CRDT (LSEQ/Yjs)**。每个字符对应一个 **Identifier**：`[timestamp, client_id]`。  
- **插入**：生成全局唯一 **Identifier**，把字符插入到有序列表中。  
- **删除**：标记对应 Identifier 为 **tombstone**（不实际移除，便于后续合并）。  
- **合并**：收到远端操作后，只需比较 Identifier 的字典序即可决定插入位置，**O(log N)**（使用 **B‑Tree / Skip‑List**）。  

#### 代码示例（伪代码）  

```go
type Identifier struct {
    Timestamp uint64 // Lamport
    ClientID  string // UUID of editing client
}

type Char struct {
    ID   Identifier
    Val  rune
    Del  bool // tombstone
}

// 文档状态保存在内存的有序结构
type CRDTDocument struct {
    chars []*Char // sorted by ID
    mutex sync.RWMutex
}

// 插入操作
func (d *CRDTDocument) Insert(pos int, ch rune, clientID string) Identifier {
    d.mutex.Lock()
    defer d.mutex.Unlock()
    ts := atomic.AddUint64(&globalLamport, 1)
    id := Identifier{Timestamp: ts, ClientID: clientID}
    // 在 pos 前后生成一个新的 ID，使其字典序位于正确位置
    // 这里省略具体的 ID 生成算法（如 LSEQ）
    newChar := &Char{ID: id, Val: ch}
    d.chars = insertAt(d.chars, pos, newChar)
    return id
}

// 删除操作
func (d *CRDTDocument) Delete(id Identifier) {
    d.mutex.Lock()
    defer d.mutex.Unlock()
    for _, c := range d.chars {
        if c.ID == id {
            c.Del = true
            break
        }
    }
}
```

- **持久化**：每隔 **N** 条操作或 **T** 秒（如 1 s）把 **全量状态快照**（压缩后）写入对象存储；同时把每条操作写入 **Kafka** 供回放。  

### 4. 离线编辑同步流程  

1. **本地缓存**：  
   - 使用 **IndexedDB** 保存 `ops_buffer[]`（每条操作的完整 CRDT Op），以及 **本地 Lamport 时钟**。  
2. **网络恢复**：  
   - 客户端把 **本地最高 Lamport** 与服务器返回的 **当前全局 Lamport** 对比。  
   - 将 **ops_buffer** 按 **Lamport** 顺序发送给服务器。  
3. **服务器端合并**（CRDT）  
   - 接收客户端批量 Ops，遍历每条：  
     - 若 `op.lamport_ts <= server.max_ts`，说明已被其他客户端覆盖，直接 **ignore**（幂等）。  
     - 否则执行 **Insert/Delete**，更新本地文档状态。  
   - 完成后 **返回**：  
     - 新的 **全局 Lamport**  
     - 服务器已接受的 **op_id 列表**（客户端用于清理本地缓存）  
     - 如有冲突（OT 场景），返回 **转化后的 Ops**。  
4. **客户端更新**：  
   - 根据服务器返回的 **ack** 删除对应本地缓存。  
   - 若服务器返回 **额外的变更**（其他用户的操作），将其合并到本地 UI。  

> **为什么要保留 Lamport 时间戳？**  
> - 保证 **全局有序**（即使消息乱序到达），防止「后来的编辑覆盖了先来的编辑」的歧义。  

### 5. 权限校验流程  

1. **API Gateway** 收到请求后，先解析 **JWT**，得到 `user_id`。  
2. 调用 **ACL Service**（内部缓存 Redis）查询 `doc_id + user_id` 的 **role**。  
3. **缓存失效**：若 Redis miss，则查询 **PostgreSQL**，并写回 Redis（TTL 5 min）。  
4. **业务层** 根据 `role` 决定是否允许：  
   - `viewer` → 只能读取快照、光标不可见  
   - `commenter` → 可读、可发评论、不可编辑  
   - `editor` → 完全编辑权限  
5. **分享链接**：访问 `/share/{token}` 时，系统把 `token` 解析为 `doc_id`、`role`（如只读），并在 **Session Service** 为该匿名用户注入临时 **JWT**。  

### 6. 版本历史与回滚  

- **写入快照**：每 **N** 分钟或每 **M** 次编辑（取决于业务）生成一次 **完整快照**，存入 OSS。  
- **增量 diff**：使用 **Rope diff** 对比上一次快照，生成 **diff 文件**，压缩后存储。  
- **回滚**：用户点击回滚，系统把对应快照（或 diff+base）加载到 **编辑引擎**，生成新的 **编辑操作**（相当于「撤销」所有后续操作），并写入 **Ops Log**，确保所有协作者看到同一状态。

## 第六步：扩展性与高可用设计  

### 1. 热点识别  

| 热点 | 产生原因 | 影响 |
|------|----------|------|
| **编辑会话实例** | 同一文档的所有实时操作都落在同一节点 | CPU、内存、网络带宽成为瓶颈 |
| **Ops Log（Kafka）分区** | 每文档对应 1 分区，为保证顺序 | 分区数限制并发文档数 |
| **元数据 DB 读写** | 权限校验、文档列表查询 | 读写竞争、锁等待 |
| **对象存储带宽** | 版本快照上传/下载 | 大文件 IO 延迟 |

### 2. 横向扩容方案  

#### 2.1. 文档会话拆分（Shard）  

- **哈希分片**：`shard_id = hash(doc_id) % SHARD_COUNT`。  
- **动态扩容**：采用 **Consistent Hash Ring**，当节点数增加时，只迁移少量文档的会话。  
- **迁移流程**：  
  1. 新节点加入 Ring，计算受影响的 `doc_id` 范围。  
  2. 老节点把对应 **状态快照**（CRDT 文档）写入 **Redis**（或对象存储）并通知客户端重新 **connect**。  
  3. 客户端在 **WebSocket 重连** 时自动落到新节点。  

#### 2.2. Kafka 分区扩容  

- **每文档单独分区** 已满足顺序需求。若文档数量超过 **Kafka 最大分区数**（常见 10k–50k），可以：  
  - **多租户分区**：把同一文档的 **Ops** 拆分成 **子流**（如 `doc_id:segment_index`），每段对应一个分区，客户端合并。  
  - **使用 **KSQL** 或 **Kafka Streams** 做 **合并**，保证最终顺序。  

#### 2.3. 元数据 DB 横向扩展  

- **读写分离**：主库处理写入（文档创建、ACL 变更），多个 **只读副本** 供查询。  
- **分库分表**：按 **owner_id**（用户）或 **doc_id** 范围做水平分片，降低单库压力。  

#### 2.4. 缓存层  

- **Redis** 用作：  
  - **会话映射**（doc_id → edit_node）  
  - **ACL 缓存**（doc_id+user_id → role）  
  - **热点文档的最新快照**（供快速恢复离线编辑）  
- **缓存失效策略**：TTL 5 min + 主动失效（文档写入后）  

#### 2.5. 对象存储 CDN  

- **快照与版本文件** 使用 **对象存储 + CDN**（边缘节点缓存），减轻核心网络压力。  
- **冷热分层**：  
  - 最近 30 天的快照放在 **SSD**（如 COS）  
  - 旧版本转入 **归档存储**（如 Glacier）  

### 3. 高可用设计  

| 组件 | 冗余方式 | 故障检测 & 恢复 |
|------|----------|-----------------|
| API Gateway | 多实例 + **ELB**（负载均衡） | 健康检查，自动剔除 |
| Session Service | 多实例 + **Consistent Hash** | 心跳失效后重新路由 |
| EditNode (CRDT 引擎) | **主备**（同一 shard 两副本） | 同步状态到 **Redis**，故障时从备份恢复 |
| Kafka | 多副本 (replication factor ≥3) | 自动 leader 选举 |
| PostgreSQL | 主从双机 + **Patroni** 进行自动故障转移 | 读写分离，写故障切换 |
| Redis | 主从 + Sentinel / Redis Cluster | 自动故障转移 |
| 对象存储 | 多 AZ 冗余 | 云服务本身提供 99.999% 可用性 |

> **为什么要这么多层冗余？**  
> - 实时编辑对 **延迟** 和 **一致性** 极其敏感，一旦任意节点卡顿都会直接导致用户卡顿。  
> - 多副本 + 自动切换可以在 **秒级** 内恢复服务，满足 **99.99%** 的可用性目标。  

### 4. 性能优化技巧  

1. **批量写入**：编辑引擎每 100 条 Ops 合并一次写入 Kafka，降低网络请求次数。  
2. **压缩**：WebSocket 消息开启 **gzip**（浏览器自动），减少带宽。  
3. **增量推送**：只推送 **变更区域**（如光标位置、字符区间），避免全量文档广播。  
4. **客户端渲染**：使用 **Virtual DOM** + **diff**，只更新受影响的行，降低前端渲染开销。  
5. **热点文档预热**：将最近活跃文档的 **CRDT 状态** 放在 **Redis**，新用户加入时直接读取，省去加载过程。

## 第七步：常见面试追问与回答  

### Q1️⃣：**OT 与 CRDT 选哪个？如果选 OT，如何解决转化服务器的单点瓶颈？**  

**回答要点**  
- **CRDT**：天然支持离线、并发高、无中心化转化，适合 5×10⁴+ 并发。  
- **OT**：实现成熟，适合小规模；如果业务对兼容性有要求可以先用 OT。  
- **OT 单点瓶颈**：  
  1. **分片**：同一文档所有操作落在同一转化实例（通过 Consistent Hash）。  
  2. **多实例水平扩容**：每个实例内部使用 **单线程事件循环**，避免锁竞争。  
  3. **转化结果写入 Kafka**，其他实例只负责 **广播**，不参与转化。  
  4. **故障转移**：实例挂掉后，Session Service 重新分配连接，新实例从 Kafka **回放** 最新状态恢复。  

### Q2️⃣：**当并发编辑人数从 5×10⁴ 突升到 10⁵，系统的热点会出现在哪里？如何应对？**  

**回答要点**  
- **热点**：  
  1. **单个 EditNode**（CPU、内存、网络）  
  2. **Kafka 分区**（单文档 1 分区）  
  3. **Redis 会话映射**（读写竞争）  
- **扩容方案**：  
  - **分片细化**：把同一文档再细分为 **子文档块**（如 1k 行为一个块），每块对应独立 **EditNode**，使用 **Operation Routing Table** 把光标所在块的操作路由到对应节点。  
  - **多分区 Kafka**：把文档的 Ops 按块分配到不同分区，保持每块内部顺序。  
  - **负载均衡**：使用 **SDN** 或 **L4 负载** 将流量均匀分配到不同 AZ。  
  - **客户端分区感知**：前端把编辑范围（行号）发送给后端，后端返回对应 **块 ID**，以后所有操作走该块的通道。  

### Q3️⃣：**离线编辑时产生的冲突如何检测并合并？请给出具体数据结构与步骤。**  

**回答要点**  
- **使用 CRDT（如 RGA/LSEQ）**，每个字符都有全局唯一 **Identifier**（Lamport+client_id）。  
- **离线期间**：客户端本地 **ops_buffer** 按生成顺序保存，每条 op 带 **lamport_ts**。  
- **恢复网络**：  
  1. 客户端把本地最高 lamport（`local_max`) 与服务器返回的 `global_max` 进行比较。  
  2. 将 `ops_buffer` 按 `lamport_ts` 排序，逐条发送。  
  3. 服务器对每条 op 检查：  
     - 若 `op.lamport_ts <= global_max`，说明已被其他客户端处理，**忽略**（幂等）。  
     - 否则执行 **Insert/Delete**，并更新 `global_max`。  
  4. 服务器返回 **ACK 列表** 与最新 `global_max`。  
  5. 客户端删除已 ACK 的 ops，若有冲突（同一位置的 Insert），CRDT 的 Identifier 排序自动决定最终顺序，**无需额外冲突检测**。  

- **若使用 OT**：需要 **转化函数** `transform(op_local, op_remote)`，在恢复时把本地操作与服务器未见过的远端操作两两转化，得到可直接执行的本地 ops。  

### Q4️⃣：**如何保证文档的最终一致性？**  

**回答要点**  
- **CRDT**：基于 **数学可交换性**，所有副本在收到相同集合的 ops 后必然收敛。  
- **OT**：通过 **转化函数** 保证 **同序列化**（所有副本执行相同的转化后得到相同状态），并在 **服务器** 保存 **历史变换日志** 供回放。  
- **持久化**：所有 ops 同步写入 **Kafka**（持久化日志），系统崩溃后可 **重放** 生成相同状态。  
- **快照**：定期生成 **完整快照**，启动时先加载快照再回放后续 ops，保证快速恢复且一致。  

### Q5️⃣：**如果某个用户的网络延迟很高（>500 ms），会不会影响整体协作？怎么缓解？**  

**回答要点**  
- **CRDT 的优势**：即使本地操作延迟到达，其他用户仍能继续编辑，最终状态仍会收敛。  
- **本地回显**：客户端在本地先执行操作并展示（乐观 UI），不必等待服务器确认。  
- **延迟隐藏**：使用 **操作压缩**（批量发送）和 **心跳**，让网络不佳的用户仍能保持会话。  
- **后端**：对每个编辑实例设置 **操作超时**，若某用户长时间无心跳，则将其从会话中剔除，防止资源占用。

## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 考虑因素 | 最终选择 | 思考过程 |
|------|----------|----------|----------|
| **OT 还是 CRDT** | 并发人数（5×10⁴），离线编辑需求，团队经验，实现复杂度 | **先实现 OT（MVP） → 预留 CRDT 迁移路径** | 1）OT 实现成熟、代码量少，能快速交付；<br>2）CRDT 更适合大并发和离线，但实现难度高，面试时间有限。<br>3）在答案中说明两者优劣，并给出迁移方案（抽象编辑接口、统一 Op 格式），展示对系统演进的思考。 |
| **文档会话的分片策略** | 单节点 CPU/内存上限、网络带宽、状态同步成本、故障恢复复杂度 | **Consistent Hash + 子块分片（按行/段）** | 1）直接把所有用户连到同一实例会导致热点；<br>2）纯粹的哈希分片可以均衡负载，但单文档仍是热点；<br>3）进一步把文档拆成块，每块单独转化/存储，兼顾局部一致性与整体收敛。 |

### 2️⃣ 新手最容易犯的错误  

1. **忽视** **一致性与延迟的权衡**：直接把所有编辑操作写入数据库再广播，导致 **>200 ms** 延迟。  
   - 正确做法：先在内存/缓存层处理实时协作，异步持久化到日志系统。  

2. **把存储设计成单体**：把文档、版本、元数据放在同一个关系型库，导致 **写放大** 与 **查询瓶颈**。  
   - 正确做法：使用 **专用的对象存储** 保存大文件、**Kafka** 保存日志、**PostgreSQL** 只存元数据。  

3. **忘记** **幂等性**：对编辑操作没有唯一 `op_id`，导致重试时产生重复插入。  
   - 解决：每个客户端生成 **全局唯一 op_id**（UUID）并在服务端做幂等校验。
