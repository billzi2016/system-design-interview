# 第 3 天：设计 Twitter/X

> 生成日期：2026-05-23

---

## 1️⃣ 题目背景  
Twitter（亦称 X）是一款面向全球的实时微博客社交平台，用户可以发布短文本（Tweet）、关注他人、转发、点赞以及进行实时互动。面试任务是设计一个能够支撑全球数亿活跃用户的“Tweet 发布与时间线分发”系统。

## 2️⃣ 面试场景设定  
> **面试官**：*“我们现在需要设计 Twitter 的核心服务——用户发推和时间线的实时分发。请从高层到细节逐步展开你的设计思路，包括系统架构、数据模型、扩展性和容错机制等。”*

## 3️⃣ 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **发布 Tweet**：用户可以发布最长 280 字符的文本、图片/视频（单条最多 4 张图片或 1 条视频）并可选择公开或仅限关注者可见。 |
| 2 | **关注/取消关注**：用户可以关注或取关其他用户，系统实时维护关注关系。 |
| 3 | **时间线读取**：用户打开主页或个人页时，系统返回按时间倒序排列的最新 Tweet 列表（包括自己和关注的用户）。 |
| 4 | **转发 & 点赞**：用户可以对任意可见 Tweet 进行转发（Retweet）或点赞，转发会出现在关注者的时间线中。 |
| 5 | **推文删除 & 隐私修改**：用户可以删除已发布的 Tweet，或修改可见范围（公开 ↔ 仅关注者）。 |
| 6 | **防刷频率**：对同一用户的发布频率进行限制（如 1 秒内最多 1 条），并提供基本的内容审核（关键词过滤）。 |

## 4️⃣ 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 2.5 亿 | 全球活跃用户规模，假设 70% 为活跃发布/阅读用户。 |
| **QPS（写入请求）** | 50,000 TPS | 约 10% 的 DAU 每秒发布一次 Tweet（280 字），峰值 2 倍（100k TPS）。 |
| **QPS（读取请求）** | 300,000 TPS | 每位活跃用户平均每分钟刷 10 次时间线，峰值 5 倍（1.5M TPS）。 |
| **平均延迟** | < 200 ms（读取）<br> < 100 ms（写入） | 从用户发起请求到收到响应的端到端时延。 |
| **可用性** | ≥ 99.99%（每月累计宕机 ≤ 5 分钟） | 关键业务必须具备多可用区冗余。 |
| **存储规模** | 约 150 PB（原始 Tweet + 多副本 + 媒体文件） | 假设每条 Tweet 平均 1 KB，媒体文件 10 KB/条，保留 3 年并有 3 副本。 |

## 5️⃣ 系统边界  

**本题范围内**（需要设计/讨论）：  
- Tweet 的写入路径、持久化、索引、分片与副本策略。  
- 关注关系的存储与查询（用户‑关注者图）。  
- 时间线的实时分发与离线合并（拉取 vs 推送模型）。  
- 基础的缓存层、热点数据处理、流量削峰。  
- 容错、灾备、监控告警的基本思路。

**本题范围外**（不必深入）：  
- 私信、群聊、话题（Hashtag）趋势分析。  
- 广告投放、推荐系统（个性化推荐）。  
- 内容审核的机器学习模型实现。  
- 第三方登录、OAuth、支付等账户安全细节。  
- 法律合规（GDPR、数据脱敏）细节实现。

## 6️⃣ 提示与追问  

1. **数据模型与分片**  
   - *“如果用户的关注关系非常不均衡（明星用户拥有数千万粉丝），你会如何设计关注关系的存储和查询？”*  

2. **时间线的实现方式**  
   - *“在高并发写入的情况下，你更倾向于使用 ‘推模型’ 还是 ‘拉模型’ 来构建时间线？两者的优缺点是什么？”*  

3. **缓存与热点处理**  
   - *“针对热点用户的时间线，你会如何利用缓存层（如 Redis）来降低后端压力？缓存失效和一致性如何保证？”*

## ## 解题思路总览

1. **先把需求翻译成技术指标**  
   - 功能点 → 必须的系统模块（写、读、关系、分发、审核）。  
   - 非功能点 → QPS、延迟、可用性、存储量 → 用来指导选型（单机/分布式、缓存、复制、分片）。

2. **从最小可行系统（MVP）出发**  
   - 先实现 **单机 + 单库 + 直接查询**，确保功能完整。  
   - 再一步步把 **瓶颈点**（写入、读取、热点）拆分成独立服务或加层。

3. **常用设计工具**  
   - **容量估算**（DAU、TPS、数据量） → 决定机器规格、分片数。  
   - **CAP & BASE** → 解释为什么在社交媒体里我们可以接受**最终一致性**。  
   - **数据流图（DFD）**、**时序图** → 帮助面试官快速了解流程。  

4. **始终围绕“三大核心”**：**性能**、**可用性**、**可扩展性**。每加入一个新组件，都要检查这三点是否得到改善。

## ## 第一步：理解需求与规模估算

### 1️⃣ 功能需求拆解（对应模块）

| 功能 | 对应系统子模块 | 关键操作 |
|------|----------------|----------|
| 发布 Tweet | **Tweet Service**（写入） | 接收、校验、持久化、写入消息队列 |
| 关注/取消关注 | **Social Graph Service** | 维护用户‑关注者关系 |
| 时间线读取 | **Timeline Service**（读） | 合并/拉取用户的 Tweet，返回给前端 |
| 转发 & 点赞 | **Interaction Service** | 记录交互、触发 Timeline 更新 |
| 删除 & 隐私修改 | **Tweet Service**（更新） | 软删、权限变更、回写缓存 |
| 防刷 & 内容审核 | **RateLimiter / Content Filter** | 限流、关键词过滤、返回错误码 |

> **为什么要划分这么细？**  
> - 每个子模块的 **读写特性** 不同，后面可以针对性扩容（写多读少 vs 读多写少）。  
> - 业务耦合度低，**单独升级/容错** 更容易。

### 2️⃣ 规模估算（核心指标）

| 指标 | 计算方式 | 结果（峰值） |
|------|----------|--------------|
| **DAU** | 2.5 亿 | — |
| **活跃发布用户** | 70% × 10% = 7% → 1.75 亿人/天 | — |
| **写入 QPS** | 1.75e8 / 86400 ≈ 2 025 TPS（平均）<br>峰值 2× → **≈ 4 k TPS**（保守）<br>**面试给的 50k‑100k TPS** 作为上限 | 50 k–100 k |
| **读取 QPS** | 每活跃用户每分钟 10 次 → 2.5e8 × 10 / 60 ≈ 4.2 M TPS（平均）<br>峰值 5× → **≈ 21 M TPS**（极端）<br>**面试给的 300k‑1.5M TPS** 为可接受范围 | 300 k–1.5 M |
| **单条 Tweet 大小** | 文本 1 KB + 媒体 10 KB（最多 4 张/1 视频）| 11 KB |
| **每日写入量** | 100 k TPS × 1 KB × 86400 ≈ **8.6 TB**（仅文本）<br>加媒体约 10 倍 → **≈ 86 TB** |
| **存储需求（3 副本、3 年）** | 86 TB × 365 × 3 ≈ **94 PB**（约 150 PB 包含索引 & 预留） | 150 PB |

> **如果不做容量估算**，很容易在面试中出现“选型随意、后期不可行”的毛病。这里的数字帮助我们决定 **分片数、机器规格、缓存容量**。

## ## 第二步：高层架构设计

### 1️⃣ MVP（最小可行系统）结构

```
+-------------------+      +-------------------+
|   Frontend (Web)  | ---> |  API Gateway /    |
+-------------------+      |  Load Balancer    |
                           +-------------------+
                                   |
                +------------------+-------------------+
                |                  |                   |
          Tweet Service      Social Graph Service   Timeline Service
                |                  |                   |
                +------------------+-------------------+
                                   |
                             MySQL (单库单表)
                                   |
                               Redis Cache
```

- **单体+单库**：所有写、读都直接落库。  
- **优点**：实现快、易调试、面试演示完整流程。  
- **缺点**：**水平扩展受限**，写入/读取瓶颈明显。

#### 关键技术选型解释

| 组件 | 选型 | 为什么要这么选 | 如果不这么选会怎样 |
|------|------|----------------|-------------------|
| **API Gateway / LB** | Nginx / Envoy + Consul | 统一入口、路由、限流、TLS，支持灰度发布 | 直接暴露内部服务，安全、运维成本高 |
| **Auth Service** | JWT + OAuth2 | 分离鉴权，减轻业务服务压力，易水平扩容 | 每个业务服务都要实现鉴权，代码重复、错误率上升 |
| **RateLimiter** | Redis‑based token bucket (分布式) | 低延迟、全局统一配额 | 本地限流不跨实例，刷频用户仍能绕过 |
| **Message Queue** | Kafka (高吞吐、持久) | 写入解耦、可做重放、天然分区 | 直接同步写库会导致写入延迟升高，系统不易伸缩 |
| **Timeline / Fanout Workers** | Go / Java + 多线程 | CPU 密集型批处理，易水平扩容 | 单进程无法满足 100k TPS 的写入分发 |
| **Timeline Store** | Cassandra / ScyllaDB (宽列) | 适合 **时间序列写入**，查询单行低延迟 | 传统 RDBMS 在大量时间线写入时会出现热点、锁争用 |
| **Social Graph DB** | HBase / DynamoDB (键值) + 二级索引 | 关注/粉丝查询是 **点查询**，键值模型天然支持 | 使用关系型 DB 会导致 JOIN、锁、扩展困难 |
| **Redis Cache** | 主从 + Cluster | 读取热点（热门用户时间线）亚毫秒，支持 LRU 驱逐 | 直接查询后端存储导致读延迟 30‑100 ms，影响用户体验 |
| **CDN** | CloudFront / 阿里云 OSS + CDN | 媒体文件大、读多写少，缓存加速 | 直接从对象存储读取会产生网络瓶颈、成本高 |

> **设计原则**：**写**走 **Kafka** → **Worker** → **Timeline Store**（**异步**），**读**走 **Cache → Timeline Store**（**同步**），这样可以做到 **写入 100 ms 内返回**（只等 Kafka ack），而 **时间线最终一致**（几百 ms 内同步）。

## ## 第三步：数据库设计

### 1️⃣ 数据模型（核心表）

> **注意**：这里使用 **宽表**（宽列）设计，方便一次读取完整时间线。

| 表/列名 | 说明 | 主键/索引 | 备注 |
|--------|------|-----------|------|
| **Tweet** | 推文主表 | `tweet_id (UUID)` PK | 包含 `user_id, text, media_urls, visibility, created_at, is_deleted` |
| **User_Timeline** | 用户时间线（写入 fanout 后的结果） | `user_id` PK + `tweet_id` Clustering Key (desc) | 每条记录对应一条可见的 tweet；使用 **Cassandra** 的宽列特性：一行保存该用户最近 N 条 tweet（如 5000 条） |
| **User_Follow** | 关注关系 | `user_id` PK + `followed_id` Set | 通过 **Redis Set** + **MySQL** 持久化 |
| **User_Follower** | 粉丝列表（逆向） | `user_id` PK + `follower_id` Set | 只保留 **明星用户** 的粉丝列表在 **HBase**，普通用户用 **Redis** |
| **Interaction** | 点赞/转发等交互 | `tweet_id` PK + `type_user` Map | 用 **Cassandra** 计数列（counter） |
| **Rate_Limit** | 频率控制 | `user_id` PK | Redis TTL 计数 |

#### 关键设计点

1. **宽列时间线表（User_Timeline）**  
   - 每个 `user_id` 对应一行，内部列是 `tweet_id:timestamp`（倒序）。  
   - **读取**只需一次 `SELECT`（Cassandra）即可返回前 N 条。  
   - **写入**（fanout）时，将 tweet_id 追加到关注者的列族。  

2. **关注关系的分层存储**  
   - **热点用户（粉丝 > 1M）**：使用 **HBase** 分区存储，按 `user_id` 分区、`follower_id` 排序。  
   - **普通用户**：使用 **Redis Set**（内存）+ **MySQL** 持久化，支持 **O(1)** 判断是否已关注。  

3. **分片（Sharding）策略**  
   - **Tweet 表** 按 **user_id hash** → 多库（MySQL）或 **topic**（Kafka）分区。  
   - **User_Timeline** 按 **user_id hash** → Cassandra **token ring** 自动分片。  
   - **Social Graph** 按 **user_id** 分区，保证同一个用户的关注列表落在同一分区，减少跨分区查询。  

4. **副本数**  
   - **Cassandra**：`Replication Factor = 3`（跨 3 AZ），保证读写可用性。  
   - **MySQL**：主从复制（1 主 + 2 从），同城跨机房双活。  
   - **Redis**：主从 + Sentinel，自动故障转移。  

### 2️⃣ 索引与查询

| 查询场景 | 需要的索引/结构 | 说明 |
|----------|----------------|------|
| **读取用户时间线** | `User_Timeline(user_id)`（主键） | 直接定位行，倒序读取前 N 列 |
| **查看单条 Tweet** | `Tweet(tweet_id)` | 主键查，返回完整信息 |
| **判断是否关注** | `User_Follow(user_id)` Set | O(1) 判断 |
| **获取粉丝列表（明星）** | `User_Follower(user_id)` 分区表 | 分页扫描 |
| **点赞计数** | `Interaction(tweet_id)` Counter | 原子递增 |
| **查询最近 N 条全局热门 Tweet** | **ElasticSearch**（全文/热度） | 侧边功能，非必须 |

## ## 第四步：核心 API 设计

> **RESTful** 方案易于演示，实际可使用 **gRPC** 进一步降低序列化开销。

### 1️⃣ 统一返回结构

```json
{
  "code": 0,
  "msg": "OK",
  "data": { ... }
}
```

- `code = 0` 表示成功，非 0 为业务错误或系统错误。  
- 统一错误码方便前端统一处理。

### 2️⃣ 关键 API 列表（示例）

| 方法 | 路径 | 请求体 | 说明 | 关键点 |
|------|------|--------|------|--------|
| **POST** | `/tweets` | `{ "text":"...", "media":[url], "visibility":"PUBLIC|FOLLOWERS_ONLY" }` | 发布 Tweet | 1️⃣ 速率校验、关键词过滤 → 2️⃣ 写入 Kafka (topic= tweets) → 3️⃣ 返回 tweet_id（即时） |
| **GET** | `/users/{uid}/timeline?limit=20&cursor=xxx` | — | 拉取时间线 | 从 Redis 缓存读取；缓存未命中 → 查询 Cassandra；使用 cursor 实现分页（基于 tweet_id） |
| **POST** | `/users/{uid}/follow` | `{ "target_id":123 }` | 关注 | 1️⃣ 判断是否已关注 → 2️⃣ 写入 Redis Set + MySQL async → 3️⃣ 发送粉丝变更事件到 Kafka |
| **DELETE** | `/users/{uid}/follow/{target_id}` | — | 取关 | 同上，删除集合成员 |
| **POST** | `/tweets/{tid}/like` | — | 点赞 | 幂等：先检查 Redis set 是否已点赞 → 若未点赞，写入 Interaction Counter + 发送事件 |
| **POST** | `/tweets/{tid}/retweet` | — | 转发 | 与发布类似：生成新 tweet（type=RETWEET），写入同一流程 |
| **DELETE** | `/tweets/{tid}` | — | 删除 | 软删 `is_deleted=1` → 发送删除事件 → Timeline Worker 过滤掉该 tweet |
| **PATCH** | `/tweets/{tid}` | `{ "visibility":"FOLLOWERS_ONLY" }` | 隐私修改 | 更新 Tweet 表 → 重新投递到 Fanout Worker（重新分发） |

### 3️⃣ 幂等性 & 防重放

- **写请求**使用 **Idempotency-Key**（UUID）由前端生成，后端在 Redis 中记录 `key → result`（TTL 24h），重复请求直接返回已记录的结果。  
- **点赞/转发** 同理，使用 `user_id:tweet_id:action` 组合键做幂等控制。

### 4️⃣ 限流 & 内容审核

- **RateLimiter**：在 API Gateway 调用 Redis **Lua 脚本**实现 **令牌桶**，返回 `429 Too Many Requests`。  
- **Content Filter**：发布前走 **Trie**（内存）或 **机器学习服务**（异步），如果命中违规词返回 `400 Bad Request`。

## ## 第五步：详细组件设计

### 1️⃣ 写入路径（发布 Tweet）

```
Client → API GW → Auth → RateLimiter → Tweet Service → Kafka (tweets topic)
      ↘︎                 ↘︎
   Media Service      Content Filter
```

#### 步骤拆解

| 步骤 | 说明 | 关键技术 |
|------|------|----------|
| **1. 鉴权** | 校验 JWT、获取 `user_id` | Nginx + Lua |
| **2. 限流** | 令牌桶，1 秒 1 条 | Redis Lua 脚本 |
| **3. 内容审核** | 关键词过滤（Trie）+ 敏感图片检测（异步） | 本地 Trie + 后置 ML 服务 |
| **4. 媒体上传** | 客户端直传至对象存储（S3），返回 URL | Pre‑signed URL |
| **5. 写入 Kafka** | 生成 `TweetMessage {tweet_id, user_id, text, media, visibility, timestamp}`，发送到 `tweets` 分区 | Kafka Producer (acks=all) |
| **6. 同步响应** | 只等 Kafka **ack**，返回 `tweet_id` 给前端（<100 ms） |  |
| **7. 异步 Fanout** | **TweetWorker** 消费 `tweets`，查询关注者列表 → 为每位关注者写入 `User_Timeline`（Cassandra）<br>如果粉丝数 > 100k，走 **批量写入**（Cassandra batch） | Kafka Consumer + ThreadPool |
| **8. 缓存更新** | 将热点用户的时间线写入 Redis（`ZSET`），保持最新 N 条 | Redis ZADD + EXPIRE |
| **9. 计数/统计** | 发送到 **Analytics** Topic（PV、UV） | Kafka → ClickHouse |

> **为什么使用 Kafka？**  
> - **解耦**：写入不必等 fanout 完成，降低前端感知延迟。  
> - **持久**：突发流量（明星用户）时，Kafka 能做 **缓冲**，防止下游服务瞬间崩溃。  
> - **可重放**：若 Timeline Worker 出错，只需重新消费。

### 2️⃣ 读取路径（时间线）

```
Client → API GW → Auth → Timeline Service → Redis Cache → Cassandra
```

#### 读取流程

| 步骤 | 说明 | 关键技术 |
|------|------|----------|
| **1. 鉴权** | 与写入相同 | JWT |
| **2. 缓存查询** | 从 Redis `ZSET(user_id)` 拉取最新 N 条 tweet_id | `ZRANGE`（倒序） |
| **3. 缓存击穿** | 若 key 不存在或已失效，走 **缓存预热**：<br>① 查询 Cassandra（单行）<br>② 将结果写回 Redis（设置 TTL） | **Cache‑Aside** 模式 |
| **4. 数据回填** | 根据 tweet_id 批量读取 **Tweet** 表（MySQL）或 **Cassandra**（宽列）获取文本、媒体等 | `SELECT ... WHERE tweet_id IN (…)` |
| **5. 合并 & 排序** | 已经是倒序，直接返回；若跨分区则在业务层合并 |  |
| **6. 分页** | 使用 **cursor**（上一次返回的最小 tweet_id）实现 **基于 ID 的分页**，避免 **offset** 带来的全表扫描 | `WHERE tweet_id < cursor ORDER BY tweet_id DESC LIMIT N` |

#### 缓存失效 & 一致性

- **写入后**：Timeline Worker **立即**把新 tweet 推入热点用户的 Redis ZSET（`ZADD`），并 **设置** `EXPIRE 1h`（热点时间线常驻 1 小时）。  
- **删除/隐私修改**：发送 `TweetDelete`/`TweetUpdate` 事件，Worker **遍历** 受影响的用户时间线，**标记**（软删）或 **删除**对应 tweet_id。若已在 Redis 中，则 `ZREM`。  
- **一致性模型**：**最终一致**（几百 ms 内同步），满足社交平台对实时性的容忍度。

### 3️⃣ 关注关系管理

1. **关注/取关 API**  
   - 写入 Redis Set：`SADD user:{uid}:followed {target_id}`  
   - 异步写入 MySQL（持久化） → **双写**（先写缓存，后写库）  
   - 发送 `FollowEvent` 到 Kafka，**Fanout Worker** 读取后对新关注者的时间线进行**回填**（把目标用户最近 N 条 tweet 写入该用户的 timeline）。  

2. **粉丝列表查询**  
   - 对 **普通用户**：直接 `SMEMBERS`（Redis） → 结果可能在 **10k** 以内。  
   - 对 **明星用户**：使用 **HBase** 分区存储，每个分区 1M 粉丝 → 支持 **分页扫描**。  

3. **热点用户的 Fanout**  
   - **批处理**：明星用户的每条 tweet 产生的 fanout 可能是 **10M+**。  
   - 采用 **两阶段写入**：<br>① 将 tweet 写入 **“明星用户专用 fanout 队列”**（Kafka 分区），<br>② **批量写入** 到 **Cassandra**（使用 `BATCH`），**并行**写入到多个节点，降低单节点压力。  

### 4️⃣ 点赞 & 转发（交互）

| 操作 | 写入路径 | 计数更新 |
|------|----------|----------|
| **点赞** | `POST /tweets/{tid}/like` → Interaction Service → Kafka `likes` topic | Consumer 增加 `Interaction(tweet_id).like_counter`（Cassandra Counter） |
| **转发** | 同发布 Tweet，只是 `visibility=PUBLIC`，`type=RETWEET`，并在 `Tweet` 中记录 `origin_tid` | 同时触发 **Fanout**，让转发也出现在关注者时间线 |

- **幂等**：使用 **Redis Set** `user:{uid}:liked:{tid}` 防止重复计数。  

### 5️⃣ 防刷频率 & 内容审核

- **RateLimiter**（全局）使用 **Redis** 的 **Lua 脚本**，确保原子性：  
  ```lua
  local key = KEYS[1]
  local limit = tonumber(ARGV[1])
  local now = tonumber(ARGV[2])
  local ttl = tonumber(ARGV[3])
  local cur = redis.call('INCR', key)
  if cur == 1 then
      redis.call('EXPIRE', key, ttl)
  end
  if cur > limit then
      return 0   -- 超限
  else
      return 1   -- 正常
  end
  ```
- **关键词过滤**：  
  - **Trie** 结构存放敏感词，**O(L)**（L 为文本长度）匹配。  
  - 若匹配，返回错误码 `403` 并记录审计日志。

## ## 第六步：扩展性与高可用设计

### 1️⃣ 多可用区（AZ）部署

| 组件 | 部署策略 |
|------|----------|
| **API Gateway / LB** | **跨 AZ**（Anycast IP），每个 AZ 1 实例，使用 **Health Check** 自动剔除失效节点 |
| **Tweet Service / Interaction Service** | **无状态**，横向扩容，部署在每个 AZ |
| **Kafka** | **3‑副本**（跨 AZ），使用 **Rack‑aware** 分配保证同一分区副本不在同 AZ |
| **Cassandra** | **NetworkTopologyStrategy**（每 AZ 至少 2 节点） |
| **Redis** | **Cluster + Sentinel**，主节点分布在不同 AZ，故障时自动迁移 |
| **MySQL** | **双主双从**（跨 AZ），使用 **Galera** 或 **MySQL Group Replication** 实现写写同步 |
| **CDN** | 全球边缘节点，缓存媒体文件 |

> **不做跨 AZ**：单点故障导致整站不可用，违背 99.99% 可用性。

### 2️⃣ 容灾 & 数据恢复

- **备份**：每日对 MySQL、Cassandra 使用 **快照**（S3/OSS），保留 30 天。  
- **灾难恢复（DR）**：在不同地域保留 **只读副本**（如使用 **Aurora Global Database**），在主站故障时切流至 DR 区。  
- **Kafka MirrorMaker**：将生产集群的 topic 同步到异地集群，防止单机房数据丢失。  

### 3️⃣ 监控、告警、限流

| 维度 | 监控指标 | 工具 |
|------|----------|------|
| **系统** | CPU、内存、磁盘 I/O、网络流量 | Prometheus + Grafana |
| **业务** | TPS、成功率、平均延迟、错误率（4xx/5xx） | OpenTelemetry + Jaeger |
| **队列** | Kafka lag、消费者吞吐、消费延迟 | Kafka Exporter + Prometheus |
| **缓存** | Redis hit rate、eviction、slowlog | Redis Exporter |
| **限流** | RateLimiter 拒绝率、令牌桶剩余 | 自定义仪表板 |
| **告警** | 任何指标超过阈值 5 分钟 → PagerDuty/钉钉 | Alertmanager |

### 4️⃣ 自动弹性伸缩

- **水平伸缩（HPA）**：基于 **CPU**、**QPS**、**Kafka lag** 自动扩容 Pod（K8s）或实例（ASG）。  
- **热点自动缓存**：使用 **Redis LRU** + **热点检测**（访问次数 top 5%）自动迁移到 **专用缓存节点**。  

### 5️⃣ 数据一致性策略

| 场景 | 一致性需求 | 采用方案 |
|------|------------|----------|
| **发布 Tweet** | **最终一致**（几百 ms） | 写入 Kafka → 异步 fanout |
| **时间线读取** | **强一致**（最新的 1 条必须可见） | 写入后立即写入 Redis（热点） |
| **点赞计数** | **弱一致**（可略有延迟） | Cassandra Counter 异步批量刷到 MySQL 统计表 |
| **关注关系** | **强一致**（防止重复关注） | Redis Set + MySQL 事务写入（双写） |

## ## 第七步：常见面试追问与回答

### Q1️⃣ 关注关系非常不均衡（明星用户几千万粉丝），如何存储与查询？

**回答思路**：

1. **分层存储**  
   - **普通用户**：使用 **Redis Set** + **MySQL** 持久化，查询 O(1)。  
   - **明星用户**：使用 **HBase**（列式存储）或 **Cassandra** 按 `user_id` 分区，每行保存 **粉丝 ID 列**，支持 **分页扫描**（`Scan` + `Filter`）。

2. **写入策略**  
   - 当明星用户发布 tweet 时，**不在写入阶段遍历所有粉丝**（会产生巨大的 fanout），而是 **写入专用 “明星 fanout 队列”**，**批量**写入每批 10k‑100k 粉丝的时间线。  

3. **读取策略**  
   - **拉模型**：用户读取自己的时间线时，只需要查询 **自己** 的 `User_Timeline`，不必关心目标用户的粉丝数。  
   - **推模型**（回填）：新关注明星时，系统会 **一次性把该明星最近 N 条 tweet** 批量写入新关注者的时间线（离线任务），避免每次读取都去联表。

4. **热点缓存**  
   - 明星的 **最新 100 条 tweet** 放入 **Redis ZSET**，所有关注者在读取时间线时直接命中缓存，减少对 HBase 的访问。

> **不做分层**：所有用户都放在同一张表会导致 **单表热点**（写放大、读放大），严重影响性能。

---

### Q2️⃣ 推模型 vs 拉模型，哪种更适合高并发写入的场景？

| 维度 | **推模型（Fanout）** | **拉模型（Pull）** |
|------|----------------------|-------------------|
| **写入延迟** | 需要将 tweet 写入每个关注者的时间线 → **写放大**，对热点用户压力大 | 写入仅一次 → **写入快** |
| **读取延迟** | 读取直接从本地时间线表 → **毫秒级**，无需额外聚合 | 每次读取需要聚合关注者的 tweet → **查询成本高**，尤其关注者多时 |
| **热点处理** | 明星用户的 fanout 需要 **批处理**、**分区**，实现复杂 | 明星用户的 tweet 仍然是单条写入，读取时聚合成本高 |
| **一致性** | 写完后即可看到 → **强实时** | 读取时可能有 **短暂延迟**（聚合/缓存失效） |
| **实现难度** | 需要 **消息队列 + Worker**，处理写放大 | 相对简单，只要提供 **查询 API** |

**结论**（针对本题）：

- **核心时间线**采用 **混合方案**：  
  - **普通用户**（关注者 ≤ 10k）使用 **推模型**（fanout），保证读取极快。  
  - **明星用户**（关注者 > 10M）采用 **拉模型 + 预计算**：将明星的最新 N 条 tweet **缓存**（Redis/ZSET），并在用户读取时 **合并**（一次性查询两份：普通时间线 + 明星缓存）。  

> **如果只选一种**：**推模型**更符合 **读取低延迟**（<200 ms）需求，但必须对热点用户做特殊处理（分区、批量写入），否则写入会成为瓶颈。

---

### Q3️⃣ 针对热点用户的时间线，如何利用缓存降低后端压力？缓存失效和一致性怎么保证？

**答案要点**：

1. **缓存层次**  
   - **一级缓存**：**Redis ZSET**，键 `timeline:{uid}`，成员 `tweet_id`，分值 `timestamp`。  
   - **二级缓存**：**CDN** 只缓存媒体文件。  

2. **写入同步**  
   - 当 **TweetWorker** 成功写入 **Cassandra**，立即 **ZADD** 新 tweet 到 **热点用户**（关注者数 > 100k）对应的 `timeline:{uid}`。  
   - 对于 **删除/隐私修改**，发送 **TweetDelete** 事件 → **ZREM**。  

3. **缓存失效策略**  
   - **TTL**：`timeline:{uid}` 设为 **1 小时**，热点用户会被频繁访问，TTL 自动刷新。  
   - **主动刷新**：读取 API 若发现 **缓存未命中**，触发 **异步预热任务**：从 Cassandra 拉取最近 N 条 tweet 写入 Redis。  

4. **一致性保证**  
   - **最终一致**：写入后 0‑300 ms 内缓存与后端同步。  
   - **读‑写冲突**：使用 **Redis Lua 脚本**实现原子写入（`ZADD` + `EXPIRE`），避免并发导致的脏数据。  
   - **缓存穿透**：对不存在的用户（被攻击的 UID），返回 **空集合** 并缓存 **空结果**（`TTL=5 min`），防止频繁查询底层 DB。  

5. **监控**  
   - 监控 **Redis hit rate**，若低于 90% 则说明热点用户缓存失效，需要 **扩容**或 **调大 TTL**。  

> **不使用缓存**：每次读取都要访问 **Cassandra**，在高并发下读延迟会突破 200 ms，用户体验下降。

---

### Q4️⃣ 如果系统需要支持“用户在 5 分钟内编辑已发布的 tweet”，该如何实现？

1. **软删 + 版本**：在 `Tweet` 表增加 `edit_version`、`updated_at`。  
2. **写入流程**：编辑请求走 **Tweet Service** → **Kafka `tweet_edit` topic** → **EditWorker** → **更新 MySQL / Cassandra** 并 **写入新版本**。  
3. **Timeline 更新**：EditWorker 将 **edit event** 推送到 **Timeline Workers**，它们在对应用户的 **Redis ZSET** 中找到旧 tweet_id，**replace** 为新 tweet（或直接更新缓存）。  
4. **幂等性**：使用 `tweet_id + edit_version` 作为唯一键，防止重复处理。  

> **如果不做软删**，直接覆盖旧记录，可能导致 **读到旧数据**（缓存未刷新），影响一致性。

## ## 心得与反思

### 1️⃣ 本题最难的 1–2 个设计决策

| 决策 | 挑战点 | 思考过程 |
|------|--------|----------|
| **关注关系的热点处理** | 明星用户粉丝数可能上亿，单表查询/写入会成为瓶颈。 | ① 先评估粉丝数分布（大部分用户 < 10k，少数明星 > 1M）。<br>② 采用 **分层存储**（普通用户 → Redis+MySQL，明星 → HBase）。<br>③ 设计 **批量 fanout** 与 **拉模型预缓存** 两种路径，兼顾写入与读取。 |
| **推模型 vs 拉模型的混合** | 写放大 vs 读取延迟的权衡。 | ① 列出两者优缺点，结合 QPS/延迟目标。<br>② 通过 **容量估算**（写入 100k TPS）判断单纯推模型在明星用户会导致 **写放大 10^6‑10^7**，不可接受。<br>③ 决定 **混合**：普通用户推，明星用户拉 + 缓存。 |

### 2️⃣ 新手最容易犯的错误（至少 2 条）

| 错误 | 结果 | 正确做法 |
|------|------|----------|
| **直接把所有关注关系放在单张关系表（MySQL）**，不区分热点 | 查询明星粉丝列表慢、写入时锁竞争严重，导致 **读写阻塞**。 | 使用 **分层存储**：普通用户 Redis+MySQL，明星用户 HBase/Column‑store。 |
| **在写入 Tweet 时同步 Fanout 所有关注者**（阻塞写） | 前端响应时间 > 1 s，写入 TPS 受限，系统无法达标。 | **异步化**：写入 Kafka → Worker 异步 Fanout，前端只等 **Kafka ack**（<100 ms）。 |
| **缓存只做写穿透（写入时不更新缓存）** | 读取热点数据仍命中 DB，造成 **缓存失效率高**，压力未降。 | **写回/写穿**：写入成功后立即更新 Redis（热点用户）或使用 **Cache‑Aside** 结合 **主动预热**。 |
| **忽视幂等性**，导致重复计数/重复写入 | 数据不准确、用户体验差。 | 为每个写请求生成 **Idempotency‑Key**，在 Redis 中记录结果，实现幂等。 |
