# CDN 内容分发网络 系统设计面试题

## 1️⃣ 题目背景  
内容分发网络（CDN）是部署在全球各地的边缘节点，用于缓存和加速用户对静态资源（如图片、视频、JS/CSS）以及动态内容的访问。通过就近取回数据，能够显著降低响应时间、减轻源站压力并提升可用性。

## 2️⃣ 面试场景设定  
> **面试官**：  
> “假设我们要为一个全球活跃用户数约 5 亿、日均产生 10 TB 静态资源的媒体平台设计一套 CDN 系统。请从整体架构出发，说明你会如何设计这套系统来满足高并发访问、低延迟以及高可用的需求。”  

（候选人可以先概述整体思路，然后逐步展开各个关键模块的设计）

## 3️⃣ 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **内容缓存与分发**：边缘节点能够缓存源站的静态资源（图片、音视频、CSS/JS），并根据用户请求返回最近的副本。 |
| 2 | **缓存刷新 & 失效**：支持源站主动或定时刷新缓存（如 `PURGE`、`Invalidate`），保证热点内容的实时性。 |
| 3 | **流量调度 & 负载均衡**：在同一地域的多个边缘节点之间进行流量分配，防止单点过载。 |
| 4 | **智能路由**：基于用户 IP、网络条件、内容类型等因素，将请求路由至最优节点（最小 RTT、带宽最优）。 |
| 5 | **访问日志 & 统计**：记录每次请求的来源、命中率、延迟等信息，供后端分析和计费使用。 |
| 6 | **安全防护**：提供基础的 DDoS 防护、访问控制（IP 黑名单/白名单）以及 TLS 加密传输。 |

## 4️⃣ 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **每日活跃用户（DAU）** | 5 × 10⁸ | 全球范围内的活跃用户数 |
| **峰值 QPS（每秒请求数）** | 200 k QPS | 假设 10% 的用户在同一秒发起 2 次请求 |
| **平均响应延迟** | ≤ 50 ms（边缘节点）<br>≤ 150 ms（跨区域） | 包括网络 RTT 与服务器处理时间 |
| **可用性** | 99.99%（月度） | 年度累计不可用时间 < 5 小时 |
| **存储容量** | 150 TB（边缘缓存） | 覆盖全球 100+ 边缘节点的总缓存容量，满足 3‑day 热点缓存需求 |
| **缓存命中率** | ≥ 85%（整体）<br>≥ 95%（热点区域） | 通过智能预取与分层缓存实现 |

> *注：上述数字均为估算，可根据实际业务场景进行微调。*

## 5️⃣ 系统边界  

**本题范围内需要设计的部分**  
- 边缘节点的缓存层、存储结构与淘汰策略  
- 全局调度层（DNS/Anycast、GSLB）与智能路由算法  
- 缓存刷新/失效机制（API、时间窗口）  
- 监控、日志收集与基本统计报表  
- 基础的安全防护（TLS、IP 过滤、速率限制）  

**本题不考虑的部分**  
- 源站的业务逻辑与存储（假设已有可靠的对象存储）  
- 高级安全功能（WAF、Bot 管理、细粒度鉴权）  
- 多媒体转码与自适应码流（如 HLS/DASH）  
- 边缘计算（函数即服务）或实时数据处理（流处理）  
- 计费系统与商务层面的 SLA 合同细节  

## 6️⃣ 提示与追问  

1. **缓存淘汰策略**：如果热点内容超过 150 TB，如何在保证热点命中率的前提下设计多层缓存（如 LRU + LFU + 预取）？  
2. **跨地域容灾**：当某个大区的边缘节点全部失效，系统如何快速切换流量到其他可用节点，并且对用户透明？  
3. **扩容与弹性**：在突发流量（如大型活动）期间，如何实现边缘节点的快速扩容（例如使用容器化/无服务器）并保持一致的路由策略？

# 题解

## 解题思路总览
> **目标**：从“只要能跑起来的最小系统”一步步演进，构建一套 **全球化、低延迟、高可用、易扩展** 的 CDN。  
> **思路**：  
> 1. **先把需求拆解** → 明确功能、非功能、规模。  
> 2. **画出最小可行架构**（单个边缘节点 + 简单 DNS 负载均衡），验证核心流程：用户 → DNS → Edge → 源站。  
> 3. **逐层加入**：多地域、多节点、智能路由、缓存淘汰、刷新、监控、安全。每加入一个子系统，都要解释 **为什么需要它**、**不加会出现什么问题**。  
> 4. **把每个子系统拆成**：数据模型、API、内部实现、选型理由。  
> 5. **最后讨论**：扩容、容灾、常见面试追问以及个人心得。  

下面我们按照 **“需求 → 高层 → 细节 → 可用性 → 进阶”** 的顺序展开，每一步都写明 **决策原因** 与 **若不这么做的后果**，帮助完全没有系统设计经验的同学逐步建立完整的思考框架。

## 第一步：理解需求与规模估算

| 维度 | 业务需求 | 关键指标 |
|------|----------|----------|
| **用户规模** | 全球活跃用户 5×10⁸（DAU） | - |
| **流量规模** | 日均 10 TB 静态资源（上传到源站）<br>峰值 QPS 200 k | 200 k QPS ≈ 1.2 M RPS（每秒请求次数） |
| **时延要求** | 边缘 ≤ 50 ms，跨区 ≤ 150 ms | 包括网络 RTT + 服务器处理 |
| **可用性** | 99.99%（月度） | 不可用时间 < 5 h/30 d |
| **缓存容量** | 150 TB（全局） | 目标覆盖 3 天热点 |
| **命中率** | ≥ 85% 全局，≥ 95% 热点 | 通过预取、分层缓存实现 |
| **安全** | TLS、IP 黑白名单、DDoS 基础防护 | - |

### 规模换算（帮助选型）

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日请求次数** | 5×10⁸ DAU × 2 次/人 ≈ 1 B 次/日 | 约 11.6 k QPS（均值） |
| **峰值 QPS** | 给定 200 k QPS | 约 17 × 均值，说明流量具有明显波峰 |
| **每秒流量** | 200 k req/s × 100 KB（平均对象） ≈ 20 GB/s | 需要 **上行 160 Gbps** 的网络带宽（单个大区） |
| **边缘存储** | 150 TB / 100 个节点 ≈ 1.5 TB/节点 | 每个节点使用 **NVMe SSD**（写放大低） |
| **热点对象数** | 假设 30 % 请求命中 10 % 热点对象 → 10 TB × 30 % = 3 TB 热点 | 需要 **专门的热缓存层**（例如 LFU） |

> **为什么要做这些换算？**  
> 1. **容量选型**：缓存、网络、计算资源必须基于真实流量才能不出现瓶颈。  
> 2. **成本评估**：了解每个节点需要多少硬件，帮助后续的 **弹性伸缩** 与 **容灾** 设计。  
> 3. **性能目标**：把时延目标拆解为 **网络 RTT**（≈ 20 ms）+ **处理时间**（≈ 30 ms）等，指导后面的 **协议栈** 与 **缓存实现**。

## 第二步：高层架构设计

### 1. 最小可行系统（MVP）

```
[User] → DNS（Anycast） → [Edge Node] → (Cache Miss) → [Origin Server / Object Store]
```

- **DNS + Anycast**：把用户的域名解析到最近的 **Anycast IP**，路由层面把流量导向最近的边缘 PoP（Point of Presence）。  
- **Edge Node**：单机（或少量实例）负责 **缓存、请求转发、TLS 终止**。  
- **Origin**：假设已有对象存储（S3、COS），提供 **GET** 接口。

> **不做 DNS/Anycast**：所有请求都会落到单一中心，网络 RTT 轻易超过 150 ms，且中心节点会成为单点故障（违背高可用）。

### 2. 完整全局架构（分层）

```
                         +-------------------+
                         |   全局调度层       |
                         |  (DNS + Anycast)  |
                         +--------+----------+
                                  |
               +------------------+-------------------+
               |                                      |
        +------+-------+                      +-------+------+
        |  GSLB / Geo  |                      |  GSLB / Geo |
        |  Load Balancer (Region)             |  Load Balancer (Region)
        +------+-------+                      +-------+------+
               |                                      |
   +-----------+-----------+              +-----------+-----------+
   |   区域调度（Region）  |              |   区域调度（Region）  |
   |  (健康检查、权重)    |              |  (健康检查、权重)    |
   +-----------+-----------+              +-----------+-----------+
               |                                      |
   +-----------+-----------+          +---------------+--------------+
   |   边缘 PoP（City）    |  … … …   |   边缘 PoP（City）            |
   |  多节点 (Nginx/Envoy) |          |  多节点 (Nginx/Envoy)          |
   +-----------+-----------+          +---------------+--------------+
               |                                      |
        +------+-------+                     +--------+--------+
        |   Edge Cache |                     |   Edge Cache   |
        |  (SSD + RAM) |                     |  (SSD + RAM)   |
        +------+-------+                     +--------+--------+
               |                                      |
        +------+-------+                     +--------+--------+
        |  Origin /   |                     |  Origin /      |
        |  Object Store|                     |  Object Store  |
        +--------------+                     +----------------+
```

**核心组件解释**  

| 组件 | 作用 | 关键技术/选型 | 为什么要这么做 |
|------|------|---------------|----------------|
| **Anycast DNS** | 把用户请求路由到最近的 **边缘 PoP**（基于 BGP 路径）。 | **Route53 / Cloudflare DNS** + **BGP Anycast** | 能在 **毫秒级** 内把流量送到最近的节点，降低网络 RTT。 |
| **GSLB（全局服务器负载均衡）** | 在同一大区内部进一步分配流量（基于健康、权重、实时 RTT）。 | **NGINX Plus**, **OpenResty**, **Alibaba SLB**, **F5 GTM** | 防止单个 PoP 故障导致用户不可达，实现 **99.99%** 可用性。 |
| **边缘节点（Edge Cache）** | **缓存层**（RAM + NVMe SSD） + **业务入口**（TLS、HTTP、速率限制）。 | **Varnish**, **NGINX + Lua**, **Envoy**, **OpenResty**；存储使用 **RocksDB/LMDB** 作为键值索引。 | 支持 **高并发、低延迟**，且可以灵活扩展插件（如 PURGE、日志）。 |
| **缓存刷新服务** | 接收源站的 **PURGE/Invalidate** 请求或定时失效。 | **RESTful API + Kafka** 消息队列；边缘节点订阅刷新消息。 | 让源站能够 **即时失效** 热点内容，保证内容新鲜度。 |
| **监控/日志** | 收集 **访问日志、命中率、延迟、错误率**。 | **Prometheus + Grafana**, **ELK**（Filebeat → Logstash → Elasticsearch），或 **ClickHouse** 实时分析。 | 为 **容量规划、异常检测、计费** 提供数据。 |
| **安全层** | TLS 终端、IP 黑白名单、基础 DDoS 防护。 | **Cert-Manager + ACME**, **iptables + fail2ban**, **Scrubbing Center**（合作 ISP）。 | 防止 **流量劫持、恶意请求**，满足合规。 |

> **不做 GSLB**：单点 PoP 故障时所有用户请求会返回错误，服务不可用。  
> **不做边缘缓存层**：所有请求都回源，网络带宽和源站压力会成倍增长，时延无法达标。

## 第三步：数据库设计

CDN 本身不是传统的关系型业务系统，主要 **元数据**（对象信息、缓存策略、刷新记录、节点健康）需要持久化。我们采用 **两套存储**：

1. **关系型 DB（MySQL/PostgreSQL）**：用于 **强一致、事务性** 的元数据，如域名‑PoP映射、租户配置、刷新任务表。  
2. **键值/时序库（Redis + ClickHouse）**：用于 **高速读取**（对象 ID → 缓存位置）以及 **监控/日志**（时序数据）。

### 1. 关键数据模型（RDBMS）

| 表名 | 主键 | 重要字段 | 说明 |
|------|------|----------|------|
| `cdn_domain` | `domain_id` | `domain_name`, `owner_id`, `origin_endpoint` | 每个 CDN 域名对应的源站入口。 |
| `origin_server` | `origin_id` | `endpoint_url`, `auth_token`, `region` | 源站对象存储信息（可以是多活）。 |
| `edge_node` | `node_id` | `city`, `region`, `ip`, `status`, `capacity_tb` | 边缘节点信息，供 GSLB 使用。 |
| `cache_rule` | `rule_id` | `path_pattern`, `ttl_seconds`, `cache_type` (LRU/LFU) | 不同路径的缓存策略（如 `/video/*` 用 LFU，`/static/*` 用 LRU）。 |
| `purge_task` | `task_id` | `domain_id`, `object_key`, `status`, `created_at` | 源站发起的刷新任务。 |
| `ip_acl` | `acl_id` | `domain_id`, `ip_cidr`, `type` (black/white) | 每个域名的访问控制列表。 |

> **为什么使用关系型库？**  
> - **事务**：刷新任务需要“插入 + 通知”原子操作。  
> - **查询关联**：例如列出某个 region 所有可用节点，需要 join。  
> - **可维护性**：业务配置（TTL、路径规则）频繁变更，关系模型更直观。

### 2. 键值存储（缓存索引）

| Key 格式 | Value | 失效策略 |
|----------|-------|----------|
| `obj:{domain}:{hash(key)}` | `{node_id}:{offset}:{size}` | TTL = 对象的 `cache_rule.ttl`，或 **LRU/LFU** 失效。 |
| `node_status:{node_id}` | `{last_heartbeat, load, health_score}` | 10 s 心跳更新。 |
| `purge:{domain}:{object_key}` | `timestamp` | 1 h（防止重复 PURGE）。 |

- **实现**：使用 **Redis Cluster**（水平扩展）或 **TiKV**（分布式 KV）来保证 **高可用、强一致**。  
- **访问流程**：Edge 节点在收到请求时先查询本地 **LRU/LFU** 索引（内存），未命中再查询 Redis（跨节点命中），最后回源。

### 3. 监控/日志时序库

- **表结构（ClickHouse）**  

```sql
CREATE TABLE cdn_access_log (
    ts               DateTime,
    domain           String,
    node_id          UInt32,
    client_ip        IPv4,
    object_key       String,
    http_status      UInt16,
    resp_time_ms     UInt16,
    cache_status     Enum8('HIT' = 1, 'MISS' = 2, 'STALE' = 3, 'BYPASS' = 4),
    bytes_sent       UInt64
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(ts)
ORDER BY (domain, node_id, ts);
```

- **查询示例**：实时查询 **命中率** `SELECT sum(cache_status='HIT')/count() FROM cdn_access_log WHERE ts > now() - INTERVAL 5 MINUTE;`

> **为什么要单独时序库？**  
> - **写入吞吐**：每日 QPS 200 k，日志量约 10 TB/天，传统 RDBMS 处理不了。  
> - **分析性能**：ClickHouse 对聚合查询（命中率、热点对象）极快，支持实时监控面板。

## 第四步：核心 API 设计

下面列出 **源站、运维、监控** 三类对外/内部 API，采用 **RESTful + JSON**（或者 gRPC 用于内部高频通信）。

### 1. 缓存刷新（Purge） API

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| `POST` | `/api/v1/purge` | `{ "domain": "cdn.example.com", "object_key": "/video/123.mp4", "type": "invalidate|delete" }` | `{ "task_id": "uuid", "status":"queued" }` | 源站调用，生成刷新任务，写入 `purge_task` 表并推送至 **Kafka**。 |
| `GET` | `/api/v1/purge/{task_id}` | - | `{ "status":"completed|failed", "detail": "..."} ` | 查询任务进度。 |

- **内部实现**：API Server 将任务写入 MySQL + 发送 **Kafka** 消息 `topic=purge`. 每个 Edge 节点订阅该 topic，收到对应 `domain` 与 `object_key` 后立即 **删除本地缓存**（LRU/LFU 表项），并返回 **ACK** 给 Kafka。  

> **不使用异步队列**：同步调用会因为网络延迟导致源站请求超时，且一次性刷新大量对象会压垮 Edge。

### 2. 配置管理 API（运维）

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| `POST` | `/api/v1/edge/nodes` | `{ "city":"Tokyo","region":"APAC","ip":"203.0.113.10" }` | `{ "node_id":123 }` | 新增 Edge 节点（自动加入 GSLB）。 |
| `PUT` | `/api/v1/cache/rules/{rule_id}` | `{ "ttl_seconds":3600, "cache_type":"LFU" }` | `{ "updated":true }` | 调整缓存策略。 |
| `GET` | `/api/v1/metrics/health` | - | `{ "node_id":123, "status":"healthy", "load":0.42 }` | 查询节点健康。 |

- **安全**：所有管理 API 必须 **JWT + Role‑Based Access Control**，并通过 **TLS** 传输。

### 3. 边缘节点内部 API（高频）

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| `GET` | `http://127.0.0.1:8080/cache/lookup?key=...` | - | `{ "hit":true, "node_id":..., "offset":..., "size":... }` | Edge 本地缓存查询（内存+SSD）。 |
| `POST` | `http://127.0.0.1:8080/cache/evict` | `{ "key":"..." }` | `{ "evicted":true }` | 本地驱逐（用于 PURGE）。 |
| `GET` | `http://127.0.0.1:8080/healthz` | - | `200 OK` | 健康检查，供 GSLB 使用。 |

> **为什么要区分内部/外部 API？**  
> - **内部 API** 频繁、延迟敏感，直接调用本地进程或使用 **Unix Domain Socket**，避免网络往返。  
> - **外部 API** 需要鉴权、审计，流量相对较低。

## 第五步：详细组件设计

### 5.1 边缘节点（Edge Cache）实现细节

#### 5.1.1 请求流程
1. **TLS 终止** → Nginx/Envoy 完成 TLS handshake，获取客户端 IP（X‑Forwarded‑For）。  
2. **路由决策** → 根据 **Host** 与 **Path** 匹配 `cache_rule`（在内存缓存的规则表）。  
3. **本地缓存查询**  
   - **内存索引**：使用 **Cuckoo Filter** + **HashMap** 判断是否命中（O(1)）。  
   - **SSD 查找**：若内存命中但数据在 SSD，读取对应块（使用 **direct I/O** 防止 page cache 冲刷）。  
4. **命中** → 直接返回 **200 + 缓存内容**，记录 **HIT**、响应时间。  
5. **未命中** → 通过 **HTTP/2**（或 **QUIC**）向 **Origin** 发起请求。  
   - **并发控制**：同一对象的并发请求会合并（single flight）避免缓存击穿。  
   - **返回后**：根据 `cache_rule.ttl` 写入 **内存 + SSD**（异步写入队列），并返回给用户。  
6. **日志收集** → 将 **access log** 发送到 **Kafka**（或直接写 ClickHouse）进行聚合。  

#### 5.1.2 缓存淘汰策略
- **冷热分层**  
  - **热层（10 % 容量）**：使用 **LFU**（计数衰减）+ **预取**（基于最近 5 min 热点分析）。  
  - **冷层（90 % 容量）**：使用 **LRU**（基于最近访问时间）。  
- **实现**  
  - **LFU**：每分钟对计数做一次 **衰减**（`cnt = cnt * 0.9`），防止老热点永久占位。  
  - **LRU**：采用 **Clock‑Hand**（近似 O(1)）替代链表，降低内存占用。  
- **预取**：每日凌晨通过 **ClickHouse** 分析热点对象，提前将它们 **push** 到热层（写入 SSD），并在 **TTL** 内不淘汰。  

> **不做冷热分层**：热点对象和冷门对象混在一起，热点被频繁淘汰导致整体命中率下降，尤其在 150 TB 限制下难以满足 95% 热点命中率。

#### 5.1.3 刷新机制
- **主动刷新**（源站调用 `/api/v1/purge`）  
  - 任务写入 MySQL → Kafka → Edge 节点消费。  
  - 节点收到后 **删除本地索引**，若对象正被请求，则返回 **Stale‑while‑revalidate**（返回旧内容并后台重新拉取）。  
- **定时失效**（TTL）  
  - 每个对象在写入时记录 **expire_at**，边缘节点维护 **时间轮**（Timing Wheel）在到期时自动驱逐。  

#### 5.1.4 安全防护
- **TLS**：使用 **Let's Encrypt** 自动签发证书，Edge 节点通过 **cert‑manager** 动态更新。  
- **IP ACL**：在 Nginx 配置 `allow/deny`，规则从 MySQL 同步到本地 **Redis**，每分钟刷新。  
- **速率限制**：基于 **token bucket**（每 IP/客户端）实现每秒最多 10 req，防止突发 DDoS。  
- **基础 DDoS**：在 **Anycast** 前部署 **Scrubbing Center**（合作 ISP）进行流量清洗；Edge 节点仅处理已清洗流量。  

### 5.2 全局调度层（DNS + GSLB）

| 功能 | 选型 | 实现要点 |
|------|------|----------|
| **Anycast IP** | **BGP Anycast**（运营商提供） | 把同一个 /24 IP 广播到全球 PoP，路由自动选最近节点。 |
| **DNS 解析** | **Route53 + Latency‑Based Routing** 或 **OpenDNS** 自建 | 根据用户 DNS 查询的 **RTT** 返回最近的 Anycast IP。 |
| **GSLB** | **NGINX Plus** + **Consul** 或 **Alibaba SLB** | 健康检查（HTTP/HTTPS、TCP） → 动态权重 → 负载均衡。 |
| **健康检查** | **Active probing** 每 5 s → 记录 `latency`、`error_rate` | 结果写入 **Redis**，GSLB 实时读取。 |

> **为什么要把 DNS 与 GSLB 分层？**  
> - **DNS** 负责把用户尽量引到最近的 **大区**（跨洲），只需要 **毫秒级** 延迟。  
> - **GSLB** 在大区内部做 **细粒度负载均衡**（考虑节点健康、实时 RTT），能够在同一大区内部快速切换。  

### 5.3 监控、日志与统计

1. **日志采集**  
   - Edge 节点使用 **Filebeat** 将本地 access.log 推送到 **Kafka**（topic=access_log）。  
   - Kafka → **ClickHouse** (via **Kafka engine**) 实时写入。  

2. **监控指标**  
   - **Node Exporter** + **Prometheus** 抓取 CPU、内存、磁盘 I/O、网络。  
   - **Custom Exporter** 暴露 CDN 特有指标：`cdn_cache_hits_total`, `cdn_cache_misses_total`, `cdn_purge_latency_seconds`.  

3. **告警**  
   - **Alertmanager** 根据阈值（如命中率 < 80%）发送 Slack/邮件。  

4. **报表**  
   - Grafana Dashboard 展示 **实时命中率、热点 Top‑10、节点负载、错误率**。  
   - 每日/每周批处理生成 **容量规划报告**（ClickHouse + Spark）供运维参考。  

### 5.4 容灾与弹性

| 场景 | 处理方式 | 关键技术 |
|------|----------|----------|
| **单节点故障** | GSLB 自动剔除失效节点，流量切到健康节点 | **Consul** 健康检查 + **NGINX Plus** 动态权重 |
| **整座大区失效** | Anycast DNS 回退到 **第二最近** 大区的 Anycast IP；GSLB 将流量分配到跨大区节点 | **BGP Route Flap Damping** 防止瞬时回弹；**Global Failover**（TTL=60 s） |
| **突发流量** | **自动扩容**：使用 **Kubernetes + Cluster Autoscaler**，在流量超过阈值时快速启动 **新 Edge Pod**（基于容器镜像） | **Horizontal Pod Autoscaler (HPA)** + **Custom Metrics**（QPS） |
| **缓存失效** | **预热**：在高峰前 10 min 根据热点列表向新节点发起 **预取** 请求 | **CronJob** + **ClickHouse** 热点分析 |

> **不做跨大区回退**：大区全死后用户请求会全部 5xx，违背 99.99% 可用性。

## 第六步：扩展性与高可用设计

### 6.1 水平扩展（Scale‑Out）

| 层级 | 扩容方式 | 关键点 |
|------|----------|--------|
| **DNS** | 添加更多 **Anycast** 节点（多运营商） | 需要 BGP 同步，避免路由冲突 |
| **GSLB** | 将 **GSLB** 部署为 **无状态**，使用 **Consul KV** 存储状态，随意增加实例 | 通过 **Service Discovery** 动态感知节点 |
| **Edge 节点** | **容器化**（Docker）+ **K8s**，基于 **Pod** 规模扩容 | 共享 **SSD**（NVMe）通过 **CSI** 动态卷 |
| **缓存层** | **分片**：对象 hash%N → 指定 SSD 分区，水平增加 SSD 磁盘 | 需要 **一致性哈希** 保证对象定位不变 |
| **存储** | **对象存储**（S3）本身是无限弹性，Edge 只需要 **读** 权限 | 多 AZ 跨区复制确保源站高可用 |

### 6.2 高可用（HA）细节

1. **数据层（MySQL）**  
   - 主从复制 + **GTID**，自动故障转移（**MHA**）  
   - **读写分离**：业务配置查询走从库，写操作走主库。  

2. **Redis / KV**  
   - **Cluster 模式** + **哨兵**（Sentinel）监控故障转移。  
   - 使用 **持久化**（AOF）防止重启数据丢失。  

3. **Kafka**  
   - **多副本（replication factor=3）**，**ISR** 保证消息不丢失。  

4. **Edge 节点**  
   - **无状态**：所有状态保存在 KV/DB，节点可以随时下线/上线。  
   - **滚动升级**：使用 **Blue/Green** 部署，新版本先在小流量节点上跑，确认无误后全量切换。  

5. **网络**  
   - **Anycast** + **BGP 多路径**，即使单个 ISP 故障，流量会自动切换到其他路径。  

### 6.3 性能优化技巧

| 目标 | 手段 | 效果 |
|------|------|------|
| **降低网络 RTT** | **QUIC** (UDP+TLS) 替代 TCP；在 Edge 直接缓存 TLS 会话票据（Session Ticket） | 约 10‑15 ms RTT 降低 |
| **提升磁盘读写** | **NVMe SSD** + **Direct I/O** + **分块对齐**（4 KB） | SSD 随机读 300 kIOPS，满足 20 GB/s 峰值 |
| **减少 CPU 负载** | **Byte‑range 过滤**（只返回需要的 Range），**压缩**（Brotli） | 同时服务更多 QPS，降低带宽费用 |
| **防止缓存击穿** | **单飞**（single flight）+ **热点热点锁**（分布式锁） | 同一对象只请求一次回源，降低 Origin 压力 |
| **提升命中率** | **热点预取** + **机器学习预测**（基于过去 24 h） | 命中率提升 3‑5%（对应数十 TB 流量） |

## 第七步：常见面试追问与回答

| 追问 | 思考点 | 参考答案 |
|------|--------|----------|
| **Q1：如果热点内容超过 150 TB，缓存容量怎么办？** | 需要 **多层缓存** 与 **预取** 机制。 | ① **热层**（LFU）只保留最近 10 %（≈ 15 TB）最热对象；② **冷层**（LRU）使用剩余 135 TB；③ **跨区域预取**：热点在多个大区复制，提升局部命中率；④ **热点预测**（基于 ClickHouse 分析）提前把热点写入热层。 |
| **Q2：大区节点全失效时，如何做到用户透明切换？** | **Anycast + DNS 回退** 与 **GSLB** 双重机制。 | 1）Anycast IP 在全局路由层已布置多条路径；2）DNS TTL 设为 60 s，若用户查询到失效大区的 IP，路由会因 BGP 收敛自动切换到第二最近的 Anycast；3）GSLB 检测到节点不可用后，立即把流量调度到跨大区可用节点。 |
| **Q3：突发活动（如世界杯）时，如何快速扩容 Edge 节点？** | **容器化 + 自动弹性伸缩**。 | 在 K8s 中预置 **NodePool**（GPU/高频 SSD），配置 **HPA** 基于 **QPS**、**CPU** 指标。活动前 5 min 开启 **预热脚本**，将热点对象预先 **pre‑warm** 到新节点。 |
| **Q4：缓存失效与 CDN 失效的区别？** | **缓存失效** 是 **对象层面**（如 PURGE），**CDN 失效** 是 **节点/服务不可用**（如节点宕机）。 | 缓存失效通过 **PURGE API**，只删除对应对象；节点失效触发 **GSLB 健康检查**，把流量切走。 |
| **Q5：如果每个对象大小分布不均，如何避免大文件占满 SSD？** | **分层存储** + **对象分片**。 | 大文件（> 10 MB）采用 **分块存储**（每块 4 MB），只缓存最近访问的块；使用 **对象大小阈值**（如 > 100 MB）直接回源不缓存。 |
| **Q6：为什么不把缓存直接放在数据库里？** | **性能与成本**。 | 数据库（MySQL）写入延迟 ms 级，吞吐不够；且 **SSD** 直接提供高并发随机读写，更适合大对象缓存。 |

## 心得与反思

### 1. 本题最难的 1‑2 个设计决策及思考过程

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **① 边缘缓存淘汰策略（冷热分层 + LFU/LRU）** | - 需要兼顾 **命中率** 与 **存储成本**。<br>- 不同业务对 **TTL**、**实时性** 要求差异大。 | 1）先估算热点对象占比（约 10 % 热点占 30 % 流量）。<br>2）决定把 **10 % 容量** 用 **LFU**，因为 LFU 更适合长期热点。<br>3）其余 90 % 用 **LRU**，实现简单且对短期热点友好。<br>4）实现时采用 **近似 LFU**（计数衰减）防止计数溢出。 |
| **② 全局路由（Anycast + DNS + GSLB）** | - 需要 **多层次** 的路由才能兼顾 **跨洲延迟** 与 **单大区容灾**。<br>- Anycast 本身对 BGP 依赖，调试复杂。 | 1）先确认目标 **RTT ≤ 150 ms**，单靠 DNS 难以保证。<br>2）引入 **Anycast**，把同一 IP 广播到所有 PoP，路由层面选最近节点。<br>3）Anycast 失效时仍可能选到故障节点，故再加 **GSLB** 在大区内部做健康检查与流量分配。<br>4）最终形成 **三层**：DNS（全球），Anycast（跨洲），GSLB（大区内部）。 |

### 2. 新手最容易犯的错误（至少 2 条）

| 错误 | 结果 | 正确做法 |
|------|------|----------|
| **① 把 CDN 设计成“单点”**：只部署一个中心节点或只靠 DNS 直接指向源站。 | 高延迟、单点故障、无法满足 99.99% 可用。 | 必须 **多节点**、**Anycast**、**GSLB**，实现真正的分布式。 |
| **② 缓存层只用 **LRU**，不考虑热点对象大小/访问频次。 | 热点对象被频繁淘汰，整体命中率低，尤其在 150 TB 容量限制下。 | 采用 **冷热分层**（LFU + LRU）或 **分层缓存**，并配合 **预取** 与 **TTL** 策略。 |
| **③ 监控/日志只在边缘节点本地保存** | 失去全局视角，无法快速定位热点或故障，运维成本高。 | 集中 **日志/监控**（Kafka → ClickHouse / Prometheus），并提供实时 Dashboard。 |
| **④ 缓存刷新同步使用 **同步 HTTP**，导致源站请求阻塞。 | 大批量刷新时源站响应慢，甚至超时。 | 使用 **异步消息队列（Kafka）**，边缘节点 **订阅** 刷新事件，做到 **脱耦**。 |
