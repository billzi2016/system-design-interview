# 系统设计面试题：Instagram

## 1. 题目背景
Instagram 是一款以图片/短视频为核心的社交媒体平台，用户可以通过上传、浏览、点赞、评论等方式进行内容分享和社交互动。系统需要支撑海量用户的实时内容分发和交互。

## 2. 面试场景设定
> **面试官**：  
> “我们现在要设计一个类似 Instagram 的平台，请从高层架构出发，说明你会如何实现核心的图片/视频发布、动态流（Feed）以及用户交互功能。重点关注系统的可扩展性、延迟和存储成本。”

## 3. 功能性需求
| 编号 | 需求描述 |
|------|----------|
| 1 | **用户注册/登录**：支持邮箱、手机号和第三方 OAuth 登录。 |
| 2 | **内容发布**：用户可以上传图片或 60 秒以内的短视频，支持多图/多视频（最多 10 条）并可添加文字、话题标签。 |
| 3 | **动态流（Feed）**：为每个用户展示关注用户的最新内容，按时间和个性化算法排序，支持无限下拉加载。 |
| 4 | **点赞 & 评论**：对任意内容进行点赞、发表评论，支持评论的回复与删除。 |
| 5 | **关注/取关**：用户可以关注或取消关注其他用户，系统实时更新对应的 Feed。 |
| 6 | **内容搜索**：按用户、标签、地点搜索公开的图片/视频。 |

## 4. 非功能性需求（带估算）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 3000 万 | 假设业务在全球范围内运营，目标覆盖 3% 市场渗透。 |
| **QPS（请求峰值）** | 150,000 QPS | 峰值时间段（欧美晚间）约为日均请求的 1.5 倍，主要集中在 Feed 拉取、点赞、评论等交互。 |
| **平均响应时延** | < 200 ms（95% 请求） | Feed 拉取、点赞、评论等交互需保持低延迟，以提升用户体验。 |
| **可用性** | ≥ 99.9%（每月累计停机 ≤ 43.2 分钟） | 关键业务（登录、Feed、内容发布）需高可用。 |
| **存储容量** | 约 150 PB | 按 30 % 活跃用户每日上传 5 MB 图片或 20 MB 视频计算，保留 2 年原始文件 + 多级缓存。 |
| **数据一致性** | 最终一致性（如点赞计数） | 对于社交交互的计数可接受短暂的延迟同步。 |

## 5. 系统边界
### 本题范围（需要设计）
- 用户身份验证与会话管理  
- 内容上传、存储（对象存储 + CDN）  
- Feed 流的生成、排序、分页  
- 点赞、评论的写入与查询  
- 关注关系的维护与实时更新  
- 基础的搜索（用户、标签）  

### 不在本题范围（可不考虑细节）
- 短视频转码与多码率适配  
- 广告投放系统  
- 直播、IGTV、Reels 等长视频功能  
- 私信、群聊、Story（24 小时阅后即焚）  
- 机器学习模型的离线训练与特征工程（只需说明高层思路）  

## 6. 提示与追问
1. **缓存策略**：如果让你设计 Feed 的缓存层，你会如何划分热点与冷数据？采用哪些缓存失效或预热机制？  
2. **数据分区**：在用户关系（关注/粉丝）和内容存储上，你会如何进行分库分表或分片？如何保证高并发写入时的负载均衡？  
3. **扩展性**：面对突发的热点事件（如明星发布热门视频），系统如何快速扩容以防止热点压垮单点？请说明你的弹性伸缩方案。

# 题解

# Instagram 系统设计完整解答

> **温馨提示**：本篇答案把 “从零开始 → 逐步进化 → 高可用” 这条思路写得很细，适合 **完全没有系统设计经验的同学**。阅读时请把每一小节当作一块拼图，先弄清楚 **“要干什么”，再弄清楚 “怎么干”，最后考虑 “如果坏了怎么恢复”。  

---

## ## 解题思路总览  

1. **先把需求拆解成最小可运行的系统**（MVP），只保留最核心的功能：用户登录、内容上传、Feed 拉取、点赞/评论、关注。  
2. **估算流量 & 存储**，把业务规模量化，帮助后面选技术栈、分片方式。  
3. **画出高层架构图**，把请求的路径、主要组件标出来。  
4. **细化每个子系统的内部实现**：数据库模型、缓存、消息队列、搜索引擎等。  
5. **逐步加入容错、扩容、弹性**：冗余、读写分离、分片、限流、异步处理、CDN 等。  
6. **准备面试常见追问**：缓存失效、热点写入、数据一致性、监控告警等。  

> **为什么要这样做？**  
> - 先做 MVP 能帮助面试官看到你能把大系统拆成可实现的小块，避免“一开始就把所有细节堆到一起”导致结构混乱。  
> - 量化后再选技术，避免盲目“选最流行的”，而是“选最匹配业务规模的”。  
> - 高层图帮助面试官快速定位你的思路，细节再逐层展开。

## ## 第一步：理解需求与规模估算  

### 1. 功能需求要点（重点复述）

| 编号 | 功能 | 关键点 |
|------|------|--------|
| 1 | 注册/登录 | 多种方式（邮箱、手机号、OAuth），需要安全的会话管理 |
| 2 | 内容发布 | 图片/≤60s 视频，最多 10 条，文字 + 标签 |
| 3 | Feed | 关注用户的最新内容，时间+个性化排序，支持无限下拉 |
| 4 | 点赞/评论 | 支持回复、删除，计数要求最终一致 |
| 5 | 关注/取关 | 实时影响 Feed |
| 6 | 搜索 | 按用户、标签、地点搜索公开内容 |

### 2. 非功能需求 & 业务假设  

| 指标 | 目标值 | 业务假设 |
|------|--------|----------|
| DAU | 30 M | 全球 3% 市场渗透 |
| 峰值 QPS | 150 k | 1.5×日均峰值，主要集中在 Feed 拉取、点赞 |
| 响应时延 | <200 ms (95%) | 前端交互体验 |
| 可用性 | ≥99.9% | 关键服务高可用 |
| 存储 | ~150 PB | 30% 活跃用户每日 5 MB（图）或 20 MB（短视频），保留 2 年 |
| 一致性 | 最终一致 | 点赞/评论计数可容忍几秒延迟 |

> **估算示例**（帮助后续容量规划）  

- **每日上传量**  
  - 活跃用户 = 30 M × 30% = 9 M  
  - 假设 70% 上传图片，30% 上传视频  
  - 图片日均 = 9 M × 70% × 5 MB ≈ **31.5 TB**  
  - 视频日均 = 9 M × 30% × 20 MB ≈ **54 TB**  
  - **合计 ≈ 85 TB / 天** → **≈ 31 PB / 年**，两年保留约 **62 PB**，再加冗余、缓存、备份，≈ 150 PB（与题目给的数字相符）。  

- **Feed 拉取 QPS**  
  - 假设每位活跃用户平均每分钟拉一次 Feed（滚动加载），则 QPS ≈ 30 M / 60 ≈ **500 k**，但实际只有 30% 用户高频使用 → **≈150 k QPS**，符合需求。  

- **点赞/评论写入 QPS**  
  - 假设每位活跃用户平均 0.2 次点赞、0.05 次评论/分钟 → 30 M × (0.2+0.05)/60 ≈ **125 k QPS**。

## ## 第二步：高层架构设计  

下面给出 **从 MVP 到完整系统** 的逐层演进图。为方便阅读，使用 **ASCII 图**，实际面试可手绘或用 PPT。

### 2.1 最小可用系统（MVP）

```
+-------------------+          +-------------------+          +-------------------+
|   客户端 (App)    |  <--->   |   API Gateway     |  <--->   |   应用服务器 (Web)|
+-------------------+          +-------------------+          +-------------------+
                                        |
                                        v
                               +-------------------+
                               |   Auth Service    |
                               +-------------------+
                                        |
                                        v
                               +-------------------+
                               |   Media Service   |
                               +-------------------+
                                        |
                                        v
                               +-------------------+
                               |   Feed Service    |
                               +-------------------+
```

**说明**  

- **API Gateway**：统一入口，做路由、限流、鉴权（JWT）等。  
- **Web Service**（单体或少量实例）：处理注册、登录、发布、点赞、评论、关注等业务逻辑。  
- **Auth Service**：负责用户信息、密码、Token。  
- **Media Service**：直接把文件写入对象存储（如 S3）并返回 URL。  
- **Feed Service**：查询关注列表，从 DB 拉取最新帖子，返回给前端。  

**为什么先这么做？**  

- **实现最快**：只需要几台机器，代码逻辑清晰。  
- **验证业务模型**：先确认 Feed 排序、点赞计数等核心功能是否可行。  
- **缺点**：**单点瓶颈**（所有请求都走同一个 Web Service），**不具备弹性**，**数据一致性只能靠单库事务**，不适应大规模。  

### 2.2 逐步加入关键中间件

```
+-------------------+          +-------------------+          +-------------------+
|   客户端 (App)    |  <--->   |   API Gateway     |  <--->   |   Load Balancer   |
+-------------------+          +-------------------+          +-------------------+
                                        |
          +-----------------------------+------------------------------+
          |                            |                              |
          v                            v                              v
+-------------------+      +-------------------+          +-------------------+
|   Auth Service    |      |   Media Service   |          |   Feed Service    |
+-------------------+      +-------------------+          +-------------------+
          |                            |                              |
          v                            v                              v
+-------------------+      +-------------------+          +-------------------+
|  User DB (RDB)    |      |   Object Store    |          |   Feed DB (NoSQL) |
+-------------------+      +-------------------+          +-------------------+
```

**新增组件**  

- **Load Balancer**：水平扩展 Web 服务。  
- **对象存储（Object Store）**：如 Amazon S3、Aliyun OSS，提供高可用、分布式存储。  
- **Feed DB**：使用 **分布式 NoSQL**（如 Cassandra、ScyllaDB）做 **时间线**，读写分离、支持高并发。  

**为什么加入这些？**  

- **对象存储** 解决大文件（图片/视频）的持久化与 CDN 加速。  
- **NoSQL** 能在 **写入**（新帖子）和 **读取**（Feed）之间提供 **高吞吐**，并且天然支持 **水平扩展**。  
- **Load Balancer** 让我们可以 **弹性伸缩** Web 实例，满足峰值 QPS。  

### 2.3 完整高可用、可伸缩系统（目标）  

```
                               +-------------------+
                               |   CDN (Edge)      |
                               +-------------------+
                                        ^
                                        |
+-------------------+      +-------------------+      +-------------------+
|   客户端 (App)    |<---->|   API Gateway     |<---->|   WAF / DDoS防护   |
+-------------------+      +-------------------+      +-------------------+
                                        |
                +-----------------------+-----------------------+
                |                       |                       |
                v                       v                       v
        +---------------+        +---------------+      +---------------+
        | Auth Service  |        | Media Service |      | Feed Service  |
        +---------------+        +---------------+      +---------------+
                |                       |                       |
        +-------+-------+       +-------+-------+       +-------+-------+
        |   User DB (RDB) |       |   Object Store  |       |   Feed DB (NoSQL) |
        +-----------------+       +-----------------+       +-------------------+
                |                       |                       |
                |                       |                       |
                v                       v                       v
        +---------------+        +---------------+      +---------------+
        |  Cache (Redis)|        |  CDN Edge Cache|      |  Cache (Redis)|
        +---------------+        +---------------+      +---------------+
                |                       |                       |
                v                       v                       v
        +---------------+        +---------------+      +---------------+
        |  Message Queue|<------->|  Async Workers|<---->|  Search Service|
        +---------------+        +---------------+      +---------------+
```

**关键组件说明**  

| 组件 | 作用 | 关键技术/实现 |
|------|------|---------------|
| **API Gateway** | 统一入口、鉴权、限流、灰度发布 | Kong / Zuul / Envoy |
| **WAF / DDoS 防护** | 防止恶意流量、网络攻击 | Cloudflare / AWS WAF |
| **Auth Service + User DB** | 用户信息、密码、OAuth、Token | MySQL / PostgreSQL + 读写分离 + 主从复制 |
| **Media Service + Object Store** | 文件上传、转码（可选）、CDN 分发 | OSS / S3 + multipart upload + CloudFront/OSS CDN |
| **Feed Service + Feed DB** | 生成 & 查询用户时间线 | Cassandra / ScyllaDB（按用户分区） |
| **Cache (Redis)** | 热门 Feed、点赞计数、关注列表的缓存 | Redis Cluster |
| **Message Queue** | 解耦异步任务（生成时间线、计数、搜索索引） | Kafka / Pulsar |
| **Async Workers** | 消费 MQ，完成 Feed 写入、计数累加、搜索同步 | Go / Java / Python |
| **Search Service** | 基于标签、地点、用户的全文检索 | Elasticsearch + OpenSearch |
| **监控 & 链路追踪** | Prometheus + Grafana、Jaeger、Alertmanager | 统一监控平台 |

**为什么这样分层？**  

1. **解耦**：发布→写入时间线、写入计数、写入搜索索引均通过 MQ 异步完成，互不影响。  
2. **弹性**：每个子系统可独立水平扩容，热点只需在对应层面加机器。  
3. **高可用**：关键数据（User DB、Feed DB、Object Store）均采用多副本、跨 AZ 部署。  
4. **低延迟**：热点 Feed、点赞计数放在 Redis 缓存，读写均在内存完成，满足 <200 ms。

## ## 第三步：数据库设计  

### 1. 关系型数据库（User DB）  

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| **users** | `user_id (BIGINT PK)` | `email, phone, password_hash, oauth_provider, created_at, status` | 基础用户信息 |
| **user_profile** | `user_id PK` | `username, bio, avatar_url, gender, birthdate, location` | 公开资料 |
| **followers** | `user_id PK, follower_id PK` | `followed_at` | **关注关系**，采用 **双向存储**（分别保存关注者列表 & 被关注者列表） |
| **sessions** | `session_id PK` | `user_id, jwt_token, expires_at, device_info` | 登录态管理（可选） |

**分库分表**  

- **users**、**user_profile** 按 **user_id** 范围分库（如 1000 万/库），再在库内水平分表（hash）。  
- **followers** 采用 **两张表**：`user_following`（存某用户关注的对象）和 `user_followers`（存某用户的粉丝），均 **按 user_id** 分库分表，保证 **查询关注列表** 高效。  

### 2. 对象存储（Media）  

- **目录结构（示例）**：`/users/{user_id}/{yyyyMMdd}/{media_id}.{ext}`  
- **元数据**（存入 **Media DB**，可用 NoSQL）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `media_id` | STRING PK | 唯一标识（UUID） |
| `user_id` | BIGINT | 所属用户 |
| `type` | ENUM(`image`,`video`) | 媒体类型 |
| `url` | STRING | OSS / S3 公网 URL（带签名） |
| `size_bytes` | BIGINT | 大小 |
| `duration_ms` | INT | 视频时长（图片为 0） |
| `thumbnail_url` | STRING | 视频封面 |
| `tags` | ARRAY<STRING> | 关联的话题标签 |
| `location` | GEOPOINT | 可选位置信息 |
| `created_at` | TIMESTAMP | 上传时间 |
| `status` | ENUM(`active`,`deleted`) | 软删标记 |

### 3. 时间线（Feed DB）  

采用 **Cassandra**（或 ScyllaDB） 的 **宽列模型**：

| 主键（复合） | 列族 | 说明 |
|--------------|------|------|
| **partition key**: `user_id` |  | **每个用户的 Feed** 存在同一分区 |
| **clustering key**: `post_time DESC` |  | 按时间倒序排列，支持分页 |
| **列**: `post_id`, `media_ids (list)`, `caption`, `tags`, `like_count`, `comment_count`, `is_deleted` |  | 具体帖子信息（复制自 Media DB） |

**写入方式**：当用户 **A** 发布新帖，系统会将 **A 的粉丝列表**（可能上万）写入 **每个粉丝的分区**（fanout 写入）。  

**热点处理**：  
- 对 **超大粉丝量（>10k）** 的明星账号采用 **“拉取模型 + Fanout on Read”**（后面详细说明）。  
- 普通用户使用 **“Fanout on Write”**（写入粉丝时间线），因为粉丝数相对可接受。  

### 4. 点赞 & 评论计数（Redis + DB）  

| 结构 | 说明 |
|------|------|
| **Redis Hash** `post:like:{post_id}` | `user_id -> timestamp`，用于快速判断用户是否已点赞、展示点赞列表（前 N） |
| **Redis String** `post:like_cnt:{post_id}` | 实时计数（原子 INCR） |
| **MySQL 表** `post_likes` | 持久化，定期（如每小时）从 Redis 持久化到 MySQL，支持全量统计 |
| **MySQL 表** `post_comments` | `comment_id PK`, `post_id FK`, `user_id`, `parent_comment_id`, `content`, `created_at`, `status` |

> **为什么使用 Redis + DB 双写？**  
> - **Redis** 提供 **毫秒级** 读取/写入，满足点赞交互的低延迟要求。  
> - **MySQL** 负责 **持久化**，防止 Redis 故障导致数据丢失。通过 **异步落库**（Kafka）实现最终一致。  

### 5. 搜索索引（Elasticsearch）  

- **文档结构**：`post_id, user_id, caption, tags, location, created_at, media_type`  
- **分词**：中文使用 IK Analyzer，英文使用 standard analyzer。  
- **倒排索引**：支持标签、关键词、地理位置范围查询。  

> **为何单独建搜索引擎？**  
> - **关系型/NoSQL** 不擅长全文检索和多字段复合查询。  
> - ES 支持 **实时同步**（通过 Logstash/Canal），并且可以水平扩容。

## ## 第四步：核心 API 设计  

下面列出 **最常用的 10+ API**，包括请求/响应示例（JSON），并说明背后涉及的子系统。

> **约定**：所有 API 均使用 **HTTPS + JWT** 进行鉴权，返回统一的 `{code, message, data}` 结构。

| # | API | 方法 | URL | 功能 | 关键子系统 |
|---|-----|------|-----|------|------------|
| 1 | **注册** | POST | `/api/v1/auth/register` | 邮箱/手机号注册，发送验证码 | Auth Service、User DB |
| 2 | **登录** | POST | `/api/v1/auth/login` | 账号密码或 OAuth，返回 JWT | Auth Service、User DB |
| 3 | **上传媒体** | POST (multipart) | `/api/v1/media/upload` | 多文件上传，返回 media_id 列表 | Media Service、Object Store、Media DB |
| 4 | **发布帖子** | POST | `/api/v1/posts` | 关联 media_id、文字、标签，写入 Feed DB、推送 MQ | Feed Service、Message Queue |
| 5 | **获取 Feed** | GET | `/api/v1/feed?cursor=...&limit=20` | 分页读取用户时间线，先读缓存 → DB | Feed Service、Redis Cache、Feed DB |
| 6 | **点赞** | POST | `/api/v1/posts/{post_id}/like` | 增加点赞计数，写入 Redis，异步落库 | Like Service、Redis、Message Queue |
| 7 | **取消点赞** | DELETE | `/api/v1/posts/{post_id}/like` | 删除点赞，计数-1 | 同上 |
| 8 | **发表评论** | POST | `/api/v1/posts/{post_id}/comments` | 写入 MySQL，返回 comment_id | Comment Service、MySQL |
| 9 | **获取评论列表** | GET | `/api/v1/posts/{post_id}/comments?cursor=...&limit=20` | 支持分页、层级（parent_id） | Comment Service、MySQL、Redis 缓存 |
|10| **关注用户** | POST | `/api/v1/users/{target_id}/follow` | 写入 followers 表，触发 Feed 重算 | Follow Service、MySQL、Message Queue |
|11| **取消关注** | DELETE | `/api/v1/users/{target_id}/unfollow` | 同上 |
|12| **搜索** | GET | `/api/v1/search?type=post&keyword=...&tag=...&location=...` | 调用 ES，返回匹配 post_id 列表 | Search Service、Elasticsearch |

### 示例：发布帖子（核心流程）

```json
POST /api/v1/posts
{
  "media_ids": ["m123","m124"],
  "caption": "在海边的午后 #旅行 #海风",
  "tags": ["旅行","海风"],
  "location": {"lat": 31.2304, "lon": 121.4737}
}
```

**后端处理步骤**  

1. **鉴权**：API Gateway 解析 JWT，获取 `user_id`。  
2. **写入 MySQL（Posts 表）**：生成 `post_id`，保存元数据（事务）。  
3. **写入 Feed DB**（Fanout on Write）：
   - 从 **followers** 表读取 `user_id` 的粉丝列表（或使用缓存）。  
   - 对每个粉丝 `fid`，向 **Feed DB** 的分区 `fid` 插入一行（异步通过 Kafka）。  
4. **推送 MQ**：`post_created` 事件 → **Async Workers**：
   - 更新 **Redis** 缓存（热点 Feed）。  
   - 同步到 **Elasticsearch**（搜索）。  
5. **返回**：`{code:0, data:{post_id:"p987"}}`。  

> **如果不走异步**，写入过程会阻塞用户，导致 **响应时间 > 500 ms**，无法满足 <200 ms 的目标。

## ## 第五步：详细组件设计  

### 1. **用户身份认证 & 会话管理**  

| 需求 | 设计 |
|------|------|
| **多方式登录** | - **邮箱/手机号**：发送一次性验证码（SMS / Email）<br>- **OAuth**：使用 Facebook/Google 登录，获取第三方 `access_token`，后端再换取用户信息并生成本系统的 JWT。 |
| **Token** | 使用 **JWT（HS256）**，包含 `user_id、exp、iat`。<br>Token 失效时间 24h，刷新 token 采用 `refresh_token`（存于 Redis，绑定设备）。 |
| **安全** | - 密码使用 **bcrypt** + `salt` 存储。<br>- 登录失败计数+验证码防暴力。<br>- 支持 **多因素认证（MFA）**（可选）。 |
| **会话失效** | 在 **Redis** 中维护 `token:blacklist`，用户注销或密码修改时将旧 token 加入黑名单，TTL 与 token 失效时间相同。 |

### 2. **内容上传 & 存储**  

1. **前端**：先向 **/media/upload** 发起 **分块（multipart）** 上传，获取 **uploadId**，每块成功后返回 **etag**，最后完成上传。  
2. **后端**：  
   - **Media Service** 负责向对象存储发起 **multipart upload**（OSS SDK），返回 **预签名 URL** 给前端直接上传（减轻服务带宽压力）。  
   - 上传成功后，Media Service 写入 **Media DB**（NoSQL），记录 **media_id、url、metadata**。  
3. **CDN**：对象存储配置 **全局加速**，Edge 节点缓存图片/视频，降低回源延迟。  
4. **视频转码**（如有）放在 **异步 Worker**，转码完成后更新 **thumbnail_url** 与 **多码率 URL**。  

### 3. **Feed 生成策略**  

#### 3.1 Fanout on Write（写时推送）  

- **适用对象**：普通用户（粉丝数 < 10k）。  
- **流程**：  
  1. 新帖子写入 **Post 表**。  
  2. 从 **followers** 表读取粉丝列表（缓存 + DB）。  
  3. 对每个粉丝写入 **Feed DB**（Cassandra）对应分区。  
  4. 写入成功后返回。  

- **优点**：用户读取 Feed 时仅一次 DB 查询（快速），写入成本可接受。  
- **缺点**：**写放大**（一条帖子可能写入上万行），对大 V（明星）不适用。  

#### 3.2 Fanout on Read（读时推送）  

- **适用对象**：粉丝数极大（>10k）的明星或热点账号。  
- **实现**：  
  - **不把明星的帖子写入每个粉丝的时间线**。  
  - 当粉丝请求 Feed 时，**从明星的个人时间线**（单独表 `user_posts_{star_id}`）读取最近 N 条，**合并**到用户自己的时间线后排序返回。  
  - 为了避免每次都做全表扫描，使用 **Redis Sorted Set** 缓存明星最近 100 条帖子，读取时直接取前 N。  

- **优点**：写入成本 O(1)（只写入自己表），读时稍多一步合并。  
- **缺点**：读时需要额外的 **合并排序**，但通过 **Redis** 缓存可以把延迟控制在 50 ms 以内。  

#### 3.3 统一 Feed Service 接口  

```go
func GetUserFeed(userID int64, cursor string, limit int) ([]Post, string, error) {
    // 1. 先查 Redis Cache（user:{id}:feed）是否命中
    // 2. 未命中 → 从 Feed DB（Cassandra）查询分页
    // 3. 同时读取热点明星的 SortedSet（Redis）并合并
    // 4. 排序后返回，并写回 Cache（LRU+TTL）
}
```

### 4. **点赞、评论计数的最终一致性**  

- **写路径**：  
  1. 客户端请求点赞 → API Gateway → Like Service → **Redis INCR** `post:like_cnt:{post_id}`。  
  2. 同时向 **Kafka** 发送 `like_event`（包含 `post_id, user_id, type(add/remove)`）。  
- **异步落库**：Worker 消费 `like_event`，更新 **MySQL** `post_likes` 表并累计到 `post_like_cnt`（聚合）。  
- **读取**：前端展示点赞数时直接读 **Redis**，若 Redis 故障则回源 MySQL（延迟几秒）。  

> **为什么采用这种方式？**  
> - **Redis** 的原子操作保证 **毫秒级** 计数，满足交互需求。  
> - **Kafka + Worker** 将 **业务数据持久化**，防止 Redis 故障导致永久丢失。  

### 5. **关注关系的实时更新**  

- **写入**：`POST /users/{target}/follow` → **Follow Service** → 写入 **followers** 表（双向），并向 **Kafka** 发送 `follow_event`。  
- **异步处理**：Worker 根据 `follow_event`  
  - 为 **新粉丝** 拉取 **target** 最近 N 条帖子（从 `user_posts_target` 表），写入该粉丝的 Feed 分区（一次性批量写入）。  
  - 同时更新 **Redis** 中的 `user:{fid}:following` 与 `user:{tid}:followers` 集合（用于快速检查是否已关注）。  
- **取消关注**：相似流程，删除对应记录并 **标记** 已删除的 Feed 条目（软删），读时过滤。  

### 6. **搜索实现**  

1. **写同步**：  
   - 每当 **Post** 创建或 **Tag** 更新时，Post Service 将 **事件** 发送至 **Kafka**。  
   - **ES Ingest Worker** 消费，构造文档并写入 **Elasticsearch**。  
2. **查询**：  
   - `/search` API 调用 **Elasticsearch DSL**，支持：  
     - **全文**（caption、tags）  
     - **过滤**（用户、时间范围、地理位置）  
     - **排序**（相关度、最新）  
3. **热点缓存**：热门搜索关键词的结果放入 **Redis**（TTL 5 min），进一步降低 ES 负载。  

### 7. **监控、日志、告警**  

| 监控维度 | 关键指标 | 工具 |
|----------|----------|------|
| **系统** | CPU、Mem、Disk I/O、网络流量 | Prometheus + Node Exporter |
| **服务** | QPS、RT (p95)、错误率 | Grafana + Prometheus |
| **业务** | 发布成功率、Feed 拉取成功率、点赞计数偏差 | 自定义仪表盘 + Alertmanager |
| **链路** | 请求追踪（从 API Gateway 到 DB） | Jaeger / Zipkin |
| **日志** | 结构化日志（JSON），集中收集 | ELK Stack (Filebeat + Logstash + Kibana) |
| **告警** | CPU>80% 连续 5min、RT>200ms、服务不可用 | Alertmanager → Slack/Email |

## ## 第六步：扩展性与高可用设计  

### 1. **水平扩容（弹性伸缩）**  

| 场景 | 自动伸缩方案 |
|------|--------------|
| **Web/API** | 基于 **Kubernetes HPA**（根据 CPU/QPS）或 **AWS Auto Scaling Group**（根据网络流量） |
| **Redis** | **Redis Cluster**，自动分片；节点数可动态增加，使用 **proxy**（Twemproxy）实现透明路由 |
| **Cassandra** | 添加节点后自动 **重新平衡**（Rebalance），支持线性扩容 |
| **Kafka** | 增加 **Broker**，分区数提前预留，使用 **Kafka Cruise Control** 自动负载均衡 |
| **对象存储** | 本身是 **分布式**，无需手动扩容，只需监控 **容量阈值**，开启 **跨区域复制** |

### 2. **热点事件的应急方案**  

- **明星发布热门视频** 时，**写放大** 会导致 `followers` 表大量 Fanout。  
- **应对措施**：  
  1. **预判**：在 `follow_event` 中记录 **粉丝数**，若超过阈值（如 50k），系统自动切换该用户至 **Fanout on Read** 模式。  
  2. **缓存热点**：将该用户最近 500 条帖子放入 **Redis Sorted Set**，并在 Feed 读取时直接合并。  
  3. **限流**：对 **发布接口** 设置 **QPS** 限流，防止瞬时写入洪峰导致后端崩溃。  
  4. **弹性队列**：Kafka 分区数提前预留（如 500 分区），确保热点事件的消息能分散到多个 broker。  

### 3. **灾备与容错**  

| 组件 | 容灾方案 |
|------|----------|
| **User DB (MySQL)** | 主从双活（跨 AZ），GTID 同步，自动故障转移（MHA） |
| **Feed DB (Cassandra)** | 每个键空间（Keyspace）设置 **NetworkTopologyStrategy**，副本因子 3，跨三可用区 |
| **Redis** | 主从复制 + Sentinel 自动故障转移，或 **Cluster** 多主 |
| **Kafka** | 每个分区副本数 3，使用 **ISR**（In-Sync Replicas），Broker 失效时 Leader 自动迁移 |
| **对象存储** | 多 AZ 多 Region 同步复制（OSS/OSS 跨地域复制） |
| **CDN** | 边缘节点自动回源，故障时切换至最近可用节点 |
| **API Gateway** | 多实例部署，使用 **DNS 轮询 + 健康检查** 或 **Global Load Balancer**（如 Cloudflare） |

### 4. **数据一致性策略**  

| 场景 | 一致性要求 | 采用方案 |
|------|------------|----------|
| **用户注册/登录** | 强一致性 | 主库同步写入，读写分离但读取主库（或使用 **Read-After-Write**） |
| **帖子发布** | **最终一致性**（短暂延迟可接受） | 写入 MySQL → 通过 MQ 异步写入 Feed DB、ES |
| **点赞计数** | **最终一致性**（几秒误差可接受） | Redis 原子计数 → 异步落库 |
| **关注关系** | **强一致性**（用户必须立即看到关注结果） | 写入 MySQL 并同步到 Redis 缓存，读时优先查询 Redis |
| **搜索索引** | **最终一致性** | 写入 ES 异步，搜索时可能出现 1‑2 秒延迟 |

> **为什么不统一强一致？**  
> - **强一致** 需要 **分布式事务**，在跨数据中心的高并发场景下会显著增加 **延迟**，不符合 200 ms 的交互要求。  
> - **最终一致** 能让我们利用 **异步流水线**（Kafka + Worker）实现 **高吞吐** 与 **弹性**，同时在用户侧感知的差异几乎为零。

## ## 第七步：常见面试追问与回答  

| 追问 | 参考答案 |
|------|----------|
| **1. Feed 缓存如何划分热点与冷数据？** | - **热点**：活跃用户（DAU 前 10%）的 Feed 采用 **Redis LRU Cache**（TTL 5 min），并预先 **预热**（每日凌晨批量读取最新 N 条写入）。<br>- **冷数据**：不活跃用户直接查询 **Cassandra**，并在返回后写入 **Cache**（写回策略）。<br>- **失效**：TTL 到期后自动失效；用户主动刷新时强制 **Cache Miss**。 |
| **2. 如何保证用户关注/取关的实时性？** | - 关注操作写入 **MySQL** + **Redis Set**（`user:{uid}:following`、`user:{uid}:followers`），写完即返回成功。<br>- 异步 **Worker** 将最近 N 条目标用户的帖子写入新粉丝的 Feed（批量写入），如果目标是热点用户则使用 **Fanout on Read**，不必立即写入所有粉丝。 |
| **3. 点赞计数的最终一致性会出现多少误差？** | - 在 **Redis** 中计数是 **原子 INCR**，所以即时读到的值是 **准确的**。<br>- 只要 **Redis** 正常，误差为 **0**。若 Redis 故障，系统回退读取 **MySQL** 持久计数，可能会出现 **几秒内的延迟**（因为异步落库尚未完成）。 |
| **4. 对象存储的多地域部署如何影响延迟？** | - **用户上传** 时直接向最近的 OSS 区域 **预签名 URL** 上传，避免跨地域网络。<br>- **读取** 时 CDN 自动将对象缓存至离用户最近的 Edge 节点，**读取延迟** 通常在 **30‑50 ms**。<br>- 跨地域复制是后台异步进行，不影响前端体验。 |
| **5. 如果某个分区的 Feed DB 热点爆炸，怎么办？** | - 采用 **分区键** 为 `user_id`，热点用户的 `user_id` 采用 **散列分片**（如 `user_id % N`），可以在 **写入时** 将同一用户的时间线分散到多个节点。<br>- 对于极端热点（明星），切换到 **Fanout on Read**，只在 **用户 Feed** 中保存 **指针**（明星 ID），实际内容在明星专属表中。 |
| **6. 如何做灰度发布新功能（比如新 Feed 排序算法）？** | - 在 **API Gateway** 使用 **Canary Routing**（基于用户属性或随机比例）将 **10%** 流量转到 **新版本 Service**。<br>- 通过 **Feature Flag**（如 LaunchDarkly）在代码层控制功能开关。<br>- 监控新版本的 **错误率、RT**，若异常自动回滚。 |
| **7. 什么时候使用关系型数据库，什么时候使用 NoSQL？** | - **关系型**（MySQL）：用户信息、关注关系、事务性强的数据，需要 **ACID** 保证。<br>- **NoSQL**（Cassandra、Redis）：高吞吐、写放大场景（Feed、计数、热点缓存），对 **一致性要求低**。 |
| **8. 系统的监控告警策略如何设计？** | - **SLA**：RT p95 < 200 ms，错误率 < 0.1%。<br>- **阈值**：CPU>80% 持续 5 min、QPS>峰值 1.2×、Redis miss rate > 30% 等触发 **PagerDuty** 报警。<br>- **仪表盘**：分层展示（系统、服务、业务）便于定位根因。 |
| **9. 如何处理图片/视频的非法内容审查？** | - 上传后 **异步 Worker** 调用 **内容审核服务**（如阿里云内容安全），返回审核结果。<br>- 若不合规，标记 `status=blocked`，从 Feed、搜索中剔除，同时通知用户。 |
| **10. 如何实现“无限下拉加载”而不产生重复/遗漏？** | - 使用 **基于时间的 Cursor**（`last_post_time`）或 **基于 ID 的分页**（`post_id`），每次请求返回 `next_cursor`。<br>- 前端保存已展示的 `cursor`，确保 **顺序不变**，即使有新内容插入也不会导致重复。 |

## ## 心得与反思  

### 1. 本题最难的 1-2 个设计决策  

| 决策 | 思考过程 |
|------|----------|
| **Feed 的写入策略（Fanout on Write vs Fanout on Read）** | 需要在 **写放大** 与 **读放大** 之间找到平衡。通过分析用户粉丝分布（大多数用户粉丝 < 10k，少数明星 > 100k），决定采用 **混合模型**：普通用户使用 **写时推送**（实现快速 Feed），明星使用 **读时合并**（避免写入爆炸）。这个决策牵涉到数据模型、缓存、异步任务的设计，几乎决定了后端的核心架构。 |
| **点赞计数的最终一致性实现** | 直接在 DB 中计数会导致 **锁竞争**，延迟大；全放在缓存又会有 **持久化风险**。最终采用 **Redis 原子计数 + Kafka 异步落库** 的方案，兼顾 **低延迟** 与 **数据安全**。实现时要注意 **幂等消费**、**消息顺序**（同一 post_id 的事件放同一分区），否则会出现计数不准的情况。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **把所有业务放在单体服务里**  
   - 结果是 **单点瓶颈**、**部署困难**，难以实现弹性伸缩。正确做法是 **模块化**（Auth、Media、Feed、Like 等），每个服务独立部署，使用 **API Gateway** 统一入口。  
2. **忽视缓存失效和热点写入的冲突**  
   - 例如直接把 Feed 完全缓存，忘记在用户关注/取关、发布新内容时更新缓存，导致 **脏数据**。应在业务关键路径加入 **缓存同步**（写穿/写回）或 **消息驱动的缓存刷新**。
