# 第 76 天：设计 对象存储系统（类 S3）

> 生成日期：2026-03-11

---

## 题目背景  
对象存储系统（类似 Amazon S3）是一种面向海量非结构化数据的分布式存储服务，提供统一的 HTTP/HTTPS 接口，供用户上传、下载、管理文件。它常被用于备份、媒体存储、日志归档、机器学习数据集等场景。

---

## 面试场景设定  
**面试官**：  
“今天我们来讨论设计一套类似 S3 的对象存储系统。请你从零开始，说明系统的整体架构、关键技术选型以及如何满足下面给出的功能和性能需求。可以先从高层设计说起，然后逐步细化到存储、元数据、访问控制等模块。”

---

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **对象上传（PUT）**：支持单对象最大 5 TB，分片上传（Multipart Upload），可在上传过程中暂停/恢复。 |
| 2 | **对象下载（GET）**：支持范围读取（Range Requests），以及通过预签名 URL 实现临时访问。 |
| 3 | **对象删除（DELETE）**：支持单对象删除、批量删除以及版本化删除（保留历史版本）。 |
| 4 | **桶（Bucket）管理**：创建/删除桶，配置访问控制列表（ACL）和策略（Policy），以及跨区域复制（CRR）开关。 |
| 5 | **元数据查询**：提供对象属性（大小、MD5、创建时间、标签）查询接口，以及列举桶内对象的分页 API。 |
| 6 | **审计日志**：记录对象的上传、下载、删除等操作日志，供后续审计与计费使用。 |

## 非功能性需求  

| 指标 | 目标值 | 备注 |
|------|--------|------|
| **日活跃用户（DAU）** | 500 万 | 包含企业客户与开发者平台用户 |
| **写入 QPS** | 30 k PUT/POST 请求 | 峰值时段（上午 10:00‑12:00） |
| **读取 QPS** | 80 k GET 请求 | 包括 Range 读取 |
| **平均响应延迟** | ≤ 150 ms（99% 请求） | 包括网络传输层的排队与处理 |
| **可用性** | ≥ 99.99%（年均） | 单地区可用性 ≥ 99.95% |
| **存储容量** | 10 EB（预计 3 年内） | 按对象大小均匀分布，支持弹性扩容 |
| **数据耐久性** | 11 9（99.999999999%） | 多副本 + 错误纠删码（Erasure Coding） |

# 题解

# 解题思路总览  

下面是一份 **从最小可用系统到完整高可用、可扩展的对象存储系统** 的完整设计路线。  
- **先把需求拆解** → 明确功能、非功能、边界。  
- **估算规模** → 计算流量、存储、并发等指标，指导后面的容量与选型。  
- **搭建高层框架** → API 网关、鉴权、元数据服务、对象存储层、审计日志等大块。  
- **细化每块的实现** → 选型、数据模型、协议、容错方案。  
- **逐步加入 HA、扩容、跨地域复制** → 让系统满足 99.99% 可用、10 EB 容量、11 9 耐久等目标。  

> **为什么要分阶段？**  
> - 面试官往往先关注 **核心功能**（上传/下载/元数据），再考察 **可扩展性** 与 **细节实现**。  
> - 逐层递进的思路可以帮助面试官快速看到你对系统整体的把控，同时也展示你对每个技术点的深入理解。  

下面我们按 **七个章节** 逐步展开，每一步都会说明 **“为什么这样做”**，以及 **不这么做会有什么缺点”。**

## 第一步：理解需求与规模估算  

### 1. 功能需求梳理  

| 编号 | 功能 | 关键点 |
|------|------|--------|
| 1 | 对象上传（PUT） | 支持单对象最大 5 TB，**分片上传**（Multipart Upload），**暂停/恢复** |
| 2 | 对象下载（GET） | **Range Requests**，**预签名 URL**（临时访问） |
| 3 | 对象删除（DELETE） | 单对象、批量、**版本化**（保留历史） |
| 4 | 桶（Bucket）管理 | 创建/删除，**ACL/Policy**，**跨区域复制（CRR）** 开关 |
| 5 | 元数据查询 | 对象属性、标签，**分页列举** |
| 6 | 审计日志 | 记录所有关键操作，供审计、计费使用 |

### 2. 非功能需求关键指标  

| 指标 | 目标 | 对系统的影响 |
|------|------|--------------|
| DAU | 5 M | 需要 **水平扩展** 的前端网关、鉴权、元数据层 |
| 写入 QPS | 30 k PUT/POST | **写放大**（分片、复制）需考虑 **磁盘 I/O** 与 **网络带宽** |
| 读取 QPS | 80 k GET | **热点对象** 可能导致 **读热点**，需要 **缓存** |
| 响应延迟 | ≤150 ms (99%) | **低延迟路径** 必须在 **同区** 完成，跨区操作走异步 |
| 可用性 | 99.99% 年均 | **多副本** + **故障转移** |
| 存储容量 | 10 EB (3 年) | **弹性扩容**、**冷热分层** |
| 数据耐久性 | 11 9 | **多副本 + 纠删码**（Erasure Coding） |

### 3. 粗略容量与流量计算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日上传数据量** | 30 k PUT/s × avg 5 MB (保守) × 86400 ≈ 12.96 PB/天 | 实际会更低，但需要支撑峰值 |
| **每日下载数据量** | 80 k GET/s × avg 5 MB × 86400 ≈ 34.56 PB/天 | 读取热点会放大 |
| **总存储需求** | 10 EB = 10,000 PB | 需要 **数千台** 存储节点（假设 100 TB/节点） |
| **网络带宽** | 30 k PUT × 5 TB (worst‑case) ≈ 150 TB/s → 远远超实际；实际峰值约 30 k × 10 MB ≈ 2.4 Gb/s | 需要 **10+ Gbps** 入口/出口，且做 **流量分层**（公网+内部） |

> **注意**：这些是 **上限估算**，实际业务中对象大小分布往往呈长尾，平均大小会远低于 5 TB。估算的目的在于 **确保系统设计有足够的余量**，不至于在高峰崩溃。

### 4. 边界划分（面试范围）  

- **必须实现**：上传/下载/删除、分片、桶管理、元数据、审计、鉴权、基本 HA。  
- **可不细化**：计费、SDK、TLS 细节、完整监控平台、对象生命周期。

## 第二步：高层架构设计  

下面给出 **最小可用（MVP）** 与 **完整 HA** 两个层次的架构示意，帮助你在面试中按需展开。

### 1. MVP（最小可用系统）  

```
+-------------------+      +-------------------+      +-------------------+
|   客户端 SDK /   | ---> |   API Gateway    | ---> |   元数据服务      |
|   浏览器/CLI    |      | (REST/HTTPS)     |      | (RDBMS)           |
+-------------------+      +-------------------+      +-------------------+
                                 |                         |
                                 |                         |
                                 v                         v
                          +-------------------+   +-------------------+
                          |   对象存储节点   |   |   审计日志服务   |
                          | (分片写入/读取)  |   | (Kafka + ES)    |
                          +-------------------+   +-------------------+
```

**解释**  
- **API Gateway**：统一入口，完成 **鉴权、流量控制、路由**。  
- **元数据服务**：使用关系型数据库（如 MySQL）存储 **bucket、object、version、ACL**。  
- **对象存储节点**：每台机器负责 **若干磁盘**（本地文件系统或块存储），提供 **PUT/GET** 接口。  
- **审计日志**：写入 Kafka，后续消费到 Elasticsearch（或 ClickHouse）做查询。  

**不这么做的后果**  
- 直接让客户端访问存储节点会 **泄露内部地址**，且 **鉴权、流量控制** 失效。  
- 不使用统一的 **API Gateway**，难以实现 **限流、熔断、灰度发布**。

### 2. 完整 HA & 可扩展架构  

```
                                 +-------------------+
                                 |   DNS / CDN       |
                                 +-------------------+
                                          |
                                          v
                                 +-------------------+
                                 |   全球负载均衡    |
                                 +-------------------+
                                          |
                         +----------------+----------------+
                         |                                 |
                         v                                 v
                +-------------------+            +-------------------+
                |   区域入口层     |            |   区域入口层     |
                | (Edge Proxy)     |            | (Edge Proxy)     |
                +-------------------+            +-------------------+
                         |                                 |
                         v                                 v
          +--------------------------+      +--------------------------+
          |   API Gateway (Cluster)  |      |   API Gateway (Cluster) |
          +--------------------------+      +--------------------------+
                         |                                 |
                         |  +---------------------------+  |
                         +->|   鉴权/签名服务 (Auth)     |<-+
                            +---------------------------+
                         |
                         v
          +----------------------------------------------+
          |                元数据服务层                 |
          |  - MySQL/PostgreSQL (主从/分片)               |
          |  - MetaCache (Redis)                         |
          +----------------------------------------------+
                         |
                         v
          +----------------------------------------------+
          |                对象存储层 (Data Plane)      |
          |  - 存储节点 (对象分片)   + Erasure Coding   |
          |  - 元数据 → Chunk Mapping (Consistent Hash)|
          |  - 多副本/纠删码 (N=10, K=6)                |
          |  - 分布式文件系统 / 对象块服务 (Ceph, MinIO)|
          +----------------------------------------------+
                         |
                         v
          +----------------------------------------------+
          |               审计日志 & 监控层             |
          |  - Kafka → Flink → ClickHouse/ES            |
          |  - Metrics: Prometheus + Grafana            |
          +----------------------------------------------+
                         |
                         v
          +----------------------------------------------+
          |            跨区域复制 (CRR) Service          |
          |  - 异步复制队列 (Kafka) → 目标 Region         |
          |  - 冲突解决：Version+Timestamp+BucketPolicy |
          +----------------------------------------------+
```

**关键技术选型说明**  

| 模块 | 备选技术 | 选型理由 |
|------|----------|----------|
| API Gateway | Kong / Nginx + Lua | 高并发、插件化（鉴权、限流） |
| 鉴权/签名 | HMAC‑SHA256（AWS V4）| 与 S3 兼容，安全性成熟 |
| 元数据存储 | MySQL（主从）+ Redis Cache | 事务性强，查询灵活；缓存提升热点查询性能 |
| 对象存储层 | **Ceph（RADOS）** 或 **MinIO** (基于分布式对象存储) | 原生支持 **Erasure Coding**、**多副本**、**水平扩容** |
| 分片/Chunk 映射 | Consistent Hash + Virtual Nodes | 数据均衡，节点增删对全局影响小 |
| 审计日志 | Kafka + Flink + ClickHouse/Elasticsearch | 实时写入，离线分析，高压缩率 |
| 跨区域复制 | 双向 Kafka Mirror / Pulsar + S3‑compatible API | 异步、容错、可监控的复制流水线 |
| 监控 | Prometheus + Grafana + Alertmanager | 开源、易集成、支持自定义告警 |

> **为什么不直接使用单机文件系统？**  
> 单机容量受限、单点故障、难以实现 11 9 耐久。使用 **分布式对象存储** 能够 **水平扩容**、**自动纠错**，并且 **对外提供统一的对象 API**。

## 第三步：数据库设计  

元数据是系统的“脑”，需要保证 **强一致性**、**事务性**，同时要能够 **快速查询**（列举、属性获取）。下面给出 **核心表结构** 与 **索引** 设计。

### 1. 关键实体  

| 实体 | 说明 |
|------|------|
| **User** | AccessKey、SecretKey、账户状态 |
| **Bucket** | bucket_name、owner_id、创建时间、region、ACL、policy、crr_enabled |
| **Object** | object_id（UUID）、bucket_id、key（路径+文件名）、size、md5、etag、storage_class、created_at、updated_at、is_latest、version_id |
| **ObjectVersion** | version_id、object_id、size、md5、etag、storage_location（chunk ids）、created_at、is_delete_marker |
| **MultipartUpload** | upload_id、bucket_id、key、initiated_at、status（IN_PROGRESS/DONE/ABORTED） |
| **MultipartPart** | upload_id、part_number、etag、size、chunk_location |
| **Tag** | object_id、tag_key、tag_value |
| **AuditLog** (写入 Kafka，后端落库) | operation、user_id、bucket、key、timestamp、request_id、src_ip、result |

### 2. 表结构示例（MySQL）  

```sql
-- Bucket 表
CREATE TABLE bucket (
    bucket_id   BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(63) NOT NULL,
    owner_id    BIGINT UNSIGNED NOT NULL,
    region      VARCHAR(32) NOT NULL,
    acl_json    JSON,
    policy_json JSON,
    crr_enabled BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uq_bucket_name (owner_id, name)
);

-- Object（最新版本指针）
CREATE TABLE object (
    object_id   BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    bucket_id   BIGINT UNSIGNED NOT NULL,
    `key`       VARCHAR(1024) NOT NULL,
    latest_version_id BIGINT UNSIGNED,
    size        BIGINT UNSIGNED,
    md5         CHAR(32),
    etag        VARCHAR(128),
    storage_class VARCHAR(32) DEFAULT 'STANDARD',
    is_deleted  BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_bucket_key (bucket_id, `key`),
    UNIQUE KEY uq_object (bucket_id, `key`)
);

-- ObjectVersion（每个历史版本）
CREATE TABLE object_version (
    version_id  BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    object_id   BIGINT UNSIGNED NOT NULL,
    version_seq BIGINT UNSIGNED NOT NULL,   -- 递增序列
    size        BIGINT UNSIGNED,
    md5         CHAR(32),
    etag        VARCHAR(128),
    location    TEXT,                       -- JSON: [{chunk_id, node_id}, ...]
    is_delete_marker BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_object (object_id),
    UNIQUE KEY uq_version (object_id, version_seq)
);
```

> **为什么使用两张表（object + object_version）？**  
> - **object** 用于快速定位最新版本（GET 时常用）。  
> - **object_version** 保存历史记录，实现 **版本化删除** 与 **回滚**。  
> - 采用 **强一致的关系型 DB**，可以保证 **事务性**（如创建 bucket、写入 object、更新 version），避免元数据不一致。

### 3. 缓存层  

- **Redis**（或 **TiKV**）存放热点 **bucket+key → object_id** 映射、**access token → user info**。  
- 缓存失效采用 **写穿透（write‑through）**：写入 DB 后同步更新缓存。  
- 缓存过期时间设为 **5‑10 min**，防止脏数据长期存在。

### 4. 数据一致性模型  

- **元数据**：**强一致**（读写走同一 MySQL 主库，或使用 **Galera** 多主同步）。  
- **对象数据**：**最终一致**（多副本/纠删码异步同步），因为对象写入后 **返回成功** 前已经在多数副本完成写入（Quorum），读到旧版本的概率极低。  

> **不采用强一致的原因**：  
> - **写放大**：每写一次对象都要同步到全部副本，会导致 **写入延迟** 失控。  
> - **耐久性要求** 已经通过 **N+K** 纠删码满足，**最终一致** 足以保证业务正确性。

## 第四步：核心 API 设计  

下面列出 **对外 RESTful/HTTPS API**（兼容 AWS S3），并说明每个 API 的 **关键实现点**。

### 1. 鉴权与签名  

- **AccessKey / SecretKey**：用户在系统注册后获得。  
- **签名算法**：采用 **AWS Signature V4**（HMAC‑SHA256），请求头 `Authorization` 包含 `Credential`, `SignedHeaders`, `Signature`。  
- **实现要点**：  
  - 在 **API Gateway** 插件层完成签名校验。  
  - 对 **预签名 URL**，服务端在生成 URL 时将有效期、请求信息写入查询参数 `X-Amz-Expires`，客户端直接使用。  

### 2. Bucket API  

| 方法 | 路径 | 说明 | 关键校验 |
|------|------|------|----------|
| `PUT` | `/bucket-name` | 创建 bucket | 检查名称合法性、唯一性、配额 |
| `DELETE` | `/bucket-name` | 删除 bucket（必须为空） | 权限、空检查 |
| `GET` | `/bucket-name?policy` | 获取 bucket policy | ACL 检查 |
| `PUT` | `/bucket-name?policy` | 设置 bucket policy | JSON Schema 校验 |
| `PUT` | `/bucket-name?versioning` | 开启/关闭版本化 | `Enabled|Suspended` |
| `PUT` | `/bucket-name?crr` | 开启跨区域复制 | 目标 Region、复制规则 |

### 3. Object 上传（Multipart）  

1. **Initiate Multipart Upload**  

   ```
   POST /bucket-name/object-key?uploads
   → 返回 uploadId
   ```

   - 在 `multipart_upload` 表创建记录，返回唯一 `uploadId`（UUID）。  

2. **Upload Part**  

   ```
   PUT /bucket-name/object-key?partNumber=1&uploadId=xxxx
   → 请求体：分片数据（5 MB~5 GB）
   ```

   - 每个 Part 写入 **对象存储节点**（采用 **写入流** → 生成 Chunk ID、ETag）。  
   - 在 `multipart_part` 表记录 `partNumber`, `etag`, `size`, `chunk_location`。  

3. **Complete Multipart Upload**  

   ```
   POST /bucket-name/object-key?uploadId=xxxx
   Body: <CompleteMultipartUpload> (list of partNumber+etag)
   ```

   - **验证**：所有 Part 是否已上传、ETag 是否匹配。  
   - **原子合并**：在 **元数据 DB** 开启事务，写入 `object_version`（聚合所有 chunk ids），更新 `object.latest_version_id`，删除 `multipart_upload` 记录。  
   - **返回**：`ETag`、`VersionId`。  

4. **Abort Multipart Upload**  

   - 删除未完成的分片（异步回收），删除 `multipart_upload` 记录。  

> **为什么要使用分片上传？**  
> - 单对象最大 5 TB，网络中断不可避免。  
> - 分片可以 **并行上传**，提升吞吐。  
> - 支持 **暂停/恢复**，只需记录已上传的 Part 列表。  

### 4. 对象下载  

| 方法 | 路径 | 说明 | 关键点 |
|------|------|------|--------|
| `GET` | `/bucket-name/object-key` | 普通下载 | 读取最新版本的 Chunk 列表，流式返回 |
| `GET` | `/bucket-name/object-key?versionId=xxx` | 指定版本下载 | 读取对应 Version 的 Chunk |
| `GET` | `/bucket-name/object-key?partNumber=1&uploadId=xxx` | 下载分片（仅在 multipart 完成前调试） | 同上 |
| `GET` | `/bucket-name/object-key?X-Amz-Expires=3600&X-Amz-Signature=...` | 预签名 URL | 直接走 CDN/Edge，免鉴权 |

- **Range Requests**：解析 `Range: bytes=start-end`，只读取对应 Chunk 的子区间，**避免全对象读取**。  
- **缓存**：热点对象可放在 **CDN / Edge Cache**（如 CloudFront），降低后端读负载。  

### 5. 对象删除  

| 方法 | 路径 | 说明 | 关键点 |
|------|------|------|--------|
| `DELETE` | `/bucket-name/object-key` | 删除最新版本 | 在 `object_version` 中写入 **Delete Marker**（若开启版本化），并更新 `object.is_deleted` |
| `DELETE` | `/bucket-name/object-key?versionId=xxx` | 删除指定历史版本 | 直接删除对应 `object_version`（或标记为删除），回收 Chunk |
| `POST` | `/bucket-name?delete` (XML body) | 批量删除 | 事务批量写入 Delete Marker，返回成功/失败列表 |

> **版本化删除**：即使用户执行 DELETE，也会保留 **Delete Marker**，后续 `GET` 若没有 `versionId` 会返回 404，历史版本仍可恢复。  

### 6. 元数据查询  

- **HEAD Object**：返回对象属性（size、ETag、Last-Modified、Metadata）而不返回内容。  
- **GET Bucket (List Objects)**：分页返回 `key`, `size`, `last_modified`, `storage_class`，支持 `prefix`、`delimiter`（模拟目录结构）。  

### 7. 审计日志  

- 每一次 **成功或失败** 的 API 调用，API Gateway 将 **日志结构体** 写入 **Kafka topic** `audit-log`.  
- 结构体示例（JSON）：

```json
{
  "request_id": "c1a2b3...",
  "timestamp": "2026-05-26T12:34:56Z",
  "user_id": 12345,
  "access_key": "AKIA...",
  "operation": "PUT_OBJECT",
  "bucket": "my-bucket",
  "key": "photos/2024/01/img001.jpg",
  "version_id": "v123",
  "client_ip": "203.0.113.12",
  "region": "cn-north-1",
  "status": "200",
  "bytes_sent": 5242880,
  "error_code": null
}
```

- 后端消费后持久化到 **ClickHouse**（高压缩、支持秒级查询）或 **Elasticsearch**（支持全文搜索）。

## 第五步：详细组件设计  

下面从 **每个核心组件** 的内部实现、容错、选型细节展开说明。

### 1. API Gateway & Edge Layer  

| 功能 | 方案 | 关键实现 |
|------|------|----------|
| **请求路由** | Nginx + Lua（OpenResty）或 Kong | 根据 `Host`（bucket）或 `X-Amz-Target` 路由到对应内部服务 |
| **鉴权** | HMAC‑SHA256 校验插件 | 读取 `Authorization`，使用 `SecretKey` 重新计算签名，时间戳校验防重放 |
| **限流/熔断** | Redis Token Bucket + Hystrix | 按 AccessKey 限制 QPS，防止恶意刷流 |
| **协议转换** | HTTP ↔ gRPC (内部) | 对象存储节点内部使用 gRPC，提升二进制传输效率 |
| **日志** | Access Log → Kafka (topic: `gateway-access`) | 包含请求耗时、状态码、客户端 IP |

**容错**：  
- **多实例** + **LVS/NGINX** 负载均衡；  
- **健康检查**：每 5 s 对实例做 HTTP/2 心跳，异常剔除。  

### 2. 鉴权/签名服务  

- **存储**：AccessKey、SecretKey 加盐后哈希存入 MySQL；  
- **缓存**：Redis 里保存 `access_key → {user_id, secret_hash}`，TTL 5 min。  
- **签名校验流程**：  
  1. 解析请求头，提取 `X-Amz-Date`, `X-Amz-Expires`, `SignedHeaders`。  
  2. 使用 `SecretKey` 按 AWS V4 规范生成 `CanonicalRequest` → `StringToSign` → `Signature`。  
  3. 对比签名，若不匹配返回 `403 Forbidden`。  

> **常见坑**：时钟不同步会导致签名失效，必须要求 **所有服务器时间同步**（NTP）并在校验时允许 **±5 min** 的窗口。

### 3. 元数据服务  

- **主库**：MySQL 8.x + **Group Replication**（同步复制，自动故障转移）。  
- **读写分离**：写入走主库，读取走 **只读实例**（提升查询并发）。  
- **分库分表**：`bucket_id` 为分库键，`object_id` 为分表键（每 1 B objects 一个表），降低单表行数。  
- **事务**：创建 bucket → 写入 bucket 表、创建默认 ACL、写入审计日志，全部在同一事务内完成。  

#### 3.1 元数据缓存（MetaCache）  

- **Key**：`bucket_id:key → object_id`（字符串拼接）。  
- **更新策略**：  
  - **写后立即更新**（write‑through）。  
  - **TTL 10 min**，防止热点对象长期占用内存。  

### 4. 对象存储层（Data Plane）  

#### 4.1 存储节点结构  

```
[Node] = {
    id: UUID,
    ip: "10.0.1.23",
    disks: [ /dev/nvme0n1, /dev/nvme1n1, ... ],
    zone: "cn-north-1a"
}
```

- 每块磁盘使用 **XFS** 或 **ext4**，开启 **direct I/O**（O_DIRECT）避免 page cache 双写放大。  
- **块大小**：默认 8 MiB，适配大对象写入。  

#### 4.2 数据布局  

- **Chunk**（分片） → **Chunk ID**（UUID） → **物理路径** `/data/<node-id>/<hash-prefix>/<chunk-id>`。  
- **Erasure Coding**：采用 **EC(10,6)**（10 data + 6 parity），每 8 MiB 的数据块生成 6 个校验块，分布在不同 **故障域**（机架、可用区）。  

| 编码方式 | 数据块数 (k) | 冗余块数 (m) | 容错能力 | 存储系数 |
|----------|--------------|--------------|----------|----------|
| 3副本    | 1            | 2            | 任意 2 失效 | 3× |
| EC(10,6) | 10           | 6            | 任意 6 失效 | 1.6× |

> **为什么选 EC(10,6) 而非 3 副本？**  
> - **成本**：存储系数从 3× 降到 1.6×，对 10 EB 规模的成本差异巨大。  
> - **耐久性**：EC 同样可以达到 **11 9** 级别，只要选对 `k,m`。  
> - **缺点**：写放大约为 **k+m/k ≈ 1.6**，读放大略高，需要 **读时重建**，但通过 **局部缓存**（热点块）可缓解。  

#### 4.3 写入流程  

1. **客户端** → **API Gateway** → **对象存储服务（gRPC）**  
2. **对象服务** 将对象切分为 **Chunk**（默认 8 MiB），每个 Chunk 交给 **EC Encoder**，生成 **Data+Parity**。  
3. **Consistent Hash** 决定每个 Chunk/Parity Block 的目标 **存储节点**。  
4. **写入**：使用 **二阶段提交**（prepare → commit）确保 **至少 k+1** 副本成功后返回成功给客户端。  

#### 4.4 读取流程  

1. **查询元数据** → 获得 Chunk 列表。  
2. **并行请求** 至 **k** 个存储节点读取 Data Chunk；若有节点失效，则从 **Parity** 重建。  
3. **流式合并** → 返回给客户端。  

#### 4.5 故障恢复  

- **监控**：每块磁盘、节点、网络均上报 **健康状态**（Prometheus）。  
- **自动修复**：当检测到 Chunk 丢失或节点宕机，系统调度 **Repair Worker** 拉取剩余块进行 **重建**，并写入新节点。  
- **恢复时间目标 (RTO)**：≤ 30 min（依据业务 SLA）  

### 5. 跨区域复制（CRR）  

| 步骤 | 说明 |
|------|------|
| **触发** | 当对象写入成功（commit）且 bucket 的 `crr_enabled = true`，发送 **复制事件** 到 Kafka topic `crr-event`。 |
| **复制队列** | 目标 Region 的 **CRR Service** 消费事件，拉取源对象的 **Chunk 列表** 与 **元数据**。 |
| **冲突解决** | 使用 **VersionId + Last-Modified Timestamp**：若目标已存在同名对象且版本更新，则采用 **最新写入覆盖**；若两者同时写入，则保留 **两条版本**（因为版本化已开启）。 |
| **目标写入** | 与本地区写入流程相同，采用 **EC** 或 **3 副本**（依据目标 Region 的策略）。 |
| **一致性** | CRR 为 **异步**，复制延迟一般为 **秒到分钟**，满足“最终一致”。 |
| **监控** | 复制成功率、延迟、错误率等指标写入监控系统，告警阈值：失败率 > 0.1% → 人工介入。 |

> **为什么采用异步复制而非强同步？**  
> - 强同步会将跨地域网络 RTT（≥ 100 ms）加入写延迟，破坏 **150 ms** 的响应目标。  
> - 大多数业务对跨区副本的 **强一致性** 要求不高，只需要 **灾备** 能力。  

### 6. 审计日志系统  

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| **日志收集** | API Gateway → Kafka (topic: `audit-log`) | 高吞吐、持久化 |
| **实时处理** | Flink Job | 过滤、聚合、写入 ClickHouse |
| **离线查询** | ClickHouse | 支持秒级聚合查询（如 “某用户 24h 内下载次数”） |
| **可视化** | Grafana + ClickHouse datasource | 报表、趋势图 |
| **保留策略** | 热数据 30 天 → 冷数据 1 年 → 归档至对象存储（Glacier） | 符合合规要求 |

### 7. 监控与告警  

- **指标**（Prometheus）  
  - `api_gateway_qps`, `api_gateway_latency_ms`  
  - `metadata_db_latency`, `metadata_db_error_rate`  
  - `storage_node_cpu`, `storage_node_disk_io`, `storage_node_error_rate`  
  - `crr_lag_seconds`, `crr_error_rate`  
  - `replication_under_repair_jobs`  
- **告警**（Alertmanager）  
  - QPS > 80% of capacity → 自动弹性伸缩  
  - 单节点磁盘使用 > 85% → 触发扩容  
  - CRR 延迟 > 5 min → 触发人工排查

### 1. 横向扩容（Scale‑out）  

| 维度 | 扩容方式 | 关键点 |
|------|----------|--------|
| **API 层** | 增加 Gateway 实例 + 负载均衡（LVS/NGINX） | 采用 **无状态**，水平扩展简单 |
| **元数据层** | **分库**（按租户或 region） + **读写分离** | 通过 **MySQL Proxy** 或 **ShardingSphere** 实现动态路由 |
| **对象存储节点** | 增加 **Node**，使用 **Consistent Hash** 自动迁移部分 Chunk | **Rebalance** 过程使用 **后台流量限制** 防止突发带宽占满 |
| **审计日志** | Kafka **分区** 增加，消费者 **水平扩容** | 保证 **吞吐 ≥ 100k events/s** |
| **CRR** | 多 Region 的 **复制集群**，每个 Region 独立消费 | 防止单点故障，提升跨区容灾能力 |

### 2. 高可用（HA）设计  

| 组件 | 故障场景 | HA 方案 |
|------|----------|----------|
| **API Gateway** | 单实例宕机 | 多实例 + **LVS** 健康检查 |
| **鉴权服务** | DB 主库失效 | **MySQL Group Replication** 自动切换 |
| **元数据 DB** | 整个 Region 故障 | **跨 Region 多活**（双活）+ **双向同步**（GTID） |
| **对象存储节点** | 磁盘失效、节点宕机 | **EC(10,6)** 可容忍 6 块失效；自动 **Repair** |
| **网络** | 链路抖动 | 多链路 **BGP** 互备，使用 **公网 + 私网** 双通道 |
| **审计日志** | Kafka 故障 | **多副本**（replication.factor=3） + **MirrorMaker** 跨集群备份 |

### 3. 容灾恢复（DR）  

- **备份**：元数据 DB 每日全量快照 + 增量日志（binlog）。  
- **恢复步骤**：  
  1. 在新 Region 部署 **相同拓扑**（Gateway、DB、Storage）  
  2. 恢复 DB 快照 → 重放 binlog  
  3. 从对象存储的 **冷备份（Glacier）** 恢复 Chunk（若有）  
- **RTO / RPO**：  
  - **RPO**（数据丢失） ≤ 5 min（通过跨 Region 同步 binlog）  
  - **RTO**（恢复时间） ≤ 2 h（自动化脚本）  

### 4. 性能优化细节  

| 场景 | 优化手段 |
|------|----------|
| **上传峰值** | - 客户端并行上传多个 Part <br> - API Gateway 使用 **HTTP/2** 多路复用 <br> - 存储节点开启 **零拷贝**（sendfile） |
| **下载热点** | - 使用 **CDN/Edge Cache** <br> - 在存储节点本地开启 **LRU** 块缓存 <br> - 支持 **Range**，只拉取请求的字节 |
| **元数据查询** | - **MetaCache**（Redis）存热点对象 <br> - **分页查询**使用 **cursor**，避免 OFFSET 大查询 <br> - 读写分离，热点读走只读实例 |
| **跨区域复制** | - **压缩**（Snappy）后发送，降低网络带宽 <br> - **批量复制**：一次性发送多个 Chunk 任务 <br> - **复制调度**：在非高峰时段加速复制 |

## 第七步：常见面试追问与回答  

下面列出面试官经常会追问的细节，并给出 **完整、条理化的答案**，帮助你在面试中快速组织语言。

### 1. “在分片上传完成后，如何保证对象的最终一致性？如果写入过程中出现节点故障，你会采用哪种一致性模型？”  

**回答思路**  

1. **分片上传阶段**：每个 Part 写入时采用 **两阶段提交**（prepare → commit）。  
   - **prepare**：写入临时文件并写入 **metadata**（part 表）标记为 `PENDING`。  
   - **commit**：所有 Part 均成功后，`CompleteMultipartUpload` 在元数据库开启事务，把所有 Part 合并成 **ObjectVersion**，并标记 `COMMITTED`。  

2. **最终一致性**：  
   - **元数据**采用 **强一致**（MySQL 主库事务），保证 **对象版本的原子性**。  
   - **对象数据**采用 **最终一致**（EC/N+K），因为写入时已经保证 **多数（k+m/2+1）** 副本成功，若有少数节点故障，系统会 **后台修复**。  

3. **节点故障处理**：  
   - 若在 **prepare** 阶段节点失效，客户端会收到错误，需 **重新上传该 Part**。  
   - 若在 **commit** 阶段（元数据提交成功）但部分 Chunk 丢失，后台 **Repair Worker** 会依据 EC 算法重建缺失块，保证对象可读。  

4. **一致性模型**：  
   - **元数据** → **强一致（Linearizability）**  
   - **对象数据** → **最终一致（Eventual Consistency）**，满足 S3 的设计原则，也能在 150 ms 延迟目标内完成写入。  

**要点**：强调 **两阶段提交**、**元数据事务**、**EC 容错** 与 **后台修复**，解释为什么不需要强一致的对象层（性能、成本原因）。

### 2. “请说明跨区域复制的实现思路，包括冲突解决、复制延迟以及如何保证目标区域的耐久性？”  

**回答要点**  

1. **触发机制**：对象写入成功后，`metadata` 触发 **CRR Event**（Kafka）。  
2. **复制流程**：  
   - **CRR Service**（目标 Region）消费事件，读取源对象的 **Chunk 列表** 与 **元数据**。  
   - 使用 **异步流水线**：① 拉取源 Chunk → ② 通过压缩/加密 → ③ 发送至目标存储节点（gRPC）。  
3. **冲突解决**：  
   - **VersionId** + **Last-Modified**：若目标已有同名对象且版本号更高，则保留两者（因为版本化已开启）。  
   - 若两者 **时间戳相同**（极少），使用 **源 Region 的写入顺序号** 进行决定。  
4. **复制延迟**：  
   - **异步**，典型延迟 10 s~5 min，受网络带宽与目标 Region 写入压力影响。  
   - 通过 **监控 `crr_lag_seconds`**，若超过阈值可开启 **加速复制**（提升并发）。  
5. **目标区域耐久性**：  
   - 目标 Region 采用 **相同或更高的纠删码**（如 EC(12,4)）或 **3 副本**，保证 **11 9** 级别。  
   - 同时在目标 Region 再次写入 **审计日志**，实现 **双活审计**。  

**为什么采用异步**：保持写入路径低延迟，满足 150 ms SLA；跨区域网络不可预测，强同步会导致严重的用户感知延迟。

### 3. “面对 10 EB 级别的存储需求，你会如何设计磁盘分层、冷热数据分离以及自动扩容的策略？”  

**回答结构**  

1. **磁盘分层**  
   - **热层**：使用 **NVMe SSD**（或高速 HDD）提供 **低延迟**，存放最近 30 天的活跃对象（依据访问频率）。  
   - **温层**：使用 **SATA HDD**（较大容量、成本低），存放 30 天~6 个月的对象。  
   - **冷层**：使用 **对象归档存储**（如 **Glacier‑like**），存放 >6 个月不活跃的数据。  

2. **冷热分离实现**  
   - **元数据**中记录 **last_access_timestamp**。  
   - **后台调度**（Cron + Spark）定期统计热点，使用 **迁移任务**把对象从冷层搬回热层（访问时自动升温），或把不活跃对象下沉。  
   - **迁移过程**：读取 Chunk → 写入目标层 → 更新 **location** 元数据 → 删除旧块（异步）。  

3. **自动扩容**  
   - **存储节点**采用 **容器化**（K8s）或 **裸金属** 自动化部署。  
   - **容量监控**：当节点磁盘使用率 > 85% 时，触发 **扩容脚本**：  
     1. 新增 **Node**（挂载新磁盘）  
     2. **Rebalance**：Consistent Hash 重新分配部分 Chunk 到新节点（使用 **流量限制** 防止突发）  
   - **弹性伸缩**：使用 **Kubernetes Operator** 管理 **Ceph/MinIO** 集群，自动根据 **CPU/IO/容量** 进行扩容或缩容。  

4. **成本控制**  
   - 通过 **存储层次** 与 **EC** 结合，热层使用 **3 副本**（性能），冷层使用 **EC(12,4)**（更高压缩率），整体成本比全副本下降 **60%+**。  

**为什么要分层**：单一存储介质既无法满足 **低延迟** 又会导致 **成本爆炸**，分层能在 **性能** 与 **成本** 之间取得平衡。

### 4. “如果元数据库出现热点（比如大量对象列举请求），你会怎么做？”  

**回答要点**  

1. **读写分离**：将 **查询** 请求路由到 **只读副本**，写请求仍走主库。  
2. **分页 Cursor**：避免 `OFFSET`，使用 **key‑based cursor**（`next_marker`），降低全表扫描成本。  
3. **缓存**：对 **热点前缀**（如 `logs/2024/05/`) 使用 **Redis** 缓存对象列表的 **ID 列表**，TTL 1 min。  
4. **二级索引**：在 **MySQL** 上建立 **覆盖索引** (`bucket_id, key`) 只返回需要的列，避免回表。  
5. **分库**：根据 **bucket_id** 或 **region** 将对象表水平分库，降低单库并发。  

---

### 5. “对象上传时如果出现网络抖动，导致部分 Part 丢失，你会怎么保证上传的完整性？”  

**回答要点**  

- **分片校验**：每个 Part 在上传结束后返回 **ETag（MD5）**，客户端在 `CompleteMultipartUpload` 时把每个 Part 的 ETag 发给服务端，服务端再次校验。  
- **重试机制**：客户端 SDK 内部对 **非 2xx** 响应自动 **指数退避重试**。  
- **幂等性**：`Upload Part` 接口是 **幂等** 的，同一个 `uploadId + partNumber` 再次上传会覆盖旧的 Part，防止重复数据。  
- **后台检查**：`CompleteMultipartUpload` 前，服务端会检查 **所有 Part 是否齐全**，缺失则返回 **MissingPart** 错误，客户端可重新上传缺失的 Part。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **对象数据的耐久性方案（副本 vs. Erasure Coding）** | 需要在 **成本**、**写入延迟**、**恢复时间**、**存储系数** 之间权衡。 | ① 计算 10 EB 存储的原始成本（3 副本 → 30 EB 物理容量） ② 评估 EC(10,6) 的存储系数 1.6×，成本下降约 50% ③ 分析写放大：EC 写放大 1.6，仍在 150 ms 延迟预算内。 ④ 考虑故障恢复复杂度：EC 需要重建，设计了 **Repair Worker** 自动化。最终选 **EC(10,6)** 为主，热点对象使用 **3 副本** 做加速。 |
| **跨区域复制的强度与一致性模型** | 需要兼顾 **写入延迟**、**网络带宽**、**灾备要求**。 | ① S3 官方采用 **异步复制**，参考其设计。 ② 计算跨区 RTT（≈100 ms）对 150 ms SLA 的冲击，发现强同步不可接受。 ③ 为满足 **耐久性**，在目标区域仍使用 **EC**，并在复制完成后写入 **审计日志**。 ④ 设计 **冲突解决**（VersionId+Timestamp）以保证业务的**幂等性**。最终采用 **异步、最终一致** 的 CRR。 |

### 2. 新手最容易犯的错误（至少 2 个）  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把对象数据也放在关系型数据库里** | 随着对象体积增长，DB IO、磁盘空间、备份成本都会爆炸，且难以满足 5 TB 单对象。 | 使用 **专门的对象存储层**（Ceph/MinIO），元数据单独放关系型 DB。 |
| **只关注强一致性**，把所有写都走 **同步复制** | 会导致写入延迟大幅提升，难以满足 150 ms SLA，且成本随副本数线性增长。 | 对 **元数据**采用强一致，对 **对象数据**采用 **最终一致 + EC**，并在后台进行 **Repair**。 |
| **忽视分片上传的幂等性** | 客户端网络抖动导致重复上传同一 Part，产生脏数据或浪费存储。 | 在 **Upload Part** 接口设计 **幂等键**（uploadId+partNumber），返回 ETag，**Complete** 前再次校验。 |
| **只在单机上做缓存**，忽视缓存失效/一致性 | 当节点故障或缓存失效时会出现 “对象找不到” 的 500 错误。 | 使用 **分布式缓存（Redis Cluster）**，并实现 **写穿透/写回** 机制。 |
