# 第 23 天：设计 直播带货系统

> 生成日期：2026-05-03

---

## 题目背景  
直播带货系统是一套面向消费者的实时视频购物平台，主播通过弹幕、秒杀、优惠券等方式在直播间推销商品，用户可以在观看直播的同时下单购买。系统需要在数万并发观众的高峰时段保持流畅的观看体验并实时完成交易。

## 面试场景设定  
> **面试官**：我们现在要设计一个支撑 **每日峰值 30 万并发观众**、**每秒 2 k 订单**的直播带货平台。请从系统架构、核心模块、扩展性和可靠性等方面进行整体设计，并说明你在容量规划和故障容错上的考虑。

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **主播管理**：主播开播、结束直播、切换摄像头/画面、推送商品信息。 |
| 2 | **实时视频播放**：观众低延迟观看直播流，支持弹幕、礼物特效、画中画等交互。 |
| 3 | **商品展示 & 购物车**：主播推送商品卡片，观众可点击查看详情、加入购物车、立即下单。 |
| 4 | **秒杀/抢购**：在直播期间开启限时抢购，系统需保证抢购秒级响应并防止超卖。 |
| 5 | **订单支付**：支持多种支付方式（支付宝、微信、信用卡），下单后实时生成订单并推送支付结果。 |
| 6 | **实时统计 & 推荐**：实时统计观看人数、弹幕量、成交额，向观众推荐相关商品或活动。 |

## 非功能性需求  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（活跃用户）** | 500 万 | 包括观众、主播、商家后台用户 |
| **峰值并发观众** | 30 万 | 单场直播最高同时在线人数 |
| **订单 QPS** | 2 k (2000) | 峰值秒级下单请求数 |
| **直播流延迟** | ≤ 1 秒 | 端到端（推流 → 播放）延迟 |
| **系统可用性** | 99.99%（年平均） | 每年累计故障时间 ≤ 52.56 分钟 |
| **存储容量** | 视频原始流 30 TB/天、商品/订单数据 5 TB/年 | 包含冷/热存储分层 |

## 系统边界  

**本题范围内**  
- 直播推流、分发、播放链路（CDN、协议选型）  
- 观众交互（弹幕、礼物、抢购）  
- 商品信息管理、下单、支付流程（不涉及第三方支付网关实现）  
- 高并发订单处理、库存防超卖机制  
- 基础监控、告警、自动伸缩策略  

**不考虑**  
- 主播入驻、实名认证、内容审核（假设已在外部系统完成）  
- 商家后台的商品上架、库存补货管理  
- 详细的财务结算、税务、发票系统  
- 客服、退货、售后流程  
- 机器学习推荐模型的实现细节（仅要求提供数据入口）  

## 提示与追问  

1. **容量规划**：如果直播峰值观众从 30 万提升到 100 万，你会如何重新设计 CDN 与业务层的扩容方案？  
2. **库存防超卖**：在秒杀场景下，如何在分布式环境中实现高并发的库存扣减，避免超卖且保证性能？请说明使用的技术（如 Redis Lua 脚本、消息队列、分布式锁等）及其优缺点。  
3. **容错与灾备**：当某一地区的 CDN 节点全部失效时，系统如何保证观众的观看体验不中断？请描述跨地域备份、流切换以及监控告警的设计。

## 解题思路总览

1. **先把业务拆成「读」和「写」两类**  
   * 读：观看直播、弹幕、商品信息、统计数据 → 需要高吞吐、低延迟，适合 **缓存 + CDN**。  
   * 写：开播/下单/秒杀/弹幕发送 → 需要 **强一致**、**事务**，适合 **消息队列 + 分布式事务**。  

2. **从最小可用系统（单机 + 本地缓存）出发**，一步步加入：  
   * 负载均衡 → 多机水平扩展  
   * CDN / 分片存储 → 大流量视频分发  
   * Redis + MQ → 高频交互（弹幕、秒杀）  
   * MySQL / ClickHouse → 业务数据 + 大数据分析  
   * 多可用区、跨地域灾备 → 99.99% SLA  

3. **每个功能点对应的关键技术**  
   * **直播流**：RTMP/RTMFP → HLS/FLV → CDN（边缘节点）  
   * **弹幕/礼物**：WebSocket → Kafka → Redis 缓存弹幕列表  
   * **秒杀**：Redis 原子扣库存 + MQ 异步落库 → 防超卖、削峰  
   * **订单**：业务服务 → 事务消息 → 最终一致 + 幂等设计  

4. **容量规划**：用 **峰值并发 × 平均请求耗时** 估算 QPS；用 **流量 × 视频码率** 估算带宽；再加上 **安全系数 1.5~2**。  

5. **容错思路**：**多副本 + 主备切换 + 健康检查**，把单点故障压到 **毫秒级**，并在监控报警体系里做到 **预警 + 自动伸缩**。

## 第一步：理解需求与规模估算

### 1.1 功能拆解

| 大模块 | 子功能 | 关键交互 |
|--------|--------|----------|
| 主播管理 | 开播、结束、切换画面、推商品 | 主播 → 推流服务器 → CDN |
| 实时视频 | 观看、弹幕、礼物、画中画 | 观众 ← CDN ← 推流 |
| 商品 & 购物车 | 商品卡片、详情、加入购物车 | 前端 → 商品服务 → DB |
| 秒杀/抢购 | 开启抢购、下单、扣库存 | 前端 → 秒杀服务 → Redis/MQ |
| 订单支付 | 生成订单、支付回调、状态同步 | 前端 → 订单服务 → 支付网关 |
| 实时统计 & 推荐 | 观看人数、弹幕量、成交额 | 各服务 → 统计/推荐服务 |

> **注意**：这里把「实时」的交互（弹幕、礼物、秒杀）单独列出来，因为它们对 **时延**、**并发** 的要求最高，往往需要专门的高速通道（WebSocket + MQ）。

### 1.2 关键非功能指标

| 指标 | 计算方式 | 备注 |
|------|----------|------|
| 峰值并发观众 | 30 万 | 单场最高同屏人数 |
| 观众带宽需求 | 30 万 × 1.5 Mbps ≈ 45 Gbps | 假设 720p、码率 1.5 Mbps，实际会采用多码率自适应 |
| 订单 QPS | 2 k | 峰值每秒下单请求数 |
| 弹幕 QPS | 30 万 × 0.5 条/秒 ≈ 150 k | 估算每位观众 0.5 条/秒，取保守值 |
| 秒杀 QPS | 2 k（在抢购窗口） | 与订单 QPS 相同量级 |
| 数据存储 | 视频原始流 30 TB/天、业务数据 5 TB/年 | 冷/热分层（对象存储 + 本地 SSD） |

### 1.3 初步容量规划（保守 1.5 倍安全系数）

| 资源 | 需求 | 说明 |
|------|------|------|
| **CDN 带宽** | 45 Gbps × 1.5 ≈ **68 Gbps** | 边缘节点需支持跨省/跨国流量 |
| **推流入口** | 30 Mbps（单主播） | 多主播可以使用负载均衡 + 按业务分组 |
| **WebSocket 连接** | 30 万 | 使用 **Nginx + Lua** 或 **Envoy** 做连接层，后端使用 **Kafka** 分流 |
| **Redis 实例** | 150 k QPS（弹幕） + 2 k QPS（秒杀） | 采用 **主从 + 分片**（Cluster） |
| **MySQL** | 2 k QPS 写 + 10 k QPS 读 | 读写分离 + 分库分表（订单库、商品库） |
| **Kafka** | 200 k QPS（弹幕） + 10 k QPS（订单） | 分区数 ≥ 12（每 10k QPS 一个分区） |

> **如果不做容量估算**，面试官会直接问 “峰值怎么支撑？” 你只能说 “加机器”，显得不专业。  
> **做好估算** 能让你自然展开后面的架构设计，展示系统思维。

### 2.1 最小可用系统（MVP）

+-------------------+      +-------------------+      +-------------------+
       |                         |                         |
       | 观看 (HLS/FLV)          |                         |
       v                         v                         v
+-------------------+      +-------------------+      +-------------------+
|   前端网页/APP    | <--- |   WebSocket Server| <--- |   弹幕/礼物服务   |
+-------------------+      +-------------------+      +-------------------+

+-------------------+      +-------------------+      +-------------------+
|   商品/下单服务   | <--- |   API Gateway    | ---> |   MySQL (单实例) |
+-------------------+      +-------------------+      +-------------------+

**特点**  
* 单点部署，所有业务跑在同一台机器上。

**问题**  
* **单点故障** → 任意组件挂掉全局不可用。  
* **扩展性差** → 30 万并发根本撑不住。

#### 关键设计点解释

| 层级 | 作用 | 为什么要这样设计 | 不这样会有什么问题 |
|------|------|------------------|--------------------|
| **入口层（Edge）** | DNS → WAF → CDN → Global Load Balancer → L4/L7 LB | 把 **流量** 先在 **边缘** 过滤、缓存，降低回源压力；全局负载均衡把用户路由到最近的可用区 | 直接把流量打到业务集群会导致 **网络拥塞**、**DDoS** 难以防御 |
| **推流层** | RTMP/RTMFP 接收主播流 → 转码 → 存储 | 主播使用 **RTMP** 推流最成熟；转码生成 **多码率 HLS/FLV**，满足不同网络带宽的观众；存入对象存储做**事后回放** | 不转码只能一种码率，低网速用户卡顿；不存储无法提供回放、审计 |
| **实时交互层** | WebSocket + Kafka + Redis | **WebSocket** 保持长连接，秒级弹幕/礼物响应；**Kafka** 做水平扩展、持久化；**Redis** 缓存最近弹幕供新用户快速拉取 | 只用 HTTP 轮询延迟大；仅靠数据库写入弹幕会导致 **写放大**、**锁竞争** |
| **商品/订单业务层** | API Gateway + 微服务 | **Gateway** 统一入口、限流、鉴权；微服务拆分（商品、购物车、订单、秒杀）实现 **独立扩容** 与 **团队自治** | 单体服务在高并发时 **CPU/内存瓶颈**，改动牵一发而动全身 |
| **数据层** | MySQL 主从/分库分表、Redis Cluster、Kafka、ClickHouse、对象存储 | **关系型数据库** 负责事务、库存、订单；**Redis** 负责高速读写（弹幕、秒杀库存）；**Kafka** 负责可靠的异步消息；**ClickHouse** 负责 **实时 OLAP** 统计；**对象存储+CDN** 负责大容量视频 | 只用 MySQL 存弹幕会导致 **慢查询**；没有 ClickHouse 统计实时数据会导致 **仪表盘卡顿** |
| **监控/告警** | Prometheus + Grafana + AlertManager | 开箱即用的时序监控，能 **自动发现**、**多维度聚合**，对 **QPS、延迟、错误率** 报警 | 没有监控，故障只能等到用户投诉，恢复时间难以控制 |

## 第三步：数据库设计

> **原则**：  
> 1. **业务强一致**：订单、库存、用户钱包必须使用 **事务**（MySQL）  
> 2. **高并发读**：弹幕、商品信息、排行榜使用 **缓存**（Redis）或 **列式库**（ClickHouse）  
> 3. **写分离**：读写分离 + 分库分表，避免热点

### 3.1 关系型数据库（MySQL）模型

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| `users` | `user_id` | `nickname, avatar, balance` | 所有平台用户（观众、主播、商家） |
| `live_rooms` | `room_id` | `host_id, status, start_time, end_time` | 直播间元信息 |
| `products` | `product_id` | `title, price, stock, sale_start, sale_end` | 商品基本信息（秒杀商品标记 `is_flash`） |
| `carts` | `cart_id` | `user_id, product_id, qty, created_at` | 购物车（TTL 30 天） |
| `orders` | `order_id` | `user_id, total_amount, status, created_at` | 订单头 |
| `order_items` | `order_item_id` | `order_id, product_id, qty, price` | 订单明细 |
| `inventory_log` | `log_id` | `product_id, change, source, ts` | 库存变更审计（秒杀、普通下单） |
| `flash_sale` | `flash_id` | `product_id, start_time, end_time, stock_total` | 秒杀专用表，记录秒杀库存快照 |

#### 分库分表方案

* **订单库**：基于 `order_id`（或者 `user_id`）做 **水平分库**（如 4 库），每库内部 **分表**（`order_00~order_99`） 按 **时间** 或 **订单号后缀**。  
* **商品库**：读多写少，**单库**即可，但 **商品表** 按 `product_id` 分表（每 1 万 商品一个表） 防止热点。  

> **如果不分库**，在 2 k QPS 的订单高峰下，单实例 MySQL 的 **写入** 能力（≈ 1 k TPS）会成为瓶颈。

### 3.2 缓存层（Redis）

| 用途 | 数据结构 | 失效策略 | 示例 |
|------|----------|----------|------|
| **弹幕缓存** | `Sorted Set (ZSET)` `room_id:danmu` | 保留最近 5 分钟弹幕（`ZREMRANGEBYRANK`） | `ZADD room:123:danmu ts "弹幕内容"` |
| **秒杀库存** | `String` `flash_stock:{product_id}` | 永不过期，库存扣减后 **不再更新**（通过 Lua 脚本） | `SET flash_stock:1001 5000` |
| **商品详情** | `Hash` `product:{id}` | 5 分钟 TTL（热点商品可延长） | `HMSET product:1001 title "..." price 99 stock 500` |
| **用户购物车** | `Hash` `cart:{user_id}` | 永不过期（业务主动清理） | `HSET cart:2001 1001 2` |
| **防重放/幂等** | `String` `order:uniq:{order_id}` | 30 分钟 TTL | `SETNX order:uniq:12345 1 EX 1800` |

#### 秒杀库存扣减的 Lua 脚本（防止超卖）

```lua
-- 参数: KEYS[1] = flash_stock:product_id
-- 返回: 1 = 扣减成功, 0 = 库存不足
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock and stock > 0 then
    redis.call('DECR', KEYS[1])
    return 1
else
    return 0
end
```

> **为什么用 Lua**：Redis 在单线程模型下执行 Lua 脚本是原子操作，**不需要分布式锁**，性能极高（数十万 QPS）。  
> **如果改用分布式锁**（如 RedLock），会产生网络往返、锁竞争，导致 **秒杀成功率下降**。

### 3.3 消息队列（Kafka）

| Topic | 生产者 | 消费者 | 关键字段 |
|-------|--------|--------|----------|
| `danmu` | WebSocket Gateways | 弹幕消费服务、审计系统 | `room_id, user_id, content, ts` |
| `order_created` | 订单服务 | 订单支付服务、库存扣减服务、CRM | `order_id, user_id, items, total, ts` |
| `flash_order` | 秒杀服务 | 库存扣减（Redis）+ 订单落库（异步） | `order_id, user_id, product_id, qty, ts` |
| `payment_result` | 第三方支付回调 | 订单状态更新服务 | `order_id, pay_status, pay_time` |
| `statistic` | 各业务（观看、弹幕、成交） | 实时统计聚合（ClickHouse） | `metric, value, ts` |

> **分区**：为保证 **顺序性**，`order_created` 按 `order_id%N` 分区；`danmu` 按 `room_id%N` 分区，**每个直播间的弹幕都落在同一个分区**，消费者可以顺序写入 ClickHouse。

#### 秒杀下单业务流（伪代码）

```go
func FlashBuy(userID, productID, qty int) (orderID string, err error) {
    // 1. 幂等校验（防刷）
    token := fmt.Sprintf("flash:%d:%d", userID, productID)
    ok, _ := redis.SetNX(token, 1, 5*time.Minute)
    if !ok { return "", ErrRepeatBuy }

    // 2. 扣库存（Lua 原子）
    script := redis.NewScript(luaFlashDeduct)
    res, err := script.Run(redisClient, []string{fmt.Sprintf("flash_stock:%d", productID)})
    if err != nil || res.(int64) == 0 {
        return "", ErrOutOfStock
    }

    // 3. 生成订单号（全局唯一，使用 Snowflake）
    orderID = snowflake.Generate()
    // 4. 发送异步消息落库
    msg := FlashOrderMsg{OrderID: orderID, UserID: userID, ProductID: productID, Qty: qty}
    kafkaProducer.Produce("flash_order", msg)
    return orderID, nil
}
```

> **为什么要把落库放到 MQ**：秒杀高峰时 **写库是瓶颈**，把业务拆成「快速扣库存」 + 「异步持久化」两步，能把 2 k QPS 降到 **几百 QPS** 的 DB 写入压力。

## 第五步：详细组件设计

### 5.1 推流与 CDN 分发

1. **主播端**：使用 **OBS** 或自研 SDK，推 **RTMP** 到 **Ingress LB**（L4） → **nginx‑rtmp**。  
2. **转码**：FFmpeg 集群 **水平扩展**（Docker + Kubernetes），生成 **1080p/720p/480p** 三档 **HLS**（`.m3u8`）和 **FLV**（`*.flv`）流。  
3. **对象存储**：转码后的视频切片直接写入 **OSS**（S3 兼容），路径如 `live/{room_id}/{timestamp}.ts`。  
4. **CDN**：  
   * **边缘缓存** 10 s‑30 s（热点切片），使用 **分层缓存**（Edge → Regional → Origin）。  
   * **回源** 采用 **分块拉取**（Range），降低 Origin 负载。  
   * **多地域节点**（亚太、华北、华东）配置 **Anycast** DNS，用户就近解析。  

> **延迟控制**：  
> * **推流 → 转码** 采用 **硬件加速（NVENC）**，转码延迟 < 300 ms。  
> * **CDN Edge → 客户端** 使用 **HTTP/2** + **TCP Fast Open**，保证端到端 ≤ 1 s。

### 5.2 实时弹幕与礼物系统

1. **WebSocket Gateways**（水平扩容，基于 **Envoy** + **Lua**）  
   * 接收客户端弹幕，校验频率（防刷：每 2 s ≤ 5 条）。  
   * 把弹幕写入 **Kafka `danmu`**（持久化）并 **publish** 到 **Redis Pub/Sub**（低延迟广播）。  
2. **弹幕消费服务**（Kafka Consumer）  
   * 将弹幕写入 **ClickHouse**（实时统计）和 **MySQL**（审计）。  
   * 同时更新 **Redis ZSET**（最近 5 min）供新用户快速拉取。  
3. **礼物特效**  
   * 客户端发送 `gift` 消息 → 进入同样的 Kafka 流 → **消费服务** 更新用户积分、触发 UI 动画。  

> **防刷**：  
> * **IP + UID 频率限制**（Redis 计数器）  
> * **敏感词过滤**（离线词库 + Bloom Filter）  

### 5.3 商品、购物车、订单微服务

| 微服务 | 关键职责 | 主要技术栈 |
|--------|----------|------------|
| **Product Service** | 商品查询、库存查询（读） | Go/Java + MySQL + Redis Cache |
| **Cart Service** | 购物车增删改查（TTL） | Go + Redis Hash |
| **Order Service** | 订单创建、状态流转、支付回调 | SpringBoot + MySQL + Kafka |
| **Flash Sale Service** | 秒杀库存扣减、限流、排队 | Go + Redis Lua + Kafka |
| **Payment Service** | 接收第三方回调、幂等处理 | Node.js + MySQL + RabbitMQ (事务消息) |

#### 订单创建流程（普通商品）

1. **前端** → **API Gateway** → **Order Service**  
2. **Order Service**  
   * 检查购物车、调用 **Product Service** 验证库存（读缓存）。  
   * **Begin Transaction** → 写 **orders**、**order_items** → **扣减 MySQL 库存**（乐观锁 `WHERE stock=old_stock`）。  
   * **Commit** → 发送 **`order_created`** 到 **Kafka**（异步生成发货任务）。  
3. **Payment Service** 监听 `order_created`，生成 **预支付单**（调用支付宝/微信 SDK），返回 **支付链接**。  

> **幂等性**：所有写操作（下单、支付回调）先 **SetNX** 业务唯一键（如 `order:uniq:{order_id}`），防止重复消费。

### 5.4 实时统计与推荐

1. **统计**：各业务将关键指标（观看人数、弹幕量、成交额）写入 **Kafka `statistic`**。  
2. **ClickHouse** 实时消费，构建 **物化视图**（如 `live_viewers_daily`、`sales_minute`），供 **Grafana** 报表。  
3. **推荐**：仅提供 **数据入口**（用户行为日志、点击流），后续可接 **机器学习平台**（如 TensorFlow Serving）进行 **实时召回**。  

### 5.5 监控、日志、告警

| 维度 | 工具 | 示例指标 |
|------|------|----------|
| **时序监控** | Prometheus + Alertmanager | `http_request_duration_seconds`, `cpu_usage`, `redis_latency` |
| **日志** | ELK (Filebeat → Logstash → Elasticsearch) | 业务日志、异常堆栈 |

## 第六步：扩展性与高可用设计

### 6.1 横向扩容

| 组件 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **CDN Edge** | 增加节点、Anycast DNS | 带宽、并发连接数 |
| **推流服务器** | **Stateless**（nginx‑rtmp）+ **自动发现**（Consul） | 并发推流数 |
| **转码集群** | **容器化**（K8s）+ **水平 pod** | 每秒转码片段数 |
| **WebSocket Gateways** | **无状态**（只做转发）| 连接数（30 万） |
| **Redis Cluster** | **分片**（hash slot）| QPS > 200 k |
| **Kafka** | **分区**（topic partitions）| QPS > 300 k |
| **MySQL** | **读写分离** + **分库分表**| TPS > 5 k（写） |
| **ClickHouse** | **分布式表** + **副本** | 查询并发 > 10 k |

> **为什么要做到无状态**：只要服务本身不保存会话，就可以 **随时新增/剔除实例**，配合 **自动伸缩**（K8s HPA）实现 **弹性**。

### 6.2 高可用（HA）设计

#### 6.2.1 多可用区（AZ）部署

```
Region A (华北)                Region B (华东)
+-------------------+          +-------------------+
| LB (Anycast)      |<-------->| LB (Anycast)      |
|   └─> CDN Edge   |          |   └─> CDN Edge   |
|   └─> Service A  |          |   └─> Service B  |
+-------------------+          +-------------------+
```

* **数据同步**：  
  * MySQL 主从采用 **双活**（Region A 主库 → Region B 只读）+ **GTID** 同步。  
  * Redis 使用 **CRDT**（如 **Redis Enterprise**）或 **Active‑Active** 方案，保证跨 AZ 数据一致。  
  * Kafka 使用 **跨集群 MirrorMaker**，保证消息不丢失。  

* **流量切换**：  
  * **健康检查**（Probe）发现某 AZ CDN/服务不可用，**DNS TTL**（60 s）+ **Anycast** 自动切到存活 AZ。  
  * **客户端** 可通过 **WebSocket 心跳** 重连到最近可用节点。  

#### 6.2.2 故障检测 & 自动恢复

| 监控项 | 阈值 | 触发动作 |
|--------|------|----------|
| CDN Edge 错误率 > 5%（5 min） | 自动 **流量切回** 主节点 |
| Redis 单节点不可用 | **故障转移**（Sentinel） |
| Kafka 分区 ISR < 2 | **告警** + **自动扩容** |
| 推流服务器 CPU > 80% | **水平扩容**（K8s HPA） |
| 订单库事务回滚率 > 0.1% | **滚动回滚** + **流量降级** |

#### 6.2.3 跨地域灾备（Region Disaster Recovery）

1. **备份**：  
   * MySQL **全量备份**（Xtrabackup）+ **增量 binlog**，每日上传到 **对象存储**（多区域）。  
   * Redis **RDB + AOF** 双模备份。  
2. **故障演练**：每月进行 **Region 切换演练**，验证业务在 5 min 内切换完成。  

> **如果不做跨地域冗余**，单点故障（如北京机房网络中断）会导致 **全站不可用**，违背 99.99% SLA。

### 6.3 扩容到 100 万并发观众（追问 1）

| 资源 | 当前配置（30 万） | 目标（100 万） | 扩容方案 |
|------|------------------|----------------|----------|
| **CDN 带宽** | 68 Gbps | ≈ 230 Gbps | 增加 **CDN 边缘节点**（双倍覆盖），使用 **多 CDN 供应商**（如阿里云 + 腾讯云）做 **负载均衡** |
| **WebSocket 连接** | 30 万 | 100 万 | 拆分 **Gateway** 为 **5 层**（每层 20 万），使用 **TCP 复用**（QUIC）降低连接数 |
| **Redis Cluster** | 3 shard | 10 shard | **水平扩容**（每 shard 8 GB 内存），使用 **Key Hash Tag** 保证热点键分布均匀 |
| **Kafka** | 12 分区 | 40 分区 | 增加 **topic partitions**，每台 **Broker** 8 核 CPU、64 GB 内存，扩到 **15 台** |
| **MySQL** | 2 主 + 2 从（每库 1 TB） | 4 主 + 8 从（每库 2 TB） | **分库**（业务按 region、category）+ **GTID** 多主复制 |
| **转码** | 5 节点（GPU） | 20 节点 | **容器化 GPU**（K8s）+ **自动伸缩**（基于推流数） |

> **关键点**：**瓶颈从 CDN → 连接 → 缓存 → 消息队列 → 数据库**，逐层放大容量，同时 **保持无状态**，确保每一层都能独立横向扩容。

### 6.4 秒杀防超卖（追问 2）

| 方案 | 步骤 | 优点 | 缺点 |
|------|------|------|------|
| **Redis Lua 脚本** | 1. `SETNX` 幂等 token <br>2. `EVAL` 原子扣库存 <br>3. 成功则写入 Kafka | 超高 QPS（>100 k)，**原子性**，无需分布式锁 | 只能保证 **单机** 原子，若库存数据分片需自行保证键分布 |
| **分布式锁（RedLock）** | 1. 加锁 <br>2. 检查库存 <br>3. 扣减 <br>4. 释放锁 | 通用，适用于多资源事务 | 网络往返、锁竞争导致 **吞吐下降**，在高并发下容易出现 **锁超时** |
| **库存预减 + MQ** | 1. 预先把库存写入 **Redis** <br>2. 扣减后 **异步落库** <br>3. MQ 消费失败重试 | 高并发削峰，**可靠落库** | 需要 **补偿机制**（库存回滚）避免异常导致库存不一致 |
| **乐观锁（MySQL）** | 1. `UPDATE product SET stock = stock - 1 WHERE id = ? AND stock > 0` <br>2. 检查受影响行数 | 简单、事务一致 | MySQL 写入瓶颈，**不适合 2 k QPS+** 的秒杀高峰 |

> **推荐组合**：**Redis Lua 扣库存 + Kafka 异步持久化**。  
> *Redis* 负责 **毫秒级响应**，*Kafka* 负责 **最终一致性**，即使出现 **Redis 故障**，也可以从 **Kafka** 重放来恢复库存。

### 6.5 跨地域 CDN 故障容错（追问 3）

1. **多 CDN 供应商**：在 DNS 解析阶段返回 **多组 CNAME**（如 `cdn1.example.com`, `cdn2.example.com`），客户端随机挑选。  
2. **健康探测**：每个 CDN 节点每 5 s 发送 **心跳** 到 **监控中心**；若错误率 > 5% 连续 3 次，标记为 **不可用**。  
3. **流切换**：  
   * **Edge 代理层**（如 Cloudflare Workers）在检测到节点故障后，**动态改写 URL** 指向备份 CDN。  
   * 客户端 **HLS** 支持 **多 URL**（`#EXT-X-MEDIA`），播放器自动切换。  
4. **回源容灾**：Origin 服务器部署在 **双活**（北京、上海），使用 **对象存储跨区域复制**，确保切换后仍能拉取切片。  
5. **监控告警**：在 **Grafana** 中绘制 **CDN 错误率、RTT**，若超过阈值，触发 **PagerDuty** 报警并自动 **启动流量切换脚本**。  

> **不做多 CDN** 时，一旦该 CDN 节点宕机，所有观众的播放请求都会 502/504，导致 **观看体验瞬间崩溃**。多 CDN + 动态切换可把影响范围限制在 **几秒钟**。

## 第七步：常见面试追问与回答

### 7.1 容量从 30 万 → 100 万观众的扩容思路（已在第 6.3 小结）

**核心思路**：**把每一层的瓶颈放大 3‑4 倍**，并 **确保每层都是无状态、水平可扩**。  
- **CDN**：多供应商、增加边缘节点、使用 Anycast + 短 TTL DNS。  
- **WebSocket**：拆分网关、使用 QUIC/HTTP3 减少连接数。  
- **缓存**：Redis Cluster 规模化、分片键设计。  
- **消息**：Kafka 分区数提升、Broker 数量翻倍。  
- **存储**：分库分表、双活 MySQL、ClickHouse 分布式表。  

### 7.2 秒杀防超卖的实现细节（已在第 6.4 小结）

- **Redis Lua** 原子扣库存 → **Kafka** 异步持久化  
- **幂等 token** 防刷  
- **库存回滚**：若 MQ 消费失败，使用 **补偿消息** 把 Redis 库存 +1，保证最终一致性。  

### 7.3 CDN 节点全失效时的容错方案（已在第 6.5 小结）

- **多 CDN + 动态切换**（DNS + Edge Worker）  
- **双活 Origin + 对象存储跨区域复制**  
- **监控+自动化流量切换脚本**，在 5 s 内完成切换。  

### 7.4 为什么选 Kafka 而不是 RabbitMQ？

| 特性 | Kafka | RabbitMQ |
|------|-------|----------|
| **吞吐** | 高（百万级 QPS） | 中等 |
| **持久化** | 基于磁盘顺序写，成本低 | 基于磁盘随机写，成本高 |
| **消费者模型** | 拉取（pull），天然支持分区消费 | 推送（push），难以控制消费速率 |
| **扩容** | 通过增加 Partition/Broker 线性扩展 | 需要重新绑定队列 |
| **使用场景** | 实时日志、弹幕、订单异步落库 | 事务性、可靠投递（低 QPS） |

秒杀、弹幕等**高并发、低延迟**场景更适合 **Kafka**。

### 7.5 如何保证用户支付回调的幂等？

1. **业务唯一键**：`order:pay:{order_id}`，`SETNX` 幂等锁（TTL 30 min）。  
2. **状态机**：订单状态只能从 `WAIT_PAY` → `PAID` → `FINISHED`，不允许回退。  
3. **日志**：所有回调写入 **MySQL** + **Kafka**，便于审计和重放。  

### 7.6 为什么要把弹幕先写入 Kafka 再落库？

- **实时性**：弹幕需要毫秒级推送，直接写 DB 会导致 **写放大** 与 **锁竞争**。  
- **持久化**：Kafka 持久化，保证弹幕不丢失，后期可以 **回放**（审计、热榜）。  
- **解耦**：弹幕消费（统计、存档）与推送（WebSocket）相互独立，互不影响。

## 心得与反思

### 1️⃣ 本题最难的设计决策

| 决策 | 思考过程 |
|------|----------|
| **秒杀库存的实现** | 初始想法是直接在 MySQL 用乐观锁，但在 2 k QPS+的高并发下 MySQL 写入成为瓶颈。随后评估了 **Redis 原子脚本**、**分布式锁**、**预减+MQ** 三种方案，综合考虑 **吞吐、可靠性、实现成本**，最终选定 **Redis Lua + Kafka**，并补上 **幂等 token**、**补偿回滚**，确保即使 Redis 故障也能从 Kafka 恢复。 |
| **跨地域 CDN 容错** | 只靠单 CDN 的高可用会在极端网络故障时失效。考虑到 **用户对观看体验的极端敏感**（≤ 1 s 延迟），决定采用 **多 CDN + 动态流切换**，并在 **DNS**、**Edge Worker**、**播放器** 多层做容错。这个方案牵涉到 **运维、成本、监控**，所以在答题时需要阐明每一步的实现细节与监控点。 |

### 2️⃣ 新手最容易犯的错误

| 错误 | 解释 |
|------|------|
| **把所有业务都放到单体 MySQL** | 这样在 2 k QPS、30 万并发的场景下会出现 **锁竞争、慢查询、单点宕机**。正确做法是 **读写分离 + 分库分表**，把热点业务（秒杀、订单）拆到 **Redis/Kafka**。 |
| **忽视弹幕/礼物的实时性**，直接写库 | 弹幕频率极高（150 k QPS），若每条都落库会导致 **磁盘 I/O 爆炸**。应使用 **内存缓存 + MQ**，先推送再异步持久化。 |
| **只关注“高并发”，不考虑“高可用”** | 只提升 QPS 能力而不做容错，单点故障仍会导致系统不可用，违背 99.99% SLA。需要在每层加入 **冗余、健康检查、自动切换**。 |
| **忘记幂等** | 支付回调、秒杀下单等外部回调可能被重试，若没有幂等保护会导致 **重复扣款、超卖**。务必在业务入口使用 **唯一键 + SetNX**。 |

### 3️⃣ 学习建议与可延伸方向

| 建议 | 说明 |
|------|------|
| **系统思维**：先画出 **业务流程图**，再划分「读」与「写」两类，明确每类的 **性能瓶颈** 与 **一致性要求**。 |
| **分布式缓存**：深入学习 **Redis Cluster**、**Lua 脚本**、**事务**，掌握 **原子操作** 与 **热点键分布**。 |
| **消息队列**：熟悉 **Kafka** 的 **分区、ISR、消费者组**，了解 **Exactly‑Once** 的实现方式。 |
| **容器化与自动伸缩**：动手搭建 **K8s**，实践 **水平 Pod 自动伸缩（HPA）**，体会 **无状态服务** 的好处。 |
| **监控告警**：学习 **Prometheus** 抓取业务指标、**Grafana** 可视化、**Alertmanager** 自动化。 |
| **性能压测**：使用 **JMeter / Locust** 对 **WebSocket、REST、Kafka** 进行压测，验证 **QPS、延迟、错误率** 是否符合预估。 |
| **灾备演练**：每月一次 **Region 切换** 演练，培养 **故障定位 + 快速恢复** 能力。 |
| **高级主题**：了解 **分布式事务（TCC、Saga）**、**服务网格（Istio）**、**统一日志（ELK）**，为后续更复杂的业务（如订单全链路）打基础。 |

---

> **结语**：系统设计不是一次性写完的「图」或「文档」，而是 **不断抽象、拆分、验证** 的过程。把业务拆成「读」和「写」两类，先确保 **高并发的读**（CDN、缓存）和 **强一致的写**（事务、幂等），再围绕 **扩展、容灾、监控** 逐层加固，就能从「能跑」走向「能稳跑、能伸缩、能容错」。祝你在面试中自信满满、条理清晰，早日成为独当一面的系统架构师！ 🚀
