# 微博（Weibo）系统设计面试题

## 1. 题目背景
微博是一个面向大众的实时社交媒体平台，用户可以发布文字、图片、视频等内容并关注他人的动态。系统需要在海量用户并发下，保证内容的快速分发、检索和交互体验。

## 2. 面试场景设定
> **面试官**：  
> “假设我们要从零开始设计一个类似微博的微型博客系统，核心目标是支持每日数亿活跃用户发布和阅读短内容。请你先从整体架构出发，说明系统的主要组成模块以及它们之间的交互方式。”

## 3. 功能性需求
| 编号 | 功能描述 |
|------|----------|
| 1 | **用户注册/登录**：支持手机号、邮箱或第三方 OAuth 登录，完成身份认证与会话管理。 |
| 2 | **发布微博**：用户可以发布文字（≤ 2000 字）并可附带图片/视频，支持编辑、删除、草稿保存。 |
| 3 | **关注/粉丝关系**：用户可以关注其他用户，系统维护关注链路并支持批量关注/取关。 |
| 4 | **时间线（首页）**：为每个用户实时生成关注人的最新微博列表，支持下拉刷新、分页加载。 |
| 5 | **互动**：对微博进行点赞、评论、转发，且这些互动可以被实时展示在时间线和微博详情页。 |
| 6 | **通知与消息**：当关注的人发布新微博、点赞/评论/转发你的微博时，向用户推送站内通知。 |

## 4. 非功能性需求（关键指标）

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 5 × 10⁸（5 亿） | 目标覆盖全国主流用户。 |
| **QPS（读请求）** | 2 × 10⁵（200 k） | 主要为时间线读取、微博详情、评论列表等。 |
| **QPS（写请求）** | 5 × 10⁴（50 k） | 包括发微博、点赞、评论、转发等写操作。 |
| **平均响应延迟** | ≤ 100 ms（95% 请求） | 前端页面交互的感知延迟。 |
| **可用性** | 99.9%（每月累计宕机不超过 43.2 分钟） | 业务对持续可用要求较高。 |
| **存储规模** | 10 PB（原始媒体 + 索引） | 估算 5 亿用户 × 平均 10 GB 数据（包括图片、视频、日志等）。 |

> 注：上述指标为 **初始规模**，面试时可讨论随业务增长的扩展策略。

## 5. 系统边界

**本题范围（需设计）**
- 微博的核心发布、阅读、互动、关注、时间线生成与推送。
- 用户身份认证、会话管理以及基本的安全防护（防刷、验证码）。
- 高并发读写的存储、缓存、消息队列、负载均衡等基础设施。
- 数据一致性方案、容错与灾备（多活/主备）。

**不在本题范围（可忽略）**
- 广告投放系统、商业营销、品牌合作等增值业务。
- 实时直播、短视频平台（如抖音）的大规模转码与 CDN。
- 私信/聊天、群组、活动报名等功能。
- 内容审核的机器学习模型实现细节（只需说明会有审核层）。
- 第三方开放平台 API（只需说明可对外提供）。

1. **先把需求拆成最小可运行（MVP）**：只保留最核心的「用户登录 → 发微博 → 读取时间线」三个功能，快速画出单体结构。

3. **逐层演进**：在 MVP 基础上，逐步加入 **缓存、消息队列、分库分表、读写分离、容灾**，每一步都解释「为什么需要」以及「如果不加会出现什么」。  
4. **把每个功能映射到具体的微服务**（用户、微博、关系、时间线、互动、通知），并用文字画出**高层架构图**。  
5. **细化存储模型**：列出主要业务实体（User、Weibo、Follow、Like、Comment 等）的字段、主键、索引、分片键。  
6. **设计 API**：从客户端角度出发，列出关键的 RESTful/GRPC 接口，说明请求/响应、幂等性、鉴权方式。  
7. **组件选型**：针对每类需求（高并发读、写放大、全局唯一 ID、异步通知）挑选成熟的开源或商业技术，并说明优缺点。

9. **准备面试追问**：提前思考时间线实现、热点计数、跨地域复制等常见“刀口”问题，给出简洁而有深度的答案。

## ## 第一步：理解需求与规模估算  

### 1. 功能需求梳理  

| 编号 | 功能 | 核心子功能 | 是否必选（MVP） |
|------|------|------------|-----------------|
| 1 | 用户注册/登录 | 手机/邮箱/第三方 OAuth，Session / Token | ✅ |
| 2 | 发布微博 | 文字 ≤2000，图片/视频上传，草稿 | ✅ |
| 3 | 关注/粉丝 | 关注、取关、批量操作 | ✅ |
| 4 | 时间线（首页） | 拉取关注人的最新微博，分页、下拉刷新 | ✅ |
| 5 | 互动 | 点赞、评论、转发 | ✅ |
| 6 | 通知 | 新微博、点赞、评论、转发的站内提醒 | ✅ |

> **MVP（最小可运行系统）**：只要实现 1、2、3、4 四个功能，即可完成「用户 → 发微博 → 看关注人动态」的闭环。后面的互动、通知属于**增强功能**，在后期逐步上线。

### 2. 非功能性需求换算  

| 指标 | 业务意义 | 计算方式 | 结果 |
|------|----------|----------|------|
| **DAU** | 每日活跃用户数 | 已给 5×10⁸ | 5 亿 |
| **写 QPS** | 发微博、点赞、评论、关注等 | 50 k /s | 5 万次写 |
| **读 QPS** | 时间线、微博详情、评论列表 | 200 k /s | 20 万次读 |
| **峰值**（假设 2×峰值） | 高峰时 1 小时内 QPS 可能翻 2 倍 | 400 k 读 / 100 k 写 | 需要留足弹性 |
| **存储** | 文字 + 媒体 | 10 PB 初始 | 约 10⁷ GB |
| **响应时延** | 前端交互感知 | 95% ≤ 100 ms | 需要缓存 + 近端部署 |
| **可用性** | 99.9% → MTBF ≈ 43.2 min/月 | 业务不可接受长时间宕机 | 多活 + 自动故障转移 |

### 3. 关键瓶颈点初步定位  

| 场景 | 可能的瓶颈 | 产生原因 |
|------|------------|----------|
| **写入微博** | 单条写入后同步到所有关注者的时间线（写放大） | 若采用推模型，热门用户关注者数可能达千万 |
| **点赞/评论计数** | 高并发写入导致单行热点 | 直接更新关系型库的计数字段会产生锁竞争 |
| **时间线读取** | 大量并发读，单点 DB 读取压力 | 需要缓存、分片或预计算 |
| **媒体存储** | 大文件上传、下载带宽 | 需要对象存储 + CDN |
| **会话/鉴权** | 鉴权中心单点故障 | 需要分布式会话或 JWT |

## ## 第二步：高层架构设计  

下面先给出 **文字版的高层架构图**（可自行在白板上画成方框），随后解释每块的职责与交互。

```
+-------------------+        +-------------------+        +-------------------+
|   客户端 (Web/M    | <---> |   API Gateway /   | <---> |   Load Balancer   |
|   iOS/Android)    |        |   Edge Layer      |        +-------------------+
+-------------------+        +-------------------+                |
                                   |                          |
                 +-----------------+-------------------+      |
                 |                 |                   |      |
        +--------v-----+   +-------v------+   +--------v------+ |
        | Auth Service |   | User Service |   |  Follow Service| |
        +--------------+   +--------------+   +---------------+ |
                 |                 |                   |      |
                 +--------+--------+--------+----------+      |
                          |                 |                 |
                +---------v-----+   +-------v------+   +------v-------+
                |  Weibo Service|   | Timeline    |   | Interaction  |
                | (Write Path)  |   | Service     |   | Service      |
                +---------------+   +------------+   +--------------+
                          |                 |                 |
               +----------v----------+      |      +----------v----------+
               |   Message Queue    |<-----+------|   Cache (Redis)    |
               +--------------------+             +--------------------+
                          |                                 |
                +---------v----------+          +-----------v-----------+
                |   Storage Service  |          |   Search Service     |
                | (MySQL / TiDB)     |          | (Elasticsearch)     |
                +--------------------+          +----------------------+
                          |
                +---------v----------+
                |   Object Store      |
                |   (OSS / S3)        |
                +---------------------+
```

### 关键组件解释  

| 组件 | 作用 | 选型理由 |
|------|------|----------|
| **API Gateway / Edge** | 统一入口，做限流、鉴权、协议转换、灰度发布 | Kong / Nginx + Lua；能快速扩容、支持插件 |
| **Load Balancer** | 将请求分发到后端服务实例 | L4（硬件）或 L7（软件）均可，确保会话亲和性 |
| **Auth Service** | 登录、Token（JWT）签发、会话校验、验证码、防刷 | 使用 **JWT**（无状态）+ **Redis** 保存黑名单；验证码服务选 **Aliyun SMS** / **Geetest** |
| **User Service** | 用户信息 CRUD、头像、个人设置 | 关系型 DB（MySQL）+ Cache |
| **Follow Service** | 维护 **关注 / 粉丝** 关系，提供 **批量关注/取关** 接口 | 关注关系是 **稀疏图**，使用 **HBase / Cassandra** 或 **MySQL 分表**（水平分片） |
| **Weibo Service** | 微博的写入、编辑、删除、草稿；生成唯一 ID（Snowflake） | 写路径需要 **高可用**、**低延迟**，使用 **分布式事务**（TCC）或 **可靠消息** |
| **Timeline Service** | **时间线生成**（推/拉模型混合），对外提供分页读取接口 | 结合 **Redis**（热点用户的预推） + **MySQL**（冷数据） |
| **Interaction Service** | 点赞、评论、转发的写入、计数、去重 | 计数使用 **Redis + DB 异步落库**，防止热点写 |
| **Message Queue** | 跨服务解耦、异步处理（写入后推送时间线、生成通知） | Kafka（高吞吐） + Pulsar（多租户）均可 |
| **Cache (Redis)** | 读热点缓存：用户信息、关注列表、时间线、计数 | 支持 **LRU**、**TTL**，在热点缓存失效时回源 DB |
| **Storage Service (MySQL/TiDB)** | 业务核心关系数据的持久化（用户、微博、评论、关系） | 支持 **水平分片**、**强一致性**，TiDB 原生分布式 |
| **Search Service (Elasticsearch)** | 全文检索、热点榜单、模糊搜索 | 通过 **倒排索引** 支持关键词搜索、过滤 |
| **Object Store** | 大文件（图片、视频）存储，配合 **CDN** 加速 | 阿里云 OSS / AWS S3，具备 **分块上传**、**生命周期**管理 |

> **为什么要把这些功能拆成独立服务？**  
> - **职责单一**：每个服务只专注自己业务，代码更易维护。  
> - **水平扩容**：热点服务（时间线、互动）可以单独扩容，不会拖慢其它模块。  
> - **容错隔离**：某服务故障（如搜索）不会导致整个系统不可用。  

如果直接把所有业务放进单体 SpringBoot 项目，随着流量增长会出现 **单点瓶颈、部署困难、代码耦合** 等问题，难以满足 **200k QPS** 的读需求。

## ## 第三步：数据库设计  

### 1. 总体原则  

| 需求 | 采用的数据库 | 说明 |
|------|---------------|------|
| 关系型事务（用户、微博、评论） | **TiDB**（MySQL 协议的分布式 NewSQL） | 支持强一致性事务、水平分片、自动水平扩容 |
| 稀疏图（关注/粉丝） | **Cassandra** 或 **HBase** | 按用户 ID 分区，读取关注列表快，写入可扩展 |
| 计数热点（点赞、转发） | **Redis**（内存计数） + **TiDB**（持久化） | 写入 Redis，定时批量落库，降低 DB 写放大 |
| 全文搜索 | **Elasticsearch** | 关键词、标签搜索 |
| 大文件（图片、视频） | **对象存储（OSS/S3）** + **CDN** | 只保存 URL，避免 DB 存储大对象 |

> **如果只使用单一 MySQL**：在 5 × 10⁸ 用户、10 PB 存储的规模下，单库写入、查询会产生 **锁竞争、磁盘 I/O 瓶颈**，并且水平扩容成本极高。  

### 2. 关键表结构（以 TiDB 为例）  

#### 2.1 User 表  

| 字段 | 类型 | 说明 | 索引 |
|------|------|------|------|
| user_id | BIGINT (PK, auto‑increment via Snowflake) | 全局唯一 ID | 主键 |
| mobile | VARCHAR(20) | 手机号，唯一 | 唯一索引 |
| email | VARCHAR(64) | 邮箱，唯一 | 唯一索引 |
| password_hash | CHAR(64) | 加盐后 SHA256 | - |
| nickname | VARCHAR(32) | 显示名 | - |
| avatar_url | VARCHAR(256) | 对象存储 URL | - |
| created_at | TIMESTAMP | 注册时间 | - |
| status | TINYINT | 0 正常 / 1 冻结 | - |

#### 2.2 Weibo（微博）表  

| 字段 | 类型 | 说明 | 索引 |
|------|------|------|------|
| weibo_id | BIGINT (PK) | Snowflake 全局唯一 ID | 主键 |
| user_id | BIGINT | 发微博用户 | 单列索引（查询用户微博） |
| content | TEXT | 文本内容（≤2000） | - |
| media_urls | JSON | [{type:"image",url:"..."}, …] | - |
| created_at | TIMESTAMP | 发布时间 | - |
| status | TINYINT | 0 正常 / 1 删除 / 2 草稿 | - |
| like_cnt | INT | 点赞数（冗余） | - |
| comment_cnt | INT | 评论数（冗余） | - |
| repost_cnt | INT | 转发数（冗余） | - |

> **冗余计数**：为了读取时间线时不必再聚合，直接存放在微博记录中，由 **Interaction Service** 负责异步更新。  

#### 2.3 Follow（关注关系）表（Cassandra）  

| 列族 | 主键 | 说明 |
|------|------|------|
| follow_by_user | **partition_key = user_id**, clustering_key = followee_id | “我关注的用户列表” |
| follower_of_user | **partition_key = user_id**, clustering_key = follower_id | “我的粉丝列表” |

- **写入**：一次写两条（双向），利用 **Batch** 保证原子性。  
- **查询**：分页读取关注列表（LIMIT + token） → 用于时间线生成。  

#### 2.4 Like 表  

| 字段 | 类型 | 说明 |
|------|------|------|
| user_id | BIGINT | 点赞者 |
| weibo_id | BIGINT | 被点赞微博 |
| created_at | TIMESTAMP | 点赞时间 |

- **唯一约束**：(user_id, weibo_id) 防止重复点赞。  
- **计数**：写入时先更新 Redis `weibo:like:{weibo_id}`，异步落库。  

#### 2.5 Comment 表  

| 字段 | 类型 | 说明 |
|------|------|------|
| comment_id | BIGINT (PK) | Snowflake |
| weibo_id | BIGINT | 所属微博 |
| user_id | BIGINT | 评论者 |
| parent_id | BIGINT | 回复的评论 ID（0 为根评论） |
| content | TEXT | 评论内容 |
| created_at | TIMESTAMP | 时间 |
| like_cnt | INT | 点赞计数（冗余） |

#### 2.6 Timeline（预推）表（Redis Sorted Set）  

- **Key**：`timeline:user:{user_id}`  
- **Member**：`weibo_id`  
- **Score**：`created_at`（时间戳）  

> **为什么用 Redis Sorted Set**：能够在 **O(log N)** 时间完成 **时间线分页**（ZRANGE）以及 **删除过期微博**（ZREMRANGEBYSCORE）。  

### 3. 分片与路由  

| 表 | 分片键 | 分片数 | 说明 |
|----|--------|--------|------|
| User、Weibo、Comment | **user_id**（或 weibo_id） | 128~256 | 基于 **Hash** 分片，均匀分布 |
| Follow | **user_id** | 256 | 按用户 ID 分区，读写热点均匀 |
| Like | **weibo_id** | 128 | 通过微博 ID 将热点集中到少数分片，配合热点分离（热点分片） |

TiDB 自动完成 **水平分片**，我们只需要设置 **分片键**（user_id）并开启 **分区表**（range/hash）。

## ## 第四步：核心 API 设计  

下面列出 **客户端最常用的 8 条 API**，使用 **RESTful**（也可以改成 **gRPC**）示例。每条都标明请求、响应、鉴权、幂等性。  

| 编号 | HTTP 方法 | 路径 | 功能 | 请求体（JSON） | 响应体（JSON） | 鉴权 | 幂等性 |
|------|-----------|------|------|----------------|----------------|------|--------|
| 1 | POST | /api/v1/auth/login | 登录（手机号/邮箱+密码） | `{ "login": "13800138000", "password": "xxxx" }` | `{ "access_token":"jwt...", "refresh_token":"..." }` | ❌（公开） | ✅（同一凭证返回相同 token） |
| 2 | POST | /api/v1/auth/register | 注册 | `{ "mobile":"...", "email":"...", "password":"...", "captcha":"..." }` | `{ "user_id":12345 }` | ❌ | ✅ |
| 3 | POST | /api/v1/weibo | 发布微博 | `{ "content":"Hello", "media_urls":[...] }` | `{ "weibo_id":987654321 }` | ✅（JWT） | ✅（幂等：若 client 重试，用 client‑generated request_id 去重） |
| 4 | GET | /api/v1/weibo/{weibo_id} | 查看微博详情 | - | `{ "weibo_id":..., "user":{}, "content":"", "media":[], "stats":{...} }` | ✅ | ✅ |
| 5 | POST | /api/v1/follow/{target_user_id} | 关注 | `{ "action":"follow" }` | `{ "result":"ok" }` | ✅ | ✅（幂等） |
| 6 | DELETE | /api/v1/follow/{target_user_id} | 取关 | - | `{ "result":"ok" }` | ✅ | ✅ |
| 7 | GET | /api/v1/timeline | 拉取首页时间线 | `?cursor=12345&limit=20` | `{ "list":[{weibo...}], "next_cursor":67890 }` | ✅ | ✅ |
| 8 | POST | /api/v1/weibo/{weibo_id}/like | 点赞 | `{ "action":"like" }` | `{ "like_cnt":101 }` | ✅ | ✅（幂等：重复点赞返回相同计数） |

### 关键实现细节  

1. **统一鉴权**：所有需要登录的 API 必须在 **Gateway** 层校验 **JWT**，并把 `user_id` 注入到请求 Header（X-User-Id）。  
2. **幂等性**：对写操作（发微博、点赞、关注）使用 **Idempotency-Key**（客户端生成 UUID）或 **业务键**（如 `user_id+weibo_id`），在后端做 **去重**（利用 Redis SET）。  
3. **分页**：时间线采用 **基于时间戳的游标（cursor）**，而不是传统的 page number，避免数据倾斜和“漏页”。  
4. **错误码**：统一返回 `code`、`msg`、`data` 结构，便于前端统一处理。

## ## 第五步：详细组件设计  

### 5.1 鉴权与安全  

| 步骤 | 说明 | 技术 |
|------|------|------|
| **注册** | 校验手机号/邮箱唯一性 → 发送验证码 → 保存 `password_hash`（bcrypt） | Spring Security + Redis（验证码） |
| **登录** | 检查密码 → 生成 **JWT**（签名 HS256） → 返回 Access+Refresh Token | JWT + RSA（可选） |
| **会话** | 前端把 Access Token 放在 `Authorization: Bearer xxx`，后端每次校验；Refresh Token 用于短时间内换新 Access | Redis 保存 Refresh Token 黑名单，支持单点下线 |
| **防刷** | 登录、发微博、点赞等高频接口加 **滑动窗口限流**（Redis Lua） | 令牌桶/漏桶 |
| **审计日志** | 所有写操作记录到 **Kafka**，供离线风控、审计 | Kafka → Hadoop/ClickHouse |

### 5.2 ID 生成（全局唯一）  

- 使用 **Snowflake**（Twitter）或 **Leaf**（美团）实现 **64bit** 长整型 ID。  
- **结构**：`timestamp(41bit) + datacenterId(5bit) + workerId(5bit) + sequence(12bit)`。  
- **优点**：递增、局部有序，便于 **时间线** 排序、**分片**（按时间段切分）。  

### 5.3 媒体存储  

1. **上传**：客户端直传 OSS/S3（获取预签名 URL），避免经过业务服务器，减轻带宽压力。  
2. **压缩/转码**：后台异步消费 **Kafka** 中的 `media_upload` 事件，调用 **FFmpeg**/第三方转码服务生成不同分辨率，存入同一对象路径。  
3. **CDN**：配置全局加速节点，缓存常访问的图片/短视频，降低对象存储读请求。  

### 5.4 关注关系与写放大治理  

#### 5.4.1 推模型（热点用户）  

- **思路**：当用户 A 发微博，系统把微博 ID 推送到 **所有关注者** 的 Timeline 缓存（Redis Sorted Set）。  
- **问题**：若 A 的粉丝数为 10M，单次写入会产生 10M 条写操作，产生 **写放大**。  

#### 5.4.2 拉模型（大多数普通用户）  

- **思路**：用户读取时间线时，后台查询 **关注列表**，从 **Weibo 表** 按时间倒序获取最近 N 条。  
- **优势**：写操作只写一次，读时按需聚合，适合 **粉丝数 ≤ 10k** 的用户。  

#### 5.4.3 **混合方案**（实际生产常用）  

| 用户类型 | 采用模型 | 说明 |
|----------|----------|------|
| **普通用户**（粉丝 ≤ 10k） | **拉** | 读取时实时聚合，降低写放大 |
| **明星/大V**（粉丝 > 10k） | **推** + **热点分片** | 预先把微博写入热点 Timeline（Redis Cluster），并在 DB 中保留原始记录供回源 |
| **普通用户的热点时间线** | **Hybrid Cache**：先查 Redis，命中则返回，未命中则查询 DB 并写回缓存 | 兼顾实时性与缓存命中率 |

**实现细节**  

- **Kafka** 主题 `weibo-publish`：Weibo Service 发送微博 ID、作者 ID。  
- **Timeline Worker** 订阅该主题，判断作者粉丝数是否超过阈值（如 10k）。  
  - 若 **小于阈值** → 不推，直接返回成功。  
  - 若 **大于阈值** → 批量读取粉丝列表（分片），对每 1000 条粉丝做一次 `ZADD`（pipeline），写入对应用户的 Redis Timeline。  
- **热点分片**：把所有“大V”用户的 Timeline 放在单独的 Redis Cluster（专用机器），防止热点导致整个缓存失效。  

### 5.5 点赞/评论计数的热点治理  

1. **写入流程**  
   - 客户端调用 `POST /weibo/{id}/like` → API Gateway → Interaction Service。  
   - Interaction Service 首先在 **Redis** `INCR weibo:like:{weibo_id}`，返回最新计数。  
   - 同步向 **Kafka** `like-event` 发送事件 `{user_id,weibo_id,ts}`。  

2. **落库**  
   - **定时批任务**（每 5 秒）或 **消费组**（Kafka Consumer）读取 `like-event`，聚合同一个 `weibo_id` 的增量，执行 **UPDATE weibo SET like_cnt = like_cnt + inc WHERE weibo_id = ?**。  
   - 通过 **乐观锁**（version）或 **行锁** 防止并发冲突。  

3. **读侧**  
   - 微博详情页直接查询 Redis 中的计数（若不存在则回源 DB），确保 **实时**。  

> **如果直接在 MySQL 上 `UPDATE` 点赞计数**，热点微博的同一行会产生 **行锁竞争**，导致 QPS 下降，甚至出现死锁。使用 **缓存+异步落库** 能把 1000+ QPS 的写请求压缩成每秒一次的批量更新。  

### 5.6 消息队列与异步流程  

| 场景 | 生产者 | Topic | 消费者 | 目的 |
|------|--------|-------|--------|------|
| 微博发布 | Weibo Service | `weibo-publish` | Timeline Worker、Search Sync、Notify Worker | 推送时间线、同步到 ES、生成站内通知 |
| 点赞/评论 | Interaction Service | `like-event`、`comment-event` | Stats Worker、Notify Worker | 计数落库、发送通知 |
| 媒体上传 | 前端（直接 OSS） | `media-upload` | Media Process Worker | 转码、生成缩略图、更新 Weibo 表 |
| 审核日志 | 全部服务 | `audit-log` | Log Analyzer | 风控、审计、异常检测 |

- **Kafka**：高吞吐、分区可扩展、支持 **Exactly‑once**（开启事务）保证不丢失。  
- **消费者组**：每个业务（Timeline、Search、Notify）单独部署多个实例，实现 **水平扩容**。  

### 5.7 缓存策略  

| 缓存对象 | Key 示例 | TTL | 失效策略 |
|----------|----------|-----|----------|
| 用户信息 | `user:{user_id}` | 24h | LRU，热点用户永不失效 |
| 关注列表 | `follow:{user_id}` | 12h | 定期全量刷新（Batch） |
| 微博详情 | `weibo:{weibo_id}` | 6h | 写入后主动刷新 |
| 时间线 | `timeline:user:{user_id}` | 1h（滚动） | 定时清理 30 天前的微博 |
| 点赞计数 | `weibo:like:{weibo_id}` | 0（永不过期） | 计数变动时不刷新 TTL |
| 评论列表 | `comments:{weibo_id}` | 1h | 追加写入后局部更新 |

> **缓存失效的一致性**：  
> - **写后读**：写完微博后立即把微博写入 Redis Timeline（推模型）或把微博写入 DB 并 **删除** 相关缓存（拉模型），保证下次读取能命中最新数据。  
> - **双写**：对热点计数（like）采用 **写进缓存 + 异步落库**，不需要同步失效。  

### 5.8 搜索服务  

- **写入**：Weibo Service 发布微博后，同步发送到 **Kafka `weibo-sync`**，Search Service 消费后调用 **Elasticsearch Bulk API** 把 `weibo_id, content, tags, user_id, created_at` 建立倒排索引。  
- **查询**：前端搜索接口 `GET /search?q=xxx&page=1&size=20` → Search Service → ES。  
- **热点过滤**：可以在 ES 上加 **热度字段**（like_cnt）做排序，提升热点内容曝光。  

### 5.9 监控、告警、自动伸缩  

| 监控维度 | 工具 | 报警阈值示例 |
|----------|------|--------------|
| **系统层** | Prometheus + Grafana | CPU > 80% 持续 5min |
| **服务层** | Jaeger（分布式追踪） | 单接口 latency > 200ms |
| **业务指标** | ClickHouse 实时埋点 | QPS > 250k、错误率 > 0.5% |
| **缓存** | Redis Sentinel + Redis Exporter | 缓存命中率 < 70% |
| **消息队列** | Kafka Cruise Control | 消费延迟 > 30s |
| **存储** | TiDB Dashboard | 复制延迟 > 5s |

- **自动伸缩**：基于 **CPU/内存** 或 **自定义 QPS** 指标，使用 **K8s HPA**（Horizontal Pod Autoscaler）或 **阿里云弹性伸缩**。

## ## 第六步：扩展性与高可用设计  

### 6.1 读写分离 & 数据分片  

1. **读写分离**：TiDB 支持 **读写分离**，写请求落在 **TiDB PD + TiKV** 主节点，读请求走 **TiDB Server** 的 **Replica**。  
2. **分库分表**：  
   - **User/Weibo/Comment** 按 **user_id** hash 分到不同库（128），每库再水平拆分成 16 张表（range on weibo_id）。  
   - **Follow** 按 **user_id** hash 分片（Cassandra keyspace），保证 **单个分区的热点** 可通过 **节点扩容** 解决。  

### 6.2 多活与跨地域部署  

| 层级 | 方案 | 关键技术 |
|------|------|----------|
| **DNS** | Anycast + GSLB | 阿里云 SLB Global、Cloudflare |
| **前端** | 多地域 CDN | 静态页面、图片、JS |
| **业务层** | **双活**（北京、上海） | 数据采用 **双向同步**（TiDB Multi‑Region、Cassandra Multi‑DC） |
| **缓存** | **Redis Cluster** 跨地域复制（Active‑Active） | Dragonfly / Codis + CRDT |
| **消息队列** | **Kafka MirrorMaker 2**（跨 DC） | 保证同一 Topic 在不同地域都有副本 |
| **存储** | **对象存储跨区域复制**（OSS 多地域） | 读写最近地域，写入同步到其他区域（异步） |

- **故障切换**：当某地域的 TiDB 节点不可用，流量自动切到另一地域的 TiDB 只读副本，写请求通过 **全局事务管理**（分布式事务框架如 **Seata**）转发至主地域。  
- **RTO / RPO**：  
  - **RTO**（恢复时间）< 5 min（自动 DNS 切换 + 业务容错）  
  - **RPO**（数据丢失）≈ 0 s（同步复制）对业务关键表（User、Weibo），对非关键表（日志）可接受几秒延迟。  

### 6.3 容灾与备份  

- **全量快照**：TiDB 每日夜间做 **全量快照**（TiDB Backup），保留 30 天。  
- **增量日志**：通过 **Binlog**（TiDB Binlog）实时推送到 **OSS**，用于 Point‑in‑Time 恢复。  
- **灾备演练**：每季度进行一次 **跨地域灾备演练**，验证 RTO、RPO。  

### 6.4 防止单点故障（SPOF）  

| 组件 | 冗余方式 |
|------|----------|
| API Gateway | 多实例 + L4 LB |
| Auth Service | 多副本 + 共享 Redis |
| DB（TiDB） | PD + TiKV 多副本（3副本） |
| Cache（Redis） | 主从复制 + Sentinel / Cluster |
| MQ（Kafka） | 多 Broker + 多副本（3） |
| 对象存储 | 多 AZ 同城复制 + 跨地域镜像 |
| 监控系统 | 多实例 Prometheus + Alertmanager 集群 |

### 6.5 性能压测与容量规划  

1. **基准测试**：使用 **JMeter** / **Locust** 对时间线读取、微博发布、点赞进行 1‑2 小时的 **峰值压测**，确保 QPS 能达 1.5× 预估值。  
2. **容量模型**：  
   - **CPU**：每台 TiDB Server 8 核可支撑约 30k QPS 读。  
   - **网络**：每台 Kafka Broker 需要 10 Gbps 网络带宽，确保 200k QPS 消息流不拥塞。  
   - **存储**：对象存储需要 10 PB，按 100 TB/节点 计算，需要约 100 台存储节点。

## ## 第七步：常见面试追问与回答  

### 7.1 时间线的粉丝流实现：推模型 vs 拉模型  

**问题**：“我们是采用推模型还是拉模型来为用户生成时间线？如何在关注数千万的热门用户时控制写放大？”  

**回答思路**  

1. **概念复述**：  
   - **推模型**：用户发布微博时，系统把微博写入所有粉丝的时间线缓存。写操作随粉丝数线性增长。  
   - **拉模型**：用户读取时间线时，系统实时查询关注列表并聚合最近微博。写操作固定，读操作随关注数增长。  

2. **业务分析**：  
   - **大多数用户**的粉丝数在 **千级以下**，写放大可接受。  
   - **明星/大V**的粉丝数可能 **上千万**，单条微博如果采用推模型会产生 **数千万次写**，导致热点写放大、缓存压力、网络带宽瓶颈。  

3. **混合方案**（推荐）  
   - **阈值划分**：设定阈值 `FANOUT_THRESHOLD = 10,000`。  
   - **普通用户**（粉丝 ≤ 阈值） → **推模型**：在微博发布后，Timeline Worker 批量写入粉丝的 Redis Timeline。  
   - **热点用户**（粉丝 > 阈值） → **拉模型 + 热点缓存**：  
     - 微博只写入一次到 DB。  
     - 为热点用户维护 **专属热点 Timeline**（Redis Cluster），只缓存最近 500 条微博。  
     - 普通用户读取时间线时，先查询 **普通用户的 Timeline**（推），再查询 **热点用户的最新微博**（拉）并合并排序。  

4. **写放大控制技术**  
   - **批量写入**：Timeline Worker 使用 **Redis Pipeline** 或 **Lua 脚本**，一次性把 10k 条 `ZADD` 命令发送给同一个 Redis 实例，减少网络往返。  
   - **分片热点**：把所有热点用户的 Timeline 放在 **独立的 Redis Cluster**，不与普通用户共享缓存资源，避免热点导致全局缓存失效。  
   - **异步降级**：当写入延迟超过阈值，系统自动降级为 **拉模型**（不再推送），保证可用性。  

5. **优势总结**  
   - **写放大受控**：大多数用户仍使用推模型保持时间线低延迟。  
   - **读性能稳定**：热点用户的微博直接在查询时拉取，不需要遍历巨大的粉丝列表。  
   - **实现简单**：只需要在 Timeline Worker 中判断粉丝数并走不同路径，业务代码改动不大。

### 7.2 点赞/评论计数的强一致性与热点写入  

**问题**：“点赞/评论等高并发写入时，如何保证计数的强一致性，同时又不导致热点写入瓶颈？”  

**回答思路**  

1. **强一致性需求**：用户在页面看到的点赞数应与实际数基本一致（误差 ≤ 1）。  

2. **单点热点问题**：如果每次点赞直接 `UPDATE weibo SET like_cnt = like_cnt + 1 WHERE weibo_id = ?`，同一行会被大量并发请求竞争行锁，导致 QPS 下降甚至死锁。  

3. **分层计数方案**  

   - **层级 1 – 内存计数**：  
     - **Redis** `INCR weibo:like:{weibo_id}` 完成瞬时计数，返回新值给前端。  
     - **优势**：单机（或 Cluster）内存操作毫秒级，天然避免行锁。  

   - **层级 2 – 异步持久化**：  
     - 将每一次点赞事件写入 **Kafka** `like-event`（包含 `weibo_id, user_id, ts`）。  
     - **Stats Worker** 按 `weibo_id` 聚合增量（比如 5 秒内累计 1000 次），一次性执行 `UPDATE weibo SET like_cnt = like_cnt + 1000 WHERE weibo_id = ?`。  

   - **层级 3 – 最终一致性校验**：  
     - 每天或每小时跑 **批任务**（MapReduce）把 **Redis 计数** 与 **DB 计数** 对比，修正偏差（容错）。  

4. **防止计数丢失**：  
   - **Kafka** 开启 **Exactly‑once** 事务，确保每条点赞事件只落库一次。  
   - **Redis** 使用 **持久化**（AOF + RDB）防止机器宕机导致计数丢失。  

5. **如果必须强一致**（如金融类业务）**：可以使用 **分布式锁**（Redlock）或 **Paxos/Raft** 写入单行，但会牺牲吞吐，通常不推荐在社交媒体这种对计数容忍微小延迟的场景。

### 7.3 跨地域部署的同步复制与容灾  

**问题**：“在跨地域部署时，如何实现数据的同步复制和故障切换？请说明读写分离、分片策略以及灾备恢复时间目标（RTO/RPO）。”  

**回答思路**  

1. **跨地域网络**：使用 **专线 / VPC Peering** + **GSLB**（全局负载均衡）把用户请求路由到最近的地域。  

2. **读写分离**：  
   - **主写地域**（如北京）负责所有写请求（User、Weibo、Follow）。  
   - **从读地域**（如上海、广州）持有 **TiDB/ TiKV** 的 **只读副本**，通过 **Raft** 同步复制（同步复制模式），保证 **RPO ≈ 0 s**。  

3. **分片策略**：  
   - **水平分片**（hash on user_id）在所有地域保持相同的 **分片规则**，这样同一用户的数据始终落在同一分片上，跨地域复制只涉及同一分片的 **多副本**。  
   - **热点分片**（明星用户）可以单独在每个地域 **预写**（写扩散），通过 **双写**（写到主 + 写到热点副本）降低跨地域网络延迟。  

4. **同步复制技术**：  
   - **TiDB** 原生支持 **Multi‑Region Deployment**，通过 **Placement Driver (PD)** 控制副本分布。  
   - **Cassandra** 支持 **NetworkTopologyStrategy**，可以在不同 DC（Data Center）之间保持 **同步写入**（QUORUM）或 **异步复制**（DC\_LOCAL\_QUORUM）。  

5. **故障切换（灾备）**：  
   - **自动检测**：使用 **Prometheus Alertmanager** 监控 PD、TiKV、Kafka、Redis 的健康状态。  
   - **切换流程**：当主写地域不可用，GSLB 将流量切换到备份地域；备份地域的 TiDB **提升为写主**（通过 PD 的 `region-switch` 接口），原来的只读副本升为写节点。  
   - **数据一致性**：在切换前，需要确认 **未同步的日志**（Kafka、Binlog）已经落库，或通过 **事务日志回放** 完成补偿。  

6. **RTO / RPO**：  
   - **RTO**（恢复时间）目标 < **5 分钟**：自动化脚本完成 PD、TiKV、Kafka 的切换，且业务层（API）无感知。  
   - **RPO**（数据丢失）目标 **0 s** 对关键表（User、Weibo、Follow），因为使用 **同步复制**；对非关键日志可接受 **几秒** 延迟。  

7. **监控与演练**：  
   - 每周跑一次 **灾备演练**（模拟 PD/TiKV 故障），记录切换时间并优化脚本。  
   - 使用 **TiDB Dashboard**、**Kafka MirrorMaker**、**Redis Sentinel** 统一监控跨地域同步延迟。

### 7.4 其他可能追问的简要答案（供参考）  

| 追问 | 简要回答 |
|------|----------|
| **“如何防止恶意刷粉/刷赞？”** | ① 业务层限流（IP、UID、设备） ② 机器学习风控（异常行为检测） ③ 点赞/关注加入 **幂等键** 防止重复请求 ④ 关键操作写入 **Kafka audit-log** 供离线风控 |

| **“搜索时如何防止脏数据返回？”** | 采用 **双写机制**：写入 DB 后立即发送到 Kafka，同步写入 ES；ES 更新失败时回滚 DB（通过事务日志）或标记为 **待同步**，查询时过滤 `status=deleted` |

## ## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **时间线的推‑拉混合模型** | 必须在 **写放大** 与 **读延迟** 之间找到平衡。热门用户的粉丝数可能是 **几千万**，直接推会导致 **网络、缓存、CPU** 的灾难性压力。 | - 先算出 **写放大 = 粉丝数 × QPS**，发现明星用户的写放大 > 10⁶。<br>- 再评估 **拉模型** 的查询成本：查询关注列表 + 合并排序，每秒可接受 200k 读。<br>- 于是提出 **阈值分流**：普通用户推，热点用户拉。<br>- 为防止热点用户拉查询仍然慢，额外在 **热点用户专属缓存**（Redis Cluster）上做 **预热**，只缓存最近 N 条微博。 |
| **点赞/评论计数的强一致性 vs 热点写入** | 点赞量在热门微博上每秒可达 **上万**，单行更新会出现 **行锁争用**，但业务又要求 **计数几乎实时**。 | - 先列出几种方案：① 直接 DB 更新；② 分布式锁；③ 缓存 + 异步落库。<br>- 评估 **成本**：直接 DB → 读写冲突；分布式锁 → 额外网络往返。<br>- 选定 **Redis INCR + Kafka 异步批量落库**，实现 **写入 1000 倍** 的吞吐，且计数在 1‑2 ms 内返回。<br>- 加入 **Exactly‑once** 的 Kafka 保证不丢事件，满足业务的 **“几乎强一致”** 要求。 |

### 2. 新手最容易犯的错误（至少 2 条）  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有业务放在单体服务里** | 随着 QPS 增长，单体的 **CPU、内存、部署** 成为瓶颈，代码耦合度高，故障影响全局。 | **微服务化**：把业务拆成 **用户、微博、关注、时间线、互动** 等独立服务，分别水平扩容。 |
| **直接在 MySQL 上做热点计数** | 热点微博的 **点赞/评论** 会产生行锁竞争，导致写吞吐下降甚至死锁。 | **使用缓存 + 异步落库**：Redis INCR + Kafka 批量写入，降低写放大。 |
| **时间线采用单一推模型** | 对明星用户的写放大会导致 **网络、Redis、磁盘** 瞬时崩溃。 | **混合推‑拉模型**：根据粉丝数阈值选择不同策略，热点用户走拉或专属缓存。 |
| **忽视幂等性** | 重试时产生 **重复微博、重复点赞**，导致数据不一致。 | 在所有写接口加 **Idempotency-Key** 或业务唯一键，后端使用 **Redis Set** 去重。 |
