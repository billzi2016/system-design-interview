## 题目背景
知乎是一个以「高质量问答」为核心的社区平台，用户可以在上面提问、回答、评论、收藏以及关注感兴趣的话题和其他用户。面试任务是设计一个能够支撑海量用户并保证内容检索、推荐和交互高效、可靠的 **知乎问答平台**。

| 步骤 | 目标 | 关键点 |
|------|------|--------|
| **1. 需求拆解 & 规模估算** | 把抽象的业务需求转化为可度量的指标（QPS、存储、并发等） | 关注峰值、增长曲线、读写比例 |

## 第一步：理解需求与规模估算

### 1.1 功能需求拆解

| 编号 | 功能 | 子功能点 | 读/写比例（预估） |
|------|------|----------|-------------------|
| 1 | 注册/登录 | 发送验证码、OAuth、生成 token | 写 1% / 读 99%（登录验证） |
| 2 | 提问/回答 | 发帖、编辑、删除、上传媒体 | 写 5% / 读 95% |
| 3 | 评论/赞同/收藏 | 点赞/收藏（高并发）、评论二级回复 | 写 10% / 读 90% |
| 4 | 内容检索 | 关键词搜索、过滤、排序 | 写 2% / 读 98% |
| 5 | 推荐 | 首页流、相关推荐 | 写 1% / 读 99%（实时计算属于写） |
| 6 | 运营后台 | 审核、统计、灰度发布 | 写 1% / 读 99% |

> **为什么要做读写比例估算？**  
> 读写比例决定了我们是走 **读写分离**、**缓存热点** 还是 **强一致性**。知乎的大多数请求是 **读取**（浏览问题、答案），所以我们会把 **读** 设计成 **快速**、**可缓存**，而 **写** 采用 **异步**、**最终一致** 的方式。

### 1.2 非功能性指标量化

| 指标 | 目标 | 计算方式 |
|------|------|----------|
| **DAU** | 300k（3 年 1M） | 300k / 24h ≈ 3.5k QPS（均摊） |
| **峰值 QPS** | 5,000 req/s | 包含搜索、首页、点赞等混合 |
| **响应时延** | ≤ 200 ms（99%） | 前端渲染完成时间，包括网络 + 后端处理 |
| **可用性** | 99.95%（≈ 22 min/月） | 单点故障必须通过冗余、自动切换解决 |
| **存储** | 150 TB 原始 + 500 TB 索引/备份 | 需考虑分布式文件系统或对象存储 |
| **一致性** | 最终一致（赞同、收藏） | 通过异步消息、缓存更新实现 |
| **扩展性** | 支持 10×负载 | 采用 **水平扩展**、容器化、弹性伸缩 |

### 1.3 粗略容量 & 流量估算

假设 **每条问题/答案平均 500B**（文字）+ **图片/视频 200KB**（按 30% 的帖子有媒体），则：

- **每日新增内容**：  
  - 提问 5k 条、回答 15k 条 → (5k+15k) × (0.5KB + 0.2MB*0.3) ≈ 6 GB/天  
- **5 年累计** ≈ 10 TB（仍远小于 150 TB），说明 **媒体占比更大**，需要 **对象存储（OSS）** 负责大文件，数据库只存元数据。

**读流**（浏览）：

- 假设每位活跃用户平均浏览 20 条内容/天 → 300k × 20 = 6 M 次阅读/天 ≈ 70 req/s（均摊），峰值 5k req/s 由 **热门问题** 短时间内的流量提升。

**写流**（点赞、评论）：

- 点赞/收藏 10% QPS → 500 req/s，峰值 1k req/s。需要 **高并发写**、**防止缓存击穿**。

> **结论**：系统整体 **读多写少**，但 **写的并发热点**（点赞、收藏）必须设计 **防击穿/写穿** 机制；**搜索** 与 **推荐** 是 **计算密集**，需要专门的 **离线/实时** 计算平台。

## 第二步：高层架构设计

### 2.1 基本原则

| 原则 | 解释 |
|------|------|
| **单一职责** | 每个服务只负责一种业务（如用户服务、内容服务），降低耦合 |
| **无状态** | 业务服务不保存会话，便于水平扩容，状态放在 **Redis**、**数据库** |
| **分层** | 前端 → API Gateway → 业务层 → 数据层（DB/ES） |
| **异步化** | 高并发写操作（点赞、收藏）先写入 **MQ**，后续异步落库、缓存更新 |
| **容错** | 多副本、自动故障转移、灰度发布 |

### 2.3 数据流向示例（点赞场景）

1. **客户端** → **API Gateway**（鉴权） → **内容服务**（点赞 API）  
2. 内容服务 **写入 Redis**（赞同计数 + 用户‑内容关系）并 **发送 Kafka 消息**（`LikeEvent`）  
3. **消费者**（异步落库服务）消费 `LikeEvent`，更新 **MySQL** 中的 `like_count`（持久化）以及 **ES** 中对应文档的 `like` 字段（搜索排序）  
4. **缓存失效**：更新完后发送 **Cache Invalidate** 消息，通知其他业务节点更新本地缓存  

> **为什么要走这么多步骤？**  
> - **Redis** 提供毫秒级读写，满足 **即时展示**（点赞数立即增长）。  
> - **Kafka** 解耦写入路径，防止数据库写入瓶颈，保证 **高可用**（写失败可重试）。  
> - **异步落库** 让 **MySQL** 只负责 **最终一致**，不影响前端响应时间。  
> - **ES** 必须同步，才能在搜索结果中按赞同数排序。

## 第三步：数据库设计

### 3.1 关系型数据库（MySQL）— 核心业务数据

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| `users` | `user_id` (BIGINT) | `phone/email/openid`, `password_hash`, `status` | 用户基本信息，业务唯一 |
| `topics` | `topic_id` | `name`, `description`, `follower_cnt` | 话题标签 |
| `questions` | `question_id` | `user_id`, `title`, `content`, `status`, `create_time`, `update_time`, `answer_cnt`, `view_cnt` | 问题元数据 |
| `answers` | `answer_id` | `question_id`, `user_id`, `content`, `status`, `create_time`, `update_time`, `like_cnt`, `comment_cnt` | 回答元数据 |
| `comments` | `comment_id` | `parent_id` (question/answer/comment), `user_id`, `content`, `create_time`, `status` | 支持二级评论 |
| `likes` | `like_id` (BIGINT) | `user_id`, `target_type`(question/answer/comment), `target_id`, `create_time` | 记录点赞行为，防重复 |
| `favorites` | `fav_id` | `user_id`, `target_type`, `target_id`, `create_time` | 收藏行为 |
| `follow_user` | `id` | `follower_id`, `followee_id`, `create_time` | 关注关系 |
| `follow_topic` | `id` | `user_id`, `topic_id`, `create_time` | 关注话题 |

**分库分表策略**  

- **水平分表**：`questions`, `answers`, `comments` 按 `question_id` 哈希到 **N** 个库（如 8），每库再按 `question_id` 范围拆分为 **M** 张表（如 16），形成 **128** 个实际表。这样单表数据 ≤ 1B，查询定位快。  
- **用户表**：`users` 采用 **垂直分库**（用户中心库）并 **读写分离**（主库写，多个从库读）。  
- **热点表**：`likes`、`favorites` 采用 **双写**（MySQL + Redis）+ **Kafka** 异步持久化，防止写冲突。

> **为什么不把所有数据放在 NoSQL（如 Mongo）里？**  
> - **事务需求**：提问/回答需要 **跨表**（question ↔ answer）原子性，关系型 DB 更擅长。  
> - **复杂查询**：如 **多条件过滤**（话题、时间、赞同数）在 MySQL 上使用索引更高效。  
> - **一致性**：赞同、收藏等 **强一致**（必须防重复）更适合关系型 DB。

### 3.2 缓存层（Redis）

| 目的 | 数据结构 | 示例键名 | 失效策略 |
|------|----------|---------|----------|
| **热点问答阅读** | `Hash` (question_id → JSON) | `question:detail:{qid}` | 永不过期，手动失效或定时刷新 |
| **赞同计数** | `String`（原子递增） | `question:like:{qid}`、`answer:like:{aid}` | 1 h TTL + 实时写回 DB |
| **用户收藏/点赞状态** | `Set`（用户已赞的目标） | `user:likes:{uid}` | 永不过期，定时清理过期（软删） |
| **排行榜** | `Sorted Set` | `topic:hot`、`question:hot` | 按天/周滚动，TTL 24 h |
| **会话/Token** | `String` | `session:{token}` | 30 min 失效，刷新时延长 |

> **热点缓存失效**：  
> - **写穿（Write-Through）**：写操作直接写入 **Redis**，再异步写 DB。  
> - **写回（Write-Back）**：写入 DB 后 **延迟** 更新 Redis，适用于 **赞同**（容忍几秒不一致）。  
> - **缓存失效（Cache Invalidate）**：更新 DB 后发送 **Invalidate** 消息，所有节点删除对应键，下一次读取时重新加载。

### 3.3 搜索引擎（Elasticsearch）

| 索引 | 主键 | 需要倒排的字段 | 业务用途 |
|------|------|----------------|----------|
| `question_idx` | `question_id` | `title`, `content`, `topic_names` | 关键字搜索、过滤 |
| `answer_idx` | `answer_id` | `content` | 对答案内容搜索 |
| `user_idx` | `user_id` | `nickname`, `bio` | 用户搜索 |
| `topic_idx` | `topic_id` | `name`, `description` | 话题搜索 |

- **同步方式**：写入 MySQL 后 **Kafka** 触发 **Logstash** / **Filebeat** → **ES** 更新（近实时 < 1 s）。  
- **增量更新**：仅更新变更字段（如 `like_cnt`），降低重建成本。  
- **索引分片**：根据 **预估文档量**（10 B）设定 20‑30 主分片，每片 30 GB，副本 1。  

> **为什么使用 ES 而不是直接 MySQL LIKE 查询？**  
> - **全文检索**（分词、倒排）在 ES 中天生高效。  
> - 支持 **复杂过滤、排序、聚合**（按赞同、时间），并能水平扩展。  

### 3.4 推荐特征存储（HBase / ClickHouse）

- **离线特征**（用户画像、历史行为）使用 **HBase** 或 **ClickHouse** 按天写入，供 **Spark** / **Flink** 读取。  
- **实时特征**（最近 5 分钟的点击/点赞）使用 **Redis Streams** 或 **Kafka**，在 **Flink** 中做 **滑动窗口** 统计，写回 **Redis**（实时热点推荐）。

## 第四步：核心 API 设计

> **原则**：API 必须 **幂等**、**统一鉴权**、**返回统一结构**，并在文档里标注 **限流** 与 **错误码**。

### 4.1 统一返回结构

```json
{
  "code": 0,               // 0=成功，非0=错误
  "msg": "OK",
  "data": { … }            // 业务数据
}
```

### 4.2 鉴权方式

- **OAuth2.0** + **JWT**（短期 Access Token + 长期 Refresh Token）  
- 登录成功后返回 **JWT**，每次请求在 `Authorization: Bearer <token>` 中携带。  
- **API Gateway** 负责验证 JWT、解析用户 ID，转发到业务服务。

### 4.3 关键业务 API 示例

| API | 方法 | 路径 | 输入 | 输出 | 幂等性 | 备注 |
|-----|------|------|------|------|--------|------|
| 注册 | POST | /api/v1/auth/register | `{phone,email,password}` | `{user_id, token}` | **是**（同一手机号只能注册一次） | 验证码校验 |
| 登录 | POST | /api/v1/auth/login | `{login_type, credential, password}` | `{token, refresh}` | **是** | 支持手机号/邮箱/OAuth |
| 发问题 | POST | /api/v1/question | `{title, content, media_ids, topic_ids}` | `{question_id}` | **否**（每次都会新建） | 写入 MySQL + Kafka |
| 查看问题 | GET | /api/v1/question/{qid} | - | `{question_detail}` | **是** | 先查 Redis，未命中查 MySQL 再写回 |
| 点赞 | POST | /api/v1/like | `{target_type, target_id}` | `{like_cnt}` | **是**（幂等：内部检查是否已赞） | 写 Redis + Kafka |
| 取消点赞 | DELETE | /api/v1/like | `{target_type, target_id}` | `{like_cnt}` | **是** | 同上 |
| 搜索 | GET | /api/v1/search | `q, topic, sort, page, size` | `{total, list[]}` | **是** | 调用 ES |
| 推荐首页 | GET | /api/v1/feed | `page, size` | `{list[]}` | **是** | 调用推荐服务，缓存热点 |
| 评论 | POST | /api/v1/comment | `{parent_type, parent_id, content}` | `{comment_id}` | **否** | 写入 MySQL + Kafka |
| 关注话题 | POST | /api/v1/follow/topic | `{topic_id}` | `{followed:true}` | **是** | 幂等 |

### 4.4 幂等实现细节（以点赞为例）

1. **客户端** 发送 `POST /like`。  
2. **业务服务**：  
   - 先查询 **Redis Set** `user:likes:{uid}` 是否已存在 `target_key`。  
   - 若不存在，使用 **Redis Lua 脚本**（原子检查+写）把 `target_key` 加入 Set，同时 `INCR` 计数键。  
   - 生成 `LikeEvent` 推送 Kafka。  
   - 返回最新计数。  

> **为什么用 Lua 脚本？**  
> - **原子性**：防止并发导致计数错增/错减。  
> - **性能**：一次网络往返完成检查+写入，比先读后写两次更快。

## 第五步：详细组件设计

### 5.1 API Gateway & 统一鉴权

- **技术选型**：**Kong** / **NGINX + Lua** 或 **Spring Cloud Gateway**  
- **功能**：  
  - 负载均衡（轮询 / IP 哈希）  
  - **限流**（IP/用户/接口）  
  - **鉴权**（JWT 校验）  
  - **灰度发布**（Canary、A/B）  
  - **日志采集**（请求链路）  

### 5.2 业务服务（微服务化）

| 服务 | 语言/框架 | 部署方式 | 关键依赖 |
|------|-----------|----------|----------|
| Auth Service | Java Spring Boot | Docker + K8s | MySQL、Redis、OAuth SDK |
| Post Service | Go / Gin | Docker + K8s | MySQL、Redis、Kafka、OSS |
| Search Service | Java SpringBoot + ES client | Docker | Elasticsearch |
| Recommend Service | Python Flask + TensorFlow / PyTorch | Docker | Spark/Flink, Redis, HBase |
| Stats Service (运营) | Java + Flink | Docker | Kafka, ClickHouse |

> **为什么采用微服务？**  
> - **业务边界清晰**（用户、内容、搜索、推荐）  
> - **独立伸缩**：搜索流量激增时只扩 ES + Search Service。  
> - **技术多样**：推荐可以用 Python，内容用 Go，互不影响。

### 5.3 消息队列（Kafka）

- **Topic 划分**  
  - `question_created`、`answer_created`、`like_event`、`comment_created`、`user_action`（浏览、收藏）  
- **分区**：基于 `target_id` hash，确保同一热点问题的消息落在同一分区，便于顺序消费。  
- **消费者组**：  
  - **Persist Service**（落库）  
  - **ES Sync Service**（同步索引）  
  - **Recommend Real‑Time Service**（实时特征）  

> **为什么选 Kafka 而不是 RabbitMQ？**  
> - **高吞吐**（每秒上万条）  
> - **持久化 + 消费位点管理**（保证不丢数据）  
> - **天然的分区模型**，适合水平扩容。

### 5.4 缓存失效与写穿策略

#### 5.4.1 热点问题阅读缓存

- **写入路径**：  
  1. **用户浏览** → **Post Service** 读取 Redis `question:detail:{qid}`。  
  2. 若未命中 → 查询 MySQL → 将结果写入 Redis（`SETEX` 12 h）。  
- **更新路径**：  
  - 当 **编辑**、**删除**、**赞同数变化** 时，发送 **Cache Invalidate** 消息到 **Kafka**。  
  - **Cache Invalidate Service** 收到后调用 **Redis DEL**，确保下次读取到最新数据。  

#### 5.4.2 赞同计数写穿（Write‑Through）

- **步骤**  
  1. **业务层** 直接 `INCR` Redis `question:like:{qid}`（原子）。  
  2. 同时 **生产** `like_event` 消息。  
  3. **异步落库** 消费 `like_event` → **MySQL** `questions.like_cnt` 加 1。  
- **失效窗口**：因为 **MySQL** 与 **Redis** 可能有 **秒级** 差异，前端可以容忍 1‑2 s 的不一致。  

> **防止缓存击穿**：  
> - **互斥锁**（Redis `SETNX` + TTL）在热点键失效时只允许一个请求去查询 DB，其他请求等待或返回旧值。  
> - **热点预热**：每天凌晨使用 **批任务** 将昨天的热门问题提前写入 Redis。  

### 5.5 搜索与索引同步

- **实时写入**：  
  - 当 **question/answer** 创建或编辑时，产生 **Kafka** 消息 `doc_update`。  
  - **ES Sync Service** 消费后调用 **Bulk API** 更新对应文档（只更新变更字段），延迟 < 500 ms。  
- **离线全量重建**：  
  - 每周或每月使用 **Spark** 读取 MySQL 全量数据，重新生成 ES 索引（`reindex`），防止脏数据积累。  
- **索引压缩**：  
  - 使用 **doc_values**、**stored_fields** 仅保留必要字段，降低磁盘占用。  

### 5.6 推荐系统整体框架

#### 5.6.1 离线特征计算（批处理）

1. **每日离线作业**（Spark）读取 **用户行为日志**（点击、点赞、收藏、关注）从 **HDFS** / **ClickHouse**。  
2. **特征工程**：  
   - **兴趣向量**：基于话题分布、关键词 TF‑IDF。  
   - **活跃度**：最近 7 天活跃天数、行为频次。  
3. **模型训练**（协同过滤、DeepFM 等）输出 **用户‑物品打分矩阵**。  
4. **写入特征存储**（HBase）或 **模型文件**（OSS）供在线服务查询。  

#### 5.6.2 实时特征（流处理）

- **Flink** 从 **Kafka**（`user_action`）读取最近 5‑30 min 的行为，做 **滑动窗口计数**，更新 **Redis** 中的 **实时兴趣标签**（`user:rt_tags:{uid}`）。  
- **推荐服务** 在返回首页流时，先读取 **实时标签** 与 **离线兴趣向量** 融合（加权），排序后返回。  

#### 5.6.3 推荐 API 示例

```http
GET /api/v1/feed?page=1&size=20
Authorization: Bearer xxx
```

- **内部流程**  
  1. **Auth Service** 验证 token，得到 `uid`。  
  2. **Recommend Service** 调用 **Redis** 读取 `user:rt_tags:{uid}`（实时）和 **HBase** 读取 `user:profile:{uid}`（离线）。  
  3. 通过 **召回模型**（基于相似用户/话题）得到 **候选集合**（1000 条）。  
  4. **排序模型**（GBDT）对候选集合打分，返回前 20 条。  
  5. 结果写入 **Cache**（`user:feed:{uid}:page:{p}`）供下次快速读取。  

> **为什么要分离离线与实时？**  
> - **离线** 能使用 **大规模特征**、**复杂模型**（训练耗时小时），提升精度。  
> - **实时** 能捕捉 **最新行为**，让用户的最新点赞/关注立即影响推荐。  

### 5.7 运营后台（内容审核、统计）

- **内容审核**  
  - **机器审**：文本使用 **文本敏感词模型**（FastText） + 图片审查（第三方云服务）。  
  - **人工审**：审核任务写入 **Kafka** → **审核系统**（内部 UI）取任务。  
  - **结果回写**：审核通过后更新 `question.status`、`answer.status`，发送 **Cache Invalidate**。  

- **统计**  
  - **实时 PV/UV**：使用 **ClickHouse** + **Kafka**（日志）做 **流式聚合**，支持 **秒级** 看板。  
  - **离线报表**：每日 Spark 作业生成 **DAU/活跃度**、**增长曲线**，写入 **MySQL** / **DataWarehouse**。  

### 5.8 监控、日志、告警

| 组件 | 监控指标 | 工具 |
|------|----------|------|
| **业务服务** | QPS、错误率、响应时延、CPU/内存、线程池饱和度 | Prometheus + Grafana |

## 第六步：扩展性与高可用设计

### 6.1 高可用（HA）关键点

| 组件 | HA 方案 | 容灾策略 |
|------|----------|----------|
| **API Gateway** | 多实例 + Nginx/HAProxy 负载均衡 | 区域跨 AZ 部署，健康检查 |
| **业务微服务** | K8s Deployment + HPA（水平自动扩容） | 多副本、Pod 重启、滚动升级 |
| **MySQL** | 主从复制 + **GTID**，**ProxySQL** 读写分离 | 主库故障自动切换（MHA） |
| **Redis** | 主从复制 + Sentinel / Redis Cluster | 自动故障转移 |
| **Kafka** | 多 Broker + Replication Factor=3 | 任意 1 Broker 死机不影响 |
| **Elasticsearch** | 主分片 + 副本=1 | 分片迁移、跨机房跨 AZ |
| **对象存储 (OSS)** | 多 AZ 同步复制 | CDN 回源容错 |
| **推荐模型** | 离线模型保存在 OSS，多节点读取 | 采用容器化，滚动升级 |

> **为什么要在每层都做 HA？**  
> - 任何单点故障都会直接导致 **可用性 < 99.95%**。  
> - 分层 HA 能把故障局部化，降低恢复时间（MTTR）。

### 6.2 灾备（DR）方案

1. **冷备中心**（同城/跨城）  
   - 同步 MySQL Binlog 到备库，Redis 同步到备集群，Kafka MirrorMaker 复制 Topic。  
2. **故障切换**  
   - **DNS/负载均衡** 切换到备中心的 IP。  
   - 数据同步延迟 ≤ 5 s（通过 **GTID**、**Kafka Replication** 实现）。  
3 **演练**  
   - 每月进行一次 **灾备演练**，验证 RTO（恢复时间） ≤ 30 min。

### 6.3 弹性伸缩（Autoscaling）

- **CPU/内存 + QPS** 为触发指标（Prometheus + K8s HPA）。  
- **缓存热点**：热点问题数量激增时，**Redis Cluster** 自动扩容（分片迁移）。  
- **搜索流量突增**：使用 **K8s Horizontal Pod Autoscaler** 为 **Search Service** 添加副本，同时 **ES** 通过 **Shard Rebalancing** 自动分配负载。  

### 6.4 灰度发布 & A/B Test

- **Canary Deployment**：新版本只推 5% 流量，监控错误率、时延。  
- **Feature Flag**（如 LaunchDarkly）控制功能开关，支持 **AB Test**（不同推荐模型对比）。  
- **回滚**：K8s 只要把 Deployment `revision` 回滚即可，几秒完成。

## 第七步：常见面试追问与回答

### 7.1 热点问题的缓存失效与写穿

> **问**：如果热点问题的阅读量在短时间内激增，如何设计缓存失效和写穿策略来保证赞同数的实时性？

**回答要点**：

1. **赞同计数使用 Redis 原子递增**（`INCR`）实现 **写穿（Write‑Through）**，写入 Redis 同时发送 **Kafka `like_event`**，异步落库。  
2. **热点问题详情**（标题、内容、图片）缓存 **不失效**，采用 **TTL = 永久**，只有在 **编辑/删除** 时主动 **Cache Invalidate**（通过 Kafka 发送失效消息）。  
3. **防止缓存击穿**：  
   - **互斥锁**（`SETNX`）保证同一时间只有一个请求查询 DB。  
   - **热点预热**：每天凌晨批任务把昨日 Top‑N 热点写入 Redis。  
4. **实时性保证**：前端每次点击赞同后直接读取 Redis 的 `like_cnt`（毫秒级），若需要更高一致性（比如必须确保不会出现负数），在 **落库成功**后再返回成功标识（可通过 **两段式提交** 实现）。  

### 7.2 搜索索引同步与资源消耗

> **问**：在高并发写入的场景下，如何保证全文搜索索引的及时同步，同时控制索引重建的资源消耗？

**回答要点**：

1. **增量同步**：写入 MySQL 后立即 **生产** `doc_update` 到 Kafka，**ES Sync Service** 使用 **Bulk API** 按批次（如 500 条）更新，仅修改变更字段，延迟 < 1 s。  
2. **背压机制**：Kafka Consumer 设置 **max.poll.records** 与 **processing thread pool**，防止瞬时写入冲击 ES。  
3. **异步批处理**：如果写入量异常大，先写入 **中间缓存（Redis List）**，后台 **Flink** 按时间窗口批量同步。  
4. **全量重建**：采用 **离线 Spark** 每周/每月一次，使用 **滚动索引**（创建新索引 → 切换 alias）避免服务中断。  
5. **资源控制**：  
   - ES 设置 **刷新间隔**（`index.refresh_interval=30s`）降低磁盘 IO。  
   - **分片数** 预估后设定，防止单分片过大导致搜索慢。  

### 7.3 推荐实时性

> **问**：当用户刚刚点赞或关注一个话题时，如何让推荐结果在几秒内体现这次行为？

**回答要点**：

1. **实时行为流**：用户的点赞/关注事件写入 **Kafka `user_action`**，**Flink** 消费后在 **滑动窗口**（如 5 min）内统计 **兴趣标签**，更新 **Redis** 中 `user:rt_tags:{uid}`（哈希）。  
2. **推荐服务**：在返回首页 Feed 前，先读取 **Redis 实时标签** 与 **离线画像**（HBase），合并后使用 **召回模型**（基于标签相似度）得到候选集合。  
3. **权重融合**：实时标签权重设为 **0.6**，离线画像 **0.4**（可调），保证最新行为对推荐影响显著。  
4. **缓存**：推荐结果写入 `user:feed:{uid}:page:{p}`，TTL 5 min，保证下次请求命中最新数据。  

### 7.4 限流与防刷

> **问**：如何防止同一用户短时间内频繁点赞导致热点缓存失效或 DB 写入压力？

**回答要点**：

1. **接口层限流**：在 **API Gateway** 使用 **令牌桶**（Token Bucket）对 `/like` 接口每秒 5 次进行限流。  
2. **业务层防刷**：在 **Redis** 为每个 `(uid, target_id)` 设定 **TTL=10s** 的 **SETNX** 锁，若已存在直接返回已赞成功，避免重复写入。  
3. **分布式计数**：使用 **Redis Lua 脚本** 同时判断是否已赞、递增计数，保持原子性。  
4. **监控告警**：监控 `like_event` 的 **Consumer Lag**，若出现异常增长立即触发报警。

## 心得与反思

### 1️⃣ 本题最难的设计决策

| 决策 | 思考过程 |
|------|----------|
| **缓存与一致性**（赞同/收藏实时性 vs. 数据库压力） | - 先要明确业务对 **实时性** 的容忍度：赞同数可接受几秒误差 → 采用 **写穿 + 异步落库**。<br>- 再考虑 **热点**：热点问题的赞同会产生 **写放大**，如果每次都走 DB 会瞬时压垮 MySQL。<br>- 于是引入 **Redis 原子递增 + Kafka**，把高并发写转移到内存层，保证 10k QPS 以上的并发仍能 1ms 返回。<br>- 最后补上 **Cache Invalidate** 与 **互斥锁**，防止缓存击穿。 |
| **推荐系统的实时与离线融合** | - 推荐需要 **精度**（离线大模型）和 **时效**（实时兴趣）。<br>- 需要划分 **特征来源**：离线特征存 HBase，实时特征存 Redis。<br>- 决定使用 **Flink** 做实时窗口统计，原因是它天然支持 **Exactly‑once**，且能在秒级输出。<br>- 为了不把推荐服务变成单点瓶颈，采用 **微服务 + 缓存**，并把 **召回** 与 **排序** 分层，保证即使实时特征暂不可得仍能返回离线结果。 |

### 2️⃣ 新手最容易犯的错误

| 错误 | 解释 | 正确做法 |
|------|------|----------|
| **把所有数据都放在关系型库** | 直接把图片、全文、点赞等放在 MySQL，导致磁盘 IO、查询慢、扩展受限。 | 按业务划分：**MySQL** 负责事务强一致的核心数据；**Redis** 缓存热点；**ES** 负责全文搜索；**对象存储** 负责大文件。 |
| **忽视幂等性** | 点赞、收藏等接口如果直接写库，重复请求会产生重复记录或计数错误。 | 在业务层使用 **Redis Set** + **Lua 脚本** 检查幂等；在 MQ 消费端使用 **唯一 ID** 去重。 |
| **把所有请求都同步阻塞** | 点赞后等 MySQL 事务提交再返回，导致响应 > 300 ms，影响前端体验。 | 采用 **写穿 + 异步落库**，先返回给前端，后续通过 MQ 保证最终一致。 |
| **只关注读的性能** | 只做缓存读，却没有考虑写的并发与一致性，导致热点写崩溃。 | 同时设计 **写路径**（锁、限流、异步）以及 **读路径**（缓存、预热）。 |

### 3️⃣ 学习建议与可延伸方向

| 建议 | 说明 |
|------|------|
| **系统思维** | 先从 **业务** 出发，划分 **读/写**、**热点**、**一致性**，再选技术。不要盲目跟风使用最新框架。 |
| **从小实验做起** | 用 **Docker Compose** 搭建 MySQL + Redis + Kafka + ES 的本地集群，跑一些压测（`ab`、`wrk`），体会 **瓶颈点**。 |
| **深入掌握分布式一致性** | 阅读《Designing Data‑Intensive Applications》《CAP Theorem》；了解 **事务、幂等、最终一致** 的实现细节。 |
| **学习消息队列的细节** | Kafka 的 **分区、复制、消费者组、offset** 机制；如何做到 **Exactly‑once**。 |
| **推荐系统入门** | 了解 **协同过滤、Embedding、深度学习**，尝试使用 **Spark MLlib** 或 **TensorFlow Recommenders** 实现简单模型。 |
| **监控与灰度** | 学习 **Prometheus + Grafana**、**ELK**、**Istio**，掌握 **金丝雀发布** 与 **回滚**。 |
| **安全与合规** | 虽然本题不要求，但实际产品必须考虑 **验证码、防刷、GDPR** 等，适当了解。 |

> **一句话总结**：系统设计不是堆砌技术，而是 **在业务约束下，用最合适的工具搭建一个 **可扩展、可维护、可监控** 的系统。**把每一次“为什么”都说清楚，面试官自然会对你的思路点赞。**
