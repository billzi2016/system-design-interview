## 实时多人同步引擎（帧同步）系统设计题

### 1. 题目背景
实时多人同步引擎是用于 **对等（P2P）或服务器‑客户端** 环境下的 **帧同步** 游戏（如格斗、RTS、竞技类）中的核心组件。它负责在 **毫秒级** 的时间窗口内收集、广播并统一处理所有玩家的输入指令，确保所有玩家的游戏状态保持 **严格一致**。

### 2. 面试场景设定
> **面试官**：  
> “我们现在要为一款预计在全球发行的 1v1 对战格斗游戏设计实时帧同步引擎。请你从零开始设计整个系统，重点考虑如何在 **低延迟**、**高并发**、**容错** 的前提下保证所有玩家的游戏状态保持一致。请先阐述你的整体思路，然后逐步展开关键模块的设计。”

### 3. 功能性需求
| 编号 | 需求描述 |
|------|----------|
| **F1** | **玩家输入收集**：在每一帧（固定时间步，如 16 ms）内收集所有参与玩家的输入指令（按键、摇杆等），并对外提供统一的输入接口。 |
| **F2** | **帧号同步**：为每一帧分配全局唯一递增的 **帧号（tick）**，并确保所有客户端使用相同的帧号进行状态演算。 |
| **F3** | **输入广播 & 回放**：将收集到的玩家输入打包后广播给所有参与玩家；在网络抖动或丢包时，支持 **回放** 已确认的历史帧以恢复一致性。 |
| **F4** | **延迟补偿 & 滞后缓冲**：实现 **客户端预测** 与 **服务器回滚**（或“帧延迟缓冲”）机制，使玩家感受到流畅的操作体验。 |
| **F5** | **房间/匹配管理**：提供创建、加入、解散房间的接口，并在房间内维护玩家列表、帧同步状态等元数据。 |
| **F6** | **异常检测 & 重连**：检测玩家掉线、卡顿等异常情况，支持 **快速重连** 并在重连后恢复到当前帧的同步状态。 |

### 4. 非功能性需求
| 编号 | 指标 | 具体要求 |
|------|------|----------|
| **N1** | **并发用户数（DAU）** | 目标 **1,000,000** 日活用户，峰值 **30,000** 并发房间（每房间 2 人）。 |
| **N2** | **每秒请求数（QPS）** | 预计 **30,000 rooms × 60 frames/s = 1.8M** 条输入上报请求；系统需支撑 **≥ 2M QPS**。 |
| **N3** | **端到端延迟** | 从玩家完成一次输入到该输入在对手端生效的 **99%** 延迟 ≤ **80 ms**。 |
| **N4** | **可用性** | 年度可用性 **≥ 99.9%**（单点故障容忍），故障恢复时间 ≤ **30 秒**。 |
| **N5** | **存储容量** | 每场对局保存最近 **10 秒** 的完整帧输入日志用于回滚，估算每场约 **200 KB**；每日需存储 **≈ 2 TB**（包括历史归档）。 |

### 5. 系统边界
**本题需考虑的范围**  
- 网络层协议选型（UDP/TCP、可靠传输实现）  
- 帧同步核心算法（输入收集、帧号分配、回滚/预测）  
- 房间调度、匹配与状态管理  
- 容错与异常恢复（掉线、重连、分区）

**不在本题范围**

- 客户端渲染、音视频同步等非帧同步部分  
- 第三方支付、社交系统、排行榜等外围业务  
- 完整的 CDN、静态资源分发方案

3. **横向扩展**：面对 2M QPS 的输入上报，如何对 **房间调度服务** 进行分片或无状态化设计，以实现弹性伸缩？

1. **先把需求拆解**：功能需求（F1~F6） + 非功能需求（N1~N5）。  
2. **估算规模**：并发房间数、QPS、存储量等，用来指导后面的容量规划。

4. **把原型拆解为「业务」+「基础设施」**，形成高层架构（网关、房间调度、同步核心、持久层、监控）。

> **核心理念**：**先做好「正确的事」**（正确的算法、正确的数据模型），再去「把事做好」——即在此基础上做高可用、弹性伸缩、监控告警。

## 第一步：理解需求与规模估算  

| 需求 | 关键点 | 对系统的直接影响 |
|------|--------|------------------|
| **F1** 输入收集 | 每帧（≈16 ms）收集每位玩家的输入 | 需要 **高频、低延迟** 的上报接口；每帧必须保证 **全体玩家都有输入**（或用默认空输入） |
| **F2** 帧号同步 | 全局递增的 **tick**，所有客户端使用同一 tick | 必须有 **唯一且有序** 的编号生成机制；分布式环境下要避免冲突 |
| **F3** 输入广播 & 回放 | 把所有玩家的输入一次性广播；网络抖动时回放历史帧 | 需要 **可靠传输**、**缓存** 已确认帧、**回滚** 机制 |
| **F4** 延迟补偿 | 客户端预测 + 服务器回滚/帧缓冲 | 设计 **帧延迟缓冲**（例如 2‑3 帧）以及 **回滚** 需要保存历史状态 |
| **F5** 房间/匹配 | 创建/加入/解散房间，维护元数据 | 需要 **房间调度服务**，支持 **水平拆分** |
| **F6** 异常检测 & 重连 | 探测掉线、快速恢复 | 需要 **心跳**、**重连协议**、**状态同步** |

### 规模估算  

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **并发房间** | 峰值 30 k 房间（2 人/房间） | 30 000 |
| **每秒输入上报** | 30 k 房间 × 2 玩家 × 60 帧/s ≈ **3.6 M** 条 | 约 3.6M 条 |
| **输入大小** | 每条输入 ≈ 20 B（tick + 按键状态） | 约 72 MB/s ≈ 0.6 Gbps |
| **存储需求** | 200 KB/局 × 30 k 局/秒 × 3600 s ≈ 21 GB/h → 约 2 TB/天 | 2 TB/天 |
| **延迟预算** | 端到端 ≤ 80 ms（99%） | 需要 **<30 ms** 的服务器内部处理 + 网络往返 |
| **可用性** | 99.9% → 年均停机 ≤ 8.76 h；单点故障恢复 ≤ 30 s | 需要 **多副本、快速故障转移** |

> **结论**：系统必须是 **无状态化**（或极少状态）+ **水平扩展** 的，网络层必须使用 **UDP + 自研可靠层**，并在服务器侧做 **帧缓冲 + 回滚**。

## 第二步：高层架构设计  

下面给出 **文字版架构图**，随后解释每个模块的职责与交互。

| ①输入上报 ②帧渲染 |        | ①UDP收发 ②负载均衡|        | ①房间调度服务    |
+-------------------+        +-------------------+        | ②帧同步服务      |
                                                      | ③回滚/预测服务   |
                                                      +-------------------+
                                                            |
                                                            |   +-------------------+
                                                            +-->| 持久化日志服务    |
                                                            |   | (Kafka / Pulsar) |
                                                            |   +-------------------+
                                                            |
                                                            |   +-------------------+

### 关键模块解释  

| 模块 | 主要职责 | 为什么要这么划分 |
|------|----------|-------------------|
| **边缘网关**（UDP + 负载均衡） | 负责 **收发 UDP 包**，做 **协议解码**、**限流**、**防 DDOS**，并把上报的输入写入 **消息队列**（Kafka） | UDP 能提供最小的网络延迟；网关做流量入口，避免业务服务直接暴露在公网 |
| **房间调度服务**（Stateless） | 接收玩家匹配/创建请求，生成 **房间 ID**，把玩家映射到 **特定的同步核心实例**（通过一致性哈希） | 让每个房间的同步逻辑只落在 **一个实例**，避免跨机器同步冲突；调度服务本身无状态，易横向扩展 |
| **帧同步服务**（Stateful per‑room） | - 为每帧分配 **全局唯一 tick**（基于房间本地递增）<br>- 收集本房间所有玩家的输入（从消息队列拉取）<br>- 组装 **帧包** 并广播给房间内所有玩家<br>- 维护 **最近 N 帧的输入日志**（内存 + 持久化） | 这是业务核心，需要 **强一致性**（同一房间的所有玩家必须看到相同的帧） |
| **回滚/预测服务** | - 当收到的输入“晚到”或出现冲突时，**回滚**到对应 tick，重新执行后续帧<br>- 客户端可做 **本地预测**（前 2‑3 帧），服务器回滚后发纠正包 | 解决网络抖动导致的“卡顿”。回滚只在服务器端执行，保持客户端简单 |
| **持久化日志服务** | 将每帧的输入日志持久化到 **Kafka**（实时）+ **Cold‑Storage**（如 S3）用于事后回放、审计 | 需求 N5 要保存最近 10 s 的完整日志，Kafka 天然支持 **滚动窗口**，且具备 **高吞吐** |
| **监控/告警平台** | 收集 **延迟、丢包率、房间数、CPU/内存** 等指标；自动弹性伸缩 | 满足 **可用性 N4**，帮助快速定位故障 |

> **为什么不直接把所有功能放在同一台服务器？**  
> - **单点故障**：一台机器宕机会导致所有房间不可用。  
> - **扩展受限**：随着并发房间数增长，CPU、网络、内存会成为瓶颈。  
> - **维护成本高**：不同职责混杂在一起，难以单独调优。

## 第三步：数据库设计  

帧同步系统的持久化需求并不复杂，主要是 **元数据**（房间、玩家、匹配）以及 **日志**（帧输入）。我们采用 **两层存储**：

1. **关系型数据库（MySQL）**：存放 **房间/玩家元数据**，需要强一致性和事务。  
2. **分布式日志（Kafka）+对象存储（S3/MinIO）**：存放 **帧输入日志**，高写入吞吐、可按时间窗口查询。

### 1. MySQL 表结构（示例）

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| `users` | `user_id` (BIGINT) | `nickname`, `region`, `last_login` | 玩家基本信息 |
| `rooms` | `room_id` (BIGINT) | `state` (enum: WAITING, PLAYING, FINISHED), `created_at`, `updated_at` | 房间元数据 |
| `room_players` | `(room_id, player_idx)` (COMPOSITE PK) | `user_id`, `joined_at`, `ready` | 房间‑玩家映射，`player_idx` 为 0/1 |
| `room_tick_state` | `(room_id, tick)` (PK) | `input_blob`, `snapshot_blob` | **可选**：若需要持久化每帧的状态快照（用于灾备） |

> **为什么不把帧输入直接写入 MySQL？**  
> - **写入频率**：每秒 3.6M 条写入，MySQL 难以支撑。  
> - **时序特性**：Kafka 天然支持 **顺序消费**，适合回放。  

### 2. Kafka Topic 设计  

| Topic 名称 | 分区数 | 消息结构 | 备注 |
|------------|--------|----------|------|
| `room-input` | 依据 `room_id` hash（如 6 000 分区） | `{room_id, tick, player_idx, input_bytes}` | 输入上报的实时流 |
| `room-broadcast` | 同上 | `{room_id, tick, aggregated_input}` | 服务器组装后广播给客户端 |
| `room-replay` | 同上（或专用） | `{room_id, tick, input_blob}` | 用于回滚或事后回放 |

> **分区数选取**：每个分区对应 **单线程消费**，保证同一房间的消息顺序。分区数≈并发房间数的 1/5~1/10，保证负载均衡且不超出 Kafka 单分区吞吐上限（约 100k msg/s）。

### 3. 快照（可选）  

如果服务器需要在 **故障恢复** 时快速恢复到最近的帧，可以每隔 `K` 帧（如 30 帧 ≈ 0.5 s）将 **完整状态快照** 写入对象存储，并在 `room_tick_state` 表记录 `snapshot_id`。恢复时先读取最近快照，再回放之后的输入。

## 第四步：核心 API 设计  

下面给出 **客户端 ↔︎ 服务器** 的关键 REST / UDP 接口（使用 protobuf 定义），并解释每个字段的意义。

### 1. 匹配 / 房间管理（REST，使用 HTTPS）

| 接口 | 方法 | URL | 请求体 | 响应体 | 说明 |
|------|------|-----|--------|--------|------|
| 创建房间 | `POST` | `/api/v1/rooms` | `{user_id, game_mode}` | `{room_id, player_idx, tick_interval_ms}` | 返回房间 ID、玩家在房间中的序号、帧间隔 |
| 加入房间 | `POST` | `/api/v1/rooms/{room_id}/join` | `{user_id}` | `{room_id, player_idx, current_tick}` | 若房间已满返回错误 |
| 结束房间 | `POST` | `/api/v1/rooms/{room_id}/close` | `{}` | `{}` | 触发回放归档、资源回收 |
| 心跳检测 | `GET` | `/api/v1/rooms/{room_id}/heartbeat` | `{}` | `{status: "OK"}` | 用于负载均衡的健康检查 |

> **为什么采用 REST 而不是 UDP**：匹配、房间创建等不频繁且需要可靠性，使用 HTTP 更易调试、兼容 CDN、支持鉴权。

### 2. 输入上报（UDP，proto）

> **数据包结构（ProtoBuf）**  

```proto
message InputPacket {
  uint64 room_id = 1;       // 所属房间
  uint32 tick   = 2;        // 客户端预估的帧号
  uint8  player_idx = 3;    // 0 / 1
  bytes  input = 4;         // 按键位、摇杆等，长度 ≤ 16B
  uint64 timestamp = 5;    // 客户端本地时间，便于延迟统计
}
```

- **发送频率**：每帧一次（≈ 60 fps）。  
- **可靠性实现**：在 UDP 上层实现 **轻量可靠协议**（见下文），只对 **当前帧** 进行 **NACK 重传**，不对已确认的旧帧重传，以免累计延迟。

### 3. 服务器广播（UDP，proto）

```proto
message FrameBroadcast {
  uint64 room_id = 1;
  uint32 tick    = 2;
  map<uint8, bytes> inputs = 3; // player_idx → input bytes
  uint64 server_timestamp = 4;
}
```

- **广播方式**：服务器单播（或多播）给房间内的两位玩家。  
- **客户端处理**：若本地已预测该 tick，直接比对；若不匹配则 **回滚** 并重放。

## 第五步：详细组件设计  

### 5.1 网络层：UDP + 可靠传输实现  

#### 5.1.1 为什么选 UDP？

- **最小的网络往返时延**（RTT ≈ 30‑50 ms），符合 80 ms 端到端目标。  
- **无拥塞控制**：TCP 的慢启动、拥塞窗口在高频率小包场景会导致额外延迟。  

#### 5.1.2 可靠层（轻量版）  

| 功能 | 实现方式 | 细节 |
|------|----------|------|
| **序列号** | 每个 **room_id** 下的 **tick** 充当序列号 | 接收端通过 `tick` 判断是否收到最新帧 |
| **确认（ACK）** | 服务器对每个客户端的 **InputPacket** 发送 **ACK**（只包含 `room_id+tick+player_idx`） | 客户端收到 ACK 后即可丢弃该包 |
| **负向确认（NACK）** | 客户端检测到 **缺失的 tick**（例如 100、102 收到但 101 未收到）则发送 **NACK** 请求服务器重发该 tick | NACK 包体积极小，仅 `room_id+missing_tick` |
| **重传策略** | **仅针对缺失的 tick** 重传一次，若仍未收到则进入 **回滚/预测** 机制 | 重传次数受限，避免无限重传导致延迟爆炸 |
| **超时阈值** | `tick_interval * 2`（如 32 ms） | 超时后即视为丢失，触发 NACK |

> **关键点**：**不对已经确认的旧帧做任何重传**，因为即使偶尔丢失也可以通过 **回滚 + 预测** 纠正。这样避免了 TCP 那种“全局阻塞”效应。

### 5.2 帧同步核心算法  

下面用伪代码展示 **服务器端每帧的处理流程**（假设每个房间对应一个 `RoomSyncWorker` 协程）：

```go
// 参数：frameInterval = 16ms (60fps)
for {
    start := time.Now()
    tick := atomic.AddUint32(&room.nextTick, 1)   // 全局唯一递增 tick

    // 1. 收集本帧所有玩家的输入（从 Kafka 拉取或本地缓冲区）
    inputs := make(map[uint8]Input) // player_idx -> input
    for _, idx := range [2]uint8{0, 1} {
        pkt := room.inputBuffer.Pop(tick, idx) // 若不存在返回 nil
        if pkt == nil {
            // 使用上一次的输入或空输入（防止卡死）
            inputs[idx] = room.lastInput[idx]
        } else {
            inputs[idx] = pkt.input
            room.lastInput[idx] = pkt.input
        }
    }

    // 2. 记录本帧日志（内存+Kafka）
    logEntry := FrameLog{
        RoomID: room.id,
        Tick:   tick,
        Inputs: inputs,
    }
    room.recentLogs.Append(logEntry)        // 环形缓冲区，保存最近 10s
    kafkaProducer.Produce("room-input", logEntry)

    // 3. 广播给客户端
    broadcast := FrameBroadcast{
        RoomID: room.id,
        Tick:   tick,
        Inputs: serialize(inputs),
    }
    udpSender.SendTo(room.players[0].addr, broadcast)
    udpSender.SendTo(room.players[1].addr, broadcast)

    // 4. （可选）回滚检测
    // 若收到的某个玩家的输入是“迟到”的（tick > currentTick），
    // 则触发回滚到该 tick，重新执行从该 tick 起的所有帧
    if needRollback {
        room.RollbackTo(lateTick)
    }

    // 5. 等待本帧结束（保证固定时间步）
    elapsed := time.Since(start)
    sleepTime := frameInterval - elapsed
    if sleepTime > 0 {
        time.Sleep(sleepTime)
    }
}
```

#### 5.2.1 回滚实现细节  

- **保存的内容**：每帧 **输入日志** + **状态快照**（可选）。  
- **回滚步骤**：  
  1. **恢复**到 `lateTick-1` 的 **状态快照**（若不存在，则从最早日志逐帧重放）。  
  2. **重新执行** `lateTick` 到 `currentTick` 的所有帧，使用新的输入。  
  3. **重新广播**这些帧的结果（服务器会在帧号相同的情况下发送 `Correction` 包，客户端只需要覆盖本地预测的结果）。  

- **性能控制**：  
  - **快照间隔**设为 30‑60 帧，最大回滚长度 ≤ 2‑3 帧（因为网络抖动通常 < 2 帧）。  
  - **回滚线程**使用 **单独协程池**，避免阻塞主帧循环。  

### 5.3 延迟补偿（客户端预测）  

- 客户端在发送 **InputPacket** 后，立即使用本地**预测**执行该 tick（把自己的输入直接写入本地状态），**不等待服务器确认**。  
- 当收到 **FrameBroadcast** 时，若服务器的输入 **与本地预测一致**，则直接继续；若不一致，客户端执行 **回滚** 到对应 tick 并 **重放**（类似服务器的回滚）。  
- **帧延迟缓冲**：客户端可以在渲染层面保持 **2‑3 帧的渲染延迟**（即显示 `tick-2`），这样可以在收到服务器帧前有时间纠正误差，提升平滑度。

> **为什么要在客户端做预测？**  
> - 网络往返 30‑50 ms，在格斗游戏里每帧 16 ms，若等服务器确认会出现明显卡顿。  
> - 预测+回滚可以在 **大多数情况下**（> 95%）保持本地操作的即时感。

### 5.4 异常检测与快速重连  

| 场景 | 检测方式 | 恢复流程 |
|------|----------|----------|
| **掉线**（UDP 包丢失） | 客户端连续 **N**（如 5）个 tick 未收到服务器 `FrameBroadcast` → 触发 **心跳超时** | 客户端发送 **ReconnectRequest**（UDP），携带最近收到的 `tick` 与本地最新输入日志。服务器校验后，将该玩家重新加入房间的 **输入缓冲**，并 **从该 tick** 开始重新发送后续帧。 |
| **卡顿**（本地帧率下降） | 客户端监测自身 **帧渲染时间**，若超过阈值（如 50 ms）则上报 **LagReport** | 服务器可临时 **放宽帧延迟缓冲**（增加预测帧数），或在房间内 **降低帧率**（只在极端情况下使用）。 |
| **分区故障**（房间所在实例挂掉） | 监控平台检测 **实例健康**（CPU/内存/心跳） | 使用 **一致性哈希** 把该房间迁移到另一实例：先将最近 **10 s** 的日志从 Kafka 拉取，恢复状态后继续运行。玩家感知的影响在 **< 1 s**（符合恢复时间 ≤ 30 s） |

### 5.5 房间调度的无状态化实现  

- **入口**：玩家先调用 **REST** 创建/加入房间，返回 **room_id**。  
- **负载均衡**：网关层使用 **一致性哈希**（基于 `room_id`）把同一房间的 UDP 包路由到同一 **RoomSyncWorker** 实例。  
- **水平扩展**：只要新增实例并加入哈希环，已有房间的路由不变，新的房间会自动落到新实例上。  
- **状态迁移**：若实例故障，哈希环重新计算，失效房间的 `room_id` 会映射到别的实例；新实例通过 **Kafka** 拉取该房间最近的日志并 **快速恢复**。

> **不做哈希路由会有什么问题？**  
> - 同一房间的输入会分散到多台机器，需要跨机器 **分布式锁** 或 **全局一致性协议**，极大增加延迟和复杂度。  
> - 通过哈希让 **每个房间的所有状态始终落在单机**，天然保证顺序性，简化实现。

## 第六步：扩展性与高可用设计  

### 6.1 横向扩展（满足 N1、N2）  

| 维度 | 扩展方式 | 关键技术 |
|------|----------|----------|
| **网关层** | **水平增加** UDP + TCP 入口节点，使用 **L4**（如 LVS）或 **Kubernetes Service** 做负载均衡 | Anycast IP、Health‑Check |
| **RoomSyncWorker** | **按房间分片**（哈希），每台机器运行数千个协程（Go/Vert.x） | 协程/Actor 模型，CPU 亲和性 |
| **Kafka** | **分区**（每 6 000 分区），**Broker** 横向扩容，**复制因子** ≥ 3 | 自动均衡、分区再平衡 |
| **MySQL** | **读写分离** + **水平分库**（按 region） | ProxySQL、Vitess |
| **对象存储** | **弹性对象存储**（S3/MinIO）自动扩容 | 生命周期规则归档旧日志 |

### 6.2 容错与灾备（满足 N4）  

1. **多副本**：Kafka、MySQL、RoomSyncWorker（热备）均保持 **≥3** 副本。  
2. **快速故障转移**：  
   - **网关**：LVS/Envoy 检测节点不可用后自动剔除。  
   - **RoomSyncWorker**：使用 **心跳**（每 100 ms）写入 ZooKeeper/Etcd；检测失效后调度器将对应房间迁移到备用实例。  
3. **数据恢复**：  
   - **日志**：Kafka 持久化 24 h，宕机后可以 **重放**。  
   - **快照**：每 0.5 s 写入对象存储，恢复时先加载最近快照。  
4. **灰度发布**：使用 **Canary** 或 **蓝绿** 部署新版本，确保不影响正在进行的对局。

### 6.3 监控、告警、限流  

| 监控项 | 监控工具 | 报警阈值 |
|--------|----------|----------|
| **帧处理时延** | Prometheus + Grafana | > 5 ms (单帧) |
| **网络丢包率** | InfluxDB + Telegraf (UDP 收包统计) | > 2% |
| **房间异常率** | 自定义 metric `room_error_total` | > 0.1% |
| **CPU/内存** | cAdvisor + Kubernetes HPA | CPU > 80% |
| **QPS** | Kafka consumer lag | Lag > 10k msgs |

**限流**：在网关层对 **每个玩家的输入速率** 做 **Token Bucket** 限制，防止恶意刷包导致后端压力激增。

### 6.4 性能优化技巧  

- **零拷贝 UDP**：使用 `recvmsg` + `sendmsg`，避免在网关和同步服务之间的内存复制。  
- **批量写入 Kafka**：每帧把两位玩家的输入合并成一条消息，减少网络往返。  
- **使用 lock‑free 队列**：在 `RoomSyncWorker` 内部使用 **MPSC** 环形缓冲区，降低锁竞争。  
- **CPU 亲和性**：把同一房间的工作线程绑定到同一核，提升缓存命中率。

## 第七步：常见面试追问与回答  

| 追问 | 参考答案（要点） |
|------|-----------------|
| **Q1：只用 UDP，如何实现可靠传输且不导致延迟爆炸？** | - **帧号 + ACK/NACK**：每帧使用唯一 tick，服务器对每个玩家的 InputPacket 立即返回 ACK。<br>- **仅对当前帧进行 NACK 重传**：如果缺失的 tick 在 **2‑3 帧** 内未收到，则触发 NACK，服务器重发该帧。<br>- **不重传已确认的旧帧**：即使偶尔丢失，也可以通过 **回滚+预测** 纠正，避免全局阻塞。<br>- **超时阈值**设为 `2 * frameInterval`（≈ 32 ms），保证在 80 ms 端到端目标内完成重传。 |
| **Q2：回滚需要保存哪些状态？高并发场景下如何控制开销？** | - **必须保存**：①每帧的 **玩家输入**（已写入 Kafka），②**游戏状态快照**（可选，仅每 30‑60 帧保存一次）。<br>- **内存缓存**：使用 **环形缓冲区** 保存最近 10 s 的输入（约 200 KB/局），占用极小。<br>- **快照策略**：不对每帧做快照，采用 **间隔快照 + 增量回放**，降低磁盘 I/O 与内存占用。<br>- **回滚频率控制**：网络抖动导致的回滚通常 ≤ 2 帧，故 CPU 额外消耗可接受。 |
| **Q3：2 M QPS 的输入上报，如何对房间调度服务进行分片或无状态化设计？** | - **无状态化**：调度服务只负责 **路由**（返回房间 ID、对应实例），不保存会话。<br>- **一致性哈希**：根据 `room_id` 把同一房间的所有 UDP 包路由到同一实例，保证顺序性。<br>- **水平扩容**：新增实例后只需在哈希环中加入节点，已有房间不受影响。<br>- **背压**：在网关层对每个玩家的输入速率做 Token‑Bucket 限流，防止瞬时冲击压垮后端。 |
| **Q4：如果玩家网络质量很差，导致频繁回滚，会不会影响游戏体验？** | - **帧延迟缓冲**：客户端渲染时保持 2‑3 帧延迟，使得回滚只在 **渲染层面** 修正，不直接导致卡顿。<br>- **动态调节**：服务器监测该玩家的 **丢包率**，若超过阈值，可 **降低帧率**（如 30 fps）或 **增加预测帧数**，降低回滚频率。<br>- **玩家提示**：在 UI 上给出网络状态提示，引导玩家切换更好网络。 |
| **Q5：为什么不直接把所有房间放在同一个进程的协程池里？** | - **单点故障**：进程挂掉会导致所有房间不可用。<br>- **资源竞争**：CPU、内存、网络带宽在单机上有限，难以支撑 30 k 并发房间的 2 M QPS。<br>- **弹性伸缩**：分布式部署后可根据负载动态扩容/缩容。 |
| **Q6：如果要支持 4‑人、8‑人等大房间，架构会有什么变化？** | - **帧同步服务**仍保持 **单实例负责一个房间**，但 **输入聚合**的复杂度随玩家数线性增长。<br>- **消息分区**需要根据 **room_id** + **player_idx** 组合来确保顺序。<br>- **回滚成本**上升，因为需要保存更多玩家的输入；可通过 **压缩**（protobuf + Snappy）减小日志体积。<br>- **网络带宽**：每帧广播的包大小随玩家数增大，需要 **MTU** 调整或 **分片** 发送。 |
| **Q7：系统如何做到 99% 的端到端延迟 ≤ 80 ms？** | - **网络层**：使用 UDP，避免 TCP 拥塞控制。<br>- **网关 → 同步服务**：采用 **同城内** 高速网络，往返 ≤ 20 ms。<br>- **帧处理**：每帧固定 16 ms，内部处理（收集、广播） ≤ 5 ms。<br>- **客户端预测**：玩家本地操作即时响应，实际输入延迟只影响 **对手端**，在 80 ms 范围内完成。 |
| **Q8：如果某个房间所在实例崩溃，玩家会看到什么？恢复过程是怎样的？** | - **检测**：监控平台 5 s 内感知实例失联。<br>- **迁移**：调度器在新实例上启动对应 `RoomSyncWorker`，从 Kafka 拉取最近 10 s 的输入日志，恢复最新快照并回放。<br>- **玩家感知**：在 **< 1 s** 内收到 **重连** 包，客户端会短暂出现 **画面冻结**（≈ 1‑2 帧），随后继续正常。符合 **30 s** 故障恢复要求。 |

## 心得与反思  

### 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **① UDP 可靠传输 + 回滚机制** | 必须在 **极低延迟** 与 **可靠性** 之间取得平衡，传统的 TCP 直接不满足；但纯 UDP 会导致丢包不可恢复。 | - 先明确 **帧同步的容错模型**：即使丢失某帧也能通过回滚纠正。<br>- 于是把 **可靠性局限在「当前帧」**，采用 **ACK/NACK + 仅重传**，防止累计延迟。<br>- 再考虑 **回滚成本**，决定只在 **2‑3 帧** 范围内回滚，避免 CPU 爆炸。 |
| **② 房间调度的无状态化 + 哈希路由** | 需要兼顾 **顺序性**（同房间的所有输入必须顺序处理）与 **水平扩展**（30 k 房间）。传统的中心化调度会成为瓶颈。 | - 分析「同房间所有消息必须落在同一机器」的根本原因是 **顺序一致性**。<br>- 引入 **一致性哈希**，把 **room_id** → **实例** 的映射固定，使得 **每个房间天然单机化**，既保证顺序，又无需全局锁。<br>- 为容错，使用 **哈希环 + 多副本**，在实例失效时重新映射，确保可恢复。 |

### 新手最容易犯的错误（至少 2 条）  

1. **把「全局唯一帧号」放在中心化服务**  
   - 这样会导致 **单点瓶颈**，并且跨实例的网络往返会把帧间隔拉长，直接破坏 16 ms 的时间步。正确做法是 **每个房间本地递增**（在同一实例内部），只要房间不迁移，帧号就是全局唯一的。  

2. **把所有玩家的输入直接写入关系型数据库**  
   - 关系库写入吞吐量有限，且每帧只有几毫秒的处理时间窗口，写库会成为**性能瓶颈**。应使用 **高吞吐的分布式日志（Kafka）** 进行实时写入，必要时再持久化到对象存储或冷库。
