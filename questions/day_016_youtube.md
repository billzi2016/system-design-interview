> 生成日期：2026-05-10

## 题目背景  
YouTube 是全球最大的视频分享与播放平台，用户可以上传、观看、评论、点赞以及订阅自己感兴趣的频道。面试目标是让候选人设计一个支撑核心视频服务的后端系统。

---

## 面试场景设定  
**面试官**：  
“我们现在要设计一个能够支撑全球用户观看和上传视频的核心系统。请先从高层次阐述你的整体架构思路，然后逐步展开关键模块的设计细节。我们重点关注系统的可扩展性、低延迟以及高可用性。”

---

## 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| 1 | **视频上传**：用户可以上传任意时长、任意分辨率的视频，系统需要对视频进行转码、生成封面图和多分辨率切片。 |
| 2 | **视频播放**：用户可以在网页或移动端观看视频，系统需要根据网络情况自动切换码率（ABR），并支持快进/暂停等基本控制。 |
| 3 | **推荐/订阅**：用户可以订阅频道，系统为其生成个性化的首页推荐列表和订阅更新流。 |
| 4 | **评论与互动**：用户可以对视频发表评论、点赞、点踩，并查看其他用户的互动。 |
| 5 | **搜索**：支持按关键字、标签、频道等维度搜索视频，返回相关度排序的结果。 |
| 6 | **统计与计费**（可选）：记录播放次数、观看时长，用于广告计费与内容创作者收益分成。 |

## 非功能性需求  

| 指标 | 目标值（估算） | 备注 |
|------|---------------|------|
| **DAU（日活跃用户）** | 3 亿 | 包括观看、上传、评论等所有活跃用户 |
| **QPS（请求峰值）** | 150 万请求/秒 | 高峰期（美国/欧洲/亚洲同时在线） |
| **播放延迟** | < 150 ms（首帧） | 端到端从用户点击到首帧渲染 |
| **可用性** | 99.99%（月均累计不可用时间 < 44 分钟） | 多可用区容灾 |
| **存储容量** | 10 EB（原始 + 转码后） | 预计每年增长 30% |
| **转码时延** | ≤ 30 分钟/视频（720p） | 关键业务 SLA |
| **搜索响应时间** | ≤ 200 ms（99% 请求） | 包含全文索引和过滤 |

# 📚 极致详细的 YouTube 核心视频服务系统设计手把手教程  

> **写给完全没有系统设计经验的同学**  
> 本文从“最小可用系统”一步步演进到“全球可用、低延迟、可水平扩展”的完整架构。每一次设计决策都会解释 **为什么** 这么做，**不这么做会有什么后果**，帮助你在面试中思路清晰、表达有条理。

## 解题思路总览
1. **先把需求拆解** → 功能需求、非功能需求、边界、业务流程。  
2. **做规模估算** → 用 DAU、QPS、存储等指标推算系统容量。  
3. **画出高层图** → 把系统划分为若干子系统（上传、转码、存储、分发、播放、推荐、评论、搜索、监控）。  
4. **挑选合适的存储/缓存/消息队列** → 关系型、NoSQL、对象存储、CDN、Redis 等。  
5. **细化每个子系统的 API、数据模型、内部流程**。  
6. **逐步加入容灾、扩容、限流、监控**，从单点到多可用区。  
7. **准备面试追问**：一致性、热点写入、缓存失效等。  

下面我们一步步展开。

## 第一步：理解需求与规模估算  

### 1. 功能需求拆解
| 编号 | 关键点 | 细化子功能 |
|------|--------|------------|
| 1 | 视频上传 | 大文件分块上传 → 触发转码 → 生成封面 & 多码率切片 |
| 2 | 视频播放 | ABR（自适应码率） → CDN 加速 → 首帧低延迟 |
| 3 | 推荐/订阅 | 关注频道 → 生成用户首页流（最近、热门） |
| 4 | 评论互动 | 发评论、点赞/点踩、分页查询 |
| 5 | 搜索 | 关键字、标签、频道、排序（相关度、时间） |
| 6 | 统计计费（可选） | 播放次数、观看时长、广告计费（不做实现） |

> **注意**：本题不要求实现复杂的机器学习推荐或广告系统，只要能解释 **如何支撑** 这些需求即可。

### 2. 非功能需求关键指标  

| 指标 | 目标 | 业务意义 |
|------|------|----------|
| **DAU** | 3 亿 | 说明系统必须支持全球规模的并发用户 |
| **QPS 峰值** | 150 万 /s | 包含上传、播放、评论、搜索等所有入口 |
| **首帧延迟** | <150 ms | 用户体验核心——点击即见 |
| **可用性** | 99.99% | 多 AZ、自动故障转移 |
| **存储** | 10 EB（年增长 30%） | 视频体量巨大，需要对象存储 + 生命周期管理 |
| **转码时延** | ≤30 min/720p | 视频上线速度，直接影响用户粘性 |
| **搜索响应** | ≤200 ms (99%) | 必须在毫秒级返回结果，搜索是高频操作 |

### 3. 业务流概览（文字版）  

1. **上传** → 客户端分块 → 上传服务 → 对象存储 (原始文件) → 触发转码任务 → 转码后产物写回对象存储 → 更新元数据 → 生成 CDN 缓存键。  
2. **播放** → 客户端请求播放页面 → 播放服务返回 CDN URL + 清晰度列表 → 客户端向 CDN 拉流 → CDN 边缘缓存命中 → 首帧返回。  
3. **评论** → 客户端 POST → 评论写入服务 → 写入分片 NoSQL + 事件写入 Kafka → 异步落库 + 更新计数。  
4. **搜索** → 客户端查询 → 搜索服务 → 读取倒排索引 → 返回视频 ID 列表 → 再调用视频元数据服务补全信息。

## 第二步：高层架构设计  

### 1. 最小可用系统（MVP）  
只需要 **单体后端 + 本地磁盘 + 单节点数据库**，满足功能验证。  

```
[Client] --> [API Gateway] --> [Video Service (upload/playback)] --> [Local Disk]
                     |
                     +--> [MySQL] (元数据、用户、评论)
```

- **优点**：实现快、部署简单。  
- **缺点**：单点故障、无法承载大流量、磁盘容量受限、扩展困难。  

> **面试提示**：先把 MVP 说清楚，展示你能从最小系统开始思考，然后自然过渡到分布式。  

### 2. 逐步演进到分布式高可用架构  

下面给出 **完整的生产级高层图**（文字 + 简单 ASCII），你可以在白板上画成框图：

```
                         +-------------------+
                         |   CDN / Edge      |
                         +--------+----------+
                                  |
                +-----------------+-----------------+
                |                                   |
         +------+------++                     +------+------+
         |  Load Balancer|                     |  Load Balancer|
         +------+------++                     +------+------+
                |                                   |
   +------------+------------+        +-------------+------------+
   |                         |        |                          |
+--v--+   +--v--+   +--v--+   +--v--+   +--v--+   +--v--+   +--v--+   +--v--+
|API GW| |Auth | |Rate | |Cache|   |Upload| |Trans| |Play | |Search|
|      | |Srv  | |Lim  | |Srv  |   |Srv   | |Srv  | |Srv  | |Srv   |
+--+---+   +--+--+   +--+--+   +--+---+   +--+---+   +--+---+   +--+---+
   |          |          |          |          |          |
   |          |          |          |          |          |
+--v----------v----------v----------v----------v----------v--------+
|                     Message Queue (Kafka)                     |
+--+----------+----------+----------+----------+----------+------+
   |          |          |          |          |          |
   |          |          |          |          |          |
+--v--+   +---v---+  +---v---+  +---v---+  +---v---+  +---v---+   +--v--+
|Obj   |  |Trans- |  |Meta- |  |Comment| |Search | |Metrics|  |Log  |
|Store |  |coder  |  |data DB| |DB (Cassandra) | (ES)  | (TSDB) | (ELK)|
+------+  +-------+  +-------+  +-------+  +-------+  +-------+  +------+
```

**关键组件解释**  

| 组件 | 作用 | 为什么需要 |
|------|------|------------|
| **API Gateway** | 统一入口、协议转换、鉴权、限流 | 防止客户端直连内部服务，简化安全与运维 |
| **Auth Service** | Token 校验（OAuth2/JWT） | 分离认证，支持多端统一登录 |
| **Rate Limiter** | 防止单用户/IP 爆炸式请求 | 保障系统稳定，防 DDOS |
| **Cache Service (Redis)** | 热点视频元数据、评论计数、用户订阅列表 | 缓解 DB 读压，降低延迟 |
| **Upload Service** | 分块上传、文件完整性校验、上传完成回调 | 大文件上传必需的分片/重传机制 |
| **Transcoder Service** | 异步转码、封面生成、切片 (HLS/DASH) | 转码耗时长，采用消息队列解耦 |
| **Playback Service** | 根据用户网络返回合适码率的 CDN URL | ABR 必须在业务层决定 |
| **Search Service** | 倒排索引、全文检索、过滤、排序 | 高并发、低延迟的搜索需求 |
| **Message Queue (Kafka)** | 事件流（上传完成、转码完成、评论写入） | 解耦、削峰、保证可靠投递 |
| **Object Store (S3/MinIO/OSS)** | 原始文件 + 转码后分段存储（多分辨率） | 具备高耐久、几乎无限容量 |
| **Metadata DB (MySQL/PostgreSQL)** | 视频、用户、频道、订阅、播放记录等结构化数据 | 强一致性、事务需求 |
| **Comment DB (Cassandra/ScyllaDB)** | 高并发写入的评论、点赞等 | 按视频分区，写入快，读可接受 |
| **Search Engine (Elasticsearch)** | 关键字、标签、标题全文索引 | 支持 200ms 响应 SLA |
| **Metrics DB (Prometheus/TSDB)** | 监控、报警、业务指标 | 可观测性 |
| **Log System (ELK)** | 日志集中、追踪、审计 | 故障排查 |

> **为什么要把转码、搜索、评论分别使用不同的存储技术？**  
> - 转码是 **CPU/GPU 密集**，但产出是文件，适合对象存储。  
> - 评论写入量极高且对强事务要求不高，使用 **宽列数据库** 能水平扩展。  
> - 搜索需要 **倒排索引**，专门的搜索引擎（ES）在全文检索方面表现更好。

## 第三步：数据库设计  

### 1. 元数据关系型库（MySQL）  
| 表名 | 主键 | 关键字段 | 索引设计 | 备注 |
|------|------|----------|----------|------|
| `users` | `user_id` (BIGINT) | `email`, `username`, `created_at` | 唯一索引 `email`, `username` | 用户基本信息 |
| `channels` | `channel_id` | `owner_user_id`, `title`, `description` | 索引 `owner_user_id` | 频道/创作者 |
| `videos` | `video_id` | `channel_id`, `title`, `duration`, `status`, `created_at` | 索引 `channel_id`, `status`, `created_at` | 视频元数据 |
| `video_files` | `video_id` + `resolution` | `object_key`, `size`, `codec`, `bitrate` | 主键复合索引 | 同一视频的不同分辨率文件 |
| `subscriptions` | `user_id` + `channel_id` | `subscribed_at` | 主键复合索引 | 用户订阅关系 |
| `playback_logs` | `log_id` (auto) | `user_id`, `video_id`, `play_start_ts`, `duration` | 索引 `video_id`, `user_id` | 统计用（可后续迁移到 TSDB） |

- **为什么使用关系型库？**  
  - 视频元数据需要 **事务一致性**（如删除视频时要同步删除多条记录），并且查询模式相对固定（主键或二级索引），关系型库最合适。  
- **分表/分库**：视频表按 **`video_id` 哈希** 或 **`created_at`（时间）** 分库，防止单库写热点。  

### 2. 评论宽列库（Cassandra）  

| 表名 | 主键 | 列 | 说明 |
|------|------|----|------|
| `comments` | **分区键** `video_id` **聚簇键** `comment_id` (timeuuid) | `user_id`, `content`, `created_at`, `like_cnt`, `reply_to` | 读取时可一次性拉取某视频的前 N 条评论，写入不需要全局唯一 ID，timeuuid 保证有序。 |
| `comment_likes` | **分区键** `comment_id` | `user_id` | 用于防止重复点赞，后期可做去重计数。 |

- **为什么用 Cassandra？**  
  - 写放大极低，支持 **每秒万级写**，天然 **按 video_id 分区**，热点视频的写入会分散到多节点。  
  - 读取时可以使用 **分页查询**（`LIMIT` + `token`），满足前端分页需求。  

### 3. 对象存储路径约定  

```
/videos/{video_id}/{resolution}/{segment_index}.ts
/videos/{video_id}/cover.jpg
/videos/{video_id}/manifest.m3u8
```

- **强一致性**：对象存储本身提供 **读后写一致**（如 S3 的 **Read‑After‑Write**），保证上传完毕后立即可读取。  
- **元数据指向**：`video_files` 表保存每个分辨率的 `object_key`，读取时先查询 DB 再去对象存储。  

### 4. 搜索索引（Elasticsearch）  

| 索引 | 必要字段 | 分析器 | 示例映射 |
|------|----------|--------|----------|
| `videos` | `video_id`, `title`, `description`, `tags`, `channel_id`, `status`, `created_at` | `standard` + `edge_ngram`（自动补全） | `title` 使用 `text`，`tags` 使用 `keyword`，`created_at` 用 `date` |

- **为什么使用 ES？**  
  - 支持 **倒排索引 + 相关度排序**，查询性能在 **ms** 级别。  
  - 可水平扩展，适配 **150 万 QPS** 中的大量搜索请求。

## 第四步：核心 API 设计  

下面列出最常用的 **RESTful** 接口（实际可用 gRPC/GraphQL），并说明请求/响应、错误码、幂等性。  

### 1. 上传相关  

| 方法 | 路径 | 功能 | 请求体 | 响应 | 幂等性 |
|------|------|------|--------|------|--------|
| `POST` | `/api/v1/videos/init` | 初始化上传，返回 `upload_id`、分片信息 | `{ "title": "...", "description": "...", "channel_id": "...", "size": 123456 }` | `{ "upload_id": "...", "chunk_size": 5MB, "chunks": N }` | ✅ |
| `PUT` | `/api/v1/videos/{upload_id}/chunks/{chunk_index}` | 上传单个分片（支持断点续传） | 二进制分片 | `204 No Content` | ✅（相同分片可覆盖） |
| `POST` | `/api/v1/videos/{upload_id}/complete` | 合并分片，触发转码 | - | `{ "video_id": "...", "status": "processing" }` | ✅ |

**实现要点**  
- **分块大小**：5 ~ 10 MB（兼顾网络抖动与请求开销）。  
- **校验**：每块带 **MD5**，服务端返回校验结果。  
- **幂等**：`upload_id` 由后端生成，基于用户 ID + 时间戳 + UUID，保证唯一。  

### 2. 播放相关  

| 方法 | 路径 | 功能 | 请求参数 | 响应 | 备注 |
|------|------|------|----------|------|------|
| `GET` | `/api/v1/videos/{video_id}/manifest` | 获取 HLS/DASH manifest URL（已加 CDN） | `?quality=auto` | `{ "manifest_url": "https://cdn.example.com/.../manifest.m3u8" }` | CDN 会做缓存 |
| `GET` | `/api/v1/videos/{video_id}/metadata` | 返回视频基本信息（标题、时长、分辨率列表） | - | `{ "title": "...", "duration": 300, "resolutions": ["360p","720p","1080p"] }` | 可从 Redis 缓存快速返回 |

### 3. 评论互动  

| 方法 | 路径 | 功能 | 请求体 | 响应 | 幂等性 |
|------|------|------|--------|------|--------|
| `POST` | `/api/v1/videos/{video_id}/comments` | 发表评论 | `{ "content": "...", "reply_to": null }` | `{ "comment_id": "...", "created_at": "..."} ` | ✅（同一 `comment_id` 重复提交返回冲突） |
| `GET` | `/api/v1/videos/{video_id}/comments` | 分页获取评论 | `?page=1&size=20` | `{ "comments": [...], "next_token": "..." }` | - |
| `POST` | `/api/v1/comments/{comment_id}/like` | 点赞 | - | `{ "like_cnt": 123 }` | ✅（幂等：内部使用 **唯一用户‑评论** 记录） |

### 4. 订阅 & 推荐  

| 方法 | 路径 | 功能 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/channels/{channel_id}/subscribe` | 关注频道 | - | `{ "subscribed_at": "..."} ` |
| `GET` | `/api/v1/users/{user_id}/feed` | 获取订阅视频列表（倒序） | `?page=1&size=20` | `{ "videos": [...], "next_token": "..." }` |
| `GET` | `/api/v1/users/{user_id}/recommend` | 简单推荐（最新 + 热门） | - | `{ "videos": [...] }` |

### 5. 搜索  

| 方法 | 路径 | 功能 | 请求体 | 响应 |
|------|------|------|--------|------|
| `GET` | `/api/v1/search` | 关键字搜索 | `?q=cat&tags=funny&sort=rel` | `{ "hits": [{ "video_id": "...", "title": "..."}], "total": 12345 }` |

**安全与限流**  
- 所有写操作（上传、评论、点赞）均需要 **JWT** 鉴权。  
- **Rate Limiter**：对每个 IP / token 限制 QPS（如 10 req/s），超过返回 `429 Too Many Requests`。

## 第五步：详细组件设计  

### 1. 上传与分块存储  
1. **客户端** → 调用 `init` 获取 `upload_id` 与分块大小。  
2. **分块上传**：每块直接写入 **对象存储的临时目录**（如 `tmp/{upload_id}/{chunk_index}`），**不经过业务服务**，降低网关压力。  
3. **完整性校验**：每块返回 `ETag`（MD5），客户端在完成后发送 `complete`，服务端检查所有块的 MD5 列表与总文件 MD5 是否匹配。  
4. **合并**：`complete` 触发 **Lambda/函数**（或后台 Worker）在对象存储内部执行 **multipart copy**，生成 `original/{video_id}.mp4`。  

> **不采用**：直接把所有分块写入关系型库，会导致 **磁盘 I/O 与事务开销**，不适合大文件。  

### 2. 转码流水线  
- **消息队列**：`upload_complete` 事件写入 Kafka **`video-transcode`** topic。  
- **转码 Worker**（基于容器化的 FFmpeg、GPU）消费该 topic：  
  1. 下载原始文件（S3） → 本地磁盘或内存映射。  
  2. 生成多码率（240p、360p、480p、720p、1080p）HLS 切片 + 封面图。  
  3. 上传切片、manifest、封面回对象存储的对应目录。  
  4. 更新 `videos.status = 'ready'`、写入 `video_files`。  
- **并发控制**：使用 **消费者组**、**限流**（如每台机器最多 4 个并发转码），确保 **转码时延 ≤ 30 min**。  

### 3. 播放流控（ABR）  
- **客户端**：使用 **Media Source Extensions** 或 **native HLS/DASH** 播放器。  
- **服务端**：`manifest` 中列出每种分辨率的切片 URL（已 CDN 加速）。  
- **动态码率切换**：播放器根据 **网络带宽测算**（下载最近 3 个切片的速度）自动切换 `resolution`。  
- **首帧优化**：在转码完成后，**将第一个切片的第一个帧（I‑frame）提前复制到 CDN 边缘的热点缓存**（预热），实现 **<150 ms 首帧**。  

### 4. CDN 与边缘缓存  
- **缓存键**：`/videos/{video_id}/{resolution}/{segment_index}.ts` → TTL **30 天**（不变的切片）。  
- **热点预热**：每日统计 **播放次数 Top 10k** 视频，将其 **全分辨率切片** 主动推送到 CDN 边缘节点（使用 CDN 提供的 **push API**）。  
- **失效策略**：  
  - **新上传/转码完成** → 立即 **purge** 对应旧缓存（若视频被重新编辑）。  
  - **删除视频** → **批量 purge** 所有关联对象键。  

### 5. 评论写入路径（解决热点）  
1. **写入口**：API Gateway → Comment Service → 写入 **Kafka `comment-write`** + **Cassandra**（同步写）。  
2. **热点缓冲**：对于 **瞬时热点视频**（如热点 10 秒内写入量 > 5k/s），使用 **Redis 计数器** 做 **写放大**：先写入 Redis，后台异步批量落库到 Cassandra。  
3. **去重 & 防刷**：每条评论在写入前检查 **用户‑视频‑时间窗口**（如 30 秒内同内容不允许重复），使用 **Redis Set** 实现。  
4. **点赞**：采用 **幂等写**（先判断是否已点赞），再在 **Cassandra** 中 **increment** `like_cnt`，并写入 `comment_likes` 表防重复。  

### 6. 搜索实现  
- **同步索引**：视频元数据写入 MySQL 成功后，业务服务发送 **`video-index`** 事件到 Kafka，**ES Indexer** 消费并做 **upsert**。  
- **异步容错**：若 ES 写入失败，事件保存在 **Kafka**，重试机制保证最终一致性。  
- **查询路径**：API Gateway → Search Service → ES → 返回 video_id 列表 → 批量查询 MySQL（或 Redis 缓存）补全标题、封面等信息。  

### 7. 监控、日志、告警  
| 维度 | 工具 | 关键指标 |
|------|------|----------|
| **系统监控** | Prometheus + Grafana | CPU、内存、网络、请求 QPS、错误率 |
| **业务指标** | ClickHouse / TSDB | 播放次数、转码时延、上传成功率、评论写入 TPS |
| **日志** | ELK (Filebeat → Logstash → Elasticsearch) | 请求链路日志、异常栈、慢查询 |
| **告警** | Alertmanager | 响应时间 > 200 ms、转码队列堆积 > 10 min、CDN 缓存命中率 < 70% |

## 第六步：扩展性与高可用设计  

下面把系统从 **单机** 演进到 **多可用区、多 Region**，并说明每一步的 **痛点** 与 **解决方案**。  

### 1. 横向扩容（Scale‑out）  

| 组件 | 扩容方式 | 关键技术点 |
|------|----------|-----------|
| API Gateway | 增加实例 → 配合 **L4 Load Balancer**（如 ALB） | 采用 **无状态**，会话保持交给前端（JWT） |
| Auth Service | 多副本 + Redis Session 共享（若需要） | JWT 无状态，基本不需要扩容 |
| Upload Service | **分片直写对象存储** → 只受网络带宽限制 | 通过 **对象存储多 AZ** 自动复制 |
| Transcoder Workers | **容器集群（K8s）**，水平 Pod 扩容 | 动态伸缩（基于 Kafka 队列长度） |
| Metadata DB | **读写分离**（主库 + 多只读副本）+ **分库分表**（video_id hash） | 采用 **Galera** / **MySQL Group Replication** 提供自动故障转移 |
| Comment DB | **Cassandra** 天生水平扩展，添加节点即可 | 采用 **NetworkTopologyStrategy** 跨 AZ |
| Search (ES) | **分片 + 副本**，每个 Region 部署独立集群，跨 Region **跨集群复制** | 通过 **CCR (Cross‑Cluster Replication)** 保持一致性 |
| CDN | 多节点边缘覆盖，自动路由最近节点 | 使用 **Anycast DNS** 实现全球低延迟 |

### 2. 多可用区容灾（Active‑Active）  

- **数据复制**：对象存储开启 **跨 AZ 同步复制**（如 S3 Cross‑Region Replication）。  
- **数据库高可用**：MySQL 主从使用 **半同步**，故障时自动切换到最近的从库（使用 **Orchestrator**）。  
- **Kafka**：跨 AZ **MirrorMaker** 保持 Topic 副本，确保消息不丢失。  
- **服务发现**：使用 **Consul / etcd**，服务实例在不同 AZ 注册，客户端通过 DNS 轮询。  

### 3. 灾难恢复（DR）  

| 场景 | 恢复策略 |
|------|----------|
| 单 AZ 完全失效 | 其他 AZ 已有完整副本（对象存储、DB、Kafka），流量自动切换至健康 AZ |
| Region 完全失效 | 在另一个 Region 部署 **只读副本**（MySQL、Cassandra、ES），利用 **DNS 切换**（TTL 60s）进行灾备切换；数据同步使用 **跨 Region 流复制**（对象存储、Kafka） |
| 数据库崩溃 | 使用 **备份 + PITR**（MySQL binlog）恢复到最近时间点，恢复期间 **只读** 访问，等写入恢复后切回读写模式 |  

### 4. 关键性能瓶颈的预防  

| 瓶颈 | 预防措施 |
|------|----------|
| **转码队列积压** | 动态扩容转码 Worker；使用 **分层转码**（先 360p → 再 720p/1080p）降低峰值算力需求 |
| **评论热点写入** | 先写入 Redis 缓冲，再批量落库；对热点视频开启 **分片键**（video_id + shard） |
| **搜索延迟** | 为热点查询开启 **热缓存**（Redis）+ **预计算排序**；ES 通过 **副本** 提升并发读取 |
| **CDN 缓存穿透** | 对不存在的对象返回 **404** 并在 CDN 设置 **负载均衡**；对异常请求返回 **错误码** 并限流 |

## 第七步：常见面试追问与回答  

### Q1️⃣：**同一视频的不同分辨率切片在读取时如何保证强一致性？**  

**回答要点**  
1. **对象存储的读后写一致（Read‑After‑Write）**：大多数云对象存储（S3、OSS）在 **PUT 完成** 后立即可以 **GET** 到最新内容。  
2. **元数据表（video_files）** 采用 **强事务**（MySQL）保存每个分辨率对应的 `object_key` 与 `etag`。读取时先查询该表，拿到最新的 `etag` 再访问对象存储。  
3. **更新流程**：如果需要替换已发布的切片（比如重新转码），先 **写入新对象**（新 key），再 **在事务中更新 video_files**，最后 **发送 CDN purge**。在此期间旧对象仍然可用，读到的始终是 **事务中最新的 key**，因此对外呈现强一致。  

**不这样做的后果**：若直接覆盖同一 object_key，CDN 可能缓存旧内容导致用户看到旧分辨率或画质，产生不一致体验。  

---

### Q2️⃣：**在 CDN 与边缘缓存层，如何设计缓存失效和热点视频的预热机制？**  

**回答要点**  
- **缓存失效（TTL）**：切片内容几乎不变，TTL 设为 **30 天**，配合 **Cache‑Control: immutable**。  
- **主动预热**：每天凌晨统计 **前 24 小时播放次数 Top 10k** 视频，使用 CDN 提供的 **push API** 把所有分辨率切片推送至边缘节点。  
- **失效时机**：  
  1. **视频删除** → 调用 CDN **purge** 所有关联路径。  
  2. **重新转码**（如作者上传新版本） → 先生成新对象 key，更新 DB 后 **purge** 旧 key，避免缓存击穿。  
- **热点穿透防护**：对 **查询不到的对象**（非法 video_id）返回 **404** 并在 CDN 配置 **短 TTL（60 s）**，防止恶意请求占满缓存。  

**不做预热的后果**：热点视频首次请求仍需回源到对象存储，导致 **首帧延迟 > 150 ms**，影响用户体验。  

---

### Q3️⃣：**面对突发的评论潮（如热门视频上线），如何设计评论写入路径以避免热点写入导致的性能瓶颈？**  

**回答要点**  
1. **写入分区**：Cassandra 按 **`video_id` + `shard_id`**（如 `hash(comment_id) % 10`）做二级分区，把热点视频的写入分散到 **10** 个子分区。  
2. **写入缓冲**：在 API 层先写入 **Redis List**（或 **Stream**），后台 Consumer 批量写入 Cassandra（每 100 条一次批处理），大幅降低单条写入的网络往返。  
3. **限流 & 速率控制**：对单个 IP / UserID 设置 **QPS 上限**（如 5 条/s），超过返回 `429`，防止刷评论攻击。  
4. **幂等 & 防刷**：利用 **Redis Set** 记录最近 30 秒内同用户的评论内容哈希，重复提交直接返回成功但不写入。  

**不做分片或缓冲**：所有评论直接写入单表，热点视频会导致 **单节点写入速率饱和**，出现 **写入延迟>1s**，甚至 **节点崩溃**。  

---

### Q4️⃣：**如果用户上传超大文件（比如 8 GB），如何保证上传的可靠性和低延迟？**  

**回答要点**  
- **分块上传 + 断点续传**：客户端把文件切成 5 ~ 10 MB 块，每块独立请求，失败后只重传该块。  
- **并行上传**：允许 **4~8 条并行流**（受网络带宽限制），显著提升总上传速率。  
- **上传完成校验**：客户端在所有块上传完后，发送 `complete`，服务器端比对 **MD5 列表** 与 **整体 MD5**，若不匹配返回错误，客户端重新上传错误块。  
- **后台直传到对象存储**：块直接写入对象存储的 **multipart upload** 接口，避免先落到业务服务器，降低网关压力。  

---

### Q5️⃣：**如何监控转码时延并在 SLA 触发时自动扩容？**  

**回答要点**  
- **监控指标**：`video_transcode_queue_length`（Kafka 待消费消息数） + `average_transcode_time`（Worker 处理单条消息的平均时长）。  
- **告警阈值**：当 `queue_length > 10,000` 或 `average_time > 25 min` 时触发 **自动伸缩**。  
- **自动伸缩**：Kubernetes HPA（Horizontal Pod Autoscaler）根据自定义指标（Kafka lag）动态增加转码 Worker 副本数。  
- **扩容后**：新节点加入后自动消费队列，队列长度快速下降，保证 **30 min SLA**。

## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 思考过程 | 最终方案 |
|------|----------|----------|
| **如何同时满足超低首帧延迟 (<150 ms) 与海量存储** | 首帧延迟受 CDN、网络 RTT、缓存命中率影响；而海量视频又导致热点分布不均。最初想把所有视频都预热到 CDN，成本太高。于是权衡 **热点预热 + 缓存失效策略**，只对 Top 10k 视频进行主动推送，其他视频依赖 **CDN 读取原始切片**，并把 **首帧切片单独标记为 immutable**，确保缓存命中时几乎为 0 ms。 | **热点预热 + immutable 首帧切片 + CDN 边缘缓存** |
| **评论写入的热点与强一致性** | 评论量在热点视频上会瞬间冲刺到上万 TPS，单表写入会出现写放大。考虑使用 **Cassandra**（宽列）但仍担心单分区热点。最终决定 **二级分区 + Redis 缓冲**，把写入压力平滑到后台批处理。 | **二级分区 + Redis 缓冲 + 批量写入** |

### 2️⃣ 新手最容易犯的错误  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有业务都放在单体服务里** | 随着流量增长，单体无法水平扩展，故障影响全局。 | **微服务化**：每个业务（上传、转码、播放、评论、搜索）独立服务，使用轻量网关做统一入口。 |
| **忽视数据一致性与缓存失效** | 只考虑读写性能，忘记在对象存储、CDN、DB 之间保持一致性，导致用户看到旧视频或错误数据。 | **读写分离 + 事务更新元数据 + CDN purge**，确保任何一次更新在用户侧都是原子可见的。 |
| **直接让客户端上传到业务服务器** | 大文件会把业务服务器的磁盘、网络压垮。 | **客户端直传对象存储的 multipart upload**，业务层只负责控制和回调。 |
| **把评论直接写入 MySQL** | 写入高并发时会锁表、导致慢查询。 | **使用宽列/NoSQL（Cassandra）**，并结合 **写缓冲**。 |
| **没有监控和告警** | 系统出问题时不知道根因，恢复慢。 | **全链路监控**（Prometheus + Grafana） + **业务指标**（ClickHouse） + **自动伸缩**。 |
