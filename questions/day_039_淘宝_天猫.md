# 系统设计面试题：淘宝 / 天猫核心交易系统

## 1. 题目背景
淘宝 / 天猫是中国最大的 C2C 与 B2C 电商平台，提供商品浏览、搜索、下单、支付、物流跟踪等全链路购物服务。面试官希望你从 **“核心交易系统”**（即用户下单 → 支付 → 订单生成 → 库存扣减）的角度进行系统设计。

## 2. 面试场景设定
> **面试官**：  
> “假设我们要重新设计淘宝/天猫的核心交易系统，支撑用户的下单、支付、订单生成以及库存扣减全流程。请先从整体架构出发，描述关键模块、数据流以及如何满足业务高并发和高可用的要求。我们可以先聚焦在 2025 年“双11”期间的峰值需求上。”  

（面试官会在你回答后根据你的方案进行细化追问）

## 3. 功能性需求
| 编号 | 需求描述 |
|------|----------|
| 1 | **商品浏览 & 加入购物车**：用户可以搜索商品、查看详情并将商品加入购物车。 |
| 2 | **下单 & 预校验**：用户提交订单时，系统需要检查商品价格、促销规则、用户收货地址合法性等。 |
| 3 | **库存扣减（强一致性）**：下单成功后立即扣减库存，防止超卖。 |
| 4 | **支付集成**：对接支付宝、微信支付等渠道，支持同步/异步回调，保证支付成功后订单状态更新。 |
| 5 | **订单状态管理**：订单从“待支付”→“已支付”→“待发货”→“已完成”等全流程状态变更，支持查询与售后。 |
| 6 | **幂等与容错**：在网络抖动或重复请求的情况下，保证下单、支付、库存扣减等操作的幂等性。 |

## 4. 非功能性需求
| 指标 | 目标值 | 说明 |
|------|--------|------|
| DAU（活跃用户） | 5 × 10⁸ | 包括移动端 & PC 端 |
| 峰值 QPS（下单） | 150 万 QPS | 双11 24 h 峰值，单秒最高 150 万订单请求 |
| 平均响应时间（下单 → 库存扣减完成） | ≤ 200 ms | 95% 请求在 200 ms 内返回 |
| 可用性 | ≥ 99.99%（年均） | 关键交易链路（下单、支付、库存）全年累计宕机不超过 52.56 分钟 |
| 数据存储量 | 订单数据 5 PB、商品库存 2 TB | 包含历史订单、审计日志、库存快照等 |
| 跨地域容灾 | 同城 99.9% 响应、跨省 99.99% 可用 | 多活部署，容灾切换 RTO ≤ 5 min，RPO ≤ 1 min |

## 5. 系统边界
### 本题范围（需要设计）
- **核心交易链路**：下单、库存扣减、支付回调、订单状态流转。  
- **高并发调度**：包括限流、分库分表、分布式锁、消息队列等。  
- **一致性方案**：强一致性库存、最终一致性订单状态。  
- **容灾与监控**：多活部署、故障自动切换、业务健康监控。  

### 不在本题范围（可不考虑）
- 商品推荐、搜索引擎、营销活动（满减、秒杀）细节。  
- 商品评价、社区、直播、内容运营等。  
- 第三方物流的全链路跟踪（仅需订单状态到“已发货”）。  
- 支付安全（如风控、风控模型）细节，只需保证支付成功回调的可靠性。

## 解题思路总览
1. **从业务切入**：先把“下单 → 库存扣减 → 支付 → 订单状态流转”拆成**独立的业务子系统**（下单系统、库存系统、支付系统、订单系统）。  
2. **从最小可用系统（MVP）做起**：只用单体服务、单库、同步调用，先验证业务完整性。  
3. **逐步加入**：  
   - **并发控制**（限流、熔断、分布式锁）  
   - **异步化**（消息队列）  
   - **数据分片**（分库分表、读写分离）  
   - **容灾**（多活、跨地域复制）  
   - **可观测性**（日志、监控、链路追踪）  
4. **一致性模型**：库存必须 **强一致**（防止超卖），订单状态、支付可以采用 **最终一致**（通过可靠的消息与幂等设计实现）。  
5. **每一次技术选型，都要回答**：  
   - **为什么要这样**（解决什么痛点）  
   - **不这么做会出现什么问题**（性能、可靠性、成本）  

下面按照上述思路，逐章展开。

## 第一步：理解需求与规模估算  

### 1. 功能性需求梳理  

| 编号 | 关键业务点 | 需要关注的技术细节 |
|------|-----------|-------------------|
| 1 | 商品浏览、加入购物车 | 主要是 **读**，高并发缓存，后面不深入 |
| 2 | 下单 & 预校验 | 需要 **价格、促销、地址** 验证，**事务** 入口 |
| 3 | 库存扣减（强一致） | 防止 **超卖**，必须 **原子** 扣减 |
| 4 | 支付集成 | **异步回调**，幂等，防止重复收款 |
| 5 | 订单状态管理 | 状态机、查询、可回滚（售后） |
| 6 | 幂等与容错 | 网络抖动、重复请求、幂等键设计 |

### 2. 非功能性需求量化  

| 指标 | 目标 | 计算方式/来源 |
|------|------|----------------|
| DAU | 5 × 10⁸ | 估算 500M 活跃用户，日均请求 2‑3 次 |
| 峰值 QPS（下单） | 1.5 M QPS | 150 万 QPS ≈ 150 万 * 1s，约 13.5 亿单/天 |
| 响应时间 | ≤200 ms | 业务链路 5 步，单步 ≤40 ms |
| 可用性 | 99.99% 年均 | 年宕机 ≤52.56 min |
| 数据规模 | 订单 5 PB，库存 2 TB | 订单 10 KB/单 × 5 B ≈ 5 PB；库存 100 GB × 多副本 |
| 跨地域容灾 | RTO ≤5 min, RPO ≤1 min | 多活、双活、异步复制 |

### 3. 关键瓶颈点（先打草稿）  

| 业务阶段 | 可能的瓶颈 | 对应指标 |
|----------|------------|----------|
| 下单预校验 | DB 读取、业务校验 | QPS、响应时延 |
| 库存扣减 | 并发写、分布式锁 | 强一致性、超卖 |
| 支付回调 | 高并发回调、幂等 | QPS、可靠性 |
| 订单写入 | 大批量写、分库分表 | QPS、存储扩容 |

> **小结**：在设计之前，我们已经把业务拆成了**四个核心子系统**，并且明确了每个子系统的 **性能目标** 与 **一致性需求**。后面的架构将围绕这些目标展开。

## 第二步：高层架构设计  

### 1. 最小可用系统（MVP）  

```
+-------------------+      +-------------------+
|   Web / Mobile   | ---> |   API Gateway     |
+-------------------+      +-------------------+
                               |
                               v
                      +-----------------+
                      |   Transaction   |
                      |   Service (Mono)|
                      +-----------------+
                               |
                               v
                      +-----------------+
                      |   MySQL (single)|
                      +-----------------+
```

- **单体 Service** 包含：下单、库存扣减、订单写入、支付回调（同步调用）  
- **单库** 用 InnoDB，开启 **行锁** 保证库存扣减的原子性  
- **优势**：实现快速原型，验证业务流；**缺点**：无法支撑 150 万 QPS，单点故障。  

> **面试官可能追问**：如果流量翻倍怎么办？  
> **回答**：我们在后面会把单体拆成微服务、引入异步消息、做分库分表、实现多活。

#### 关键模块解释  

| 模块 | 作用 | 为什么要单独拆出来 |
|------|------|-------------------|
| **API Gateway** | 统一入口、限流、鉴权、灰度发布 | 防止直达业务层，统一治理 |
| **防刷/限流/熔断** | 防止恶意流量冲垮后端 | 业务高峰时保护核心服务 |
| **业务微服务（Order/Stock/Payment）** | 职责单一、可独立水平扩容 | 避免单体服务成为性能瓶颈 |
| **消息队列（Kafka）** | 异步化库存扣减、订单状态、支付回调 | 解耦、削峰、提供可靠投递 |
| **Redis（缓存+分布式锁）** | 商品详情、库存预热、幂等键、限流计数 | 读写分离、快速响应 |
| **MySQL（分库分表）** | 事务强一致的核心数据 | 行锁 + 主键分片保证写入性能 |
| **Elasticsearch** | 订单检索、审计、报表 | 读多写少的搜索需求 |
| **日志/审计系统** | 全链路追踪、故障回溯 | 满足合规与监控需求 |

#### 架构演进路线（图）  

1. **MVP** → 2. **单体拆分**（API GW + 三个微服务） → 3. **引入 MQ**（库存扣减异步） → 4. **读写分离 + 缓存** → 5. **分库分表** → 6. **多活跨地域**  

> **面试官追问**：“如果库存强一致性要求导致写入延迟怎么办？”  
> **回答**：采用 **“先预扣 + 最终确认”** 方案：①在 Redis 中做 **预扣（原子 decrement）**，②异步写入 MySQL 并使用 **唯一库存事务 ID** 做幂等，③回滚时再加回。这样既保持强一致，又把耗时的磁盘写入搬到异步路径。

## 第三步：数据库设计  

### 1. 选型原则  

| 需求 | 推荐存储 | 原因 |
|------|----------|------|
| **强一致的库存** | **MySQL（InnoDB）** + **Redis（原子递减）** | 行锁 + 事务保证不超卖；Redis 预扣降低 DB 压力 |
| **订单大规模写入** | **MySQL（分库分表）** + **Kafka** 作为写入缓冲 | 关系模型适合事务，Kafka 提供削峰 |
| **查询/报表** | **Elasticsearch** | 支持复杂搜索、聚合，读写分离 |
| **幂等键/限流计数** | **Redis**（String / Bitmap） | 单机内存快，TTL 失效自动回收 |
| **审计日志** | **HDFS + Hive** 或 **ClickHouse** | 大批量写入、列式存储便于分析 |

### 2. 库表设计（核心）  

#### 2.1 商品（goods）  

| 字段 | 类型 | 备注 |
|------|------|------|
| goods_id | BIGINT PK | 全局唯一 |
| name | VARCHAR(200) | 商品名称 |
| price | DECIMAL(12,2) | 当前售价 |
| stock_total | BIGINT | 库存总量（只读） |
| version | BIGINT | 乐观锁版本号 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 最后更新时间 |

> **说明**：`version` 用于 **乐观锁** 防止并发更新冲突（若采用乐观锁方案）。  

#### 2.2 库存（stock）  

| 字段 | 类型 | 备注 |
|------|------|------|
| stock_id | BIGINT PK | 同 goods_id（或复合键） |
| goods_id | BIGINT FK | 对应商品 |
| quantity | BIGINT | 可售库存（实际可扣减） |
| reserved | BIGINT | 已预扣但未确认的库存（用于预扣） |
| version | BIGINT | 乐观锁版本 |
| updated_at | DATETIME | 更新时间 |

> **业务模型**：**可售库存 = quantity - reserved**。当下单成功后，`reserved` 加 1；支付成功后 `reserved` 减 1，`quantity` 减 1（真正扣减）。  

#### 2.3 订单（order）  

| 字段 | 类型 | 备注 |
|------|------|------|
| order_id | BIGINT PK | 全局唯一（雪花算法） |
| user_id | BIGINT | 购买用户 |
| status | TINYINT | 0‑待支付、1‑已支付、2‑待发货、3‑已完成、4‑已关闭 |
| total_amount | DECIMAL(12,2) | 订单总价 |
| payment_id | BIGINT | 对应支付记录 |
| create_time | DATETIME | 订单创建时间 |
| update_time | DATETIME | 状态更新时间 |
| version | BIGINT | 乐观锁（防止并发更新） |

#### 2.4 订单明细（order_item）  

| 字段 | 类型 | 备注 |
|------|------|------|
| id | BIGINT PK | 主键 |
| order_id | BIGINT FK | 所属订单 |
| goods_id | BIGINT | 商品 |
| price | DECIMAL(12,2) | 下单时的单价 |
| quantity | INT | 购买数量 |
| discount | DECIMAL(12,2) | 折扣金额（可选） |

#### 2.5 支付（payment）  

| 字段 | 类型 | 备注 |
|------|------|------|
| payment_id | BIGINT PK | 全局唯一 |
| order_id | BIGINT FK | 关联订单 |
| channel | VARCHAR(20) | alipay / wechat |
| amount | DECIMAL(12,2) | 支付金额 |
| status | TINYINT | 0‑待回调、1‑成功、2‑失败 |
| callback_time | DATETIME | 支付回调时间 |
| created_at | DATETIME | 创建时间 |

### 3. 分库分表策略  

- **订单库**：按照 **order_id** 的 **后 3 位**（0~999）做 **水平分表**，每 10,000 张表（每库 100 表） → 每库约 **50 GB**（假设 10 KB/单）。  
- **商品/库存库**：按 **goods_id** 的 **哈希**（% 128）划分 **128 库**，每库约 **15 GB**（2 TB / 128）。  

> **为什么不把库存放在同库**：库存写热点极高，独立库可以专门调优（更高的 `innodb_buffer_pool_size`、更快的磁盘）。  

### 4. 强一致库存实现方案（两种）  

| 方案 | 实现步骤 | 优点 | 缺点 |
|------|----------|------|------|
| **A. DB 行锁（悲观锁）** | 1. 在 MySQL `stock` 表上 `SELECT ... FOR UPDATE` <br>2. 检查 `quantity - reserved >= qty` <br>3. `UPDATE stock SET quantity = quantity - qty, version = version+1 WHERE stock_id=? AND version=?` | 直接使用事务，最安全 | 高并发下锁竞争严重，吞吐受限 |
| **B. Redis 预扣 + MySQL 异步落库** | 1. `DECRBY` Redis key `stock:{goods_id}`（原子）<br>2. 若返回值 < 0，回滚 `INCRBY` 并返回库存不足<br>3. 成功后写入 Kafka `stock-deduct` 事件<br>4. 消费端持久化到 MySQL 并做幂等校验 | 极高并发，Redis 低延迟，削峰 | 需要可靠的 MQ + 幂等保证，极端情况下仍可能出现 **短暂不一致**（但业务上可接受） |
| **推荐**：在“双11”期间，**方案 B** 更符合 150 万 QPS 的需求，**方案 A** 作为备份或在库存非常紧张的商品上使用。

## 第四步：核心 API 设计  

> **原则**：**幂等、简洁、明确的错误码**。所有外部调用（包括内部微服务之间）均采用 **HTTP+JSON** 或 **gRPC**（内部推荐 gRPC）实现。下面给出最核心的几组 API。  

### 1. 下单 API（Order Service）  

| 方法 | 路径 | 请求体 | 响应体 | 说明 |
|------|------|--------|--------|------|
| `POST` | `/order/create` | ```json { "user_id":123, "items":[ {"goods_id":1001,"quantity":2}, {"goods_id":1002,"quantity":1} ], "address_id":987, "client_token":"uuid-xxxx" } ``` | ```json { "order_id": "202505270001", "status":0, "expire_at":"2025-05-27T15:30:00Z" } ``` | **幂等键** `client_token`（UUID）在 Redis 中保存 24h，重复请求直接返回已有 `order_id`。 |
| `GET` | `/order/{order_id}` | - | ```json { "order_id":..., "status":..., "items":[...], "total_amount":... } ``` | 查询订单详情（读缓存优先） |

**内部实现流程（伪代码）**  

```go
func CreateOrder(req CreateOrderReq) (CreateOrderResp, error) {
    // 1. 幂等检查
    if orderID := redis.Get("order:client_token:" + req.ClientToken); orderID != "" {
        return GetOrderResponse(orderID)
    }

    // 2. 预校验（价格、地址、促销） → 业务服务层
    // 3. 调用 Stock Service 预扣库存（Redis 原子 DECR）
    // 4. 若任一商品库存不足，回滚已扣减库存并返回错误
    // 5. 生成全局唯一 order_id (Snowflake)
    // 6. 写入 MySQL (order, order_item)（事务）
    // 7. 写入 Kafka topic "order_created"（供后续支付、审计）
    // 8. 写入幂等键到 Redis（TTL 24h）
    // 9. 返回成功
}
```

### 2. 库存扣减（Stock Service）  

| 方法 | 路径 | 请求体 | 响应体 |
|------|------|--------|--------|
| `POST` | `/stock/deduct` | ```json { "goods_id":1001, "quantity":2, "order_id":"2025...", "client_token":"uuid-xxxx" } ``` | ```json { "success":true } ``` |

- **幂等**：`order_id` + `goods_id` 组合在 Redis 中做 **SETNX**（TTL 30min），防止同一订单重复扣减。  
- **返回码**：`0` 成功、`-1` 库存不足、`-2` 已扣减（幂等）  

### 3. 支付回调（Payment Service）  

> **外部支付渠道（支付宝、微信）** 会 **POST** 到我们提供的 **/payment/callback**，内容示例：  

```json
{
  "order_id":"2025...",
  "payment_id":"pay_12345",
  "channel":"alipay",
  "amount":199.00,
  "status":"SUCCESS",
  "sign":"xxxxx"
}
```

**处理要点**  

1. **验签**（支付方提供公钥）  
2. **幂等键**：`payment_id` 在 Redis 中 `SETNX`，已处理直接返回 `200 OK`  
3. **事务**：更新 `payment` 表、更新 `order` 状态、发送 `order_paid` Kafka 消息（用于后续发货）  
4. **异常回滚**：若订单已关闭或已支付，直接返回成功，避免重复扣钱  

### 4. 发货（简化）  

| 方法 | 路径 | 请求体 | 响应体 |
|------|------|--------|--------|
| `POST` | `/order/ship` | ```json { "order_id":"2025...", "logistics_no":"SF12345678" } ``` | ```json { "success":true } ``` |

- **业务规则**：只能对 `status = 已支付` 的订单进行发货。  
- **实现**：更新订单状态为 `待发货 → 已发货`，写入物流信息，发送 `order_shipped` 事件。

### 2. 限流、熔断、降级  

| 场景 | 方案 | 关键指标 |
|------|------|----------|
| **全局下单流量** | **令牌桶（Token Bucket）** + **统一阈值** | 1.5M QPS |
| **单商品秒杀** | **库存预热 + Redis 限流**（`stock:{gid}`）| 防止瞬间超卖 |
| **支付回调** | **熔断器（Hystrix）** | 当回调异常率 > 50% 连续 5s，短路进入降级返回成功（业务侧补偿） |

### 3. 分布式锁（防止超卖）  

| 场景 | 方案 | 代码示例（Redis） |
|------|------|-------------------|
| **库存预扣** | **SETNX + Expire**（乐观锁） | `SET stock_lock:1001 nx ex 5` |
| **订单号生成** | **Snowflake**（本地算法） | 通过 **Zookeeper** 选主获取机器 ID，保证唯一性 |
| **支付回调幂等** | **Redis SETNX**（payment_id） | `SETNX payment:pay_12345 1 EX 3600` |

> **为什么不直接用 DB 行锁**：在 150 万 QPS 的峰值，DB 行锁会导致大量的等待、死锁，吞吐下降。Redis 原子操作的 **微秒级** 延迟更适合高并发抢占。  

### 4. 消息队列（Kafka）  

| 主题 | 生产者 | 消费者 | 关键属性 |
|------|--------|--------|----------|
| `order_created` | Order Service | Stock Service（预扣）、Payment Service（发起支付） | **至少 3 副本**，**ACK=all**，**压缩** |
| `stock_deduct` | Stock Service（预扣成功） | MySQL Stock Consumer（持久化） | **事务**：先写 DB，再提交 offset |
| `order_paid` | Payment Service | Order Service（状态更新）、Logistics Service | **幂等消费**（使用 `order_id` 作为唯一键） |
| `order_shipped` | Logistics Service | Order Service（状态流转） | **延迟消息**：用于订单超时未发货的自动关闭 |

**幂等消费实现**（Kafka Consumer）  

```java
public void onMessage(ConsumerRecord<String, OrderPaidEvent> record) {
    String orderId = record.key();
    if (processedCache.putIfAbsent(orderId, true)) { // Redis SETNX
        // 正常业务处理
        orderDao.updateStatus(orderId, PAID);
    } else {
        // 已处理，直接 ack
    }
}
```

### 2. 容灾与跨地域部署  

#### 2.1 多活架构（同城 + 跨省）  

```
Region A (华东)          Region B (华北)
+-------------------+   +-------------------+
| API GW (Active)   |   | API GW (Active)   |
| Order Service (A) |   | Order Service (B) |
| Stock Service (A) |   | Stock Service (B) |
| Payment Service(A)|   | Payment Service(B)|
| MySQL Primary(A)  |   | MySQL Primary(B)  |
| Redis Cluster(A)  |   | Redis Cluster(B)  |
| Kafka(A)          |   | Kafka(B)          |
+-------------------+   +-------------------+
        ^                     ^
        |   数据复制 (双向)   |
        +---------------------+
```

- **数据同步**  
  - **订单**：使用 **MySQL 双向复制**（GTID），冲突极少（因为订单 ID 全球唯一）。  
  - **库存**：采用 **Redis 主从+双向同步** + **Kafka 跨地域 Mirror**，确保 **库存变更** 在两地几乎实时（RPO ≤ 1 min）。  
- **流量切换**  
  - **DNS 级别**（AliDNS）配合 **GeoDNS**，故障时 **TTL** 设为 30s，实现 **5 min** 内切换（RTO ≤ 5 min）。  
  - **全局限流**：在每个 Region 使用 **统一的限流 Token Bucket**（基于 Redis Cluster 跨 Region），防止两地同时超卖。  

#### 2.2 灾难恢复（DR）  

| 步骤 | 操作 | 时长 |
|------|------|------|
| 1 | 监控检测到 Region A 网络不可达 | ≤ 1 min |
| 2 | 自动把 DNS 指向 Region B（TTL=30s） | ≤ 30 s |
| 3 | 启动 Region B 的 **写入代理**，把写请求路由到 Region B 主库 | ≤ 1 min |
| 4 | 业务侧对库存使用 **乐观锁 + 幂等**，防止两地同时扣减同一库存 | 持续 |

### 3. 数据安全与合规  

- **事务日志**：MySQL binlog、Kafka 消费位点、Redis AOF，满足 **RPO ≤ 1 min**。  
- **审计日志**：所有订单、支付、库存变更写入 **HDFS + Hive**，保留 2 年。  
- **脱敏**：用户敏感信息（手机号、身份证）加密存储（AES-256），仅业务服务可解密。  

### 4. 性能优化技巧  

| 场景 | 优化手段 |
|------|----------|
| **库存热点** | - **热点商品预热到 Redis** <br>- **使用分布式计数器（Redis Lua 脚本）** <br>- **使用局部缓存 + 失效回写** |
| **订单写入** | - **批量写入**（一次 INSERT 多行） <br>- **Kafka + MySQL 双写**（异步） |
| **支付回调高并发** | - **回调排队（Kafka）**，消费端 **幂等** <br>- **使用线程池、限流** |
| **查询订单** | - **读写分离**，热点订单放到 **Redis Cache**（TTL 5 min） <br>- **Elasticsearch** 进行全文检索 |

## 第七步：常见面试追问与回答  

下面列出常见的**深度追问**，并给出 **思考过程 + 参考答案**，帮助你在现场快速组织语言。  

### 1. 库存扣减的实现细节  

**问题**：如果采用乐观锁+消息队列，如何保证在双11高峰期不出现库存超卖？  

**思考路径**：  
1. **先从业务角度说明需求**：库存必须强一致，超卖不可接受。  
2. **解释乐观锁的工作原理**：在 DB 中保存 `version`，更新时带上旧 version，若被其他事务修改则返回 0，重试。  
3. **引入消息队列的作用**：将库存扣减的**业务**（如生成订单）与**持久化**（写库）解耦，削峰。  
4. **完整流程**：  
   - **Step1**：下单服务先在 Redis 做预扣（原子 `DECR`），返回成功或失败。  
   - **Step2**：若成功，发送 `stock_deduct` 事件到 Kafka。  
   - **Step3**：Stock Consumer 消费该事件，使用 **乐观锁**（`UPDATE stock SET quantity = quantity - ? , version = version+1 WHERE goods_id=? AND version=?`）写入 MySQL。  
   - **Step4**：若更新受影响行数为 0（乐观锁冲突），说明有并发冲突，重新回滚 Redis（`INCR`）并发送补偿消息（或直接通知业务端 “库存不足”。）  
5. **幂等保证**：消费端在 Redis 中 `SETNX stock_deduct:{order_id}`，若已处理直接 ACK，防止重复扣减。  

**参考答案**（简洁版）：  

> “我们采用 **‘先预扣 → 异步持久化 + 乐观锁’** 的两段式方案。下单时在 Redis 中原子 `DECR`，如果返回负数则直接回滚，保证不超卖。成功后把扣减事件写入 Kafka，消费者在 MySQL 上使用 **乐观锁**（version）进行扣减。因为 Kafka 保证 **至少一次** 投递，消费者端通过 `SETNX` 幂等键防止重复扣减。即使在 150 万 QPS 的高峰，Redis 的毫秒级原子操作能处理并发，MySQL 通过乐观锁避免脏写，整体实现强一致且不超卖。”  

### 2. 幂等性设计  

**问题**：支付回调可能重复发送，如何在订单系统实现幂等？  

**思考路径**：  
- **核心点**：唯一标识（`payment_id`） + 幂等键 + 原子操作。  
- **实现方案**：  
  1. **接收回调**，先校验签名。  
  2. 使用 **Redis** `SETNX payment:{payment_id}`，TTL 2h。  
     - 若返回 1，说明是首次处理，继续后续业务。  
     - 若返回 0，说明已经处理过，直接返回成功，避免重复业务。  
  3. 在业务层，**在同一个数据库事务**中更新 `payment` 表、`order` 状态表。  
  4. 为防止 Redis 故障导致幂等失效，**在 MySQL** 同时加唯一索引 `UNIQUE(payment_id)`，在插入时若冲突直接捕获并返回。  

**参考答案**（要点突出）：  

> “我们把支付回调的 **`payment_id`** 视为幂等键。回调到达后，第一步在 Redis 用 `SETNX payment:{payment_id}`（TTL 2h）做原子检查；若成功，则在同一个事务里写入 `payment` 表并把对应 `order` 状态改为 `已支付`，最后发送 `order_paid` 消息。若 `SETNX` 返回 0，说明已经处理过，直接返回 200。即使 Redis 故障，MySQL 的唯一约束 `UNIQUE(payment_id)` 仍能防止重复写入，保证幂等性。”  

### 3. 跨地域容灾切换  

**问题**：当某个可用区因网络故障导致下单请求不可达，系统如何快速切流到备区并保证库存不冲突？  

**思考路径**：  
1. **流量层面**：使用 **全局 DNS + 健康检查**，故障检测到后把 DNS TTL 降到 30s，指向备区。  
2. **业务层面**：  
   - **全局唯一 ID**（Snowflake）在每个可用区都有独立的机器 ID，避免冲突。  
   - **库存预扣**采用 **Redis 集群的跨地域复制**（主从双向），并在每个 Region 都有 **本地缓存**，保证请求落在本地即可快速扣减。  
   - 为防止 **“双写冲突”**，在 **库存扣减** 前使用 **全局分布式锁**（基于 **Paxos/etcd**）或者 **基于业务唯一键的幂等**（`order_id+goods_id`）。  
3. **数据同步**：  
   - **MySQL 双向复制**（GTID），冲突概率极低（因为 ID 全局唯一）。  
   - **Kafka MirrorMaker** 跨 Region，同步 `stock_deduct`、`order_created` 事件。  

**参考答案**（结构化）：  

> “我们在流量入口使用 **AliDNS + 健康检查**，当某个可用区的 API GW 探测不到健康状态时，DNS 会在 30 s 内把流量切到另一个可用区，实现 ≤ 5 min 的 RTO。  
> 业务层面，每个 Region 都部署 **独立的 Redis Cluster**，并通过 **双向复制** 保持库存数据的一致性。下单时先在本地 Redis 预扣，如果扣减成功，立即写入本地 Kafka；消费者再把扣减持久化到本地 MySQL，同时通过 **Kafka MirrorMaker** 把同一条事件同步到备区。  
> 为防止同一库存在两个 Region 被并行扣减，我们在 **库存扣减** 前使用 **全局唯一幂等键**（`order_id+goods_id`）在 Redis 中 `SETNX`，若已存在直接返回成功，避免重复扣减。这样即使切流，库存仍然保持强一致且不出现超卖。”  

### 4. 为什么不把所有业务放在同一个事务里？  

**答案要点**：  
- **单体事务**会锁住 **库存、订单、支付** 三张表，导致 **锁竞争**，吞吐极低。  
- **支付回调**是 **异步** 的，若放在同事务会导致 **长事务**（数秒到分钟），影响数据库 GC、导致阻塞。  
- **拆分**成 **事务 + 消息** 的方式，**订单** 与 **库存** 通过 **可靠消息** 进行最终一致，**支付** 与 **订单状态** 通过 **幂等回调** 解耦。  

### 5. 如何评估系统的成本与收益？  

- **成本**：硬件（数十台 MySQL、Redis、Kafka 节点），运维（自动化部署、监控）。  
- **收益**：通过 **缓存+异步** 把峰值 QPS 从 150 万降到 **30 万**（核心 DB 直接写入），提升 **可用性**（故障切换 5 min），避免 **超卖** 带来的品牌损失。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 为什么最难 | 思考过程 |
|------|------------|----------|
| **库存强一致 vs. 高并发** | 需要在 **毫秒级** 响应的同时 **绝对不超卖**，两者天然冲突。 | ① 先列出常见方案（DB 行锁、乐观锁、Redis 预扣、分布式锁）。<br>② 计算峰值并发对 DB 行锁的等待时间 → 明显不可接受。<br>③ 决定采用 **Redis 预扣 + Kafka 异步持久化 + 乐观锁**，把热点放在内存，降低 DB 竞争。<br>④ 通过 **幂等键** 防止重复扣减，确保最终一致。 |
| **全局幂等设计** | 支付回调、下单请求、库存扣减都可能 **重复**，若幂等失效会导致 **重复扣钱/超卖**。 | ① 确定每类业务的唯一标识（`client_token`、`payment_id`、`order_id+goods_id`）。<br>② 评估存放幂等键的介质：**Redis**（高速、TTL）+ **DB 唯一约束**（持久化防失效）。<br>③ 设计 **幂等检查 → 业务处理 → 幂等键写入** 的完整流程，确保 **先检查后写**，防止 **TOCTOU**。 |

### 2. 新手最容易犯的错误（至少 2 条）  

| 错误 | 结果 | 正确做法 |
|------|------|----------|
| **把所有业务都放在单体事务里**（订单、库存、支付） | 长事务锁表、吞吐严重下降、容易出现死锁，导致 QPS 只能几千。 | **拆分业务**：下单 → 订单写入 + 库存预扣（异步），支付回调独立处理，使用 **可靠消息 + 幂等**。 |
| **忽视幂等性**，只在业务代码里判断一次 | 重复请求（网络重试、支付回调重复）会导致 **重复扣款、订单重复**，业务灾难。 | 为每类外部请求设计 **全局唯一幂等键**，使用 **Redis SETNX** + **DB 唯一约束** 双保险。 |
| **只靠 DB 锁实现库存防超卖** | 在双11 150 万 QPS 场景下锁竞争导致 **响应时间 > 2 s**，直接超出 SLA。 | 引入 **Redis 预扣**、**乐观锁** 或 **分布式锁**，把热点放到内存，降低 DB 锁竞争。 |
| **没有对外部依赖（支付、物流）做降级** | 第三方回调慢或不可达导致业务积压，最终导致超时或系统崩溃。 | 使用 **熔断器**、**回调重试+幂等**、**补偿机制**，确保单点故障不蔓延。 |

### 3. 学习建议和可延伸的方向  

| 方向 | 推荐学习资源 | 说明 |
|------|--------------|------|
| **分布式事务** | 《分布式系统：概念与设计》、阿里巴巴《CANAL & RocketMQ》 | 理解 **二阶段提交、可靠消息、TCC**，帮助处理更复杂的业务（如秒杀、预售）。 |

| **消息队列深度** | Kafka 官方文档、Confluent 实战课程 | 学习 **Exactly‑once、事务生产者、消费者幂等**，提升消息系统的可靠性。 |
| **CAP 与一致性模型** | 《Principles of Distributed Systems》、Google Spanner 论文 | 对 **强一致 vs. 最终一致** 的权衡有更系统的认识。 |
| **容灾与多活** | 《Site Reliability Engineering》、阿里云容灾最佳实践 | 理解 **RTO/RPO**、跨地域复制、全局调度的实现方式。 |

| **系统容量规划** | 《容量规划实战》、Netflix Simian Army | 学会 **负载模型、压测、容量预估**，为双11等大促做好准备。 |
