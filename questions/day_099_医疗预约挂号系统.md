# 第 99 天：设计 医疗预约挂号系统

> 生成日期：2026-02-16

---

## 1️⃣ 题目背景  
医疗预约挂号系统是面向患者、医院和医生的线上平台，提供挂号、预约、排队、费用结算等服务，帮助患者在手机/PC 上快速完成就诊预约，减轻线下排队压力。

## 2️⃣ 面试场景设定  
> **面试官**：  
> “假设我们要为全国 3000 多家医院设计一套高并发的医疗预约挂号系统，要求在高峰期（如上午 9:00‑10:00）仍能保持秒级响应。请从架构角度说明你会怎么设计这套系统，重点考虑可用性、扩展性以及数据一致性。”  

（面试官随后会要求你逐步展开功能划分、流量估算、关键组件设计等）

## 3️⃣ 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| **F1** | **患者注册/登录**：支持手机号、身份证、第三方（微信/支付宝）登录，完成身份验证。 |
| **F2** | **科室/医生查询**：按医院、科室、医生、时间段、排班等维度检索可预约的号源。 |
| **F3** | **挂号预约**：患者选择号源后完成预约，系统需生成唯一预约号并发送短信/推送确认。 |
| **F4** | **预约管理**：患者可查看、修改、取消预约（受医院取消规则限制），并支持排队/叫号提醒。 |
| **F5** | **支付结算**：支持线上支付（微信、支付宝）和线下支付，保证支付成功后才生成预约。 |
| **F6** | **运营后台**：医院管理员可以发布号源、设置排班、查看实时预约情况、导出报表。 |

（实际实现时可以按优先级聚焦 F1‑F4，F5‑F6 作为加分项）

## 4️⃣ 非功能性需求  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU** | 3,000,000（日活） | 包括患者、医院管理员、医生端。 |
| **QPS（峰值）** | 30,000 req/s | 早上 9:00‑10:00 预约高峰，主要集中在查询（70%）和下单（30%）。 |
| **平均响应时延** | ≤ 200 ms（95% 请求） | 包括查询、下单、取消等核心接口。 |
| **可用性** | ≥ 99.9%（月可用时长 ≥ 719 h） | 关键业务（预约、支付）必须高可用。 |
| **数据存储** | 约 500 TB（累计） | 预约记录、日志、审计、患者健康档案等，需支持冷热分离。 |
| **一致性要求** | **强一致**（预约/支付）+ **最终一致**（报表、日志） | 防止超额预约，保证金流准确。 |

## 5️⃣ 系统边界  

| 属于本题范围 | 不在本题范围 |
|--------------|-------------|
| • 患者端（Web / Mobile）<br>• 医院/医生后台管理<br>• 预约、取消、支付核心业务流<br>• 高可用、弹性伸缩、数据一致性方案<br>• 监控、报警、灰度发布机制 | • 电子病历完整系统（仅需挂号关联患者 ID）<br>• 医保/社保结算细节<br>• 实时视频问诊、远程会诊<br>• 医院内部 HIS（住院、药品、费用）深度集成<br>• AI 诊疗、健康推荐等高级功能 |

## 6️⃣ 提示与追问  

1. **高并发防超卖**  
   - “如果同一号源在 1 秒内被 10,000 人抢购，你会怎么保证不出现超额预约？”  

2. **数据一致性与事务**  
   - “支付成功后需要强一致的预约记录，你会选择哪种分布式事务方案？为什么？”  

3. **灾备与容错**  
   - “遇到单个地区数据中心宕机，系统如何快速切流并保证预约业务不中断？”  

（面试官可根据你的回答继续深入，如：缓存失效策略、限流算法、消息队列可靠投递、CAP 权衡等）

# 题解

## 解题思路总览
> **目标**：从「最小可用系统」一步步演进到「全局高可用、可水平扩展、强一致+最终一致」的医疗预约挂号平台。  
> **思路**：  
> 1. **先把业务拆成最小可运行的功能块**（注册、查询、预约），用单体或简单的微服务实现，先验证业务流程是否完整。  
> 2. **在此基础上评估流量、瓶颈、失效点**，决定哪些组件需要拆分、加缓存、加限流。  
> 3. **逐层引入分布式技术**（读写分离、分库分表、消息队列、分布式锁、CAP 权衡），每一步都要解释「为什么要这么做」以及「不这么做会有什么风险」。  
> 4. **最终形成完整的架构图**，并对每个关键点给出实现细节、备选方案、监控/灰度/容灾方案。  

下面我们按照 **需求 → 规模 → 架构 → 数据 → API → 组件 → 高可用** 的顺序展开，确保每一步都有“**做了什么** + **为什么这么做** + **不这么做会怎样**” 的完整解释，帮助新手把握系统设计的全局视角。

## 第一步：理解需求与规模估算

| 维度 | 业务描述 | 关键指标 | 设计影响 |
|------|----------|----------|----------|
| **用户** | 患者、医院管理员、医生（登录、查询、下单） | DAU≈3,000,000；峰值 QPS≈30,000 | 必须支撑海量并发、跨地域的读写请求 |
| **功能** | F1‑F4 为核心（注册/登录、查询、预约、管理）<br>F5‑F6 为加分（支付、后台） | 查询占 70%，下单/取消占 30% | 读请求远多于写请求，读写比例决定缓存、读写分离策略 |
| **时延** | 95% 请求 ≤ 200 ms | 前端交互体验、抢号成功率 | 需要低延迟网络、就近访问、缓存热数据 |
| **一致性** | 预约/支付强一致，报表/日志最终一致 | 防止超卖、保证金流准确 | 必须在关键路径使用强一致机制（分布式锁、事务） |
| **容量** | 累计约 500 TB（历史预约、日志、健康档案） | 冷/热分离、归档 | 采用冷热分层存储（MySQL + Hive/ES） |
| **可用性** | ≥ 99.9%（月可用≥719 h） | 关键业务（预约、支付）必须容错 | 多 AZ、自动容灾、灰度发布、监控报警 |

### 1️⃣ 业务流简化（最小可用系统）
```
患者 -> 登录 -> 查询号源 -> 选号 -> 下单（生成预约） -> 发送短信/推送
```
只要这条链路跑通，就已经满足了 F1‑F4 中的核心需求。后面再逐步加入支付、后台、报表等。

### 2️⃣ 流量估算（粗略）
- **DAU** 3,000,000 → **活跃用户**（上午 9‑10 点）≈ 10% = 300,000 人
- **峰值 QPS** 30,000  
  - 查询（70%）= 21,000 QPS  
  - 预约/取消（30%）= 9,000 QPS  
- **单用户请求数**：查询一次 → 1 次；预约一次 → 1 次查询 + 1 次下单  
- **峰值写入**（预约/取消）≈ 9,000 QPS → 每秒约 9,000 条预约记录（包括事务日志）

> **为什么要算这些？**  
> - 评估 **CPU、内存、网络、磁盘 I/O** 的上限，决定实例规格。  
> - 判断是否需要 **流量削峰**（如排队、令牌桶）以及 **限流**。  
> - 为 **容量规划**（数据库写入、缓存热度）提供依据。

## 第二步：高层架构设计

### 2.1 最小可用系统（单体+单库）
```
[客户端] → HTTP/HTTPS → 单体服务 (SpringBoot) → 单库 MySQL
```
- **优点**：实现成本最低、调试最快。  
- **缺点**：单点故障、难以水平扩展、写入热点（同一号源抢购）会导致锁竞争，无法满足 30,000 QPS。

### 2.2 第一步分层（前后端分离 + 读写分离）
```
[客户端] → API Gateway → 业务服务 (微服务) → 
   ├─ Redis（缓存） → MySQL 主库（写） → MySQL 从库（读）
   └─ MQ（Kafka） → 异步处理（短信、日志、报表）
```
**关键点解释**  

| 组件 | 作用 | 为什么要加 |
|------|------|------------|
| **API Gateway** | 统一入口、鉴权、限流、路由 | 防止直接暴露内部服务，统一流控、灰度发布 |
| **业务微服务** | 按业务划分（User、Schedule、Appointment、Payment） | 便于独立扩容、故障隔离 |
| **Redis 缓存** | 热门号源、医生信息、用户会话 | 读取热点数据（查询 70%）在 1~5 ms 内返回，降低 DB 读压 |
| **MySQL 主从** | 主库负责写（预约、取消、支付），从库负责读（查询） | 读写分离提升 QPS，避免写锁影响查询 |
| **Kafka** | 可靠的消息队列，保证短信、账务、报表异步且不丢失 | 支撑 **最终一致** 场景，解耦高峰写入与慢任务 |
| **限流/熔断** | Sentinel/RateLimiter | 防止突发流量压垮后端，保证整体可用性 |

> **不加这些层会怎样？**  
> - **无缓存**：查询全部落到 MySQL，从库可能因复制延迟出现热点，导致 QPS 达不到 30k。  
> - **无限流**：秒级抢号时 10,000 并发瞬间到达，单实例会瞬间挂掉。  
> - **无消息队列**：同步调用短信、账务会拖慢主业务路径，影响 200 ms SLA。

**说明**  

- **多 AZ（可用区）**：每个服务在至少 2 个 AZ 部署，使用 **LVS/NGINX**+**EIP** 实现跨 AZ 负载均衡。  
- **分库分表**：医院、科室、医生等维度做水平切分（例如 `hospital_id % 4` → 4 库），预约表再按日期或号源 ID 分表，避免单表 100+ 亿行导致查询慢。  
- **Redis Cluster**：提供高可用、自动分片，热点号源放在热点槽位，使用 **TTL** 防止脏数据。  
- **Kafka 多分区**：保证每个号源对应唯一分区（或使用 **Key**），保证同一号源的订单在同一个分区顺序消费，防止并发超卖。  
- **统一的 Config/Discovery**（Spring Cloud Config + Nacos/Eureka）实现服务注册、动态配置、灰度发布。  
- **监控/告警**：每个关键指标（QPS、RT、错误率、锁竞争、Kafka 延迟）都有告警阈值，配合自动伸缩（ASG）实现 **弹性伸缩**。  

> **不做多 AZ/容灾会怎样？**  
> - 单点故障（机房、网络、DB 主库宕机）会导致系统整体不可用，违背 99.9% SLA。  
> - 没有自动切流，恢复时间可能是**小时级**，对业务影响极大。

## 第三步：数据库设计

### 3.1 业务模型（核心表）

| 表名 | 说明 | 主键 | 关键索引 | 分库/分表方案 |
|------|------|------|----------|--------------|
| `user` | 患者/管理员/医生基本信息 | `user_id`（UUID） | `mobile`（唯一）<br>`id_card`（唯一） | 按 `user_id % 4` → 4 库 |
| `hospital` | 医院信息 | `hospital_id` | `city_id` | 1 库（数量有限） |
| `department` | 科室信息 | `dept_id` | `hospital_id` | 1 库 |
| `doctor` | 医生信息（排班关联） | `doctor_id` | `hospital_id`<br>`dept_id` | 1 库 |
| `schedule` | 医生排班、号源数量（每日/时段） | `schedule_id` | `doctor_id`<br>`date` | 按 `hospital_id % 4` → 4 库，**号源分表**：`schedule_202311`、`schedule_202312` 等 |
| `appointment` | 预约订单（核心强一致） | `appointment_id`（UUID） | `user_id`<br>`schedule_id`<br>`status` | 按 `schedule_id % 8` → 8 库、**日期分表**：`appoint_20231101`、`appoint_20231102` |
| `payment` | 支付流水（外部支付平台回调） | `payment_id` | `appointment_id` | 按 `appointment_id % 4` → 4 库 |
| `message_log` | 短信/推送日志（最终一致） | `msg_id` | `appointment_id` | 单库（写少） |

### 3.2 强一致性实现（预约/支付）

1. **预减库存 + 悲观锁**  
   - **步骤**：  
     1. 用户下单 → **先在 Redis** 使用 `INCRBY` 对 `schedule:{schedule_id}:available` 做 **原子减 1**（Lua 脚本保证原子性）。  
     2. 若返回值 < 0，则 **回滚**（`INCR`）并返回号源已满。  
     3. 成功后写入 **预约表**（MySQL）并发送 **Kafka** 消息。  
   - **为什么**：Redis 的原子操作在 10k QPS 级别仍能保持毫秒级延迟，避免 MySQL 行锁竞争导致写入慢。  
   - **不这样做的后果**：直接在 MySQL `UPDATE schedule SET available = available - 1 WHERE id=? AND available>0`，在高并发时会产生大量死锁/锁等待，吞吐难以达到 30k QPS。

2. **支付强一致**（两阶段提交）  
   - **方案**：使用 **TCC（Try‑Confirm‑Cancel）** 或 **可靠消息+幂等**。  
   - **实现**：  
     - **Try**：预约已生成（已扣库存），调用支付平台下单，返回 **交易号**，状态置为 `PAYING`。  
     - **Confirm**：支付成功回调 → 在同一个业务事务（MySQL）中更新 `appointment.status = PAID`、`payment.status = SUCCESS`。  
     - **Cancel**：支付失败/超时 → 通过 **补偿事务**（消息队列消费）回滚预约：恢复 Redis 库存、更新预约状态为 `CANCELLED`。  
   - **为什么选择 TCC**：比 2PC 更轻量，适合高并发，且可以保证 **强一致**（只有支付成功才真正完成预约）。  

### 3.3 最终一致场景（报表、日志）

- **Kafka → Flink/Spark Streaming → Hive/ES**  
- 把 **预约变更**（创建、取消、支付成功）写入 Kafka，离线计算每日/每医院报表，写入 **Hive 表**（冷热分离）或 **Elasticsearch** 供后台查询。  
- **幂等性**：消息体自带唯一 `msg_id`，消费端做去重。  

### 3.4 冷热分离

| 数据类型 | 热/冷 | 存储 | 读取频率 | 失效策略 |
|----------|-------|------|----------|----------|
| 实时号源、医生排班、用户会话 | 热 | Redis Cluster + MySQL 主库 | 频繁（查询 70%） | TTL 5 min（排班） |
| 预约明细（最近 30 天） | 热 | MySQL 主库（分表） | 常查询、改删 | 主库保留 6 个月 |
| 历史预约（>6 个月） | 冷 | Hive / OSS（归档） | 报表、审计 | 按月归档至 OSS，保留 5 年 |
| 日志、审计 | 冷 | ELK（Elasticsearch） | 低频检索 | 按天滚动索引，30 天保留热索引 |

## 第四步：核心 API 设计

> **原则**：RESTful + 统一错误码 + 幂等设计（尤其是下单、取消）。

### 4.1 鉴权统一

| 接口 | 方法 | URL | 鉴权 | 幂等性 |
|------|------|-----|------|--------|
| 登录 | POST | `/api/v1/auth/login` | - | ✅ |
| 刷新 token | POST | `/api/v1/auth/refresh` | Refresh token | ✅ |

**返回**：JWT（含 `user_id`、`role`），在 Header `Authorization: Bearer <token>` 中传递。

```http
      "schedule_id": "sch_100001",
      "doctor_id": "doc_9876",
      "time_slot": "09:00-09:30",
      "total": 30,
      "available": 12,
      "price": 20
    },
    ...
  ]
}
```

- **缓存**：查询结果先写入 Redis，Key 结构 `schedule:{hospital_id}:{dept_id}:{date}`，TTL 5 min。  
- **防止缓存穿透**：对不存在的号源返回空列表并写入 **布隆过滤器**，后续直接返回空。

### 4.3 创建预约（核心强一致）

```http
POST /api/v1/appointments
Body:
{
  "schedule_id": "sch_100001",
  "patient_id": "user_12345",
  "pay_method": "WECHAT"
}
```

**业务流程图（文字版）**

1. **接口层**：先做 **防刷**（限流+验证码） → 防止恶意抢号。  
2. **Redis 扣库存**（Lua 脚本） → 返回成功/失败。  
3. **生成预约号**（UUID） → 写入 **MySQL appointment**（状态 `PENDING`）。  
4. **发送 Kafka 消息** `appointment_created`（包含预约信息）。  
5. **调用支付中心**（下单） → 返回 `payment_id`。  
6. **支付回调** → 更新 `appointment.status = PAID`、`payment.status = SUCCESS`（同一事务）。  
7. **成功响应**：返回 `appointment_id`、`appointment_no`。

**幂等**：使用 **业务唯一键** `user_id + schedule_id`（在同一天同一科室只能预约一次）做唯一索引，重复请求返回已存在的预约 ID。

### 4.4 取消预约

```http
DELETE /api/v1/appointments/{appointment_id}
```

- **检查规则**：只能在预约时间前 30 分钟内取消（医院配置）。  
- **业务步骤**：  
  1. **状态校验** → 若已 `PAID`，先走 **退款流程**（调用支付中心）。  
  2. **恢复库存**：`INCR schedule:{schedule_id}:available`（Lua 原子）。  
  3. **更新预约状态** `CANCELLED`。  
  4. **发送 Kafka** `appointment_cancelled`（用于日志、报表）。

### 4.5 查询我的预约

```http
GET /api/v1/users/{user_id}/appointments?status=ALL
```

- **读缓存**：最近 7 天的预约放在 Redis `user:appointments:{user_id}`（Sorted Set），TTL 1 天。  
- **分页**：使用 `last_id`（基于时间倒序）做游标分页，避免 OFFSET 大查询。

### 4.6 错误码约定（示例）

| Code | 含义 | 说明 |
|------|------|------|
| 0 | SUCCESS | - |
| 1001 | 参数错误 | 参数校验不通过 |
| 2001 | 号源已满 | Redis 扣库存失败 |
| 3001 | 支付失败 | 调用支付中心返回错误 |
| 4001 | 鉴权失败 | token 不合法 |
| 5000 | 系统异常 | 未捕获异常 |

> **为什么统一错误码？**  
> - 前端可以统一处理（如弹框、重试）。  
> - 面试官关注你对 **可观测性** 的考虑，统一错误码是第一步。

## 第五步：详细组件设计

### 5.1 API Gateway & 鉴权

- **技术选型**：Kong / Nginx + Lua 插件 或 Spring Cloud Gateway。  
- **功能**：  
  - **统一入口**：路由到对应微服务。  
  - **限流**：全局 QPS 30k，**IP + UID** 双重限流（令牌桶）。  
  - **鉴权**：解析 JWT，注入 `user_id`、`role` 到 Header。  
  - **灰度发布**：基于 `user_id%100` 实现灰度流量切换。  

### 5.2 认证中心（Auth Service）

- **实现**：使用 **OAuth2.0** + **JWT**（RS256 非对称加密）。  
- **登录方式**：手机号+验证码、身份证+实名认证、微信/支付宝 OAuth。  
- **安全**：密码不在系统保存，使用 **SM2/SM3**（国密）或 **bcrypt**。  

### 5.3 缓存层（Redis Cluster）

| 结构 | Key 示例 | TTL | 备注 |
|------|----------|-----|------|
| **String** | `schedule:{schedule_id}:available` | 0（永久） | 号源库存，原子递增/递减 |
| **Hash** | `doctor:{doctor_id}:info` | 10 min | 医生基本信息 |
| **Sorted Set** | `user:appointments:{user_id}` | 1 day | 按预约时间倒序，用于快速查询 |
| **Bitmap** | `user:daily:login:{date}` | 30 day | 登录签到统计（运营侧） |
| **Bloom Filter** | `bf:schedule:{date}` | 7 day | 防止缓存穿透（不存在的 schedule_id） |

**防止热点键**：如果某个热门号源的 `available` 变更频繁，可使用 **分片计数**（如 `schedule:{id}:slot_0`, `slot_1`）再聚合，降低单键写入压力。

### 5.4 消息队列（Kafka）

| Topic | 生产者 | 消费者 | 关键属性 |
|-------|--------|--------|----------|
| `appointment_created` | Appointment Service | 短信服务、报表服务、支付补偿 | **Key = schedule_id**（确保同一号源顺序消费） |
| `appointment_cancelled` | Appointment Service | 短信服务、报表服务 | 同上 |
| `payment_result` | 支付中心回调 | Appointment Service（确认/回滚） | **幂等**（payment_id） |
| `audit_log` | 各服务（统一日志） | 日志系统（ELK） | 高可靠（副本 ≥ 3） |

**可靠投递**：生产者使用 **事务**（enable.idempotence=true）确保 **Exactly‑Once** 语义；消费者开启 **自动提交** 前手动 **offset**，防止重复消费。

### 5.5 业务微服务划分

| 服务 | 业务范围 | 主要依赖 | 扩容粒度 |
|------|----------|----------|----------|
| **User Service** | 注册、登录、用户信息管理 | Auth Service、MySQL `user` | 业务轻，CPU 2C/4G 即可 |
| **Schedule Service** | 医院/科室/医生排班、号源查询 | Redis、MySQL `schedule` | 读多写少，读侧可水平扩容 |
| **Appointment Service** | 预约创建、取消、查询、状态流转 | Redis、MySQL `appointment`、Kafka、Payment Service | 高并发热点，需多实例、热点调度 |
| **Payment Service** | 调用第三方支付、退款、对账 | 第三方 SDK、MySQL `payment`、Kafka | 与支付平台并发匹配，建议 2C/8G |
| **Notification Service** | 短信、推送、邮件 | 第三方短信/推送、Kafka | 业务轻，按流量弹性伸缩 |
| **Admin Backend Service** | 号源发布、报表查询 | MySQL、Hive、Elasticsearch | 读写均衡，可独立伸缩 |

### 5.6 防超卖（高并发抢号）实现细节

1. **令牌桶预热**：每个 `schedule_id` 对应 **N** 个令牌（等于号源 `total`），放入 Redis List `schedule:{id}:tokens`。  
2. **抢号流程**：  
   - `LPOP schedule:{id}:tokens` → 成功则进入后续预约流程。  
   - 若返回 `null` → 直接返回 “号源已满”。  
3. **幂等恢复**：若后续流程因支付失败需要回滚，**LPUSH** 令牌回库。  
4. **热点保护**：使用 **Redis 分片**（如 `schedule:{id}:tokens:0/1/2`）或 **Lua 脚本** 把 `LPOP` 与库存扣减合并，保证“一次请求只消耗一次令牌”。  

> **为什么不直接在 MySQL 行锁？**  
> - 行锁在 10k QPS 场景下会产生大量 **死锁/等待**，响应时间超出 200 ms。  
> - Redis 单机可处理 **百万 QPS** 的原子操作，且延迟在 **1‑2 ms**，更符合抢号秒级需求。

### 5.7 限流与防刷

| 场景 | 限流点 | 算法 | 参数 |
|------|--------|------|------|
| 全局入口 | API Gateway | 令牌桶（全局） | QPS ≤ 35k |
| 预约下单 | Appointment Service | **滑动窗口**（用户+IP） | 5 次/秒/用户 |
| 登录验证码 | Auth Service | **漏桶**（手机号） | 1 条/分钟 |
| 短信推送 | Notification Service | **固定窗口**（手机号） | 3 条/小时 |

**实现**：使用 **Redis** 的 `INCR` + `EXPIRE` 或 **Sentinel** 的 **RateLimiter**，配合 **熔断**（Hystrix/Resilience4j）在异常激增时快速降级。

### 5.8 监控、日志、灰度

- **监控**：Prometheus 抓取微服务 `metrics`（QPS、RT、错误率、CPU、内存、Redis 命中率、Kafka 延迟）。Grafana 看板分为 **业务层**、**系统层**、**异常告警**。  
- **日志**：统一使用 **ELK**，结构化 JSON 日志，关键业务链路（TraceId）通过 **MDC** 传递，方便 **分布式追踪**（Jaeger）。  
- **灰度**：在 Gateway 中配置 **Canary**，通过 `user_id%100 < 5` 将 5% 流量路由到新版本实例，观察 **错误率、RT** 再全量发布。

## 第六步：扩展性与高可用设计

### 6.1 水平扩容

| 维度 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **API Gateway** | 添加实例 + LVS/NGINX 负载均衡 | QPS 30k → 60k |
| **业务服务** | 按微服务粒度伸缩（K8s HPA） | CPU/Memory 使用率 70% 触发扩容 |
| **Redis** | **Cluster** 扩容节点（slot 重新分配） | 读写 QPS 100k+ |
| **MySQL** | 主从水平扩容 + 分库分表 | 写入 20k/s（单库） |
| **Kafka** | 增加分区数、Broker | 单分区吞吐 1M msg/秒 |
| **支付/短信** | 第三方供应商弹性伸缩 | 业务峰值不受限 |

### 6.2 容灾与故障切换

1. **跨地域多活**（如华北、华东、华南）  
   - **DNS 负载均衡**（阿里云 SLB + Anycast）把流量分到最近的 Region。  
   - **数据同步**：MySQL 使用 **Binlog + Canal** 同步到另一 Region 的只读库；Redis 使用 **CRDT** 或 **Active‑Active** 方案（如 **Redis Enterprise Geo-Replication**）。  
2. **单 AZ 故障恢复**  
   - **Pod/实例自愈**（K8s 自动调度）  
   - **MySQL 主库故障转移**（MHA 或 Orchestrator）  
   - **Kafka 副本自动切换**（ISR）  
   - **Redis Sentinel** 自动故障切换。  
3. **灾备演练**  
   - 每月一次 **全链路故障模拟**（关闭一个 AZ），验证 **自动切流**、**数据一致性**、**业务可用性**。  

### 6.3 数据一致性 CAP 权衡

| 场景 | CAP 需求 | 选型 |
|------|----------|------|
| 预约/支付 | **C + A**（强一致、可用） | 使用 **分布式锁 + TCC**（强一致），通过 **同步 RPC** 保证可用。 |
| 查询号源 | **C + P**（强一致 + 分区容错） | 读缓存 + 主从复制（最终一致），若缓存失效则回源查询。 |
| 报表 | **A + P**（可用 + 分区容错） | 离线计算（Hive）→ 最终一致。 |
| 短信/通知 | **A + P** | 异步 Kafka → 最终一致。 |

> **为什么区分？**  
> - 关键业务必须 **避免超卖**，所以选强一致（牺牲一点可用性在极端故障下仍能保证不超卖）。  
> - 大多数查询业务对 **可用性** 要求更高，容忍几秒钟的复制延迟即可。

### 6.4 灰度发布与回滚

1. **蓝绿部署**：在同一集群保留两套完整服务实例，切流前先做 **流量镜像**（复制真实请求但不返回给用户），观察错误率。  
2. **金丝雀**：使用 **Istio** 或 **Kong** 的 **Canary** 插件，将 1%→5%→10% 流量逐步切入新版本。  
3. **快速回滚**：所有服务使用 **无状态**，回滚只需要切回旧版 Service，数据库兼容采用 **向后兼容** 的 schema 演进（新增字段、非必填）。

## 第七步：常见面试追问与回答

### Q1️⃣ 高并发防超卖
> **“如果同一号源在 1 秒内被 10,000 人抢购，你会怎么保证不出现超额预约？”**

**回答要点**：

1. **预先把号源数量放入 Redis 作为令牌**（`schedule:{id}:tokens` List），每个号源对应 `total` 条 token。  
2. **抢号时先执行 `LPOP`（原子）获取 token**：  
   - 成功 → 进入后续预约事务（写 DB、生成预约号）。  
   - 失败（返回 nil） → 直接返回 “号源已满”。  
3. **Lua 脚本把 `LPOP` 与库存检查、幂等校验一次性完成**，避免网络往返导致的竞争窗口。  
4. **后置补偿**：若后续支付/业务失败，`LPUSH` 把 token 归还，确保库存最终一致。  
5. **热点键分片**：如果单个 Redis 实例压力过大，可将 token列表拆成若干子列表 `schedule:{id}:tokens:0~n`，使用 `hash` 决定放哪一个，进一步分摊并发。  
6. **为什么不直接在 MySQL 加锁？**  
   - 行锁在 10k QPS 场景会产生大量死锁、锁等待，导致响应时间 > 200 ms。  
   - Redis 原子操作在同等并发下 **毫秒级**，更符合抢号秒级需求。  

### Q2️⃣ 分布式事务与强一致
> **“支付成功后需要强一致的预约记录，你会选择哪种分布式事务方案？为什么？”**

**推荐方案**：**TCC（Try‑Confirm‑Cancel）** 或 **可靠消息+幂等事务**。  

- **TCC 步骤**  
  1. **Try**：预约已创建（扣库存），向支付中心发起预支付，返回 `payment_id`，状态设为 `PAYING`。  
  2. **Confirm**：支付成功回调 → 在同一个本地事务里更新 `appointment.status = PAID`、`payment.status = SUCCESS`。  
  3. **Cancel**：支付失败或超时 → 通过补偿事务回滚预约（恢复库存、标记 `CANCELLED`）。  

- **优点**  
  - **强一致**：只有 Confirm 完成才算正式预约，避免“支付成功但预约未生成”。  
  - **高可用**：每一步都是本地事务，避免两段式提交的阻塞和锁定。  
  - **幂等**：每个业务 ID（`appointment_id`、`payment_id`）都有唯一键，重复回调只会更新一次。  

- **与 2PC 对比**  
  - 2PC 在高并发下需要 **全局锁**，会导致资源长时间占用，响应时间难以满足 200 ms。  
  - TCC 通过 **业务层面拆分**，把长事务拆成短事务，性能更好。  

- **实现细节**  
  - **消息可靠投递**：使用 Kafka 确认投递，消费端幂等处理。  
  - **超时补偿**：定时任务扫描 `status=PAYING` 超过 15 min 的记录，自动走 Cancel 流程。  

### Q3️⃣ 灾备与容错
> **“遇到单个地区数据中心宕机，系统如何快速切流并保证预约业务不中断？”**

**答复要点**：

1. **跨 Region 多活**  
   - 每个 Region 部署完整的 **API Gateway + 微服务 + MySQL 主库 + Redis 集群**。  
   - 使用 **全局 DNS（Anycast）+ SLB** 将用户流量路由到最近的 Region。  

2. **数据同步**  
   - **MySQL Binlog + Canal** → 将写入的业务数据实时同步到备份 Region 的只读库。  
   - **Redis Active‑Active**（CRDT）或 **双写 + 异步同步**，保证热点号源的库存在两个 Region 基本一致。  

3. **故障检测与自动切换**  
   - 采用 **健康检查**（心跳）监控每个 Region 的 Gateway、服务实例、DB、Redis。  
   - 当检测到某 Region 健康度 < 阈值，DNS/SLB 自动剔除该 Region，剩余 Region 立即承接流量。  

4. **业务层容错**  
   - **预约请求**如果在本地 Region 的库存抢不到（因同步延迟导致库存不一致），会回退到 **全局锁**（如 Zookeeper 分布式锁）跨 Region 再次尝试，确保不出现超卖。  
   - **支付回调**采用 **幂等**、**全局唯一流水号**，即使回调在另一个 Region 也能正确关联到对应预约。  

5. **恢复流程**  
   - 故障 Region 修复后，使用 **双写**方式逐步把写入流量恢复，期间通过 **流量镜像**验证数据一致性。  

### Q4️⃣ 缓存失效与雪崩
> **“如果 Redis 宕机导致缓存失效，怎么防止 DB 瞬时压力激增？”**

**方案**：

1. **热点缓存预热**：启动时批量加载最近 1 天的号源数据到 Redis。  
2. **熔断+降级**：在 API 层检测到 Redis 访问异常（超时 > 50 ms），直接走 **读库**，但对查询 QPS 做 **限流**（如 5k QPS），防止 DB 瞬时爆炸。  
3. **缓存降级**：对热点数据（热门号源）使用 **本地缓存**（Guava）+ **Bloom Filter**，减少对 Redis 的依赖。  
4. **多级缓存**：在 Redis 前再加一层 **CDN/Edge Cache**（如阿里云 CDN），把公开的科室、医生信息缓存到边缘节点，降低中心 Redis 访问量。  

### Q5️⃣ 监控与告警
> **“你会监控哪些关键指标，怎么快速定位故障？”**

**关键指标**（按层级）：

| 层级 | 指标 | 阈值（示例） | 目的 |
|------|------|-------------|------|
| **网关** | QPS、4xx/5xx 比例、响应时间 (p95) | QPS > 35k → 报警 | 流量异常、限流失效 |
| **业务服务** | CPU/Memory、GC 时间、异常率、业务 RT(p95) | CPU > 80% | 资源瓶颈 |
| **Redis** | 命中率、慢查询数、复制延迟、内存使用率 | 命中率 < 80% | 缓存雪崩 |
| **MySQL** | TPS、慢查询、复制延迟、连接数、锁等待时间 | 复制延迟 > 5s | 数据不一致 |
| **Kafka** | 消费延迟、ISR 数、磁盘使用率 | 消费延迟 > 5s | 消息堆积导致补偿失效 |
| **支付** | 支付成功率、回调超时率 | 回调超时 > 1% | 金额风险 |
| **业务** | 预约成功率、超卖次数、库存回滚次数 | 超卖 > 0 | 关键业务错误 |

**定位流程**：

1. **告警触发** → 自动生成 **PagerDuty/钉钉** 通知。  
2. **查看仪表盘**：先定位是 **网络层**（网关异常）还是 **后端层**（服务、DB、缓存）。  
3. **链路追踪**：使用 **Jaeger** 查看慢请求的完整调用链，定位具体微服务的瓶颈点。  
4. **日志检索**：在 ELK 中搜索关键错误码（如 2001 号源已满、5000 系统异常），确认业务异常根因。  
5. **自动化脚本**：如 Redis 复制延迟超阈值，脚本自动执行 **主从切换** 并通知运维。

## 心得与反思

### 1️⃣ 本题最难的设计决策

| 决策 | 思考过程 |
|------|----------|
| **抢号防超卖方案** | 初始想法是直接在 MySQL 使用 `UPDATE ... WHERE available>0`，但在 10k QPS 并发下会出现锁竞争、死锁，导致响应时间远超 200 ms。于是转向 **Redis 原子操作 + 令牌桶**，再通过 **Lua 脚本** 把库存扣减、幂等校验一次性完成。实现时需要权衡 **热点键**（单 schedule_id）可能导致单节点压力过大，最终决定采用 **分片 token 列表** 或 **热点迁移**（把热点号源提前分配到专用 Redis 实例）。 |
| **支付与预约的强一致** | 传统的 **2PC** 方案在高并发支付场景下会导致全局锁，极易产生超时。通过调研，选定 **TCC**（业务层面的 Try‑Confirm‑Cancel）+ **可靠消息**，把“预扣库存”与“支付确认”解耦成短事务，既能保证 **强一致**，又保持 **高可用**。实现时要特别注意幂等性（业务 ID、重复回调）以及 **补偿任务**（超时自动 Cancel）。 |

### 2️⃣ 新手最容易犯的错误

| 错误 | 说明 | 如何避免 |
|------|------|----------|
| **把所有业务写在单体里** | 初学者往往直接把用户、预约、支付全部放进一个服务，导致 **代码耦合、部署难、扩容受限**。 | 先划分 **业务边界**（用户、排班、预约、支付），每块做独立的微服务，接口通过 **REST/gRPC** 通信。 |
| **只关注功能不考虑流量** | 只实现了 F1‑F4，忽视了 QPS、并发抢号的热点，导致系统在高峰期宕机。 | 在需求分析阶段就做 **流量预估**，并据此决定 **缓存、限流、分布式锁** 等防护措施。 |
| **盲目使用强一致** | 对所有读请求都使用分布式事务，导致系统性能急剧下降。 | 区分 **强一致**（预约、支付）与 **最终一致**（报表、日志），只在关键路径使用事务。 |
| **缓存失效不做降级** | 认为缓存 100% 命中，未考虑 Redis 故障导致 DB 瞬时压垮。 | 在 API 层实现 **缓存降级**（读取 DB + 限流），并做好 **熔断**、**预热**。 |
| **忽视监控与告警** | 没有监控关键指标，故障发现慢，恢复时间长。 | 设计时即加入 **Prometheus+Grafana**、**ELK**、**链路追踪**，并定义 **SLA/SLI**。 |

### 3️⃣ 学习建议和可延伸方向

| 学习阶段 | 推荐内容 |
|----------|----------|
| **基础** | 熟悉 **HTTP、REST、JWT、OAuth2**，了解 **MySQL 基础事务、索引、分库分表**。 |
| **中级** | 掌握 **Redis 高并发原子操作（Lua 脚本）**、**Kafka 消息可靠投递**、**Spring Cloud / Spring Boot 微服务**。 |
| **高级** | 深入 **分布式事务（TCC、SAGA）**、**CAP 理论**、**多活容灾（Active‑Active）**、**流量治理（限流、熔断、灰度发布）**。 |
| **实战** | 搭建 **线上演练环境**（K8s + Istio + Prometheus），实现 **预约抢号**、**支付补偿**、**灾备切换**。 |
| **延伸** | 了解 **分布式锁（RedLock、Zookeeper）**、**一致性哈希**、**热点分片**、**CRDT**、**业务链路追踪**（Jaeger）以及 **机器学习在排队预测** 的可能性。 |
