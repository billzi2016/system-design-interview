# 第 11 天：设计 微信（WeChat）

> 生成日期：2026-05-15

---

## 1. 题目背景  
微信（WeChat）是一款面向全球用户的即时通讯与社交平台，提供文字、语音、视频聊天、朋友圈、公众号、小程序等多种服务。面试者需要设计一个 **高并发、低时延且高度可用** 的核心聊天系统（包括单聊与群聊），满足数亿日活用户的实时通信需求。

---

## 2. 面试场景设定  

**面试官**：  
> “我们现在要从零开始设计微信的核心聊天系统，你可以先从整体架构思路说起吗？重点请考虑 **消息的可靠投递、顺序保证以及高并发场景下的扩展性**。在此基础上，我们会进一步探讨存储、推送以及容灾等细节。”

---

## 3. 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **单聊**：用户之间能够发送文字、图片、语音、表情、短视频等多媒体消息，要求消息在 1 秒内到达对方。 |
| 2 | **群聊**：支持 1 万 人以内的大型群组，消息需在群成员间保持 **强序一致性**（即所有成员看到的顺序相同）。 |
| 3 | **消息离线存储 & 拉取**：用户离线期间的消息需要持久化，用户上线后能够拉取最近 30 天的聊天记录（约 5 GB/用户）。 |
| 4 | **已读回执 & 消息撤回**：实现已读回执、消息撤回、编辑（限 2 分钟内）等交互功能。 |
| 5 | **多端同步**：同一账号在 PC、Web、Android、iOS 多端登录时，聊天状态（已读、已发送、未读计数）保持实时同步。 |
| 6 | **安全与审计**：对所有消息进行端到端加密（传输层 TLS），并在服务器侧对违规内容做实时检测（机器学习/规则），记录审计日志。 |

---

## 4. 非功能性需求  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 5 × 10⁸ | 目标覆盖全球主要市场的活跃用户数 |
| **QPS（请求峰值）** | 2 × 10⁶ 发送/接收请求/秒 | 峰值期间（如节假日）约 10 % 的 DAU 同时活跃 |
| **消息投递延迟** | ≤ 300 ms（95%）<br>≤ 1 s（99%） | 包括网络传输、服务器处理、推送链路 |
| **系统可用性** | 99.99% 年可用率（约 52 分钟年宕机） | 支持跨地域容灾、热备 |
| **存储容量** | 约 2 EB（Exabyte）<br>（5 GB/用户 × 5 × 10⁸ 用户 × 0.8 冗余系数） | 采用冷热分离、分块压缩、对象存储 |
| **水平扩展性** | 通过增加机器线性提升 QPS | 设计无状态服务、分片路由、自动扩容 |

---

## 5. 系统边界  

**本题范围（需要设计）**  
- 消息的 **实时投递**、**顺序保证** 与 **可靠存储**（包括离线存储、历史检索）。  
- **推送层**（APNs、FCM、WebSocket）与 **多端同步**。  
- **高可用、容灾、扩容** 机制（分区、复制、负载均衡）。  
- 基础 **安全加密** 与 **审计日志**。  

**不在本题范围（可不必详细设计）**  
- 微信的 **朋友圈、公众号、小程序** 等社交与业务功能。  
- **支付、红包、视频号** 等金融与内容分发业务。  
- **全局搜索、智能推荐** 等高级 AI 功能。  
- 具体的 **前端 UI/交互** 实现细节。  

---

## 6. 提示与追问  

1. **消息顺序与强一致性**  
   - “如果一个群聊有 10 k 成员，如何在保证所有成员看到相同顺序的前提下实现高并发写入？请说明你会选用哪种一致性模型、分片策略以及冲突解决方案。”

2. **离线存储与冷热分离**  
   - “考虑到每位用户需要保存最近 30 天的聊天记录，你会如何设计存储结构来兼顾读写性能、成本与容灾？冷热分离的阈值与迁移策略是什么？”

3. **容灾与跨地域同步**  
   - “在出现跨地域网络分区（如某个 IDC 故障）时，系统如何保证 **消息不丢失** 且 **最终一致**？请描述你的多活部署、复制协议以及故障恢复流程。”

# 题解

## 解题思路总览
> **目标**：从「最小可用」的原型系统出发，层层递进，最终得到一个 **高并发、低时延、强一致、可水平扩展、跨地域容灾** 的微信核心聊天系统设计。  
> **方法**：  
> 1. **先读需求**，把功能、非功能、约束全部列出来。  
> 2. **估算规模**，把用户数、QPS、存储量转化为具体的机器/网络指标。  
> 3. **画高层框图**，把流量入口、业务处理、持久化、推送、监控等大块拆开。  
> 4. **细化每块**：选择合适的技术/模型（如分区、复制、消息队列），并解释为什么不选其它方案。  
> 5. **从最小可用**（单机+单库）一步步演进到 **多活、分区、异步复制、冷热分离**。  
> 6. **补齐 API、数据结构、故障恢复流程**，并准备面试官常追问的答案。  

下面按照 **七个章节** 逐层展开，每一步都会说明 **“为什么这样做”** 与 **“如果不这么做会有什么问题”**，帮助新手形成系统化的思考方式。

## 第一步：理解需求与规模估算

| 类别 | 关键点 | 说明 |
|------|--------|------|
|**功能**|单聊、群聊、离线存储、已读回执、多端同步、审计|需要兼顾实时性和可靠性。|
|**非功能**|DAU 5×10⁸、峰值 QPS 2×10⁶、延迟 ≤300 ms (95%)、年可用 99.99%|直接决定系统容量、分区数、容灾方案。|
|**约束**|消息大小多样（文字≈1 KB、图片/视频≈几 MB），30 天历史 ≈5 GB/用户|影响存储压缩、冷热分离策略。|

### 1.1 业务量估算

| 项目 | 计算过程 | 结果 |
|------|----------|------|
|**每日活跃用户**|5×10⁸| |
|**峰值并发用户**|10% × DAU ≈ 5×10⁷| |
|**平均每用户每秒发送消息**|假设 0.4 条/秒（活跃用户每分钟 24 条）| |
|**峰值发送 QPS**|5×10⁷ × 0.4 ≈ 2×10⁷*（发送+接收）|实际需求给出 2×10⁶，保守取 2M，说明每条消息在系统内部会被多次处理（写入、推送、回执），所以整体吞吐约 2M。|
|**存储需求**|5 GB/用户 × 5×10⁸ 用户 ≈ 2.5 EB（未计冗余）|考虑 80% 冗余，约 2 EB。|
|**单条消息大小**|文字 1 KB，图片 100 KB，视频 1 MB，取平均 100 KB|用于估算网络带宽和磁盘 I/O。|
|**峰值网络流量**|2 M 条/s × 100 KB ≈ 200 GB/s ≈ 1.6 Tbps|实际业务中图片/视频会走 CDN，核心消息（文字+指针）流量远小于此。|

> **结论**：系统必须 **水平拆分**（分区/分片），**无状态** 前端入口，**异步写入** 与 **批量推送** 来削峰；存储层采用 **冷热分离 + 对象存储**，避免单机磁盘瓶颈。

### 1.2 关键性能指标拆解

| 指标 | 目标 | 对应技术点 |
|------|------|------------|
|**延迟 ≤300 ms**|95% 消息在 300 ms 内送达|WebSocket 长连接、热点分区、就近调度、异步落库|
|**强顺序一致性（群聊）**|所有成员看到相同顺序|全局递增序列号（Logical Clock）+ 单分区写入|
|**高可用 99.99%**|年宕机 ≤52 min|多活双活 IDC、自动故障转移、无单点|
|**线性扩容**|QPS 随机器数线性增长|无状态服务、统一路由、分片键均匀分布|

## 第二步：高层架构设计

### 2.1 最小可用（MVP）原型

```
[Client] --WebSocket--> [Load Balancer] --TCP--> [ChatServer]
                               |
                               +--> [MySQL] (消息持久化)
                               |
                               +--> [Redis] (用户在线状态、消息队列)
```

- **ChatServer**：单体服务，负责收发、落库、推送。  
- **MySQL**：事务保证消息不丢失，单库满足小流量原型。  
- **Redis**：缓存在线状态，做简单的 **publish/subscribe**。

> **缺点**：单点、无法支撑 2M QPS、群聊强序一致性受限、存储不够弹性。

### 2.2 演进到分布式高可用架构

```
                       +-------------------+
                       |  Global DNS / CDN |
                       +-------------------+
                                 |
               +-----------------+-----------------+
               |          全球负载均衡 (L4/L7)      |
               +-----------------+-----------------+
                                 |
      +----------+----------+----------+----------+----------+
      |   Region A (北京)   |   Region B (硅谷)   |  Region C (新加坡)|
      +----------+----------+----------+----------+----------+
                |                         |                         |
   +------------+------------+   +--------+--------+   +------------+------------+
   | 入口网关 (API GW) + LB   |   | 入口网关 + LB   |   | 入口网关 + LB   |
   +------------+------------+   +--------+--------+   +------------+------------+
                |                         |                         |
   +------------+------------+   +--------+--------+   +------------+------------+
   |   消息路由层 (Router)   |   |   消息路由层   |   |   消息路由层   |
   +------------+------------+   +--------+--------+   +------------+------------+
                |                         |                         |
   +------------+------------+   +--------+--------+   +------------+------------+
   | 业务处理层 (Chat Service) |   | 业务处理层   |   | 业务处理层   |
   +------------+------------+   +--------+--------+   +------------+------------+
                |                         |                         |
   +------------+------------+   +--------+--------+   +------------+------------+
   |   消息队列 (Kafka)      |   |   消息队列    |   |   消息队列    |
   +------------+------------+   +--------+--------+   +------------+------------+
                |                         |                         |
   +------------+------------+   +--------+--------+   +------------+------------+
   |  持久化层 (TSDB/ HBase)  |   |  持久化层   |   |  持久化层   |
   +------------+------------+   +--------+--------+   +------------+------------+
                |                         |                         |
   +------------+------------+   +--------+--------+   +------------+------------+
   |   推送服务 (APNs/FCM)    |   |   推送服务   |   |   推送服务   |
   +--------------------------+   +----------------+   +----------------------+
```

#### 关键组件解释  

| 组件 | 作用 | 为什么需要它 |
|------|------|---------------|
|**全局 DNS / CDN**|把用户流量路由到最近的 Region|降低 RTT，提升 300 ms 延迟目标|
|**入口网关 + LB**|统一入口、限流、鉴权、协议升级|防止直接攻击单点、实现灰度发布|
|**消息路由层**|根据 **ConversationID**（单聊/群聊）做分区路由|保证同一会话的所有消息落在同一个分区，实现强顺序|
|**Chat Service**|业务核心：收发、回执、撤回、编辑|保持无状态，便于水平扩容|
|**Kafka (或 Pulsar)**|消息的 **持久化写入日志** 与 **异步消费**|解耦写入与推送，削峰；提供 at‑least‑once + 顺序保证（分区内）|
|**TSDB/HBase / Cassandra**|聊天历史的冷热分离存储|写一次、读多次，支持大容量、低成本对象存储|
|**推送服务**|向 WebSocket、APNs、FCM、SMS 等渠道发送 |不同终端需要不同协议，统一抽象|
|**监控/告警/链路追踪**|Prometheus、Grafana、Zipkin|及时发现异常，满足 99.99% 可用|

> **核心设计**：**ConversationID**（单聊: `uid1_uid2`，群聊: `groupId`） 作为 **分区键**，所有属于同一会话的消息都进入同一 **Kafka 分区**，从而在该分区内部保证 **FIFO**。对 **10 k 成员的大群**，我们把 **groupId** 再 hash 到 **N=128** 个子分区（子分区内部顺序），再在客户端侧做 **全局序号**（Lamport Clock）合并，保证全体成员看到 **相同的全局顺序**（后文细化）。

## 第三步：数据库设计

### 3.1 业务数据模型（核心表）

| 表名 | 主键 | 关键字段 | 备注 |
|------|------|----------|------|
|`Message`|`msg_id (BIGINT auto-incr)`|`conversation_id`, `sender_id`, `msg_type`, `payload`, `timestamp`, `seq_in_conv`|存放 **原始消息体**（文字、URL、meta）|
|`MessageSeq`|`conversation_id`|`next_seq (BIGINT)`|用于生成 **递增序列号**（强顺序）|
|`UserStatus`|`user_id`|`online_node`, `last_heartbeat`|Redis 哈希，TTL 30 s|
|`ReadAck`|`conversation_id + user_id`|`last_read_seq`|已读指针|
|`RecallLog`|`msg_id`|`recall_by`, `recall_time`|撤回审计|
|`AuditLog`|`log_id`|`msg_id`, `action`, `detail`, `timestamp`|违规检测日志|

> **为什么用关系型 + KV**：  
> - **Message** 需要 **事务**（发送 + 状态同步）以及 **二级索引**（按时间查询），适合 **MySQL / TiDB**（水平分片）。  
> - **UserStatus/ReadAck** 为 **热点**，使用 **Redis** 提供毫秒级读写。  
> - **历史大数据**（30 GB/用户）不放在 MySQL，而是 **对象存储 + HBase**（列式）做冷热分离。

### 3.2 分区（Sharding）策略

| 数据 | 分区键 | 分区数 | 说明 |
|------|--------|--------|------|
|`Message`|`conversation_id`|`2000`（每个 Region 再拆分）|保证同会话落同库，同分区内顺序|
|`MessageSeq`|`conversation_id`|同上|与 Message 同步|
|`UserStatus`|`user_id`|`500`|均匀 hash，避免热点|
|`ReadAck`|`user_id`|同上|热点用户会分布在多个 Redis 实例|

> **不采用** `user_id` 直接分区的原因是 **群聊** 会导致 **写入热点**（同一群的 10 k 成员同时写入），而 `conversation_id` 能把写入均匀分布到多个分区。

### 3.3 冷热分离存储

1. **热数据（最近 7 天）**  
   - 存放在 **TSDB/HBase**（列式压缩），读写延迟 < 10 ms。  
   - 数据结构：`rowkey = conversation_id|day|msg_id`，列族存放 `payload`、`metadata`。  
2. **冷数据（7‑30 天）**  
   - **定时任务**（每天 02:00）把 7 天前的数据 **压缩**（Parquet）后写入 **对象存储**（OSS / S3）。  
   - 采用 **分层索引**：元数据（msg_id、offset）仍保留在 HBase，真正内容在对象存储。  
3. **归档（30 天后）**  
   - 迁入 **低频存储**（Glacier），仅在用户主动拉取时解档。  

> **阈值选择**：  
> - 7 天是 **业务访问热点**（用户经常查询最近对话），也是 **缓存命中率** 的拐点。  
> - 30 天对应 **业务需求**（历史记录保存时长），超过 30 天可自行删除或转为长期归档。

## 第四步：核心 API 设计

> 为了让面试官看到你能把业务拆成 **接口**，这里列出 **主要的 REST + WebSocket** 接口（请求/响应示例、幂等性说明）。

| 接口 | 方法 | 路径 | 说明 | 关键参数 | 幂等性 |
|------|------|------|------|----------|--------|
|**登录**|POST|/api/v1/auth/login|返回 JWT + 长连接 token|`username,password`|是|
|**获取离线消息**|GET|/api/v1/conversations/{cid}/messages|拉取自 `last_seq` 起的消息|`cid`, `last_seq`, `limit`|是|
|**发送消息**|POST|/ws（WebSocket）|实时发送，返回 `msg_id`|`conversation_id`, `msg_type`, `payload`|**否**（但通过 **client‑generated UUID** 实现幂等）|
|**已读回执**|POST|/api/v1/conversations/{cid}/read|上报已读的最大 seq|`cid`, `last_read_seq`|是|
|**撤回消息**|POST|/api/v1/messages/{msg_id}/recall|仅 2 min 内可撤回|`msg_id`, `reason`|是|
|**编辑消息**|PUT|/api/v1/messages/{msg_id}|仅 2 min 内可编辑|`msg_id`, `new_payload`|是|
|**获取会话列表**|GET|/api/v1/users/{uid}/conversations|返回最近 N 条会话元信息|`uid`, `page`, `size`|是|
|**推送 Token 注册**|POST|/api/v1/push/register|绑定 APNs/FCM token|`device_id`, `platform`, `push_token`|是|

### 4.1 消息发送流程（伪代码）

```python
# client -> websocket
msg = {
    "cid": "group_12345",
    "type": "text",
    "payload": "hello world",
    "client_msg_id": "uuid-xxxx"   # 幂等标识
}
ws.send(json.dumps(msg))

# Server side (Chat Service)
def on_message(ws, raw):
    msg = json.loads(raw)
    # 1. 幂等校验
    if dup = IdempotencyCache.check(msg["client_msg_id"]):
        ws.send(dup.response)      # 重复请求直接返回上一次的结果
        return

    # 2. 生成全局顺序号（conversation 分区的 seq）
    seq = MessageSeqService.next_seq(msg["cid"])

    # 3. 组装持久化记录
    record = Message(
        cid=msg["cid"], sender=ws.user_id,
        type=msg["type"], payload=msg["payload"],
        timestamp=now(), seq_in_conv=seq,
        client_msg_id=msg["client_msg_id"]
    )

    # 4. 写入 Kafka（持久化日志）
    kafka_producer.send(topic="msg_log", key=msg["cid"], value=record)

    # 5. 返回 ack 给客户端
    ack = {"msg_id": record.id, "seq": seq}
    ws.send(json.dumps(ack))
```

> **为什么走 Kafka**：  
> - **高吞吐**（单分区可达 100k+ msg/s），并提供 **分区内部顺序**。  
> - **持久化** 防止因业务服务崩溃导致消息丢失。  
> - **异步消费**（推送服务）实现 **写入快、推送慢** 的解耦。

### 4.2 消息消费（推送）流程

```
Consumer (Push Service) --> 读取 Kafka 分区
   |
   +-- 判断接收方是否在线 (Redis UserStatus)
   |        |
   |        +-- 在线 -> 通过对应节点的 WebSocket 推送
   |        +-- 离线 -> 写入离线消息表（Message）并触发 APNs/FCM
   |
   +-- 写入历史库（TSDB/HBase）  <-- 异步落库
```

> **幂等**：Kafka 消费使用 **offset** 作为幂等标识，消费失败时可回滚重试；推送成功后在 Redis 中标记 `msg_id:delivered`。

## 第五步：详细组件设计

### 5.1 入口层（API Gateway + LB）

| 需求 | 方案 | 细节 |
|------|------|------|
|**协议统一**|NGINX/Envoy|终端使用 **WebSocket**（升级 HTTP）和 **HTTPS**（REST）|
|**限流/防刷**|Rate‑Limiter（基于 IP + UID）|令牌桶算法，防止突发流量压垮后端|
|**鉴权**|JWT + OAuth2|每个请求携带签名，网关解析后写入 `X-User-Id` Header|
|**灰度发布**|Canary / Blue‑Green|基于 Header 或 Cookie 控制流量分配|

> **不使用**传统 **L4** 直接转发的原因是：需要 **统一鉴权、限流、监控**，否则每个业务节点都要实现这些功能，导致代码重复、运维成本上升。

### 5.2 消息路由层（Conversation Router）

- **核心功能**：把 **conversation_id** 映射到 **Kafka 分区 + DB 分片**。  
- **实现**：采用 **一致性哈希**（Ring）+ **虚拟节点**（100 per physical node），保证 **负载均衡** 且 **节点上下线时最小迁移**。  
- **容灾**：每个虚拟节点对应 **双写**（主/备），备节点故障时路由自动切换。

### 5.3 核心业务层（Chat Service）

- **无状态**：所有状态放在 **Redis / DB**，服务实例可以随意上下线。  
- **线程模型**：采用 **Netty**（Java）或 **aiohttp**（Python）实现 **NIO**，单机可支撑上万并发连接。  
- **请求流水线**：  
  1. **解析** → 2. **幂等检查** → 3. **Seq 生成** → 4. **写入 Kafka** → 5. **返回 ACK** → 6. **记录监控**。

### 5.4 消息日志（Kafka 集群）

| 参数 | 取值 | 说明 |
|------|------|------|
|**分区数**|`2000`（每 Region）|足以支撑 2M QPS（每分区约 1k QPS）|
|**副本数**|`3`|保证单机故障不丢数据|
|**压缩**|`lz4`|降低网络带宽|
|**Producer**|**事务**模式（Exactly‑once）|避免重复写入|
|**Consumer**|Push Service（Group ID = region）|每个分区只被同一消费组的一个实例消费，保持顺序|

### 5.5 持久化层（冷热存储）

1. **热库**：**TiDB**（分布式 MySQL）或 **Cassandra**（列族）  
   - 主键：`(conversation_id, seq)`  
   - **二级索引**：`timestamp` 用于分页查询。  
2. **冷库**：**对象存储** + **Parquet** 文件  
   - 定时 **Spark**/Flink 作业将 7 天前的数据压缩归档。  
   - **元数据表**（HBase）保存 `msg_id → (bucket, offset)`，快速定位。  

### 5.6 推送层

| 场景 | 渠道 | 实现方式 |
|------|------|----------|
|**在线 Web/PC**|WebSocket|长连接，服务器主动 `push` |
|**移动端 iOS**|APNs|Push Service 调用 Apple HTTP/2 API |
|**移动端 Android**|FCM|同上，使用 Google FCM |
|**离线/未登录**|短信/邮件|备选方案，低优先级 |

- **推送失败重试**：指数退避 + 死信队列（DLQ），超过阈值后写入 **审计日志**。  
- **多端同步**：每次 **已读/撤回** 事件写入 **Kafka**，所有在线端的 Chat Service 消费后更新本地缓存并向客户端推送。

### 5.7 安全与审计

- **传输加密**：全部使用 **TLS 1.3**，WebSocket `wss://`。  
- **消息体加密**（端到端可选）：客户端使用 **AES‑GCM** 对称密钥，密钥通过 **Diffie‑Hellman** 握手后在服务器端不保存，仅用于**内容审计**的密文解密（合规要求）。  
- **内容审计**：消费 Kafka 消息的 **审计服务**，使用 **ML 模型 + 规则** 检测违规，违规消息写入 **AuditLog**，并触发 **人工复审**。  
- **日志审计**：所有关键操作（登录、发送、撤回、读回执）写入 **ELK**（Elasticsearch + Logstash + Kibana）系统，保留 90 天，供合规查询。

## 第六步：扩展性与高可用设计

### 6.1 水平扩容

| 维度 | 扩容方式 |
|------|----------|
|**入口层**|增加 LB 与 API GW 实例，DNS 轮询 |
|**Kafka**|扩增 **分区**（重新分配），每个新机器承担一部分分区 |
|**Chat Service**|添加更多无状态容器（K8s Pod），自动注册到 Service Registry |
|**Redis**|使用 **Cluster**，水平扩展热点键（UserStatus） |
|**TSDB/HBase**|扩容节点，RegionServer 自动均衡 |

> **关键点**：**conversation_id** 通过 **一致性哈希** 保证写入均匀，**新增节点** 只会导致 **~1/N** 的键迁移，不会出现全局热点。

### 6.2 高可用 & 容灾

#### 6.2.1 多活双活部署（跨 Region）

```
Region A (北京)  <--同步--> Region B (硅谷)
   |                     |
   +--- 同步写入 Kafka (MirrorMaker) ----+
   |                                     |
   +--- 同步写入 TSDB (Binlog Replication) 
```

- **数据复制**：  
  - **Kafka MirrorMaker**：实时把每个分区的日志同步到另一 Region（异步 200 ms SLA）。  
  - **TiDB / MySQL Binlog**：使用 **PXC** 或 **TiDB** 的双活复制。  
- **读写路由**：  
  - **用户请求** 由最近的 Region 处理。  
  - **写入** 同时写入本地和远端（双写），若远端不可达则 **本地先成功**，后续 **补偿**。  
- **故障转移**：  
  1. 检测到 Region A 完全不可用（心跳、监控告警）。  
  2. DNS 将该 Region 的流量切换到 Region B（TTL 低）。  
  3. 客户端自动重连到最近的入口。  

#### 6.2.2 单点故障消除

| 单点 | 替代方案 |
|------|----------|
|**API Gateway**|部署 **多实例** + **全局 L4**（Anycast）|
|**Chat Service**|无状态容器，K8s 自动伸缩|
|**Redis**|Cluster + 主从复制|
|**Kafka Broker**|每个分区 3 副本，跨机房|
|**TSDB**|多副本 Region，使用 Raft 共识|

#### 6.2.3 数据不丢失保证

- **Kafka**：`acks=all` + `min.insync.replicas=2` → 只要两副本落盘即认为成功。  
- **MySQL/TiDB**：使用 **InnoDB** 双写事务，**binlog** 持久化。  
- **幂等**：客户端 **UUID** + **服务端 Idempotency Cache** 防止网络重传导致的重复写入。  

#### 6.2.4 灾难恢复演练（DR）

1. **模拟网络分区**（Region A 与外网断连）。  
2. **监控触发**：Region A 进入 **只读** 模式（写请求返回 “暂时不可用”，前端提示用户稍后重试）。  
3. **异步补偿**：Region A 缓存写入的日志，等网络恢复后使用 **Kafka MirrorMaker** 回放到 Region B。  
4. **数据对账**：利用 **checksum**（MD5）比对两 Region 的 **MessageSeq** 表，确保最终一致。

### 6.3 性能优化技巧

| 场景 | 优化点 | 说明 |
|------|--------|------|
|**热点群聊**|**分片子分区** + **局部顺序**|将大群 `groupId` 再 hash 为 `sub_id`（128），每个子分区内部保持顺序，客户端在收到各子分区的消息后按 **全局 Lamport Clock** 合并。|
|**大文件**|**对象存储 CDN**|客户端上传图片/视频先到 OSS，消息体只保存 **URL + MD5**，降低 Kafka/DB 写入压力。|
|**已读回执**|**批量上报**|客户端每 200 ms 上报一次 `last_read_seq`，服务端聚合写入 DB，减轻写入频率。|
|**离线推送**|**批量推送**|对同一用户的多条离线消息合并为一条 **摘要**（如 “您有 12 条未读消息”），减少 APNs/FCM 调用次数。|

## 第七步：常见面试追问与回答

### 7.1 **“如果一个群聊有 10 k 成员，如何在保证所有成员看到相同顺序的前提下实现高并发写入？”**

1. **顺序模型**：采用 **强序一致性**（**全局递增序号**）  
   - 每条消息在 **MessageSeq** 表里获得 **conversation_seq**（单调递增）。  
2. **写入路径**：  
   - **单写入口** → **Kafka 同一分区**（`groupId` hashed → `sub_partition`） → **Consumer** 按顺序消费。  
3. **冲突解决**：  
   - **同一分区内部**天然 FIFO，冲突不存在。  
   - **跨子分区**（为避免热点）使用 **Lamport Clock**：发送端在每条消息里附带 **local_timestamp**，消费者在聚合时取 **max(lamport, local_timestamp)** 作为全局序号。  
4. **高并发写入**：  
   - **分散子分区**（如 128）让 10 k 成员的写请求分布到多个 Kafka 分区，每个分区仍保持顺序。  
   - **生产者端**使用 **批量发送**（batch.size、linger.ms）提升吞吐。  

> **如果不做子分区**：所有 10 k 成员的写请求都会落到同一个 Kafka 分区，导致 **单分区写入上限 ~100k msg/s**，在峰值时容易形成瓶颈。  

### 7.2 **“离线存储与冷热分离怎么做？阈值与迁移策略是什么？”**

- **冷热阈值**：  
  - **7 天**：热数据（TSDB/HBase）保持 **高读写性能**。  
  - **7‑30 天**：转为 **冷数据**（Parquet + OSS），每天一次 **Spark**/Flink 批处理。  
  - **30 天后**：归档到 **低频存储**（Glacier/OSS Infrequent Access）。  
- **迁移流程**：  
  1. **抽取**：从 HBase 按 `rowkey = conversation_id|day|msg_id` 批量读取。  
  2. **压缩**：使用 **Snappy/Parquet** 写入对象存储。  
  3. **写入索引**：在 HBase 中写入 **元数据表** `cold_msg_index`，指向 OSS 对象位置。  
  4. **删除原始**：确认写入成功后，安全删除热库对应分区（TTL + 手工校验）。  
- **为什么这样做**：  
  - 热库读写延迟 < 10 ms，满足 **实时聊天**；  
  - 冷库使用 **对象存储**，成本低至 **0.01 USD/GB/月**，满足 **EB 级** 存储需求；  
  - 分层迁移避免 **一次性大规模 IO** 对线上业务造成冲击。  

### 7.3 **“跨地域网络分区时，系统如何保证消息不丢失且最终一致？”**

1. **写入双写**：在 **Chat Service** 写入本地 Kafka + 同步写入远端 Region 的 **MirrorMaker**（异步但 **ack** 至少 2 副本）。  
2. **本地成功**：只要本地 Kafka 复制成功，即认为业务成功返回给客户端。  
3. **网络分区**：远端 MirrorMaker 失联，本地继续接受写入，产生 **未同步日志**。  
4. **恢复后**：启动 **补偿任务**，读取本地 Kafka **未同步 offset**，重新推送到远端。  
5. **冲突处理**：因为每条消息都有全局唯一 **msg_id**（UUID）和 **seq**，补偿时若远端已有相同 msg_id，则直接 **忽略**（幂等）。  
6. **最终一致**：所有 Region 的 **MessageSeq** 表通过 **双向复制** 保持一致，查询任意 Region 都能得到相同的历史记录。  

> **如果只在单 Region 写入**：网络分区后远端 Region 将丢失这段时间的所有消息，导致 **数据不一致**，无法满足业务容灾要求。  

### 7.4 **“为什么不直接把消息存入 MySQL 而不是走 Kafka？”**

- **写入延迟**：MySQL 单行事务在高并发下（2M QPS）会出现 **锁竞争**，延迟超过 300 ms。  
- **水平扩展**：MySQL 水平分片需要 **全局事务**，实现复杂。  
- **解耦需求**：发送消息后需要 **推送**、**审计**、**离线存储** 多个下游；Kafka 天然 **发布/订阅**，每个下游独立消费，互不影响。  
- **顺序保证**：Kafka **分区内部 FIFO**，比 MySQL 的 InnoDB 更易保证 **写入顺序**（只要同一分区）。  

### 7.5 **“如何防止消息重复投递？”**

- **生产者端**：**client_msg_id**（UUID）+ **幂等缓存**（Redis）  
- **Kafka**：`enable.idempotence=true`，Producer 自动对同一 `producerId+sequence` 重复发送进行去重。  
- **消费者端**：消费时根据 **msg_id** 检查 **Redis** `delivered_set`，若已存在直接丢弃。

## 心得与反思

### 1. 本题最难的 1‑2 个设计决策

| 决策 | 难点 | 思考过程 |
|------|------|----------|
|**强顺序一致性 + 大群写入**|群成员数 10 k，单分区写入瓶颈；又要保证全体成员看到同一顺序。|① 先确认必须在 **同一会话** 内使用 **单一全局序号**。<br>② 评估单分区吞吐上限（≈100k msg/s）不足以支撑峰值。<br>③ 引入 **子分区**（hash → 128）把写请求分摊，保持 **每子分区内部顺序**，再通过 **Lamport Clock** 在客户端合并全局序。|
|**跨地域容灾 + 消息不丢失**|网络分区时既要保证 **低时延**（不能同步阻塞），又要保证 **最终一致** 与 **不丢失**。|① 采用 **双写 + 异步复制**（Kafka MirrorMaker、Binlog）。<br>② 设计 **幂等写入**（UUID、msg_id）确保补偿不产生重复。<br>③ 明确恢复流程：监控 → 本地缓存 → 补偿任务 → 对账。|

### 2. 新手最容易犯的错误（≥2 个）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
|**把所有业务放在单体数据库**|单点瓶颈、难以水平扩容、写入冲突导致延迟 > 300 ms|使用 **分区 + Kafka** 进行写入解耦，热点分散到多个机器。|
|**只关注单聊，忽视群聊的顺序问题**|大群消息乱序，用户体验极差，业务规则冲突|在架构中加入 **Conversation Router**，把 **groupId** 再 hash 成子分区，保证 **分区内部顺序**。|
|**把离线消息直接存入 MySQL**|磁盘 IO、索引膨胀，30 GB/用户的历史记录导致 DB 爆炸|采用 **冷热分离**：热库（TSDB/HBase）+ 冷库（对象存储），并使用 **批量归档**。|
|**没有幂等设计**|网络抖动导致消息重复投递，用户收到多条相同消息|在 **客户端** 生成 **UUID**，服务端使用 **Idempotency Cache**，并开启 **Kafka Producer 幂等**。|

### 3. 学习建议与可延伸方向

1. **系统设计基础**  
   - 熟悉 **CAP 定理、BASE、PACELC**，能快速判断是选 **强一致** 还是 **最终一致**。  
   - 多阅读 **大厂公开架构**（如微博、知乎、抖音）对比不同业务的取舍。

2. **关键技术深入**  
   - **Kafka**：了解 **ISR、复制因子、事务**，以及 **Exactly‑once** 的实现原理。  
   - **分布式时钟**：Lamport Clock、Hybrid Logical Clock（HLC），用于全局顺序的场景。  
   - **冷热分层**：掌握 **HBase**/**Cassandra** 的列式存储模型，了解 **Parquet/ORC** 的压缩与查询特性。

3. **实战练习**  
   - 用 **Docker Compose** 搭建 **Kafka + Redis + MySQL**，实现简易的即时聊天原型。  
   - 用 **JMeter/Locust** 做压测，观察 **吞吐、延迟、错误率**，练习 **瓶颈定位** 与 **扩容**。  

4. **面试技巧**  
   - **结构化回答**：需求 → 规模 → 高层图 → 关键点 → 细节 → 备选方案。  
   - **提前准备**：画图工具（excalidraw、draw.io）可快速展示分区、流向。  
   - **主动提问**：如 “我们更看重消息投递的实时性还是存储的一致性？” 显示对业务的思考深度。
