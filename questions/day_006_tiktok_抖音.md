# 第 6 天：设计 TikTok（抖音）

> 生成日期：2026-05-20

---

## 题目背景
TikTok（抖音）是一款以 **短视频浏览、创作与分享** 为核心的社交媒体平台。用户通过滑动观看个性化推荐的短视频，并可以上传、互动、关注感兴趣的创作者。

## 面试场景设定
> **面试官**：*“我们现在要设计一个能够支撑全球数亿用户的 TikTok 视频流系统。请从高层次出发，先说明你会怎么划分系统的主要模块，并重点阐述视频推荐流（Feed）以及视频存储/分发的设计思路。”*

## 功能性需求
| 编号 | 功能描述 |
|------|----------|
| 1 | **视频上传**：用户可以上传 15 秒至 3 分钟的短视频，支持多种分辨率（720p、1080p、4K）和常见编码格式。 |
| 2 | **个性化视频 Feed**：用户打开 APP 后看到基于兴趣的连续短视频流，支持无缝滑动切换。 |
| 3 | **互动操作**：点赞、评论、分享、关注创作者，实时更新计数。 |
| 4 | **用户个人页**：展示用户的个人信息、已发布视频、粉丝/关注列表。 |
| 5 | **视频搜索 & 话题页**：通过关键词或话题标签检索视频。 |
| 6 | **内容审核**：对上传视频进行机器+人工审核，违规内容不可出现在 Feed 中。 |

## 非功能性需求（关键指标 & 估算）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 1.2 B（全球） | 假设 30% 的全球互联网用户活跃 |
| **QPS（Feed 请求）** | 150 k requests/s | 每位活跃用户平均 5 次 Feed 拉取/分钟 |
| **单次 Feed 延迟** | < 100 ms（99% 请求） | 包括推荐计算、缓存命中、CDN 传输 |
| **系统可用性** | 99.9%（月度） | 容错、灾备、灰度发布 |
| **视频存储规模** | 500 PB | 按 30 天热点 + 冷数据归档 3 年计算 |
| **吞吐量（上传）** | 30 GB/s 峰值 | 假设峰值时每秒 30 万条 100 MB 视频上传 |

## 系统边界
**本题需要设计的范围**
- 视频上传、转码、存储、分发（CDN）；
- 推荐系统（离线模型 + 实时召回）及 Feed 接口；
- 基础用户信息、关注关系、互动计数的 CRUD；
- 内容审核的接口与流程（不需要实现机器学习模型本身）。

**不在本题范围**
- 广告投放系统、营销活动；
- 完整的社交体系（私信、群组、直播）；
- 大数据分析平台、BI 报表；
- 细粒度的运营后台与监控报警系统（仅需讨论监控要点）；

## 提示与追问
1. **推荐系统的实现细节**  
   - *“如果实时召回需要在 10 ms 内返回 100 条候选视频，你会怎么设计缓存层、倒排索引以及热点热点分层？”*

2. **视频存储与分发的成本控制**  
   - *“面对 500 PB 的海量视频，你如何在保证高可用的前提下，利用冷热分层、对象存储与 CDN 做成本优化？”*

3. **高并发下的写放大与一致性**  
   - *“用户点赞/评论的计数需要强一致还是最终一致？请说明你的数据模型（如 Redis+MySQL）以及如何防止热点写入导致的性能瓶颈。”*

# 题解

## 解题思路总览  

本篇答案把 **从“我只会写一个最小可用的 Demo”**，一步步升级成 **“支撑全球 10⁸‑10⁹ 活跃用户的高可用分布式系统”** 的全过程全部写出来，**每一个设计决策都会解释“为什么要这么做、如果不这么做会有什么风险”。  
阅读顺序建议：  
1️⃣ 先读 **需求与规模估算**，弄清业务边界和性能目标。  
2️⃣ 再看 **高层架构**，把系统拆成若干职责明确的子系统。  
3️⃣ 接着深入 **数据模型、API、关键组件** 的实现细节。  
4️⃣ 最后学习 **扩展性 / 高可用** 的技巧以及面试官常追问的点。  

> **温馨提示**：本答案把每一步都写成 “先做 X → 再做 Y”，帮助你在面试中展现 **渐进式思考**，而不是一次性直接抛出“超大规模”方案导致思路混乱。

## 第一步：理解需求与规模估算  

| 类别 | 关键点 | 说明 |
|------|--------|------|
| **功能性** | 视频上传、转码、存储、分发；Feed 推荐、点赞/评论/分享；用户信息、关注关系；内容审核 | 只需要实现核心流程，其他（广告、直播等）可以暂时忽略 |
| **非功能性** | **DAU 1.2 B**，**QPS 150 k**，**Feed 延迟 <100 ms**，**可用性 99.9%**，**存储 500 PB**，**上传峰值 30 GB/s** | 这些指标决定后面的 **容量规划、缓存层、容错** 等设计 |
| **业务边界** | 只设计后端系统，前端 UI、运营后台、报表等不在范围 | 让讨论聚焦在核心技术难点上 |

### 1.1 关键指标拆解  

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **每日活跃用户 (DAU)** | 1.2 B | 直接使用 |
| **Feed 请求 QPS** | DAU × 5 次/分钟 ÷ 60 | 1.2 B × 5 / 60 ≈ **100 k** (保守估计) → 题目给 150 k，取上限 |
| **上传并发** | 假设峰值 10% 的 DAU 同时上传，平均视频大小 100 MB | 1.2 B × 0.1 × 100 MB / 1 s ≈ **12 TB/s**，但题目限定 30 GB/s，说明 **真实峰值** 只占很小比例，取 30 GB/s 作为系统设计上限 |
| **存储** | 30 天热点 + 冷数据 3 年 | 500 PB（已给出） |

### 1.2 业务流梳理  

```
[用户] → (1) 上传视频 → (2) 转码 + 审核 → (3) 存储 & CDN → (4) 写入 Feed 索引
[用户] → (5) 打开 App → 请求 Feed → (6) 推荐系统召回 + 排序 → (7) 返回视频列表
[用户] → (8) 点赞/评论/关注 → (9) 计数/关系写入 → (10) 实时刷新 UI
```

## 第二步：高层架构设计  

### 2.1 先画最小可用系统（MVP）

```
+-------------------+      +-------------------+      +-------------------+
|   Mobile Client  | ---> |   API Gateway     | ---> |   单体服务 (Go)   |
| (Feed/Upload UI) |      |   (REST/GraphQL)  |      | (业务+DB)         |
+-------------------+      +-------------------+      +-------------------+
```

- **优点**：实现快，代码量少。  
- **缺点**：**单点故障、水平扩展困难、无法满足 150k QPS、难以做灰度发布**。  

> **面试官** 常会问：**“如果流量翻倍怎么办？”** → 这时我们需要 **拆分**。

### 2.2 逐步拆分成微服务（业务分层）

```
                     +-------------------+
                     |   API Gateway     |
                     +---------+---------+
                               |
   +---------------------------+---------------------------+
   |            |               |               |         |
   v            v               v               v         v
Upload Svc   Feed Svc      Interaction Svc  User Svc   Audit Svc
   |            |               |               |         |
   |            |               |               |         |
   v            v               v               v         v
ObjectStore   Cache/Redis    Redis+MySQL      MySQL    ML/Audit
   +            +               +               +         +
   |            |               |               |         |
   v            v               v               v         v
   CDN <----------------------------+-----------------------+
```

- **Upload Service**：负责文件接收、分片上传、转码、写入对象存储。  
- **Feed Service**：处理 Feed 拉取请求，调用 **召回层** + **排序层**，返回缓存命中的视频 ID 列表。  
- **Interaction Service**：点赞、评论、分享计数，使用 **Redis** 做热点写，定时落库。  
- **User Service**：用户信息、关注关系、个人页数据。  
- **Audit Service**：机器审查 + 人工审核工作流。  

### 2.3 关键外部组件

| 组件 | 角色 | 选型理由 |
|------|------|----------|
| **对象存储 (Object Store)** | 视频原文件 & 转码产物 | 支持 **海量**、**高吞吐**、**分区容错**（如 AWS S3、阿里云 OSS、Ceph） |
| **CDN** | 静态文件（视频）分发，降低回源压力 | 全球节点、**边缘缓存**、**带宽计费**相对低 |
| **消息队列 (Kafka / Pulsar)** | 解耦上传 → 转码 → 审核 → 推荐索引的异步流程 | 高吞吐、持久化、支持 **分区** 与 **消费组** |
| **缓存层 (Redis Cluster)** | 热门视频 ID、用户点赞计数、关注列表、Feed 候选集合 | 读写延迟 < 1 ms，支持 **分片** 与 **持久化** |
| **搜索/倒排索引 (Elasticsearch / OpenSearch)** | 视频标签、话题检索、实时召回 | 支持 **全文检索**、**聚合**，可做 **倒排索引** |
| **离线推荐平台 (Spark / Flink + Hive / HDFS)** | 大规模离线特征计算、模型训练 | 批处理能力强，适合 **日/周** 召回模型 |
| **实时特征流 (Flink / Spark Structured Streaming)** | 实时用户行为流（点击、停留） | 支持 **低延迟**、**状态管理**，为召回提供最新特征 |
| **监控/告警 (Prometheus + Grafana, Jaeger)** | 系统可观测性 | **时序指标**、**链路追踪**、**告警** |

## 第三步：数据库设计  

### 3.1 总体数据模型划分  

| 类别 | 读写特性 | 推荐存储 | 说明 |
|------|----------|----------|------|
| **视频元信息** | 写少读多（上传后基本不变） | **关系型 DB (MySQL/PG)** + **二级缓存 (Redis)** | 需要事务保证唯一性（ID、唯一 URL） |
| **用户信息 & 关注关系** | 写少读多（注册/关注） | **关系型 DB** + **缓存** | 关注关系是 **图**，可以在 Redis 中用 **ZSET** 存储粉丝/关注列表 |
| **互动计数 (点赞、评论、分享)** | 写多读多 | **Redis (原子计数)** → **异步落库 MySQL** | 采用 **最终一致**，秒级同步到持久层 |
| **视频标签 / 话题** | 写少读多 | **Elasticsearch** | 用于倒排检索、实时召回 |
| **推荐候选集合** | 写多读多（实时更新） | **Redis Sorted Set** + **MySQL 备份** | 召回层返回 **Sorted Set** 按分值排序的 video_id 列表 |
| **审核状态** | 写少读多 | **MySQL** + **缓存** | 审核结果决定是否进入 Feed |

### 3.2 关键表结构（示例）  

```sql
-- videos
CREATE TABLE videos (
    video_id      BIGINT PRIMARY KEY,
    user_id       BIGINT NOT NULL,
    title         VARCHAR(255),
    description   TEXT,
    duration_sec  INT,
    resolution    ENUM('720p','1080p','4K'),
    codec         VARCHAR(32),
    status        ENUM('uploading','processing','approved','rejected') DEFAULT 'uploading',
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user (user_id),
    INDEX idx_status (status)
);

-- video_files (不同分辨率的对象存储路径)
CREATE TABLE video_files (
    video_id      BIGINT,
    resolution    ENUM('720p','1080p','4K'),
    object_url    VARCHAR(1024),
    PRIMARY KEY (video_id, resolution),
    FOREIGN KEY (video_id) REFERENCES videos(video_id)
);

-- users
CREATE TABLE users (
    user_id       BIGINT PRIMARY KEY,
    username      VARCHAR(64) UNIQUE,
    nickname      VARCHAR(64),
    avatar_url    VARCHAR(1024),
    create_ts     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status        ENUM('active','banned') DEFAULT 'active'
);

-- follows (用户关注关系)
CREATE TABLE follows (
    follower_id   BIGINT,
    followee_id   BIGINT,
    create_ts     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (follower_id, followee_id)
);

-- interaction_counters (持久化计数，周期同步)
CREATE TABLE interaction_counters (
    video_id      BIGINT PRIMARY KEY,
    like_cnt      BIGINT DEFAULT 0,
    comment_cnt   BIGINT DEFAULT 0,
    share_cnt     BIGINT DEFAULT 0,
    last_sync_ts  TIMESTAMP
);
```

> **为什么要把计数单独抽出来？**  
> - 防止 **热点写**（同一视频的点赞频繁）导致 **MySQL 行锁**。  
> - 在 Redis 中使用 **INCR**，秒级落库，兼顾 **强一致**（用户看到的点赞数即时）与 **高吞吐**（不影响主库）。

### 3.3 数据库分片 & 读写分离  

| 数据 | 分片键 | 分片策略 | 读写分离 |
|------|--------|----------|----------|
| videos / video_files | video_id (hash) | **水平分片**（如 256 分片） | 主库写，读库复制（异步） |
| users / follows | user_id (hash) | **水平分片** | 同上 |
| interaction_counters | video_id (hash) | **水平分片**（热点视频可单独放在专用节点） | 主库写，读库复制 |

> **不做分片** 的后果：单库写入峰值会超过 **10k TPS**，导致 **CPU、IO、锁竞争**，系统很快挂掉。

## 第四步：核心 API 设计  

下面列出 **最常用的 8+ 个 RESTful 接口**（也可以用 gRPC），并说明 **请求/响应、鉴权、限流**。  

### 4.1 上传相关 API  

| 方法 | 路径 | 功能 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/videos/init` | 初始化上传（获取 upload_id、分片信息） | `{ "user_id":123, "filename":"xxx.mp4", "size":12345678 }` | `{ "upload_id":"u-abc123", "chunk_size":4MB, "chunks":31 }` |
| `PUT` | `/api/v1/videos/{upload_id}/chunks/{num}` | 上传单个分片（支持断点续传） | 二进制流 | `202 Accepted` |
| `POST` | `/api/v1/videos/complete` | 完成上传，触发转码 & 审核 | `{ "upload_id":"u-abc123", "title":"xxx", "tags":["fun"] }` | `{ "video_id":987654321, "status":"processing" }` |

> **为什么要分三步？**  
> - 大文件 **分片上传** 能够在网络抖动时 **重试**，提升成功率。  
> - `init` 返回 **唯一 upload_id**，后端可以在 **对象存储** 中直接写入，避免流经业务服务器，减轻 **网络带宽**。  

### 4.2 Feed 拉取 API  

| 方法 | 路径 | 功能 | 请求参数 | 响应 |
|------|------|------|----------|------|
| `GET` | `/api/v1/feed` | 拉取下一个视频列表 | `user_id`, `last_video_id`(分页锚点), `page_size`(默认 20) | `{ "videos":[ {"video_id":123,"url":"https://...","like_cnt":1234,...}, ... ], "next_cursor":"..." }` |

- **实现细节**：在 Service 层先查询 **Redis Sorted Set** `feed:{user_id}`（已排好序的 video_id），若 **缓存未命中** 再走 **召回 + 排序** 流程并 **写回缓存**。  

### 4.3 互动相关 API  

| 方法 | 路径 | 功能 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/videos/{video_id}/like` | 点赞/取消赞 | `{ "user_id":123, "action":"like|unlike" }` | `{ "like_cnt": 12345 }` |
| `POST` | `/api/v1/videos/{video_id}/comment` | 发表评论 | `{ "user_id":123, "content":"好玩!" }` | `{ "comment_id": 9876, "comment_cnt": 567 }` |
| `POST` | `/api/v1/users/{target_id}/follow` | 关注/取消关注 | `{ "user_id":123, "action":"follow|unfollow" }` | `{ "follow_cnt": 1023 }` |

> **计数更新策略**：API 直接 **INCR** Redis 中对应 key（如 `video:123:like`），随后异步 **写入 MySQL**（每分钟一次批量落库），保证 **强一致感知**（用户立即看到变更）且 **不阻塞**。  

### 4.4 内容审核 API（内部）  

| 方法 | 路径 | 功能 | 触发点 |
|------|------|------|--------|
| `POST` | `/internal/audit/video` | 将视频 ID 推入机器审查队列 | 上传完成后 |
| `POST` | `/internal/audit/result` | 接收机器/人工审查结果 | 审核系统回调 |

- **审查结果** 更新 `videos.status`，若 `approved` 则 **写入 Feed 索引**；若 `rejected` 则 **标记为不可见**。

## 第五步：详细组件设计  

### 5.1 视频上传、转码、存储流程  

1. **客户端** 调用 `init` 获得 `upload_id` 与分片信息。  
2. **分片上传** 直接写入 **对象存储**（S3 Multipart Upload），**边缘节点**（如 CDN 边缘）可以提供 **预签名 URL**，降低业务服务器流量。  
3. **完成上传** 时，业务服务器向 **Kafka** 发送 `VideoUploaded` 事件（payload：video_id、object_key）。  
4. **转码服务**（基于 **FFmpeg** + **容器化**）消费该事件，使用 **弹性容器（K8s Job）** 进行 **多分辨率**（720p/1080p/4K）转码，产出新对象存储路径。  
5. **转码完成** 再发 `VideoTranscoded` 事件，触发 **内容审核**。  

> **为什么使用对象存储的分片上传？**  
> - **网络层面**：用户的上传流量直接走 **对象存储** 的 **高速通道**，不占用业务服务器的带宽。  
> - **可靠性**：对象存储自带 **分片校验**，失败分片可以单独重试，降低整体失败率。  

### 5.2 内容审核（机器 + 人工）  

| 步骤 | 说明 |
|------|------|
| **机器审查** | 使用 **CNN/Transformer** 对视频抽帧进行违规检测（暴恐、低俗等），把结果写入 `audit_result` 表。 |
| **人工复审** | 对机器打“疑似”或“低置信度”的视频推送到 **后台工单系统**，人工确认后更新 `status`。 |
| **结果分发** | `status = approved` → 推送 `VideoApproved` 事件到 **召回系统**； `rejected` → 标记 `visibility = false`，删除 CDN 缓存。 |

> **实时性要求**：机器审查 **≤ 200 ms**，人工审查 **≤ 30 min**（不影响热点视频的即时上线）。  

### 5.3 推荐系统（离线 + 实时）  

#### 5.3.1 离线召回（批处理）

1. **特征计算**（Spark）  
   - 用户特征：兴趣标签、历史观看向量（Embedding）  
   - 视频特征：标签向量、作者热度、最近互动等  
2. **模型训练**（XGBoost / DeepFM）  
   - 目标：CTR、播放时长、完播率  
3. **生成候选集合**  
   - 按用户划分（User Partition），输出 `user_id → [candidate_video_id]`，写入 **HDFS** 再导入 **Redis**（`feed:{user_id}`）  
4. **定时刷新**（每 4 小时）  

#### 5.3.2 实时召回（低延迟）

| 场景 | 需求 | 设计 |
|------|------|------|
| **热点视频** | 10 ms 内返回 100 条候选 | 将 **热点视频 ID** 按 **标签**、**地区**、**时间段** 存入 **Redis Sorted Set**（score = 热度）<br>查询时直接 **ZREVRANGE**，返回 Top‑N |
| **用户即时兴趣** | 基于最近 30 s 行为召回 | 使用 **Flink** 维护 **用户最近点击流**，实时更新 **用户‑标签倒排表**（Redis Hash） |
| **冷门视频** | 必须保证新上传视频能被看到 | 新视频入库后立即写入 **倒排索引（Elasticsearch）**，并在召回时加入 **“新鲜度 boost”** |

#### 5.3.3 排序层（Ranking）

- **特征聚合**：从 Redis、MySQL、ES 拉取候选视频的 **实时特征**（点赞数、热度、作者粉丝数）  
- **模型推理**：使用 **TensorFlow Serving** 进行 **CTR + 观看时长** 预测，返回 **score**  
- **业务规则**：强制过滤违规、已屏蔽、已看过的内容 | 通过 **Score + BusinessWeight** 排序后返回前 `page_size` 条  

> **为什么要把召回/排序拆分？**  
> - **召回** 需要 **极高的 QPS**（读取热点集合），可以全靠 **缓存**，不必每次跑模型。  
> - **排序** 只在 **候选集合**（几十到几百）上执行 **复杂模型**，资源消耗可控。  

### 5.4 互动计数与热点写优化  

1. **热点键分散**：对 **点赞计数** 使用 **hash 分片**（如 `video:like:{video_id % 256}`），每个分片对应独立 Redis 实例，避免单点热点。  
2. **写入流程**：  
   - **用户点击** → API → `INCRBY video_like:{shard}` → 返回最新计数（从 Redis 读取）。  
   - **异步批量落库**：每 5 s 收集一次 `INCR` 结果，写入 MySQL `interaction_counters`（`UPDATE ... SET like_cnt = like_cnt + delta`）。  
3. **防止超卖**：采用 **乐观锁**（版本号）或 **CAS**（Redis `WATCH`）保证 **幂等**。  

### 5.5 视频分发（CDN）  

| 步骤 | 说明 |
|------|------|
| **上传完毕 & 转码** | 将每种分辨率的对象 URL 写入 **MySQL**，并 **预热** 到 **CDN**（PUT 请求触发缓存）。 |
| **客户端请求** | Feed 返回的视频 URL 已经是 **CDN 加速的域名**（如 `https://cdn.tiktok.com/vid/123_1080p.mp4`）。 |
| **缓存失效** | 当视频被 **删除/违规** 时，发送 **Purge** 请求至 CDN，确保边缘节点及时失效。 |
| **冷热分层**  
  - **热点 30 天** → 存储在 **高性能 SSD**（对象存储的 “标准” 类）  
  - **冷数据** → 移动到 **低频/归档**（如 S3 Glacier）并在访问时 **自动回溯**（费用低） |
| **成本控制** | 使用 **生命周期规则** 自动将 **30 天未访问** 的对象转到 **低频**，并对 **4K** 版保留更短的热存周期（因为体积大）。 |

## 第六步：扩展性与高可用设计  

### 6.1 横向扩展（Scale‑out）  

| 组件 | 扩展方式 | 关键指标 |
|------|----------|----------|
| **API Gateway** | **负载均衡**（L4/7）+ **自动伸缩**（K8s HPA） | QPS 150 k，目标 CPU < 70% |
| **Upload Service** | **无状态** → 多副本；对象存储分片写入本地 **Nginx** 代理 | 支持 **30 GB/s**，每副本 2 GB/s |
| **转码集群** | **容器化 Job**，按需求弹性伸缩（K8s Queue） | 每 1 GB 视频约 2 min 转码，峰值 30 GB/s → 1500 并发 Job |
| **Redis Cluster** | **分片**（hash slot）→ 256 分片，**主从复制**（1 主 1 备） | 读写 QPS 1 M，单片 < 10k QPS |
| **MySQL** | **读写分离** + **水平分片**（User‑Id, Video‑Id） | 写入峰值 50 k TPS，读库复制延迟 < 100 ms |
| **Kafka** | **分区**（topic × 64）+ **多副本**（3） | 每分区吞吐 1 MB/s，整体 30 GB/s |
| **CDN** | **全球边缘节点** + **智能调度** | 目标 P99 视频加载 < 100 ms（首帧） |

### 6.2 容错与灾备  

| 场景 | 方案 |
|------|------|
| **单节点故障** | 所有业务均为 **无状态**，使用 **负载均衡 + 自动重启**；Redis、MySQL 采用 **主从复制 + 自动故障转移**（如 **Sentinel**、**Orchestrator**） |
| **跨机房灾备** | **多活**（至少 3 个大区）<br>1. **数据同步**：MySQL 使用 **双向复制**（或 **TiDB**）<br>2. **对象存储跨 Region 同步**（CRR）<br>3. **DNS 负载均衡**（GeoDNS）切流 |
| **网络分区** | 采用 **幂等 API**（重试 ID）+ **业务回滚**（上传失败回滚对象） |
| **CDN 缓存雪崩** | 设置 **缓存预热**、**热点降级**（回源限流） |
| **服务升级** | **灰度发布**（Canary）+ **回滚**（K8s Deployment） |

### 6.3 监控、告警与限流  

| 维度 | 关键指标 | 工具 |
|------|----------|------|
| **性能** | QPS、P99 latency、CPU/Memory、网络 I/O | Prometheus + Grafana |
| **业务** | 视频上传成功率、转码成功率、审核通过率、Feed 曝光/点击率 | Grafana + Alertmanager |
| **错误** | 5xx 错误率、Kafka 死信队列大小、Redis 超时 | ELK Stack |
| **流量控制** | **每用户** QPS 限流（令牌桶），**全局** 入口限流（API Gateway） | Nginx + Lua / Envoy RateLimiter |

> **如果不做监控**，系统在高峰时段出现 **卡顿** 或 **数据丢失** 时，你根本不知道根因在哪里，面试官会直接扣分。

## 第七步：常见面试追问与回答  

### 1️⃣ “实时召回需要在 10 ms 内返回 100 条候选视频，你会怎么设计缓存层、倒排索引以及热点分层？”  

**回答要点**  

1. **热点分层**  
   - 将 **最近 1 h、24 h、7 d** 的热点视频分别放入 **3 个 Redis Sorted Set**（`hot:1h`, `hot:24h`, `hot:7d`），Score = **热度指数**（播放数、点赞数加权）。  
   - 对 **标签**、**地区** 再细分（如 `hot:1h:tag:dance:cn`），保证查询时 **键空间有限**，可在 **单机** 读取完成。  

2. **倒排索引**  
   - 使用 **Elasticsearch** 为每个视频建立 **标签、标题、描述、作者 ID** 等倒排字段。  
   - 为 **实时性** 创建 **近实时索引**（refresh interval 1 s），并在 **转码完成** 时即写入。  

3. **查询流程（伪代码）**  

   ```text
   // 1. 根据用户兴趣标签取对应热点集合
   candidate_set = ZUNIONSTORE tmp_hot
        [hot:1h:tag:dance:cn, hot:24h:tag:comedy:us, ...]   // O(1) 合并

   // 2. 若集合不足 100 条，从 ES 召回
   if ZCARD(candidate_set) < 100:
       es_res = ES.search(
           query = bool(
               should = [terms(tag, user_interests), term(region, user_region)]
           ),
           size = 200
       )
       // 将 ES 返回的 video_id 加入 candidate_set
       ZADD candidate_set score_of_es_doc video_id

   // 3. 直接返回前 100 条（已排好热度顺序）
   result = ZREVRANGE candidate_set 0 99
   ```

4. **为什么能做到 10 ms**  
   - **全部在内存**（Redis）完成 **合并 + 排序**，单机操作延迟 < 2 ms。  
   - **ES 查询** 只在集合不足时才触发，且只拉取 **200 条**，耗时 ~ 5 ms（近实时索引）。  
   - 网络往返（客户端 → API Gateway）在同城机房 < 3 ms，整体 < 10 ms。  

### 2️⃣ “面对 500 PB 的海量视频，你如何在保证高可用的前提下，利用冷热分层、对象存储与 CDN 做成本优化？”  

**回答要点**  

1. **冷热分层策略**  
   - **热存储（0‑30 天）**：对象存储 **Standard**（SSD），配合 **多 AZ 冗余**（3 副本），保证 **99.999%** 可用性。  
   - **冷存储（30‑180 天）**：对象存储 **Infrequent Access**（IA），费用下降 70%，仍保持 **快速读取**（几百 ms）。  
   - **归档（>180 天）**：对象存储 **Glacier/Deep Archive**，费用再下降 90%，恢复时间 1‑4 h（对老视频接受度高）。  

2. **生命周期规则**（示例）  

   ```json
   {
     "Rules": [
       { "ID":"to-IA", "Prefix":"videos/", "Status":"Enabled",
         "Transition": { "Days":30, "StorageClass":"STANDARD_IA"} },
       { "ID":"to-Glacier", "Prefix":"videos/", "Status":"Enabled",
         "Transition": { "Days":180, "StorageClass":"GLACIER"} }
     ]
   }
   ```

3. **CDN 缓存**  
   - **热点视频**（播放次数 Top 1 %）设置 **长缓存 TTL（12 h）**，配合 **Cache‑Control** 头部。  
   - **冷视频** 采用 **短 TTL（5 min）**，避免 CDN 持久占用空间。  

4. **成本计算（粗略）**  

   | 类别 | 大小（PB） | 单价（$/GB/月） | 月费用 |
   |------|------------|----------------|--------|
   | 热存储 | 30 | $0.023 | $690 k |
   | 冷存储 | 150 | $0.0125 | $1.875 M |
   | 归档 | 320 | $0.004 | $1.28 M |
   | **合计** | 500 | — | **≈ $3.8 M / 月** |

5. **高可用**  
   - 对象存储 **跨 AZ** 多副本，自动容错。  
   - CDN **多节点** 失效自动切回源。  
   - **元数据**（video 表）保存在 **MySQL 双活**，即使对象存储出现局部故障，仍可返回 **备用 URL**（回源至其他 Region）。  

### 3️⃣ “用户点赞/评论的计数需要强一致还是最终一致？请说明你的数据模型以及如何防止热点写入导致的性能瓶颈。”  

**回答要点**  

1. **业务需求**：用户在滑动时希望立即看到点赞数变化 → **强感知一致**（用户感知）。但系统整体可以容忍 **几秒** 的延迟 → **最终一致** 足以。  

2. **模型**  

   - **写路径**：API → **Redis**（原子 INCR） → **返回最新值**（读 Redis）  
   - **异步同步**：每 5 s 将 **增量（delta）** 写入 MySQL `interaction_counters`（批量 UPDATE）  

   ```text
   // Redis key 设计
   video:like:{shard_id}  => hash { video_id => count }
   // shard_id = video_id % 256
   ```

3. **热点写分散**  

   - **Hash 分片**：把同一视频的计数 **拆到不同 Redis 实例**（256 分片），单实例写入量 ≈ 总写入 / 256。  
   - **幂等性**：客户端在网络抖动时可能重发请求，使用 **唯一请求 ID** + **SETNX** 防重放。  

4. **落库策略**  

   - **批量 UPDATE**：`UPDATE interaction_counters SET like_cnt = like_cnt + ? WHERE video_id = ?`，一次写入 10 k 条。  
   - **持久化**：MySQL 采用 **InnoDB**，开启 **binary log**，支持 **Point‑In‑Time Recovery**。  

5. **如果不做上述分散**，单个热点视频（如明星作品）可能在 **秒级** 产生 **10⁶ 次点赞**，导致 **Redis 单实例 QPS** 超过 **100 k**，出现 **CPU/网络瓶颈**，甚至 **数据丢失**。

## 心得与反思  

### 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **1️⃣ 推荐系统的实时召回与缓存层** | 必须在 **10 ms** 内返回 **100 条** 候选，且要兼顾 **新视频曝光** 与 **热点分层**。 | - 先分析 **数据规模**（几千万视频）→缓存不可能全部放。 <br>- 将 **热点** 按时间窗口、标签、地域拆分到 **Redis Sorted Set**，保证 O(1) 读取。 <br>- 采用 **倒排索引（ES）** 只在缓存不足时补充，保持 **查询成本** 可控。 <br>- 通过 **ZUNIONSTORE** 合并多个集合，保持 **排序** 已在内存完成，避免二次排序。 |
| **2️⃣ 海量视频的冷热分层 + 成本控制** | 500 PB 数据量庞大，直接放在高性能存储成本天文数字。 | - 先估算 **热点比例**（≈ 1%）→决定 **热/冷/归档** 三层。 <br>- 结合 **对象存储生命周期** 自动迁移，降低运维复杂度。 <br>- 计算 **费用** 与 **可用性**，验证 **SSD+IA+Glacier** 组合能在 **99.999%** 可用的前提下把成本控制在可接受范围。 |

### 新手最容易犯的错误（至少 2 条）  

1. **一次性把所有功能堆进一个巨型单体**  
   - **后果**：部署、扩容、容错都极其困难，面试官会质疑你的 **可扩展性思维**。  
   - **正确做法**：先划分 **业务边界**（上传、Feed、互动、审核），每块做 **无状态服务**，再逐步拆分成微服务。  

2. **忽视 **热点写** 的影响**（尤其点赞、关注计数）  
   - **后果**：单点 Redis/DB 被瞬间击穿，导致 **系统崩溃**，且难以解释为什么要使用 **分片+异步落库**。  
   - **正确做法**：在设计计数时，先做 **写分散**（hash 分片）、**幂等**、**批量持久化**，并在监控中设定 **写入 QPS 阈值**。
