# 第 40 天：设计 拼多多秒杀系统

> 生成日期：2026-04-16

---

## 题目背景  
拼多多秒杀系统是电商平台在大促活动期间（如“双11”“618”）提供的高并发抢购功能，用户在极短的时间窗口内以超低价抢购限量商品。系统需要在毫秒级响应的同时，保证库存不出现超卖、用户体验流畅。

## 面试场景设定  
**面试官**：  
“假设我们要为拼多多实现一套支持 1 秒内完成 10 万笔订单的秒杀系统，请你从需求、架构到实现细节完整设计该系统。先从整体思路说起，随后我们会逐步深入细节。”  

---

## 功能性需求  

1. **秒杀商品展示**  
   - 前端实时展示即将开始的秒杀商品列表、倒计时、限量库存等信息。  

2. **秒杀下单**  
   - 用户在秒杀开始后点击“立即抢购”，系统必须在 200 ms 内返回下单成功或失败的结果。  

3. **库存严格控制**  
   - 必须防止超卖，保证每件商品的实际扣减库存不超过预设库存。  

4. **订单生成与持久化**  
   - 成功抢购后生成订单，写入订单库并发送订单确认消息（如短信、站内通知）。  

5. **防刷/限流**  
   - 对同一用户、同一 IP、同一设备进行访问频率限制，防止恶意刷单。  

6. **秒杀结果查询**  
   - 用户可查询秒杀是否成功、订单状态以及支付入口。  

---

## 非功能性需求  

| 指标 | 目标值 | 备注 |
|------|--------|------|
| **并发 QPS** | 100,000 QPS（峰值） | 包括下单请求、查询请求等 |
| **单请求延迟** | ≤ 200 ms（99% 请求） | 包括网络、业务处理、返回 |
| **可用性** | 99.99%（月度） | 包括秒杀期间的整体可用 |
| **库存一致性** | 强一致性（无超卖） | 通过分布式锁/事务等手段实现 |
| **数据持久化** | 秒杀结束后 5 min 内完成全部订单落库 | 采用异步落库+双写机制 |
| **流量峰值** | 10 M 并发连接 | 需要 CDN、负载均衡等支撑 |

**本题范围内**  
- 秒杀商品的前端展示、下单、库存扣减、订单写入、结果查询。  
- 防刷、限流、流量高可用设计。  
- 关键技术选型（缓存、消息队列、分布式锁、数据库等）以及容量估算。  

**不在本题范围**

- 运营后台的秒杀活动配置页面（只需假设已有接口提供商品、库存信息）。

# 题解

## 解题思路总览  

> **目标**：在 1 秒内完成 10 万笔订单的秒杀系统。  
> **核心难点**：  
> 1. **超高并发**（100 k QPS）下的 **库存强一致性**（防止超卖）  
> 2. **毫秒级响应**（≤ 200 ms）  
> 3. **削峰填谷**：瞬时流量需要被平滑到后端可承受的处理速率  
> 4. **防刷/限流**：保护系统不被恶意流量击垮  

本解答将 **从需求拆解 → 规模估算 → 逐层演进架构 → 关键实现细节** 进行完整展开，力求每一步都解释「**为什么这么做**」以及「**不这么做会怎样**」，帮助没有系统设计经验的同学一步步搭建起可演进的秒杀系统。

---

## 第一步：理解需求与规模估算  

| 需求 | 关键指标 | 业务含义 |
|------|----------|----------|
| **秒杀商品展示** | 实时库存、倒计时 | 前端需要快速读取最新库存信息 |
| **秒杀下单** | 200 ms 响应，成功/失败立即返回 | 用户体验关键 |
| **库存严格控制** | 强一致性、无超卖 | 业务合规、用户信任 |
| **订单生成** | 5 min 内全部落库 | 秒杀结束后批量持久化 |
| **防刷/限流** | 按用户/IP/设备限速 | 防止刷单、系统被攻击 |
| **秒杀结果查询** | 低延迟查询订单状态 | 业务闭环 |

### 1.1 流量峰值估算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **下单请求 QPS** | 100 000 req/s（题目要求） | 100 k QPS |
| **查询请求 QPS** | 假设 30% 用户在抢购后查询 | 30 k QPS |
| **总 QPS** | 100 k + 30 k | **130 k QPS** |
| **并发连接数** | 10 M 并发（题目给出） → 主要是 TCP 连接数，需使用 **长连接 + 连接池** | 10 M |
| **单机能处理请求数**（经验值） | 2 k QPS（单台高配机器） | 需要 **≈ 50 台** 业务机器做水平扩展（130 k / 2 k） |

> **如果不做并发估算**，直接把所有请求压在几台机器上，会导致 CPU/网络瞬间打满，出现 **超时、错误率飙升**，甚至宕机。

### 1.2 数据规模  

- **商品 SKU**：假设一次秒杀只涉及 10‑100 种商品，每种商品库存 ≤ 10 000。  
- **订单量**：峰值 100 k 条订单。单条订单大小约 1 KB → 100 MB 数据在 1 秒内产生。  

> **不考虑数据规模**，在后端写入时若直接落库，会造成 **磁盘 I/O 瓶颈**，导致响应时间远超 200 ms。

## 第二步：高层架构设计  

### 2.1 从最小可用系统（MVP）出发  

```
[用户] → HTTP → [API Gateway] → [秒杀服务] → Redis（库存） → MySQL（订单）
```

- **API Gateway**：统一入口、做限流、鉴权、流量监控。  
- **秒杀服务**（单体）负责：  
  - 检查活动时间、用户合法性  
  - 原子扣减 Redis 库存（Lua 脚本）  
  - 生成订单写入 MySQL（同步）  
- **Redis**：使用 **单点**（或主从）实现库存的原子操作。  
- **MySQL**：普通单库，事务保证订单写入成功。

> **缺点**：  
> - 单点 Redis 成为瓶颈，且没有削峰；  
> - 同步写库导致响应时间受磁盘 I/O 影响；  
> - 整体可用性低，单机故障即全盘崩溃。  

### 2.2 向高可用分布式系统演进  

```
                 ┌─────────────────────┐
                 │   CDN / WAF          │
                 └───────▲─────▲───────┘
                         │     │
                 ┌───────┴─────┴───────┐
                 │   Load Balancer     │
                 └───────▲─────▲───────┘
                         │     │
               ┌─────────┘     └─────────┐
               │                         │
      ┌────────▼───────┐      ┌──────────▼───────┐
      │ API Gateway    │      │   API Gateway    │
      └───────▲────────┘      └───────▲──────────┘
              │                       │
   ┌──────────▼─────────────┐ ┌───────▼─────────────┐
   │ 秒杀业务微服务集群 (N) │ │   结果查询微服务   │
   └───────▲───────▲───────┘ └───────▲───────▲─────┘
           │       │               │       │
   ┌───────▼─────┐ ┌─▼───────┐ ┌─────▼─────┐ ┌─▼───────┐
   │ Redis 集群  │ │ MQ (Kafka│ │ MySQL 主从 │ │ Elastic │
   │（库存）    │ │ /Rocket)│ │（订单）   │ │Search   │
   └─────────────┘ └─────────┘ └────────────┘ └─────────┘
```

**关键组件说明**  

| 组件 | 作用 | 为什么要这么设计 |
|------|------|-------------------|
| **CDN / WAF** | 静态资源缓存、基础 DDoS 防护 | 大量用户会请求商品列表、图片，提前缓存可以削减后端压力；WAF 阻止恶意流量 |
| **Load Balancer** | 分发 TCP/HTTP 连接，提供健康检查 | 保证流量均匀落在多台机器上，单点故障不影响整体 |
| **API Gateway** | 鉴权、统一限流、熔断、灰度发布 | 统一入口，业务服务只关注业务逻辑；限流在入口层做可以提前拦截非法流量 |
| **秒杀业务微服务** | 业务核心：检查活动、扣库存、投递订单消息 | 微服务化实现 **水平扩展**，每台机器只负责一部分请求 |
| **Redis 集群（库存）** | 采用 **主从+分片**，提供 **原子扣减** | 单机 Redis 无法支撑 100 k QPS，集群提供高并发读写、容错 |
| **消息队列 (Kafka / RocketMQ)** | 异步落库、削峰、可靠传输 | 将秒杀请求的“下单”转为可靠的 **event**，业务线程只负责快速返回，消费端可以慢慢落库 |
| **MySQL 主从** | 订单持久化、事务保证 | 主库写入订单，读从库供查询；主从复制提供高可用 |
| **ElasticSearch** | 实时订单查询、状态过滤 | ES 对高并发查询友好，且可以做全文、聚合，提升查询响应 |
| **监控/告警** | Prometheus + Grafana，链路追踪（Jaeger） | 及时发现瓶颈、异常，满足 99.99% 可用性目标 |

> **不做削峰**（直接同步写库）会导致 **数据库写入压力瞬间飙到 100 k QPS**，远超过常规 MySQL 的写入能力（≈ 5 k QPS），系统很快崩溃。

## 第三步：数据库设计  

### 3.1 库存（Redis）  

- **Key 设计**  
  - `seckill:stock:{seckill_id}:{sku_id}` → 整数，表示剩余库存  
  - `seckill:info:{seckill_id}` → 哈希，存放活动开始/结束时间、总库存等元信息  

- **原子扣减**（Lua 脚本）  

```lua
-- 参数: KEYS[1] = stock_key, ARGV[1] = 请求数量 (通常为 1)
local stock = tonumber(redis.call('GET', KEYS[1]))
if not stock then
    return -1   -- 库存键不存在，活动未初始化
end
if stock <= 0 then
    return 0    -- 库存已耗尽
end
redis.call('DECRBY', KEYS[1], ARGV[1])
return 1        -- 扣减成功
```

> **为什么使用 Lua**：  
> - **原子性**：整个脚本在 Redis 中一次性执行，避免并发下的“检查‑扣减”竞争条件。  
> - **性能**：一次网络往返即可完成检查和扣减，比先 `GET` 再 `DECR` 多一次 RTT 更慢。  

### 3.2 订单表（MySQL）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `order_id` | BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT | 全局唯一订单号 |
| `user_id` | BIGINT NOT NULL | 下单用户 |
| `seckill_id` | BIGINT NOT NULL | 秒杀活动 id |
| `sku_id` | BIGINT NOT NULL | 商品 SKU |
| `status` | TINYINT NOT NULL DEFAULT 0 | 0‑待支付，1‑已支付，2‑已取消 |
| `create_time` | DATETIME NOT NULL | 订单创建时间 |
| `pay_time` | DATETIME NULL | 支付成功时间 |
| `msg_id` | VARCHAR(64) NOT NULL | 对应 MQ 消息唯一标识（幂等） |
| **唯一索引** (`user_id`,`seckill_id`,`sku_id`) | 防止同一用户多次抢同一 SKU |  

> **幂等设计**：消息投递到 MQ 时生成全局唯一 `msg_id`，落库时使用 `INSERT ... ON DUPLICATE KEY UPDATE`，保证 **同一条下单消息只会写一次**。

### 3.3 订单查询（ElasticSearch）  

- **索引结构**  
  - `order_id`, `user_id`, `seckill_id`, `status`, `create_time`（date）  
- **同步方式**：消费 MQ 中的订单落库成功事件，异步写入 ES。  

> **不做 ES**，直接查询 MySQL，面对 30 k QPS 查询会导致 **读库压力剧增**，影响写入性能。

## 第四步：核心 API 设计  

| API | 方法 | 路径 | 参数 | 返回 | 关键实现点 |
|-----|------|------|------|------|-----------|
| **获取秒杀商品列表** | GET | `/seckill/{seckill_id}/items` | `seckill_id` | 商品列表（包含实时库存） | 从 Redis `MGET` 批量获取库存，返回前端 |
| **下单（抢购）** | POST | `/seckill/{seckill_id}/order` | `user_id`, `sku_id` | `{orderId, status}`（成功/失败） | 1. API Gateway 限流 <br>2. 秒杀服务调用 Lua 脚本扣库存 <br>3. 库存成功后发送订单消息到 MQ <br>4. 立即返回 `orderId`（预生成） |
| **查询订单状态** | GET | `/order/{order_id}` | `order_id` | `{status, payUrl}` | 读取 ES（或缓存）快速返回 |
| **获取秒杀活动信息** | GET | `/seckill/{seckill_id}` | `seckill_id` | 活动起止时间、商品列表 | 读取 MySQL 活动表 + Redis 库存 |

### 4.1 下单流程（伪代码）  

```java
// API Gateway 限流 + 鉴权后进入 SecKillService
public OrderResponse placeOrder(Long userId, Long seckillId, Long skuId) {
    // 1. 检查活动状态（从 Redis 缓存的活动信息读取）
    if (!activityService.isRunning(seckillId)) {
        return fail("秒杀未开始或已结束");
    }

    // 2. 调用 Lua 脚本原子扣库存
    Long result = redisTemplate.execute(luaScript,
            Collections.singletonList(stockKey(seckillId, skuId)),
            Collections.singletonList("1"));
    if (result == null || result <= 0) {
        return fail("库存不足");
    }

    // 3. 预生成订单号（雪花算法）并构造消息
    long orderId = IdGenerator.nextId();
    OrderMessage msg = new OrderMessage(orderId, userId, seckillId, skuId);
    // 4. 发送到 MQ，确保可靠投递（事务性 Producer）
    mqProducer.send(msg);

    // 5. 快速返回给前端
    return success(orderId);
}
```

> **为什么提前生成 orderId 并返回**：  
> - 前端需要展示「抢购成功」的视觉反馈，若等到真正落库后才返回，会导致 **响应时间 > 200 ms**。  
> - 订单号唯一性通过 **雪花算法**（或 Redis INCR）保证。

## 第五步：详细组件设计  

### 5.1 流量入口层（CDN → WAF → LB → API Gateway）  

| 设计要点 | 解释 |
|----------|------|
| **CDN 缓存商品图片、静态页面** | 大幅降低后端请求，削峰效果显著 |
| **WAF（Web Application Firewall）** | 基于 IP、UA、行为规则拦截已知攻击，防止大规模 DDoS |
| **L4/L7 负载均衡** | L4（TCP）实现长连接复用，L7（HTTP）做 Header 路由（如灰度） |
| **API Gateway 限流** | **令牌桶**（per‑user）+ **漏斗**（全局）<br>① 每个用户 1 s 只能发 5 次请求 <br>② 整体 QPS 设为 150 k，超出直接返回 429 |
| **熔断/降级** | 当秒杀服务不可用时，返回「系统繁忙」而不是阻塞 |

> **不做入口层限流**：流量直接冲到业务服务，瞬时请求会把业务机器的线程池耗尽，导致 **连接阻塞、超时**。  

### 5.2 秒杀业务微服务  

- **部署**：容器化（Docker）+ K8s 自动伸缩（Horizontal Pod Autoscaler）  
- **线程模型**：Netty / Spring WebFlux（非阻塞） → 减少每个请求占用的线程数，提升并发度。  
- **核心处理流程**：  
  1. **快速返回**：只做库存扣减 + MQ 投递，耗时主要是一次 Redis‑Lua 调用 + MQ 发送（约 1‑2 ms）  
  2. **业务降级**：当 Redis 延迟 > 10 ms，返回「抢购过于火爆」而非排队  

- **热点商品分片**：将热点 SKU 按 `hash(seckillId, skuId) % N` 分配到不同 Redis 分片，避免单个节点热点。  

### 5.3 Redis 集群（库存）  

| 参数 | 推荐值 | 原因 |
|------|--------|------|
| **节点数** | 6（3 主 + 3 从） | 奇数节点保证多数写成功，容错 1 节点失效 |
| **每个主节点内存** | 32 GB（使用 SSD） | 10 k SKU × 10 k 库存 = 100 M 数据，留足 10 倍冗余 |
| **持久化** | AOF（每秒同步） | 防止因宕机导致库存丢失 |
| **网络** | 10 Gbps 内部专网 | 满足 100 k QPS 的写入需求 |
| **监控** | Redis Sentinel + Prometheus | 自动故障转移、告警 |

> **不使用 Redis 集群**，单机 Redis 在高并发写入时会出现 **CPU 饱和、延迟激增**，导致库存扣减失败或超时。  

### 5.4 消息队列（Kafka / RocketMQ）  

- **主题**：`seckill_order`（分区数 = 30，建议 ≥ 业务机器数）  
- **生产者**：同步发送（`acks=all`）保证消息落盘后再返回成功给业务层  
- **消费者**：订单落库服务（多实例）采用 **消费者组**，实现水平消费。  
- **幂等**：消息体中包含唯一 `msg_id`，消费者在写库前先查询 `msg_id` 是否已存在，使用 `INSERT ... ON DUPLICATE KEY UPDATE`。  

| 场景 | 处理方式 |
|------|----------|
| **消息积压** | 通过 **流控**（生产者限速）和 **分区扩容**；监控 `lag` 告警 |
| **消息重复** | 消费端幂等处理 + **事务性 Producer**（Kafka 0.11+） |
| **消费失败** | 死信队列（DLQ），人工介入或补偿脚本 |  

### 5.5 订单落库服务  

1. **消费 MQ** → 解析订单消息  
2. **检查幂等**（`msg_id`）  
3. **写入 MySQL**（事务）  
4. **写入 ES**（异步）  
5. **发送确认消息**（可选，用于用户中心推送）  

> **为何不直接在秒杀服务里写 MySQL**：  
> - **写库耗时 10‑20 ms**（磁盘 I/O）会拉高整体响应，违背 200 ms SLA。  
> - **同步写库** 会导致 **流量峰值直接映射到 DB**，难以扩展。  

### 5.6 结果查询服务  

- **读取路径**：先查询 **Redis 缓存**（`order_status:{orderId}`），若未命中再查 **ES** → **MySQL**（最后兜底）。  
- **缓存失效策略**：订单状态在 5 min 内保持在 Redis，之后淘汰，防止缓存穿透。

   - **Device (UA/DeviceID) Bucket**：每秒 10 次

## 第七步：常见面试追问与回答  

### 7.1 库存一致性实现  

**问**：如果使用 Redis 的原子操作和 Lua 脚本，你如何保证在高并发下仍然不出现超卖？是否需要额外的补偿机制？

**答**：  
- **Lua 脚本的原子性**：Redis 在执行脚本期间会 **阻塞该 key 所在的分片**，保证脚本内部的 `GET` + `DECRBY` 是一次原子操作，竞争条件被彻底消除。  
- **超卖的根本原因**是 **库存扣减成功但后续写库失败**，导致库存已扣但订单未落库。为此我们采用 **消息队列**：扣减成功后立即 **投递订单消息**，即使写库失败也可以 **通过补偿脚本**（消费端幂等重试）恢复。  
- **补偿机制**：  
  1. **定时任务**扫描 Redis 中 `stock_key` 与 MySQL 订单表的差异，若发现扣减但订单缺失，则重新投递补偿消息或回滚库存（使用 `INCRBY`）。  
  2. **幂等消费**确保同一条消息多次消费只会生成一条订单，避免重复扣减。  

**结论**：**Redis Lua** 保证扣减不超卖；**MQ + 幂等消费** 防止因后端异常导致的库存与订单不一致；**补偿任务** 进一步确保最终一致性。

### 7.2 流量削峰与异步落库  

**问**：请详细说明秒杀请求如何进入消息队列并异步落库的完整流程，包括容错和幂等处理。

**答**：

1. **秒杀请求到达业务服务** → **Lua 扣减库存**（原子）  
2. **库存成功** → **生成全局唯一 `msg_id` 与 `order_id`**（雪花算法）  
3. **构造 `OrderMessage`**（包含用户、商品、时间戳、msg_id）  
4. **发送到 MQ**（Producer `acks=all`，同步等待 broker 确认）  
   - 若发送失败，业务层返回「系统繁忙」并可重试（最多 2 次）  
5. **业务层立即返回 `order_id`** 给前端（响应 ≤ 200 ms）  

**消费者侧**（订单落库服务）：

```java
while (true) {
    ConsumerRecord<String, OrderMessage> record = consumer.poll();
    OrderMessage msg = record.value();

    // 幂等检查：先查询 msg_id 是否已存在
    if (orderDao.existMsgId(msg.getMsgId())) {
        // 已处理过，直接提交 offset
        consumer.commitSync();
        continue;
    }

    // 开始事务
    try {
        connection.setAutoCommit(false);
        orderDao.insert(msg);               // INSERT ... (msg_id)
        connection.commit();
        // 异步写入 ES（不影响事务）
        esClient.indexAsync(msg);
        // 发送成功确认（可选）
        consumer.commitSync();
    } catch (Exception e) {
        connection.rollback();
        // 记录错误，Kafka 自动重试（或发送到 DLQ）
    }
}
```

- **容错**：  
  - **Producer**：`retries` + `acks=all` 保证不丢失。  
  - **Consumer**：异常时不提交 offset，Kafka 会 **重新投递**；若多次失败，进入 **DLQ**，后续人工或补偿脚本处理。  
- **幂等**：`msg_id` 唯一键约束 + `INSERT ... ON DUPLICATE KEY UPDATE`，确保同一条消息多次消费只产生一条订单。

### 7.3 防刷与限流策略  

**问**：在秒杀活动期间，如何基于用户、IP、设备三维度进行限流？如果出现突发流量暴涨，你会怎样动态扩容？

**答**：

#### 1️⃣ 三维限流实现（基于 Redis）  

```lua
-- key: limit:user:{userId}
-- token bucket: capacity=5, refillRate=5 tokens/s
local token = redis.call('GET', KEYS[1])
if not token then token = 5 end
if tonumber(token) > 0 then
    redis.call('DECR', KEYS[1])
    return 1   -- 通过
else
    return 0   -- 被限流
end
```

- **User‑ID Bucket**：`capacity=5, refill=5/s`（每秒最多 5 次）  
- **IP Bucket**：`capacity=20, refill=20/s`（防止同网段刷）  
- **Device Bucket**（DeviceID/UA+IMEI）：`capacity=10, refill=10/s`  

在 **API Gateway** 中依次检查三桶，只要任意一桶返回 0，即返回 **429 Too Many Requests**。  

#### 2️⃣ 动态扩容方案  

| 触发条件 | 扩容动作 |
|----------|----------|
| **Redis 延迟 > 10 ms**（监控指标） | 自动 **水平扩容 Redis 分片**（增加节点，重新哈希） |
| **MQ lag > 5 s** | **增加 Topic 分区数**，启动更多消费者实例 |
| **CPU/内存利用率 > 80%**（业务 Pod） | K8s HPA 按 CPU/RT 伸缩 Pod 数 |
| **入口层 QPS 达到阈值** | 动态添加 API Gateway 实例，更新 LB 配置 |

**实现**：利用 **Prometheus Alertmanager** 触发 **K8s HorizontalPodAutoscaler** 或 **自定义脚本**（调用云厂商 API）进行弹性伸缩。

## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 思考过程 |
|------|----------|
| **库存强一致性 vs 高并发** | 初始想直接使用 **分布式事务**（2PC）来保证库存与订单原子提交，结果发现 2PC 在 100 k QPS 场景下 **网络往返次数多、锁等待时间长**，会导致响应远超 200 ms。于是转向 **Redis 原子扣减 + MQ 异步落库** 的方案，结合 **幂等消费** 与 **补偿任务**，在保证一致性的同时满足性能要求。 |
| **削峰方案的选型** | 曾考虑 **直接写库 + MySQL 主从复制**，但在高并发写入时主库写入瓶颈（≈5 k QPS）无法支撑 100 k QPS。后来决定引入 **消息队列**，把瞬时写入转化为 **异步消费**，实现 **流量削峰**。这一步的关键是保证 **消息不丢失** 与 **消费幂等**，于是加入 **唯一 msg_id** 与 **事务性 Producer**。 |

### 2️⃣ 新手最容易犯的错误  

1. **把全部业务放在单体服务里**  
   - 结果：无法水平扩容、单点故障导致全站不可用。  
   - 正确做法：**微服务化** + **容器化**，使用 **K8s** 或 **服务网格** 实现弹性伸缩。  

2. **仅依赖数据库事务防止超卖**  
   - 结果：在高并发下数据库锁竞争激烈，响应时间爆炸。  
   - 正确做法：**在缓存层（Redis）做原子扣减**，再通过 **MQ** 异步持久化，降低 DB 直接参与的路径。  

### 3️⃣ 学习建议和可延伸的方向  

| 建议 | 说明 |
|------|------|
| **深入掌握 Redis Lua 与分布式锁** | 练习编写高效的 Lua 脚本、了解 RedLock 的局限，能帮助在秒杀、抢购等高并发场景中实现原子操作。 |
| **熟悉消息队列的可靠投递和幂等消费** | 学习 Kafka / RocketMQ 的事务生产者、Exactly‑once 语义以及消费端的幂等实现（唯一键、幂等表）。 |
