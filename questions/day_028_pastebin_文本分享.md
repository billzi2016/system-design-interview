# Pastebin 文本分享系统设计题

## 1️⃣ 题目背景  
Pastebin 是一个在线文本存储与分享平台，用户可以快速粘贴短文本（代码片段、日志、笔记等）并生成唯一的短链接，供他人查看或下载。系统需要在海量短文本的写入、读取和过期删除之间保持高可用和低延迟。

## 2️⃣ 面试场景设定  
> **面试官**：我们想构建一个类似 Pastebin 的文本分享服务，要求用户能够随时创建、读取、编辑（可选）以及删除短文本，并且文本可以在一定时间后自动过期。请你从零开始设计整个系统的架构，考虑高并发、存储成本以及可用性等方面。  

（随后请候选人从需求分析、系统概览、关键模块、扩展性等角度展开讨论）

## 3️⃣ 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| F1 | **创建粘贴**：用户提交任意长度（上限 1 MB）的纯文本，系统返回唯一的短链接（如 `https://pb.io/abcd1234`）。 |
| F2 | **读取粘贴**：通过短链接访问文本内容，支持匿名访问或登录后查看私有粘贴。 |
| F3 | **设置过期时间**：创建时可选择过期策略（如 10 分钟、1 小时、1 天、永久），系统自动在过期后删除。 |
| F4 | **粘贴可见性**：支持公开、仅限链接访问、登录用户私有三种可见性。 |
| F5 | **编辑/删除**：拥有编辑/删除权限的用户可以在粘贴未过期前修改或手动删除。 |
| F6 | **统计与搜索**：提供粘贴访问次数统计、最近 10 条公开粘贴的列表（可选搜索关键字）。 |

## 4️⃣ 非功能性需求  

| 编号 | 指标 | 估算值 | 说明 |
|------|------|--------|------|
| N1 | **日活跃用户 (DAU)** | 500,000 | 以中等流量的开发者社区为基准 |
| N2 | **峰值 QPS (读取)** | 15,000 QPS | 读取请求远高于写入，峰值约为写入 QPS 的 5 倍 |
| N3 | **平均响应时延** | ≤ 80 ms（读） / ≤ 150 ms（写） | 99.9% 请求满足该时延 |
| N4 | **可用性** | 99.95%（年可用时间 ≈ 4.38 h） | 支持多地域灾备 |
| N5 | **存储容量** | 约 200 TB（假设 500 KB 平均文本，30 天保留） | 采用冷热分层存储，30 天后归档或删除 |
| N6 | **数据持久性** | 99.9999%（单对象） | 使用多副本或纠删码保障持久性 |

## 5️⃣ 系统边界  

**本题范围内**（需要设计或考虑的）：  
- 文本的写入、读取、过期自动删除、可见性控制  
- 短链接生成与冲突解决  
- 高并发访问下的缓存、负载均衡与容错  
- 数据持久化（主存储 + 冷归档）与备份策略  
- 基础监控、日志与报警（SLA/SLI）  

**本题范围外**（不必实现/讨论的）：  
- 用户注册、登录、OAuth 第三方认证（可视为已有统一身份服务）  
- 复杂全文搜索、标签系统、评论功能  
- 高级编辑器（语法高亮、文件上传）  
- 付费套餐、流量计费、广告投放等商业化功能

## ## 第一步：理解需求与规模估算  

### 1. 功能需求（F）  
| 编号 | 关键点 | 实现要点 |
|------|--------|----------|
| F1 | 创建粘贴（≤1 MB）返回唯一短链 | 需要 **短链生成**、**文本存储**、**返回 URL** |
| F2 | 读取粘贴（匿名/登录） | **鉴权**、**可见性判断**、**高并发读取** |
| F3 | 过期时间（TTL） | **不同 TTL**（10 min、1 h、1 day、永久） → **自动删除** |
| F4 | 可见性（公开 / 链接访问 / 私有） | **权限模型**（owner、public、unlisted） |
| F5 | 编辑/删除（未过期） | **写操作**、**乐观锁**防止并发冲突 |
| F6 | 统计/搜索（访问计数、最近公开） | **计数**（increment）、**列表**（分页） |

### 2. 非功能需求（N）  
| 编号 | 指标 | 估算值 | 关键影响 |
|------|------|--------|----------|
| N1 | DAU | 500 k | 决定整体并发、机器规模 |
| N2 | 峰值读取 QPS | 15 k | 读写比例约 5:1，缓存必不可少 |
| N3 | 响应时延 | ≤80 ms（读）/≤150 ms（写） | 需要 **近端缓存、异步处理** |
| N4 | 可用性 | 99.95% | 多可用区、故障转移 |
| N5 | 存储容量 | 200 TB（30 天） | **冷热分层**、归档/删除策略 |
| N6 | 持久性 | 99.9999% 单对象 | 多副本或纠删码 |

### 3. 规模粗算（帮助后面选型）

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日新粘贴** | DAU × 平均粘贴数/人（假设 2） | 1 M 条 |
| **写入 QPS** | 1 M / 86400 ≈ 12 req/s（峰值 5×≈60 req/s） | 低 |
| **读取 QPS** | 15 k 峰值，日均约 5 k | 高 |
| **每日存储** | 1 M × 0.5 MB ≈ 500 GB | 30 天 ≈ 15 TB 原始数据，压缩/归档后 200 TB 预估 |
| **访问计数增长** | 每条粘贴平均 100 次阅读 → 100 M 计数写入/天 | 需要高效计数方案（Redis） |

> **为什么要算这些？**  
- 决定 **机器数**、**网络带宽**、**缓存容量**。  
- 防止后期出现「CPU/IO 瓶颈」而不得不大幅重构。

## ## 第二步：高层架构设计  

### 1. 最小可运行（MVP）架构  

```
[Client] → DNS → CDN (静态页面) → LB → API Server (单体) → MySQL
```

- **优点**：实现最快、验证业务流程。  
- **缺点**：单点故障、读写性能不足、无法弹性扩容。  

### 2. 逐步演进的分层架构  

```
                                   +-------------------+
                                   |   CDN (HTML/JS)   |
                                   +---------+---------+
                                             |
                                   +---------v---------+
                                   |   Global DNS LB   |
                                   +---------+---------+
                                             |
                +----------------------------+----------------------------+
                |                         LB (Layer 4)                  |
                +----------------------------+----------------------------+
                |                        |                            |
      +---------v--------+      +--------v--------+        +----------v----------+
      |  API Gateway /   |      |  API Gateway /  |        |  API Gateway /      |
      |  Edge Service    |      |  Edge Service   |        |  Edge Service       |
      +---------+--------+      +--------+--------+        +----------+----------+
                |                        |                            |
         +------v------+          +------v------+               +-----v------+
         |  Auth Service|          |  RateLimiter|               |  Logging  |
         +------+-------+          +------+-------+               +-----------+
                |                        |                            
         +------v------+          +------v------+                    
         | Paste Service| <----> | Cache Layer |  (Redis / CDN)    
         +------+-------+          +------+-------+                    
                |                        |                            
         +------v------+          +------v------+                    
         |   DB (MySQL) |  <----> |   Search/TS   | (Elasticsearch)|
         +--------------+          +--------------+                    
                |                                                
         +------v------+                                        
         |  Object Store| (S3/MinIO)   <-- 归档/备份 -->  Cold Storage
         +--------------+                                        
```

**关键层级解释**  

| 层级 | 作用 | 典型技术 | 为什么需要 |
|------|------|----------|------------|
| **CDN** | 静态前端（HTML、JS、CSS）快速就近访问 | CloudFront、Akamai | 降低入口延迟、减轻 Origin 负载 |
| **全局 DNS + LB** | 把流量分到最近的可用区 | Anycast DNS + Anycast LB | 业务容灾、跨地域负载均衡 |
| **API Gateway / Edge Service** | 统一入口、限流、鉴权、日志 | Kong、AWS API GW、Envoy | **防止恶意流量**、统一监控 |
| **Paste Service** (业务微服务) | 处理创建/读取/编辑/删除业务 | Spring Boot / Go / Node.js | **业务解耦**，方便独立扩容 |
| **Cache Layer** | 读热点缓存、计数器、短链映射 | Redis（cluster）+ CDN edge cache | **读 QPS 远高于写**，必须缓存 |
| **关系型 DB** | 元数据（短链、owner、TTL、可见性） | MySQL（Aurora）或 PostgreSQL | **事务性、强一致**，需要唯一约束 |
| **对象存储** | 文本内容（原始 1 MB） | S3、MinIO、COS | **大容量、低成本**，天然支持 **分片/多副本** |
| **搜索/统计** | 最近公开粘贴、访问计数、关键字搜索（可选） | Elasticsearch / ClickHouse | **热点列表**、**分析** |
| **归档/冷存储** | 超过保留期或永久粘贴的长期存储 | Glacier、Coldline | **降低成本**、满足 30 天保留后归档需求 |

> **不做这些层级会出现的典型问题**  
- **无缓存** → 读取 15 k QPS 直接打到 DB，导致 CPU/IO 饱和。  
- **单点 DB** → 失效即全站不可用，违背 99.95% SLA。  
- **没有限流/鉴权** → 公开 API 易被爬虫刷流，导致服务崩溃。

## ## 第三步：数据库设计  

### 1. 数据模型  

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| **paste_meta** | `id` (bigint auto‑increment) | `short_key` (varchar, unique), `owner_id`, `visibility` (enum), `ttl_seconds`, `expire_at` (datetime), `created_at`, `updated_at`, `is_deleted` (bool) | 保存 **元信息**，短链映射到对象存储 key |
| **paste_content** | `paste_id` (fk → paste_meta.id) | `content_url` (varchar), `content_md5` | 实际文本存放在对象存储，**content_url** 形如 `s3://bucket/<uuid>` |
| **paste_stats** | `paste_id` (pk) | `view_count` (bigint) | 访问计数，使用 **Redis** 同步落库（异步批写） |
| **user** (如已有) | `user_id` | `username`, `email` | 只在需要私有粘贴时使用 |

> **为什么拆分 `paste_meta` 与 `paste_content`？**  
- **元数据** 体积小、频繁读写，适合放在 **关系型 DB**（索引快）。  
- **内容** 最大 1 MB，频繁写入会导致 **行锁** 与 **磁盘碎片**，放对象存储可利用 **分块、并行上传**，且成本更低。  

### 2. 短链唯一性 & 索引  

- **唯一索引**：`UNIQUE(short_key)` 防止冲突。  
- **TTL 索引**：`INDEX(expire_at)` 用于定时扫描过期粘贴。  
- **可见性索引**：`INDEX(visibility, created_at)` 支持「最近公开」查询。  

### 3. 事务与并发  

- **创建粘贴**：  
  1. 生成 `short_key`（后面章节讲）。  
  2. 在同一事务中 **插入 paste_meta** → **获取 id**。  
  3. 将文本上传对象存储，返回 `content_url`。  
  4. **更新 paste_content**（可以异步，但为保持强一致，推荐在事务外的 **两阶段提交**：先写 DB，再确认对象存储成功）。  

- **编辑**：使用 **乐观锁**（`updated_at`）防止并发覆盖。  

### 4. 数据持久化 & 备份  

| 组件 | 持久化方式 | 副本/纠删码 | 备份频率 |
|------|------------|------------|----------|
| MySQL | InnoDB binlog + 自动备份 | 3‑副本（主‑从‑从）或 **X‑Cluster** | 每日全备 + 每小时增量 |
| Redis | RDB + AOF | 主‑从（至少 2 副本） | 每 5 min AOF 同步 |
| 对象存储 | 多 AZ 多副本 | S3 标准 3‑副本，或 **纠删码**（EC） | 持续持久化，无需额外备份 |

## ## 第四步：核心 API 设计  

下面给出 **RESTful** 风格的接口（实际可用 gRPC、GraphQL，视团队而定），并标注 **鉴权、缓存、异常**。

| 方法 | 路径 | 功能 | 请求体 | 响应 | 关键实现点 |
|------|------|------|--------|------|------------|
| `POST` | `/api/v1/paste` | 创建粘贴 | `{ "content": "string", "ttl": "10m/1h/1d/permanent", "visibility": "public|unlisted|private" }` | `{ "short_url": "https://pb.io/abcd1234", "expire_at": "2026-06-01T12:00:00Z" }` | - **短链生成**（后）<br>- **写入 DB**（事务）<br>- **对象存储上传**（异步）<br>- **返回前写入缓存**（Redis） |
| `GET` | `/api/v1/paste/{short_key}` | 读取粘贴 | — | `{ "content": "string", "owner": "uid", "visibility": "...", "created_at": "...", "expire_at": "...", "view_count": 123 }` | - **缓存优先**（Redis → DB → 对象存储）<br>- **权限检查**（public/unlisted/private）<br>- **计数自增**（Redis INCR） |
| `PATCH` | `/api/v1/paste/{short_key}` | 编辑粘贴（仅 owner） | `{ "content": "...", "ttl": "...", "visibility": "..." }` | `{ "msg": "updated" }` | - **乐观锁** (`If-Unmodified-Since`) <br>- **重新上传对象**<br>- **更新缓存** |
| `DELETE` | `/api/v1/paste/{short_key}` | 删除粘贴（owner 或管理员） | — | `{ "msg": "deleted" }` | - **软删** (`is_deleted=1`) + **异步硬删**<br>- **清理缓存** |
| `GET` | `/api/v1/paste/public/latest?limit=10` | 获取最近公开粘贴列表 | — | `[{ "short_key": "...", "title": "...", "created_at": "..."}]` | - **查询索引** (`visibility=public`)<br>- **缓存热点列表**（Redis sorted set） |
| `GET` | `/api/v1/paste/{short_key}/stats` | 访问计数（可选） | — | `{ "view_count": 12345 }` | - **从 Redis 读取**，若缺失则回源 DB |

> **安全与防刷**  
- **RateLimiter**（基于 IP + UserID）在 API Gateway 层实现，阈值 100 req/s/IP。  
- **Captcha**（可选）针对异常创建速率。  
- **短链长度**足够防止暴力枚举（6 位 Base62 → 56 B 组合，约 2.2B），再配合 **IP 黑名单**。

## ## 第五步：详细组件设计  

### 1. 短链接生成 & 冲突处理  

| 方案 | 生成方式 | 长度 | 冲突概率 | 适用场景 | 备注 |
|------|----------|------|----------|----------|------|
| **自增 ID + Base62** | DB 自增 `id` → 转成 Base62 | 6‑8 位（取 6 位时约 2.2B） | **理论无冲突**（唯一） | 简单、可预测 | 需要 **分区** 防热点（后文） |
| **随机 6 位 Base62** | `crypto/rand` 生成 6 位 | 6 位 | **碰撞概率**≈ 1‑% 在 10M 条后，需要重试 | 高并发、分布式 | 需要 **冲突检测**（唯一索引） |
| **Snowflake + Base62** | 64‑bit 雪花 ID → Base62 | 8‑10 位 | **全局唯一** | 多地域、可排序 | 需要时钟同步 |

**推荐方案**：**Snowflake + Base62**  
- **全局唯一**（机器 ID + 时间戳 + 序列），无需中心化自增，避免热点写入单点 DB。  
- **可排序**（时间递增），有助于「最近公开」查询。  

**冲突防止**：  
- 对于 **随机方案**，在生成后立即 **INSERT** 带唯一约束的 `short_key`，若冲突则 **重试**（最多 3 次）。  
- 对于 **Snowflake**，冲突几乎不存在，唯一约束仍保留防止意外错误。

**热点分区**：  
- **写入分布**：Snowflake 自带机器 ID，写入分散在不同 DB 实例或分区。  
- **读取分布**：短链前缀可映射到 **分片路由**（如使用 **Sharding‑Sphere**），把相同前缀的请求路由到对应分片，减轻热点。  

### 2. 缓存层设计  

| 缓存目标 | 数据 | 缓存策略 | 失效方式 | 备注 |
|----------|------|----------|----------|------|
| **短链 → 内容** | `content`（<=1 MB） | **Cache‑Aside**（先查 Redis，未命中再读 DB+OSS） | **TTL = min(原粘贴TTL, 30 min)**，过期后自动失效 | 读取热点，直接返回，避免 OSS 频繁访问 |
| **粘贴元信息** | `paste_meta`（owner、visibility、expire_at） | **Read‑Through**（写时同步） | **TTL = 原 TTL** | 与业务逻辑紧耦合，确保一致性 |
| **访问计数** | `view_count` | **Redis INCR**（写‑only），**异步批写**至 DB（每分钟） | **永不过期**（除非粘贴删除） | 高并发计数，避免 DB 争抢 |
| **最近公开列表** | `sorted set` (score=created_at, member=short_key) | **Write‑Behind**（写入 DB 后同步到 Redis） | **TTL = 5 min**（每次查询刷新） | 用于首页热点展示 |
| **短链冲突检查** | `short_key` | **写时唯一索引**（DB）+ **Redis Set**（快速冲突检测） | **永久**（除非删除） | 只在创建时使用 |

**为什么采用 Cache‑Aside + Read‑Through？**  
- **Cache‑Aside** 让业务层决定何时刷新缓存，避免缓存写入导致 **写放大**（写一次多写一次）。  
- **Read‑Through**（在读取时自动加载）适用于 **元数据**，因为它们经常被读取但很少变更。  

### 3. 过期粘贴自动删除  

#### 方案比较  

| 方案 | 实现方式 | 适用 TTL 场景 | 优点 | 缺点 |
|------|----------|---------------|------|------|
| **DB TTL + 自动过期** | 为 `expire_at` 建立 **定时任务**（MySQL Event）| 所有 | 简单、事务内完成 | 大量短时间 TTL（10 min）时任务频繁 |
| **Redis 延迟队列** | `ZADD` (score = expire_ts) → **消费** (Redis Streams) | 10 min、1 h | 高精度、低延迟 | 需要额外消费者进程 |
| **消息队列 + 延迟消息** | Kafka/ RabbitMQ **Delay Topic**（如 RabbitMQ TTL）| 1 h、1 d | 分布式、可靠 | 不适合极短 TTL（10 min） |
| **Time Wheel**（时间轮） | 本地或分布式时间轮（如 **HashedWheelTimer**）| 10 min‑1 d | O(1) 删除，低资源 | 实现复杂，需要持久化防丢失 |

**推荐组合**  

1. **短期 TTL (≤1 h)**：使用 **Redis 延迟队列**（`ZSET` + `ZRANGEBYSCORE`）  
   - 创建时 `ZADD expiry_set expire_ts short_key`  
   - **消费者** 每秒轮询 `ZRANGEBYSCORE`，取出已到期的 key，执行 **软删**（DB `is_deleted=1`）并从 **Cache**、**Object Store** 中删除。  

2. **长期 TTL (≥1 h) / 永久**：使用 **DB 定时任务**（如 **MySQL Event** 或 **Cron**）每天一次批量清理 `expire_at < now()`。  

> **为什么分两套？**  
- **Redis** 能在 **秒级** 触发，适合 10 min、1 h 这种对及时删除有需求的场景。  
- **DB 定时任务** 只需要每天跑一次，成本低，适合大量「几天后」的过期数据。  

#### 删除流程（示例）  

```
[Create Paste] → DB INSERT (expire_at) → ZADD expiry_set (if ttl ≤ 1h) → Return short_url
[Expiry Worker] (Redis) → ZRANGEBYSCORE now → for each key:
    1. DB UPDATE paste_meta SET is_deleted=1 WHERE short_key=?
    2. DEL Redis cache (short_key)
    3. Async delete object from OSS (S3 DeleteObject)
```

### 4. 读写分离 & 负载均衡  

- **读写分离**：主库处理写入，多个只读副本提供查询。  
  - **复制方式**：MySQL **semi‑sync**（保证至少 1 个副本同步）+ **GTID** 自动故障转移（使用 **Orchestrator** 或 **MHA**）。  
  - **读路由**：API 层使用 **读写分离库**（如 **MySQL Proxy**、**HAProxy**）或在代码里根据操作选择 `master`/`replica`。  

- **负载均衡**：  
  - **L4**（TCP）或 **L7**（HTTP）层使用 **NGINX/Envoy**，配合 **健康检查** 自动剔除故障节点。  
  - **全局负载**：多地域使用 **Anycast DNS + Geo‑DNS**，把用户请求路由到最近的 Region。  

### 5. 监控、日志、报警（SLA/SLI）  

| 维度 | 指标 | 收集方式 | 报警阈值 |
|------|------|----------|----------|
| **系统** | CPU、内存、磁盘 I/O | Node Exporter + Prometheus | >80% 持续 5 min |
| **服务** | API latency (p95)、错误率 (5xx) | OpenTelemetry + Prometheus | latency > 120 ms、error_rate > 1% |
| **业务** | QPS、短链创建成功率、删除成功率、缓存命中率 | Prometheus (custom metrics) | cache_hit < 70% |
| **安全** | RateLimiter 触发次数、异常登录 | Falco / WAF | rate_limit_exceed > 1000/min |
| **存储** | Redis memory、MySQL replica lag、对象存储可用性 | Redis Exporter、MySQL Exporter | replica_lag > 5 s |

> **日志**：统一使用 **JSON** 格式，写入 **ELK**（或 **EFK**) 集群，配合 **Kibana** 做业务日志检索（创建、读取、编辑）。

## ## 第六步：扩展性与高可用设计  

### 1. 横向扩展（Scale‑Out）  

| 组件 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **API/业务服务** | **容器化 + K8s** 自动水平 pod 扩容（HPA）| CPU、QPS |
| **Redis** | **Cluster**（分片）或 **Sentinel** 多实例 | 请求数、内存 |
| **MySQL** | **读写分离** + **sharding**（按短链前缀）| 写入量、热点 |
| **对象存储** | **多 AZ** 自动扩容（S3 本身） | 存储容量、IOPS |
| **缓存 CDN** | **边缘节点** 全球分布 | 静态资源访问 latency |

### 2. 容灾与多地域部署  

- **主备 Region**：  
  - **Active‑Active**：两套完整系统（API、DB、Redis），使用 **全局负载均衡**（如 Cloudflare Load Balancer）实现流量自动切换。  
  - **数据同步**：MySQL **GTID** + **Binlog** 跨 Region 同步；Redis **Active‑Active**（Redis Enterprise）或 **CRDT**。  
  - **对象存储跨 Region Replication**（S3 Cross‑Region Replication）。  

- **灾难恢复 (DR)**：  
  - **备份**：每日全量快照 + 按分钟增量（对象存储、MySQL）。  
  - **故障转移**：自动检测 Region 健康，触发 **Route53** 切换或 **DNS TTL** 短化（30 s）。  

### 3. 数据分区（Sharding）  

- **依据**：`short_key` 前两位（Base62） → 64 个分片。  
- **好处**：  
  - **写热点** 分散（不同机器负责不同前缀）。  
  - **读热点** 也分散，**缓存** 能更均匀使用。  
- **实现**：使用 **Sharding‑Sphere** 或自行在业务层路由到对应 DB 实例。  

### 4. 运营指标与容量规划  

| 项目 | 计算公式 | 结果 | 伸缩策略 |
|------|----------|------|----------|
| **Redis 容量** | (热点粘贴数 × avg size) ≈ 200k × 0.5 MB ≈ 100 GB | 采用 3‑节点 32 GB 版 + LRU | 当内存 > 70% 自动扩容 |
| **MySQL 存储** | 30 days × 500 KB × 1M/day ≈ 15 TB | 采用 3‑副本 2 TB/节点 | 自动扩容磁盘 |
| **对象存储** | 200 TB (30 days) | S3 标准存储 | 按需付费，自动弹性 |

## ## 第七步：常见面试追问与回答  

### 1️⃣ 短链接冲突与分区  

**追问**：如果采用 6 位 Base62 编码，如何保证全局唯一且避免热点分区？  

**回答要点**  

1. **唯一性**  
   - 6 位 Base62 ⇒ 62⁶ ≈ 56.8 B 可用组合。  
   - 采用 **Snowflake**（64 bit）生成唯一 ID，随后 **Base62** 编码取低 6‑8 位。  
   - **唯一约束**在 DB 中仍保留，防止极端错误。  

2. **冲突概率**  
   - 只要不截断超过 6 位，冲突几乎为 0。  
   - 若硬要 6 位，可 **在生成前检测 Redis Set**（已存在的短链集合），冲突概率在 10M 条数据时约 0.2%，可通过 **重试**解决。  

3. **热点分区**  
   - **写热点**：Snowflake 的机器 ID 把不同机器的 ID 隔离，写入 DB 时分散到 **不同分片**（如基于短链前两位路由）。  
   - **读热点**：使用 **Redis** 与 **CDN** 缓存热点短链，热点粘贴在缓存中占用比例高，降低 DB 访问。  
   - **分片策略**：`shard_id = hash(short_key) % N`，N 为分库/分表数，确保 **负载均匀**。  

### 2️⃣ 过期粘贴的清理机制  

**追问**：在 10 分钟、1 小时、1 天等不同 TTL 场景下，你会选用何种技术（TTL 数据库、定时任务、消息队列）来实现高效的自动删除？  

**回答要点**  

| TTL | 推荐技术 | 实现细节 |
|-----|----------|----------|
| **≤ 1 h** (10 min、1 h) | **Redis 延迟队列**（ZSET） | `ZADD expiry_set expire_ts short_key`，消费者每秒轮询 `ZRANGEBYSCORE now`，批量软删并异步清除对象存储。 |
| **> 1 h** (1 d、永久) | **MySQL 定时任务 / Daily Cron** | 每天 00:00 执行 `DELETE FROM paste_meta WHERE expire_at < now() AND is_deleted=0`，配合 **分区表**（按天分区）提升效率。 |
| **永久** | **不加入任何删除机制**，只在用户手动删除时软删。 | 通过后台 **归档**（对象存储转至 Glacier）降低成本。 |

**为什么不只用单一方案？**  
- **Redis** 适合 **秒级** 精准删除，但如果每秒需要处理上万条会占用大量内存。  
- **DB 定时任务** 适合 **批量** 清理，成本低但不够实时。两者结合可兼顾 **实时性** 与 **资源利用率**。  

### 3️⃣ 读写分离与缓存策略  

**追问**：针对读取 QPS 高的情况，你会如何设计缓存层（热点缓存、冷热缓存、Cache‑Aside vs Write‑Through）以及缓存失效策略？  

**回答要点**  

1. **缓存层级**  
   - **一级缓存**：**CDN Edge**（缓存短链对应的 HTML 页面） → 解决 **全球** 延迟。  
   - **二级缓存**：**Redis**（键：`paste:{short_key}`）存放 **完整文本**（≤1 MB）与 **元信息**。  
   - **三级缓存**：**本地进程 LRU**（如 Go `bigcache`）存放 **热点**（Top 1% 访问）进一步降低网络 RTT。  

2. **缓存策略**  
   - **Read‑Through (Cache‑Aside)** 对 **内容**：读取时先查询 Redis，若 miss 再从 DB + OSS 拉取并写回 Redis。  
   - **Write‑Through** 对 **元数据**（visibility、expire_at）：写入 DB 时同步写入 Redis，确保读时不产生不一致。  

3. **失效 / 更新**  
   - **TTL**：缓存的 TTL 与粘贴的 TTL 同步（`min(pasteTTL, 30m)`），防止过期后仍被缓存。  
   - **主动失效**：编辑、删除、过期时立即 **DEL** Redis key，确保后续读取命中最新状态。  
   - **LRU/Eviction**：Redis 设置 `maxmemory-policy allkeys-lru`，热点自然保留。  

4. **冷热缓存**  
   - **热点**（访问次数 > 1000 次/天）自动提升 **缓存 TTL**（如 1 h → 6 h），降低后端压力。  
   - **冷数据**（访问稀疏）保持短 TTL（5 min），让内存空间腾出给热点。  

5. **为什么不直接使用 Write‑Through 对内容？**  
   - 内容最大 1 MB，写一次会导致 **网络+磁盘双写**，写放大严重。读多写少的场景下 **Cache‑Aside** 更经济。

## ## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 思考过程 |
|------|----------|
| **短链生成方式** | 必须在 **全局唯一**、**低冲突**、**可水平扩展**、**易于路由分片** 四者之间取舍。最初倾向于 **自增 ID + Base62**，但在多 Region、写入热点情况下会导致单点瓶颈。最终选择 **Snowflake + Base62**，因为它天然提供唯一性且分散写入，同时还能保留时间顺序帮助最近查询。|
| **过期删除机制** | 需要兼顾 **实时性**（10 min）与 **资源消耗**（每天成千上万条）。单一方案（如全用 Redis）会导致内存压力；单用 DB 定时任务则延迟太高。通过 **TTL 分层**（Redis 延迟队列 + DB 批任务）实现了成本与时效的平衡。|

### 2️⃣ 新手最容易犯的错误（≥2）  

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **只设计单机 DB**，忽视 **读写分离** 与 **缓存** | 读取 QPS 达 15 k 时，单库瞬间崩溃，响应时间远超 SLA。 | 采用 **Redis** 缓存热点，**主从复制** + **只读副本** 进行读写分离。 |
| **短链冲突直接使用随机 6 位**，不做唯一校验 | 随机冲突概率随数据量指数增长，导致创建失败率升高，用户体验差。 | 使用 **全局唯一 ID**（Snowflake）或 **冲突检测 + 重试**，并在 DB 上加唯一索引。 |
| **过期删除只靠手动脚本**，不考虑不同 TTL 的实时性 | 短期 TTL 粘贴会在过期前仍可被访问，违反业务需求。 | 按 TTL 分层使用 **Redis 延迟队列**（秒级）和 **DB 定时任务**（批量）。 |
| **把所有业务放在单个微服务**，不考虑 **模块化** | 难以单独扩容编辑、统计、删除等功能，导致资源浪费。 | 将 **Paste Service**、**Stats Service**、**Cleanup Service** 拆分，使用 **消息队列** 解耦。 |
