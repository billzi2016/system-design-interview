## 智能家居控制系统 - 系统设计面试题

### 1. 题目背景  
智能家居控制系统（Smart Home Control System）为用户提供统一的移动端/语音端入口，能够对家中的灯光、空调、门锁、安防摄像头、窗帘等 IoT 设备进行实时监控与远程控制。目标是实现低延迟的交互体验并保障系统的高可用与安全。

### 2. 面试场景设定  
> **面试官**：  
> “假设我们要为一家新成立的智能家居公司设计一套可扩展的云端控制平台，能够支撑数百万用户同时管理他们的家庭设备。请你从零开始设计系统的整体架构，重点说明数据流、关键组件以及如何满足性能和可靠性要求。”

### 3. 功能性需求（核心功能）  
1. **设备注册与管理**  
   - 用户通过手机 App 扫描设备二维码或手动输入序列号完成设备绑定。  
   - 支持设备信息的增删改查（名称、房间、类型、状态等）。

2. **实时状态查询**  
   - 用户打开 App 即可看到所有已绑定设备的当前状态（开/关、温度、湿度等），延迟 ≤ 200 ms。

3. **远程控制指令**  
   - 用户对任意设备发出开/关、调节参数等指令，系统需在 300 ms 内下发至设备并返回成功/失败反馈。

4. **场景联动与自动化**  
   - 支持用户自定义“场景”（如“回家模式”）或基于时间/传感器触发的自动化规则，系统负责批量下发指令。

5. **多端同步**  
   - 同时在手机 App、Web 控制台、语音助理（如 Alexa、Google Assistant）上操作，状态保持强一致。

6. **安全与权限控制**  
   - 支持基于 OAuth2 的用户身份认证、设备级权限（主用户/子用户）以及指令加密传输。

### 4. 非功能性需求（关键指标）  
| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 1,500,000 | 目标在系统上线第一年实现 150 万日活 |
| **QPS（指令请求峰值）** | 25,000 req/s | 高峰期（上午 8‑10、晚上 7‑10）每秒约 2.5 万条控制指令或查询请求 |
| **单次指令/查询延迟** | ≤ 300 ms（99th percentile） | 包括网络、服务处理、下发至设备的全链路时延 |
| **系统可用性** | 99.99%（每月累计宕机 ≤ 4.38 小时） | 业务对连续性要求高，尤其是安全/门锁等关键设备 |
| **存储容量** | 约 2 TB（设备元数据、用户日志、规则引擎数据） | 按 30 天日志保留计算，平均每用户 1 KB/天的设备状态日志 |

### 5. 系统边界  
**本题范围（需要设计）**  
- 云端 API 网关、身份认证、设备管理服务、状态缓存、指令路由、场景/规则引擎、监控告警。  
- 与设备的通信协议（MQTT/CoAP）在云端的接入层实现。  
- 数据持久化（关系型 DB、时序库）与缓存（Redis）。  
- 多端（移动、Web、语音）统一的 API 设计与鉴权。  

**不在本题范围（可忽略）**  
- 具体的硬件设备设计、芯片固件实现。  
- 本地网关（如 Zigbee、Thread）与云端的桥接细节。  
- 第三方语音平台的集成细节（仅要求提供统一 API）。  
- 详细的 UI/UX 交互设计与前端实现。  

### 6. 提示与追问  
1. **“如果在高峰期 QPS 突然翻倍，系统如何弹性伸缩？”**  
2. **“请说明你会选用哪种消息队列/协议来实现指令下发，如何保证指令的幂等性和可靠投递？”**  
3. **“在设备离线后，用户仍然可以查看最近状态，系统如何设计状态缓存和持久化？”**

## 解题思路总览
1. **先把需求拆成功能块**：设备注册、状态查询、指令下发、场景联动、多端同步、安全等。  
2. **估算流量 & 存储**：把业务指标（DAU、QPS、延迟）转成具体的并发数、带宽、存储大小。  
3. **从最小可用系统 (MVP) 入手**：只用单体服务 + 单库 + 简单缓存，先保证功能完整、延迟可接受。  
4. **逐层演进**：在 MVP 基础上，引入 **API 网关、服务拆分、缓存层、消息队列、分布式数据库、容灾**，每一步都解释为什么要拆、拆了有什么好处。  
5. **安全 & 可观测**：在每层都加入鉴权、加密、监控、日志、告警。  
6. **面试常见追问**：准备伸缩、幂等、离线状态等细节答案。

## 第一步：理解需求与规模估算  

### 1. 功能需求快速列表  

| 功能 | 关键点 | 业务指标 |
|------|--------|----------|
| 设备注册/管理 | 扫码/手输、绑定/解绑、元数据 CRUD | 需持久化、一次性写入 |
| 实时状态查询 | 读取最新状态、延迟 ≤200 ms | 读多写少 |
| 远程控制指令 | 开/关、调参、300 ms 响应 | 高并发写 |
| 场景/自动化 | 批量下发、多触发源 | 业务复杂度提升 |
| 多端同步 | 强一致、实时推送 | 需要 Pub/Sub |
| 安全 | OAuth2、TLS、细粒度权限 | 全链路加密 |

### 2. 规模估算  

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **活跃用户** | DAU = 1.5 M | 1,500,000 |
| **并发在线用户** | 假设 10% 同时在线 ≈ 150k | 150,000 |
| **峰值 QPS** | 25,000 req/s（题目给出） | 25k |
| **设备数** | 每用户平均 8 台设备 → 12 M 设备 | 12,000,000 |
| **每设备状态大小** | 约 200 B（属性+时间戳） | 2.4 GB 总状态 |
| **日志存储** | 1 KB/用户/天 × 30 天 ≈ 45 GB/天 → 1.35 TB/月 | 约 2 TB（题目给出） |
| **网络带宽** | 25k QPS × 200 B ≈ 4 GB/s（上行+下行）≈ 32 Gbps峰值 | 需负载均衡 + CDN |

> **为什么要做这些估算？**  
> - 能帮助判断 **单机能否支撑**、**需要多少机器**、**缓存大小**。  
> - 没有估算直接设计，很容易出现 **资源瓶颈**（如 DB 连接耗尽）或 **成本失控**。

## 第二步：高层架构设计  

### 1. MVP（最小可用系统）  

```
+-------------------+      +-------------------+
|   Mobile / Web   | ---> |   API Server      |
+-------------------+      +-------------------+
                               |
                               v
                       +---------------+
                       |   MySQL       |
                       +---------------+
```

- **API Server**：单体 Spring Boot / Django，负责全部业务。  
- **MySQL**：存放用户、设备、规则等结构化数据。  
- **优点**：实现最快、代码最少，能验证业务。  
- **缺点**：单点故障、水平扩展困难、无法满足 300 ms 全链路延迟。

### 2. 演进到分布式高可用架构  

下面的 **完整系统图** 用 **Mermaid** 表示，便于面试时直接画在白板上：

```mermaid
graph TD
    subgraph Edge
        Client[Mobile/Web/Voice] -->|HTTPS| APIGW[API Gateway]
    end

    subgraph Auth
        APIGW --> AuthS[Auth Service (OAuth2)]
    end

    subgraph Core
        APIGW -->|REST| UserS[User Service]
        APIGW -->|REST| DeviceS[Device Service]
        APIGW -->|REST| RuleS[Rule Engine Service]
        APIGW -->|REST| CmdS[Command Service]

        DeviceS -->|Cache| Redis[Redis Cache]
        CmdS -->|Queue| MQTTB[MQTT Broker]
        CmdS -->|Queue| Kafka[Kafka Topic]
        RuleS -->|Pub/Sub| MQTTB
        RuleS -->|Pub/Sub| Kafka
    end

    subgraph Persistence
        UserS -->|RDBMS| MySQL[MySQL Cluster]
        DeviceS -->|RDBMS| MySQL
        RuleS -->|RDBMS| MySQL
        CmdS -->|TSDB| InfluxDB[InfluxDB]
    end

    subgraph Monitoring
        Prom[Prometheus] --> Grafana[Grafana]
        DeviceS -->|Metrics| Prom
        CmdS -->|Metrics| Prom
        MQTTB -->|Metrics| Prom
    end

    subgraph Offline
        DeviceS -->|Event| EventBus[Kafka] 
        EventBus -->|Replay| CmdS
    end
```

#### 关键组件解释  

| 组件 | 作用 | 为什么要单独拆出来 |
|------|------|----------------------|
| **API Gateway** | 统一入口、限流、路由、TLS 终止、灰度发布 | 防止每个服务都暴露在公网，集中治理安全 & 监控 |
| **Auth Service (OAuth2)** | 登录、Token 发放、校验 | 业务解耦、支持第三方登录、统一权限模型 |
| **User Service** | 用户信息、子账号、权限 | 将用户相关业务独立，后续可水平扩展 |
| **Device Service** | 设备注册、元数据、状态查询（读缓存） | 设备管理是高频读，需要缓存层 |
| **Command Service** | 接收控制指令、幂等检查、下发至 MQTT/ Kafka | 指令路径独立，便于重试、审计 |
| **Rule Engine Service** | 场景/自动化规则、触发器、批量指令 | 业务复杂度高，单独服务易水平扩容 |
| **Redis** | 热状态缓存、热点设备快速读取 | 读多写少，缓存降低 DB 压力，满足 ≤200 ms |
| **MQTT Broker** | 轻量、低延迟的设备推/拉协议 | IoT 设备普遍支持 MQTT，保持长连接、双向通讯 |
| **Kafka** | 可靠的消息队列，用于指令、事件、审计日志 | 支持持久化、幂等、重放，适合离线设备 |
| **InfluxDB / TSDB** | 存储时间序列状态、历史曲线 | 设备状态是时序数据，TSDB 查询更快 |
| **MySQL Cluster** | 主数据（用户、设备、规则） | 关系型 DB 提供事务、一致性 |
| **Prometheus + Grafana** | 监控、告警、容量预警 | 及时发现异常，满足 99.99% SLA |

#### 数据流概览（指令下发）  

1. **用户** → API Gateway → **Command Service**（REST）  
2. **Command Service** 进行 **幂等性校验**（基于指令 ID） → 写入 **Kafka**（持久化）  
3. 同时把指令推送至 **MQTT Broker**（实时）  
4. **设备**（保持 MQTT 长连）收到指令 → 执行 → 回执通过 MQTT → **Command Service** → 更新 **Redis** + **InfluxDB**  
5. **Command Service** 返回统一的成功/失败响应给用户（≤300 ms）  

> **如果不使用消息队列**，指令只走 MQTT，出现网络抖动或设备离线时指令会丢失，无法保证 **可靠投递**。  
> **如果不做幂等性**，用户重复点击会导致设备状态不确定，尤其是“开灯”这类幂等指令也会产生重复日志。

## 第三步：数据库设计  

### 1. 关系型库（MySQL）结构  

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| **users** | `user_id` (PK) | `email`, `password_hash`, `status` | 主账号信息 |
| **user_devices** | `(user_id, device_id)` (PK) | `role` (owner/guest), `alias`, `room` | 多对多，记录绑定关系与权限 |
| **devices** | `device_id` (PK) | `serial_number`, `type`, `model`, `firmware_version`, `registered_at` | 设备静态信息 |
| **device_status_latest** | `device_id` (PK) | `status_json`, `updated_at` | **热点读**，与 Redis 同步 |
| **rules** | `rule_id` (PK) | `user_id`, `name`, `trigger_type`, `trigger_cfg`, `action_cfg`, `enabled` | 场景/自动化规则 |
| **rule_actions** | `action_id` (PK) | `rule_id`, `device_id`, `command_json`, `order` | 规则对应的多条指令 |
| **audit_logs** | `log_id` (PK) | `user_id`, `device_id`, `action`, `timestamp`, `detail` | 业务审计、合规需求 |

#### 设计要点  

- **分库分表**：设备数 12 M，单库可以承受；若业务增长到 100 M，可按 **region** 或 **user_id hash** 分库。  
- **只读副本**：`device_status_latest` 读取频繁，部署只读副本供查询，主库只负责写入。  
- **外键**：为保证数据一致性，可在业务层实现，生产环境可关闭外键提升写入性能。  

### 2. 缓存层（Redis）  

| Key | 类型 | 说明 |
|-----|------|------|
| `device:status:{device_id}` | Hash | `{online:1, temperature:23, ...}` 实时状态，TTL 5 min（防止僵尸键） |
| `user:devices:{user_id}` | Set | 绑定的设备 ID 列表，快速查询用户拥有设备 |
| `command:idempotent:{cmd_id}` | String | `processed` 标记，用于幂等检查，TTL 24 h |
| `rule:trigger:{rule_id}` | Sorted Set | 触发时间戳 → rule_id，用于时间驱动的自动化 |

> **为什么要把热点状态放到 Redis**  
> - **读写分离**：设备状态每秒更新数万次，直接写 MySQL 会产生大量锁竞争。  
> - **低延迟**：Redis 读写毫秒级，满足 **≤200 ms** 的查询要求。  
> - **失效策略**：即使 Redis 故障，仍可从 MySQL/TSDB 回拉最近一次状态，保证可用性。  

### 3. 时序库（InfluxDB / TimescaleDB）  

- **measurement**：`device_status`  
- **tags**：`device_id`, `type`  
- **fields**：`temperature`, `humidity`, `power_state` 等  
- **timestamp**：UTC 毫秒  

用于 **历史趋势**（如 30 天温度曲线）以及 **离线设备最近状态** 的回放。

## 第四步：核心 API 设计  

> **统一的 RESTful + WebSocket**，其中实时推送走 **WebSocket**（或 Server‑Sent Events），指令请求走 **HTTPS POST**。  

### 1. 鉴权机制  

| 步骤 | 说明 |
|------|------|
| **登录** | `POST /oauth/token` → 返回 **JWT**（access_token） + **Refresh Token** |
| **校验** | API Gateway 通过 **OAuth2 资源服务器** 中间件校验 JWT，放入 `user_id` 上下文 |
| **细粒度权限** | 在业务服务中检查 `user_devices` 表的 `role`，决定是否能操作该设备 |

### 2. 关键 API 列表  

| 方法 | 路径 | 功能 | 请求体示例 | 响应示例 |
|------|------|------|-----------|----------|
| `POST` | `/devices/register` | 绑定新设备（二维码/序列号） | `{ "serial": "SN12345", "alias": "客厅灯", "room": "客厅" }` | `{ "device_id": "dev-abc123", "status": "registered" }` |
| `GET` | `/devices` | 查询当前用户所有设备 + 最新状态 | - | `[{ "device_id": "...", "alias":"客厅灯", "status": {...}}]` |
| `GET` | `/devices/{device_id}/status` | 单设备状态（读缓存） | - | `{ "online": true, "temperature": 22 }` |
| `POST` | `/devices/{device_id}/command` | 发送控制指令 | `{ "cmd_id":"uuid", "command": {"power":"on"} }` | `{ "result":"accepted", "cmd_id":"uuid" }` |
| `POST` | `/rules` | 创建场景/自动化规则 | `{ "name":"回家模式","trigger":{"type":"time","cron":"0 18 * * *"},"actions":[{"device_id":"dev-1","command":{"power":"on"}}] }` | `{ "rule_id":"rule-xyz" }` |
| `GET` | `/rules` | 列出用户规则 | - | `[{ "rule_id":"...", "name":"回家模式", "enabled":true }]` |
| `GET` | `/ws/updates` | **WebSocket**：实时推送设备状态变化、指令执行结果 | - | `{"type":"status_update","device_id":"dev-1","payload":{...}}` |

#### 幂等性设计（关键）  

- **客户端生成唯一 `cmd_id`（UUID）**，随指令一起发送。  
- **Command Service** 在 Redis 中检查 `command:idempotent:{cmd_id}` 是否已存在，若已存在直接返回之前的结果。  
- 这样即使用户网络抖动导致 **重发**，也不会产生重复操作。  

### 3. 错误码约定（参考 HTTP + 自定义业务码）  

| HTTP | 业务码 | 含义 |
|------|--------|------|
| 200 | 0 | 成功 |
| 400 | 1001 | 参数错误 |
| 401 | 1002 | 未认证 / Token 失效 |
| 403 | 1003 | 权限不足 |
| 404 | 1004 | 设备不存在 |
| 409 | 1005 | 幂等指令已处理 |
| 429 | 1006 | 触发限流 |
| 500 | 2001 | 服务内部错误 |

## 第五步：详细组件设计  

下面把每个核心组件的内部实现细节展开，帮助你在面试时能**深入解释**每个决定背后的技术点。  

### 1. API Gateway  

- **实现选型**：Kong、AWS API Gateway、或 Nginx + Lua。  
- **功能**：  
  - **TLS 终止**（HTTPS → HTTP），减轻后端服务负担。  
  - **统一限流**：基于 `user_id` 的令牌桶（Burst = 10，Rate = 5 req/s）防止恶意刷接口。  
  - **路由**：把 `/devices/**`、`/rules/**`、`/commands/**` 分别路由到对应微服务。  
  - **日志**：记录请求 ID、来源 IP、响应时间，供后端追踪。  
- **为什么要单独放**：若直接把流量暴露到每个服务，**安全、限流、监控**都要在每个服务里重复实现，运维成本翻倍。  

### 2. Auth Service  

- **协议**：OAuth2.0 Authorization Code + PKCE（移动端安全）。  
- **实现**：Keycloak / Auth0（SaaS）或自建 Spring Security OAuth。  
- **Token**：JWT，**签名**使用 RSA256，**claims** 包含 `sub=user_id`, `exp`, `iat`。  
- **刷新机制**：Refresh Token 存在 **Redis**，设定 **7 天** 有效期，失效后强制重新登录。  

### 3. Device Service  

#### 3.1 注册流程  

1. **App** 读取二维码 → 发送 `serial_number`、`user_id` 至 `/devices/register`。  
2. **Device Service** 校验 **serial_number** 是否在 **Device Registry**（预先导入的设备列表）中。  
3. 写入 **devices** 表、**user_devices** 关联表。  
4. **生成 MQTT 客户端凭证**（clientId = `device_id`, password = 随机 token）并写入 **Redis**，返回给设备（或通过云端下发）。  

#### 3.2 状态更新  

- **设备** 持久化 **心跳**（每 30 s）通过 **MQTT** 发布到主题 `devices/{device_id}/heartbeat`。  
- **MQTT Broker** 配置 **持久化会话**（QoS 1），确保离线期间消息不丢失。  
- **后端** 订阅该主题，收到心跳后：  
  - 更新 **Redis** `device:status:{device_id}`。  
  - 写入 **InfluxDB** `device_status`（时间序列）。  
  - 若状态改变，推送 **WebSocket** 消息给在线用户。  

### 4. Command Service  

| 步骤 | 细节 |
|------|------|
| **接收指令** | HTTP POST，解析 `cmd_id`、`device_id`、`command` |
| **幂等检查** | `SETNX command:idempotent:{cmd_id} "processing"` → 若已存在直接返回 |
| **写入 Kafka** | Topic `device-commands`（分区 = `device_id` hash），确保同一设备指令顺序 |
| **同步下发** | 同步推送到 **MQTT**（QoS 1），并记录 **messageId** |
| **回执处理** | 设备完成后向 `devices/{device_id}/cmd/ack` 发送 ACK，Command Service 收到后更新 Redis + MySQL `audit_logs`，并把幂等键设为 `completed` |
| **超时处理** | 若 5 s 未收到 ACK，进入 **重试队列**（Kafka 延迟主题或使用 **RabbitMQ** DLX） |
| **响应客户端** | 只要 **已成功写入 Kafka** 即返回 `accepted`（保证 300 ms），后续成功/失败通过 **WebSocket** 或轮询告知用户 |  

#### 幂等与可靠投递  

- **Kafka**：持久化、**Exactly‑once** 语义（开启事务），保证指令不被重复消费。  
- **MQTT QoS 1**：至少一次投递，若设备离线，Broker 会在设备重新上线后**重新推送**。  
- **指令唯一 ID**：所有重试使用同一个 `cmd_id`，在设备端也实现 **幂等**（若已经执行相同 `cmd_id`，直接 ACK）。  

### 5. Rule Engine Service  

- **触发源**：  
  - **时间**（Cron） → 使用 **Quartz Scheduler**。  
  - **设备事件**（温度超过阈值） → 订阅 **Kafka** `device-events`，使用 **Kafka Streams** 进行流式过滤。  
- **执行**：匹配规则后，生成 **批量指令**（一次性写入 Kafka），交给 **Command Service** 统一下发。  
- **事务性**：规则触发和指令写入在同一 **Kafka 事务** 中完成，防止 “触发了但指令未下发”。  

### 6. MQTT Broker  

| 选型 | 理由 |
|------|------|
| **EMQX** / **HiveMQ**（商业版） | 支持 **百万级并发连接**、分布式集群、插件化鉴权、QoS 1/2、桥接到 Kafka |  
| **Mosquitto**（轻量） | 适用于原型验证、单机可跑数千连接 |  

- **鉴权插件**：在连接阶段校验 **device token**（存于 Redis），不通过则断开。  
- **主题结构**：  
  - `devices/{device_id}/cmd`（平台下发指令）  
  - `devices/{device_id}/state`（设备上报状态）  
  - `devices/{device_id}/heartbeat`（心跳）  
- **持久化**：开启 **持久化存储**（LevelDB/SQLite）防止 broker 重启导致指令丢失。  

### 7. 监控、日志、告警  

| 项目 | 实现方式 |
|------|----------|
| **指标采集** | Prometheus Exporter 集成在每个微服务（HTTP latency、QPS、错误率） |
| **可视化** | Grafana Dashboard（实时 QPS、CPU、内存、MQTT 连接数） |
| **分布式追踪** | OpenTelemetry + Jaeger，跨服务请求链路追踪（关键链路：API → Command → MQTT） |
| **日志** | ELK（Filebeat → Logstash → Elasticsearch）或 Loki，日志统一结构化（JSON） |
| **告警** | Alertmanager 根据 SLA（响应时间 > 300 ms, 错误率 > 1%）触发 PagerDuty / 微信告警 |

## 第六步：扩展性与高可用设计  

### 1. 水平伸缩  

| 维度 | 伸缩方式 | 关键指标 |
|------|----------|----------|
| **API Gateway** | 增加实例 + **负载均衡**（L4/7） | 每实例 QPS ≤ 10k |
| **微服务** | **Kubernetes** Deployment + HPA（基于 CPU/请求延迟） | 自动扩容至满足峰值 25k QPS |
| **数据库** | 主从复制 + **读写分离**；MySQL 使用 **ProxySQL** 路由读写；必要时使用 **分库分表**（user_id hash） | 读写分离提升 5‑10 倍吞吐 |
| **Redis** | 集群模式（Cluster Slots） → 支持 10k QPS/节点 | 键均匀分布、无单点 |
| **MQTT Broker** | 集群部署（EMQX 支持水平扩容） | 连接数可达 1M+ |
| **Kafka** | 增加 Broker 与 Partition，分区数 = `设备数 / 1000`（约 12k） | 保障每台设备独占一个分区，顺序消费 |

### 2. 容灾与灾备  

| 层级 | 容灾方案 |
|------|----------|
| **跨地域** | 在两个 AZ（或两个可用区）部署同一套集群，使用 **跨 AZ 同步复制**（MySQL GTID, Kafka MirrorMaker） |
| **备份** | MySQL 全量快照 + binlog 增量；Redis 持久化（RDB+AOF）+ 定期复制；InfluxDB 备份 |  
| **故障转移** | 使用 **VIP + Keepalived** 或 **Service Mesh**（Istio）实现自动切流；K8s 中的 **PodDisruptionBudget** 保证最少副本数 |  
| **蓝绿部署** | API Gateway 支持 **canary**，逐步把流量切到新版本，降低回滚风险 |  

### 3. 数据一致性策略  

| 场景 | 一致性要求 | 方案 |
|------|------------|------|
| **用户-设备绑定** | 强一致（必须马上生效） | 写入 MySQL 主库后返回，读请求走 **读写分离**（主库） |
| **设备状态查询** | 最终一致（可接受几秒延迟） | 读 Redis，若缺失回源 MySQL/TSDB |
| **指令下发** | **强一致**（用户必须知道指令是否成功） | 采用 **两阶段提交**：① 写 Kafka（事务）② 等待设备 ACK（通过 MQTT）再标记成功；若超时返回失败 |  

### 4. 安全加固  

| 维度 | 措施 |
|------|------|
| **传输层** | 全链路 TLS（API、MQTT） |
| **身份认证** | OAuth2 + JWT，Token 失效时间短（15 min） |
| **设备鉴权** | MQTT 连接使用 **用户名 = device_id**, **密码 = 设备 token**（存 Redis） |
| **细粒度授权** | `user_devices.role`（owner/guest）控制指令范围 |
| **审计** | 所有指令、登录、绑定操作写入 **audit_logs**，满足合规 |

## 第七步：常见面试追问与回答  

### Q1. “如果在高峰期 QPS 突然翻倍，系统如何弹性伸缩？”  

**回答要点**：  
1. **自动伸缩（Auto‑Scaling）**：  
   - **API Gateway** + **K8s HPA** 基于 **CPU/请求延迟** 动态扩容。  
   - **Redis Cluster** 自动分片，增加节点后 **resharding**（无业务中断）。  
   - **Kafka**：提前预留 **Broker**，通过 **磁盘扩容** 和 **分区再均衡**（Kafka Cruise Control）应对流量激增。  
2. **限流降级**：  
   - 在网关层对单用户 QPS 进行硬限制，防止单个恶意用户占满资源。  
   - 对非关键业务（如日志写入）采用 **异步写入**（Kafka）或 **批量写**。  
3. **容量预留**：  
   - 在设计阶段预留 **30%~50% 的峰值冗余**（比如 30k QPS 处理能力），确保突发流量不至于触发资源耗尽。  
4. **快速回滚**：  
   - 使用 **蓝绿/灰度部署**，如果新版本出现性能问题，可立即切回旧版，保证 SLA。  

### Q2. “请说明你会选用哪种消息队列/协议来实现指令下发，如何保证指令的幂等性和可靠投递？”  

**选型**：  
- **协议**：**MQTT**（QoS 1） + **TLS**，因为 IoT 设备普遍支持，轻量、保持长连接、双向通信。  
- **消息队列**：**Kafka**（持久化、分区、事务）负责 **指令的可靠存储与重试**。  

**可靠投递**：  
1. **写入 Kafka**：使用 **事务**（`producer.beginTransaction()`），确保指令写入成功后才提交。  
2. **同步推送 MQTT**：指令写入后立即发布到对应主题，QoS 1 保证 **至少一次** 投递，离线设备会在重新连接后收到。  
3. **ACK 回执**：设备执行后向 `devices/{device_id}/cmd/ack` 回报，若未收到 ACK，系统进入 **重试**（Kafka 延迟队列或 DLX）。  

**幂等性**：  
- 客户端（App）生成 **全局唯一 `cmd_id`（UUID）**。  
- **Command Service** 在 Redis 中使用 `SETNX` 保存 `cmd_id → processed`，若已存在直接返回上一次的结果。  
- **设备端** 也保存最近处理过的 `cmd_id`，若收到重复指令直接返回 ACK，避免重复操作。  

### Q3. “在设备离线后，用户仍然可以查看最近状态，系统如何设计状态缓存和持久化？”  

**实现思路**：  

1. **状态写入路径**：  
   - 设备在线时通过 MQTT 心跳/状态上报，**后端**把数据写入 **Redis**（实时缓存）和 **InfluxDB**（时序持久化）。  
2. **离线检测**：  
   - 若 **30 s** 未收到心跳，标记 `device:status:{device_id}.online = false`（TTL 自动失效）。  
3. **查询时序**：  
   - **App** 请求 `/devices/{id}/status` → **Device Service** 先读 **Redis**。  
   - 若 Redis 中不存在（离线或缓存失效），从 **InfluxDB** 读取最近一条记录（最近 5 min）并返回。  
4. **缓存失效策略**：  
   - **TTL** 为 5 min，防止 stale 数据长期占用内存。  
   - **写回**：每次写入 InfluxDB 时同步更新 Redis（Cache‑Aside 模式）。  

这样即使设备离线，用户仍能看到 **最近一次成功上报的状态**，并在 UI 中标记为 “离线”。  

### Q4. “如果用户想要在 10 万用户同时创建场景规则，系统如何避免规则执行冲突？”  

**回答要点**：  
- **规则引擎采用** **事件驱动 + 幂等事务**。  
- 每条规则在触发时会生成 **唯一执行 ID**，写入 **Kafka**（事务），消费端（Command Service）在执行前检查该执行 ID 是否已处理（Redis 幂等键）。  
- 若同一设备在同一时间收到多条指令，**Command Service** 按 **device_id 分区** 的顺序消费，确保指令顺序执行。  
- **冲突检测**：在规则创建时进行 **冲突校验**（同一时间点同一设备的互斥操作），通过 **规则管理库**（MySQL）做唯一约束（`device_id + trigger_time`），避免业务层冲突。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策  

| 决策 | 为什么难 | 思考过程 |
|------|----------|----------|
| **指令的可靠投递 + 幂等** | IoT 设备网络不稳定，指令可能丢失或重复。要兼顾 **低延迟**（≤300 ms）和 **强一致**，涉及多层协议（HTTP、Kafka、MQTT）以及事务/幂等实现。 | - 先确定业务必须“用户感知成功”。<br>- 选定 **Kafka** 负责持久化、**MQTT QoS1** 负责实时下发。<br>- 设计 **全局唯一 cmd_id** + Redis 幂等键，防止重发导致重复执行。 |
| **实时状态查询的高并发与离线兼容** | 需要在 **200 ms** 内返回最新状态，同时要在设备离线时仍能提供最近数据。涉及缓存一致性、时序库、失效策略。 | - 将热点状态放在 **Redis**，采用 **Cache‑Aside**。<br>- 设备每 30 s 心跳，更新 Redis & InfluxDB。<br>- 设定 TTL，离线后从 InfluxDB 回拉最近记录。<br>- 通过 **读写分离** 避免 MySQL 成为瓶颈。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **“一开始就把所有功能全堆在单体服务里”**  
   - 结果是 **难以水平扩展**、**故障会波及全系统**。  
   - 正确做法：先实现 **MVP**，随后按业务边界拆分为 **微服务**（Device、Command、Rule），并在 API Gateway 前做统一入口。  

2. **忽视 **幂等性** 与 **消息可靠性**，直接用 HTTP → 直接调用设备**  
   - 在网络抖动或设备离线时会产生 **指令丢失/重复**，导致用户体验差甚至安全风险（门锁）。  
   - 正确做法：使用 **唯一指令 ID**、**幂等缓存**、**持久化队列**（Kafka）+ **QoS1 MQTT** 双保险。
