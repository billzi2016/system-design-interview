# 第 43 天：设计 优惠券 / 红包系统

> 生成日期：2026-04-13

---

## 题目背景
优惠券/红包系统为电商、生活服务、社交等平台提供 **发放、领取、核销** 的能力，帮助业务实现促销、用户激活与留存。系统需要在高并发下保证优惠券的唯一性、有效性以及及时失效。

---

## 面试场景设定
> **面试官**：  
> “现在请你设计一个 **优惠券/红包系统**，要求能够支撑我们平台的日常促销活动。请先从整体架构出发，说明系统的核心模块以及它们之间的交互，并重点解释你在 **高并发发放** 与 **核销** 场景下的设计思路。”

---

## 功能性需求
1. **优惠券/红包发放**  
   - 支持 **批量发放**（如活动全量、指定用户、用户分层）  
   - 支持 **定时/周期** 发放（活动开始/结束自动触发）

2. **领取/领取校验**  
   - 用户可主动领取或系统自动投放到用户账户  
   - 防止同一用户重复领取同一批次（幂等校验）

3. **核销/使用**  
   - 在订单结算或支付前校验优惠券是否满足使用规则（金额阈值、商品范围、时间窗口等）  
   - 支持 **部分使用**（如满 100 减 20，剩余金额继续可用）  

4. **状态管理 & 失效**  
   - 自动失效（到期、已使用、被撤回）  
   - 支持 **手动撤回**（活动异常或违规）  

5. **查询与统计**  
   - 用户可查询自己持有的优惠券列表、状态、使用期限  
   - 业务方可查询 **发放量、领取率、核销率** 等关键指标  

6. **安全与防刷**  
   - 防止恶意抢券、刷券、重复使用（如 IP 限速、验证码、业务签名）  

---

## 非功能性需求（带具体估算）

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（活跃用户）** | 15 Million | 包括电商、社交等多个业务线 |
| **QPS（高峰期请求）** | 3,000 req/s | 发放、领取、核销高峰叠加 |
| **平均响应时延** | ≤ 80 ms | 主要接口（领取、核销） |
| **可用性** | 99.99%（每月累计宕机 < 4.3 min） | 业务对促销活动高度敏感 |
| **存储容量** | 约 600 TB（≈ 1 billion 条优惠券记录，含历史） | 需要支持 3 年以上的历史数据查询 |
| **数据一致性** | 强一致性（核销必须单点成功） | 防止同一券被多次使用 |
| **扩展性** | 支持水平扩容，节点数可弹性伸缩至 200+ | 业务活动峰值可瞬时提升 5 倍 |

## 系统边界
**本题需实现的范围**  
- 优惠券/红包的 **发放、领取、核销、状态管理** 与 **基本查询**

- 第三方渠道（如短信、邮件）推送的实现细节  
- 跨业务线的 **统一财务结算**（仅关注优惠券本身的价值流转）

## 解题思路总览
> **核心目标**：在 **高并发** 场景下 **安全、准确** 地完成「发放 → 领取 → 核销」全链路，同时满足业务查询、统计与防刷需求。  
> **思考框架**：  
> 1. **先把需求拆成业务流程**（发放、领取、核销、状态管理、查询）。  
> 2. **估算流量/数据量**，判断是否需要 **水平扩容、分库分表、缓存**。  
> 3. **画出最小可运行（MVP）系统**，确认每个业务点的 *读写* 方向与一致性要求。  
> 4. **逐步加入**：  
>    - **幂等/唯一性**（ID 生成、业务键唯一约束）  
>    - **异步化**（定时发放、失效、统计）  
>    - **防刷/限流**（滑动窗口、验证码）  
>    - **高可用**（多活、容灾、降级）  
> 5. **最终形成**：**业务层 → 缓存层 → 持久层 → 消息队列 → 定时任务** 的完整闭环。  

下面按章节展开，每一步都写出「**为什么**」和「**不这么做会怎样**」。

## 第一步：理解需求与规模估算

| 需求 | 关键点 | 关联的技术难点 |
|------|--------|----------------|
| **发放** | 批量、定时、分层、唯一/幂等 | 大批量写、幂等、限流 |
| **领取** | 主动或自动投放、避免重复领取 | 幂等校验、热点竞争 |
| **核销** | 订单结算前校验、部分使用、强一致性 | 事务、分布式锁、缓存击穿 |
| **状态管理** | 自动失效、撤回 | 定时任务、状态机 |
| **查询/统计** | 用户持券列表、业务指标 | 大规模读、OLAP |
| **安全/防刷** | IP 限速、验证码、签名 | 防刷、限流、灰度 |

### 1.1 业务规模估算（依据题目给出的非功能需求）

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 15 M | 最高峰同时在线 ≈ 5 M |
| **日新增券** | 假设 0.5% 用户领取一次，单次 1 张 → 75 M | 3 年累计 ≈ 1 B 张 |
| **QPS** | 3 000 req/s（发放+领取+核销） | 峰值 5 × → 15 000 req/s |
| **存储** | 1 B 条记录 × 600 B（每条约 600B） ≈ 600 TB | 需要冷热分层存储 |
| **强一致性** | 核销必须单点成功，避免双扣 | 需要事务或分布式锁 |
| **响应时延** | ≤ 80 ms | 需要缓存、快速路径 |

> **如果直接上大规模分库分表**：实现复杂、运维成本高，且在 **MVP** 阶段没有必要。先用 **单库单表**（或分库+分表）跑通业务，再逐步演进。

## 第二步：高层架构设计

### 2.1 MVP（最小可运行系统）  
```
[Client] → API Gateway → Coupon Service (REST) → MySQL
                                   ↘
                                   Redis (Cache)
```
- **API Gateway**：统一入口，做鉴权、限流、日志。  
- **Coupon Service**：业务实现（发放、领取、核销）。  
- **MySQL**：持久化，使用 **InnoDB**，保证事务性。  
- **Redis**：热点券状态缓存（用户领券列表、核销锁）。

> **不加入消息队列、定时任务的风险**：  
> - 高并发批量发放会直接压垮 MySQL。  
> - 失效、撤回等后台任务难以准时执行，导致“过期券仍可用”。

### 2.2 完整分布式架构（满足 99.99% 可用、5 倍峰值）

```
                              ┌─────────────┐
                              │   API GW    │
                              └─────┬───────┘
                                    │
                ┌───────────────────┼───────────────────┐
                │                   │                   │
          ┌─────▼─────┐       ┌─────▼─────┐       ┌─────▼─────┐
          │  Auth/    │       │  Rate     │       │  Log/    │
          │  Security │       │  Limiter  │       │  Metrics │
          └─────┬─────┘       └─────┬─────┘       └─────┬─────┘
                │                   │                   │
          ┌─────▼─────┐       ┌─────▼─────┐       ┌─────▼─────┐
          │ Coupon   │◄──────►│   Cache   │◄──────►│  MQ (Kafka)│
          │ Service  │       │ (Redis)   │       │ (异步任务) │
          └─────┬─────┘       └─────┬─────┘       └─────┬─────┘
                │                   │                   │
          ┌─────▼─────┐       ┌─────▼─────┐       ┌─────▼─────┐
          │ MySQL    │       │ TSDB (ClickHouse)││  Scheduler │
          │ (主库/从库)│       │ 统计/报表 │       │(Quartz)   │
          └─────┬─────┘       └─────┬─────┘       └─────┬─────┘
                │                   │                   │
          ┌─────▼─────┐       ┌─────▼─────┐       ┌─────▼─────┐
          │ Order   │       │  CDN/OSS │       │  Alert/   │
          │ Service │       │  (附件) │       │  Ops      │
          └──────────┘       └─────────┘       └───────────┘
```

#### 关键组件解释

| 组件 | 作用 | 为什么要有 |
|------|------|------------|
| **API Gateway** | 统一入口、统一鉴权、限流、灰度发布 | 防止直接暴露内部服务，统一治理流量 |
| **Auth / Security** | JWT / OAuth2，防刷签名 | 防止恶意调用、伪造请求 |
| **Rate Limiter** | 本地+分布式（令牌桶）限流 | 抢券、秒杀场景下防止瞬时流量冲垮后端 |
| **Coupon Service** | 业务核心（发放、领取、核销、查询） | 业务拆分，便于水平扩容 |
| **Cache (Redis)** | 用户领券列表、核销锁、热点券状态 | 读写分离、降低 DB 压力，满足 80 ms 响应 |
| **Message Queue (Kafka)** | 异步处理：批量发放、失效、统计、审计日志 | 峰值流量削峰，保证业务的 **最终一致性** |
| **MySQL** | 主库写、从库读，事务强一致 | 核销必须单点成功，避免双扣 |
| **TSDB / ClickHouse** | 大规模统计、报表查询 | 600 TB 历史数据不适合放在 OLTP 库 |
| **Scheduler (Quartz / XXL‑Job)** | 定时任务：批量发放、失效、撤回 | 精准时间点触发，避免业务逻辑耦合到主流程 |
| **Order Service** | 只做优惠券校验的调用方 | 解耦，保证 Coupon Service 不依赖订单实现 |
| **Alert / Ops** | 监控、告警、自动伸缩 | 达到 99.99% 可用目标 |

> **不使用 MQ 的后果**：高峰期全部请求直接打到 MySQL，写入热点（同一批次券）导致行锁竞争，响应时间瞬间冲到秒级，甚至出现 **死锁**。  

> **不做缓存的后果**：每次领取/核销都要查询 MySQL，读写比例 1:1 时延大，难以满足 80 ms SLA。

## 第三步：数据库设计

### 3.1 业务实体模型（简化版）

```sql
-- 1. Coupon Batch（批次/活动）: 描述一次发放活动
CREATE TABLE coupon_batch (
    batch_id        BIGINT PRIMARY KEY,          -- 分布式唯一 ID（Snowflake）
    name            VARCHAR(64),
    total_qty       BIGINT NOT NULL,            -- 总发行量（0 表示无限）
    issued_qty      BIGINT NOT NULL DEFAULT 0,  -- 已发放数量
    status          TINYINT NOT NULL DEFAULT 1, -- 1:进行中 2:已结束 3:撤回
    start_time      DATETIME,
    end_time        DATETIME,
    rule_json       JSON,                       -- 使用规则（门槛、商品范围等）
    create_time     DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;

-- 2. User Coupon（用户领取的券）
CREATE TABLE user_coupon (
    uc_id           BIGINT PRIMARY KEY,          -- 唯一 ID（Snowflake）
    batch_id        BIGINT NOT NULL,
    user_id         BIGINT NOT NULL,
    coupon_code     VARCHAR(32) NOT NULL,        -- 业务唯一键，防刷
    status          TINYINT NOT NULL DEFAULT 0, -- 0:未使用 1:已使用 2:已失效 3:已撤回
    amount          DECIMAL(10,2) NOT NULL,     -- 面值或折扣额
    used_amount     DECIMAL(10,2) DEFAULT 0,    -- 已使用（部分使用场景）
    valid_start     DATETIME,
    valid_end       DATETIME,
    create_time     DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uq_user_batch (user_id, batch_id)   -- 防止同一用户重复领取同批次
) ENGINE=InnoDB;

-- 3. Coupon Usage Log（核销审计）
CREATE TABLE coupon_usage_log (
    log_id          BIGINT PRIMARY KEY,
    uc_id           BIGINT NOT NULL,
    order_id        BIGINT NOT NULL,
    used_amount     DECIMAL(10,2) NOT NULL,
    used_time       DATETIME DEFAULT CURRENT_TIMESTAMP,
    rule_snapshot   JSON,                       -- 使用时规则快照
    INDEX idx_uc_id (uc_id)
) ENGINE=InnoDB;
```

### 3.2 关键设计点

| 设计点 | 目的 | 可能的坑 & 规避方式 |
|--------|------|---------------------|
| **分布式唯一 ID（Snowflake）** | 生成全局唯一、递增的 `batch_id`、`uc_id`，避免业务冲突 | 若使用自增 ID，跨库分库会冲突 |
| **唯一键 `uq_user_batch`** | 防止同一用户多次领取同一批次（幂等） | 必须在业务层捕获 `Duplicate entry` 异常并返回幂等成功 |
| **状态字段（status）** | 统一管理「未使用/已使用/失效/撤回」 | 业务状态迁移必须走统一的 **状态机**，避免直接 UPDATE 产生不一致 |
| **规则 JSON** | 灵活存储多变的使用门槛（满减、商品类目等） | 查询时若需要过滤，建议在业务层解析，而不是在 SQL 中做 JSON 查询 |
| **分表策略** | 1 B 条记录 → 按 `user_id` 哈希分 64~128 表 | 初期不做分表，等到单表行数 > 1 亿 再拆分；提前预留 `batch_id` 与 `user_id` 的分片键 |
| **只读从库** | 业务查询（用户券列表）走从库，写走主库 | 确保从库延迟 < 1 s，防止用户领取后立即查询不到 |
| **审计日志表** | 追踪每一次核销，支持对账 | 采用 **Append‑Only**，不更新，保证审计完整性 |

### 3.3 事务与强一致性

- **领取**：  
  1. 检查批次状态、库存（`issued_qty < total_qty`）  
  2. 插入 `user_coupon`（唯一键保证幂等）  
  3. `UPDATE coupon_batch SET issued_qty = issued_qty + 1`  
  4. **事务提交**（MySQL InnoDB）  

- **核销**：  
  1. **先在 Redis 加锁**（key: `lock:uc:{uc_id}`）避免并发双扣  
  2. 检查 `user_coupon` 状态、有效期、使用规则  
  3. **UPDATE user_coupon SET status=1, used_amount=... WHERE uc_id=? AND status=0**（乐观锁 `WHERE status=0`）  
  4. 写入 `coupon_usage_log`  
  5. **事务提交**（写库）  
  6. 释放 Redis 锁  

> **不加锁的后果**：在 5 k 并发的核销场景下，同一券可能被多次扣减，导致财务对账错误。

## 第四步：核心 API 设计

> **原则**：RESTful + 明确的幂等性设计；所有写操作返回业务唯一的 `request_id`，便于追踪。

| 方法 | 路径 | 业务含义 | 请求体（示例） | 响应 | 幂等性 |
|------|------|----------|----------------|------|--------|
| **POST** | `/api/v1/coupon/batches` | 创建发放批次（活动） | `{ "name":"双11全场满100减20","total_qty":1000000,"start_time":"2024-11-11T00:00:00Z","end_time":"2024-11-11T23:59:59Z","rule":{...} }` | `{ "batch_id":1234567890 }` | **是**（业务 ID 重复提交返回同 batch_id） |
| **POST** | `/api/v1/coupon/batches/{batch_id}/publish` | 手动触发批次发放（可配合定时任务） | `{}` | `{ "published":true }` | **是**（幂等） |
| **POST** | `/api/v1/coupon/receive` | 用户领取券 | `{ "user_id":98765, "batch_id":1234567890 }` | `{ "uc_id":55555555, "coupon_code":"ABCD1234" }` | **是**（基于 `uq_user_batch` 幂等） |
| **GET** | `/api/v1/coupon/user/{user_id}` | 查询用户持券列表（分页） | `?status=0&offset=0&limit=20` | `{ "total":123, "list":[...] }` | **否**（查询本身） |
| **POST** | `/api/v1/coupon/redeem` | 订单核销优惠券 | `{ "user_id":98765, "uc_id":55555555, "order_id":112233, "order_amount":120.00 }` | `{ "discount":20.00, "final_amount":100.00 }` | **是**（基于 `uc_id` 幂等） |
| **POST** | `/api/v1/coupon/batches/{batch_id}/revoke` | 手动撤回批次（所有未使用券失效） | `{}` | `{ "revoked":true }` | **是** |
| **GET** | `/api/v1/metrics/coupon` | 业务统计（发放量、领取率、核销率） | `?start=2024-11-01&end=2024-11-30` | `{ "issued":..., "claimed":..., "redeemed":... }` | **否** |

### 幂等实现要点

1. **请求幂等键**（`Idempotency-Key`）在前端生成，后端在 **Redis** 中保存 `key → result`，30 min 失效。  
2. **业务唯一键**：`uq_user_batch` 防止重复领取；`uc_id` 本身唯一，核销时 **只允许一次成功**（乐观锁 + Redis 锁）。  
3. **重试策略**：对 MQ、定时任务使用 **幂等消费**（消费前检查状态）。

## 第五步：详细组件设计

### 5.1 发放模块（批量/定时）

1. **业务入口**：`POST /batches/{batch_id}/publish`  
2. **流程**  
   - **读取批次配置**（MySQL）  
   - **生成待发放任务**：使用 **Kafka** 的 **批次主题**（`coupon_batch_{batch_id}`），每条消息携带 `user_id`、`batch_id`、`coupon_code`（可提前预生成或在消费端生成）  
   - **消费者（发放 Worker）**：水平扩容的 **Kafka Consumer Group**，负责把消息写入 `user_coupon`（带事务），并更新 `issued_qty`。  
   - **幂等**：在写库前先 `SELECT ... FOR UPDATE` 检查 `uq_user_batch`，若已存在直接 **ACK**，不再插入。  

3. **技术选型**  
   - **Kafka**：高吞吐、持久化、支持重放。  
   - **Spring Batch / Flink**（可选）：如果需要复杂的 **分层投放**（不同用户标签），可在 **Kafka Producer** 前做一次离线计算，生成不同 `user_id` 列表。  

4. **失效/撤回**  
   - **撤回**：写 `batch.status=3`，随后 **异步任务**遍历 `user_coupon`（`status=0`）改为 `status=2`（失效），使用 **分片扫描**（MySQL 8.0 `ROW_NUMBER` + 分批）或 **ClickHouse** 实时更新。  

### 5.2 领取模块

| 场景 | 关键点 | 实现 |
|------|--------|------|
| **主动领取** | 防止同批次并发抢领 | 1️⃣ 前端限流（秒杀页面）<br>2️⃣ API Gateway 限流（IP+用户）<br>3️⃣ **Redis 分布式锁** `lock:batch:{batch_id}`，加锁后检查 `issued_qty < total_qty`，再写库。 |
| **系统投放**（如新用户注册赠送） | 必须保证 **幂等** | 在业务层先查 `uq_user_batch`，若已存在直接返回；否则走正常写库流程。 |
| **高并发抢券**（秒杀） | 1. **热点预热**<br>2. **令牌桶**<br>3. **库存预扣** | - **库存预扣**：在 Redis 中维护 `stock:{batch_id}`（原子 `decrby`），当库存为负时直接返回抢购失败。<br>- 成功后异步写入 MySQL（Consumer）。<br>- 失败请求直接返回，不进入 DB。 |

> **不做库存预扣的后果**：所有请求直接打到 MySQL，行锁竞争导致 QPS 大幅下降，甚至出现 **Deadlock**。

### 5.3 核销模块

1. **调用方**：订单系统在 **结算前**调用 `POST /coupon/redeem`。  
2. **流程**（伪代码）  

```text
1. 接收请求，校验签名 + 参数合法性
2. Redis SETNX lock:uc:{uc_id} (ttl=5s)
   - 若返回 0，说明已有核销在进行，直接返回错误（防止并发双扣）
3. 从 MySQL（主库）读取 user_coupon（SELECT ... FOR UPDATE）
4. 校验：
   - status == 0
   - now BETWEEN valid_start AND valid_end
   - order_amount >= rule.threshold
   - 商品范围匹配（rule.sku_set）
5. 计算 discount（满减、比例等），更新：
   UPDATE user_coupon 
   SET status = IF(used_amount+discount >= amount, 1, 0),
       used_amount = used_amount + discount
   WHERE uc_id = ? AND status = 0;
6. 写入 coupon_usage_log
7. 提交事务
8. 删除 Redis 锁
9. 返回 discount、final_amount 给订单系统
```

3. **强一致性保障**  
   - **行锁 + 乐观锁**：`WHERE status=0` 防止并发更新。  
   - **Redis 锁**：防止在事务外的业务重入（如同一个请求因为网络重试导致再次进入）。  

4. **部分使用**  
   - `used_amount` 记录已使用金额；若 `used_amount < amount`，仍保持 `status=0`，可继续使用。  

### 5.4 状态管理 & 自动失效

| 场景 | 触发方式 | 实现 |
|------|----------|------|
| **到期失效** | **定时任务**（Quartz）每分钟扫描 `user_coupon` 中 `valid_end < now AND status=0` | 批量 `UPDATE user_coupon SET status=2 WHERE ...`（分片） |
| **已使用失效** | **核销成功**时直接更新 `status=1` | 事务内完成 |
| **撤回失效** | **后台管理员**调用撤回 API | 写 `batch.status=3`，异步扫描并更新对应 `user_coupon` 为 `status=2` |
| **异常失效**（如库存不匹配） | **监控告警**触发补偿脚本 | 手工或自动修正 |

> **失效任务不做分片**的风险：一次全表 `UPDATE` 会锁表，导致业务写入阻塞。分批（每批 10k）+ `WHERE id > last_id` 能有效避免。

### 5.5 查询 & 统计

1. **用户持券查询**  
   - **读从库**（只读）+ **Redis 缓存**（热点用户）。  
   - Cache key：`user:coupon:list:{user_id}:{status}`，TTL 5 min，更新时 **双删**（写库后删缓存）或 **延迟双删**。  

2. **业务指标统计**  
   - **实时**：使用 **Kafka → ClickHouse**，将 `coupon_usage_log`、`user_coupon` 的变更写入 ClickHouse，实时聚合。  
   - **离线**：每日/每周跑 **Spark/Presto** 报表。  

3. **API 示例**  
   ```http
   GET /api/v1/metrics/coupon?start=2024-11-01&end=2024-11-30
   ```
   - 后端查询 ClickHouse，返回 `issued`, `claimed`, `redeemed`, `revoked` 等字段。  

### 5.6 防刷 / 限流

| 防刷点 | 手段 | 实现细节 |
|--------|------|----------|
| **接口层** | IP + UserId 令牌桶 | `RateLimiter`（Redis Lua 脚本）每秒 X 次，突发 Y 次 |
| **领取抢券** | 验证码 + 短信（业务方） | 前端在秒杀页弹出图形验证码，后端校验后放行 |
| **签名** | HMAC（业务私钥） | 请求头 `X-Signature` = HMAC(secret, body) |
| **黑名单** | 违规 IP/设备ID | Redis Set `blacklist:{ip}`，拦截所有请求 |
| **机器学习**（可选） | 风险评分 | 通过日志分析异常行为，动态调节限流阈值 |

> **不做限流**：在秒杀活动期间，瞬时 10 万并发请求会直接压垮后端 DB，导致业务不可用。

## 第六步：扩展性与高可用设计

### 6.1 水平扩容

| 组件 | 横向扩容方式 | 关键指标 |
|------|--------------|----------|
| **API Gateway** | 多实例 + LVS/NGINX 负载均衡 | QPS ≤ 20k |
| **Coupon Service** | Stateless → 任意节点均可部署，使用 **Consul/Eureka** 注册 | CPU 80% 报警 |
| **Redis** | 主从复制 + 哨兵，或 **Cluster**（分片） | 单节点 QPS ≤ 100k |
| **MySQL** | 主从复制 + 分库分表（user_id 哈希） | 单库 QPS ≤ 10k |
| **Kafka** | 分区数 ≥ 消费并发数（每分区一个 Consumer） | 分区 1000+ |
| **ClickHouse** | 多节点分布式表 | 实时聚合 QPS ≥ 50k |

> **不做分库分表**的后果：单表行数 > 1 B 时，索引体积巨大，查询和写入性能急剧下降，甚至出现 **磁盘 I/O 瓶颈**。

### 6.2 高可用（HA）策略

| 故障场景 | 解决方案 |
|----------|----------|
| **单点故障（服务实例）** | 多实例 + 健康检查 → 自动下线 |
| **Redis 主节点宕机** | 哨兵自动故障转移；客户端使用 **Redis Sentinel** 或 **Cluster** |
| **MySQL 主库故障** | 主从切换（MHA）或 **双活（Galera）**，业务自动切到从库写 |
| **Kafka Broker 故障** | 多副本（replication.factor ≥ 3），自动 leader 迁移 |
| **网络分区** | 超时重试 + 幂等消费，保证不重复扣减 |
| **灾难恢复** | 同城双活 + 异地备份（OSS/对象存储），每日快照 + PITR（Point‑In‑Time Recovery） |

### 6.3 流量削峰（秒杀抢券）完整方案

1. **入口层**  
   - **CDN + WAF**：过滤恶意流量、Bot。  
   - **API Gateway**：统一限流（全局 QPS）+ **IP+用户令牌桶**。  

2. **抢券队列**  
   - **预热库存**：活动开始前将 `stock:{batch_id}` 写入 Redis，使用 **原子 DECR**。  
   - **令牌桶**：每秒生成 N 张抢券令牌，只有拿到令牌的请求才能进入下一步。  

3. **业务层**  
   - **无锁抢券**：`stock = redis.decrby(key, 1)`；若 `stock >= 0`，生成 **领券消息**发送到 Kafka。  
   - **异步落库**：Kafka Consumer 负责写 `user_coupon`，成功后回写 `ResultCache`（Redis）供前端轮询。  

4. **回滚机制**  
   - 若 Kafka 消费失败（比如 DB 写入冲突），在 **补偿任务**中将 `stock` 加回（`redis.incrby`），确保库存最终一致。  

5. **监控报警**  
   - **库存残余**、**Kafka 消费延迟**、**Redis QPS** 实时监控，异常自动降级（如关闭抢券入口，转为排队）。  

> **不做削峰**：全部请求直接落库，导致 DB 瞬时超载、服务不可用，甚至 **数据丢失**（请求被直接丢弃）。

### 6.4 监控、日志、告警

| 维度 | 关键指标 | 工具 |
|------|----------|------|
| **业务** | 发放成功率、领取成功率、核销成功率、失效率 | Grafana + Prometheus |
| **性能** | QPS、RT、CPU/内存、磁盘 I/O、网络流量 | Prometheus Node Exporter |
| **可靠性** | Kafka 消费延迟、Redis 命中率、MySQL 主从复制延迟 | Kafka Manager, Redis Sentinel, MySQL Replication Monitor |
| **异常** | 订单核销异常、库存负数、幂等冲突 | ELK（日志）+ Alertmanager |

## 第七步：常见面试追问与回答

### Q1：**高并发发放时如何保证唯一性与幂等？**  
**答**：  
1. **全局唯一 ID**（Snowflake）确保 `batch_id`、`uc_id` 在所有节点唯一。  
2. **业务唯一键** `uq_user_batch`（`user_id + batch_id`）在 `user_coupon` 表上建立唯一索引，数据库层面天然幂等。  
3. **幂等键**：前端传 `Idempotency-Key`，后端在 Redis 中保存请求结果 30 min，重复请求直接返回缓存结果。  
4. **写入流程**：采用 **MySQL 事务** + **乐观锁**（`WHERE status=0`），若出现 `Duplicate entry`，捕获异常返回幂等成功。  

> **不做唯一键**：在 10 万并发抢券时，同一用户可能在不同节点同时插入，导致 **重复券**，业务财务出现漏扣。

### Q2：**核销流程中如果使用了缓存，如何防止“缓存击穿”导致的重复核销？**  
**答**：  
- **双写一致性**：核销时先 **加分布式锁**（Redis `SETNX`），锁定 `uc_id`，只有获得锁的请求才能进入 DB 事务。锁的 TTL 设为 5 s，防止死锁。  
- **缓存失效策略**：核销成功后 **删除对应的缓存**（`del coupon:uc:{uc_id}`），并在写库后 **双删**（先删，事务提交后再删一次），降低脏数据概率。  
- **缓存预热**：热点券的状态放在 Redis 中，读直接命中；若不存在则走 DB，返回后写入缓存（**Cache‑Aside**）。  
- **降级**：若 Redis 故障，直接走 DB（虽然延迟稍高），但保证业务正确性。  

> **不加锁**：两台机器几乎同时读取缓存为 “未使用”，都去写 DB，导致 **双扣**，财务对账出现异常。

### Q3：**秒杀式抢券（10 万并发在 1 秒内抢完）如何设计水平扩容与流量削峰？**  
**答**：  
1. **流量入口**：使用 CDN+WAF+API‑Gateway 做全局限流，分配 **全局令牌桶**（每秒 N 张）控制进入系统的请求速率。  
2. **库存预扣**：在 Redis 中维护 `stock:{batch_id}`，使用 **原子 DECR**，成功的请求进入下一个环节，失败直接返回 “抢完”。  
3. **异步写库**：成功抢到的请求只产生 **Kafka 消息**（`user_id, batch_id, coupon_code`），不直接写 DB。Kafka Consumer **水平扩容**（分区数 ≥ 消费并发），并在消费端完成事务写入 `user_coupon`。  
4. **补偿回滚**：若消费失败（如 DB 写入冲突），补偿任务将 `stock` 加回，保证最终库存一致。  
5. **监控**：实时监控 **Redis QPS**、**Kafka 延迟**、**库存剩余**，若出现异常自动 **降级**（关闭抢券入口或转为排队）。  

> **不做削峰**：所有 10 万请求直接到 MySQL，行锁竞争导致系统卡死，甚至宕机，用户体验极差。

### 其它可能追问

| 追问 | 简要回答 |
|------|----------|
| **如何实现批量发放的定时任务？** | 使用 **Quartz**（或 **XXL‑Job**）在业务库里保存 `schedule_time`，任务触发时查询待发放的 `batch_id`，生成 Kafka 消息进行异步投放。 |
| **为什么核销必须强一致，而查询可以 eventual？** | 核销涉及金钱扣减，必须一次成功一次失败，防止重复扣款；查询只要求用户看到最新状态，短暂的 **读写延迟** 可以接受。 |
| **如果 MySQL 主库宕机，如何保证核销不丢失？** | 使用 **双写**（主从同步）+ **GTID** 备份；故障切换后新主库继续接受写入，Kafka 消费的事务日志可做 **补偿**。 |
| **如何处理跨库事务（如订单系统扣款）？** | 采用 **可靠消息 + 最终一致性**：Coupon Service 先核销成功，发送 `CouponUsed` 事件；订单系统监听后完成扣款，若扣款失败再回滚（补偿消息）。 |

## 心得与反思

### 1️⃣ 本题最难的设计决策

| 决策 | 考量 | 思考过程 |
|------|------|----------|
| **抢券的库存预扣 vs 直接落库** | **性能 vs 数据一致性**。抢券瞬时并发极高，直接写库会导致行锁竞争、事务冲突；但预扣到 Redis 需要后续补偿保证最终一致性。 | - 先评估峰值 QPS（10 万/秒） → MySQL 写入不可能。<br>- 设计 **Redis 原子 DECR** + **Kafka 异步落库**，并加入 **补偿回滚**，在业务层接受最终一致性。<br>- 为防止库存负数，设置 **Lua 脚本** 检查并返回错误码。 |
| **核销的强一致性实现** | **防止双扣** 与 **高并发** 冲突。<br>需要在 1 ms 内完成锁定、校验、更新。 | - 比较 **分布式锁**（Redis） vs **数据库行锁**（SELECT … FOR UPDATE）。<br>- 发现单纯 DB 行锁在高并发下仍会产生锁等待，导致响应 > 80 ms。<br>- 采用 **先 Redis 锁 + 乐观锁** 双保险，确保业务层快速返回，同时在 DB 中做最终校验。 |

### 2️⃣ 新手最容易犯的错误（至少两条）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有写请求直接落到单库** | 读写竞争、锁冲突、响应时间爆炸，导致 SLA 失效。 | 使用 **消息队列削峰**，热点数据预写入 **Redis**，异步落库。 |
| **忽略幂等性**（如领取、核销） | 重试时产生重复记录，导致双扣或重复发放。 | 在 DB 建唯一索引、使用 **Idempotency-Key**，业务层捕获冲突返回成功。 |
| **缓存与数据库双写不一致** | 缓存脏数据导致用户看到错误券状态（如已使用仍显示未使用）。 | 采用 **Cache‑Aside + 双删** 或 **事务性写缓存**（如 Redis 事务 + MySQL）。 |
| **没有监控/告警** | 故障发现慢，业务影响大。 | 为每个关键路径（库存、Kafka 延迟、Redis QPS）设置 **Prometheus** 监控和 **Alertmanager** 告警。 |

### 3️⃣ 学习建议与可延伸方向

1. **系统设计基础**  
   - 熟悉 **CAP 定理**、**BASE** 与 **强一致** 场景的取舍。  
   - 掌握 **分布式唯一 ID**（Snowflake、Leaf）和 **分库分表** 的实现原理。  

2. **关键技术深耕**  
   - **Redis**：Lua 脚本、分布式锁、Cache‑Aside、热点预热。  
   - **Kafka**：分区、消费者组、事务、Exactly‑once 语义。  
   - **MySQL**：事务隔离级别、行锁、GTID、主从切换。  

3. **性能调优实战**  
   - 用 **JMeter/Locust** 进行抢券压测，观察锁竞争、CPU/IO、GC。  
   - 通过 **Explain** 分析 SQL，优化索引与覆盖索引。  

4. **高级主题**（可在后期扩展）  
   - **分布式事务**（TCC、Saga）在跨系统扣款场景的应用。  
   - **机器学习** 防刷模型：基于行为特征动态调节限流阈值。  
   - **多活跨地域**（Active‑Active）架构，实现全球秒级抢券。  

> **一句话总结**：  
> *在高并发优惠券系统里，**先把流量削峰**（前置限流、库存预扣），再 **确保幂等与强一致**（唯一键、分布式锁），最后 **做好监控与弹性伸缩**，系统才能在秒杀级别的流量下仍然保持 **正确** 与 **可用**。*
