# 今日头条信息流推荐系统设计题

## 1. 题目背景
今日头条是一款基于机器学习的内容分发平台，向海量用户实时推送个性化的新闻、短视频等信息流。系统需要在毫秒级完成用户兴趣建模与内容排序，以提升用户黏性和平台收益。

## 2. 面试场景设定
> **面试官**：  
> “假设我们要为今日头条设计一套高可用、低延迟的信息流推荐系统。请从整体架构出发，说明你会怎样支撑每天数亿用户的实时推荐需求，并重点阐述核心模块的设计思路。”

## 3. 功能性需求
| 编号 | 功能描述 |
|------|----------|
| 1 | **用户登录 & 身份鉴权**：支持手机号、第三方 OAuth 登录，返回唯一用户 ID。 |
| 2 | **实时兴趣模型**：基于用户最近的点击、浏览、点赞、分享等行为，实时更新用户画像。 |
| 3 | **内容召回 & 过滤**：从全量内容库中快速召回与用户兴趣匹配的候选文章/视频，并过滤违规、重复、已读等内容。 |
| 4 | **排序 & 打分**：使用机器学习模型（如 GBDT、DeepFM）对候选集合进行打分并生成最终展示顺序。 |
| 5 | **曝光监控 & 反馈收集**：记录每条内容的曝光、点击、停留时长等行为，上报给离线训练系统。 |
| 6 | **个性化配置**：支持用户自行屏蔽/关注特定主题、作者，系统在召回/排序时遵循这些偏好。 |

## 4. 非功能性需求
| 指标 | 目标值 | 说明 |
|------|--------|------|
| DAU（日活跃用户） | **2 亿** | 包括国内外全部活跃用户。 |
| QPS（查询每秒） | **50,000 req/s** | 推荐请求峰值，包含首页、频道页、下拉加载等。 |
| 单次推荐延迟 | **≤ 100 ms**（99% 请求） | 包括召回、排序、返回全过程。 |
| 可用性 | **99.9%（月均）** | 计划维护窗口 ≤ 4 h/月。 |
| 存储容量 | **5 PB**（原始内容 + 索引 + 行为日志） | 预计每年增长 30%。 |
| 数据一致性 | **最终一致性** 对行为日志；**强一致性** 对用户画像（实时更新）。 |

## 5. 系统边界
**本题范围内**  
- 推荐核心流程：登录 → 兴趣模型 → 内容召回 → 排序 → 返回。  
- 实时特征计算、模型推理、流量控制、监控报警。  
- 基础设施选型（缓存、消息队列、存储、计算框架）以及扩展/容灾设计。

**不在本题范围**  
- 内容生产（编辑、UGC 上传、审核系统）。  
- 广告投放、商业化计费、收益结算。  
- 客户端 UI 细节、离线批量训练流水线的实现细节（仅需说明数据入口）。  
- 法律合规、隐私合规（除违规过滤需求外）。

## 6. 提示与追问
1. **数据倾斜与热点问题**：如果某些热门作者或热点事件导致召回请求集中到少数节点，你会如何进行负载均衡和热点分散？  
2. **模型热更新**：在保证服务不宕机的前提下，如何实现推荐模型的灰度发布和快速回滚？  
3. **容灾恢复**：当召回服务所在的数据中心故障时，系统如何在 5 秒内切换到备份中心，仍然满足 100 ms 延迟要求？

## 解题思路总览

1. **先把需求拆解**——把功能需求和非功能需求分别列出，明确哪些是必须的、哪些是可选的。  
2. **做规模估算**——把日活、QPS、存储等数字转化为硬件资源的大致需求，帮助后面的容量规划。  
3. **从最小可用系统（MVP）出发**——先画出最简化的单机/单进程实现，确保我们能把“推荐”完整跑通。  
4. **逐层演进到分布式高可用**——在 MVP 基础上加入缓存、消息队列、水平扩展、容灾等能力，形成完整的生产级架构。  
5. **每个模块落地到具体技术选型**——数据库、缓存、搜索/召回、模型推理、日志采集等都要给出实现方案和理由。  
6. **补全 API、数据模型、监控告警**——让面试官看到你不仅会画图，还会思考接口、协议、监控等细节。  
7. **准备常见追问**——热点、模型热更新、跨机房容灾等都是面试常点，这里提前准备好答案。  

> **核心思路**：**先满足功能，再满足性能** → **先满足性能，再满足可靠性**。在每一步都要问自己「如果不这么做，系统会出现什么问题？」。

## 第一步：理解需求与规模估算

### 1. 功能需求回顾

| 编号 | 关键点 | 需要实现的最小功能 |
|------|--------|-------------------|
| 1 | 登录 & 鉴权 | 统一的 UID，支持多种登录方式 |
| 2 | 实时兴趣模型 | 行为实时写入、用户画像即时更新 |
| 3 | 内容召回 & 过滤 | 快速返回候选集合，过滤违规、已读等 |
| 4 | 排序 & 打分 | 机器学习模型对候选打分 |
| 5 | 曝光监控 & 反馈 | 记录曝光、点击、停留等行为 |
| 6 | 个性化配置 | 用户可屏蔽/关注主题、作者 |

> **最小可用系统（MVP）** 只需要：  
> - 登录接口 → 用户画像查询 → 简单召回（基于标签） → 直接返回（不做复杂排序） → 行为日志写入。

### 2. 非功能需求关键指标

| 指标 | 目标 | 业务意义 |
|------|------|----------|
| DAU | 2 亿 | 系统规模基准 |
| QPS | 50 k req/s | 峰值流量，决定水平扩展规模 |
| 延迟 | ≤100 ms（99%） | 用户体验关键，必须在前端渲染前返回 |
| 可用性 | 99.9% 月均 | 业务容忍的停机时间 < 4 h/月 |
| 存储 | 5 PB | 包含原始内容、日志、索引 |
| 一致性 | 行为最终一致、画像强一致 | 行为可以稍后回放，画像必须实时用于推荐 |

### 3. 粗略容量估算（帮助后面选型）

| 项目 | 估算方式 | 结果 |
|------|----------|------|
| **推荐请求 QPS** | 50 k req/s ≈ 4.3 B/天 | 每天约 4.3 B 次推荐调用 |
| **单次召回候选数** | 100 条 × 4.3 B ≈ 430 B 条 | 召回层需要 **高并发读** |
| **行为日志写入** | 1 条曝光 + 0.2 条点击 ≈ 1.2 条/请求 | 约 5 B 条日志/天 |
| **用户画像更新** | 每条行为实时写入 → 5 B 条/天 | 需要 **低延迟、强一致** 的 KV 存储 |
| **内容库大小** | 5 PB（约 2 TB 文本 + 3 PB 视频） | 需要分布式对象存储（OSS/MinIO） |
| **索引体积** | 召回用倒排索引约 200 TB | 需要支持高并发查询的搜索引擎（Elasticsearch） |

> **如果不做这些估算**，很容易在后期发现“CPU/内存/磁盘不足”，导致系统崩溃或延迟失控。

## 第二步：高层架构设计

下面先给出 **最小可用系统（MVP）** 的结构图，然后逐步加入 **高可用、水平扩展、容灾** 的组件。

### 1. MVP 架构（单机/单进程）

```
[客户端] -> HTTP/HTTPS -> [API Gateway] -> 
   -> [Auth Service] (登录、签发 Token) 
   -> [Recommendation Service] 
          |- 调用 [User Profile Store] (Redis)
          |- 调用 [Simple Recall] (基于标签的本地索引)
          |- 返回候选列表
   -> [Log Collector] (写入本地文件或 MySQL)
```

- **优点**：逻辑完整，代码量最小，易于验证业务流程。  
- **缺点**：单点故障、无法支撑 50k QPS、无法满足 100 ms 延迟。

### 2. 生产级分布式架构（层次化）

```
+-------------------+      +-------------------+      +-------------------+
|   客户端 (App/Web) | --> |   API Gateway /   | --> |   流量控制 Layer  |
|                     |      |   Edge Proxy      |      | (RateLimiter, WAF)|
+-------------------+      +-------------------+      +-------------------+
                                    |
            +-----------------------+-----------------------+
            |                                               |
    +-------------------+                         +-------------------+
    |  Auth Service     |                         |  Config Service   |
    |  (OAuth, JWT)     |                         | (User prefs)      |
    +-------------------+                         +-------------------+
            |                                               |
            +-----------------------+-----------------------+
                                    |
                          +-------------------+
                          | Recommendation   |
                          |   Orchestrator   |
                          +-------------------+
          |-------------------|-------------------|-------------------|
          |                   |                   |                   |
   +-------------+   +---------------+   +----------------+   +--------------+
   |  Real‑time  |   |  Recall Service|   | Ranking Service|   |  Log Service |
   | Feature Pip|   | (ES/ANN)      |   | (Model Server) |   | (Kafka)      |
   +-------------+   +---------------+   +----------------+   +--------------+
          |                   |                   |
   +-------------+   +---------------+   +----------------+
   |  User Profile|   | Content Store |   | Model Store   |
   |   Store     |   | (OSS/HBase)   |   | (TensorFlow   |
   | (Redis+MySQL) |  +---------------+   | Serving)      |
   +-------------+                         +----------------+
```

**关键层级解释**：

| 层级 | 负责的职责 | 选型示例 | 为什么要这么划分 |
|------|------------|----------|-------------------|
| **API Gateway / Edge** | 统一入口、TLS 终止、限流、灰度发布 | Kong / APISIX | 让后端业务无需关心网络安全、流量治理 |
| **Auth Service** | 登录、鉴权、签发 JWT | Spring Boot + MySQL + Redis | 单独拆分方便水平扩展，且安全隔离 |
| **Config Service** | 用户个性化配置（屏蔽/关注） | MySQL + Cache | 配置变化不频繁，可使用强一致读 |
| **Recommendation Orchestrator** | 调度召回、排序、聚合，返回统一结果 | gRPC / Protobuf | 业务编排中心，易于插拔新模型 |
| **Real‑time Feature Pipeline** | 行为实时写入、特征计算、画像更新 | Kafka + Flink / Spark Structured Streaming | 高吞吐、低延迟、Exactly‑Once 语义 |
| **Recall Service** | 从全量内容库快速召回候选 | Elasticsearch + Faiss (ANN) | 支持倒排索引 + 向量相似搜索，满足热点召回 |
| **Ranking Service** | 模型推理、打分、排序 | TensorFlow Serving / TorchServe + gRPC | GPU/CPU 可弹性伸缩，支持在线 A/B |
| **Log Service** | 曝光/点击/停留日志收集 | Kafka + HDFS / Iceberg | 最终一致、离线训练入口 |
| **User Profile Store** | 实时画像（特征向量） | Redis（热点）+ MySQL（持久） | 读写分离、强一致（使用 Redis Cluster + 写后同步 MySQL） |
| **Content Store** | 原始内容、图片/视频文件 | OSS / Ceph / MinIO + CDN | 大容量、低成本、读多写少 |
| **Model Store** | 模型二进制、配置、版本信息 | Git + MinIO + MySQL | 统一管理，支持灰度发布 |

> **如果把所有组件都放在同一个进程里**，最直观的问题是：**无法水平扩展、单点故障、无法满足 100 ms 的延迟**。分层设计把不同的 **资源需求**（CPU、IO、网络）分离，才能针对性扩容。

## 第三步：数据库设计

> **原则**：**读写分离 + 垂直拆分**，每类数据选最合适的存储。

### 1. 用户相关

| 表/集合 | 主键 | 主要字段 | 说明 |
|--------|------|----------|------|
| `users` (MySQL) | `uid` | `phone`, `oauth_id`, `created_at`, `status` | 永久存储用户基本信息，强一致 |
| `user_token` (Redis) | `uid` | `jwt_token`, `expire_ts` | 登录态缓存，TTL 1 h |
| `user_profile` (Redis Hash) | `uid` | `interest_vector`(float[128]), `recent_clicks`(list), `last_update_ts` | 实时画像，热点缓存 |
| `user_config` (MySQL + Cache) | `uid` | `blocked_topics`, `followed_authors`, `pref_weights` | 低频变更，可使用二级缓存 |

**写入路径**：用户行为 → Kafka → Flink → 更新 `user_profile` (Redis) 同步到 MySQL (持久化)。  
**强一致实现**：使用 **Redis事务 + Write‑Through**，确保写成功后才返回给业务层。

### 2. 内容（文章/视频）

| 表/集合 | 主键 | 主要字段 | 说明 |
|--------|------|----------|------|
| `contents` (MySQL) | `cid` | `title`, `author_id`, `category`, `publish_ts`, `status` | 基础元数据，查询时使用 |
| `content_vector` (Faiss / ES) | `cid` | `embedding`(float[256]) | 用于向量召回（相似度） |
| `content_text` (OSS) | `cid` | `text_path` | 大文本或视频文件，CDN 加速 |
| `content_status` (MySQL) | `cid` | `is_violation`, `is_deleted` | 过滤用的标记位 |

**召回索引**：  
- **倒排索引**（Elasticsearch）存储 `category`、`tags`、`author_id` 等结构化字段。  
- **向量索引**（Faiss/ANN）存储 `embedding`，支持相似度召回。  
- 两者 **联合查询**（过滤 + 向量）得到候选集合。

### 3. 行为日志

| Topic | 消息结构 | 目的 |
|-------|----------|------|
| `exposure_log` | `{uid, cid, ts, position}` | 曝光监控、CTR 计算 |
| `click_log` | `{uid, cid, ts, dwell_time}` | 反馈特征、模型训练 |
| `like_log` | `{uid, cid, ts}` | 强化兴趣模型 |
| `share_log` | `{uid, cid, ts}` | 增强热点信号 |

**持久化**：Kafka → Flink → 写入 HDFS/Iceberg → 离线特征工程。  
**实时监控**：Kafka → Prometheus Exporter → Grafana。

### 4. 召回 & 排序缓存

| 缓存 | Key | Value | 失效策略 |
|------|-----|-------|----------|
| `recall:uid:page` | `uid:page` | `[cid1,cid2,...]` (JSON) | 5 min TTL（热点用户可命中） |
| `rank:uid:session` | `uid:sessionId` | 排序后列表 | 1 min TTL（防止同一次滑动重复计算） |

**为什么要缓存**：召回和排序往往在短时间内会被同一用户多次请求（下拉加载），缓存可把 **CPU/模型推理** 成本降到 10% 以下。

## 第四步：核心 API 设计

> **接口设计原则**：**RESTful + Protobuf (gRPC)** 双协议并存。移动端倾向使用 **HTTPS+JSON**，内部服务间使用 **gRPC**，既易调试又高效。

### 1. 登录 & 鉴权

| 方法 | 路径 | 请求 | 响应 | 说明 |
|------|------|------|------|------|
| `POST` | `/api/v1/login` | `{type:"phone|oauth", phone?, oauth_token?}` | `{uid, access_token, refresh_token, expires_in}` | 登录成功返回 JWT，`access_token` 放在 `Authorization: Bearer` 头部 |
| `POST` | `/api/v1/refresh` | `{refresh_token}` | `{access_token, expires_in}` | 刷新 token，防止频繁登录 |

### 2. 推荐查询

| 方法 | 路径 | 请求 | 响应 | 关键字段 |
|------|------|------|------|----------|
| `GET` | `/api/v1/recommend` | `uid`, `page=1`, `page_size=20`, `scene=home|channel` | `{items:[{cid, title, cover_url, author, score}], next_page_token}` | **实时画像** 在服务端读取后参与召回；**next_page_token** 用于下拉加载防止分页失效 |
| `POST` | `/api/v1/recommend/batch` | `{uid, scenes:[home, channel]}` | `{scene: items[]}` | 批量获取多场景结果，减少网络往返（适用于首页+频道同时渲染） |

### 3. 行为上报（曝光/点击等）

| 方法 | 路径 | 请求 | 响应 |
|------|------|------|------|
| `POST` | `/api/v1/feedback/exposure` | `{uid, cid, position, ts}` | `{status:"ok"}` |
| `POST` | `/api/v1/feedback/click` | `{uid, cid, ts, dwell_time}` | `{status:"ok"}` |
| `POST` | `/api/v1/feedback/like` | `{uid, cid, ts}` | `{status:"ok"}` |
| **实现**：客户端直接 **异步** 调用，服务端把请求写入 **Kafka**（防止业务侧阻塞），返回立即成功。

### 4. 用户个性化配置

| 方法 | 路径 | 请求 | 响应 |
|------|------|------|------|
| `GET` | `/api/v1/user/config` | `uid` | `{blocked_topics:[], followed_authors:[]}` |
| `POST` | `/api/v1/user/config` | `{uid, blocked_topics?, followed_authors?}` | `{status:"ok"}` |

### 5. 健康检查 & 监控

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/healthz` | 返回 200 表示进程存活 |
| `GET` | `/readyz` | 检查依赖（Redis、MySQL、Kafka）是否可用 |
| `GET` | `/metrics` | Prometheus 格式指标（QPS、延迟、错误率） |

> **不提供**：直接暴露内部数据库查询接口。这样做会导致 **安全泄漏、耦合度高**。

## 第五步：详细组件设计

下面把核心组件拆解到实现细节，包括 **数据流向、技术选型、容错机制**。

### 1. 登录 & 鉴权 Service

- **技术栈**：Spring Boot + Spring Security + MySQL（持久化用户信息） + Redis（Token 缓存）。
- **流程**：
  1. 客户端提交手机号或 OAuth Token。
  2. Service 调用第三方 OAuth（如微信）或验证码服务验证。
  3. 若用户不存在则 **创建**（写入 MySQL）。
  4. 生成 **JWT**（payload 包含 `uid`、`exp`），写入 Redis（键：`uid:token`，TTL=1h）实现 **单点登录**（登录即踢掉旧 token）。
  5. 返回 `access_token` 给客户端。
- **高可用**：使用 **Nginx+K8s** 做水平扩容，MySQL 主从+读写分离，Redis Sentinel 或 Cluster。

### 2. 实时特征计算（行为 → 画像）

#### 数据流

```
[客户端] -> Kafka (topic: user_behavior) -> Flink Job -> 
   1) 写入 User Profile Store (Redis)   // 实时画像
   2) 写入 HDFS/Iceberg (Parquet)      // 离线特征
   3) 写入 MySQL (行为表)               // 备份
```

- **Kafka**：分区键为 `uid`，保证同一用户行为有序。分区数 **≥ QPS/10k**（假设每秒 5 万请求，分区 10‑20）。
- **Flink**：
  - **Exactly‑Once**：使用 **Kafka Transactional** + **Checkpoint**。
  - **窗口**：滑动窗口（最近 1h、24h）计算点击率、兴趣向量增量。
  - **模型**：轻量的 **Embedding + Sum**（如 Word2Vec 版）更新 `interest_vector`（128 维）。
- **Redis 更新**：使用 **Hash** 存储向量，`HSET uid interest_vector <base64>`；写完后 **publish** 一个 `profile_updated` 事件（供召回服务热更新缓存）。
- **容错**：Flink Job 多实例（Active‑Standby），Checkpoint 到 **HDFS**。

### 3. 召回 Service

#### 目标

- **毫秒级**返回 **上百** 条候选内容。
- 支持 **结构化过滤**（标签、作者、违规）+ **向量相似**（兴趣向量）。

#### 技术选型

| 需求 | 方案 | 说明 |
|------|------|------|
| 结构化倒排检索 | Elasticsearch (ES) | 支持分词、布尔过滤、聚合，水平扩展。 |
| 向量 ANN 检索 | Faiss (GPU) + ES Vector Plugin | 将 `interest_vector` 与 `content_vector` 做余弦相似度，返回 Top‑K。 |
| 高并发读 | ES 集群 + 读写分离 | 只查询，写入（新内容）走单独的 **写入节点**。 |
| 热点分散 | **热点分片** + **路由**（把同一作者或热门标签的文档分散到不同分片） | 防止单个 ES 分片成为瓶颈。 |

#### 流程

```
1. Recommendation Orchestrator 拉取用户画像 (Redis)
2. 构造 ES Query:
   - bool filter: status=online, NOT in blocked_topics/blocked_authors
   - knn: {vector: interest_vector, k: 200}
3. ES 返回候选 CID 列表 (带 score)
4. 调用过滤服务（去重、已读过滤） → 形成候选集合 (约 100 条)
```

#### 缓存

- **热点用户**（活跃度 Top 5%）的召回结果放入 **Redis**（key: `recall:{uid}:{scene}`），TTL 5 min。
- **热点内容**（阅读量 Top 0.1%）在 ES 中使用 **routing** 分散到多个分片。

### 4. 排序（Ranking） Service

#### 需求

- **模型**：GBDT、DeepFM、或最近流行的 **Two‑Tower** 向量模型。  
- **实时推理**：每条请求的候选数 **≈ 100**，每次推理必须 ≤ **5 ms**。  
- **灰度发布**：新模型上线后需要 **A/B 流量**，并能快速回滚。

#### 实现方案

| 组件 | 技术 | 关键配置 |
|------|------|----------|
| 模型存储 | **MinIO** + **MySQL**（版本表） | 二进制文件 `.pb`、`.pt` |
| 推理服务 | **TensorFlow Serving**（CPU/GPU）或 **Triton Inference Server** | 支持 **gRPC**、**REST**，多模型多版本 |
| 负载均衡 | **Envoy** + **Istio**（流量分配） | 按权重路由到不同模型实例 |
| 缓存 | **Redis** (key:`rank:{uid}:{cid}`) | 若同一用户短时间内多次请求相同候选，可直接返回已有打分 |

#### 推理流程

```
1. Recommendation Orchestrator 把候选列表 + 用户画像一起发给 Ranking Service (gRPC)
2. Ranking Service 将向量特征拼接（用户向量 + 内容向量）喂给模型
3. 模型返回每条内容的 **CTR 预估分**，加权后排序
4. 返回前 20 条（或根据 pageSize）给 Orchestrator
```

#### 灰度发布 & 回滚

- **模型版本表**（MySQL）记录 `model_id, version, status, rollout_percent`。  
- **Istio VirtualService** 根据 `rollout_percent` 将请求分流到不同 `model_id` 的实例。  
- 若新模型表现异常，只需把 `status=paused`，流量自动回到旧模型，**几秒内完成**。

### 5. 曝光 & 行为日志采集

- **客户端**：在渲染完成后 **异步** POST `/feedback/exposure`（批量一次性上报 10 条）。  
- **服务端**：收到后立即 **写入 Kafka**（`exposure_log`），返回成功。  
- **下游**：Kafka → Flink → 写入 **Iceberg** 表（列式存储），供离线模型训练使用。  
- **实时监控**：Flink 中聚合 `exposure` / `click`，实时计算 **CTR**，推送到 **Prometheus**，形成仪表盘报警（CTR 低于阈值告警）。

### 6. 个性化配置 Service

- **存储**：MySQL（持久） + Redis（热点缓存）。  
- **API**：`GET/POST /user/config`。  
- **过滤**：召回时读取用户配置，加入 **bool must_not** 条件（如 `author_id NOT IN blocked_authors`），确保不返回用户屏蔽的内容。

### 7. 监控、报警、日志

| 维度 | 关键指标 | 监控工具 |
|------|----------|----------|
| 系统层 | CPU/内存/磁盘IO、网络流量 | Prometheus + Node Exporter |
| 服务层 | QPS、成功率、错误率、平均/99% 延迟 | Prometheus + Grafana |
| 业务层 | 曝光量、点击率（CTR）、转化率、模型分布 | Grafana + ClickHouse 实时查询 |
| 异常 | Kafka 堆积、Flink 延迟、Redis 命中率下降 | Alertmanager 邮件/钉钉/Slack |

> **不做监控** 的后果：系统出问题时只能靠“用户投诉”发现，恢复时间会从 **分钟** 变成 **小时**，直接影响 SLA。

## 第六步：扩展性与高可用设计

### 1. 水平扩展（Scale‑out）

| 组件 | 扩容方式 | 关键指标 |
|------|----------|----------|
| API Gateway | 增加 Nginx/Envoy 实例 + LVS 负载均衡 | QPS |
| Auth Service | K8s Deployment → 多副本，Session 通过 JWT 无状态 | TPS |
| Redis (Profile) | **Cluster**（分片）+ 主从复制 | 读写吞吐 |
| Kafka | 增加 **partition**，每个分区对应 **Broker**，使用 **MirrorMaker** 跨机房复制 | 消费延迟 |
| Flink | **TaskManager** 横向扩容，State Backend 为 **RocksDB**（本地磁盘） | 事件处理延迟 |
| Elasticsearch | 按需增删 **节点**，使用 **shard allocation** 动态平衡 | 查询 QPS |
| TensorFlow Serving | **ReplicaSet** + **HorizontalPodAutoscaler**，GPU/CPU 自动弹性 | 推理 QPS |

> **为什么要水平扩展**：单机无法支撑 50 k QPS，且故障时整个系统不可用。水平扩展配合 **无状态** 设计是保证 **99.9%** 可用的根本。

### 2. 数据热点与分布式均衡

- **热点作者 / 事件**：把同一作者的内容使用 **自定义路由键**（`author_id % shard_num`）分散到不同 ES 分片。  
- **热点用户画像**：对活跃用户使用 **专属 Redis 分片**（如 `uid % 128`），并在 Flink 中使用 **KeyBy** 按 `uid` 分区，确保同一用户状态只落在一个 Task。  
- **写入热点**：Kafka 分区键使用 **uid**，但热点用户会导致单分区写入过载。解决办法：**二级分区**——`uid_hash = hash(uid) % 128`，再加 **随机前缀**，保证写入均匀。

### 3. 容灾恢复（跨机房）

| 场景 | 方案 | 切换时间目标 |
|------|------|--------------|
| **召回服务** 故障 | 在另一机房部署 **同等规模 ES + Faiss**，使用 **DNS 轮询** + **Global Load Balancer**（如 Alibaba Cloud SLB）实现 **主动健康检查**。| ≤ 5 s（DNS TTL 1 s） |
| **Kafka** 故障 | **MirrorMaker** 同步到备份集群；消费者支持 **Fail‑over**（消费组在两套集群间切换）。| ≤ 10 s |
| **Redis** 故障 | **Redis Cluster** 跨可用区（跨 AZ）复制，故障转移通过 **Sentinel** 完成。| ≤ 3 s |
| **模型服务** 故障 | 多模型实例分布在不同机房，**Istio** 按权重路由；若发现异常错误率升高，自动降级为 **默认模型**（本地缓存）。| ≤ 2 s |

> **不做跨机房同步** 会导致机房宕机后 **全部流量** 被阻塞，业务不可用时间会远超 SLA。

### 4. 灰度发布 & A/B 测试

- **流量分层**：`10%` 新模型 → `90%` 老模型 → 通过 **Istio** 控制。  
- **指标监控**：实时比较 **CTR、曝光量、错误率**；若新模型指标跌破阈值，自动 **回滚**。  
- **实验平台**：使用 **OpenTelemetry** + **Prometheus** 收集每个实验组的细粒度指标，存入 **ClickHouse** 供离线分析。

### 5. 安全与合规

- **Token 加密**：使用 **RSA256** 私钥签名，过期时间短（1 h）。  
- **数据脱敏**：日志中不记录 `uid` 明文，统一使用 **hash(uid)**。  
- **访问控制**：API Gateway 配置 **IP 白名单**、**RateLimiter**（每用户 30 req/s）防止爬虫攻击。  

> **忽略安全** 会导致 **数据泄露**，在面试中是致命错误。

## 第七步：常见面试追问与回答

下面列出几类常被追问的点，并给出 **思路清晰、结构化** 的答案示例。

### 1. 数据倾斜与热点问题

> **问**：如果某些热门作者或热点事件导致召回请求集中到少数节点，你会如何进行负载均衡和热点分散？

**回答要点**：

1. **召回层分片**  
   - 使用 **自定义路由**：`author_id % shard_num` 把同一作者的内容分散到不同 ES 分片。  
   - 对热点事件（如“北京奥运会”）使用 **标签哈希** 再加 **随机前缀**，防止单一标签占满一个分片。

2. **查询路由**  
   - 在召回服务层面 **并行** 发起多路查询：对热点标签对应的多个分片并行查询，随后在业务层做 **合并去重**。  

3. **缓存**  
   - 对热点用户的召回结果做 **短期缓存**（TTL 5 min），降低对后端查询的压力。  

4. **监控与自动扩容**  
   - 在 Grafana 中监控每个分片的 QPS、查询延迟，配合 **K8s HPA** 动态扩容 ES 节点或 **Shard Rebalance**。  

> **不做分片**：热点请求会把单个节点打满，导致 **查询超时**，直接影响 100 ms SLA。

### 2. 模型热更新

> **问**：在保证服务不宕机的前提下，如何实现推荐模型的灰度发布和快速回滚？

**回答结构**：

1. **模型管理**  
   - 所有模型二进制存放在 **MinIO**，MySQL 记录 `model_id, version, status, rollout_percent`。  

2. **推理服务**  
   - 使用 **TensorFlow Serving**（支持多模型、多版本），每个模型实例对应一个 **model_id**。  

3. **流量路由**  
   - 通过 **Istio VirtualService** 按 `rollout_percent` 将请求分流到不同 `model_id` 的实例。  

4. **灰度监控**  
   - 为每个模型实例打上 **Prometheus label**，实时监控 `ctr`, `error_rate`。  

5. **回滚**  
   - 若新模型指标下降，直接在 MySQL 将 `rollout_percent` 设为 `0`，Istio 自动把流量切回旧模型，整个过程几秒内完成。  

> **不使用流量路由**：只能全量下线旧模型，导致 **服务不可用** 或 **业务冲击**。

### 3. 跨机房容灾切换

> **问**：当召回服务所在的数据中心故障时，系统如何在 5 秒内切换到备份中心，仍然满足 100 ms 延迟要求？

**回答要点**：

1. **双活部署**  
   - 在两个可用区分别部署完整的 **召回集群**（ES+Faiss）和 **Redis**。  

2. **全局负载均衡**  
   - 使用 **DNS 低 TTL（1 s）+ Anycast IP** 或 **云厂商 Global Load Balancer**，健康检查探测召回服务 `/healthz`。  

3. **数据同步**  
   - **内容索引**通过 **跨集群复制**（ES Cross‑Cluster Replication）实时同步。  
   - **热点缓存**（Redis）使用 **双写 + 异步复制**，保证两边一致性。  

4. **故障检测 & 自动切换**  
   - 健康检查失败后，GLB 自动把流量切换到备份机房，客户端 DNS TTL 失效后重新解析。  
   - 切换后 **热点缓存** 可能出现 **短暂 miss**，但会在几毫秒内回源到 DB，仍在 100 ms 预算内。  

> **不做双活**：机房故障导致全局不可用，恢复时间可能是 **分钟级**，违背 99.9% SLA。

### 4. 如何保证 100 ms 延迟？

**回答思路**：

1. **延迟预算拆解**  
   - 网络 + 负载均衡：5 ms  
   - Auth / Token 校验：2 ms  
   - 画像读取（Redis）：3 ms  
   - 召回查询（ES+ANN）：30 ms  
   - 过滤 & 去重：5 ms  
   - 排序模型推理（GPU）：10 ms  
   - 响应序列化 & 传输：5 ms  
   - **总计** ≈ 70 ms，留出余量。  

2. **关键技术**  
   - **缓存**（Redis）消除 DB 访问。  
   - **并行化**：召回 + 排序可异步并行，使用 **Futures**。  
   - **热点缓存**：热点用户/内容直接命中。  
   - **批量 RPC**：一次网络请求获取所有候选内容的元信息。  

3. **监控**  
   - 在 Grafana 中拆分每一步的延迟，设置 **99%** 告警阈值，快速定位瓶颈。  

> **不拆解延迟**：只能在整体 SLA 报告中看到异常，难以定位具体模块问题。

## 心得与反思

### 1. 本题最难的设计决策

| 决策 | 思考过程 | 最终选择 |
|------|----------|----------|
| **实时画像的存储与一致性** | - 强一致性需要同步写入，但实时写入大量（5 B 行为/天）会导致瓶颈。<br>- 弱一致性会导致用户最近的兴趣不被及时利用，影响推荐质量。 | 采用 **Redis 写‑Through + MySQL 持久化**，配合 **Flink Exactly‑Once**，在 **Redis** 中实现强一致（事务+回写），在 **MySQL** 中做持久化备份，实现 **强一致 + 最终一致** 的折中。 |
| **召回层的热点分散** | - 直接把所有热点作者的文档放在同一分片，查询会集中在少数机器。<br>- 采用 **全局二级索引** 会增加系统复杂度。 | 使用 **自定义路由键**（`author_id % shard_num`）把同一作者分散到不同 ES 分片，并在召回服务层面 **并行查询多个分片**，同时配合 **热点缓存**。 |

### 2. 新手最容易犯的错误（至少 2 条）

1. **把所有组件都写在同一个进程里**  
   - 结果：单点故障、无法横向扩容、难以满足 50 k QPS 与 100 ms 延迟。  
   - 正确做法：把 **无状态** 的业务（API、Orchestrator）和 **有状态** 的存储（Redis、Kafka）分离，使用 **容器化 + K8s** 实现弹性伸缩。

2. **忽略监控与告警**  
   - 只关注实现功能，忘记给每个关键指标打点。系统出现卡顿时只能靠人工排查。  
   - 正确做法：在每个服务入口/出口埋 **Prometheus** 指标（QPS、Latency、ErrorRate），并在 **Grafana** 中配置 **99% 延迟**、**错误率** 告警阈值。

### 3. 学习建议和可延伸的方向

| 方向 | 推荐学习资源 | 说明 |
|------|--------------|------|
| **分布式缓存（Redis）** | 《Redis 实战》 + 官方文档 | 理解缓存穿透、击穿、雪崩的防御措施。 |
| **流式计算（Flink）** | 《Streaming Systems》, Flink 官方教程 | 掌握 Exactly‑Once、状态后端、窗口计算。 |
| **搜索与向量召回** | Elasticsearch 官方文档、FAISS 论文与实战 | 学会倒排索引、向量 ANN、跨集群复制。 |
| **模型服务化** | TensorFlow Serving / Triton 官方文档 | 了解多模型版本管理、GPU 推理优化。 |
| **可观测性** | 《Prometheus 与监控实战》, OpenTelemetry | 统一指标、日志、链路追踪的实现方式。 |
| **容灾与多活** | 《Designing Data-Intensive Applications》章节 | 学习 CAP、跨机房复制、全局负载均衡。 |

> **练手项目**：可以先搭建一个 **小型推荐系统**（用户画像存 Redis，召回使用 Elasticsearch，排序直接用 LightGBM），再逐步加入 Kafka+Flink、模型服务、跨机房同步，体会从 **单机 MVP** 到 **生产级高可用** 的演进过程。

本篇答案从 **需求拆解 → 规模估算 → MVP → 分层架构 → 细化到每个组件的技术选型与实现细节 → 高可用扩展 → 面试追问**，一步一步把“怎么把 2 亿 DAU、50 k QPS、≤ 100 ms 的推荐系统”完整地画出来。

- 为什么选这个技术（性能、成本、生态）？

- 怎么保证高可用、监控、容灾？

这样既展现了你的 **系统全局观**，也体现了 **细节落地能力**，会让面试官对你印象深刻。祝你面试顺利 🚀！
