# 短视频推荐系统设计题

## 1. 题目背景
短视频推荐系统负责为海量用户实时推送符合兴趣的短视频流，核心目标是提升用户粘性和平台内容消费。系统需要在毫秒级别完成个性化排序，并支持日活跃用户数（DAU）级别的高并发请求。

## 2. 面试场景设定
> **面试官**：  
> “现在请你设计一个 **短视频推荐系统**，要求能够支撑 **每日数千万活跃用户**，在 **用户打开 APP 后 200 ms** 内返回一个包含 20 条视频的推荐列表。请从整体架构出发，说明你会如何划分模块、选型技术以及如何满足高并发和低延迟的需求。”

## 3. 功能性需求
1. **获取首页推荐流**：用户打开 APP 时返回个性化的 20 条视频列表。  
2. **视频点赞/点踩**：用户对推荐视频进行喜欢或不喜欢的反馈，系统实时更新用户兴趣模型。  
3. **浏览历史记录**：记录用户观看的视频 ID、观看时长、停留时长等，用于后续推荐。  
4. **兴趣标签管理**：系统根据用户行为动态生成/更新兴趣标签（如音乐、运动、搞笑），并在推荐时加权使用。  
5. **冷启动处理**：对新注册用户或新上传视频提供初始推荐策略（基于相似用户、内容特征等）。  
6. **推荐结果实时刷新**：在用户滑动观看过程中（如每滑动 5 条视频），实时触发增量推荐，保持内容新鲜感。

## 4. 非功能性需求
| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU** | 5,000 万 | 高峰期同时在线约 2,000 万 |
| **QPS（推荐查询）** | 30,000 QPS | 假设每位活跃用户平均 6 次请求/小时 |
| **单次推荐响应时延** | ≤ 200 ms（99% 响应 ≤ 300 ms） | 包括模型推理、排序、缓存命中等 |
| **可用性** | 99.9%（月度 SLO） | 单点故障容忍，自动故障转移 |
| **存储规模** | 用户行为数据 5 PB，视频元数据 500 TB | 采用冷热分层存储，近期数据放在 SSD，历史数据放在对象存储 |
| **模型更新频率** | 实时增量更新 + 每日离线全量训练 | 实时特征流式计算，离线模型每 24 小时更新一次 |

## 5. 系统边界
**本题需要设计的范围**  
- 用户行为采集、特征实时计算、离线特征批处理。  
- 推荐召回、粗排、精排三层架构及其数据流。  
- 缓存层（热点推荐、热点视频）设计。  
- 高可用部署、容灾、监控告警框架。  
- 与 **视频上传、播放**、**用户登录/社交** 等业务的接口定义（但不需实现这些业务本身）。

**不在本题范围**  
- 视频内容审核、版权管理、广告投放系统。  
- 实际的视频存储、转码、 CDN 分发。  
- 详细的机器学习模型结构（只要求模型输入/输出、训练/推理方式）。  
- 前端 UI/交互细节（只关注后端 API 与数据流）。

## 6. 提示与追问
1. **数据倾斜如何处理？**  
   - “如果某些热门视频的点击量占比过高，会导致召回阶段热点视频压垮某些节点，你会采用哪些方法缓解？”  

2. **实时特征与离线特征的融合策略？**  
   - “在精排阶段，你会如何权衡实时行为特征（如最近 5 分钟的点击）与离线画像特征？”  

3. **缓存失效和一致性问题**  
   - “推荐结果会被缓存到热点缓存层，如何设计缓存失效策略，确保用户兴趣变化后能快速看到新的推荐？”

# 短视频推荐系统设计完整解答  

> **适读对象**：刚入行的后端同学，对系统设计几乎没有实战经验。  
> **目标**：从最小可跑的 MVP（最小可用系统）一步步展开，解释每一步“为什么”，帮助你形成完整的系统设计思路，而不是仅记住答案。

## ## 解题思路总览  

1. **先把需求拆成「功能」+「非功能」**，再把「非功能」转化为具体的数值（DAU、QPS、延迟等），这一步决定后面所有容量估算和技术选型。  
2. **画出「数据流」**：用户行为 → 特征计算 → 召回 → 粗排 → 精排 → 推荐返回。把每一步的输入/输出、处理时长、依赖的系统都标出来。  
3. **从 MVP 开始**：只需要一个单机服务完成「召回 + 排序」并返回 20 条视频。  
4. **逐层加入**：  
   - **缓存层**（热点推荐、热点视频）  
   - **离线特征批处理**（MapReduce / Spark）  
   - **实时特征流计算**（Flink / Spark‑Streaming）  
   - **召回、粗排、精排三层**（分布式 RPC）  
   - **高可用/容灾**（多机房、读写分离、限流）  
5. **每加入一个新组件，都要问自己**：  
   - *为什么需要它？*（瓶颈、可靠性、扩展性）  
   - *不加会出现什么问题？*  
   - *选哪种技术最合适？*（成本、团队熟悉度、生态）

## ## 第一步：理解需求与规模估算  

### 1. 功能性需求（复述）

| 功能 | 关键点 |
|------|--------|
| 首页推荐流 | 20 条视频，个性化，200 ms 返回 |
| 点赞/点踩 | 实时写入用户兴趣模型 |
| 浏览历史 | 记录观看时长、停留时长 |
| 兴趣标签 | 动态生成/更新，权重参与排序 |
| 冷启动 | 对新用户/新视频提供可接受的推荐 |
| 实时刷新 | 每滑动 5 条触发一次增量推荐 |

### 2. 非功能性指标转化为容量

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 5,000 万 | 高峰同时在线约 2,000 万 |
| **每用户请求频次** | 6 次/小时 | 每天 144 次/用户 |
| **QPS（推荐查询）** | 2,000 万 × 6 / 3600 ≈ **33,333 QPS** | 取保守值 30,000 QPS |
| **返回 20 条视频** | 每条约 200 KB（meta+封面） | 单次响应 ≈ 4 MB（实际压缩后 <1 MB） |
| **总流量** | 30,000 QPS × 1 MB ≈ **30 GB/s** | 需要 CDN + 负载均衡 |
| **行为日志** | 假设每用户 50 条行为/天，日志 200 B/条 | 5,000 万 × 50 × 200 B ≈ **500 GB/天** ≈ **180 TB/年**（实际会更大，考虑视频观看时长等） |
| **用户画像** | 1 KB/用户（离线特征） | 5,000 万 × 1 KB ≈ **50 TB** |
| **视频元数据** | 2 KB/视频，假设 10 亿视频 | 2 KB × 10⁹ ≈ **2 PB**（冷热分层） |

> **小结**：  
> - **查询层（推荐 API）** 需要 **毫秒级**、**高并发**，是系统的「前端」瓶颈。  
> - **写入层（行为采集）** 需要 **高吞吐**、**低延迟**，但对时效要求稍低（秒级），可以使用异步日志。  
> - **离线特征** 与 **实时特征** 必须并行跑，后者负责最近几分钟的兴趣变化，前者负责稳态画像。

### 1. 整体层次结构（从左到右）

```

│  前端 APP / H5   │←→│   API Gateway (L4)    │
└─────▲───────▲─────┘   └───────▲───────▲───────┘
      │       │                 │       │
      │       │                 │       │

      │   │ 负载均衡│   │   推荐服务层 │ │
      │   └───▲───┘   └───────▲───────┘ │
      │       │               │       │
      │   ┌───┴───┐   ┌───────┴───────┐ │
      │   │ 缓存层 │   │   召回/粗排/精排 │ │
      │   └───▲───┘   └───────▲───────┘ │
      │       │               │       │
      │   ┌───┴─────┐   ┌─────┴─────┐ │
      │   │   实时特征  │   │ 离线特征  │ │
      │   │   计算层   │   │ 批处理层 │ │
      │   └───▲───────┘   └─────▲─────┘ │
      │       │               │       │
      │   ┌───┴───────┐   ┌─────┴───────┐
      │   │ 行为日志收集 │   │ 元数据存储  │
      │   └─────▲──────┘   └─────▲───────┘
      │         │                │
   ┌──┴─────┐ ┌─┴─────┐      ┌────┴─────┐
   │ Kafka │ │ Flink │      │ HBase /  │
   └───────┘ └───────┘      │  ClickHouse│
                              └───────┘
```

> **层次说明**  

| 层级 | 主要职责 | 关键技术（示例） |
|------|----------|-------------------|
| **API Gateway / L4** | 统一入口、鉴权、限流、灰度发布 | Kong / Nginx + Lua |
| **负载均衡** | 将请求分发到多实例的推荐服务 | LVS / Nginx / Consul DNS |
| **推荐服务层** | 业务入口，拆分为 **缓存层** + **召回/粗排/精排** 三层 RPC | Spring Boot / Go + gRPC |
| **缓存层** | 热点推荐、热点视频、用户画像快照 | Redis Cluster (LRU+TTL) / CDN（边缘缓存） |
| **召回层** | 基于粗粒度特征快速返回 **数千** 候选视频 | **基于向量检索**（FAISS、Milvus）+ 倒排索引（Elasticsearch） |
| **粗排层** | 对召回的候选进行 **千级** 排序，使用轻量模型 | XGBoost / LightGBM 在线服务（BertServing） |
| **精排层** | 最终 **20 条** 排序，使用深度学习模型（CTR、兴趣流） | TensorFlow Serving / Pytorch‑Serve |
| **实时特征计算** | 计算最近 5~10 分钟的点击、停留等特征，写入 **用户画像缓存** | Flink / Spark‑Streaming + Redis |
| **离线特征批处理** | 每日离线画像、视频特征、召回索引重建 | Spark + Hive / HDFS + ClickHouse |
| **行为日志收集** | 收集点击、播放、点赞等，保证不丢失 | Kafka（高可靠）+ MirrorMaker（跨机房） |
| **元数据存储** | 视频基本信息、标签、时长、上传时间等 | MySQL（业务）+ HBase（宽表）+ ClickHouse（分析） |

### 2. 为什么要分层？

- **缓存层** 能把 **99%** 的热点请求在 **<5 ms** 直接返回，降低后端算力压力。  
- **召回/粗排/精排** 三层分别对应 **不同的时延/计算强度**：召回只要 **毫秒级**，粗排 **10 ms**，精排 **≤ 30 ms**。如果把所有逻辑塞到同一个服务，CPU、内存、模型加载会相互争抢，难以保证 200 ms SLA。  
- **实时特征** 与 **离线特征** 分离，实时流负责“新鲜度”，离线批处理负责“全局性”。两者合并在精排阶段即可得到既新鲜又稳健的排序分数。

### 3. 初始 MVP（最小可跑系统）

| 组件 | 实现方式 |
|------|----------|
| API + 负载均衡 | 单机 Spring Boot + Nginx |
| 推荐逻辑 | 只用 **召回 + 简单规则排序**（基于用户兴趣标签 + 视频热度） |
| 缓存 | 本地 LRUMap（后续替换为 Redis） |
| 行为日志 | 写入本地文件 → 后期迁移到 Kafka |
| 离线特征 | 手工脚本生成 CSV → 导入 MySQL |

> **目标**：在本地机器上完成「打开 APP → 返回 20 条视频」的全链路，验证业务模型和数据结构。之后再逐步拆分、扩容。

## ## 第三步：数据库设计  

### 1. 数据模型划分原则  

| 类别 | 访问模式 | 读写比例 | 推荐技术 |
|------|----------|----------|----------|
| **用户画像（离线）** | 按用户查询，少量更新 | 读 99% / 写 1% | HBase / Cassandra（宽表） |
| **用户画像（实时）** | 读写频繁（秒级） | 读/写均衡 | Redis（Hash）+ TTL |
| **视频元数据** | 按视频 ID 查询，批量读取 | 读多写少 | MySQL（业务）+ ClickHouse（分析） |
| **行为日志** | 顺序写，后端批处理 | 写 100% | Kafka（topic）+ HDFS（冷存） |
| **召回索引** | 向量相似搜索、倒排查询 | 读频繁，偶尔全量重建 | Milvus / Faiss + Elasticsearch |
| **模型参数** | 在线服务读取，周期更新 | 读多写少 | OSS / NFS + 自动同步脚本 |

### 2. 关键表/宽表设计（示例）

#### 2.1 用户画像（离线） `user_profile`

| 列名 | 类型 | 说明 |
|------|------|------|
| `user_id` | BIGINT | 主键 |
| `interest_vector` | BINARY(1024) | 128‑dim float 向量（归一化） |
| `tag_weights` | MAP<VARCHAR, FLOAT> | 兴趣标签 + 权重 |
| `last_active_ts` | BIGINT | 最近活跃时间 |
| `profile_version` | INT | 用于缓存失效的版本号 |

> **存储**：HBase 表 `profile:offline`，列族 `p`，行键 `user_id`。  
> **冷热分层**：最近 30 天的用户画像放在 **SSD** 节点，历史画像归档到 **对象存储**（OSS）供离线特征计算。

#### 2.2 用户画像（实时） `user_rt_profile`（Redis）

```text
HASH key: rt_profile:{user_id}
field   value
-------------------------
interest_vector   <binary>
last_5min_events  <JSON>   // 最近 5 分钟点击/停留
profile_version   <int>
TTL  300s  // 自动失效，防止脏数据长期占用
```

#### 2.3 视频元数据 `video_info`

| 列名 | 类型 | 说明 |
|------|------|------|
| `video_id` | BIGINT | 主键 |
| `title` | VARCHAR(200) | 视频标题 |
| `duration` | INT | 秒 |
| `upload_ts` | BIGINT | 上传时间 |
| `tags` | ARRAY<VARCHAR> | 业务标签 |
| `hot_score` | FLOAT | 热度（最近 24h 交互加权） |
| `embedding` | BINARY(1024) | 内容向量（用于召回） |

> **存储**：MySQL（业务查询）+ ClickHouse（聚合分析）+ Milvus（向量检索）。  
> **冷热**：最近 7 天的 `hot_score`、`embedding` 放在 **内存+SSD**，旧数据仅保留在对象存储。

#### 2.4 行为日志（Kafka Topic）

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | BIGINT | |
| `video_id` | BIGINT | |
| `event_type` | ENUM('play','pause','finish','like','dislike','share') | |
| `event_ts` | BIGINT | |
| `play_time` | INT | 实际播放时长（秒） |
| `stay_time` | INT | 停留时长（秒） |

> **Topic**：`user_behavior`，分区数 ≈ **30**（根据 QPS 计算，每分区吞吐 10k‑20k msg/s），使用 **KeyBy(user_id)** 保证同一用户行为在同一分区，方便实时流计算。

### 3. 数据一致性与时效

- **行为日志 → 实时特征**：使用 **at‑least‑once** 消费，Flink 通过 **状态快照 + Checkpoint** 保证幂等。  
- **实时特征 → 缓存**：写入 Redis 时使用 **乐观锁 + version**，防止并发覆盖。  
- **离线特征 → 召回索引**：每天全量重建后，**双写**（新索引上线后逐步切流）避免查询空洞。

## ## 第四步：核心 API 设计  

### 1. API 列表（REST + gRPC 混合）

| 方法 | URL/接口 | 输入 | 输出 | 说明 |
|------|----------|------|------|------|
| `GET /feed` | `GET /api/v1/feed?user_id=123&version=5` | `user_id`、`client_version`、`last_feed_ts`（可选） | `FeedResponse`（20 条 video_id + meta） | 首页推荐入口，支持 **版本号** 用于灰度实验 |
| `POST /feedback` | `POST /api/v1/feedback` | `FeedbackRequest`（user_id, video_id, type） | `200 OK` | 点赞/点踩/分享，写入 Kafka |
| `GET /user/profile` | `GET /api/v1/user/{user_id}/profile` | `user_id` | `UserProfile` | 用于调试/AB 测试 |
| `GET /video/{video_id}` | `GET /api/v1/video/{video_id}` | `video_id` | `VideoMeta` | 播放页获取视频信息（标题、标签） |
| `POST /refresh` | `POST /api/v1/refresh` | `user_id` | `200 OK` | 前端主动请求 **增量刷新**（滑动 5 条后），内部调用增量召回 RPC |

> **返回结构示例（FeedResponse）**  

```json
{
  "user_id": 123,
  "feed_version": 5,
  "items": [
    {
      "video_id": 987654321,
      "title": "搞笑猫咪合集",
      "cover_url": "https://cdn.example.com/cover/987654321.jpg",
      "duration": 15,
      "tags": ["搞笑","动物"],
      "score": 0.92   // 精排打分，仅调试用
    }
    // ... 19 条
  ],
  "next_cursor": "ts=1654234000&offset=20"
}
```

### 2. API 设计要点（新手常犯错误的解释）

| 常见错误 | 正确做法 | 为什么 |
|----------|----------|--------|
| **把所有业务参数放在 body**，忽略 query 参数的可缓存性 | 对 **GET** 请求使用 **query string**，便于 CDN/Edge 缓存 | 200 ms 目标需要利用 HTTP 缓存层 |
| **返回完整视频文件** | 只返回 **元数据**（ID、封面、时长），实际视频流由 CDN 播放 | 减少网络传输、降低 API Server 负载 |
| **单次返回 100 条** | 按需求返回 **20 条**，并支持 **分页/游标** | 避免一次性计算过多候选，降低时延 |
| **不返回模型打分** | 业务侧可以不需要，但调试时建议返回 **score**（仅在 debug 模式） | 有助于快速定位排序问题 |

### 3. API 性能优化

- **限流**：在 API Gateway 对 `GET /feed` 进行 QPS 限流（用户级、IP级），防止雪崩。  
- **熔断**：当召回/精排服务出现异常时，快速返回 **缓存热点** 或 **默认排序**，保持可用性。  
- **压缩**：返回 JSON 使用 **gzip**，客户端开启压缩；若流量敏感，可改为 **Protobuf**（gRPC）。

## ## 第五步：详细组件设计  

下面逐个拆解关键组件的内部实现、技术选型、数据流、容错策略。

### 5.1 行为日志采集 & 可靠传输  

1. **客户端 SDK**（Android / iOS）  
   - 在 UI 触发（播放、点击、停留）时 **异步**写入本地 SQLite/文件缓存，批量上报。  
   - 使用 **HTTPS + Protobuf**，压缩后发送到 **LogCollector**（NGINX + protobuf‑decoder）。  

2. **LogCollector → Kafka**  
   - NGINX 负责 **流量限速**、**IP 访问控制**。  
   - 后端 Java/Python **LogAggregator** 将接收到的 protobuf 批量写入 **Kafka**，使用 **acks=all**、**replication=3** 确保不丢失。  

3. **Kafka Topic**：`user_behavior`，**分区** = 30~50（根据 QPS 估算），**Key** = `user_id`，保证同一用户的日志落在同一分区，方便后续 **实时特征** 计算的 **KeyBy**。  

4. **容错**  
   - **生产者重试**（指数退避），写入失败写入 **Dead Letter Queue**。  
   - **Kafka MirrorMaker** 跨机房复制，灾备机房可直接消费。

### 5.2 实时特征计算（Flink）  

| 步骤 | 说明 | 关键点 |
|------|------|--------|
| **Source** | Flink 从 `user_behavior` 读取，使用 **Checkpoint**（每 5s） | 确保 **Exactly‑once** 语义 |
| **KeyBy(user_id)** | 将同一用户的行为聚合到同一算子实例 | 保证状态局部性 |
| **Window** | **Sliding Event‑time Window**：大小 5 min，滑动 1 min | 捕获最近 5 min 行为 |
| **State** | **MapState** 保存 `video_id → play_time/like_cnt` | 状态大小 ≤ 10 KB/用户 |
| **Feature Generation** | 计算 **CTR、停留率、最近点击向量** 等 | 生成 `UserRealtimeFeature` |
| **Sink** | 写入 Redis（Hash）+ 写入 HBase（增量画像） | TTL 300 s，防止旧特征残留 |

**示例 Flink 代码（伪）**  

```scala
val env = StreamExecutionEnvironment.getExecutionEnvironment
env.enableCheckpointing(5000)

val behavior = env.addSource(new KafkaSource("user_behavior"))
  .assignTimestampsAndWatermarks(WatermarkStrategy.forMonotonousTimestamps())
  .keyBy(_.userId)

val realtimeFeat = behavior
  .window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1)))
  .process(new ProcessWindowFunction[Behavior, UserRealtimeFeature, Long, TimeWindow]{
    override def process(key: Long, ctx: Context, elems: Iterable[Behavior], out: Collector[UserRealtimeFeature]): Unit = {
      // 统计最近 5 分钟的点击、停留、点赞等
      val feat = calcFeature(elems)
      out.collect(feat)
    }
  })

realtimeFeat.addSink(new RedisSink())
```

### 5.3 离线特征批处理（Spark）  

1. **数据源**：每日凌晨从 HDFS/OSS 读取 **行为日志**（已落盘 Parquet），以及 **视频元数据**、**用户画像**。  
2. **特征工程**：  
   - **用户画像**：聚合最近 7 天、30 天、90 天的交互特征（播放次数、停留时长、点击率），生成 **稀疏向量** + **兴趣标签权重**。  
   - **视频特征**：使用 **深度学习模型**（如 ResNet‑3D）抽取 **内容向量**，并计算 **热度分**（最近 24h 交互加权）。  
3. **存储**：特征写回 HBase（用户画像）和 Milvus（向量索引），并写入 ClickHouse 用于离线分析。  
4. **调度**：使用 **Airflow** 编排 DAG，确保 **依赖**（行为 → 画像 → 索引）顺序执行。  

### 5.4 召回层  

#### 5.4.1 召回策略组合  

| 召回方式 | 说明 | 优缺点 |
|----------|------|--------|
| **基于兴趣标签的倒排** | 查询 `tag -> video_id` 索引，返回热门视频 | 简单、低延迟，但受热点视频影响大 |
| **向量相似度检索** | 用用户兴趣向量在 Milvus 中做 **ANN** 检索，返回 Top‑K（如 5000） | 捕捉细粒度相似度，计算稍重 |
| **协同过滤（User‑CF）** | 基于相似用户的历史点击，返回候选 | 需要离线预计算相似度矩阵 |
| **热门流** | 最近 1h 内热度最高的 N 条 | 保证热点内容，防止冷门沉默 |

**实现细节**  

- **并行调用**：在召回服务（gRPC）内部，使用 **Fork‑Join** 并行发起上述四个子召回请求，**统一聚合**后去重。  
- **去重**：使用 **HashSet**（video_id）去除重复，保留 **召回来源权重**（后面粗排会用到）。  
- **返回数量**：统一返回 **3000–5000** 条候选，供后续粗排筛选。  

#### 5.4.2 召回系统架构  

```
[API] → [Cache (热点召回)] → [召回服务 (gRPC)]
          ↑                     ↓
    [Redis 召回缓存]   →  [Milvus]   ←  [Elasticsearch] ← [用户兴趣向量]
```

- **热点召回缓存**：对 **活跃用户**（最近 1h）缓存上一次召回结果，TTL 1 min，直接返回，降低向量检索压力。  
- **Milvus**：使用 **IVF‑PQ** + **HNSW**，支持 **每秒 10 万次** 向量查询，**单节点** 64 GB RAM + 2 TB SSD，满足 30k QPS（并行 8 路）。  

### 5.5 粗排层  

- **输入**：召回的候选集合（3000‑5000 条），每条带有 **召回来源权重**、**视频基本特征**。  
- **模型**：轻量 **GBDT**（LightGBM）或 **Logistic Regression**，特征包括：  
  - 用户兴趣向量与视频向量的 **余弦相似度**  
  - 视频热度、时长、发布时间  
  - 用户对该标签的历史点击率  
  - 召回来源权重（标签召回、向量召回、热门）  
- **实现**：部署为 **TensorFlow‑Serving** 或 **PMML**（在线评分服务），**毫秒级**返回 **分数**。  
- **输出**：对每条候选打 **粗排分**，取 **Top‑1000** 进入精排。  

> **为什么不直接进入精排？**  
> 精排模型通常是 **深度神经网络**（CTR、兴趣流），计算成本高，若直接对 3000+ 条视频做一次推理会导致 **200 ms** 目标破裂。粗排先过滤掉 99% 的不相关视频，只保留少量高质量候选。

### 5.6 精排层  

- **模型**：  
  - **CTR 预估模型**（DeepFM、DIN）输出点击概率  
  - **兴趣流模型**（基于 Transformer）捕捉用户最近行为序列  
  - **多任务模型**：点击率 + 停留时长 + 点赞率  
- **输入特征**：  
  - **实时特征**（最近 5 分钟点击向量、停留时长）  
  - **离线画像**（兴趣标签权重、长期点击率）  
  - **视频特征**（内容向量、热度、标签）  
  - **上下文特征**（时间段、设备、地域）  
- **融合方式**：在模型 **前端**做特征拼接，**后端**使用 **TensorFlow Serving** 提供 **REST/gRPC** 接口。  
- **输出**：对 **Top‑1000** 再次打分，取 **Top‑20** 返回给前端。  

> **实时特征 vs 离线特征权重**  
> - **实时特征**（最近 5 min）权重 **30%**，保证新鲜感。  
> - **离线画像**（长期兴趣）权重 **70%**，保证稳健性。  
> - 权重可以在 **AB 测试** 中动态调节。

### 5.7 缓存层设计  

| 缓存对象 | 失效策略 | 备注 |
|----------|----------|------|
| **热点召回结果** | TTL 60 s + 主动驱逐（热点用户） | 当用户行为产生实时特征变化时，立即 **Cache Invalidate** |
| **热点视频元数据** | TTL 5 min，定时刷新热度 | 视频热度更新后需要刷新 |
| **用户画像快照** | TTL 300 s（实时特征），每日全量刷新（离线画像） | 用版本号 `profile_version` 做一致性校验 |
| **推荐结果**（20 条） | **无缓存**（因为每次都需要最新排序），但可以 **预取**（后台提前算） | 通过 **预计算服务** 为活跃用户提前生成下一批推荐，放入 **Redis List** |

**缓存失效实现示例**（伪代码）  

```java
// 点赞后，更新用户兴趣标签权重
void onLike(UserLikeEvent e){
    // 1. 写入 Kafka
    kafkaProducer.send("user_behavior", e);
    // 2. 直接删除对应用户的缓存
    redis.del("rt_profile:" + e.userId);
    // 3. 触发增量召回刷新（异步）
    asyncCall("refreshFeed", e.userId);
}
```

### 5.8 高可用 & 容灾  

1. **多机房部署**  
   - **北京**（主站） + **上海**（灾备）  
   - 业务流量通过 **全局负载均衡**（阿里云 SLB）进行 **DNS 轮询**，灾备机房保持 **只读** 状态，随时切换。  

2. **服务冗余**  
   - 每个微服务（召回、粗排、精排） **水平扩容** 至 **N+1**，使用 **Kubernetes**（或 **Mesos**）进行 **Pod 自动恢复**。  
   - **状态ful** 服务（Redis、HBase）使用 **主从复制 + 自动故障转移**（Sentinel、Zookeeper）。  

3. **数据备份**  
   - **Kafka**：3 副本 + **MirrorMaker** 跨机房同步。  
   - **HDFS/OSS**：每日快照 + **对象版本**。  
   - **MySQL**：GTID 主从复制，异地灾备。  

4. **监控 & 告警**  
   - **指标**：QPS、延迟、错误率、缓存命中率、Kafka 延迟、Flink Checkpoint 成功率。  
   - **工具**：Prometheus + Grafana（时序监控），Alertmanager（告警），ELK（日志分析）。  
   - **异常处理**：当 **召回服务** 响应超时 > 80% 时，自动降级为 **热点缓存**，并触发 **弹性伸缩**（增加召回实例）。

## ## 第六步：扩展性与高可用设计  

### 6.1 横向扩展（Scale‑out）  

| 维度 | 扩容方式 | 关键点 |
|------|----------|--------|
| **API 层** | 增加 **Nginx** 实例 + **Consul** 服务发现 | 采用 **无状态**，可以随意增加 |
| **召回/粗排/精排** | 基于 **K8s** 自动水平伸缩（HPA） | 通过 **QPS/CPU** 指标触发 |
| **向量检索** | **Milvus** 集群（Shard + Replication） | 每个 Shard 负责一部分向量，查询时并行路由 |
| **实时特征** | **Flink** JobManager/TaskManager 增加 Task Slots | 按分区数伸缩，保持 **KeyBy** 分区不变 |
| **缓存** | **Redis Cluster** 扩容（分片） | 采用 **hash slot**，确保热点均匀分布 |
| **存储** | **HBase** RegionServer 自动拆分 | 数据热点导致 Region 迁移，监控 RegionServer 负载 |

### 6.2 垂直扩展（Scale‑up）  

- 当 **单机 CPU** 成为瓶颈（如深度模型推理），可以使用 **GPU**（Tesla T4）部署 **TensorFlow Serving**，或 **模型蒸馏**（轻量化）降低计算量。  

### 6.3 业务容灾与降级  

| 场景 | 降级方案 | 触发条件 |
|------|----------|----------|
| **召回服务不可用** | 直接返回 **热点缓存**（热门视频+标签召回） | 召回错误率 > 5% / 响应超时 > 100 ms |
| **粗排/精排模型超时** | 使用 **规则排序**（热度+标签相似度） | 单实例 CPU 100% 持续 30 s |
| **实时特征计算延迟** | 使用 **最近一次离线特征** 替代 | Flink Checkpoint 失败超过 3 次 |
| **缓存失效** | 回源至 **MySQL/HBase** | Redis 响应错误率 > 2% |

### 6.4 数据倾斜处理（针对追问）  

1. **热点视频** 导致召回节点负载不均衡  
   - **热点分离**：对热度 Top 0.1% 的视频，单独建立 **热点表**（Redis ZSET）并在召回时 **加权抽样**，防止同一节点请求过多。  
   - **分片均衡**：在 Milvus、Elasticsearch 中使用 **hash 分片**，确保热点向量分布到多个节点。  
2. **用户兴趣倾斜**（少数活跃用户请求量巨大）  
   - 对 **活跃用户** 使用 **专属缓存**（User‑Specific Cache），并限制 **每秒请求上限**（滑动窗口）防止 DoS。  
   - **分布式限流**：在 API Gateway 使用 **Token Bucket** 对每个用户 ID 进行速率限制。  

### 6.5 实时特征与离线特征融合（针对追问）  

- **特征拼接**：在精排模型输入层，将 **实时向量**（长度 32）与 **离线向量**（长度 128）拼接为 160 维向量。  
- **特征归一化**：实时特征使用 **指数衰减**（α=0.7），离线特征使用 **L2 正规化**，确保数值尺度相近。  
- **动态加权**：在模型训练时加入 **time_decay** 参数，学习实时特征的权重随时间衰减；在推理时可根据 **最近行为次数** 调整权重。  

### 6.6 缓存失效与一致性（针对追问）  

- **版本号机制**：用户画像在写入 Redis 时带上 **profile_version**（自增），每次实时特征更新后 **+1**。  
- **缓存读侧检查**：业务层在读取用户画像前，先读取 **profile_version**，若本地缓存版本旧于 Redis，则 **强制刷新**。  
- **主动失效**：点赞、评论等关键行为完成后，**同步调用** `CacheInvalidationService`，立即删除对应用户的 **rt_profile** 与 **召回缓存**。  
- **软失效**：使用 **TTL** 结合 **LRU**，让不活跃用户的缓存自然过期，避免长时间脏数据。

## ## 第七步：常见面试追问与回答  

### 7.1 数据倾斜如何处理？

**回答要点**  
1. **热点分离 + 多副本**：把热度最高的 0.1% 视频单独存放在 Redis ZSET，召回时对热点进行 **加权抽样**，并在多个机器上复制热点索引。  
2. **均匀分片**：向量检索使用 **hash 分片**（Milvus），保证热点向量分散到不同节点。  
3. **流量调度**：在 API Gateway 对热点用户/视频做 **限流**（Token Bucket），防止单一用户压垮后端。  
4. **监控与自动扩容**：监控每个召回节点的 QPS，热点节点超过阈值自动 **水平扩容**（新增 Shard）。  

### 7.2 实时特征与离线特征的融合策略？

**回答要点**  
- **特征层级**：实时特征捕获「最近 5 min」的兴趣变化，离线特征提供「长期画像」。  
- **拼接 + 加权**：在精排模型输入层，将两类向量拼接，使用 **Learnable 权重**（如 attention）让模型自行决定实时/离线的贡献。  
- **衰减因子**：实时特征使用 **指数衰减**（α=0.8），离线特征使用 **固定权重**，在模型训练时让 loss 同时约束两者。  
- **AB 测试**：上线前做 **Canary**，对比「只用离线」vs「实时+离线」的点击率、停留时长，确保实时特征提升显著后再全面推广。  

### 7.3 缓存失效和一致性问题？

**回答要点**  
- **主动失效**：关键行为（点赞、收藏）完成后，立即调用 **CacheInvalidationService** 删除对应用户的实时画像缓存与召回缓存。  
- **版本号机制**：每次写入用户画像时自增 `profile_version`，读取时比较本地缓存版本，若不匹配则强制刷新。  
- **TTL + LRU**：热点缓存设置 **短 TTL**（如 60 s），并使用 **LRU** 驱逐不活跃键，保证新鲜度。  
- **最终一致性**：即使短时间出现「旧缓存」导致轻微不一致，也在业务层容忍（推荐结果仍可接受），并在下次刷新时自动纠正。  

### 7.4 为什么要把「召回、粗排、精排」拆成三层？

- **召回**：目标是 **召回规模**（千级‑万级），只需要 **粗糙相似度**，计算轻量，强调 **低延迟**。  
- **粗排**：在召回的候选中做 **轻量机器学习**（GBDT），进一步筛选到 **千级**，仍然在 **毫秒级** 完成。  
- **精排**：对 **千级** 候选使用 **深度学习**（CTR、兴趣流），计算成本高，但因为候选数少，仍能在 **<30 ms** 完成。  
> **不拆的后果**：一次性对上万条候选跑深度模型，CPU/GPU 资源会被瞬间耗尽，导致 **200 ms SLA** 失效；同时模型调试、灰度发布也会变得非常困难。  

### 7.5 如何实现「每日数千万活跃用户」的高可用？

- **多机房双活**：主备机房均部署完整业务栈，使用 **全局负载均衡**（DNS+SLB）实现自动容灾。  
- **服务无状态**：所有业务服务（召回、粗排、精排）保持 **无状态**，请求可以路由到任意实例。  
- **数据复制**：Kafka、MySQL、Redis、HBase 均采用 **多副本**（≥3），并开启 **自动故障转移**。  
- **弹性伸缩**：基于 **Prometheus** 指标（CPU、QPS、延迟）自动触发 **K8s HPA**，在高峰期自动扩容。  
- **灰度发布**：使用 **Canary** 或 **Feature Flag**（LaunchDarkly）逐步把新模型/新功能推给 1%→10%→全量，降低风险。

## ## 心得与反思  

### 1. 本题最难的 1-2 个设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **召回、粗排、精排三层划分** | 需要在 **延迟** 与 **模型精度** 之间找到平衡点。 | ① 先估算每层模型的 **计算成本**（GBDT ≈ 0.2 ms/条，DeepCTR ≈ 10 ms/条）<br>② 结合业务需求（200 ms 总时延）推算可接受的候选规模 → 召回 5k、粗排 1k、精排 20。<br>③ 再考虑 **水平扩展** 与 **灰度发布**，三层解耦后每层可以独立升级。 |
| **实时特征与离线特征的融合** | 实时特征更新频繁、离线特征体量大，如何保证 **一致性** 与 **有效性**。 | ① 明确两类特征的 **时间窗口**（5 min vs 30 day）。<br>② 采用 **特征拼接 + 可学习权重**，让模型自行平衡。<br>③ 为避免「实时特征覆盖离线特征」导致漂移，使用 **版本号 + TTL** 机制，在缓存层做强制刷新。 |

### 2. 新手最容易犯的错误（至少 2 条）  

| 错误 | 解释 | 正确做法 |
|------|------|----------|
| **把所有业务写进一个单体服务** | 随着 QPS 增大，CPU、内存、IO 争用导致 **延迟爆炸**，且无法独立扩容。 | 将 **召回、粗排、精排**、**日志采集**、**特征计算** 拆成 **独立微服务**，使用 **无状态** 设计，便于水平扩容。 |
| **忽视缓存失效导致用户兴趣变化慢** | 只依赖 **TTL**，当用户点赞后仍可能看到旧推荐，体验下降。 | 在关键行为（点赞、点踩、收藏）后 **主动删除** 对应用户的 **实时画像缓存** 与 **召回缓存**，并使用 **版本号** 检查一致性。 |
| **只考虑离线批处理** | 离线模型每 24 h 更新，无法捕捉用户的即时兴趣，导致推荐不够新鲜。 | 引入 **实时特征流**（Flink）和 **增量召回**，在精排阶段把实时特征融合进去。 |
| **只用热点缓存不做降级** | 当召回服务挂掉时，系统直接不可用。 | 设计 **多级降级**：召回失效 → 直接返回 **热点视频缓存**；模型超时 → 使用 **规则排序**。 |

### 3. 学习建议与可延伸方向  

1. **系统设计方法论**  
   - 先 **拆解需求 → 估算规模 → 画出数据流**，再逐层加入「缓存」「容错」「扩容」等非功能需求。  
   - 练习 **从 0 到 1** 的 MVP 设计，然后逐步添加「高可用」层次，帮助思路结构化。  

2. **关键技术栈深度学习**  
   - **Kafka + Flink**：从「日志采集」到「实时特征」的完整流水线，建议动手搭建一个小项目（如实时 UV 统计）。  
   - **向量检索（Milvus / Faiss）**：了解 ANN（近似最近邻）原理，实践「图片/文本相似搜索」项目。  
   - **TensorFlow‑Serving / TorchServe**：部署深度模型，练习「模型热更新」与「灰度发布」。  

3. **性能调优实战**  
   - 学习 **CPU/内存/网络** 监控，掌握 **压测工具**（JMeter、Locust）对 QPS、P99 延迟进行评估。  
   - 熟悉 **缓存穿透、击穿、雪崩** 的防护措施（布隆过滤器、热点降级、熔断）。  

4. **业务层面的思考**  
   - 推荐系统不是单纯的技术堆砌，**业务目标**（提升 DAU、CTR）要体现在 **模型指标** 与 **AB 测试** 中。  
   - 关注 **隐私合规**（GDPR、数据脱敏）和 **可解释性**（为什么给用户推荐这个视频），这在面试中常被追问。
