# 第 13 天：设计 小红书

> 生成日期：2026-05-13

---

## 题目背景
小红书是一款以生活方式分享为核心的内容社区，用户可以发布图文、短视频笔记，浏览他人的种草、旅行、美食等经验，并通过社区互动（点赞、评论、关注）形成兴趣圈层。  
本题要求你从“海量笔记的生成、分发、检索与互动”四个维度出发，设计一个支撑核心业务的系统。

## 面试场景设定
> **面试官**：  
> “假设我们要为小红书设计一个全新的笔记服务（包括发布、推荐、搜索、互动等），请先从高层次描述系统的主要模块和数据流。随后我们会逐步深入到存储选型、缓存层、伸缩方案等细节。你可以先从功能需求和非功能需求开始梳理吗？”

## 功能性需求
1. **笔记发布**  
   - 支持图文、短视频（单视频 ≤ 60 s）上传，自动生成多分辨率的图片/视频缩略图。  
2. **笔记浏览 & 推荐**  
   - 按用户兴趣、地域、时间等因素进行实时个性化推荐，返回 20 条笔记列表。  
3. **全文检索**  
   - 支持关键词搜索，返回匹配度排序的笔记，并支持高亮显示。  
4. **互动功能**  
   - 点赞、评论（单层回复）、收藏、转发。每条笔记的互动计数需实时展示。  
5. **用户关系**  
   - 关注/取关、粉丝/关注列表，关注动态实时推送。  
6. **内容安全**  
   - 上传内容的图片/视频、文字进行机器审查（违规检测、敏感词过滤），不合规的内容直接拦截或标记。

## 非功能性需求
| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（日活跃用户）** | 200 M | 包括阅读、发布、互动等全链路 |
| **QPS（峰值请求）** | 250 k req/s | 笔记首页、推荐流、搜索等热点业务的峰值 |
| **单次请求时延** | ≤ 100 ms（95%） | 包括 API 网关到业务层的总时延 |
| **可用性** | 99.9%（每月累计宕机 ≤ 43 分钟） | 对外提供的笔记服务必须高可用 |
| **存储规模** | 500 PB（图片 + 视频） | 按 30 % 增长率/年预估 |
| **写入吞吐** | 5 k 笔记/s（含图片/视频） | 高峰期的发布并发写入 |

## 系统边界
**本题范围内（需设计）**  
- 笔记的上传、存储、处理（转码、压缩、审查）  
- 推荐与搜索的后端服务（召回、排序、索引）  
- 互动计数的实时更新（点赞、评论、收藏）  
- 基础的用户关系管理（关注、粉丝）  
- 内容安全审查的集成流程  

**本题范围外（不考虑）**  
- 用户注册/登录、账号安全（如验证码、密码加密）  
- 私信、社群、直播等非笔记核心业务  
- 广告投放系统、商业化结算  
- 第三方登录、外部分享（如分享到微信）  

## 提示与追问
1. **数据一致性**：如果采用分布式计数（如点赞），如何在保证高并发的同时提供强一致或最终一致的保证？  
2. **缓存策略**：在推荐流和搜索结果中，哪些数据适合放在缓存层，缓存失效策略怎么设计？  
3. **扩容与灾备**：面对突增的流量（如大型活动或热点话题），系统如何实现快速水平扩容？灾备中心的切换方案如何设计？

# 题解

## 解题思路总览
本篇答案把 **“从零到可用、再到高可用”** 的全过程拆成七个章节，力求每一步都像老师手把手教学生一样，**先讲 *为什么*，再讲 *怎么做*，最后说明 *不这么做会怎样***。  
- **第一步** 先把业务需求、规模、性能指标全部写清楚，帮助你把“看不见的东西”变成可量化的数字。  
- **第二步** 绘制 **高层架构图**，把所有功能块划分成独立的服务（上传、处理、推荐、搜索、互动、用户关系、内容安全）。  
- **第三步** 讨论 **数据模型** 与 **存储选型**（关系型、NoSQL、全文检索、对象存储），并说明每种存储的利弊。  
- **第四步** 给出 **对外 API**（RESTful/GraphQL）以及内部 RPC 接口的设计要点。  
- **第五步** 深入每个核心组件（**上传流、媒体处理、内容审查、推荐系统、搜索引擎、计数服务、关系服务**）的内部实现细节、缓存、异步任务、幂等性等。  
- **第六步** 讲 **扩容、容灾、故障恢复、监控告警**，并针对 **高并发计数、缓存失效、突发流量** 给出完整方案。  
- **第七步** 汇总 **常见面试追问**（强一致 vs 最终一致、热点缓存、灰度发布等）并给出回答思路。  
- **心得与反思** 总结最难的决策、常见坑以及进一步学习的方向。  

> **阅读建议**：如果你第一次接触系统设计，建议先通读一遍整体结构，再回到每一章节逐行细读；如果你已有一定经验，可以直接跳到感兴趣的章节（例如计数一致性或推荐系统实现）。

## 第一步：理解需求与规模估算

### 1. 功能需求拆解
| 大功能 | 子功能 | 关键点 |
|--------|--------|--------|
| **笔记发布** | 图文上传、短视频上传、自动生成多分辨率缩略图 | 大文件上传、秒传、断点续传、媒体转码 |
| **笔记浏览 & 推荐** | 实时个性化推荐、分页返回 20 条 | 用户画像、地域/时间过滤、召回+排序 |
| **全文检索** | 关键字搜索、匹配度排序、高亮显示 | 反向索引、分词、同义词、热词 |
| **互动功能** | 点赞、单层评论、收藏、转发、计数实时展示 | 高并发计数、去重、消息推送 |
| **用户关系** | 关注/取关、粉丝/关注列表、关注动态推送 | 双向关系、粉丝流、关系缓存 |
| **内容安全** | 文本敏感词、图片/视频机器审查、拦截或标记 | 离线模型、实时审核、审核结果回流 |

### 2. 非功能需求（关键指标）

| 指标 | 目标值 | 业务含义 |
|------|--------|----------|
| **DAU** | 200 M | 每日活跃用户数，决定整体流量规模 |
| **峰值 QPS** | 250 k req/s | 最高并发请求量，主要集中在首页、推荐、搜索 |
| **单次请求时延** | ≤ 100 ms（95%） | 前端交互流畅度要求 |
| **可用性** | 99.9%（≤ 43 min/月） | 必须做到热备、快速故障切换 |
| **存储规模** | 500 PB（图片+视频） | 对象存储容量要求，年增长 30% |
| **写入吞吐** | 5 k 笔记/s | 高峰期发布并发写入，包含媒体文件 |

> **粗略换算**（帮助估算资源）  
- **上传流**：5 k 笔记/s × (平均 2 MB 图文 + 8 MB 短视频) ≈ **70 GB/s** 原始流量（压缩/CDN 后会下降）。  
- **阅读流**：200 M DAU × 10 次打开/天 ≈ 2 B 次请求/天 ≈ 23 k req/s（峰值 5 倍 => 115 k req/s）→ 与 250 k QPS 目标相吻合，说明搜索、推荐占大头。  
- **计数更新**：每条笔记平均 30 次交互/分钟 → **10 M 计数写入/s**（后面会用分布式计数+异步落库来削峰）。

## 第二步：高层架构设计

### 1. 架构图（文字版）

```
+-------------------+      +-------------------+      +-------------------+
|   客户端 (APP/WEB) | <--->|   API Gateway /   | <--->|   统一监控 & 日志   |
|                   |      |   Edge Proxy      |      +-------------------+
+-------------------+      +-------------------+
           |                        |
           | HTTP/HTTPS (REST)      |
           v                        v
+-------------------+      +-------------------+      +-------------------+
|   鉴权/限流/灰度   | <--->|   路由层 (Service   | <--->|   配置中心 (Nacos) |
|   (OAuth2, JWT)   |      |   Mesh / Envoy)   |      +-------------------+
+-------------------+      +-------------------+
           |                        |
   +-------+--------+   +-----------+-----------+   +-------------------+
   |                |   |                       |   |                   |
   v                v   v                       v   v                   v
+--------+   +------------+   +-------------------+   +-------------------+
|上传服务|   |笔记服务 (CRUD)|   |推荐服务 (召回+排序) |   |搜索服务 (ES)      |
+--------+   +------------+   +-------------------+   +-------------------+
   |                |                |                       |
   |  (文件流)      |                |                       |
   v                v                v                       v
+------------+   +------------+   +------------+   +-------------------+
|对象存储 (OSS) |   |关系库 (MySQL) |   |召回缓存 (Redis) |   |全文索引 (ES集群) |
+------------+   +------------+   +------------+   +-------------------+
   |                |                |                       |
   |   (异步任务)   |   (业务事件)   |   (实时特征)          |
   v                v                v                       v
+-------------------+   +-------------------+   +-------------------+
|媒体处理服务 (FFmpeg) |   |计数服务 (Redis+Kafka) |   |内容安全服务 (ML) |
+-------------------+   +-------------------+   +-------------------+
```

> **解释**  
1. **API Gateway** 负责统一入口、限流、灰度发布、日志收集。  
2. **鉴权/限流层** 把用户身份（JWT）解析后注入请求上下文，防止恶意刷接口。  
3. **业务微服务** 按功能拆分：**上传服务**（负责大文件分块上传、秒传），**笔记服务**（CRUD、元数据存储），**推荐服务**（召回、排序），**搜索服务**（ES），**计数服务**（点赞/评论/收藏），**内容安全服务**（机器审查）。  
4. **对象存储**（如阿里云 OSS、Amazon S3）保存原始图片/视频以及转码后、缩略图等多分辨率文件。  
5. **关系库**（MySQL/PolarDB）保存结构化数据（笔记元信息、用户关系、评论等），采用 **读写分离 + 分库分表**。  
6. **缓存层**（Redis）用于热点计数、推荐召回、用户关系等的快速读写。  
7. **消息队列**（Kafka）解耦高并发写入与异步处理（媒体转码、内容审查、计数持久化）。  
8. **监控**（Prometheus + Grafana）+ **日志**（ELK）+ **链路追踪**（SkyWalking）保证 99.9% 可用性。  

### 2. 为什么采用微服务 + 异步

| 场景 | 直接同步的风险 | 微服务+异步的好处 |
|------|----------------|-------------------|
| 大文件上传 | 需要在单个请求里完成上传、转码、审查，耗时可能 > 5 s，导致超时、前端卡死 | 上传即返回 202，后续转码、审查、索引通过 MQ 异步完成，提升感知响应 |
| 高并发计数 | 每次点赞直接写 MySQL，锁竞争会导致 QPS 爆炸 | 先写入 Redis（单机 10⁵ QPS），定时或批量落库，兼顾性能与最终一致性 |
| 推荐/搜索 | 实时查询数据库会产生大量 I/O，响应慢 | 把热点数据预热到 Redis/ES，查询直接命中内存，满足 100 ms SLA |

## 第三步：数据库设计

### 1. 数据模型概览

| 表/集合 | 主要字段 | 主键 | 备注 |
|--------|----------|------|------|
| **users** (MySQL) | user_id (BIGINT PK), nickname, avatar_url, gender, birthdate, region_id, created_at | PK | 用户基本信息 |
| **notes** (MySQL) | note_id PK, user_id FK, title, content (TEXT), cover_url, media_type (enum: IMAGE, VIDEO), status (enum: DRAFT, PUBLISHED, REJECTED), created_at, updated_at | PK | 笔记元信息 |
| **note_media** (MySQL) | media_id PK, note_id FK, media_url, width, height, size_bytes, format, resolution_type (ORIGINAL/THUMB/HD) | PK | 多分辨率媒体关联 |
| **comments** (MySQL) | comment_id PK, note_id FK, user_id FK, parent_comment_id (NULL), content, created_at | PK | 单层回复（parent_comment_id 为 NULL 表示根评论） |
| **user_follows** (MySQL) | follower_id PK, followee_id PK, follow_time | 复合 PK | 关注关系（双向索引） |
| **note_counts** (Redis Hash) | key = `note:{note_id}` → fields: like_cnt, comment_cnt, collect_cnt, share_cnt | - | 实时计数，定时落库到 MySQL `note_stats` |
| **note_stats** (MySQL) | note_id PK, like_cnt, comment_cnt, collect_cnt, share_cnt, last_sync_at | PK | 持久化计数，供离线分析 |
| **search_index** (Elasticsearch) | _id = note_id, title, content, tags, user_id, region_id, created_at, like_cnt, comment_cnt | - | 全文检索 + 排序字段 |
| **recommend_cache** (Redis Sorted Set) | key = `rec:user:{user_id}` → score = relevance, member = note_id | - | 推荐召回结果（热点用户） |
| **media_audit_log** (MongoDB) | audit_id PK, media_url, audit_result (PASS/REJECT), reason, audit_time | PK | 审核记录，支持离线分析 |

> **说明**  
- **结构化数据**（用户、笔记、评论、关系）采用 **关系型数据库**，因为需要强事务（发布笔记时要保证笔记与媒体关联原子性）以及复杂关联查询。  
- **计数** 与 **热点召回** 放在 **Redis**，因为 **读写频率极高、对实时性要求强**。  
- **全文检索** 使用 **Elasticsearch**，天然支持倒排索引、分词、高亮、排序。  
- **媒体审查日志** 用 **MongoDB**（文档型），因为审查字段不固定、查询主要是基于时间窗口的分析。  

### 2. 分库分表策略

| 数据 | 分库依据 | 分表依据 | 目标 |
|------|----------|----------|------|
| **users** | 按 **region_id**（省/市）分库 | 按 **user_id** hash 分表（100 表） | 防止单库写热点、均匀分布 |
| **notes** | 按 **create_date**（月份）分库 | 按 **note_id** hash 分表（200 表） | 写入热点（当日笔记）分散 |
| **comments** | 按 **note_id** hash 分库 | 按 **comment_id** hash 分表（500 表） | 单笔记评论量大时仍能水平扩展 |
| **user_follows** | 按 **follower_id** hash 分库 | 按 **followee_id** hash 分表 | 关注查询均衡分布 |
| **note_stats** | 与 **notes** 同库同表（同步） | - | 统计与业务保持一致性 |

> **不做分库分表的后果**：单库写入会在高峰期（写 5 k 笔记/s）出现 **锁竞争、IO 饱和**，导致 **响应超时**，并且 **备份/恢复** 成本极高。  

### 3. 索引设计（MySQL 示例）

```sql
CREATE INDEX idx_note_user   ON notes(user_id);
CREATE INDEX idx_note_status ON notes(status);
CREATE INDEX idx_comment_note ON comments(note_id);
CREATE INDEX idx_follow_follower ON user_follows(follower_id);
CREATE INDEX idx_follow_followee ON user_follows(followee_id);
```

- **复合索引**：`idx_note_user_status`（(user_id, status)）用于查询用户的已发布笔记。  
- **全文索引**：在 `notes.title`、`notes.content` 上使用 **Elasticsearch** 而非 MySQL，避免 MySQL 全文索引的扩展性瓶颈。  

### 4. 数据一致性方案

| 场景 | 强一致性需求 | 采用的方案 |
|------|--------------|------------|
| **笔记发布**（元数据 + 媒体） | 必须原子 | 使用 **两段提交（2PC）** 或 **业务层事务补偿**：先写 `notes`（状态 DRAFT），再写 `note_media`，成功后将 `status= PUBLISHED`，若任一步失败回滚。 |
| **点赞计数** | 实时展示 → **最终一致** 可接受 | **Redis + Kafka**：先 INCR 到 Redis，Kafka 异步落库到 MySQL `note_stats`，每分钟一次批量同步。 |
| **关注关系** | 需要强一致（用户是否能看到关注动态） | **MySQL 事务** + **双写到 Redis**（缓存），写成功后立即返回；Redis 失效时回源 MySQL。 |
| **内容审查结果** | 必须在笔记对外可见前完成 | **同步审查**（图片/视频）或 **异步审查 + 状态锁**：笔记状态为 `UNDER_REVIEW`，审查通过后切换为 `PUBLISHED`，不通过直接 `REJECTED` 并返回错误。 |

## 第四步：核心 API 设计

> 采用 **RESTful** 风格，统一返回结构 `{code, message, data}`，错误码分业务域（`0` 成功、`1xx` 参数错误、`2xx` 业务错误、`5xx` 系统错误）。

### 1. 笔记发布（图片/视频）  

| 方法 | 路径 | 请求体 | 响应 | 说明 |
|------|------|--------|------|------|
| `POST` | `/api/v1/notes` | `multipart/form-data` <br> - `title` (string) <br> - `content` (string) <br> - `media_type` (enum) <br> - `files[]` (binary) | `{code:0, data:{note_id:"12345", status:"UNDER_REVIEW"}}` | 立即返回 **202**，媒体文件会异步上传至对象存储并触发审查、转码。 |
| `GET` | `/api/v1/notes/{note_id}` | - | `{code:0, data:{...note_detail...}}` | 从缓存或 MySQL 读取，若 `status != PUBLISHED` 返回 403。 |
| `PUT` | `/api/v1/notes/{note_id}` | JSON（title、content） | `{code:0}` | 只能编辑 `DRAFT` 或 `REJECTED` 状态的笔记。 |

#### 幂等性
- **上传接口**使用 `client-generated UUID`（`X-Idempotency-Key`）防止网络抖动导致的重复创建。  
- **服务器端**先检查 `idempotency_key` 是否已存在，若已存在直接返回之前的 `note_id`。

### 2. 推荐列表  

| 方法 | 路径 | 参数 | 响应 | 说明 |
|------|------|------|------|------|
| `GET` | `/api/v1/recommendations` | `user_id` (必填) <br> `page` (默认 1) <br> `size` (默认 20) | `{code:0, data:{list:[{note_id,...}]}}` | 先从 **Redis** `rec:user:{user_id}` 读取召回列表；若缓存 miss，调用 **召回服务** 生成并写回缓存（TTL 5 min）。 |

### 3. 搜索  

| 方法 | 路径 | 参数 | 响应 | 说明 |
|------|------|------|------|------|
| `GET` | `/api/v1/search` | `q` (keyword) <br> `page` <br> `size` | `{code:0, data:{list:[{note_id, highlight,...}]}}` | 调用 **Elasticsearch**，返回高亮片段；热点关键字结果可放在 **Redis** `search:hot:{q}` 缓存 1 min。 |

### 4. 互动计数  

| 方法 | 路径 | 请求体 | 响应 | 说明 |
|------|------|--------|------|------|
| `POST` | `/api/v1/notes/{note_id}/like` | `{user_id}` | `{code:0, data:{like_cnt:123}}` | **幂等**：内部检查 `user_like:{user_id}:{note_id}`，若已点过直接返回当前计数。 |
| `POST` | `/api/v1/notes/{note_id}/comment` | `{user_id, content}` | `{code:0, data:{comment_id}}` | 生成评论后写入 MySQL，计数 INCR Redis。 |

## 第五步：详细组件设计

下面逐个拆解关键业务组件的内部实现细节、技术选型、容错与扩展点。

### 1. 上传服务 & 对象存储

#### 1.1 分块上传（支持大文件、断点续传）
- **前端**：使用 **Tus** 或 **S3 Multipart Upload** 协议，先向 **Upload Service** 申请 `upload_id`、分块大小、签名 URL。  
- **Upload Service**（Go/Java）：
  - 接收 `POST /upload/init` → 返回 `upload_id`、预签名的分块上传 URL（直接指向 OSS）。  
  - **状态表** `upload_sessions`（MySQL）记录已上传分块的 ETag，支持查询已完成块数，实现断点续传。  
  - **幂等性**：每块上传成功后返回块号 + MD5，服务器端校验后写入 `upload_sessions`。  

#### 1.2 秒传（MD5 去重）
- 客户端先计算文件 MD5，调用 `GET /upload/check?md5=xxx`。  
- **Upload Service** 在 **Redis** 中维护 `md5:{hash} → object_key`，若已存在直接返回对象 URL，省去上传。  

#### 1.3 完成上传后触发异步任务
- **OSS** 通过 **EventBridge** 或 **OSS 回调** 发送 **ObjectCreated** 事件到 **Kafka** `media-raw` 主题。  
- **媒体处理服务** 订阅该主题，执行：
  1. **转码**（FFmpeg）→ 生成 1080p、720p、480p 以及 **缩略图**。  
  2. **内容审查**：将文件 URL 发送给 **AI 审查微服务**（深度学习模型），得到 PASS/REJECT。  
  3. **生成索引**：将笔记 ID、媒体信息、审查结果写入 **MySQL** `note_media` 与 **Elasticsearch**。  
  4. **状态回写**：若审查通过，将 `notes.status = PUBLISHED`；否则 `REJECTED` 并返回错误给前端。  

#### 1.4 容错与重试
- **Kafka** 消费采用 **幂等消费**：在 `media_processing` 表记录 `message_id`，消费前先检查是否已处理。  
- **FFmpeg** 转码失败 → 发送告警并写入审查日志，人工介入。  

### 2. 内容安全审查服务

| 子系统 | 技术 | 核心流程 |
|--------|------|----------|
| 文本审查 | **BERT + 关键词库**（离线模型） | 接收 `content` → 分词 → 置信度阈值判断 → 返回 PASS/REJECT |
| 图片审查 | **CNN（InceptionV3）** + **OSS 回调** | 图片 URL → 拉取 → 识别违规内容（暴恐、涉黄） → 返回结果 |
| 视频审查 | **视频帧抽样 + 目标检测**（每秒抽 1 帧） | 视频 URL → 抽帧 → 逐帧审查 → 若任一帧违规则 REJECT |

- **实时性**：文本审查同步返回；图片/视频审查采用 **异步**（因为需要下载文件），返回后通过 **Kafka** `audit-result` 主题回写 `note.status`。  
- **模型更新**：使用 **在线学习**（每日离线批训练）并通过 **灰度发布**（AB Test）验证新模型。  

### 3. 推荐系统

#### 3.1 召回层（离线 + 实时）
- **离线召回**（每日批处理）：
  1. **用户画像**（兴趣标签、地域、活跃时间）存放在 **HBase** / **Hive**。  
  2. **物品特征**（标签、热点）同样存储在大数据仓库。  
  3. **召回算法**：基于 **协同过滤**（ALS）+ **内容相似**（标签匹配） → 生成 **Top‑N**（200）笔记列表。  
  4. 结果写入 **Redis Sorted Set** `rec:user:{user_id}`（score = 召回分）。  

- **实时召回**（热点、最近 1 h）：
  - **流式计算**（Flink / Spark Streaming）监控 **用户行为日志**（点击、点赞），即时更新热点笔记集合 `hot_notes:{region}`。  
  - 对于热点用户直接 **合并** 实时热点与离线召回，写回同一 Sorted Set。  

#### 3.2 排序层（实时）  
- **特征**：用户兴趣匹配度、笔记热度（点赞/评论/收藏/转发）、新鲜度、地域相似度、内容安全等级。  
- **模型**：**GBDT + LR**（线上服务）或 **DeepFM**（离线训练后导出模型）。  
- **实现**：排序服务为 **gRPC** 接口，接收召回列表，批量查询特征（Redis/DB），返回 **Top‑20**。  

#### 3.3 缓存与失效
- **缓存对象**：召回列表（Sorted Set），TTL 5 min；热点笔记详情（JSON）缓存 10 min。  
- **失效策略**：  
  - **主动失效**：当笔记被删除、违规或计数大幅变化时发送 **CacheInvalidate** 消息至 Kafka，消费端删除对应键。  
  - **被动失效**：TTL 过期后自动淘汰。  

### 4. 搜索服务（Elasticsearch）

#### 4.1 索引结构
```json
{
  "mappings": {
    "properties": {
      "title": {"type":"text","analyzer":"ik_max_word"},
      "content": {"type":"text","analyzer":"ik_max_word"},
      "tags": {"type":"keyword"},
      "user_id": {"type":"long"},
      "region_id": {"type":"integer"},
      "created_at": {"type":"date"},
      "like_cnt": {"type":"long"},
      "comment_cnt": {"type":"long"}
    }
  }
}
```
- 使用 **中文分词器 IK**（或自研词典）实现高质量分词。  
- **热点加权**：`like_cnt`、`comment_cnt` 作为 **function_score** 的 boost 参数。  

#### 4.2 同步更新策略
- **写入路径**：笔记发布成功后 **Kafka** `note-published` → **ES 同步消费者** 把笔记写入 ES。  
- **删除/下线**：同样通过 **Kafka** `note-removed` → ES 删除。  
- **增量更新**：计数变化（点赞）通过 **Kafka** `note-count-changed` → ES **Update API**（脚本增量）。  

#### 4.3 高可用
- ES 集群 **3 主节点 + 3 副本**（跨机房跨 AZ），使用 **Shard**（每 50 GB）划分，总计约 10 TB 索引容量（可水平扩容）。  

### 5. 互动计数服务

#### 5.1 计数写入流程
1. **前端** 调用 `POST /like` → **API Gateway** → **计数服务**（Go）。  
2. **计数服务** 在 **Redis** `INCR` `note:{note_id}:like_cnt`，并在 **Redis Set** `user_like:{user_id}` 中记录已点赞的 note_id（用于幂等校验）。  
3. 同时发送 **Kafka** `note-like` 事件（包含 note_id、user_id），供 **异步落库** 与 **推荐特征更新**。  

#### 5.2 计数持久化（最终一致）
- **消费者**（Java）批量消费 `note-like`，每 5 s 或 10 k 条记录一次 **批量更新** MySQL `note_stats`（`UPDATE note_stats SET like_cnt = like_cnt + ? WHERE note_id = ?`）。  
- 若消费失败，Kafka **重试** 并写入 **Dead Letter Queue**，人工或自动补偿。  

#### 5.3 防刷机制
- **频率限制**：对同一 `user_id`+`note_id` 30 s 内只能点赞一次（Redis TTL），超出返回错误。  
- **行为分析**：结合 **实时风控**（Flink）检测异常点赞峰值，触发 **风控报警**。  

### 6. 用户关系服务

#### 6.1 关注/取关流程
- **写库**：MySQL `user_follows`，使用 **事务** 同时写入两条记录（正向/反向）。  
- **缓存**：  
  - `followees:{user_id}` → Set（存 followee_id）  
  - `followers:{user_id}` → Set（存 follower_id）  
- **异步通知**：写成功后发送 `follow-event` 到 Kafka，**推送服务**（WebSocket/消息队列）实时向被关注者推送 “新粉丝”。  

#### 6.2 关注动态流
- **动态流**（用户最近发布的笔记）存入 **Redis Stream** `user_feed:{user_id}`，消费者（推荐服务）把新笔记推送到关注者的 Feed。  
- 对于 **大 V**（粉丝 > 10 k），采用 **“粉丝拉模型”**（Fanout） + **消息队列**，防止热点导致写放大。  

### 7. 缓存层设计

| 缓存键 | 存储结构 | 失效时间 | 更新方式 |
|--------|----------|----------|----------|
| `note:{note_id}` | Hash（title, cover, stats） | 5 min | 计数变更时 **INCR**，笔记编辑时 **SET** |
| `rec:user:{user_id}` | Sorted Set（score, note_id） | 5 min | 召回任务完成后 **ZADD**，热点笔记变动 **ZINCRBY** |
| `search:hot:{keyword}` | String（JSON） | 1 min | 搜索热点更新后 **SET** |
| `followees:{user_id}` | Set | 10 min | 关注/取关时 **SADD/ SREM** |
| `user_like:{user_id}` | Set | 30 days | 点赞时 **SADD**，定期清理 |

- **热点缓存**：针对 **首页推荐**、**热点搜索词**、**热门笔记**，采用 **二级缓存**（本地 LRU + Redis）降低网络 RTT。

## 第六步：扩展性与高可用设计

### 1. 水平扩容

| 维度 | 关键指标 | 扩容方式 |
|------|----------|----------|
| **API 层** | QPS 250 k | **无状态**，增加 **Gateway/Load Balancer**（Nginx+Consul）实例，使用 **IP Hash** 或 **Session Sticky**（若需要） |
| **业务微服务** | 每个服务 QPS 10‑30 k | **容器化 (Docker) + K8s** 自动伸缩（Horizontal Pod Autoscaler） |
| **Redis** | 计数写入 10 M/s | **分片（Cluster）**，每片 3‑5 个节点，使用 **读写分离**，热点键可采用 **热点分片**（如 `note:{热点ID}` 单独分片） |
| **MySQL** | 写 5 k 笔记/s + 计数落库 | **读写分离** + **分库分表**（上文已说明），使用 **ProxySQL** 做路由 |
| **Kafka** | 事件流 100 k/s | **分区**（topic 按业务分区），**Broker** 横向扩容，**Producer** 开启 **Batch**、**Compression** |
| **Elasticsearch** | 搜索 QPS 100 k | **增加 Shard 数**，使用 **跨 AZ 复制**，查询路由到最近的副本 |
| **对象存储** | 70 GB/s 原始流量 | **多 AZ 多 Region**，使用 **CDN** 加速读取，**分片上传**（S3 multipart） |

#### 自动扩容策略
- **监控指标**：CPU、内存、QPS、延迟、Redis 命中率、Kafka lag。  
- **阈值**：CPU > 70% 持续 2 min → **水平扩容**；Kafka lag > 10 s → **增加消费者实例**。  

### 2. 容灾与灾备

| 组件 | 主备方案 | 切换机制 |
|------|----------|----------|
| **API Gateway / LB** | 多可用区（AZ）部署，使用 **跨 AZ 负载均衡**（SLB） | 自动健康检查，故障 AZ 自动剔除 |
| **MySQL** | 主-从复制（GTID），同城双活（双主）+ 异地备份（RDS 多 AZ） | **MHA** 自动故障转移，手动 **DR** 切换到异地只读 → 写恢复 |
| **Redis** | **主从复制 + Sentinel**，跨 AZ 部署 | Sentinel 触发 **Failover**，客户端自动切换 |
| **Kafka** | **多副本**（replication factor 3），跨 AZ | Broker down 自动选举 ISR，生产者重试 |
| **Elasticsearch** | **跨 AZ 副本**（每个 shard 2 副本） | 节点故障时副本升为主 |
| **对象存储** | 多 Region 同步（OSS 跨 Region 复制） | DNS 自动指向最近可用 Region |
| **媒体处理 / 内容审查** | 多实例 **K8s** 部署，使用 **Pod Anti-Affinity** 分散 | Pod crash 自动重建，水平弹性伸缩 |

#### 灾备演练
- **日常**：使用 **Chaos Monkey** 模拟单点故障，验证自动切换时延 < 30 s。  
- **周/月**：全链路灾备演练（停掉一个 AZ），确保业务可在 **5 min** 内恢复。  

### 3. 监控、告警与限流

- **指标收集**：Prometheus 抓取 **CPU、Mem、QPS、Latency、ErrorRate、Redis hit率、Kafka lag、Elasticsearch Query Latency**。  
- **告警阈值**：错误率 > 1%，延迟 > 150 ms，Redis hit率 < 80% → 触发 PagerDuty。  
- **链路追踪**：SkyWalking（或 Jaeger）记录每个请求的跨服务调用链，帮助定位热点。  
- **限流**：API Gateway 使用 **令牌桶**（Token Bucket）对每个 IP、每个用户（基于 UID）进行 QPS 限流，防止恶意刷请求。  

### 4. 数据安全与合规

- **传输加密**：全部使用 **HTTPS/TLS**，对象存储使用 **签名 URL**。  
- **存储加密**：OSS 开启 **服务端加密**（SSE），MySQL 使用 **Transparent Data Encryption**。  
- **隐私合规**：对用户敏感字段（手机号、邮箱）做 **脱敏**，日志中不记录原始内容。  
- **审计日志**：所有关键操作（发布、删除、审核、计数）写入 **审计系统**（Kafka → Elasticsearch），满足监管要求。

## 第七步：常见面试追问与回答

| 追问 | 关键点 | 参考答案 |
|------|--------|----------|
| **1. 点赞计数要强一致吗？如果要怎么实现？** | 业务上只要求**实时展示**，不需要每次都强一致。 | 采用 **Redis + Kafka** 的两段式写入：<br>1）先在 Redis `INCR`，返回给前端（实时）。<br>2）异步发送 Kafka，后台批量落库 MySQL。<br>如果业务必须强一致（如抽奖计数），可以使用 **Paxos/Raft** 的 **分布式计数器**（如 TiKV）或 **单机强一致的 Redis Cluster**（使用 `WAIT` 命令确保复制成功）。 |
| **2. 推荐召回缓存失效怎么办？** | 召回是离线生成，缓存失效会导致冷启动。 | 1）缓存 **TTL** 设为 5 min，失效后立刻触发 **异步召回任务**（使用调度系统 Cron）重新生成。<br>2）热点用户（活跃度 Top 5%) 采用 **预热**（每分钟刷新一次），保证缓存命中率。 |
| **3. 为什么搜索使用 ES 而不是 MySQL 全文索引？** | ES 天生为倒排索引、分片、水平扩展设计。 | MySQL 全文索引在 **海量数据**（500 PB）下 **查询延迟**、**索引构建**、**分词**都会成为瓶颈。ES 支持 **多副本、跨 AZ、函数评分**，并且可以 **实时同步**（Kafka）而不影响主库。 |
| **4. 大文件上传时，如何保证不因网络抖动导致重复写入？** | 需要幂等控制和断点续传。 | 采用 **上传 Session ID** + **MD5 校验**（秒传），服务端在 `upload_sessions` 表里记录已上传块的 ETag。客户端再次上传时先检查块是否已存在，若已存在直接跳过。 |
| **5. 内容审查如果误判会怎样？** | 误判会影响用户体验，需要回滚。 | 设计 **审查结果回滚接口**：如果用户申诉成功，后台将笔记状态从 `REJECTED` 改为 `PUBLISHED`，并重新写入搜索/推荐索引；同时记录审查日志供模型迭代。 |
| **6. 系统如何应对突发流量（如明星直播带来的流量峰值）？** | 需要弹性伸缩和流量削峰。 | 1）使用 **CDN** 缓存图片/视频，降低对象存储带宽。<br>2）**限流**：对热点笔记的阅读请求进行 **令牌桶** 限流，超额请求返回 `429`。<br>3）**预热缓存**：在活动前将热点笔记、搜索热点词提前写入 Redis。<br>4）**流量削峰**：使用 **消息队列** 将写操作（点赞、评论）排队处理，防止 DB 瞬时写爆。 |
| **7. 如何保证数据库的事务不因分库分表导致跨库事务难实现？** | 采用 **业务层补偿事务**（Saga）或 **软事务**。 | 对于跨库写（如笔记 + 媒体），先在 **笔记库** 插入 `status=DRAFT`，再写 **媒体库**，成功后更新笔记状态为 `PUBLISHED`。若任一步失败，执行补偿（删除已写记录）并回滚。也可以使用 **分布式事务框架**（Seata）在强一致性要求不高的场景下。 |

## 心得与反思

### 1. 本题最难的 1‑2 个设计决策及思考过程

| 决策 | 困难点 | 思考路径 |
|------|--------|----------|
| **① 实时计数的强/最终一致性** | 点赞、评论等计数需要 **毫秒级展示**，但写入 MySQL 并发受限。 | 先评估业务容忍度：**展示**可以稍有误差 → 采用 **Redis + Kafka** 两段式写入；若业务要求 **抽奖计数** 必须强一致 → 引入 **分布式计数器（Raft）** 或 **单机强一致 Redis**。最终方案兼顾两类需求：默认使用 **最终一致**，提供 **强一致计数服务**（可选）给特殊业务。 |
| **② 推荐召回的离线/实时混合** | 推荐必须 **个性化、时效性**，但离线批处理成本高，实时计算资源又有限。 | 先划分 **特征**：用户画像（离线） vs 实时热点（流式）。构建 **离线召回**（每日）生成基础候选集合；再用 **实时流**（Flink）注入最近 1 h 热点笔记。最终在 **排序服务** 合并两部分并做细粒度打分。这样兼顾 **规模** 与 **时效**，并且可以在流量激增时只扩展 **实时流** 部分。 |

### 2. 新手最容易犯的错误（至少 2 条）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有数据都放在单库单表**（比如把笔记、评论、用户都放在同一个 MySQL 实例） | 写入热点、锁竞争、单点瓶颈，导致 **QPS 只能到几千**，无法支撑 250 k req/s。 | 采用 **读写分离 + 分库分表**，按照业务特征（时间、地域、ID 哈希）划分；热点数据使用 **分片** 或 **NoSQL**。 |
| **同步调用全部下游服务**（上传完即同步转码、审查、索引） | 请求链路过长，单次响应时间轻易突破 100 ms SLA，甚至超时。 | **异步化**：上传返回 202，后续转码、审查、索引、状态回写通过 **消息队列** 完成；前端轮询或 WebSocket 监听状态。 |
| **计数直接写入 MySQL**（每次点赞都 UPDATE） | 计数写入并发高，导致 **行锁**、**事务冲突**，吞吐不到 10 k/s。 | 采用 **Redis 原子 INCR** + **Kafka 异步落库**，并使用 **幂等键** 防止重复计数。 |
| **忽视缓存失效**（直接把热点数据永远缓存） | 数据不一致、业务错误（如违规笔记仍被推荐）。 | 设计 **主动失效**（业务事件触发 `CacheInvalidate`） + **TTL**，并在热点更新时同步刷新。 |
