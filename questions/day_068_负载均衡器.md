# 第 68 天：设计 负载均衡器

> 生成日期：2026-03-19

---

## 1. 题目背景  
负载均衡器（Load Balancer）是位于客户端与后端服务之间的流量调度层，负责把请求按照一定策略分发到多台业务服务器，以提升系统的吞吐、容错和可用性。面试中我们希望你设计一个能够支撑大规模互联网业务的 **分布式层级负载均衡系统**。

## 2. 面试场景设定  
> **面试官**： “假设我们要为一家每日活跃用户（DAU）约 5000 万、峰值 QPS 达到 200 万的电商平台提供全局负载均衡服务。请你从零开始设计这套负载均衡系统，重点说明架构、核心组件以及如何满足高可用、低延迟的需求。”  

（面试官随后会让你逐步展开设计，关注流量入口、调度策略、健康检查、扩容与故障恢复等关键点。）

## 3. 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| F1 | **全局流量入口**：提供统一的公网 IP /域名，接受来自浏览器、移动端等客户端的 TCP/UDP/HTTP(S) 请求。 |
| F2 | **多层调度**：实现 DNS 负载均衡 → 边缘代理（L4/L7） → 后端业务集群的三级调度，支持基于轮询、加权轮询、最小连接数、响应时间等算法。 |
| F3 | **健康检查**：对后端实例进行主动（TCP/HTTP/HTTPS）和被动（错误率、超时）监控，自动剔除异常节点并支持快速恢复。 |
| F4 | **会话保持 / 粘性**：对需要会话粘性的业务（如购物车、登录态）提供基于 Cookie、IP 哈希或 Token 的会话保持能力。 |
| F5 | **动态扩容/缩容**：支持在流量高峰或节点故障时，快速新增/删除后端实例，且调度策略能够实时感知变化。 |
| F6 | **监控与告警**：实时上报请求量、错误率、延迟、节点状态等指标，提供仪表盘与阈值告警。 |

## 4. 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **日活用户（DAU）** | 5,000 万 | 设计需支撑全站流量。 |
| **峰值 QPS** | 2,000,000 QPS（约 2 M） | 包括 HTTP、HTTPS、TCP 等混合流量。 |
| **平均响应延迟** | ≤ 2 ms（负载均衡层） | 不计入后端业务处理时间，仅指调度决策与转发耗时。 |
| **99.99% 可用性** | 年度累计不可用时间 ≤ 52.56 分钟 | 需要多活部署、故障自动切换。 |
| **吞吐/带宽** | ≥ 200 Gbps（入口总带宽） | 需要横向扩展的高速网络转发能力。 |
| **数据持久化** | 健康检查日志、配置变更日志保留 30 天 | 用于审计与故障分析。 |

## 5. 系统边界  

**本题范围（需要设计）**  
- 公网入口的 DNS 负载均衡、边缘代理（L4/L7）实现  
- 调度算法、健康检查机制、会话保持方案  
- 节点的自动注册/下线、扩容/缩容流程  
- 监控、日志、告警体系的基本框架  
- 高可用部署（多可用区、多地域）与故障恢复策略  

**不在本题范围（可不必详细设计）**  
- 具体的业务后端服务实现（如商品搜索、订单系统）  
- 完整的 CDN、WAF、DDoS 防护等安全层（可在讨论中简单提及）  
- 底层硬件网络设备的详细选型（如 ASIC、DPDK 实现细节）  
- 数据库或持久化存储的内部实现（仅需说明日志存储方式）

3. **扩展性与成本**

## 解题思路总览
1. **先把系统拆成“控制平面 + 数据平面”。**  
   - 控制平面负责配置、健康检查、元数据同步、扩容/缩容等“慢路径”。  
   - 数据平面负责高速转发、调度决策、会话保持等“快路径”。  

2. **从最小可用（MVP）开始**：  
   - 只要一个公网入口（单 IP）+ 简单的轮询调度 + 健康检查 → 能满足基本功能 F1、F2、F3。  

3. **逐步给系统加层**：  
   - **多层调度**（DNS → 边缘 L4/L7 → 集群内部 LB）  
   - **多可用区/多地域**、**会话保持**、**动态扩容**、**监控告警**、**持久化日志**。  

4. **每一次扩展都要回答“为什么要加？”**  
   - 如：**为什么要做 DNS 级别的负载均衡？** → 为了把流量粗粒度分散到不同地域，降低单点网络瓶颈、提升容灾。  

5. **把非功能需求（QPS、延迟、可用性）映射到技术选型**：  
   - 2 M QPS → 需要 **水平扩展的无状态转发**（基于 DPDK / eBPF / ASIC），并且 **网络带宽** 至少 200 Gbps。  
   - 2 ms 延迟 → **控制平面与转发平面严格分离**、**热点路由表放在内存/TCAM**。  

下面按章节展开。

## 第一步：理解需求与规模估算  

| 需求编号 | 关键点 | 估算要点 |
|----------|--------|----------|
| F1 | 公网统一入口，支持 TCP/UDP/HTTP(S) | 需要 **Anycast + Anycast DNS**，提供 1~3 个公网 IP（多地域） |
| F2 | 多层调度（DNS → L4/L7 → 集群） | ① DNS 负责粗粒度流量分发 ② Edge L4/L7 负责细粒度调度 |
| F3 | 健康检查（主动+被动） | 主动：每秒一次 TCP/HTTP 检查；被动：收集错误率、RTT |
| F4 | 会话保持 | Cookie、IP Hash、Token 三种常用方案 |
| F5 | 动态扩容/缩容 | 与容器/虚拟机编排系统（K8s、ECS）对接 |
| F6 | 监控告警 | 需要实时采集 **每秒 QPS、错误率、RTT、节点状态**，保留 30 天日志 |

### 规模估算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日活跃用户** | 5,000 万 | 已知 |
| **峰值 QPS** | 2,000,000 QPS | 已知 |
| **每个请求大小**（HTTP Header + TCP/IP）≈ 1 KB | 2 M × 1 KB = 2 GB/s ≈ 16 Gbps（仅业务层） |
| **实际入口流量**（考虑 TLS 握手、重传、UDP）≈ **200 Gbps** | 需 200 Gbps 入口带宽 |
| **每台 Edge 机器处理能力**（基于 DPDK）≈ 10 M QPS | 200 M QPS / 10 M ≈ 20 台（保守冗余 2×） → **40 台** |
| **健康检查频率** 1 s/实例，假设 10 k 实例 → 10 k 检查/s，CPU 负载可接受 |
| **配置/状态存储**：每分钟写入约 10 k 条实例状态，30 天 ≈ 4.3 B 条，**单表 100 GB** 级别，关系型 DB（MySQL）+ 分区足够 |

> **结论**：系统必须是 **横向可扩展**、**无状态转发**、**多活部署**，并在 **控制平面** 使用 **强一致性存储**（如 etcd/Consul）保证全局配置一致。

## 第二步：高层架构设计  

### 1. 整体结构（文字版）

```
+-------------------+        +-------------------+        +-------------------+
|   DNS (Anycast)   | -----> |   Edge LB 集群    | -----> |   后端业务集群    |
|  (全球负载均衡)   |        |  L4/L7 代理 (N)   |        | (多 AZ/多 Region) |
+-------------------+        +-------------------+        +-------------------+
          |                         |                         |
          |   控制平面 (Config Service, Health Service,  |
          |   Service Registry, Metrics, Alarm)        |
          +-------------------+--------------------------+
```

### 2. 关键层次解释  

| 层级 | 主要职责 | 典型实现 | 为什么要这么划分 |
|------|----------|----------|-------------------|
| **DNS 负载均衡** | 把全球流量粗粒度分配到最近的 Edge 区域（基于地理、网络 RTT） | Anycast + GeoDNS（如 AWS Route 53、Alibaba Alidns） | 防止单点网络瓶颈，降低跨地域时延，提升容灾能力 |
| **Edge L4/L7 代理** | 高速转发、细粒度调度、会话保持、TLS 终止 | 基于 **DPDK**、**eBPF** 或 **硬件 ASIC** 的软/硬件 LB（如 Nginx Plus、Envoy、F5、腾讯 CLB） | 1) 需要 2 ms 内完成调度决策；2) 支持 L4（TCP/UDP）和 L7（HTTP）混合；3) 可以在同一层做健康检查、会话保持 |
| **后端业务集群** | 真正的业务处理（商品、订单等） | 任意业务框架（SpringBoot、Node.js 等） | 只负责业务，不参与流量调度，保持业务层的 **水平扩展** |
| **控制平面** | 配置管理、实例注册/下线、健康状态同步、监控告警 | **分布式 KV（etcd/Consul） + 微服务（Go/Java）** | 控制平面不参与数据转发，故可以采用强一致性存储，保证全局视图一致；同时可以独立横向扩容 |

### 3. 数据平面 vs 控制平面  

| 维度 | 数据平面 | 控制平面 |
|------|----------|----------|
| **处理对象** | 单个网络包/请求 | 配置、元数据、监控数据 |
| **时延要求** | ≤ 2 ms | 低于秒级即可 |
| **状态** | **无状态**（仅依赖本地缓存的路由表） | 强一致性（KV、日志） |
| **伸缩方式** | 横向增加转发节点 | 增加控制服务实例 + KV 集群规模 |

## 第三步：数据库设计  

> **注意**：这里的“数据库”指的是 **控制平面持久化存储**，包括业务配置、健康状态、监控日志。后端业务自己的 DB 不在本设计范围。

### 1. 选型  

| 需求 | 推荐方案 | 说明 |
|------|----------|------|
| 强一致性、事务 | **MySQL（主从+半同步）** 或 **TiDB** | 配置变更必须全局可见，防止“脑裂”。 |
| 大规模时序监控 | **TSDB（Prometheus + Thanos）** | QPS、RTT、错误率是时序数据。 |
| KV 注册/服务发现 | **etcd / Consul** | 内置 Raft，实现强一致的键值存储，适合做控制平面状态。 |
| 长期日志存储（30 天） | **ClickHouse** 或 **Elasticsearch** | 支持高吞吐写入、快速查询。 |

下面给出 **MySQL**（配置/健康）和 **Prometheus**（监控）两张核心表的示例结构。  

### 2. 表结构（Markdown + SQL）  

#### 2.1 `lb_instance` – 后端实例元信息  

```sql
CREATE TABLE lb_instance (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ip VARCHAR(45) NOT NULL,               -- IPv4/IPv6
    port INT NOT NULL,
    region VARCHAR(32) NOT NULL,           -- 如 cn‑gz, cn‑sh
    az VARCHAR(32) NOT NULL,               -- 可用区
    weight INT DEFAULT 100,                -- 加权轮询
    status ENUM('UP','DOWN','MAINTENANCE') DEFAULT 'UP',
    last_heartbeat TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uq_ip_port (ip, port)
) ENGINE=InnoDB;
```

#### 2.2 `lb_health_check` – 主动健康检查记录  

```sql
CREATE TABLE lb_health_check (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    instance_id BIGINT NOT NULL,
    check_type ENUM('TCP','HTTP','HTTPS') NOT NULL,
    interval_sec INT NOT NULL DEFAULT 5,
    timeout_ms INT NOT NULL DEFAULT 2000,
    success_cnt INT DEFAULT 0,
    fail_cnt INT DEFAULT 0,
    last_result ENUM('SUCCESS','FAIL') NULL,
    last_checked_at TIMESTAMP NULL,
    FOREIGN KEY (instance_id) REFERENCES lb_instance(id) ON DELETE CASCADE,
    INDEX idx_instance_time (instance_id, last_checked_at)
) ENGINE=InnoDB;
```

#### 2.3 `lb_config` – 负载均衡器全局配置（JSON）  

```sql
CREATE TABLE lb_config (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(64) NOT NULL,
    version BIGINT NOT NULL DEFAULT 0,          -- 乐观锁
    config_json JSON NOT NULL,                  -- 包含调度算法、会话保持策略等
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;
```

#### 2.4 `lb_metric` – 时序监控（写入 Prometheus）  

> 直接写入 Prometheus 的 `pushgateway` 或使用 `remote_write`，表结构不必在关系型库中。  

### 3. 数据一致性与同步  

- **实例注册/心跳** → 写入 `lb_instance` → **etcd** 通过 Watch 机制同步到所有 Edge 节点的本地缓存（路由表）。  
- **健康检查结果** → 更新 `lb_instance.status` → 触发 **Consul/etcd** 事件，Edge 节点在下一个调度周期感知状态变化。

## 第四步：核心 API 设计  

> **原则**：RESTful + JSON，配合 **OpenAPI（Swagger）** 生成文档，方便运维和自动化。  

| 类别 | API | 方法 | 请求体 | 响应 | 说明 |
|------|-----|------|--------|------|------|
| **实例管理** | `/api/v1/instances` | POST | `{ip,port,region,az,weight}` | `{id}` | 新增后端实例（自动注册到 etcd） |
| | `/api/v1/instances/{id}` | PUT | `{weight, status}` | `200` | 更新权重或手动下线 |
| | `/api/v1/instances/{id}` | DELETE | - | `204` | 删除实例 |
| **健康检查** | `/api/v1/healthchecks` | POST | `{instance_id, type, interval, timeout}` | `{id}` | 创建主动检查任务 |
| | `/api/v1/healthchecks/{id}` | DELETE | - | `204` | 删除检查 |
| **负载均衡配置** | `/api/v1/lb/config` | GET | - | `{config_json, version}` | 获取全局配置 |
| | `/api/v1/lb/config` | PUT | `{config_json, version}` | `200` | 更新配置（乐观锁） |
| **会话保持** | `/api/v1/lb/sticky` | POST | `{type:'cookie'|'ip_hash'|'token', ttl}` | `200` | 设置会话保持策略 |
| **监控查询** | `/api/v1/metrics` | GET | `?metric= qps&start=...&end=...` | `{datapoints}` | 查询时序指标（内部转发到 Prometheus） |
| **扩容/缩容** | `/api/v1/scale` | POST | `{action:'add'|'remove', count, region, az}` | `202` | 调用编排系统（K8s/ECS） |

### 关键 API 细节  

1. **幂等性**：POST `/instances` 返回实例 ID；后续 PUT/DELETE 使用 ID，保证幂等。  
2. **权限控制**：使用 **OAuth2 + RBAC**，运维人员只能读/写特定 Region 的配置。  
3. **灰度发布**：在 `lb_config` 中加入 `rules` 字段，可针对特定路径/用户做 **Canary** 调度。

## 第五步：详细组件设计  

### 1. DNS 负载均衡层  

| 功能 | 实现细节 | 关键参数 |
|------|----------|----------|
| **Anycast IP** | 在全球 BGP 网络上发布同一个 /32 IP，路由到最近的 Edge PoP | 1 – 3 个 Anycast IP（分别对应 HTTP、HTTPS、TCP） |
| **GeoDNS** | 基于用户 IP 的地理位置或 RTT，返回最近的 Edge IP 列表（最多 2‑3 条） | TTL 30 s（可动态下发） |
| **故障切换** | 当某区域 Edge 节点失联，DNS 控制平面把对应 IP 从返回列表中剔除 | 自动刷新（每分钟） |

> **不做 DNS** 的后果：所有流量都进入单一入口，带宽瓶颈、单点故障、跨地域时延大。  

### 2. Edge L4/L7 代理  

#### 2.1 数据平面核心流程（伪代码）  

```go
func handleConn(conn net.Conn) {
    start := time.Now()
    // 1. 读取 5‑tuple (srcIP, srcPort, dstIP, dstPort, proto)
    flow := parseFlow(conn)

    // 2. 本地缓存查询路由表 (etcd watch → in‑memory map)
    backend := router.Lookup(flow)

    // 3. 会话保持检查
    if sticky := stickyManager.Check(flow, conn); sticky != nil {
        backend = sticky
    }

    // 4. 统计最小连接数、响应时间等动态权重（可选）
    if backend.Algorithm == "least_conn" {
        backend = router.LeastConn()
    }

    // 5. 计数器、连接计数 + 限流（Token Bucket）
    if !rateLimiter.Allow(flow) {
        conn.Write([]byte("HTTP/1.1 429 Too Many Requests\r\n\r\n"))
        conn.Close()
        return
    }

    // 6. 转发 (DPDK fast path)
    forward(conn, backend)

    // 7. 统计指标
    latency := time.Since(start)
    metrics.Record(flow, backend, latency)
}
```

#### 2.2 关键技术点  

| 技术 | 为什么选它 | 备注 |
|------|------------|------|
| **DPDK**（Data Plane Development Kit） | 零拷贝、用户态高速转发，单核可达 20 Mpps，满足 2 M QPS & 2 ms 延迟 | 需要 NIC 支持，部署在裸金属或支持 SR‑IOV 的虚拟机 |
| **eBPF/XDP** | 代码在内核层运行，几乎无额外上下文切换，适合 **L4**（TCP/UDP）转发 | 通过 `tc` 或 `XDP` 实现流量镜像、监控 |
| **Envoy / Nginx Plus** | 已实现 L7 过滤、TLS 终止、灰度路由，易于二次开发插件 | 业务侧可以直接使用插件（Lua、Go）实现自定义调度 |
| **Consistent Hash Ring** | 会话保持时，**IP Hash** 需要环形哈希，保证节点增删时最小迁移 | 与 etcd 同步后更新环形结构 |
| **Ring‑Buffer + Batching** | 健康检查结果写入 etcd 前先放入本地 RingBuffer，批量提交降低网络开销 | 典型批次 10 ms/100 条 |

#### 2.3 健康检查模块  

- **主动检查**：每台 Edge 节点拥有 **HealthChecker** 协程，按 `interval` 对实例发 TCP SYN 或 HTTP GET。结果写入本地缓存并通过 **gRPC** 上报给 **Control Service**。  
- **被动检查**：拦截转发返回的错误码（5xx、RST）和超时，累计错误率。如果错误率 > 5%（窗口 30 s）则标记为 `DOWN`。  

#### 2.4 会话保持实现  

| 场景 | 方案 | 实现细节 |
|------|------|----------|
| **Cookie** | **基于 Set‑Cookie**，后端返回 `Set-Cookie: LBID=xxxx; Path=/; Max-Age=3600`，Edge 读取该 Cookie 并路由到对应实例 | Edge 需要 **Cookie 解析**，保持映射 `cookie→instanceID`（LRU 缓存） |
| **IP Hash** | 对客户端 IP 计算 `hash(IP) % N`，定位环形哈希上的实例 | 当实例上下线时，只迁移相邻的哈希区间 |
| **Token** | 前端携带 JWT，其中 `lb_instance_id` 字段，由业务生成 | Edge 解码 JWT（无需签名校验），直接路由 |

> **为什么提供多种方式**：不同业务对会话粘性的要求不同，Cookie 适合 Web，IP Hash 适合无状态 API，Token 适合微服务间调用。  

### 3. 控制平面服务  

| 服务 | 功能 | 关键技术 |
|------|------|----------|
| **Config Service** | 读取/写入 `lb_config`，推送配置变更至 Edge（gRPC stream） | **etcd** + **gRPC**（Server‑Side Streaming） |
| **Health Service** | 汇总主动/被动健康检查结果，写入 MySQL、触发告警 | **Kafka**（异步日志） + **Prometheus**（指标） |
| **Service Registry** | 实例注册/心跳 → 写入 `lb_instance`，并同步到 etcd | **REST API** + **Watch** |
| **Metric Service** | 收集 Edge 上报的 QPS/RTT → 暴露 `/metrics` 给 Prometheus | **Pushgateway** 或 **remote_write** |
| **Scale Service** | 与容器编排平台（K8s）或云主机 API 对接，实现弹性伸缩 | **K8s Operator** 或 **Terraform** |

### 4. 监控、日志、告警体系  

1. **监控**  
   - **Prometheus** 抓取 Edge 暴露的 `lb_requests_total{status="2xx"}`、`lb_latency_seconds_bucket` 等指标。  
   - **Grafana** 看板：实时 QPS、错误率、平均延迟、每个实例的健康状态。  

2. **日志**  
   - **Access Log**（请求/响应）采用 **JSON**，写入 **ClickHouse**（列式存储）以支持高速写入和快速聚合查询。  
   - **Health Log**（检查结果）写入 **MySQL**，保留 30 天。  

3. **告警**  
   - **Alertmanager** 根据阈值（如 QPS > 1.8M、错误率 > 2%）触发钉钉/Slack/邮件。  
   - **故障自动化**：当某 AZ 所有实例 `DOWN`，Alertmanager 调用 **Scale Service** 自动把流量切到其它 AZ 并启动新实例。

## 第六步：扩展性与高可用设计  

### 1. 多可用区 / 多地域部署  

| 层级 | 多 AZ/多地域实现方式 | 目的 |
|------|--------------------|------|
| DNS  | **Anycast + GeoDNS** → 根据用户 IP 返回最近的 Edge IP | 降低跨区域时延、分散带宽 |
| Edge  | 每个地域部署 **N** 台 Edge（N≥3），同一 Region 内使用 **L4/L7** 互相备份 | 防止单机、单机房故障 |
| 控制平面 | **etcd** 集群跨 3 个 AZ（每 AZ 1 个节点），使用 **Raft** 保证强一致 | 防止控制面单点失效 |
| 后端业务 | **跨 AZ 多活**，使用 **分布式事务**（如 Saga）或 **读写分离**，业务自行容错 | 保证业务层面的高可用 |

### 2. 故障恢复流程  

1. **实例失联**（主动检查失败） → Health Service 标记 `DOWN` → etcd 触发 **Watch**，Edge 删除该实例路由。  
2. **Edge 节点宕机** → 其它 Edge 节点已在 **同一 Anycast IP**，BGP 自动切回；Control Service 检测不到心跳后 **自动补偿**（拉起新实例）。  
3. **全 AZ 失联** → DNS 控制平面把该 AZ 的 IP 从返回列表中剔除；同时 **Scale Service** 在其它 AZ 扩容（通过云 API），并更新 `lb_instance`。  

> **一致性保证**：所有 Edge 只读取 **etcd** 中的最新路由表；etcd 使用 **线性化读**（lease）保证所有节点在同一时间看到相同的实例状态。  

### 3. 动态扩容/缩容  

| 场景 | 操作步骤 |
|------|----------|
| **流量突增**（如双 11） | 1. 监控系统触发 QPS 超阈值 → Alertmanager 调用 **Scale Service** <br>2. Scale Service 调用 **K8s Horizontal Pod Autoscaler** 或 **云 API** 创建新实例 <br>3. 新实例启动后向 **Service Registry** 注册 → etcd 同步 → Edge 实时感知 |
| **流量下降** | 1. QPS 持续低于阈值 30 min → Scale Service 触发 **scale‑down** <br>2. 选取低负载实例下线，先从 **lb_instance** 标记 `MAINTENANCE` → 等连接 drain 完成后 **DELETE** |
| **故障节点快速恢复** | 被动检测到错误率下降 → Health Service 将 `status` 从 `DOWN` → `UP` → etcd 更新 → Edge 恢复流量 |

### 4. 成本控制  

- **按需弹性**：Edge 节点使用 **裸金属/专用实例**（固定带宽），而 **后端业务** 采用 **云弹性实例**，只在高峰期扩容。  
- **流量分层**：DNS 只做粗粒度调度，真正的 200 Gbps 入口在 **Edge**，避免在公网层面购买过多带宽。  
- **冷热实例**：在非高峰期，后端业务保留 **低配实例**（只处理少量请求），高峰时再拉起 **高配实例**，通过 **自动伸缩** 完成。

## 第七步：常见面试追问与回答  

### Q1. **调度策略的细化**  
> *“如果后端集群中有部分实例的响应时间显著高于其他实例，你会如何在调度算法中体现这一点？”*  

**回答要点**：  

1. **实时性能监控**：在 Edge 上为每个实例维护 **滑动窗口平均响应时间（RTT）**，通过 **Prometheus** 暴露 `instance_latency_seconds`。  
2. **加权最小连接数**：调度算法改为 **Weighted Least Connections**，权重 = `1 / avg_latency`（或 `max(1, baseline / latency)`），响应慢的实例权重自动降低。  
3. **熔断**：若某实例的 **95% 延迟 > 阈值**（如 500 ms），直接标记为 **UNHEALTHY**，剔除出路由表。  
4. **动态权重更新**：每 5 秒从监控系统拉取最新指标，更新 **路由表**（etcd）并推送给 Edge。  

> 通过 **指标驱动的调度**，系统能自适应实例性能波动，避免慢实例拖慢整体 QPS。  

---

### Q2. **故障恢复与一致性**  
> *“当某个可用区全部节点失联时，系统如何保证流量不出现黑洞？节点状态在不同调度层之间如何保持一致？”*  

**回答要点**：  

1. **跨层状态同步**  
   - **健康检查** → `lb_instance.status` 写入 MySQL + **etcd**（通过 CDC/Change Data Capture）。  
   - **Etcd Watch** 在所有 Edge 节点实时感知状态变化，立即从本地路由表剔除对应实例。  

2. **多 AZ DNS 失效转移**  
   - **GeoDNS** 配置 **健康探针**（HTTP/HTTPS）检测 Edge IP 可达性。  
   - 若探针连续失败（如 3 次），自动从 DNS 响应列表中 **剔除该 AZ**，客户端随后重试得到其它 AZ 的 IP。  

3. **流量黑洞防护**  
   - **Edge** 在收到所有后端实例 `DOWN`（路由表为空）时，返回 **503 Service Unavailable** 并在后台触发 **自动扩容**（在其他 AZ 新建实例），同时告警。  
   - **控制平面** 通过 **etcd quorum** 保证即使一个 AZ 故障，仍能对配置做写入（多数节点仍存活）。  

---

### Q3. **扩展性与成本**  
> *“在流量高峰期需要临时提升带宽 3 倍，你会如何快速完成扩容？如何在保持低成本的前提下做到弹性伸缩？”*  

**回答要点**：  

1. **快速带宽扩容**  
   - **边缘云提供商**（如阿里云 CDN+SLB、AWS Global Accelerator）支持 **按需带宽**，通过 API 在几分钟内把 **Anycast IP** 的 **带宽上限** 调整至 600 Gbps。  
   - 同时 **弹性伸缩** Edge 实例（使用容器化部署）横向扩容 3 倍，保证每台机器的 **CPU/内存** 能支撑相应 QPS。  

2. **成本控制**  
   - **按使用计费**：只在峰值时段（如 12 h）开启 **高带宽**，峰后立即降回基础带宽。  
   - **预热策略**：提前 30 min 使用 **Warm‑up** 机制启动新实例，避免突发流量导致冷启动导致的性能抖动。  
   - **冷热实例分层**：在非峰期使用 **低配实例**（低成本），峰期 **自动升级** 为 **高配实例**（CPU×2、内存×2），通过 **云服务的弹性伸缩组** 完成。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **① 数据平面 vs 控制平面的边界** | 必须在 **2 ms** 延迟限制下既保证高速转发，又要让配置动态生效。 | - 先把所有 *慢* 的业务（健康检查、扩容）放到控制平面。<br>- 再评估转发层能否在 **用户态**（DPDK）完成，确认 **无状态**、**本地缓存**即可满足调度。 |
| **② 多层调度的一致性** | DNS、Edge、后端三层都有自己的路由表，如何保持它们在故障或扩容时同步？ | - 采用 **etcd** 作为全局单一事实来源。<br>- 通过 **Watch** + **gRPC** 实时推送给 Edge。<br>- DNS 使用 **健康探针** 与 **etcd** 状态联动，自动剔除不可用 IP。 |

### 2. 新手最容易犯的错误（至少 2 条）  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有功能都塞进单机 LB** | 试图让一台服务器完成 DNS、健康检查、监控、转发，导致 **单点故障、性能瓶颈**。 | 按 **层级**（DNS → Edge → 控制）拆分，确保每层都可以水平扩展。 |
| **忽视状态同步延迟** | 只在 Edge 本地缓存路由表，忘记 **更新策略**，导致剔除的节点仍被调度，引发 5xx 错误。 | 使用 **etcd Watch** + **心跳/TTL** 机制，确保状态在 **秒级** 内在所有节点统一。 |
| **只考虑平均延迟** | 只看 2 ms 平均，忽视 **尾部延迟**（99th、99.9th），在高并发时出现“卡顿”。 | 在监控中加入 **百分位延迟**，并在调度算法中使用 **响应时间权重**。 |

### 3. 学习建议和可延伸的方向  

| 方向 | 推荐学习资源 | 说明 |
|------|--------------|------|
| **网络编程与高性能转发** | 《DPDK 实战》、Linux `XDP` 官方文档、`netmap` | 掌握用户态零拷贝、内核 BPF，才能满足 2 ms 延迟。 |
| **分布式一致性** | 《Designing Data‑Intensive Applications》、etcd/Consul 官方手册 | 理解 Raft、Lease、Watch，保证控制平面强一致。 |
| **服务网格 / Envoy** | Envoy 官方文档、Istio 体系结构 | 学习 L7 路由、流量镜像、熔断等高级特性，可在 Edge 层复用。 |
| **弹性伸缩 & 监控** | Prometheus + Alertmanager 实战、Kubernetes HPA | 熟悉指标采集、自动化扩容的闭环。 |
| **安全与防 DDoS** | 云厂商 DDoS 防护白皮书、WAF 原理 | 虽不在本题范围，但在真实项目中必须考虑。 |

> **一句话总结**：把 **“高可用 + 高性能 + 可扩展”** 这三把钥匙分别放在 **DNS层、Edge层、控制层**，并用 **一致性存储** 把它们紧密联结，你的分布式层级负载均衡系统就已经具备了面试官想看的完整答案。
