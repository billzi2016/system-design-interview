# 第 49 天：设计 美团外卖配送系统

> 生成日期：2026-04-07

---

## 1. 题目背景  
美团外卖配送系统负责在用户下单后，将餐品从商家配送到用户手中，涉及骑手调度、路径规划、实时状态跟踪等关键业务。系统需要在高并发、低时延的环境下，保证配送效率和用户体验。

## 2. 面试场景设定  
> **面试官**：  
> “我们现在要设计美团外卖的配送系统，核心是如何在订单高峰期把订单快速、精准地匹配给合适的骑手，并实时跟踪配送进度。请你从需求出发，给出系统的整体架构设计思路，包括主要模块、数据流和关键技术选型。”

## 3. 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **下单后订单分配**：系统根据商家位置、骑手当前位置、配送距离、骑手负载等因素，将新订单实时匹配给最合适的骑手。 |
| 2 | **骑手接单与拒单**：骑手收到推送后可接受或拒绝，系统需在规定时间内重新调度。 |
| 3 | **路径规划与导航**：为每个已接单的骑手生成最优送达路径，支持实时路况更新和动态重新规划。 |
| 4 | **实时状态跟踪**：用户、商家、客服均可查看订单当前所处的状态（待取餐、配送中、已送达）以及骑手位置。 |
| 5 | **异常处理**：包括骑手迟到、取消、订单失效、路线不可达等场景的自动识别与二次调度。 |
| 6 | **统计与报表**：对配送时长、骑手响应时长、订单完成率等关键指标进行实时统计，供运营监控与调度优化使用。 |

## 4. 非功能性需求  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（活跃用户）** | 2,000 万 | 包括下单用户、骑手、商家后台用户。 |
| **峰值 QPS（订单下单）** | 15,000 QPS | 早晚高峰时段（12:00‑14:00、18:00‑20:00）。 |
| **订单匹配时延** | ≤ 200 ms（99%） | 从订单生成到骑手收到推送的时间。 |
| **路径规划时延** | ≤ 100 ms（99%） | 单次路径计算耗时。 |
| **系统可用性** | 99.99%（每月累计宕机 ≤ 4.38 小时） | 包含核心匹配、状态同步等关键服务。 |
| **存储容量** | 约 5 PB | 订单历史、轨迹日志、骑手轨迹数据（保留 90 天）。 |

## 5. 系统边界  

**本题范围（考虑）**  
- 订单从创建到最终送达的完整配送闭环（匹配、接单、路径规划、状态同步、异常处理）。  
- 实时骑手位置上报与地图路况的集成。  
- 调度算法（基于距离、负载、ETA 等因子）的设计与实现。  
- 高可用、伸缩的服务架构（微服务、消息队列、缓存等）。  

**不考虑（超出范围）**  
- 餐品的制作、商家内部订单管理系统。  
- 支付结算、优惠券、会员体系等前置业务。  
- 司机/骑手的薪酬结算、绩效考核系统。  
- 具体的前端 UI/UX 设计细节。  
- 第三方配送合作（如同城快递）与跨城市配送。

# 题解

# 美团外卖配送系统设计详解（手把手教学）

> **温馨提示**：下面的内容从“只要一个最小可用系统”一步步演进到“企业级高可用、可伸缩的完整架构”。每一步都解释「为什么这么做」，帮助你在面试中思路清晰、表达完整。

---

## 解题思路总览

1. **先把需求拆解成核心业务**：订单匹配、骑手接单、路径规划、状态同步、异常处理、统计报表。  
2. **估算规模**（DAU、QPS、数据量），把系统从「单机」想到「需要水平扩展」的程度。  
3. **从最小可用系统（MVP）出发**，先画出 **高层框图**，明确每个模块的职责。  
4. **逐层细化**：  
   - 数据库选型与表结构（事务 vs. 只读、冷热分离）  
   - API 接口（REST / gRPC）和调用链  
   - 关键组件实现细节（调度算法、实时位置、路况）  
5. **在每个关键点加入容错、缓存、异步化**，让系统满足 **低时延 & 高可用** 的非功能需求。  
6. **最后准备面试追问**：算法、强一致性、降级、弹性伸缩等。  

> **核心思路**：先把业务流弄清楚 → 用最简单的技术实现 → 再一步步加入“分布式要素”。这样可以避免“一开始就把系统搞得太复杂”，也更容易在面试里把思路说清楚。

## 第一步：理解需求与规模估算

### 1. 功能性需求拆解

| 编号 | 功能 | 关键子任务 |
|------|------|------------|
| 1 | **订单分配** | 计算骑手-订单距离、负载、ETA → 选出最优骑手 |
| 2 | **骑手接/拒单** | 推送、超时重调度、状态回写 |
| 3 | **路径规划** | 起点→取餐点→送达点的最短/最快路径，实时路况 |
| 4 | **实时状态跟踪** | 位置上报、状态流转、前端/商家/客服查询 |
| 5 | **异常处理** | 迟到、取消、不可达 → 自动二次调度 |
| 6 | **统计报表** | 实时指标（配送时长、响应时长） + 历史报表 |

> **注意**：这些功能大多数都是 **强实时**（毫秒级）和 **强一致**（订单状态不能出现“已接单” & “未接单”并存）。

### 2. 非功能性需求 & 规模估算

| 指标 | 估算值 | 计算方式/备注 |
|------|--------|--------------|
| **DAU** | 2,000 万 | 包括用户、骑手、商家 |
| **峰值 QPS（下单）** | 15,000 QPS | 12‑14、18‑20 两个高峰 |
| **订单匹配时延** | ≤ 200 ms（99%） | 包括调度、推送、骑手 ACK |
| **路径规划时延** | ≤ 100 ms（99%） | 单次路径计算 |
| **可用性** | 99.99% | 月均宕机 ≤ 4.38 h |
| **存储容量** | ~5 PB（90 天） | 轨迹日志、历史订单、日志等 |

**大致流量**（用于后面容量规划）：

- **下单**：15k QPS → 1.3B 单日订单（假设 24h）  
- **骑手位置上报**：每位骑手 1 次/秒 → 10k骑手 * 1s ≈ 10k QPS（实际会更多，取 30k QPS 预留）  
- **路径规划**：每单 1‑2 次 → 30k QPS  

> **如果不做容量估算**，很容易在设计时把关键瓶颈（如数据库写入、消息队列吞吐）忽略，导致后期“扩容成本爆炸”。

## 第二步：高层架构设计

### 1. 最小可用系统（MVP）

```
+-------------------+      +-------------------+      +-------------------+
|   前端/APP (用户)  | ---> |   API Gateway    | ---> |   订单服务 (单体) |
+-------------------+      +-------------------+      +-------------------+
                                    |
                                    v
                           +-------------------+
                           |   调度引擎 (单体) |
                           +-------------------+
                                    |
                                    v
                           +-------------------+
                           |   推送服务 (MQ)   |
                           +-------------------+
                                    |
                                    v
                           +-------------------+
                           |   骑手端 App      |
                           +-------------------+
```

- **优势**：实现最快，代码量最少，能在 30 分钟内跑通业务流。  
- **缺点**：**单点故障**、**水平扩展难**、**难以满足 200 ms 匹配时延**。

### 2. 逐步演进到微服务+分布式

下面的架构是面试常见的 “标准答案”，同时兼顾 **可伸缩性**、**容错**、**监控**：

```
+-------------------+          +-------------------+          +-------------------+
|   前端/APP (用户)  |  HTTPS   |   API Gateway    |  gRPC/HTTP|   鉴权/限流服务   |
+-------------------+--------->+-------------------+--------->+-------------------+
                                          |
                                          v
                               +---------------------------+
                               |   业务路由 (Dispatcher)   |
                               +---------------------------+
                                          |
                +-------------------------+--------------------------+
                |                         |                          |
                v                         v                          v
    +-------------------+   +--------------------+    +---------------------+
    |  订单服务 (Order) |   |  调度服务 (Match) |    |  位置服务 (Location)|
    +-------------------+   +--------------------+    +---------------------+
                |                         |                          |
                v                         v                          v
    +-------------------+   +--------------------+    +---------------------+
    |  订单库 (RDB)     |   |  调度缓存 (Redis) |    |  轨迹库 (TSDB)      |
    +-------------------+   +--------------------+    +---------------------+
                |                         |                          |
                v                         v                          v
    +-------------------+   +--------------------+    +---------------------+
    |  消息队列 (Kafka) |   |  路径规划服务 (Map) |    |  推送服务 (MQ)      |
    +-------------------+   +--------------------+    +---------------------+
                |                         |                          |
                +------------+------------+------------+-------------+
                             |                         |
                             v                         v
                     +-------------------+   +-------------------+
                     |  骑手端 APP (WebSocket) |   |  商家/客服端 (Web) |
                     +-------------------+   +-------------------+
```

#### 关键技术选型说明

| 组件 | 推荐技术 | 为什么 |
|------|----------|--------|
| **API Gateway** | Kong / Nginx + Lua / Spring Cloud Gateway | 统一入口、限流、鉴权、灰度发布 |
| **鉴权/限流** | JWT + Redis 滑动窗口 | 轻量、支持分布式 |
| **订单服务** | Spring Boot + MySQL (InnoDB) | 事务强一致、成熟生态 |
| **调度服务** | Java / Go + Redis + Kafka | Redis 用作 **实时位置缓存 & 排队**，Kafka 负责 **异步调度任务** |
| **位置服务** | Go + GeoHash + TiDB (或 ClickHouse) | 大批量写入、空间索引快速查询 |
| **路径规划** | 高德/腾讯地图 API + 本地路网（GraphHopper）+ Redis 缓存 | 实时路况、缓存热点路径，满足 ≤100 ms |
| **消息队列** | Apache Kafka（高吞吐）+ RocketMQ（低延迟）| 订单、调度、轨迹均使用 **分区**，支持 **Exactly‑once** |
| **推送** | MQTT / WebSocket + FCM/APNs | 实时、低功耗 |
| **监控/日志** | Prometheus + Grafana + ELK (Elasticsearch‑Logstash‑Kibana) | 可观测性、报警 |
| **容错/降级** | Hystrix / Resilience4j + Sentinel | 限流、熔断、降级回退 |
| **容器化** | Docker + Kubernetes (k8s) | 自动弹性伸缩、滚动升级 |

> **如果不做这些选型**，比如直接用单体 + MySQL，**高并发下写入锁竞争**、**路径计算阻塞**、**推送延迟**都会成为系统瓶颈，无法满足 200 ms 匹配时延。

## 第三步：数据库设计

### 1. 数据库划分原则

| 业务 | 读写特性 | 选型 | 说明 |
|------|----------|------|------|
| 订单核心（创建、状态变更） | 强一致、事务 | **MySQL（InnoDB）** | 需要 ACID，订单状态不能出现冲突 |
| 骑手位置信息（实时） | 高写入、时序查询 | **TiDB / ClickHouse**（列式 + 时序） | 支持大批量写入、快速时间范围查询 |
| 轨迹日志（历史） | 归档、分析 | **HDFS + Parquet** 或 **OSS** | 大容量、离线分析 |
| 路径缓存（热点路径） | 读多写少 | **Redis**（TTL） | 低时延、自动失效 |
| 调度缓存（骑手可用状态） | 读写均衡、分布式锁 | **Redis + RedLock** | 实时选骑手、避免抢单竞争 |

### 2. 核心表结构（示例）

#### 2.1 订单表（order）

| 字段 | 类型 | 说明 |
|------|------|------|
| order_id | BIGINT PK | 全局唯一（雪花算法） |
| user_id | BIGINT | 下单用户 |
| merchant_id | BIGINT | 餐厅 |
| status | TINYINT | 0:待分配,1:待取餐,2:配送中,3:已完成,4:已取消 |
| created_at | DATETIME | 下单时间 |
| assigned_rider_id | BIGINT | 当前匹配的骑手（NULL 表示未分配） |
| pickup_lat/lng | DOUBLE | 商家坐标 |
| delivery_lat/lng | DOUBLE | 用户坐标 |
| estimate_eta | INT | 预计送达时间（秒） |
| version | INT | 乐观锁（防并发更新） |
| ... |  | 其他业务字段（菜品、优惠等） |

> **乐观锁**（`version`）是防止 **并发抢单** 的关键手段：在调度服务写 `assigned_rider_id` 前检查 `version`，若不匹配则说明已被其他调度抢占，直接放弃。

#### 2.2 骑手状态表（rider_status）

| 字段 | 类型 | 说明 |
|------|------|------|
| rider_id | BIGINT PK | |
| cur_lat | DOUBLE | 实时位置 |
| cur_lng | DOUBLE | |
| last_report_ts | BIGINT | 最近上报时间（epoch ms） |
| status | TINYINT | 0:空闲,1:已接单,2:配送中,3:休息 |
| load | INT | 当前负载（已接单数） |
| version | INT | 乐观锁 |
| ttl | INT | 自动失效时间（5s） |

> **Redis** 中使用 **Hash** 存储，每个骑手一个 `hash`，并配合 **ZSET**（基于距离）快速查询最近空闲骑手。

#### 2.3 轨迹表（rider_track）

| 字段 | 类型 | 说明 |
|------|------|------|
| rider_id | BIGINT | |
| ts | BIGINT | 上报时间（epoch ms） |
| lat | DOUBLE | |
| lng | DOUBLE | |
| speed | DOUBLE | 可选，计算实时 ETA 用 |

- 使用 **ClickHouse** 按 `rider_id`、`ts` 分区，支持毫秒级查询。

#### 2.4 路径缓存表（path_cache） - Redis

- **Key**：`path:{origin_hash}:{dest_hash}`  
- **Value**：JSON `{polyline: [...], distance: xxx, duration: xxx, ts: epoch}`  
- **TTL**：5 min（路况会变化）

### 3. 数据一致性方案

| 场景 | 需要强一致性 | 解决方案 |
|------|---------------|----------|
| 订单状态更新 | ✅ | 使用 **数据库事务** + **消息事务**（Kafka事务） |
| 骑手状态（可用/已接单） | ✅ | **Redis 分布式锁** + **乐观锁** |
| 轨迹写入 | ❌（最终一致） | **异步写入 ClickHouse**，写入失败走 DLQ（Dead Letter Queue） |
| 路径缓存 | ❌（可容忍旧路径） | **Cache‑Aside**，缓存失效后重新查询 |

> **如果不保证强一致性**，比如直接在 MySQL 更新后异步发送 Kafka，可能出现“订单已被骑手接单但消息未投递”，导致调度服务误认为订单仍未分配，出现 **双骑手** 的灾难性错误。

## 第四步：核心 API 设计

> **原则**：REST 用于 CRUD（订单、骑手），gRPC 用于高频、低时延的内部调用（调度、位置推送），统一返回结构。

### 1. 公共返回结构

```json
{
  "code": 0,               // 0=成功, >0=错误码
  "msg": "OK",
  "data": {...}            // 业务数据
}
```

### 2. 关键外部 API（用户/商家端）

| 方法 | 路径 | 功能 | 请求示例 | 响应关键字段 |
|------|------|------|----------|--------------|
| POST | /api/v1/orders | 创建订单 | `{userId, merchantId, items[], deliveryLat, deliveryLng}` | `orderId, status=0` |
| GET  | /api/v1/orders/{id} | 查询订单状态 | - | `status, riderId, eta` |
| POST | /api/v1/riders/{riderId}/accept | 骑手接受订单 | `{orderId}` | `orderId, status=2` |
| POST | /api/v1/riders/{riderId}/reject | 骑手拒单 | `{orderId}` | `orderId, status=0` |
| GET  | /api/v1/riders/{riderId}/track | 拉取骑手实时轨迹 | `?since=timestamp` | `trackList[]` |

### 3. 内部高频 API（gRPC）

```proto
service MatchService {
  // 调度入口，传入订单信息，返回骑手ID（若无则返回空）
  rpc DispatchOrder(DispatchRequest) returns (DispatchResponse);
  // 骑手位置上报（流式）
  rpc ReportLocation(stream LocationReport) returns (LocationAck);
}
```

- **DispatchOrder**：内部调用，使用 **Redis** 快速查询最近空闲骑手，返回 `riderId`。若返回空，则写入 **Kafka** `order_retry` 主题，由后台调度任务重试。
- **ReportLocation**：骑手端通过 **WebSocket** → **gRPC**，每秒上报一次。服务端写入 **TiDB**（时序）并更新 **Redis** 缓存。

### 4. 异步消息（Kafka 主题）

| 主题 | 生产者 | 消费者 | 目的 |
|------|--------|--------|------|
| order_created | 订单服务 | 调度服务 | 新订单进入调度 |
| order_assigned | 调度服务 | 推送服务、订单服务 | 推送给骑手 & 写库 |
| rider_location | 位置服务 | 调度服务、路径规划 | 实时位置更新 |
| order_event | 订单服务 | 统计服务 | 生成运营报表 |
| anomaly_alert | 调度/位置服务 | 调度服务 | 异常二次调度 |

> **如果不使用消息队列**，所有服务都直接同步调用，**峰值 QPS** 时会导致 **链路阻塞**，系统吞吐不到需求。

## 第五步：详细组件设计

下面逐个拆解核心组件的实现细节、选型、容错与性能优化。

### 1. 调度服务（Match Service）

#### 1.1 工作流程

1. **订单创建** → 写入 MySQL → **事务** 中同步发送 `order_created` Kafka 消息（**事务性生产**）。
2. **调度服务** 消费 `order_created`，进入 **调度引擎**：
   - 从 **Redis** 读取所有 **空闲骑手**（ZSET 按距离排序）。
   - 过滤掉 **负载 >阈值**、**距离 > 5km**、**最近上报时间 > 5s** 的骑手。
   - 计算 **综合评分**：`score = w1*distance + w2*load + w3*ETA`（权重可动态调节）。
   - 取 **Top‑N**（默认 3）候选，依次尝试 **抢单**（乐观锁 `version`）。
3. **抢单成功** → 更新 `order.assigned_rider_id`、`order.status=1`、`rider_status.load+=1`，写入 MySQL、Redis。
4. **推送** → 通过 **MQ（RocketMQ）** 发 `order_assigned`，推送服务把订单推给骑手端。
5. **超时未接** → 调度服务监听 `order_accept_timeout`（基于 **Kafka 延迟队列**），重新进入调度流程。

#### 1.2 防止“抢单”竞争的技术

- **乐观锁**：`UPDATE order SET assigned_rider_id=?, version=version+1 WHERE order_id=? AND version=?`
- **分布式锁**（RedLock）：在抢单前对 `order:{orderId}` 加锁，确保同一时间只有一个调度实例在处理。
- **幂等性**：Kafka 消息使用 **唯一键**（orderId），消费端幂等处理（去重表）。

#### 1.3 高峰期策略（动态调度）

| 场景 | 调整措施 |
|------|----------|
| **骑手供给不足** | 降低 `w2`（负载权重），扩大搜索半径至 8km；开启 **“批量调度”**（一次分配多个相邻订单给同一骑手） |
| **骑手拥堵** | 引入 **实时路况 ETA**（从路径规划服务获取），把 **ETA** 放入评分公式；对拥堵路段的骑手 **降权** |
| **订单积压** | 开启 **“抢单模式”**：向多骑手推送同一订单，取最快接受的；使用 **秒级失效**（order_valid_until）防止长时间占用 |

### 2. 位置服务（Location Service）

- **上报协议**：骑手端使用 **WebSocket** + **protobuf**，每秒一次，内容 `{riderId, lat, lng, timestamp}`。
- **入口层**：`nginx + stream` → `location-gateway`（Go），做 **限流**（每秒 ≤ 2 条）和 **鉴权**。
- **处理**：写入 **TiDB**（时序） + **Redis**（Hash）+ **ZSET**（距离索引）。  
  - **Hash**：`rider:{id}` → 实时位置、状态。  
  - **ZSET**（geo）使用 **GeoHash** 作为成员，`GEOADD riders_geo lng lat riderId`，支持 **半径查询**（`GEORADIUS`）。
- **容错**：若写入 TiDB 失败，先写入 **本地缓冲队列**，后台异步重试；若 Redis 故障，直接返回错误，调度服务会降级为 **基于 MySQL 的查询**（慢但可用）。

### 3. 路径规划服务（Map Service）

#### 3.1 方案

- **主方案**：调用 **高德/腾讯地图** **路线规划 API**（返回最短路径、预计耗时）。  
- **备选**：本地部署 **GraphHopper**（基于 OpenStreetMap）+ **Redis** 缓存热点路径。  

#### 3.2 调用流程

1. 调度服务或骑手端请求路径时，先查询 **Redis Cache**（Key = `path:{originHash}:{destHash}`）。  
2. **缓存未命中** → 调用外部地图 API（带实时路况），返回 `polyline、distance、duration`。  
3. 将结果 **写入 Redis**，TTL 5 min。  
4. 若外部 API **超时**（>80 ms），直接返回 **近似路径**（直线距离 * 1.3），并记录 **降级日志**。

#### 3.3 性能优化

- **并发限流**：使用 **令牌桶**（RateLimiter）防止瞬时流量冲击外部 API（如高峰时段）。  
- **批量预取**：对同一商家附近的订单，提前批量请求路径，缓存后供后续调度使用。  
- **热点路径**：统计 **同一商家 → 同一区域** 的请求频次，预热到本地图引擎。

### 4. 推送服务（Push Service）

- **技术**：基于 **MQTT**（低功耗、持久会话）+ **APNs/FCM**（移动端推送）。  
- **流程**：  
  1. `order_assigned` 消息到达推送服务。  
  2. 查找骑手的 **设备 token**（MySQL `rider_device` 表）。  
  3. 通过 **MQTT** 将订单信息推送到骑手端的 **WebSocket**，如果 WebSocket 不在线，则走 **FCM**。  
- **可靠性**：使用 **QoS 1**（至少一次）+ **消息幂等**（骑手端去重）。  
- **监控**：统计 **推送成功率**、**平均时延**，若低于阈值（如 95% < 150 ms）触发 **告警**。

### 5. 异常处理（Anomaly Service）

| 异常场景 | 检测方式 | 处理策略 |
|----------|----------|----------|
| **骑手迟到**（实际 ETA > 预计 + 5min） | 定时比较位置 vs. 预计轨迹（位置服务） | 发送 **迟到提醒** 给骑手；若迟到超过阈值，自动 **二次调度**（重新匹配） |
| **骑手取消** | 接收 `reject` 消息或心跳失联 | 立即 **回滚订单状态**（status=0），重新进入调度 |
| **路线不可达**（路径规划返回错误） | 调用地图 API 失败 | 降级为 **直线距离** 估算；若仍不可达，标记商家为 **不可配送** 并通知用户 |
| **系统故障**（调度服务宕机） | 心跳监控（Prometheus） | 自动 **切换至备份调度实例**；如果全部不可用，进入 **手工调度** 模式（运营后台） |

> **关键点**：所有异常处理必须 **幂等**，防止同一个订单被多次二次调度。

### 6. 统计与报表（Analytics Service）

- **实时指标**（Kafka → Flink）：
  - `order_created_per_minute`、`order_assigned_latency`、`rider_accept_rate`  
- **离线报表**（Spark / Hive）：
  - 日/周/月 **配送时长**、**骑手效率**、**订单完成率**  
- **数据存储**：实时指标写入 **Prometheus**（监控）+ **ClickHouse**（OLAP），离线报表写入 **Hive**（HDFS）供 BI 使用。

## 第六步：扩展性与高可用设计

### 1. 弹性伸缩（Auto‑Scaling）

| 层级 | 伸缩指标 | 触发方式 |
|------|----------|----------|
| **API Gateway** | CPU、QPS、请求延时 | k8s HPA（Horizontal Pod Autoscaler） |
| **订单/调度服务** | Kafka lag、CPU、内存 | 自动扩容 Pod + **Kafka 分区** 增加 |
| **位置服务** | 位置上报 QPS、Redis 命中率 | 动态增加 **Redis Cluster** 节点 |
| **路径规划** | 外部 API 调用量、缓存命中率 | 增加本地 GraphHopper 实例，或 **预热缓存** |
| **推送服务** | MQTT 连接数、推送延时 | 横向扩容实例 + **负载均衡** |

- **水平扩容**：所有无状态服务（Gateway、调度、位置、推送）均采用 **Docker + k8s**，**Stateless**，通过 **Service** 进行负载均衡。  
- **垂直扩容**：对 **数据库**（MySQL、Redis）采用 **主从复制 + 读写分离**，必要时使用 **分库分表**（按城市、时间分片）。

### 2. 高可用（HA）设计要点

| 组件 | HA 方案 | 容错点 |
|------|----------|--------|
| **API Gateway** | 多实例 + **LVS/NGINX** 负载均衡 + **健康检查** | 单点故障 → 自动剔除 |
| **MySQL** | 主‑从双机 + **Semi‑Sync**，故障切换使用 **MHA** 或 **Orchestrator** | 主库挂掉 → 自动切换 |
| **Redis** | **Cluster**（3 主 3 从） + **故障转移**（Redis Sentinel） | 节点失效 → 自动选举 |
| **Kafka** | 多 broker（≥3）+ **Replication** (RF=3) + **Controller** 选举 | Broker 故障 → 仍可消费 |
| **服务实例** | **k8s** Deployment + **PodDisruptionBudget** | 滚动升级时不影响整体可用性 |
| **路径规划外部 API** | **双源**（高德 + 腾讯）+ **本地缓存** | 第三方 API 超时 → 降级本地路径 |
| **监控告警** | **Prometheus** + **Alertmanager** → **PagerDuty** | 故障未被发现 → 业务受影响 |

### 3. 数据一致性与事务

- **订单状态**：使用 **数据库事务 + Kafka 事务**，保证 **“写库成功 → 消息一定投递”**（Exactly‑once）。  
- **骑手状态**：使用 **Redis 分布式锁 + 乐观锁**，在写库前先锁定，防止并发抢单。  
- **最终一致**：轨迹、统计数据使用 **异步写入**，业务对这类数据容忍 **秒级延迟**。

### 4. 降级与熔断

| 场景 | 降级措施 |
|------|----------|
| **路径规划超时** | 直接返回 **直线距离估算**，并在后台补偿 |
| **调度服务不可用** | 直接进入 **手工调度**（运营后台）或 **抢单模式**，向所有空闲骑手广播 |
| **推送服务故障** | 暂停推送，改为 **短信通知**（通过第三方短信服务） |
| **Redis 故障** | 回退到 **MySQL** 查询骑手位置（慢但可用） |
| **Kafka 堆积** | 限流新订单进入调度，使用 **本地缓存队列** 暂存，防止系统崩溃 |

> **不要忘记**：所有降级路径都必须 **记录日志** 并 **告警**，防止在高峰期产生“黑洞”。

### 5. 安全与合规

- **鉴权**：统一使用 **OAuth2 + JWT**，每个微服务通过 **网关** 校验。  
- **数据加密**：骑手位置、订单信息在 **传输层 TLS** 加密，敏感字段（手机号）在 **存储层** 使用 **AES** 加密。  
- **审计日志**：关键操作（订单状态变更、骑手接单）写入 **审计库**（只读），满足监管需求。

## 第七步：常见面试追问与回答

### 1. “如果骑手数量远大于待配送订单，如何避免‘抢单’导致的资源竞争？”

**回答要点**：

1. **乐观锁 + 版本号**：在 `order` 表加 `version`，更新时检查 version，只有一次成功。  
2. **分布式锁（RedLock）**：在抢单前对 `order:{id}` 加锁，确保同一时间只有一个实例在处理。  
3. **幂等消费**：Kafka 消息使用 `orderId` 作为唯一键，消费端先检查是否已处理（去重表）。  
4. **批量调度**：若骑手数量远大，采用 **“批量匹配”**：一次把若干相邻订单分配给同一骑手，减少锁竞争次数。  

**不这么做的后果**：大量并发抢单会导致 **写冲突、事务回滚**，严重时出现 **订单多骑手同时接单**，用户体验灾难。

---

### 2. “在高峰期出现大面积骑手拥堵时，你会如何动态调整匹配策略？”

**回答要点**：

- **实时路况**：从路径规划服务获取每条道路的 ETA，加入调度评分公式 `score = w1*distance + w2*load + w3*ETA`。  
- **负载阈值**：对同一区域的骑手设置 **负载上限**（如每位骑手最多 2 单），超出则降低其匹配权重。  
- **扩大搜索半径**：将搜索范围从 5km 扩大到 8km 或 10km，利用相对空闲的外环骑手。  
- **批量调度**：对同一商家的多个订单一次性派给同一骑手，形成 “路线合并”，减少单车往返次数。  
- **动态权重调节**：运营后台提供 **调度控制面板**，可以实时调高/调低 `w2`（负载）或 `w3`（ETA）权重。

---

### 3. “订单状态在多服务之间是如何保证强一致性的？”

**回答要点**：

1. **数据库事务**：订单状态的每一次变更都在 MySQL 中使用 **事务** 完成。  
2. **Kafka事务**：在同一个事务里先提交 DB，随后 **提交 Kafka 事务**，确保 **“写库成功 → 消息必达”**。  
3. **幂等消费**：下游服务（调度、推送）消费时先检查 `order_status_log` 表，若已处理则直接 ACK，防止重复执行。  
4. **两段提交（2PC）**：在极端情况下（如跨库），可以使用 **分布式事务管理器（Seata）** 实现两段提交。  

**不保证强一致** 会导致 **“状态错位”**（如用户已看到已配送，实际骑手未接单），极大影响用户信任。

---

### 4. “如果路径规划服务宕机，系统应如何降级保持业务可用？”

**回答要点**：

- **缓存回退**：先查询 **Redis** 中的路径缓存，若未命中则使用 **直线距离估算**（`distance = haversine(lat1,lng1,lat2,lng2) * 1.3`）并返回。  
- **本地图引擎**：提前部署 **GraphHopper**（离线路网），在外部 API 不可用时切换到本地引擎。  
- **延迟补偿**：后续在后台异步调用外部 API，更新缓存，以便后续订单使用更精确的 ETA。  
- **告警**：通过 **Prometheus** 监控路径规划调用错误率，一旦超过阈值立即报警并触发 **运维手动切换**。

---

### 5. “面对突发的流量峰值（如双十一），你会在哪些层面做弹性伸缩？”

**回答要点**：

| 层面 | 伸缩手段 |
|------|----------|
| **入口层**（API Gateway） | **水平扩容** + **IP 限流**（防止 DDOS） |
| **业务层**（订单、调度、位置） | **k8s HPA** 基于 **CPU/QPS/Kafka Lag** 自动扩容；**Kafka 分区** 预留足够分区数 |
| **缓存层**（Redis） | **Cluster** 自动分片，**水平扩容**（增加节点）并使用 **Rebalancing** |
| **数据库层**（MySQL） | **读写分离** + **分库分表**（按城市或时间）；**热点表**使用 **分表**（如订单表） |
| **消息层**（Kafka） | **Broker** 增加，**复制因子** 3；**生产者** 开启 **批量发送**（batch.size） |
| **路径规划** | **预热热点路径** 缓存；**本地图引擎** 扩容；**限流** 防止外部 API 触发限额 |
| **监控告警** | **自动扩容** 与 **阈值报警** 紧密结合，确保伸缩成功后及时恢复阈值 |

> **注意**：弹性伸缩的前提是 **无状态**（或状态可外部化），因此在设计时要把 **Session**、**锁**等状态存入 **Redis** 或 **数据库**，而不是放在单个实例的内存中。

## 心得与反思

### 1. 本题最难的 1‑2 个设计决策及思考过程

| 决策 | 困难点 | 思考路径 |
|------|--------|----------|
| **调度算法的并发安全** | 高并发抢单会导致大量数据库冲突、订单状态不一致。 | - 先从业务层面明确「一次只能有一个骑手接单」的强制规则。<br>- 决定使用 **乐观锁 + 分布式锁** 双保险。<br>- 再考虑 **幂等消费**，防止消息重复导致二次写库。<br>- 通过 **压测**（JMeter）验证锁竞争对 QPS 的影响，最终确定 **RedLock + version** 为最佳折中。 |
| **路径规划的低时延 & 降级** | 实时路况查询需调用外部第三方 API，网络波动会直接影响配送 ETA，进而影响调度决策。 | - 评估外部 API **SLA**（95% ≤ 80 ms）是否满足需求。<br>- 设计 **双源+本地缓存** 方案：先查 Redis，未命中再调用外部；若外部超时则回退 **直线距离估算**。<br>- 引入 **GraphHopper** 作为本地备份，避免单点故障。<br>- 在压测中模拟外部 API 延迟，验证降级路径对整体时延的影响，确保 ≤ 200 ms。 |

### 2. 新手最容易犯的错误（至少 2 条）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有业务写在单体服务里**，没有拆分微服务。 | 难以水平扩展，单点故障导致整个配送系统不可用。 | 按业务域拆分为 **订单、调度、位置、路径、推送** 等独立服务，使用 **API Gateway** 统一入口。 |
| **忽视消息的幂等性和事务性**，直接在服务之间同步调用。 | 高并发时出现 **重复消费**、**状态错位**（如双骑手接单）。 | 使用 **Kafka 事务**、**唯一键去重**、**数据库乐观锁**，确保 **Exactly‑once** 语义。 |
| **只关注功能实现，忽略监控/降级**。 | 系统故障时难以定位，无法快速恢复。 | 统一 **Prometheus + Grafana** 监控；在每个关键路径加入 **熔断/降级**（Hystrix、Sentinel）。 |
| **对实时位置信息直接写入 MySQL**。 | 写入压力极大，导致 DB 瓶颈，延迟 > 200 ms。 | 使用 **时序数据库**（TiDB/ClickHouse） + **Redis** 缓存，MySQL 只保存少量状态。 |

### 3. 学习建议和可延伸的方向

1. **系统设计基础**：先掌握 **CAP、BASE、CAP Theorem、分布式事务**，再学习 **微服务、消息队列、缓存、负载均衡**。  
2. **算法与数据结构**：熟悉 **GeoHash、K‑最近邻、最短路径（Dijkstra、A*）**，能够在面试中快速说明调度评分模型。  
3. **实战项目**：自行实现一个 **简化版外卖调度**（使用 Spring Boot + Redis + Kafka），跑压测，体会锁竞争与消息幂等的实际表现。  
4. **可观测性**：学习 **OpenTelemetry**、**Jaeger**，在分布式系统里追踪每一步请求链路，面试时可以说出“**全链路追踪**”。  
5. **容灾与降级**：阅读 **Netflix OSS**（Hystrix、Ribbon、Eureka）或 **Alibaba Sentinel** 的设计思想，能在追问时给出成熟的降级方案。  

> **最后的提醒**：系统设计面试不是让你画出完美的图，而是考察 **思考框架**、**权衡取舍** 与 **沟通表达**。先从 **业务目标** 出发，层层拆解，最后再补齐 **可用性、伸缩性、监控**，你的答案自然会完整且有说服力。
