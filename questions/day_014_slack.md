# 面试题：设计类 Slack 的企业即时通讯系统

## 1. 题目背景
Slack 是一款面向团队的即时通讯与协作平台，提供基于频道（Channel）和私聊的消息交流、文件共享、搜索以及与第三方服务的集成等功能。面试官希望你在限定的业务范围内，设计一个能够支撑大规模企业用户的核心聊天系统。

## 2. 面试场景设定
> **面试官**：  
> “我们现在要设计一个类似 Slack 的企业即时通讯系统。请先从整体架构出发，说明你会如何支撑 **数千万活跃用户** 的实时聊天、文件传输和全文搜索。随后我们会逐步深入到存储、消息投递可靠性、伸缩性等细节。”

## 3. 功能性需求
| 编号 | 功能描述 |
|------|----------|
| 1 | **频道（Channel）和私聊**：用户可以在公开或私密频道中发送文字、表情、代码块等；支持一对一私聊。 |
| 2 | **消息历史 & 全文搜索**：用户可以随时检索过去 90 天内的聊天记录（包括文字、文件名、链接）。 |
| 3 | **文件上传与预览**：支持上传图片、文档、压缩包等，文件大小 ≤ 1 GB，能够在聊天窗口直接预览常见格式。 |
| 4 | **实时通知**：在用户在线时推送实时消息；离线时通过移动端推送（APNs / FCM）或邮件提醒。 |
| 5 | **第三方集成（Webhook / Bot）**：能够接收外部系统的事件推送（如 GitHub、Jira），并以 Bot 形式在相应频道发送消息。 |
| 6 | **权限与角色管理**：支持普通成员、管理员、所有者等角色，控制频道的创建、邀请、删除等操作。 |

## 4. 非功能性需求
| 指标 | 目标值（估算） | 说明 |
|------|----------------|------|
| **DAU（Daily Active Users）** | 50 M | 全球企业用户每日活跃人数 |
| **QPS（峰值请求）** | 10 k req/s | 峰值时每秒请求数（发送消息、拉取历史、文件下载等） |
| **平均端到端延迟** | ≤ 200 ms | 消息从发送端到接收端的感知延迟 |
| **可用性** | 99.9%（月度） | 包括消息投递、文件存取等核心功能的可用性 |
| **存储容量** | 1 PB/年 | 以 90 天保留期计，包含消息、索引、文件元数据 |
| **一致性要求** | 最终一致性 + 事务保证（发送消息成功即对发送方可见） | 关键是对发送方的强一致，接收方可接受短暂的复制延迟 |

## 5. 系统边界（本题范围 / 不考虑的功能）
**本题范围内**  
- 核心实时聊天（文字、表情、代码块）  
- 文件上传/下载（≤ 1 GB）与元数据存储  
- 消息持久化、检索（全文搜索）  
- 基本的推送通知与离线提醒  
- 简单的 Bot/Webhook 接入与权限控制  

**本题不考虑**  
- 视频/语音通话、屏幕共享等媒体流功能  
- 企业级合规归档、审计日志（如 GDPR、HIPAA）  
- 多租户计费、企业组织结构的细粒度管理  
- 客户端 UI/UX 细节、跨平台 SDK 实现  
- 高级机器学习功能（如智能回复、情感分析）  

## 6. 提示与追问
1. **消息投递可靠性**：  
   - “如果用户 A 在网络不稳定的情况下发送了一条消息，系统如何保证该消息不丢失且仅投递一次？”  

2. **扩展性与分区策略**：  
   - “面对 10 k QPS 的峰值请求，你会如何划分数据分区（Channel、User）以及选用什么一致性模型？”  

3. **全文搜索实现**：  
   - “考虑到消息量大且需要支持实时搜索，你会选用哪种搜索引擎/索引结构，如何保证搜索延迟在 200 ms 以内？”

# 题解

# 设计类 Slack 的企业即时通讯系统（完整解答）

> **温馨提示**：下面的每一步都假设你是系统设计的新人。我们会从“只有一台服务器、只能跑 10 QPS” 的 **最小可用系统** 出发，逐层加上缓存、分片、容错、搜索等功能，帮助你形成完整的思路。阅读时请把每一章节当成“课堂”，先理解概念，再看实现细节，最后再去思考面试官可能的追问。

## ## 解题思路总览

| 步骤 | 目的 | 关键点 |
|------|------|--------|
| 1️⃣ 理解需求 & 规模估算 | 明确**功能**、**非功能**、**边界**，把业务转换为技术指标（DAU、QPS、存储） | 业务拆解 → 关键指标 → 估算 |
| 2️⃣ 高层架构草图 | 把系统划分为**客户端、API 网关、业务服务、存储、搜索、推送**等大块 | 单体 → 微服务、分层、职责单一 |
| 3️⃣ 数据模型 & 选型 | 设计**消息、频道、用户、文件**等表结构，决定 **关系型/NoSQL** 组合 | 读写比例、事务需求、扩展性 |
| 4️⃣ API 设计 | 把业务拆成 **REST / gRPC** 接口，定义请求/响应，考虑 **幂等** 与 **鉴权** | 接口粒度、错误码、限流 |
| 5️⃣ 详细组件设计 | 逐个剖析 **消息写入、投递、持久化、搜索、文件、推送** 的实现细节 | 流程图、关键技术、缓存/队列 |
| 6️⃣ 扩展性 & 高可用 | 讨论 **分区、复制、负载均衡、容灾、监控**，保证 99.9% SLA | 分片键、CAP、异步复制 |
| 7️⃣ 常见追问 | 预设面试官可能的 5~8 个问题并给出思考路径 | 可靠性、最终一致、成本、升级 |
| 8️⃣ 心得 & 反思 | 归纳本题的难点、常见错误、学习路径 | 复盘、进阶方向 |

> **核心思路**：先让系统 **能跑通**（最小可用），再让它 **跑得稳**（容错、监控），最后让它 **跑得快**（分片、缓存、搜索）。每一次“加”都是有目的的，而不是盲目堆砌。

## ## 第一步：理解需求与规模估算

### 1️⃣ 功能性需求拆解

| 功能编号 | 关键子功能 | 对应的系统模块 |
|----------|------------|----------------|
| 1 | 频道/私聊、文字/表情/代码块 | **消息服务**、**权限服务** |
| 2 | 消息历史、全文搜索（90 天） | **持久化**、**搜索引擎** |
| 3 | 文件上传、预览、大小 ≤ 1 GB | **文件服务**、**对象存储** |
| 4 | 实时推送、离线推送、邮件 | **推送服务**、**通知服务** |
| 5 | Webhook/Bot 事件入口 | **集成服务**、**消息生产者** |
| 6 | 角色/权限管理 | **用户/权限服务** |

> **为什么要拆**：面试官往往会针对每个子功能追问实现细节。如果我们一开始就把“聊天系统”当成一个黑盒，后面很容易卡住。

### 2️⃣ 非功能性需求转化为技术指标

| 指标 | 估算过程 | 结果 |
|------|----------|------|
| **DAU** | 50 M → 假设 20% 同时在线 → 10 M 并发连接 | 10 M 同时在线 |
| **峰值 QPS** | 发送消息、拉取历史、文件下载综合 → 10 k req/s（约 1 req/用户/秒） | 10 k req/s |
| **消息吞吐** | 每条消息 200 B（文字） → 10 k × 200 B = 2 MB/s ≈ 17 TB/天（含文件元数据） | 约 6 PB/年 |
| **存储** | 90 天保留 → 6 PB/年 × 0.25（90/365） ≈ 1.5 PB 有效存储 | 1 PB/年（题目要求） |
| **延迟** | 客户端 → API Gateway → 消息服务 → 投递 → 客户端 ≤ 200 ms | 需要低延迟网络、异步投递 |
| **可用性** | 99.9% → 单点故障容忍 ≤ 8.76 h/年 | 多 AZ、冗余、自动恢复 |

> **常见误区**：只看 QPS 而忽略 **带宽**、**存储增长**。实际设计时要把 **数据体积**（文字+文件）算进去，否则后期容量会爆炸。

### 3️⃣ 边界划分（Scope）

- **纳入**：核心聊天、文件、搜索、推送、Bot。
- **排除**：语音/视频、合规归档、计费、细粒度组织结构。

> **目的**：让面试官看到你能**聚焦核心**，而不是“一口气想实现所有功能”。如果后面被问到不在范围的功能，你可以说“可以在此基础上再扩展”。

## ## 第二步：高层架构设计

下面给出 **文字版** 的层次结构，随后每层会展开说明。

```
+------------------------------------------------------------+
|                       Client（Web/Mobile/Desktop）       |
+--------------------+--------------------+------------------+
                     |                    |
               +-----v-----+        +-----v-----+
               | API Gateway (REST/gRPC, TLS, Auth) |
               +-----+-----+        +-----+-----+
                     |                    |
            +--------v--------+   +-------v-------+
            |   Load Balancer (L7)   |   Rate Limiter |
            +--------+--------+   +-------+-------+
                     |                    |
        +------------v------------+-------------------+
        |   微服务层（业务拆分）  |   公共组件层        |
        |                         | (Cache, MQ, Metrics)|
        +-----------+------------+-------------------+
                    |                              |
   +----------------v----------------+   +----------v----------+
   |   消息服务 (Message Service)   |   |   文件服务 (File Service) |
   +-----------+--------------------+   +-----------+----------+
               |                                    |
   +-----------v-----------+            +-----------v-----------+
   |   消息写入库 (MySQL/TSDB)   |            |   对象存储 (S3/OSS) |
   +-----------+-----------+            +-----------+-----------+
               |                                    |
   +-----------v-----------+            +-----------v-----------+
   |   消息队列 (Kafka)    |            |   CDN / 预览服务       |
   +-----------+-----------+            +-----------+-----------+
               |                                    |
   +-----------v-----------+            +-----------v-----------+
   |   投递服务 (Push/WS)   |            |   搜索服务 (Elasticsearch) |
   +-----------------------+            +---------------------------+
```

### 2.1 各层职责说明

| 层级 | 主要职责 | 选型理由 |
|------|----------|----------|
| **客户端** | UI、加密、重连、离线缓存 | 采用 **WebSocket**+**HTTP** 双通道，满足实时+文件上传需求 |
| **API Gateway** | 统一入口、TLS、鉴权、限流、路由 | **Kong / Envoy**，支持插件化，易于灰度发布 |
| **Load Balancer** | 将请求分发到后端实例，提供跨 AZ 负载均衡 | **L7（如 Nginx、ALB）**，能根据路径/Header 路由到不同微服务 |
| **微服务层** | 业务解耦：**Message Service、File Service、Auth Service、Search Service、Push Service** | 微服务让团队可以独立扩容、部署；使用 **Spring Boot / Go / Node** 任意语言均可 |
| **公共组件** | **Redis**（热点缓存、用户在线状态）、**Kafka**（异步消息、投递）、**Prometheus + Grafana**（监控） | 经典组合，成熟、社区丰富 |
| **持久化层** | **MySQL**（事务、强一致）、**Cassandra / ScyllaDB**（大规模写入）、**对象存储**（文件） | 结合关系型与宽列存储，兼顾事务和海量数据 |
| **搜索层** | **Elasticsearch**（倒排索引、实时搜索） | 支持全文、聚合、近实时（秒级）索引 |
| **推送层** | **WebSocket Server**（在线实时） + **APNs/FCM**（移动端） + **邮件**（SMTP） | 多种渠道保证离线提醒 |

> **为什么不直接把所有业务放在一台服务器？**  
> - **扩展性**：10 k QPS 需要数十台机器；单点故障无法满足 99.9%。  
> - **职责单一**：搜索、文件、消息的负载模型不同，分离后可以使用最适合的技术栈。  
> - **团队协作**：不同业务线可以独立迭代，不会互相阻塞。

## ## 第三步：数据库设计

> **目标**：兼顾 **强一致性**（发送方必须立即可见） 与 **高写入吞吐**（每秒数千条消息），并且 **支持 90 天的检索**。

### 3.1 数据模型概览

| 实体 | 主键 | 重要字段 | 说明 |
|------|------|----------|------|
| **User** | `user_id (UUID)` | `email, name, avatar_url, status` | 基本信息，放在 **MySQL**（事务） |
| **Channel** | `channel_id (UUID)` | `name, type(public|private|dm), owner_id, created_at` | 频道元数据 |
| **ChannelMember** | `(channel_id, user_id)` 复合 PK | `role, joined_at, mute_until` | 多对多关联表 |
| **Message** | `message_id (UUID)` | `channel_id, sender_id, content, type(text|image|file|code), created_at, edit_flag` | 主表，写入 **MySQL**（分区） |
| **MessageAttachment** | `attachment_id (UUID)` | `message_id, file_id, preview_url` | 与文件关联 |
| **File** | `file_id (UUID)` | `owner_id, bucket_path, size, mime, sha256, uploaded_at` | 对象存储的元数据，放 **MySQL** 或 **Cassandra** |
| **SearchIndex** | - | `inverted_index` | 由 **Elasticsearch** 维护，来源于 Message + File name |

### 3.2 分区 & 归档策略

- **Message 表**：按 **`channel_id`** 哈希分区 + **时间窗口（每月）**。  
  - **为什么**：查询往往是“某个频道的最近 N 条消息”。哈希分区保证同一频道的数据在同一分区，时间窗口帮助 **90 天归档**（超过 90 天的数据可以迁移到冷存储或直接删除）。
- **ChannelMember**：因为成员数量相对固定，放在单独的 **MySQL** 实例即可。
- **File 表**：若文件数量极大（> 100 B），可以使用 **Cassandra** 按 `owner_id` 分区，读取文件元数据时只需要单行查询。

### 3.3 事务与一致性

- **发送消息流程**  
  1. **写入 MySQL（Message）** – 使用 **事务** 确保 `INSERT` 成功后返回 `message_id`。  
  2. **同步写入 Kafka**（生产者） – 事务性生产（Kafka Transactional Producer）确保 **消息写入+写入队列原子**。  
  3. **返回给客户端** – 此时发送方已完成强一致写入，收到 `message_id`，可以展示在 UI。  
- **读取消息**  
  - 读取 MySQL（或 **读写分离** 的从库）获取历史，热点消息可放 **Redis** 缓存（最近 100 条）加速。

> **不使用事务的后果**：若写库成功但未写入 Kafka，消息会“丢失”，接收方永远收不到；若只写入 Kafka 而 DB 失败，系统重启后消息会“漂移”。使用 **两段提交** 或 **Kafka事务** 可以避免。

### 3.4 索引设计

| 索引 | 字段 | 用途 |
|------|------|------|
| PK `message_id` | - | 唯一定位 |
| IDX `channel_id, created_at DESC` | `channel_id, created_at` | 拉取频道最新 N 条 |
| IDX `sender_id, created_at` | `sender_id, created_at` | 用户个人历史查询 |
| FULLTEXT (Elasticsearch) | `content, file_name` | 全文搜索 |
| PK `file_id` | - | 文件唯一定位 |
| IDX `owner_id, uploaded_at` | `owner_id, uploaded_at` | 用户文件列表 |

## ## 第四步：核心 API 设计

> **原则**：**RESTful** + **gRPC** 双协议（REST 用于浏览器/移动端的 CRUD，gRPC 用于内部高频调用）。所有请求必须带 **JWT**（或 OAuth2）进行鉴权，返回统一的 **ErrorCode**。

### 4.1 鉴权 & 授权

| 接口 | 鉴权方式 | 说明 |
|------|----------|------|
| `/api/v1/login` | 账号密码 → JWT | 登录返回短期 JWT + Refresh Token |
| `/api/v1/refresh` | Refresh Token | 续签 |
| 业务接口 | JWT (Bearer) | 在网关统一校验并注入 `user_id` |

### 4.2 常用 API 列表（仅示例）

| 方法 | 路径 | 描述 | 请求体示例 | 响应 |
|------|------|------|-----------|------|
| **POST** | `/api/v1/channels` | 创建频道（公开/私密） | `{ "name":"dev", "type":"private", "member_ids":[...] }` | `{ "channel_id":"..." }` |
| **GET** | `/api/v1/channels/{channel_id}/messages?limit=50&cursor=...` | 拉取历史（分页） | - | `{ "messages":[...], "next_cursor":"..." }` |
| **POST** | `/api/v1/channels/{channel_id}/messages` | 发送消息（文字） | `{ "content":"Hello", "type":"text" }` | `{ "message_id":"...", "created_at":... }` |
| **POST** | `/api/v1/files` (multipart) | 上传文件 | `file` + `metadata` | `{ "file_id":"...", "url":"..." }` |
| **GET** | `/api/v1/search?q=bug&channel_id=...` | 全文搜索 | - | `{ "hits":[...] }` |
| **POST** | `/api/v1/webhooks/github` | 接收 GitHub 事件 | GitHub payload | `202 Accepted` |
| **POST** | `/api/v1/push/register` | 注册设备 token（APNs/FCM） | `{ "device_token":"...", "platform":"ios" }` | `200 OK` |

### 4.3 幂等性设计

- **发送消息**：客户端在发起前生成 **client_message_id**（UUID），随请求一起发送。服务端在写库前先检查该 ID 是否已存在，若已存在返回已有的 `message_id`。这样即使网络重试，也只会产生 **一次** 消息。
- **文件上传**：使用 **分块上传 + MD5 校验**，每块都有唯一 `chunk_id`，服务端幂等保存。

### 4.4 错误码约定（示例）

| Code | 含义 |
|------|------|
| 0 | 成功 |
| 1001 | 参数校验错误 |
| 2001 | 未授权（JWT 失效） |
| 3001 | 权限不足（非频道成员） |
| 4001 | 消息写入冲突（client_message_id 重复） |
| 5001 | 内部服务错误（DB、Kafka） |

> **如果不统一错误码**：前端会陷入“到底是网络问题还是业务错误”的困惑，调试成本翻倍。

## ## 第五步：详细组件设计

下面按功能模块逐层展开实现细节、技术选型、关键流程以及常见的 **故障点** 与 **防御措施**。

### 5.1 消息写入与持久化

#### 5.1.1 流程图（文字版）

1. **客户端** → **API Gateway**（HTTPS） → **Message Service**（REST）  
2. **Message Service**  
   - 生成 `client_message_id`（若未提供）  
   - **事务**：`INSERT INTO message`（MySQL）  
   - **写入 Kafka**（事务 Producer）`topic = channel_{channel_id}`，键 = `channel_id`，值 = 消息体（JSON）  
   - **返回** `message_id` 给客户端  
3. **Kafka Consumer**（投递服务）  
   - 读取消息，写入 **Redis**（热点缓存）  
   - 调用 **Push Service**（WebSocket/FCM）进行实时投递  

#### 5.1.2 技术点解释

| 技术 | 作用 | 替代方案 | 关键配置 |
|------|------|----------|----------|
| **MySQL InnoDB** | 事务强一致，支持二级索引 | PostgreSQL | 主从复制（GTID），写库专用 |
| **Kafka Transactional Producer** | 写库 + 写队列原子，防止“写库成功/写队列失败” | RabbitMQ（事务不够） | `transaction.timeout.ms`、`enable.idempotence=true` |
| **Redis (Cluster)** | 最近 N 条消息的热点缓存，降低 DB 读压 | Memcached | LRU、TTL（5 min） |
| **WebSocket Server (Netty / Go)** | 实时推送，保持长连接 | SSE（不支持双向） | 心跳检测、连接数上限（10 k / 实例） |

#### 5.1.3 防止“消息丢失”或 “重复投递”

- **幂等消费**：消费者在写入 Redis 前先检查 `message_id` 是否已存在。若已存在直接 ACK。
- **回放机制**：若用户发现缺失消息，可通过 **offset 重置** 拉取 Kafka 中的历史（保留 24 h）。
- **监控**：监控 `kafka.lag`、`db.write_latency`、`ws.delivery_success_rate`。

### 5.2 实时投递（Push Service）

| 场景 | 目标 | 实现 |
|------|------|------|
| **在线用户** | 0–200 ms 内送达 | **WebSocket** 长连接 → 服务器直接 push |
| **离线移动端** | ≤ 5 s（APNs/FCM） | **Push Server** 读取未送达消息列表 → 调用 **APNs** / **FCM** API |
| **邮件提醒** | 关键事件（被 @） | **SMTP Worker**（异步） |

#### 关键实现细节

- **用户在线状态**：`SETEX online:{user_id} 300` 于 Redis，每次 WS 收到心跳刷新 TTL。离线检测通过键失效实现。
- **消息分发策略**：  
  - **单播**（私聊） → 直接推送对应 `user_id`。  
  - **多播**（频道） → 维护 **Channel → 在线成员列表**（Redis Set），遍历一次推送；若成员 > 10 k，采用 **批量推送**（分片）或 **Fanout Service**（Kafka 分区）。
- **推送可靠性**：WS ACK → 若 3 次未收到 ACK，转为 **APNs/FCM**（移动端）或 **邮件**。

### 5.3 文件上传、存储与预览

#### 5.3.1 流程

1. 客户端 **分块上传**（默认 5 MB/块），每块带 **MD5**。  
2. **File Service** 接收块，写入 **对象存储**（如 **Amazon S3**、**Aliyun OSS**）的临时目录。  
3. 所有块上传完成后，调用 **完成合并 API**：服务端校验整体 **SHA-256**，生成正式对象路径 `bucket/{file_id}`。  
4. **生成预览**：  
   - 对图片/PDF 使用 **ImageMagick** / **PDFium** 生成缩略图（存入对象存储）。  
   - 对 Office 文档使用 **LibreOffice headless** 转 PDF → 生成预览。  
5. **写入 File 元数据** 到 MySQL（或 Cassandra），返回 `file_id`、`download_url`、`preview_url` 给客户端。  
6. **消息服务** 再把 `file_id` 关联到消息（MessageAttachment），并推送。

#### 5.3.2 选型解释

| 选项 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **对象存储（S3/OSS）** | 高可用、自动分片、CDN 集成、成本低 | 读/写延迟 ~ 100 ms（可接受） | 大文件（≤1 GB） |
| **自建分布式文件系统（Ceph）** | 可私有化、低延迟 | 运营成本高、需要维护 | 对合规要求极高的企业 |
| **分块上传** | 断点续传、网络波动容忍 | 实现稍复杂 | 1 GB 大文件必备 |
| **预览服务（微服务）** | 统一转码、可水平扩容 | 需要额外算力 | 常用文档、图片 |

#### 5.3.3 防止“文件丢失”或 “恶意上传”

- **MD5/SHA 校验**：每块、整体校验确保完整性。  
- **上传限流**：对单用户并发块数、总流量做限制，防止 DoS。  
- **病毒扫描**：使用 **ClamAV** 或云厂商提供的 **Content Moderation**，在文件入库前扫描。

### 5.4 全文搜索（Search Service）

#### 5.4.1 需求回顾

- **搜索范围**：文字内容、文件名、链接（URL）  
- **时效性**：实时或近实时（秒级）  
- **保留期限**：90 天  

#### 5.4.2 技术选型

| 方案 | 说明 | 适配度 |
|------|------|--------|
| **Elasticsearch** | 倒排索引、分片、复制、聚合、近实时 | ✅ 完全匹配需求 |
| **OpenSearch** | AWS 兼容版，功能相同 | ✅ 替代方案 |
| **Solr** | 老牌搜索，部署复杂 | ❌ 不推荐 |
| **自建倒排 + KV** | 高度定制、开发成本大 | ❌ 资源有限时不选 |

#### 5.4.3 索引结构

```json
{
  "mappings": {
    "properties": {
      "message_id":   { "type": "keyword" },
      "channel_id":   { "type": "keyword" },
      "sender_id":    { "type": "keyword" },
      "content":      { "type": "text", "analyzer": "standard" },
      "file_name":    { "type": "text", "analyzer": "standard" },
      "created_at":   { "type": "date" }
    }
  }
}
```

- **分片数**：根据 **QPS** 与 **存储量** 估算。假设每月 30 TB 索引，单节点可处理 2 TB，采用 **30** 主分片 + **1** 副本。
- **刷新间隔**：`refresh_interval: 1s`（默认 1s）确保秒级可搜索。

#### 5.4.4 写入流程

1. **Message Service** 插入 DB → **Kafka** 同步 `message_created` 事件。  
2. **Search Consumer**（独立微服务）消费该事件，调用 **Elasticsearch Bulk API** 批量写入（每 5 s 或 5 k 条）。  
3. **删除/过期**：每天跑 **删除任务**（基于 `created_at`），对超过 90 天的文档执行 `DELETE BY QUERY`。

#### 5.4.5 查询流程

- 客户端 → **API Gateway** → **Search Service**（REST） → **Elasticsearch** → 返回 `message_id` 列表 → 前端再调用 **Message Service** 批量获取详情（或直接返回部分字段）。  

#### 5.4.6 防止搜索延迟超标

- **热点缓存**：热门关键词的搜索结果放 **Redis**（TTL 30 s）。  
- **并发控制**：对同一用户的搜索请求做 **Rate Limiter**（如 5 QPS），防止突发查询冲垮 ES。  
- **监控指标**：`search_latency_p95`, `es_node_cpu`, `heap_usage`，出现异常即时扩容。

---

### 5.5 Bot / Webhook 集成

| 步骤 | 说明 |
|------|------|
| **Webhook 接收** | **API Gateway** 通过 **HMAC** 验签（GitHub `X-Hub-Signature`）确保来源可信。 |
| **事件转化** | **Webhook Service** 将外部 JSON 转为内部 `bot_message` 结构（包括 `channel_id`, `text`, `attachments`）。 |
| **投递** | 与普通用户消息相同，写入 DB、Kafka，交由 **Push Service** 投递。 |
| **权限** | Bot 账户拥有 **Bot** 角色，只能在特定频道发消息，防止滥用。 |

> **如果直接把外部 webhook 内容推送到客户端**，会面临安全风险（XSS、钓鱼）以及 **格式不统一** 的问题。统一走内部消息路径可以复用所有可靠性机制。

### 5.6 监控、日志、运维

| 维度 | 关键指标 | 采集方式 |
|------|----------|----------|
| **可用性** | `api_success_rate`, `ws_connected_clients` | Prometheus + Alertmanager |
| **性能** | `api_latency_p95`, `db_write_latency`, `kafka_lag` | Jaeger（分布式追踪） |
| **容量** | `db_storage_used`, `es_disk_usage`, `object_storage_bytes` | Grafana + CloudWatch |
| **错误** | `http_5xx_rate`, `consumer_error_rate` | ELK（日志聚合） |
| **安全** | `failed_auth`, `webhook_signature_fail` | SIEM |

- **告警策略**：  
  - `api_latency_p95 > 300ms` → 触发 **自动扩容**（水平扩容 API 实例）。  
  - `kafka_lag > 5min` → **通知运维** 并 **自动增加 consumer 并发**。

## ## 第六步：扩展性与高可用设计

### 6.1 分区（Sharding）策略

| 维度 | 分区键 | 目的 | 备注 |
|------|--------|------|------|
| **Message** | `channel_id`（Hash） + `month`（Time） | 同频道数据局部化，便于查询；时间分区帮助 90 天归档 | 采用 **MySQL 分区表** + **Kafka 分区**（`channel_id`） |
| **User** | `user_id`（UUID） | 负载均衡，避免热点 | 只读（Profile） |
| **File** | `owner_id`（Hash） | 同用户文件聚集，便于列表查询 | 可用 **Cassandra** |
| **Search** | `channel_id`（Hash） | 保证同一频道的倒排索引在同一 ES 节点 | 通过 **routing** 参数实现 |

> **如果不做分区**：单表会快速膨胀到 TB 级别，查询慢、备份恢复成本高；写入热点导致 **锁竞争**，影响 QPS。

### 6.2 CAP 与一致性模型

| 场景 | 需要的属性 | 取舍 |
|------|------------|------|
| **发送消息** | **强一致**（发送方必须看到） + **可用** | 采用 **单主 MySQL** + **同步写入 Kafka**（事务），在网络异常时返回错误，防止“已发送未投递”。 |
| **消息投递** | **可用 + 最终一致**（接收方可接受几秒延迟） | 投递通过 **异步队列**，消费者宕机时消息保存在 Kafka，重启后继续投递。 |
| **搜索** | **最终一致**（秒级延迟可接受） | ES 采用 **副本**，写入延迟 1s，查询直接读副本。 |
| **文件存储** | **强一致**（上传成功后立即可下载） | 对象存储本身提供 **强一致**（S3 2020+），或使用 **多AZ副本**。 |

### 6.3 容错与灾备

| 组件 | 冗余方式 | 故障切换 |
|------|----------|----------|
| **API Gateway** | 多实例 + **L7** 负载均衡（跨 AZ） | 健康检查自动剔除 |
| **MySQL** | 主‑从 + **semi‑sync**（确保写成功后返回） + **跨地域复制** | 自动故障转移（MHA） |
| **Kafka** | 多副本（replication factor=3） | ISR（In‑Sync Replicas）保证不丢失 |
| **Redis** | 主从 + **Cluster**（分片） + **哨兵**监控 | 故障转移 30 s 内完成 |
| **Elasticsearch** | 主副本（replica=1） + **跨 AZ shard allocation** | 节点故障自动重新分配 |
| **对象存储** | 多 AZ 多副本（S3 标准） | 自动恢复 |
| **WebSocket** | 多实例 + **Sticky Session**（基于用户 ID） | 若实例挂掉，客户端重新握手连接到其他实例 |

### 6.4 水平扩容路径

| 维度 | 触发阈值 | 扩容方式 |
|------|----------|----------|
| **API** | `cpu > 70%`、`latency > 300ms` | **Auto Scaling Group**（按实例数） |
| **Message DB** | `row count > 500M per shard`、`write latency > 20ms` | **增加分区**（新月份自动创建），或 **分库**（按 `channel_id` 前缀拆分） |
| **Kafka** | `broker disk > 70%`、`lag > 1M` | **增加 broker**，重新平衡分区 |
| **Redis** | `key count > 100M`、`eviction_rate > 5%` | **增加节点**（Cluster 自动 re-shard） |
| **Elasticsearch** | `heap > 75%`、`search latency > 200ms` | **增加节点**，重新分配分片 |
| **File Service** | `upload QPS > 2k` | **增加上传网关**（Nginx + S3 presigned URL） |

> **为什么要用自动化扩容**：手动扩容耗时，无法应对突发流量（比如公司发布大事件导致聊天激增）。自动化配合 **预测模型**（基于历史 DAU）能提前预留容量。

### 6.5 迁移与灰度发布

- **蓝绿部署**：新版本部署到独立环境，切流量前做 **健康检查**（API latency、错误率）。  
- **金丝雀发布**：对 1% 用户使用新服务，监控异常后逐步提升比例。  
- **数据迁移**：采用 **双写**（写入旧库 + 新库）+ **后台同步**，确保业务不中断。

## ## 第七步：常见面试追问与回答

下面列出面试官常问的 8 类问题，每类给出**思路、关键点、示例答案**，帮助你在现场快速组织语言。

| 编号 | 追问 | 思路要点 | 示例回答 |
|------|------|----------|----------|
| 1 | **消息投递可靠性**：网络抖动时如何确保“只投递一次”？ | 幂等 `client_message_id`、Kafka 事务、消费端幂等检查、重试机制 | “客户端先生成唯一的 `client_message_id`，服务端在写 DB 前检查是否已存在；写库成功后使用 Kafka Transactional Producer 把同一条消息写入 Kafka，保证写库+写队列原子。消费者在投递前再次检查 `message_id` 是否已投递，若已投递直接 ACK。这样即使网络重试，也只会出现一次消息”。 |
| 2 | **分区策略**：10 k QPS 时怎么划分数据？ | 按 `channel_id` 哈希 + 时间窗口，兼顾热点与归档；Kafka 与 MySQL 分区保持一致 | “我们把每条消息的 `channel_id` 做哈希分区，保证同一频道的所有消息落在同一分区，查询时不跨分区；再按月份做时间分区，方便 90 天归档。Kafka 主题也使用同样的分区键，这样消费端可以顺序读取”。 |
| 3 | **全文搜索实现**：如何保证搜索延迟 ≤200 ms？ | 使用 Elasticsearch，近实时索引（1s 刷新），热点缓存，分片合理，查询只返回 ID 再补充详情 | “消息写入后会产生 `message_created` 事件推送到 Kafka，Search Service 每秒批量写入 ES。ES 刷新间隔设为 1 s，几乎实时可搜索。搜索时先在 ES 获取 `message_id` 列表，返回给前端，前端再通过 Message Service 批量查询详情，热点关键词的结果再缓存到 Redis，整体延迟在 100‑150 ms”。 |
| 4 | **文件上传大文件**：1 GB 文件怎么保证不丢失、支持断点续传？ | 分块上传、MD5 校验、对象存储的分段合并、幂等记录、后台病毒扫描 | “客户端把文件切成 5 MB 块，每块带 MD5，服务端在对象存储的临时目录写入块并记录块号。若网络中断，客户端只需要重新上传缺失块。全部块上传完成后，File Service 发起合并请求，校验整体 SHA‑256，成功后移动到正式路径并写入 DB”。 |
| 5 | **推送系统设计**：离线用户如何保证收到消息？ | 在线状态保存在 Redis，离线时写入未送达表或 Kafka，定时任务推送 APNs/FCM，邮件备用 | “用户在线时 WS 直接推送；离线时我们在 Redis 中记录 `offline:{user_id}`，消息写入后会写入 `undelivered` 表（或专门的 Kafka topic），推送服务定时扫描该表，使用 APNs/FCM 推送到移动端。若仍未收到，则发送邮件”。 |
| 6 | **高可用**：单点故障怎么办？ | 多 AZ 部署、主从复制、健康检查、自动故障转移、数据多副本 | “所有关键服务（API、DB、Kafka、Redis、ES）都部署在至少 3 个 AZ。API 通过 L7 负载均衡做健康检查，实例挂掉自动剔除。MySQL 使用主‑从 + semi‑sync，故障时 MHA 自动切换。Kafka 每个分区 3 副本，至少保证 ISR 为 2，确保即使掉机也不丢数据”。 |
| 7 | **扩容**：QPS 突增到 100 k，怎么快速扩容？ | 自动扩容组、水平拆分、热点缓存、预热实例、限流保护 | “我们在云上使用 Auto Scaling Group，监控 CPU、延迟、请求数，一旦超过阈值自动增加 API 实例。消息写入通过分库分表，新增分片后只需要重新分配 Kafka 分区即可。对热点频道使用 Redis 缓存最近 200 条消息，减轻 DB 压力”。 |
| 8 | **安全**：如何防止恶意 Bot 滥发消息？ | Bot 账户角色、配额、速率限制、审计日志、Webhook 验签 | “Bot 只能使用专门的 Bot Token，系统为每个 Bot 设置每日发送上限（比如 10 k 条）。对 Bot 调用的 API 加入 IP 白名单和 HMAC 验签，所有 Bot 行为写入审计日志，异常行为触发告警”。 |

> **技巧**：回答时先 **复述需求**，再 **逐层解释**（可靠性 → 幂等 → 重试），最后给出 **具体实现**（Kafka 事务、Redis TTL），让面试官看到你思路清晰、细节完整。

## ## 心得与反思

### 1️⃣ 本题最难的设计决策

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **消息的强一致 + 实时投递** | 必须让发送方立刻看到消息（强一致），但投递本身是异步的，容易出现“写成功、投递失败”的情况。 | - 先列出所有可能的失败点（DB、Kafka、网络）<br>- 评估每种方案的 CAP 权衡<br>- 选定 **Kafka Transactional Producer** + **client_message_id 幂等**，实现“一次写入，两阶段提交”。 |
| **全文搜索的实时性 vs 成本** | 需要秒级搜索，使用 ES 可满足功能，但索引成本与集群规模会随消息量急速增长。 | - 计算每日新增消息量（≈ 5 TB）→ 估算 ES 索引大小（≈ 1.5×原始）<br>- 通过 **分片 + 只保留 90 天**、**冷热分层**（最近 30 天主分片 5 副本，30‑90 天降副本）控制成本<br>- 再加 **Redis 缓存热点查询**，降低查询压力。 |

### 2️⃣ 新手最容易犯的错误（至少 2 条）

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有业务放在单体数据库** | 业务增长后单库写入成为瓶颈，备份恢复时间线性增长。 | 采用 **分区/分库**（按 channel、时间），并使用 **Kafka** 进行异步写入与解耦。 |
| **忽视幂等性** | 网络抖动导致客户端多次重试，产生重复消息，用户体验极差。 | 为每条消息生成 **client_message_id**，服务端在写库前检查唯一性；Kafka 使用 **事务**，消费者做幂等消费。 |
| **只考虑读写性能，忽略离线/推送** | 实时聊天成功，但离线用户收不到消息，系统不完整。 | 设计 **在线状态缓存（Redis）**、**未投达队列**、**APNs/FCM** 推送链路。 |
| **搜索直接查询 DB** | 随着数据量暴涨，全文检索查询慢、CPU 高。 | 引入 **Elasticsearch**，采用 **倒排索引** 与 **近实时刷新**。 |

### 3️⃣ 学习建议 & 可延伸方向

1. **系统设计基础**  
   - 阅读《Designing Data‑Intensive Applications》（Martin Kleppmann），理解 CAP、事务、日志等概念。  
   - 学会画 **C4 模型** 与 **时序图**，帮助在面试时快速表达架构。

2. **核心技术实战**  
   - **Kafka**：搭建本地集群，练习事务生产者、Exactly‑once 消费。  
   - **Elasticsearch**：学习索引映射、分片、滚动查询，体会搜索延迟与资源的关系。

   - **多租户、权限细粒度**：实现组织结构树、ABAC。
