# 第 26 天：设计 Google Drive

> 生成日期：2026-04-30

---

## 题目背景  
Google Drive 是一款云存储与协作平台，用户可以在其中上传、下载、同步文件，并支持多人实时编辑、分享和版本管理。面试中要求你设计一个类似的分布式云文件存储系统。

---

## 面试场景设定  
> **面试官**：*“我们现在要设计一个可以服务全球数十亿用户的云文件存储系统（类似 Google Drive），请先从整体架构角度说明你会如何满足用户的上传、下载、同步和共享需求。接下来我们会逐步深入细节。”*  

---

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **文件上传/下载**：支持任意大小（从几 KB 到数 GB）的文件上传、分块上传、断点续传以及高速下载。 |
| 2 | **文件同步**：跨多端（PC、移动端、Web）实时同步文件状态，保证同一账号下的文件在不同设备上保持一致。 |
| 3 | **共享与权限控制**：用户可以生成分享链接或邀请他人协作，支持只读、可编辑、可评论等细粒度权限。 |
| 4 | **版本管理**：对每一次修改生成版本快照，用户能够查看、回滚或比较历史版本。 |
| 5 | **垃圾回收/回收站**：删除文件后进入回收站，保留一定时间后彻底清除，并提供批量恢复功能。 |
| 6 | **全文搜索**：基于文件名、元数据以及文档内容提供快速搜索能力。 |

---

## 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 2 × 10⁸ | 假设覆盖全球主流市场的活跃用户规模。 |
| **QPS（上传/下载请求峰值）** | 150 k QPS | 峰值时段（如工作日 19:00‑21:00）内的并发请求数。 |
| **平均响应延迟** | ≤ 200 ms（读取） / ≤ 500 ms（写入） | 包括网络往返、鉴权、元数据查询等。 |
| **可用性** | 99.99% 年可用率 | 每年累计不可用时间 ≤ 52.56 分钟。 |
| **存储容量** | 5 EB（Exabyte） | 预计 5 EB 原始文件数据，压缩/去重后实际使用约 3 EB。 |
| **数据一致性** | 最终一致性 + 强一致性（对元数据） | 文件元数据（目录、权限）要求强一致性，文件内容可接受最终一致性。 |

## 系统边界  

**本题范围（需要设计）**  
- 文件的分块存储、上传/下载流控、断点续传。  
- 元数据服务（目录树、权限、版本信息）。  
- 同步协议（跨设备的变更推送与冲突解决）。  
- 共享链接生成、权限校验、访问审计。  
- 基础的监控、限流与容错机制（如副本、故障转移）。  

**不在本题范围（可不做详细设计）**  
- 客户端 UI/交互细节（仅关注后端 API）。  
- 细粒度的反病毒、内容审查（可视为外挂服务）。  
- 计费、企业租户隔离、合规性审计（如 GDPR）等业务层面的实现。  
- 第三方编辑器（如 Google Docs）实时协作的复杂 OT/CRDT 细节。

## 提示与追问  

1. **冲突解决策略**：当两台设备同时对同一文件进行修改时，你会采用什么冲突检测与合并方案？  
2. **数据去重与压缩**：在海量用户上传相似文件（如系统镜像、常用文档）时，你会如何设计全局去重和增量压缩？  
3. **跨地域复制与一致性**：为了满足低延迟访问，你会如何在多个数据中心之间复制文件？在保证强一致性的前提下，如何控制复制延迟？

## 第一步：理解需求与规模估算

### 1. 功能性需求拆解

| 编号 | 功能 | 关键技术点 |
|------|------|------------|
| 1 | 文件上传/下载（分块、断点续传） | **分块存储、分片上传、MD5/CRC 校验、流控、CDN** |
| 2 | 文件同步（多端） | **事件总线、增量变更、冲突检测、推送/拉取** |
| 3 | 共享与权限 | **ACL、签名链接、审计日志** |
| 4 | 版本管理 | **快照、增量存储、元数据变更日志** |
| 5 | 回收站 | **软删、TTL、批量恢复** |
| 6 | 全文搜索 | **倒排索引、文档解析、分布式搜索集群** |

> **注意**：功能之间有层级依赖——上传是底层，搜索依赖元数据、版本等。

### 2. 非功能性需求拆解

| 指标 | 目标 | 计算方式 / 备注 |
|------|------|-----------------|
| DAU | 2×10⁸ | 估算日活用户数 |
| QPS（峰值） | 150 k | 采用 3×峰值比例（1 min 内峰值） |
| 响应延迟 | ≤200 ms 读 / ≤500 ms 写 | 包含网络 RTT、鉴权、元数据查询 |
| 可用性 | 99.99% | 年不可用 ≤ 52.6 min |
| 存储容量 | 5 EB 原始 | 假设平均文件 5 MB → 约 1 B 文件数 1 trillion |
| 一致性 | 强一致元数据、最终一致文件块 | 业务容忍度划分 |

### 3. 流量、存储、并发的粗略估算

| 项目 | 计算公式 | 结果 |
|------|----------|------|
| **上传 QPS** | 150k × 60%（上传占比） ≈ 90k QPS | 约 90 000 次/秒的上传请求 |
| **平均上传大小** | 5 MB（保守） |  |
| **上传带宽需求** | 90k × 5 MB ≈ 450 GB/s ≈ 3.6 Tbps | 需要 **多区域 CDN + 分布式写入** |
| **存储增长速率** | 90k × 5 MB × 86400 s ≈ 38 PB/天 | 5 EB 目标在 3–4 年内可达 |
| **元数据写入 QPS** | 90k（每块上传一次元数据） | 需要 **强一致 KV**（如 TiKV、CockroachDB） |
| **搜索请求 QPS** | 150k × 20% ≈ 30k QPS | 需要 **分布式搜索集群**（Elasticsearch/Opensearch） |

> **如果不做这些估算**，设计时很容易低估网络、存储、CPU 的需求，导致后期频繁“宕机、卡顿”。

## 第二步：高层架构设计

> 先画 **最小可用系统（MVP）**，再逐步加 **高可用、跨地域** 组件。下面的 ASCII 图帮助你快速抓住整体结构。

### 2.1 MVP（单机/单AZ）架构

```
+-------------------+       +-------------------+
|   客户端 (Web/APP) | <---> |   API Gateway    |
+-------------------+       +-------------------+
                                 |
                                 v
                        +-------------------+
                        |  Auth Service     |
                        +-------------------+
                                 |
                                 v
+-------------------+   +-------------------+   +-------------------+
|  Upload Service   |---|  Metadata Service |---|  Download Service |
+-------------------+   +-------------------+   +-------------------+
        |                         |                     |
        v                         v                     v
+-------------------+   +-------------------+   +-------------------+
|  Object Store (FS)|   |  Metadata DB (RDB)|   |  Object Store (FS)|
+-------------------+   +-------------------+   +-------------------+
```

**说明**  

| 组件 | 负责什么 | 为什么先这么设计 |
|------|----------|-------------------|
| API Gateway | 统一入口、流控、限速 | 防止客户端直接访问内部服务，便于后期灰度发布 |
| Auth Service | OAuth2 / JWT 鉴权 | 分离安全，支持多租户、单点登录 |
| Upload / Download Service | 分块上传/下载、分片校验 | 业务逻辑清晰，后期可以水平扩展 |
| Metadata Service | 目录树、权限、版本信息（强一致） | 需要事务、唯一约束，使用关系型 DB |
| Object Store | 实际文件块（对象） | 大文件使用 **分布式对象存储**（如 Ceph、MinIO） |
| Metadata DB | 元数据持久化 | 读写频繁，需要 **强一致**、二级索引 |

> **不这样做的风险**：把上传、下载、元数据混在一起会导致 **单点瓶颈**、**事务冲突**，后期难以水平扩容。

### 2.2 进阶：全局分布式高可用架构

```
                        +-------------------+
                        |   CDN (边缘节点)  |
                        +-------------------+
                                 |
                                 v
+-------------------+   +-------------------+   +-------------------+
|  Global API GW   |---|   Global Auth     |---|   Rate Limiter    |
+-------------------+   +-------------------+   +-------------------+
        |                         |                     |
        v                         v                     v
+-------------------+   +-------------------+   +-------------------+
|   Upload Service |---|   Metadata Service|---|   Download Service|
+-------------------+   +-------------------+   +-------------------+
        |                         |                     |
        v                         v                     v
+-------------------+   +-------------------+   +-------------------+
|  Object Store (R) |   |  Metadata DB (R)  |   |  Object Store (R) |
|  (Multi‑AZ)       |   |  (Strong Cons.)  |   |  (Multi‑AZ)       |
+-------------------+   +-------------------+   +-------------------+
        ^                         ^                     ^
        |                         |                     |
        |                         |                     |
        |   +---------------------+---------------------+
        |   |   Replication / Sync Service (Async)   |
        |   +----------------------------------------+
        |                     |
        v                     v
+-------------------+   +-------------------+
|  Search Service   |   |  Sync Service     |
|  (Elasticsearch) |   |  (Kafka + Workers)|
+-------------------+   +-------------------+
```

**关键点解释**  

| 组件 | 作用 | 为什么要加 |
|------|------|------------|
| **CDN** | 静态文件（已上传的块）就近缓存，降低下载 RTT | 满足 **≤200 ms** 读取延迟 |
| **Global API GW** | 跨区域统一入口，做 **路由**（就近或最近） | 避免用户请求跨洲漂移 |
| **Rate Limiter** | 防止恶意刷流、突发流量 | 保证 **可用性**、**成本** 可控 |
| **Replication / Sync Service** | 异步跨 AZ/Region 复制对象块 | **最终一致**、**灾备** |
| **Search Service** | 构建倒排索引，支持全文搜索 | 满足搜索 QPS |
| **Sync Service** | 基于 **Kafka** 的变更日志，推送文件变更到各端 | 实现多端实时同步 |
| **Metadata DB (强一致)** | 使用 **分布式事务**（两阶段提交）或 **Raft** 实现强一致 | 目录、权限必须强一致，否则会出现 “文件在 A 端看得到、在 B 端看不到” 的混乱 |

> **不做跨地域复制**：单点灾难会导致整个地区不可用，违背 **99.99%** SLA。

## 第三步：数据库设计

> **核心原则**：**元数据强一致** → 关系型/强一致 KV；**文件块大容量** → 对象存储 + 异步复制。

### 3.1 元数据模型（强一致）

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| `users` | `user_id` (UUID) | email, password_hash, quota, status | 用户信息 |
| `folders` | `folder_id` (UUID) | `owner_id`, `parent_id`, `name`, `created_at`, `updated_at` | 目录树，`parent_id` 为自引用 |
| `files` | `file_id` (UUID) | `owner_id`, `folder_id`, `file_name`, `size`, `content_hash`, `current_version_id`, `deleted_at` (NULL=未删) | 文件基本信息 |
| `file_versions` | `version_id` (UUID) | `file_id`, `version_number`, `size`, `content_hash`, `created_at`, `metadata` (JSON) | 版本快照 |
| `file_chunks` | `chunk_id` (UUID) | `file_id`, `version_id`, `chunk_index`, `chunk_hash`, `storage_key` | **每块对应对象存储的键** |
| `acl_entries` | `acl_id` (UUID) | `resource_type`(file/folder), `resource_id`, `grantee_id`, `permission`(READ/WRITE/COMMENT), `expiry` | 权限控制表 |
| `share_links` | `link_id` (UUID) | `resource_type`, `resource_id`, `token`, `permission`, `creator_id`, `expire_at`, `max_uses` | 分享链接 |
| `audit_logs` | `log_id` (UUID) | `user_id`, `action`, `resource_type`, `resource_id`, `timestamp`, `detail` | 访问审计（可归档） |

**索引建议**  

- `files.owner_id + folder_id` → 列出文件列表  
- `file_versions.file_id + version_number` → 快速定位历史版本  
- `acl_entries.resource_type + resource_id` → 权限检查（常用）  
- `share_links.token` → 通过短链快速定位资源  

**事务需求**  

- 创建文件 → **插入 `files`、`file_versions`、`file_chunks`、`acl_entries`** 必须在同一事务内完成。  
- 更新 ACL → 只修改 `acl_entries`，需要 **行级锁** 防止并发冲突。  

> **若使用 NoSQL（如 DynamoDB）**：可以把 `files`、`file_versions` 合并为单表（单个 Partition Key = `file_id`），但要自行实现事务或使用 DynamoDB Transactions。对新手而言，**关系型 + 分布式 KV** 更易解释。

### 3.2 文件块存储（对象存储）

| 属性 | 说明 |
|------|------|
| **存储介质** | 分布式对象存储（Ceph、MinIO、Amazon S3） |
| **块大小** | 4 MiB（可调） → 兼顾大文件分片数和上传并发 |
| **去重策略** | **内容哈希（SHA‑256）** 作为 `storage_key`，相同块只存一次 |
| **副本数** | 3 副本（跨机房） → 失效自动恢复 |
| **写入路径** | **写时复制（Copy‑on‑Write）**：上传新块不覆盖旧块，保持历史版本的完整性 |
| **元数据** | `chunk_hash`、`size`、`storage_key` 存在 `file_chunks` 表，**不在对象存储里**（避免全局遍历） |

> **不做块去重**：会导致 **存储浪费 30%+**，尤其是系统镜像、常用文档。  

### 3.3 搜索索引（倒排索引）

| 方案 | 关键点 |
|------|--------|
| **Elasticsearch / OpenSearch** | 每次文件元数据或内容变更，异步写入 ES；使用 **分片 + 副本** 保证查询延迟 < 100 ms |
| **文本提取** | 对 Office、PDF、图片（OCR）等使用 **Tika**、**Vision API** 预处理后写入索引 |
| **权限过滤** | 在查询时加入 `owner_id` 或 `shared_with` 过滤，或在 ES 中使用 **Document-level security** |

> **不做搜索**：用户只能按文件名/路径浏览，体验大幅下降，尤其在 **亿级文件** 环境下。

## 第四步：核心 API 设计

> 下面列出 **RESTful + JSON** 风格的主要接口，包含 **请求/响应示例**、**幂等性**、**鉴权** 说明。实际实现可基于 **gRPC**、**GraphQL**，但面试时先写 REST 更直观。

| 方法 | 路径 | 功能 | 关键参数 | 幂等性 | 鉴权 |
|------|------|------|----------|--------|------|
| `POST /v1/files/upload/init` | 初始化分块上传 | 返回 `upload_id`, 推荐块大小, 已存在块列表（秒传） | `file_name`, `size`, `mime_type`, `parent_folder_id` | ✅（创建） | JWT |
| `PUT /v1/files/upload/{upload_id}/chunk/{index}` | 上传单块 | **流式** `PUT`，附带 `Content-MD5` | `upload_id`, `index`, `chunk_data` | ✅（同 `index` 幂等） | JWT |
| `POST /v1/files/upload/{upload_id}/complete` | 完成上传 | 服务器校验所有块完整性，写入元数据、生成版本 | `upload_id` | ✅（完成一次） | JWT |
| `GET /v1/files/{file_id}/download` | 下载文件（全文件） | 支持 **Range** 头部实现断点续传 | `file_id` | ✅（只读） | JWT / ACL |
| `GET /v1/files/{file_id}/chunks/{index}` | 下载单块（内部使用） | 返回块内容 | `file_id`, `index` | ✅ | 内部鉴权 |
| `GET /v1/folders/{folder_id}/list` | 列出目录 | 支持分页、排序 | `folder_id`, `page_token`, `page_size` | ✅ | JWT |
| `POST /v1/files/{file_id}/share` | 创建分享链接 | 返回短链 `token` | `file_id`, `permission`, `expire_at`, `max_uses` | ✅ | JWT |
| `GET /share/{token}` | 通过短链访问 | 解析 token → 检查权限 → 重定向到下载 URL | `token` | ✅ | 无（Token 本身携带权限） |
| `POST /v1/files/{file_id}/version/{version_id}/restore` | 回滚到历史版本 | 创建新版本，指向旧块 | `file_id`, `version_id` | ✅ | JWT / ACL |
| `GET /v1/search` | 全文搜索 | 支持 `q`, `owner_id`, `file_type`, `date_range` | `q`, `filters`, `page_token` | ✅ | JWT |
| `POST /v1/sync/poll` | 客户端轮询变更 | 返回自 `last_seq` 起的变更列表（JSON） | `last_seq` | ✅ | JWT |

### 幂等性说明

- **上传/下载**：使用 **upload_id + chunk_index** 组合键保证 **幂等**（相同块可以重复 `PUT`，服务器只保留一次）。
- **创建资源**：返回唯一 ID，**POST** 本身不幂等，但客户端可以先 **GET** 检查是否已存在（如秒传）。
- **删除/恢复**：采用 **软删**（`deleted_at`），`DELETE` 只标记，真正物理删除走 **后台 GC**。

### 鉴权与授权流程

1. 客户端携带 **JWT**（在登录后由 Auth Service 颁发）  
2. API GW 解析 JWT → 获取 `user_id` → **速率限制**（每用户/全局）  
3. 业务服务（Upload/Download）再次调用 **ACL Service**：  
   - 输入：`user_id`, `resource_type`, `resource_id`  
   - 输出：`ALLOW/DENY` + 权限级别  
4. 对于 **share token**，直接在 **Share Service** 验证 token 与对应权限，无需额外登录。

> **不做统一鉴权**：每个微服务自行校验会导致 **安全漏洞**、**实现重复**，且难以统一 **限流**。

## 第五步：详细组件设计

### 5.1 分块上传 & 断点续传

**流程图（文字版）**

1. **客户端** → `POST /upload/init` → 返回 `upload_id`、`chunk_size`、已存在块列表（通过文件 `content_hash` 检查秒传）  
2. 客户端分块读取文件，每块 **PUT** 到 `PUT /upload/{upload_id}/chunk/{index}`，附带 **MD5** 用于完整性校验。  
3. 服务器把块 **写入对象存储**，块的 `storage_key = SHA256(chunk_data)`。若该 `storage_key` 已存在（全局去重），直接返回成功，不再写入对象。  
4. **客户端** 完成所有块后 → `POST /upload/{upload_id}/complete`  
   - 服务器校验 **所有块的 MD5 列表** 与客户端提供的整体 `file_hash` 是否匹配。  
   - 在 **事务** 中创建 `files`、`file_versions`、`file_chunks` 三张表记录。  
   - 返回 **文件 ID**、**当前版本号**。  

**关键实现细节**

| 关键点 | 实现方式 | 为什么要这样 |
|--------|----------|---------------|
| **块大小** | 4 MiB（可调） | 平衡 **并发上传** 与 **磁盘 I/O**，兼容移动网络。 |
| **MD5/CRC 校验** | 客户端计算并放在 `Content-MD5` 头；服务端再次校验 | 防止网络传输错误或恶意篡改。 |
| **幂等上传** | 使用 `upload_id + chunk_index` 作为唯一键，**PUT** 可重复 | 防止网络抖动导致的重复写入。 |
| **秒传（Instant Upload）** | 客户端先算全文件 `SHA‑256`，向服务端查询是否已存在相同 `content_hash`，若有直接返回已有 `file_id` | 大幅降低重复上传流量（常见于系统镜像、常用文档）。 |
| **写时复制** | 新块永远写入新 `storage_key`，不覆盖旧块 | 保证 **版本回滚** 与 **共享文件** 的数据不被破坏。 |

### 5.2 断点续传（下载）

- 使用 **HTTP Range** 头部：客户端请求 `GET /files/{id}`，携带 `Range: bytes=0-4194303`（第一个 4 MiB）。  
- **Download Service** 读取 `file_chunks` 表，定位对应块的 `storage_key`，从对象存储流式返回。  
- 支持 **多线程并行下载**（每个线程请求不同 Range），提升大文件下载速度。  

### 5.3 文件同步协议

**核心思路**：**事件流 + 客户端轮询/长轮询**  

1. **变更写入日志**：所有元数据写操作（上传完成、删除、移动、权限变更）在 **Metadata Service** 写入 **Kafka**（或 Pulsar）主题 `metadata-changelog`。  
2. **同步服务**（消费者）读取日志，按 **user_id** 分区，生成 **增量变更包**（如 `{"op":"UPDATE","file_id":"...","version":3}`）。  
3. **客户端** 定期 `POST /sync/poll`，携带上一次的 `last_seq`（Kafka offset）。服务返回自该 offset 起的所有变更。  
4. **冲突检测**：  
   - 每个文件有 **版本号（单调递增）**。客户端在提交修改时必须带上 `base_version`。  
   - 若服务器发现 `base_version` < `current_version`，说明已有冲突 → 返回 **409 Conflict**，并提供 **冲突文件的最新内容** 给客户端自行合并或提示用户。  

**冲突解决策略**（对应面试追问）  

| 场景 | 方案 | 实现要点 |
|------|------|----------|
| 同时编辑同一文件（不同块） | **乐观锁 + 自动合并**（块级） | 只要块索引不冲突，服务器接受并生成新版本；若块索引冲突（同一块被改），返回冲突。 |
| 同时编辑同一文件的 **元数据**（重命名、移动） | **全局版本号 + 最后写者赢**（LWW） | 元数据表使用 **row_version** 字段，写时比较并更新。 |
| 同时编辑 **同一块**（如协作编辑） | **返回冲突** → 客户端弹框让用户选择 **覆盖/保留/手动合并** | 因为实时协作文档不在本系统范围（可交给 Google Docs），这里采用保守策略。 |

### 5.4 共享链接与权限校验

1. **创建分享**：`POST /files/{id}/share` → 生成 **唯一 token**（例如 Base62 编码的 96 位随机数），存入 `share_links` 表。  
2. **访问时**：`GET /share/{token}` → 解析 token → 查 `share_links` → 校验 `expire_at`、`max_uses` → 生成 **一次性下载 URL**（签名的对象存储 URL，有效期 5 min），返回给前端。  
3. **权限细粒度**：`permission` 可是 `READ`、`WRITE`、`COMMENT`。**ACL Service** 在每次访问时检查 **ACL + Share** 两条记录的 **并集**。  

**安全注意**  

- **短链**：长度至少 10‑12 字符，防止暴力猜测。  
- **访问审计**：每次通过分享链接下载都写入 `audit_logs`（包括 IP、User-Agent），便于事后追踪。  

### 5.5 版本管理与回收站

- **每次写入**（上传完成、手动保存）都会创建 **新 `file_versions`**，并把旧块的 `storage_key` **保持不变**（实现写时复制）。  
- **回收站**：删除文件时，仅将 `files.deleted_at = now()`，并在 `file_versions` 中标记 `is_deleted = true`。后台 **GC**（每 24 h）根据 `deleted_at` 与 **保留天数**（30 天）真正删除块并回收存储。  
- **恢复**：`POST /files/{id}/restore` → 将 `deleted_at = NULL`，若需要恢复到某个历史版本，则创建新的 `file_versions` 指向旧块。  

### 5.6 监控、限流、容错

| 维度 | 监控指标 | 实现方式 |
|------|----------|----------|
| **流量** | QPS、带宽、错误率 | **Prometheus + Grafana**（服务端自曝 metrics） |
| **存储** | 总容量、块副本健康、对象存储 I/O | Ceph/Metrics, S3 CloudWatch |
| **业务** | 上传成功率、下载延迟、同步延迟、冲突次数 | 自定义仪表盘 |
| **限流** | 每用户 QPS、IP 黑名单 | **RateLimiter**（令牌桶）在 API GW 实现 |
| **熔断** | 对象存储不可用时自动降级为 **只读** | **Hystrix/Resilience4j** |
| **灾备** | 跨 AZ/Region 副本延迟、恢复时间（RTO） | **Chaos Monkey** 演练，验证自动故障转移 |

> **不做监控**：在真实生产环境中，**无法定位故障**，导致 SLA 失守。

## 第六步：扩展性与高可用设计

### 6.1 横向扩容（Scale‑out）

| 组件 | 扩容方式 | 关键技术 |
|------|----------|----------|
| API Gateway | **无状态**，水平增加实例 + **L4/L7** 负载均衡（Envoy、NGINX） | 自动伸缩（K8s HPA） |
| Auth Service | **读写分离**：读从缓存（Redis） → 写到 DB | JWT 无状态，可随意复制 |
| Upload/Download Service | **无状态** → 多实例 | 对象存储本身是分布式 |
| Metadata Service | **分库分表**（用户/目录分区）+ **Raft**（TiKV） | 读写分离、强一致 |
| Object Store | **增加 OSD**（Ceph）或 **S3 分区** | 自动平衡、再平衡 |
| Search Service | **水平分片** + **副本**（ES） | 通过 **滚动升级** 防止查询失效 |
| Sync Service | **Kafka 分区**（按 user_id）+ **消费者组** | 增加消费者实例提升并发 |

### 6.2 多活数据中心（跨 Region）

1. **用户就近路由**：DNS/Anycast + **Geo‑IP** 将请求路由到最近的 **Region**。  
2. **元数据强一致**：使用 **Google Spanner‑like** 两段提交或 **CockroachDB** 的 **跨 Region 分布式事务**。  
3. **对象存储复制**：  
   - **主动复制**：写入本地 Region 后，后台 **复制服务**（基于 **Rsync/CRDT**) 将块同步到其他 Region。  
   - **读写策略**：**写入本地** → **读** 先查询本地副本，若缺失再 **跨 Region 拉取**（异步）  
4. **冲突与一致性**：  
   - **元数据**：采用 **强一致**（2PC） → **写延迟** 约 150 ms（跨三中心）仍在写入 SLA（≤500 ms）范围内。  
   - **文件块**：最终一致 → 复制延迟 ≤ 5 min（可接受）  
5. **灾备切换**：  
   - **自动故障检测**（Health‑Check） → 当某 Region 失联，**DNS TTL** 失效后自动切到次近 Region。  
   - **数据恢复**：使用 **对象存储的跨 Region 副本**，元数据通过 **跨 Region 备份**（每天全量快照）恢复。  

### 6.3 容错与降级

| 场景 | 降级措施 |
|------|----------|
| 对象存储写入异常 | **返回 503**，客户端重试；同时 **开启只读模式**，不接受新上传 |
| Metadata DB 高延迟 | **读取走缓存**（Redis） → **写入缓冲队列**（Kafka）后异步落库 |
| CDN 故障 | **回源到对象存储**（直接拉取） |
| 搜索节点宕机 | **返回“仅返回文件名/路径”**，或 **使用本地缓存**（最近 1000 条搜索） |

## 第七步：常见面试追问与回答

### 1️⃣ 冲突解决策略

> **问题**：两台设备同时编辑同一个文件，会产生冲突，如何检测并合并？

**回答要点**  

1. **乐观锁 + 版本号**：每次修改必须带上 `base_version`。服务器比较 `base_version` 与当前 `current_version`。  
2. **冲突检测**：  
   - **块级冲突**：如果两次修改涉及 **不同块索引**，直接合并（写时复制），生成新版本。  
   - **同块冲突**：返回 **409 Conflict**，并把最新块内容返回给客户端，由用户决定覆盖或手动合并。  
3. **自动合并策略**（可选）：  
   - 对于 **文本文件**，可使用 **diff‑3** 合并算法。  
   - 对二进制文件直接冲突，让用户手动处理。  
4. **最终一致性**：冲突解决后，**同步服务**把新版本写入 `metadata-changelog`，所有客户端在下一次轮询时得到统一的最新状态。  

**为什么不直接覆盖**：覆盖会导致用户数据不可恢复，违反 **用户期望**。  

### 2️⃣ 数据去重与压缩

> **问题**：海量用户上传相似文件（系统镜像、常用文档），如何做全局去重和增量压缩？

**回答要点**  

1. **全局内容哈希**（SHA‑256）作为 **块的唯一标识**（`storage_key`）。  
2. **秒传**：客户端先计算整文件 `SHA‑256`，向服务端查询是否已存在相同 `content_hash`。若存在，直接返回已有 `file_id`，不产生任何块。  
3. **块级去重**：每块上传时服务端计算块哈希，若 `storage_key` 已存在，仅记录 `file_chunks` 中的引用，不写入对象存储。  
4. **增量压缩**：  
   - 对 **相似但不相同** 的大文件（如虚拟机镜像）使用 **增量差分算法**（rsync‑style）生成 **delta 块**，只存增量。  
   - 可在 **后台压缩服务**（基于 **Apache Parquet** 或 **ZSTD**）对 **小文件**（< 1 MiB）进行 **批量压缩**，提升存储利用率。  
5. **去重副作用**：**引用计数** 必须在对象存储层或元数据层维护，删除文件时只能在 **引用计数为 0** 时真正删除块。  

**为什么不直接存每个文件**：海量相同文件会导致 **存储成本** 暴涨，且 **网络带宽** 也会被浪费。  

### 3️⃣ 跨地域复制与一致性

> **问题**：为了降低访问延迟，需要在多个数据中心复制文件，如何保证强一致性（元数据）和复制延迟？

**回答要点**  

1. **元数据强一致**：采用 **分布式 NewSQL（CockroachDB / TiDB）**，内部使用 **Raft** 多副本共识协议。写入需要 **多数节点确认**，跨 Region 复制延迟约 100‑200 ms，仍在 **写入 SLA ≤500 ms** 范围。  
2. **文件块最终一致**：  
   - **本地写入** → **异步复制服务** 把块推送到其他 Region（使用 **Gossip**/**Rsync**）  
   - **复制延迟** 设定 **SLA ≤5 min**（对用户感知不大）  
   - 若复制期间出现故障，块仍可从 **最近的 Region** 读取，因对象存储本身提供 **多副本**。  
3. **复制策略**：  
   - **主动复制**（写完即复制）对热点文件使用 **双写**（本地+目标 Region）  
   - **被动复制**（后台）对冷文件采用 **定时任务**（如每 12 h）  
4. **冲突控制**：因为 **块是不可变**（写时复制），跨 Region 的写冲突只能发生在 **元数据**，而元数据使用 **强一致**，所以冲突被自然避免。  
5. **读路由**：客户端 DNS 解析到最近 Region；若本地不存在对应块，服务会 **透明回源**（跨 Region 拉取），并在本地缓存。  

**为什么不采用完全异步写**：若元数据也异步，会导致 **目录/权限不一致**，出现“我在 A 地看到文件，B 地却看不到”的情况，破坏用户体验。

## 心得与反思

### 本题最难的 1‑2 个设计决策及思考过程

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **元数据强一致 vs 文件内容最终一致** | 需要在同一系统中兼顾 **事务** 与 **高吞吐**。如果全部强一致，写入延迟会飙升；如果全部最终一致，目录/权限会出现冲突。 | 1️⃣ 明确业务需求：目录、权限必须强一致，文件内容可容忍短暂不一致。 <br>2️⃣ 选型：使用 **分布式 NewSQL**（Raft）存放元数据，**对象存储**存放块。 <br>3️⃣ 设计写时复制，保证块不可变，简化跨 Region 复制。 |
| **全局去重与引用计数** | 海量块的哈希冲突概率极低，但管理 **引用计数**（防止误删）非常复杂。错误的计数会导致 **数据丢失** 或 **存储泄漏**。 | 1️⃣ 采用 **content‑hash = storage_key** 直接映射到对象存储路径。 <br>2️⃣ 在 **元数据层**（`file_chunks`）维护 **外键**，删除文件时仅标记，后台 GC 根据 **无引用** 删除对象。 <br>3️⃣ 为防止并发删除/创建导致计数错误，引入 **乐观锁 + 幂等删除**（先写删除标记，再异步清理）。 |

### 新手最容易犯的错误（≥2 条）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把上传、下载、元数据放在同一个单体服务** | 造成 **单点瓶颈**，扩容困难；写入冲突高，响应延迟大。 | 按职责拆分：**Upload Service**、**Download Service**、**Metadata Service**，全部保持 **无状态**，通过统一的 **API Gateway** 进行路由。 |
| **忽视分块去重/秒传** | 大量重复文件导致 **存储成本** 爆炸、网络带宽浪费。 | 在 **上传初始化** 时进行 **全文件 hash** 检查；块级 **hash** 再次检查，使用 **写时复制**，仅在对象存储中存一次。 |
| **只用单机 MySQL 存元数据** | 当 QPS 达到 150 k 时会出现 **CPU/IO 饱和**，且单点故障违背 99.99% SLA。 | 采用 **水平分片 + 多副本** 的 NewSQL（TiDB、CockroachDB）或 **分布式 KV（TiKV）**，配合 **读写分离** 与 **缓存**。 |
| **把搜索功能直接放在关系库** | 文字搜索慢、无法支持全文检索，查询延迟 > 秒。 | 使用 **倒排索引**（Elasticsearch），异步写入，查询走专用搜索集群。 |

### 学习建议和可延伸的方向

1. **系统设计基础**：阅读《Designing Data‑Intensive Applications》《系统设计面试之道》；熟悉 **CAP、BASE、ACID** 概念。  
2. **分布式存储**：深入了解 **对象存储（Ceph、MinIO）**、**分布式文件系统（HDFS）**、**块级去重**（Content Defined Chunking）。  
3. **强一致数据库**：实验 **CockroachDB**、**TiDB**，理解 **Raft**、**Two‑Phase Commit** 的实现细节。  
4. **消息队列 & 事件溯源**：搭建 **Kafka**、**Pulsar**，练习 **Exactly‑Once** 消费和 **CDC**（Change Data Capture）。  
5. **同步协议**：实现一个简易的 **client‑side polling** 与 **WebSocket** 推送系统，体会 **冲突检测** 与 **版本控制**。  
6. **安全与审计**：了解 **OAuth2、JWT、签名 URL**、**ACL**、**审计日志** 的最佳实践。  
7. **演练面试**：把本案例的每个章节写成 5‑10 分钟的 PPT，练习 “先说概念、再说实现、最后说权衡”。  

> **核心思路**：先把 **业务需求 → 数据模型 → API → 关键组件** 按层次拆解，每一步都要解释 **为什么这么做、如果不这么做会怎样**，再在每层加入 **扩展/高可用** 的手段。这样即使面对面试官的追问，你也能从容应对。
