# 第 10 天：设计 Telegram

> 生成日期：2026-05-16

---

## 题目背景
Telegram 是一款基于云端的即时通讯应用，支持一对一聊天、群组、频道以及跨平台的文件传输。它以高并发、低延迟和强隐私保护（端到端加密）而闻名，全球用户数已超过 7 亿。

---

## 面试场景设定
> **面试官**：  
> “今天我们来一起设计一个类似 Telegram 的即时通讯系统。请先从整体架构出发，说明你会如何支撑 **每天 2 亿活跃用户** 的 **实时消息收发、群聊、媒体文件共享** 以及 **端到端加密** 等核心需求。我们先聚焦在消息的写入/读取路径和系统的可扩展性上，后面再讨论细节实现。”

---

## 功能性需求
1. **单聊**：用户之间可以发送文本、表情、图片、视频、音频等多媒体消息，支持已读回执。  
2. **群聊**：支持最多 200,000 人的超大群，提供消息广播、管理员权限、成员邀请/踢除等功能。  
3. **频道（Broadcast）**：单向推送，允许无限订阅者接收消息，适用于新闻、博客等场景。  
4. **端到端加密**：私聊（Secret Chat）采用 E2E 加密，确保只有通信双方可以解密内容。  
5. **离线消息存储**：用户离线时消息应在服务器持久化，登录后能够快速同步未读消息。  
6. **文件与媒体存储**：支持单文件最大 2 GB，提供 CDN 加速下载与分块上传/断点续传。

---

## 非功能性需求（带估算）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **日活跃用户 (DAU)** | 200 M | 全球范围的日活用户数 |
| **峰值 QPS（消息发送）** | 120 K msg/s | 假设峰值时 1% 用户同时发送 1 条消息 |
| **端到端加密消息延迟** | ≤ 150 ms | 从发送方客户端到接收方客户端的感知延迟（不含网络 RTT） |
| **系统可用性** | 99.99%（月度） | 计划停机时间 ≤ 4.38 小时/年 |
| **存储容量** | 150 PB（原始媒体+消息） | 按 30 天热存储 + 冷存归档，平均每条消息 1 KB，文件 2 GB，假设 2 PB/天 写入 |
| **吞吐带宽** | 30 Tbps（峰值） | 包括文本、图片、视频的上行/下行流量 |

# 题解

# Telegram‑类即时通讯系统设计全解答  

> **温馨提示**：下面的每一步都从“最小可运行系统”出发，逐层加入容错、扩展性和性能优化。即使你是第一次接触系统设计，也能跟着思路一步步搭建出完整的答案。  

---  

## ## 解题思路总览  

1. **先把需求拆成最小可用（MVP）**：只实现单聊、基本的写入/读取、登录鉴权。  
2. **在 MVP 基础上逐步加功能**：群聊广播、频道、媒体、E2E 加密、离线同步。  
3. **每加一个功能，问自己三件事**  
   - *为什么需要它？*（业务价值）  
   - *会产生哪些新瓶颈？*（性能、可靠性）  
   - *如何用现有或新建组件解决？*（技术选型）  
4. **从单机到分布式**：先把功能跑通，再把流量拆分到多机、跨地域。  
5. **把“横向扩展”和“高可用”作为后置目标**，在设计时预留分区、冗余、故障转移的接口。  

下面按照 **需求 → 架构 → 数据 → API → 细节组件 → 可扩展/HA → 面试追问** 的顺序展开。

## ## 第一步：理解需求与规模估算  

| 需求 | 关键点 | 业务影响 |
|------|--------|----------|
| **单聊** | 文本、表情、图片/视频（≤2 GB）<br>已读回执 | 需要实时写入、持久化、快速查询 |
| **群聊** | 超大群 200k 成员<br>广播、管理员权限 | 消息要在 **150 ms** 内送达 99% 成员 |
| **频道** | 单向广播、无限订阅 | 类似推送服务，流量更偏向下行 |
| **E2E 加密** | 私聊端到端，密钥只能在客户端持有 | 增加密钥协商、设备同步的复杂度 |
| **离线存储** | 离线用户登录后同步未读 | 必须持久化、支持增量拉取 |
| **媒体文件** | 2 GB 上限，分块上传/断点续传，CDN 加速 | 大流量、带宽压力、存储成本 |

### 规模估算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 题目给出 | 200 M |
| **峰值发送 QPS** | 1% 同时发 1 条 → 200 M × 0.01 = 2 M 条/秒 ≈ **120 K msg/s**（考虑并发系数） |
| **每日消息量** | 120 K msg/s × 3600 s × 12 h ≈ **5 B 条**（5 × 10⁹） |
| **每日媒体流量** | 假设 10% 消息带媒体，平均 500 KB → 5 B × 0.1 × 500 KB ≈ **250 TB** |
| **存储** | 30 天热存 150 PB（题目） + 冷归档 | 需要分层冷热存储 |
| **网络带宽** | 30 Tbps 峰值（题目） | 约 3.75 TB/s 进/出 |

> **结论**：系统必须 **水平拆分**（按用户/会话分区），使用 **无中心瓶颈** 的异步流转（消息队列、Push Service），并且 **跨地域** 部署 CDN 与存储。

## ## 第二步：高层架构设计  

### 1️⃣ MVP（单机版）结构  

```
[Client] <--HTTPS/GRPC--> [API Gateway] <---> [Auth Service]
                                          |
                                          v
                                    [Message Service]
                                          |
                                          v
                                   [Relational DB (MySQL)]
                                          |
                                          v
                                 [File Storage (FS/OSS) ]
```

- **API Gateway**：统一入口，做流量控制、鉴权、协议转换。  
- **Auth Service**：OAuth2 / JWT，生成短时 token。  
- **Message Service**：业务核心，处理「写入」与「读取」请求。  
- **MySQL**：存储消息元数据（user_id, chat_id, seq, status）。  
- **文件存储**：本地磁盘或对象存储（OSS）用于大文件。  

> **为什么先这么简单？**  
> - 易于实现、验证业务模型。  
> - 能快速定位瓶颈（CPU、IO、DB），为后续拆分提供依据。  

### 2️⃣ 扩展到生产级（分布式）结构  

```
                                   +---------------------+
                                   |   CDN (Edge Cache) |
                                   +----------+----------+
                                              |
                +-----------------------------+----------------------------+
                |                                                          |
      +---------v----------+                                     +-------v--------+
      |   Load Balancer    |                                     |   Global DNS   |
      +---------+----------+                                     +-------+--------+
                |                                                        |
    +-----------+-----------+                                +-----------+-----------+
    |   API Gateway (N)    |                                |   Push Service (N)   |
    +-----------+-----------+                                +-----------+-----------+
                |                                                        |
   +------------+------------+                         +----------------+----------------+
   |  Auth Service (Shard)   |                         |  Message Service (Shard)       |
   +------------+------------+                         +----------------+----------------+
                |                                                        |
   +------------+------------+                         +----------------+----------------+
   |  Chat Router (Consistent Hash) |                 |  Message Queue (Kafka / Pulsar) |
   +------------+------------+                         +----------------+----------------+
                |                                                        |
   +------------+------------+                         +----------------+----------------+
   |  User‑Shard DB (MySQL/PG) |                     |  Chat‑Shard DB (Cassandra/Scylla) |
   +------------+------------+                         +----------------+----------------+
                |                                                        |
   +------------+------------+                         +----------------+----------------+
   |  Media Service (Object Store) |                 |  Media CDN (Edge Nodes)          |
   +----------------------------+                 +-----------------------------------+
```

#### 关键组件说明  

| 组件 | 作用 | 关键技术 |
|------|------|----------|
| **Load Balancer / DNS** | 把流量分到最近的入口节点，提供 **跨地域** 访问 | Anycast DNS + L4 LB (Envoy, Nginx) |
| **API Gateway** | 统一入口，限流、鉴权、协议转化（HTTP/2、WebSocket） | Kong / APISIX |
| **Auth Service** | 登录、Token 发放、设备管理，使用 **JWT + Refresh Token** | Redis + MySQL |
| **Chat Router** | 根据 **chat_id** 进行 **一致性哈希**，把请求路由到对应的 Message Service 实例 | Consistent Hash Ring |
| **Message Service** | 负责写入消息、写入 MQ、写入 DB、返回 seq_id | Go/Java, Netty |
| **Message Queue** | 异步解耦，提供 **高吞吐** 的广播能力（Kafka / Pulsar） | 分区+副本 |
| **Chat‑Shard DB** | 存储 **大量** 消息（TB‑PB 级），选用 **宽列存储**（Cassandra、Scylla、DynamoDB） | 按 chat_id 分区 |
| **User‑Shard DB** | 存储用户信息、会话、已读状态（事务需求） | MySQL（主从） |
| **Push Service** | 长连（WebSocket、GRPC）或 **APNs/FCM** 推送，负责把 **消息 ID** 推给在线设备 | Redis Pub/Sub + NATS |
| **Media Service + CDN** | 分块上传、断点续传、对象存储、边缘缓存 | S3兼容存储 + CloudFront/Alibaba CDN |
| **E2E Key Service (client‑side)** | 只在客户端运行，负责 Diffie‑Hellman、Session Key、Key Sync | **Signal Protocol**（Double Ratchet） |

> **为什么要把写入和推送分离？**  
> - 写入是 **持久化**，需要强一致性，往往受 DB 写入延迟影响。  
> - 推送是 **实时**，对时延要求更高，使用 **异步队列** 能让两者独立扩容。

## ## 第三步：数据库设计  

### 1️⃣ 消息元数据模型（宽列存储）  

| 列族（Column Family） | 主键（PK） | 重要字段 | 说明 |
|------------------------|------------|----------|------|
| `messages` | `chat_id` (分区键) + `seq_id` (排序键) | `sender_id`、`msg_type`、`payload_hash`、`timestamp`、`status` | 每条消息的 **不可变** 信息，`seq_id` 为自增（全局单调递增） |
| `read_state` | `user_id` + `chat_id` | `last_read_seq`、`unread_count` | 读取进度，放在 **用户库**（MySQL）因为需要事务性更新 |
| `chat_info` | `chat_id` | `type`（single/group/channel）<br>`member_count`<br>`admin_list`<br>`created_at` | 群组属性，读多写少，放在 **关系库** 或 **缓存** |
| `attachment` | `msg_id` | `file_id`、`bucket`、`size`、`mime_type` | 大文件的指针，实际文件在对象存储 |

**分区策略**  

- **单聊**：`chat_id = hash(userA_id, userB_id)` → 保证两端消息在同一分区，降低跨分区读写。  
- **群聊**：`chat_id` 为群唯一标识，所有成员写入同一分区（水平扩容靠 **分区键** 的哈希均匀性）。  
- **频道**：同群聊，只是 **只写**（发布者）和 **只读**（订阅者）角色分离。  

> **为什么不直接用单一 MySQL？**  
> - 200 M DAU、120 K msg/s 的写入量会把单机或单主从的事务日志压垮。  
> - 宽列存储天然支持 **追加写**（Cassandra 的 LSM Tree），对**顺序写**和**大批量读取**（分页）友好。  

### 2️⃣ 用户与鉴权模型（关系库）  

| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| `users` | `user_id` | `phone_number`、`password_hash`、`created_at` | 基本用户信息 |
| `devices` | `(user_id, device_id)` | `public_key`、`last_seen`、`push_token` | 多设备登录、推送 token |
| `sessions` | `session_token` | `user_id`、`expires_at`、`refresh_token_hash` | JWT 对应的会话信息，存 Redis（TTL） |
| `chat_members` | `(chat_id, user_id)` | `role`、`joined_at`、`is_muted` | 群成员关系，查询成员列表时使用 |

> **为什么把 Session 放在 Redis？**  
> - 读取频率极高（每次 API 调用都要校验），而且 **TTL** 能自动失效，天然满足 **99.99%** 可用性。  

### 3️⃣ 媒体文件元数据（对象存储 + DB）  

- **对象存储**：S3 兼容 API，文件分块（Multipart）上传，返回 `file_id`（UUID）。  
- **元数据表 `files`**（MySQL）  
  - `file_id`（PK）  
  - `owner_id`  
  - `bucket`、`key`、`size`、`mime_type`、`hash`  
  - `created_at`、`expire_at`（可用于临时文件）  

> **冷热分层**：  
> - **热**（最近 30 天）放在 **SSD** 对象存储，**读写延迟 < 5 ms**。  
> - **冷**（归档）转到 **Glacier / Deep Archive**，成本低、读延迟可接受（数小时）。

## ## 第四步：核心 API 设计  

> 所有 API 均使用 **HTTPS + JSON**（或 **gRPC**），在实际实现中可通过 **API Gateway** 做统一路由、限流、监控。  

### 1️⃣ 鉴权类  

| 方法 | URL | 请求 | 响应 | 说明 |
|------|-----|------|------|------|
| `POST` | `/auth/login` | `{phone, password, device_id, push_token}` | `{access_token, refresh_token, expires_in}` | 登录并注册设备信息 |
| `POST` | `/auth/refresh` | `{refresh_token}` | `{access_token, expires_in}` | 刷新 JWT |
| `POST` | `/auth/logout` | `Authorization: Bearer <token>` | `200 OK` | 删除 Session（Redis） |

### 2️⃣ 单聊/群聊消息  

| 方法 | URL | 请求 | 响应 | 说明 |
|------|-----|------|------|------|
| `POST` | `/chats/{chat_id}/messages` | `Authorization` + `{msg_type, payload, attachment_id?}` | `{msg_id, seq_id, timestamp}` | **写入**消息，返回全局递增 `seq_id`（客户端用来做 UI 排序） |
| `GET` | `/chats/{chat_id}/messages?after=seq_id&limit=50` | `Authorization` | `{messages:[...]}` | **拉取**未读或分页消息 |
| `POST` | `/chats/{chat_id}/read` | `{last_read_seq}` | `200 OK` | 更新已读状态（写入 `read_state`） |
| `POST` | `/chats/{chat_id}/members` | `{user_id, role}` | `200 OK` | 添加/移除成员（仅群/频道） |

### 3️⃣ 媒体文件  

| 方法 | URL | 请求 | 响应 | 说明 |
|------|-----|------|------|------|
| `POST` | `/files/initiate` | `{file_name, size, mime_type}` | `{upload_id, part_size, urls[]}` | 返回 **分块上传 URL**（预签名） |
| `PUT` | `/files/{upload_id}/part/{part_number}` | **二进制块** | `200 OK` | 上传每个块 |
| `POST` | `/files/{upload_id}/complete` | `{parts:[{ETag, PartNumber}]}` | `{file_id, url}` | 完成后返回文件 ID 与 CDN URL |
| `GET` | `/files/{file_id}` | — | **302** → CDN URL | 下载（CDN 自动缓存） |

### 4️⃣ E2E 加密（仅客户端协议）  

| 方法 | URL | 请求 | 响应 | 说明 |
|------|-----|------|------|------|
| `POST` | `/e2e/key-exchange` | `{chat_id, client_pub_key, device_id}` | `{server_nonce, server_pub_key}` | **Diffie‑Hellman** 交换，服务器仅转发，不保存密钥 |
| `POST` | `/e2e/key-sync` | `{chat_id, encrypted_session_key, target_device_id}` | `200 OK` | 用 **端到端加密的**会话密钥同步多设备（加密后存 DB，仅服务器转发） |

> **注意**：真实的 E2E 加密实现往往在 **Signal Protocol**（Double Ratchet）之上，这里只列出最小交互。

## ## 第五步：详细组件设计  

### 1️⃣ 消息写入路径  

1. **客户端** → **API Gateway**（HTTPS）  
2. **Gateway** → **Auth Service** 校验 JWT（Redis）  
3. **Gateway** 将请求转发至 **Chat Router**（根据 `chat_id` 计算哈希）  
4. **Chat Router** 把请求路由到对应的 **Message Service 实例**（水平扩容）  
5. **Message Service**  
   - 生成 **全局唯一 `seq_id`**（使用 **Snowflake** 或 **Redis INCR**）  
   - 将 **消息体**（如果是文本/emoji）直接写入 **Chat‑Shard DB**（Cassandra）  
   - 若有 **附件**，先返回 `attachment_id`（已上传至对象存储），再写入 `payload_hash`  
   - 写入成功后 **生产**一条 **Kafka 事件**：`{chat_id, msg_id, seq_id, type}`  
   - 立即返回 `msg_id, seq_id` 给客户端（保证 **≤150 ms** 的感知延迟）  

6. **Kafka 消费组**（Push Service）  
   - 按 **chat_id 分区** 消费，确保同一聊天的消息顺序不乱  
   - 读取消息后 **查询在线设备**（Redis `online_user:{user_id}`）  
   - 对每个在线 **WebSocket / GRPC** 连接推送 **msg_id + seq_id**（轻量）  

> **为什么采用两段提交？**  
> - **写库** → **推送** 分离，写库故障不影响实时推送，推送延迟只受网络和 MQ 影响，易于 **水平扩容**。  

### 2️⃣ 消息读取（离线同步）  

1. 客户端登录后，**Push Service** 把 **未送达的 msg_id 列表**（或最近 `seq_id`）推送给客户端。  
2. 客户端调用 **GET /chats/{chat_id}/messages?after=last_seq** 拉取缺失消息。  
3. **Message Service** 在 **Chat‑Shard DB** 按 `chat_id`+`seq_id` 范围查询，返回 **分页**（默认 50 条）。  
4. 若消息含 **attachment_id**，返回 CDN URL（带签名，有限期）供客户端直接下载。  

> **为什么不把完整消息放进推送？**  
> - 消息体（尤其是大文件）会导致 **推送服务** 带宽飙升，且不易做 **重试**。  
> - 把 **元信息**（msg_id、seq_id）推送，真正内容走 **数据库 + CDN**，更符合 **CDN** 的缓存优势。  

### 3️⃣ 群聊/频道的广播模型  

#### 3.1 传统 **全网广播**（不可行）  

- 直接把每条消息发送给 **200k** 成员 → **200k** * **120 K QPS** = **24 B** 推送，远超带宽。  

#### 3.2 **分层（Hierarchical）推送**  

1. **Chat Service** 把消息写入 **Kafka**（分区 = `chat_id % N`）。  
2. **Push Service** 维护 **“分区消费者组”**：每个消费者负责 **若干子群**（例如 1k 成员）。  
3. **子群** 再 **分配** 给 **Edge Push Nodes**（部署在不同机房/CDN 边缘），每个节点只需要 **推送本地在线成员**。  
4. **在线状态** 通过 **Redis** 的 **Bitmap** 或 **Set** 存储，**边缘节点** 只查询本地域的在线用户。  

> **优势**  
> - **带宽** 按 **地域** 分散，单机压力 ≤ **10 K QPS**。  
> - **延迟** 只受 **单跳** 网络影响，典型 **50‑100 ms**。  
> - **故障隔离**：某节点宕机，只影响该地域的推送，其他节点不受影响。  

#### 3.3 频道（单向）  

- 频道的 **发布者**（如官方账号）写入 **单独的 Kafka Topic**（只读），**所有订阅者** 通过 **Push Service** 的 **广播订阅** 接收。  
- 因为 **订阅者数量可能是几千万**，采用 **分布式 Pub/Sub**（如 **Google Cloud Pub/Sub**、**Kafka MirrorMaker**）将 **消息复制** 到各区域的 **Edge Push Nodes**，再按本地在线用户推送。  

### 4️⃣ E2E 加密关键流程  

#### 4.1 单次聊天的密钥协商（基于 Signal 双向 Ratchet）  

1. **双方各生成一次性 DH 公钥** `X_A`, `X_B`（Curve25519）。  
2. **客户端 A** 将 `X_A` 发送到服务器的 **/e2e/key-exchange** 接口，服务器仅把它转发给 **B**（不保存）。  
3. **B** 同样发送 `X_B`。  
4. 两端使用 **X25519** 计算共享密钥 `K = DH(X_A, X_B)`。  
5. **K** 再通过 **HKDF** 派生 **Root Key**、**Chain Key**、**Message Keys**（Double Ratchet）。  

> **服务器角色**：只做 **转发**，不持久化密钥，满足 **端到端** 的安全要求。  

#### 4.2 多设备同步  

- **每个设备** 都有自己的 **长期身份密钥**（Curve25519）与 **一次性预钥**（Pre‑Key）。  
- 当 **新设备** 登录时：  
  1. 客户端 **A** 向服务器请求 **对方 B** 的 **Pre‑Key Bundle**（包含 B 的身份公钥、一次性预钥、签名）。  
  2. A 使用 **X3DH**（Triple Diffie‑Hellman）生成 **共享会话密钥**。  
  3. A 把 **会话密钥（已加密）** 通过 **/e2e/key-sync** 发给 **B**，服务器只做 **转发**。  
  4. **B** 解密后把同样的会话密钥加密给 **A 的其他设备**（遍历设备列表），实现 **多端同步**。  

> **安全要点**：  
> - 会话密钥永远不落地服务器，只在 **加密的消息体** 中出现。  
> - 若设备丢失，用户可以在 **Web/APP** 中撤销该设备的 **Pre‑Key**，防止后续窃取。  

### 5️⃣ 媒体文件的分块上传 & CDN 加速  

1. **客户端** 调用 **/files/initiate**，服务返回 **upload_id**、**分块大小**（如 8 MB）和 **预签名 URL**（S3 `PUT`）。  
2. 客户端并行上传 **每个块**（最多 10 并发），支持 **断点续传**（只重传失败块）。  
3. 所有块成功后，调用 **/files/complete**，服务器校验 **ETag**（MD5）并在对象存储生成 **File ID**。  
4. **Message Service** 在发送消息时，只携带 **File ID**。  
5. **CDN**（如 CloudFront）自动拉取对象存储内容，边缘节点缓存后提供 **低时延下载**。  

> **为什么不把文件直接放进消息库？**  
> - 大文件写入 **Cassandra** 会导致 **写放大**，并且查询时会把大量二进制数据拉回，影响 **读性能**。  
> - 使用 **对象存储 + CDN** 能做到 **弹性扩容**、**按需付费**，且 **边缘缓存** 能显著降低带宽成本。

## ## 第六步：扩展性与高可用设计  

### 1️⃣ 横向拆分（Sharding）  

| 维度 | 拆分方式 | 目的 |
|------|----------|------|
| **用户** | `user_id % N` → **User‑Shard DB**（MySQL） | 读写热点均衡 |
| **聊天** | `chat_id % M` → **Chat‑Shard DB**（Cassandra） | 消息写入均匀，防止热点 |
| **消息队列** | Kafka **Topic** 按 `chat_id` 分区 | 保证同一聊天顺序消费 |
| **Push Service** | 按 **地域** + **子群**（1k 成员） | 降低单节点带宽、提升局部容错 |

> **扩容**：当某分片负载升高，只需要 **增加对应分片的机器**，不影响其他分片。  

### 2️⃣ 多活跨地域部署  

- **核心数据**（用户、会话、聊天元数据）采用 **多活** + **强一致性**（如 **Google Spanner**、**CockroachDB**），保证 **登录/跨地域切换** 不丢失会话。  
- **媒体文件** 放在 **对象存储的多区域复制**（如 S3 Cross‑Region Replication），配合 **Anycast DNS** 把用户请求路由到最近的 **CDN Edge**。  

### 3️⃣ 高可用（HA）  

| 组件 | 失效场景 | 备援方案 |
|------|----------|----------|
| **API Gateway** | 单节点宕机 | 多实例 + **L4 Anycast** |
| **Auth Service** | Redis 故障 | 主从复制 + Sentinel 自动切换 |
| **Message Service** | 实例挂掉 | **无状态**，请求可自动路由到其他实例 |
| **Chat‑Shard DB** | 节点失联 | **复制因子 ≥ 3**（Cassandra），自动读写切换 |
| **Kafka** | Broker 故障 | **副本数 ≥ 3**，ISR（In‑Sync Replicas）保证不丢失 |
| **Push Service** | Edge 节点失效 | 同区域其他节点接管，客户端自动重新建立 WebSocket |
| **CDN** | 边缘节点失效 | 自动回源到对象存储，全球覆盖不受影响 |

> **SLA 目标**：  
> - **99.99%** 可用 → 每个子系统 **MTTR ≤ 5 min**，**MTBF ≥ 30 days**。  
> - **数据持久性**：Cassandra + Kafka 多副本 → **≥ 99.9999%**。  

### 4️⃣ 监控、告警与自动伸缩  

- **Metrics**（Prometheus）收集：QPS、RTT、错误率、CPU/内存、磁盘 I/O、网络流量。  
- **Alertmanager** 根据阈值（如 99th‑percentile 延迟 > 120 ms）触发告警。  
- **Kubernetes HPA** / **自研 Autoscaler** 根据 **CPU、QPS、消息队列积压** 动态扩容 **Message Service** 与 **Push Service**。  
- **Chaos Engineering**（如 Gremlin）定期演练节点失效，验证自动故障转移。

## ## 第七步：常见面试追问与回答  

### Q1️⃣ 数据分区与跨地域写入一致性如何保证？  

**回答思路**：  
1. **分区键**（`chat_id`）决定写入的 **Chat‑Shard**，同一个聊天始终落在同一分片，避免跨分片事务。  
2. 跨地域使用 **多活数据库**（如 **CockroachDB**）提供 **分布式事务**（两阶段提交）或 **强一致的读写**（Raft）保证 **全局唯一 seq_id**。  
3. 对于 **低延迟** 场景，**本地写入**后 **异步复制** 到其他区域（使用 **CDC + Kafka MirrorMaker**），**读写** 仍在本地域完成，**最终一致性** 只在 **离线同步** 场景体现。  

### Q2️⃣ 200k 人的超大群聊，怎么在 150 ms 内送达 99% 成员？  

**回答思路**：  
- **层级推送**：消息写入后进入 **Kafka**，每个 **子群**（~1k 成员）对应一个 **Push Worker**，在 **Edge** 节点直接向本地在线设备推送。  
- **在线状态** 用 **Redis Bitmap** 按地域分片，**单次查询 O(1)**。  
- **并行推送**：每个 Edge 节点使用 **异步 I/O（epoll）**，一次可以并发推送上千条 WebSocket 消息。  
- **统计**：假设每个 Edge 节点负责 10k 成员，带宽 10 Gbps → 10k × 100 byte（消息体） ≈ 1 Gbps，足以在 **≤100 ms** 完成。  
- **容错**：若某节点慢，其他节点仍可在 **150 ms** 前完成 99% 成员的推送。  

### Q3️⃣ 多设备下的端到端加密密钥如何安全同步？  

**回答思路**：  
1. **每台设备** 生成 **长期身份密钥**（Curve25519）+ **一次性预钥**。  
2. 当 **新设备** 加入时，**旧设备** 使用 **X3DH** 与对方的 **Pre‑Key Bundle** 计算 **共享会话密钥**。  
3. **会话密钥** 使用 **对方设备的公钥** 加密后，**通过服务器的 `/e2e/key-sync`** **转发**（服务器只能看到密文）。  
4. **旧设备** 再把相同密钥加密给 **其他已登录设备**（遍历设备列表），实现 **多端同步**。  
5. 若用户在 **Web** 端登录，**Web Crypto API** 负责密钥生成，**不泄露**给服务器。  

> **核心原则**：**服务器只转发**，**密钥永不落地**，满足 **端到端** 的安全模型。  

### Q4️⃣ 为什么不直接使用传统关系型数据库存储所有消息？  

**回答**：  
- **写入吞吐**：120 K msg/s → 每秒几万行插入，单机 MySQL 难以支撑，写放大导致 **磁盘 I/O** 瓶颈。  
- **扩展成本**：水平拆分（sharding）在 MySQL 上实现复杂，需要 **全局事务**、**二阶段提交**，延迟高。  
- **查询模式**：消息多为 **顺序读取**（按时间/seq），宽列存储（Cassandra）天然 **按分区键+聚簇键** 排序，支持 **高效分页**。  
- **冷热分层**：Cassandra 可直接在 **SSD** 上保留 **30 天热数据**，随后 **TTL** 自动迁移到 **更慢的存储**，而 MySQL TTL 需要手动清理。  

### Q5️⃣ 文件上传的分块大小怎么选？  

**回答**：  
- **网络特性**：大多数宽带在 **10‑30 Mbps** 以上，8 MB 分块在 **2‑5 秒** 内完成，兼顾 **错误恢复** 与 **并行度**。  
- **对象存储限制**：多数 S3 兼容服务支持 **最小 5 MB** 分块，**最大 5 GB**，8 MB 兼容所有平台。  
- **并发**：客户端可并行上传 **5‑10** 块，提升整体吞吐（单文件可达 **80‑100 MB/s**），满足 **2 GB** 大文件在 **30 秒** 内完成。

## ## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 思考过程 |
|------|----------|
| **群聊广播模型** | 初始想法是直接把消息推给所有在线用户，结果发现 **带宽** 与 **并发连接数** 会爆炸。通过 **分层推送 + 边缘节点** 的思路，把广播问题转化为 **局部推送**，大幅降低单点压力，同时满足 **150 ms** 延迟目标。 |
| **端到端加密多设备同步** | 端到端加密本身已经复杂，再加上多设备需要共享会话密钥。先把 **Signal/X3DH** 的概念抽象出来，明确“服务器只做转发”。随后设计 **key‑sync** 接口，确保 **密钥只在客户端加解密**，并用 **预签名** 防止中间人攻击。这个流程在面试中最容易卡点，需要把 **安全模型** 与 **实现细节** 同时阐述清楚。 |

### 2️⃣ 新手最容易犯的错误（至少两点）  

1. **把所有功能一次性全部写进最初的架构**  
   - 结果是图太复杂、难以解释，面试官会觉得你缺乏 **抽象分层** 能力。正确做法是先 **MVP**（单聊+持久化），再逐步引入群聊、文件、加密等模块。  

2. **忽视 **数据分区** 与 **路由** 的细节**  
   - 直接说 “把消息存到 MySQL” 会被追问 **写入吞吐** 与 **热点**。一定要说明 **分区键**、**一致性哈希**、**跨地域复制**，并解释如果不做会导致的 **单点瓶颈** 与 **数据不一致**。
