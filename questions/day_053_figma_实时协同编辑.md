# 题目：Figma 实时协同编辑系统设计

## 1. 题目背景
Figma 是一款基于浏览器的 UI/UX 设计工具，支持多人同时在同一个设计稿上进行编辑、评论和原型预览。面试任务是设计其核心的 **实时协同编辑** 能力，使得多个用户能够在毫秒级别看到彼此的修改，并保持文档的一致性和完整性。

## 2. 面试场景设定
> **面试官**：  
> “假设我们要为 Figma 构建一个能够支撑全球数百万活跃设计师实时协同编辑的后端系统。请从零开始，设计该系统的核心架构，重点说明如何实现低延迟的同步、冲突解决以及高可用的存储。我们可以先从功能需求和非功能需求开始讨论。”

## 3. 功能性需求
| 编号 | 功能描述 |
|------|----------|
| **F1** | **多人同步编辑**：多个用户可以在同一文件的同一画布或不同画布上同时编辑图形、文本、属性等，实时同步至所有在线用户。 |
| **F2** | **光标/选区可视化**：每个用户的光标、选中对象以及编辑状态（如正在拖拽、正在编辑文本）实时展示给其他人。 |
| **F3** | **操作历史与撤销/重做**：系统记录每一次操作，支持跨用户的撤销/重做，并能在需要时回滚到任意历史版本。 |
| **F4** | **冲突检测与自动合并**：对同一对象的并发修改进行冲突检测，采用 OT（Operational Transformation）或 CRDT 进行自动合并，保证最终状态一致。 |
| **F5** | **离线编辑与同步**：用户在网络中断后仍可继续编辑，恢复网络后系统自动将离线操作合并进实时协同流。 |
| **F6** | **权限控制**：基于文件的共享链接或团队权限，控制用户的 **只读 / 评论 / 编辑** 权限，并在同步时强制校验。 |

## 4. 非功能性需求
| 编号 | 指标 | 估算值 | 备注 |
|------|------|--------|------|
| **N1** | **日活跃用户（DAU）** | 2,000,000 | 估计全球设计师活跃用户数 |
| **N2** | **并发编辑会话数（Peak Concurrency）** | 150,000 | 高峰时刻约 7.5% 的 DAU 同时编辑 |
| **N3** | **每秒请求数（QPS）** | 120,000 | 包括光标移动、属性修改、心跳等 |
| **N4** | **端到端编辑延迟** | ≤ 150 ms（99% 请求） | 用户在 UI 中看到他人修改的时间 |
| **N5** | **系统可用性** | 99.99% 年可用性（≈ 52.6 min 年宕机） | 必须支持全球 24/7 服务 |
| **N6** | **存储规模** | 5 PB（年度累计设计稿及历史版本） | 采用对象存储 + 增量快照压缩 |

## 5. 系统边界
**本题需要考虑的范围**  
- 实时协同编辑的核心数据流（操作分发、冲突解决、持久化）。  
- 用户身份鉴权、文件权限校验。  
- 高可用的实时服务（多区部署、容错、负载均衡）。  
- 基础的离线缓存同步机制。  

**本题不需要实现的功能**  
- UI 前端实现（光标渲染、编辑工具栏等）。  
- 设计稿的渲染引擎、向量图形库。  
- 完整的文件分享、项目管理、计费系统。  
- 第三方插件生态、导入/导出功能。  
- 详细的监控告警系统（只需在答案中提及监控思路即可）。

## 6. 提示与追问
1. **冲突解决细节**  
   - “如果两位用户同时修改同一个矩形的宽度和颜色，系统会如何决定最终状态？”  
   - “请比较 OT 与 CRDT 在本场景下的优缺点，并说明你最终的选型理由。”

2. **扩展性与分区策略**

3. **离线同步与冲突**  
   - “离线期间用户可能产生大量操作，恢复网络后如何高效地将这些操作批量同步并合并？”  
   - “是否需要在客户端保留操作日志？如果是，日志的持久化和清理策略是什么？”

## 解题思路总览
> **核心目标**：让 **多个用户** 能在 **毫秒级** 看到彼此的编辑，并且 **数据最终一致**（即使出现离线、冲突、节点故障等极端情况），同时系统能够 **支撑全球数百万活跃用户**。

实现思路分为 **四层**：

| 层级 | 负责的核心职责 | 关键技术 |
|------|----------------|----------|
| **1. 接入层** | 鉴权、负载均衡、WebSocket/QUIC 长连 | CDN、API Gateway、TLS、WebSocket、QUIC |
| **2. 实时协同层** | 收集用户操作、冲突解决（OT/CRDT）、实时分发、光标同步 | **CRDT**（或 OT）+ 实时消息总线（Kafka/Redis‑Stream）+ 多副本 **协作服务器**（Collaborative Service） |
| **3. 持久化层** | 可靠保存文档状态、增量快照、历史版本 | 对象存储（S3/MinIO）+ 增量日志（Append‑only）+ 元数据库（Cassandra/Scylla） |
| **4. 辅助层** | 权限校验、离线缓存同步、监控告警、审计 | Auth Service、Permission Service、Sync Service、Prometheus+Grafana |

**从 MVP 到完整系统的演进路径**：

1. **MVP**：单机协作服务 + 内存操作日志 + 本地文件持久化（适用于 10‑100 并发会话）  
2. **水平扩展**：将协作服务拆成无状态的 **协作节点**，使用 **分片（sharding）** 按文件 ID 分配；引入 **消息队列** 做可靠的操作广播。  
3. **高可用**：多可用区（AZ）部署，每个文件拥有 **主‑备副本**；使用 **Raft/etcd** 选举 leader，保证写入顺序。  
4. **全局弹性**：使用 **全球负载均衡** + **边缘节点**（CDN + Cloudflare Workers）做 **光标/心跳** 的超低延迟转发；**离线同步** 采用 **客户端日志压缩 + 合并**。  

下面把每一步拆解得更细。

## 第一步：理解需求与规模估算  

### 1.1 功能需求拆解  

| 编号 | 关键点 | 对应系统子模块 |
|------|--------|----------------|
| **F1** | 多人同步编辑 | 实时协同层（CRDT/OT）+ 消息分发 |
| **F2** | 光标/选区可视化 | 轻量心跳/位置广播（实时层） |
| **F3** | 操作历史、撤销/重做 | 增量日志 + 版本管理服务 |
| **F4** | 冲突检测、自动合并 | 冲突解决算法（CRDT/OT） |
| **F5** | 离线编辑、同步 | 客户端操作日志 + Sync Service |
| **F6** | 权限控制 | Auth/Permission Service，写入前校验 |

> **注意**：F3 与 F4 强耦合——撤销/重做的实现必须基于同一套操作日志（OT/CRDT 的变换序列或状态向量）。

### 1.2 非功能需求拆解  

| 编号 | 关键指标 | 对系统设计的影响 |
|------|----------|-----------------|
| **N1**、**N2**、**N3** | 大规模并发（150k 会话、120k QPS） | 必须水平扩展、分片、无状态服务、异步消息 |
| **N4** | ≤150 ms 端到端延迟（99%） | 使用 **长连接（WebSocket/QUIC）**，在 Edge 节点做转发，避免跨地域往返 |
| **N5** | 99.99% 可用性 | 多 AZ、自动故障转移、读写分离、幂等写入 |
| **N6** | 5 PB 年存储 | **对象存储** + **增量压缩**，历史版本存放在低频存储，热点快照放在 SSD |

### 1.3 规模估算（粗略）  

| 维度 | 计算方式 | 结果 |
|------|----------|------|
| **峰值会话** | 2M DAU × 7.5% = **150k** 同时编辑会话 |
| **每会话 QPS** | 假设 5 条操作/秒（光标、属性、心跳） | 150k × 5 ≈ **750k** 事件/秒 |
| **消息体大小** | 平均 200 B（操作描述） | 750k × 200 B ≈ **150 MB/s** ≈ **1.2 Gbps** |
| **存储** | 每文件 10 MB，平均 20 版本/天 → 200 MB/天/文件 | 100k 活跃文件 → 20 TB/天 → 年约 7 PB（与 N6 接近） |

> 以上是上限估算，实际会有分布式热点（大型企业一次性 10k 同时编辑），需要 **分片** 与 **热点文件的专属节点**。

## 第二步：高层架构设计  

下面给出 **从零到全局可用** 的演进路线图，配合关键组件说明。

### 2.1 MVP（单机）架构  

```
+-------------------+       +-------------------+
|   前端浏览器 (Web) | <---> |  Collaborative   |
|   (WebSocket)     |       |  Service (单实例)|
+-------------------+       +-------------------+
          |                         |
          | (持久化)                | (本地文件系统)
          v                         v
    +------------+            +-----------+
    |   Log DB   |            |   FS/Blob |
    +------------+            +-----------+
```

- **优点**：概念清晰、实现快。  
- **缺点**：单点故障、无法支撑并发、没有分区、延迟受单机性能限制。  

> **面试技巧**：先从 MVP 说起，展示你能把系统拆解成最小可运行单元，随后再阐述如何逐步演进。  

### 2.2 关键技术选型  

| 需求 | 备选技术 | 选型理由 |
|------|----------|----------|
| 长连接、低延迟 | **WebSocket**、**QUIC (HTTP/3)** | 浏览器原生支持，QUIC 更好抗丢包、RTT 较低 |
| 实时消息广播 | **Kafka**、**Redis Streams**、**NATS** | Kafka 持久化、分区能力强；Redis Streams 超低延迟；可组合使用（Kafka 负责持久化，Redis 负责实时转发） |
| 冲突解决 | **CRDT**（如 Yjs、Automerge） vs **OT** | CRDT 天生无中心化、离线友好，适合跨地域；OT 更成熟于文字编辑，复杂度高。综合考虑 **选择 CRDT**（后文对比） |
| 持久化 | **对象存储 (S3/MinIO)** + **增量日志** | 对象存储成本低、弹性好；增量日志用于恢复、回滚 |
| 元数据 & 分片信息 | **Cassandra / ScyllaDB** | 高写入吞吐、自动分区、无单点 |
| 权限校验 | **OAuth2 + JWT** + **RBAC** | 标准、可跨服务传递 |
| 监控 | **Prometheus + Grafana** | 开源、易集成 |  

### 2.3 扩展到分布式的 **核心架构图**  

```
                     ┌───────────────────────────────────────┐
                     │               CDN / Edge               │
                     │   (TLS termination, WebSocket proxy)   │
                     └───────┬───────────────────────┬───────┘
                             │                       │
               ┌─────────────▼─────────────┐ ┌───────▼───────┐
               │   API Gateway / LB (global)│ │   Auth Service│
               └─────────────┬─────────────┘ └───────┬───────┘
                             │                       │
                ┌────────────▼─────────────┐ ┌───────▼───────┐
                │   Collaborative Service  │ │ Permission Srv│
                │  (stateless, per‑shard) │ │ (RBAC, ACL)  │
                └───────┬───────┬───────┬───┘ └───────┬───────┘
                        │       │       │             │
   ┌────────────────────▼─────┐ ┌─▼───────▼─────┐ ┌─────▼─────┐
   │   Message Bus (Kafka)    │ │   Redis Stream│ │   Raft/etcd│
   │   (persisted ops)        │ │ (real‑time)   │ │ (leader‑election)│
   └───────┬──────────────────┘ └───────┬───────┘ └─────┬─────┘
           │                            │               │
   ┌───────▼───────┐            ┌───────▼───────┐ ┌─────▼─────┐
   │  Shard #1    │            │  Shard #2    │ │   …       │
   │ (Collab Node)│            │ (Collab Node)│ │           │
   └─────┬───────┘            └─────┬───────┘ └─────┬─────┘
         │                          │               │
   ┌─────▼─────┐              ┌─────▼─────┐   ┌─────▼─────┐
   │   Log DB │              │   Log DB │   │   Log DB │
   │ (Cassandra│              │ (Cassandra│   │ (Cassandra│
   │  /Scylla)│              │  /Scylla)│   │  /Scylla)│
   └─────┬─────┘              └─────┬─────┘   └─────┬─────┘
         │                          │               │
   ┌─────▼─────┐              ┌─────▼─────┐   ┌─────▼─────┐
   │   Object  │              │   Object  │   │   Object  │
   │   Store   │              │   Store   │   │   Store   │
   │ (S3/MinIO)│              │ (S3/MinIO)│   │ (S3/MinIO)│
   └───────────┘              └───────────┘   └───────────┘
```

**关键点说明**：

1. **CDN/Edge**：把 WebSocket 握手、心跳、光标广播放在离用户最近的节点，降低 RTT。  
2. **API Gateway + LB**：全局负载均衡，把请求路由到最近的 **Collaborative Service** 实例。  
3. **Collaborative Service**：**无状态**，只负责业务逻辑（CRDT 合并、权限校验），所有持久化交给后端。  
4. **Message Bus**：  
   - **Kafka** 持久化所有操作日志（为回放、审计、快照提供依据）。  
   - **Redis Streams** 为实时分发提供低延迟通道（消费者是同一 shard 的协作节点）。  
5. **Shard**：每个文档（文件）通过 **hash(fileId) % N** 分配到具体的 **Collab Node**。同一 shard 内的所有操作通过同一个 **Redis Stream** 顺序消费，保证 **CRDT** 的 **因果一致**。  
6. **Log DB (Cassandra/Scylla)**：保存 **增量操作日志**（每条操作的元信息），支持 **快速回放** 与 **历史版本查询**。  
7. **Object Store**：存放 **完整快照**（每 N 次操作或每 X 分钟生成一次），用于 **快速恢复** 与 **历史回滚**。

## 第三步：数据库设计  

### 3.1 元数据模型（文件、用户、权限）  

| 表/集合 | 主键 | 主要字段 | 说明 |
|--------|------|----------|------|
| **users** | `user_id` (UUID) | `email`, `name`, `hashed_pwd`, `status` | 基础用户信息 |
| **teams** | `team_id` | `name`, `owner_id` | 团队组织 |
| **team_members** | `team_id + user_id` | `role` (owner/admin/member) | RBAC 关系 |
| **files** | `file_id` (UUID) | `owner_id`, `team_id`, `created_at`, `updated_at`, `size`, `current_version` | 设计稿元信息 |
| **file_permissions** | `file_id + user_id` | `role` (read/comment/edit), `expires_at` | 细粒度共享链接权限 |

> **实现**：使用 **关系型 DB**（PostgreSQL）或 **文档库**（MongoDB）均可，读写不频繁，事务需求较强。  

### 3.2 操作日志（增量）  

**结构（Cassandra 表）**  

| 主键 | `file_id` (partition) + `seq_num` (clustering) |
|------|------------------------------------------------|
| `op_id` | UUID |
| `user_id` | UUID |
| `timestamp` | bigint (epoch ms) |
| `op_type` | enum (INSERT, UPDATE, DELETE, CURSOR_MOVE, ... ) |
| `payload` | blob (CRDT op JSON) |
| `client_version` | bigint (客户端本地版本) |
| `server_version` | bigint (写入后全局版本) |

- **分区键**：`file_id`，保证同一文件的日志都落在同一节点，读写局部化。  
- **聚簇键**：`seq_num` 单调递增，提供 **顺序读取**（回放）和 **范围查询**（历史版本）。  

**写入流程**：

1. 客户端发送操作 → **Collaborative Service** → **CRDT** 合并 → 生成 **全局序号**（从 Raft/etcd 获取） → 写入 **Kafka**（持久化） + **Redis Stream**（实时） → 同步写入 **Cassandra**（log DB)。  

### 3.3 快照（完整文档状态）  

- **存储位置**：对象存储（S3）  
- **文件命名**：`snapshots/{file_id}/{version}.json.gz`  
- **内容**：完整的 **CRDT 状态**（如 Yjs 的 `Y.Doc` JSON）压缩后存放。  
- **生成策略**：  
  - **时间驱动**：每 5 min 生成一次（防止日志无限增长）。  
  - **操作数驱动**：每 10,000 条操作生成一次。  

**元数据表**（Cassandra）  

| 主键 | `file_id + version` |
|------|----------------------|
| `snapshot_path` | string |
| `created_at` | timestamp |
| `size_bytes` | bigint |
| `is_full` | boolean (true) |

### 3.4 读写路径概览  

| 场景 | 读取路径 | 写入路径 |
|------|----------|----------|
| **实时编辑** | 客户端 → WebSocket → **Collaborative Service** → **CRDT** 本地合并 → **Redis Stream** → 同步到其他客户端 | 同上 + **Kafka** 持久化 + **Cassandra** 增量日志 |
| **离线恢复** | 客户端本地日志 → 发送至 **Sync Service** → **CRDT** 合并 → **Kafka** + **Cassandra** | 同实时 |
| **历史版本查看** | 读取 **snapshot** + **增量日志**（从目标版本到最新） → 客户端回放 | N/A |
| **撤销/重做** | 读取对应 **seq_num** 操作 → 逆向 CRDT 变换（CRDT 天生支持） | 写入撤销操作（新操作） |

## 第四步：核心 API 设计  

> **原则**：**幂等**、**安全**（鉴权），**最小化网络往返**（一次请求尽量完成一次业务），**统一错误码**。

### 4.1 鉴权与会话  

| 方法 | 路径 | 描述 | 请求头 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/auth/login` | 使用邮箱/密码登录，返回 JWT | `Content-Type: application/json` | `{access_token, refresh_token, expires_in}` |
| `POST` | `/api/v1/auth/refresh` | 刷新 token | `Authorization: Bearer <refresh>` | 同上 |
| `GET` | `/api/v1/auth/me` | 获取当前用户信息 | `Authorization: Bearer <access>` | `{user_id, email, name}` |

### 4.2 文件元数据  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/files` | 创建新文件（返回 file_id） | `{name, team_id?, init_content?}` | `{file_id, created_at}` |
| `GET` | `/api/v1/files/{file_id}` | 查询文件元信息 + 权限 | – | `{file_id, owner_id, size, current_version, permission}` |
| `DELETE` | `/api/v1/files/{file_id}` | 删除文件（仅 owner） | – | `204 No Content` |

### 4.3 实时协同（WebSocket）  

> **WebSocket 只负责实时操作流**，所有持久化、权限校验在后端完成后再推送。  

**握手 URL**：`wss://collab.figma.com/ws?fileId={file_id}&access_token={jwt}`  

**消息格式（JSON）**  

```json
{
  "type": "op",                 // "op" | "cursor" | "ack" | "error"
  "clientId": "c-12345",       // 客户端唯一标识（用于 ACK）
  "seq": 0,                     // 客户端本地序号（递增）
  "payload": {                 // 依据 type 不同结构
    "op": {                    // 当 type == "op"
      "opId": "uuid",
      "userId": "uuid",
      "crdt": { ... }          // CRDT 操作 JSON
    },
    "cursor": {                // 当 type == "cursor"
      "x": 123,
      "y": 456,
      "selectedIds": ["node-1","node-2"]
    }
  }
}
```

**服务器返回**  

- **ACK**：`{ "type":"ack", "clientId":"c-12345", "globalSeq": 987654 }`（告知客户端已落库的全局序号）  
- **Broadcast**：把其它用户的 `op`/`cursor` 推给所有已订阅该 `fileId` 的客户端。  

### 4.4 离线同步（HTTP）  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/files/{file_id}/sync` | 客户端把离线操作批量上传 | `{operations: [{seq, crdt, clientVersion}], clientVersion}` | `{serverVersion, missingOps: []}` |

- **业务流程**：  
  1. 客户端把本地 **操作日志**（按 seq）一次性发送。  
  2. 服务端按 **CRDT** 合并顺序写入 Kafka + Cassandra，返回 **最新全局版本号**。  
  3. 若服务端检测到 **缺失的中间操作**（网络抖动导致部分未收到），在响应里返回 `missingOps`，客户端补发。  

### 4.5 版本 & 撤销/重做  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `GET` | `/api/v1/files/{file_id}/versions` | 列出可回滚的快照/版本 | – | `[{version, created_at, size}]` |
| `POST` | `/api/v1/files/{file_id}/revert` | 回滚到指定版本 | `{targetVersion}` | `{newVersion}` |
| `POST` | `/api/v1/files/{file_id}/undo` | 发起撤销（对当前用户的最近一次操作） | – | `{opId, newVersion}` |
| `POST` | `/api/v1/files/{file_id}/redo` | 发起重做 | – | 同上 |

> **实现要点**：撤销本质上是 **新建一个反向 CRDT 操作**，写入同一日志流，这样所有客户端都能统一看到撤销效果。

## 第五步：详细组件设计  

下面把 **每个核心组件** 拆成子模块，说明内部职责、交互协议、容错手段以及关键实现细节。

### 5.1 入口层（CDN + Edge WebSocket Proxy）  

- **功能**：TLS 终止、IP 过滤、近用户路由（Anycast DNS），把 WebSocket 连接转发到最近的 **Collaborative Service**。  
- **实现**：使用 **Cloudflare Workers / AWS CloudFront + ALB**，在 Edge 节点维持 **WebSocket Keep‑Alive**。  
- **为什么要在 Edge 做转发**：  
  - **降低 RTT**：用户的光标移动每 50 ms 必须感知，跨洲传输会导致 >150 ms，违背 N4。  
  - **防止 DoS**：在 Edge 层进行速率限制、IP 黑名单，减轻后端压力。  

### 5.2 API Gateway / 负载均衡  

- **职责**：统一鉴权、流量治理、灰度发布、路由到不同的 **Collab Service** 实例。  
- **实现**：Kong / Kong‑Enterprise 或 **AWS API Gateway + Lambda Authorizer**。  
- **关键点**：  
  - **JWT 校验**：在网关层完成，后端只收到已校验的 `userId`，避免每个服务都要重复鉴权。  
  - **熔断 & 限流**：针对每个 `fileId` 设定 QPS 上限，防止单文件热点导致整个节点崩溃。  

### 5.3 Collaborative Service（核心业务）  

#### 5.3.1 架构与部署  

- **无状态**：所有业务状态保存在 **Redis Streams**（实时）和 **Cassandra**（持久化），因此节点可以随时上下线。  
- **水平扩展**：基于 **文件 hash** 的 **分片**（Shard）把请求路由到对应的实例组。  

#### 5.3.2 关键模块  

| 模块 | 说明 | 关键算法 |
|------|------|----------|
| **Auth & Permission Filter** | 根据 `userId` + `fileId` 检查编辑/只读权限，拒绝非法写入 | RBAC 表 + ACL 检查 |
| **CRDT Engine** | 负责把收到的操作转换为 **CRDT**（如 Yjs 的 `Y.Map`、`Y.Array`）并生成 **全局版本号** | **LWW‑Element‑Set** + **RGA**（可变向量） |
| **Sequencer / Version Manager** | 为每条操作分配单调递增的 **global_seq**（借助 **etcd** 的 **lease/sequence**） | **Raft** 中的 **log index** |
| **Real‑time Broadcaster** | 把已合并的操作写入 **Redis Stream**，并通过 **Pub/Sub** 推送给同 shard 所有客户端 | `XADD`, `XREADGROUP` |
| **Persistence Writer** | 异步把操作写入 **Kafka**（持久化）和 **Cassandra**（增量日志） | 采用 **异步批量写**（batch size 100）降低延迟 |
| **Snapshot Scheduler** | 定时读取最新 CRDT 状态，序列化后写入对象存储 | **Cron + Worker**，使用 **GZIP** 压缩 |
| **Sync Service** | 接收离线批量同步请求，做 **幂等合并** 与 **冲突检测** | 基于 **CRDT** 的 **合并函数**（commutative） |

#### 5.3.3 CRDT 选型细节  

| CRDT 类型 | 适用场景 | 优点 | 缺点 |
|-----------|----------|------|------|
| **Yjs / Automerge (sequence‑based)** | 文本、图形属性的有序列表 | **强一致性**、离线友好、成熟库 | 需要 **对象模型映射**（把节点属性映射成 JSON） |
| **LWW‑Element‑Set** | 单属性（颜色、可见性） | 简单、冲突解决直观（最近写入赢） | 只能解决 “最后写入” 冲突，无法捕获多属性复合修改 |
| **RGA (Replicated Growable Array)** | 有序层级结构（画布层级、组/子组） | 保证 **因果顺序**，适合树形结构 | 实现稍复杂，日志体积略大 |

**综合考虑**：Figma 的对象模型是 **树形（页面 → 画布 → 图层 → 节点）**，且每个节点属性是 **键值对**。  
→ **采用 Yjs**（底层使用 **RGA + LWW-Map**），能够一次性同步 **节点增删**、**属性变更**，并且 **离线合并** 无需中心服务器。  

### 5.4 消息总线（Kafka + Redis Streams）  

- **Kafka**：  
  - **持久化**：每条操作写入 **topic = fileId**（分区数 = 3×副本），提供 **Replay**、**审计**。  
  - **消费者**：**Snapshot Worker**、**Analytics**、**Backup**（异地灾备）。  
- **Redis Streams**：  
  - **低延迟广播**：每个 shard 有独立 `stream:fileId`，协作节点 `XREADGROUP` 读取后立即推给 WebSocket 客户端。  
  - **幂等**：使用 `clientId+seq` 作为 **Message ID**，防止网络抖动导致重复投递。  

### 5.5 持久化层（Cassandra + Object Store）  

- **Cassandra**（或 **ScyllaDB**）  
  - **写放大**：采用 **宽表**（每条日志占 1 行），分区键 `file_id`，聚簇键 `seq_num`。  
  - **TTL**：对 **旧日志**（超过 30 天）设定 TTL，降低存储压力（历史版本仍保存在快照）。  
- **对象存储**（S3/MinIO）  
  - **快照**：每次快照写入 `PUT /snapshots/{fileId}/{version}.json.gz`。  
  - **Lifecycle Policy**：90 天后转入 **Glacier**，满足 N6 中的 **归档需求**。  

### 5.6 权限服务（Auth + Permission）  

- **Auth Service**：基于 **OAuth2 + JWT**，统一登录、单点登录（SSO）。  
- **Permission Service**：查询 `file_permissions` 表，返回 **enum**（READ / COMMENT / EDIT）。  
- **安全检查点**：在 **API Gateway**、**Collaborative Service**、**Sync Service** 三层都做权限校验，防止“越权写入”。  

### 5.7 监控、日志、告警  

| 监控项 | 实现方式 |
|--------|----------|
| **实时延迟**（客户端→服务→客户端） | 在 WebSocket 消息中携带 `client_ts`、`server_ts`，Prometheus 记录 99th percentile |
| **QPS / 并发会话数** | API Gateway 计数器 + Redis `INCRBY` 统计 |
| **错误率** | HTTP 5xx / WebSocket error 计数 |
| **资源使用** | Node CPU/Memory（cAdvisor） |
| **异常流量** | WAF + Rate limiting，触发 PagerDuty 报警 |
| **日志** | 使用 **ELK**（Filebeat → Logstash → Kibana）收集操作日志、业务日志 |

## 第六步：扩展性与高可用设计  

### 6.1 分片（Sharding）策略  

1. **Hash 分片**：`shardId = hash(fileId) % N`，N 为当前集群的 shard 数（如 256）。  
2. **热点文件**：对 **大文件**（> 50 MB 或 > 5k 并发）采用 **子分片**：将其内部 **层级节点** 再哈希到不同的协作节点（类似 **分布式锁表**）。  
3. **动态迁移**：使用 **Consistent Hash Ring**，当某个 shard 负载过高时可 **添加新节点**，自动迁移部分文件的元数据（metadata 在 Cassandra 中，迁移成本低）。  

> **为什么要分片**：防止单节点成为 **热点**，保证 **线性扩展**（N6 中的 QPS 目标）。  

### 6.2 多 AZ 高可用  

| 组件 | 多 AZ 方案 | 容错方式 |
|------|------------|----------|
| **Collaborative Service** | 每 AZ 部署同等数量的实例，使用 **Global Load Balancer**（Anycast） | 客户端自动连接最近 AZ，若节点故障，LB 自动切换 |
| **Kafka** | **跨 AZ Replication**（ISR），副本跨 AZ | 任何副本失效，ISR 自动恢复 |
| **Redis** | **Cluster Mode** + **Active‑Active**（CRDT based） | 写入任意节点，数据在各 AZ 同步 |
| **Cassandra** | **NetworkTopologyStrategy**（每 AZ 至少 3 副本） | 任意 AZ 故障，仍能读取写入 |
| **Object Store** | **跨区域复制**（S3 Cross‑Region Replication） | 读取最近副本，灾备恢复 5 分钟内完成 |

### 6.3 故障恢复流程  

1. **单节点 Crash**（Collaborative Service）  
   - **检测**：Health check → LB 标记为 UNHEALTHY。  
   - **恢复**：新实例自动注册到 **Service Discovery**（Consul/Eureka），接管该 shard 的流量。  
2. **网络分区**（跨 AZ）  
   - **写入**：使用 **Quorum**（Kafka ISR、Cassandra）保证多数节点写成功。  
   - **恢复**：分区恢复后，Kafka 自动进行 **leader election**，Cassandra 进行 **hinted handoff**。  
3. **对象存储故障**  
   - **备份**：跨区域复制 + **版本控制**。  
   - **回滚**：从最近的副本恢复快照。  

### 6.4 大规模协作（10k+ 同时编辑）  

- **分层广播**：  
  1. **Edge 节点** 收到光标/属性小流量（< 5 KB/s），直接转发给同区域的 **WebSocket Proxy**。  
  2. **核心协作节点** 只处理 **结构化变更**（增删、属性批量），每 50 ms 合并一次，降低消息频率。  
- **写入节流**：  
  - 对 **同一对象的高频修改**（如同一矩形的宽度）使用 **本地去重**（在客户端只保留最新值），服务器端只接受 **最终状态**（**LWW**），避免日志膨胀。  
- **分布式锁（可选）**：对于 **极端热点对象**（如共享颜色变量），可以引入 **乐观锁 + 版本号**，冲突概率降低。  

### 6.5 离线同步的高效实现  

1. **客户端本地日志结构**（SQLite / IndexedDB）  
   - **每条记录**：`{localSeq, crdtOp, timestamp}`。  
   - **压缩**：每 500 条合并为一次 **Batch**（JSON + gzip）。  
2. **服务器端批量合并**  
   - 接收 Batch → 按 `localSeq` 排序 → **CRDT Merge** → 生成 **全局 seq** → 批量写入 Kafka + Cassandra（一次性写入 10‑100 条）  
   - **幂等**：使用 `clientId+localSeq` 去重，若已存在直接返回 ACK。  
3. **冲突检测**  
   - CRDT 天然 **冲突无害**：不同属性的改动会自动合并；同属性采用 **LWW**（最新时间戳赢）。  
   - 对于 **业务级冲突**（如删除后修改属性），在 **CRDT** 中使用 **tombstone**（删除标记）并在合并时检查。  

### 6.6 成本与容量规划  

| 资源 | 估算需求 | 选型建议 |
|------|----------|----------|
| **WebSocket 连接** | 150k 并发 × 2 KB 心跳/秒 ≈ 300 MB/s | 使用 **NGINX+Lua** 或 **Envoy**，开启 **TCP Fast Open** |
| **Kafka** | 150 MB/s 持久化（含压缩） | 8 节点 3 副本，磁盘使用 **NVMe** |
| **Redis** | 200k QPS，数据 < 5 GB | **Cluster** + **replication factor 3** |
| **Cassandra** | 5 TB 写入/天 | 12 节点，RF=3，使用 **SSD** |
| **对象存储** | 7 PB/年 | S3 Intelligent‑Tiering + Glacier for > 90 days |
| **CPU** | 每协作节点 2‑4 核心，处理 CRDT | 使用 **c5.large**（AWS）或 **自研容器** |

## 第七步：常见面试追问与回答  

下面列出面试官经常追问的细节，并提供 **完整答案**（包括思考过程），帮助你在现场快速组织语言。

### 7.1 冲突解决细节  

#### 7.1.1 示例：两位用户同时修改同一个矩形的 **宽度** 与 **颜色**  

- **操作 A（用户1）**：`set width = 200`（timestamp 10:00:01.123）  
- **操作 B（用户2）**：`set color = #FF0000`（timestamp 10:00:01.124）  

**CRDT 处理过程**  

1. 两条操作分别以 **LWW‑Element‑Set** 形式写入同一 **属性 Map**（`width`、`color`）。  
2. 合并时，**宽度** 和 **颜色** 属于 **不同键**，CRDT 直接把两者都保留下来（**无冲突**）。  
3. 最终状态：`width = 200`，`color = #FF0000`。  

> **为什么不需要额外冲突检测**：CRDT 的 **属性 map** 天然是 **键分离**，只有同一键的并发写才会产生冲突。  

#### 7.1.2 OT 与 CRDT 对比  

| 维度 | OT (Operational Transformation) | CRDT |
|------|----------------------------------|------|
| **实现难度** | 需要 **中心化的转化函数**，对每种操作都写转换规则，代码量大 | 大多数库（Yjs、Automerge）已实现，使用即得 |
| **离线/分区容忍** | 必须在服务器保持 **全序**，离线时需要回放并重新变换，复杂 | **自然并发合并**，离线后直接把本地操作发回即可 |
| **性能** | 每次变换都要遍历历史操作，**O(N)**，在大文件上可能卡顿 | 合并 **O(1)**（基于 LWW / RGA），适合高频编辑 |
| **一致性保证** | 通过 **变换函数** + **中心序列号** 达到 **意图保留**，实现较难 | **强最终一致性**（CRDT 本身属性），易验证 |
| **选型结论** | 适用于 **纯文本协作**（Google Docs） | 更适合 **结构化、图形对象**（Figma） |

**最终选型**：**CRDT**（Yjs）——离线友好、实现成本低、对图形对象的属性合并天然支持。

### 7.2 扩展性与分区策略  

#### 7.2.1 150k 并发会话的分区方案  

1. **文件级哈希分片**（hash(fileId) % 256） → 每个 shard 对应一组 **Collaborative Service** 实例。  
2. **热点文件子分片**：当同一文件的 QPS > 2k 时，进一步把该文件的 **层级节点** 按 **nodeId** 再哈希分配到 **子协作实例**（类似 **分布式渲染**），降低单实例压力。  
3. **全局路由表**：使用 **Consistent Hash Ring** + **Service Discovery**（etcd），客户端或 API Gateway 根据 `fileId` 找到对应的 **入口节点**。  

> **不做分片的后果**：所有请求集中到少数机器，CPU/网络瓶颈迅速出现，导致 **延迟 > 500 ms**，违背 N4。  

#### 7.2.2 大企业一次性 10,000 人共同编辑  

- **横向扩容**：为该文件临时 **扩容至 50‑100 个协作节点**，每个节点负责约 100‑200 人的实时流。  
- **分层广播**：  
  - **Edge → Region → File‑Group**：Edge 将光标/小变更转发给同 Region 的 **Group Hub**，Group Hub 再聚合后推给内部协作节点。  
  - **批量合并**：每 20 ms 将同一对象的多次属性改动压缩为 **单条 LWW 操作**，减少消息量。  
- **资源预留**：在系统调度层（K8s/HPA）为 **热点文件** 预留 **Burst Capacity**（CPU、网络、内存），确保瞬时流量不被 throttling。  

### 7.3 离线同步与冲突  

#### 7.3.1 客户端离线日志的存储与清理  

| 阶段 | 客户端实现 | 说明 |
|------|------------|------|
| **编辑期间** | **IndexedDB**（浏览器）或 **SQLite**（桌面）存储 `[{localSeq, crdtOp, ts}]` | 本地持久化防止页面刷新丢失 |
| **同步成功** | **删除** 已确认的日志，保留 **未 ACK** 的部分 | 通过服务器返回的 `globalSeq` 进行对比 |
| **日志压缩** | 每 500 条合并为 **gzip batch**，写入 `Sync Batch` 接口 | 减少网络请求次数 |
| **清理策略** | **TTL**（30 天）+ **手动清理**（用户登出时） | 防止本地磁盘无限增长 |

#### 7.3.2 恢复网络后的合并流程  

1. **客户端** 将本地批次发送到 `/sync` 接口，附带 **最高本地 seq**。  
2. **服务器** 检查 **全局 seq** 与 **clientVersion** 的差距：  
   - 若服务器已有的操作 **比 clientVersion 更新**，返回 **missingOps**（从服务器拉取缺失的操作），客户端先 **回放** 再 **发送本地批次**。  
3. **服务器端** 对每条 **CRDT 操作** 进行 **幂等合并**（使用 `clientId+localSeq` 去重），生成 **全局 seq**，写入 Kafka + Cassandra。  
4. **服务器** 将 **全局 seq** 返给客户端，客户端标记本地日志已持久化。  

> **关键点**：**CRDT 的可交换性** 让离线操作不需要先解决冲突，合并后自然得到一致状态。  

### 7.4 监控与故障处理  

| 监控指标 | 阈值（示例） | 报警方式 |
|----------|--------------|----------|
| **WebSocket 延迟**（client→server→client） | > 120 ms（99%） | PagerDuty + Slack |
| **协作节点 CPU** | > 80% 持续 5 min | 自动水平扩容 |
| **Kafka ISR 副本数 < 2** | – | 邮件告警 |
| **Redis Stream Lag** | > 200 ms | 自动重启节点 |
| **Cassandra Write Latency** | > 30 ms | 增加写入节点 |

**故障演练**：每月进行 **Chaos Monkey**，随机关闭一台协作节点，验证系统自动切换、数据不丢失、延迟仍在 SLA 范围内。

## 心得与反思  

### 1. 本题最难的设计决策  

| 决策 | 思考过程 |
|------|----------|
| **CRDT vs OT** | 初看 OT 好像更“精准”（保持意图），但实现复杂、离线难以保证全序。经过对 Figma **对象模型**（树形、属性键值）分析，发现 **CRDT（Yjs）** 能直接映射为 **Map + List**，天然支持离线合并，且已有成熟开源实现。于是决定采用 CRDT，省去自研 OT 的高风险。 |
| **分片与热点处理** | 150k 并发看似均匀，但实际**热点文件**会出现指数级流量。最初想单纯用 **hash(fileId) % N**，但担心单文件负载过高导致节点崩溃。于是引入 **子分片**（把文件内部的节点再哈希），并配合 **动态扩容**（按热点实时加节点），兼顾负载均衡和一致性。 |

### 2. 新手最容易犯的错误  

| 错误 | 解释 | 正确做法 |
|------|------|----------|
| **把所有数据放在单一数据库** | 只用关系库保存所有日志、快照、权限等，会导致写入瓶颈、扩展困难。 | 按业务划分存储：**Cassandra** 保存增量日志、**对象存储** 保存快照、**PostgreSQL** 保存元数据、**Redis** 做实时广播。 |
| **忽视网络延迟的根因** | 只关注后端处理时间，忘记 **跨地域 RTT**，导致无法满足 150 ms SLA。 | 在 **Edge** 部署 WebSocket 代理、使用 **QUIC**，并把 **光标/心跳** 放在离用户最近的节点。 |
| **把冲突解决写在业务层** | 在业务代码里手写“最后写入赢”，会在离线同步时出现不可预料的状态。 | 使用 **CRDT**，让冲突解决在数据结构层自动完成，业务层只负责业务校验。 |
| **不考虑幂等性** | 网络抖动导致重复操作，系统状态出现双计数。 | 所有写入（WebSocket、HTTP）都要携带 **唯一 clientId+seq**，服务器在写入前做去重检查。 |

### 3. 学习建议与可延伸方向  

1. **系统设计基本功**  
   - 熟悉 **CAP 定理、BASE**，学会在 **一致性 vs 可用性** 之间权衡。  
   - 多练 **高并发 QPS 估算** 与 **容量规划**，面试时能快速给出数字。  

2. **深入 CRDT**  
   - 阅读 **“A comprehensive study of CRDTs”**（Shapiro et al.）以及 **Yjs**、**Automerge** 源码。  
   - 在本地实现一个小型 **协同绘图** demo，体会属性合并的细节。  

3. **实时系统实践**  
   - 熟悉 **WebSocket、QUIC** 的握手与心跳机制。  
   - 探索 **Redis Streams** 与 **Kafka** 的区别，了解 **Exactly‑once** 与 **at‑least‑once** 的语义。  

4. **分布式存储**  
   - 学习 **Cassandra/Scylla** 的 **data model** 与 **tunable consistency**。  
   - 了解 **对象存储 Lifecycle Policy**，如何实现冷热分层。  

5. **监控与故障演练**  
   - 搭建 **Prometheus + Grafana**，熟悉 **SLO/SLI** 的定义。  
   - 练习 **Chaos Engineering**（Chaos Monkey、Litmus）来验证高可用设计。
