# 面试题目：设计 iCloud（个人云存储与同步）系统

## 1. 题目背景
iCloud 是 Apple 为个人用户提供的云端存储与同步服务，支持照片、文档、备份、通讯录、日历等数据在多台 iOS/macOS 设备之间无缝同步。用户可以随时随地访问、分享和恢复自己的数据。

## 2. 面试场景设定
> **面试官**：  
> “我们现在要设计一个类似 iCloud 的个人云存储与同步系统，核心目标是让全球数以亿计的 Apple 设备能够安全、快速地上传、下载和同步用户的各种数据。请先从需求出发，整体把握系统的关键组成部分，然后我们再逐步深入细节。你可以先概述一下你的设计思路吗？”

## 3. 功能性需求（核心功能）

| 编号 | 功能描述 | 关键点 |
|------|----------|--------|
| 1 | **文件上传/下载**：用户可以把任意文件（照片、文档、备份等）上传到云端，并在其他设备上下载 | 支持分块上传、断点续传、流量压缩 |
| 2 | **跨设备同步**：同一用户的多台设备之间的文件、相册、通讯录、日历等数据保持实时或准实时同步 | 采用版本冲突解决策略、增量同步 |
| 3 | **文件共享与协作**：用户可以生成分享链接或邀请其他 Apple ID 共同编辑/查看文件 | 权限控制（只读/可编辑/过期链接） |
| 4 | **数据备份与恢复**：自动备份设备全盘或增量备份，支持一键恢复到任意历史时间点 | 加密备份、分层存储 |
| 5 | **安全与隐私**：端到端加密、身份认证、访问审计 | 客户端加密密钥管理、双因素认证 |
| 6 | **离线访问与缓存**：在设备离线时仍能访问最近使用的文件，随后在网络恢复后自动同步 | 本地缓存策略、冲突合并 |

## 4. 非功能性需求（关键指标）

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 2.5 亿 | 假设 30% 的全球 iPhone/iPad/macOS 活跃用户使用 iCloud |
| **QPS（上传/下载请求峰值）** | 150,000 QPS | 峰值时段（上午 9:00–11:00、晚上 20:00–22:00） |
| **单次上传/下载延迟** | ≤ 300 ms（99% 请求） | 包括网络往返 + 服务端处理 |
| **系统可用性** | 99.99% 年可用性（约 52.6 分钟宕机） | 必须满足企业用户对备份的高可靠性需求 |
| **存储容量** | 3 EB（Exabytes） | 预计 2026 年全球 iCloud 存储总量 |
| **数据恢复时间目标 (RTO)** | ≤ 5 分钟（对单用户备份恢复） | 关键业务需求 |

> *注：上述数字为面试估算，实际产品会依据市场调研与业务增长动态调整。*

## 解题思路总览
1. **先把业务拆成最小可跑通的功能**（单机版文件上传/下载），确保我们能写出一套完整的请求‑响应流程。  
2. **逐步加入分布式必需的特性**：分块上传、元数据服务、对象存储、鉴权、缓存、异步任务。  
3. **再往上加全局需求**：多地域、冷热分层、端到端加密、审计、监控、限流。  
4. **每加一种新特性，都要问**：  
   - *为什么现在需要它？*（性能、可靠性、成本）  
   - *不加会出现什么问题？*（单点故障、瓶颈、数据不一致）  
5. **最后把所有块拼成一张全局时序图**，帮助面试官快速看到“请求从入口到落库再到返回”的完整路径。

> **核心理念**：从 **“先满足功能再满足非功能”** → **“先单点再拆分”** → **“先同步再异步”** 的递进式思考。

## 第一步：理解需求与规模估算

### 1.1 功能需求快速梳理
| 编号 | 功能 | 必须实现的最小子功能 | 后期可演进的高级特性 |
|------|------|-------------------|-------------------|
| 1 | 文件上传/下载 | 单文件完整上传、下载 | 分块、断点续传、压缩、CDN 加速 |
| 2 | 跨设备同步 | 同一用户在两台设备间看到相同文件列表 | 增量同步、冲突检测、实时推送 |
| 3 | 文件共享 | 生成只读链接 | 可编辑协作、过期链接、访问审计 |
| 4 | 备份恢复 | 将设备全盘打包上传、单文件恢复 | 增量备份、历史快照、跨设备恢复 |
| 5 | 安全隐私 | 用户登录、Token 校验、传输加密（TLS） | 端到端加密、双因素、审计日志 |
| 6 | 离线缓存 | 本地磁盘缓存最近文件 | 本地数据库同步、冲突合并 |

> **最小可用系统（MVP）**：实现 **1**（单文件上传/下载）+ **5**（登录鉴权）+ **4**（备份的“全盘上传”），其他功能可以在后续迭代中逐步加入。

### 1.2 非功能需求估算（粗略计算）

| 项目 | 估算方式 | 结果 |
|------|----------|------|
| **DAU** | 2.5 × 10⁸（已给） | - |
| **每日上传/下载次数** | 假设每位活跃用户平均每天上传 2 GB、下载 1 GB，平均文件大小 5 MB ⇒ 每用户约 600 次操作。<br>QPS ≈ 2.5e8 × 600 / 86400 ≈ 1.74 × 10⁶（峰值 150k QPS 为保守） | 1.74 M 请求/天 ≈ 150k QPS 峰值 |
| **存储总量** | 3 EB（已给） | - |
| **网络带宽** | 峰值 QPS × 平均响应大小（5 MB） ≈ 150k × 5 MB ≈ 750 GB/s ≈ 6 Tbps（需要 CDN+分片） | - |
| **延迟目标** | 99% ≤ 300 ms | 需要就近调度、缓存、异步写入 |
| **可用性** | 99.99% = 52.6 min/year | 多活多地域部署 + 自动故障转移 |

> **为什么要做这些估算？**  
- 让我们在选技术栈时有 “容量感知”，避免“一开始就选最贵的”或“选不适配的”。  
- 在面试里直接给出数字，显得你对业务有深度理解。

## 第二步：高层架构设计

### 2.1 MVP（单机版）结构

```
+-------------------+        +-------------------+
|   iOS/macOS 客户端 | <----> |  API Gateway (HTTP)|
+-------------------+        +-------------------+
                               |
                               v
                        +---------------+
                        |   Auth Service|
                        +---------------+
                               |
                               v
                        +---------------+
                        |   File Service|
                        +---------------+
                               |
                               v
                        +---------------+
                        |  本地文件系统 |
                        +---------------+
```

- **API Gateway**：统一入口，做路由、限流、日志。  
- **Auth Service**：OAuth2 + JWT，返回短期 Access Token。  
- **File Service**：接收完整文件流，写入本地磁盘，返回文件 ID。  

> **不这么做的后果**：直接把业务代码写在网关里，后期想拆分成微服务会很痛苦；单点故障导致整个系统不可用。

### 2.2 生产级分布式架构（逐层拆解）

```
+-----------------------+   +-------------------+   +-------------------+
|   全球 CDN / Edge     |   |   API Gateway     |   |   Rate Limiter    |
+-----------------------+   +-------------------+   +-------------------+
          |                         |                       |
          v                         v                       v
+-------------------+   +-------------------+   +-------------------+
|   Auth Service    |   |   Sync Service    |   |   Share Service   |
+-------------------+   +-------------------+   +-------------------+
          |                         |                       |
          v                         v                       v
+-------------------+   +-------------------+   +-------------------+
|   Metadata DB    (R/W) |   Metadata DB (R)   |   Metadata DB (R) |
+-------------------+   +-------------------+   +-------------------+
          |                         |                       |
          v                         v                       v
+-------------------+   +-------------------+   +-------------------+
|   Object Store (S3‑like)  |   |   Object Store (S3‑like) |
+-------------------+   +-------------------+   +-------------------+
          |                         |
          v                         v
+-------------------+   +-------------------+
|   CDN Cache (Hot) |   |   Cold Storage    |
+-------------------+   +-------------------+
```

#### 关键模块解释
| 模块 | 主要职责 | 为什么要单独拆出来 |
|------|----------|-------------------|
| **CDN / Edge** | 静态文件（图片、文档）就近分发，降低延迟 | 直接访问对象存储会跨地域、延迟高 |
| **API Gateway** | 统一入口、TLS 终结、流量治理、灰度发布 | 让后端服务专注业务，不处理网络细节 |
| **Auth Service** | 登录、Token 发放、密钥轮换 | 业务系统不想耦合身份验证，安全隔离 |
| **Sync Service** | 处理设备间的增量同步、冲突检测、推送 | 同步逻辑与文件上传/下载分离，便于水平扩展 |
| **Share Service** | 生成/校验分享链接、权限校验 | 权限模型复杂，单独服务便于审计 |
| **Metadata DB** | 保存文件/目录的元信息、版本、权限 | 元数据查询频繁，需要高并发读写分离 |
| **Object Store** | 实际存放二进制数据（Blob），支持分块、版本 | 大文件、冷热分层都在这里实现 |
| **CDN Cache** | 热数据（最近访问的文件）在边缘缓存 | 大幅降低对象存储的读取成本 |
| **Cold Storage** | 冷数据（长期归档）使用更低成本存储 | 成本控制关键，满足 3 EB 规模 |

> **不做分层存储的风险**：所有数据都放在高性能 SSD，成本指数级爆炸；而且热数据占比往往只有 10%~20%。

## 第三步：数据库设计

> **核心原则**：元数据必须 **强一致**（用户看到的文件列表不能出现冲突），而二进制对象可以 **最终一致**（上传成功后异步复制到多副本）。

### 3.1 元数据模型（关系型 + KV）

#### 3.1.1 关系型表（如 MySQL / PostgreSQL / Aurora）
| 表名 | 主键 | 关键字段 | 说明 |
|------|------|----------|------|
| `users` | `user_id` (UUID) | `email, password_hash, mfa_secret` | 用户基本信息 |
| `devices` | `device_id` (UUID) | `user_id, platform, last_sync_ts` | 每台设备的注册信息 |
| `folders` | `folder_id` (UUID) | `user_id, parent_id, name, created_at, updated_at` | 目录结构（树形） |
| `files` | `file_id` (UUID) | `folder_id, user_id, name, size, mime, hash, created_at, updated_at, latest_version_id` | 文件的抽象层 |
| `file_versions` | `version_id` (UUID) | `file_id, object_key, size, hash, created_at, is_deleted` | 每一次上传对应的对象存储 key |
| `shares` | `share_id` (UUID) | `file_id, creator_user_id, permission, expires_at, token` | 分享链接信息 |
| `audit_logs` | `log_id` (UUID) | `user_id, action, resource_type, resource_id, ts, ip, client_version` | 访问审计 |

- **Why UUID?** 分布式系统里不想依赖自增 ID 的中心化生成器，UUID 能在各节点独立生成且冲突概率极低。  
- **Why separate `file_versions`?** 支持 **版本回滚**、**增量同步**、**软删**（标记为 deleted），而不必改 `files` 表。

#### 3.1.2 KV/缓存（如 Redis）
| 键 | 示例 | 作用 |
|----|------|------|
| `user:{user_id}:quota` | `500GB` | 限速/配额检查 |
| `session:{token}` | `{user_id, exp}` | 快速鉴权 |
| `file_meta:{file_id}` | `{name, size, latest_version}` | 高频读取，降低 DB 压力 |
| `sync_queue:{user_id}` | List | 记录待同步的变更，供 Sync Service 拉取 |

### 3.2 对象存储（Object Store）结构
```
bucket/
   └─ user_id/
        └─ file_id/
               └─ version_id
```
- **路径设计**：`user_id/file_id/version_id` 让删除、迁移、权限校验都非常直观。  
- **元数据（ETag）**：存储对象的 MD5/SHA256，用于 **断点续传** 和 **完整性校验**。  

### 3.3 事务与一致性
- **上传流程**：  
  1. 客户端先向 **Sync Service** 请求 **上传凭证**（预签名 URL）。  
  2. 客户端直接向对象存储 **PUT** 分块。  
  3. 完成后调用 **Sync Service /files/uploadComplete**，服务端在 **metadata DB** 中创建 `file_versions` 条目并更新 `files.latest_version_id`（**事务**）。  

- **冲突检测**：在 `uploadComplete` 时检查 `files.updated_at` 是否比客户端带的 `client_version` 新，若更高则返回冲突错误，客户端重新拉取最新元数据后再上传。

> **不使用事务的后果**：文件实际写入成功，但元数据未更新，用户在别设备上看不到文件，导致“丢失”。

## 第四步：核心 API 设计

> **设计原则**：RESTful + JSON + HTTP/2，兼容移动端的网络特性（慢速网络、流量计费）。

| 场景 | HTTP 方法 | URL | 请求体（JSON） | 响应体 | 备注 |
|------|-----------|-----|----------------|--------|------|
| 登录 | POST | `/auth/login` | `{email, password, mfa_code?}` | `{access_token, refresh_token, expires_in}` | 使用 **OAuth2 Resource Owner Password**，实际可改为 Apple Sign‑In |
| 获取上传凭证（分块） | POST | `/files/{file_id}/uploadUrl` | `{file_name, size, mime, client_version}` | `{upload_id, part_size, urls: [ {part_number, url} ]}` | **预签名 URL**，客户端直接 PUT 到对象存储 |
| 完成上传 | POST | `/files/{file_id}/complete` | `{upload_id, parts: [{part_number, etag}] , client_version}` | `{file_id, version_id, metadata}` | 服务端验证 ETag，创建 `file_versions` 记录 |
| 下载文件 | GET | `/files/{file_id}/download` | `?version_id=` | **302 Redirect** → CDN/对象存储 URL | 支持 Range Header 进行断点续传 |
| 列出目录 | GET | `/folders/{folder_id}/list` | `?page=&size=` | `{items: [{type,file_id/folder_id,name,metadata}]}` | **缓存** `folder_id` 元数据 |
| 同步变更（长轮询/WebSocket） | GET | `/sync/changes` | `?since_ts=` | `{changes: [...]}` | 实时推送或轮询 |
| 创建分享链接 | POST | `/shares` | `{file_id, permission, expires_at}` | `{share_url, token}` | **只读/可编辑** |
| 访问分享链接 | GET | `/shares/{token}` | - | `{file_id, metadata, download_url}` | 直接返回对象存储 URL，保持 **匿名** |
| 备份恢复 | POST | `/backup/{device_id}/restore` | `{snapshot_ts}` | `{status, progress_url}` | 异步任务，返回进度查询接口 |

### 4.1 错误码约定（示例）

| HTTP 状态码 | 错误码 | 说明 |
|-------------|--------|------|
| 400 | `INVALID_PARAMETER` | 参数校验不通过 |
| 401 | `UNAUTHORIZED` | Token 失效或缺失 |
| 403 | `FORBIDDEN` | 权限不足或分享链接已失效 |
| 409 | `CONFLICT` | 版本冲突，需要客户端重新拉取 |
| 429 | `RATE_LIMITED` | 超过速率限制 |
| 500 | `INTERNAL_ERROR` | 服务内部异常 |

> **为什么要统一错误码**：移动端往往只根据状态码做重试，细粒度错误码帮助定位问题，提升用户体验。

## 第五步：详细组件设计

下面逐个拆解关键组件的内部实现细节、选型理由以及可能的备选方案。

### 5.1 API Gateway & Edge
- **技术选型**：Nginx + Lua（OpenResty） 或 Kong（插件化）  
- **职责**：TLS 终结、IP 限流、路由、灰度发布、请求日志、统一监控埋点。  
- **为何使用插件化网关**：能够在不改业务代码的前提下快速添加 **WAF、Auth、限流** 等功能；若直接在业务服务里实现，会导致代码耦合，后期难以演进。  

### 5.2 Auth Service
- **实现方式**：  
  - 使用 **OAuth2 + JWT**（HS256 或 RS256）  
  - **Refresh Token** 长期有效（30 天），存储在 **Redis**（可撤销）  
- **多设备密钥共享**：  
  - 每台设备在首次登录时生成 **Device Secret**，加密存入用户的 **Keybag**（在服务器加密后存储）。  
  - 通过 **Key Derivation Function (HKDF)** 为每个设备生成 **File Encryption Key**（用于 E2EE）。  
- **为何使用 JWT**：无状态，网关可以直接验证，无需每次请求都查询数据库，降低延迟。  

### 5.3 Sync Service（核心同步逻辑）
#### 5.3.1 增量同步机制
1. **客户端**在每次启动或网络恢复时，读取本地 **Change Log**（SQLite），把 **本地变更**（创建/修改/删除）发送给服务端。  
2. **服务端**把变更写入 **Change Stream**（Kafka / Pulsar），并写入 **metadata DB**（事务）。  
3. **服务端**把成功的变更写入 **UserSyncState**（Redis），更新 `last_sync_ts`。  
4. **服务端**通过 **Server Push**（WebSocket/GRPC）把 **其他设备**的变更推送过去。

#### 5.3.2 冲突检测
- **乐观锁**：在每次写入 `files` 表时，要求客户端提供 `client_version`（`updated_at` 时间戳或 `version` 整数）。  
- **冲突策略**：  
  - **文件**：保留两个版本，标记为 `conflict`，客户端弹出冲突 UI，用户自行选择合并或保留。  
  - **目录/元数据**：采用 **Last Writer Wins**（最新时间戳覆盖），因为冲突概率极低且影响范围小。  

#### 5.3.3 实时推送
- **技术**：使用 **GRPC bidirectional streaming** 或 **WebSocket**。  
- **负载均衡**：在 API Gateway 之上放置 **Envoy**，对长连接做连接复用、健康检查。  

### 5.4 Object Store（核心存储层）
- **选型**：自研对象存储（参考 Ceph / MinIO）或商业化对象存储（如 AWS S3、Google Cloud Storage），但本题更倾向自研，以便实现 **端到端加密、分层冷热、跨地域复制**。  
- **分块上传**：客户端一次性上传 **4 MB~8 MB** 块，服务器返回 **ETag**，完成后再提交。  
- **多副本**：采用 **Erasure Coding**（如 8+2）+ **跨 AZ 同步**，既保证耐久度（> 99.999999999%）又降低存储成本。  

#### 5.4.1 冷热分层
| 层级 | 访问频率 | 存储介质 | 迁移策略 |
|------|----------|----------|----------|
| **Hot** | > 10 %/day | NVMe SSD | 1 h 内未访问 → 自动降为 Warm |
| **Warm** | 1 %–10 %/day | SATA SSD | 7 day 未访问 → 自动降为 Cold |
| **Cold** | < 1 %/day | 磁带/低频 HDD | 30 day 未访问 → 归档（Glacier） |

- **自动迁移实现**：使用 **对象存储的 Lifecycle Policy**（基于对象的 `last_accessed` 元数据），定时任务（Cron）触发 **复制 + 删除**。  
- **为何分层**：直接把所有数据放在 SSD 成本高达数十亿美元；冷热分层让 90%+ 的冷数据以 **0.01 USD/GB/月** 的价格存储，显著降低运营成本。  

### 5.5 Share Service
- **生成分享链接**：随机 **128‑bit Token**，使用 **HMAC‑SHA256** 防篡改。  
- **权限校验**：在访问链接时，先校验 Token、权限、过期时间，随后返回 **对象存储预签名 URL**（只读或写入）。  
- **审计**：每一次分享链接的访问都写入 **audit_logs**（异步 Kafka → ClickHouse），用于后期统计和安全追踪。  

### 5.6 Backup & Restore Service
- **备份方式**：  
  - **全量备份**：把设备的文件系统快照压缩后上传为单个对象（`.tar.gz`），配合 **增量快照**（只上传变化的块）。  
  - **增量备份**：使用 **块级指纹（Rabin Fingerprinting）**，只上传新增块，降低带宽。  
- **恢复**：用户选择历史时间点，系统查询对应的 **快照元数据**，把对象下载到设备或直接通过 **恢复任务** 推送到目标设备。  
- **为何采用块级增量**：全量备份 100 GB 设备每次都要上传 100 GB，流量成本和时间不可接受。  

### 5.7 监控、日志、限流
| 维度 | 关键指标 | 工具 |
|------|----------|------|
| **监控** | QPS、CPU、磁盘 I/O、对象存储写入延迟、同步延迟 | Prometheus + Grafana |
| **日志** | 请求日志、错误堆栈、审计日志 | ELK（Filebeat → Logstash → Elasticsearch） |
| **限流** | 每用户每秒上传次数、每 IP 下载速率 | Redis Token Bucket + API Gateway 插件 |
| **告警** | 95% 延迟 > 300 ms、磁盘使用 > 80% | Alertmanager → PagerDuty |

> **不做监控的后果**：故障只能被动发现，恢复时间（MTTR）大幅提升，面试官会直接扣分。

## 第六步：扩展性与高可用设计

### 6.1 多地域部署
- **地域划分**：按照 **北美、欧洲、亚洲** 三大主流区域，每个区域内部署 **多个 AZ**（可用区）。  
- **流量路由**：使用 **Anycast DNS + Geo‑IP** 将用户请求路由到最近的区域。  
- **跨区域复制**：对象存储使用 **异步复制**（最终一致），元数据（MySQL）使用 **双主复制**（Galera）或 **TiDB**（分布式事务）保证强一致。  

### 6.2 容灾恢复（RTO/RPO）
| 场景 | RPO | RTO | 方案 |
|------|-----|-----|------|
| 单 AZ 故障 | ≤ 5 s（对象存储副本） | ≤ 30 s | 自动切换至同区域另一 AZ |
| 整个 Region 故障 | ≤ 10 min（跨区异步复制） | ≤ 5 min | DNS 切换到备份 Region，恢复服务 |
| 数据库主节点失效 | 0（同步复制） | ≤ 1 min | 自动故障转移（Pacemaker + Corosync） |

### 6.3 自动伸缩（Autoscaling）
- **API/业务层**：使用 **Kubernetes HPA**（基于 CPU/QPS）进行 Pod 自动伸缩。  
- **对象存储**：采用 **分片扩容**（Ceph OSD 增加）或 **对象存储服务弹性节点**。  
- **缓存层**：Redis Cluster 自动分片，水平扩容时不需要停机。  

### 6.4 成本控制
| 成本要素 | 控制手段 |
|----------|----------|
| **存储** | 冷热分层、生命周期自动归档、对象压缩（gzip） |
| **网络** | CDN 缓存热点文件、分块上传避免重复传输、压缩传输（HTTP/2 + gzip） |
| **计算** | 按需伸缩、无用的实例使用 **Spot**（抢占式）或 **Preemptible** |
| **运维** | 统一监控告警，提前发现浪费资源 |  

### 6.5 安全与合规
- **端到端加密**：  
  - 客户端生成 **文件加密密钥（FEK）**，使用 **AES‑256‑GCM** 加密文件。  
  - FEK 再使用 **用户的公开钥（User PK）** 加密，上传至 **Key Management Service (KMS)**。  
  - 多设备共享：每台设备持有用户的 **私钥（受 Secure Enclave 保护）**，解密 FEK 后本地再解密文件。  
- **密钥轮换**：KMS 支持 **定期自动轮换**，旧密钥仍能解密历史数据（密钥版本号存于元数据）。  
- **审计合规**：所有写操作（上传、删除、分享）记录 **不可篡改日志**（写入 **Append‑Only** 的 ClickHouse），满足 GDPR、CCPA 要求。

## 第七步：常见面试追问与回答

### 1️⃣ “如果用户在两台设备上同时编辑同一个文件，会产生冲突，你会采用怎样的冲突检测与合并策略？”

**回答思路**  
1. **乐观锁 + 版本号**：`files` 表里维护 `version`（递增整数）或 `updated_at`。客户端每次上传时必须带上自己认知的版本号。  
2. **服务端校验**：若提交的版本号不是当前最新，则返回 **409 CONFLICT**，并在响应体里返回最新的 `metadata`（包括最新的 `version_id`）。  
3. **冲突处理**：  
   - **文件**（二进制）无法自动合并，保留两个版本：`file_versions` 中新增一条 `conflict=true`，客户端 UI 提示用户手动合并（例如打开两份文件进行对比）。  
   - **结构化数据**（如通讯录、日历）可以采用 **CRDT（Conflict‑free Replicated Data Type）** 或 **Last Writer Wins**。  
4. **同步通知**：冲突产生后，`Sync Service` 会把冲突事件推送到所有在线设备，确保用户及时看到。

**不采用乐观锁的风险**：会出现 **脏写**，后面的写覆盖前面的写，导致用户数据不可恢复，体验极差。

---

### 2️⃣ “面对 3 EB 的冷热数据，你会如何设计对象存储的分层（热、温、冷）以及自动迁移策略？”

**回答要点**  
- **对象标签（Tag）**：每上传对象默认打上 `last_accessed=timestamp`、`storage_class=hot`。  
- **Lifecycle Policy**：  
  - **Hot → Warm**：若对象最近 24 h 未被访问，则自动把 `storage_class` 改为 `warm`（迁移到 SATA SSD）。  
  - **Warm → Cold**：若对象 7 day 未被访问，则改为 `cold`（磁带/低频 HDD）。  
  - **Cold → Glacier**：若 30 day 未被访问，则归档至 **Glacier**，只在需要恢复时才解冻（数小时）。  
- **实现方式**：  
  - **对象存储自带**（如 Ceph’s Tiering）或 **自研**：后台 **Cron**（每小时）读取 **对象元数据**（`last_accessed`），触发 **复制 + 删除**。  
  - **元数据同步**：迁移完成后更新 **metadata DB** 中的 `object_key` 与 `storage_class`。  
- **成本控制**：通过 **冷热分层**，90%+ 冷数据成本降低 80%~90%，仍能满足 **RTO ≤ 5 min**（因为冷数据在恢复时先在热层预取）。  

**如果不做分层**：所有数据都放在高性能 SSD，成本随存储增长呈指数，3 EB 级别的 SSD 费用超过数十亿美元，业务不可持续。

---

### 3️⃣ “端到端加密的密钥管理如何在不泄露用户明文的前提下实现多设备共享？”

**关键点**  
1. **用户主密钥（UMK）**：在用户第一次登录时，客户端在安全硬件（Secure Enclave / TPM）里生成 **随机 256‑bit 私钥**，只保存在本地。  
2. **公钥上报**：客户端把 **UMK 的公钥**（PEM）上传到 **KMS**，关联到 `user_id`。  
3. **文件加密密钥（FEK）**：每个文件（或每个块）生成 **随机 AES‑256‑GCM 密钥**。  
4. **密钥封装**：  
   - 使用 **UMK 公钥** 对 **FEK** 进行 **RSA‑OAEP** 加密，得到 `encrypted_fek`。  
   - `encrypted_fek` 随文件元数据一起存入 `file_versions`。  
5. **多设备读取**：设备登录后，从 **KMS** 拉取用户的 **UMK 公钥**（已在本地保存的私钥解密），再用 **私钥** 解密 `encrypted_fek`，得到 FEK，最后解密文件。  
6. **密钥轮换**：KMS 支持 **密钥版本**，当用户更换设备或启用二步验证时，重新生成 UMK 私钥，旧文件的 FEK 仍可使用旧私钥解密，兼容性通过 **密钥版本号** 管理。  

**安全保证**：  
- **服务器从不持有用户明文私钥**，只能看到加密后的 `encrypted_fek`。  
- **即使对象存储泄漏**，没有 UMK 私钥也无法解密文件。  

---

### 4️⃣ “如果某个 Region 故障，用户的同步会受多大影响？如何保证 RTO ≤ 5 min？”

**答要点**  
- **跨 Region 异步复制**：对象存储与元数据（MySQL）使用 **异步复制**，在 2–3 分钟内将增量同步到备份 Region。  
- **DNS/Anycast 切换**：故障检测（通过 Pingdom + 监控）触发 **Route53（或自研）** 的 **Health Check**，把用户的 DNS 解析切换到备份 Region（几秒内完成）。  
- **客户端容错**：客户端 SDK 在收到 **5xx** 错误或 **超时** 时自动切换到 **备份 Endpoint**（同一 API 接口），无需用户手动干预。  
- **恢复步骤**：  
  1. **故障检测** → 2 秒内触发 DNS 切换。  
  2. **流量切换** → 10 秒内全部请求落到备份 Region。  
  3. **数据同步完成** → 3 分钟内最新的写入已在备份 Region 可读。  
- **整体 RTO** ≈ **< 5 分钟**，满足题目要求。  

---

### 5️⃣ “如果 QPS 峰值达 150k，单次上传/下载延迟要 ≤ 300 ms，如何保证？”

**关键技术**  
- **就近 CDN**：热点文件走 **Edge CDN**，网络往返 < 50 ms。  
- **分块上传**：客户端并行上传 4–8 块（并发 4），单块大小 5 MB，单块延迟 ≈ 100 ms，整体感知 < 300 ms。  
- **后端异步写**：上传完成后立即返回成功，实际对象写入采用 **后台写入队列**（Kafka），不阻塞请求。  
- **缓存**：`file_meta:{file_id}` 存 Redis，读取一次元数据耗时 < 2 ms。  
- **负载均衡**：使用 **L4**（TCP）+ **L7**（HTTP/2）双层 LB，快速路由到最空闲的业务实例。

## 心得与反思

### 1️⃣ 本题最难的 1‑2 个设计决策及思考过程
| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **端到端加密的密钥共享** | 必须在不泄露明文的前提下让多台设备都能解密同一文件；同时要支持密钥轮换、设备丢失等场景。 | - 先明确**谁持有私钥**（客户端安全硬件）<br>- 再确定**密钥包装**方式（使用用户公钥加密文件密钥）<br>- 再考虑**多设备同步**（把公钥存 KMS，设备登录后拉取）<br>- 最后加入**密钥版本**和**轮换策略**以兼容旧数据。 |
| **冷热分层与自动迁移** | 3 EB 数据规模、成本控制与 RTO 之间的平衡。 | - 统计访问热点比例（10% 热、30% 温、60% 冷）<br>- 设计**对象标签+Lifecycle Policy**实现自动迁移<br>- 评估**迁移延迟**对恢复时间的影响，确保冷数据在需要时能在 5 min 内预热<br>- 预估成本节约（SSD vs HDD vs Glacier），确保业务可持续。 |

### 2️⃣ 新手最容易犯的错误（至少两条）
| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有业务都写在单体服务里** | 难以水平扩展、单点故障、代码难维护 | 从 **MVP** 开始，逐步拆分成 **API Gateway → Auth/Sync/File Service** 等独立微服务 |
| **忽视元数据的强一致性**，直接把文件写入对象存储后返回成功 | 可能出现“文件已经上传但在 UI 看不到”的情况，用户体验极差 | 在 **上传完成回调** 中使用 **数据库事务** 先写入 `file_versions`，成功后才返回 200 |
| **只考虑功能，不做容量/成本估算** | 面试官会质疑方案可落地性 | 通过 **DAU、存储总量、热点比例** 进行 **粗略容量规划**，并在设计中体现 **冷热分层、自动伸缩** |
| **把加密只做传输层 TLS** | 失去端到端加密的优势，数据在存储层可能被内部人员或泄漏读取 | 实现 **客户端文件加密 + KMS 包装**，服务器仅存密文 |
