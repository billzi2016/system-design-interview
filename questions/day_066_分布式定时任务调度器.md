# 第 66 天：设计 分布式定时任务调度器

> 生成日期：2026-03-21

---

## 题目背景  
分布式定时任务调度器是一套在大规模集群环境下，能够可靠地执行用户提交的定时或周期性任务（如报表生成、消息推送、缓存预热等）的系统。它需要保证任务在预定时间点触发、支持高并发、并在节点故障时仍能正常运行。

---

## 面试场景设定  
**面试官**：  
> “我们现在需要设计一套能够在多机房、多租户环境下运行的分布式定时任务调度器。请从整体架构出发，说明你会如何满足下面的功能和性能要求，并重点讨论任务的可靠性和水平扩展能力。”

---

## 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| **F1** | **任务创建/管理**：用户可以通过 API 或 UI 创建一次性或周期性任务，支持 Cron 表达式、固定延迟、指定时间点等多种调度方式。 |
| **F2** | **任务分片与负载均衡**：调度器能够把大量任务均匀分配到集群中的多个执行节点，支持动态扩容/缩容。 |
| **F3** | **任务容错与重试**：节点宕机或任务执行异常时，系统自动检测并在其他健康节点上重新调度，支持自定义重试次数、间隔和幂等控制。 |
| **F4** | **任务监控与告警**：提供任务执行状态（成功、失败、超时）、执行日志、指标统计，支持对异常任务触发告警（如邮件、Webhook）。 |
| **F5** | **权限与多租户**：不同业务线/租户只能管理和查看自己创建的任务，支持基于 RBAC 的细粒度权限控制。 |
| **F6** | **任务结果回调**：任务完成后可选调用用户提供的 HTTP 回调 URL 或写入消息队列，以便业务方进行后续处理。 |

| **N6** | **水平扩展** | 添加节点后 QPS 线性提升，单节点最大吞吐 ≥ 800 QPS，CPU 使用率 ≤ 70% |

# 题解

# 分布式定时任务调度器系统设计答案  

> **温馨提示**：本答案把“从 0 到 1”整个思考过程都展开写出来，力求让 **第一次接触系统设计的同学** 能够一步步跟随、明白每个决策背后的原因。阅读时请先通读一遍整体结构，再回到每个章节细细体会。  

---  

## 解题思路总览  

1. **先把需求拆成“最小可用系统”(MVP)**，只实现最核心的功能（任务创建 → 存储 → 按时触发 → 执行 → 回调），用最简单的单机模型验证业务流程是否正确。  
2. **逐步加入非功能需求**：并发、容错、扩容、多租户、监控等。每加入一种非功能需求，就在已有的 MVP 上做“演进”，把对应的组件抽象出来并实现分布式化。  
3. **围绕四大系统设计原则**：  
   - **单一职责**：每个服务/进程只负责一件事（如调度中心只负责选任务，执行节点只负责跑任务）。  
   - **显式依赖**：通过消息队列、服务注册中心等明确交互，而不是隐式共享内存。  
   - **故障隔离**：任何单点故障都要有备份或降级路径。  
   - **弹性伸缩**：所有关键路径的容量都可以水平扩展。  

下面按照 **七个章节**（从需求到实现，再到面试追问）逐层展开。

## 第一步：理解需求与规模估算  

| 需求 | 关键点 | 对系统的直接影响 |
|------|--------|-----------------|
| **F1** 任务创建/管理 | 支持一次性、周期性、Cron、延迟等多种触发方式 | 需要统一的 **任务模型**，以及能够解析 Cron 的库；存储层要支持 **高写入**。 |
| **F2** 任务分片与负载均衡 | 大量任务均匀分配，动态伸缩 | 必须把任务 **切分** 成独立的 **调度单元**，调度中心要做到 **无状态**，调度算法要支持 **一致性哈希** 或 **分区**。 |
| **F3** 容错与重试 | 节点宕机、任务异常重试，幂等控制 | 需要 **任务状态持久化**、**心跳检测**、**分布式锁**，以及 **重试策略**（次数、间隔）。 |
| **F4** 监控告警 | 任务状态、日志、指标 | 引入 **监控系统**（Prometheus）、**日志系统**（ELK），并提供 **告警规则**。 |
| **F5** 多租户、RBAC | 隔离租户数据、细粒度权限 | 任务表需要 **租户字段**，权限校验在 **API 网关** 或 **服务层**。 |
| **F6** 结果回调 | HTTP 回调或写入 MQ | 调度节点在任务完成后调用 **异步回调**，需要 **重试** 与 **幂等**。 |

### 规模估算  

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 10 000 租户，假设 10% 活跃 | 1 000 活跃租户 |
| **任务创建频率** | 平均每租户 5 条任务/天 | 5 000 条新任务/天 ≈ 0.058 QPS |
| **调度触发 QPS** | 峰值 5 000 QPS（系统给定） | 约 4.3 亿次/日 |
| **单节点目标吞吐** | 单节点 ≤ 800 QPS，CPU ≤ 70% | 需要 **≈ 7 台执行节点** 在峰值时满足 5 000 QPS（5 000 / 800 ≈ 6.25） |
| **存储容量** | 任务元数据 10 TB（包括历史） | 需要 **分布式存储**（如 MySQL Cluster、TiDB、Cassandra）并做 **冷热分离**。 |

> **为什么要先做这些数字**  
> - 能帮助我们判断是 **单机** 还能 **多机**，以及每个组件的 **容量上限**。  
> - 在面试中，**量化** 往往比“能做到”更有说服力。

## 第二步：高层架构设计  

### 1. MVP（单机）结构  

```
+-------------------+
|   API Gateway    |  <-- REST / gRPC
+--------+----------+
         |
+--------v----------+
|   Task Service    | 业务层：校验、权限、持久化
+--------+----------+
         |
+--------v----------+
|   Scheduler (单机)|
|  - 解析 Cron    |
|  - 时间轮(时间槽)|
+--------+----------+
         |
+--------v----------+
|   Executor (线程池)|
|   - 执行用户代码  |
+--------+----------+
         |
+--------v----------+
|   Callback/Result |
+-------------------+
```

**优点**：实现快速，验证业务模型。  
**缺点**：单点故障、容量受限、无法满足高并发。  

### 2. 逐步演进到分布式架构  

> 下面的图是 **从左到右** 每一步加入的关键组件。  

```
+-------------------+      +-------------------+      +-------------------+
|   API Gateway    | ---> |   Auth / RBAC    | ---> |   Load Balancer   |
+-------------------+      +-------------------+      +-------------------+
          |                         |                         |
          v                         v                         v
+-------------------+      +-------------------+      +-------------------+
|   Task Service    | <--->|   Meta Store (SQL) |<--->|   Config Center   |
+-------------------+      +-------------------+      +-------------------+
          |                         ^                         |
          |                         |                         |
          v                         |                         v
+-------------------+      +-------------------+      +-------------------+
|   Scheduler      | <--->|   Distributed    |<--->|   Leader Election |
| (Stateless)      |      |   Lock Service   |      |   (Zookeeper)    |
+-------------------+      +-------------------+      +-------------------+
          |                         |                         |
          v                         v                         v
+-------------------+      +-------------------+      +-------------------+
|   Executor Pool  | <--- |   Task Queue (MQ) | -->  |   Health Checker |
+-------------------+      +-------------------+      +-------------------+
          |                         |                         |
          v                         v                         v
+-------------------+      +-------------------+      +-------------------+
|   Result Store   | <--->|   Callback Service|<--->|   Alerting System|
+-------------------+      +-------------------+      +-------------------+
```

#### 关键组件解释  

| 组件 | 作用 | 选型建议 | 为什么要它 |
|------|------|----------|------------|
| **API Gateway** | 统一入口，做流量控制、鉴权、限流 | Kong / APISIX | 防止业务服务直接暴露，易于灰度发布 |
| **Auth / RBAC** | 多租户权限校验 | Casbin + JWT | 细粒度权限，租户隔离 |
| **Task Service** | 业务 API（创建/查询/删除） | Spring Boot / Go‑Gin | 保持业务逻辑与调度解耦 |
| **Meta Store** | 持久化任务元数据、调度状态 | MySQL（Aurora）/ TiDB（水平扩展） | 关系型存储提供强一致性事务，适合元数据 |
| **Config Center** | 动态参数（如重试次数、调度粒度） | Nacos / Apollo | 不重启服务即可调参 |
| **Leader Election** | 选出唯一的调度 Master，负责任务分配 | Zookeeper / etcd | 防止多节点同时调度同一任务（避免重复执行） |
| **Distributed Lock** | 对每个任务加锁，确保同一时间只能被一台 Executor 抢到 | RedLock（Redis）或 Zookeeper 锁 | 再次保障 **幂等性** |
| **Scheduler (Stateless)** | 读取时间轮/时间槽，计算“该执行哪些任务”，写入 **Task Queue** | Java Timer Wheel / Hashed Wheel Timer | 只负责把任务放进队列，保持无状态，易水平扩展 |
| **Task Queue (MQ)** | 解耦调度与执行，提供可靠投递 | Kafka（高吞吐）或 Pulsar | 支持 **至少一次** 投递 + 消费位点保存 |
| **Executor Pool** | 真正运行用户业务代码（Java、Shell、Docker） | 线程池 + Docker / Kubernetes Job | 资源弹性，可对不同语言/环境做隔离 |
| **Health Checker** | 心跳监测执行节点状态，发现故障后触发 **Failover** | Consul / 自研心跳 | 实时感知节点存活，配合 Leader 做调度恢复 |
| **Result Store** | 保存执行日志、返回值、状态 | Elasticsearch + S3（冷热分离） | 支持搜索、归档，满足 10 TB 存储需求 |
| **Callback Service** | 统一处理用户回调（HTTP、MQ） | Async HTTP client + Retry | 统一重试、幂等、降级 |
| **Alerting System** | 监控告警（Prometheus + Alertmanager） | 邮件/Slack/Webhook | 及时发现异常任务或系统故障 |

> **为什么一步步加**  
> - **单点故障**：如果直接一口气把所有功能都写在同一个服务里，任何故障都会导致整体不可用。  
> - **扩展性**：把调度、执行、存储解耦后，每一层都可以独立水平扩容。  
> - **运维**：监控、配置中心、日志系统都是后期必须投入的运维设施，提前预留接口可以降低后期改动成本。

## 第三步：数据库设计  

### 1. 任务元数据表（Task）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 全局唯一任务 ID（使用 Snowflake） |
| `tenant_id` | BIGINT | 所属租户 |
| `name` | VARCHAR(128) | 业务名称 |
| `type` | ENUM('ONCE','CRON','FIXED_DELAY','FIXED_RATE') | 调度方式 |
| `cron_expr` | VARCHAR(64) | Cron 表达式（仅 CRON） |
| `trigger_time` | TIMESTAMP | 首次触发时间（UTC） |
| `interval_ms` | BIGINT | 固定间隔（毫秒），仅 FIXED_* |
| `payload` | JSON | 用户业务参数（如 HTTP POST body） |
| `callback_url` | VARCHAR(256) | 任务完成回调地址 |
| `retry_policy` | JSON | 包含 `max_retries`, `interval_ms`, `backoff_factor` |
| `status` | ENUM('ACTIVE','PAUSED','DELETED') | 任务生命周期 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 最近一次修改 |
| `version` | BIGINT | 乐观锁字段（防并发更新） |

**设计要点**  
- **全局唯一 ID**：使用 Snowflake 或 MySQL `auto_increment` + 租户前缀，确保跨机房唯一。  
- **乐观锁**：更新任务时防止并发写冲突。  
- **JSON**：灵活存储租户自定义字段，避免频繁 schema 变更。  

### 2. 任务调度记录表（Task_Instance）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `instance_id` | BIGINT PK | 每一次调度的唯一标识 |
| `task_id` | BIGINT FK | 对应的任务 |
| `scheduled_at` | TIMESTAMP | 计划执行时间 |
| `started_at` | TIMESTAMP | 实际开始时间 |
| `finished_at` | TIMESTAMP | 完成时间 |
| `node_id` | VARCHAR(64) | 执行节点标识 |
| `status` | ENUM('PENDING','RUNNING','SUCCESS','FAIL','TIMEOUT') | 当前状态 |
| `retry_count` | INT | 已经重试的次数 |
| `result` | JSON | 成功返回或异常信息 |
| `log_path` | VARCHAR(256) | 日志文件在对象存储的路径 |
| `created_at` | TIMESTAMP | 记录创建时间 |

**为什么要独立的实例表**  
- **审计**：满足 **10 TB** 的历史记录需求。  
- **监控**：可直接做聚合统计（成功率、平均时延）。  
- **幂等**：通过 `instance_id` + `node_id` 的唯一约束，防止同一实例被多节点重复执行。  

### 3. 分区 & 冷热策略  

- **分区键**：`scheduled_at`（按天或小时） + `tenant_id`（二级分区）  
  - 有助于 **定时清理** 过期数据，避免全表扫描。  
- **冷热分离**：  
  - 最近 7 天的数据保存在 **主库**（快速查询）。  
  - 超过 7 天的归档至 **对象存储（S3）+ Hive/Presto**，供离线分析。  

### 4. 选型对比  

| 选型 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **MySQL Cluster / Aurora** | 事务强一致、熟悉度高 | 横向扩容受限（写热点） | 元数据（任务、实例） |
| **TiDB** | 自动水平分片、强一致 | 运维相对复杂 | 大规模租户、写入热点 |
| **Cassandra** | 写吞吐高、无单点 | 只能实现最终一致、查询不灵活 | 大规模历史日志（只读） |
| **Elasticsearch** | 近实时搜索、聚合 | 不适合事务 | 任务执行日志检索 |

> **结论**：核心元数据使用 **MySQL/Aurora**（或 TiDB），历史日志放 **Elasticsearch + OSS**。

**创建任务请求**  

```json

  "name": "每日报表生成",
  "type": "CRON",
  "cron_expr": "0 0 2 * * ?",   // 每天 02:00 UTC
  "payload": { "reportId": 12345 },
  "callback_url": "https://biz.example.com/callback",
  "retry_policy": { "max_retries": 3, "interval_ms": 60000 },

```

**成功响应**  

```json
{
  "code": 0,
  "message": "OK",
  "data": {
    "id": 9876543210123,
    "version": 1,
    "status": "ACTIVE",
    "created_at": "2026-05-26T08:12:34Z"
  }
}
```

### 3. 参数校验与幂等  

- **幂等 Token**：`POST /tasks` 支持 `Idempotency-Key` 头部，防止网络重试导致重复创建。  
- **字段校验**：Cron 表达式使用 **cron-utils**，时间戳必须是未来时间。  
- **租户校验**：所有请求必须携带 JWT，JWT 中包含 `tenant_id`，后端统一校验是否匹配。  

### 4. 错误码约定  

| Code | 含义 | 场景 |
|------|------|------|
| 0 | 成功 | — |
| 4001 | 参数非法 | Cron 语法错误、时间冲突 |
| 4002 | 租户未授权 | RBAC 检查失败 |
| 4040 | 任务不存在 | 查询或操作不存在的 ID |
| 4090 | 乐观锁冲突 | 并发更新 |
| 5000 | 系统内部错误 | DB/ MQ 异常 |

## 第五步：详细组件设计  

### 5.1 调度中心（Scheduler）  

#### 5.1.1 时间轮（Hashed Wheel Timer）  

- **原理**：把时间轴划分为若干个槽（比如 1 秒一个槽），每个槽维护一个链表，存放到期时间在该槽的任务。  
- **实现要点**：  
  - **精度**：槽大小 200 ms → 满足 **95% 任务 ±200 ms**。  
  - **滚动指针**：每 200 ms 向前移动一个槽，取出该槽的任务放入 **Task Queue**。  
  - **持久化**：启动时从 **Meta Store** 加载所有 **ACTIVE** 且 **未来** 的任务到时间轮。每次 **新增/更新** 任务时同步更新时间轮。  

#### 5.1.2 防重复执行机制  

1. **Leader Election**：使用 Zookeeper/etcd 选举唯一的调度 Master。所有调度节点只读时间轮，只有 Master 能把任务写入 **Task Queue**。  
2. **分布式锁**（RedLock）：写入 Task Queue 前，对每个 `task_id + scheduled_at` 加锁，锁成功才投递。即使出现 Master 切换，也不会导致两次投递。  
3. **幂等 Instance ID**：`instance_id = hash(task_id, scheduled_at)`，Executor 在消费时检查 `Task_Instance` 是否已存在（唯一约束），若已存在直接 **ACK** 并丢弃。  

### 5.2 任务队列（Message Queue）  

- **选型**：Kafka（分区数 ≥ 10，复制因子 3）  
- **分区策略**：**task_id % partitions**，保证同一任务的所有实例都落在同一分区，保持顺序。  
- **消费者组**：每个 **Executor** 属于同一消费组，Kafka 自动实现 **负载均衡**。  

| 场景 | 参数 |
|------|------|
| **吞吐** | 5 000 QPS → Kafka 1 分区可支撑 ~10k msg/s，保守设 20 分区 |
| **可靠性** | `acks=all`、`enable.idempotence=true`，确保 **至少一次** 投递且不重复 |
| **回溯** | 消费位点保存在 Kafka，节点重启后可从上一次位点继续消费 |  

### 5.3 执行节点（Executor）  

#### 5.3.1 运行模型  

- **容器化**：每个 Executor 运行在 **Kubernetes Pod**，Pod 内部使用 **ThreadPoolExecutor**（或 Go 协程）并发执行任务。  
- **资源隔离**：使用 **Namespace + ResourceQuota**，防止单租户任务耗尽集群资源。  
- **任务跑步环境**：  
  - **轻量脚本**：直接在进程内调用 Java / Python 脚本。  
  - **复杂业务**：启动 **Docker container**（镜像由租户提供）或 **K8s Job**。  

#### 5.3.2 幂等与重试  

- **幂等标识**：`instance_id` 作为业务方传递的 **业务幂等键**，执行前先查询 `Task_Instance` 表是否已经成功。  
- **重试策略**：读取任务的 `retry_policy`，在 Executor 捕获异常后判断是否满足 `retry_count < max_retries`，若满足则 **延迟放回** Task Queue（使用延迟队列或 Kafka 的 `delay` 插件）。  
- **超时控制**：每个任务配置 `timeout_ms`，Executor 使用 **Future.get(timeout)** 或 **Context with deadline**，超时即标记 `TIMEOUT`，并执行回调。  

#### 5.3.3 日志收集  

- **Stdout/Stderr** → **Sidecar**（Fluent Bit） → **Elasticsearch** + **S3**（冷存）  
- **结构化日志**：JSON 包含 `task_id, instance_id, level, message, timestamp`，便于后期搜索和告警。  

### 5.4 结果回调（Callback Service）  

- **异步队列**：Executor 完成后把回调信息写入 **Callback MQ**（Kafka Topic `callback-events`），专门的 **Callback Worker** 负责发送 HTTP POST。  
- **重试与降级**：使用 **指数退避** + **熔断器**（Hystrix）防止回调方不可达导致资源耗尽。  
- **幂等**：回调请求带 `X-Instance-Id`，业务方应实现 **幂等**（如 DB upsert）。  

### 5.5 监控与告警  

| 指标 | 采集方式 | 报警阈值 |
|------|----------|----------|
| 调度延迟（scheduled_at → started_at） | Prometheus exporter 在 Executor 记录 | 95% ≤ 200 ms，99% ≤ 500 ms |
| 任务成功率 | 统计 `Task_Instance.status` | 失败率 > 1% → 告警 |
| 节点心跳丢失 | Consul 健康检查 | 连续 3 次未响应 → 触发 Failover |
| MQ 堆积长度 | Kafka JMX | 队列长度 > 10 万 → 扩容 |
| API QPS | API Gateway 统计 | 超过 8 000 QPS → 自动扩容 |  

- **报警渠道**：Email、Slack、Webhook。  
- **Dashboard**：Grafana 可视化 **任务流转**、**节点状态**、**租户维度**。

## 第六步：扩展性与高可用设计  

### 6.1 防止单点故障（SPOF）  

| 组件 | 冗余方式 | 关键技术 |
|------|----------|----------|
| API Gateway | 多实例 + L4 LB | Nginx/Envoy |
| Auth / RBAC | 多实例 + Redis Session 共享 | JWT + Casbin |
| Task Service | 多实例 + MySQL 主从 / Aurora | 读写分离 |
| Meta Store | 主-从复制 + 自动故障转移 | MySQL Group Replication |
| Scheduler（Leader） | **Zookeeper/etcd** 选举，只有 Leader 写 Task Queue | 选举超时 5 s |
| Distributed Lock | 多节点 Redis 集群（RedLock） | 5 个节点跨机房 |
| Task Queue | Kafka 多副本（RF=3） | 自动分区再均衡 |
| Executor | 多实例（K8s Deployment） + HPA | CPU/QueueLength 触发扩容 |
| Callback Service | 多实例 + 重试队列 | Kafka + Worker Pool |
| Monitoring | 多副本 Prometheus + Alertmanager HA |  |  

### 6.2 自动水平扩容（Auto‑Scaling）  

1. **指标采集**：K8s HPA 根据 **CPU 使用率**、**Executor Queue Length**、**Kafka Lag** 三个指标。  
2. **扩容策略**：  
   - **CPU > 70%** 持续 2 分钟 → 增加 1~2 个 Executor Pod。  
   - **Kafka Lag > 10k** → 触发 **Kafka Cluster** 自动扩容（增加 Broker）+ **Executor** 横向扩容。  
   - **突发流量**（秒杀活动） → 预先在 **业务侧** 调用 **Scale‑Out API**（自研）把 **Executor Deployment** 的 `replicas` 设为预计峰值，K8s 在 30 s 内完成。  

3. **缩容**：当 **CPU < 30%** 且 **Lag < 1k** 持续 5 分钟，安全缩容至最小副本数（保证 **最低 3** 实例保持容错）。  

### 6.3 故障转移（Failover）  

- **调度 Master 挂掉**：Zookeeper 检测会话失效，重新选举出新 Master。新 Master 从 **Meta Store** 重新加载未完成的任务（状态为 `PENDING`）并恢复时间轮。  
- **Executor 节点失联**：Health Checker 发现心跳超时 → 将该节点上 **RUNNING** 状态的实例标记为 `UNKNOWN`，并在 **Scheduler** 中重新投递（基于 `retry_policy`）。  
- **Kafka Broker 故障**：副本自动切换，生产者/消费者透明重连。  
- **数据库主库故障**：使用 **MySQL Group Replication**，自动选举新 Primary，业务层使用 **读写分离代理**（ProxySQL）无感切换。  

### 6.4 数据一致性策略  

| 场景 | 一致性要求 | 方案 |
|------|------------|------|
| 任务元数据（创建/更新） | **强一致**（避免重复创建） | 事务提交到 MySQL，使用 **唯一索引** (`tenant_id, name`) |
| 任务调度记录 | **最终一致**（统计报表） | 先写入 **Task_Instance**（事务），后异步同步到 **ES**（Logstash） |
| 任务执行结果回调 | **最终一致**（业务方自行幂等） | MQ **至少一次** 投递 + 幂等键 |
| 任务状态（RUNNING → SUCCESS/FAIL） | **强一致**（防止重复执行） | 在 Executor 中 **先写入 DB**（行锁），成功后 ACK Kafka |  

> **为什么要区分**：强一致需要同步阻塞，会影响吞吐；而报表、日志等可以接受 **延迟**，使用 **异步复制** 能显著提升整体性能。

## 第七步：常见面试追问与回答  

### Q1. 调度一致性：如果同一个任务在同一时间被多个节点选中执行，你会如何避免重复执行？  

**回答要点**：  
1. **Leader 选举**：只有 Leader 能把任务写入 Task Queue。  
2. **分布式锁**：在写入前对 `task_id + scheduled_at` 加 RedLock，锁成功才投递。  
3. **幂等 Instance ID**：Consumer 在消费时检查 `Task_Instance` 表是否已存在对应 `instance_id`（唯一约束），若已存在直接 ACK。  
4. **最终防线**：Executor 在执行前再次查询 DB，若状态已经是 `SUCCESS`，直接返回。  

> **不这样做的后果**：会导致业务方数据重复、计费异常，甚至产生不可逆的业务错误。  

---

### Q2. 高可用实现：请详细说明调度中心（如 Master）选举、故障转移以及持久化日志（Write‑Ahead Log）的设计思路。  

**回答结构**：  
- **选举**：使用 Zookeeper/etcd 的 **ephemeral sequential node**，所有调度实例创建临时节点，最小序号的实例成为 Master。失联后节点自动删除，触发新选举。  
- **故障转移**：选举过程是 **O(1)**（只需要监听节点变化），新 Master 立即从 Meta Store 加载最近的 **时间轮状态**（通过快照 + 增量日志）。  
- **WAL**：调度中心在把任务写入 Task Queue 前，先把 **调度决定**（task_id, scheduled_at, target_queue_offset）写入 **WAL 表**（MySQL/InnoDB）并 **fsync**，保证即使 Master 崩溃也能在恢复时重放未完成的调度。  
- **持久化策略**：WAL 每 1 s **批量刷盘**，使用 **RAID 1+0** 磁盘；同时把 WAL 复制到 **备份 Kafka Topic**，提供跨机房容灾。  

---

### Q3. 扩容策略：在突发流量（如秒杀活动）下，系统如何快速感知并完成水平扩容？涉及哪些指标和自动化流程？  

**回答要点**：  
1. **监控指标**：  
   - **Kafka Lag**（未消费消息数）  
   - **Executor Queue Length**（内部缓冲）  
   - **CPU/Memory 使用率**  
2. **阈值**：如 `Kafka Lag > 50k` 或 `CPU > 80%` 持续 30 s。  
3. **自动化流程**：  
   - **Prometheus Alertmanager** 触发 **Webhook** → 调用 **K8s Horizontal Pod Autoscaler (HPA)** 或自研 **Scale‑Out Service**。  
   - **Scale‑Out Service** 同时 **预热** 新的 **Executor Pods**（拉取镜像、初始化容器），并 **动态注册** 到 **Service Discovery**。  
   - **扩容完成后**，新的 Executor 自动加入同一 **Consumer Group**，Kafka 重新平衡分区，立即开始消费。  
4. **预热策略**：在业务方预计活动前 5 min，通过 **API** 把 **预期 QPS** 发送给调度系统，系统提前 **创建** 额外节点，保证活动开始时已有可用资源。  

---

### Q4. 任务执行时如果出现“超时”，系统怎么处理？  

**回答要点**：  
- **Executor** 使用 **Context with deadline**（Go）或 **Future.get(timeout)**（Java）控制执行时间。  
- 超时后立即标记 `Task_Instance.status = TIMEOUT`，记录 `duration = timeout_ms`。  
- **回调**：仍然触发 **Callback Service**，携带 `status=TIMEOUT`，业务方自行决定是否补偿。  
- **重试**：若 `retry_policy.max_retries` > 已重试次数，任务会被重新投递到 **Delay Queue**（Kafka 延迟插件），后续再次调度。  

---

### Q5. 多租户数据隔离如何实现？  

**回答要点**：  
- **数据库层**：所有业务表都有 `tenant_id` 列，且对 `tenant_id` 建立 **二级索引**。  
- **查询层**：统一在 DAO 层加入 `WHERE tenant_id = :currentTenant`，防止泄漏。  
- **权限层**：JWT 中携带 `tenant_id` 与 `role`，Casbin 策略文件基于 `tenant_id` 动态生成。  
- **网络层**：不同租户可以走不同 **Ingress** 域名，做 **流量隔离**（可选）。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 困难点 | 思考路径 |
|------|--------|----------|
| **任务去重与幂等** | 调度中心可能出现 “脑裂” 导致同一任务被多个 Master 投递；Executor 可能因网络重试重复消费。 | 1）采用 **Zookeeper/etcd Leader Election**，只允许唯一 Master；2）在投递前使用 **分布式锁**（RedLock）确保唯一写入；3）在 **Task_Instance** 表建立 `UNIQUE(task_id, scheduled_at)` 约束，Executor 再次检查；4）业务方提供 **幂等键**（instance_id）做二次防护。 |
| **高并发调度的时间轮实现** | 需要在毫秒级精度下处理 5 000 QPS，同时保证持久化不成为瓶颈。 | 1）把 **时间轮** 作为 **内存结构**，只在系统启动/任务变更时从 DB 加载；2）调度线程每 200 ms 轮询一次，把到期任务批量写入 **Kafka**；3）写入前通过 **WAL** 先落盘，避免因进程 crash 丢失任务；4）使用 **批量写**（批量 Insert）降低 DB IO。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **把调度、执行、存储全部塞进同一个服务**  
   - 结果：单点故障、扩展受限、难以做灰度发布。  
   - 建议：**明确职责边界**，调度只负责把任务放入队列，执行只负责消费并跑业务。  

2. **忽视任务的幂等性**  
   - 只靠调度中心选举，忘记网络重试、消费者重复消费的情况。  
   - 建议：**在业务层面也实现幂等**（如 DB upsert、唯一键），并在系统层加入唯一约束与分布式锁。
