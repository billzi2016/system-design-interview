# 第 92 天：设计 用户行为埋点分析系统

> 生成日期：2026-02-23

---

## 题目背景  
用户行为埋点分析系统用于在移动端、Web 端等多种客户端实时收集用户的点击、浏览、转化等行为数据，并提供离线/实时的数据查询、漏斗分析、用户画像等业务洞察，帮助产品、运营、营销等团队做数据驱动决策。

## 面试场景设定  
> **面试官**：  
> “我们现在要设计一套支持海量用户行为埋点的分析平台。请从零开始，先说出整体的技术方案思路，然后重点说明数据采集、传输、存储以及查询的关键设计点。我们关注的是系统的可扩展性、低延迟以及成本控制。”

## 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| 1 | **实时采集**：客户端 SDK 能在毫秒级别将埋点事件上报至后端，支持自定义属性、事件序列号、时间戳。 |
| 2 | **离线批处理**：对历史行为数据进行离线聚合（如日活、周活、漏斗、留存），生成报表并支持定时调度。 |
| 3 | **实时查询**：提供近实时（≤ 5 s）查询能力，支持按事件、属性、时间范围的过滤，返回聚合结果（PV、UV、转化率等）。 |
| 4 | **用户画像 & 分群**：基于行为属性（如最近 N 天活跃、购买频次）生成标签，支持多维度分群查询。 |
| 5 | **埋点管理平台**：提供 UI，供产品团队定义/修改埋点事件、属性、采样率，实时下发至客户端 SDK。 |
| 6 | **数据安全与合规**：对敏感字段（如手机号、ID）进行脱敏或加密，满足 GDPR/CCPA 等合规要求。 |

## 非功能性需求  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU（活跃用户数）** | 2 × 10⁶ | 高峰期每日活跃用户约 200 万 |
| **QPS（上报请求）** | 10 000 req/s | 平均每用户 5 条事件/分钟，峰值约 10k QPS |
| **单条事件延迟** | ≤ 5 s（实时查询）<br>≤ 200 ms（上报成功回执） | 实时分析需要低延迟，采集回执要求快速 |
| **可用性** | ≥ 99.9%（月度） | 业务对数据完整性和查询可用性要求高 |
| **存储量** | 约 5 PB/年 | 假设每条事件 300 B，日均 1 TB，全年约 365 TB，压缩后约 5 PB（包括备份、索引） |
| **成本** | 控制在 30 万/月以内 | 包含云资源、网络、存储、监控等费用 |

## 系统边界  

**本题范围内**（需要设计）

- 查询层（实时 OLAP、离线报表、用户画像查询）。

- 监控报警、容量规划、灾备切换。

**本题范围外**（不必详细设计）  
- 客户端 UI/UX（仅说明 SDK 接口即可）。  
- 第三方广告、推荐系统的深度集成。

- 法律合规审计流程（仅说明需要脱敏/加密）。

3. **扩容与容错**：面对突发流量（如营销活动导致 QPS 突升至 100 k），系统的横向扩展和故障恢复机制如何设计？

# 题解

## 解题思路总览  

1. **从需求出发**：先把功能性、非功能性需求全部列出来，找出关键指标（如 QPS、延迟、存储规模）。  
2. **最小可用系统（MVP）**：只保留最核心的「上报‑写入‑查询」三环路，使用最简单的组件快速验证可行性。  
3. **逐步演进**：在 MVP 基础上，按**可用性 → 扩展性 → 成本 → 安全**的顺序，加入负载均衡、消息队列、实时流处理、离线批处理、冷热分层存储、元数据中心等。  
4. **每一次加东西，都要问**：  
   - 为什么要加？解决了什么痛点（如热点、容错、压缩）？  
   - 不加会出现什么问题？  
   - 选的技术有什么 trade‑off？  

下面按章节把整个设计过程一步一步展开，**每一步都解释“为什么这么做”。**

## 第一步：理解需求与规模估算  

| 需求编号 | 核心点 | 对系统的直接影响 |
|----------|--------|------------------|
| 1 | 实时采集（毫秒级上报） | 需要低延迟的 API、快速回执、可靠的网络协议 |
| 2 | 离线批处理 | 必须持久化原始事件、提供大规模离线计算框架 |
| 3 | 实时查询（≤5 s） | 需要近实时 OLAP 引擎、冷热分层存储 |
| 4 | 用户画像 & 分群 | 需要聚合后的维度表、标签计算作业 |
| 5 | 埋点管理平台 | 需要元数据服务、配置下发、版本控制 |
| 6 | 数据安全与合规 | 脱敏/加密、访问控制、审计日志 |

### 关键业务指标（KPI）

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **DAU** | 2 M | 高峰期每日活跃用户 |
| **上报 QPS** | 10 k req/s | 5 条事件/分钟 × 2 M ≈ 166 k/min ≈ 2.8 k/s，留有峰值因子，取 10 k |
| **单条事件大小** | ~300 B | 包含事件名、属性、时间戳、用户 ID 等 |
| **每日写入量** | 1 TB | 10 k req/s × 300 B × 86 400 s ≈ 260 GB（压缩前），实际业务常有压缩、采样，取 1 TB 为安全上限 |
| **年存储** | 5 PB | 1 TB × 365 ≈ 365 TB，考虑多副本、索引、冷热分层后约 5 PB |
| **实时查询延迟** | ≤5 s | 必须在流处理 → 实时存储 → 查询层之间保持 <5 s 的窗口 |
| **上报回执** | ≤200 ms | 客户端感知上报成功，需在 API 层快速返回 |
| **可用性** | 99.9% | 月均不可用时间 < 43 min |
| **成本** | ≤30 万/月 | 需要在云资源、存储、网络之间做好性价比权衡 |

### 初步容量规划（以 1 TB/日 为例）

| 组件 | 估算资源 | 备注 |
|------|----------|------|
| **API 网关 + LB** | 2 台（CPU 8 核, 32 GB） | 负载均衡、限流、TLS 终止 |
| **消息队列（Kafka）** | 3 × 12 分区（每分区 1 GB） | 保障 10 k QPS、容错 |
| **实时流处理（Flink / Spark Structured Streaming）** | 5 台 (CPU 16, 64 GB) | 1.5×峰值并行度 |
| **实时存储（ClickHouse / Druid）** | 6 台 (CPU 24, 128 GB, SSD) | 支持 5 s 延迟查询 |
| **离线存储（HDFS / Iceberg on S3）** | 30 TB (原始) + 100 TB (压缩后) | 复制因子 2 |
| **离线计算（EMR / Dataproc）** | 按需弹性伸缩 | 周期性聚合作业 |
| **元数据中心（MySQL / PostgreSQL）** | 2 台 HA | 埋点配置、用户标签元信息 |

## 第二步：高层架构设计  

下面先画出 **最小可用系统（MVP）**，再逐层演进到完整的高可用分布式架构。  

### 2.1 MVP（单机版）  

```
[客户端 SDK] → HTTP POST /event → [单体 API Server] → 本地文件/SQLite → 简单查询 API
```

- **优点**：实现快、验证业务模型。  
- **缺点**：单点故障、无法支撑 10k QPS、查询慢、没有安全/脱敏。  

> **新手常犯错**：直接把所有功能塞进单体，忽视了伸缩性和可靠性。  

### 2.2 生产级高层架构  

```
┌───────────────────────────────────────────────────────┐
│                     客户端 SDK                        │
│  (HTTP/HTTPS + gRPC, 支持压缩、采样、重试、限流)      │
└───────▲───────────────────────▲───────────────────────┘
        │                           │
        │   1. 上报请求（JSON/Protobuf）   │
        │                           │
┌───────▼───────────────────────▼───────────────────────┐
│            API 网关 + 负载均衡 (L7 LB)                │
│  - TLS 终止、流量治理、灰度发布、IP 限流             │
│  - 鉴权/签名（HMAC）                                 │
└───────▲───────────────────────▲───────────────────────┘
        │                           │
        │   2. 写入消息队列 (Kafka)   │
        │   - 分区键：user_id%N 或 hash(event)      │
        │   - 可靠投递 (acks=all)                       │
┌───────▼───────────────────────▼───────────────────────┐
│            实时流处理层 (Flink / Spark Structured)   │
│  - 数据清洗、字段校验、脱敏/加密、时间窗口对齐       │
│  - 侧写到实时存储（ClickHouse） + 侧写到离线湖      │
│  - 生成标签、分群（实时/近实时）                    │
└───────▲───────────────────────▲───────────────────────┘
        │                           │
        │   3. 实时存储 (ClickHouse / Druid)            │
        │   - 预聚合物化视图、分区、索引                  │
        │   - 支持 SQL/REST 查询，延迟 ≤5 s             │
┌───────▼───────────────────────▼───────────────────────┐
│          离线存储 (Iceberg on S3 / HDFS)               │
│  - 原始事件湖、压缩 Parquet、元数据 (Hive Metastore) │
│  - 按天/小时分区，支持快照回滚                     │
└───────▲───────────────────────▲───────────────────────┘
        │                           │
        │   4. 离线批处理 (EMR / Dataproc)               │
        │   - Spark 作业、每日/周聚合、漏斗、留存      │
        │   - 结果写回 ClickHouse（离线视图）或 Hive   │
┌───────▼───────────────────────▼───────────────────────┐
│            查询层 (统一查询 API)                     │
│  - 实时查询 → ClickHouse                              │
│  - 离线报表 → Hive/Presto                               │
│  - 用户画像/分群 → ElasticSearch（倒排）或 ClickHouse│
└───────▲───────────────────────▲───────────────────────┘
        │                           │
        │   5. 元数据中心 (MySQL/PG)                      │
        │   - 埋点配置、SDK 版本、标签字典               │
        │   - 配置下发 via CDN / gRPC                     │
└───────┴───────────────────────┴───────────────────────┘
```

**关键设计点**  

| 位置 | 为什么要这样做 | 不这样做会出现的问题 |
|------|----------------|-------------------|
| **API 网关 + LB** | 统一入口、限流、灰度、TLS，解耦后端升级 | 单点入口压力爆炸、无法做流量治理、证书更新困难 |
| **Kafka** | 高吞吐、持久化、天然的背压、支持多消费者 | 直接写库会导致写入突发、数据丢失、消费者耦合 |
| **Flink** | 实时脱敏、窗口聚合、Exactly‑once 语义 | 业务侧在写库时无法保证数据一致性，热点难均衡 |
| **ClickHouse** | 列式存储 + 索引，查询聚合快（毫秒级），支持物化视图 | 使用行式 KV 或关系库查询聚合慢，成本高 |
| **Iceberg + Parquet** | 高压缩、快照、支持时间旅行，成本低 | 直接存 MySQL/Redis 费用爆炸、不可扩展 |
| **离线 Spark** | 大规模离线批处理，能做全量/增量聚合 | 实时层只做近实时，离线层缺失全量校验、历史回溯 |
| **统一查询 API** | 对外只暴露统一接口，内部路由到实时/离线，统一鉴权 | 前端需要分别调用不同系统，维护成本大 |
| **元数据中心** | 配置中心化，支持灰度发布、版本回滚 | SDK 难以统一管理，埋点改动需要全量更新客户端 |

## 第三步：数据库设计  

### 3.1 事件原始表（写入 Kafka → Iceberg）  

| 列名 | 类型 | 说明 |
|------|------|------|
| `event_id` | STRING | UUID，唯一标识 |
| `event_name` | STRING | 业务定义的事件，如 `page_view` |
| `user_id` | STRING | 脱敏/加密后用户唯一标识 |
| `device_id` | STRING | 设备唯一标识 |
| `event_timestamp` | TIMESTAMP(6) | 客户端产生时间（ms） |
| `ingest_timestamp` | TIMESTAMP(6) | 后端接收时间 |
| `session_id` | STRING | 会话标识（可选） |
| `properties` | MAP<STRING,STRING> | 动态属性（键值对），JSON 序列化后存 MAP |
| `sampling_rate` | DOUBLE | 当前事件采样率（0~1） |
| `app_version` | STRING | 客户端版本 |
| `region` | STRING | 国家/地区（IP 解析） |
| `is_test` | BOOLEAN | 是否为测试数据（灰度） |

- **分区**：`ingest_date = DATE(ingest_timestamp)`，**子分区** `event_name`（热门事件再细分）。  
- **压缩**：Parquet + Snappy/ZSTD，Iceberg 自动管理快照。  

### 3.2 实时聚合表（ClickHouse）  

1. **原始事件表（事实表）**（用于回溯）  

```sql
CREATE TABLE events
(
    event_date   Date,
    event_time   DateTime,
    event_name   LowCardinality(String),
    user_id      String,
    device_id    String,
    properties   Nested(
        key   String,
        value String
    ),
    sampling_rate Float32
) ENGINE = MergeTree()
PARTITION BY event_date
ORDER BY (event_name, event_date, user_id);
```

2. **物化视图：PV、UV 按日/小时**  

```sql
CREATE MATERIALIZED VIEW mv_daily_pv_uv
ENGINE = SummingMergeTree()
PARTITION BY event_date
ORDER BY (event_name, event_date)
AS SELECT
    event_date,
    event_name,
    count()               AS pv,
    uniqExact(user_id)    AS uv
FROM events
GROUP BY event_date, event_name;
```

- **为什么用物化视图**：实时查询只扫描聚合表，毫秒级返回，避免全表扫描。  

### 3.3 用户画像标签表  

| 列名 | 类型 | 说明 |
|------|------|------|
| `user_id` | STRING | 脱敏后唯一标识 |
| `tag_key` | STRING | 如 `active_last_7d` |
| `tag_value` | STRING/INT | 标签具体值 |
| `update_ts` | TIMESTAMP | 最近更新时间 |

- **存储**：ClickHouse（宽表）或 Elasticsearch（倒排+聚合），视查询频率而定。  

### 3.4 元数据表（MySQL）  

| 表名 | 关键字段 | 说明 |
|------|----------|------|
| `event_definitions` | `event_name`, `description`, `property_schema` | 产品侧定义的事件元信息 |
| `sdk_versions` | `version`, `download_url`, `checksum` | SDK 包管理 |
| `sampling_rules` | `event_name`, `rate`, `effective_from` | 动态采样配置 |
| `tag_definitions` | `tag_key`, `calc_expression`, `update_cycle` | 画像标签计算规则 |

## 第四步：核心 API 设计  

### 4.1 上报 API（POST /api/v1/events）  

```http
POST /api/v1/events HTTP/1.1
Host: analytics.example.com
Content-Type: application/json
Accept: application/json
Authorization: HMAC sha256 ...   # 防篡改、鉴权

{
  "events": [
    {
      "event_name": "page_view",
      "timestamp": 1727359212345,
      "properties": {
        "page": "/home",
        "referrer": "google"
      },
      "user_id": "MD5(13800138000)",   # 脱敏后
      "device_id": "A1B2C3D4",
      "session_id": "S12345",
      "sampling_rate": 0.5,
      "app_version": "2.3.1"
    },
    ...
  ]
}
```

- **压缩**：客户端可选 `gzip`、`snappy`（Content-Encoding），网关自动解压。  
- **批量上报**：一次最多 50 条事件，降低请求次数。  
- **返回**：`200 OK` + `{ "code":0, "msg":"ok", "failed_ids": [] }`，在 200 ms 内返回。  

### 4.2 查询 API（GET /api/v1/analytics)  

| 参数 | 必填 | 说明 | 示例 |
|------|------|------|------|
| `event_name` | 是 | 过滤的事件 | `page_view` |
| `start` | 是 | 起始时间（ISO8601） | `2024-09-01T00:00:00Z` |
| `end` | 是 | 结束时间 | `2024-09-01T23:59:59Z` |
| `group_by` | 否 | 维度字段（逗号分隔） | `page,region` |
| `metric` | 否 | 需要的聚合指标（pv,uv,avg_duration） | `pv,uv` |
| `filter` | 否 | 属性过滤表达式（JSON） | `{"page":"home"}` |
| `type` | 否 | `realtime`|`offline`（默认 realtime） | `realtime` |

- **实现**：后端根据 `type` 路由到 ClickHouse（实时）或 Presto/Hive（离线）。  

### 4.3 元数据 API（GET /api/v1/metadata/events）  

返回所有已注册的埋点事件及属性 schema，供 SDK 动态更新。

## 第五步：详细组件设计  

### 5.1 客户端 SDK  

- **语言**：Android（Java/Kotlin）、iOS（Swift/ObjC）、Web（JS）  
- **核心功能**：  
  1. **埋点收集**：`track(eventName, properties)`  
  2. **本地缓存 & 批量发送**：使用环形缓冲区，网络不可达时本地磁盘持久化（SQLite），后台任务每 30 s 或达到阈值自动上报。  
  3. **压缩 & 加密**：对整个 payload 使用 `gzip` + `AES-128-GCM`（对称密钥由服务端下发并定期轮换）。  
  4. **采样**：SDK 在本地读取 `sampling_rules`（从 CDN 拉取），对不需要的高频事件进行采样，减轻后端压力。  
  5. **重试 & 幂等**：每条事件生成 `event_id`（UUID），上报成功后本地标记，失败则指数退避重试。  

### 5.2 API 网关 & 负载均衡  

- **选型**：AWS API Gateway + CloudFront（或自建 Kong/Nginx + LVS）  
- **功能**：  
  - **TLS 终止**：统一使用 HTTPS，降低后端证书管理成本。  
  - **流量治理**：IP 限流、请求速率（每 IP 10 rps）防止 DoS。  
  - **灰度发布**：通过 `X-Canary` 头部路由到新版本服务。  
  - **日志审计**：Access Log 送到 ELK，供安全审计。  

### 5.3 消息队列（Kafka）  

| 参数 | 设计 |
|------|------|
| **Topic** | `event_raw`（主流） |
| **分区数** | 24 分区（每台机器 4 核，考虑 10 k QPS，预留 2×峰值） |
| **键** | `user_id`（hash）或 `event_name`（热点时改用 `event_name+random`） |
| **副本数** | 3（保证容错） |
| **压缩** | `snappy`（Kafka 内部压缩） |
| **消费者组** | `realtime_processor`, `offline_ingester` |

- **热点处理**：如果 `page_view` 占 30% 流量，单一分区会成为热点。**解决方案**：  
  1. **复合键**：`event_name + hash(user_id)`，把同一事件的不同用户分散到多个分区。  
  2. **使用 Kafka 的 **`partitioner`** 自定义实现，使热点事件随机分配。  

### 5.4 实时流处理（Flink）  

- **部署**：Kubernetes 上的 Flink JobManager + TaskManager（按需弹性伸缩）。  
- **关键算子**：  
  1. **Source**：`FlinkKafkaConsumer`，开启 **Exactly‑once**（使用事务式写入 Kafka + Checkpoint）。  
  2. **Map**：JSON → POJO，字段校验、脱敏（MD5、AES），异常事件写入 **Dead‑Letter Queue**（另建 Kafka Topic）。  
  3. **Window**：**Tumbling Event‑Time Window**（1 min）进行 `pv/uv` 计数；**Session Window** 用于会话分析。  
  4. **Sink**：  
     - **ClickHouseSink**（使用 JDBC 批量写入） → 实时聚合表。  
     - **IcebergSink**（Flink‑Iceberg Connector） → 原始 Parquet 文件。  
  5. **Side Output**：实时标签计算结果写入 **Redis**（缓存）或 **ElasticSearch**（倒排）供画像查询。  

- **容错**：Checkpoint 间隔 5 s，状态后端使用 **RocksDB**（本地磁盘），Checkpoint 存到 **S3**。  

### 5.5 实时存储（ClickHouse）  

- **集群模式**：`Distributed` + `ReplicatedMergeTree`（多副本），每台节点 2× SSD（NVMe）+ 1× HDD（备份）。  
- **分区 & 索引**：  
  - **分区**：`event_date`（天） + `event_name`（热点分区）  
  - **索引**：`primary key (event_name, event_date, user_id)`，**Skipping Index**（`minmax`）加速时间范围过滤。  
- **物化视图**：预先定义好常用聚合（PV/UV/转化率），实时写入后即能查询。  

### 5.6 离线存储（Iceberg on S3）  

- **表结构**：同上 **原始事件表**，但使用 **Iceberg** 的 **Snapshot** 功能实现 **时间旅行**（回滚错误写入）。  
- **分区**：`ingest_date`、`event_name`（可选二级分区）。  
- **压缩**：Parquet + ZSTD（压缩率 3–5×），成本大幅降低。  

### 5.7 离线批处理（Spark）  

- **调度**：Airflow / AWS Step Functions，每日 02:00（全量）+ 每小时增量。  
- **作业**：  
  1. **读取 Iceberg 表** → **过滤** → **聚合**（日活、周活、漏斗、留存）。  
  2. **写回 ClickHouse**（离线聚合表）或 **Hive**（供 Presto 查询）。  
  3. **生成画像标签**（基于最近 N 天行为），写入 ClickHouse `user_tags` 表。  
- **增量策略**：使用 **Iceberg 的 `metadata`** 判断新增文件，仅处理新增分区，降低作业时长。  

### 5.8 查询层  

- **统一查询网关**（Spring Boot / Go）提供 `/analytics`、`/tags`、`/metadata` 三类接口。  
- **路由策略**：  
  - **实时** → ClickHouse（SQL）  
  - **离线** → Presto/Trino（SQL）读取 Hive 或 Iceberg。  
- **缓存**：热点聚合结果（如今日 PV）写入 **Redis**（TTL 5 min），降低 ClickHouse 负载。  
- **授权**：基于 **OAuth2** + **RBAC**，不同业务团队只能查询自己命名空间的数据。  

### 5.9 元数据中心  

- **数据库**：MySQL 主从（GTID），提供强一致性。  
- **服务**：`MetadataService`（RESTful）对外提供：  
  - 埋点定义查询  
  - SDK 版本检查  
  - 采样规则下发（带时间戳的配置）  
- **同步**：配置变更写入 **Kafka** `metadata_change` 主题，SDK 客户端通过 **Long-Polling** 或 **WebSocket** 实时获取最新配置。  

### 5.10 监控、报警、容量规划  

| 维度 | 关键指标 | 监控工具 |
|------|----------|----------|
| **采集层** | 请求 QPS、错误率、上报延迟 | Prometheus + Grafana |
| **Kafka** | Lag、吞吐、ISR、磁盘使用 | Burrow + Kafka Manager |
| **Flink** | Checkpoint 成功率、背压、处理延迟 | Flink Dashboard + Prometheus |
| **ClickHouse** | 查询 QPS、慢查询、磁盘 I/O | ClickHouse System Tables + Grafana |
| **存储** | S3 使用量、Iceberg Snapshot 数量 | CloudWatch / Datadog |
| **安全** | 敏感字段访问日志、异常加解密错误 | ELK + WAF |

- **报警阈值**：如 `Kafka Lag > 5 min`、`Flink Checkpoint 失败率 > 1%`、`ClickHouse CPU > 80% 持续 5min`。

## 第六步：扩展性与高可用设计  

### 6.1 横向扩容  

| 组件 | 扩容方式 | 关键点 |
|------|----------|--------|
| **API 网关** | 增加节点 + 自动弹性伸缩组 (ASG) | 统一健康检查、统一 DNS（Route53） |
| **Kafka** | 增加 **Broker** 并 **Rebalance** 分区 | 通过 `kafka-reassign-partitions.sh`，保持副本因子 3 |
| **Flink** | 动态增加 **TaskManager** 实例，调度器自动分配算子 | 使用 `slot` 机制，确保每个算子拥有足够的 CPU/内存 |
| **ClickHouse** | 添加 **Shard**，使用 **Distributed** 表自动路由 | 通过 **ZooKeeper** 管理集群元数据 |
| **Iceberg** | S3 本身弹性，增加 **Catalog** 实例（Hive Metastore） | 只要元数据服务水平扩容即可 |

### 6.2 容错与灾备  

| 场景 | 设计 |
|------|------|
| **单节点故障** | 所有层使用 **多副本**（Kafka ISR=3，ClickHouse ReplicatedMergeTree，MySQL 主从） |
| **机房故障** | 多可用区部署：API GW、Kafka、Flink、ClickHouse 各跨 2 AZ；使用 **跨 AZ VPC Peering** 保持网络连通 |
| **数据恢复** | **备份策略**：ClickHouse 每日快照 → S3；Iceberg 自动快照 + 30 天保留；MySQL binlog + PITR |
| **业务回滚** | 元数据中心提供 **版本化** 配置，SDK 支持回滚到旧版本；Kafka 支持 **时间旅行消费**（从特定 offset 重放） |
| **网络分区** | 使用 **Circuit Breaker**（Hystrix）在 API 层快速降级，返回 “数据稍后可查询” 给前端，避免全链路阻塞 |

### 6.3 成本控制  

- **冷热分层**：最近 7 天数据放在 **ClickHouse**（SSD），超过 30 天归档至 **Iceberg**（S3）并开启 **Glacier** 归档。  
- **压缩与列式**：ClickHouse 列式压缩率约 2–3×，Iceberg Parquet + ZSTD 约 5×，显著降低存储费用。  
- **弹性伸缩**：使用 **K8s HPA** 或 **云原生 Auto Scaling**，在业务低谷（夜间）缩容 50%。  
- **预估费用**（以 AWS 为例）：  
  - EC2/EMR 计算：约 12 万/月  
  - S3（标准 + Glacier）≈ 8 万/月  
  - ClickHouse（自建）≈ 5 万/月（实例+磁盘）  
  - 其他（Kafka、监控、网络）≈ 3 万/月  
  - **合计** ≈ 28 万/月 < 30 万/月

## 第七步：常见面试追问与回答  

### Q1️⃣ 数据倾斜与热点  
> **如果“页面打开”事件占 30% 流量，流处理层会不会卡住？如何避免？**

**回答思路**  

1. **上报阶段**：在 SDK 端做**采样**（如 10%），但因为页面打开是关键指标，采样率不能太低。  
2. **Kafka 分区**：  
   - **复合键**：`event_name + hash(user_id)`，把同一种事件的不同用户分散到多个分区。  
   - **自定义分区器**：对热点事件使用 **随机数** 再加 `event_name`，确保分区均匀。  
3. **Flink 并行度**：对热点事件使用 **KeyBy** 后的 **分区并行度** 设置为 **N > 1**（如 8），让同一事件的不同 key 在不同 TaskManager 处理。  
4. **负载均衡**：在 ClickHouse 中对热点 `event_name` 再 **子分区**（如 `event_name + hour`），防止单表写入瓶颈。  
5. **监控**：设置 **Kafka Lag**、**Flink Backpressure** 报警，出现倾斜时快速 **扩容** 或 **调优** 分区数。

### Q2️⃣ 存储选型权衡  
> **实时查询需要 5 s 延迟、离线聚合需要高压缩率，怎么在 ClickHouse、HBase、Iceberg 中选？**

**回答要点**  

| 系统 | 读/写延迟 | 存储成本 | 查询能力 | 适用场景 |
|------|------------|----------|----------|----------|
| **ClickHouse** | 写 10–50 ms，查询 ms‑level | 较高（SSD） | OLAP 聚合快，支持物化视图 | **实时** PV/UV、近实时仪表盘 |
| **HBase** | 写 5–20 ms，查询秒级 | 低‑中（HDD） | 行键查询快，范围扫描慢 | 实时用户画像的键值查询 |
| **Iceberg (Parquet on S3)** | 写批量（分钟） | **最低**（对象存储） | 大规模离线扫描，支持时间旅行 | **离线** 漏斗、留存、全量回溯 |

**方案**：  
- **实时层**使用 **ClickHouse**（满足 ≤5 s），因为它的列式压缩、向量化执行能在 SSD 上实现毫秒级聚合。  
- **离线层**使用 **Iceberg + Parquet**（成本最低），配合 **Spark** 做每日/每周全量聚合。  
- **HBase** 可作为 **辅助**，存储 **用户画像标签**（快速点查询），但不做聚合。  

### Q3️⃣ 突发流量（QPS 100 k）如何应对？  

**思路**  

1. **API 网关弹性伸缩**：使用云原生 **Auto Scaling Group**，根据 CPU/网络 QPS 自动扩容至 10×。  
2. **限流 & 流量削峰**：在网关层设置 **令牌桶**，对单 IP、单 AppID 限流；超出部分进入 **Message Queue Buffer**（Kafka）或 **S3 EventBridge** 暂存。  
3. **Kafka 扩容**：提前准备 **分区 200+**，突发时 **快速扩容**（Kafka 0.11+ 支持在线添加分区）。  
4. **Flink 弹性**：开启 **dynamic scaling**（Kubernetes HPA），任务管理器可在几分钟内新增至 50+ 实例。  
5. **后端缓存**：热点聚合结果（如当天 PV）提前写入 **Redis**，在突发期间直接返回，减少 ClickHouse 负载。  
6. **降级策略**：若仍出现瓶颈，API 返回 **202 Accepted**，并在后台完成写入，前端提示“数据稍后可查询”。

## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **实时层 vs 离线层的存储选型** | 必须兼顾 **低延迟**（≤5 s）和 **高压缩**（成本 ≤30 万/月），而常见的 OLAP 数据库（ClickHouse）在成本上不如对象存储。 | - 先列出需求：实时查询、聚合、容量、成本。<br>- 评估 ClickHouse 的查询速度 vs 存储成本（SSD 高），决定仅保留最近 7‑30 天数据。<br>- 将历史数据迁移至 Iceberg（Parquet+ZSTD）实现 5× 压缩，满足成本目标。<br>- 通过 **冷热分层** 结合 **数据生命周期管理**（TTL）实现两者互补。 |
| **热点事件的分区与并行** | “页面打开”类事件占比 30% 会导致 Kafka 单分区、Flink KeyBy 甚至 ClickHouse 写入热点。 | - 先分析热点产生的根源（同一 event_name、相同 user_id）。<br>- 设计 **复合分区键**（event_name + hash(user_id)），在 Kafka、Flink、ClickHouse 多层面分散。<br>- 为防止业务侧改动导致倾斜，加入 **动态分区数** 与 **监控报警**（Kafka Lag、Flink Backpressure）。<br>- 通过 **实验**（压测）确认 24‑48 分区、并行度 8‑16 能支撑 10 k QPS 并留有余量。 |

### 2️⃣ 新手最容易犯的错误  

| 错误 | 影响 | 如何避免 |
|------|------|----------|
| **把所有业务都塞进单体服务** | 单点故障、伸缩受限、代码难维护。 | 从一开始就划分 **采集层 → 流处理 → 存储 → 查询** 四层，使用 **消息队列** 解耦。 |
| **忽视数据倾斜** | 某一分区/节点成为瓶颈，导致延迟爆炸、丢数据。 | 在设计 Kafka 分区键、Flink KeyBy、ClickHouse 分区时，使用 **复合键**、**随机前缀**，并在监控中加入 **倾斜检测**。 |
| **把实时和离线使用同一套存储** | 实时查询慢、离线成本高，难以满足两端需求。 | 明确 **冷热分层**：实时使用 ClickHouse，离线使用 Iceberg/Parquet。 |
| **只考虑功能不计成本** | 方案落地后预算超支，无法交付。 | 在选型阶段就对比 **存储成本、计算费用**，做 **容量预估**，并加入 **TTL、归档** 策略。 |
| **安全/合规只在最后才考虑** | 敏感数据泄露、合规审计不通过。 | 从 SDK 开始就实现 **脱敏/加密**，并在流处理层统一处理。 |

### 3️⃣ 学习建议和可延伸方向  

1. **基础功**：

   - **ClickHouse**：物化视图、分区/索引、分布式表、复制机制。  
   - **Iceberg**：表格快照、时间旅行、分区演进。  

3. **实战项目**：  
   - 在个人或开源项目中搭建 **小型埋点平台**（如使用 `kafka → Flink → ClickHouse`），从采集到查询全链路跑通。
