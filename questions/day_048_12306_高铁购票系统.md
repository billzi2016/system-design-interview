## 1. 题目背景
12306 是中国铁路客户服务中心提供的线上高铁/动车票务平台，面向全国数亿用户提供车次查询、车票预订、支付、改签、退票等业务。面试官希望了解你在高并发、大流量、强一致性场景下的系统设计能力。

## 2. 面试场景设定
**面试官**：

## 3. 功能性需求（核心 4‑6 项）

|------|----------|--------|
| 1 | **车次/余票查询** | 支持多条件过滤（出发站、到达站、日期、车次、座席等级），实时返回余票信息。 |
| 2 | **车票预订（抢票）** | 高并发下的秒级下单，防止超卖；支持排队/令牌桶控制。 |

| 4 | **改签/退票** | 在规定时间窗口内修改或取消订单，涉及余票回滚和退款流程。 |

# 题解

# 12306 高铁购票系统 – 详细系统设计方案  

> **写给**：刚入行的后端小伙伴  
> **目标**：从 **最小可用系统** 一步步演进到 **高可用、强一致、可伸缩** 的完整架构，并在每一步说明 **为什么要这么做**、**不这么做会有什么风险**。  

> **阅读建议**：先通读一遍总览章节，随后按章节顺序细读。每个章节后面的“思考点”帮助你自行验证是否真正理解。

## 解题思路总览  

1. **需求拆解** → 功能点、非功能点、边界。  
2. **流量与容量估算** → 把抽象的“120k QPS”转化为 CPU、网络、存储需求。  
3. **从单体到微服务**：先画出 **最小可用系统**（单体+单库），再逐层拆分为 **API 网关、业务服务、缓存、消息队列、持久化**。  
4. **强一致性 vs 最终一致性**：订单/支付必须强一致，余票库存可以用 **乐观锁+缓存** 实现 **“强一致+高并发”**。  
5. **削峰、排队、令牌桶**：在抢票高峰引入 **分布式限流 + 排队系统**，保证公平并防止系统崩溃。  
6. **容灾与跨地域**：使用 **多活/灾备双活** + **全局唯一 ID** + **幂等设计**，实现 **99.99%+** 的可用性。  
7. **监控告警 & 自动化运维**：从 **业务指标**、**系统指标**、**日志** 三层入手，构建 **Prometheus+Grafana + Alertmanager** 体系。  

下面把每一步拆开来讲。

## 第一步：理解需求与规模估算  

### 1. 功能需求（核心 4‑6 项）  

| 编号 | 功能 | 必要子功能 | 关键技术点 |
|------|------|-----------|------------|
| 1 | 车次/余票查询 | 多条件过滤、实时余票展示 | 缓存、读写分离 |
| 2 | 车票预订（抢票） | 高并发下的下单、库存扣减、排队/令牌桶 | 分布式锁、限流、消息队列 |
| 3 | 支付结算 | 第三方支付回调、事务性订单生成 | 幂等、两阶段提交（或可靠消息） |
| 4 | 改签/退票 | 余票回滚、退款、状态变更 | 逆向库存、事务补偿 |
| 5 | 用户身份管理 | 登录、实名认证、会话维护 | JWT、OAuth2、验证码防刷 |
| 6 | 订单查询 & 电子客票 | 订单历史、二维码生成 | 静态文件 CDN、加密签名 |

> **思考点**：哪些业务必须强一致？（订单、支付、库存）哪些可以接受 **最终一致性**？（查询缓存、统计报表）  

### 2. 非功能需求（关键指标）  

| 指标 | 估算值 | 背后隐含的实现难点 |
|------|--------|--------------------|
| DAU | 5,000 万 | 大量并发登录、Session 共享、用户画像存储 |
| QPS 峰值 | 120,000（查询 80k + 下单 40k） | 需要 **水平扩展**、**削峰**、**高效缓存** |
| 响应时延 | 查询 <200ms、下单/支付 <500ms | 网络 IO、DB 延迟、锁竞争必须控制在毫秒级 |
| 可用性 | ≥ 99.99%（年故障时间 < 52.56 min） | 多机房容灾、自动故障转移、灰度发布 |
| 存储规模 | ~150 TB（10 年历史） | 冷/热数据分层、归档、备份策略 |
| 数据一致性 | 订单/支付强一致，余票缓存最终一致 | 需要 **分布式事务** 或 **可靠消息** 方案 |

> **思考点**：如果把 **可用性** 降到 99.9%（年故障 8.76 小时），对业务有什么影响？（春运期间大量退票、舆情危机）  

### 3. 边界划分（本题需要实现的范围）  

- **内建**：登录、查询、下单、支付、改签、退票、订单查询、电子客票。  
- **外部**：第三方支付、短信/邮件验证码、时刻表只读。  

> **思考点**：把 **支付** 视为 **外部系统**，为何要在我们系统内部做 **幂等** 与 **事务**？（防止支付成功回调重复、资金安全）

## 第二步：高层架构设计  

### 1. 从单体到分层的演进路径  

| 阶段 | 主要组件 | 目标 | 典型技术 |
|------|----------|------|----------|
| **A. 最小可用系统** | 单体 Web + 单库（MySQL） | 快速验证业务流程 | Spring Boot + MySQL |
| **B. 横向拆分** | 前端负载均衡、业务服务拆分（用户、查询、订单、支付） | 逐步提升并发 | Nginx/SLB、Spring Cloud、MySQL 主从 |
| **C. 引入缓存 & 消息** | Redis、Kafka/RabbitMQ、分布式锁 | 解决热点、削峰、异步化 | Redis Cluster、Kafka |
| **D. 高可用 & 多活** | 多可用区（AZ）部署、全局负载均衡、灾备双活、全局唯一 ID、幂等 | 99.99%+ 可用，跨地域容灾 | LVS+Keepalived、Consul/Eureka、Snowflake、TCC/可靠消息 |
| **E. 监控/运维** | Prometheus、Grafana、ELK、灰度发布 | 可观测、快速定位故障 | Prometheus、Grafana、Kibana、Istio/Envoy |

> **思考点**：在 **B** 阶段，如果直接上 **Kafka**，会出现哪些风险？（业务拆分不彻底导致消息丢失、事务不一致）  

### 2. 关键高层图（Mermaid）

> **说明**：

## 第三步：数据库设计  

### 1. 选型原则  

| 数据 | 访问特征 | 选型 |
|------|----------|------|
| 订单、支付、用户、票务（强一致） | 事务、关联查询、强 ACID | **关系型 MySQL**（Percona / Aurora） |
| 余票缓存、会话、热点查询 | 高并发读/写、TTL | **Redis Cluster**（内存 + 持久化） |
| 大日志、审计、业务事件 | 顺序写、离线分析 | **Kafka + ClickHouse**（实时 OLAP） |
| 冷归档（历史订单） | 大批量写、低频读 | **对象存储 + Hive/Presto** |

> **思考点**：为什么不把余票直接放在 MySQL？（锁竞争、写放大、响应时延 > 200ms）  

### 2. 关系型库（MySQL）核心表结构  

#### 2.1 用户表（user）  

| 字段 | 类型 | 备注 |
|------|------|------|
| user_id | BIGINT UNSIGNED PK | **全局唯一 ID**（Snowflake） |
| mobile | VARCHAR(20) | 唯一，登录凭证 |
| password_hash | CHAR(64) | bcrypt |
| real_name | VARCHAR(50) | 实名认证 |
| id_number | VARCHAR(20) | 加密存储 |
| created_at | DATETIME | 注册时间 |
| updated_at | DATETIME | 最近登录/修改时间 |

> **索引**：`UNIQUE (mobile)`，`INDEX (real_name)`（查询实名用户）。  

#### 2.2 车次信息（train）  

| 字段 | 类型 | 备注 |
|------|------|------|
| train_id | BIGINT PK | 业务唯一，非全局 |
| train_no | VARCHAR(10) | 车次号 |
| departure_station | VARCHAR(10) |
| arrival_station | VARCHAR(10) |
| departure_time | DATETIME |
| arrival_time | DATETIME |
| seat_type | ENUM('商务座','一等座','二等座','硬座','硬卧') |
| total_seats | INT |
| price | DECIMAL(10,2) |
| version | INT | **乐观锁**（库存扣减） |

> **备注**：车次表是**只读**，可以放在只读库或缓存。  

#### 2.3 余票库存（seat_inventory）  

| 字段 | 类型 | 备注 |
|------|------|------|
| inventory_id | BIGINT PK |
| train_id | BIGINT FK |
| travel_date | DATE |
| seat_type | ENUM |
| total | INT |
| available | INT |
| version | INT | **乐观锁**字段 |
| updated_at | TIMESTAMP |

> **关键点**：**库存扣减**使用 **乐观锁 + CAS**，在业务层配合 **Redis 分布式锁** 防止热点冲突。  

#### 2.4 订单表（order）  

| 字段 | 类型 | 备注 |
|------|------|------|
| order_id | BIGINT PK |
| user_id | BIGINT FK |
| train_id | BIGINT |
| travel_date | DATE |
| seat_type | ENUM |
| passenger_name | VARCHAR(50) |
| passenger_id_number | VARCHAR(20) |
| status | ENUM('CREATED','PAID','CANCELLED','REFUNDED','EXPIRED') |
| price | DECIMAL(10,2) |
| create_time | DATETIME |
| pay_time | DATETIME NULL |
| expire_time | DATETIME |
| version | INT | 乐观锁（防并发修改） |

> **索引**：`INDEX (user_id, create_time)`，`INDEX (status, travel_date)`（改签/退票扫描）  

#### 2.5 支付表（payment）  

| 字段 | 类型 | 备注 |
|------|------|------|
| payment_id | BIGINT PK |
| order_id | BIGINT FK |
| third_party | ENUM('ALIPAY','WECHAT') |
| third_party_trade_no | VARCHAR(64) |
| amount | DECIMAL(10,2) |
| status | ENUM('INIT','SUCCESS','FAILED') |
| callback_time | DATETIME NULL |
| created_at | DATETIME |
| updated_at | DATETIME |

> **幂等点**：`third_party_trade_no` + `order_id` 建唯一索引，防止回调重复写入。  

### 3. 缓存层（Redis）  

| Key 结构 | 示例 | TTL | 说明 |
|----------|------|-----|------|
| `user:session:{token}` | `user:session:abc123` | 30 min | JWT 失效前保存用户基本信息（用于鉴权） |
| `seat:stock:{trainId}:{date}:{seatType}` | `seat:stock:1001:2024-02-15:二等座` | 0（永久） | 余票实时库存，使用 **原子递减 INCRBY** |
| `train:list:{from}:{to}:{date}` | `train:list:北京:上海:2024-02-15` | 5 min | 车次列表缓存，更新后主动失效 |
| `order:detail:{orderId}` | `order:detail:200000001` | 1 day | 订单详情缓存，读写分离后加速查询 |

> **失效策略**：  
- **库存变更**（下单/改签/退票）后 **立即更新 Redis**，若更新失败回滚 DB。  
- **查询热点**（车次列表）每 5 分钟主动刷新，或在 **车次信息更新** 时推送失效消息。

## 第四步：核心 API 设计  

下面给出 **RESTful**（也可使用 **gRPC**）的核心接口示例，强调 **幂等性**、**鉴权**、**限流**。  

### 1. 鉴权（登录）  

| 方法 | URL | 请求 | 响应 | 说明 |
|------|-----|------|------|------|
| POST | `/api/v1/auth/login` | `{ "mobile": "13800138000", "password": "******" }` | `{ "access_token":"jwt...", "expires_in":3600 }` | 登录成功返回 JWT，放入 **Authorization: Bearer** 头部。 |
| POST | `/api/v1/auth/logout` | `-`（JWT） | `200 OK` | 将 token 加入 **黑名单**（Redis Set），实现 **强制下线**。 |

> **思考点**：为什么要把 token 放进 Redis 黑名单而不是仅靠 JWT 失效？（即时注销、防止 token 被盗用）  

### 2. 余票查询  

| 方法 | URL | 参数 | 响应 | 说明 |
|------|-----|------|------|------|
| GET | `/api/v1/trains` | `from=北京&to=上海&date=2024-02-15&seat_type=二等座` | `{ "trains": [ { "train_id":1001, "departure":"08:00", "arrival":"12:30", "stock": 23 }, … ] }` | 先查询 **Redis**，若 miss 再落库；返回 **可用余票**（stock）。 |
| GET | `/api/v1/trains/{trainId}` | `date=2024-02-15` | 车次详情 + 库存 | 适用于“单车次详情”。 |

> **限流**：对查询接口使用 **IP+用户** 双层限流（QPS 1000/秒），防止爬虫刷库。  

### 3. 抢票下单（核心）  

> **业务流程**：  
> 1️⃣ 检查用户登录、实名信息；  
> 2️⃣ **令牌桶**（全局限流）判断是否有抢票资格；  
> 3️⃣ **分布式锁** + **乐观锁** 扣减库存；  
> 4️⃣ 生成 **订单**（状态 `CREATED`），写入 MySQL 主库；  
> 5️⃣ 将 **订单消息** 发送至 Kafka（下单成功事件）；  
> 6️⃣ 返回 **订单 ID** 给前端，前端轮询支付状态。  

| 方法 | URL | 请求体 | 响应 | 幂等性 |
|------|-----|--------|------|--------|
| POST | `/api/v1/orders` | `{ "train_id":1001, "date":"2024-02-15", "seat_type":"二等座", "passenger": {"name":"张三","id_no":"123456199001011234"} }` | `{ "order_id":200000001, "status":"CREATED" }` | **幂等键**：`Idempotency-Key`（UUID）+ `user_id`，服务端先检查 Redis 幂等表。 |

> **关键点**：  
- **库存扣减**：`Redis DECRBY` + **Lua 脚本**（原子检查>0）→成功后写 DB（事务）。若 DB 写失败再 **ROLLBACK** Redis。  
- **防超卖**：使用 **乐观锁 version** + **CAS**，若 version 不匹配直接返回 “余票不足”。  

### 4. 支付回调（异步）  

> 第三方支付会向我们系统 **POST** 回调，必须 **幂等**。  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| POST | `/api/v1/payments/callback` | `{ "order_id":200000001, "trade_no":"2024021512345678", "status":"SUCCESS", "amount":123.00, "sign":"xxx" }` | `200 OK` | 通过 **签名校验**，使用 **RabbitMQ/Kafka** 写入支付成功事件，后续业务（改状态、发送票据）消费。 |

> **事务实现**：  
- **可靠消息**（Kafka）+ **本地事务**（MySQL） → 先在本地 DB 更新订单状态为 `PAID`，再发送 **支付成功** 消息。若消息发送失败，事务回滚。  

### 5. 改签 / 退票  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| POST | `/api/v1/orders/{orderId}/change` | `{ "new_train_id":1002, "new_date":"2024-02-20", "new_seat_type":"一等座" }` | `{ "new_order_id":200000005, "status":"CREATED" }` | 改签视为 **新建订单**，原订单标记 `CANCELLED`，并将原余票回滚。 |
| POST | `/api/v1/orders/{orderId}/refund` | `-` | `{ "refund_id":300001, "status":"PROCESSING" }` | 退票先检查是否在可退时间窗口，调用支付系统退款接口，成功后标记 `REFUNDED`。 |

> **库存回滚**：采用 **补偿事务**（TCC）或 **消息回滚**，确保 **余票恢复** 与 **订单状态** 一致。  

### 6. 电子客票查询  

| 方法 | URL | 参数 | 响应 | 说明 |
|------|-----|------|------|------|
| GET | `/api/v1/tickets/{orderId}` | `-` | `{ "qr_code_url":"https://cdn.xxx.com/ticket/200000001.png" }` | 二维码在对象存储生成（一次性），返回 CDN 链接，前端直接展示。 |

## 第五步：详细组件设计  

### 1. API Gateway  

- **职责**：统一入口、流量治理、鉴权、灰度发布、统一日志。  
- **实现**：  
  - **Nginx + Lua**（轻量）或 **Kong**、**Spring Cloud Gateway**。  
  - **限流**：全局令牌桶（QPS 200k）+ **用户级滑动窗口**（防刷）。  
  - **安全**：TLS、IP 黑名单、WAF（防SQL注入、XSS）。  

### 2. 鉴权中心（Auth Service）  

- **技术**：Spring Security + JWT + Redis 黑名单。  
- **关键点**：  
  - **登录失败次数**（IP+账号）放入 Redis，超过阈值返回验证码。  
  - **实名校验**：调用 **身份认证服务**（外部），返回 `real_name_verified` 标记。  

### 3. 余票查询服务（Search Service）  

- **读写分离**：查询走 **读库 + Redis**，写操作只在 **订单服务** 中进行。  
- **热点车次**：使用 **热点缓存**（Redis LRU）+ **预热**（在高峰前 30 分钟预加载当天所有车次库存）。  

### 4. 订单服务（Order Service）  

- **核心流程**：  
  1. **令牌桶**（全局） → 检查用户是否还有抢票资格。  
  2. **分布式锁**（Redis `SETNX` + expiration）锁定 **train_id+date+seat_type**，防止同一时刻并发扣库存。  
  3. **库存扣减**：`Lua` 脚本原子检查 `available > 0` 并 `DECR`.  
  4. **事务写库**：使用 **MySQL XA** 或 **Spring Transaction**（本地事务）写入 `order` 与 `seat_inventory`（乐观锁）。  
  5. **消息投递**：成功后向 **Kafka** 发送 `order.created`。  

- **幂等实现**：  
  - **Idempotency-Key**（客户端生成） + **Redis** `SET key value NX EX 86400`。  
  - 若键已存在，直接返回已创建的 `order_id`。  

### 5. 支付服务（Payment Service）  

- **调用链**：`Order Service → Payment Service → 第三方支付`.  
- **防重放**：回调携带 `trade_no`，在 **payment** 表上唯一索引。  
- **可靠消息**：  
  - 支付成功后 **本地事务** 先更新 `order.status = PAID`，再 **发送** `payment.success` 到 Kafka。  
  - 消费者（如 **Ticket Service**）在消费成功后 **确认**（Kafka commit）。  

### 6. 改签/退票服务（Change Service）  

- **业务模型**：改签 = **取消旧订单 + 新建订单**（两段事务）。  
- **实现**：采用 **TCC（Try‑Confirm‑Cancel）**：  
  - **Try**：检查新车次余票、锁定资源。  
  - **Confirm**：扣除新余票、标记旧订单 `CANCELLED`、回滚旧余票。  
  - **Cancel**：任一步失败，回滚已锁定资源。  

### 7. 电子客票服务（Ticket Service）  

- **生成**：订单 `PAID` 后，异步消费 `payment.success`，生成 **QR Code**（使用 **zxing**），存入 **OSS**（如阿里云 OSS），返回 CDN URL。  
- **防伪**：二维码内容为 **签名的订单信息**（order_id + secret），检票系统通过公钥验证。  

### 8. 消息队列（Kafka）  

| Topic | 生产者 | 消费者 | 关键属性 |
|-------|--------|--------|----------|
| `order.created` | Order Service | Payment Service、Change Service | 订单创建，触发支付 |
| `payment.success` | Payment Service | Ticket Service、统计服务 | 支付成功，生成票 |
| `order.cancelled` | Change/Refund Service | Inventory Service、统计服务 | 余票回滚 |
| `ticket.generated` | Ticket Service | 前端推送、审计 | 电子客票生成 |

- **分区**：`orderId % 12`（12 分区），保证同一订单的所有消息顺序消费。  

### 9. 缓存层（Redis）  

| 结构 | 用途 | 失效策略 |
|------|------|----------|
| **String** | 余票库存 `stock:{trainId}:{date}:{seat}` | **原子递减**，永不过期（业务层自行恢复） |
| **Hash** | 用户会话 `session:{token}` | 30 min TTL，活跃时续期 |
| **Set** | 幂等键 `idem:{userId}:{key}` | 24 h（一次性） |
| **Sorted Set** | 排队令牌 `queue:{date}` | 按时间排序，TTL 1 h |

> **为什么要使用 Lua 脚本**：Redis 本身的 `DECR` 不是“先检查后递减”，Lua 脚本可以一次性完成 “如果 stock>0 则 stock--，否则返回错误”。

## 第六步：扩展性与高可用设计  

### 1. 流量削峰与排队（回答提示 1）  

| 场景 | 方案 | 实现细节 |
|------|------|----------|
| **瞬时 QPS > 200k** | **全局令牌桶 + 本地滑动窗口** | - **令牌桶**（Redis `INCR` + `EXPIRE`）每秒放入 100k token，只有拿到 token 的请求才进入下单流程。<br>- **本地滑动窗口**（每台机器维持 1s 内请求计数）进一步防止单点突发。 |
| **公平排队** | **分布式排队队列**（Redis ZSET） | - 请求进入 `ZSET`（score = timestamp + 随机），消费端按顺序弹出，弹出前再次检查库存。<br>- 若排队时间超过阈值（如 30 s）返回 “排队超时”。 |
| **降级** | **只读模式**（查询）| 高峰期间关闭抢票入口，仅提供查询，等到流量恢复后再开启。可以通过 **Feature Flag**（Apollo）动态切换。 |

> **为什么不直接加大机器数**：单纯扩容只能提升 **处理能力**，但 **库存竞争** 仍会导致超卖；**排队/令牌桶** 在业务层面限制并发，防止 DB、Redis 产生热点导致宕机。  

### 2. 余票库存一致性（回答提示 2）  

#### 读路径  

1. **前端查询** → **API Gateway** → **Search Service** → **Redis**（`stock:{trainId}:{date}:{seat}`）  
2. 若 Redis **未命中** → 访问 **MySQL 只读库** → 将结果写入 Redis（**写回**）。  

#### 写路径（下单）  

```text
① 客户端请求抢票
② Order Service 通过 Redis Lua 脚本原子检查并 DECR（stock > 0 ? stock-- : fail)
③ 若 Lua 成功，返回 “锁定成功”，进入本地事务：
    - INSERT order (status=CREATED)
    - UPDATE seat_inventory SET available=available-1, version=version+1 WHERE train_id=? AND date=? AND seat_type=? AND version=oldVersion
④ 若 DB 更新失败（乐观锁冲突），回滚 Redis DECR（INCR）并返回 “余票不足”
⑤ 成功后发送 Kafka order.created
```

#### 缓存失效策略  

| 失效场景 | 处理方式 |
|----------|----------|
| **库存扣减成功** | **不失效**，直接在 Redis 中递减。 |
| **订单取消/退票** | **INCR** Redis 对应键；如果库存已经为 **最大值**（total），直接 **删除** 并从 DB 重新加载。 |
| **系统异常（如 Redis 重启）** | **持久化**（AOF + RDB）+ **启动时全量加载**（从 MySQL 读取当日所有余票到 Redis）。 |
| **库存异常（负数）** | 警报触发，自动 **回滚**（检查 DB 与 Redis 差异）并人工干预。 |

> **为什么不在每次查询时直接访问 DB**：每秒上万次查询如果都落库，DB 将成为瓶颈；缓存层提供 **毫秒级** 读，极大降低延迟。  

### 3. 容灾与跨地域部署（回答提示 3）  

#### 3.1 多活/灾备架构  

```
[用户] → [Global L4 Load Balancer (DNS + Anycast)] → 
    ├─[Region A] (主站) → 业务集群、MySQL 主/从、Redis Cluster、Kafka
    └─[Region B] (灾备) → 同构备份、只读 MySQL、Redis Slave、Kafka Mirror
```

- **数据同步**：  
  - **MySQL 主从**（基于 GTID）实时复制到灾备区。  
  - **Redis 主从**（异步复制） + **RDB** 跨地域同步（每 5 秒）。  
  - **Kafka MirrorMaker** 将 Topic 同步到灾备集群。  

#### 3.2 故障转移流程  

| 故障类型 | 检测方式 | 自动切换步骤 |
|----------|----------|--------------|
| **业务机器宕机** | Liveness Probe (K8s) | Service 自动重新调度到健康节点。 |
| **Redis 单点故障** | Sentinel/Cluster failover | 自动选举新 master，客户端自动切换。 |
| **MySQL 主库不可用** | MHA（MySQL High Availability）| 自动提升最近的 slave 为 master，切换 DNS/SLB。 |
| **整机房网络中断** | 监控告警（Ping、链路时延）| Global L4 将流量切换到 **Region B**；灾备站点已经同步到最新数据，业务继续。 |
| **支付回调丢失** | Kafka 消费位点持久化 | 消费者重启后从上一次提交的 offset 继续消费，保证 **不漏**。 |

#### 3.3 幂等与防重复  

- **全局唯一 ID**：使用 **Snowflake**（41 位时间戳 + 10 位机器码 + 12 位序列）生成 **order_id**、**payment_id**，保证跨机房不冲突。  
- **业务幂等**：所有写操作（下单、支付回调、改签、退款）都 **检查唯一键**（如 `order_id`、`payment_id`、`Idempotency-Key`），若已存在直接返回成功。  

> **为什么要采用 **双活** 而不是单活 + 主备**：春运期间网络抖动可能导致跨区域切换时间过长，双活可以 **无感知** 切换，满足 **99.99%** 可用性要求。  

### 4. 监控、告警与自动化运维  

| 维度 | 关键指标 | 监控工具 | 告警阈值 |
|------|----------|----------|----------|
| **业务** | QPS、成功率、排队长度、库存负数次数 | Prometheus + Grafana | QPS > 80% 设定上限、排队 > 5000、库存负数 > 0 |
| **系统** | CPU、内存、磁盘 I/O、网络流量 | Node Exporter + Prometheus | CPU > 85% 持续 5min |
| **中间件** | Redis 延迟、复制滞后、Kafka 消费位点滞后 | Redis Exporter、Kafka Exporter | 延迟 > 10ms、复制滞后 > 5s |
| **日志** | 错误率、异常栈 | ELK (Filebeat + Logstash + Kibana) | 错误率 > 0.1% |
| **业务链路** | 端到端时延（OpenTelemetry） | Jaeger + Prometheus | 订单创建 > 500ms |

- **自愈**：当某节点 CPU 持续高负载，自动 **水平扩容**（K8s HPA）。  
- **灰度发布**：使用 **Canary** 或 **蓝绿** 部署，监控新版本错误率，一旦异常回滚。

## 第七步：常见面试追问与回答  

### 1️⃣ 流量削峰与排队机制  

> **问题**：如果在春运抢票高峰时，系统瞬时 QPS 超过 200k，如何进行流量削峰并保证公平性？  

**回答思路**：

1. **全局限流（令牌桶）**  
   - 在 Redis 中维护一个 **令牌桶**（key `ticket:token_bucket`），每秒以固定速率（如 150k）往桶里放 token。  
   - 请求抢票前先 `GET token`，若取不到直接返回 “系统繁忙”。这样把流量压在 **可控范围** 内。  

2. **本地滑动窗口**  
   - 每台机器维护 1 秒窗口计数，若本机 QPS 超过阈值（如 5k），直接拒绝或排队。防止单点机器被压垮。  

3. **分布式排队（ZSET）**  
   - 抢票请求在拿到令牌后，放入 **Redis ZSET** `queue:{date}`（score = 当前时间戳 + 随机），消费者按序弹出。  
   - 弹出前再次检查余票，如果余票不足则直接返回 “抢票失败”。  
   - 排队超时（如 30 s）返回 “排队超时”，用户可自行刷新或放弃。  

4. **公平性保证**  
   - 采用 **先到先得** 的 ZSET 排序，且每个用户只能持有 **一个排队位**（用 `SETNX` 锁定 `queue_user:{userId}`），防止刷票。  
   - 对于 VIP/会员可以在 **令牌桶** 中预留一定比例的 token，实现差异化服务（业务层可配置）。  

5. **降级策略**  
   - 当排队长度持续增长且系统负载逼近上限时，**关闭抢票入口**（只提供查询），并在页面提示 “当前抢票高峰，请稍后”。  

> **为什么要在业务层做限流**：单纯在 **负载均衡** 或 **网络层** 做限流只能削减流量，但无法保证业务关键资源（余票）不被超卖。业务层的 **令牌桶+排队** 能把“抢票机会”公平分配给真正的用户。  

### 2️⃣ 余票库存一致性  

> **问题**：请详细说明余票库存的读写路径、缓存失效策略以及如何防止超卖？  

**回答要点**：

1. **读取路径**  
   - **API Gateway → Search Service → Redis**（Key `stock:{trainId}:{date}:{seat}`）。  
   - 若 Redis **命中**，直接返回；若 **miss**，查询 MySQL 只读库，并写回 Redis（`SETEX`）。  

2. **写入路径（抢票）**  
   - **Lua 脚本**在 Redis 中原子执行：  
     ```lua
     local key = KEYS[1]
     local stock = tonumber(redis.call('GET', key))
     if stock and stock > 0 then
         redis.call('DECR', key)
         return 1
     else
         return 0
     end
     ```  
   - 脚本返回 `1` 表示扣库存成功；返回 `0` 表示余票不足。  

3. **持久化同步**  
   - 成功扣 Redis 后，进入 **本地事务**：  
     - `INSERT order`（status=CREATED）  
     - `UPDATE seat_inventory SET available=available-1, version=version+1 WHERE ... AND version = oldVersion`（乐观锁）  
   - 若 DB 更新 **失败**（版本冲突），在业务层 **回滚** Redis（`INCR`）并返回 “抢票失败”。  

4. **防止超卖的关键点**  
   - **原子检查+递减**（Redis Lua）防止并发读‑写产生的竞争。  
   - **乐观锁**确保 MySQL 中的库存与 Redis 同步，防止因网络延迟导致的“双减”。  
   - **幂等**：每个下单请求都有唯一的 `Idempotency-Key`，防止用户因网络重试产生的重复扣库存。  

5. **缓存失效策略**  
   - **库存变化**（下单、改签、退票）**不失效**，直接在 Redis 中递减/递增。  
   - **系统异常**（Redis 重启）采用 **AOF + RDB** 持久化，启动时全量恢复。  
   - **业务手动刷新**：后台定时（每 5 分钟）比对 MySQL `available` 与 Redis `stock`，若不一致执行 **强制同步**。  

> **如果只使用 DB 乐观锁而不加 Redis**：在高并发时每个请求都要走 DB，锁冲突率高，QPS 难以突破 10k；**加入 Redis** 能把热点库存操作搬到内存，提升并发度。  

### 3️⃣ 容灾与跨地域部署  

> **问题**：在出现机房网络故障或单点故障时，系统如何快速切换到灾备中心，保证订单不丢失、支付不重复？  

**回答要点**：

1. **跨地域全局负载均衡**  
   - 使用 **DNS Anycast + L4 Global Load Balancer**（如阿里云 SLB Global）将用户流量路由到最近可用的 Region。  

2. **数据同步机制**  
   - **MySQL 主从**（GTID）实时复制到灾备区，保证事务日志同步。  
   - **Redis 主从**（异步） + **RDB/AOF** 跨地域同步（每 5 s）。  
   - **Kafka MirrorMaker** 将业务 Topic 同步到灾备 Kafka。  

3. **故障检测与自动切换**  
   - **健康检查**（TCP/HTTP）+ **链路监控**（Ping、Traceroute）。  
   - 当主站网络不可达，SLB 自动将流量切换到 **Region B**，灾备站点已具备最新的 MySQL、Redis、Kafka 数据。  

4. **订单不丢失**  
   - 所有 **写操作** 必须走 **Kafka**（可靠消息），即使主库挂掉，消息仍在 Kafka 中持久化。灾备消费组在切换后继续消费，完成订单落库。  

5. **支付不重复**  
   - **支付回调**通过 **幂等唯一键**（`trade_no`）在 `payment` 表上唯一索引。即使回调在灾备恢复后重复到达，业务层会检测到已存在记录，直接返回成功，不会产生二次扣款。  

6. **全局唯一 ID**  
   - 采用 **Snowflake**（或 **Sonyflake**）生成全局唯一 `order_id`、`payment_id`，跨机房不冲突，保证数据关联性。  

7. **灾备演练**  
   - 每月进行一次 **灾备切换演练**（手动或自动），验证 **SLB 切换、数据同步延迟、消息消费** 是否满足 **< 1 分钟** 的恢复时间目标（RTO）。  

> **如果只在单机房内部做 HA**：当机房网络整体失效（如光纤中断），即使内部有多副本也无法对外提供服务，导致业务不可用，违背 **99.99%** 的可用性要求。跨地域双活才能真正实现业务不间断。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **余票库存的强一致性 + 高并发** | 需要在 **毫秒级** 完成库存检查、扣减、持久化，且避免超卖。 | - 先考虑 **全局锁**（如 Zookeeper）→锁竞争太大。<br>- 再尝试 **单机乐观锁** →仍有热点冲突。<br>- 最终组合 **Redis Lua 原子扣减** + **MySQL 乐观锁**，把热点放在内存，事务只做校验，极大降低冲突概率。 |
| **削峰排队与公平性** | 春运瞬时流量 200k+，如果直接放开会导致系统崩溃；又要保证“先到先得”。 | - 先想 **限流**（IP、全局）→不够公平。<br>- 再考虑 **令牌桶** + **排队队列**（ZSET）→实现了流量控制 + 排队顺序。<br>- 细化 **幂等键**、**用户排队位**，防止刷票。 |

### 2. 新手最容易犯的错误（≥2 条）  

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有业务写在单体服务里，直接连接 MySQL** | 高并发时 DB 成为瓶颈，响应时间 > 500 ms，系统容易宕机。 | 将 **热点（余票）** 放到 **Redis**，业务层使用 **缓存 + 异步消息**，实现读写分离。 |
| **只依赖乐观锁或数据库事务，忽视缓存层的并发** | 仍会出现超卖或缓存脏数据（如负库存），导致用户投诉。 | 在 **Redis** 用 **Lua 脚本** 做原子检查‑扣减，确保在内存层面先防止冲突，再落库。 |
| **支付回调不做幂等处理** | 第三方回调重试会导致订单、资金重复扣减，产生财务纠纷。 | 使用 **唯一交易号** + **唯一索引**，在业务层返回 **已处理** 状态，保证回调幂等。 |
| **灾备只做数据备份，忘记同步消息队列** | 主库宕机后，未消费的下单消息丢失，导致订单缺失。 | 同步 **Kafka**（MirrorMaker），并在灾备恢复后继续消费。 |

### 3. 学习建议与可延伸方向  

1. **系统设计基本功**  
   - 熟练掌握 **CAP**、**BASE**、**一致性模型**，能快速判断哪些数据需要强一致。  
   - 多练 **读写分离、缓存击穿、缓存雪崩、缓存穿透** 的防御手段。  

2. **分布式锁与限流**  
   - 学习 **Redis Lua**、**Redisson**、**Zookeeper**、**Etcd** 的使用场景与局限。  
   - 实践 **令牌桶**、**漏桶**、**滑动窗口** 的实现代码。  

3. **可靠消息 & 事务**  
   - 了解 **可靠消息（Transactional Outbox）**、**Kafka Exactly‑Once**、**TCC** 三种事务方案的优缺点。  
   - 手写一个 **Outbox** 示例：业务表 + 消息表 + 定时任务。  

4. **容灾与多活**  
   - 掌握 **MySQL GTID**、**MHA**、**PXC**、**TiDB** 等分布式数据库的主备切换。  
   - 练习 **Kafka MirrorMaker**、**Redis Replication** 跨地域配置。  

5. **监控与自动化**  
   - 熟悉 **Prometheus** 的 **exporter** 编写，掌握 **Alertmanager** 的路由规则。  
   - 学会使用 **OpenTelemetry** 对业务链路做端到端追踪。  

> **小练习**：用 Docker‑Compose 搭建一个包含 **Nginx → SpringBoot → MySQL + Redis + Kafka** 的简易抢票原型，验证 **Lua 扣库存 + Kafka 可靠消息** 的完整闭环。
