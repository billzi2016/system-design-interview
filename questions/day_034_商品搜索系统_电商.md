# 第 34 天：设计 商品搜索系统（电商）

> 生成日期：2026-04-22

---

## 题目背景  
商品搜索系统是电商平台的核心入口，帮助用户通过关键词、筛选、排序等方式快速定位并浏览感兴趣的商品。系统需要在海量商品数据和高并发请求下，保证搜索的相关性、实时性和低延迟。

## 面试场景设定  
**面试官**：  
> “我们现在要设计一个面向全国用户的商品搜索系统。请你从零开始构建整个系统的架构，重点考虑如何支撑每天数十亿次的搜索请求、如何保证搜索结果的相关性与实时性，以及系统的可扩展性和高可用性。你可以先从整体思路说起，然后逐步展开细节。”

## 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| 1 | **关键词搜索**：用户输入任意关键词，系统返回匹配的商品列表。 |
| 2 | **多维过滤**：支持按类目、品牌、价格区间、销量、评分、属性（如颜色、尺寸）等进行过滤。 |
| 3 | **排序功能**：支持按综合相关度、销量、价格（升/降）、上新时间等排序。 |
| 4 | **自动补全 & 推荐**：在用户输入时提供搜索词联想、热搜词、个性化推荐。 |
| 5 | **分页 & 游标**：返回分页结果，支持翻页、无缝滚动，保证结果一致性。 |
| 6 | **实时索引更新**：商品上下架、价格、库存等变动能够在秒级内反映到搜索结果中。 |

## 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 日活跃用户 (DAU) | 1500 万 | 包括移动端、PC 端用户。 |
| QPS (峰值) | 45,000 QPS | 高峰期（双11、618）期间的请求峰值。 |
| 单次搜索响应时间 | ≤ 120 ms (99.9% 请求) | 包括网络传输、业务处理、返回前端。 |
| 可用性 | 99.99% (月均累计不可用时间 ≤ 4.38 小时) | 必须实现跨地域容灾与无单点故障。 |
| 存储规模 | 商品全文索引 300 TB，倒排索引 + 辅助属性索引 150 TB | 基于 10 亿 SKU，平均 200 KB/商品的文本与属性数据。 |
| 数据刷新时延 | ≤ 5 秒 | 商品状态、价格、库存等变更实时写入搜索索引。 |

## 系统边界  

**本题范围内需要考虑的内容**  
- 商品数据的抓取、清洗、特征抽取与索引构建。  
- 搜索请求的路由、查询解析、倒排索引检索、排序与聚合。  
- 自动补全、热词、个性化推荐的实时计算与缓存。  
- 高可用架构（多活中心、容灾切换）与水平扩展。  
- 监控、日志、限流、灰度发布等运维能力。  

**本题范围外（不必实现）**  
- 商品详情页的渲染、购物车、订单支付等业务流程。  
- 推荐系统的深度学习模型训练（只需考虑轻量化实时推荐即可）。  
- 第三方广告投放、联盟营销等外部业务。  
- 法律合规、审计日志等合规性细节。  

## 提示与追问  

1. **索引更新策略**：如果采用增量实时索引与离线全量重建相结合，你会如何设计数据管道和一致性保障？  
2. **多租户与类目隔离**：在不同业务线（服装、数码、食品）之间，是否需要独立索引或共享索引？如何在保证查询性能的同时实现资源隔离？  
3. **搜索相关性排序**：在保证低延迟的前提下，你会使用哪些排序模型（BM25、向量检索、学习排序）？如何进行模型的在线评估与回滚？

## 第一步：理解需求与规模估算  

### 1. 功能需求拆解  

| 编号 | 功能 | 核心要点 |
|------|------|----------|
| 1 | 关键词搜索 | 分词、倒排检索、相关度排序 |
| 2 | 多维过滤 | 类目、品牌、价格区间、销量、评分、属性等过滤条件 |
| 3 | 排序 | 综合相关度、销量、价格（升/降）、上新时间 |
| 4 | 自动补全 & 推荐 | 热词、联想、个性化（基于用户历史） |
| 5 | 分页 & 游标 | 深度分页、游标保持结果一致性 |
| 6 | 实时索引更新 | 商品上下架、价格、库存在 **≤5 秒** 内同步到搜索层 |

### 2. 非功能需求（业务指标）  

| 指标 | 目标值 | 解释 |
|------|--------|------|
| DAU | 1500 万 | 需要支撑的活跃用户数 |
| 峰值 QPS | 45 000 | 高峰时每秒搜索请求数 |
| 单次搜索响应时间 | ≤120 ms（99.9%） | 包括网络+业务+返回 |
| 可用性 | 99.99%（月累计 ≤4.38 h） | 必须跨地域容灾 |
| 存储规模 | 索引 300 TB + 辅助 150 TB | 10 亿 SKU、200 KB/商品 |
| 数据刷新时延 | ≤5 秒 | 实时写入搜索索引的时效要求 |

### 3. 规模预估（帮助后面做容量规划）  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日搜索请求数** | 1500 万 DAU × 3 次/日（保守） = 4.5 亿 |  |
| **峰值 QPS** | 45 000 QPS（已给） |  |
| **单机 QPS 能力**（假设 1 台搜索节点 5 000 QPS）| 45 000 / 5 000 ≈ 9 台 | 需要 **≥9** 台搜索节点（考虑冗余 1.5 倍 → 14 台） |
| **索引大小** | 10 亿 SKU × 200 KB ≈ 200 TB（原始） + 额外倒排、分词、向量等 → 300 TB | 采用 **分片**、**冷热分层** 设计 |
| **实时写入流量** | 假设 10 % 商品每天有变更 → 1 亿 条写入/天 ≈ 1 200 TPS | 需要 **Kafka**/ **Pulsar** 作为消息队列，写入吞吐 2 k TPS 以上 |

> **如果不做这些预估**，直接选型可能会导致 **硬件资源不足、查询延迟爆炸**，面试官会抓住这点扣分。

## 第二步：高层架构设计  

下面给出 **从最小可用到全链路** 的逐层演进图（文字版），你可以自行在白板上画出类似结构。

### 2.1 最小可用系统（MVP）

```
[客户端] -> [API Gateway] -> [Search Service] -> [Elasticsearch 集群]
```

- **API Gateway**：统一入口，做限流、鉴权、灰度发布。
- **Search Service**：解析请求、构造 ES DSL、返回结果。
- **Elasticsearch**：单机或两机副本，存放倒排索引、属性索引。

> **不足**：单机/单可用区、没有缓存、无实时写入、无法支撑 45k QPS。  

### 2.2 扩展为可横向扩展的分层架构

```
                               +-------------------+
                               |   CDN / Edge Cache|
                               +--------+----------+
                                        |
+-------------------+   +----------------v-------------------+   +-------------------+
|   客户端 (Web/APP) |   |          API Gateway (L4/L7)       |   |   监控/日志平台   |
+-------------------+   +----------------+-------------------+   +-------------------+
                                 |  (限流、鉴权、灰度)
          +----------------------+---------------------+----------------------+
          |                      |                     |                      |
+----------------+   +--------------------+   +----------------+   +-------------------+
| Search Service |   | AutoComplete Service|   | Recommendation |   |   Metrics/Alert   |
+--------+-------+   +----------+----------+   +--------+-------+   +-------------------+
         |                        |                     |
         |   +--------------------v---------------------v-------------------+
         |   |                Elasticsearch 集群 (多AZ 多副本)                |
         |   +--------------------+---------------------+-------------------+
         |                        |                     |
         |   +--------------------v---------------------v-------------------+
         |   |               Redis / Memcached 缓存层 (热点词/结果)          |
         |   +--------------------+---------------------+-------------------+
         |                        |                     |
         |   +--------------------v---------------------v-------------------+
         |   |            Kafka / Pulsar (实时写入队列)                     |
         |   +--------------------+---------------------+-------------------+
         |                        |                     |
         |   +--------------------v---------------------v-------------------+
         |   |   实时索引写入服务 (Logstash / Flink)   |   离线全量构建 (Spark) |
         |   +---------------------------------------------------------------+
```

#### 关键模块说明  

| 模块 | 作用 | 关键技术选型 | 为什么选 |
|------|------|--------------|----------|
| **API Gateway** | 统一入口、限流、鉴权、灰度发布 | Kong / APISIX / Alibaba Cloud API Gateway | 轻量、插件化、支持多活 |
| **Search Service** | 业务层：解析、过滤、排序、分页、游标 | Spring Boot + Spring Cloud OpenFeign | 熟悉的 Java 生态，易做灰度 |
| **AutoComplete Service** | 联想词、热词、个性化推荐 | 基于 Redis 的前缀树 + 简易模型 | 低延迟（≤10 ms） |
| **Elasticsearch 集群** | 主搜索引擎，倒排+属性索引 | ES 8.x + 热/冷节点分层 | 成熟、支持 BM25、向量检索、分片复制 |
| **Redis / Memcached** | 热点查询缓存、热点词缓存 | Redis Cluster (主从) | 读写分离、缓存击穿保护 |
| **Kafka / Pulsar** | 实时写入的消息队列 | Kafka 3.x（或 Pulsar） | 高吞吐、持久化、消费组灵活 |
| **实时索引写入服务** | 将商品变更写入 ES（增量） | Logstash + ES Ingest Pipeline / Flink | 支持高并发、Exactly‑once |
| **离线全量构建** | 周期性全量重建倒排索引，修复漂移 | Spark / Hadoop + ES Bulk API | 处理海量历史数据、降低碎片 |
| **监控/日志平台** | 业务指标、链路追踪、告警 | Prometheus + Grafana + Loki + Jaeger | 开源、可视化、易扩展 |

> **如果直接选单机 MySQL** 作为搜索引擎，**倒排查询会非常慢**，且难以支持多维过滤，面试官会立刻挑出这点。  

### 2.3 多活跨地域容灾  

```
                +--------------------+       +--------------------+
                |   Region A (华北)   |       |   Region B (华东)   |
                +--------------------+       +--------------------+
                |  API GW + Search   | <---> |  API GW + Search   |
                |  Service + ES     |       |  Service + ES      |
                +--------------------+       +--------------------+
                        ^  ^                         ^  ^
                        |  |   数据复制（跨 Region）   |  |
                        +--+---------------------------+--+
                                    Kafka MirrorMaker
```

- **双活**：两套完整业务链路（同样的 ES 索引），流量由 **全局负载均衡（如 Nginx+LVS 或云厂商的 GSLB）** 按健康度分发。  
- **跨 Region 数据同步**：使用 **Kafka MirrorMaker** 或 **Pulsar Replication** 将商品变更同步到另一 Region 的 ES。  
- **故障切换**：监控感知单个 Region 不可用后，自动将 DNS/SLB 指向另一 Region（秒级切换）。

## 第三步：数据库设计  

搜索系统的核心是 **倒排索引**，但我们仍需要 **关系型/键值数据库** 来存放商品的 **原始元数据**、**库存/价格快照**、**用户行为日志** 等。下面分别说明。

### 3.1 关系型数据库（MySQL）  

| 表名 | 关键字段 | 说明 |
|------|----------|------|
| `product` | `product_id PK`, `title`, `description`, `category_id`, `brand_id`, `price`, `stock`, `status (上架/下架)`, `create_time`, `update_time` | 商品主表，供后台管理、离线 ETL 使用。 |
| `category` | `category_id PK`, `name`, `parent_id` | 类目树结构。 |
| `brand` | `brand_id PK`, `name` | 品牌表。 |
| `product_attr` | `product_id PK`, `attr_key`, `attr_value` | 可变属性（颜色、尺码）键值对。 |
| `sku` | `sku_id PK`, `product_id FK`, `price`, `stock`, `status` | 多 SKU 场景（如同一商品不同颜色）。 |

- **分库分表**：按 `product_id` 哈希分到 4~8 个库，防止单库写入热点。  
- **同步方式**：使用 **Canal** 或 **Debezium** 捕获 binlog，推送到 **Kafka**，供实时索引写入。  

### 3.2 键值/缓存数据库（Redis）  

| Key 模式 | 示例 | 用途 |
|----------|------|------|
| `search:hot_words` | `search:hot_words -> ["iPhone", "夏季连衣裙"]` | 热搜词榜单，TTL 5 min。 |
| `autocomplete:prefix:{prefix}` | `autocomplete:prefix:iph -> ["iphone 12","iphone 13"]` | 前缀联想，使用 **Sorted Set** 按热度排序。 |
| `search:result:{query}:{page}` | `search:result:iphone:1 -> JSON` | 热点查询缓存（LRU）。 |
| `product:stock:{sku_id}` | `product:stock:12345 -> 58` | 实时库存查询，秒级更新。 |

- **缓存失效策略**：热点查询使用 **TTL+LRU**，写入时 **Cache‑Aside**（先写库，再删缓存）。  
- **防止缓存击穿**：使用 **互斥锁（Redis 分布式锁）** 或 **布隆过滤器** 过滤不存在的查询。  

### 3.3 搜索引擎（Elasticsearch）索引结构  

```json
PUT product_index
{
  "settings": {
    "number_of_shards": 48,
    "number_of_replicas": 1,
    "refresh_interval": "1s",          // 实时写入
    "analysis": {
      "analyzer": {
        "ik_max_word": { "type": "ik_max_word" }   // 中文分词示例
      }
    }
  },
  "mappings": {
    "properties": {
      "product_id":   { "type": "keyword" },
      "title":        { "type": "text", "analyzer": "ik_max_word" },
      "description": { "type": "text", "analyzer": "ik_max_word" },
      "category_id":  { "type": "keyword" },
      "brand_id":     { "type": "keyword" },
      "price":        { "type": "double" },
      "stock":        { "type": "integer" },
      "status":       { "type": "keyword" },   // up/down
      "attributes": {
        "type": "nested",
        "properties": {
          "key":   { "type": "keyword" },
          "value": { "type": "keyword" }
        }
      },
      "sales":        { "type": "long" },
      "rating":       { "type": "float" },
      "create_time":  { "type": "date" },
      "update_time":  { "type": "date" }
    }
  }
}
```

- **分片数**：48 shards（每 shard 6 GB），可水平扩展到 8 台机器（每台 6 shards），**预留 1.5 倍冗余**。  
- **副本数**：1（跨 AZ），保证高可用。  
- **属性使用 `nested`**，支持 **属性过滤**（如颜色=红色）且不会出现 **跨文档匹配** 的错误。

## 第四步：核心 API 设计  

### 4.1 统一响应结构  

```json
{
  "code": 0,                     // 0 成功，非 0 为错误码
  "msg": "OK",
  "traceId": "20230527-xxxxxx",  // 链路追踪 ID
  "data": { ... }                // 业务数据
}
```

### 4.2 搜索 API  

| 方法 | 路径 | 参数（GET） | 说明 |
|------|------|------------|------|
| `GET` | `/api/v1/search` | <ul><li>`q`（必填）关键词</li><li>`category`（可选）类目ID</li><li>`brand`（可选）品牌ID</li><li>`price_min`、`price_max`（可选）价格区间</li><li>`attr_color`、`attr_size`（可选）属性过滤</li><li>`sort`（可选）`relevance`、`sales`、`price_asc`、`price_desc`、`newest`</li><li>`page`、`size`（分页）</li><li>`cursor`（可选）深度分页游标</li></ul> | 业务入口，返回 `total`, `hits[]`, `cursor`。 |
| **返回示例** | | | |
| | | | |
```json
{
  "code":0,
  "msg":"OK",
  "traceId":"20230527-12345",
  "data":{
    "total":124578,
    "page":1,
    "size":20,
    "hits":[
      {
        "product_id":"100001",
        "title":"Apple iPhone 13 128GB",
        "price":5999,
        "brand":"Apple",
        "score":12.34,
        "attributes":[{"key":"颜色","value":"蓝色"}],
        "stock":23
      }
    ],
    "cursor":"eyJwYWdlIjoxLCJzaXplIjoyMH0="   // Base64 编码的分页状态
  }
}
```

### 4.3 自动补全 API  

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| `GET` | `/api/v1/autocomplete` | `prefix`（必填）用户输入的前缀 | 返回前 10 条联想词，带热度排序。 |
| `GET` | `/api/v1/hotwords` | `size`（可选，默认 10） | 返回全站热搜榜。 |

### 4.4 实时索引刷新 API（内部调用）  

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| `POST` | `/internal/v1/product/update` | `product_id`、`op_type`（`upsert` / `delete`） | 业务后台或 Kafka 消费端调用，触发 ES 增量写入。 |
| **安全** | 只允许内部 IP / Token 鉴权。 | | |

### 4.5 错误码约定  

| Code | 含义 | 场景 |
|------|------|------|
| 0 | 成功 | — |
| 1001 | 参数错误 | 必填字段缺失、类型错误 |
| 2001 | 商品不存在 | `product_id` 不在库中 |
| 3001 | 搜索超时 | ES 响应 > 80 ms |
| 4001 | 系统内部错误 | 未捕获异常 |
| 5001 | 资源不可用 | ES 节点全部失效 |

> **为什么要统一错误码**：面试官会关注你是否考虑 **可观测性** 与 **前端友好**，不统一会导致调试成本飙升。

## 第五步：详细组件设计  

下面从 **数据流** 的角度，对每个核心组件做深入剖析，包括 **技术选型、内部实现要点、容错与监控**。

### 5.1 商品数据采集 & ETL  

1. **数据来源**  
   - 商品后台（MySQL）  
   - 第三方供应商 API（JSON）  
   - 运营后台手工 CSV 上传  

2. **采集方式**  
   - **全量导入**：使用 **Sqoop** 或 **DataX** 将 MySQL 全量 dump 到 **HDFS**。  
   - **增量捕获**：部署 **Canal**（MySQL binlog） → **Kafka topic `product_change`**。  

3. **清洗/特征抽取**  
   - **Spark Structured Streaming** 读取 `product_change`，做 **去重、字段校验、分词**（使用 `jieba` / `ik_analyzer`）。  
   - 生成 **搜索文档**（JSON）并写入 **Kafka topic `product_doc`**，供后续写入 ES。  

4. **容错**  
   - **Kafka** 保证 **至少一次** 投递，Spark 设置 **checkpoint** 到 HDFS，重启后从 checkpoint 恢复。  
   - **幂等写入**：在写入 ES 时使用 **document id = product_id**，避免重复。  

5. **监控**  
   - **Kafka lag**、**Spark job 延迟**（Prometheus + Grafana）。  
   - **异常记录**：若出现字段缺失或非法字符，写入 **DLQ（Dead Letter Queue）** 并报警。  

### 5.2 实时索引写入服务  

| 步骤 | 实现方式 | 关键点 |
|------|----------|--------|
| **消费** | Flink Job（或 Logstash）订阅 `product_doc` | **Exactly‑once** 语义（Flink Checkpoint + Kafka Transaction） |
| **转换** | 将业务字段映射到 ES Mapping（标题、属性、价格等） | 处理 **nested** 属性结构、价格字段统一为 `double` |
| **写入** | ES Bulk API（每批 5 KB / 100 条） | 设置 **`refresh_interval=1s`**，兼顾实时与写入吞吐 |
| **回写缓存** | 写入成功后，更新 Redis `product:stock:{sku_id}` | 防止缓存与 DB 不一致 |
| **错误处理** | Bulk 失败条目写入 **DLQ**，并重试 3 次 | 记录错误原因（如映射冲突） |  

> **为什么不直接在业务服务里写 ES？**  
> - **解耦**：搜索服务只负责查询，写入业务单独跑，避免写入慢导致搜索响应超时。  
> - **弹性**：实时写入可以独立扩容，峰值写入不影响查询节点。  

### 5.3 离线全量构建  

- **触发时机**：每周一次或每月一次（视业务需求），或者当增量碎片率超过阈值时。  
- **流程**  
  1. **全量抽取**：使用 **Sqoop** 将 MySQL 全量导出到 **HDFS Parquet**。  
  2. **离线计算**：Spark 读取 Parquet → **重新分词、属性聚合** → 生成 **全量 ES 文档**。  
  3. **全量写入**：先在新索引（如 `product_index_v2`）完成写入 → **切流**（使用 Alias）切换到新索引。  
- **切流策略**  
  - 使用 **ES Alias**：`product_search` → 指向当前活跃索引。  
  - 切换时 **先将新索引设置为只读**，再切换 alias，最后删除旧索引或做冷备。  

### 5.4 搜索请求处理流程  

```
[客户端] --HTTPS--> [API GW] --HTTP--> [Search Service] --gRPC--> [ES Cluster]
          |                 |                                 |
          |                 |--Cache (Redis)--> 检查热点结果   |
          |                 |                                 |
          |                 |--构造 DSL (过滤+排序)            |
          |                 |                                 |
          |                 |<--返回原始文档集合（hits）       |
          |                 |                                 |
          |                 |--业务层二次加工（加库存、价格）  |
          |                 |                                 |
          |                 |--返回统一 JSON 响应              |
```

**关键实现细节**  

1. **查询缓存**：  
   - **查询键**：`search:{q}:{filters}:{sort}:{page}`（MD5）  
   - **缓存失效**：`TTL = 30 s`（热点词会频繁更新）  
   - **写穿透**：若缓存 miss，直接查询 ES，返回后写入缓存。  

2. **过滤 & 排序实现**  
   - **过滤**：使用 **bool filter**（`term`, `range`, `nested`）保证 **filter 走倒排不参与打分**，提升性能。  
   - **排序**：  
     - 默认 **BM25**（ES 自带） → 相关度排序。  
     - 销量、价格、上新等 **基于字段排序**（`docvalue_fields`）+ **script_score** 进行二次加权（如 **综合排序 = 0.7*BM25 + 0.3*销量**）。  

3. **深度分页（游标）**  
   - 使用 **`search_after`** + **`sort`** 组合返回 **游标 token**（Base64 编码），避免 `from+size` 导致的深分页性能问题。  

4. **结果二次加工**  
   - 从 Redis 拉取 **实时库存**、**最新价格**（因为 ES 刷新间隔 1 s，仍有微小延迟）。  
   - 合并后返回给前端，确保 **库存/价格一致性**。  

### 5.5 自动补全 & 热词  

| 业务 | 实现方案 | 数据来源 | 更新频率 |
|------|----------|----------|----------|
| **前缀联想** | Redis **Sorted Set** + **Trie**（可选）| 搜索日志（用户输入）+ 商品标题 | 每 5 min（批量统计） |
| **热词榜** | Redis ZSET `search:hot_words`，Score = 最近 24h 访问次数 | 实时计数（Kafka → Flink）| 实时（每分钟刷新前 10） |
| **个性化推荐** | 简易协同过滤（用户最近 10 次搜索关键词）| 用户行为日志 | 实时（Flink）|

- **查询路径**：API → Redis → 返回 Top N。  
- **防刷**：对同一 IP/用户的同一前缀查询做 **限流**（QPS ≤ 20）并记录 **滑动窗口**。  

### 5.6 监控、日志、告警  

| 维度 | 监控指标 | 采集方式 |
|------|----------|----------|
| **系统层** | CPU、内存、磁盘 I/O、网络带宽 | Node Exporter + Prometheus |
| **业务层** | QPS、成功率、平均延迟、95/99 响应时间、错误码分布 | Spring Boot Actuator → Prometheus |
| **搜索层** | ES query latency、search threads、cache hit ratio、segment count | Elastic Exporter |
| **实时流** | Kafka lag、Flink processing time、错误率 | Prometheus + Grafana |
| **缓存层** | Redis hit/miss、eviction、CPU | Redis Exporter |
| **异常告警** | 响应时间 > 200 ms、ES 节点不可用、Kafka lag > 30 s | Alertmanager + Slack/Email |

- **链路追踪**：使用 **Jaeger**（OpenTelemetry）把从 API GW 到 ES 的每一步链路记录下来，定位慢查询根因。

## 第六步：扩展性与高可用设计  

### 6.1 水平扩展（Scale‑out）  

| 组件 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **API Gateway** | 增加实例 + L4/7 负载均衡（Nginx/LVS） | QPS、连接数 |
| **Search Service** | 增加容器/微服务实例（K8s） | CPU、内存、并发数 |
| **Elasticsearch** | 添加节点 → 自动重新分配分片 | 节点磁盘、网络、分片数 |
| **Kafka** | 增加 Broker、分区数 | 每秒写入/读取吞吐 |
| **Redis** | 主从集群 + 分片（Redis Cluster） | QPS、热点键比例 |
| **Flink** | 扩展 TaskManager 机器 | 并行度、延迟 |

> **为什么采用分片而不是单机**：搜索请求需要 **并行倒排**，单机无法满足 45 k QPS 以及 120 ms 延迟的要求。  

### 6.2 多活跨地域容灾  

1. **全局负载均衡**（GSLB）  
   - 基于 **健康检查**（API GW / Search Service）和 **网络延迟**，把用户请求路由到最近的 Region。  

2. **跨 Region 数据同步**  
   - **Kafka MirrorMaker**：把 `product_change` 同步到另一个 Region 的 Kafka。  
   - **Flink CDC** 在两边独立消费，保证 **最终一致性**（基于 product_id 的 upsert）。  

3. **灾备切换流程**  
   - **监控**检测到某 Region ES 节点全部不可用 → 自动触发 **DNS 切换** 或 **GSLB 权重改为 0**。  
   - **数据恢复**：故障恢复后，利用 **快照（snapshot）** 将新 Region 的数据回写到旧 Region，保持双向同步。  

### 6.3 高可用细节  

| 场景 | 解决方案 | 关键实现 |
|------|----------|----------|
| **单节点故障** | ES 副本、Redis 主从、Kafka 副本 | 自动故障转移（ES 选举、Redis Sentinel） |
| **网络分区** | **CAP** 取舍：搜索倾向 **可用 + 分区容错**（使用 **读写分离**） | 写入采用 **Kafka** 持久化，避免网络瞬时不可达导致数据丢失 |
| **热点爆炸** | **限流**（令牌桶）+ **热点缓存**（热点查询提前预热） | API GW + Redis 缓存热点词 |
| **部署升级** | **蓝绿部署** + **灰度发布**（Canary） | K8s Deployment + Istio 流量分配 |
| **磁盘故障** | **ES 快照**（S3 / OSS） + **磁盘 RAID10** | 每日自动快照，恢复时从快照恢复 |

### 6.4 性能调优要点  

- **倒排索引压缩**：开启 **`index.codec: best_compression`**，减少磁盘 IO。  
- **查询路由**：在 Search Service 通过 **`preference=_shards:2,3`** 将查询固定到特定分片，提升缓存命中。  
- **并发搜索**：开启 **`search.max_concurrent_shard_requests`**，控制每秒并发查询的 shard 数。  
- **冷/热分层**：将 **历史商品**（上架 > 180 天）迁移到 **成本更低的冷节点**，热节点专注于最新商品，降低热点竞争。

## 第七步：常见面试追问与回答  

### Q1️⃣ 索引更新策略：增量实时 + 离线全量，如何保证一致性？  

**回答要点**  

1. **增量实时**：  
   - 商品变更 → **Canal → Kafka → Flink** → **ES upsert**（`doc_id = product_id`）。  
   - 使用 **Kafka Transaction** + **Flink checkpoint** 实现 **Exactly‑once**，避免重复或丢失。  

2. **离线全量**：  
   - 周期性 **全量导入** 到新索引 → **Alias 切换**。  
   - 切换前，**暂停增量消费** 5 s，确保最新增量已经落库。  

3. **最终一致性**：  
   - **双写**（实时 + 离线）会产生短暂的 **“写‑写冲突”**，但因为 **ES Upsert** 采用 **最新 timestamp** 决策，冲突后保留最新状态。  
   - **监控**：对比 **实时索引的文档数** 与 **全量索引的文档数**，若差异超过阈值报警。  

### Q2️⃣ 多租户 / 类目隔离：是否需要独立索引？  

**回答要点**  

- **业务角度**：类目间搜索相关度模型差异大（服装关注属性，数码关注规格）。  
- **技术实现**：  
  - **共享倒排索引**（统一字段）+ **类目过滤**（`term` filter）可以满足大多数查询。  
  - 对于 **资源隔离**（防止单类目热点拖慢全站），可在 **ES 中使用 **`routing`** 按 `category_id` 路由，将同类目数据集中到同一分片，提升局部缓存命中。  
- **独立索引**：如果业务线 **SLA** 差异极大（如金融类目需要 10 ms），可以 **为该类目单独部署一套 ES 集群**，通过 **API Gateway** 按类目路由。  

### Q3️⃣ 搜索相关性排序模型：BM25、向量检索、学习排序，如何在 120 ms 内完成？  

**回答要点**  

1. **BM25**：默认倒排评分，查询成本最低（仅遍历倒排列表），是 **baseline**。  
2. **向量检索**（如商品图片/语义向量）  
   - 只在 **特定场景**（如“相似商品”）启用，使用 **HNSW** 索引（ES 8.x 支持）。  
   - 为保证 **120 ms**，向量检索放在 **二阶段**：先用 BM25 过滤 top‑N（如 200），再在这 200 条内部做向量打分。  

3. **学习排序（Learning to Rank, LTR）**  
   - 使用 **XGBoost / LightGBM** 训练模型，输出 **score**。  
   - **离线特征**（商品点击率、转化率） + **实时特征**（库存、价格）在 **Search Service** 进行 **特征拼装**，把特征向量发送到 **模型微服务**（TensorFlow Serving）得到最终排序分数。  
   - 为控制时延，**模型推理**在 **搜索节点本地**（模型轻量化），或使用 **缓存**（相同 query 的模型输出）。  

4. **在线评估 & 回滚**  
   - **A/B Test**：流量划分 10%（新模型） vs 90%（BM25），监控 **转化率、点击率、搜索时延**。  
   - **Canary Deploy** + **自动回滚**：若新模型的 **错误率或时延** 超阈值（如 >150 ms），立即切回旧模型。  

### Q4️⃣ 如果流量突发到 200k QPS，系统怎么应对？  

**回答要点**  

1. **预留弹性**：在架构中预留 **1.5‑2 倍** 的容量（如搜索节点 30 台），并使用 **自动伸缩（K8s HPA）**。  
2. **限流 + 熔断**：API GW 使用 **令牌桶** 对每个用户/IP 限流；后端使用 **Hystrix**/**Resilience4j** 做熔断。  
3. **热点降级**：对热点查询直接返回 **缓存**，或降级为 **仅返回 ID 列表**（不做排序），降低 CPU 消耗。  
4. **流量削峰**：使用 **消息队列（Kafka）** 将搜索请求缓冲（若后端 CPU 达到阈值），通过 **背压** 控制写入速率。  

### Q5️⃣ 数据一致性如何在搜索和库存系统之间保持？  

**回答要点**  

- **库存**：写入订单系统后，先 **写库 → 发送库存变更 Kafka** → **Flink** 计算库存 **实时写入 Redis**，**同时写入 ES**（更新 `stock` 字段）。  
- **读时**：Search Service 先从 **ES** 读取商品基本信息，再 **从 Redis** 拉取 **最新库存**，确保 **库存** 与 **搜索结果** 同步（≤5 秒延迟）。  
- **最终一致性**：如果 Redis 与 ES 不一致（极端网络分区），前端仍展示 **Redis** 的库存，后端 **后台任务** 每分钟做一次 **对账**，发现差异自动修复。

## 心得与反思  

### 1️⃣ 本题最难的 1‑2 个设计决策  

| 决策 | 挑战点 | 思考过程 |
|------|--------|----------|
| **实时索引与离线全量的协同** | 需要在 **秒级** 写入的同时，又要保证 **全量索引的一致性**，避免脏数据。 | ① 采用 **Canal → Kafka → Flink → ES Upsert** 实现 **Exactly‑once**；② 离线全量采用 **Alias 切换**，切换前短暂暂停增量流；③ 通过 **时间戳 + 版本号** 解决冲突。 |
| **跨地域多活容灾** | 业务要求 99.99% 可用，跨省网络延迟、数据同步成本大。 | ① 设计 **双活**（两个 Region 完全对等），所有业务（写、读）均通过 **全局负载均衡**；② 用 **Kafka MirrorMaker** + **Flink CDC** 实现 **双向同步**，并在每条消息上加 **全局唯一 ID** 防止重复；③ 引入 **ES Alias** + **快照**，实现快速灾备恢复。 |

### 2️⃣ 新手最容易犯的错误（至少两条）  

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把倒排索引和关系数据库混为一谈**，直接用 MySQL 做全文搜索。 | 查询性能极差，无法满足 45k QPS 与多维过滤。 | 使用 **专门的搜索引擎（Elasticsearch）**，倒排 + 属性索引；关系库仅存原始元数据。 |
| **忽视缓存层与热点问题**，所有请求直接落到 ES。 | 产生 **热点热点**，CPU/IO 飙升，导致响应超时。 | 引入 **Redis** 缓存热点查询、热词、自动补全；对热点查询使用 **Cache‑Aside** 与 **TTL**。 |
| **全局锁/单点写入**，所有商品变更走同一进程写 ES。 | 写入吞吐受限，实时性无法达标。 | **解耦写入**：使用 **消息队列** + **分布式流处理**（Flink）实现 **水平写入**。 |
| **深分页直接使用 from+size**。 | 当用户翻到后面的页码时，ES 需要遍历大量倒排列表，延迟暴增。 | 使用 **search_after** 或 **游标**（基于排序字段），避免深分页性能瓶颈。 |
