# 去中心化存储系统设计面试题（IPFS）

## 题目背景
IPFS（InterPlanetary File System）是一个基于内容寻址的点对点分布式文件系统，旨在为互联网提供去中心化的存储与内容分发。用户可以将任意文件上传至网络，系统通过哈希生成唯一的 CID（Content Identifier），并在全网节点之间复制、检索。

## 面试场景设定
> **面试官**：  
> “假设我们要在现有的云服务平台上实现一个面向企业用户的 IPFS‑like 去中心化存储服务，请你从零开始设计整体架构。包括如何接入用户、文件的写入/读取、数据持久化、以及系统的可扩展性和可靠性。先把核心思路说出来，随后我们会逐步深入细节。”

## 功能性需求
1. **文件上传**：用户通过 API 将任意大小的文件（单文件 ≤ 5 GB）上传，系统返回唯一的 CID。  
2. **文件下载**：用户提供 CID，系统在全网查找并返回文件内容，支持分块流式下载。  
3. **版本管理**：同一文件多次上传时，生成新的 CID，系统需维护历史版本关系（可通过 DAG 链接）。  
4. **存储付费 & 配额**：企业可购买不同的存储套餐（如 10 TB、100 TB），系统实时监控并限制超额写入。  
5. **节点激励 & 复写策略**：提供激励机制让自愿节点加入网络，并根据预设复制因子（默认为 3）自动分配副本。  
6. **安全与访问控制**：支持基于公钥的加密上传/下载，企业可为特定 CID 设置 ACL（只读/只写/私有）。

## 非功能性需求
| 指标 | 目标值 | 说明 |
|------|--------|------|
| **日活用户 (DAU)** | 30,000 | 企业研发人员、运维团队等 |
| **写入 QPS** | 1,200 req/s | 峰值时 2 倍峰值写入（约 2,400 req/s） |
| **读取 QPS** | 3,500 req/s | 包括 CDN 缓存命中与全网检索 |
| **平均延迟** | 写入 ≤ 800 ms（含内容哈希与分片），读取 ≤ 300 ms（缓存命中） |
| **可用性** | 99.95%（月均累计不可用时间 ≤ 21.6 分钟） |
| **总存储容量** | 150 PB（首年） | 采用冷热分层，热数据 ≤ 5 PB，其余归档至冷节点 |

## 系统边界
**本题范围内**  
- 设计 IPFS 核心协议的实现（内容寻址、DAG、块分发）。  
- 节点管理、激励模型、复制因子调度。  
- API 网关、身份认证、计费与配额控制。  
- 基础监控、日志、容错（副本、分区容错）设计。

**不考虑**  
- 具体的区块链激励代币经济模型细节（仅需抽象激励接口）。  
- 跨地域网络优化（如 BGP 优化、专线接入）。  
- 与传统对象存储（S3、OSS）兼容的网关层实现（可在扩展阶段讨论）。  
- 法律合规审计、GDPR 数据删除的实现细节。

## 提示与追问
1. **数据一致性**：如果一个文件的多个副本同时被写入不同版本，如何保证最终的 DAG 结构是有序且可回滚的？  
2. **节点失效恢复**：当某些存储节点因硬件故障下线，系统如何快速重新分配副本并保持复制因子？涉及哪些监控与调度机制？  
3. **性能瓶颈**：在写入高峰期，内容哈希计算和块切分会成为 CPU 瓶颈，你会采用哪些技术手段（如硬件加速、分布式哈希）来提升吞吐？

# 题解

# 去中心化存储系统设计面试题（IPFS）完整解答  

> **目标**：帮助零经验的后端新人，从需求拆解、最小可用系统（MVP）到高可用分布式架构，逐层、逐步、带理由地完成 **IPFS‑like 去中心化存储服务** 的系统设计。  

---  

## 解题思路总览  

1. **先把需求拆成**  
   - **功能需求**（上传、下载、版本、付费、激励、ACL）  
   - **非功能需求**（容量、QPS、延迟、可用性）  
2. **估算规模**：用户数、并发、存储量，得到 **CPU、网络、磁盘** 的粗略需求。  
3. **画出最小可用系统（MVP）**：只保留核心「内容寻址 + DAG + 简单副本」以及 API 网关、身份认证。  
4. **逐步添加**：配额、计费、ACL、激励、监控、容错、冷热分层、CDN缓存等。  
5. **每一步都要问**：  
   - *为什么要这么做？*  
   - *不这么做会出现什么问题？*  
   - *有什么可选方案？为什么最终选这个？*  

下面按章节展开，每一步都兼顾 **为什么** 与 **怎么实现**。

## 第一步：理解需求与规模估算  

### 1. 功能需求关键点  

| 功能 | 关键技术点 | 需要解决的难点 |
|------|------------|----------------|
| 文件上传 | 分块、流式写入、内容哈希（CID） | 大文件切分、哈希性能、写入顺序保证 |
| 文件下载 | CID → 节点定位 → 分块流式返回 | 低延迟检索、分块合并、容错 |
| 版本管理 | DAG（有向无环图）| 版本链的维护、回滚、冲突处理 |
| 存储付费 & 配额 | 实时计量、配额校验、计费系统 | 防止超额写入、计费精准 |
| 节点激励 & 复制因子 | 副本调度、节点注册、激励接口 | 副本数不足、节点失效恢复 |
| 安全 & ACL | 公钥加密、签名、细粒度访问控制 | 密钥管理、ACL 细粒度、审计日志 |

### 2. 非功能需求关键指标  

| 指标 | 目标 | 业务含义 |
|------|------|----------|
| DAU | 30 k | 同时在线的企业研发/运维人员 |
| 写入 QPS | 1.2k (峰值 2.4k) | 每秒最多 2,400 次文件上传请求 |
| 读取 QPS | 3.5k | 每秒最多 3,500 次下载请求 |
| 写入延迟 ≤ 800 ms | 包括分块、哈希、写入副本 | 需要并行化、硬件加速 |
| 读取延迟 ≤ 300 ms | 缓存命中 | CDN + 本地节点缓存 |
| 可用性 99.95% | 月不可用 ≤ 21.6 min | 多 AZ、自动故障转移 |
| 存储容量 150 PB (首年) | 热 5 PB，冷 145 PB | 冷热分层、不同存储介质 |

### 3. 粗略容量与并发估算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **日写入量** | 1.2k QPS × 86,400 s × 平均文件 10 MB | ≈ 1 TB / day |
| **年写入量** | 1 TB × 365 ≈ 365 TB | 约占热存储 5 PB 的 7% |
| **热数据上限** | 5 PB / 365 TB ≈ 13.7 天的热写入 | 超出后进入冷层 |
| **网络带宽** | 峰值写入 2.4k × 5 GB = 12 TB/s (理论) → 实际受限于平均文件大小，假设 10 MB → 240 Mbps ≈ 30 MB/s | 需要 10 Gbps 以上的内部骨干网 |
| **CPU 需求** | 哈希（SHA‑256）≈ 200 MB/s/核 → 5 GB 文件需要 25 s/核 → 并行 100 核可满足 800 ms | 采用多核 + SIMD + 硬件加速（GPU/FPGA） |

> **结论**：  
- **CPU** 与 **哈希** 是瓶颈，需要 **并行**、**硬件加速**。  
- **网络** 需要 **10 Gbps+** 骨干，**内部节点间** 使用 **RDMA / gRPC** 高效传输。  
- **存储** 采用 **分层**（热 SSD、冷 HDD/对象存储）并配合 **副本策略**。

## 第二步：高层架构设计  

### 1. MVP（最小可用系统）  

```
┌─────────────┐      ┌───────────────────────┐
│   API GW    │←→TLS│   Auth Service (JWT)   │
└─────┬───────┘      └───────┬───────────────┘
      │                      │
      ▼                      ▼
┌─────────────┐      ┌───────────────────────┐
│  Upload Svc │      │   Download Svc        │
└─────┬───────┘      └───────┬───────────────┘
      │                      │
      ▼                      ▼
┌─────────────────────────────────────────┐
│   Content Router (hash → node mapping) │
└─────┬───────────────────────┬───────────┘
      │                       │
   ┌──▼───┐               ┌───▼───┐
   │Node1│ … …… … …… … │NodeN│
   └──────┘               └──────┘
```

**组件解释**  

- **API Gateway**：统一入口，做流量控制、限流、日志、TLS 终端。  
- **Auth Service**：基于企业 PKI，颁发 **JWT**（包含企业 ID、配额信息）。  
- **Upload/Download Service**：处理流式分块、哈希、DAG 生成、块分发。  
- **Content Router**：使用 **分布式哈希表（DHT）**（如 Kademlia）把 CID 映射到存储节点列表（副本）。  
- **存储节点**：负责 **块存储**（块 = 固定大小 256 KB），提供 **Put/Get** RPC。  

**为什么这么简**  

- **最小化**：只保留核心功能，易于在面试中快速阐述。  
- **可扩展**：每个模块都是**无状态**的（除了 DHT），横向扩容简单。  
- **不做**：不在 MVP 加入计费、ACL、激励、冷热分层，避免“一开始就太复杂”。  

### 2. 完整企业级架构（在 MVP 基础上逐层叠加）

└───────▲───────────────────────▲───────────────────────┘
        │                       │
┌───────┴───────┐   ┌───────────┴─────────────┐
│  API Gateway  │   │   Monitoring & Alert    │
└───────▲───────┘   └───────────▲─────────────┘
        │                       │
   ┌────┴─────┐          ┌──────┴─────┐
   │ Auth Svc │          │ Billing Svc│
   └────▲─────┘          └──────▲──────┘
        │                       │
   ┌────┴─────┐          ┌──────┴───────┐
   │Upload Svc│          │Download Svc │
   └────▲─────┘          └──────▲───────┘
        │                       │
   ┌────┴─────┐          ┌──────┴───────┐
   │Chunking │          │Chunking/Cache│
   └────▲─────┘          └──────▲───────┘
        │                       │
   ┌────┴─────┐          ┌──────┴───────┐

   └────▲─────┘          └──────▲───────┘
        │                       │

        │                       │
   ┌────┴───────────────────────┴─────┐
   │         Storage Cluster          │
   │  ┌─────┬─────┬─────┬─────┬─────┐ │
   │  │Node│Node│Node│Node│Node│…│ │

**新增模块说明**  

| 模块 | 作用 | 关键技术 | 为什么要加 |
|------|------|----------|------------|
| **CDN / Edge Cache** | 靠近用户的分块缓存，降低读取延迟 | HTTP/2、对象缓存、Geo‑DNS | 达到 300 ms 读取目标 |
| **Billing Service** | 实时计费、配额扣减、套餐管理 | 流式计费、Redis+Kafka 计数器 | 防止企业超额写入、提供账单 |
| **Monitoring & Alert** | 指标采集、告警、自动伸缩 | Prometheus + Alertmanager + Grafana | 满足 99.95% 可用性 SLA |
| **激励接口** | 与外部区块链/代币系统对接，记录贡献 | gRPC 接口、签名校验 | 促进自愿节点加入 |
| **冷热分层管理** | 将不活跃块迁移到成本更低的冷节点 | 阶段性迁移任务、分层存储策略 | 控制存储成本，满足 150 PB 目标 |
| **ACL Service** | 对 CID 进行细粒度访问控制 | 基于公钥的签名 + 权限表 (RBAC) | 满足企业安全需求 |
| **DAG Service** | 维护文件‑块‑版本之间的有向无环图 | Merkle‑DAG、IPLD | 支持版本回滚、历史查询 |
| **Chunking / Hash Service** | 并行切块、硬件加速哈希 | SIMD、GPU、FPGA、线程池 | 解决写入 CPU 瓶颈 |

## 第三步：数据库设计  

> **原则**：业务元数据（用户、配额、CID ↔ 块映射、版本 DAG）使用 **强一致性** 数据库；块本身采用 **对象存储**（分布式文件系统）。  

### 1. 关系型/文档型元数据库  

| 表/集合 | 主键 | 关键字段 | 备注 |
|--------|------|----------|------|
| **users** | `user_id` (UUID) | `org_id`, `public_key`, `plan_id`, `created_at` | 企业/用户信息 |
| **plans** | `plan_id` | `name`, `storage_quota_bytes`, `price_per_gb` | 套餐定义 |
| **buckets** (可选) | `bucket_id` | `org_id`, `name`, `acl_policy` | 多租户分区 |
| **files** | `cid` (CID) | `owner_org_id`, `size_bytes`, `created_at`, `dag_root_cid`, `current_version` | 文件元信息 |
| **file_versions** | `(cid, version)` | `parent_cid`, `block_cids[]`, `timestamp` | 版本链 |
| **blocks** | `block_cid` | `size`, `node_ids[]`, `replication_factor`, `created_at` | 块元数据，指向存储节点 |
| **node_status** | `node_id` | `ip`, `region`, `capacity_bytes`, `used_bytes`, `online`, `last_heartbeat` | 节点管理 |
| **billing** | `record_id` | `org_id`, `action`(upload/download), `bytes`, `timestamp` | 计费流水 |
| **acl_rules** | `rule_id` | `cid`, `principal`(org/user/pubkey), `perm`(read/write), `expires_at` | 细粒度 ACL |

**为什么要分离**  

- **块数据量巨大**，不适合放在关系库，只保存 **元数据**（块所在节点、大小、复制因子）。  
- **文件元数据** 需要强一致性（防止配额超支），使用 **PostgreSQL** 或 **CockroachDB**（分布式强一致）  
- **计费/日志** 采用 **Append‑only** 的 **ClickHouse**/**TimescaleDB**，便于 OLAP 分析。  

### 2. 存储节点块持久化  

- 每个节点本地使用 **分布式对象存储**（如 **Ceph RADOS**、**MinIO**）或 **本地 SSD/HDD**，块文件名 = `block_cid`（二进制）。  
- 采用 **分层存储**：热节点使用 **NVMe SSD**，冷节点使用 **HDD**，块在迁移时更新 `node_status` 表的 `node_ids[]`。  

### 3. 数据一致性方案  

| 场景 | 一致性需求 | 方案 |
|------|------------|------|
| **配额检查 + 写入** | 强一致（防止超额） | 在 **Upload Service** 中先 **事务性** 检查 `org.quota_used + new_size ≤ quota_limit`，使用 **分布式锁**（Redis RedLock）或 **两阶段提交** |
| **块副本注册** | 最终一致 | 写入 `blocks` 表后异步触发 **副本调度**，若副本不足由 **调度器** 补齐 |
| **版本 DAG 更新** | 强一致 | **文件表** `files` 与 `file_versions` 同一事务提交，确保 DAG 链完整 |
| **节点失效恢复** | 最终一致 | 监控发现节点失联后，调度器重新分配副本，更新 `blocks` 表的 `node_ids[]`（异步） |

## 第四步：核心 API 设计  

> **原则**：RESTful + gRPC 双协议。REST 供外部企业调用，gRPC 供内部服务高效通信。  

### 1. 认证与授权  

| 方法 | 路径 | 请求头 | 响应 | 说明 |
|------|------|--------|------|------|
| POST | `/auth/token` | `X-Public-Key: <pubkey>`<br>`X-Signature: <sig of timestamp>` | `{access_token, expires_in}` | 使用企业 **公钥签名** 登录，返回 **JWT**（内含 org_id、配额信息） |
| GET  | `/auth/me` | `Authorization: Bearer <jwt>` | `{org_id, plan, quota_used, quota_limit}` | JWT 校验后返回用户信息 |

### 2. 文件上传  

| 方法 | 路径 | 请求 | 响应 | 备注 |
|------|------|------|------|------|
| POST | `/files` | **流式** `multipart/form-data` 或 **gRPC** `UploadRequest`（分块）<br>Header: `Authorization`、`X-File-Name`、`X-File-Size` | `{cid, version, dag_root_cid}` | **分块**：客户端先把文件切块（256 KB），每块带 `block_index`，服务端返回 `block_cid` 列表，最终生成 `CID` |
| GET  | `/files/{cid}/status` | — | `{status: "COMPLETED|PENDING", uploaded_blocks, total_blocks}` | 客户端轮询或使用 **WebSocket** 推送进度 |

**实现细节**  

1. **客户端**：使用 **并行管道**（如 `ThreadPoolExecutor`），每个块调用 **Upload Service** 的 `PutBlock`（gRPC），并在本地计算 **SHA‑256**（或 **Blake3**）得到 `block_cid`。  
2. **服务端**：收到块后 **验证签名**（防篡改），将块写入本地节点磁盘并 **返回确认**。  
3. **块登记**：写入 `blocks` 表，记录 `node_id`、`replication_factor`。  
4. **完成**：所有块上传成功后，**DAG Service** 把块 CIDs 按顺序组织成 **Merkle‑DAG**，根哈希即为文件 **CID**，写入 `files`、`file_versions`。  

### 3. 文件下载  

| 方法 | 路径 | 请求 | 响应 | 备注 |
|------|------|------|------|------|
| GET | `/files/{cid}` | Header: `Range: bytes=0-`（支持断点）<br>`Authorization` | **流式**文件内容（分块） | 服务端先查询 **DHT** 找到包含块的节点列表，按顺序拉取块，合并返回 |
| GET | `/files/{cid}/dag` | — | `{dag_root_cid, block_cids[]}` | 用于 **客户端自行组装**（大文件、带宽受限场景） |

**实现细节**  

- **内容路由**：`Content Router` 根据 `cid` → **块列表** → **节点列表**（优先最近、健康节点）。  
- **并行下载**：客户端可并行请求多个块（最多 8 条并发），提升吞吐。  
- **缓存**：热点块写入 **CDN Edge**，边缘节点直接返回。  

### 4. 版本管理  

| 方法 | 路径 | 请求 | 响应 |
|------|------|------|------|
| GET | `/files/{cid}/versions` | — | `{versions: [{version, cid, timestamp}]}` |
| POST| `/files/{cid}/rollback` | `{target_version}` | `{new_cid, new_version}` |

- **回滚**：在 `file_versions` 表中创建新记录，指向旧的 `block_cids`，生成新 `CID`（不覆盖历史）。  

### 5. 配额查询  

| 方法 | 路径 | 请求 | 响应 |
|------|------|------|------|
| GET | `/orgs/{org_id}/quota` | — | `{used_bytes, limit_bytes}` |

- **实时**：计费服务每次写入后立即更新 `quota_used`（使用 **Redis** 原子递增），同步落库。  

### 6. ACL 管理  

| 方法 | 路径 | 请求 | 响应 |
|------|------|------|------|
| POST | `/acls` | `{cid, principal, perm, expires_at}` | `{rule_id}` |
| DELETE| `/acls/{rule_id}` | — | `200 OK` |

- **校验**：每次下载前，`Download Service` 调用 **ACL Service** 验证 JWT 中的 `org_id` 与规则匹配。

## 第五步：详细组件设计  

### 1. Upload Service  

| 子模块 | 功能 | 技术实现 | 关键指标 |
|--------|------|----------|----------|
| **Chunker** | 将上传流切成 256 KB 块 | 客户端 SDK（Go/Java）使用 **goroutine** 并行切块；服务端使用 **netty** 流式处理 | 每块 0.5 ms 切分（CPU SIMD） |
| **Hash Engine** | 计算块 CID（SHA‑256/Blake3） | **硬件加速**：GPU (CUDA) 或 **FPGA**，也可使用 **SIMD**（AVX2）| 200 MB/s/核，4 核即可满足 800 ms 目标 |
| **Block Store API** | 把块写入本地节点 & 复制 | **gRPC** `PutBlock(block_cid, data)`；内部采用 **Raft** 或 **Paxos** 保证副本写成功 | **写入延迟 ≤ 300 ms**（含网络） |
| **DAG Builder** | 生成 Merkle‑DAG、根 CID | 按块顺序构建 **IPLD**（Merkle‑Tree），根哈希即 CID | **CPU 计算 ≤ 100 ms** |
| **Metadata Writer** | 写入 `files`、`file_versions`、`blocks` 表 | 使用 **CockroachDB** 事务 | **事务时延 ≤ 50 ms** |
| **Quota Enforcer** | 检查并扣减配额 | **Redis** 原子 `INCRBY`，失败回滚 | **并发安全** |

**流程图**  

```
Client → Upload Svc (Chunker + Hash) → Block Store (PutBlock) → DAG Builder → Metadata Writer → Response(CID)
```

### 2. Download Service  

| 子模块 | 功能 | 技术实现 | 关键指标 |
|--------|------|----------|----------|
| **CID Resolver** | 从 CID 查询块列表 | 查询 `blocks` 表 + DHT | ≤ 5 ms |
| **Node Selector** | 选取最近、健康的节点 | **Latency‑based** + **负载均衡**（Consistent Hash） | ≤ 10 ms |
| **Parallel Fetcher** | 并行拉取块 | **gRPC** `GetBlock(block_cid)`，使用 **协程**/线程池 | 每块 2 ms（网络+磁盘） |
| **Reassembly** | 按顺序拼接块、校验根哈希 | **SHA‑256** 校验 | ≤ 5 ms |
| **Rate Limiter / ACL Check** | 访问控制 & 限流 | JWT → ACL Service → Token Bucket | 保障安全 & 防刷 |

### 3. DHT（内容寻址路由）  

- 采用 **Kademlia** 风格的分布式哈希表。  
- **节点 ID** = `SHA‑256(node_public_key)`，保证随机性。  
- **存储**：每个块的 **primary node** 负责在 DHT 中登记 `block_cid → node_id[]`（副本列表）。  
- **查询路径**：最多 3‑5 跳即可定位到拥有副本的节点。  

**为什么用 Kademlia**  

- **去中心化**：符合 IPFS 的设计哲学。  
- **高容错**：即使部分节点失效，仍能通过 **路由表** 找到副本。  
- **可伸缩**：节点增删不影响全局结构。  

### 4. 副本调度器  

| 步骤 | 触发条件 | 动作 |
|------|----------|------|
| **节点上线** | 心跳检测 `online = true` | 重新评估当前块的副本分布，若有 **低于复制因子** 的块，调度 **复制任务** |
| **节点下线** | 心跳超时（>30 s） | 标记为 **offline**，触发 **恢复任务**，寻找其他节点补齐副本 |
| **容量阈值** | 节点 `used/total > 80%` | 将部分块迁移至空闲节点，更新 `blocks.node_ids` |
| **冷热分层** | 块最近访问时间 > 30 天 | 将块移动到 **冷节点**（HDD），复制因子保持不变 |

- **实现**：使用 **Kafka** 事件流，**Scheduler Service** 读取 `node_status` 与 `blocks` 表，生成 **复制/迁移任务**，交给 **Worker**（内部 gRPC）执行。  

### 5. 计费 & 配额系统  

- **实时计费**：每次 **Upload** 调用 `quota_enforcer` 前，先写入 **Kafka** `billing_events`（topic: `upload`），**Billing Service** 订阅后计算费用并更新 **Redis** `org_quota_used`。  
- **配额检查**：在 **Upload Service** 中 **原子** `GETSET`（Redis）+ **Lua 脚本** 确保不超额。  
- **账单查询**：`/orgs/{org_id}/billing` → 从 **ClickHouse** 拉取历史记录，生成 PDF。  

### 6. 监控、日志、告警  

| 维度 | 采集方式 | 关键指标 |
|------|----------|----------|
| **业务** | Prometheus Exporter（Upload/Download） | QPS、成功率、平均延迟、错误码 |
| **系统** | Node Exporter、cAdvisor | CPU、内存、磁盘 I/O、网络 |
| **链路** | OpenTelemetry (Tracing) | 请求路径、跨服务时延 |
| **日志** | Fluent Bit → Elasticsearch | 请求日志、错误日志、审计日志 |
| **告警** | Alertmanager | QPS > 阈值、节点离线、复制因子下降 |

## 第六步：扩展性与高可用设计  

### 1. 横向扩容  

| 组件 | 扩容方式 | 关键点 |
|------|----------|--------|
| **API Gateway** | 增加实例 + **L4**（NGINX/Envoy）负载均衡 | 支持 **TLS 终端**，统一限流 |
| **Auth Service** | 无状态，水平扩容 | 使用 **Redis** 共享会话/撤销列表 |
| **Upload/Download Service** | **Stateless**，使用 **消息队列**（Kafka）做异步任务 | 通过 **分区** 实现请求路由 |
| **DHT 节点** | 新增节点自动加入，Kademlia 自动更新路由表 | 需要 **节点健康检测** |
| **Storage Nodes** | 增加磁盘/节点，**CRUSH**（如 Ceph）重新分配块 | **数据再平衡** 需要后台任务 |
| **Scheduler** | 多实例 + Leader Election（ZooKeeper/etcd）避免冲突 | 任务去重、幂等执行 |

### 2. 容错与故障恢复  

| 故障类型 | 检测方式 | 恢复措施 |
|----------|----------|----------|
| **节点宕机** | 心跳 (Redis/ZooKeeper) 超时 | 调度器重新复制缺失副本 |
| **网络分区** | 多点 Ping、BGP 路由监控 | 暂停对该分区的写入，等恢复后同步 |
| **存储磁盘故障** | 磁盘 SMART、块错误率 | 自动迁移块至其他节点，标记坏块 |
| **服务实例崩溃** | Kubernetes Liveness Probe | 自动重启 Pod，流量自动切换 |
| **复制因子下降** | 定时任务检查 `blocks.replication_factor` | 触发 **补副本** 任务 |

### 3. 冷热分层策略  

1. **热层**（SSD）  
   - 最近 7 天访问的块自动保留在热节点。  
   - 使用 **LRU** + **访问计数**（Redis）决定热度。  

2. **冷层**（HDD / 归档对象存储）  
   - 超过 30 天未访问的块迁移至冷层，副本数可降至 2。  
   - 冷层节点使用 **低功耗**、**大容量**硬盘，成本更低。  

3. **冷热切换**  
   - **迁移任务**由 **Scheduler** 每日触发，使用 **copy‑on‑write**，保证迁移期间副本不低于 3。  

### 4. CDN 与边缘缓存  

- **热点文件**（如公司内部文档）自动发布到 **CDN**，使用 **Cache‑Control** 头部控制 TTL。  
- **Edge Node** 同时作为 **存储节点**（写入同步），读取时优先从最近的 Edge 拉取。  

### 5. 安全加固  

| 维度 | 措施 |
|------|------|
| **传输** | 全链路 TLS 1.3，双向认证（客户端证书） |
| **存储** | 块在写入前使用 **AES‑256-GCM** 加密，密钥由 **KMS**（云密钥管理）托管 |
| **访问控制** | JWT + ACL Service，最小权限原则 |
| **审计** | 所有上传/下载操作写入 **审计日志**（ELK）并保留 1 年 |
| **灾备** | 多地域（至少 3 AZ）复制，跨 Region 复制因子为 2（跨区副本） |

## 第七步：常见面试追问与回答  

### Q1️⃣ 数据一致性：**“如果一个文件的多个副本同时被写入不同版本，如何保证最终的 DAG 结构是有序且可回滚的？”**  

**回答要点**  

1. **写入流程采用** **两阶段提交（2PC）**：  
   - **Prepare 阶段**：所有目标副本节点收到块后 **暂存**（写入临时文件），返回 `prepared`。  
   - **Commit 阶段**：Coordinator（Upload Service）在所有 `prepared` 成功后，发送 `commit`，节点正式移动到正式存储区并更新 `blocks` 表。  
2. **版本号（Monotonic）**：每个 **org** 拥有全局递增的 **Version Counter**（存于 **CockroachDB**），写入时先获取 `next_version`，确保 **线性**。  
3. **冲突检测**：若同一时间两次 `Prepare` 获得相同 `version`，只有 **一个** 能成功 `Commit`，另一个返回 **冲突错误**，客户端重新发起（指数退避）。  
4. **DAG 链**：`file_versions` 表记录 `parent_cid`，形成 **链式 DAG**；回滚时只需指向历史 `cid`，不修改已有块。  

**如果不这样做**：可能出现 **写入覆盖**、**版本混乱**，导致文件内容不确定，用户无法回滚。  

### Q2️⃣ 节点失效恢复：**“当某些存储节点因硬件故障下线，系统如何快速重新分配副本并保持复制因子？”**  

**回答要点**  

1. **心跳监控**：每个节点每 5 s 上报 `heartbeat` 到 **etcd**。失联 > 30 s 触发 **故障事件**。  
2. **故障事件写入 Kafka** → **Scheduler Service** 读取。  
3. **调度器算法**：  
   - **查询** `blocks` 表，找出受影响的块 `replication_factor < target`。  
   - **选取** 目标节点：满足 **磁盘剩余 ≥ block_size**、**网络延迟低**、**地域分散**。  
   - **创建复制任务**，发送 **CopyBlock** RPC 给源节点，将块复制到目标节点。  
4. **复制完成后**：原子更新 `blocks.node_ids[]`，并在 **DHT** 重新登记。  
5. **幂等性**：复制任务使用 **唯一任务 ID**，重复消费不会导致多副本。  

**若不做**：复制因子下降，导致 **数据可靠性** 降低，可能出现 **单点丢失**。  

### Q3️⃣ 性能瓶颈：**“在写入高峰期，内容哈希计算和块切分会成为 CPU 瓶颈，你会采用哪些技术手段提升吞吐？”**  

**回答要点**  

| 手段 | 实现方式 | 预期提升 |
|------|----------|----------|
| **硬件加速** | - **GPU**（CUDA）执行 SHA‑256 并行<br>- **FPGA** 专用哈希加速卡 | 5‑10× 计算吞吐 |
| **SIMD 向量化** | 使用 **AVX2/AVX‑512** 实现块级并行哈希 | 单核 2‑3× |
| **分布式哈希服务** | 将哈希任务拆分到 **Hash Workers**（独立微服务），采用 **Kafka** 分片 | 横向扩展，线性提升 |
| **零拷贝切分** | 使用 **mmap** + **splice** 系统调用，避免用户态拷贝 | 减少 CPU 负载、提升 I/O |
| **批量写入** | 将多个块聚合后一次性写入磁盘（SSD） | 降低磁盘 IOPS，提升写入速率 |
| **流式压缩+加密** | 在切块前先 **压缩**（LZ4）再 **哈希**，减少块大小 | 减少网络传输、块数 |

**权衡**：硬件加速需要额外成本，适合 **峰值** 场景；分布式哈希服务可以在 **云** 环境弹性伸缩，成本更灵活。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **副本调度与故障恢复** | 需要在 **强一致性**（配额、版本）和 **最终一致性**（块副本）之间找到平衡。 | - 首先确定哪些数据必须强一致（配额、版本链），使用事务型数据库。<br>- 其余块副本采用 **异步复制**，通过 **调度器** 监控复制因子。<br>- 设计 **故障检测**（心跳）和 **补副本** 流程，确保即使在网络分区也能逐步恢复。 |
| **版本化 Merkle‑DAG 与回滚** | 如何在去中心化环境下保持 **可回溯**、**不覆盖** 的历史，同时不产生巨大的存储开销。 | - 采用 **IPLD**（内容可寻址的 Merkle‑DAG），每一次上传生成新的根 CID。<br>- 将 **版本链** 记录在 `file_versions` 表中，父指针指向上一个 CID，形成 **链式 DAG**。<br>- 回滚时只生成新根 CID 指向旧块，不删除旧块，保持 **不可变性**。<br>- 通过 **复制因子** 保证旧块的持久性。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **把块数据直接放进关系数据库**  
   - **后果**：磁盘空间、I/O 成本爆炸，查询慢，难以横向扩容。  
   - **正确做法**：块存储在 **对象存储**（本地或云），数据库仅保存 **元数据**（块 CID → 节点映射）。  

2. **忽视配额的强一致性**  
   - **后果**：在高并发写入时，配额可能被超支，导致计费、业务冲突。  
   - **正确做法**：在写入前使用 **分布式锁** 或 **Redis Lua 脚本** 原子检查并扣减配额；配额变更持久化到强一致的数据库。
