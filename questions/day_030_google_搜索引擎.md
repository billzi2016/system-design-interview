# 第 30 天：设计 Google 搜索引擎

> 生成日期：2026-04-26

---

## 1. 题目背景  
Google 搜索引擎是全球最大、最常用的网页检索系统，用户输入关键词后返回相关网页、图片、视频等结果。面试中需要你从零开始设计一个能够支撑海量数据、高并发查询的搜索系统（核心检索与排序功能）。

## 2. 面试场景设定  
> **面试官**：  
> “假设我们要为全球用户重新设计一套类似 Google 的搜索引擎，要求能够每天处理数十亿次查询，并在毫秒级返回高质量的搜索结果。请从系统结构、数据流、关键技术选型等角度完整地描述你的设计思路。”  

（随后可让候选人自行划分模块、说明每层职责等）

## 3. 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| F1   | **关键词检索**：用户输入任意长度的关键词，系统返回匹配的网页列表。 |
| F2   | **搜索结果排序**：基于 PageRank、语义相关性、用户行为等多维度对结果进行实时排序。 |
| F3   | **多媒体检索**：支持图片、视频、新闻等垂直搜索，返回对应类型的结果。 |
| F4   | **自动纠错 & 推荐**：对拼写错误进行纠正，提供 “Did you mean” 与相关查询建议。 |
| F5   | **地域/语言本地化**：根据用户 IP 与语言偏好返回对应地区、语言的结果。 |
| F6   | **索引更新**：支持每日数十TB 新增/修改网页的增量索引，保证 24h 内收录。 |

## 4. 非功能性需求  

| 编号 | 指标 | 估算值 | 备注 |
|------|------|--------|------|
| N1   | **每日活跃用户 (DAU)** | 5 × 10⁸ 人 | 按全球用户基数估算 |
| N2   | **查询吞吐量 (QPS)** | 2 × 10⁵ QPS 峰值 | 高峰时段约 200k 次/秒 |
| N3   | **查询响应时延** | 99% ≤ 200 ms，99.9% ≤ 500 ms | 包括网络+检索+排序 |
| N4   | **系统可用性** | ≥ 99.99%（年度累计停机 ≤ 52 分钟） | 采用多活+灾备 |
| N5   **存储规模** | **网页原始数据**：≈ 100 PB<br>**倒排索引**：≈ 30 PB | 按每天 20 TB 新增、10 年保留计算 |
| N6   | **扩展性** | 支持水平扩容至 10× 当前容量，且扩容时间 ≤ 2 小时 | 采用无中心化设计 |

## 5. 系统边界  

**本题需要设计并讨论的范围**  
- 用户查询入口、负载均衡、搜索请求路由  
- 爬虫系统的概念设计（不要求实现细节）  
- 文档存储、倒排索引结构、分布式索引分片与合并  
- 排序模型（PageRank、学习排序）与实时计算框架  
- 缓存层（热点查询缓存、结果缓存）  
- 高可用与灾备方案、监控告警指标  

**不在本题范围（可略过）**  
- 浏览器端 UI/前端交互细节  
- 广告投放系统、商业化模型  
- 语音搜索、自然语言对话等高级交互  
- 法律合规、隐私保护的具体实现（仅需说明会考虑）  
- 硬件层面的网络拓扑、机房选址细节  

## 6. 提示与追问  

1. **索引分片与路由**  
   - “如果倒排索引按照 term → doc 列表进行分片，如何保证查询时只访问少量分片并且避免热点分片？”  

2. **排序与机器学习**  
   - “在实时排序阶段，你会怎样设计特征存储与特征计算流水线，以支持千级 QPS 的学习排序模型？”  

3. **灾备与容错**  
   - “当一个数据中心失联时，如何保证查询仍能在 500 ms 内返回近似正确的结果？”  

（面试官可根据候选人回答进一步追问数据同步、一致性模型、缓存失效策略等细节。）

# 题解

## 解题思路总览  

1. **从最小可用系统（MVP）出发**——先把“用户输入关键词 → 返回匹配文档列表”做到。  
2. **逐步加入非功能需求**：高并发、低延迟、可扩展、容灾、排序、纠错等。  
3. **每加入一个新特性，都要问三件事**  
   - *为什么需要它*（业务价值、风险）？  
   - *怎么实现*（技术选型、数据流）？  
   - *不这么做会有什么后果*？  
4. **模块化思考**：把系统划分为 **入口层、路由层、检索层、排序层、缓存层、存储层、监控/运维层**，每层只负责自己的职责，降低耦合。  
5. **把抽象的概念落到具体的实现细节**：比如倒排索引怎么分片、怎样做热点分片均衡、特征如何实时计算、容灾如何做到 99.99% 可用。  

下面我们按 **需求 → 规模 → 架构 → 细节** 的顺序，层层递进，帮助零经验的同学把“从 0 到有” 的完整思考过程串起来。

## 第一步：理解需求与规模估算  

### 1. 功能性需求梳理  

| 编号 | 关键点 | 实现要点 |
|------|--------|----------|
| **F1** | 关键词检索 | **倒排索引**（term → doc list）是核心。需要支持多词、布尔、短语、前缀等查询。 |
| **F2** | 结果排序 | 多维特征：PageRank、向量相似度、点击率、地域、实时热点。需要 **离线排序模型 + 在线学习排序**。 |
| **F3** | 多媒体检索 | 为图片、视频、新闻分别建立专属倒排索引（视觉特征、元数据）。查询入口统一，后端按垂直分流。 |
| **F4** | 自动纠错 & 推荐 | 拼写纠错、同义词扩展、查询建议。实现方式：**Trie/DAWG** + **基于查询日志的语言模型**。 |
| **F5** | 本地化 | IP → 地域、语言首选；在检索时加 **过滤/加权**。 |
| **F6** | 索引更新 | **增量抓取 → 增量索引 → 合并**，保证 24h 内收录。 |

### 2. 非功能性需求转化为数字  

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| **DAU** | 5×10⁸ | 直接给出。 |
| **峰值 QPS** | 2×10⁵ | 200 k QPS ≈ 1.2 B/天。 |
| **单次查询时延** | 99% ≤ 200 ms, 99.9% ≤ 500 ms | 包括网络+业务逻辑。 |
| **原始网页存储** | 100 PB | 10 年 × 10 TB/天 ≈ 36 PB；留有冗余、快照等。 |
| **倒排索引大小** | 30 PB | 按 1 TB/10 ⁹ 条 doc 计，实际压缩后约 30% 原始大小。 |
| **可用性** | 99.99% → 年停机 ≤ 52 min | 多活+灾备。 |
| **水平扩容** | 10× | 必须是 **无中心化**（无单点）设计。 |

### 3. 规模换算（帮助后面选型）  

| 项目 | 估算值 | 备注 |
|------|--------|------|
| **每日查询数** | 200 k QPS × 86 400 s ≈ 1.73 × 10¹⁰ 次 | 约 17 B 次/天 |
| **查询字数** | 平均 3‑5 词 | 每词 10 bytes → 50 bytes/查询 |
| **查询流量** | 1.73 × 10¹⁰ × 50 B ≈ 850 GB/天 ≈ 10 Gbps | 网络带宽必须预留 2‑3 倍冗余 |
| **索引写入** | 每天 20 TB 新增文档 | 增量索引写入吞吐 ≈ 250 MB/s |
| **查询并发** | 峰值 200 k QPS → 200 k 同时查询 | 每台机器假设能处理 2 k QPS，需要 **≥100 台** 计算节点（粗略） |

## 第二步：高层架构设计  

### 1. 从 MVP 到完整系统的层次划分  

```
┌───────────────────────┐
│   1. 客户端 / CDN / DNS │
└─────┬─────────────────┘
      │
┌─────▼─────┐
│ 2. 负载均衡 │（L4/L7）
└─────┬─────┘
      │
┌─────▼─────────────────────┐
│ 3. API 网关 / 请求路由层   │
│   - 入口鉴权、限流、日志   │
│   - 路由到对应的检索集群   │
└─────┬─────────────────────┘
      │
┌─────▼─────────────────────┐
│ 4. 检索层（倒排索引服务）   │
│   - 分片路由、并行检索      │
│   - 结果合并、去重         │
└─────┬─────────────────────┘
      │
┌─────▼─────────────────────┐
│ 5. 排序层（学习排序）      │
│   - 离线特征/模型加载       │
│   - 实时特征计算（点击、热点）│
│   - 打分 & Top‑K           │
└─────┬─────────────────────┘
      │
┌─────▼─────────────────────┐
│ 6. 缓存层（热点查询缓存）   │
│   - 本地 LRU/LFU           │
│   - 分布式缓存（Redis/Memcached）│
└─────┬─────────────────────┘
      │
┌─────▼─────────────────────┐
│ 7. 存储层                     │
│   - 原始文档存储（对象存储）   │
│   - 倒排索引存储（分布式文件系统）│
│   - 元数据/特征库（NoSQL）      │
└─────────────────────────────┘
```

> **为什么要这么层层拆分？**  
> - **单一职责**：每层只关注自己能做的事，便于独立扩容和故障隔离。  
> - **并行化**：检索层可以在多台机器上并行取倒排列表，排序层再并行打分。  
> - **缓存命中率提升**：热点查询在进入检索层前就能被拦截，极大降低后端压力。  
> - **容灾**：每层都可以部署多活副本，单点故障不影响整体可用性。

### 2. 关键技术选型（解释原因）  

| 模块 | 备选技术 | 推荐技术 | 选型理由 |
|------|----------|----------|----------|
| **负载均衡** | LVS, HAProxy, Nginx, Envoy | **Envoy**（L7） | 支持动态路由、灰度发布、统一的 telemetry，便于后续服务网格化。 |
| **API 网关/路由** | Kong, Zuul, Spring Cloud Gateway | **Kong + 自研路由插件** | 插件化、插件生态丰富；可以把查询路由、限流、鉴权统一管理。 |
| **倒排索引服务** | Elasticsearch, Solr, 自研 Lucene‑style | **自研基于 Lucene 的分布式倒排索引** | 为了满足 **PB 级数据、毫秒级响应**，需要自行控制分片、压缩、查询并行度。 |
| **排序计算** | Spark‑ML, TensorFlow Serving, 自研在线特征计算 | **自研特征服务 + TensorFlow Serving** | 离线模型大规模训练使用 Spark；在线推理使用 TF Serving，提供低延迟 RPC。 |
| **分布式缓存** | Redis Cluster, Aerospike, TiKV | **Redis Cluster** + **本地 LRU** | 生态成熟、支持持久化、读写分离；本地缓存进一步降低网络 RTT。 |
| **对象存储** | Ceph, MinIO, 商业云 OSS | **Ceph / 商业云 OSS** | 支持 PB 级、分片冗余、统一命名空间。 |
| **分布式文件系统（索引）** | HDFS, Apache Parquet + S3, 自研 Columnar FS | **HDFS + Columnar 编码（ORC/Parquet）** | 适合顺序写、批量压缩；支持 MapReduce 式的离线合并。 |
| **监控/告警** | Prometheus + Grafana, OpenTelemetry | **Prometheus + Alertmanager + OpenTelemetry** | 可观测性全链路覆盖（latency, QPS, cache hit, replica lag）。 |

## 第三步：数据库设计  

### 1. 主要存储实体  

| 实体 | 存储方式 | 关键字段 | 备注 |
|------|----------|----------|------|
| **文档（网页/图片/视频）** | 对象存储（Blob） + 元数据表（NoSQL） | `doc_id, url, title, content_hash, crawl_ts, language, region, media_type` | 文档本体大块存放，元数据用于快速过滤。 |
| **倒排索引** | 分布式文件系统（列式压缩） | `term → posting_list`（doc_id, tf, positions, field） | 按 term 哈希分片，每片 10‑100 GB，压缩后 30% 左右。 |
| **Term Dictionary** | KV Store（RocksDB/LevelDB） | `term → term_id, df, shard_id` | 词典用于 term → shard 快速定位。 |
| **特征库** | NoSQL（Cassandra/ScyllaDB） | `doc_id → {pagerank, url_score, click_rate, freshness}` | 离线特征，供排序层读取。 |
| **实时特征** | 流处理状态存储（Apache Flink State Backend） | `doc_id → {current_qps, recent_clicks}` | 每分钟刷新一次，用于实时加权。 |
| **查询日志 & 推荐模型** | 大数据平台（Hive + Spark） | `query_id, user_id, ip, timestamp, clicked_doc_id` | 用于离线模型训练、拼写纠错词典更新。 |

### 2. 索引分片策略  

1. **Term‑Hash 分片**：  
   - 对每个 term 计算 **64‑bit hash** → `hash % N_SHARDS`，得到所在倒排分片。  
   - 这样查询时只需 **路由到对应的 N_SHARDS**（通常是 1‑3 个分片），避免全库扫描。  

2. **热点分片均衡**：  
   - **热点 term**（如 “news”, “weather”）会导致某些分片请求量过大。  
   - 采用 **“分片复制 + 负载均衡路由”**：同一个 term 的 posting list 在多个节点上保持 **副本**（Primary + N‑1 Replicas），查询入口使用 **一致性哈希 + 动态权重** 把请求分散到副本。  
   - 当某个副本负载高时，路由器会自动把请求转到负载较低的副本，保证 **热点分片不成为瓶颈**。  

3. **文档‑层面分片（DocID）**：  
   - 每个 posting list 内的 `doc_id` 按 **DocID 范围** 再细分为子块（Block），块大小 4‑8 KB，便于 **局部压缩、跳表定位**。  

> **不做分片会有什么问题？**  
> - 单机查询需要遍历全部倒排列表，磁盘 IO 成为天堑，响应时间达秒级。  
> - 任何一次硬件故障都会导致全部数据不可用，违背 99.99% 可用性。  

### 3. 数据一致性模型  

| 操作 | 一致性要求 | 实现方式 |
|------|------------|----------|
| **查询** | **最终一致性**（用户可以接受几分钟的索引延迟） | 读取最近可用副本，若副本落后则返回旧结果，后台同步。 |
| **索引写入（增量）** | **强一致性**（写入成功后必须能被查询到） | 采用 **两阶段提交**：① 写入 Primary → 同步写入多数 Replicas；② 完成后返回成功。 |
| **实时特征更新** | **近实时**（分钟级） | Flink/Storm **Exactly‑once** 状态更新，写入 NoSQL，使用 **乐观锁** 防止写冲突。 |

## 第四步：核心 API 设计  

> **原则**：RESTful + Protobuf（高效二进制） 双协议，满足浏览器和内部服务的不同需求。

### 1. 对外查询 API（用户侧）  

| 方法 | URL | 请求体（JSON / Protobuf） | 响应体 | 说明 |
|------|-----|---------------------------|--------|------|
| `POST` | `/search` | ```json { "query": "string", "lang": "en", "region": "US", "page": 1, "size": 10, "filters": { "media_type": ["web","image"] }, "user_id": "optional" }``` | ```json { "request_id": "uuid", "total": 12345, "hits": [ { "doc_id": "12345", "title": "...", "snippet": "...", "url": "...", "score": 9.8, "media_type": "web" }, ... ], "suggestion": "Did you mean: ...", "latency_ms": 132 }``` | **POST** 避免 URL 长度限制；返回 `request_id` 便于追踪。 |
| `GET` | `/autocomplete?prefix=xxx` | - | ```json { "suggestions": ["xxx1","xxx2"] }``` | 自动补全，用 Trie + 热门查询缓存。 |

### 2. 内部服务 API  

| 方法 | URL | 功能 | 备注 |
|------|-----|------|------|
| `GET /index/term/{term_id}` | 倒排分片查询 | 返回 posting list（压缩二进制） | 内部使用 Protobuf，支持 **分片并行**。 |
| `POST /rank` | 排序服务 | 输入 doc_id 列表 + 实时特征 → 输出排序后列表 | 采用 **TensorFlow Serving** 的 `Predict` 接口。 |
| `POST /feature/update` | 实时特征写入 | Flink 将聚合结果写入 | 幂等写，使用 `doc_id` 作为主键。 |
| `POST /cache/invalidate` | 缓存失效 | 当文档更新或删除时，主动刷新热点缓存 | 采用 **消息队列**（Kafka）广播。 |

### 3. API 设计细节  

- **统一错误码**：`0` 成功，`1` 参数错误，`2` 鉴权失败，`3` 资源不可用，`4` 超时。  
- **限流**：在 API 网关层使用 **令牌桶**（Token Bucket）对每个 IP / UserID 限流，防止恶意刷流。  
- **鉴权**：对搜索日志做匿名化处理，使用 **OAuth2** 或 **JWT**（可选）进行用户身份校验。  
- **压缩**：查询返回的 `hits` 使用 **gzip/ Brotli**；倒排列表使用 **VarByte + Simple-9** 编码。  

> **如果不使用压缩会怎样？**  
> - 单个查询返回 10 条结果，大约 2 KB；但倒排列表在内部服务之间传输可能达到 **几 MB**，网络带宽瞬间被耗尽，导致 QPS 下降。

## 第五步：详细组件设计  

下面逐层展开每个关键组件的内部实现细节、数据流以及选型原因。

### 1. 入口层（CDN + DNS）  

- **CDN**：就近缓存静态资源（JS、CSS、搜索建议 JSON），降低回源压力。  
- **DNS**：使用 **Anycast** 将用户请求路由到最近的 **入口数据中心**，实现地理负载均衡。  

### 2. 负载均衡 & API 网关  

| 任务 | 具体实现 |
|------|----------|
| **L4 负载** | Envoy 使用 **ECMP** 将 TCP 连接均匀分配到多台 API 网关机器。 |
| **L7 路由** | 基于 **HTTP Header**（`Accept-Language`, `X-Region`) 以及 **查询内容**（关键词类型）进行路由到不同的 **检索集群**（Web、Image、Video）。 |
| **限流/熔断** | Envoy 的 **Rate Limit Service** + **Circuit Breaker**，对异常热点查询自动降级。 |
| **日志/追踪** | 注入 **OpenTelemetry** Trace ID，后续全链路监控使用。 |

### 3. 检索层（分布式倒排索引服务）  

#### 3.1 查询路径  

1. **API 网关** 将 `query` 拆分为 **term 列表**（分词器 → Tokenizer）。  
2. 对每个 term，查询 **Term Dictionary**（KV Store）得到 `term_id` 与 **所在分片 ID**。  
3. **路由器**（内部服务）把同一分片的多个 term 合并，发 **并行 RPC** 到对应 **倒排分片节点**。  
4. 每个分片节点：  
   - 读取 **posting list**（磁盘 → 内存映射），解码后做 **布尔/交集/并集** 计算。  
   - 根据 **DocID** 与 **过滤条件**（region、media_type）做 **快速过滤**（位图）。  
   - 返回 **Top‑K（未排序）doc_id + 评分（TF‑IDF）**，只返回 **doc_id + term‑frequency**，不返回完整文档。  
5. **合并层**（聚合节点）把各分片返回的 Top‑K 合并，去重后形成 **候选集合（约 1k‑5k）**，交给排序层。  

#### 3.2 并行度与优化  

| 优化点 | 说明 | 预期收益 |
|--------|------|----------|
| **倒排列表压缩** | VarByte + SIMD 解码 | CPU 解码成本降低 30% |
| **跳表（Skip List）** | 在 posting list 中每 128 条 doc 放一个 skip pointer | 快速定位交集时跳过大量无关 doc |
| **内存映射 (mmap)** | 大文件直接映射到进程地址空间 | 减少磁盘 I/O 系统调用 |
| **批量 RPC** | 同一次查询把所有 term 的请求打包一次发送 | 减少网络 RTT，提升 QPS |
| **分片副本负载均衡** | 动态权重 + 监控指标（CPU、QPS） | 防止热点分片单点过载 |

#### 3.3 索引写入流程（增量）  

1. **爬虫** 把新抓取的网页写入 **消息队列**（Kafka） → **Parser** 解析出 **token、metadata**。  
2. **索引构建服务**（基于 Lucene）在 **微批**（每 5 min）里生成 **增量倒排段**（Segment）。  
3. **段上传** → **分片节点** 按 **term‑hash** 分配到对应分片。  
4. **合并服务**（后台）定期把小段合并成大段（Merge），压缩率提升，查询性能提升。  
5. **索引元数据**（segment version、doc count）写入 **Zookeeper/etcd**，供路由器感知最新分片状态。  

> **如果直接在查询时实时写入倒排索引会怎样？**  
> - 写入延迟会显著增加查询路径的锁竞争，导致 **查询延迟失控**，且合并成本不可控。  

### 4. 排序层（学习排序）  

#### 4.1 特征体系  

| 类别 | 示例特征 | 计算方式 |
|------|----------|----------|
| **内容特征** | TF‑IDF, BM25, 词向量相似度 | 离线预计算，存入特征库 |
| **链接特征** | PageRank, 入链数, 出链数 | 周期性离线 PageRank 计算（MapReduce） |
| **行为特征** | CTR, 最近 1h 点击次数, 用户历史偏好 | 实时流处理（Flink） |
| **上下文特征** | Query length, Region, Device type | API 网关直接注入 |
| **模型特征** | 深度学习向量（BERT） | 离线模型推理后存入特征库 |

#### 4.2 排序模型  

- **离线训练**：使用 **Lambda Architecture**  
  - **Batch**（Spark ML）训练 **GBDT / LambdaMART**，利用历史点击、跳转、停留时间。  
  - **Online**（FTRL、深度学习）微调模型，捕捉实时热点。  
- **在线推理**：  
  - **输入**：候选 doc_id 列表 + 实时特征（从 NoSQL 拉取）  
  - **模型**：TensorFlow Serving **SavedModel**（多层感知机 + 树模型融合）  
  - **输出**：每个 doc 的 **打分**，再 **Top‑K**（默认 10）返回给前端。  

#### 4.3 性能优化  

| 技巧 | 作用 |
|------|------|
| **特征缓存**（热点 doc） | 将常用特征放入 Redis，查询时 1‑2 ms 读取，避免 DB 随机 I/O。 |
| **模型分片**（sharding） | 把模型权重切分，分布式推理（多节点并行），降低单机 CPU 占用。 |
| **批量推理** | 同一查询把所有 candidate 合并一次调用模型，减少 RPC 开销。 |
| **异步回写** | 排序完成后把最新点击特征写回流处理系统，做到 **读写分离**。 |

### 5. 缓存层  

1. **热点查询缓存**（查询 → 完整结果）  
   - 使用 **Redis Cluster**，键为 `hash(query+region+lang)`，值为完整 JSON 结果。  
   - **TTL**：5‑10 min，热点查询在此期间直接返回。  
2. **局部 LRU/LFU**（每台查询节点本地）  
   - 保存最近 10 k 条查询的 **候选 doc_id 列表**，避免跨网卡的 RPC。  
3. **缓存失效策略**  
   - 当对应文档的 **PageRank**、**实时点击** 更新时，发送 **Kafka 消息** → **Cache Invalidate Service** → 删除/更新相关缓存键。  

> **不做缓存的后果**：每一次查询都要走完整的倒排 → 合并 → 排序流程，CPU、磁盘、网络压力成倍增加，峰值 QPS 很难突破 30k。  

### 6. 存储层  

| 存储 | 角色 | 关键配置 |
|------|------|----------|
| **对象存储（Ceph / OSS）** | 原始网页、图片、视频二进制 | 多副本（EC 纠删码）+ 多AZ 同步 |
| **分布式文件系统（HDFS）** | 倒排索引段文件 | 复制因子 3、块大小 256 MB、压缩 ORC |
| **NoSQL（Cassandra）** | 特征库、元数据 | 写放大低、线性扩展、TTL 支持 |
| **KV Store（RocksDB）** | Term Dictionary | 按 `term_id` 排序、支持前缀扫描 |
| **流处理状态（Flink RocksDB State Backend）** | 实时特征 | 支持 exactly‑once、checkpoint 到 HDFS |

### 7. 监控、日志、告警  

- **Metrics**：QPS、P99 latency、cache hit ratio、segment merge latency、replica lag。  
- **Tracing**：OpenTelemetry 跨服务链路，定位慢查询节点。  
- **Alerting**：CPU > 80% 持续 5 min、Replica Lag > 30 s、Cache Miss Spike > 2× 基线。  
- **Dashboard**：Grafana 展示每层吞吐、错误率、热点 term 分布。

## 第六步：扩展性与高可用设计  

### 1. 水平扩容路径  

| 维度 | 扩容方式 | 关键点 |
|------|----------|--------|
| **查询节点** | 增加 API 网关 + 检索节点 | 负载均衡自动感知新节点，Hash 分片不需要迁移。 |
| **倒排分片** | 新增 **分片节点**，重新平衡 term‑hash 范围 | 使用 **Consistent Hash Ring**，只搬迁受影响的少量 term。 |
| **缓存层** | 扩容 Redis Cluster（slot 迁移） | Redis 支持在线迁移，业务不感知。 |
| **排序模型** | 增加 TensorFlow Serving 实例，使用 **Kubernetes HPA** 自动伸缩 | 通过 **gRPC** 负载均衡实现无感知扩容。 |
| **存储** | 对象存储自动扩容（增加 OSD） | Ceph 通过 CRUSH map 动态平衡数据。 |

> **扩容时间 ≤ 2 小时**：通过 **无中心化的元数据服务（etcd）** 实时发布新节点信息，所有服务读取同一配置，无需人工迁移。  

### 2. 多活 + 灾备  

1. **地域多活**：在 **北美、欧洲、亚洲** 三大洲分别部署完整的 **入口 → 检索 → 排序 → 缓存 → 存储** 链路。  
2. **全局 DNS Anycast** 自动把用户请求路由到最近的活跃地区。  
3. **跨地域数据复制**  
   - **倒排索引**：使用 **跨集群复制**（基于 HDFS DistCp）每日增量同步，延迟 ≤ 6 h。  
   - **特征库**：Cassandra 多数据中心复制（NetworkTopologyStrategy），保证 **读写在本地**，写入跨中心异步。  
   - **对象存储**：使用 **跨 AZ 纠删码**（EC 6+3），单中心失效仍可提供完整数据。  
4. **故障切换流程**  
   - **检测**：使用 **heartbeat**（Consul）监测每个数据中心健康状态。  
   - **自动切换**：若某中心失联，Global Load Balancer 将流量切到其他两中心，DNS TTL 设置为 **30 s**，快速刷新。  
   - **业务层降级**：热点分片副本在其他中心不可用时，使用 **近似 Top‑K**（只取局部分片）保证 **≤ 500 ms** 响应。  

### 3. 容错细节  

| 场景 | 失效组件 | 备选方案 | 延迟影响 |
|------|----------|----------|----------|
| **单节点倒排分片宕机** | 该分片的 Primary | 自动切换到 Replica（快速选举） | < 5 ms（内部 RPC） |
| **Cache 节点失效** | Redis 单实例 | 自动路由到其他 slot 主节点 | 1‑2 ms |
| **排序服务不可用** | TensorFlow Serving | 使用 **离线模型 + 规则排序**（回退） | 延迟 ↑ 30 ms，排序质量略降 |
| **跨地区网络中断** | 两个数据中心之间的链路 | 只使用本地中心完整数据 | QPS 下降 ~30%，但仍满足 SLA（500 ms） |

## 第七步：常见面试追问与回答  

### Q1️⃣ 索引分片与路由——如何保证查询只访问少量分片并避免热点？  

**回答要点**  
1. **Term‑Hash 分片**：每个 term 通过一致性哈希映射到唯一分片，查询时只需要访问对应分片（通常 1‑3 个），而不是遍历全部。  
2. **副本路由**：热点 term（高 DF）在多个节点上保持 **Primary + N‑1 Replicas**，路由器根据 **实时负载（CPU、QPS）** 将请求分散到负载最轻的副本。  
3. **热点分片均衡**：定期统计 term 的访问频率，若某分片负载持续 > 80% → **拆分该分片**（重新 hash 迁移一部分 term 到新分片），并更新路由表。  
4. **查询并行**：即使查询包含多个 term，仍然可以 **并行向多个分片发起 RPC**，整体延迟受慢最慢分片支配，但通过 **副本负载均衡** 可把慢分片的概率降到 < 5%。  

### Q2️⃣ 排序与机器学习——千级 QPS 的学习排序模型如何支撑？  

**回答要点**  
- **特征离线/实时分层**：大部分特征（PageRank、词向量）在离线批处理完成，存入特征库，读取成本极低。实时特征（最近点击）使用 **流处理**（Flink）写入 NoSQL，TTL 短，保持最新。  
- **模型轻量化**：使用 **GBDT/LambdaMART** 作为主模型，模型文件 < 10 MB，推理时只需 **few hundred FLOPs**，适合 CPU 推理。  
- **批量推理**：一次查询的候选集（1k‑5k doc）一次性送入模型，避免对每条 doc 进行单独 RPC。  
- **水平扩展**：TensorFlow Serving 部署在 **Kubernetes**，使用 **HPA** 根据 CPU 使用率自动扩容，保证 QPS 高峰时仍能保持 **< 5 ms** 推理时延。  
- **降级方案**：若模型服务不可用，直接使用 **BM25+规则加权** 进行排序，仍能满足 SLA（略低的相关度）。  

### Q3️⃣ 灾备与容错——单个数据中心失联时，如何保证 500 ms 内返回近似正确的结果？  

**回答要点**  
1. **全局 Anycast DNS + 负载均衡**：请求自动切换到仍然可用的其他中心。  
2. **本地完整倒排索引**：每个中心保存 **完整的倒排分片**（通过每日增量复制），因此即使失去一个中心，其他中心仍拥有全部 term → 能完整检索。  
3. **缓存热备**：热点查询缓存是 **全局同步**（通过双向复制），失联中心的热点缓存会在其他中心已有。  
4. **降级查询**：如果某中心的 **实时特征**（点击、热点）不可用，排序层回退使用 **离线特征 + 规则**，仍能在 **< 500 ms** 完成。  
5. **容错时延预算**：  
   - 网络路由切换：< 30 ms（DNS TTL）  
   - 检索层：< 150 ms（并行查询 + 合并）  
   - 排序层：< 80 ms（离线模型）  
   - 响应组装 + 传输：< 40 ms  
   - 合计 ≈ 300 ms，留有余量应对突发。  

### Q4️⃣ “如果倒排索引按照 term → doc 列表进行分片，如何保证查询时只访问少量分片并且避免热点分片？”  

**回答要点**（与 Q1 类似）  
- 采用 **哈希分片 + 多副本**，查询只需要定位 term 对应的分片。  
- 对 **高频 term**（DF > 10⁶）采用 **双层分片**：先按照 term 哈希定位大分片，再在该分片内部 **二级哈希**（或 **range 分片**）进一步细分，分布到多台机器。  
- **路由层**维护 **热点 term → 多分片映射表**，动态把查询请求轮询到不同子分片，实现负载均衡。  

### Q5️⃣ “实时特征如何保证千级 QPS 的写入与读取一致性？”  

- **写入**：使用 **Flink Exactly‑Once**，对每条点击事件进行 **KeyBy(doc_id)**，状态存入 **RocksDB State Backend**，并 **checkpoint** 到 HDFS。  
- **读取**：排序层通过 **读‑写分离**，从 **Cassandra**（或 **ScyllaDB**) 读取最新特征，读取路径采用 **读副本**，保证查询不被写阻塞。  
- **延迟**：Flink → Cassandra 的 **异步写**，平均 1‑2 s 可见，满足业务对 “近实时” 的要求。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **倒排索引的分片与热点均衡** | 需要在 **查询效率** 与 **存储均衡** 之间找到折中。若分片过细，合并成本高；若分片过粗，热点导致单节点 CPU/IO 爆炸。 | - 先用 **term‑hash** 把大多数 term 均匀分配。<br>- 统计 **DF**，对 **高频 term** 采用 **二级分片** 或 **多副本并行查询**。<br>- 实时监控分片 QPS，触发 **自动拆分/合并**。 |
| **学习排序的实时特征与低延迟** | 排序模型需要 **千级 QPS**，且要结合 **实时点击**、**热点** 等特征，特征获取与模型推理都可能成为瓶颈。 | - 将 **大多数特征** 预计算离线，放入 **NoSQL**，只把 **实时特征**（近 1 min）放入 **流处理状态**。<br>- 采用 **批量推理**，一次 RPC 处理所有候选 doc，降低网络开销。<br>- 通过 **模型轻量化 + 横向扩容**，确保 CPU 能在 5 ms 内完成一次预测。 |

### 2. 新手最容易犯的错误（至少 2 条）  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有功能一次性放进单体服务** | 会导致 **代码臃肿、扩容困难、单点故障**，并且难以满足 200 ms 的时延目标。 | 采用 **微服务/分层** 架构：入口 → 检索 → 排序 → 缓存 → 存储，每层独立伸缩。 |
| **只考虑查询性能，忽略写入和索引更新** | 搜索引擎的 **数据新鲜度** 与 **增量索引** 同样关键，忽视会导致 “数据滞后 > 24 h”。 | 设计 **增量抓取 → 增量索引 → 合并** 流程，并把 **写入路径** 与 **查询路径** 解耦。 |
| **缓存只做单层本地缓存** | 只缓存本机热点，跨机器热点仍然会打到后端，导致热点聚集。 | **多层缓存**：本地 LRU + 分布式 Redis + CDN，形成 **从 L1 到 L3** 的层级缓存。 |
| **只用单副本来保证可用性** | 单副本在硬件故障、网络分区时会导致服务不可用，无法满足 99.99% SLA。 | **多副本 + 多活**：每个分片至少 3 副本，跨机房/跨地区分布。 |
