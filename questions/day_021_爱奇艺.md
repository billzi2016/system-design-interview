# 系统设计面试题：爱奇艺视频平台

## 1. 题目背景
爱奇艺是国内领先的在线视频点播平台，提供海量影视、综艺、动漫等内容的在线播放、下载和弹幕互动。面试官希望你从零设计一个能够支撑 **日活跃用户 2 亿**、 **峰值 QPS 80,000** 的高可用、低延迟的视频点播系统。

---

## 2. 面试场景设定  
> **面试官**：  
> “我们现在需要为爱奇艺构建一个全球化的视频点播系统，能够在高峰期同时为数亿用户提供流畅的观看体验。请从需求分析、系统架构、关键技术选型等方面进行设计，并重点说明如何保证 **低延迟、强并发、数据安全**。”  

---

## 3. 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **用户登录 / 注册**：支持手机号、邮箱、第三方 OAuth（微信、QQ）登录，登录后保持会话。 |
| 2 | **视频点播**：用户可以搜索、浏览、播放任意视频，支持 **快进/快退、切换清晰度（1080p/4K）**，并在播放过程中无感知切换。 |
| 3 | **弹幕与实时互动**：在播放过程中实时发送/接收弹幕、点赞、评论，弹幕需要精准同步到视频时间轴。 |
| 4 | **内容推荐**：基于用户画像、观看历史、实时热点生成 **首页推荐**、**猜你喜欢** 列表，支持离线批处理和在线实时召回。 |
| 5 | **离线缓存 / 下载**：用户可将视频缓存到本地或云端，支持 **断点续传** 与 **版权有效期** 校验。 |
| 6 | **付费/会员管理**：提供付费点播、会员订阅、优惠券、积分等功能，保证支付安全并实时更新会员状态。 |

## 4. 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（日活跃用户）** | 200 million | 覆盖移动端、Web、TV 多端。 |
| **峰值 QPS** | 80,000 requests/s | 包括播放请求、弹幕推送、搜索、推荐等。 |
| **平均播放延迟** | ≤ 200 ms（客户端收到首帧） | 包括 CDN 拉取、边缘缓存、协议优化。 |
| **系统可用性** | ≥ 99.99%（月度 SLO） | 计划内维护 ≤ 5 min/次，故障自动切换。 |
| **存储容量** | 视频原始库 30 PB，缓存层 5 PB | 采用冷热分层、对象存储与 CDN 双层缓存。 |
| **弹幕实时性** | ≤ 100 ms 从发送到接收 | 使用低延迟消息队列与边缘节点分发。 |

## 解题思路总览
> **核心思路**：先把系统 **最小可用（MVP）** 搭建出来——只需要登录、点播、弹幕三个功能，使用最简单的单机组件即可验证业务流程。随后逐层 **拆分**、**复制**、**缓存**、**容错**，把单机升到分布式，最终满足 **DAU 2 亿、峰值 QPS 80k、99.99% 可用** 的指标。

| 步骤 | 目标 | 关键技术 | 为什么这样做 |
|------|------|----------|--------------|
| 1️⃣ | 需求拆解 & 业务容量估算 | 业务模型、公式 | 只有知道“大概多少请求、多少存储”，才能判断是否需要水平扩展 |
| 2️⃣ | 绘制 **高层框图**（客户端 → API 网关 → 业务层 → 存储层） | API Gateway、负载均衡、微服务 | 把系统划分成 **职责清晰的层**，便于后续独立扩容 |
| 3️⃣ | 选型 **持久化**（关系型、NoSQL、对象存储） | MySQL、MongoDB、Redis、OSS/MinIO | 不同数据（用户、视频元数据、弹幕）有不同的访问特征，需要对应的存储 |
| 4️⃣ | 设计 **核心 API**（登录、点播、弹幕） | RESTful / gRPC、统一返回体 | API 是面试官最容易抓住的切入口，写清楚入参/出参、错误码 |
| 5️⃣ | 细化每个 **业务组件**（认证中心、视频分发、弹幕流、推荐） | JWT、CDN、FFmpeg、Kafka、Spark、Flink | 让抽象的模块落到可实现的技术栈上 |
| 6️⃣ | 引入 **缓存、限流、熔断、容灾** 等非功能设施 | Redis、RateLimiter、Hystrix、Cassandra、跨地域多活 | 直接对应需求表中的 **低延迟、强并发、数据安全** |
| 7️⃣ | 准备 **面试追问**（缓存失效、弹幕顺序、一致性、弹性伸缩） | 解释原理、给出权衡 | 面试官往往通过细节问题判断你的深度，提前准备能加分 |

下面我们一步步展开。

## 第一步：理解需求与规模估算

### 1. 功能性需求拆解
| 编号 | 业务子系统 | 关键点 |
|------|------------|--------|
| 1 | **用户中心** | 注册、登录、会话、OAuth、密码/验证码安全 |
| 2 | **视频处理流水线** | 上传 → 转码 → 存储 → CDN 上线 |
| 3 | **点播服务** | 播放请求、ABR、自适应切换、断点续传 |
| 4 | **弹幕/互动** | 实时发送、时间轴同步、点赞/评论 |
| 5 | **推荐系统** | 离线模型 + 在线召回 |
| 6 | **付费/会员** | 订单、支付回调、权限校验、积分/优惠券 |

> **最小可用系统（MVP）**：实现 1、2、3、4 四个子系统，其他可后期按需接入。  

### 2. 业务规模估算（保守估计）

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 题目给出 2×10⁸ | 200 M |
| **峰值 QPS** | 80 k（包括所有接口） | 80 000 req/s |
| **播放请求占比** | 假设 60% 为视频播放（包括分片请求） | 48 k req/s |
| **弹幕请求占比** | 20% 发送/拉取弹幕 | 16 k req/s |
| **搜索/推荐占比** | 10% | 8 k req/s |
| **登录/付费占比** | 10% | 8 k req/s |
| **平均每个播放请求的分片数** | 采用 HLS/DASH 2 s 切片 → 1 分钟视频 ≈ 30 切片 | 48 k × 30 = 1.44 M 切片/s |
| **峰值流量** | 每片 2 Mbps（720p） → 2 Mbit × 1.44 M ≈ 2.88 Tbps | 约 360 GB/s（压缩后约 45 GB/s） |

> **结论**：  
- **网络**：需要 CDN + 边缘缓存，单机带宽根本无法满足。  
- **存储**：原始视频 30 PB（假设 10 M 部视频，每部 3 GB） → 必须使用对象存储 + 多级冷热分层。  
- **计算**：登录、推荐、弹幕等需要 **水平扩容** 的 stateless 服务。  

### 3. 非功能需求拆解成技术指标

| 需求 | 对应技术点 |
|------|------------|
| 低延迟（首帧 ≤200 ms） | CDN 边缘缓存、TCP/QUIC、HTTP/2、ABR、预热 |
| 高可用（99.99%） | 多可用区、跨地域灾备、自动故障转移、健康检查 |
| 弹幕实时 ≤100 ms | Kafka + 消费者分区 + Edge 推送、WebSocket/QUIC |
| 数据安全 | TLS、OAuth、JWT、支付安全（PCI DSS） |
| 容量 30 PB + 5 PB 缓存 | OSS（对象存储）+ 分层缓存（Redis/LRU 本地）+ CDN |

## 第二步：高层架构设计

下面给出 **从 MVP 到完整高可用系统** 的层次化图（使用 Mermaid 语法，面试时可以手绘或 PPT）：

```mermaid
graph TD
    subgraph 客户端
        A[移动端 / Web / TV] -->|HTTPS/WebSocket| B[API Gateway]
    end
    subgraph 边缘层
        B --> C[Auth Service (JWT)]
        B --> D[Video Edge Proxy]
        B --> E[弹幕 Edge Server]
        D -->|分片| CDN[CDN 节点]
        E -->|实时推送| MQ[Kafka Edge Topic]
    end
    subgraph 核心业务层
        C --> F[User Service (MySQL + Redis)]
        D --> G[Play Service (Metadata DB + Object Store)]
        E --> H[Bullet Service (MongoDB + Redis)]
        B --> I[Recommendation Service (Spark + Redis Cache)]
        B --> J[Payment Service (MySQL + MQ)]
    end
    subgraph 数据持久层
        G -->|原始/转码文件| OSS[对象存储 (30 PB)]
        G -->|转码后分片| CDN
        F -->|用户信息| MySQL_User[MySQL]
        H -->|弹幕| Mongo_Bullet[MongoDB]
        I -->|模型/特征| HDFS[HDFS + Spark]
        J -->|订单| MySQL_Pay[MySQL]
    end
    subgraph 监控/运维
        K[Prometheus] -->|指标| L[Alertmanager]
        K -->|日志| M[ELK]
    end
```

### 关键层次划分

| 层级 | 主要职责 | 典型技术 |
|------|----------|----------|
| **客户端** | 发起请求、播放/弹幕 UI、使用 CDN 拉流 | HLS/DASH、QUIC、WebSocket |
| **API Gateway / 负载均衡** | 统一入口、路由、限流、鉴权、灰度发布 | Kong / Nginx + Consul / Istio |
| **边缘层** | 靠近用户的缓存与实时推送，降低时延 | CDN、Edge Cache、Kafka Edge、Envoy |
| **业务层（微服务）** | 登录、用户、播放、弹幕、推荐、支付等 **业务逻辑** | Spring Boot / Go / Node.js，容器化（Docker）+编排（K8s） |
| **持久层** | 大容量对象存储、关系型/文档库、时序/缓存 | OSS/MinIO、MySQL、MongoDB、Redis、HDFS |
| **监控/运维** | 监控、日志、告警、灰度发布、自动扩缩容 | Prometheus、Grafana、ELK、Kubernetes HPA |

> **为什么要这么分层？**  
- **职责单一**：每层只负责自己擅长的事，出现问题时定位更快。  
- **弹性伸缩**：例如弹幕层流量突增，只需要横向扩展 Kafka & Edge Server，而不影响播放服务。  
- **跨地域容灾**：业务层可以在不同 AZ 部署，持久层使用多活对象存储，故障时自动切流。

## 第三步：数据库设计

不同业务数据的访问特征差异很大，**同构**（只用一种数据库）会导致资源浪费或性能瓶颈。下面分别说明每类数据的模型、主键、分库分表、索引及备份策略。

### 1. 用户信息（User Service）

| 表名 | 主键 | 关键字段 | 访问模式 | 建议存储 |
|------|------|----------|----------|----------|
| `users` | `user_id` (BIGINT auto) | phone, email, password_hash, nickname, avatar_url, created_at | 读：登录、个人中心（QPS 较高）<br>写：注册、修改资料 | **MySQL（主从）+ Redis 缓存** |
| `user_sessions` | `session_token` (VARCHAR) | user_id, expire_at, device_info | 读写均衡，TTL 30d | **Redis（TTL）** |
| `user_wallet` | `user_id` | balance, frozen_amount, last_update | 写：支付、充值<br>读：会员校验 | **MySQL + Binlog 同步至 Kafka** |

**分库分表**：  
- 按 `user_id % N`（N=16）水平分库，降低单库写入压力。  
- 每库内部再按月份归档 `user_sessions`（TTL），防止无限增长。  

**备份**：  
- 主从复制（异地灾备）+ 每日全量快照（OSS）+ 增量 binlog。  

### 2. 视频元数据（Play Service）

| 表名 | 主键 | 关键字段 | 访问模式 | 建议存储 |
|------|------|----------|----------|----------|
| `videos` | `video_id` (BIGINT) | title, description, category_id, duration, status, created_at, updated_at | 读：列表、详情、推荐召回（高 QPS）<br>写：上传完成、转码状态更新 | **MySQL（只读副本）** |
| `video_files` | `video_id` + `quality` (1080p/4K) | file_path, size, md5, codec, bitrate | 读：CDN 拉流、断点续传 | **对象存储（OSS） + MySQL 记录路径** |
| `video_cdn_status` | `video_id` | cdn_node_list, last_sync_time | 写：CDN 同步任务 | **Redis（热点缓存）** |

**分库**：  
- 按 `category_id` 或 `video_id % 32` 分库，保证热点视频均匀分布。  

**索引**：  
- `title`（FULLTEXT）用于搜索；`category_id`、`status`（普通 B‑Tree）用于列表过滤。  

### 3. 弹幕（Bullet Service）

弹幕是 **写多读多、时间轴强关联** 的典型场景，适合文档型数据库 + 缓存。

| 集合/表 | 主键 | 关键字段 | 访问模式 | 建议存储 |
|--------|------|----------|----------|----------|
| `bullet_comments` (MongoDB) | `_id` (ObjectId) | video_id, user_id, content, timestamp_ms, color, type | 读取：按 `video_id` + 时间范围查询（高并发）<br>写：单条弹幕插入 | **MongoDB Sharded**（基于 `video_id`） |
| `bullet_cache_{video_id}` (Redis Sorted Set) | `score = timestamp_ms` | comment_json | 读取热点视频弹幕（前 5 min） | **Redis**（LRU + 失效策略） |

**写入流程**：  
1. 客户端 → API Gateway → Bullet Service（Kafka Producer）  
2. 写入 MongoDB（持久化） & 同时写入 Redis ZSET（实时展示）  
3. 消费端（Edge Server）从 Redis 拉取并通过 WebSocket 推送给观看者  

**备份**：MongoDB 副本集 + 每日快照到 OSS。  

### 4. 推荐系统（离线/实时）

| 表/文件 | 主键 | 关键字段 | 访问模式 | 建议存储 |
|--------|------|----------|----------|----------|
| `user_features` | `user_id` | embedding vector、兴趣标签 | 实时召回（读） | **HBase / Cassandra**（列式） |
| `item_features` | `video_id` | embedding、热度分、标签 | 召回 | **HBase** |
| `model_snapshot` | `model_version` | Spark/Flink 训练模型文件 | 离线更新 | **HDFS + OSS** |
| `rec_cache` (Redis) | `user_id` | 推荐列表（JSON） | 读：首页、猜你喜欢 | **Redis**（TTL 5 min） |

**为什么不放在 MySQL？**  
- 推荐特征向量维度大、查询需要 **列式扫描**、写入频繁（实时特征更新），NoSQL 更合适。

## 第四步：核心 API 设计

> **原则**：**RESTful** 用于大多数 CRUD 场景，**gRPC** 用于内部高频流媒体/弹幕交互（低延迟），**WebSocket** 用于实时弹幕推送。

下面列出 **最关键的 6 套 API**，包括请求/响应示例、错误码、鉴权方式。面试时可以挑选 2–3 套详细讲解。

### 1. 登录 / 注册 API（Auth Service）

| 方法 | 路径 | 鉴权 | 请求体 | 响应体 | 可能错误码 |
|------|------|------|--------|--------|------------|
| POST | `/api/v1/auth/register` | 无 | `{ phone/email, password, sms_code }` | `{ user_id, token, expires_at }` | 400（参数错误）<br>429（短信验证码频率） |
| POST | `/api/v1/auth/login` | 无 | `{ login_type, identifier, password_or_code }` | 同上 | 401（凭证错误）<br>403（账号被封） |
| POST | `/api/v1/auth/refresh` | JWT | `{ refresh_token }` | `{ token, expires_at }` | 401（刷新令牌失效） |

**实现要点**：  
- 使用 **JWT**（短期 access token + 长期 refresh token）实现无状态鉴权。  
- 登录成功后把 `user_id` 写入 Redis `user_sessions`，设置 TTL，供后续单点登录（SSO）检查。  

### 2. 视频播放请求（Play Service）

| 方法 | 路径 | 鉴权 | 请求体 | 响应体 | 说明 |
|------|------|------|--------|--------|------|
| GET | `/api/v1/video/{video_id}/play` | JWT | `?quality=1080p&token=xxx` | `{ cdn_url, token, expire, segment_template }` | **ABR**：返回 HLS/DASH **master.m3u8** 或 **manifest.mpd**，其中 `segment_template` 为 `$Number$.ts`，CDN URL 已经做过鉴权签名（防盗链）。 |
| POST | `/api/v1/video/{video_id}/report` | JWT | `{ event, position, duration }` | `{ status: "ok" }` | 客户端上报播放结束/异常，用于计费、推荐模型。 |

**实现要点**：  
- **鉴权**：在返回 CDN URL 前先校验用户是否有观看权限（会员、付费）。  
- **签名**：使用 HMAC‑SHA256 对 URL+expire 生成防盗链 token，CDN 端验证。  
- **分片**：每 2 s 生成一个 `.ts`/`.m4s`，支持 **Range** 请求，实现断点续传。  

### 3. 弹幕发送 / 拉取（Bullet Service）

| 方法 | 路径 | 鉴权 | 请求体 | 响应体 | 说明 |
|------|------|------|--------|--------|------|
| POST | `/api/v1/bullet/{video_id}` | JWT | `{ content, color, type, timestamp_ms }` | `{ bullet_id, status:"queued" }` | 服务器收到后写入 Kafka + Redis ZSET，返回 ID 用于撤回。 |
| GET | `/ws/bullet/{video_id}` | JWT (WebSocket 握手) | — | **实时弹幕流**（JSON） | Edge Server 订阅 Redis ZSET，按时间顺序推送。 |

**实现要点**：  
- **顺序保证**：弹幕写入 **Kafka Partition = video_id % N**，保证同一视频的弹幕在同一分区有严格顺序。  
- **实时性**：Edge Server 通过 **Redis Pub/Sub** 监听新弹幕，几乎零延迟推送。  

### 4. 推荐召回 API（Recommendation Service）

| 方法 | 路径 | 鉴权 | 请求体 | 响应体 |
|------|------|------|--------|--------|
| GET | `/api/v1/recommend/home` | JWT | `?page=1&size=20` | `{ videos: [{video_id, title, cover, score}] }` |
| GET | `/api/v1/recommend/like` | JWT | `?user_id=xxx` | 同上 |

**实现要点**：  
- 先从 **Redis** 拉取缓存的召回列表，若未命中则走 **HBase** 读取用户特征+物品特征做在线召回（Flink 实时计算），最后写回缓存。  

### 5. 会员/付费 API（Payment Service）

| 方法 | 路径 | 鉴权 | 请求体 | 响应体 |
|------|------|------|--------|--------|
| POST | `/api/v1/payment/checkout` | JWT | `{ video_id, price, payment_method }` | `{ order_id, payment_url }` |

**实现要点**：  
- 支付回调使用 **签名+IP 白名单**，防止伪造。  
- 订单写入 **MySQL**，并同步到 **Kafka**，后续 **会员状态服务** 消费更新 `user_wallet`。  

### 6. 搜索 API（Search Service）

## 第五步：详细组件设计

下面分别从 **身份验证、视频处理流水线、播放分发、弹幕实时系统、推荐系统、会员计费** 六个核心模块进行深度剖析，说明 **数据流、关键技术选型、容错与扩展点**。

### 1. 身份验证与会话管理

1. **流程**  
   - 客户端 → API Gateway → Auth Service（登录/注册） → 生成 **JWT access_token**（15 min）+ **refresh_token**（30 day）  
   - `access_token` 放在 **HTTP Header Authorization**，`refresh_token` 放在 **HttpOnly Cookie**（防 XSS）  
   - 每次请求，Gateway 通过 **JWT 校验**（公钥/私钥），若过期则走 `/refresh` 接口  
   - `user_sessions`（Redis）保存 **session_id → user_id**，TTL 与 token 同步，用于 **强制下线**（如付费失效）  

2. **选型理由**  
   - **JWT**：无状态，横向扩容不受 Session 共享限制。  
   - **Redis**：提供快速的 Session 查找、失效、黑名单功能。  

3. **容错**  
   - JWT 校验在每台 Gateway 本地完成，单点故障概率极低。  
   - Redis 使用 **主从+哨兵**，自动故障转移。  

### 2. 视频上传 → 编码 → CDN 分发流水线

1. **上传**  
   - 客户端直接把原始文件 **PUT** 到对象存储（OSS）**预签名 URL**，避免网关成为流量入口。  
   - OSS 触发 **ObjectCreated** 事件（Kafka） → **Transcode Service** 拉取原始文件进行多码率转码（FFmpeg /自研）。  

2. **转码**  
   - 多码率（1080p、4K、HDR）+ **分片**（2 s）+ **自适应码率（ABR）** 生成 `m3u8`/`mpd` 清单。  
   - 转码完成后，将分片写回 OSS，生成 **metadata** 记录到 `video_files` 表。  

3. **CDN 同步**  
   - OSS 与 CDN 采用 **拉取（Pull）模式**：CDN 节点周期性拉取热点对象；  
   - 对于新上线的热门视频，**预热**脚本主动向 CDN 发起 **Cache Purge/Prefetch**，确保首帧在边缘已有缓存。  

4. **防盗链**  
   - CDN 支持 **Signed URL**，在返回 `play` 接口时加上 HMAC‑SHA256 签名（有效期 5 min），防止 URL 被外泄。  

5. **扩展点**  
   - 转码服务使用 **K8s Job** + **GPU/CPU Auto‑Scaling**，峰值时自动增加 Pod。  
   - CDN 采用 **多厂商（阿里云、腾讯云、AWS CloudFront）** 多活，降低单厂商故障风险。  

### 3. 播放请求路由与 ABR

1. **请求路径**  

   ```
   客户端 --> API Gateway --> Play Service --> Redis (权限缓存) 
            --> OSS (元数据查询) --> CDN (分片拉流) 
   ```

2. **ABR（自适应码率）实现**  
   - **Manifest**（master.m3u8）列出不同 bitrate 的子 playlist。  
   - 客户端根据 **当前网络带宽**、**CPU** 动态切换分片。  
   - **切换无感知**：播放器在后台预下载下一个码率的分片，播放时切换 `src`。  

3. **缓存策略**  
   - **热点视频**的 `manifest` 与前 10 分片在 **Redis** 中缓存（TTL 10 min），避免每次都查询 OSS。  
   - **冷门视频**直接走 OSS → CDN，利用 CDN 的 **LRU** 缓存。  

4. **容错**  
   - 若 CDN 失效，Play Service 回退至 **OSS 直连**（HTTP Range），仍能保证播放，只是延迟稍高。  

### 4. 弹幕实时推送系统

1. **整体架构**  

   ```
   客户端 (WebSocket) <---> Edge Bullet Server <---> Redis ZSET / PubSub <---> Kafka Partition <---> Bullet Service (MongoDB)
   ```

2. **写入流程**  
   - 客户端发送弹幕 → Edge Server → **Kafka Producer**（topic: bullet_{video_id % N})。  
   - Kafka 消费后 **写入 MongoDB**（持久化）+ **写入 Redis ZSET**（实时排序）。  

3. **读取/推送**  
   - Edge Server 持续 **订阅 Redis ZSET**（`ZRANGE`）获取最新弹幕，使用 **WebSocket**（或 **QUIC**）推送给所有观看同一视频的客户端。  
   - 为保证 **顺序一致性**，所有弹幕在同一 **Kafka 分区** 内有严格的 offset 顺序，Edge Server 按 offset 顺序读取并推送。  

4. **延迟控制**  
   - **Kafka → Redis** 采用 **pipeline**，从消费到写入 Redis 平均 < 5 ms。  
   - **WebSocket 推送** 再加上网络 RTT（北京-上海约 30 ms），整体 ≤ 100 ms。  

5. **容错**  
   - Kafka 多副本（replication factor=3）保证写入不丢失。  
   - Edge Server 使用 **无状态**，可水平扩展，客户端会自动重新连接到最近的节点。  

### 5. 推荐系统（离线 + 实时）

1. **离线批处理**（每日/每 6h）  
   - 使用 **Spark** 从 **HDFS** 读取用户日志（观看时长、收藏、搜索）→ 生成 **用户画像向量** → 写入 **HBase**（列式存储）。  
   - 同步 **视频特征**（标签、热度、内容相似度）到 `item_features` 表。  

2. **实时召回**（秒级）  
   - **Flink** 监听 **Play Report**（Kafka）实时更新用户最近兴趣（滑动窗口），写回 **Redis**（用户实时热点列表）。  
   - 当客户端请求首页推荐时，**Recommendation Service** 先读 **Redis**（最近兴趣 + 热门榜单），若缓存未命中则查询 **HBase** 并做 **向量相似度** 召回（Approximate Nearest Neighbor, 如 HNSW）  

3. **缓存层**  
   - **Redis** 保存 **用户 → 推荐列表**（TTL 5 min），热点用户使用 **热点缓存**（LRU + 过期刷新）  
   - **全局热门榜**（每分钟更新）存放在 **Redis Sorted Set**，边缘 CDN 可直接拉取。  

4. **容错**  
   - Spark / Flink 作业采用 **Checkpoint** + **State Backend**（HDFS）实现故障恢复。  
   - 推荐 API 使用 **熔断**（Hystrix）保护，若 HBase 超时，直接返回 **缓存热点**。  

### 6. 会员计费与权限校验

1. **计费流程**  
   - **前端** 调用 `/checkout`，后端生成 **订单号**，调用 **第三方支付网关**（支付宝/微信）返回支付链接。  
   - 支付成功后，**支付网关**回调 `/notify`，后端校验签名 → 写入 **订单表**（MySQL） → 发送 **Kafka** 事件 `order_success`。  

2. **会员状态更新**  
   - **Member Service** 消费 `order_success`，更新 `user_wallet`、`user_membership`（有效期、类型）并写入 **Redis**（权限缓存）。  

3. **播放权限校验**  
   - Play Service 在返回 CDN URL 前，从 **Redis** 拉取 `user_id:membership`，判断是否有观看该视频的权限（免费/付费/会员）。  

4. **安全措施**  
   - 所有支付回调必须 **HTTPS + IP 白名单 + RSA 签名**。  
   - 数据库采用 **AES‑256** 加密存储关键字段（如支付流水号）。

## 第六步：扩展性与高可用设计

### 1. 弹性伸缩（Auto‑Scaling）

| 场景 | 触发指标 | 伸缩动作 |
|------|----------|----------|
| 播放峰值（如热门综艺首播） | CDN 边缘命中率 < 70% **或** API Gateway CPU > 70% | 自动 **扩容** Play Service Pod、Edge Bullet Server、Redis Cluster |
| 弹幕突增 | Kafka lag > 5,000 条/partition | 增加 **Kafka Consumer** 实例、扩容 Redis ZSET Shard |
| 推荐模型更新 | Flink 作业延迟 > 2 min | 动态调高 Flink 并行度、加大 HBase 读写预留容量 |

- 使用 **Kubernetes Horizontal Pod Autoscaler (HPA)** 结合 **Custom Metrics**（如 QPS、CPU、Kafka lag）实现自动伸缩。  

### 2. 跨地域灾备（Multi‑Active）

1. **多可用区（AZ）**  
   - 每个业务服务（Auth、Play、Bullet、Recommend）部署在 **3+ AZ**，使用 **负载均衡**（L4）进行跨 AZ 流量分发。  

2. **多地域（跨省）**  
   - 采用 **全局流量管理**（如阿里云 Global Server Load Balancer）把用户请求路由到最近的 **Region**。  
   - **数据同步**：  
     - **MySQL** 使用 **双向同步**（DRDS）或 **TiDB** 多活。  
     - **MongoDB** 使用 **跨地域复制集**。  
     - **对象存储** 使用 **跨地域复制**（OSS 同步到多个 region）。  

3. **故障切换**  
   - **健康检查**（TCP/HTTP）发现某 AZ/Region 故障后，DNS/GLB 自动剔除该节点。  
   - **服务降级**：如 CDN 故障，Play Service 回退到 **OSS 直连**；弹幕系统故障则回退到 **轮询长轮询**（Polling）模式。  

### 3. 缓存失效与预热策略

| 缓存层 | 失效原因 | 处理方式 |
|--------|----------|----------|
| CDN Edge | TTL 过期、热点视频被驱逐 | **预热脚本**：热门视频上线前，调用 CDN API `prefetch` 拉取前 30 s 分片；使用 **热点预测模型**（基于历史播放量）提前预热。 |
| Redis (metadata) | LRU 驱逐 | **双写**：写入 MySQL 同时写入 Redis；读不到时回源 MySQL 并 **异步写回** Redis（Cache Aside）。 |
| 推荐缓存 | 用户兴趣变化快 | **实时刷新**：Flink 产生的用户兴趣更新写入 Redis，TTL 5 min 自动失效；热点用户使用 **热点缓存**（永不过期，后台定时刷新）。 |

### 4. 数据安全与合规

- **传输层**：全部使用 **TLS 1.3**，HTTP/2 + QUIC（移动端）降低 RTT。  
- **存储加密**：对象存储开启 **服务端加密**（AES‑256），MySQL、MongoDB 使用 **磁盘加密**。  
- **审计日志**：所有敏感操作（登录、付费、弹幕删除）写入 **Kafka审计Topic** → ELK 集群供审计查询。  
- **GDPR/国内合规**：用户可以 **删除账号**，系统执行 **软删** → 30 天后硬删（批处理），并在对象存储中同步删除对应视频副本（仅对付费下载有效）。

## 第七步：常见面试追问与回答

下面列出面试官最可能追问的 **5 大热点**，并给出 **思路 + 关键点**，帮助你在现场快速组织语言。

### 1. **缓存失效与热点预热**  
> **问**：如果热点视频突然升温（比如明星直播），CDN 缓存不一定命中，如何保证首帧低延迟？  

**回答要点**  
- **多级缓存**：Edge CDN + 本地 Redis（metadata）+ 客户端缓存（MPEG‑DASH 的 `#EXT-X-MAP`）  
- **预热机制**：  
  1. **预测模型**（基于历史趋势、社交媒体热度）提前 5 min 标记为热点。  
  2. 自动调用 CDN **Prefetch API** 拉取前 30 s 分片到边缘节点。  
  3. 若仍未命中，**Play Service** 直接返回 **OSS 直连 URL**（带签名）并在后台 **异步写入 CDN**。  
- **回源容错**：CDN 失效时，使用 **HTTP 302** 重定向到 OSS，仍在 200 ms 内返回首帧（因为 OSS 同样部署在同城机房）。  

### 2. **弹幕顺序一致性与低延迟**  
> **问**：弹幕需要严格按照时间轴顺序展示，如何在分布式系统里保证？  

**回答要点**  
- **单分区写入**：同一视频的弹幕写入同一 **Kafka Partition**（`partition = video_id % N`），Kafka 天然保证 **offset 顺序**。  
- **消费端顺序**：Edge Bullet Server 只消费一个分区，保持 **offset** 单调递增。  
- **时间戳校准**：客户端发送的 `timestamp_ms` 必须是 **服务器时间**（后端统一校正），防止客户端时钟漂移。  
- **延迟控制**：Kafka → Redis **pipeline** < 5 ms；WebSocket 推送 + 网络 RTT ≈ 30 ms，总体 ≤ 100 ms。  

### 3. **跨地域灾备如何做到无感知切换**  
> **问**：如果华东 Region 故障，用户还能观看视频吗？  

**回答要点**  
- **全局负载均衡（GLB）**：DNS + Anycast 把用户请求路由到最近可用 Region。故障检测到后，GLB 自动剔除故障 Region。  
- **数据同步**：  
  - **对象存储**：OSS 多活复制，同一文件在华南、华北都有副本。  
  - **MySQL**：使用 **双主**（阿里云 PolarDB）实现跨 Region 同步，或使用 **TiDB** 的多活特性。  
  - **MongoDB**：跨地域副本集，写入后异步复制。  
- **CDN 回源**：即使某 Region 的边缘节点失效，CDN 会自动回源到最近的 OSS 副本，仍能提供流媒体。  
- **无感知**：客户端只感知 **首帧延迟略升**（从 150 ms 到 250 ms），业务层不需要特殊降级处理。  

### 4. **如何实现秒级弹性伸缩**  
> **问**：热门综艺首播时，QPS 瞬间冲到 200k，系统如何快速扩容？  

**回答要点**  
- **K8s HPA + Custom Metrics**：监控 **API Gateway QPS、CPU、Kafka lag**，当阈值超过设定值时自动 **ScaleOut** Pod。  
- **预热池**：在非高峰期保留 **冷备 Pod**（ReplicaSet 设为 5），快速加入 Service。  
- **无状态服务**：所有业务（Play、Bullet、Recommend）均为 **无状态**，只依赖外部存储，横向扩容不受 Session 影响。  
- **限流+熔断**：在流量超出预期时，先在 Gateway 做 **令牌桶限流**，保护后端不被击垮，同时把超出部分 **降级**（如弹幕改为轮询）。  
- **自动化部署**：使用 **GitOps**（Argo CD）配合 **Helm** Chart，快速滚动升级或回滚。  

### 5. **推荐系统的离线 vs 实时召回怎么配合**  
> **问**：离线模型每 6 小时跑一次，实时召回怎么保证不冲突？  

**回答要点**  
- **离线模型** 负责 **全局特征、长尾视频** 的向量计算，写入 **HBase**，周期性更新。  
- **实时召回** 只负责 **最近兴趣、热点榜单**，从 **Kafka** 消费用户行为（播放、点赞）实时更新 **Redis**（用户热点集合）。  
- **融合策略**：  
  1. **Score = α·offline_score + β·online_score**（α、β 动态调节）。  
  2. 推荐 API 先查询 **Redis**（快速返回）+ **HBase**（补齐缺口） → 合并去重。  
- **冲突解决**：若同一 video 同时出现在离线与实时结果，取 **最高 Score**，并在 **日志** 中标记供后续模型迭代。

## 心得与反思

### 本题最难的 1‑2 个设计决策及思考过程

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **弹幕的顺序一致性 & 低延迟** | 弹幕量大（每秒上万条），跨地域分布，必须保证 **全局时间顺序** 同时 **≤100 ms**。 | 1️⃣ 先分析业务：弹幕是 **写一次、读多次**，时间轴是唯一的排序键。<br>2️⃣ 选用 **Kafka** 分区保证顺序；分区数 = 视频并发量的上限（如 10k 分区）。<br>3️⃣ 为避免跨机房网络带来的时延，弹幕 Edge Server **就近**消费，直接写入 **Redis ZSET**（本地有序集合），再推送 WebSocket。<br>4️⃣ 通过 **pipeline**、**批量写入**降低写延迟，最终实现 ≤100 ms。 |
| **全局低延迟的首帧播放** | 200 ms 包括 DNS、TLS 握手、CDN 拉取、ABR 切片，用户遍布全国乃至海外。 | 1️⃣ 将 **用户 → CDN Edge** 的路径最短化：使用 **Anycast DNS + Edge CDN**。<br>2️⃣ 采用 **预签名 URL + HLS/DASH**，首段（1‑2 s）提前 **预热** 至边缘节点。<br>3️⃣ 若热点失效，**回源 OSS** 同城机房，仍能在 200 ms 内返回首帧。<br>4️⃣ 通过 **TLS 1.3 + QUIC** 减少握手往返次数。 |

### 新手最容易犯的错误（至少 2 条）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有数据都放进单一 MySQL** | 写入热点（弹幕、日志）会导致锁竞争、磁盘 IO 爆炸，整体 QPS 受限。 | 根据访问特性分库分表，弹幕使用 **Kafka + MongoDB/Redis**，视频元数据使用 **MySQL**，用户会话使用 **Redis**。 |
| **忽视缓存失效与热点预热** | 热门视频在高峰期会出现 CDN 失效、首帧卡顿，用户体验大幅下降。 | 实现 **热点预测 + CDN Prefetch**，并准备 **OSS 直连回源** 方案，使用 **Cache‑Aside** 模式保证缓存一致性。 |
| **把业务逻辑写在网关层** | 网关本应轻量，业务代码膨胀导致扩展困难、故障恢复慢。 | 将业务拆成 **微服务**，网关只做路由、鉴权、限流。 |
| **未做好降级/熔断** | 在高峰或故障时，单点故障会导致全链路不可用。 | 引入 **Hystrix/Resilience4j**，在后端超时/错误率升高时返回 **默认热点** 或 **降级页面**。 |

### 学习建议和可延伸的方向

1. **系统设计基础**  
   - 阅读《系统设计面试之道》《Designing Data‑Intensive Applications》；了解 **CAP、BASE、CQRS** 等概念。  
2. **分布式消息队列**  
   - 深入学习 **Kafka**（分区、复制、消费者组）和 **RabbitMQ** 对比，动手实现 **生产者/消费者**。  
3. **缓存与 CDN**  
   - 实践 **Redis** 的 **Sorted Set**、**LRU**、**热点缓存**；了解主流 CDN（阿里云、腾讯云、CloudFront）缓存策略和 **预热 API**。  
4. **流媒体协议**  
   - 学习 **HLS、DASH、RTMP、QUIC** 的工作原理，动手搭建 **FFmpeg** 转码流水线。  
5. **推荐系统入门**  
   - 了解 **协同过滤、矩阵分解、Embedding**，使用 **Spark MLlib** 或 **TensorFlow Recommenders** 做离线模型实验。  
6. **高可用与灾备**  
   - 熟悉 **Kubernetes** 的 **StatefulSet、PodDisruptionBudget**、**Service Mesh**（Istio）实现流量治理。  
7. **面试技巧**  
   - 练习 **结构化回答**：先说 **需求 → 估算 → 架构 → 关键技术 → 容错 → 监控**，每一步都配合 **为什么**。  

> **一句话总结**：系统设计不是“一次性画完图”，而是 **从需求出发、逐层抽象、一步步加上可扩展性与容错** 的过程。只要你在每一步都能解释“**为什么这么做**”，面试官会更关注你的思考方式而不是具体实现细节。祝你面试顺利，早日成为大厂后端高手！ 🚀
