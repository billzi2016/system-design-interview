# 第 35 天：设计 Amazon 购物车与结算

> 生成日期：2026-04-21

---

## 题目背景
Amazon 购物车与结算系统负责为全球数亿用户提供“加入购物车 → 统一结算 → 生成订单”全链路的购物体验。系统需要在高并发、跨地域的环境下保持低延迟、强一致性，并支持灵活的促销、库存检查等业务规则。

---

## 面试场景设定
> **面试官**：  
> “假设我们要为 Amazon 重新设计一套购物车 + 结算系统，使其能够支撑全球用户在大促期间的海量请求，并且在用户点击‘结算’后能够在毫秒级给出可支付的订单。请先从高层次阐述你的系统拆分思路，然后逐步展开关键模块的设计。”

---

## 功能性需求

| 编号 | 需求描述 |
|------|----------|
| 1 | **添加/删除/修改商品**：用户可以把商品加入购物车、删除已选商品、修改购买数量或规格。 |
| 2 | **购物车持久化**：登录后跨设备、跨会话同步购物车；匿名用户使用本地缓存或 cookie，后续登录后合并。 |
| 3 | **库存与价格实时校验**：在加入购物车、修改数量、结算前均需校验最新库存和价格，防止超卖或价格错误。 |
| 4 | **优惠/促销计算**：支持满减、折扣码、组合促销、会员优惠等，在结算页实时展示最终应付金额。 |
| 5 | **结算流程**：生成唯一订单号、锁定库存、创建支付流水、返回可支付的订单详情（含收货地址、运费、税费）。 |
| 6 | **幂等与容错**：用户可能因网络抖动重复提交结算请求，系统必须保证幂等性并能安全回滚。 |

---

## 非功能性需求

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **日活跃用户 (DAU)** | 2.5 亿 | 包括登录与匿名用户 |
| **峰值 QPS（加入购物车）** | 120,000 QPS | 大促（Prime Day）期间 |
| **峰值 QPS（结算）** | 30,000 QPS | 大促期间，结算占比约 25% |
| **平均响应时延** | < 150 ms (加入/删除) <br> < 300 ms (结算) | 端到端，包括库存/优惠校验 |
| **可用性** | 99.99%（年均累计停机 ≤ 52 分钟） | 业务容忍极低的不可用 |
| **存储规模** | 约 500 TB（购物车快照 + 订单元数据） | 需支撑 90 天历史购物车数据，满足审计需求 |
| **数据一致性要求** | 强一致性（结算）<br> 最终一致性（购物车同步） | 结算阶段必须确保库存、价格、优惠的强一致 |
| **容灾恢复时间目标 (RTO)** | ≤ 5 分钟 | 多活跨地域容灾 |

## 系统边界

**本题需要考虑的范围**  
- 购物车的 API、持久化、跨设备同步。  
- 结算入口的订单生成、库存锁定、优惠计算、支付流水创建。  
- 高并发访问的水平扩展、缓存层、异步降压、幂等设计。  
- 跨地域数据复制（如多活写入、读写分离）以及容灾方案。  

**不在本题范围**  
- 商品搜索、详情页渲染、推荐系统。  
- 支付网关的具体实现（仅需模拟支付流水的创建）。  
- 物流配送、退货/退款、售后服务。  
- 前端 UI/UX 细节，只关注后端接口与服务设计。

# 题解

## 解题思路总览
本篇答案把 **“从零到可在全球大促中支撑 120k QPS 加入购物车、30k QPS 结算的完整系统”** 拆解成若干层次，**一步一步、由浅入深** 进行讲解。  
- **先抓住业务核心**：购物车的 CRUD、持久化、跨设备同步 → 结算的强一致性、库存锁定、优惠计算 → 幂等与容错。  
- **再把业务映射到技术组件**：API 网关、服务层、缓存、持久化、消息队列、分布式锁等。  
- **每加入一个新组件，都要问**：*为什么需要它？不加会出现什么问题？*  
- **最后再考虑扩展性、容灾、监控**，形成完整的高可用分布式方案。  

下面的章节会遵循 **“最小可用系统 → 逐步进化 → 高可用/跨地域”** 的路线图，帮助新手把握 **“需求 → 分解 → 设计 → 优化”** 的完整思考过程。

## 第一步：理解需求与规模估算

### 1. 功能需求拆解  

| 编号 | 业务点 | 关键子功能 | 强一致性要求 |
|------|--------|-----------|--------------|
| 1 | 购物车 CRUD | 加入、删除、改数量/规格 | 最终一致即可（用户感知可接受短暂不一致） |
| 2 | 购物车持久化 & 跨设备同步 | 匿名缓存 → 登录合并 → 多端同步 | 最终一致 |
| 3 | 库存/价格实时校验 | 加入、改数量、结算前校验 | 结算时必须强一致 |
| 4 | 优惠/促销计算 | 满减、优惠码、会员价、组合促销 | 结算时强一致（防止优惠被滥用） |
| 5 | 结算流程 | 生成订单号、锁库存、创建支付流水、返回可支付订单 | 强一致 |
| 6 | 幂等 & 容错 | 重复提交、网络抖动 | 强一致（幂等性必须保证） |

### 2. 非功能需求量化  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| DAU | 2.5 亿 | 包括登录/匿名 |
| 峰值加入购物车 QPS | 120 k | 大促期间 |
| 峰值结算 QPS | 30 k | 大促期间 |
| 响应时延 | <150 ms (加入/删除) <br> <300 ms (结算) | 包括外部依赖（库存、优惠） |
| 可用性 | 99.99%（≈52 min/yr） | 业务容忍极低的不可用 |
| 存储规模 | 500 TB (购物车快照 + 订单) | 90 天历史数据 |
| RTO | ≤5 min | 多活跨地域容灾 |

> **规模换算**（帮助新手感受压力）  
- 加入购物车：120 k QPS ≈ **1.04 亿次/天**。  
- 结算：30 k QPS ≈ **2.6 亿次/天**（假设每次结算产生 1 条订单）。  
- 假设每条购物车记录 2 KB、每条订单元数据 5 KB，90 天数据量轻易超过 **500 TB**。

### 3. 关键瓶颈点  
1. **库存同步**：必须在结算时同步到库存系统，防止超卖。  
2. **优惠计算**：涉及多规则，计算复杂度高。  
3. **跨地域同步**：购物车需要在不同地域快速同步，网络延迟是挑战。  
4. **幂等/事务**：结算涉及多资源（库存、订单、支付），必须保证原子性。

## 第二步：高层架构设计

### 1. MVP（最小可用系统）  
```
[Client] → API Gateway → Cart Service ↔ Cart DB (MySQL)
                     ↓
               Checkout Service ↔ Order DB (MySQL)
                     ↘
                 Inventory Service (sync)
                 Promotion Service (sync)
```
- **API Gateway**：统一入口，做路由、鉴权、流量控制。  
- **Cart Service**：提供购物车 CRUD，使用 **关系型数据库**（强一致读写）保存持久化数据。  
- **Checkout Service**：负责结算，内部调用库存、优惠等同步服务。  
- **Inventory / Promotion Service**：提供实时查询（读‑写分离的缓存层）  

**为什么这样设计？**  
- 业务拆分清晰，职责单一，便于后续水平扩展。  
- 使用 **关系型 DB** 能快速实现强一致的事务（对 MVP 足够）。  
- 不引入分布式锁、消息队列等复杂组件，降低实现风险。  

### 2. 逐步进化的完整高可用架构  

```
                               ┌─────────────────────┐
                               │  Global CDN / WAF   │
                               └───────┬─────┬───────┘
                                       │     │
                               ┌───────▼─────▼───────┐
                               │   API Gateway (R)   │
                               └───────┬─────┬───────┘
            ┌───────────────────────▼─────▼───────┐
            │  Region A                     Region B │
            │  (US‑East)                     (EU‑West)│
            └───────┬─────┬───────┬───────┬───────┘
                    │     │       │     │
          ┌─────────▼─┐ ┌─▼─────────▼─┐ ┌─▼─────────┐
          │  Cart Svc │ │ Checkout Svc│ │ Cart Svc │
          └─────┬─────┘ └─────┬───────┘ └─────┬─────┘
                │            │               │
        ┌───────▼─────┐ ┌────▼─────┐   ┌─────▼─────┐
        │ Cart Cache │ │ Order DB│   │ Cart DB   │
        └───────┬─────┘ └────┬─────┘   └─────┬─────┘
                │            │               │
                │            ▼               │
                │   ┌─────────────────┐      │
                └──►│ Inventory Svc   │◄─────┘
                    └───────┬─────────┘
                            │
                        ┌───▼─────┐
                        │ Promo Svc│
                        └─────────┘
```
- **多活 Region**：每个 Region（如美洲、欧洲、亚洲）都有完整的 **Cart/Checkout** 服务，做到 **读写分离 + 本地化**，降低跨地域 RTT。  
- **全局负载均衡 + CDN**：把流量路由到最近的 Region，提升用户感知时延。  
- **缓存层（Redis / DynamoDB Accelerator）**：购物车的热点数据、库存/促销的只读副本，减少 DB 直接压力。  
- **异步消息（Kafka / Pulsar）**：库存扣减、订单日志、审计等写操作通过可靠消息队列实现 **最终一致**，并提供 **回滚** 能力。  
- **分布式锁（Redis RedLock / Zookeeper）**：结算阶段对同一 SKU 的库存进行 **预扣**，防止并发超卖。  
- **幂等键存储（MySQL / DynamoDB）**：每一次结算请求的唯一 **request_id** 写入表，重复请求直接返回已有结果。  

> **不这样做的后果**  
- 只在单 Region 部署，网络跨境时延 > 300 ms，无法满足 150 ms 响应目标。  
- 直接在业务服务里写库不使用缓存，峰值 QPS 时 DB 直接崩溃。  
- 没有幂等控制，用户重复点击“立即支付”会产生多笔订单，造成财务混乱。

## 第三步：数据库设计

### 1. 购物车数据模型（MySQL + Redis Cache）

| 表名 | 说明 | 主键 | 关键字段 |
|------|------|------|----------|
| `cart` | 每个用户的购物车主表 | `user_id` | `updated_at` |
| `cart_item` | 购物车中每个商品的明细 | (`user_id`, `sku_id`) | `quantity`, `price_at_add`, `selected` |
| `cart_snapshot` | 90 天历史快照（归档） | `snapshot_id` (auto) | `user_id`, `data_json`, `created_at` |

**为什么拆两张表？**  
- `cart` 用来快速判断用户是否已有购物车，支持 **分区**（基于 `user_id` hash），写入成本低。  
- `cart_item` 记录每个 SKU，使用 **复合主键** 保证同用户同 SKU 只能有一条记录，方便 **upsert**（INSERT … ON DUPLICATE KEY UPDATE）。  

**缓存设计**  
- **Redis Hash**：`cart:{user_id}` → `{sku_id}:{quantity}:{price_at_add}:{selected}`  
- **TTL**：30 天（匿名用户）或永久（登录用户）。  
- **写穿/读穿**：写操作先更新 DB 再 **异步写缓存**（Kafka → Consumer），读操作先查缓存，缓存未命中再回源 DB。  

### 2. 订单与结算模型（强一致事务）

| 表名 | 说明 | 主键 | 关键字段 |
|------|------|------|----------|
| `order` | 订单头信息 | `order_id` (UUID) | `user_id`, `status`, `total_amount`, `created_at` |
| `order_item` | 订单明细 | (`order_id`, `sku_id`) | `quantity`, `price`, `discount_amount` |
| `order_payment` | 支付流水 | `payment_id` | `order_id`, `pay_status`, `pay_amount`, `pay_time` |
| `order_snapshot` | 归档历史（90 天） | `snapshot_id` | `order_id`, `json_blob`, `archived_at` |
| `checkout_idempotency` | 幂等键表 | `request_id` | `order_id`, `status`, `created_at` |

**事务边界**  
- 结算时在 **单个 MySQL 实例**（或使用 **分布式事务框架**）开启 **可重复读**事务：  
  1. 检查库存（`SELECT … FOR UPDATE`）  
  2. 计算优惠（调用 Promo Service，返回折扣）  
  3. 写入 `order`、`order_item`、`order_payment`  
  4. 更新库存（`UPDATE inventory SET available = available - ? WHERE sku_id=? AND version=old_version`）  
  5. 插入幂等键  

如果任何一步失败，事务回滚，保证 **原子性**。

### 3. 库存模型（强一致）  

| 表名 | 说明 | 主键 | 关键字段 |
|------|------|------|----------|
| `inventory` | SKU 库存 | `sku_id` | `available`, `reserved`, `version` |
| `inventory_log` | 库存变更日志（审计） | `log_id` | `sku_id`, `delta`, `type`, `order_id`, `created_at` |

- **乐观锁** (`version`) 用于防止并发扣减失败。  
- **预扣/释放**：结算时先 `UPDATE … SET reserved = reserved + qty, version = version + 1 WHERE sku_id=? AND available - reserved >= qty`，成功后再在事务提交后 **真正扣减**（`available = available - qty, reserved = reserved - qty`）。

## 第四步：核心 API 设计

> **原则**：RESTful + 幂等、统一错误码、响应体结构化、兼容前端缓存（ETag、If-None-Match）。

| 方法 | 路径 | 功能 | 请求体 | 响应 | 幂等性 |
|------|------|------|--------|------|--------|
| POST | `/cart/items` | 加入商品 | `{ "sku_id": "123", "quantity": 2, "spec": {...} }` | `201 Created` + `cart_item` | **非幂等**（会累加），**可通过** `PUT /cart/items/{sku_id}` 实现幂等 |
| PUT | `/cart/items/{sku_id}` | 更新数量/规格 | `{ "quantity": 3, "spec": {...} }` | `200 OK` | **幂等** |
| DELETE | `/cart/items/{sku_id}` | 删除商品 | - | `204 No Content` | **幂等** |
| GET | `/cart` | 获取完整购物车 | - | `{ "items": [...], "total_price": … }` | **幂等** |
| POST | `/checkout` | 发起结算 | `{ "request_id": "uuid", "address_id": "...", "promo_code": "...", "payment_method": "ALIPAY" }` | `{ "order_id": "...", "pay_url": "...", "expires_at": "..." }` | **幂等**（`request_id` 防重） |
| GET | `/order/{order_id}` | 查询订单状态 | - | `{ "status": "...", "amount": ..., "items": [...] }` | **幂等** |

### 幂等实现细节（Checkout）

1. **客户端生成全局唯一 `request_id`**（UUID v4）。  
2. **Checkout Service 在入口先查 `checkout_idempotency` 表**：  
   - 若存在且 `status=SUCCESS`，直接返回已生成的 `order_id` 与支付信息。  
   - 若不存在，则插入 `request_id`（状态 `PENDING`），继续后续业务。  
3. **事务提交后**，更新 `checkout_idempotency` 为 `SUCCESS` 并填充 `order_id`。  
4. **异常回滚**：把记录标记为 `FAILED`，客户端可重新发起。  

> **不做幂等的风险**：用户因网络抖动重复提交，会产生 **重复订单**、**库存多扣**、**支付多扣**，导致极其难以排查的脏数据。

## 第五步：详细组件设计

### 1. API Gateway  

- **功能**：路由、限流、鉴权、统一日志、灰度发布。  
- **实现**：使用 **Kong / Envoy**（支持插件化）或 **AWS API Gateway**。  
- **限流策略**：  
  - **全局 QPS**：120k / 30k 对应每个 Region 分配 1/3‑1/4。  
  - **用户级**：每秒最多 5 次加入、2 次结算，防止刷单。  
- **安全**：OAuth2 + JWT，JWT 中携带 `user_id`、`device_id`。  

### 2. Cart Service  

#### 2.1 流程图（文字版）

```
[Client] → API GW → Cart Service
   ↓            (鉴权)
   → 检查 Redis Cache (cart:{user_id})
       ↙                ↘
   命中 → 返回缓存
   未命中 → 查询 MySQL → 写入 Redis → 返回
   写操作 → 先写 MySQL (INSERT/UPDATE) → 发送 Kafka “cart_change” → 异步刷新 Redis
```

#### 2.2 关键技术点  

| 场景 | 关键实现 | 为什么这样 |
|------|----------|-----------|
| **加入/更新** | `INSERT … ON DUPLICATE KEY UPDATE` + **Kafka** 事件 | 保证 DB 事务成功后再异步更新缓存，避免缓存击穿 |
| **删除** | `DELETE FROM cart_item WHERE user_id=? AND sku_id=?` → 同步 `cart_change` | 同上，保持缓存最终一致 |
| **跨设备合并** | 登录后：读取匿名缓存 → 调用 `MergeCart(user_id, anon_cart)` → **幂等**的批量 upsert | 合并时避免重复写入、冲突（后写覆盖前写） |
| **离线/冲突** | 使用 **版本号（client_version）**，冲突时采用 “最新时间戳胜出” 或 “用户手动选择” | 防止网络抖动导致的**写丢失** |

#### 2.3 缓存失效策略  

- **TTL**：30 天（匿名） / 永久（登录）  
- **主动失效**：写成功后立即 `DEL cart:{user_id}`，下次读再拉回 DB（读‑写分离，避免脏读）  
- **热点预热**：大促期间对活跃用户提前加载缓存，使用 **Bloom Filter** 判断是否可能存在缓存，降低 Redis 负载  

### 3. Checkout Service  

#### 3.1 流程（事务 + 异步）

```
1. 接收 checkout 请求 (request_id)
2. 幂等校验 → 已完成直接返回
3. 开启 DB 事务 (READ COMMITTED)
4. 读取用户购物车 (FOR UPDATE) → 获取商品列表
5. 调用 Inventory Service (预扣库存) → 成功返回
6. 调用 Promotion Service (计算优惠) → 返回折扣明细
7. 计算最终应付金额
8. 写入 Order、OrderItem、Payment、Idempotency 表
9. 提交事务
10. 发送 Kafka “order_created” → downstream (支付、日志、审计)
```

#### 3.2 库存预扣细节  

- **预扣**：在事务中对 `inventory` 表执行 `UPDATE … SET reserved = reserved + qty, version = version + 1 WHERE sku_id=? AND available - reserved >= qty`。  
- **失败回滚**：若 `affectedRows = 0`，说明库存不足，立即返回 **库存不足错误**，事务回滚。  
- **最终扣减**：在订单创建成功后，消费 `order_created` 消息的 **Inventory Consumer** 执行真正的 `available = available - qty, reserved = reserved - qty`。这样即使 Checkout Service 崩溃，预扣的库存仍在 `reserved` 中，后续补偿程序可以释放（如果超时未生成订单）。

#### 3.3 优惠计算  

- **Promo Service** 提供 **RESTful** 接口 `POST /promo/calc`，输入：商品列表、用户信息、优惠码。  
- **返回**：每个 SKU 的折扣、整体满减、优惠码抵扣、最终应付。  
- **缓存**：促销规则经常变化但查询频繁，使用 **Redis** 进行 **热点规则缓存**，并在规则变更时 **发布 Kafka “promo_update”**，让所有节点同步。  

#### 3.4 幂等实现  

- **Request ID**：业务层先 `INSERT IGNORE INTO checkout_idempotency(request_id, status='PENDING')`。  
- **事务内**：如果成功生成订单，`UPDATE checkout_idempotency SET status='SUCCESS', order_id=?, updated_at=NOW() WHERE request_id=?`。  
- **重复请求**：直接查询该表返回已有 `order_id` 与支付信息。  

### 4. Inventory Service  

- **读写分离**：主库 MySQL 保存 `available`、`reserved`，从库提供查询（强一致通过 `read-after-write` 机制）。  
- **高并发锁**：使用 **乐观锁**（`version`） + **数据库行锁** (`SELECT … FOR UPDATE`) 来避免 **超卖**。  
- **降级方案**：如果库存查询延迟 > 30 ms，直接返回 **“库存不可用”**，前端可提示“稍后再试”，保护后端。  

### 5. Promotion Service  

- **规则存储**：使用 **MongoDB**（文档模型）或 **Redis JSON** 保存 **规则树**（如：`{type: "FULL_REDUCTION", threshold: 200, discount: 20}`）。  
- **规则引擎**：基于 **Drools** 或自研 **Rule DSL**，在每次结算时 **一次性计算**，避免多次调用。  
- **高可用**：部署 **双活** 实例，使用 **Consul** 做服务发现与健康检查。  

### 6. 异步消息系统（Kafka）  

| Topic | 生产者 | 消费者 | 目的 |
|-------|--------|--------|------|
| `cart_change` | Cart Service | Cart Cache Updater | 同步缓存 |
| `order_created` | Checkout Service | Inventory Consumer、Payment Consumer、Audit Service | 库存扣减、生成支付流水、审计日志 |
| `promo_update` | Promo Admin | Promo Service Instances | 同步促销规则缓存 |
| `order_status` | Payment Service | Notification Service | 发送短信/邮件 |

- **可靠性**：开启 **ACK=all**，副本数 3，确保不丢失。  
- **幂等消费**：消费者使用 **Message ID + DB 唯一键**，防止重复处理。  

### 7. 监控 & 报警  

| 指标 | 阈值 | 目的 |
|------|------|------|
| **API QPS** | > 80% 峰值阈值 | 提前扩容 |
| **Cart Service 99th latency** | > 150 ms | 检查缓存击穿 |
| **Checkout 事务回滚率** | > 0.1% | 检查库存/优惠冲突 |
| **Kafka lag** | > 5000 msgs | 防止消息积压 |
| **Redis miss rate** | > 20% | 增加缓存容量或热点预热 |
| **Error rate (5xx)** | > 0.5% | 自动触发降级 |

使用 **Prometheus + Grafana** 收集、可视化；**Alertmanager** 发送 PagerDuty / 微信报警。

## 第六步：扩展性与高可用设计

### 1. 横向扩展（Scale‑out）

| 层级 | 扩容方式 | 关键点 |
|------|----------|--------|
| **API Gateway** | 增加实例 + L4/7 负载均衡（AWS ALB / Nginx） | 会话保持（Sticky）对购物车非必须 |
| **Cart Service** | **无状态**，通过 **Consistent Hash** 将请求路由到对应 Redis 分片 | 业务只依赖 DB & Cache，水平扩容无锁 |
| **Checkout Service** | **有状态事务**，但可以 **分区**（如按 `user_id` hash）并使用 **分布式事务**（Seata） | 防止单点事务瓶颈 |
| **Redis Cache** | **分片（Cluster）**，每片 6 主/1 备，自动故障转移 | 需要做好键的散列，防止热点 |
| **MySQL** | **读写分离** + **水平分区**（按 `user_id` 范围或 `order_id` 日期） | 通过 **ProxySQL** 实现路由，避免热点表锁 |

### 2. 多活跨地域容灾

1. **数据复制**  
   - **Cart DB**：使用 **MySQL Group Replication**（多主）或 **TiDB**（分布式 SQL）实现 **跨 Region 双向同步**（最终一致）。  
   - **Order DB**：采用 **双活** + **全局唯一 ID（Snowflake / UUID）**，写入本地 Region，**异步复制**到其他 Region 作为备份。  

2. **故障切换流程**  
   - **监控发现 Region A 网络异常** → 自动把 **DNS**（Route53）指向 **Region B** 的 API GW。  
   - **用户会话迁移**：因为 **JWT** 已经携带 `user_id`，只要 **Cart Service** 能在 B 区读取 A 区的同步数据即可。  
   - **RTO ≤ 5 min**：使用 **自动化脚本**（Terraform + Ansible）在 5 分钟内完成流量切换和数据库只读/写切换。  

3. **数据冲突解决**  
   - **购物车**：采用 **CRDT（Last‑Write‑Wins）** 或 **版本向量**，冲突时保留最新修改时间的记录。  
   - **订单**：因为订单号全局唯一，冲突概率极低；若出现双写，使用 **幂等键** 防止重复。  

### 3. 防止超卖的综合方案

| 步骤 | 手段 | 说明 |
|------|------|------|
| **预扣** | `reserved` 字段 + 乐观锁 | 结算时先预留库存，防止并发扣减 |
| **限流** | Token Bucket（每 SKU 限流） | 防止同一 SKU 瞬时请求冲击 |
| **排队** | **Redis Stream** 或 **Kafka** 排队 | 对热点 SKU（秒杀）使用排队，先入先出 |
| **回滚** | **Compensation Service**（30 s 未生成订单释放 `reserved`） | 防止因服务崩溃导致库存永久锁定 |
| **监控** | 库存 `reserved / available` 比例报警 | 实时发现异常预扣增长 | 

### 4. 降级与熔断

- **购物车读取失败** → 返回 **空购物车**，前端提示“已失效，请重新加载”。  
- **优惠服务不可用** → 使用 **默认优惠（无优惠）**，并在后端补偿。  
- **库存服务超时** → **快速失败**，返回 “库存查询异常，请稍后重试”。  
- 使用 **Hystrix / Resilience4j** 实现 **熔断**，防止单点故障扩散。

## 第七步：常见面试追问与回答

### 1. “如果在大促期间，某商品的库存被抢购一空，用户已经把它加入购物车并进入结算，你会如何保证用户体验并避免超卖？”

**回答思路**  
1. **预扣+超时回滚**：结算阶段先在 `inventory` 表预扣 `reserved`，若 30 s 内未成功生成订单则自动释放。  
2. **实时库存回显**：在购物车页面使用 **库存查询 API**（读‑写分离）实时返回 `available - reserved`，若为 0，前端显示 “已抢完”。  
3. **排队/秒杀**：对热点 SKU 使用 **Redis Stream** 排队，先排队后抢购，给用户排队等待的感知（如倒计时）。  
4. **友好提示**：若结算时库存不足，立即返回错误码 **OUT_OF_STOCK**，前端弹窗建议 “改选其他规格” 或 “稍后再试”。  

**不采用**：仅在结算时检查库存（不预扣）会出现 **并发超卖**，导致订单异常、用户投诉、财务损失。  

### 2. “购物车数据需要跨设备同步，考虑到网络抖动和离线情况，你会如何设计数据一致性和冲突解决？”

**回答要点**  
- **最终一致**：采用 **CRDT（LWW‑Element‑Set）** 或 **版本向量**。每次写入都带上 **timestamp**（服务器时间）和 **client_version**。  
- **同步机制**：  
  1. 客户端每隔 5 s 发 `GET /cart/changes?since=last_sync_ts` 拉取增量。  
  2. 本地修改时先写 **本地缓存**，随后 `POST /cart/sync` 把增量上传，服务器按 **timestamp** 合并。  
- **冲突规则**：  
  - 同一 SKU 的 **数量** 采用 **最新时间戳** 覆盖。  
  - **选中/未选中** 状态同理。  
  - **删除** 操作带 `deleted_at`，比任何更新都新。  
- **离线场景**：本地保存所有操作日志（SQLite），网络恢复后一次性批量同步。  

**不这样做的风险**：直接覆盖会导致 **写丢失**，用户在两台设备分别修改同一商品会出现不一致的购物车，影响体验。  

### 3. “请说明你会如何实现结算请求的幂等性，尤其是在用户网络不稳定、重复提交的场景下。”

**实现细节**  
- **客户端**：在每次结算前生成 **全局唯一的 `request_id`**（UUID v4），随请求一起发送。  
- **服务端**：在 Checkout Service 入口使用 **幂等键表** `checkout_idempotency(request_id, status, order_id, created_at)`。  
  - **步骤**：  
    1. `INSERT IGNORE INTO checkout_idempotency(request_id, status='PENDING')` → 若已存在直接查询。  
    2. 在事务内部完成订单生成后 `UPDATE checkout_idempotency SET status='SUCCESS', order_id=?, updated_at=NOW() WHERE request_id=?`。  
    3. 若事务回滚，更新为 `FAILED`。  
- **查询**：重复请求时直接返回 **已生成的 `order_id` 与支付信息**，不再次扣库存。  

**关键点**：幂等键必须 **持久化**（MySQL/DynamoDB），且 **唯一**，否则在服务宕机后可能丢失状态导致重复下单。  

### 4. “如果库存服务不可用，结算还能继续吗？如何设计容错？”

- **熔断**：在 Checkout Service 调用 Inventory Service 前使用 **Hystrix** 检查熔断状态。  
- **降级**：如果熔断，直接返回 **库存不可用** 错误，前端提示 “系统繁忙，请稍后”。**不** 直接放行，否则会产生超卖。  
- **补偿**：若库存服务恢复后，**补偿任务**（定时任务）检查 **`order_status = PENDING`** 且 **库存已锁定**，完成扣减或取消订单。  

### 5. “如何保证全局唯一的订单号且可追踪？”

- **雪花算法（Snowflake）**：64 位 ID = **timestamp(41) + datacenter_id(5) + machine_id(5) + sequence(12)**，能够在 69 年内保证全局唯一且递增。  
- **业务编码**：在 Snowflake 基础上再拼接 **业务前缀**（如 `ORD-20240527-`），方便运维查询。

## 心得与反思

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **库存预扣 + 预留机制** | 必须在高并发下既防止超卖，又避免长时间占用库存导致“假缺货”。 | 1) 先评估库存的 **读‑写比例**（写占 30%），决定使用 **行锁** + **乐观锁**。<br>2) 引入 `reserved` 字段，将 “预扣” 与 “真正扣减” 分离。<br>3) 设计 **回滚补偿**（30 s 未完成订单自动释放），防止因异常导致库存永远锁定。 |
| **跨地域购物车最终一致 + 冲突解决** | 跨地域复制延迟可能导致用户在两台设备看到不同购物车，冲突时怎样合并？ | 1) 明确 **业务容忍度**：购物车只需要 **最终一致**，不必强实时。<br>2) 选用 **CRDT**（LWW‑Element‑Set）或 **版本向量**，每次写都带时间戳。<br>3) 通过 **增量同步**（`since` 参数）降低网络带宽。<br>4) 实现 **离线日志**，保证网络抖动时仍能恢复。 |

### 2. 新手最容易犯的错误（至少 2 条）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有业务都塞进单个单体服务** | 难以水平扩展，单点故障导致整套系统不可用。 | 按业务边界拆分为 **Cart Service、Checkout Service、Inventory Service、Promotion Service**，每个服务保持 **无状态**（除必要的事务）。 |
| **在结算阶段只做“查询库存”而不做锁/预扣** | 并发请求会导致 **超卖**，产生订单与实际库存不符的灾难。 | 在结算事务中使用 **FOR UPDATE** 或 **乐观锁**，预先 **reserved** 库存，确保并发安全。 |
| **忽视幂等性** | 用户网络抖动会产生 **重复订单、重复扣款**，影响财务与用户体验。 | 为 **每一次结算** 引入 **全局唯一 request_id**，在 DB 中保存幂等键，重复请求直接返回已有结果。 |
| **只用本地缓存，不做缓存失效/同步** | 缓存击穿导致瞬时 DB 瓦片，甚至出现 **脏读**（用户看到过时库存）。 | 实现 **写穿、读穿**、**主动失效**，并通过 **Kafka** 异步同步缓存，监控 **Cache Miss Rate**。 |

### 3. 学习建议和可延伸的方向

1. **系统设计基础**  
   - 熟悉 **CAP**、**BASE**、**一致性模型**，了解何时选强一致、何时选最终一致。  
   - 学习 **微服务划分原则**（单一职责、业务聚合、数据所有权）。  

2. **事务与分布式事务**  
   - 掌握 **两阶段提交（2PC）**、**Saga**、**TCC**、**Seata** 等分布式事务方案。  
   - 实践 **MySQL 行锁、乐观锁** 与 **Redis 分布式锁** 的使用场景。  

3. **高并发 & 限流**  
   - 了解 **令牌桶、漏桶、滑动窗口** 等限流算法，能在代码层实现或使用 **Envoy/NGINX**。  

4. **消息队列与幂等消费**  
   - 熟悉 **Kafka**（分区、副本、消费组）以及 **幂等消费** 的实现方式。  

5. **缓存设计**  
   - 学习 **Redis** 的数据结构（Hash、Sorted Set、Stream）以及 **缓存失效策略**（TTL、主动失效、热点预热）。  

6. **容灾与多活**  
   - 了解 **MySQL Group Replication、TiDB、CockroachDB** 等多活数据库。  
   - 掌握 **DNS Failover、Global Load Balancer**、**RTO/RPO** 的设计。  

7. **实战项目**  
   - 可以在个人项目或开源项目中实现 **简化版购物车/结算**，重点练习 **幂等、事务、缓存同步**。  

> **一句话总结**：系统设计的核心是 **“把业务拆成最小的强一致单元”，在这些单元上使用 **事务/锁** 保证一致性，在它们之间用 **异步、最终一致** 来提升可用性与伸缩性。**
