# 系统设计面试题：腾讯文档（Tencent Docs）

## 1. 题目背景
腾讯文档是面向企业和个人用户的在线文档编辑平台，提供文字、表格、演示等多种文档类型的 **实时协同编辑**、**云端存储** 与 **多端同步** 能力。用户可以在浏览器或移动端共同编辑同一份文档，并随时查看历史版本与评论。

## 2. 面试场景设定
> **面试官**：  
> “我们现在要设计一套能够支撑 **全球数亿用户** 同时使用的 **实时协同编辑** 系统——腾讯文档。请你从零开始搭建核心架构，重点考虑 **高并发实时同步**、**数据一致性** 与 **高可用**。先请你梳理一下系统的主要功能和非功能需求，然后我们再逐步深入细节。”

## 3. 功能性需求（Core Features）

| 编号 | 功能描述 |
|------|----------|
| 1 | **实时协同编辑**：多用户在同一文档上同时编辑，编辑操作在 200 ms 内同步到其他在线用户的页面。 |
| 2 | **文档权限管理**：支持查看、评论、编辑、管理员四种角色；可以通过链接、企业组织或微信/QQ 授权进行共享。 |
| 3 | **版本控制 & 历史回溯**：自动保存每一次提交，用户可以查看、比较并恢复任意历史版本。 |
| 4 | **离线编辑与同步**：客户端可在离线状态下编辑，恢复网络后自动合并冲突并同步至云端。 |
| 5 | **评论/批注 & @ 提及**：用户可以在文档任意位置添加评论、回复并 @ 其他成员，通知实时推送。 |
| 6 | **多格式导入/导出**：支持 Word、Excel、PDF 等常见格式的上传、预览以及导出。 |

> **注**：本题关注 **1‑4** 四个功能的实现细节，5、6 可作简要说明。

## 4. 非功能性需求（Key Metrics）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| DAU（日活跃用户） | **2 亿** | 包括企业版与个人版用户。 |
| 同时在线编辑文档数 | **150 万** | 峰值时段的并发编辑文档数量。 |
| QPS（编辑操作请求） | **200,000 ops/s** | 每秒编辑指令（字符插入/删除、格式变更等）请求数。 |
| 端到端延迟 | **≤ 200 ms**（95% 以内） | 编辑操作从客户端发送到其他客户端可见的总时延。 |
| 可用性 | **99.9%**（月均停机 ≤ 43.2 分钟） | 关键协同服务（同步服务、权限服务）需保持高可用。 |
| 存储规模 | **≥ 500 PB**（文档原始数据 + 版本历史） | 采用对象存储，支持冷热分层。 |
| 数据一致性 | **弱一致 + 业务层冲突解决** | 实时编辑对强一致要求不高，最终一致即可；离线同步需保证冲突可检测、可回滚。 |

## 5. 系统边界

- 计费、企业租户隔离、审计日志等运营层面功能。

## 解题思路总览  

1. **先把需求拆成**  
   - **功能需求**（实时协同编辑、权限、版本、离线同步）  
   - **非功能需求**（并发、延迟、可用、存储、数据一致性）  
2. **做规模估算** → 为后面的容量规划、选型提供依据。  
3. **从最小可用系统（MVP）出发**：只实现核心编辑同步 + 简单存储，快速验证可行性。  
4. **逐步加入**：分布式协同服务、权限服务、版本库、离线合并、消息推送、容灾。  
5. **每加入一个组件，都要回答**：  
   - 为什么需要它？  
   - 不加入会出现什么问题？  
   - 该怎么实现才能满足性能/可用要求？

## 第一步：理解需求与规模估算  

### 1. 功能需求要点  

| 编号 | 关键点 | 需要的后端能力 |
|------|--------|----------------|
| 1 | 实时协同编辑（≤200 ms） | 高吞吐编辑协议、低延迟推送、冲突解决 |
| 2 | 权限管理（四种角色） | 鉴权、ACL、共享链路 |
| 3 | 版本控制 | 持久化增量/快照、历史查询、回滚 |
| 4 | 离线编辑与同步 | 本地缓存、离线日志、冲突检测/合并 |
| 5 | 评论/@ 提及 | 实时通知、消息队列（可选） |
| 6 | 多格式导入/导出 | 文件存储、转码服务（外部） |

> **本题重点**：1‑4。后面章节会把 5、6 作为补充说明。

### 2. 非功能需求量化  

| 指标 | 目标 | 计算方式 | 备注 |
|------|------|----------|------|
| DAU | 2 亿 | 假设 5% 同时在线 → 1,000 万在线用户 | 需要弹性伸缩 |
| 同时编辑文档数 | 150 万 | 估算每文档平均 4 人在线 | 峰值压力点 |
| QPS（编辑操作） | 200 k ops/s | 150 万 文档 × 1.3 ops/s（平均） | 需要分布式消息/协同引擎 |
| 端到端延迟 | ≤200 ms | 客户端 → 边缘 → 协同服务 → 推送 → 客户端 | 关键 SLA |
| 可用性 | 99.9% | 关键服务（协同、权限）必须多活 | 单点故障不可接受 |
| 存储规模 | 500 PB | 文档原始 + 版本历史（增量） | 采用对象存储 + 分层冷热 |
| 数据一致性 | 弱一致 + 冲突解决 | 实时编辑不要求强一致，离线合并要可回滚 | 选 CRDT/OT 方案 |

### 3. 粗略容量估算  

| 项目 | 计算公式 | 结果 |
|------|----------|------|
| 编辑操作大小 | 平均 50 B/操作 | 200k ops/s × 50 B ≈ 10 MB/s ≈ 864 GB/天 |
| 每日新增编辑日志 | 864 GB | 需要压缩/归档 |
| 文档存储 | 假设平均 200 KB/文档 × 2 亿文档 ≈ 40 TB | 只算最新快照，历史增量另算 |
| 版本历史 | 预计 5‑10 倍快照大小 → 200‑400 TB | 采用增量压缩后放对象存储 |
| 业务元数据（权限、ACL） | 10 KB/文档 × 2 亿 ≈ 2 TB | 放关系型/文档型 DB |

> **小结**：系统需要支撑 **10 MB/s** 的实时编辑流、**PB 级** 的持久化存储，并且要在 **200 ms** 内完成编辑同步。

## 第二步：高层架构设计  

### 1. MVP（最小可用系统）结构  

```
Client (Web / Mobile)
   |
   | 1. WebSocket / HTTP
   v
Gateway (API Gateway + L4 LB)
   |
   | 2. 只路由到「协同服务」 + 「存储服务」
   v
Collaborative Service (单实例)
   |
   | 3. 持久化到对象存储 (S3/OSS)
   v
Object Storage
```

**实现要点**  

- **WebSocket** 维持长连接，实时推送编辑操作。  
- **协同服务** 使用 **OT**（后文会对比 OT/CRDT），在单机上完成变换、广播。  
- **对象存储** 直接写入快照文件（JSON/Protobuf），无需 DB。  

**不足**：单实例会成为瓶颈，故障会导致全部编辑不可用。  

---

### 2. 逐步演进到分布式高可用架构  

```
+---------------------------------------------------------------+
|                         CDN / Edge Proxy                    |
|  (把用户请求路由到最近的 Region，做流量切分、TLS 终止)      |
+---------------------------+-----------------------------------+
                            |
                            v
+-------------------+   +-------------------+   +-------------------+
|  Region A LB      |   |  Region B LB      |   |  Region C LB      |
| (L4+L7)           |   | (L4+L7)           |   | (L4+L7)           |
+--------+----------+   +--------+----------+   +--------+----------+
         |                       |                       |
         v                       v                       v
+-------------------+   +-------------------+   +-------------------+
|  Gateway/API GW   |   |  Gateway/API GW   |   |  Gateway/API GW   |
+--------+----------+   +--------+----------+   +--------+----------+
         |                       |                       |
         v                       v                       v
+-------------------+   +-------------------+   +-------------------+
|  Collab Service   |   |  Collab Service   |   |  Collab Service   |
|  (多实例, Sharding) |   |  (多实例, Sharding) |   |  (多实例, Sharding) |
+--------+----------+   +--------+----------+   +--------+----------+
         |                       |                       |
         |  +-------------------+-------------------+   |
         |  |   Message Queue (Kafka/RocketMQ)    |   |
         |  +-------------------+-------------------+   |
         |                       |                       |
         v                       v                       v
+-------------------+   +-------------------+   +-------------------+
|  Permission Svc   |   |  Version Svc      |   |  Offline Sync Svc |
+-------------------+   +-------------------+   +-------------------+
         |                       |                       |
         v                       v                       v
+-------------------+   +-------------------+   +-------------------+
|  Meta DB (MySQL/   |   |  Version Store    |   |  Offline Log Store |
|   PostgreSQL)      |   |  (Object Store)  |   |  (Kafka+Redis)    |
+-------------------+   +-------------------+   +-------------------+
         |                       |                       |
         v                       v                       v
+---------------------------------------------------------------+
|                     Object Storage (OSS)                     |
+---------------------------------------------------------------+
```

#### 关键组件说明  

| 组件 | 作用 | 为什么要单独拆出来 |
|------|------|-------------------|
| **CDN / Edge Proxy** | 最近访问路由、TLS 终止、DDoS 防护 | 把流量压在边缘，降低核心网络压力 |
| **Load Balancer** (L4+L7) | 按 Region 分流、故障转移 | 实现跨地域的高可用 |
| **API Gateway** | 鉴权、流量控制、限流、灰度发布 | 防止恶意请求直接冲击内部服务 |
| **Collaborative Service** | 实时编辑协议处理（OT/CRDT）、广播 | 需要水平扩展，按文档 ID 分片 |
| **Message Queue** | 跨实例广播编辑操作、异步落库 | 解耦、削峰，保证顺序（分区） |
| **Permission Service** | 角色校验、共享链路、Token 生成 | 权限是业务热点，单独缓存提升性能 |
| **Version Service** | 生成增量快照、归档、回滚 | 需要可靠持久化，支持离线恢复 |
| **Offline Sync Service** | 合并离线日志、冲突检测 | 离线场景独立处理，避免阻塞实时流 |
| **Meta DB** | 文档元信息、ACL、用户信息 | 关系型 DB 支持事务、二级索引 |
| **Object Storage** | 文档最新快照、历史版本、二进制文件 | 大容量、冷热分层、低成本 |

**不这样做的后果**  

- 把所有功能塞进一个服务，单点故障会导致编辑、权限、历史全部不可用，扩容困难。  
- 不使用消息队列，跨实例同步难以保证顺序，冲突概率激增。  
- 没有分区的 DB 在高并发读写时会出现锁竞争，导致延迟飙升。

## 第三步：数据库设计  

### 1. 元数据（Meta DB）  

使用 **MySQL（或 Aurora）** 采用 **垂直分表 + 水平分片**（DocId 范围分片）  

| 表名 | 主键 | 关键字段 | 索引 | 备注 |
|------|------|----------|------|------|
| `documents` | `doc_id` (BIGINT) | `owner_id`, `title`, `type`, `created_at`, `updated_at` | `owner_id`、`type` | 文档基本信息 |
| `doc_permissions` | `(doc_id, user_id)` | `role` (enum: VIEW, COMMENT, EDIT, ADMIN), `expire_at` | `user_id`、`role` | 细粒度 ACL |
| `doc_sharding` | `doc_id` | `shard_id` | PK | 记录文档分配到哪个协同实例（缓存） |
| `users` | `user_id` | `username`, `email`, `mobile`, `status` | `email` | 用户中心（可外部化） |

**分片策略**：  
- 采用 **consistent hashing**（如 Jump Consistent Hash）把 `doc_id` 映射到 **协同服务实例** 与 **元数据分库**。  
- 每个分库容量约 10 TB，支持 **水平扩容**。  

### 2. 实时编辑日志（Kafka + Redis）  

| 目的 | 选型 | 关键字段 |
|------|------|----------|
| **实时广播** | Kafka Topic `doc-edit-{shard}`（分区 = shard） | `doc_id`, `op_seq`, `user_id`, `op_type`, `payload`, `timestamp` |
| **离线缓存** | Redis Sorted Set `offline:{user_id}:{doc_id}`（score = op_seq） | 同上，便于快速回放 |

- **Kafka** 保证 **分区内顺序**，每个文档只落在同一分区，避免乱序冲突。  
- **Redis** 用作 **短期离线日志缓存**，断线后可快速拉取合并。

### 3. 版本存储（Object Storage）  

- **快照文件**：`doc/{doc_id}/snapshot/{timestamp}.json.gz`（全量）  
- **增量文件**：`doc/{doc_id}/delta/{from_ts}_{to_ts}.json.gz`（压缩）  
- **归档策略**：最近 30 天保留增量 + 最新快照，超过 30 天压缩为 **归档包**（冷存储）  

**元数据**（`versions` 表）记录快照与增量的映射：  

| 字段 | 说明 |
|------|------|
| `doc_id` | 文档 ID |
| `version_id` | 自增 |
| `base_ts` | 基准快照时间 |
| `delta_range` | `from_ts-to_ts` |
| `size` | 文件大小 |
| `created_at` | 生成时间 |

### 4. 权限缓存  

- **Redis**（Hash `perm:{doc_id}`）存放最新的 ACL，TTL 5 min，后台异步刷新。  
- **写穿透**：权限变更先写 DB，再更新缓存，确保强一致性。  

**小结**：  
- **强一致** 需求（权限、文档分片映射）放在关系型 DB 并配合缓存。  
- **弱一致**（编辑日志、版本）使用 Kafka + 对象存储，最终一致即可。

## 第四步：核心 API 设计  

> **原则**：REST 用于元数据（文档创建/权限），WebSocket/ProtoBuf 用于实时编辑。  

### 1. 文档生命周期 API（REST）  

| 方法 | 路径 | 描述 | 请求体 | 响应 |
|------|------|------|--------|------|
| `POST` | `/api/v1/docs` | 创建文档 | `{title, type, owner_id}` | `{doc_id, created_at}` |
| `GET` | `/api/v1/docs/{doc_id}` | 查询文档元信息 | — | `{doc_id, title, type, owner, updated_at}` |
| `DELETE` | `/api/v1/docs/{doc_id}` | 删除文档（软删） | — | `204` |
| `POST` | `/api/v1/docs/{doc_id}/share` | 添加/修改权限 | `{user_id, role, expire_at}` | `200` |
| `GET` | `/api/v1/docs/{doc_id}/versions` | 列出历史版本 | `?page=&size=` | `{versions: [...]}` |
| `POST` | `/api/v1/docs/{doc_id}/restore` | 回滚到某版本 | `{version_id}` | `200` |

### 2. 实时编辑协议（WebSocket + Protobuf）  

#### 2.1 连接握手  

```json
// client → server
{
  "type": "handshake",
  "doc_id": "123456789",
  "token": "JWT..."
}
```

- **服务端**校验 **JWT**、**权限**（READ/EDIT），返回 **session_id** 与 **last_seq**（上一次已确认的操作序号）。

#### 2.2 编辑操作（OT/CRDT）  

```proto
message Op {
  int64 doc_id = 1;
  int64 user_id = 2;
  int64 seq = 3;          // 单调递增，客户端自增
  enum OpType { INSERT = 0; DELETE = 1; FORMAT = 2; }
  OpType type = 4;
  bytes payload = 5;      // 具体插入字符或删除范围等，使用 protobuf 定义
  int64 timestamp = 6;
}
```

- **客户端**把本地编辑转成 `Op`，发送至 **WebSocket**。  
- **服务端**收到后：  
  1. **序列化校验**（seq 必须递增），若缺失则请求重传。  
  2. **冲突转换**（OT 转换函数 `transform(op, concurrentOps)`），得到 **已序列化的操作**。  
  3. **写入 Kafka**（确保持久化）并 **广播** 给同一文档的其他在线用户。  
  4 **返回 ack**（`{type:"ack", seq:xxx}`）给发送方，便于客户端本地确认。

#### 2.3 心跳 & 重连  

- 每 30 s 发送 `ping`，服务端返回 `pong`。  
- 断线后，客户端使用 `last_seq` 拉取 **缺失的操作**（从 Kafka/Redis），并继续编辑。

### 3. 离线同步 API（HTTP）  

| 方法 | 路径 | 描述 | 请求体 |
|------|------|------|--------|
| `POST` | `/api/v1/docs/{doc_id}/offline-sync` | 上传离线日志（批量） | `{base_seq, ops: [Op...]}` |

- **服务端**校验 `base_seq` 是否为当前最新 `seq`。  
  - 若相等，直接 **合并**（同 OT 转换）。  
  - 若不等，先 **拉取缺失的中间 ops**，再进行 **两段合并**。  

**返回**：合并后的最新 `seq` 与可能的冲突列表（需要 UI 提示用户）。  

### 4. 推送通知（WebSocket）  

```json
{
  "type": "comment",
  "doc_id": "123",
  "comment_id": "c456",
  "from_user": "alice",
  "content": "请检查第 3 行",
  "at_user": "bob",
  "timestamp": 1685001234567
}
```

- **Comment Service**写入 DB 后通过 **Kafka** 推送到对应 **WebSocket** 连接。  

**小结**：REST 负责 **幂等、事务**，WebSocket 负责 **低延迟、实时**；两者相辅相成。

## 第五步：详细组件设计  

下面把关键组件的内部实现、数据流和容错机制展开。

### 1. 协同服务（Collaborative Service）  

#### 1.1 选型：OT vs CRDT  

| 维度 | OT（Operational Transformation） | CRDT（Conflict‑free Replicated Data Type） |
|------|-----------------------------------|--------------------------------------------|
| **实现复杂度** | 中等，需要服务器维护转换函数 | 高，需要为每种数据结构设计 CRDT |
| **带宽** | 只传操作（小） | 同样只传操作，状态可压缩 |
| **冲突解决** | 服务器集中处理，易控制 | 各节点自行合并，天然最终一致 |
| **状态压缩** | 需要定期生成快照 | CRDT 状态可直接 GC（如 LWW‑Element） |
| **适合场景** | 文本、富文本编辑，已广泛实践（Google Docs） | 多端离线、强去中心化需求（如协同绘图） |

**本系统**：  
- 主要是 **文本/表格**，编辑冲突相对简单，且已有成熟的 **OT** 实现（如 ShareDB、Google Docs）。  
- 为了 **低延迟、易调试**，这里采用 **OT**。  
- 在离线同步阶段，可把 **CRDT** 思想（如 **LWW‑Map**）用于冲突检测的 **辅助**，但核心仍是 OT。  

#### 1.2 工作流程（单文档实例）  

1. **客户端**发送 `Op(seq, payload)`。  
2. **接收层**（WebSocket handler）  
   - 验证 `token`、`doc_id`、`role`。  
   - 把 `Op` 放入 **本地队列**（FIFO）。  
3. **处理层**（OT Engine）  
   - 读取当前 **已确认的操作列表** `C = [op1…opN]`（保存在内存）。  
   - 对新 `op` 与 `C` 中 **并发操作** 进行 **transform**：`op' = T(op, C_concurrent)`。  
   - 将 `op'` 写入 **Kafka**（持久化）并 **广播**（publish）给同文档的其他实例。  
   - 更新本地 **状态向量**（如 `doc_version = N+1`），返回 `ack(seq)`.  
4. **广播层**（Kafka Consumer）  
   - 每个实例订阅同一 **分区**，收到 `op'` 后：  
     - 再次 **transform**（如果本实例已经本地执行了同一操作），确保 **全局序列** 一致。  
     - 将 `op'` 应用到 **内存模型**（如 `Rope` 数据结构）并 **推送** 给对应的 WebSocket 客户端。  
5. **落库层**（异步）  
   - 定期（如每 5 s）把 **完整文档状态**（快照）写入 **Object Storage**。  
   - 同步 **操作日志**（Kafka）到 **Version Service** 生成增量文件。  

#### 1.3 状态数据结构  

- **Rope**（平衡二叉树）或 **Piece Table**：适合高频插入/删除，且支持 **增量快照**。  
- **内存模型**：每个实例维护 **文档树** + **已确认的 seq**。  
- **持久化**：只在 **快照** 时写入对象存储，操作日志保存在 **Kafka**（持久化 + 可回放）。  

#### 1.4 容错  

| 故障类型 | 处理方式 |
|----------|----------|
| **实例宕机** | 其它实例仍持有完整 `C`（Kafka）和 **状态向量**，新实例启动后从 **Kafka** 回放到最新 `seq`。 |
| **网络分区** | 客户端短暂失联后会 **重连**，使用 `last_seq` 拉取缺失 ops；若分区时间过长，系统会 **强制回滚**（用户提示冲突）。 |
| **消息丢失** | Kafka **副本** ≥ 3，保证不丢失；客户端若未收到 ack，超时后自动 **重发**。 |
| **热点文档** | 对热点文档使用 **单独的分片/专属实例**，并且 **限流**（如 1000 ops/s/用户）防止刷流。 |

### 2. 权限服务（Permission Service）  

- **API**：`GET /perm/{doc_id}/{user_id}` → 返回角色。  
- **实现**：  
  1. **查询 Redis 缓存**，若命中直接返回。  
  2. **缓存未命中** → 访问 **MySQL**，返回后写入 Redis。  
  3. **写操作**（共享、撤销） → 事务写 MySQL → **publish** 到 **Kafka** → **消费者** 更新对应 Redis。  
- **安全**：所有编辑请求必须先通过 **Gateway** 调用该服务进行鉴权。  

### 3. 版本服务（Version Service）  

- **增量生成**：每收到 **N**（如 500）条操作或时间阈值（如 1 min），触发 **快照** & **增量** 生成。  
- **压缩**：使用 **Snappy/Zstd** 对 JSON/ProtoBuf 进行压缩，降低存储成本。  
- **归档**：超过 30 天的增量合并为 **大包**（Parquet）存入 **冷存储**（OSS Glacier）。  
- **恢复流程**：  
  1. 读取 **最新快照**。  
  2. 按 **delta_range** 依次回放增量，得到指定版本。  

### 4. 离线同步服务（Offline Sync Service）  

#### 4.1 客户端离线日志结构  

```json
{
  "doc_id": "123",
  "base_seq": 5678,   // 客户端上一次同步的全局 seq
  "ops": [
    {"seq":1,"type":"INSERT","payload":...},
    {"seq":2,"type":"DELETE","payload":...}
  ]
}
```

#### 4.2 合并算法（伪代码）  

```python
def offline_merge(base_seq, client_ops):
    # 1. 从 Kafka 拉取 server_missing = ops[base_seq+1 : current_seq]
    server_missing = fetch_ops(base_seq+1, CURRENT_SEQ)

    # 2. 先把 server_missing 应用到本地状态（OT transform）
    state = apply_ops(initial_state, server_missing)

    # 3. 再把 client_ops 按顺序 transform 对 server_missing
    for op in client_ops:
        op_prime = transform(op, server_missing)
        state = apply_op(state, op_prime)
        server_missing.append(op_prime)   # 让后续 client ops 看到最新并发集合

    # 4. 将合并后的 ops 写入 Kafka（保持全局 seq 连续）
    write_ops_to_kafka(server_missing[base_seq+1:])   # 包含原 server + client 新 ops

    return state, new_seq
```

- **关键点**：  
  - **transform** 必须是 **可交换的**（满足 OT 的 **TP1/TP2** 条件），否则会出现不一致。  
  - 合并后 **返回最新 seq** 给客户端，客户端把本地状态同步为 `state`。  

#### 4.3 分布式实现  

- **离线同步入口**是 **HTTP**，后端调用 **Offline Sync Service**。  
- Service **读取**：  
  - **Kafka**（最新 seq、缺失 ops）  
  - **Redis**（最近 5 min 的 ops 缓存）  
- **写入**：  
  - 把合并后 **完整的 ops 列表** 重新写入 **Kafka**（顺序保证），并 **同步** 到 **Version Service**。  

### 5. 消息推送（WebSocket Server）  

- **集群化**：使用 **NGINX + TCP Load Balancer** → 将同一用户的 **WebSocket** 连接粘性分配到同一 **Gateway** 实例。  
- **会话管理**：每条连接对应 `session_id`，保存在 **Redis**（hash `ws:{session_id}` → `user_id, doc_id, conn_info`）。  
- **广播**：协同服务发布到 **Kafka**，**WebSocket Consumer** 读取并 **push** 给对应 `session_id`。

### 1. 横向扩容  

| 组件 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **API Gateway** | 增加实例 + L4 LB | QPS 200k → 400k |

### 2. 热点文档与限流  

- **热点检测**：实时监控每个 `doc_id` 的 QPS，阈值如 5k ops/s。  
- **策略**：  
  1. **分片迁移**：把热点文档迁移到 **专用实例**（弹性伸缩组）。  
  2. **操作限流**：对单用户每秒操作数做 **令牌桶** 限制（如 100 ops/s），防止刷流。  
  3. **优先级队列**：编辑操作与系统内部操作（如自动保存）分不同队列，保证编辑优先。  

### 3. 容灾与故障恢复  

| 故障 | 备份/冗余 | 恢复步骤 |
|------|-----------|----------|
| **数据中心全失** | 多 Region 同步复制（OSS、MySQL 双活） | DNS 切换至备份 Region，流量自动落到新 Region |
| **单节点崩溃** | 同步副本（Kafka ISR ≥ 2，Redis 主从） | 副本接管请求，自动重建失效节点 |
| **网络分区** | 客户端重连、幂等请求 | 超时后重新握手，使用 `last_seq` 拉取缺失 ops |
| **磁盘损坏** (对象存储) | 多 AZ 冗余、版本化存储 | 自动读取其他副本，无需人工干预 |

### 4. 监控、告警、限流  

- **Metrics** (Prometheus):  
  - `doc_edit_ops_per_sec`、`collab_service_latency`、`websocket_connection_count`、`kafka_lag`、`redis_hit_rate`。  
- **Tracing** (OpenTelemetry): 跨服务链路追踪编辑请求的全路径，定位延迟热点。  
- **Alert**:  
  - 延迟 > 150 ms（95%） → 报警。  
  - Kafka lag > 5000 → 报警。  
  - Redis hit rate < 80% → 报警。  

### 5. 安全与合规  

- **TLS** 全链路加密。  
- **JWT** + **OAuth2** 鉴权（企业微信、QQ 登录）。  
- **审计日志**（写入独立审计库）用于合规审查。  

**小结**：通过 **分片、弹性伸缩、冗余复制**，系统能够在 **200k QPS**、**150 万并发文档** 的情况下保持 **≤200 ms** 延迟和 **99.9%** 可用。

## 第七步：常见面试追问与回答  

### Q1️⃣：如果我们选择 **CRDT** 而不是 **OT**，在实现上会带来哪些优势与挑战？如何在海量文档上进行状态压缩？

**回答要点**  

| 维度 | CRDT 的优势 | CRDT 的挑战 | 状态压缩方案 |
|------|--------------|--------------|--------------|
| **冲突处理** | 自动无冲突（所有副本天然收敛） | 需要为每种数据结构实现专属 CRDT（如 RGA、LSEQ） | 使用 **Delta‑CRDT**：只同步增量而非完整状态 |
| **离线支持** | 本地副本可随时操作，合并时不需要中心服务器参与 | 合并代价可能随操作数线性增长（状态膨胀） | 定期 **压缩快照**：把内部元数据（如标记 ID）重新编号，丢弃已被覆盖的元素 |
| **实现复杂度** | 对开发者透明，客户端实现难度高 | 需要 **全局唯一 ID**（如 Lamport Clock + NodeId）来保持顺序 | 采用 **Hybrid Logical Clock (HLC)** + **Garbage Collection** 机制定期清理 tombstone |
| **网络开销** | 每次同步发送完整对象或 delta，通常比 OT 多 | 对大文档的 delta 仍可能很大 | 使用 **分块 CRDT**（每块 4KB）并 **压缩**（Zstd） |

**总结**：CRDT 能简化冲突解决、天然支持离线，但在 **大文本** 场景下 **状态膨胀** 是主要痛点，需要 **定期快照+压缩**。因此在本系统（以文本为主）更倾向 OT，而 CRDT 可作为离线子系统的补充。

---

### Q2️⃣：当编辑 QPS 突然增长到 **500 k ops/s** 时，系统的水平扩容方案如何设计？哪些数据需要强一致的分片，哪些可以使用弱一致？

**回答要点**  

1. **扩容步骤**  
   - **扩容协同服务**：增加 **实例数**，同时在 **Kafka** 中为每个文档保持 **单独分区**（Partition = doc_id hash）。只要分区数 ≥ 实例数，即可实现 **1:1** 映射。  
   - **扩容 Kafka**：增加 **Broker**，重新平衡分区，确保每个 Broker 负载均衡。  
   - **扩容 Redis**：使用 **Cluster**，增加 **slot**，保证 **读写** 通过 **MGET/MSET** 批量完成。  
   - **扩容 Permission Service**：把 **MySQL** 拆分为 **读写分离**，使用 **ProxySQL** 自动路由，必要时 **水平分表**（按 `owner_id`）。  

2. **强一致 vs 弱一致**  
   - **强一致** 必须保证 **权限** 与 **文档分片映射** 正确：  
     - `doc_permissions`、`doc_sharding` → 必须 **事务** 写入 **MySQL**，并同步到 **Redis**（写后读一致）。  
   - **弱一致** 可接受延迟的对象：  
     - **编辑日志**（Kafka） → 只要最终落库即可。  
     - **版本快照**（对象存储） → 异步写入，短暂不一致不影响编辑。  
     - **离线日志**（Redis） → 可容忍短暂失效，后续补偿。  

3. **热点文档处理**  
   - **动态分片**：当某文档 QPS 超过阈值，自动 **拆分** 为 **子文档**（如章节级别）或 **迁移** 到 **专用实例**。  
   - **限流**：对单用户 QPS 进行 **Token Bucket**，防止恶意刷流导致全局瓶颈。  

**结论**：通过 **分区+实例一对一**、**异步弱一致** 与 **强一致关键元数据分离**，系统能够平滑支撑 **500 k ops/s** 甚至更高。

### Q3️⃣：用户离线期间产生大量编辑操作，恢复网络后如何保证最终一致且不丢失任何操作？请描述冲突检测的算法流程及其在分布式环境中的实现方式。

**回答要点**  

1. **离线日志收集**  
   - 客户端在本地 **SQLite/IndexedDB** 按顺序记录每个 `Op`，并维护 **本地 seq**（从 `last_synced_seq` 开始递增）。  
2. **恢复流程**  
   - **Step 1：握手** → 客户端发送 `base_seq`（最后一次成功同步的全局 seq）给服务器。  
   - **Step 2：服务器拉取缺失 ops** → 从 **Kafka**/Redis 中取 `base_seq+1 … current_seq`（称为 **ServerMissingOps**）。  
   - **Step 3：合并** → 使用 **OT Transform** 将 `ClientOps` 与 `ServerMissingOps` 两两转换。伪代码如下（已在第5步给出）。  
   - **Step 4：写入日志** → 合并后得到的完整操作流写回 **Kafka**（保持全局顺序），并 **更新** `doc_version`。  
   - **Step 5：返回 ack** → 服务器返回最新 `seq` 与可能的 **冲突列表**（如同一位置的不同文字），客户端展示 UI 让用户手动确认或自动选择。  

3. **冲突检测细节**  
   - **冲突** 在 OT 中表现为 **两个操作在同一位置** 且 **不可兼容**（如两人同时在同一字符前插入不同内容）。  
   - **Transform** 函数会 **调整位置**：  
     - `transform(insA, insB)` → 若 `insA.pos > insB.pos`，则 `insA.pos += len(insB.text)`；  
     - `transform(delA, insB)` → 若 `delA.range` 包含 `insB.pos`，则产生 **冲突**。  
   - **冲突标记**：在合并后产生 **ConflictRecord**（`op_id, user_id, type, position, reason`），写入 **Conflict DB**（MongoDB）供 UI 提示。  

4. **分布式实现**  
   - **Kafka** 负责 **全局顺序**：每个文档对应一个分区，保证 **单分区顺序**。  
   - **离线合并服务**是 **无状态**的：接收 `base_seq` 与 `ops`，从 **Kafka** 拉取缺失 ops，完成合并后 **写回** Kafka。因为 Kafka 本身是 **持久化** 的，合并过程不需要额外事务。  
   - **幂等性**：写入 Kafka 时使用 **唯一键**（`doc_id + op_seq`），如果重复写入会被 **幂等过滤**。  

**结果**：用户离线编辑后，所有操作都能 **完整**、**有序**、**冲突可感知** 地同步到云端，系统保持 **最终一致**。

## 心得与反思  

### 1. 本题最难的设计决策  

| 决策 | 思考过程 |
|------|----------|
| **选择 OT 还是 CRDT** | 初始直觉是 CRDT 天然无冲突、离线友好，但深入后发现文本编辑的 **状态膨胀** 与 **实现复杂度** 远超 OT。通过对比实现成本、成熟度（Google Docs、ShareDB）以及对 200 ms 延迟的严格要求，最终决定使用 **OT**，并在离线合并阶段辅以 CRDT 思想（如 LWW‑Map）来帮助冲突检测。 |
| **编辑日志的存储与顺序保证** | 实时编辑要求 **全局有序**，但又要高吞吐。最开始考虑直接写 MySQL，发现锁竞争和事务开销太大。最终选用 **Kafka 分区**（每文档 1 分区）实现 **顺序写入**，同时利用 **Kafka 的副本** 提供强持久化，配合 **Redis 缓存** 加速缺失日志拉取。这个决定平衡了 **一致性、性能、容灾**。 |

### 2. 新手最容易犯的错误  

1. **把实时编辑和持久化混在同一个服务**  
   - 结果：一次写入延迟直接影响编辑体验，导致 200 ms SLA 难以达成。正确做法是 **编辑服务专注低延迟**，持久化（快照、版本）采用 **异步批处理**。  

2. **忽视操作顺序导致冲突不可收敛**  
   - 如果不使用 **单分区有序日志**（如 Kafka）或不做 **OT Transform**，不同客户端的操作会出现“先后顺序错乱”，最终文档状态不一致。应保证 **全局序列号**（doc_version）以及 **顺序广播**。
