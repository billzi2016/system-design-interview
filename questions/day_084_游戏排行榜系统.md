# 游戏排行榜系统设计题

## 1. 题目背景
游戏排行榜系统用于实时统计并展示玩家在不同游戏模式或全局的分数、排名、成就等信息，帮助玩家了解自己的竞争状态并提升黏性。系统需要在高并发下提供低延迟的查询和及时的榜单更新。

## 2. 面试场景设定
> **面试官**：  
> “假设我们正在为一款每日活跃用户约 200 万、峰值 QPS 可达 10 万的多人在线竞技游戏设计一个全局排行榜系统。请你从零开始设计该系统，重点考虑高并发写入、实时排名以及查询的性能和可用性。你会怎样拆解需求、选型以及构建整体架构？”

## 3. 功能性需求
1. **提交分数**：玩家完成一局游戏后，将本局得分（或排名分值）上报至系统。  
2. **查询个人排名**：玩家可以随时查询自己在对应游戏模式/全局的当前排名和分数。  
3. **获取排行榜**：支持分页获取前 N 名（如前 100、前 1000）以及指定区间（如第 1001~2000 名）。  
4. **排行榜历史快照**：提供每日/每周排行榜的历史快照查询（用于榜单回顾或奖励结算）。  
5. **防作弊/分数校验**：在提交分数时进行基本的合法性校验（防止异常分数突增）。  
6. **多维度榜单**：支持按不同维度（如游戏模式、地区、赛季）分别维护独立榜单。

## 4. 非功能性需求
| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（每日活跃用户）** | 2,000,000 | 约 2 百万玩家每天会产生至少一次查询或提交 |
| **峰值 QPS（写入+读取）** | 100,000 | 峰值时写入约 30,000 QPS，读取约 70,000 QPS |
| **单次查询延迟** | ≤ 100 ms（99%） | 排名查询、榜单分页返回时间 |
| **写入可见延迟** | ≤ 5 秒 | 分数上报后在榜单中可见的最长延迟 |
| **可用性** | 99.9%（月度累计） | 包括读写路径的容错 |
| **存储规模** | ≈ 5 TB | 预计每月累计历史快照约 30 GB，全年约 360 GB；加上索引、冗余等约 5 TB |

## 5. 系统边界
**本题范围内**  
- 分数提交、实时排名计算、排行榜查询、历史快照存储与查询。  
- 多维度榜单的分区实现与负载均衡。  
- 基本的防作弊校验（如分数阈值、频率限制）。  

**本题不考虑**  
- 用户登录、鉴权、账号体系。  
- 客户端 UI、展示层实现。  
- 完整的作弊检测系统（如机器学习模型）。  
- 第三方支付、道具商城等业务。  
- 跨服/跨地区的全局一致性（只需在单个大区内部实现高可用）。  

## 6. 提示与追问
1. **数据结构选型**：如果使用 Redis 的有序集合（ZSET）来维护实时榜单，你会如何解决 ZSET 在单节点容量限制以及持久化的问题？  
2. **写入热点**：在热门赛季或大型赛事期间，某些玩家的分数会集中在极短时间内大量上报，你会怎样设计写入削峰、批量写入或异步落库的方案？  
3. **历史快照存储**：如何在保证查询效率的前提下，设计每日/每周快照的生成、存储与压缩策略？是否会使用归档冷存储或增量备份？

# 题解

# 游戏排行榜系统设计完整解答

> **写给**：刚入行的后端小伙伴  
> **目标**：从最小可用系统一步步搭建到高可用、可扩展的生产级排行榜服务。每一步都会说明 **为什么** 要这么做，**不这么做** 会有什么痛点。  

---

## 解题思路总览
1. **先把需求拆成“业务”和“技术”两块**，明确哪些是**必须实现的功能**，哪些是**非功能约束**（QPS、延迟、可用性、容量）。  
2. **做粗略的流量与存储估算**，把数字写在纸上，帮助后面选技术栈、容量规划。  
3. **从最小可用（MVP）出发**：只用几台机器、单点 Redis+MySQL，快速验证业务流程。  
4. **逐层补齐**：  
   - **横向扩展**（分片、缓存）  
   - **高可用**（主备、容错、降级）  
   - **削峰/异步**（写入削峰、批处理）  
   - **持久化 & 快照**（历史榜单、归档）  
5. **每个组件都写清楚**：职责、输入/输出、关键数据结构、容错点。  
6. **面试官常追问**：准备好数据结构选型、削峰、快照等细节的答案。  

下面按照**需求 → 高层架构 → 数据库 → API → 细节实现 → 高可用 → 追问**的顺序展开。

## 第一步：理解需求与规模估算

| 需求 | 解释 |
|------|------|
| **提交分数** | 玩家一局结束后上报分数，需要在榜单中尽快生效。 |
| **查询个人排名** | 玩家随时查询自己的当前排名、分数。 |
| **获取排行榜** | 分页查询前 N 名或任意区间。 |
| **历史快照** | 每日/每周生成一次快照，供回顾、结算奖励使用。 |
| **防作弊** | 简单阈值、频率校验，防止一次性刷分。 |
| **多维度榜单** | 按游戏模式、地区、赛季等维度分别维护独立榜单。 |

### 1.1 流量估算

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 2,000,000 | 每天活跃用户 |
| **每日写入次数** | 假设每位玩家平均 1 次提交 | 2,000,000 次 |
| **每日读取次数** | 每位玩家平均 3 次查询（个人+榜单） | 6,000,000 次 |
| **峰值 QPS** | 24h → 86400s，峰值占 10% 时间 | 100,000 QPS（写 30k，读 70k） |
| **单次写入可见延迟** | ≤5s | 需要 5 s 内能查询到最新分数 |
| **单次查询延迟** | ≤100 ms (99%) | 排名、榜单分页必须快 |
| **存储** | 预计每条分数记录 32 B（uid、score、timestamp） | 2 M * 30 days ≈ 2 GB（原始）<br>快照 + 索引 ≈ 5 TB（预留） |

> **小结**：系统需要 **高并发写入**、**低延迟读取**，且 **写后可见** 需要在 **秒级**。因此实时榜单必须在 **内存** 中维护，持久化交给 **后台批处理**。

### 1.2 功能优先级（MVP 版本）

| 优先级 | 功能 | 说明 |
|--------|------|------|
| 1 | 提交分数 + 实时个人排名查询 | 核心业务，用户体验关键 |
| 2 | 分页获取排行榜 | 常用功能 |
| 3 | 多维度榜单（模式、地区） | 通过分区实现 |
| 4 | 防作弊阈值校验 | 简单阈值即可 |
| 5 | 历史快照 | 可后期离线批处理实现 |

先实现 1~2，后面逐步扩展到 3~5。

## 第二步：高层架构设计

### 2.1 最小可用系统（MVP）

```
+-----------+          +----------------+          +-----------+
|   客户端   | <------> |  API Gateway   | <------> |   Redis   |
+-----------+          +----------------+          +-----------+
                           |   ^   |
                           |   |   | (异步写入)
                           v   |   v
                       +----------+----------+
                       |  MySQL (持久化)    |
                       +-------------------+
```

- **API Gateway**：统一入口，做鉴权、限流、日志。  
- **Redis (单机 ZSET)**：维护实时排行榜（内存快），读写均在此完成。  
- **MySQL**：持久化用户分数历史、快照等。写入采用 **异步队列**（Kafka/Rabbit）→消费者落库。  

**优点**：结构清晰、开发成本低、可以快速验证业务。  
**缺点**：单点 Redis 容量/性能受限、故障时全局不可用。

### 2.2 逐步演进到生产级分布式架构

```
                         +-------------------+
                         |  CDN / Edge Cache |
                         +-------------------+
                                 |
+-----------+   HTTPS/REST   +-------------------+   HTTPS/REST   +-----------+
|   客户端   | <------------> |   API Gateway     | <------------> |   客户端  |
+-----------+                +-------------------+                +-----------+
                                 |  ▲  |  ▲
                                 |  |  |  |
               +-----------------+  |  |  +-----------------+
               |                    |                     |
        +------v------+       +-----v------+        +-----v------+
        |   Redis     |       |  Redis      |        |  Redis      |
        |   Cluster   |       |  Shard #1   | ...    |  Shard #N   |
        +-------------+       +------------+        +------------+
               |  ^                |  ^                |  ^
               |  |                |  |                |  |
   +-----------v--+-------+ +------v--+-------+ +------v--+-------+
   |  Kafka / Pulsar (写入队列) | |  Kafka / Pulsar (写入队列) |
   +---------------------------+ +---------------------------+
               |  ^                |  ^
               v  |                v  |
         +------------+       +------------+
         |   MySQL    |       |   ClickHouse  |
         |  主备集群  |       |  冷/热存储   |
         +------------+       +------------+
               |                     |
               v                     v
          +-----------+        +-----------+
          |  Spark /  |        |  Offline  |
          |  Flink    |        |  Job      |
          +-----------+        +-----------+
               |                     |
               v                     v
          +-----------+        +-----------+
          |   OSS /   |        |   Glacier |
          |   HDFS    |        |   (归档)  |
          +-----------+        +-----------+
```

#### 关键组件解释

| 组件 | 作用 | 关键技术选型 |
|------|------|--------------|
| **API Gateway** | 统一入口、限流、鉴权、灰度发布 | Kong / Nginx + Lua |
| **Redis Cluster** | 真实的 **ZSET** 分片实现，水平扩容 | Redis 6.x+ 集群模式 |
| **Kafka / Pulsar** | 写入削峰、持久化日志、保证至少一次投递 | Kafka 3.x |
| **MySQL 主备** | 业务关键的用户分数、配置、元数据的强一致存储 | Percona XtraDB Cluster |
| **ClickHouse** | 大量历史快照、统计分析的列式存储，查询速度快 | ClickHouse 22+ |
| **Spark/Flink** | 离线/近实时生成每日/每周快照、聚合榜单 | Flink Streaming |
| **OSS / HDFS / Glacier** | 冷归档、成本控制 | 阿里云 OSS + Glacier（或 S3 Glacier） |

**为什么要这么拆？**  
- **Redis Cluster** 解决单节点 ZSET 大小（几 GB）和并发瓶颈。  
- **Kafka** 把写入请求平滑到 5 s 以内的持久化层，削峰。  
- **ClickHouse** 适合 **历史快照** 的大规模查询，远比 MySQL 更省存储。  
- **离线计算**（Flink）负责 **增量快照**、**排行榜合并**，避免实时计算压力。

## 第三步：数据库设计

### 3.1 实时排行榜（Redis ZSET）数据模型

| ZSET key  | 成员 (member) | 分数 (score) |
|-----------|---------------|--------------|
| `leaderboard:mode:{mode_id}:region:{region_id}` | `uid:{user_id}` | 玩家当前累计分数（或最高分） |

- **Key 设计**：`leaderboard:mode:{mode_id}:region:{region_id}` 实现多维度分片。  
- **成员**：`uid:{user_id}` 统一前缀，方便后期做批量操作。  
- **score**：使用 **double**，支持负分或小数。  

> **注意**：ZSET 按 **score** 自动排序，`ZRANGE`、`ZREVRANK` 都是 O(log(N)+M)。  

### 3.2 持久化表（MySQL）——**user_score**（写入日志）

```sql
CREATE TABLE user_score (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    uid BIGINT UNSIGNED NOT NULL,
    mode_id INT NOT NULL,
    region_id INT NOT NULL,
    score BIGINT NOT NULL,
    submit_ts TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uq_uid_mode_region (uid, mode_id, region_id)  -- 防止重复写入
) ENGINE=InnoDB PARTITION BY HASH(mode_id) PARTITIONS 16;
```

- **目的**：提供 **强一致** 的分数历史，支持 **回滚**、**对账**。  
- **分区**：按 `mode_id` 哈希，提升写入并行度。  

### 3.3 历史快照表（ClickHouse）——**snapshot_daily**

```sql
CREATE TABLE snapshot_daily (
    snap_date Date,
    mode_id UInt32,
    region_id UInt32,
    uid UInt64,
    rank UInt32,
    score Int64,
    PRIMARY KEY (snap_date, mode_id, region_id, rank)  -- 按日期、模式、地区、排名聚簇
) ENGINE = MergeTree()
ORDER BY (snap_date, mode_id, region_id, rank);
```

- **写入方式**：每日/每周 **Flink Job** 从 Redis 拉取前 N（比如 10k）写入 ClickHouse。  
- **查询**：`SELECT * FROM snapshot_daily WHERE snap_date='2026-05-25' AND mode_id=1 ORDER BY rank LIMIT 100;`  

### 3.4 防作弊阈值表（MySQL）

```sql
CREATE TABLE score_rule (
    mode_id INT PRIMARY KEY,
    max_score_per_game BIGINT NOT NULL,   -- 单局最高分
    min_interval_seconds INT NOT NULL    -- 同一玩家提交间隔
);
```

- 在 **写入 API** 前做校验。

### 4.1 API 细节实现要点

#### 4.1.1 提交分数（/score/submit）

```go
func SubmitScore(c *gin.Context) {
    var req struct {
        UID      uint64 `json:"uid" binding:"required"`
        ModeID   int    `json:"mode_id" binding:"required"`
        RegionID int    `json:"region_id" binding:"required"`
        Score    int64  `json:"score" binding:"required"`
    }
    if err := c.ShouldBindJSON(&req); err != nil { ... }

    // 1. 防作弊阈值检查（从缓存读取 rule 表）
    rule := cache.GetScoreRule(req.ModeID)
    if req.Score > rule.MaxScorePerGame { ... }
    // 2. 限流：同一玩家 5 秒内只能提交一次
    if limiter.Allow(req.UID) == false { ... }

    // 3. 异步写入 Kafka
    evt := ScoreEvent{UID: req.UID, ModeID: req.ModeID, RegionID: req.RegionID, Score: req.Score, Ts: time.Now()}
    kafkaProducer.Produce("score_submit", evt)

    // 4. 直接更新 Redis ZSET（实时可见）
    key := fmt.Sprintf("leaderboard:mode:%d:region:%d", req.ModeID, req.RegionID)
    redisClient.ZIncrBy(ctx, key, float64(req.Score), fmt.Sprintf("uid:%d", req.UID))

    c.JSON(200, gin.H{"code":0, "msg":"ok"})
}
```

- **步骤 3+4**：先把事件投到 Kafka，确保 **持久化**；随后 **同步** 更新 Redis，让用户几乎立刻看到结果（5 s 内写入 DB 完成）。  
- **防作弊**：在业务层先过滤，后端再做一次二次校验（消费者端）。

#### 4.1.2 查询个人排名（/rank/{uid}）

```go
func GetUserRank(c *gin.Context) {
    uid := c.Param("uid")
    mode := c.Query("mode_id")
    region := c.Query("region_id")
    key := fmt.Sprintf("leaderboard:mode:%s:region:%s", mode, region)

    // ZREVRANK 返回 0‑based 排名，若不存在返回 nil
    rank, err := redisClient.ZRevRank(ctx, key, fmt.Sprintf("uid:%s", uid)).Result()
    if err == redis.Nil {
        c.JSON(404, gin.H{"code":1, "msg":"user not in leaderboard"})
        return
    }
    score, _ := redisClient.ZScore(ctx, key, fmt.Sprintf("uid:%s", uid)).Result()
    c.JSON(200, gin.H{"code":0, "data":gin.H{"rank": rank+1, "score": score}})
}
```

- **时间复杂度**：O(log(N))，即使 N=10M 仍在毫秒级。  

#### 4.1.3 分页获取排行榜（/leaderboard）

```go
func GetLeaderboard(c *gin.Context) {
    mode := c.Query("mode_id")
    region := c.Query("region_id")
    start, _ := strconv.Atoi(c.Query("start"))   // 0‑based
    limit, _ := strconv.Atoi(c.Query("limit"))

    key := fmt.Sprintf("leaderboard:mode:%s:region:%s", mode, region)
    // ZREVRANGE 返回按分数从高到低的成员
    vals, _ := redisClient.ZRevRangeWithScores(ctx, key, int64(start), int64(start+limit-1)).Result()
    // 构造返回结构
    list := make([]gin.H, 0, len(vals))
    for i, v := range vals {
        uidStr := strings.TrimPrefix(v.Member.(string), "uid:")
        uid, _ := strconv.ParseUint(uidStr, 10, 64)
        list = append(list, gin.H{
            "uid":   uid,
            "rank":  start + i + 1,
            "score": v.Score,
        })
    }
    c.JSON(200, gin.H{"code":0, "data":gin.H{"list":list, "total":-1}})
}
```

- **total** 可通过 `ZCard` 获取（O(1)），但对大榜单不一定需要返回。

#### 4.1.4 历史快照查询（/leaderboard/snapshot）

- 直接查询 **ClickHouse**，使用 **HTTP 接口**（ClickHouse 自带）或 **JDBC**。  
- 只返回 **已生成的快照**，查询延迟 < 50 ms（列式压缩）。

## 第五步：详细组件设计

### 5.1 写入路径（Score Submit）

1. **API 层** → **防作弊校验** → **限流** → **写入 Kafka**（异步）  
2. **同步更新 Redis ZSET**（实时可见）  
3. **Kafka Consumer**（分区消费） → **批量落库**（MySQL）  
4. **批量落库成功后** → **发送 ACK**（可选）给业务层，用于审计  

#### 5.1.1 削峰 & 批量落库策略

| 参数 | 取值建议 | 解释 |
|------|----------|------|
| **Kafka 分区数** | `#mode × #region`（或 30~50） | 保证同一榜单的写入顺序，避免并发冲突 |
| **消费者批次大小** | 500~1000 条 | 兼顾吞吐与延迟 |
| **写库频率** | 每 1~2 秒一次或累计 1k 条 | 达到 **5 s** 可见延迟目标 |
| **事务** | MySQL 批量 `INSERT … ON DUPLICATE KEY UPDATE` | 防止重复写入，保持幂等 |

> **不这么做的后果**：直接同步写 MySQL，写入延迟会升到 50‑200 ms，且高峰期 DB 可能出现 **写锁争用**，导致 QPS 爆炸。

### 5.2 读取路径（查询)

1. **API → Redis**（ZSET）  
   - **个人排名**：`ZREVRANK` + `ZSCORE`  
   - **榜单分页**：`ZREVRANGE`  
2. **历史快照** → **ClickHouse**（列式查询）  

#### 5.2.1 缓存层细节

- **热点玩家**（常查询）可以在 **本地缓存（Guava/ Caffeine）** 再读 Redis，降低网络 RTT。  
- **榜单分页** 常被同一时间段的玩家请求，使用 **Redis 缓存分页结果**（如 `leaderboard:mode:1:region:2:page:0:100`）+ **TTL 10 s**，防止频繁遍历 ZSET。

### 5.3 多维度分片实现

- **Key 前缀**：`leaderboard:mode:{mode_id}:region:{region_id}`  
- **Shard 规则**：在 Redis Cluster 中，每个 **hash slot** 对应一个 **实际节点**，Redis 会根据 key 的 **CRC16** 取模决定落在哪个节点。  
- **扩容**：使用 **Redis Cluster Rebalance**，迁移部分 hash slot，几乎无感知。  

### 5.4 防作弊细化

| 场景 | 检查点 | 实现方式 |
|------|--------|----------|
| 单局异常高分 | `score <= max_score_per_game` | API 直接返回错误 |
| 频繁提交 | `last_submit_ts(uid, mode, region)` 距离当前 < `min_interval_seconds` | Redis `SETNX` + `EXPIRE` 实现限速 |
| 分数突增 | 与历史平均分比较，超过阈值时写入 **审计日志** | Kafka `audit_topic` → 后台人工/机器学习审计 |

### 5.5 历史快照生成

1. **定时任务**（Cron）在每日/每周零点触发 Flink Job。  
2. Flink **读取 Redis ZSET**（使用 `SCAN` + `ZRANGE`）或 **消费 Kafka** 中的 **最终分数**（每个玩家的最新分数）。  
3. 对每个维度生成 **前 N（10k）** 排名，写入 **ClickHouse** 表 `snapshot_daily`。  
4. **压缩**：ClickHouse 默认使用 **LZ4**，查询时只解压需要的列。  
5. **归档**：超过 90 天的快照转存至 **OSS**（对象存储）+ **Glacier**（冷备），保留 2 年。

> **为什么不直接把快照写到 MySQL？**  
MySQL 在写入大量历史数据时会产生 **行锁**，查询慢，且磁盘占用大。列式存储的 ClickHouse 对于 **只读、聚合** 场景更省资源、查询更快。

### 5.6 容错 & 降级

| 故障点 | 备选方案 |
|--------|----------|
| Redis 单点故障 | **Redis Cluster** + **Replica**（主从） |
| Kafka 消费积压 | **Consumer 分组** + **背压**，超时告警，自动扩容 |
| MySQL 主库不可写 | **自动切换到只读备库**，写入只能走 Kafka，后续补偿 |
| ClickHouse 查询慢 | **查询缓存**（Cache Table）+ **分区裁剪** |
| API 网关超负载 | **限流 + 返航（fallback）** 返回最近一次缓存的排行榜 |

## 第六步：扩展性与高可用设计

### 6.1 水平扩展

| 维度 | 扩展方式 |
|------|----------|
| **写入** | 1）Kafka 分区数随业务增长线性增加；2）消费者水平扩容（同一消费组） |
| **实时榜单** | Redis Cluster 按节点增加 **hash slot**；ZSET 大小天然分散 |
| **查询** | API Gateway 采用 **负载均衡**（LVS/Nginx），后端服务 **无状态**，可以随意扩容 |
| **历史快照** | ClickHouse 按 **分区**（日期、模式）自动分片，节点增加后自动平衡 |

### 6.2 容灾方案（99.9% 可用）

1. **多可用区部署**：Redis、MySQL、Kafka 均跨 AZ 部署 **主备**，故障转移时间 < 30 s。  
2. **灰度发布**：使用 **Canary** 或 **Blue/Green** 部署 API，确保新版本不影响全局。  
3. **监控报警**：  
   - **QPS、Latency**（Prometheus + Grafana）  
   - **Redis ZSET 大小**（告警阈值 70%）  
   - **Kafka Lag**（消费者落后 > 30 s）  
   - **错误率**（5xx）  

### 6.3 数据一致性模型

| 操作 | 强一致性需求 | 实现方式 |
|------|---------------|----------|
| **分数提交** | 写入后 5 s 可见 → **最终一致性** | Redis 实时写 + Kafka 异步落库 |
| **个人排名查询** | 必须看到最近 5 s 的写入 → **读写在同一 Redis** |
| **历史快照** | 只读、无需实时 → **强一致**（快照生成时使用锁/事务） |
| **防作弊校验** | 必须在写入前完成 → **强一致**（在 API 层使用 Redis 原子操作） |

> **如果要求全局强一致**（所有节点立即看到新分数），则需要 **分布式事务**（两段提交）或 **Paxos/Raft**，但代价极高，且与本题的 **5 s 可见** 目标不匹配。

## 第七步：常见面试追问与回答

### Q1️⃣：Redis ZSET 单节点容量有限，怎么办？

**回答要点**  
- **容量限制**：单个 ZSET 最大约 **2^32‑1** 成员，实际受内存限制（每条成员+score 大约 16 B + key overhead）。  
- **解决思路**  
  1. **分片**：使用 **Redis Cluster**，把不同维度（mode+region）分别放在不同的 hash slot，天然分散。  
  2. **分段**：若单维度榜单仍然太大（如全服排行榜），可以 **按分数区间切分**（如 0‑1w、1w‑10w），只保留热点区间在内存，低区间落库。  
  3. **冷热分层**：实时榜单只保留 Top‑N（如 Top‑100k）在 Redis，剩余数据写入 MySQL/ClickHouse，查询时若超出范围再走持久层。  
- **持久化**：Redis 本身提供 **RDB + AOF**，但不适合大规模持久化。我们把 **写入日志**投到 **Kafka**，再落库，保证 **可靠性**。

### Q2️⃣：写入高峰时（比如大型赛季活动）会出现大量瞬时上报，如何削峰？

**回答要点**  
- **异步队列**：所有提交先进入 **Kafka**（或 Pulsar），写入 Redis 的操作仍同步但非常轻量（`ZINCRBY`），实际持久化在消费者端批量落库。  
- **限流/熔断**：在 API 层使用 **令牌桶**或 **漏桶** 对每个玩家、每个模式进行 QPS 限制，防止单用户刷接口。  
- **批量写库**：消费者每隔 1 s 收集 500~1000 条记录一次性写 MySQL，显著降低磁盘 IO。  
- **弹性扩容**：Kafka 分区数可以提前预留（如 200 分区），活动期间通过 **kafka-reassign-partitions** 动态扩容。  

### Q3️⃣：历史快照要怎么压缩存储？查询时会不会慢？

**回答要点**  
- **列式存储**（ClickHouse）天然压缩（LZ4），只读取需要的列（如 `uid, rank, score`），磁盘 I/O 极低。  
- **分区裁剪**：表按 `snap_date`、`mode_id`、`region_id` 分区，查询时只扫描对应分区，几乎不产生全表扫描。  
- **增量快照**：只保存 **变化的 Top‑N**，若某天与前一天完全相同，只保存 **diff**（可选），进一步降低存储。  
- **归档**：90 天后将数据迁移到对象存储（OSS）+ 冷备份（Glacier），查询时若需要历史（超过 90 天），先把对应分区恢复到 ClickHouse 再查询，业务层做 “只查询最近 90 天” 的限制。  

### Q4️⃣：如果 Redis 节点宕机，排行榜会不会瞬间不可用？

**回答要点**  
- **主从复制**：Redis Cluster 每个分片都有 **1 主 + N 从**（建议 N≥1）。主节点故障后，从节点自动晋升为新主，过程几秒内完成。  
- **客户端重连**：使用 **Redis Cluster 客户端**（如 Jedis、go-redis）会自动发现新主节点。  
- **数据恢复**：写入 Kafka 的日志可以在新主恢复后通过 **重放**（消费者重新消费）恢复遗漏的写入。  
- **降级策略**：若短时间内仍无法获取排行榜，可返回 **缓存的上一轮榜单**（TTL 10 s），保证用户体验不至于报错。  

### Q5️⃣：如果需要在榜单上显示 “玩家上一次的最高分” 或 “历史最佳”，该怎么设计？

**回答要点**  
- 在 **MySQL** `user_score` 表中 **保存最高分**（`MAX(score)`）字段。  
- 每次分数提交后，消费者在落库时使用 `INSERT … ON DUPLICATE KEY UPDATE score = GREATEST(score, VALUES(score))`。  
- 实时查询时，可从 **Redis** 读取当前分数，再从 **MySQL** 拉取历史最高分（缓存），合并返回。  
- 若对实时性要求不高，直接把 **最高分** 也写入 **Redis**（另建 ZSET `leaderboard:mode:{mid}:region:{rid}:best`），保持同步。

## 心得与反思

### 1️⃣ 本题最难的设计决策

| 决策 | 思考过程 |
|------|----------|
| **实时榜单 vs 持久化的权衡** | 必须在 **秒级可见** 与 **强一致** 之间做取舍。最终决定采用 **Redis ZSET**（实时） + **Kafka 异步落库**（持久化），因为同步写 MySQL 会导致写入延迟 > 5 s，破坏用户体验。|
| **多维度分片方案** | 直接在单个 Redis 实例存放所有维度会导致 **热点**（如全球模式榜）和 **内存爆炸**。经过分析，采用 **Redis Cluster** + **key 前缀分片**（mode+region），同时预留 **热点榜单 Top‑N** 只保留在内存，其他数据落库。|

### 2️⃣ 新手最容易犯的错误（至少两条）

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有数据都放在 MySQL**，实时查询直接走库 | MySQL 在高并发读写场景下会成为瓶颈，查询延迟难以满足 100 ms。 | 使用 **内存缓存（Redis）** 保存实时排行榜，MySQL 只负责持久化和历史查询。 |
| **写入后立即返回成功**，但没有保证落库成功 | 若 Redis 写成功但 Kafka/DB 失败，数据会丢失，导致排行榜不一致。 | 引入 **事务性写入**（Kafka 事务或两阶段提交）或在 API 层做 **重试 + 死信队列**，确保持久化成功后才返回。 |
| **忽视防作弊**，只在前端做校验 | 客户端容易被逆向，作弊分数直接写入系统。 | 在 **后端**（API）进行 **阈值校验**，并在 **消费端** 再次校验；异常写入写入 **审计日志** 供后续分析。 |
