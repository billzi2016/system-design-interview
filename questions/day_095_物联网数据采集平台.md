# 第 95 天：设计 物联网数据采集平台

> 生成日期：2026-02-20

---

## 题目背景  
物联网数据采集平台用于统一接入、存储、处理和转发来自海量 IoT 设备（如传感器、摄像头、工业机器）的时序数据，为上层业务（监控、预测、告警、分析）提供可靠、低延迟的数据支撑。

## 面试场景设定  
> **面试官**：  
> “我们公司计划搭建一个面向数千万设备的物联网数据采集平台，要求能够实时接收设备上报的数据、做轻量级的预处理并持久化，同时要支持后续的实时分析和离线计算。请你从零开始设计这样一个系统，重点说明整体架构、关键技术选型以及如何满足高并发、低时延和高可用的需求。”

## 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| 1 | **设备接入**：支持 MQTT、CoAP、HTTP/HTTPS 三种常用协议的双向通信，能够动态注册/注销设备并进行身份鉴权。 |
| 2 | **时序数据写入**：接收每秒上万条/设备的结构化/半结构化数据（如 JSON、ProtoBuf），并在写入前完成格式校验、去重、基本过滤。 |
| 3 | **实时流处理**：对采集的数据进行实时聚合、阈值告警、异常检测等轻量计算，并将结果推送至下游系统（如告警服务、Dashboard）。 |
| 4 | **持久化存储**：将原始时序数据和处理后的结果长期保存，支持按时间、设备、标签等维度的高效查询。 |
| 5 | **数据查询与订阅**：提供 RESTful API 与 WebSocket/Server‑Sent Events 接口，供业务方查询历史数据或订阅实时数据流。 |
| 6 | **运维监控**：提供平台自身的指标监控（吞吐、延迟、错误率、资源使用）和自恢复能力（故障自动切换、限流降级）。 |

## 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **并发设备数** | 10 000 000 台 | 峰值同时在线设备数 |
| **吞吐量（写入 QPS）** | 100 000 TPS（每秒上报 10 条数据/设备） | 包括协议解码、校验、写入 |
| **写入延迟** | ≤ 200 ms（99% 请求） | 从设备发送到数据落库的端到端延迟 |
| **可用性** | 99.95%（月度累计停机 ≤ 22 min） | 支持多地域容灾 |
| **存储容量** | 30 PB/年（原始数据 1 KB/条） | 需要冷热分层存储 |
| **查询响应时间** | ≤ 1 s（常规时序查询） | 支持范围查询、聚合查询 |

## 系统边界  

**本题范围内**  
- 设备接入层、数据写入流水线、实时流处理、时序存储（冷热分层）以及对外查询/订阅接口的设计。  
- 鉴权、限流、故障恢复、监控报警的高层方案。  

**本题范围外**  
- 具体的业务侧数据分析模型（机器学习、预测）实现细节。  
- 设备端固件、网络接入层（如网关硬件）实现。  
- 第三方支付、计费系统、用户账号系统（仅考虑鉴权方式）。  
- 法律合规（GDPR、数据脱敏）细节实现。

# 题解

# 物联网数据采集平台系统设计详细解答  

> **温馨提示**：本答案把 **“从零到可用 → 高可用分布式”** 的完整思路一步步展开，适合 **完全没有系统设计经验的同学**。每一个技术选型、每一次划分边界，都配有“**为什么这样做**” 与 “**不这么做会怎样**”。阅读时请跟随章节顺序，先弄清需求、再画出大概框架、最后补齐细节。

## ## 解题思路总览  

1. **先把需求拆成可度量的指标**（并发设备、TPS、延迟、存储量），再算出每个子系统的负载。  
2. **从最小可运行系统（MVP）出发**：只保留「设备接入 → 简单写入 → 查询」三条链路，快速验证业务价值。  
3. **逐步加入缺口**：协议网关、流处理、冷热存储、监控、容灾……每加一个功能，都先思考  
   - *它解决了哪个痛点？*  
   - *对已有组件有什么依赖/影响？*  
4. **每一步都写出「为什么」**（可扩展性、容错、成本、运维难度），并给出 **不这么做的后果**（单点故障、写入热点、查询慢等）。  
5. **把关键技术点列出来**：MQTT Broker、Kafka、Fluent Bit、Apache Flink、TimescaleDB / ClickHouse / InfluxDB、对象存储、Prometheus+Grafana 等。  
6. **在每个模块里补齐**：数据模型、分区键、路由策略、背压、限流、容错、监控指标。  

下面按照 **需求 → 架构 → 数据库 → API → 组件 → 高可用 → 追问** 的顺序展开。

## ## 第一步：理解需求与规模估算  

| 需求 | 关键指标 | 计算方式 | 结果 |
|------|----------|----------|------|
| **设备接入** | 峰值在线设备数 | 10 M | 需要能够管理 10 M 长连接（MQTT/CoAP）或短连接（HTTP） |
| **写入吞吐** | 每设备每秒上报条数 | 10 条 × 10 M = **100 M 条/秒**（即 100 k TPS 按题目要求） | 实际写入 QPS ≈ 100 k（每秒 100 k 条） |
| **数据大小** | 1 KB/条 | 100 k × 1 KB = **100 MB/s** ≈ **8.6 TB/天** |
| **年度存储** | 1 KB × 10 条/秒 × 10 M 设备 × 365 天 ≈ **30 PB** | 与题目给出一致 |
| **延迟要求** | 99% ≤ 200 ms | 包括网络、协议解码、校验、写入、持久化等所有环节 |
| **可用性** | 99.95% (月停机 ≤ 22 min) | 必须做到多 AZ/Region 冗余、自动故障转移 |

### 1.1 业务流程拆解  

1. **设备 → 接入层（网关）**：协议解码、鉴权、限流、去重、过滤  
2. **接入层 → 消息总线**：高吞吐、持久化、可回溯（Kafka）  
3. **消息总线 → 实时流处理**：聚合、告警、异常检测 → 推送结果（WebHook、MQ）  
4. **消息总线 → 持久化写入**：原始数据 → 热存储（时序 DB）  
5. **热存储 → 冷存储迁移**：周期性归档到对象存储（S3/OSS）  
6. **查询层**：RESTful + WebSocket/SSE → 读取热/冷数据，做聚合、分页等  
7. **运维层**：监控、告警、日志、限流、灰度发布、自动伸缩  

### 1.2 关键痛点（后续会针对性解决）  

| 痛点 | 影响 | 需要关注的技术点 |
|------|------|----------------|
| **海量长连接管理** | 连接数太多导致内存/FD耗尽、单点故障 | 负载均衡、水平扩展的 MQTT/CoAP Broker |
| **写入热点** | 同一时间大量设备写入同一分区导致写放大 | 分区键设计、写入路由、预写缓存 |
| **实时处理的背压** | 流处理慢会导致 Kafka 积压，最终导致磁盘满 | Flink 检查点、流控、水平扩容 |
| **冷热数据分层** | 直接在单一 TSDB 保存 30 PB → 性能下降、成本高 | 热库/冷库分层、归档策略 |
| **查询延迟** | 跨天、跨设备的大范围查询如果走冷库会很慢 | 多层查询路由、物化视图、缓存 |

## ## 第二步：高层架构设计  

下面先给出 **最小可用系统（MVP）**，随后逐层演进到 **完整高可用分布式方案**。  

### 2.1 MVP（单机版）  

```
[Device] --> [Gateway (MQTT+HTTP)] --> [Kafka (单节点)] --> [Flink (单实例)] 
                                 |                               |
                                 +---> [TimescaleDB (单实例)] <--+
                                 |
                                 +---> [REST API] --> [Web UI]
```

- **为什么可以从单机开始**：  
  - 先验证业务价值（数据能否完整采集、基本查询是否满足）。  
  - 开发成本低、调试方便。  
  - 只要把每个组件的 **配置**（如 Kafka 分区数）调大，就能在后期平滑迁移到集群。  

- **不足**：单点故障、容量受限、无法支撑 10 M 设备。  

### 2.2 完整高可用分布式架构  

> **整体结构（文字版）**  

```
+-------------------+        +-------------------+        +-------------------+
| 设备 (10M)        |  <---> | 负载均衡层 (L4)   |  <---> | 接入网关集群      |
| MQTT/CoAP/HTTP   |        |  (Envoy/Nginx)   |        |  - MQTT Broker    |
+-------------------+        +-------------------+        |  - CoAP Server   |
                                                          |  - HTTP API      |
                                                          +-------------------+
                                                                   |
                                                                   v
+-------------------+      +-------------------+      +-------------------+
| 统一消息总线      | <-- | 实时流处理集群    | --> | 结果推送服务      |
| Kafka (多集群)   |      | Flink / Spark    |      | WebHook, MQ, etc |
+-------------------+      +-------------------+      +-------------------+
        |                                 |
        v                                 v
+-------------------+            +-------------------+
| 热时序存储集群    |            | 冷归档存储 (OSS) |
| TSDB/ClickHouse   |            | (对象存储)       |
+-------------------+            +-------------------+
        |                                 |
        +--------> 查询层 (REST+WS) <----+
                 API网关 (Gateway)
                 - 缓存层 (Redis)
                 - 授权中心 (OAuth2 / JWT)
```

#### 关键层次说明  

| 层次 | 主要职责 | 推荐技术 | 设计要点 |
|------|----------|----------|----------|
| **负载均衡层** | 将外部流量均匀分发到网关，提供 **TLS 终结**、IP 透明度、健康检查 | Envoy、NGINX、ALB（云） | 采用 **L4/TCP** + **L7/HTTP** 双层，支持 **Session Stickiness**（MQTT 长连接） |
| **接入网关** | **协议解析、鉴权、限流、去重、过滤**；对外提供 **双向**（下行指令）<br>需要水平扩展、状态共享 | **EMQX / VerneMQ**（MQTT），**CoAP Server (libcoap)**，**Kong + Lua**（HTTP）| - **Cluster Mode**（基于 etcd）<br>- 统一 **Device Registry**（MySQL/Redis）<br>- **Back‑pressure**：当下游 Kafka 堆积超过阈值时，主动 **拒绝/降速** 连接 |
| **统一消息总线** | 高吞吐、持久化、顺序保证、可回放 | **Apache Kafka**（3‑5 个副本、分区 10k+）| - **Topic per data type**（raw、alarm、metric）<br>- **Key = deviceId** → 确保同一设备数据落在同一分区，便于顺序消费 |
| **实时流处理** | 轻量聚合、阈值告警、异常检测、下发指令 | **Apache Flink**（Checkpoint + Exactly‑once）<br>或 **Spark Structured Streaming** | - **Event‑time** + **Watermark** 处理乱序<br>- **State TTL** 防止状态爆炸 |
| **热时序存储** | 最近 30 天（或 1‑2 个月）查询、聚合、降采样 | **TimescaleDB**（PostgreSQL 扩展）<br>或 **ClickHouse**（列式）<br>或 **InfluxDB 2.x** | - **分区（Hypertable）** 按 **时间+deviceId**<br>- **压缩**、**索引**（tag）<br>- **写入并行**（multi‑writer） |
| **冷归档** | 长期存储、成本低、满足合规 | **对象存储**（Amazon S3 / Alibaba OSS）+ **Parquet/ORC** | - **批量归档**：每小时/每天把热点分区的数据导出为 **Parquet**，写入对象存储<br>- **元数据库**（Glue / Hive Metastore）记录文件路径、分区 |
| **查询层** | 对外提供 **RESTful**、**WebSocket/SSE**、**SQL** 接口 | **API Gateway**（Kong、APISIX）+ **GraphQL/REST** + **Cache (Redis)** | - **读写分离**：热点查询走热库，历史查询走冷库（Presto / Trino）<br>- **限流**：每个租户/APP 配额 |
| **运维监控** | 指标采集、告警、自动伸缩 | **Prometheus + Grafana** + **Alertmanager**<br>日志 **ELK/EFK** | - **业务指标**（设备在线数、写入延迟）<br>- **系统指标**（Kafka lag、Flink checkpoint latency） |
| **安全/鉴权** | 统一身份校验、访问控制 | **OAuth2 + JWT**、**X.509**（设备证书） | - **Device Registry** 保存 **Key/Cert**、**租户**、**标签**<br>- **动态下发** ACL（ACL → MQTT topics） |

## ## 第三步：数据库设计  

### 3.1 数据模型（原始数据）  

| 字段 | 类型 | 说明 | 是否为标签（tag） |
|------|------|------|-------------------|
| `device_id` | STRING | 唯一设备标识 | **是**（用于分区/索引） |
| `timestamp` | BIGINT (epoch ms) | 采集时间 | **是**（时间分区） |
| `metric` | STRING | 业务指标名称（如 temperature） | **是** |
| `value` | DOUBLE / STRING | 实际数值或 JSON 字符串 | 否 |
| `tags` | JSON | 业务自定义标签（如 location、firmware） | **是**（展开为列） |
| `raw_payload` | BYTES | 原始报文（保留） | 否 |
| `processed_flag` | BOOLEAN | 是否已被流处理标记 | 否 |

> **为什么把 `device_id`、`timestamp`、`metric` 设为 tag**  
> - 在时序数据库中 **tag** 会自动建立倒排索引，查询时可以 **高速过滤**。  
> - 这三列几乎是所有查询的 **必选条件**（按设备、时间、指标），放在 tag 中可以避免全表扫描。  

### 3.2 分区（Sharding）策略  

#### 3.2.1 采用 **时间 + 哈希** 双重分区  

- **时间分区**（如 **1 天** 或 **1 小时**）保证 **写入均匀**，并且 **自动过期**（DROP PARTITION）。  
- **哈希分区**：在每个时间分区内部，依据 `hash(device_id) % N` 再划分 **子分区**（N 取 10~100，视集群规模而定）。  

**好处**：  

| 场景 | 传统单时间分区 | 双重分区 |
|------|----------------|----------|
| 写入热点（同一时间大量设备） | 可能导致单分区写入冲突 | 通过哈希分散到多个子分区，写入并行度提升 |
| 查询（按设备+时间） | 需要扫描整天分区 | 只定位到对应子分区，大幅降低 I/O |
| 数据迁移 | 整天迁移代价大 | 子分区可独立迁移，支持 **冷热分层** 更细粒度 |

> **实现方式**（以 TimescaleDB 为例）  
```sql
-- 创建 hypertable，按时间分区
SELECT create_hypertable('device_metric', 'timestamp',
                         partitioning_column => 'device_id',
                         number_partitions => 64);   -- 64 为哈希分区数
```

> **如果使用 ClickHouse**：  
```sql
CREATE TABLE device_metric (
    device_id String,
    timestamp DateTime64(3),
    metric    String,
    value     Float64,
    tags      Nested(name String, value String)
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)        -- 时间分区
ORDER BY (device_id, metric, timestamp)   -- 主键支持范围查询
```

### 3.3 冷数据归档  

1. **归档触发**：**每 24 小时**，对上一天的分区执行 **导出 → 压缩 → 上传**。  
2. **导出格式**：**Parquet**（列式、压缩率高、支持 Predicate Pushdown）。  
3. **元数据管理**：使用 **Glue / Hive Metastore** 记录 `s3://bucket/device_metric/year=2026/month=05/day=25/` 等路径。  
4. **查询路径**：  
   - **热点（最近 30 天）** → 直接查询 **热库**。  
   - **历史（>30 天）** → **Presto / Trino** 读取 **对象存储** 中的 Parquet，返回给 API。  

> **不做冷热分层的后果**：  
> - 热库需要无限扩容，查询性能下降。  
> - 存储成本飙升（高性能 SSD 价格高）。  

### 3.4 其它存储选型对比  

| 方案 | 写入吞吐 | 查询延迟（热点） | 冷归档成本 | 备注 |
|------|----------|------------------|------------|------|
| **TimescaleDB** | 10k–30k TPS 单实例，集群可扩容 | ≤ 200 ms | 通过 COPY → Parquet → OSS | 基于 PostgreSQL，SQL 友好 |
| **ClickHouse** | 100k+ TPS（列式写入） | ≤ 100 ms（聚合） | 同上 | 适合大规模聚合查询 |
| **InfluxDB 2.x** | 50k TPS | ≤ 300 ms | 官方 **TSM → Object Store** | 较轻量，但生态相对较小 |
| **Cassandra + ScyllaDB** | 极高写入 | 查询需要二次建模 | 需要自行实现归档 | 适合宽表、强一致需求 |

> **推荐**：**ClickHouse + Parquet + OSS** 组合。写入快、聚合查询快、成本低。若业务对事务/SQL 兼容性要求高，可使用 **TimescaleDB**。

## ## 第四步：核心 API 设计  

### 4.1 鉴权模型  

- **设备端**：使用 **X.509 证书** 或 **预共享密钥 (PSK)** + **JWT**（短期 token）  
- **业务方（APP）**：OAuth2 授权码模式 → 获取 **JWT**，在请求头 `Authorization: Bearer <token>`  

> **为什么采用双套鉴权**  
> - 设备需要 **强身份**（证书）防止伪造；  
> - 业务方使用 **OAuth2** 便于统一管理租户、权限、审计。  

### 4.2 RESTful 接口（示例）  

| 方法 | 路径 | 描述 | 请求体 | 返回 |
|------|------|------|--------|------|
| `POST` | `/api/v1/devices/{deviceId}/metrics` | 单条/批量上报（HTTP） | `{ "timestamp": ..., "metric": "...", "value": ..., "tags": {...} }` | `202 Accepted` |
| `GET` | `/api/v1/devices/{deviceId}/metrics` | 查询历史数据 | `?metric=temperature&start=...&end=...&limit=...` | JSON 数组 |
| `GET` | `/api/v1/streams/subscribe` | WebSocket 实时订阅 | 参数同上（filter） | WebSocket 消息流 |
| `POST` | `/api/v1/alarms` | 告警上报（下游系统） | `{ "deviceId": ..., "type": ..., "severity": ..., "detail": ... }` | `201 Created` |

> **实现细节**  
> - **请求入口**：API Gateway → **Rate Limiter**（基于租户、IP） → **Auth Service** → **业务服务**（Spring Boot / Go）  
> - **写入路径**：业务服务把数据写入 **Kafka Producer**，不直接写 DB，保持 **解耦**。  

### 4.3 WebSocket / SSE  

- **用途**：业务方想要 **实时推送**（如监控大盘）。  
- **实现**：  
  - **WebSocket**：每个连接在 **API Gateway** 中分配 **唯一订阅 ID**，后端 **Redis Pub/Sub** 将 **Flink** 计算结果写入对应频道。  
  - **SSE**：对不需要双向交互的轻量客户端（如浏览器）使用，兼容性更好。  

> **为什么要提供两种方式**  
> - **WebSocket** 支持 **双向**（平台下发指令），适用于 **控制类** 场景。  
> - **SSE** 更轻量，省去心跳帧，适合 **只读** 场景，降低服务器资源占用。  

### 4.4 错误码约定（示例）  

| HTTP 状态码 | 业务码 | 含义 |
|-------------|--------|------|
| 200 | 0 | 成功 |
| 400 | 1001 | 参数错误 |
| 401 | 1002 | 鉴权失败 |
| 403 | 1003 | 权限不足 |
| 429 | 1004 | 限流（超出配额） |
| 500 | 2001 | 系统内部错误 |

## ## 第五步：详细组件设计  

下面逐个拆解关键组件的内部实现要点、选型、常见配置。  

### 5.1 接入网关（协议层）  

| 需求 | 实现方式 | 关键配置 | 监控指标 |
|------|----------|----------|----------|
| **MQTT** | EMQX **Cluster**（基于 etcd） | - **最大连接数** 12M（每节点 2M）<br>- **QoS** 0/1 支持<br>- **Bridge** → Kafka | 连接数、QoS 错误率、订阅分布、CPU/内存 |
| **CoAP** | libcoap + custom **Koa** 代理 | - **DTLS** 端口 5684<br>- **Observe** 支持 | 包丢失率、响应时间 |
| **HTTP/HTTPS** | Kong + **Lua** 插件 | - **Rate‑limit** (IP+Tenant)<br>- **Request Body Size** ≤ 2 KB | 请求 QPS、错误率、TLS 握手时长 |
| **统一鉴权** | 设备 Registry (MySQL) + **Redis** 缓存 | - **Device→Tenant→PublicKey** 表<br>- **TTL 5 min** 缓存 | 鉴权成功率、Cache 命中率 |
| **限流/背压** | 当 Kafka **lag** > 5 M，向网关发送 **BACK_PRESSURE** 信号（MQTT CONNACK 0x8F）| - **Kafka Lag监控**<br>- **动态调整** 连接速率 | Kafka Lag、网关拒绝率 |

> **不使用专门的 MQTT Broker**：如果直接在业务服务里实现 MQTT，会导致 **协议栈不成熟、连接管理混乱、难以水平扩容**。使用成熟的 EMQX 能让我们专注于业务逻辑。  

### 5.2 消息总线（Kafka）  

- **Topic 设计**  
  - `raw-data`（Key=deviceId）  
  - `alarm-event`（Key=alarmId）  
  - `downlink-cmd`（Key=deviceId）  

- **分区数**  
  - 目标 **吞吐** 100 k TPS → 假设每分区 5 k TPS，**分区数 ≈ 20**；考虑 **扩展**，预设 **200 分区**（每台 broker 10‑20 分区）。  

- **副本**：3 副本（保证 **RF=2** 时仍可写）  

- **生产者**  
  - **批量发送**（batch.size 64KB）  
  - **压缩**：`snappy`（降低网络 IO）  
  - **幂等**：`enable.idempotence=true`，防止重试导致重复写入  

- **消费者**（Flink、写入服务）  
  - **Exactly‑once**：使用 Flink Kafka connector 的 **transactional** 模式  
  - **Checkpoint interval**：30 s（可根据延迟 SLA 调整）  

> **为何不直接写 DB**：Kafka 提供 **持久化缓冲**、**水平扩展**、**回放** 能力，能够吸收突发流量峰值，且解耦了 **写** 与 **读**。  

### 5.3 实时流处理（Flink）  

- **作业结构**  

```
source (Kafka raw-data) 
   └─> map (decode + schema validation) 
   └─> keyBy(device_id) 
   └─> window (tumbling 1s) 
   └─> aggregate (avg, max, min) 
   └─> process (threshold check) 
   ├─> side-output (alarm) → Kafka alarm-event 
   └─> sink (Hot TSDB write) 
```

- **状态管理**  
  - 使用 **RocksDBStateBackend**，开启 **TTL**（30 天）防止状态无限增长。  

- **容错**  
  - **Checkpointing**：每 30 s，存储到 **HDFS** 或 **S3**。  
  - **Savepoint**：定期手动触发，支持版本升级。  

- **背压**  
  - Flink 内置 **backpressure** 机制，会自动减慢 source 读取速度。监控 `source_input_rate` 与 `checkpoint_duration`，当 lag > 5 M 时触发 **Kafka Consumer Pause**。  

> **如果改用 Spark**：批处理延迟会更高，不适合 200 ms 的实时 SLA。  

### 5.4 热时序存储（ClickHouse）  

- **表结构**（参考前文）  
- **写入方式**  
  - **INSERT** via **HTTP** 接口（批量 10k 条/次）  
  - **ClickHouse Kafka Engine**：直接把 `raw-data` Topic 挂载为表，实现 **零代码写入**。  

- **压缩**：`LZ4`（写入快、查询稍慢）或 `ZSTD`（压缩率高）可根据成本权衡。  

- **索引**  
  - **Primary Key**：`(device_id, metric, timestamp)`  
  - **TTL**：`TTL toDateTime(timestamp) + INTERVAL 30 DAY DELETE`（自动清理）  

- **查询优化**  
  - **Materialized View**：对热点聚合（如 1‑minute 平均）预计算，降低实时查询压力。  

### 5.5 冷归档（Parquet + OSS）  

- **作业**：使用 **Flink** 批作业或 **Spark**，每天 **读取上一天分区** → **写入 Parquet** → **上传 OSS**。  
- **分区目录**：`/year=2026/month=05/day=25/`，文件名 `device_metric_00001.parquet`。  
- **压缩**：`snappy`（兼容性好）或 `zstd`（压缩率更高）。  

- **查询**：  
  - **Presto/Trino** 读取对象存储，提供 **ANSI SQL** 接口。  
  - **API** 判断时间范围：若全在 30 天内，走 ClickHouse；否则走 Presto。  

### 5.6 查询层（API Gateway + Cache）  

- **缓存**：热点时间序列（最近 5 分钟）放入 **Redis**（Sorted Set）  
- **读写分离**：查询请求先检查 **Cache** → 未命中走 **ClickHouse** / **Presto** → 结果写回 Cache（TTL 30 s）。  

- **分页**：使用 **cursor**（基于 timestamp）避免 **OFFSET** 大导致全表扫描。  

- **安全**：每个租户只能查询自己 **tenant_id** 下的设备，SQL 注入防护使用 **prepared statements**。  

### 5.7 运维监控  

| 监控维度 | 关键指标 | 警报阈值 | 处理方式 |
|----------|----------|----------|----------|
| **网关** | 连接数、CPU、内存、TLS 握手时长 | 连接数 > 80% 容量 → 扩容 | 自动弹性伸缩（K8s HPA） |
| **Kafka** | **Lag**、ISR、磁盘使用率 | Lag > 5M → 触发背压 | 发送告警、自动增加 broker |
| **Flink** | 检查点成功率、状态大小、背压率 | 检查点失败 > 3 次/小时 | 重启作业、回滚 |
| **ClickHouse** | QPS、查询延迟、磁盘 I/O | 查询 P99 > 200 ms | 调整副本、增加节点 |
| **对象存储** | 读取错误率、带宽 | 读取错误 > 0.1% | 检查网络、切换区域 |
| **业务** | 写入成功率、告警触发率 | 成功率 < 99% | 排查网关/Kafka |

> **可视化**：Grafana 仪表盘 + **Alertmanager** 邮件/钉钉/Slack 通知

## ## 第六步：扩展性与高可用设计  

### 6.1 横向扩容（Scale‑out）  

| 组件 | 扩容方式 | 关键点 |
|------|----------|--------|
| **接入网关** | **K8s Deployment** + **StatefulSet**（保留连接） | 使用 **Pod Anti‑Affinity** 防止同 AZ 单点 |
| **Kafka** | 增加 **Broker**，重新分配分区 | **Rebalance** 时使用 **Rack‑aware**，避免跨机房流量 |
| **Flink** | 增加 **TaskManager**，调高 **parallelism** | **Checkpoint** 仍保存在共享存储 |
| **ClickHouse** | 增加 **Shard** 与 **Replica**，水平分片 | **Distributed Table** 跨 Shard 查询 |
| **对象存储** | 多 Region 镜像（CRR） | DNS 负载均衡 + **S3 Transfer Acceleration** |

### 6.2 多地域容灾（Active‑Active）  

1. **数据复制**：Kafka **MirrorMaker 2** 将数据同步到 **Region B**；ClickHouse **Replicated MergeTree** 跨 Region；对象存储 **跨 Region 同步**。  
2. **DNS 级别路由**：使用 **Anycast IP** 或 **GeoDNS**，将用户流量就近路由。  
3. **故障切换流程**  
   - 检测到 Region A 某关键服务不可用 → **自动切换** DNS → 客户端重新连接到 Region B。  
   - 数据一致性通过 **Kafka exactly‑once** + **Flink checkpoint** 保证。  

> **如果只做单 Region**：一旦机房网络或电力故障，整套平台不可用，达不到 99.95% SLA。  

### 6.3 数据一致性与幂等  

- **写入幂等**：Kafka **Keyed** 消息 + **Producer id**，确保同一条数据只写一次。  
- **去重**：在网关层做 **MD5** 校验 + **Cache**（最近 1 min）防止短时间重复上报。  
- **事务**：Flink → ClickHouse 使用 **Two‑Phase Commit Sink**，实现 **Exactly‑once** 写入。  

### 6.4 灾难恢复（DR）  

| 场景 | 恢复步骤 | RTO / RPO |
|------|----------|-----------|
| **Kafka** 故障 | 启动 **MirrorMaker** 目标集群的 **Consumer**，从备份 Topic 读取 | RTO < 5 min，RPO ≈ 0 |
| **ClickHouse** 丢失 | 使用 **Backup**（每日全量 + 增量）恢复至新节点 | RTO < 30 min |
| **Flink** 作业失效 | 通过 **Savepoint** 重启作业，恢复状态 | RTO < 2 min |
| **对象存储** 区域失效 | 从 **跨 Region** 副本恢复 | RTO < 10 min |

## ## 第七步：常见面试追问与回答  

> 下面列出面试官可能的追问，给出 **思路** 与 **关键要点**，帮助你在现场快速组织答案。  

### 7.1 “如果设备数量翻倍，系统还能支撑吗？”  

- **答案结构**  
  1. **瓶颈定位**：接入层（连接数）、Kafka（分区数）、TSDB（写入 TPS）是主要限制。  
  2. **横向扩容方案**：  
     - 增加 **接入网关节点**（使用 **Pod Anti‑Affinity** 保证均衡）。  
     - **Kafka** 增加 **Broker**，重新 **reassign partitions**（保持每个分区写入量在 5k TPS 以下）。  
     - **ClickHouse** 增加 **Shard**，每个 Shard 处理 1/2 设备的数据。  
  3. **容量规划**：提前预留 **磁盘 IOPS**、**网络带宽**（10 Gbps），并使用 **自动弹性伸缩**。  

### 7.2 “为什么不直接把数据写入 MySQL？”  

- **原因**  
  1. **写入吞吐**：MySQL 单实例难以支撑 100k TPS，且行存储不适合时间序列的压缩/查询模式。  
  2. **查询性能**：时序查询往往是 **范围查询 + 聚合**，MySQL 需要大量索引，磁盘占用和查询延迟都会飙升。  
  3. **冷热分层**：MySQL 缺少 **TTL**、**分区裁剪**等时序特性，管理 30 PB 数据成本极高。  

### 7.3 “如果某个设备的上报频率异常提升（比如 1000 条/秒），会不会把系统拖垮？”  

- **防护措施**  
  1. **网关限流**：基于 **deviceId** 的 **Token Bucket**，超出阈值直接返回 **429**。  
  2. **Kafka 生产端背压**：设置 **max.in.flight.requests.per.connection**、**linger.ms**，当 **Broker lag** 超过阈值时 **pause** 生产者。  
  3. **Flink 滑动窗口**：对异常设备进行 **分流**（侧输出），单独处理或直接写入 **异常存储**（MongoDB）进行后期分析。  

### 7.4 “冷热分层的迁移会不会导致查询不一致？”  

- **保证一致性**  
  1. **双写**：在归档作业启动前，**原始数据仍保留在热库**，查询层先检查 **时间范围**，若跨冷热，则 **并行查询** 两侧并 **合并结果**。  
  2. **事务**：归档完成后，用 **元数据表** 标记该分区已归档，随后 **删除热库分区**（TTL）。  
  3. **查询路由**：API 层通过 **时间戳** 自动选择 **ClickHouse**（热）或 **Presto**（冷），并统一返回统一 JSON 格式。  

### 7.5 “如何处理时序数据的乱序问题？”  

- **技术点**  
  1. **Kafka** 保证 **partition 内顺序**，但跨分区仍可能乱序。  
  2. **Flink** 使用 **Event‑time** + **Watermark**：设置 **maxOutOfOrderness = 5s**（根据业务容忍度）。  
  3. **窗口延迟**：如果数据迟到超过 watermark，使用 **side‑output** 保存到 **Dead‑Letter Queue**（Kafka）供后续补偿。  

### 7.6 “如果要在平台上实现用户自定义的聚合查询，怎么做？”  

- **方案**  
  1. **SQL 接口**：对外暴露 **Presto/Trino**（对冷热数据统一）或 **ClickHouse HTTP** 接口。  
  2. **安全沙箱**：使用 **SQL 解析器**（Apache Calcite）限制只能查询 **device_metric** 表，禁止 **DROP**、**INSERT** 等 DDL。  
  3. **资源配额**：对每个租户设置 **查询并发**、**CPU/内存** 限额，防止大查询拖垮集群。

## ## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 思考过程 |
|------|----------|
| **协议网关的选型 & 连接管理** | - 必须兼顾 **MQTT 长连接**（数百万）与 **HTTP 短连接** 的高并发。<br>- 直接自己实现协议栈会导致 **状态同步**、**水平扩展**、**容错** 的复杂度爆炸。<br>- 最终选择 **成熟的 EMQX/VerneMQ**（支持集群、分布式会话、桥接 Kafka），并配合 **Redis** 做设备注册缓存，实现 **一致性** 与 **快速鉴权**。 |
| **冷热分层的迁移策略** | - 30 PB/年数据如果全放在热库，成本不可接受且查询性能受限。<br>- 需要 **无缝迁移**（查询不出现“空白”或“重复”），并且 **归档过程** 不能影响写入吞吐。<br>- 采用 **Kafka → Flink 批作业** 按天导出 Parquet、写入 OSS，同时 **保留热库分区** 并使用 **元数据表** 做状态标记，查询层根据时间路由，实现 **查询一致性**。 |

### 2️⃣ 新手最容易犯的错误  

| 错误 | 解释 |
|------|------|
| **只关注单机 MVP，直接把所有组件放在一台机器** | 看似快速，但 **单点故障**、**资源瓶颈**（CPU、磁盘 I/O、网络）会在真实流量下立刻崩溃。面试时要明确 **水平扩展** 与 **容错** 的路线。 |
| **把 Kafka 当作 “消息队列” 而忽视分区设计** | 随意使用默认分区会导致 **热点写入**（所有设备同一分区），写入吞吐受限，甚至出现 **leader 选举抖动**。必须基于 **device_id hash** 进行分区并提前规划分区数量。 |
| **在时序数据库里直接存全部原始 JSON** | 虽然灵活，但 **查询效率** 极差，且 **压缩率低**，导致存储成本暴涨。应把结构化字段抽出来做 **tag**，原始报文可选保留（如 raw_payload），但不作为查询主键。 |
| **没有统一的背压/限流方案** | 当下游出现卡顿（Kafka 堆积、Flink 检查点慢），上游仍持续写入会导致 **磁盘写满**、**OOM**。需要在网关层、Kafka Producer、Flink Source 都实现背压或限流。 |

### 3️⃣ 学习建议与可延伸方向  

1. **系统设计基础**  
   - 阅读《Designing Data‑Intensive Applications》（Martin Kleppmann）了解 **CAP、数据模型、流批统一**。  
   - 练习 **从需求 → 规模估算 → 架构图** 的完整流程。  

2. **关键技术深入**  
   - **MQTT/CoAP**：掌握协议细节、QoS、主题过滤、桥接。  
   - **Kafka**：了解 **ISR、rebalancing、exactly‑once**。动手搭建本地集群，实验 **分区、复制**。  
   - **Flink**：从 **DataStream** 入手，学会 **Event‑time、Watermark、State TTL**。  
   - **ClickHouse / TimescaleDB**：对比 **列式存储 vs. 扩展 PostgreSQL**，熟悉 **分区、TTL、Materialized View**。  

3. **实战项目**  
   - 搭建一个 **小规模 IoT 平台**（几千设备），完整走通 **接入 → Kafka → Flink → ClickHouse → API**。  
   - 通过 **Prometheus+Grafana** 监控关键指标，练习 **故障注入**（Kill broker、网络抖动），观察系统自愈。  

4. **关注行业趋势**  
   - **Edge‑Computing**：将部分过滤/聚合下沉到网关。  
   - **Serverless**：使用 **AWS Lambda / Azure Functions** 处理突发告警。  
   - **OpenTelemetry**：统一链路追踪，帮助定位 **200 ms 延迟** 的瓶颈。  

> **一句话总结**：系统设计不是一次性“画图”，而是 **需求 → 规模 → 关键瓶颈 → 技术选型 → 可扩展/高可用方案** 的循环迭代。把每一步的 **“为什么”** 说清楚，面试官会更看重你的思考过程而不是记忆的技术点。
