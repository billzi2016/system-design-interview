# 快递物流追踪系统设计面试题

## 题目背景
快递物流追踪系统用于为寄件人、收件人以及内部运营人员提供包裹在全链路（揽收 → 中转 → 派送）的实时状态查询、异常告警和历史轨迹回溯等服务。系统需要支撑海量包裹的高并发查询，同时保证数据的时效性和可靠性。

## 面试场景设定
> **面试官**：  
> “我们现在要设计一个面向全国用户的快递物流追踪系统，请你从零开始构建它的整体架构。请先说明核心业务流程，然后围绕系统的功能、性能、可用性等方面给出你的设计方案。可以先从系统的主要功能点说起，接下来我们会逐步深入细节。”

## 功能性需求
1. **实时状态查询**：用户（寄件人/收件人）通过运单号查询当前物流状态（已揽收、已发往中转、正在派送、已签收等），并显示最近一次更新时间。  
2. **物流轨迹可视化**：展示包裹从揽收到签收的完整时间线及对应的位置信息（城市/站点），支持地图定位。  
3. **异常告警推送**：当包裹出现异常（如滞留、破损、错投等）时，系统向用户和客服推送短信/APP 推送通知。  
4. **批量查询 API**：对接企业客户，提供批量查询接口（一次最多 1000 条运单号），返回 JSON 格式的最新状态。  
5. **数据同步与回溯**：内部运营需要能够查询过去 90 天的物流历史数据，用于审计和客服分析。  
6. **多端统一入口**：提供 Web、移动端（iOS/Android）以及微信小程序统一的查询入口，统一返回同一套状态数据。

## 非功能性需求
| 指标 | 目标值 | 备注 |
|------|--------|------|
| **DAU（日活跃用户）** | 5 百万 | 包含普通消费者、企业客户、内部运营人员 |
| **QPS（查询请求峰值）** | 30 k QPS | 高峰期（上午 10:00–12:00、下午 16:00–18:00） |
| **查询延迟** | ≤ 200 ms（99% 请求） | 包括网络、后端处理、缓存等总时延 |
| **可用性** | ≥ 99.95%（年可用时间 ≥ 4.38 小时停机） | 支持自动容灾、灰度发布 |
| **存储量** | 约 300 TB（90 天日志 + 索引） | 以分布式存储为主，冷热分层 |
| **数据一致性** | 最终一致性，单笔状态更新在 5 秒内对外可见 | 兼顾高并发写入与查询性能 |

## 系统边界
**本题范围（需要设计）**  
- 包裹状态采集（与快递员/站点的上报接口）  
- 实时查询服务、批量查询 API、异常告警推送  
- 数据存储方案（热数据、冷数据、归档）与缓存层  
- 高可用/容灾架构、负载均衡、监控报警  

**本题不考虑（可略过）**  
- 第三方支付、订单管理、商品信息系统  
- 具体的配送路线优化、车辆调度算法  
- 站点内部的仓储管理系统（WMS）细节  
- 细粒度的权限体系（如 RBAC）和审计日志的合规存档  

## 提示与追问
1. **数据采集频率与可靠性**：如果站点上报的状态数据出现网络抖动或重复上报，如何保证数据的唯一性和幂等性？  
2. **缓存失效策略**：在高并发查询场景下，如何设计缓存的失效与预热机制，防止“缓存击穿”导致后端数据库瞬时压力激增？  
3. **异常告警的推送路径**：针对不同异常类型（如时效滞留 vs. 包裹破损），如何设计多渠道（短信、邮件、APP 推送）的优先级和降级策略？

## ## 解题思路总览  

1. **先把业务拆解成最小可运行单元**（采集、查询、告警），先做单体或单机原型，验证功能正确性。  
2. **估算规模** → 计算 QPS、数据量、存储大小，明确是否需要分布式。  
3. **画出高层结构图**，把流量入口、业务服务、存储、缓存、消息队列、监控等模块摆出来。  
4. **逐层细化**：  
   - 数据模型 → 关系型 vs. 时序/列式存储  
   - API & RPC 接口  
   - 缓存、幂等、容错、降级策略  
5. **在每一层都加入“为什么这样做 + 不这样会怎样”**，帮助面试官看到你的思考过程。  
6. **最后补齐可用性、扩展性、监控、运维等非功能点**，并准备面试官常见追问的答案。  

下面按照 **“最小可用 → 高可用”** 的顺序，逐章展开。

## ## 第一步：理解需求与规模估算  

### 1. 功能需求回顾  

| 功能 | 关键点 |
|------|--------|
| 实时状态查询 | 单运单查询，返回最新状态 + 时间 |
| 轨迹可视化 | 按时间顺序的节点（城市/站点）+ 位置信息 |
| 异常告警推送 | 多渠道（短信/APP），支持降级 |
| 批量查询 API | 最多 1000 条运单号，返回 JSON |
| 数据回溯 | 90 天历史查询（审计、客服） |
| 多端统一入口 | 同一套后端 API，兼容 Web、APP、微信小程序 |

### 2. 非功能需求关键指标  

| 指标 | 目标 | 说明 |
|------|------|------|
| DAU | 5 M | 包含普通用户、企业、内部 |
| 峰值 QPS | 30 k | 业务高峰 2×峰值 = 60 k QPS（预留） |
| 查询延迟 | ≤200 ms (99%) | 包括网络+业务+缓存 |
| 可用性 | ≥99.95% | 年停机 ≤4.38 h |
| 存储 | ~300 TB (90 天日志) | 需要冷热分层 |
| 数据一致性 | 最终一致，5 s 可见 | 写入延迟 <5 s |

### 3. 规模估算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每日新单** | 假设 1 M 运单/天（5 M DAU × 20% 活跃） | 1 M |
| **单日状态上报次数** | 平均 5 次（揽收→中转→派送→签收+异常） | 5 M |
| **峰值写入 QPS** | 5 M / (24×3600) ≈ 58 写/s，峰值 5× = 300 写/s（保守） |
| **查询 QPS** | 30 k QPS（已给） |
| **每条状态记录大小** | 运单号 20B + 时间戳 8B + 状态枚举 2B + 位置信息 30B ≈ 60B |
| **单日日志大小** | 5 M × 60B ≈ 300 MB → 90 天 ≈ 27 GB（压缩后 10 GB） |
| **热点缓存数据** | 30 k QPS × 0.5（80% 访问热点）≈ 15 k QPS → 需要缓存 ~10 M 条最新状态（≈600 MB） |
| **总存储** | 90 天热数据（列式）+ 冷数据（对象存储）≈ 300 TB（包括索引） |

> **结论**：查询量大、写入相对较小，**读写分离 + 缓存** 是关键；日志量虽不算 TB 级，但 90 天全量查询仍需 **分布式存储**（列式/时序）来支撑。

## ## 第二步：高层架构设计  

### 1. 初版最小可用系统（MVP）  

```
[用户端] → (HTTPS) → [API Gateway] → [单体服务] → MySQL (单实例) 
                                 ↘︎
                                 → Redis (本地缓存)
                                 → MQ (Kafka) → 触发短信/推送
```

- **为什么**：  
  - 只用单台机器、单库，快速验证业务流程（采集、查询、告警）。  
  - 代码复杂度最低，适合面试演示概念。  
- **不这样会怎样**：  
  - 难以模拟高并发、故障恢复，面试官会追问扩展思路。  

### 2. 目标高可用分布式架构（满足 30 k QPS、99.95% 可用）  

> **核心思路**：**层层解耦** → 前端入口、业务服务、缓存、持久化、异步消息、监控。  

```
                           +-------------------+
                           |   CDN / WAF       |
                           +-------------------+
                                   |
                      +---------------------------+
                      |   API Gateway / LoadBalancer (L7) |
                      +---------------------------+
                                   |
            +---------------------------------------------------+
            |               Service Mesh (Envoy)               |
            +---------------------------------------------------+
           /                     |                     \
   +----------------+   +----------------+    +----------------+
   |   Query Service|   |   Ingest Service|   |   Alarm Service|
   +----------------+   +----------------+    +----------------+
        |  |  |               |  |  |               |  |
   +----+  |  +----+    +-----+  |  +----+    +-------+  |
   |       |       |    |          |       |    |          |
Redis (Cache)   Redis (Cache)   Kafka (Event Bus)   MySQL/TSDB
   |                |               |                |
   |                |               |                |
   +---->   ClickHouse / HBase (列式)   +---->   Object Storage (OSS)
```

#### 各层职责  

| 层级 | 组件 | 关键职责 | 设计理由 |
|------|------|----------|----------|
| **入口层** | CDN + WAF + API Gateway (Kong/Zuul) | 防 DDOS、统一鉴权、流量路由、灰度发布 | 将业务逻辑与网络层解耦，提升安全性 |
| **服务层** | Query Service（REST/GRPC）<br>Ingest Service（接收站点上报）<br>Alarm Service（异常检测 & 推送） | 业务细分，水平扩展 | 单一职责，容错粒度细 |
| **缓存层** | Redis Cluster（热点状态）<br>LocalCache（Sidecar） | 读热点降压、避免 DB 雪崩 | 读写比例高，缓存命中率决定延迟 |
| **消息层** | Kafka（Topic: `tracking_events`） | 异步落库、告警、脱敏 | 高吞吐、持久化、分区保证顺序 |
| **持久化层** | MySQL（事务写入）<br>ClickHouse（时序/列式）<br>对象存储（OSS） | 事务安全 + 大规模分析 + 冷归档 | 结合 OLTP + OLAP，满足 90 天查询 |
| **监控层** | Prometheus + Grafana + Alertmanager | 指标采集、告警、容量预警 | 可观测性是高可用前提 |

> **不这样做的后果**：  
> - 直接把所有请求落到单库会导致 **读写冲突、锁竞争、CPU/IO 饱和**，查询延迟瞬间突破 200 ms。  
> - 没有消息队列，写入失败会导致数据丢失，异常告警无法可靠推送。

## ## 第三步：数据库设计  

### 1. 数据模型概览  

| 表/集合 | 主键 | 主要字段 | 说明 |
|--------|------|----------|------|
| **tracking_event**（MySQL） | `id` (auto increment) | `tracking_no` (varchar), `status` (enum), `timestamp`, `location_id`, `city`, `province`, `operator_id`, `detail` (json) | 实时写入，确保事务一致性（单条状态） |
| **tracking_latest**（Redis） | `tracking_no` | `status`, `timestamp`, `city`, `province` | 缓存最新状态，TTL 5 min（防止脏数据） |
| **tracking_history**（ClickHouse） | `tracking_no` (partition), `timestamp` (order) | 同 MySQL + `lat`, `lng` | 适合 90 天历史查询、聚合分析 |
| **alarm_event**（MySQL） | `id` | `tracking_no`, `alarm_type`, `status` (未处理/已处理), `created_at` | 记录异常告警，供客服系统查询 |
| **location**（MySQL） | `location_id` | `city`, `province`, `lat`, `lng` | 站点/网点信息，供逆向地理编码使用 |

> **为什么分表**：  
> - **tracking_event** 负责事务写入，行数每日约 5 M，单表仍在可接受范围，但为避免长时间热点，**按照日期分区**（MySQL Partition）或 **Sharding**（region）更安全。  
> - **tracking_history** 需要 **列式存储** 以支撑大量扫描查询（时间线），ClickHouse 天然适合。  
> - **alarm_event** 频率低，放在 MySQL 即可。

### 2. 关键索引  

| 表 | 索引 | 目的 |
|----|------|------|
| tracking_event | `UNIQUE (tracking_no, timestamp)` | 防止重复上报，保证幂等 |
| tracking_event | `INDEX idx_tracking_no (tracking_no)` | 快速定位单运单历史 |
| tracking_history | `PARTITION BY toYYYYMMDD(timestamp)`<br>`ORDER BY (tracking_no, timestamp)` | 时间范围查询高效 |
| alarm_event | `INDEX idx_tracking_no (tracking_no)` | 客服快速查询异常 |
| location | `PRIMARY KEY (location_id)` | 站点查询 |

### 3. 数据写入流程（确保幂等 & 唯一性）  

1. **站点/快递员上报** → **Ingest Service** → 写入 **Kafka**（`tracking_events`）。  
2. **Consumer**（单独的落库服务）读取消息：  
   - 使用 **`tracking_no + timestamp`** 生成 **业务唯一键**（或业务方提供的 `event_id`）。  
   - **先写入 MySQL**（INSERT … ON DUPLICATE KEY UPDATE），确保幂等。  
   - **写入 ClickHouse**（异步批量导入），保证历史数据。  
   - **更新 Redis**（最新状态），并 **设置 TTL** 防止永久缓存失效。  

> **如果不做幂等**：网络抖动导致同一状态多次写入，历史记录会出现重复，查询时间线混乱，告警可能误触。

### 4. 读查询路径  

| 场景 | 查询路径 | 说明 |
|------|----------|------|
| **单条实时查询** | API → Redis（`tracking_latest`）→ 若 Miss → MySQL（最新记录）→ 回写 Redis | 读热点直接命中缓存，确保 <200 ms |
| **历史轨迹查询** | API → ClickHouse（`tracking_history`） | 大批量时间范围扫描，列式压缩高效 |
| **批量查询** | API → Redis MGET（批量键） → 并发请求 MySQL 对 Miss | 批量命中率高，利用 pipeline 减少 RTT |
| **异常告警查询** | API → MySQL（`alarm_event`） | 数据量小，直接查询即可 |

## ## 第四步：核心 API 设计  

### 1. API 风格  

- **RESTful** + **JSON** 为对外统一接口（Web、APP、微信小程序）。  
- **内部服务** 使用 **gRPC**（低延迟、强类型）在 Service Mesh 中通信。  

### 2. 对外公开的关键接口  

| 方法 | 路径 | 请求示例 | 响应示例 | 说明 |
|------|------|----------|----------|------|
| `GET /api/v1/tracking/{tracking_no}` | 单条实时查询 | `GET /api/v1/tracking/YT1234567890` | `{ "tracking_no":"YT1234567890", "status":"DELIVERING", "city":"上海", "timestamp":"2026-05-26T10:23:00Z" }` | 缓存命中 → 直接返回 |
| `GET /api/v1/tracking/{tracking_no}/history?start=...&end=...` | 轨迹历史 | `GET /api/v1/tracking/YT1234567890/history?start=2026-05-20&end=2026-05-26` | `{ "tracking_no":"YT1234567890", "history":[ {"status":"PICKED","city":"北京","timestamp":"..."} , …] }` | ClickHouse 查询 |
| `POST /api/v1/tracking/batch` | 批量查询 | `{ "tracking_nos":["YT1","YT2",...] }` | `{ "results":[ {"tracking_no":"YT1","status":"..."} , … ] }` | 支持最多 1000 条 |
| `POST /api/v1/trackings` | 上报状态（内部） | `{ "tracking_no":"YT123","status":"IN_TRANSIT","location_id":123,"timestamp":"..." }` | `202 Accepted` | Ingest Service 接口，内部鉴权 |
| `GET /api/v1/alarms/{tracking_no}` | 查询异常 | `GET /api/v1/alarms/YT123` | `{ "alarms":[ {"type":"DELAY","created_at":"..."} ] }` | 客服使用 |

### 3. 幂等设计  

- **上报接口**必须接受 **`event_id`**（UUID）或 **`tracking_no+timestamp`** 组合，后端使用 **唯一键**去重。  
- **查询接口**天然幂等（GET），不产生副作用。  

### 4. 错误码统一（参考 HTTP+自定义）  

| HTTP | Code | 含义 |
|------|------|------|
| 200 | 0 | 成功 |
| 400 | 1001 | 参数错误 |
| 404 | 1002 | 运单不存在 |
| 429 | 1003 | 请求限流 |
| 500 | 2001 | 系统内部错误 |
| 503 | 2002 | 服务不可用（降级） |

### 5. 限流 & 防刷  

- **IP+UserId 维度**：使用 **Sentinel** 或 **RateLimiter**（令牌桶）在 API Gateway 层限流。  
- **批量查询**：每分钟最多 5 次，防止单用户压垮后端。

## ## 第五步：详细组件设计  

### 1. 入口层（API Gateway + CDN）  

- **CDN**：缓存静态资源（JS、CSS、图片），减轻回源压力。  
- **WAF**：防注入、CC 攻击。  
- **API Gateway**（如 Kong、Spring Cloud Gateway）：  
  - **路由**：`/api/v1/tracking/**` → Query Service  
  - **统一鉴权**（JWT / OAuth2）  
  - **灰度发布**：基于 Header/Version 切流到新版本实例  

### 2. Query Service  

- **技术栈**：Spring Boot + Spring WebFlux（异步非阻塞）或 Go + Gin。  
- **线程模型**：使用 **Reactor**（NIO）或 **goroutine**，保证高并发下 CPU 利用率。  
- **调用链**：  
  1. **先查 Redis**（MGET 支持批量）。  
  2. **Miss** → **异步查询 MySQL**（单条），**并回写 Redis**（SETEX）。  
  3. **历史轨迹** → **ClickHouse JDBC**（列式查询），分页返回。  

#### 缓存失效策略  

| 场景 | 失效方式 | 目的 |
|------|----------|------|
| 正常更新 | **写后删**（Cache‑Aside） | 写完 MySQL、ClickHouse 后 `DEL tracking_latest:{no}`，下一请求自动重新加载最新值。 |
| 关键异常 | **主动预热**（Cache‑Warm） | 当检测到 `DELAY`、`DAMAGED` 等异常，立即写入 Redis 并设置 **短 TTL（30 s）**，确保用户快速感知。 |
| 大批量失效 | **布隆过滤 + 批量删除** | 防止 **缓存击穿**：当大量热点运单在同一时间失效，使用布隆过滤判断是否需要提前预热或降级到 DB。 |

### 3. Ingest Service  

- **入口**：HTTP POST（内部） + **Kafka Producer**。  
- **可靠性**：使用 **幂等 Producer**（Kafka `enable.idempotence=true`）防止重复写入。  
- **流控**：对外提供 **限流**，内部使用 **背压**（Reactor）防止瞬时流量冲垮。  

#### 消费者（落库）  

| 组件 | 负责 | 关键点 |
|------|------|--------|
| `TrackingEventConsumer` | 从 Kafka 拉取事件，写入 MySQL、ClickHouse、更新 Redis | 使用 **事务**（MySQL）+ **批处理**（ClickHouse）提升吞吐；异常重试（指数退避） |
| `AlarmDetector` | 监听同一 Topic，检测异常规则（时效、状态跳变） | 基于 **CEP**（Flink）或 **Kafka Streams** 实时计算，生成 `alarm_event` 并推送到消息队列 `alarm_notify` |

### 4. Alarm Service  

- **消费**：`alarm_notify` Topic → **短信服务**（阿里云 / 腾讯云） + **推送服务**（JPush、微信模板消息）。  
- **降级**：  
  - **优先级**：`DAMAGED` > `DELAY` > `MISROUTED`。  
  - **渠道顺序**：APP Push → 短信 → 邮件。  
  - **降级策略**：若短信接口返回 `429` 或超时，自动改为 **邮件**（或仅记录，后续批量发送）。  

### 5. 存储层细化  

#### MySQL（事务写入）  

- **主从复制**：Primary 写，两个 Read Replica 提供查询（热点读取可走 Replica）。  
- **分区**：`tracking_event` 按 `date(timestamp)` 分区，降低单表大小。  
- **连接池**：HikariCP，最大连接数根据机器 CPU × 2。  

#### ClickHouse（历史）  

- **分区**：`toYYYYMMDD(timestamp)`，**TTL** 90 天后自动删除。  
- **压缩**：`ZSTD`，节省磁盘空间。  
- **并行导入**：使用 `INSERT INTO table VALUES` 批量写入，每批 10k 条。  

#### Redis（缓存）  

- **Cluster**：至少 3 主 3 从，支持水平扩容。  
- **数据结构**：`HASH`（`tracking_latest:{no}`）存放最新状态；使用 **TTL 5min** 防止陈旧。  
- **热点防击穿**：使用 **Lua 脚本** 实现 **单flight**（同一键并发只查询一次 DB）。  

### 6. 监控与告警  

| 维度 | 关键指标 | 收集方式 |
|------|----------|----------|
| **业务** | QPS、成功率、异常告警推送成功率 | Prometheus + Exporter（Spring/Go） |
| **系统** | CPU、内存、磁盘 I/O、网络流量 | Node Exporter |
| **缓存** | 命中率、慢查询、淘汰率 | Redis Exporter |
| **消息** | Kafka Lag、消费速率、错误率 | Kafka Exporter |
| **可用性** | 服务 Up/Down、API 响应时延 | Alertmanager + PagerDuty |

> **阈值示例**：Redis 命中率 < 80% → 触发缓存预热脚本；Kafka Lag > 10,000 → 报警运维。

## ## 第六步：扩展性与高可用设计  

### 1. 读写分离 & 缓存层  

- **写** → **MySQL Primary** → **Kafka** → **ClickHouse** + **Redis**（最新状态）  
- **读**：  
  - **热点** → Redis（Cache‑Aside）  
  - **普通查询** → MySQL Replica（负载均衡）  
  - **历史查询** → ClickHouse（列式）  

> **好处**：写不受读压制，查询几乎不影响写入性能。  

### 2. 容灾方案  

| 失效点 | 备份/冗余方式 | 自动切换 |
|--------|---------------|----------|
| **API Gateway** | 多可用区部署 + **Anycast** DNS | 由 **LVS**/云负载均衡自动切换 |
| **Query Service** | K8s Deployment 多副本 + Pod Anti‑Affinity | K8s 自动调度、滚动升级 |
| **Redis** | 主从复制 + Sentinel（自动故障转移） | Sentinel 检测后提升从为主 |
| **MySQL** | 主从双活（MGR）或 **PXC** | 自动故障转移 + 读写分离路由 |
| **Kafka** | 多副本（replication factor=3） | Leader 迁移不影响消费 |
| **ClickHouse** | 多副本分片（ReplicatedMergeTree） | 节点失效自动恢复数据 |

### 3. 灰度发布 & 蓝绿部署  

- **蓝绿**：新版本部署在独立的命名空间，切流 5% 用户进行监控，确认无异常后全量切换。  
- **灰度**：利用 API Gateway 的 **Canary** 插件，按用户属性（地区、设备）分流。  

### 4. 防止缓存击穿 & 雪崩  

1. **热点预热**：每日凌晨通过离线脚本把前一天 **TOP 10k** 运单的最新状态写入 Redis。  
2. **互斥锁（single flight）**：同一键并发查询 DB 时只允许一次 DB 请求，其他请求等待结果（Lua 脚本实现）。  
3. **熔断**：对 MySQL、ClickHouse 设置 **熔断器**（Hystrix/Resilience4j），当后端响应慢时直接返回 **缓存或降级数据**（如“数据暂不可用，请稍后再试”）。  

### 5. 横向扩展  

| 组件 | 扩容方式 |
|------|----------|
| API Gateway | 增加实例 + DNS 轮询 |
| Query Service | K8s HPA（基于 CPU/QPS） |
| Redis | 添加节点 → Rehash |
| MySQL | 分库（region）或使用 **TiDB** 兼容 MySQL 的分布式方案 |
| ClickHouse | 增加节点 → 数据自动分片复制 |
| Kafka | 增加 Partition → 消费者组水平扩容 |

## ## 第七步：常见面试追问与回答  

### 1. **数据采集频率与幂等性**  

> **问**：站点上报可能出现网络抖动、重复上报，如何保证唯一性和幂等？  

**答**：  
1. **业务端生成唯一 `event_id`（UUID）**，或使用 **`tracking_no+timestamp`** 组合。  
2. **Kafka Producer 开启幂等**（`enable.idempotence=true`），Broker 会基于 **Producer ID + Sequence** 防止重复写入。  
3. **消费落库**时使用 **`INSERT ... ON DUPLICATE KEY UPDATE`**（MySQL）或 **`UPSERT`**（ClickHouse）确保同一唯一键只写一次。  
4. **幂等校验**：在 MySQL 表上建立 **唯一索引** (`tracking_no`, `event_time`)。若重复写入会直接被抛弃或更新同样数据。  

> **不这样**：会导致历史轨迹出现重复节点，用户在时间轴上看到“同一时间出现两条记录”，异常检测也会误报。  

### 2. **缓存失效策略**  

> **问**：高并发查询时，如何防止缓存失效导致 DB 瞬时压垮？  

**答**：  
- **Cache‑Aside + 互斥锁**：查询时先尝试 `GET`，若 `MISS`，使用 **Redis SETNX** 或 **Lua** 实现 “单 flight”。只有第一个请求真正去 DB，其他请求等待结果或直接返回旧值。  
- **热点预热**：使用离线作业（Spark/Flink）每日计算热点运单，提前写入 Redis。  
- **分布式限流**：在 API Gateway 对同一 `tracking_no` 的并发请求做 **令牌桶** 限流，防止瞬时并发穿透。  
- **降级**：当 Redis 故障或命中率低于阈值（如 <70%），自动切换到 **MySQL Replica**，并在后台启动 **缓存恢复**（批量加载热点）。  

### 3. **异常告警推送路径**  

> **问**：不同异常类型的多渠道推送如何设计优先级和降级？  

**答**：  
| 异常类型 | 渠道顺序 | 降级策略 |
|----------|----------|----------|
| **破损/错投**（高危） | APP Push → 短信 → 邮件 | 若 Push 失败（FCM 返回 5xx） → 直接发送短信；短信失败 → 邮件；若全部失败 → 记录到 `alarm_event`，后续批量告警 |
| **时效滞留**（中危） | 短信 → APP Push → 邮件 | 首选短信（即时），若短信限流 → 使用 Push；Push 受限 → 邮件 |
| **轻微异常**（低危） | 邮件 → Push | 只要邮件成功即结束；若邮件系统故障，改为 Push |  

- **实现**：Alarm Service 接收 `alarm_event`，根据 `alarm_type` 读取 **配置表**（`alarm_channel_priority`），按顺序调用对应推送 SDK，**每一步记录返回码**，失败则进入下一步。  
- **幂等**：每条告警在 24 小时内只能推送一次（使用 `tracking_no+alarm_type` 唯一键），防止重复通知。  

### 4. **为何选择 ClickHouse 而不是仅 MySQL**  

- **查询模式**：历史轨迹是 **大范围、时间范围扫描**，MySQL 在 90 天、数十亿行的情况下会出现全表扫描、慢查询。  
- **列式存储优势**：ClickHouse 只读取查询涉及的列（status、timestamp、location），I/O 大幅下降。  
- **压缩率高**：相同数据量磁盘占用约 1/5，降低成本。  
- **实时近线**：使用 **Kafka → ClickHouse** 的 **异步批写**，延迟在秒级，满足 “5 s 可见”。  

### 5. **如果业务突增到 100 k QPS，如何扩容**  

- **水平扩容 Query Service**：开启 **K8s HPA**，依据 **CPU+QPS** 自动扩容到更多 Pod。  
- **Redis Cluster**：增加节点至 9 主 9 从，使用 **分片**（hash slot）降低单节点热点。  
- **MySQL Read Replica**：增加复制实例至 5~10 台，读请求通过 **ProxySQL** 按业务标签路由。  
- **ClickHouse**：添加节点形成 **分布式表**，每个节点负责一部分分区，查询自动并行。  
- **Kafka Partition**：将 `tracking_events` 主题的 **partition** 数提升到 48~96，消费者组相应扩容，提升并发消费能力。

## ## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策  

| 决策 | 思考过程 | 关键点 |
|------|----------|--------|
| **读写分离 + 缓存层** | 初始想直接把所有查询落到 MySQL，结果发现 30 k QPS 对单库读压会致命。于是分析了业务的 **读热点**（最新状态）和 **历史查询**（大批量扫描），决定采用 **Redis** 缓存最新状态、**MySQL Replica** 处理普通查询、**ClickHouse** 负责历史轨迹。 | 通过 **QPS 计算**、**数据访问模式**（热点 vs 冷）决定存储层级，确保查询延迟 ≤200 ms。 |
| **幂等与唯一键设计** | 状态上报频繁且可能因网络抖动重复，需要防止 **重复写入**、**脏数据**。深入分析了业务唯一标识（运单号 + 上报时间），决定在 MySQL 上建 **唯一索引**，并在 Kafka Producer 开启 **幂等**，消费端使用 **UPSERT**。 | 关键是把 **幂等性**前移到 **生产者**，再在 **落库**层面做唯一键校验，确保端到端无重复。 |

### 2. 新手最容易犯的错误（至少 2 条）  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把所有数据都放在单一关系型库** | 读写竞争、扩展受限、查询慢。 | 按业务划分 **热/冷**，使用 **缓存、列式存储**，实现读写分离。 |
| **忽视幂等/去重** | 重复上报导致轨迹重复、异常误报。 | 业务层生成唯一 `event_id`，数据库建唯一键，Kafka Producer 开幂等。 |
| **只考虑单机性能** | 没有考虑容灾、故障转移，系统一旦节点挂掉全挂。 | 引入 **多可用区、主从复制、自动故障转移**（Redis Sentinel、MySQL MGR、Kafka ISR）。 |
| **缓存失效不做防护** | 高并发瞬时失效导致 DB 雪崩。 | 实现 **单 flight、热点预热、熔断**，并监控缓存命中率。 |
