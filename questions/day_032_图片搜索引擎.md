# 第 32 天：设计 图片搜索引擎

> 生成日期：2026-04-24

---

## 题目背景  
图片搜索引擎是一套能够基于用户提交的查询（关键词、图片或组合）返回相关图片结果的系统。它广泛用于电商、社交平台、内容创作等场景，帮助用户快速定位所需视觉素材。

## 面试场景设定  
> **面试官**：  
> “我们现在要设计一个全球化的图片搜索引擎，核心目标是支持 **海量图片的实时检索**，并在 **千毫秒级** 给出相关度较高的结果。请你从零开始构思系统的整体架构，并说明关键技术选型和容量规划。”  

（面试官随后会让你逐步展开设计、讨论细节。）

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **关键词搜索**：用户输入文字关键词，系统返回匹配图片列表，支持多词、布尔、短语搜索。 |
| 2 | **以图搜图**：用户上传或提供图片 URL，系统返回相似/相同图片（基于视觉特征）。 |
| 3 | **过滤与排序**：支持按上传时间、图片来源、分辨率、版权类型等维度过滤，并可按相关度、热度、最新等排序。 |
| 4 | **分页/无限滚动**：返回分页结果，每页默认 20 条，支持翻页或滚动加载。 |
| 5 | **图片预览 & 元数据**：每条结果展示缩略图、标题、来源、标签、版权信息等。 |
| 6 | **搜索日志 & 热词**：记录用户搜索行为，实时统计热词，用于推荐和业务分析（只读展示）。 |

## 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **日活跃用户 (DAU)** | 5 Million | 假设全球范围的公开搜索入口。 |
| **查询 QPS** | 10 K QPS 峰值 | 峰值期间约 1 M QPS/天，平均 5 K QPS。 |
| **单次查询延迟** | ≤ 800 ms（99th percentile） | 包括网络、后端检索、排序和返回。 |
| **系统可用性** | ≥ 99.9% 年可用率 | 单点故障必须通过冗余消除。 |
| **存储容量** | 200 PB 原始图片 + 10 PB 索引/特征向量 | 预计每张图片平均 5 MB，图片总量 40 Billion。 |
| **数据更新延迟** | ≤ 5 min 见新上传图片 | 新图片在上传后 5 分钟内可被搜索到。 |

## 系统边界  

**本题范围内需要设计的部分**  
- 图片的 **上传、存储、压缩、分发**（CDN）流程（不需要实现编辑功能）。  
- **全文检索**（基于倒排索引）和 **向量检索**（基于 ANN）两套索引的构建、更新与查询。  
- **查询路由、负载均衡、缓存层**（热点关键词/特征向量缓存）。  
- **结果排序、过滤、分页**的业务逻辑。  
- **监控、日志、灰度发布**的基本方案。  

**本题范围外（可不考虑）**  
- 用户登录、权限管理、付费/版权付费结算。  
- 图片内容审核（违规图片检测）。  
- 客户端 UI/前端实现细节。  
- 跨语言多语言检索（仅中文/英文关键词）。  
- 私有化部署或离线批量检索。

## 提示与追问  

1. **索引规模与更新**：  
   - “如果每天新增 500 Million 张图片，如何保证倒排索引和特征向量索引的实时更新且不影响查询性能？”  

2. **向量检索的实现**：  
   - “在 200 PB 规模的特征向量库中，你会选哪种 ANN 算法/框架？如何在多机房部署保证低延迟？”  

3. **缓存与热点处理**：  
   - “在高并发热点关键词（如‘猫’）的情况下，如何设计缓存层和热点分片，防止热点导致的热点节点失效？”

# 图片搜索引擎系统设计完整解答  

> **温馨提示**：下面的每一步都尽量从「为什么」出发，帮助没有系统设计经验的同学建立完整的思考框架。即使你以后不直接负责某个细节，只要懂得 **需求 → 容量 → 架构 → 细化** 的闭环，就能在面试中自信地回答。

## ## 解题思路总览  

1. **需求拆解** → 明确功能点、非功能指标、边界。  
2. **容量估算** → 通过 DAU、QPS、图片规模等算出 **流量、存储、计算** 需求。  
3. **从 MVP（最小可用产品）起步** → 只实现核心搜索流程，保证 **单机/小集群** 能跑通。  
4. **逐层演进** → 加入 **分布式、缓存、索引分片、容灾、监控**，一步步满足 **高可用、低延迟、可扩展**。  
5. **技术选型** → 根据业务特点（文本检索 vs 向量检索、实时更新、海量数据）挑选成熟开源或商业方案。  
6. **细化每个子系统** → 上传、存储、CDN、倒排索引、向量索引、查询路由、结果合并、缓存、日志等。  
7. **常见追问准备** → 索引实时更新、热点缓存、跨机房 ANN、灰度发布等。  

下面我们按章节展开。

## ## 第一步：理解需求与规模估算  

### 1️⃣ 功能性需求要点  

| 编号 | 关键点 | 需要的系统能力 |
|------|--------|----------------|
| 1 | 关键词搜索（布尔、短语） | **全文倒排索引**，支持分词、同义词、过滤 |
| 2 | 以图搜图 | **向量检索**（ANN）+ **特征抽取** |
| 3 | 过滤/排序 | 多维属性过滤、二次排序（热度、时间、相关度） |
| 4 | 分页 | **游标/深度分页**，防止热点页面 |
| 5 | 预览 & 元数据 | 缩略图 CDN、元数据查询 |
| 6 | 搜索日志 & 热词 | 实时埋点、流式聚合（Kafka → Flink → Redis） |

### 2️⃣ 非功能性需求拆解  

| 指标 | 目标 | 隐含设计约束 |
|------|------|--------------|
| DAU 5M | 需要支撑全球用户并发 | **多地域部署**、**负载均衡** |
| QPS 峰值 10K | 100 ms 内完成路由、检索、排序 | **水平扩展**、**热点缓存** |
| 延迟 ≤800 ms (p99) | 包括网络、检索、排序 | **缓存**、**索引分片**、**并行查询** |
| 可用性 ≥99.9% | 年故障时间 <8.76 h | **冗余**、**故障转移** |
| 存储 200 PB 原始 + 10 PB 索引 | 超大规模持久化 | **对象存储 + 分布式文件系统**、**冷热分层** |
| 更新延迟 ≤5 min | 新图 5 min 可检索 | **近实时索引管道**、**增量写入** |

### 3️⃣ 规模粗算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **日请求量** | 10 K QPS * 86 400 s ≈ 864 M 次/天 |
| **峰值并发请求** | 10 K QPS × 平均响应时间 0.5 s ≈ 5 000 并发 |
| **每日新增图片** | 500 M 张 |
| **总图片数** | 40 B 张 (5 y) |
| **原始存储** | 40 B × 5 MB ≈ 200 PB |
| **特征向量大小** | 128 float (4 B) ≈ 0.5 KB/图 → 20 TB (全量) < 10 PB 预留空间（备份、索引结构） |
| **倒排索引大小** | 词项数 10 M，平均文档列表 10 K → 约 100 TB（压缩后） |
| **查询流量** | 假设每查询返回 20 条 → 20 × 864 M ≈ 17 B 条返回记录/天 |

> **为什么要算这些？**  
> 1. 明确 **CPU、内存、网络** 需求，防止后期踩坑。  
> 2. 估算 **索引分片数**、**机器规格**、**带宽**。  
> 3. 为 **成本评估**（云资源、存储费用）提供依据。

## ## 第二步：高层架构设计  

### 1️⃣ MVP（最小可用产品）结构  

```
[Client] → [API GW] → [Search Service] → {Text Index, Vector Index}
                     ↘               ↙
                [Metadata DB]   [Object Store (CDN)]
```

- **API GW**：统一入口，做限流、鉴权（若有）、日志。  
- **Search Service**：单机或小集群，实现关键词解析、向量召回、过滤、排序。  
- **Text Index**：Elasticsearch 单节点或小集群。  
- **Vector Index**：FAISS / Milvus 单机版。  
- **Metadata DB**：MySQL / PostgreSQL 存放图片元信息。  
- **Object Store**：S3/OSS 存原图、缩略图，配合 CDN。  

> **不这样做的后果**：  
> - 直接把查询请求打到 ES/FAISS 上，缺少统一限流和业务日志，易导致 **雪崩**。  
> - 元数据和图片混在一起，导致 **存储耦合**，难以做冷热分层。  

### 2️⃣ 逐步演进到完整分布式系统  

下面的架构图（文字版）展示了所有必备组件及其交互关系。  

```
+-------------------+      +-------------------+      +-------------------+
|   客户端 (Web/APP) | ---> |   API Gateway     | ---> |   统一流量层 (LB) |
+-------------------+      +-------------------+      +-------------------+
                                            |
                      +-------------------+-------------------+
                      |                                        |
          +-----------v-----------+                +-----------v-----------+
          |   查询路由/调度层      |                |   写入/上传调度层      |
          +-----------+-----------+                +-----------+-----------+
                      |                                        |
      +---------------+---------------+          +-------------+-------------+
      |                               |          |                           |
+-----v-----+                   +-----v-----+ +--v---+                 +------v------+
| Text Index|  (Elasticsearch) | VectorIdx| (Milvus) | Cache| (Redis) |  上传服务   |
|  Cluster  |                   | Cluster  |          | (热点) | (对象存储) |
+-----------+                   +----------+          +--------+------------+
      |                               |                     |
      |   +---------------------------+---------------------+------------------+
      |   |                                                               |
+-----v---v-----+                                               +----------v----------+
| 元数据服务    | (MySQL / TiDB)                                 |   对象存储 (OSS)    |
| (图片信息、标签) |                                               | (原图+缩略图)      |
+---------------+                                               +--------------------+
      |                                                               |
      +-----------------------+---------------------------+-----------+
                              |                           |
                         +----v----+                 +----v----+
                         | 监控&日志|                 | 灰度发布 |
                         +----------+                 +----------+
```

#### 关键模块说明  

| 模块 | 主要职责 | 选型建议 | 为什么这样选 |
|------|----------|----------|--------------|
| **API Gateway** | 鉴权、限流、统一入口、日志收集 | Kong / APIGEE / Envoy | 轻量、插件化、支持灰度 |
| **统一流量层（LB）** | 负载均衡、跨地域路由 | LVS + Anycast / Cloud LB | 低延迟、全局流量分配 |
| **查询路由/调度层** | 将请求分发到对应的 **文本** 与 **向量** 索引集群，合并结果 | 自研基于 RPC (gRPC) 的调度服务 | 可灵活控制并行度、超时、容错 |
| **文本倒排索引** | 关键词检索、过滤 | Elasticsearch (ES) 8.x | 成熟、支持多字段、分词、聚合 |
| **向量检索** | 高维相似度搜索 | Milvus 2.x (基于 IVF‑PQ) 或 Faiss + DiskANN | 支持 PB 级向量、GPU 加速、水平扩展 |
| **缓存层** | 热点关键词/向量、查询结果缓存 | Redis Cluster + LRU | 读热点压在磁盘/索引上，降低延迟 |
| **元数据服务** | 业务属性、标签、版权信息 | TiDB / MySQL 8（水平扩展） | 强一致性、二级索引、事务 |
| **对象存储 + CDN** | 原图、缩略图、压缩、分发 | 阿里云 OSS / AWS S3 + CDN (CloudFront) | 海量存储、自动冷热分层、全球加速 |
| **监控&日志** | 业务/系统指标、告警、追踪 | Prometheus + Grafana + Loki + Jaeger | 可观测性、快速定位故障 |
| **灰度发布** | 新特性/新索引模型平滑上线 | Spinnaker / Argo Rollout | 零风险发布、回滚 |

> **不这样做的风险**  
> - 直接把查询请求打到 ES/Vector 集群，缺少 **统一的限流、流量治理**，在热点时会导致 **服务雪崩**。  
> - 把缓存放在查询服务内部，难以横向扩展，热点节点容易 **OOM**。  
> - 没有监控/灰度，故障定位困难，线上改动风险大。

## ## 第三步：数据库设计  

### 1️⃣ 元数据表（关系型）  

| 表名 | 说明 | 主键 | 关键字段 | 索引 |
|------|------|------|----------|------|
| `image` | 图片基本信息 | `image_id` (BIGINT) | `url`, `status`, `upload_time`, `size`, `format` | PK, `status` 索引 |
| `image_meta` | 业务属性（标题、描述、版权） | `image_id` | `title`, `description`, `copyright` | PK |
| `image_tag` | 多对多标签 | (`image_id`, `tag_id`) | `tag_id` | 联合索引 |
| `tag` | 标签字典 | `tag_id` | `name`, `type` | PK, `name` 唯一索引 |
| `image_source` | 图片来源渠道 | `source_id` | `name`, `type` | PK |
| `image_stats` | 热度计数（点击、曝光） | `image_id` | `click_cnt`, `view_cnt`, `like_cnt` | PK |

> **为什么使用关系型 DB？**  
> - **强一致性**：搜索结果需要准确的版权、过滤属性。  
> - **多维过滤**：SQL 的二级索引在过滤、排序上更灵活。  
> - **事务需求**：上传时需要原子写入多表（元信息、标签、统计）。  

> **不使用关系型的后果**：  
> - 采用 NoSQL（如 DynamoDB）虽然扩展性好，但在 **复杂过滤、联合排序** 场景会变得笨拙，需要二次查询或在业务层做过滤，导致 **延迟** 增大。  

### 2️⃣ 倒排索引（ES）结构  

| 索引 | 文档结构（示例） |
|------|-------------------|
| `image_text` | ```json { "image_id": 12345, "title": "Cute cat", "description": "...", "tags": ["cat","animal"], "ocr_text": "...", "upload_time": "2026-05-20T12:34:56Z", "source": "user", "resolution": "1920x1080", "license": "CC0" } ``` |

- **分词器**：ik_max_word（中文）+ standard（英文）+ synonym_filter。  
- **字段类型**：`keyword`（标签、license）、`text`（title、description、ocr_text）、`date`（upload_time）、`long`（resolution_x, resolution_y）。  
- **副本数**：默认 1（跨 AZ），保证 99.9% 可用。  

### 3️⃣ 向量索引（Milvus）模型  

| 参数 | 说明 |
|------|------|
| `image_id` | 主键（与关系库保持同步） |
| `vector` | 128‑dim float（采用 ResNet‑50 / EfficientNet‑B0 提取） |
| `metadata` | JSON（可选：source、upload_time） |
| `index_type` | IVF‑PQ（nlist=4096, M=16） |
| `metric_type` | L2 或 IP（取决于特征归一化） |
| `partition` | 按 `upload_date`（每日一个分区）或 `region`（跨机房）划分，便于 **并行增量** 与 **冷热分层**。 |

> **为什么选 IVF‑PQ**：  
> - **压缩率高**（PQ 让向量压到 0.5 KB），适合 **PB 级** 数据。  
> - 支持 **磁盘索引**，可以在内存不足的情况下仍保持 **ms 级** 检索。  
> - **GPU 加速** 的训练/重建可在离线批处理时使用。  

> **不采用 IVF‑PQ 的后果**：  
> - 使用 HNSW 虽然召回率更高，但 **内存消耗** 按 **O(N×logN)** 线性增长，难以容纳 10 PB 向量。

## ## 第四步：核心 API 设计  

### 1️⃣ 搜索 API（RESTful + gRPC）  

| 方法 | 路径 | 参数 | 示例（JSON） |
|------|------|------|--------------|

```json
{
  "total": 124567,
  "page": 1,
  "size": 20,
  "hits": [
    {
      "image_id": "12345",
      "title": "Cute cat",

    ...
  ],

```

### 2️⃣ 上传 API  

| 方法 | 路径 | 参数 | 示例 |
|------|------|------|------|
| `POST` | `/api/v1/images` | `file` (multipart), `title`, `tags`, `license`, `source` | `multipart/form-data` |

- **处理流程**：API → **写入调度层** → **对象存储** (原图) → **特征抽取服务** → **向量索引写入** + **倒排索引写入** → **元数据库写入**。

- **Kafka → Flink** 实时聚合，写入 **Redis Sorted Set** `hot_words:{region}`。  
- 前端可直接查询 `GET /api/v1/hotwords?region=global`。

## ## 第五步：详细组件设计  

### 1️⃣ 上传与特征抽取流水线  

1. **用户上传** → **API Gateway** → **Upload Service**（写入调度层）。  
2. **对象存储**（分片）保存原图，返回 **ObjectKey**。  
3. **异步任务**（Kafka）将 `ObjectKey` 发送至 **Feature Extraction Service**。  
4. **特征抽取**  
   - 使用 **GPU 集群**（TensorRT、ONNX Runtime）运行 **ResNet‑50** 或 **ViT**。  
   - 输出 **128‑dim float**，写入 **Milvus**（向量索引）和 **ES**（ocr_text）。  
5. **元数据写入**：MySQL 事务写入 `image`、`image_meta`、`image_tag`。  
6. **索引同步**：Milvus/ES 提供 **增量写入 API**，在 5 分钟内完成 **刷新**（`refresh_interval=1s`）。  

> **为什么使用异步 pipeline**？  
> - 上传过程对用户要 **低延迟**（几百 ms），特征抽取是 **耗时**（秒级），分离后不阻塞前端。  
> - 通过 **Kafka** 能够 **水平扩展**，且天然支持 **重试** 与 **幂等**。  

### 2️⃣ 检索流程  

```
[Client] → API GW → Query Router
   │
   ├─► Text Search (ES)  ← query + filters
   │
   ├─► Vector Search (Milvus) ← image vector (if any) OR query vector (if textual query converted)
   │
   └─► Result Merger
        ├─ 合并两套召回（取 topK 合并 + 重排）
        ├─ 二次过滤（属性、时间范围）
        ├─ 排序（相关度 + 热度）
        └─ 分页 & 缓存写回 Redis
```

#### 关键细节  

| 步骤 | 关键技术 | 说明 |
|------|----------|------|
| **查询路由** | gRPC + Consistent Hash | 根据 `region`、`query hash` 把请求分发到对应 ES / Milvus 分片，避免跨机房网络。 |
| **文本召回** | ES `bool` + `function_score` | 支持布尔、短语、同义词，使用 **BM25** 计算初始相关度。 |
| **向量召回** | Milvus `search`（topK=100） | 采用 **IVF‑PQ**，返回 **距离**（L2）和 `image_id`。 |
| **结果融合** | 线性融合 `score = α * text_score + β * (1 - distance_norm)` | α、β 通过 **AB Test** 调优，保证视觉相似度与关键词相关度兼顾。 |
| **二次过滤** | 过滤器在 **MySQL** 中执行，或在 **ES**/Milvus**返回前做** | 大多数过滤属性（license、resolution）已经在索引中存储为 **keyword**，直接在检索阶段过滤，减少数据搬运。 |
| **排序** | 先相关度，再 **热度**（click_cnt） | 使用 **Redis ZSET** 缓存热度，实时更新。 |
| **分页** | **深度分页**：`search_after`（ES）+ `offset`（Milvus）| 避免 `from+size` 导致全表扫描，保证大页性能。 |
| **缓存** | **热点查询**：`query_key = md5(query+filters)` → Redis TTL 5 min | 只缓存 **结果 ID 列表**，实际图片 URL 仍走 CDN。 |
| **监控** | **Trace ID** 贯穿整个链路，记录每阶段耗时 | 通过 **Jaeger** 追踪慢查询，定位瓶颈。 |

### 3️⃣ 索引实时更新方案  

| 场景 | 方案 |
|------|------|
| **新增图片** | 写入后立即 **push** 到 ES `/_bulk`（`refresh_interval=1s`）和 Milvus `insert`，保证 ≤5 min 可搜索。 |
| **标签/属性变更** | 使用 **Change Data Capture (CDC)**（Canal、Debezium）捕获 MySQL 变更，推送到 **Kafka** → **ES 更新脚本**（`update_by_query`）和 **Milvus**（metadata 更新）。 |
| **删除/下线** | 逻辑删除 `status=deleted`，并在 **ES**、**Milvus** 中做 **soft delete**（过滤），异步执行 **硬删除**（压缩）任务。 |
| **索引重建** | 周期性 **全量快照** → 新集群 **滚动切换**（蓝绿部署），不影响线上查询。 |

> **为什么采用 **增量写入 + CDC**？**  
> - **低延迟**：新增图片可在几秒内写入索引。  
> - **一致性**：CDC 保证 **元数据** 与 **倒排/向量索引** 同步。  
> - **可回滚**：增量日志可在异常时回放，避免全量重建。  

### 4️⃣ 缓存与热点处理  

1. **查询结果缓存**（Redis Cluster）  
   - **Key**：`search:{region}:{query_hash}:{page}`  
   - **Value**：JSON 列表（image_id + score）  
   - **TTL**：5 min（热点查询可手动延长）  

2. **热点热点分片**（热点路由）  
   - 使用 **Consistent Hash Ring**，为热点 `query_hash` 预留 **专属节点**（虚拟节点数加倍）。  
   - 这些节点配备 **更大内存** 与 **更高带宽**，防止单点 OOM。  

3. **向量缓存**（GPU‑enabled Redis / Faiss‑GPU）  
   - 对 **前 1000 条热门向量**（如 “猫”）预先加载到 **GPU 内存**，在 Milvus 前做 **二级缓存**。  

> **如果不做热点缓存**：  
> - 大量相同查询会直接打到 **ES/Milvus**，导致 **CPU/IO 峰值**，出现 **latency spike**，进而影响 99th‑percentile SLA。  

### 5️⃣ 监控、日志、灰度发布  

| 维度 | 监控指标 | 工具 |
|------|----------|------|
| **系统** | CPU、内存、磁盘 I/O、网络带宽 | Prometheus + node_exporter |
| **服务** | QPS、成功率、平均/最大延迟、错误码分布 | Prometheus + Grafana |
| **业务** | 搜索成功率、热词分布、点击/曝光比 | ClickHouse + Grafana |
| **链路** | Trace ID、每阶段耗时 | Jaeger (OpenTelemetry) |
| **日志** | 请求/响应体（脱敏）、异常栈 | Loki + Fluent Bit |
| **灰度** | 版本占比、回滚阈值 | Argo Rollout + Flagger |

- **灰度发布**：使用 **Canary**（10% 流量） → **AB Test** 监控关键指标 → 自动提升至 100%。  
- **告警**：延迟 > 1 s、错误率 > 0.5% → PagerDuty。

## ## 第六步：扩展性与高可用设计  

### 1️⃣ 横向扩展（Scaling Out）  

| 组件 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **API GW** | 添加实例 + LVS/Anycast | CPU、请求数 |
| **查询路由** | Stateless，水平扩容 | QPS |
| **ES** | **分片**（primary + replica）<br>→ 按 `region`、`upload_date` 进行 **shard routing** | 每 shard 30 GB (默认) |
| **Milvus** | **Partition** + **Replica**<br>→ 多节点 **Distributed IVF‑PQ** | 每节点 1 TB 向量 |
| **Redis** | **Cluster**（hash slots）<br>→ 添加节点再分配 slots | 内存、热点命中率 |
| **MySQL/TiDB** | **读写分离** + **水平分片**（range/sharding） | QPS、事务数 |
| **对象存储** | 自动扩容（对象层） | PB 级存储 |

### 2️⃣ 多地域部署（跨机房）  

- **Region A (美洲)**、**Region B (亚洲)**、**Region C (欧洲)**  
- **全局流量层**使用 **Anycast IP** + **Geo‑DNS**（如 Cloudflare）将用户请求路由到最近的 Region。  
- **数据同步**  
  - **元数据**：采用 **TiDB** 多活复制（Raft）或 **MySQL Group Replication**，保证强一致性。  
  - **倒排索引**：每 Region 独立 **ES 集群**，通过 **跨集群复制**（跨站点快照）实现 **只读备份**。  
  - **向量索引**：每 Region 部署 **Milvus**，每日 **增量同步**（通过 Kafka + Milvus Replicator）保证 **全局一致**。  
- **故障切换**：若某 Region 故障，**DNS TTL** 短（30 s）让流量自动切换到其他 Region。  

> **不做多地域**：  
> - 跨洲用户网络 RTT 可能超过 200 ms，整体延迟难以满足 800 ms SLA。  

### 3️⃣ 容灾与备份  

| 目标 | 实现方式 |
|------|----------|
| **单点故障** | 所有服务 **双活**（至少 2 台机器）<br>→ 负载均衡+健康检查 |
| **数据持久化** | 对象存储 **多 AZ** 复制（3×）<br>MySQL/TSDB **binlog + X‑Cluster** |
| **索引灾备** | ES **快照** → OSS<br>Milvus **备份**（Milvus Backup） |
| **故障恢复** | 采用 **灾备演练**（每周）<br>自动化脚本：恢复 ES 快照 → 重建 Milvus 分区 |  

### 4️⃣ 性能调优要点  

1. **查询并行**：同时向 ES 与 Milvus 发起 RPC，采用 **异步 Future**，在两者返回后再合并，降低整体延迟。  
2. **向量压缩**：使用 **PQ** + **OPQ**，把向量压到 0.5 KB，减少磁盘 IO。  
3. **热点分片**：对热门关键词（如 “cat”）的倒排列表做 **复制** 到 **热点节点**，并开启 **高速缓存**。  
4. **预热**：每日凌晨对 **昨日热点** 做 **warm‑up**（查询一次），让缓存命中率提升。  
5. **限流**：对单 IP / 单用户 QPS 进行限流，防止 **DDOS** 导致后端崩溃。

## ## 第七步：常见面试追问与回答  

### Q1️⃣ **“如果每天新增 500 Million 张图片，如何保证倒排索引和特征向量索引的实时更新且不影响查询性能？”**  

**回答要点**  

1. **增量写入 + 近实时刷新**  
   - **ES**：`refresh_interval=1s`，使用 **bulk API**（每批 5k 文档）写入；写入线程池独立，不阻塞查询线程。  
   - **Milvus**：开启 **insert** 接口的 **segment flush** 策略，`flush_interval_seconds=30`，并在 **insert** 完成后立即 **load`** 到内存（或 SSD）进行 **实时查询**。  

2. **写入分流**  
   - **上传调度层**使用 **Kafka** 分区（按 `region`）将任务分配到对应的 **索引写入服务**，实现 **写入水平扩展**。  

3. **查询与写入分离**  
   - **查询路由**只访问 **已刷新** 的 **segment**，新写入的 segment 通过 **shadow index** 先不参与查询，待 `refresh` 完成后自动加入。  

4. **热点回补**  
   - 对 **热点关键词**（如 “cat”）采用 **双写**：在主索引写入的同时，写入 **热点缓存节点**（Redis）或 **副本索引**，确保热点查询不受写入冲击。  

5. **资源隔离**  
   - 为 **写入** 配置专用 **CPU/IO**（如使用 **cgroup**），防止写入占满查询节点的资源。  

**补充**：若写入速率接近查询速率，可考虑 **微批**（如 5 s）进行 **合并**，或 **分层索引**（冷热分层），把 **当天** 的数据放在 **热层**（SSD），**历史** 数据转入 **冷层**（HDD）。  

---  

### Q2️⃣ **“在 200 PB 规模的特征向量库中，你会选哪种 ANN 算法/框架？如何在多机房部署保证低延迟？”**  

**答案要点**  

1. **算法选型**  
   - **IVF‑PQ (Inverted File + Product Quantization)**：  
     - **优势**：磁盘索引友好、查询时只访问少量倒排列表、内存占用低。  
     - **适合 PB 级**：向量压缩到 ~0.5 KB，10 PB 可装在 **数千台机器** 的 SSD。  
   - **可选补充**：对 **极热点** 向量使用 **HNSW**（在 GPU/内存中）做二级快速召回，提升召回率。  

2. **框架实现**  
   - **Milvus 2.x**（原生支持 IVF‑PQ、DiskANN），已实现 **分布式索引、并行搜索、GPU 加速**。  
   - **FAISS + DiskANN**（如果团队对底层有改造需求），自行搭建 **Shard + Replication**。  

3. **多机房部署**  
   - **分区策略**：按照 **region + upload_date** 划分 **Partition**，每个 Partition 只在对应机房部署。  
   - **跨机房查询**：查询路由根据用户 IP 先定位最近机房，只在本地 **向量库** 检索；若本地召回不足（低 recall），再 **跨机房** 发起 **异步补偿查询**（返回补充结果）。  
   - **同步机制**：使用 **Kafka** + **Milvus Replicator** 将新增向量 **增量同步** 到其他机房，延迟 ≤ 5 min。  

4. **低延迟技巧**  
   - **预加载热点 Partition** 到 **NVMe SSD**，并开启 **GPU 加速**（每台机器配 1~2 块 RTX 4090）。  
   - **并行查询**：在同一查询中对多个 Partition 同时发起搜索（线程池），取 **topK** 合并。  
   - **查询缓存**：把 **热点向量的搜索结果**（ID 列表）缓存到 **Redis**，直接返回。  

---  

### Q3️⃣ **“在高并发热点关键词（如‘猫’）的情况下，如何设计缓存层和热点分片，防止热点导致的热点节点失效？”**  

**答案要点**  

1. **热点分片（Hot Shard）**  
   - **Consistent Hash Ring**：为热点 `query_hash` 添加 **虚拟节点**（如 10 倍），让它们映射到 **多台机器**（至少 3 副本）。  
   - **负载均衡**：在热点查询入口使用 **随机选择** 或 **最少连接** 策略，将请求分散到所有副本。  

2. **多级缓存**  
   - **一级缓存（Redis Cluster）**：存放 **查询结果 ID 列表**，TTL 5 min。  
   - **二级缓存（本地 LRU）**：每查询节点本地内存缓存最近 100 条热点查询的 **完整结果**（包括缩略图 URL），避免每次都访问 Redis。  
   - **向量二级缓存**：热点向量（前 10 k）放在 **GPU 内存**（Faiss‑GPU）或 **RedisAI**，快速召回。  

3. **缓存失效与降级**  
   - **写穿（Write‑Through）**：每当新图片匹配热点关键词时，立即 **推送** 到 **Redis**（使用 **ZSET** 维护分数）。  
   - **缓存雪崩防护**：设置 **不同 TTL**（随机 +/- 10%），防止同一时间大量失效。  
   - **降级**：当热点节点 CPU/内存 超过阈值（如 80%），自动 **关闭热点缓存**，改为直接查询 ES/Milvus，防止节点宕机。  

4. **监控与自动扩容**  
   - 监控 **热点查询 QPS**、**Redis 命中率**、**节点 CPU**。  
   - 当 QPS 超过阈值（如 5 K QPS）且命中率低于 70% 时，**自动扩容** Redis 分片或 **增加热点副本**。

### Q4️⃣ **“如果用户想对搜索结果做实时排序（比如‘按热度’），该怎么实现？”**  

**回答要点**  

1. **热度数据来源**：  
   - 实时 **点击/曝光** 计数写入 **Kafka** → **Flink** → **Redis Sorted Set** `hot:{image_id}`（score = click_cnt * w1 + view_cnt * w2）。  
   - 同时持久化到 **MySQL**（每日汇总），防止数据丢失。  

2. **排序实现**：  
   - **二次排序**：检索得到的 `image_id` 列表后，批量从 **Redis ZSET** 拉取对应 **热度分**（`ZRANGE`），在业务层 **合并**（`score = α * relevance + β * hot_score`）。  
   - **ES Function Score**：可以在倒排查询阶段加入 **脚本评分**，把 `hot_score` 作为 **payload**（通过 `stored_fields`）直接在 ES 中完成合并。  

3. **实时性保障**：  
   - **热度计数**使用 **增量计数**（HyperLogLog / Count-Min Sketch）在 Flink 中聚合，**秒级**刷新到 Redis。  
   - **延迟**控制在 **1 s** 以内，满足“实时”感知。

## ## 心得与反思  

### 1️⃣ 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 为什么难 | 思考路径 |
|------|----------|----------|
| **向量检索在 PB 级数据上的 ANN 选型** | 需要在 **召回率、查询延迟、存储成本、可扩展性** 四者之间找到平衡；传统 HNSW 不能直接放在磁盘，FAISS IVF‑PQ 需要自行实现分布式、容错。 | 1）先列出所有常见 ANN（HNSW、IVF‑PQ、ANNOY、DiskANN）<br>2）对比 **内存占用**（HNSW ≈ O(N×logN)）<br>3）考虑 **磁盘索引**（IVF‑PQ + PQ）<br>4）调研已有成熟分布式实现（Milvus）<br>5）最终选 IVF‑PQ + Milvus，补足热点使用 HNSW + GPU。 |
| **实时索引更新与查询性能的平衡** | 需要 **秒级** 将 500 M 新图写入倒排+向量索引，同时保证查询 QPS 10K 的低延迟。写入与查询是资源争用的“硬仗”。 | 1）明确写入路径：上传 → 对象存储 → 异步特征抽取 → 增量写入<br>2）把写入和查询 **资源隔离**（专用写入节点、cgroup）<br>3）使用 **near‑real‑time refresh**（ES 1 s、Milvus segment flush）<br>4）引入 **冷热分层**（当天数据在热层，老数据冷层）<br>5）评估 **写入批次大小** 与 **刷新频率** 对 QPS 的冲击，做压测调优。 |

### 2️⃣ 新手最容易犯的错误（至少 2 条）  

| 错误 | 结果 | 正确做法 |
|------|------|----------|
| **把搜索服务和业务逻辑直接写在同一个进程里**（没有层次划分） | 难以水平扩容，单点故障导致整体不可用，调试、监控困难。 | **分层**：API Gateway → Query Router → 独立的 Text/Vector 检索服务 → Result Merger。每层职责单一，便于独立扩容。 |
| **只关注倒排索引，忽视向量索引的冷热分层** | 向量库占用大量内存/磁盘，查询时频繁磁盘 IO，导致延迟暴涨。 | **冷热分层**：最近 1‑2 天热点向量放在 SSD+GPU，历史向量压缩到 HDD；使用 **IVF‑PQ** + **DiskANN**，并配合 **分区**、**副本**。 |
| **缓存只放结果，不做热点分片** | 热点查询直接击穿 ES/Milvus，导致节点 CPU/IO 饱和，出现 **缓存雪崩**。 | **热点分片 + 多级缓存**：在一致性哈希环上为热点增加虚拟节点；在 Redis Cluster 上使用 **热点副本**，并设置不同 TTL 防雪崩。 |
| **忽视监控与灰度发布** | 上线新特性时出现回滚困难，故障定位慢。 | 在设计阶段即加入 **Prometheus + Grafana**、**Jaeger**、**Canary/Blue‑Green** 部署流程。 |
