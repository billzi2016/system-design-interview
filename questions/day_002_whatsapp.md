# 第 2 天：设计 WhatsApp

> 生成日期：2026-05-24

---

## 1. 题目背景  
WhatsApp 是一款跨平台的即时通讯应用，主要用于一对一聊天、群聊、语音/视频通话以及多媒体消息的发送。用户通过手机或网页端随时保持联系，要求低延迟、高可靠性。

## 2. 面试场景设定  
**面试官**：  
“我们现在要设计一个能够支撑全球数十亿用户的 WhatsApp 消息系统。请你从零开始，阐述整体架构并重点说明核心的消息投递流程。先从需求入手，随后我们会逐步深入细节。”  

（面试官在候选人阐述完需求后，会继续追问扩展性、容灾、数据一致性等方面。）

## 3. 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **单聊消息**：用户可以发送文字、图片、音频、视频、文档等多媒体消息，接收方需实时收到。 |
| 2 | **群聊**：支持最多 1,000 人的群组，消息需广播给所有成员，成员加入/退出需即时生效。 |
| 3 | **离线消息**：当接收方离线时，系统需持久化未送达消息，用户上线后立即推送。 |
| 4 | **端到端加密**：所有消息在客户端加密，服务器只保存密文，确保隐私。 |
| 5 | **已读回执**：发送方能看到对方已阅读的状态（单聊 & 群聊），并能对已读状态进行撤回。 |
| 6 | **消息撤回**：发送方在 2 分钟内可撤回已发送的文字或媒体，撤回后所有终端同步删除。 |

## 4. 非功能性需求  

| 指标 | 目标值 | 备注 |
|------|--------|------|
| **日活跃用户（DAU）** | 5 亿 | 假设覆盖全球主要市场 |
| **峰值 QPS（消息发送）** | 120 万 QPS | 估算：5 亿 DAU × 2 条/分钟 ≈ 1.7M QPS，取保守峰值 1.2M |
| **端到端消息延迟** | ≤ 150 ms（99.9%） | 从发送端到接收端的感知延迟 |
| **系统可用性** | 99.99%（每月累计宕机 ≤ 4.38 小时） | 包括消息投递、登录等核心功能 |
| **存储容量** | 约 15 EB（年） | 计算：5B 用户 × 100 GB/年（平均消息+媒体） ≈ 15 EB |
| **加密性能** | 加密/解密 CPU 开销 ≤ 5% | 确保端到端加密不成为瓶颈 |

> 注：上述数值为粗略估算，面试中可根据假设进行讨论和微调。

## 5. 系统边界  

**本题范围（考虑）**  
- 消息的发送、路由、存储、加密、离线投递与回执。  
- 用户登录、设备管理（多端同步）以及基础的群组管理。  
- 高可用、水平扩展、灾备、监控告警等核心运维能力。  

**不在本题范围（不考虑）**  
- 短信/邮件备份渠道、验证码服务。  
- 社交功能（状态、朋友圈、表情包市场等）。  
- 跨平台音视频通话的媒体流传输（仅关注信令层）。  
- 客户端 UI 设计与前端细节实现。  
- 法律合规审计（如 GDPR）细节，只需简要提及合规需求。

## 6. 提示与追问  

1. **扩展性**  
   - “如果我们在 6 个月内用户数翻倍，系统的哪些层级需要先做容量规划？”  
2. **一致性与回撤**  
   - “消息撤回需要在所有已收到的设备上同步删除，你会如何保证强一致性而不牺牲性能？”  
3. **灾备与数据恢复**  
   - “在一次跨地区数据中心故障导致部分消息库不可用的情况下，如何保证离线消息不丢失且能够快速恢复？”

# 题解

## 解题思路总览  

> **目标**：帮助没有系统设计经验的同学，像搭积木一样，从最小可用系统（MVP）一步步构建出能够支撑全球数十亿用户的 WhatsApp 消息系统。  
> **思路**：  
> 1. **先读需求** → 把功能需求、非功能需求、业务假设写成可量化的指标。  
> 2. **画最小可用系统** → 只保留「发送‑存储‑投递」三大核心模块，先让系统跑通。  
> 3. **逐层演进** → 按照「水平扩展 → 高可用 → 容灾 → 监控」的顺序，给每一层添加必要的组件和设计细节。  
> 4. **每一步都问自己**：*为什么要加这个组件？不加会出现什么问题？* 通过这种自问自答的方式，帮助新人形成系统化的思考模型。  

下面我们正式开始。

## 第一步：理解需求与规模估算  

### 1. 功能需求拆解  

| 编号 | 关键点 | 需要的系统能力 |
|------|--------|----------------|
| 1 | 单聊多媒体 | **实时投递**、**大文件存储**、**带宽优化** |
| 2 | 群聊（≤1000 人） | **消息广播**、**成员变更即时生效** |
| 3 | 离线消息 | **持久化**、**离线设备缓存**、**上线拉取** |
| 4 | 端到端加密 | **密文存储**、**加密/解密在客户端**、**密钥管理** |
| 5 | 已读回执 | **状态同步**、**回执去重** |
| 6 | 消息撤回（2 min） | **撤回指令广播**、**已读/已送达的消息同步删除** |

> **注意**：功能需求里所有「实时」和「即时」的要求，都转化为 **低延迟** 与 **强一致性**（在可接受的范围内） 的技术目标。

### 2. 非功能需求量化  

| 指标 | 计算过程/假设 | 目标值 |
|------|----------------|--------|
| DAU | 5 × 10⁸ | 5 亿 |
| 峰值发送 QPS | 5 × 10⁸ × 2 条/分钟 ≈ 1.7 M，取保守 1.2 M | 1.2 M QPS |
| 消息大小（平均） | 文本 1 KB + 多媒体 100 KB（按比例折算） ≈ 30 KB | — |
| 网络带宽需求 | 1.2 M × 30 KB ≈ 36 GB/s ≈ 288 Gbps | — |
| 延迟目标 | 99.9% ≤ 150 ms | — |
| 存储量（年） | 5 B × 100 GB ≈ 15 EB | — |
| 可用性 | 99.99% → 每月 ≤ 4.38 h | — |

> **思考**：在后面的设计里，**容量估算**（CPU、内存、磁盘、网络）要围绕这些数字展开，确保每层都有足够的余量。

### 3. 业务假设（帮助划界）  

| 假设 | 说明 |
|------|------|
| 每个用户最多 5 台设备（手机、平板、PC） | 多端同步需要考虑设备 ID |
| 群组最大 1000 人，平均每个用户加入 30 个群 | 群聊广播的规模计算 |
| 消息撤回率 < 0.5% | 撤回相关的额外流量不算主流 |
| 端到端加密在客户端完成，服务器只存密文 | 服务器不参与解密，密钥不落库 |

## 第二步：高层架构设计  

### 1. 最小可用系统（MVP）  

```
+----------+      +----------+      +--------------+
|  Client  | ---> | API GW   | ---> | Message Svc  |
+----------+      +----------+      +--------------+
                                   |
                                   v
                             +------------+
                             |  DB (Message)|
                             +------------+
```

**说明**  

| 组件 | 作用 | 为什么必须有 |
|------|------|--------------|
| 客户端 | 发送/接收消息 | 业务入口 |
| API Gateway (REST/WebSocket) | 统一入口、协议转换、限流、鉴权 | 若直接让客户端连到业务服务，无法统一控制流量、TLS 终止等 |
| Message Service | 处理「发送」请求、写入 DB、返回 ACK | 核心业务逻辑，必须可水平扩展 |
| 数据库 | 持久化消息（密文） | 离线投递、历史查询的根基 |

> **如果不加 API GW** → 每个业务实例都要处理 TLS、鉴权、限流，代码重复且难以扩容。  

### 2. 逐步演进的完整系统（加入高可用、扩展、监控）  

```
                     +-------------------+
                     |   CDN / Edge GW   |
                     +-------------------+
                               |
            +------------------+------------------+
            |                                     |
    +---------------+                     +---------------+
    |  Auth Service |                     |  Rate Limiter |
    +---------------+                     +---------------+
            |                                     |
            +------------------+------------------+
                               |
                         +-----------+
                         | API GW(s) |
                         +-----------+
                               |
        +----------------------+----------------------+
        |                      |                      |
   +----------+          +----------+           +----------+
   |  Router  |          |  Push Svc|           |  Read Svc|
   +----------+          +----------+           +----------+
        |                      |                      |
   +----------+          +----------+           +----------+
   |  Queue   |<------->|  Queue   |<--------->|  Queue   |
   +----------+          +----------+           +----------+
        |                      |                      |
   +----------+          +----------+           +----------+
   |  Msg DB  |          |  Ack DB  |           |  User DB |
   +----------+          +----------+           +----------+
```

#### 关键组件解释  

| 组件 | 职责 | 为什么需要 |
|------|------|------------|
| **CDN / Edge GW** | 靠近用户的入口，做 TLS 终止、IP 过滤、流量就近路由 | 降低跨洲网络 RTT，提升延迟 |
| **Auth Service** | 登录、Token 发放、设备管理 | 统一鉴权，支持多端登录 |
| **Rate Limiter** | 防刷、流控（每用户 QPS、IP QPS） | 防止恶意流量导致系统崩溃 |
| **Router** | 根据收件人/群组决定消息落在哪个分区、写入对应 Queue | 业务解耦，写入即返回 ACK，后续投递异步进行 |
| **Push Service** | 负责从 Queue 拉取消息，推送到在线设备（WebSocket、Push 通知） | 实时性、横向扩展 |
| **Read Service** | 处理离线拉取、已读回执、撤回指令等 | 与投递路径分离，避免相互干扰 |
| **Message Queue** (Kafka / Pulsar) | 持久化、顺序保证、分区扩展 | 高吞吐、天然的容错 |
| **Message DB** (Cassandra / DynamoDB) | 长期存储密文消息、按用户/时间分区 | 支持大规模写入、查询 |
| **Ack DB** (Redis Cluster) | 存储已送达、已读、撤回等短时状态 | 高速读写、TTL 自动过期 |
| **User DB** (MySQL / Vitess) | 用户信息、设备列表、群组元数据 | 关系型查询需求（例如群成员） |

> **不加 Queue** → 消息投递路径会变成「写库 → 同时推送」的同步模式，写入延迟大，写库压力会直接影响实时性，且难以做流量削峰。

## 第三步：数据库设计  

### 1. 选型原则  

| 数据类型 | 访问特征 | 推荐存储 |
|----------|----------|----------|
| **消息（密文）** | 大批量写、按用户/时间范围读、需要水平扩展 | **宽列存储**（Cassandra / Scylla / DynamoDB） |
| **短时状态（已送达/已读/撤回）** | 高频读写、TTL、需要强一致性（局部） | **内存 KV**（Redis Cluster） |
| **用户/设备/群组元数据** | 关系查询、事务（加入/退出） | **关系型**（MySQL + Vitess/Sharding） |
| **大文件（图片、视频）** | 大对象、低频访问 | **对象存储**（S3、COS） + CDN 加速 |

### 2. 消息表（宽列）  

| 主键（PK） | 列族（Column Family） | 备注 |
|------------|-----------------------|------|
| **partition_key = user_id（或 group_id）** | **clustering_key = timestamp (逆序)** | 同一个用户/群的消息会聚在同一个分区，按时间倒序查询最近消息 |
| **message_id** (UUID) | `ciphertext`、`type`、`size`、`media_url`、`sender_id`、`status`（撤回标记） | 只存密文，`media_url` 指向对象存储 |
| **ttl** | 30 天（可配置） | 超期后自动删除，减轻存储压力 |

> **为什么使用宽列而不是传统 RDBMS** → 读取「最近 N 条」时，只需一次扫描分区的前 N 行，极低 I/O；写入时天然水平扩展。  

### 3. 短时状态表（Redis）  

| Key | Value | TTL |
|-----|-------|-----|
| `msg:delivered:{msg_id}` | Set{device_id...} | 24 h |
| `msg:read:{msg_id}` | Set{device_id...} | 24 h |
| `msg:revoke:{msg_id}` | `1`（撤回标记） | 2 min（撤回窗口） |
| `user:online:{user_id}` | Set{device_id...} | 30 s（心跳刷新） |

> **强一致性**：对撤回、已读等状态使用 **Redis Cluster** 的 **同步复制（sync）**，确保同一分区的所有节点在写成功前返回 ACK。  

### 4. 用户/群组表（MySQL）  

```sql
CREATE TABLE users (
    user_id BIGINT PRIMARY KEY,
    phone VARCHAR(20) UNIQUE,
    password_hash VARBINARY(64),
    created_at TIMESTAMP
);

CREATE TABLE devices (
    device_id BIGINT PRIMARY KEY,
    user_id BIGINT,
    push_token VARCHAR(255),
    last_seen TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);

CREATE TABLE groups (
    group_id BIGINT PRIMARY KEY,
    name VARCHAR(64),
    owner_id BIGINT,
    created_at TIMESTAMP
);

CREATE TABLE group_members (
    group_id BIGINT,
    user_id BIGINT,
    role ENUM('owner','admin','member'),
    joined_at TIMESTAMP,
    PRIMARY KEY (group_id, user_id),
    FOREIGN KEY (group_id) REFERENCES groups(group_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

> **为什么仍然保留关系型** → 群成员的增删改需要事务保证（比如一次加入操作要同时写 `group_members` 与 `users` 的 `group_count`），关系型数据库在这类强一致性场景下更成熟。

## 第四步：核心 API 设计  

### 1. 鉴权模型  

| 步骤 | 描述 |
|------|------|
| **登录** | 客户端提交手机号+一次性验证码 → Auth Service 生成 **JWT**（包含 `user_id`、`device_id`、`exp`） |
| **请求** | 每次调用 API 必须在 Header `Authorization: Bearer <jwt>`，网关负责验证并把 `user_id` 注入上下文 |
| **多端** | 同一个用户可以拥有多个 `device_id`，在 `devices` 表里维护对应的推送 token 与在线状态 |

### 2. 发送消息（单聊）  

```
POST /api/v1/message/send
Headers:
  Authorization: Bearer <jwt>
Body (JSON):
{
  "to_user_id": 12345,
  "type": "text|image|audio|video|file",
  "content": "<base64-encoded ciphertext>",   // 已在客户端加密
  "media_url": "https://cdn.example.com/obj/abc", // 可选，仅对多媒体
  "timestamp": 1695691200000
}
Response:
{
  "message_id": "uuid",
  "status": "queued"
}
```

**处理流程**  

1. API GW 校验 JWT → 取 `sender_id`、`device_id`。  
2. 调用 **Router**，Router 根据 `to_user_id` 找到对应的 **Message Queue** 分区（hash(to_user_id)）。  
3. 把消息包装为 **MessageEvent** 写入 Kafka（持久化 + 顺序）。  
4. Router 立即返回 `queued`，保证 **低延迟**（< 50 ms）。  

> **不写入 Queue 而直接写 DB** → 发送端会等待 DB ACK，导致延迟升高，且无法做削峰。  

### 3. 群聊发送  

```
POST /api/v1/group/{group_id}/message
Body 同上，只是把 to_user_id 替换为 group_id
```

**特殊处理**  

- Router 先查询 **group_members**（缓存于 Redis），得到成员列表。  
- 为每个成员计算对应的分区，将同一消息复制到 **N**（≤1000）个分区的 **Topic** 中（或使用 **Kafka的广播模式**）。  
- 为避免一次写入 1000 条记录导致热点，使用 **分区键 = hash(member_id) % P**，并在生产者端批量发送。  

### 4. 拉取离线消息  

```
GET /api/v1/message/receive?since=timestamp
Headers: Authorization
Response:
{
  "messages": [
    {"message_id": "...", "from_user_id": "...", "type":"text", "content":"..."},
    ...
  ]
}
```

- 客户端上线后（WebSocket 连接或 HTTP 长轮询）先调用此接口。  
- **Read Service** 从 **Message DB** 按 `user_id`、`timestamp` 查询最近 N 条未读消息。  
- 返回后，使用 **Ack DB** 标记为已送达。  

### 5. 已读回执  

```
POST /api/v1/message/{message_id}/read
```

- Read Service 把 `device_id` 写入 `msg:read:{message_id}` 的 Set。  
- 同时向 **Push Service** 发送 “已读” 事件，推送给发送方的所有在线设备。  

### 6. 消息撤回  

```
POST /api/v1/message/{message_id}/revoke
Body: { "reason": "mistyped" }
```

- 首先检查 **撤回窗口**（2 min）和 **发送者** 是否匹配。  
- 在 **Redis** 设置 `msg:revoke:{message_id}=1`（TTL=2 min）。  
- **Push Service** 收到撤回事件后，向所有已收到该消息的设备发送 “撤回” 指令。  
- 客户端收到后，从本地 UI、缓存以及本地数据库中删除该消息。  

> **强一致性实现**：撤回写入 Redis 使用 **同步复制**，保证在返回成功前，所有副本已同步，随后 Push Service 读取到撤回标记即可以 **幂等** 的方式广播。

## 第五步：详细组件设计  

### 1. API Gateway  

- **技术选型**：NGINX + Lua（OpenResty）或 **Envoy**。  
- **功能**：TLS 终止、IP 黑名单、限流（Token Bucket per user/device），请求路由到相应的微服务。  
- **监控**：记录 QPS、错误率、平均延迟，输出到 **Prometheus**。  

### 2. 认证服务（Auth）  

- **流程**：  
  1. 客户端发送手机号+一次性验证码（SMS） → **Auth** 验证码服务。  
  2. 验证成功后生成 **JWT**（HS256 或 RS256），签名私钥保存在 **KMS**。  
  3. 同时把 `device_id`、`push_token` 写入 **devices** 表。  

- **高可用**：部署在多 AZ，使用 **Consul/Eureka** 注册中心，状态存储在 **Redis**（短时）+ **MySQL**（持久）。  

### 3. 消息路由（Router）  

- **实现**：基于 **Kafka Producer** 的 **partitioner**，采用 **consistent hashing**（hash(user_id) % partitions）。  
- **负载均衡**：每个 Router 实例无状态，水平扩展。  
- **幂等**：生产者在发送前生成 **message_id**（UUID），若重复写入同 `message_id`，Kafka 会去重（使用 **Idempotent Producer**）。  

### 4. 消息队列（Kafka / Pulsar）  

| 参数 | 设定 | 原因 |
|------|------|------|
| **分区数** | 10 000（全局） | 支持 1.2 M QPS，每分区 100–200 TPS，避免热点 |
| **副本数** | 3 | 跨 AZ 高可用 |
| **压缩** | LZ4 | 减少网络带宽、磁盘占用 |
| **Retention** | 7 天（短期）+ **Compaction**（基于 key） | 保障离线消息不丢失，且对撤回事件可覆盖旧消息 |

### 5. 推送服务（Push）  

- **协议**：WebSocket（双向实时） + **FCM/APNs**（移动端离线推送）。  
- **连接管理**：使用 **Redis** 维护 `user:online:{user_id}` Set，保存每个在线 `device_id` 对应的 **socket id**。  
- **批量推送**：对群聊，先在 **Push Service** 按 **device_id** 分片（Batch size 100），并发发送，降低单次调用的网络开销。  

### 6. 读取服务（Read）  

- **职责**：离线消息查询、已读回执、撤回同步。  
- **查询优化**：对 **Message DB** 建立 **二级索引** `user_id + timestamp`，配合 **Cassandra 的 CLUSTERING ORDER BY** 实现 “最近 N 条” 的快速定位。  

### 7. 存储层  

- **消息存储**：Cassandra（或 Scylla）——写放大低，水平扩展。  
- **短时状态**：Redis Cluster（主从同步），使用 **Redis Streams** 也可以做消费位点追踪。  
- **对象存储**：Amazon S3 / 阿里云 OSS，配合 **CDN**（如 CloudFront）提供全球低延迟下载。  

### 8. 监控、日志、报警  

| 维度 | 工具 |
|------|------|
| **指标**（QPS、Latency、ErrorRate） | Prometheus + Grafana |
| **分布式链路追踪** | OpenTelemetry + Jaeger |
| **日志**（结构化 JSON） | ELK (Elasticsearch, Logstash, Kibana) |
| **告警** | Alertmanager（基于阈值或异常检测） |

> **不做监控** → 难以及时发现热点分区、网络抖动或服务故障，导致 SLA 下降。

## 第六步：扩展性与高可用设计  

### 1. 水平扩展  

| 层级 | 扩容方式 | 关键指标 |
|------|----------|----------|
| **API GW** | 增加实例 + DNS/SLB 负载均衡 | QPS、连接数 |
| **Router / Push / Read** | 无状态容器化，K8s 自动伸缩 | CPU、网络 I/O |
| **Kafka** | 增加 **Broker** + **分区**，使用 **Rack Awareness** 防止同机房故障 | 每分区 TPS、复制延迟 |
| **Cassandra** | 增加节点，自动均衡 token 范围 | 写入吞吐、读延迟 |
| **Redis** | 分片（Cluster） + 主从复制 | 键空间、命中率 |
| **对象存储** | 多 Region 同步（CRR） | 带宽、存储成本 |

> **容量瓶颈预判**：  
> - **发送峰值** → 先观察 **Router → Kafka** 的入站流量，若 Kafka 达到 **produce latency > 100 ms**，考虑 **增加分区**。  
> - **读取热点** → 当某个用户（大 V）频繁拉取历史消息，可在 **Cassandra** 为其开设 **专用节点**（热点分区分离）。  

### 2. 高可用（HA）  

| 场景 | 方案 |
|------|------|
| **单点故障**（API GW、Auth） | 多 AZ 部署，使用 **负载均衡器（L4/L7）** 做健康检查，自动剔除故障实例 |
| **Kafka Broker 故障** | **复制 factor=3**，故障后自动从副本恢复，生产者开启 **acks=all** 确保不丢数据 |
| **Cassandra 节点失效** | **Quorum** 读写一致性，自动重新分配 token |
| **Redis 分片故障** | **主从+哨兵** 或 **Cluster** 自动故障转移 |
| **跨地区灾备** | **Active‑Active** 多 Region 部署，每 Region 拥有独立的完整业务链路（写入同写到两个 Region），使用 **CRDT** 或 **双写** 保证最终一致性；故障时切流至备份 Region，DNS TTL 设为 5 s 进行快速切换 |
| **网络分区（CAP）** | 对 **消息投递** 使用 **AP**（可用性 + 分区容忍），对 **用户元数据**（如群成员）使用 **CP**（强一致性） via 两段事务或 **两阶段提交** |

### 3. 数据一致性与撤回  

- **撤回** 需要 **强一致**：  
  1. **写入 Redis**（同步复制） → 确认成功后返回给客户端。  
  2. **Push Service** 读取撤回键后立即广播。  
  3. 若某台机器在撤回窗口内已经把消息投递给离线用户，撤回指令会在下次 **离线拉取** 时过滤掉（因为 `msg:revoke` 仍在 Redis 中）。  

- **已读**：采用 **幂等** 的 Set 操作，读取端只需要判断 Set 中是否已经存在该 `device_id`。  

### 4. 容灾演练与数据恢复  

| 步骤 | 说明 |
|------|------|
| **1. 备份** | 对 **Message DB**（Cassandra）启用 **snapshot**（每 12 小时），同步到对象存储；对 **Redis** 使用 **RDB+AOF** 双模备份。 |
| **2. 故障模拟** | 通过 Chaos Monkey 随机关闭单个 Region 的 Kafka、Cassandra、Redis。 |
| **3. 自动切换** | 监控检测到 **Write Latency > 500 ms**，触发 **Failover**：DNS 切换至备份 Region，客户端重新握手获取新 Token。 |
| **4. 数据恢复** | 故障 Region 恢复后，从对象存储拉取最新 Snapshot，使用 **Cassandra’s incremental repair** 进行数据同步；Redis 从最新 AOF 重放。 |
| **5. 验证** | 通过 **端到端测试**（发送 → 拉取 → 已读）确保业务完整性。 |

> **不做跨 Region 双写** → 某 Region 完全失效后，离线消息会丢失，违背“消息不丢失”的业务要求。

## 第七步：常见面试追问与回答  

### Q1. “如果我们在 6 个月内用户数翻倍，系统的哪些层级需要先做容量规划？”  

**回答思路**  

| 层级 | 受影响程度 | 先做的动作 |
|------|------------|------------|
| **API Gateway / Load Balancer** | 直接受流量冲击 | 增加实例、提升 LB 并发上限，检查 TLS 终端 CPU 利用率 |
| **Message Queue (Kafka)** | 写入 QPS 成倍增长 | 预先增加 **分区数**（如从 10k → 20k），并检查 **Broker** 磁盘 I/O 与网络带宽 |
| **Router / Push / Read** | 无状态，可水平扩展 | 提前在 K8s 中配置 **自动伸缩**（CPU/内存阈值），确保容器池有余量 |
| **Message DB (Cassandra)** | 写入放大、数据量激增 | 增加节点、重新平衡 token，提前评估磁盘空间（每 PB 费用） |
| **Redis** | 短时状态增长（已读/撤回） | 扩容分片数、调整 `maxmemory-policy`，防止 OOM |
| **对象存储** | 多媒体流量提升 | 启用 **多 Region 镜像**，配置 **CDN** 缓存规则 |

> **先做**：**Queue** 与 **API GW**，因为它们是流量入口，瓶颈直接导致用户感知延迟。  

---

### Q2. “消息撤回需要在所有已收到的设备上同步删除，你会如何保证强一致性而不牺牲性能？”  

**回答要点**  

1. **撤回写入 Redis（同步复制）**：使用 **写入 ACK = all**，确保所有副本在返回成功前已持久。  
2. **幂等广播**：Push Service 读取撤回键后，以 **广播事件**（topic `revoke`) 推送给所有在线设备。设备端收到后执行 **本地删除**，若已经删除则忽略。  
3. **离线用户**：当离线用户下次拉取消息时，Read Service 首先检查 `msg:revoke:{msg_id}` 是否存在，若存在直接过滤掉，不返回给客户端。  
4. **性能优化**：撤回只在 **2 min** 窗口内有效，TTL 自动失效，撤回键在 Redis 中的生命周期极短，查询成本几乎为 **O(1)**。  

> **不使用全局分布式事务**：因为撤回只涉及短时状态，使用 **单点强一致的 KV** 足够，避免对消息库做同步写入，保持主路径的高吞吐。  

---

### Q3. “跨地区数据中心故障导致部分消息库不可用，如何保证离线消息不丢失且能够快速恢复？”  

**回答结构**  

1. **多 Region 双写**：发送时，Router 同时向两个 Region 的 Kafka **MirrorMaker** 复制同一消息（保证 **至少两份**）。  
2. **分区副本跨 AZ**：每个 Region 的 Kafka 副本分布在不同机房，防止单机房故障。  
3. **消息库（Cassandra）**：采用 **Multi‑DC**（NetworkTopologyStrategy），每个 DC 有独立的复制因子（如每 DC 3），写入时使用 **LOCAL_QUORUM**（在本地 DC 达到多数副本），确保写成功即同步到其他 DC（异步复制）。  
4. **故障切换**：监控发现 Region A 的 **write latency** 超过阈值或不可用，自动将流量切换至 Region B（DNS/Global Load Balancer）。客户端重新握手后获得 B 区的 **Token**。  
5. **恢复过程**：故障恢复后，使用 **Cassandra Repair** 或 **SSTable Streaming** 将 B 区的最新数据同步回 A 区；Kafka 使用 **MirrorMaker** 再次拉回缺失分区。  

> **不做双写** → 单 Region 故障时，所有未持久的离线消息都会丢失，违背业务“不丢失”的要求。  

---

### Q4. “如果用户在一台设备上已读消息，另一台设备仍显示未读，如何保持已读状态一致？”  

**回答**  

- 已读事件写入 **Redis Set** `msg:read:{msg_id}`，Set 中存放 **所有已读的 device_id**。  
- 每次客户端打开会话时，读取该 Set 并与本地已读列表比对，更新 UI。  
- 当任一设备发送已读请求，Push Service 将该事件 **广播** 给该用户的所有在线设备。即使设备离线，下一次登录时 **Read Service** 会把已读状态一起返回。  

> **关键点**：使用 **幂等 Set**（`SADD`），避免重复计数，且读写都是 O(1)。  

---

### Q5. “消息存储的压缩和加密会不会导致 CPU 成本过高？”  

**回答**  

- **端到端加密**在客户端完成，服务器只存密文，无需解密，CPU 开销几乎为零。  
- **压缩**（LZ4）在 **Kafka Producer** 端完成，CPU 占用约 2‑3%（相对 150 ms 延迟目标仍有余量），而可节省约 50% 网络带宽。  
- 对 **对象存储**的大文件（图片/视频），使用 **分片上传** + **服务端压缩**（如 WebP、H.264）在客户端完成，服务器只负责存储。  

> **不做压缩** → 带宽成本会急剧上升，尤其在全球范围的峰值 QPS 场景下，可能导致网络拥塞，影响延迟。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **消息撤回的强一致性实现** | 必须在已投递的设备上同步删除，同时不能让撤回路径成为系统瓶颈。 | ① 确定撤回只在 2 min 窗口内有效 → 可以使用短暂的 KV（Redis）实现强一致；② 采用 **同步复制** 确保写成功后立即返回；③ 通过 **Push Service** 的幂等广播完成同步删除；④ 对离线用户在下次拉取时过滤撤回键，保持最终一致性。 |
| **跨 Region 双写与容灾** | 需要在不显著增加写延迟的前提下，实现“消息不丢失”。 | ① 评估写入路径的关键链路：Router → Kafka → Cassandra；② 引入 **MirrorMaker**（Kafka）和 **Cassandra Multi‑DC**，实现 **异步复制**；③ 为保持 150 ms 延迟，复制放在后台，写入仍使用 **本地 quorum**；④ 设计自动故障切换与恢复流程，确保业务感知层无感。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **把所有业务都塞进单体服务**  
   - 结果：代码耦合、难以水平扩展、故障影响全局。  
   - 建议：从 **发送‑路由‑投递** 三大职责划分微服务，保持无状态，使用消息队列解耦。  

2. **忽视离线消息的持久化与削峰**  
   - 只实现 “实时推送”，一旦用户离线就直接丢失消息。  
   - 正确做法：**先写入持久化存储（Message DB）**，再异步投递；使用 **Kafka** 做削峰，确保高峰期间也不会导致写超时。  

### 3. 学习建议和可延伸的方向  

| 方向 | 推荐学习资源 | 实践方式 |
|------|--------------|----------|
| **分布式系统基础**（CAP、BASE、共识） | 《Designing Data‑Intensive Applications》、MIT 6.824 | 搭建自研的 Raft 实现，体会一致性权衡 |
| **消息队列原理**（Kafka、Pulsar） | Confluent Kafka 文档、《Kafka: The Definitive Guide》 | 用 Docker 搭建本地 Kafka，写生产者/消费者模拟消息流 |
| **宽列存储（Cassandra/Scylla）** | 官方文档、《Cassandra: The Definitive Guide》 | 部署单机 Cassandra，实验写入/查询模式 |
| **高可用与容灾**（多 Region） | Google Cloud Spanner/Alibaba Cloud 多活架构案例 | 在云上尝试 Multi‑AZ 部署，做故障切换演练 |
| **监控与链路追踪** | Prometheus + Grafana、OpenTelemetry 官方教程 | 为自己写的微服务加入指标、追踪，观察延迟分布 |
| **端到端加密** | Signal Protocol、libsodium 文档 | 实现一个简单的 E2EE 聊天 Demo，感受密钥管理难点 |

> **关键**：系统设计不是一次性完成的「答案」，而是 **不断迭代** 的过程。先把最核心的「发送‑存储‑投递」跑通，再逐步加入高可用、扩展、监控等能力。每加入一个新组件，都要问自己：*它解决了什么痛点？不加会有什么风险？* 这样的思考方式，能帮助你在面试和实际工作中快速搭建可靠的分布式系统。
