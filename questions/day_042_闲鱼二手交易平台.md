# 第 42 天：设计 闲鱼二手交易平台

> 生成日期：2026-04-14

---

## 题目背景  
闲鱼是阿里巴巴旗下的二手交易平台，用户可以在平台上发布闲置商品、浏览、私聊、下单并完成线下或线上交易。平台需要支撑海量的商品信息、即时的聊天互动以及高并发的搜索/交易流程。

## 面试场景设定  
> **面试官**：  
> “我们现在要设计一个类似闲鱼的二手交易系统，核心是**商品发布/搜索 → 私聊 → 交易闭环**。请你从整体架构出发，先说说系统的主要模块和数据流，并围绕下面的需求进行设计。”

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **用户注册/登录**：支持手机号、微信、支付宝三种方式登录，登录后获取唯一的用户 ID。 |
| 2 | **商品发布与管理**：用户可以上传商品标题、描述、图片（最多 9 张）并设定价格、所在地、标签等，支持编辑、下架、删除。 |
| 3 | **商品搜索 & 推荐**：提供关键字搜索、分类筛选、地理位置过滤，结合热点推荐、相似商品推荐。 |
| 4 | **即时私聊**：买卖双方可在商品详情页发起聊天，支持文本、图片、表情、已读回执。 |
| 5 | **下单与交易流程**：买家发起下单，平台生成订单号，支持支付宝/微信支付，支付成功后进入线下交付/平台托管（可选）。 |
| 6 | **评价系统**：交易完成后，买卖双方可互相评分、留言，评价会影响用户信用分。 |

## 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU** | 30 万 | 活跃用户数，峰值集中在晚上 8 pm‑10 pm。 |
| **QPS（搜索）** | 8 k | 每秒搜索请求峰值，搜索是最频繁的读操作。 |
| **平均响应时间** | ≤ 200 ms（搜索/列表）<br>≤ 100 ms（聊天消息） | 端到端用户感知的延迟。 |
| **可用性** | 99.9%（月可用时间 ≥ 43 200 分钟） | 包括搜索、聊天、支付等核心服务。 |
| **存储规模** | 商品信息 150 TB（≈ 5 亿条商品、30 GB/商品）<br>图片存储 300 PB（CDN+对象存储）<br>聊天日志 20 TB/年 | 需考虑冷热分层、归档。 |
| **伸缩性** | 支持 2 × 峰值流量的横向扩容 | 通过无状态服务 + 自动扩容实现。 |

## 系统边界  

**本题范围内（需要设计）**  
- 用户身份管理、商品发布/编辑、搜索/推荐、聊天、下单支付（不涉及第三方支付网关的实现，仅需说明对接方式）。  
- 数据存储结构（关系型 DB、搜索引擎、对象存储、消息队列等）以及缓存、负载均衡的选型。  
- 高可用、容灾、监控报警的基本思路。  

**本题范围外（不必详细设计）**  
- 支付网关的具体实现细节（如支付宝/微信支付的协议）。  
- 物流、线下交付的供应链系统。  
- 内容安全（图片/文字审核）和反欺诈模型。  
- 运营后台（商品审核、活动运营）和数据分析平台。  

## 提示与追问  

1. **数据一致性**：在商品下单后，如何确保商品库存（上架状态）与订单状态的强一致性？  
2. **聊天系统扩展**：如果要支持 10 M 同时在线用户的即时消息，消息的可靠投递、离线存储以及跨地域部署如何设计？  
3. **搜索与推荐的热点问题**：面对“爆款商品”突然涌入导致搜索负载激增，你会采用哪些缓存/预热、流量削峰的手段？

# 题解

## 解题思路总览  

> **目标**：从「最小可用系统」出发，逐步演进到「高可用、可横向扩容」的完整二手交易平台架构。  
> **思路**：  
> 1. **先把需求拆解**——功能、非功能、业务流程、数据流。  
> 2. **估算规模**（用户、QPS、存储），帮助后面选型。  
> 3. **画出高层框图**，把系统划分为 **用户层、业务层、数据层、基础设施层**。  
> 4. **围绕每个业务模块**（注册、商品、搜索、聊天、订单、评价）设计 **数据库表/索引、API、缓存、消息队列**。  
> 5. **再加入容灾、监控、限流、降级** 等非功能特性，使系统满足 **99.9% 可用、200 ms 响应** 等指标。  
> 6. **最后准备面试追问**（强一致性、海量聊天、热点搜索）以及 **心得反思**，帮助新手避免常见陷阱。  

下面我们一步步展开，**每一步都解释「为什么这样做」**，并说明不这样做会出现什么问题。

## 第一步：理解需求与规模估算  

### 1. 功能拆解  

| 模块 | 关键业务 | 主要接口 |
|------|----------|----------|
| **用户** | 注册/登录（手机号、微信、支付宝） | `/auth/register`, `/auth/login` |
| **商品** | 发布、编辑、下架、删除、查看 | `/item/create`, `/item/update`, `/item/delete`, `/item/get` |
| **搜索/推荐** | 关键字、分类、位置、相似/热点推荐 | `/search`, `/recommend` |
| **聊天** | 私聊（文本/图片/表情/已读） | `/chat/send`, `/chat/poll` (长轮询/WS) |
| **订单** | 下单、支付、状态流转、托管 | `/order/create`, `/order/pay`, `/order/status` |
| **评价** | 交易后互评、信用分计算 | `/review/add`, `/user/credit` |

### 2. 非功能指标（转化为技术数字）  

| 指标 | 计算方式 | 结果 |
|------|----------|------|
| **DAU** | 30 万 | 峰值 QPS≈DAU × 活跃率 / 秒 |
| **搜索 QPS** | 8 k（峰值） | 需要支撑 8 000 次/秒搜索 |
| **聊天 QPS** | 设 1 k 消息/秒（保守） | 实际可能更高，后面做弹性扩容 |
| **订单 QPS** | 约 500 req/s（假设 1% DAU 成交） | 需要强一致性 |
| **存储** | 商品 150 TB, 图片 300 PB, 聊天 20 TB/年 | 冷热分层、对象存储、归档必不可少 |
| **可用性** | 99.9% → 单点故障不可接受 | 多机房、多实例、自动故障转移 |

### 3. 初步容量估算  

| 资源 | 计算方式 | 结果 |
|------|----------|------|
| **商品 DB（关系型）** | 5 × 10⁸ 条 × 30 KB ≈ 15 TB（实际行大小 1‑2 KB，索引 2‑3 TB）| 采用 **分库分表**（按用户 ID 或地区） |
| **搜索引擎** | 5 × 10⁸ 条文档，每文档 1 KB（标题+属性）≈ 500 GB，倒排索引 2‑3 TB | 采用 **Elasticsearch / OpenSearch** 集群 |
| **聊天日志** | 20 TB/年 ≈ 55 GB/天，写入峰值 100 MB/s | 使用 **Kafka + HDFS/OSS** 归档 |
| **图片** | 300 PB（CDN+OSS）| 对象存储 + 多区域 CDN 缓存 |
| **缓存** | 热门商品/搜索结果/用户会话 | 需要 **Redis** 集群，容量约 200 GB（根据热点占比） |

> **为什么要先算**：只有明确规模，后面的 **分库分表、集群规模、机器选型** 才有依据。若不估算，直接“大而全”会导致资源浪费或容量不足，系统在高峰时容易崩溃。

## 第二步：高层架构设计  

### 1. 高层组件划分  

```
┌───────────────────────┐
│        CDN / WAF        │   ← 静态资源、流量入口、DDoS 防护
└─────────────▲───────────┘
              │
┌───────┬─────┴─────┬───────┐
│ 负载均衡（L4/L7）│   │ 负载均衡（L4/L7）│
│   API 网关      │   │   静态资源/图片 │
└───────┴─────▲─────┴───────┘
              │
        ┌─────┴─────┐
        │   网关层   │ 统一鉴权、限流、日志
        └─────▲─────┘
              │
   ┌──────────┼───────────┐
   │          │           │
   ▼          ▼           ▼
业务服务1   业务服务2   业务服务3
(User, Item,   Search,   Chat,
 Order, Review)   Recommend)
```

> **解释**  
- **CDN + WAF**：把图片、JS、CSS 这类静态资源放到离用户最近的节点，降低核心业务服务器的带宽压力；WAF 防止恶意请求直接打到业务层。  
- **负载均衡 + API 网关**：实现 **无状态**、**统一鉴权**、**限流**、**灰度发布**。如果直接让客户端请求业务服务器，升级、灰度、监控都很困难。  
- **业务服务**：按业务边界拆分成 **微服务**（或按功能拆分的模块化单体），每个服务只负责自己的业务数据和业务逻辑，避免“巨石”系统难以扩展。  

### 2. 数据层划分  

| 层级 | 技术选型 | 作用 |
|------|----------|------|
| **关系型 DB** | MySQL（Percona / Aurora） | 商品主信息、用户、订单、评价等强一致性事务 |
| **搜索引擎** | Elasticsearch 集群 | 商品全文搜索、过滤、排序、热点/相似推荐 |
| **对象存储** | OSS / MinIO（多AZ）+ CDN | 商品图片、聊天图片、附件 |
| **缓存** | Redis Cluster（读写分离） | 热门商品、用户会话、搜索热点、限流计数 |
| **消息队列** | Kafka + RocketMQ | 异步任务（图片处理、搜索索引、订单事件、聊天持久化） |
| **日志/审计** | ELK（Elasticsearch + Logstash + Kibana） | 系统日志、审计、监控查询 |
| **流式计算** | Flink / Spark Streaming | 实时热点商品、推荐模型、信用分计算 |

> **为什么分层**：不同数据访问特性（强一致 vs. 高吞吐 vs. 大容量）用最合适的存储，能让系统在 **成本、性能、可用性** 上取得平衡。若把所有数据都放在 MySQL，写放大、查询慢、扩容困难；若只用 NoSQL，事务和关联查询会受限。  

### 3. 网络与安全  

- **VPC**：业务服务器、数据库、缓存、MQ 均放在同一 VPC，不同子网划分前后端。  
- **安全组**：最小权限原则，只开放必要端口（如 443、6379、3306、9092）。  
- **HTTPS + TLS**：所有外部流量加密；内部服务间使用 mTLS（服务网格如 Istio）提升安全性。

## 第三步：数据库设计  

> **原则**：**业务一致性**（事务）交给关系型 DB，**查询性能**交给搜索引擎，**海量对象**交给对象存储。  

### 1. 关系型数据库（MySQL）表结构（示例）  

#### 1.1 用户表 `users`  

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | BIGINT PK | 全局唯一，自增或 Snowflake |
| `mobile` | VARCHAR(20) | 手机号（唯一） |
| `wechat_openid` | VARCHAR(64) | 微信唯一标识 |
| `alipay_uid` | VARCHAR(64) | 支付宝唯一标识 |
| `nickname` | VARCHAR(64) | 显示昵称 |
| `avatar_url` | VARCHAR(256) | 头像 OSS 地址 |
| `credit_score` | INT | 信誉分 |
| `created_at` | DATETIME | 注册时间 |
| `updated_at` | DATETIME | 更新时间 |

- **唯一索引**：`mobile`, `wechat_openid`, `alipay_uid` 防止同一用户多次注册。  
- **Why**：登录需要快速定位用户，唯一索引提供 O(logN) 查询。  

#### 1.2 商品表 `items`  

| 字段 | 类型 | 说明 |
|------|------|------|
| `item_id` | BIGINT PK | 全局唯一 |
| `seller_id` | BIGINT FK (`users.user_id`) | 所属卖家 |
| `title` | VARCHAR(128) | 商品标题 |
| `description` | TEXT | 详细描述 |
| `price` | DECIMAL(10,2) | 价格 |
| `status` | TINYINT | 0‑下架, 1‑在售, 2‑已成交 |
| `category_id` | INT | 分类 |
| `city_id` | INT | 所在城市 |
| `tags` | JSON | 标签数组 |
| `image_urls` | JSON | 图片 OSS 链接数组 |
| `created_at` | DATETIME | 发布时间 |
| `updated_at` | DATETIME | 最近编辑时间 |
| `version` | BIGINT | 乐观锁字段（防并发修改） |

- **分库分表**：按 `seller_id % N` 或 `city_id % N` 做水平切分，避免单表 5 亿行导致索引膨胀。  
- **Why**：商品查询（搜索）主要靠搜索引擎，但订单生成时需要检查 `status`，必须在事务中完成。  

#### 1.3 订单表 `orders`  

| 字段 | 类型 | 说明 |
|------|------|------|
| `order_id` | BIGINT PK | 全局唯一（业务号） |
| `buyer_id` | BIGINT FK (`users.user_id`) | 买家 |
| `seller_id` | BIGINT FK (`users.user_id`) | 卖家 |
| `item_id` | BIGINT FK (`items.item_id`) | 对应商品 |
| `price` | DECIMAL(10,2) | 成交价 |
| `status` | TINYINT | 0‑待付款,1‑已付款,2‑已完成,3‑已关闭 |
| `pay_time` | DATETIME | 支付时间 |
| `create_time` | DATETIME | 创建时间 |
| `update_time` | DATETIME | 更新时间 |
| `version` | BIGINT | 乐观锁（防并发下单） |

- **事务**：下单时需要 **先锁定商品**（`SELECT ... FOR UPDATE`） → 创建订单 → 更新商品 `status=2`（已成交）。  
- **Why**：保证 **商品库存（唯一）** 与 **订单状态** 强一致，防止超卖。  

#### 1.4 评价表 `reviews`  

| 字段 | 类型 | 说明 |
|------|------|------|
| `review_id` | BIGINT PK | 主键 |
| `order_id` | BIGINT FK (`orders.order_id`) | 所属订单 |
| `reviewer_id` | BIGINT FK (`users.user_id`) | 评价人 |
| `reviewed_id` | BIGINT FK (`users.user_id`) | 被评价人 |
| `score` | TINYINT | 1‑5 分 |
| `comment` | TEXT | 文字评价 |
| `created_at` | DATETIME | 时间 |
| `updated_at` | DATETIME | 更新时间 |

- **业务规则**：订单完成后才能评价，后端在 `order.status=2` 时才允许写入。  

### 2. 搜索引擎（Elasticsearch）映射示例  

```json
PUT items_index
{
  "mappings": {
    "properties": {
      "item_id": {"type":"keyword"},
      "title": {"type":"text","analyzer":"ik_max_word"},
      "description": {"type":"text","analyzer":"ik_max_word"},
      "price": {"type":"scaled_float","scaling_factor":100},
      "category_id": {"type":"integer"},
      "city_id": {"type":"integer"},
      "tags": {"type":"keyword"},
      "status": {"type":"byte"},
      "created_at": {"type":"date"},
      "seller_id": {"type":"keyword"}
    }
  }
}
```

- **同步机制**：商品创建/编辑 → 写入 MySQL → 通过 **Kafka** 发送 `item_change` 事件 → **Logstash/Canal** 同步到 ES。  

### 3. 缓存设计（Redis）  

| Key | 类型 | 示例 | 用途 |
|-----|------|------|------|
| `user:{user_id}` | Hash | `user:12345` | 个人信息、信用分（热点） |
| `item:hot:{city_id}` | Sorted Set | `item:hot:310` | 城市热点商品（score=浏览量） |
| `search:kw:{keyword}` | String (JSON) | `search:kw:iphone` | 关键词搜索结果缓存（TTL 5 min） |
| `chat:session:{userA}:{userB}` | Hash | `chat:session:1001:2002` | 最近 N 条聊天摘要，用于列表展示 |
| `rate_limit:{ip}` | String | `rate_limit:10.0.0.1` | 接口限流计数器（滑动窗口） |

- **Why**：热点数据放在内存里可以把 **读延迟** 从毫秒降到 **微秒**，显著提升用户体验。

## 第四步：核心 API 设计  

> **原则**：RESTful + JSON，统一错误码，接口幂等（POST 用唯一 `request_id`），鉴权采用 **JWT**（在网关层校验）。  

### 1. 用户模块  

| API | 方法 | 请求体 | 响应 | 说明 |
|-----|------|--------|------|------|
| `/auth/register` | POST | `{mobile, sms_code, nickname, avatar_url}` | `{user_id, token}` | 短信验证码校验，返回 JWT |
| `/auth/login` | POST | `{type:"mobile"/"wechat"/"alipay", credential}` | `{user_id, token}` | 第三方登录 → 通过统一 ID 关联 |
| `/user/profile` | GET | Header `Authorization: Bearer xxx` | `{user_id, nickname, avatar, credit_score}` | 查询个人信息 |
| `/user/profile` | PUT | `{nickname, avatar_url}` | `{success:true}` | 修改（幂等） |

### 2. 商品模块  

| API | 方法 | 请求体 | 响应 | 说明 |
|-----|------|--------|------|------|
| `/item/create` | POST | `{title, description, price, category_id, city_id, tags, images[]}` | `{item_id}` | 业务校验后写入 MySQL、发送 `item_change` 到 Kafka |
| `/item/{item_id}` | GET | — | 商品完整信息 + `seller_info` | 从缓存读取，若 miss 再查询 MySQL + ES |
| `/item/{item_id}` | PUT | `{title?, description?, price?, status?, images?}` | `{success:true}` | 使用 **乐观锁** `version` 防冲突 |
| `/item/{item_id}` | DELETE | — | `{success:true}` | 软删（status=0）+异步删除图片 |

### 3. 搜索 / 推荐模块  

| API | 方法 | 请求体 | 响应 | 说明 |
|-----|------|--------|------|------|
| `/search` | GET | `q=keyword&category=...&city=...&sort=price_desc&page=1&size=20` | `{items[], total}` | 先查 Redis 缓存（热点），未命中走 ES，返回 ID 列表后批量从 MySQL 拉详情 |
| `/recommend/hot` | GET | `city_id=&limit=10` | `{items[]}` | 读取 Redis Sorted Set `item:hot:{city_id}` |
| `/recommend/related/{item_id}` | GET | — | `{items[]}` | 通过 ES `more_like_this` 查询相似商品 |

### 4. 聊天模块（基于 WebSocket）  

| API | 方法 | 请求体 | 响应 | 说明 |
|-----|------|--------|------|------|
| `/chat/connect` | WebSocket Upgrade | Header `Authorization` | `connected` | 握手后服务器下发 `session_id` |
| `/chat/send` | POST (或 WS `msg`) | `{to_user_id, type:"text"/"image", content, request_id}` | `{msg_id, timestamp}` | 写入 Kafka `chat_msg`，消费者落库、写 Redis 消息队列 |
| `/chat/history` | GET | `peer_user_id&cursor&size` | `{messages[], next_cursor}` | 从 Redis 拉最近 N 条，历史归档从 HDFS/OSS 中查询（分页） |
| `/chat/ack` | POST | `{msg_id}` | `{success:true}` | 标记已读，更新 Redis `unread:{user_id}` 集合 |

- **Why WebSocket**：聊天需要 **双向低延迟**，HTTP 长轮询会浪费带宽和增加延迟。  

### 5. 订单模块  

| API | 方法 | 请求体 | 响应 | 说明 |
|-----|------|--------|------|------|
| `/order/create` | POST | `{item_id, buyer_id, pay_type}` | `{order_id, status:0}` | **事务**：`SELECT item FOR UPDATE` → 检查 `status=1` → `INSERT order` → `UPDATE item SET status=2` |
| `/order/pay/callback` | POST (支付网关回调) | `{order_id, pay_status, pay_time, sign}` | `{success:true}` | 幂等校验 `pay_status`，写入 `order.status=1`，发送 `order_paid` 事件 |
| `/order/{order_id}` | GET | — | 订单详情 | 从 MySQL 读取，若状态为已付款，返回支付凭证 |
| `/order/{order_id}/cancel` | POST | — | `{success:true}` | 检查状态可取消 → 回滚商品 `status=1` → 更新订单 `status=3` |

### 6. 评价模块  

| API | 方法 | 请求体 | 响应 | 说明 |
|-----|------|--------|------|------|
| `/review/add` | POST | `{order_id, score, comment}` | `{review_id}` | 检查订单已完成，写入 `reviews`，异步更新 `users.credit_score` |
| `/user/{user_id}/reviews` | GET | `page,size` | `{reviews[], avg_score}` | 拉取对方的评价列表 |

## 第五步：详细组件设计  

### 1. 用户身份管理  

- **登录流程**：  
  1. 前端收集手机号或第三方授权码。  
  2. API 网关转发到 **Auth Service**。  
  3. Auth Service 调用 **SMS/微信/支付宝** SDK 验证。  
  4. 若是新用户，写入 `users` 表（唯一约束），返回 **JWT**（包含 `user_id`、`exp`）。  
  5. JWT 在网关层验证，放入 **上下文**（如 `X-User-Id`），后续业务服务直接读取。  

- **为什么使用 JWT**：无状态、跨服务共享用户身份，避免每次请求都查询 DB，提升性能。  

### 2. 商品发布与搜索  

#### 2.1 商品写入流程  

```
Client -> API Gateway -> Item Service
Item Service:
  1. 参数校验、图片上传至 OSS（返回 URL）
  2. 写 MySQL (INSERT) -> 获取 item_id
  3. 生成 Kafka 事件 item_change (type=CREATE)
  4. 返回 item_id 给前端
```

- **异步索引**：消费者 `ItemIndexWorker` 读取 Kafka，调用 ES API **批量 upsert**。  
- **Why**：写入 MySQL 必须快速返回，索引构建耗时，异步可解耦，提升写入吞吐。  

#### 2.2 搜索请求路径  

```
Client -> API GW -> Search Service
Search Service:
  1. 先查询 Redis cache (search:kw:{keyword})
  2. Miss -> 调用 ES 查询 (分页、过滤、排序)
  3. 取出 item_id 列表 -> 批量从 MySQL (或 Elastic 的 _source) 拉商品详情
  4. 结果写入 Redis (TTL 5min) 返回
```

- **热点缓存**：热门关键词的搜索结果直接命中 Redis，降低 ES 负载。  
- **Why ES**：支持全文检索、倒排索引、聚合统计，单机 MySQL 难以满足 8k QPS+过滤需求。  

#### 2.3 推荐系统（简化版）  

- **热点推荐**：基于 Redis Sorted Set `item:hot:{city}`，实时维护（每次商品被浏览/收藏，ZINCRBY）。  
- **相似商品**：使用 ES `more_like_this`，或离线计算向量（Item2Vec）写入 ES `knn_vector` 字段，在线查询时返回 Top‑N。  

### 3. 即时私聊系统  

#### 3.1 架构总览  

```
Client <--WebSocket--> Chat Gateway (nginx+ws) <---> Chat Server Cluster
Chat Server:
  - 接收消息，写入 Kafka (topic=chat_msg)
  - 将消息写入 Redis (list per conversation) 供在线用户拉取
  - 计入未读计数 set (unread:{user_id})
  - 消费者落库到 MySQL (chat_logs) + 归档到 HDFS/OSS
```

#### 3.2 消息可靠投递  

- **生产端**：WebSocket 收到消息后，**先写入 Kafka**（同步 `acks=all`），确保消息持久化。  
- **消费端**：Chat Server **异步**从 Kafka 拉取，**原子**写入 Redis + MySQL。若 Redis 写失败（宕机），仍有 MySQL 持久化，可在恢复后回补。  
- **幂等**：每条消息带 `request_id`（UUID），写入 DB 前先 `INSERT ... ON DUPLICATE KEY UPDATE`，防止客户端重发导致重复。  

#### 3.3 离线存储 & 拉取  

- **最近 N 条**（如 100 条）保存在 Redis `list:{conv_id}`（LRU 删除），查询时直接返回。  
- **历史归档**：每小时将 Redis 列表快照写入 **HDFS**（Parquet）或 **OSS**，供后续分页查询。  

#### 3.4 跨地域部署  

- **多 IDC**：在每个大区部署独立的 Chat Server + Kafka MirrorMaker 同步 `chat_msg` 主题。  
- **用户定位**：用户首次登录时记录地域，后端将其路由到最近的 Chat Server。若跨地域聊天，消息会在两端 **双向同步**（Kafka MirrorMaker + 双写）。  

### 4. 订单与支付  

#### 4.1 下单事务  

```sql
START TRANSACTION;
SELECT status FROM items WHERE item_id = ? FOR UPDATE;
-- 若 status != 1 (在售) => abort
INSERT INTO orders (order_id, buyer_id, seller_id, item_id, price, status) VALUES (...);
UPDATE items SET status = 2 WHERE item_id = ?;
COMMIT;
```

- **锁粒度**：只锁定单条商品行，避免全表锁。  
- **Why**：防止同一商品被多个买家同时下单（强一致性需求）。  

#### 4.2 支付回调幂等  

- 支付平台回调带 **`order_id` + `out_trade_no` + `sign`**。  
- **Order Service** 首先查询 `order.status`，若已是 `已付款`，直接返回成功（幂等）。  
- 成功后发送 **`order_paid`** 事件到 Kafka，供后续 **物流、评价、信用分** 等模块消费。  

#### 4.3 托管/线下交付  

- 业务层记录 `delivery_mode`（托管/线下），不在本设计深度展开。  

### 5. 评价与信用分  

- **写入流程**：订单完成 → 调用 Review Service → 写入 `reviews` → 发送 `review_added` 事件。  
- **信用分计算**：消费者 `CreditWorker` 消费 `review_added`，使用 **滑动窗口**（最近 30 天）对用户评分做加权平均，更新 `users.credit_score`（写入 Redis + MySQL）。  

### 6. 缓存与热点处理  

| 场景 | 缓存策略 | 失效/更新方式 |
|------|----------|----------------|
| 商品详情 | **读写穿透**：先查 Redis，Miss → MySQL → 写入 Redis（TTL 30 min） | 商品编辑/下架 → **主动删**或 **更新** |
| 搜索结果 | **热点关键词缓存**（TTL 5 min） | 定时预热（每日热点） |
| 热门商品榜单 | **Redis Sorted Set**（ZINCRBY） | 实时写入，定时清理过期 |
| 聊天未读计数 | **Redis Set** `unread:{uid}` | 消息已读 → `SREM`，用户登出 → 清空 |
| 访问频率限流 | **令牌桶**（Lua 脚本） | 每分钟/秒自动恢复 |

- **为什么要缓存**：读取频率远高于写入，缓存可以把 **读延迟** 从毫秒级降到 **微秒级**，显著提升用户感知。

## 第六步：扩展性与高可用设计  

### 1. 无状态化 + 自动扩容  

- **所有业务服务**（Auth、Item、Search、Chat、Order、Review）均设计为 **无状态**：  
  - 会话信息保存在 **Redis**（Token、聊天会话）或 **数据库**。  
  - 只依赖外部资源（DB、Cache、MQ），可以随时 **水平扩容**。  
- **K8s / Docker Swarm**：使用容器编排平台，配合 **Horizontal Pod Autoscaler (HPA)** 根据 CPU / QPS 自动扩容。  

### 2. 数据库高可用  

| DB | 高可用方案 |
|----|------------|
| MySQL | 主从复制 + 半同步（确保写成功后主库同步到至少 1 个从库）<br>故障转移使用 **MHA / ProxySQL** |
| Elasticsearch | 多主分片（primary=3, replica=2）<br>跨 AZ 部署，自动分片均衡 |
| Redis | 主从复制 + Sentinel（自动故障转移）<br>或 **Redis Cluster**（分片 + 多主） |
| Kafka | 多副本（replication factor=3）+ Zookeeper（或 KRaft）<br>跨机房 MirrorMaker 实现异地灾备 |

- **为什么要多副本**：单点故障会导致服务不可用，副本可以在节点故障时快速切换，满足 **99.9%** 可用性。  

### 3. 容灾与跨地域  

- **多可用区（AZ）**：同一业务层在不同 AZ 部署多实例，使用 **负载均衡器（L4/L7）** 实现跨 AZ 自动切换。  
- **跨地域灾备**：关键数据（订单、用户）在 **两个地域** 同步（MySQL 双活或基于 MySQL GTID 的异步复制），搜索索引通过 **跨集群快照** 同步。  
- **灾备演练**：定期进行 **故障切换演练**，验证恢复时间在 **5 min** 以内。  

### 4. 流量削峰 & 限流  

- **入口限流**：API Gateway 使用 **令牌桶 + 漏斗** 对登录、搜索、下单等关键接口进行 **QPS 限流**，防止突发流量冲垮后端。  
- **热点商品预热**：对即将上架的热门商品（如“双十一”）提前 **预热缓存**（将商品信息、搜索索引提前加载到 Redis/ES），并在正式上架前 **开启流量分级**（灰度放流），逐步提升流量。  
- **降级策略**：当搜索 ES 超时，返回 **推荐热点** 或 **最近浏览**，保证页面仍可渲染，提升可用性。  

### 5. 监控、告警、日志  

| 维度 | 工具 |
|------|------|
| **指标**（QPS、RT、错误率） | Prometheus + Grafana |
| **日志** | ELK（Filebeat → Logstash → Elasticsearch） |
| **链路追踪** | OpenTelemetry + Jaeger |
| **告警** | Alertmanager（根据阈值、异常率）发送钉钉/邮件 |
| **容量预警** | 定期统计 DB、ES、OSS 使用量，触发扩容脚本 |

- **为什么要监控**：及时发现 **性能瓶颈**、**异常流量**，在 SLA 失效前自动恢复或人工干预。

## 第七步：常见面试追问与回答  

### 1. 商品下单后，如何确保商品库存（上架状态）与订单状态的强一致性？  

**回答要点**：  
- **单行锁 + 事务**：在 MySQL 中使用 `SELECT ... FOR UPDATE` 锁定商品行，随后在同一个事务里插入订单、更新商品状态。  
- **乐观锁**：在 `items` 表加入 `version` 字段，业务先读取 version → 下单时 `UPDATE items SET status=2, version=version+1 WHERE item_id=? AND version=?`，若返回 0 行说明并发冲突，需要重试。  
- **防止超卖**：只有当商品状态为“在售”(status=1) 时才允许下单，失败则返回商品已下架/已成交。  
- **幂等性**：订单号使用全局唯一 ID（Snowflake），即使客户端重发请求也只会产生一条订单。  

### 2. 支持 10 M 同时在线用户的即时消息，如何保证可靠投递、离线存储以及跨地域部署？  

| 需求 | 设计方案 | 关键技术 |
|------|----------|----------|
| **可靠投递** | 客户端 → **WebSocket** → **Kafka**（`acks=all`） → **Chat Server**（写 Redis + MySQL） | Kafka 持久化、幂等 `request_id`、事务日志 |
| **离线存储** | 最近 N 条保存在 Redis List，超过 N 的归档到 **HDFS/OSS**（Parquet） | Redis LRU、批量写入 HDFS |
| **跨地域** | 每个地域部署独立的 Chat Server + Kafka 集群；使用 **Kafka MirrorMaker** 双向同步 `chat_msg` 主题；用户首次登录时绑定最近的地域 | MirrorMaker、跨地域 DNS/Anycast |
| **水平扩容** | Chat Server 基于 **Stateless**，通过 **K8s HPA** 根据连接数自动扩容；Kafka 按分区数水平扩容 | K8s、分区均衡 |
| **消息顺序** | 同一会话的消息使用同一个分区（`partition_key = conversation_id`），保证顺序消费 | Kafka partitioning |

### 3. 搜索热点商品导致突发流量激增，如何削峰？  

- **热点缓存**：使用 **Redis Sorted Set** 维护热点榜单，搜索时先检查是否在热点列表，若是直接返回缓存结果。  
- **预热+限流**：在热点商品上架前 5 min 开启 **流量分级**（例如每秒只能放 500 条搜索请求），逐步提升到全量。  
- **异步降级**：当 ES 响应时间 > 200 ms，返回 **推荐热点** 或 **最近浏览**，并在后台继续查询，前端异步刷新。  
- **分片扩容**：ES 集群可以动态添加 **节点** 并 **reallocate shards**，提升查询并发。  
- **流量分发**：使用 **全局负载均衡（L7）** 将请求路由到 **不同地域** 的搜索集群，实现 **跨地域削峰**。  

### 4. 如果用户的信用分出现争议，系统该如何回滚或修正？  

- **审计日志**：所有评价、信用分变化写入 **审计表**（不可修改），并同步到 **Kafka** 供离线校验。  
- **可回滚**：信用分计算采用 **增量日志**（每次加/减多少），若出现争议，只需要在审计日志中找到对应记录，反向操作即可。  
- **人工介入**：提供后台 **工单系统**，管理员可以调用 **API** 手动调整 `users.credit_score`，并记录操作日志。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策  

| 决策 | 挑战 | 思考过程 |
|------|------|----------|
| **商品下单的强一致性** | 需要在高并发下防止超卖，同时又要保证系统可用性（不因为单表锁导致整个订单系统阻塞）。 | 先考虑 **全局分布式锁**（如 Redis RedLock），发现会成为性能瓶颈；随后转向 **单行锁 + 事务**（MySQL）或 **乐观锁**，结合 **分库分表**（每个库只管少量商品），最终在业务层实现 **“先查询锁定 → 下单 → 更新状态”** 的原子操作。 |
| **海量即时聊天的可靠投递与跨地域同步** | 10 M 在线用户意味着每秒可能产生数十万条消息，必须做到 **零丢失**、**低延迟**，且要在多地域之间保持一致。 | 初始想把消息直接写入 Redis List，后发现宕机会导致数据丢失；再考虑单点 Kafka 集群，跨地域网络延迟大；最终选择 **Kafka + MirrorMaker** 作为持久化和跨地域复制层，WebSocket 负责实时推送，Redis 负责热点缓存，HDFS/OSS 负责归档。通过 **分区键 = 会话 ID** 保证顺序，再加 **幂等 request_id** 防止重复投递。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **把所有数据都放在单体 MySQL**  
   - **后果**：查询热点（搜索、推荐）导致读写竞争，扩容困难，单点故障直接导致整个平台不可用。  
   - **正确做法**：把 **搜索** 移到 **Elasticsearch**，把 **图片** 存储到 **对象存储+CDN**，把 **聊天** 用 **Kafka+Redis**，只在 MySQL 保存需要强事务的数据（用户、订单、评价）。  

2. **忽视幂等性与事务边界**  
   - **后果**：在网络抖动或客户端重试时产生 **重复订单、重复消息**，导致库存错乱、信用分异常。  
   - **正确做法**：所有写操作都要 **幂等**（全局唯一 `request_id`、业务号），下单使用 **行锁/乐观锁**，支付回调使用 **状态检查**，聊天消息使用 **Kafka** 的 **`acks=all`** 并在 DB 中 **唯一键** 防重。
