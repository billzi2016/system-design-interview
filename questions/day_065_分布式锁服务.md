## 题目背景  
分布式锁服务是一套提供跨机器、跨进程的互斥/同步能力的中间件，常用于保证在分布式环境下对共享资源的安全访问（如库存扣减、订单号生成、任务调度等）。

## 面试场景设定  
> **面试官**：我们现在需要设计一套高可用的分布式锁服务，供公司内部的微服务在高并发场景下使用。请你从零开始梳理系统的整体架构、关键技术点以及如何满足业务需求。  

（面试官随后会要求你重点说明锁的获取/释放流程、容错机制、性能指标等。）

## 功能性需求  

| 编号 | 需求描述 |
|------|----------|
| 1 | **加锁**：客户端可以请求在指定的资源键上获取独占锁，支持阻塞等待和立即返回两种模式。 |
| 2 | **解锁**：客户端可以主动释放自己持有的锁，系统必须校验释放者身份，防止误删。 |
| 3 | **锁自动过期**：为防止死锁，锁必须支持 TTL（可配置），TTL 到期后自动失效并可被其他请求获取。 |
| 4 | **可重入锁**：同一客户端在同一资源上可以多次加锁，解锁时需要对应次数才能真正释放。 |
| 5 | **锁租约续期**：在业务执行时间可能超过 TTL 的情况下，客户端可以主动续租，延长锁的有效期。 |
| 6 | **监控/查询**：提供查询接口，能够查询当前锁的持有者、剩余 TTL、加锁次数等信息，供运维调试使用。 |

## 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **QPS（加锁/解锁）** | 50,000 次/秒 | 高峰期整体请求量，要求系统在单节点水平扩展后仍能保持此吞吐。 |
| **平均延迟** | < 5 ms（99% 请求） | 加锁/解锁路径的端到端响应时间，包含网络往返。 |
| **可用性** | 99.99%（月均累计不可用时间 ≤ 4.38 小时） | 必须实现无单点故障，支持自动故障转移。 |
| **数据持久化** | 10 TB（锁状态元数据） | 假设每秒产生 10 万条锁记录，保留 30 天用于审计。 |
| **一致性** | 强一致性（加锁成功即全局唯一） | 业务对锁的互斥性要求极高，不能出现“脏读”。 |

# 分布式锁服务设计详解（新手手把手教程）

> **温馨提示**：本篇从 “最小可用系统” 起步，逐层演进到 **高可用、强一致** 的完整方案。每一步都会解释 **为什么要这么做**，以及 **不这么做会有什么后果**，帮助零经验的同学把抽象的概念转化为可落地的设计。

## ## 解题思路总览  

1. **先把需求拆解**：功能需求（加锁、解锁、TTL、可重入、续租、监控） → 非功能需求（QPS、时延、可用性、一致性、持久化）。  
2. **估算规模**：算出每秒需要处理多少请求、每条锁记录的大小、30 天审计日志的存储量。  
3. **从最小可运行系统（单机 KV + 简单协议）** 起步，验证核心业务逻辑。  
4. **逐步加入**：强一致性共识、故障转移、水平扩容、监控、审计持久化。  
5. **最终形成**：**基于 Raft 共识的 KV 存储 + 多副本集群 + 客户端 SDK + 监控/告警** 的完整分布式锁服务。  

> **核心难点**：在高并发下保证 **“锁的唯一性 + 强一致性”**，并且在节点故障、网络分区时仍能保持 **可用性**。  

下面按章节一步步展开。

## ## 第一步：理解需求与规模估算  

### 1. 功能需求拆解  

| 编号 | 需求 | 关键要点 |
|------|------|----------|
| 1 | **加锁**（阻塞 / 立即返回） | - 需要原子判断键是否存在并写入锁信息<br>- 阻塞模式可以轮询或使用等待队列 |
| 2 | **解锁** | - 必须校验 “持有者身份” 防止误删<br>- 支持可重入计数递减 |
| 3 | **锁自动过期（TTL）** | - 防止死锁，TTL 到期自动失效<br>- TTL 必须在所有副本上保持一致 |
| 4 | **可重入锁** | - 同一客户端对同一资源可多次加锁<br>- 需要计数器 + 持有者唯一标识 |
| 5 | **锁租约续期** | - 客户端可以在业务执行时间超过 TTL 时延长租约<br>- 续期必须再次写入新 TTL，且保持原持有者身份 |
| 6 | **监控/查询** | - 查询锁的持有者、剩余 TTL、计数等<br>- 为运维提供审计、告警依据 |

### 2. 非功能需求拆解  

| 指标 | 目标 | 设计影响 |
|------|------|----------|
| QPS 50k/s | 高峰期总请求量 | 需要 **水平扩展**、**低延迟协议**、**高效序列化** |
| 平均延迟 <5 ms（99%） | 端到端响应时间 | 选用 **内存 KV**、**近距离网络**、**异步 IO** |
| 可用性 99.99% | 月均 ≤ 4.38 h 不可用 | 必须消除 **单点故障**，实现 **自动故障转移** |
| 持久化 10 TB（30 天审计） | 记录所有锁操作 | 采用 **分区写入**的 **时序/日志存储**（如 ClickHouse、HBase） |
| 强一致性 | 加锁成功即全局唯一 | 采用 **共识算法（Raft）** 或 **线性化读写**的 KV 存储 |

### 3. 粗略规模估算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **每秒锁操作数** | QPS = 50k（加锁+解锁） | 50 000 ops/s |
| **每分钟** | 50 000 × 60 | 3 000 000 ops |
| **每小时** | 3 000 000 × 60 | 180 000 000 ops |
| **30 天总记录数** | 180 M × 24 × 30 ≈ 130 B 条 | ~130 B 条日志 |
| **单条锁元数据大小**（键 32 B + 持有者 ID 36 B + TTL 8 B + 计数 4 B）≈ 80 B | 80 B × 130 B ≈ **10 TB**（符合需求） |
| **内存需求**（活跃锁 ≈ QPS × TTL）<br>假设平均 TTL 10 s → 50 k × 10 = 500 k 锁 | 500 k × 80 B ≈ 40 MB（非常小） |  

> **结论**：业务高峰期 **内存压力不大**，但 **持久化审计** 需要 **大容量、分布式日志存储**。

## ## 第二步：高层架构设计  

### 1. 先画出最小可运行系统（MVP）  

```
+-----------+        +-----------------+        +-----------+
|  Client   | <----> |  Lock Service   | <----> |  Single   |
|  SDK/API  | RPC/HTTP| (单机 KV + 业务逻辑) |  |  In‑mem  |
+-----------+        +-----------------+        |  Map/DB  |
                                                +-----------+
```

- **单机 KV**（如 `ConcurrentHashMap` 或 `Redis`）实现锁的原子写入/读取。  
- **优点**：实现快速，验证业务流程（加锁、解锁、TTL、可重入、续租）。  
- **缺点**：单点故障、容量受限、无法满足强一致性。  

### 2. 向高可用演进的目标体系结构  

```
+--------------------+          +---------------------------+          +-------------------+
|   Client SDK       |  gRPC/HTTP  |   Load Balancer /  Proxy   |  gRPC/HTTP  |   Raft KV Cluster   |
| (Java / Go / ...) | <------> | (Envoy / Nginx / L7)       | <------> | (3~5 节点, Leader+Followers) |
+--------------------+          +---------------------------+          +-------------------+
          |                                                            |
          |                                                            |
          |                                                            v
          |                                                  +-------------------+
          |                                                  |   Persistent Log  |
          |                                                  |   (e.g., Kafka)   |
          |                                                  +-------------------+
          |
          v
+-------------------+       +-----------------+      +-------------------+
| 监控 & 采集 (Prom) | <---> |  运维/告警系统   | <---> |  审计存储 (ClickHouse) |
+-------------------+       +-----------------+      +-------------------+
```

#### 关键组件解释  

| 组件 | 作用 | 设计要点 |
|------|------|----------|
| **Client SDK** | 为业务微服务提供 **加锁/解锁/续租/查询** 的调用封装 | - **唯一持有者 ID**（如 `clientId + threadId + UUID`）<br>- 自动重试、超时、阻塞等待实现 |
| **Load Balancer / Proxy** | 将请求均衡分发到 **Raft 集群的 Leader**（写）或 **Follower**（只读） | - 采用 **L7**（如 Envoy）实现 **健康检查**、**限流**、**熔断** |
| **Raft KV 集群** | 核心的 **强一致** 键值存储，负责锁元数据的 **原子写/读** | - 3~5 节点（多数写/多数读）<br>- Leader 负责写请求，Follower 可服务线性化读（Read‑Index） |
| **Persistent Log**（可选） | 将每一次锁操作写入 **不可变日志**，供审计/回放 | - 使用 **Kafka** 或 **Pulsar**，保留 30 天 |
| **审计存储** | 按天/分区写入 **ClickHouse**、**HBase** 或 **S3**，满足 10 TB | - 采用 **列式存储** 高效查询 |
| **监控 & 采集** | 暴露 **Prometheus** 指标（QPS、错误率、锁占用率、延迟等） | - 关键指标：`lock_acquire_latency`, `lock_hold_seconds`, `leader_changes` |
| **运维/告警系统** | 根据监控阈值自动告警、滚动升级 | - 与 **PagerDuty/Slack** 集成 |

#### 为什么要采用 **Raft** 而不是 **主从复制**？

- **强一致性**：Raft 在多数节点存活时保证线性化写入，防止出现“两个客户端都以为自己拿到锁”的情况。  
- **故障转移**：Leader 挂掉后，Raft 自动选举新 Leader，客户端只需重新定位即可。  
- **简洁性**：相对 Paxos，Raft 更易实现、易于调试。  

> 若使用 **主从异步复制**，在网络分区时可能出现 **脑裂**：两个节点各自成为 “主”，导致同一资源被两端同时加锁，违背强一致性要求。

## ## 第三步：数据库设计  

### 1. 锁元数据模型（存储在 Raft KV）  

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `key` | **String** | 业务资源唯一标识（如 `product:12345`） | `order:20230601:5678` |
| `owner_id` | **String** | 客户端唯一持有者标识（`clientId:processId:UUID`） | `svcA:pid1234:9f3c...` |
| `counter` | **Int** | 可重入计数，首次加锁为 1，后续加锁+1 | 3 |
| `expire_ts` | **Long** (epoch ms) | 锁失效时间戳（TTL） | `1685385600000` |
| `version` | **Int** | 乐观锁版本号，防止写冲突 | 5 |

- **键** (`key`) 直接作为 KV 存储的主键。  
- **值** 为上述结构的 JSON/二进制（如 Protobuf）序列化体。  

### 2. 持久化审计日志（写入 Kafka → ClickHouse）  

| 字段 | 类型 | 说明 |
|------|------|------|
| `event_time` | DateTime | 事件产生时间 |
| `event_type` | Enum(`ACQUIRE`, `RELEASE`, `RENEW`, `EXPIRE`) |
| `key` | String |
| `owner_id` | String |
| `counter` | Int |
| `ttl_ms` | Int |
| `node_id` | String | 产生日志的 Raft 节点 |

- **写入策略**：在 Raft 提交日志后 **同步**写入 Kafka，确保 **原子性**（先写 Raft，再写 Kafka）。  
- **查询**：运维可通过 ClickHouse 按时间、资源、持有者等维度查询。  

### 3. 读写路径的强一致性实现  

- **写路径**（加锁、解锁、续租） → **只能发往 Leader**。Leader 在 **日志复制**到多数节点后才返回成功，保证全局唯一。  
- **读路径**（查询锁状态） → **Read‑Index**：客户端发送 `ReadIndex` 请求到 Leader，Leader 返回当前的 **commit index**，随后客户端向任意副本发起读请求，副本只在 commit index ≥ 客户端拿到的 index 时返回，确保线性化读。

## ## 第四步：核心 API 设计  

> **设计原则**：**幂等**、**可重入**、**超时可控**、**统一错误码**。  

### 1. API 列表（使用 gRPC IDL 示例）  

```proto
syntax = "proto3";

package locksvc;

// 锁请求通用字段
message LockRequest {
  string key = 1;               // 资源键
  string client_id = 2;         // 客户端唯一标识
  int64 ttl_ms = 3;             // 期望 TTL（毫秒），0 表示使用默认
  bool block = 4;               // 是否阻塞等待（true）或立即返回（false）
}

// 加锁响应
message LockResponse {
  enum Status {
    OK = 0;
    LOCKED = 1;      // 已被其他客户端持有
    TIMEOUT = 2;     // 阻塞等待超时
    ERROR = 3;
  }
  Status status = 1;
  string owner_id = 2;          // 实际持有者（本次请求成功时返回）
  int64 expire_ts = 3;          // 锁失效时间戳（毫秒）
  int32 counter = 4;            // 当前可重入计数
}

// 解锁请求
message UnlockRequest {
  string key = 1;
  string client_id = 2;
}

// 解锁响应
message UnlockResponse {
  enum Status {
    OK = 0;
    NOT_OWNER = 1;   // 释放者不是持有者
    NOT_FOUND = 2;   // 锁不存在
    ERROR = 3;
  }
  Status status = 1;
}

// 续租请求
message RenewRequest {
  string key = 1;
  string client_id = 2;
  int64 ttl_ms = 3;   // 新的 TTL
}

// 续租响应
message RenewResponse {
  enum Status { OK = 0; NOT_OWNER = 1; NOT_FOUND = 2; ERROR = 3; }
  Status status = 1;
  int64 new_expire_ts = 2;
}

// 查询请求
message QueryRequest {
  string key = 1;
}
message QueryResponse {
  bool exists = 1;
  string owner_id = 2;
  int64 expire_ts = 3;
  int32 counter = 4;
}

// Service 定义
service LockService {
  rpc Acquire(LockRequest) returns (LockResponse);
  rpc Release(UnlockRequest) returns (UnlockResponse);
  rpc Renew(RenewRequest) returns (RenewResponse);
  rpc Query(QueryRequest) returns (QueryResponse);
}
```

### 2. SDK（伪代码）  

```go
type LockClient struct {
    svc locksvc.LockServiceClient
    clientID string // e.g. hostname:pid:uuid
}

// Acquire 带阻塞等待的实现
func (c *LockClient) Acquire(key string, ttl time.Duration, block bool) (LockHandle, error) {
    ctx, cancel := context.WithTimeout(context.Background(), ttl+5*time.Second)
    defer cancel()
    req := &locksvc.LockRequest{
        Key: key,
        ClientId: c.clientID,
        TtlMs: int64(ttl / time.Millisecond),
        Block: block,
    }
    for {
        resp, err := c.svc.Acquire(ctx, req)
        if err != nil { return nil, err }
        switch resp.Status {
        case locksvc.LockResponse_OK:
            return &LockHandle{key, resp.ExpireTs, resp.Counter}, nil
        case locksvc.LockResponse_LOCKED:
            if !block { return nil, ErrLockHeld }
            // 阻塞模式：短暂休眠后重试（指数退避）
            time.Sleep(10 * time.Millisecond)
            continue
        default:
            return nil, fmt.Errorf("acquire failed: %v", resp.Status)
        }
    }
}
```

- **幂等**：每次 `Acquire` 都携带 **相同 client_id**，若已持有锁会直接返回 `OK` 并递增计数。  
- **错误码统一**：`OK`, `LOCKED`, `NOT_OWNER`, `TIMEOUT`, `ERROR`，方便上层业务统一处理。

## ## 第五步：详细组件设计  

### 1. Raft KV 集群  

| 子模块 | 职责 | 关键实现点 |
|--------|------|-----------|
| **Raft Core** | 选举、日志复制、状态机提交 | 使用开源实现（etcd/Consul/Dragonboat），或自行实现基于 **Log Replication** 的协议 |
| **Lock State Machine** | 将 Raft 日志转换为锁元数据的增删改 | - **Command 类型**：`LOCK`, `UNLOCK`, `RENEW` <br>- **事务**：每条命令必须在 `apply` 时检查 **owner_id** 与 **counter** 的合法性 |
| **Snapshot** | 防止日志无限增长 | 周期性生成 **快照**（全量锁表），写入磁盘；恢复时先加载快照，再回放后续日志 |
| **Leader Forwarder** | 将写请求路由到 Leader | 代理层（如 `grpc-gateway`）检测 `NotLeader` 错误并自动重定向 |
| **Read‑Index Service** | 支持线性化只读 | 客户端发送 `ReadIndex` RPC，Leader 返回 `commitIdx`，随后客户端向任意节点发读请求并检查节点的 `appliedIdx` ≥ `commitIdx` |

#### 防止“锁漂移”  

- **TTL 由 Leader 完全控制**：所有写入必须先通过 Leader，TTL 统一在 Leader 时间（使用 **单调递增的逻辑时钟**）计算。  
- **续租只能在 Leader**：防止某个副本因时钟漂移误修改 TTL。  
- **心跳检测**：客户端在持锁期间每 `TTL/3` 发送一次 `Renew`，若长时间未续租，锁自然过期。  

### 2. 负载均衡 / Proxy  

- **L7 代理**（Envoy）负责 **健康检查**（Raft Leader/Followers 状态）并把 **写请求**定向到当前 Leader。  
- **写请求超时**：若 Leader 在一定时间未响应，代理返回 `UNAVAILABLE`，SDK 自动重试并重新获取最新 Leader（通过 **Raft 状态查询**）。  

### 3. 持久化审计流水线  

1. **Raft apply** 成功后，`LockStateMachine` 将操作封装成 `AuditEvent`。  
2. **同步写入** Kafka（`acks=all`），确保 **写入顺序** 与 Raft 日志保持一致。  
3. **Kafka Consumer** 按天分区写入 **ClickHouse** 表（列式压缩），支持 **秒级查询**。  

> **为什么使用 Kafka + ClickHouse** 而不是直接写入关系库？  
- **高吞吐**：Kafka 能轻松支撑 100k+ 事件/秒的持久化。  
- **离线分析**：ClickHouse 对大规模审计日志的聚合查询非常快（毫秒级），适合运维调试。  

### 4. 监控 & 告警  

| 指标 | 采集方式 | 报警阈值（示例） |
|------|----------|-----------------|
| `lock_acquire_qps` | Prometheus `counter`（每个节点） | > 60k/s（预警） |
| `lock_acquire_latency_ms` | `histogram`（99th percentile） | > 5 ms |
| `leader_changes_total` | Raft 内部计数 | > 3/小时 |
| `lock_hold_seconds` | 计数锁的持有时间分布 | 锁持有时间 > TTL*2 |
| `node_cpu_usage` / `memory_usage` | node_exporter | > 80% |

- **仪表盘**：展示当前活跃锁数量、热点资源 Top‑N、节点健康状态。

## ## 第六步：扩展性与高可用设计  

### 1. 水平扩容  

| 维度 | 扩容方式 | 关键点 |
|------|----------|--------|
| **写吞吐** | 增加 **Raft 集群节点数**（如从 3 增至 5） | 只要多数节点存活即可继续写，**写延迟略升**（多数投票） |
| **读吞吐** | **Read‑Index** 让 Followers 直接提供只读服务 | 读请求不走 Leader，显著提升 QPS；注意 **Read‑Index** 必须在 Leader 先返回 commitIdx |
| **审计存储** | 按天 **分区**，使用 **分布式 ClickHouse 集群** | 自动扩容节点，支持 PB 级数据 |
| **监控** | **Prometheus federation** 或 **Thanos** 集群 | 支持全局查询、长期存储 |

### 2. 容错与故障恢复  

| 场景 | 处理方式 | 关键机制 |
|------|----------|----------|
| **Leader 宕机** | **Raft 自动选举** → 新 Leader 产生；客户端 SDK 在收到 `NotLeader` 错误后重新定位 | 选举超时 `election_timeout`（150‑300 ms）保证快速恢复 |
| **网络分区** | **多数派存活** → 系统继续提供服务；少数派进入 **只读/不可写** 状态，返回 `UNAVAILABLE` 给客户端 | 防止脑裂，保持 **强一致性** |
| **磁盘故障** | **日志复制** 到多个磁盘/节点（RAID+跨机房复制） | 失效后可从其他副本恢复 |
| **时钟漂移** | **使用 Leader 时间** 计算 TTL，客户端不自行判断 | 消除因机器时间不一致导致的锁提前失效或延迟失效 |
| **单点故障（负载均衡器）** | **双活 Envoy** + **VIP**（keepalived）或 **云原生 Service** | 自动故障切换，不影响业务 |

### 3. 性能瓶颈与优化  

| 潜在瓶颈 | 解决方案 |
|----------|----------|
| **Raft 写延迟**（多数复制） | - 调整 **批量提交**（每 1‑2 ms 提交一次）<br>- 使用 **SSD**、**NVMe** 加速日志持久化 |
| **网络往返**（客户端 → LB → Leader） | - 部署 **近距离**（同机房）节点<br>- 使用 **HTTP/2** 或 **gRPC** 的 **多路复用** |
| **锁热点**（同一资源被频繁争抢） | - **自旋 + back‑off** 在 SDK 层降低请求频率<br>- 业务层 **分片** 锁键（如 `order:region:id`） |
| **监控采集**（高 QPS） | - **局部聚合**（每个节点本地统计后一次性上报）<br>- **Prometheus remote write** 到高性能时序库 |
| **审计写入** | - **Kafka 批量**（每 1 ms 或 1 KB）写入，减小网络开销<br>- 采用 **压缩**（Snappy）降低磁盘占用 |

## ## 第七步：常见面试追问与回答  

### 1. **一致性 vs 可用性**：如果采用主从复制而非 Raft，会出现哪些风险？如何权衡？  

- **风险**  
  - **脑裂**：网络分区后，主库和从库可能都认为自己是 “可写”，导致同一资源被两个客户端同时加锁，违背 **强一致性**。  
  - **读写不一致**：从库延迟同步，读到的锁状态可能是旧的，业务可能误以为锁已释放。  

- **权衡**  
  - **如果业务可以容忍偶尔的锁冲突**（例如非关键的缓存失效），可以使用 **主从异步复制**，获得更高的 **可用性**（写请求直接落到本地主库，分区时仍可写）。  
  - **如果业务对互斥要求极高**（库存扣减、订单号生成），必须保证 **线性化**，此时 **Raft**（或 Paxos）是必选，即使在网络分区时多数节点不可写也要牺牲可用性。  

### 2. **锁的失效与误删**：客户端异常宕机后，如何确保锁能够及时失效而不被误删？有哪些防止“锁漂移”的技术手段？  

| 手段 | 解释 |
|------|------|
| **TTL + 自动过期** | 锁在 KV 中存有 `expire_ts`，Raft 节点在每次读/写时检查时间，过期则自动删除。即使客户端崩溃，锁会在 TTL 结束后失效。 |
| **Leasable 锁（租约）** | 锁持有者必须周期性 **Renew**（续租），若客户端宕机则续租停止，锁自然过期。 |
| **单调递增的逻辑时钟** | 由 Leader 统一维护 `commit_index` 作为时间基准，避免不同节点因系统时间不同导致 TTL 失效时间不一致。 |
| **持有者唯一标识**（client_id+UUID） | 解锁/续租必须携带相同的 `owner_id`，即使有其它节点错误尝试删除，也会因 `owner_id` 不匹配被拒绝。 |
| **写入日志后再删除** | 解锁/过期操作在 Raft 日志提交后才执行，确保所有副本同步完成，防止 “锁漂移”。 |
| **Watchdog 线程** | 每个节点维护一个 **优先级队列**（按 `expire_ts` 排序），定时扫描并删除已过期锁，避免依赖客户端主动释放。 |

### 3. **性能瓶颈**：在 QPS 达到 50k/s 时，哪些环节最可能成为瓶颈？请说明如何通过水平扩展、请求合并、批处理等手段进行优化。  

| 环节 | 可能瓶颈 | 优化手段 |
|------|----------|----------|
| **Raft 写入**（日志复制） | 每条命令都要同步到多数节点 → 网络+磁盘 IO 成为瓶颈 | - **批量提交**：将多条加锁请求合并成一个日志条目（每 1‑2 ms）<br>- **增大网络带宽**（使用 10 Gbps 以上）<br>- **使用 NVMe SSD** 加速 WAL 持久化 |
| **网络往返**（客户端 → LB → Leader） | 单次往返 RTT 可能已接近 2 ms，累计影响延迟 | - **就近部署**（同机房）<br>- **使用 HTTP/2 + gRPC 多路复用** 减少握手次数 |
| **热点锁竞争** | 同一键频繁争抢导致大量重试 | - **指数退避 + 随机抖动** 在 SDK 层降低冲突概率<br>- **业务层分片**（如在业务键中加入 region、tenant） |
| **监控采集** | 高 QPS 时每个节点上报大量指标 | - **本地聚合**（每秒统计一次）后一次性推送至 Prometheus<br>- **使用 Pushgateway** 或 **remote write** 进行批量写入 |
| **审计日志写入** | 每次操作同步写 Kafka → 可能出现背压 | - **Kafka 异步批量**（`linger.ms=5`）<br>- **压缩**（Snappy）降低网络流量 |
| **锁状态查询**（只读） | 读取大量热点键导致 CPU 高占用 | - **Read‑Index** 让 Followers 承担读请求，分摊负载<br>- **缓存热点锁状态**（如本地 LRU）在节点内部短暂保存，降低磁盘/网络访问 |

## ## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策及思考过程  

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **选择 Raft 作为共识层** | - 需要权衡 **强一致性** 与 **可用性**。<br>- 实现或选型的复杂度（日志、快照、选举） | 1️⃣ 明确业务对锁的唯一性要求极高 → 只能接受 **线性化**。<br>2️⃣ 评估主从复制的脑裂风险 → 不可接受。<br>3️⃣ 调研已有开源 Raft 实现（etcd、Consul、Dragonboat），决定在面试中使用 **抽象概念** 而非自行实现细节。 |
| **如何实现可重入 + TTL + 续租 的原子性** | - 多个属性必须在同一次 **Raft 提交** 中原子更新，防止出现 “计数 + TTL 不一致”。<br>- 需要在 **状态机** 中处理 **幂等性**。 | 1️⃣ 将 **LockCommand** 设计为 **复合操作**（type, key, owner, ttl, deltaCounter）。<br>2️⃣ 在 `apply` 阶段先检查 `owner_id` 是否匹配，随后 **一次性** 更新 `counter` 与 `expire_ts`。<br>3️⃣ 若 `counter` 为 0，删除键；否则仅更新 TTL。 |

### 2. 新手最容易犯的错误（至少 2 条）  

1. **忽视时间同步导致的锁漂移**  
   - 直接使用本地系统时间计算 TTL，导致不同节点对同一锁的失效时间不一致。  
   - **纠正**：TTL 与失效时间统一由 **Raft Leader** 计算，客户端仅传递期望的 TTL 长度。  

2. **把锁的实现当成普通 CRUD，忽略幂等性**  
   - 直接 `SET key value NX`，在可重入或续租场景下会产生 **脏写**。  
   - **纠正**：在状态机里实现 **原子检查‑更新**（Check‑Then‑Set），并在 SDK 中使用 **相同 client_id** 进行重入计数。
