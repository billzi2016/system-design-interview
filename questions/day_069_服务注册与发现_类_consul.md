# 第 69 天：设计 服务注册与发现（类 Consul）

> 生成日期：2026-03-18

---

## 题目背景  
服务注册与发现系统（类似 Consul）用于在微服务架构中统一管理服务实例的注册、健康检查、元数据存储以及客户端的服务发现，帮助业务系统实现动态路由、负载均衡和容错。

## 面试场景设定  
> **面试官**：  
> “我们现在准备在公司内部搭建一套类似 Consul 的服务注册与发现平台，请你从零开始设计该系统。请先给出整体思路，之后我们会逐步深入讨论细节。”  

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **服务注册**：服务实例在启动时向注册中心上报自身的 IP、端口、协议、标签（如版本、地域）等信息。 |
| 2 | **健康检查**：支持主动（HTTP/TCP 心跳）和被动（报告状态）两种健康检查方式，能够及时剔除不可用实例。 |
| 3 | **服务发现**：客户端可以通过服务名称查询当前可用的实例列表，支持轮询、随机、基于权重的负载均衡策略。 |
| 4 | **键值存储** |提供一个轻量级的 KV 存储，用于保存配置、特性开关等元数据，支持 Watch 机制。 |
| 5 | **ACL 与鉴权**：对注册、查询、修改操作进行细粒度的访问控制，支持基于 token 的鉴权。 |
| 6 | **多数据中心/跨地域同步**：在不同可用区或地域之间保持注册信息的一致性，实现跨 DC 的服务发现。 |

## 非功能性需求  

| 指标 | 估算值 | 说明 |
|------|--------|------|
| **日活服务实例数（DAU）** | 150,000 | 包括所有微服务的实例（容器/VM） |
| **注册/注销 QPS** | 5,000 QPS | 峰值期间的注册、下线操作速率 |
| **查询 QPS** | 30,000 QPS | 客户端每秒发起的服务发现请求 |
| **平均响应时延** | ≤ 30 ms | 99.9% 请求在此时延内返回 |
| **可用性** | ≥ 99.95%（月可用） | 包括数据持久化、同步和 API 可用性 |
| **存储规模** | 约 10 TB | KV 数据 + 元数据的持久化需求（压缩后） |

## 系统边界  

**本题范围内**  
- 注册中心的 API 设计、数据模型、存储方案（持久化 + 内存缓存）  
- 健康检查机制的实现与调度  
- 服务发现的查询路径与负载均衡策略  
- 多数据中心的同步模型（如基于 Raft、 Gossip）  
- 鉴权、ACL、审计日志的基本实现  

**不考虑**  
- 客户端 SDK 的实现细节（仅需说明接口）  
- 完整的 UI 控制台（仅需提供 API 说明）  
- 对接外部监控系统（如 Prometheus） 的集成细节  
- 数据中心网络拓扑的具体部署（只需抽象模型）  

## 提示与追问  

1. **数据一致性**：如果使用 Raft 来实现多节点一致性，你会如何处理网络分区导致的脑裂问题？  
2. **扩容与容灾**：在注册中心节点数从 3 增加到 7 时，哪些组件需要重新设计或调优？  
3. **性能瓶颈**：面对查询 QPS 达到 30,000+ 时，你会怎样做缓存层的设计以及热点服务的负载均衡？

## 解题思路总览
1. **先做最小可用系统（MVP）**：只实现注册、注销、查询，使用单节点持久化 KV + 内存缓存。  
2. **逐步加入非功能需求**：健康检查、ACL、Watch、跨 DC 同步。每加一个特性，都要思考 **“是否会影响已有的性能/一致性/可用性？”**。  
3. **从单机到集群**：采用 **Raft** 保证强一致性（元数据、服务实例表），采用 **Gossip** 解决节点感知和跨 DC 同步。  
4. **缓存层的设计**：读多写少 → 把热点服务列表放到 **本地内存 + LRU**，并通过 **版本号/Watch** 保证缓存的及时失效。  
5. **负载均衡**：在查询 API 中直接返回 **加权随机/轮询** 的实例列表，客户端只做一次网络跳转。  

> **核心原则**  
- **先保证正确性**（强一致、事务），再追求 **高吞吐**（缓存、分片）。  
- **每一个新模块都要有清晰的边界**（接口、时序），便于以后单独扩容或替换。  
- **监控 & 日志** 必不可少：任何故障排查都依赖这些信息。

## 第一步：理解需求与规模估算  

| 需求编号 | 功能点 | 关键指标 | 业务价值 |
|---------|--------|----------|----------|
| 1 | 服务注册 | 注册/下线 QPS ≤ 5,000 | 动态扩容、弹性伸缩 |
| 2 | 健康检查 | 心跳频率 10 s / 30 s | 自动剔除故障实例 |
| 3 | 服务发现 | 查询 QPS ≤ 30,000，时延 ≤ 30 ms | 客户端快速定位可用实例 |
| 4 | KV 存储 | 10 TB（压缩后） | 配置、特性开关统一管理 |
| 5 | ACL & 鉴权 | 基于 token 的细粒度控制 | 安全、审计 |
| 6 | 跨 DC 同步 | 多 AZ/Region 间一致性 | 全球化业务、灾备 |

### 1.1 业务规模换算
- **日活实例数（DAU） = 150,000** → 假设每实例平均 **1 KB** 元信息（IP、端口、标签等），则 **元数据 ≈ 150 MB**，在内存中完全可放下（几百 MB）。  
- **注册/注销峰值**：5,000 QPS → 每秒约 5,000 条写请求，写入持久化层必须支持 **≥ 5,000 TPS**。  
- **查询峰值**：30,000 QPS → 读多写少，**读/写比例 ≈ 6:1**，说明缓存可以显著削峰。  
- **KV 存储 10 TB**：远大于实例元数据，需 **分片 + 多副本**（如使用 LSM‑Tree 存储引擎 RocksDB、TiKV）。  

### 1.2 关键非功能约束
| 约束 | 为什么重要 | 违背后果 |
|------|------------|----------|
| **强一致性**（注册、下线必须同步） | 防止出现“同一个实例被两次注册/未被及时剔除” | 客户端可能拿到不可用实例 → 业务故障 |
| **低时延**（≤30 ms） | 注册/发现是业务调用的前置，延迟直接放大到业务链路 | SLA 失效、用户体验差 |
| **高可用（≥99.95%）** | 注册中心是全局控制平面，一旦不可用整个微服务体系瘫痪 | 业务宕机、业务方投诉 |
| **可水平扩展** | 随着业务增长 QPS、实例数会继续上升 | 系统瓶颈、成本上升 |

## 第二步：高层架构设计  

### 2.1 从 MVP 开始的单体模型  

```
+-------------------+        +-------------------+
|   客户端 SDK/API  | <----> |  注册中心 (单节点) |
+-------------------+        +-------------------+
          |                         |
          |   HTTP/HTTPS (REST)    |
          v                         v
   +----------------+      +-----------------+
   |  持久化存储 (RDB) |    |  内存缓存 (Map) |
   +----------------+      +-----------------+
```

- **优点**：实现简单，快速验证业务流程。  
- **缺点**：单点故障、无法支撑 30k QPS、扩容困难。  

### 2.2 进入集群（核心）  

```
               +---------------------------+
               |      Load Balancer (L4)   |
               +------------+--------------+
                            |
          +-----------------+-----------------+
          |                                   |
+-------------------+               +-------------------+
|   API Gateway /   |               |   API Gateway /   |
|   HTTP Frontend   |               |   HTTP Frontend   |
+-------------------+               +-------------------+
          |                                   |
   +------+-------+                   +------+-------+
   |  Raft Leader | <---内部RPC--->   |  Raft Follower|
   +------+-------+                   +------+-------+
          |                                   |
   +------+-------+                   +------+-------+
   |  KV Store (RocksDB)  |           |  KV Store (RocksDB) |
   +----------------------+           +----------------------+
          |                                   |
   +------+-------+                   +------+-------+
   |  HealthCheck |   (Gossip)    |  HealthCheck |
   +--------------+----------------+--------------+
          |                                   |
   +------+-------+                   +------+-------+
   |  Watch/Notify|  (Pub/Sub)   |  Watch/Notify|
   +--------------+----------------+--------------+
```

**关键技术点**  

| 组件 | 作用 | 选型理由 |
|------|------|----------|
| **Load Balancer**（L4/L7）| 将请求均衡到集群中的任意前端节点| 解决入口的流量高并发，支持 DNS/IP 轮询 |
| **API Gateway / HTTP Frontend**| 负责解析 REST/gRPC、鉴权、限流| 业务层与内部 Raft、KV 解耦 |
| **Raft 集群**| 维护 **服务实例表**、**ACL、KV 元数据** 的强一致性| Raft 简单、成熟，天然提供 Leader 选举、日志复制 |
| **KV 存储 (RocksDB / TiKV)**| 持久化日志、快照、KV 数据| LSM‑Tree 适合写放大、压缩后可达 10 TB |
| **HealthCheck Scheduler**| 主动（HTTP/TCP）心跳、被动上报| 独立调度，避免阻塞主请求路径 |
| **Gossip**| 节点状态感知、跨 DC 同步元数据（弱一致）| Gossip 传播快、对网络分区容忍 |
| **Watch/Notify**| KV/服务变更推送给客户端| 实现配置热更新、服务列表自动刷新 |

> **为什么要把健康检查、Watch、Gossip 单独拆出来？**  
- **职责单一**，防止 API 前端因为慢检查卡住请求。  
- **水平扩容**：这些组件可以根据业务负载单独扩容（比如健康检查需要大量网络 I/O）。  

### 2.3 跨数据中心的同步模型  

```
DataCenter A                         DataCenter B
+-------------------+                +-------------------+
|   Raft Cluster A  | <---Gossip---> |   Raft Cluster B  |
+-------------------+                +-------------------+
```

- **Raft** 在每个 DC 内部保持强一致性（本地写入 → 本地多数节点提交）。  
- **Gossip** 负责把 **元数据变更的摘要（版本号+hash）** 传播到其他 DC，**按需拉取**（类似 Consul 的 WAN sync）。  
- **冲突解决**：使用 **时间戳 + 版本向量**，冲突概率极低；如果冲突则 **保留两侧最新版本**，并在审计日志中记录。

## 第三步：数据库设计  

### 3.1 实体模型（简化版）  

| 表/集合 | 主键 | 关键字段 | 备注 |
|--------|------|----------|------|
| **services** | `service_id` (UUID) | `name`, `tags`, `metadata` | 服务的抽象信息（唯一名称） |
| **instances** | `(service_id, instance_id)` | `ip`, `port`, `protocol`, `version`, `region`, `weight`, `status`, `last_heartbeat` | 每个实例的注册信息 |
| **kv_store** | `key` (string) | `value` (blob), `modify_index`, `create_index`, `session_id` | 支持 **Watch**、TTL、事务 |
| **acl_tokens** | `token_id` | `policy_id`, `expiry`, `description` | token 与权限映射 |
| **policies** | `policy_id` | `rules` (JSON) | 细粒度 ACL 规则 |
| **sessions** | `session_id` | `owner`, `ttl`, `create_time` | 用于 KV 锁、Watch |

> **为什么使用复合主键 `(service_id, instance_id)` 而不是单独的 `instance_id`？**  
- 能够快速定位同一服务下的所有实例（常用查询是 `WHERE service_id = ?`），避免全表扫描。  

### 3.2 存储选型对比  

| 需求 | 关系型 (MySQL/PostgreSQL) | KV + LSM (RocksDB/TiKV) | 说明 |
|------|--------------------------|--------------------------|------|
| 强事务（注册/下线） | ✅ 完整 ACID | ✅ 通过 Raft 日志实现 | 两者均可，后者更易水平扩展 |
| 大容量 KV（10 TB） | ❌ 单机容量受限，扩容复杂 | ✅ 分片、压缩、冷热分层 | 选 KV 引擎更合适 |
| 读取热点（服务列表） | ✅ 索引快速 | ✅ 内存缓存+LSM 读取快 | 两者均可，需配合缓存 |
| 多 DC 同步 | ❌ 跨库复制成本高 | ✅ Gossip + Raft 组合 | 更符合 Consul 的设计思路 |

**最终决定**：  
- **服务实例、ACL、KV 元数据** → **Raft + RocksDB**（内部持久化）。  
- **大容量 KV（如配置文件）** → **TiKV / HBase**（可做外部集群，后期按需拆分）。  

### 3.3 数据持久化与快照  

- **日志写入**：Raft 每条命令写入 **WAL**，随后同步到 **RocksDB**。  
- **快照**：当日志长度 > 10 万条或磁盘占用 > 1 GB 时，触发 **快照**，将当前状态序列化为 **snapshot 文件**（压缩后存储），并在新节点加入时用于快速同步。  

> **不做快照的后果**：日志无限增长，导致恢复时间线性增长，磁盘耗尽，节点重启慢 → 违背高可用要求。

## 第四步：核心 API 设计  

> **统一使用 REST + JSON**（可选 gRPC），因为大多数内部工具和运维脚本更熟悉 HTTP。  

### 4.1 注册/下线  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| `PUT` | `/v1/agent/service/register` | `{ "Name":"order-service", "ID":"order-1", "Address":"10.1.2.3", "Port":8080, "Tags":["v1","region=sh"], "Meta":{"weight":10}, "Check":{"TTL":"30s"}}` | `200 OK` | **幂等**：同一 `ID` 再次注册会覆盖。 |
| `PUT` | `/v1/agent/service/deregister` | `{ "ID":"order-1" }` | `200 OK` | 立即下线，触发健康检查失效。 |

### 4.2 健康检查  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| `PUT` | `/v1/agent/check/pass` | `{ "CheckID":"service:order-1" }` | `200 OK` | 被动上报成功。 |
| `PUT` | `/v1/agent/check/fail` | `{ "CheckID":"service:order-1", "Output":"timeout" }` | `200 OK` | 主动标记为失败。 |
| `GET` | `/v1/health/checks/{service}` | – | `[{ "CheckID":"service:order-1", "Status":"passing"}]` | 查询健康状态。 |

> **主动检查**（Scheduler）内部会每 `interval` 调用对应的 HTTP/TCP 检查，**如果连续 N 次失败** → 自动调用 `fail` API 并下线实例。  

### 4.3 服务发现  

| 方法 | URL | 参数 | 响应 | 说明 |
|------|-----|------|------|------|
| `GET` | `/v1/catalog/service/{service}` | `?tag=v1&near=_agent&filter=weight>5` | `[ { "Node":"10.1.2.3", "Address":"10.1.2.3", "Port":8080, "Weight":10 } ]` | 支持 **标签过滤**、**近邻优先**、**权重过滤**。 |
| `GET` | `/v1/catalog/services` | – | `{ "order-service":["v1","region=sh"], "pay-service":["v2"] }` | 列出所有服务名称及标签。 |

- **负载均衡实现**：在 API Server 中 **先读取实例列表 → 根据请求头中的 `X-Hash`（或随机）选择一个实例** → 返回 **单实例**（`/v1/catalog/service/{service}?healthy=true&lb=weighted_random`）。  

### 4.4 KV 存储  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| `PUT` | `/v1/kv/{key}` | `{ "Value":"base64", "Flags":0, "Session":"<session-id>", "TTL":"60s" }` | `200 OK` | 支持 **事务**、**TTL**、**锁**（通过 Session）。 |
| `GET` | `/v1/kv/{key}` | – | `{ "Value":"base64","ModifyIndex":12345 }` | 支持 **递归** `?recurse`。 |
| `GET` | `/v1/kv/{key}?wait=5s&index=12345` | – | **阻塞返回**（长轮询） | 实现 **Watch**，客户端可等待变更。 |

### 4.5 ACL & 鉴权  

| 方法 | URL | 请求体 | 响应 | 说明 |
|------|-----|--------|------|------|
| `PUT` | `/v1/acl/create` | `{ "Name":"svc-token", "Policy":"service-read", "TTL":"24h" }` | `{ "Token":"abc123" }` | 生成 **token**，返回给服务端。 |
| `GET` | `/v1/acl/token/{id}` | – | `{ "Policy":"service-read","Expiry":"2026-06-01" }` | 查询 token 信息。 |

- **所有 API** 必须在 **HTTP Header** `X-Consul-Token` 中携带 token，**网关**负责 **校验**（缓存 token → 权限树），未通过则返回 `403 Forbidden`。

## 第五步：详细组件设计  

### 5.1 API Gateway / HTTP Frontend  

- **技术选型**：Nginx + Lua（OpenResty）或 Envoy + WASM。  
- **职责**：  
  1. **路由**：根据 URL 把请求转到对应的内部服务（注册、查询、KV）。  
  2. **限流**：对注册/查询 QPS 做 **令牌桶**，防止突发流量压垮 Raft。  
  3. **鉴权**：读取 `X-Consul-Token` → 调用 **ACL Service** 本地缓存 → 决策是否放行。  
  4. **审计日志**：记录每条请求的 `token、method、path、status、latency`，写入 **ELK**。  

### 5.2 Raft Leader / Follower  

- **实现**：使用开源 **etcd/raft** 库（Go）或自行实现（Java）。  
- **日志结构**（简化）：  

```
type LogEntry struct {
    Index uint64
    Term  uint64
    Cmd   []byte   // protobuf: Register/Unregister/PutKV/ACL
}
```

- **提交路径**：  
  1. **API Server** 将请求序列化为 **Cmd**，调用 `raft.Propose(entry)`。  
  2. **Leader** 将日志写入本地 WAL → 同步复制给多数 **Follower**。  
  3. **多数确认**后，执行 **apply**：更新 **RocksDB**、更新 **内存缓存**、触发 **Watch**。  

- **为何使用 Raft 而不是单纯的主从复制？**  
  - **一致性保证**：Raft 在多数节点存活的情况下，**永远不会出现脑裂**。  
  - **自动选举**：节点故障后自动选出新 Leader，**无需外部选举服务**。  

### 5.3 健康检查调度器  

- **架构**：独立的 **Scheduler Service**（多实例，基于 **cron** / **timer wheel**）。  
- **流程**：  
  1. **读取**所有 **需要主动检查** 的实例列表（从内存缓存中获取）。  
  2. 按 **检查类型**（HTTP/TCP）并发发送请求（使用 **协程/线程池**），**超时阈值** = `interval / 2`。  
  3. 根据返回结果调用 **API** `PUT /v1/agent/check/pass|fail` → 实际写入 Raft。  
  4. **失败计数**：连续 `N` 次失败 → 标记实例 `Status=critical` → 自动 **deregister**（可配置）  

- **被动检查**：服务在业务代码里调用 `PUT /v1/agent/check/pass`，**不需要调度器**。  

### 5.4 Watch & 事件推送  

- **实现方式**：**Long Polling**（HTTP） + **Server‑Sent Events (SSE)** 或 **WebSocket**。  
- **核心概念**：  
  - **ModifyIndex**：每次 KV/服务变更都会递增。客户端传入上一次的 `index`，如果当前 `index` 大于传入值，则立即返回；否则 **阻塞**（最长 5 min）直至有变更或超时。  
- **内部实现**：  
  - **Watcher Manager** 按 `key` / `service` 建立 **等待队列**（Channel）。  
  - **Apply Thread** 在每次状态变更后 **遍历对应等待队列**，向所有挂起的请求写入新 `index` → 客户端收到通知。  

### 5.5 本地缓存层  

- **结构**：`map[string]*CacheEntry` + **LRU**（容量约 500 MB）。  
- **CacheEntry**：  
  ```go
  type CacheEntry struct {
      Data      interface{} // ServiceInstanceList or KV value
      Index     uint64      // Raft modify index
      ExpireAt  time.Time   // TTL for KV entries
  }
  ```
- **失效策略**：  
  1. **主动失效**：收到 **Watch** 推送后，比较 `Index`，若更大则 **更新/删除** 本地缓存。  
  2. **TTL**：KV 带 TTL 的条目自动过期。  
  3 **内存回收**：LRU 淘汰最少使用的键。  

- **为何不直接走 DB？**  
  - 读取 QPS 30k → 单次 DB I/O 成本太高，**缓存** 能把时延降到 **1 ms** 级别，满足 30 ms SLA。  

### 5.6 跨 DC 同步（WAN）  

| 步骤 | 说明 |
|------|------|
| **摘要发布** | 每个 DC 的 Raft Leader 每 `interval`（默认 10 s）生成 **Digest**：`{ServiceID, Version, Hash}` 并通过 **Gossip** 广播到其他 DC。 |
| **冲突检测** | 接收端比较本地 `Version` 与 `Digest.Version`，若不一致则请求 **差异拉取**。 |
| **差异拉取** | 使用 **gRPC** 拉取缺失的 **Log Entries**（仅元数据，不包括实例的心跳状态），并 **本地 Replay** 到 Raft。 |
| **冲突解决** | 若同一 `ServiceID` 同时在两个 DC 被修改（极少），取 **更大的 Version**（单调递增）或 **更晚的 Timestamp**。冲突记录到 **审计日志**。 |

> **Gossip vs. Raft**：  
> - **Raft** 适合 **局域网**（低延迟）内的强一致；  
> - **Gossip** 适合 **跨 WAN**（高延迟、可能分区），只需要 **最终一致** 即可。

## 第六步：扩展性与高可用设计  

### 6.1 水平扩容路线图  

| 阶段 | 目标 | 关键改动 |
|------|------|----------|
| **1. MVP（单节点）** | 验证功能 | - 单进程 + 本地 RocksDB |
| **2. 3‑节点 Raft 集群** | 防止单点故障、提供强一致 | - 引入 Raft 库、Leader 选举<br>- Load Balancer 前置 |
| **3. 5‑7 节点** | 提升容错（最多 2 节点故障） | - 调整 **Quorum**（N/2+1）<br>- 监控 Leader 负载，必要时 **手动转移** |
| **4. 多 DC 部署** | 跨地域容灾、就近访问 | - 每个 DC 部署独立 Raft 集群<br>- 加入 Gossip 同步层 |
| **5. 动态分片 KV** | 支撑 10 TB KV 数据 | - 将 KV 拆分为 **多个 TiKV 实例**（水平分片）<br>- 使用 **PD（Placement Driver）** 管理元数据 |
| **6. 读写分离** | 进一步提升 QPS | - 在 Raft 前加入 **Read‑Only Replica**（仅接受查询）<br>- 通过 **Stale‑Read** 机制提供 **微弱一致性**（可选） |

### 6.2 容灾与故障恢复  

| 故障场景 | 自动化处理 | 人工介入 |
|----------|------------|----------|
| **Leader 崩溃** | **Raft** 自动选举新 Leader（多数节点存活） | 若多数不可用，需要手动恢复（恢复快照） |
| **网络分区（脑裂）** | Raft **多数节点** 继续工作，少数节点进入 **follower** 状态并 **拒绝写入** | 检查网络，必要时手动 **重新加入** |
| **磁盘满** | 写入阻塞 → 触发 **快照** + **日志压缩**；若仍满，报警并 **只读模式** | 扩容磁盘或迁移老数据 |
| **跨 DC 同步中断** | 本地 DC 仍保持 **强一致**，对外提供 **本地视图**；Gossip 会在网络恢复后 **自动恢复** | 监控 WAN 链路，必要时手动 **触发同步** |
| **节点硬件故障** | 通过 **负载均衡** 将流量转移到健康节点 | 替换故障机器，加入新节点进行 **日志复制** |

### 6.3 性能优化点  

1. **批量写入**：注册/下线请求可在 **Gateway** 层进行 **批量合并**（每 10 ms 合并一次），一次性提交多条日志，降低 Raft 提交次数。  
2. **热点服务缓存**：对访问频率最高的 **前 100** 服务，使用 **本地热点缓存**（复制到每个 API 实例），并通过 **Watch** 保持同步。  
3. **异步健康检查**：检查结果先写入 **本地内存**，只在 **阈值突破** 时才写入 Raft，减少写放大。  
4. **压缩日志**：使用 **Snappy** 对 Raft 日志进行压缩，降低网络传输成本。  
5. **限流 & 熔断**：对注册/注销请求设置 **QPS 上限**，当超过阈值时返回 `429 Too Many Requests`，保护后端。

## 第七步：常见面试追问与回答  

### 1️⃣ 数据一致性：Raft 脑裂如何处理？  
- **Raft 天然防止脑裂**：只有 **多数节点**（> N/2）能够提交日志。即使网络分区导致 **少数节点** 与多数节点互相通信中断，少数节点会 **保持 follower 状态**，不接受写入，直至重新与多数节点建立联系。  
- **处理流程**：  
  1. 分区后，**多数** 继续工作，选出 Leader。  
  2. **少数** 会在 **Election Timeout** 超时后尝试发起选举，但因为无法得到多数投票，选举失败，仍保持 follower。  
  3. 当网络恢复，少数节点会 **自动快照同步**，追上多数的日志。  
- **如果真的出现“多数不可达”**（例如 2/3 节点挂掉），整个集群会进入 **不可写状态**（只读），这时业务层需要 **降级或缓存**，而不是继续写入导致不一致。  

### 2️⃣ 扩容从 3 节点到 7 节点，需要重新设计哪些组件？  
| 组件 | 需要改动 | 原因 |
|------|----------|------|
| **Raft 集群配置** | 更新 `cluster.conf` 中的 `node IDs`、`peer URLs`，并 **重新启动** 节点进行 **joint consensus**（两阶段添加） | 新节点必须加入日志复制链路 |
| **负载均衡/服务发现** | 将 **新节点的 IP** 加入 LB 配置，确保流量可以均匀分配 | 防止单点流量过载 |
| **监控告警** | 添加新节点的 **心跳、日志延迟、磁盘使用** 指标 | 及时发现新节点异常 |
| **Gossip 网络** | 更新 **WAN/LAN Peer List**，让 Gossip 知道新节点 | 跨 DC 同步需要完整的成员列表 |
| **快照/日志压缩** | 若原有节点日志较大，**先做快照**再添加节点，避免一次性同步大量日志 | 加速节点加入过程 |
| **客户端 SDK**（如果有） | 更新 **集群地址列表**，确保 SDK 能感知新增节点 | 防止写请求只能打到旧节点 |  

> **注意**：Raft **不需要**重新划分数据分片，因为它本身是 **全局日志复制**；如果后期采用 **分片 KV**（TiKV），则需要 **重新平衡分片**（PD 自动完成）。  

### 3️⃣ 查询 QPS 达 30,000+ 时，缓存层怎么设计？热点服务怎么负载均衡？  
1. **多级缓存**  
   - **本地 L1**（每个 API 实例） → **进程内 map + LRU**（热点 5% 服务）。  
   - **L2 共享缓存**（如 Redis Cluster） → 用于 **非热点**，可统一失效。  
2. **缓存失效**  
   - **Watch 机制**：服务表变更时，Raft **apply** 阶段会向 **Watch Manager** 推送 `ModifyIndex`，对应的 **缓存键** 被标记为 **stale**。  
   - **TTL**：对 KV 项目设定合理的 **TTL**（如 60 s），防止长时间失效。  
3. **负载均衡实现**  
   - **在查询 API** 中直接完成 **加权随机**（Weighted Random）或 **一致性哈希**（对同一客户端保持会话黏性）。  
   - **实例权重**来源于 **Meta.weight** 字段，默认 `1`。  
   - **轮询**：如果客户端要求轮询，可返回 **完整实例列表**，客户端自行轮询。  
4. **热点服务的特殊处理**  
   - 对 **访问频率最高的 100 条服务**（可通过计数器监控）采用 **复制缓存**：每个 API 实例都持有完整副本，且 **每次更新只广播一次**（通过内部 Pub/Sub），避免热点查询触发跨节点网络。  

### 4️⃣ 如何保证 KV 的强一致性与最终一致性的平衡？  
- **核心 KV（ACL、服务元数据）** 使用 **Raft** → **强一致**。  
- **大容量、读多写少的 KV（如配置文件）** 可以 **在 Raft 基础上做二级缓存**：  
  - 写入时走 Raft → **同步写入** 到 **TiKV**（分片 KV） → **异步复制** 到 **跨 DC**。  
  - 读取时先查询 **本地缓存** → 若 **stale**（Index 旧）再走 **Raft** 或 **TiKV**。  
- 通过 **版本号（ModifyIndex）** 与 **Watch**，客户端能够感知 **何时需要重新拉取**，从而在 **强一致** 与 **高吞吐** 之间取得折中。

## 心得与反思  

### 1️⃣ 本题最难的设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **Raft + Gossip 双层一致性模型** | 需要在 **强一致** 与 **跨地域容灾** 之间找到合适的边界。 | - 首先评估业务对 **服务注册** 的一致性要求（必须强） → 选 Raft。 <br>- 再评估 **跨 DC** 的容忍度（可以是最终一致） → 选 Gossip。 <br>- 通过 **版本号 + 哈希** 的方式让两层模型协同工作，避免冲突。 |
| **缓存失效策略（Watch vs TTL）** | 缓存若失效不及时会导致 **客户端拿到已下线实例**，影响业务可用性。 | - 设计 **Watch** 机制，让每次写操作都产生 **ModifyIndex**，并在 **Apply** 阶段推送。 <br>- 为防止 **Watch 丢失**（网络抖动），再加上 **TTL** 作为兜底。 <br>- 通过 **压测** 确认失效延迟 ≤ 5 ms。 |

### 2️⃣ 新手最容易犯的错误（至少两条）  

1. **忽视强一致性需求，直接使用 Eventually‑Consistent KV**  
   - 症状：服务下线后仍被发现 → 请求被路由到不可用实例 → 故障放大。  
   - 纠正：先把 **注册/注销** 放到 **Raft**，只把 **非关键配置** 放到弱一致层。  

2. **把健康检查写在同步路径里，导致写放大**  
   - 症状：每次心跳都触发 Raft 复制 → 产生大量日志，导致延迟升高。  
   - 纠正：**健康检查** 结果先在本地缓存，只有 **状态变更**（从 healthy→unhealthy）才写入 Raft。  

### 3️⃣ 学习建议与可延伸方向  

| 方向 | 推荐学习资源 | 重点 |
|------|--------------|------|
| **分布式一致性算法** | 《Designing Data‑Intensive Applications》章节、etcd Raft 源码 | 理解 Leader 选举、日志复制、快照、网络分区处理 |
| **Gossip 协议** | 《Gossip Protocols》论文、Consul 官方文档 | 了解成员感知、状态传播、冲突解决 |
| **键值存储底层实现** | RocksDB / TiKV 文档、LSM‑Tree 原理 | 学会读写放大、压缩、分片 |
| **高性能网络编程** | 《Linux 高性能服务器编程》、Go net/http & gRPC 性能调优 | 处理 30k QPS、低时延的关键点 |
| **监控与可观测性** | Prometheus + Grafana、OpenTelemetry | 实时监控 Raft 延迟、健康检查成功率 |

> **实战 tip**：  
> 1. 先在本地实现一个 **单节点** 版的 **注册/查询**（使用内存 + RocksDB），写完后再逐步加入 **Raft**（etcd/raft 包），最后再加 **Gossip**（memberlist 库）。  
> 2. 每加入一个新模块，都 **写单元测试 + 性能基准**，确保 **时延 ≤ 30 ms**，并记录 **吞吐**、**资源占用**，形成完整的 **benchmark 报告**，面试时可以直接展示。
