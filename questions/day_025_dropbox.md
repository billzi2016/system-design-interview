# 第 25 天：设计 Dropbox

> 生成日期：2026-05-01

---

## 题目背景  
Dropbox 是一款面向个人和团队的云存储与同步服务，用户可以通过桌面客户端、移动端或网页上传、下载、分享文件，并在多设备间保持文件实时同步。

## 面试场景设定  
**面试官**：  
> “假设我们要重新设计 Dropbox 的核心文件存储与同步系统，使其能够支撑全球数亿用户的日常使用。请从零开始设计该系统的架构，重点考虑高可用、高并发以及数据一致性。我们先从需求入手，你可以先列出核心功能和非功能需求吗？”

## 功能性需求  

| 编号 | 功能描述 |
|------|----------|
| 1 | **文件上传/下载**：用户可以将任意大小（上限 10 GB）的文件上传到云端，也可以随时下载到任意设备。 |
| 2 | **跨设备实时同步**：同一用户在任意设备上对文件的增删改操作，需在其他已登录设备上几秒内完成同步。 |
| 3 | **共享链接与权限控制**：用户可以为文件或文件夹生成可公开或受密码保护的共享链接，并支持只读、可编辑等权限。 |
| 4 | **版本历史**：对每个文件保留至少 30 天的版本记录，用户可以随时恢复到历史版本。 |
| 5 | **冲突解决**：当多设备并发编辑同一文件时，系统能够检测冲突并生成冲突副本供用户手动合并。 |
| 6 | **离线编辑与同步**：客户端支持离线状态下的文件增删改，恢复网络后自动同步并处理冲突。 |

## 非功能性需求  

| 指标 | 目标值 | 说明 |
|------|--------|------|
| DAU（日活跃用户） | 5 × 10⁷ | 假设全球活跃用户约 5 亿，活跃比例 10%。 |
| QPS（上传/下载请求） | 2 × 10⁵ QPS | 峰值时每秒约 20 万次文件上传/下载请求（平均文件 5 MB）。 |
| 同步延迟 | ≤ 5 秒 | 文件在源设备完成写入后，目标设备可见的时延不超过 5 秒（95% 以上请求）。 |
| 可用性 | 99.99%（每月累计宕机 < 4.38 小时） | 关键业务需高度可用。 |
| 存储容量 | 约 2 EB（exabyte） | 预计每用户平均存储 20 GB，5 × 10⁸ 用户 → 10 EB；考虑压缩、冗余后约 2 EB 有效存储需求。 |
| 带宽峰值 | 约 15 Tbps | 依据 QPS 与平均文件大小估算的上行/下行带宽需求。 |

## 系统边界  

**本题范围（需要设计）**  
- 文件的持久化存储（对象存储、分块、冗余）。  
- 元数据服务（文件目录、权限、版本、共享链接）。  
- 同步协议与冲突检测/解决机制。  
- API 网关与鉴权、速率限制。  
- 客户端与服务器之间的增量同步算法。  

**不在本题范围（可不考虑细节）**  
- 客户端 UI/交互细节（桌面、移动、网页）。  
- 文档预览、搜索、OCR、协同编辑等高级功能。  
- 计费、企业组织结构、审计日志等业务层面。  
- 第三方集成（如 Office 365、Google Docs）。  
- 区域性法规合规（GDPR、数据本地化）实现细节。  

## 提示与追问  

1. **数据一致性**：如果我们采用多副本存储，如何在保证高可用的同时实现“读已提交”（Read‑After‑Write）一致性？  
2. **冲突检测**：请详细说明在多设备并发写入同一文件时，服务器如何检测冲突并生成冲突副本的完整流程。  
3. **扩展性**：当每日活跃用户从 5 × 10⁷ 增长到 2 × 10⁸ 时，哪些组件需要水平扩容？请给出扩容策略与可能的瓶颈。

# 题解

# Dropbox 核心文件存储与同步系统设计  
*面向完全没有系统设计经验的同学，手把手从需求出发、逐层演进到可支撑全球数亿用户的高可用分布式系统。每一步都解释「为什么」以及不这么做会出现什么问题。*  

---

## 解题思路总览
1. **先把需求写清楚** → 功能、非功能、边界。  
2. **做容量 & 吞吐估算**，把抽象的「5 × 10⁷ DAU」转化为具体的 QPS、存储、网络带宽。  
3. **画最小可运行系统（MVP）**：只需要一个对象存储 + 一个元数据库 + 简单的 API 网关。  
4. **逐步加入高可用、水平扩展、容灾、缓存、异步化**，每加一个组件都要说明它解决了什么瓶颈。  
5. **细化到每个核心 API、数据模型、同步协议**，确保功能完整且易于实现。  
6. **准备面试追问**（一致性、冲突检测、扩容策略）。  
7. **总结经验教训**，帮助新手避免常见坑。

> **关键思路**：先把「能跑通」的最小系统搭好，再围绕 **可用性 / 可扩展性 / 一致性** 三大非功能需求逐层强化。这样既能让面试官看到你对业务的把握，又能展示你对分布式系统核心原则的理解。

## 第一步：理解需求与规模估算

### 1. 功能需求拆解

| 编号 | 功能 | 对系统的直接影响 |
|------|------|-------------------|
| 1 | 文件上传/下载（≤10 GB） | 大流量的 **对象存储**、**分块传输**、**带宽**。 |
| 2 | 跨设备实时同步 | **事件推送**、**变更日志**、**低延迟**。 |
| 3 | 共享链接 & 权限 | **细粒度 ACL**、**签名 URL**、**安全审计**。 |
| 4 | 版本历史（≥30 天） | **元数据增长**、**增量存储**、**恢复 API**。 |
| 5 | 冲突解决 | **冲突检测**、**冲突副本**、**客户端 UI**（但服务端要提供副本管理）。 |
| 6 | 离线编辑 & 同步 | **本地缓存 → 队列**、**幂等写入**、**重放机制**。 |

> **注意**：功能 2、5、6 强调 **同步** 与 **冲突**，这决定我们需要 **变更日志 + 事件流**，而不是单纯的「上传后立即返回成功」模型。

### 2. 非功能需求量化

| 指标 | 目标值 | 计算方式 & 说明 |
|------|--------|-----------------|
| **DAU** | 5 × 10⁷ | 业务规模的基准，用来估算并发用户数。 |
| **QPS（上传/下载）** | 2 × 10⁵ | 峰值 20 万次请求/秒，假设 70% 下载、30% 上传。 |
| **平均文件大小** | 5 MB | 便于换算带宽（QPS × size）。 |
| **带宽峰值** | ≈ 15 Tbps | 2 × 10⁵ req × 5 MB ≈ 1 PB/s = 8 Tbps（上传+下载），留 2×安全裕度。 |
| **同步延迟** | ≤ 5 秒（95%） | 需要 **实时推送** + **快速写入**。 |
| **可用性** | 99.99% | 每月 < 4.38 h 故障，总体容错必须做到 **多AZ、多Region**。 |
| **存储容量** | 2 EB 有效 | 假设 20 GB/用户 × 5×10⁸ 用户 = 10 EB 原始，压缩/去重后约 2 EB。 |

### 3. 关键瓶颈预判

| 资源 | 估算量 | 潜在瓶颈 |
|------|--------|----------|
| **对象存储** | 2 EB + 高吞吐（15 Tbps） | 网络 I/O、磁盘写入、分块合并。 |
| **元数据服务** | 约 10⁹ 条记录（文件/目录/版本） | 单点 DB 读写压力、锁竞争。 |
| **API 网关** | 2 × 10⁵ RPS | 需要水平扩容、限流、鉴权。 |
| **同步事件系统** | 每秒 ≤ 2 × 10⁵ 变更事件 | 消息队列吞吐、消费者延迟。 |
| **缓存层** | 热点元数据/共享链接 | 缓存击穿、失效一致性。 |

## 第二步：高层架构设计

### 1. MVP（最小可运行系统）

```
+-------------------+       +-------------------+
|   客户端 (Web/PC) | <---> |   API Gateway    |
+-------------------+       +-------------------+
                                 |
               +-----------------+-----------------+
               |                                   |
      +--------v--------+                  +-------v-------+
      |   Auth Service  |                  |   RateLimiter |
      +-----------------+                  +---------------+
               |
      +--------v--------+       +-------------------+
      |  Metadata DB    | <---> |  Object Storage   |
      +-----------------+       +-------------------+
```

- **API Gateway**：统一入口，做鉴权、限流、日志。  
- **Auth Service**：基于 OAuth2/JWT，生成访问令牌。  
- **Metadata DB**：关系型或分布式键值，用来保存目录树、权限、版本等。  
- **Object Storage**：如自研的分块对象系统或直接使用 S3‑compatible。  

**为什么这么简单？**  
- 能快速实现 **上传/下载**、**元数据查询**。  
- 只要保证单节点磁盘和网络够大，就可以在实验环境跑通。  

**不这么做的后果**  
- 没有冗余、单点故障会导致全部服务不可用，面试官会直接扣分。

### 2. 逐步加入的关键组件（从左到右、从下到上）

| 目标 | 新增组件 | 作用 & 解决的问题 |
|------|----------|-------------------|
| **高可用** | 多实例 API Gateway + **负载均衡(LB)** | 防止单点崩溃，流量分发。 |
| **容灾** | **多 AZ/Region** 对象存储 + **复制**（跨 AZ 3 副本） | AZ 故障时仍能读取/写入。 |
| **可扩展** | **Metadata Service** 采用 **分布式键值（如 DynamoDB、TiDB）** + **读写分离** | 解决单库写热点。 |
| **低延迟同步** | **变更日志（Change Log）** + **消息队列（Kafka / Pulsar）** + **推送服务（WebSocket / FCM）** | 5 秒内把变更推送到所有在线设备。 |
| **冲突检测** | **乐观锁 + 版本号（etag）** + **冲突副本表** | 同步时检测并生成冲突副本。 |
| **离线/增量同步** | **本地操作队列 → 服务器端幂等写入** + **分块校验（hash）** | 网络恢复后安全重放。 |
| **缓存** | **CDN**（对象下载） + **Redis**（热点元数据） | 大幅降低对象存储与 DB 的读负载。 |
| **监控/告警** | **Prometheus + Grafana** + **日志系统（ELK）** | 及时发现异常，满足 99.99% 可用性。 |

## 第三步：数据库设计

### 1. 元数据模型（关系型/分布式键值）

| 表/集合 | 主键 | 关键字段 | 备注 |
|--------|------|----------|------|
| **User** | `user_id (UUID)` | email, password_hash, plan, status | 用户基本信息 |
| **File** | `file_id (UUID)` | owner_id, parent_id, name, size, mime, created_at, updated_at, latest_version_id, hash, is_folder | 每个文件/文件夹的根元数据 |
| **FileVersion** | `version_id (UUID)` | file_id, version_number, size, hash, created_at, chunk_list (array<ChunkRef>) | 版本历史，保留 30 天 |
| **Chunk** | `chunk_id (UUID)` | storage_path, size, md5, ref_count | 对象存储的分块 |
| **Permission** | `perm_id (UUID)` | target_id (file_id/folder_id), grantee_user_id, role (READ/WRITE/OWNER), expires_at | 细粒度 ACL |
| **SharedLink** | `link_id (UUID)` | target_id, creator_id, token, role, password_hash, expires_at, click_count | 公开/受密码的分享链接 |
| **SyncLog** (Change Log) | `log_id (UUID)` | file_id, version_id, operation (CREATE/UPDATE/DELETE), initiator_user_id, timestamp | 供 Kafka 推送的变更记录 |
| **Conflict** | `conflict_id (UUID)` | file_id, base_version_id, conflicting_version_id, created_at | 记录冲突副本，供 UI 展示 |

> **为什么分离 Chunk 表？**  
- 大文件（>10 MB）会被切成 4 MB~8 MB 小块，**去重** 与 **引用计数** 能显著节省存储。  
- Chunk 只存储在对象存储里，元数据库只保存指针，读写开销小。

### 2. 选型思考

| 需求 | 推荐技术 | 解释 |
|------|----------|------|
| **强一致的元数据（目录、权限）** | **TiDB / CockroachDB**（分布式 NewSQL） | 支持事务、水平扩容，能保证 **Read‑After‑Write**（写完立刻可读）|
| **高并发的键值查询（热点文件）** | **Redis**（Cache） + **DynamoDB**（备份） | 缓存降低 DB 读压，DynamoDB 天然水平扩容 |
| **变更日志** | **Kafka / Pulsar** | 高吞吐、持久化、支持分区（按 user_id 分区）|
| **对象存储** | **自研分块存储** 或 **Ceph / MinIO** | 支持多副本、分块上传、分片合并、CDN 接入 |
| **监控/告警** | **Prometheus + Grafana** | 业界标准，易于集成 |

### 3. 一致性实现（Read‑After‑Write）

- **写路径**：客户端 → API Gateway → Metadata Service（TiDB） → **事务提交** → 返回成功。  
- **读路径**：同一请求的后续读取走 **同一分区的 Leader**，TiDB 保证强一致（使用 Raft）。  
- **跨 Region**：使用 **Active‑Active 多 Region**，每个 Region 本地写，**双写** 到另外 Region 并使用 **冲突解决（last‑write‑wins + version vector）**。  

> **不这样做的后果**：如果使用只读副本或 eventual‑consistent KV，用户可能在上传后立刻尝试下载却返回 “文件不存在”，体验极差。

## 第四步：核心 API 设计

> **原则**：RESTful + 统一错误码 + 幂等性 + 支持分块上传（Multipart）。

| API | 方法 | URL | 主要请求体 | 响应 | 备注 |
|-----|------|-----|------------|------|------|
| **Auth** | POST | `/auth/login` | `{email, password}` | `{access_token, refresh_token}` | JWT，15 min 有效，refresh 30 days |
| **列出目录** | GET | `/users/{uid}/files?parent={folder_id}` | - | `{files: [...]}` | 支持分页 (`limit`, `cursor`) |
| **创建文件/文件夹** | POST | `/users/{uid}/files` | `{name, parent_id, is_folder, size?}` | `{file_id}` | 若 `size>0`，返回 `upload_id` |
| **获取上传 URL（分块）** | POST | `/files/{file_id}/upload_url` | `{part_number, chunk_hash}` | `{url, expires}` | 使用预签名 URL，直接 PUT 到对象存储 |
| **完成上传** | POST | `/files/{file_id}/complete` | `{parts: [{part_number, etag}]}` | `{version_id}` | 服务端校验 MD5，生成 FileVersion |
| **下载文件** | GET | `/files/{file_id}/download` | - | **302 Redirect** 到 CDN URL | 支持 Range 断点续传 |
| **获取文件元数据** | GET | `/files/{file_id}` | - | `{file_id, name, size, version, hash, ...}` | 强一致读取 |
| **生成共享链接** | POST | `/files/{file_id}/share` | `{role, expires_at, password?}` | `{share_url, token}` | 链接可设置密码、失效时间 |
| **访问共享链接** | GET | `/share/{token}` | - | **302** 到下载 URL（校验密码） | 只读或可编辑（写入会走特定 API） |
| **列出历史版本** | GET | `/files/{file_id}/versions` | - | `{versions: [...]}` | 支持恢复 `POST /files/{file_id}/restore` |
| **冲突副本查询** | GET | `/files/{file_id}/conflicts` | - | `{conflicts: [...]}` | 客户端展示冲突文件 |
| **同步变更** | WS / SSE | `/sync/stream?uid={uid}&since={timestamp}` | - | **变更事件流** | 实时推送 `SyncLog` 消息 |

### 幂等性 & 重试

- **上传 ID**（`upload_id`）唯一标识一次分块上传，会话内 **PUT** 同一 `part_number` 多次返回相同 `etag`。  
- **完成上传** 接口必须是 **幂等**：若已完成，返回已有 `version_id`。  
- **所有写操作** 返回 **事务 ID**，客户端可用来做重试校验。

## 第五步：详细组件设计

### 1. 对象存储（分块 + 冗余）

1. **分块大小**：默认 4 MB（可配置 8 MB），对大文件做 **multipart upload**。  
2. **写流程**  
   - 客户端请求 `upload_id` → 返回分块上传 URL（预签名）  
   - 客户端并行 PUT 分块 → 每块返回 **ETag (MD5)**  
   - 完成后调用 `complete`，服务器校验所有 `etag` 与客户端提供的 `hash` 是否匹配。  
   - 成功后 **写入 Chunk 表**（新增或引用计数+1）并生成 **FileVersion**。  
3. **去重**：上传时先 `GET /chunks/{hash}` 检查是否已有相同 MD5，若有直接返回已有 `chunk_id`，跳过实际上传。  
4. **冗余**：每块在 **3 个不同 AZ** 保存副本（Erasure Coding 或 3‑way replication），对象存储内部负责自动恢复。  

> **不做分块的后果**：一次上传 10 GB 需要单一 TCP 连接，网络抖动会导致全局失败，且难以实现 **断点续传**。

### 2. 元数据服务（高可用读写）

- **读写分离**：TiDB 主节点负责写，Follower（只读副本）负责查询热点目录。  
- **事务**：创建文件、写入版本、更新父目录的 `size` 必须在同一个事务里完成，防止不一致。  
- **缓存**：  
  - **Redis**：热点目录列表（`GET /users/{uid}/files?parent=xxx`）缓存 5 min，写后 **Cache Invalidate**。  
  - **Local Cache**（客户端）：最近访问的文件元数据（TTL 1 min），配合 **ETag** 校验。  

### 3. 同步服务 & 变更日志

1. **写入**：每次 **FileVersion** 创建后，向 **Kafka** 生产一条 `FileChanged` 事件（包括 `file_id`, `version_id`, `operation`, `timestamp`）。  
2. **消费**：  
   - **推送服务**（WebSocket/FCM）订阅对应用户分区的事件，实时推送给在线设备。  
   - **异步任务**（如备份、索引）消费同一 Topic。  
3. **冲突检测**（关键流程，后面专门展开）。  

### 4. 冲突检测与冲突副本生成

**场景**：用户 A 与用户 B 同时编辑同一文件 `F`，各自提交了新版本 `V_A`、`V_B`。

#### 流程图（文字版）

```
Client A                      Client B
   |                             |
   |---PUT Part(s)----> Object Storage
   |---POST /complete (base_version = V0) ---> Metadata Service
                                            |
                                            v
                                      Metadata Service (TiDB)
                                      1. 检查 base_version == latest_version ?
                                      2. 若相同，创建新 FileVersion V_A
                                      3. 若不同，标记冲突
                                      |
                                      v
                               写入 Conflict 表 & 生成冲突副本
                                      |
                                      v
                              同步日志 (Kafka) -> 推送给两端
```

#### 详细步骤

| 步骤 | 说明 | 关键字段 |
|------|------|----------|
| **1. 客户端上传分块** | 与普通上传相同，得到 `upload_id`。 | `upload_id`, `chunk_hashes` |
| **2. 完成上传** | `POST /files/{file_id}/complete`，请求体里带 `base_version_id`（客户端本地最近看到的版本）。 | `base_version_id` |
| **3. 元数据服务事务** | - **读取**当前 `latest_version_id`（行锁）。<br>- **比较**`base_version_id` 与 `latest_version_id`。 | `latest_version_id` |
| **4. 若相同** | 正常写入新 `FileVersion`，更新 `latest_version_id`。 | `new_version_id` |
| **5. 若不同** | 产生冲突：<br> a. 保存上传的分块为 **冲突副本**（新 `FileVersion`，标记 `is_conflict = true`）。<br> b. 在 **Conflict** 表记录 `conflict_id`、`base_version_id`、`conflicting_version_id`。<br> c. 返回错误码 `409 Conflict`，并在响应体里返回冲突副本的 `file_id`（在 UI 中表现为 “冲突副本”）。 | `conflict_id` |
| **6. 同步推送** | 将 `FileChanged` 事件发送到 Kafka，**推送服务**把冲突信息同步到所有在线设备。 | `event_type = CONFLICT` |
| **7. 客户端处理** | UI 展示冲突文件，用户可以手动合并或保留其中一个版本。 | — |

- **幂等保证**：若客户端因网络抖动多次调用 `complete`，服务端通过 `upload_id` + `base_version_id` 判定是否已经产生冲突或成功，返回相同 `version_id`。  
- **冲突副本存储**：与普通版本同样使用分块存储，只是元数据 `FileVersion.is_conflict = true`，不计入“最新版本”。  

> **不做冲突检测的后果**：不同设备的改动会相互覆盖，用户会丢失数据，极度破坏用户信任。

### 5. 离线编辑与同步

1. **本地操作队列**：客户端在离线时把所有 **文件增删改** 记录为本地 **操作日志**（包括 `base_version_id`）。  
2. **恢复网络**：客户端按顺序发送 `complete` 请求。若出现冲突，按照上面冲突流程处理。  
3. **幂等写入**：每个离线操作都有唯一 `client_op_id`（UUID），服务端记录已处理的 `client_op_id`，防止网络重试导致同一操作多次写入。  

### 6. 共享链接实现

- **生成**：服务器生成 **随机 token**（64 位），在 `SharedLink` 表保存映射以及权限（READ/WRITE）。  
- **签名 URL**：下载时返回 **预签名的 CDN URL**（有效期 5 min），只要 token 校验通过即可。  
- **访问控制**：如果链接设置了密码，客户端在访问前需要先提交密码进行一次 **POST /share/{token}/verify**，成功后返回临时访问凭证。  

### 7. 监控 & 灰度发布

| 监控指标 | 阈值（示例） | 处理方式 |
|----------|------------|----------|
| **API Latency P99** | < 200 ms | 自动伸缩 API 实例 |
| **Upload Success Rate** | > 99.9% | 报警并回滚最近的部署 |
| **Sync Delay (95%)** | ≤ 5 s | 增加消费者实例或调高 Kafka 分区数 |
| **对象存储磁盘使用** | > 80% | 触发扩容或压缩旧块 |
| **错误码 5xx** | > 0.1% | 自动触发降级（仅只读） |

## 第六步：扩展性与高可用设计

### 1. 水平扩容点

| 组件 | 触发扩容信号 | 扩容方式 |
|------|--------------|----------|
| **API Gateway / LB** | QPS > 80% of 当前实例上限 | 增加实例，使用 **Consistent Hash** 或 **IP‑hash** 维持会话粘性 |
| **Metadata DB** | 写入 TPS > 70% of 单节点吞吐 | **分片（Shard）**：按 `user_id` hash 分区；或切换到 **TiDB** 自动水平扩容 |
| **Redis Cache** | 命中率 < 70% 或内存占用 > 80% | 增加节点，使用 **Cluster Mode**，自动分片 |
| **Kafka** | 分区 backlog > 1 GB | 增加 **partition**（按 `user_id` hash），扩容 **Broker** |
| **对象存储** | 存储容量 > 80% 或网络 I/O > 70% | **扩容磁盘**、**增加 OSD 节点**、**使用 Erasure Coding** 减少单块压力 |
| **推送服务** | 在线设备数 > 10 M | **水平扩容** WebSocket 集群，使用 **Nginx + upstream** 负载均衡 |
| **CDN** | 带宽峰值 > 80% | **按地域扩容 Edge 节点**，使用 **Anycast** |

### 2. 容灾 & 多 Region

1. **数据复制**  
   - **元数据**：跨 Region **双写**（Active‑Active），使用 **冲突解决策略**（基于 **Version Vector**）。  
   - **对象存储**：**异步跨 Region 复制**（每块写入后发送复制任务），配合 **读写分离**，读优先本地，跨 Region 读 fallback。  

2. **故障切换**  
   - **DNS 级别**（Route53）+ **Health Check**：当某 Region 健康检查失败，自动将流量切换至备份 Region。  
   - **服务注册中心**（Consul/Eureka）：各组件感知对端不可用后切换到最近的可用实例。  

3. **数据恢复**  
   - **Chunk 引用计数** 与 **垃圾回收**：定期扫描 `ref_count==0` 的块，异步删除。  
   - **版本保留**：使用 **Time‑Based TTL**（30 天）+ **手动保留**（用户标记）实现。  

### 3. 读已提交（Read‑After‑Write）实现细节

- **写成功后返回**的 `version_id` 已经写入 **Leader** 并同步到 **majority**（Raft）。此时任何后续 **读请求**（即使落在 follower）都会被 **Redirect** 到 **Leader** 或者 **Follower 已同步**，保证 **强一致**。  
- **跨 Region**：写到本地 Region Leader 后，立刻返回给客户端（**低延迟**），再异步复制到其他 Region。读请求若指定 **强一致**（默认）则强制路由到本地 Region Leader；若允许 **最终一致**（如 CDN 读取）则可以读副本。  

> **不使用 Raft/强一致**：读写可能落在不同副本，导致客户端上传后立即下载到的文件是旧的，违背业务需求。

### 4. 安全与合规（简要）

- **传输层**：全部使用 **TLS 1.3**。  
- **存储加密**：对象存储使用 **AES‑256**，密钥由 **KMS** 管理，支持 **密钥轮转**。  
- **访问控制**：每次 API 调用检查 **JWT** 与 **Permission** 表；共享链接使用 **一次性签名 URL**。  
- **审计日志**：所有写操作（上传、删除、共享）写入 **审计 Kafka Topic**，供离线分析。

## 第七步：常见面试追问与回答

### Q1. 数据一致性：多副本存储如何实现 Read‑After‑Write？

**回答要点**  
1. **元数据层**使用 **Raft** 共识（TiDB/CockroachDB）保证写入必须被多数副本确认后才返回，读请求可以直接从 Leader 或已同步的 Follower 读取，确保 **强一致**。  
2. **对象存储层**：写入分块后，返回 **ETag**（MD5）并在后台进行 **同步复制**。客户端在 **完成上传** 前不会返回成功，只有当块写入本地磁盘并写入 **副本日志**（至少 2/3）后才确认。  
3. **跨 Region**采用 **Active‑Active 双写 + Version Vector**，冲突极少且通过 **last‑write‑wins** + **用户提示** 解决。  

**如果不这么做**：读可能返回旧的块或元数据，用户会看到“上传成功但文件仍是旧的”，极大破坏用户体验。

### Q2. 冲突检测的完整流程？

（参考上文 **冲突检测与冲突副本生成** 小节）  
- 客户端在 `complete` 时带上 **base_version_id**。  
- 服务端在事务中读取当前 **latest_version_id** 并比较。  
- 若相同，正常写入新版本；若不同，创建 **冲突副本**（标记 `is_conflict`），写入 **Conflict** 表并返回 `409 Conflict`。  
- 同步日志把冲突事件推送给所有在线设备，客户端 UI 展示冲突文件，用户手动合并。  

### Q3. 当 DAU 从 5 × 10⁷ 增长到 2 × 10⁸ 时，哪些组件需要扩容？

| 组件 | 扩容策略 |
|------|----------|
| **API Gateway / LB** | 增加实例，使用 **自动伸缩组**（基于 CPU / QPS） |
| **Metadata DB** | **水平分片**：按 `user_id` hash 拆分到多个 TiDB 集群；或使用 **CockroachDB** 自动扩容 |
| **Redis Cache** | 扩容 **Cluster**，增加 slot 数 |
| **Kafka** | 增加 **partition**（每 1M 用户 1 partition），并增加 **Broker** |
| **对象存储** | 增加 **OSD** 节点，使用 **Erasure Coding** 降低单块写放大；提升 **网络带宽**（10 GbE → 40 GbE） |
| **推送服务** | 增加 **WebSocket** 节点，使用 **负载均衡** 与 **水平扩容** |
| **CDN** | 按流量购买更多 **Edge** 节点，开启 **Anycast** 加速 |  

**可能的瓶颈**  
- **元数据写热点**（同一个用户的频繁改动）→ 需要 **分区锁** 或 **乐观锁** 降低冲突。  
- **Kafka 分区数**上限导致 **consumer 并发受限** → 需要 **重新分配分区** 或 **使用多 Topic**。  
- **对象存储网络**：跨 AZ/Region 带宽可能成为瓶颈，需采用 **本地写** + **异步复制**。  

### Q4. 如果用户在离线状态下编辑同一文件，恢复网络后会出现冲突吗？

**回答**  
- **是的**，因为离线设备的 `base_version_id` 与服务器当前的 `latest_version_id` 可能已经不同。  
- 当设备恢复网络，客户端会把离线操作按顺序发送，服务器会按 **冲突检测** 流程判断并生成冲突副本。  
- 为了降低冲突概率，客户端在离线时可以 **周期性写入本地心跳**（如每 5 min 把本地 `base_version_id` 与服务器最新版本对比），但这只能在网络可用时实现。  

### Q5. 为什么要使用分块上传而不是一次性上传？

- **容错**：单块上传失败只需要重传该块，而不是整个文件。  
- **并行**：可以同时上传多个块，提升吞吐（尤其在高带宽环境）。  
- **断点续传**：网络中断后，只需要继续上传未完成的块。  
- **去重**：块级 MD5 可以在服务器端快速判断是否已经存在，避免重复写入。

## 心得与反思

### 1. 本题最难的 1‑2 个设计决策及思考过程

| 决策 | 难点 | 思考路径 |
|------|------|----------|
| **如何实现强一致的元数据读写（Read‑After‑Write）** | 必须兼顾高可用、跨 Region、写入延迟不宜过高。 | ① 评估 **单机 DB** → 不满足可用性。 ② 考虑 **主从复制** → 读写不一致。 ③ 选择 **Raft‑based NewSQL（TiDB/CockroachDB）**，保证写入必须多数确认后返回。 ④ 为跨 Region 加入 **Version Vector** + **双写**，在本地 Region 返回成功，异步复制到其他 Region，冲突极少。 |
| **冲突检测与冲突副本生成的完整流程** | 需要在保证 **幂等**、**低延迟**、**离线恢复** 的前提下，准确捕获并告知冲突。 | ① 引入 **base_version_id**（客户端感知的最新版本）作为写请求的前置条件。 ② 在元数据服务内部使用 **行锁+事务**比较最新版本，决定是否创建冲突副本。 ③ 设计 **Conflict 表** 与 **FileVersion.is_conflict** 标记，确保冲突文件不影响 “最新版本”。 ④ 将冲突信息写入 **Kafka** 推送给所有设备，保证实时告知。 ⑤ 为离线场景保留 **client_op_id** 幂等键，防止网络抖动导致重复写入。 |

### 2. 新手最容易犯的错误（至少 2 条）

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把对象存储和元数据混在一起**（把文件内容直接存在关系型 DB） | DB 写入压力极大，磁盘成本高，难以实现大文件的分块上传，扩展性差。 | 使用 **对象存储（分块）** 保存文件内容，仅在 **元数据 DB** 保存指向块的引用（ChunkRef）。 |
| **只考虑单机或单 AZ 部署**（忽视容灾） | 单点故障会导致整个平台不可用，无法满足 99.99% 可用性。 | 采用 **多 AZ/Region**，对象存储多副本、元数据服务使用 Raft 多副本、API Gateway 前置 LB。 |
| **忽略幂等性**（上传/完成接口重复调用导致数据重复） | 重试机制会产生脏数据，用户看到重复文件或冲突。 | 为每次上传生成 **upload_id**，所有写操作携带 **client_op_id**，服务端检查幂等键后返回已有结果。 |
| **把所有同步逻辑放在同步请求里**（同步时阻塞） | 同步延迟无法满足 ≤5 秒，服务器压力激增。 | 使用 **异步事件流（Kafka） + 推送服务**，写完立即返回，后续同步通过事件消费完成。 |
