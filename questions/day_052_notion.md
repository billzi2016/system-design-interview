# 第 52 天：设计 Notion

> 生成日期：2026-04-04

## 题目背景
Notion 是一款面向个人和团队的全功能协作平台，提供文档编辑、知识库、项目管理、数据库等多种工作空间形态，帮助用户以块（block）为粒度组织和共享信息。

## 面试场景设定
> **面试官**：  
> “今天我们来聊一聊如果要从零开始设计一套类似 Notion 的在线协作系统，你会如何做？请先从高层次的架构出发，阐述核心模块的职责和交互，然后再逐步细化到具体的功能实现、容量规划以及性能保证等方面。”

## 功能性需求
| 编号 | 功能描述 | 关键点 |
|------|----------|--------|
| 1 | **块（Block）编辑与渲染**：用户可以创建、编辑、删除、移动各种类型的块（文字、标题、列表、表格、代码块、嵌入媒体等），并实时预览渲染效果。 | 富文本、Markdown 支持、拖拽排序、历史版本。 |
| 2 | **实时协同编辑**：多用户同时在同一个页面上编辑，同步冲突最小化，编辑光标、选区、评论等实时可见。 | OT/CRDT、光标位置信息、冲突解决策略。 |
| 3 | **页面与工作空间组织**：用户可以创建层级化的页面/子页面，使用侧边栏进行导航；支持团队工作空间、权限管理（读/写/管理员）。 | 层级树结构、ACL、邀请链接。 |
| 4 | **数据库/表格视图**：块可以作为数据记录，支持多视图（表格、看板、日历、画廊），并可进行筛选、排序、聚合。 | 数据模型、查询语言、视图同步。 |
| 5 | **全文检索 & 标签**：对所有块内容提供即时全文搜索，支持标签、收藏、最近使用等快捷入口。 | 倒排索引、搜索高亮、分页。 |
| 6 | **离线编辑与同步**：在网络不稳定时仍能编辑，恢复网络后自动同步冲突。 | 本地缓存、同步协议、冲突合并。 |

## 非功能性需求
| 指标 | 估算值 | 说明 |
|------|--------|------|
| **日活跃用户（DAU）** | 5 Million | 假设目标市场为全球中小团队用户。 |
| **每页 QPS（读）** | 2,000 req/s | 高峰时段单个工作空间的页面访问并发。 |
| **写入 QPS（编辑/保存）** | 300 req/s | 实时协同编辑的变更上报。 |
| **页面渲染延迟** | < 150 ms（95% SLA） | 包括后端渲染、数据查询、前端渲染。 |
| **可用性** | 99.9% 年可用率 | 计划使用多 AZ 部署，自动故障转移。 |
| **存储量** | 500 TB 文档与附件 | 估算每用户平均 100 KB 文档 + 10 KB 附件，乘以 5 M DAU，保留 2 年。 |

## 解题思路总览  

1. **从需求出发**：先把功能需求、非功能需求、业务边界梳理清楚，做出**容量估算**（DAU、QPS、存储）。  
2. **从最小可用系统（MVP）开始**：只实现「单机」的块编辑、页面组织、持久化」三大核心能力，让系统能够跑通最基本的用户故事。  
3. **逐层演进**：在 MVP 基础上，逐步加入 **实时协同、离线同步、全文检索、权限控制、数据库视图**，并在每一步引入 **水平扩展、容错、监控** 的设计。  
4. **关键技术点**：  
   - **协同算法**（OT vs CRDT）  
   - **块模型 & 版本链**（支持无限历史）  
   - **混合存储**：关系型 + 文档/宽列 + 对象存储  
   - **搜索索引**：倒排索引 + 增量同步  
5. **面试常见追问**：提前准备冲突解决、索引扩展、故障恢复等答辩材料。  

> **核心原则**：**先对业务负责，再对性能负责**。先把功能正确实现，再考虑高并发与高可用。

## 第一步：理解需求与规模估算  

### 1. 功能拆解  

| 编号 | 功能 | 子功能点（最小可实现） |
|------|------|------------------------|
| 1 | 块编辑与渲染 | 创建/编辑/删除块、基本渲染（Markdown → HTML） |
| 2 | 实时协同编辑 | 同步文字块的编辑、光标位置信息 |
| 3 | 页面/工作空间组织 | 页面树（父子关系）、工作空间（租户） |
| 4 | 数据库/表格视图 | 表格块 → 行/列模型、简单筛选/排序 |
| 5 | 全文检索 & 标签 | 关键字搜索、标签过滤 |
| 6 | 离线编辑与同步 | 本地缓存（IndexedDB），恢复后同步 |

> **最小可用系统（MVP）** 只需实现 1、3 两大块，后面再逐步叠加 2、4、5、6。  

### 2. 非功能需求与容量估算  

| 指标 | 估算值 | 计算方式/解释 |
|------|--------|---------------|
| DAU | 5 M | 全球中小团队目标 |
| 页面读 QPS | 2 000 req/s | 峰值每秒 2000 次页面读取 |
| 写入 QPS（编辑） | 300 req/s | 实时编辑上报 |
| 页面渲染延迟 | <150 ms（95%） | 包括 DB + 渲染 |
| 可用性 | 99.9%（≈8.76 h/年） | 多 AZ + 自动故障转移 |
| 存储量 | 500 TB | 5 M * 100 KB 文档 + 10 KB 附件 * 2 年 ≈ 600 TB，取保守 500 TB |

### 3. 关键容量公式（帮助面试官快速判断）  

- **读请求**：  
  `R = DAU * avg_page_per_user * read_per_page_per_day / 86400`  
  假设每用户每天打开 5 页，每页平均 3 次读取 → `R ≈ 5M * 5 * 3 / 86400 ≈ 868 req/s`。取 2×安全系数 → 1.7 k → 2 k。  

- **写请求**：  
  `W = DAU * edit_rate`  
  假设 5% 的活跃用户在同一时间编辑，编辑频率 1 次/分钟 → `W ≈ 5M * 0.05 / 60 ≈ 4.2k`，但真实峰值集中在协同编辑的 **增量变更**，每条变更体积很小，取 300 req/s 为题目给值。  

- **存储**：  
  `Total = Users * (DocSize + AttachmentSize) * retention_years`  
  `5M * (100KB + 10KB) * 2 ≈ 1.1 TB`（实际远小于 500 TB），但考虑历史版本、索引、备份等，取 500 TB。  

> **为什么要做这些估算**：在面试里，你需要展示自己会先 **量化需求**，否则后面的架构设计会缺乏依据，评审会直接打“估算不足”。

## 第二步：高层架构设计  

### 1. MVP（单机）高层图  

```
+-------------------+          +-------------------+
|   Frontend (SPA)  | <------> |   API Gateway     |
+-------------------+          +-------------------+
                                   |
                        +-----------------------+
                        |   Application Server  |
                        +-----------------------+
                                   |
                     +-----------------------------+
                     |   Block Service (CRUD)      |
                     +-----------------------------+
                                   |
                     +-----------------------------+
                     |   Relational DB (Postgres)  |
                     +-----------------------------+
```

- **前端**：React/Vue 单页应用，负责渲染块、发送 CRUD 请求。  
- **API Gateway**：统一入口，做鉴权、限流、路由。  
- **Application Server**：业务层（Spring Boot / Node.js），实现块编辑、页面组织等业务逻辑。  
- **关系型 DB**：存放页面树、块结构、用户信息。  

> **不做**：分布式缓存、消息队列、搜索服务。  
> **后果**：单机能跑通功能，但无法支撑并发、容灾。  

### 2. 演进到可扩展的分布式架构  

```
               +---------------------------+
               |      CDN / Edge Cache     |
               +---------------------------+
                          |
+-----------+   +-------------------+   +-----------+
|  Frontend |<->|   API Gateway     |<->|  Frontend |
+-----------+   +-------------------+   +-----------+
                          |
        +-------------------------------------------+
        |                Service Mesh               |
        +-------------------------------------------+
          |               |               |
   +--------------+  +--------------+  +--------------+
   | Block Service|  |  Collab Svc  |  | Search Svc   |
   +--------------+  +--------------+  +--------------+
          |               |               |
   +--------------+  +--------------+  +--------------+
   |  Relational  |  |   CRDT Store |  |  ES / Solr   |
   |   DB (PG)    |  | (Redis+Log)  |  |   Cluster    |
   +--------------+  +--------------+  +--------------+
          |               |               |
   +--------------+  +--------------+  +--------------+
   | Object Store |  | Message Bus  |  |   Cache      |
   |  (S3/MinIO)  |  | (Kafka)      |  | (Redis)      |
   +--------------+  +--------------+  +--------------+
```

- **CDN**：缓存页面的静态资源、附件。  
- **API Gateway**：统一鉴权（JWT/OAuth）、限流、灰度发布。  
- **Service Mesh**（Istio/Linkerd）：实现服务间的流量控制、熔断、监控。  
- **Block Service**：负责块的 CRUD、版本管理。  
- **Collab Service**：实时协同核心，运行 OT/CRDT 引擎，使用 **Kafka** 作为变更的可靠广播渠道。  
- **Search Service**：Elasticsearch 集群，提供全文检索、标签过滤。  
- **CRDT Store**：采用 **Redis + Write‑Ahead Log** 保存 CRDT 状态，快速读取/写入。  
- **Object Store**：S3 兼容对象存储，用于附件、页面快照。  
- **Cache**：Redis 读取热点页面、权限信息，降低 DB 压力。  

> **为什么这样拆**：  
- **职责单一**（SRP）让每个服务可以独立水平扩展。  
- **实时协同** 与 **搜索** 对延迟、吞吐要求不同，分离能避免相互干扰。  
- **消息总线** 保证 **写‑读分离**，编辑操作只写入 Kafka，后端消费者负责落库、索引。

## 第三步：数据库设计  

### 1. 块（Block）模型  

| 字段 | 类型 | 说明 |
|------|------|------|
| `block_id` | UUID (PK) | 全局唯一 |
| `page_id` | UUID (FK) | 所属页面 |
| `parent_block_id` | UUID (nullable) | 父块（用于嵌套列表、子页面） |
| `type` | ENUM | text / heading / list / table / code / embed … |
| `content` | JSONB | 具体内容（Markdown、属性） |
| `position` | INT | 同层块的顺序（用于拖拽排序） |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 最后编辑时间 |
| `version` | BIGINT | 当前版本号（递增） |
| `deleted` | BOOLEAN | 软删标记 |

- **JSONB** 让不同块类型拥有灵活字段（如表格块的 `rows`、`columns`）。  
- **PostgreSQL** 的 **行级锁** + **乐观锁**（`version`） 能支持高并发的单块编辑。  

### 2. 页面（Page）模型  

| 字段 | 类型 | 说明 |
|------|------|------|
| `page_id` | UUID (PK) |
| `workspace_id` | UUID (FK) |
| `title` | TEXT |
| `parent_page_id` | UUID (nullable) |
| `path` | TEXT | 例如 `/Team/Docs/Project`，用于快速检索 |
| `created_by` | UUID |
| `created_at` | TIMESTAMP |
| `updated_at` | TIMESTAMP |
| `is_deleted` | BOOLEAN |

- **层级树** 用 **Adjacency List**（父指针）实现，配合 **Materialized Path** (`path`) 能快速获取子树。  

### 3. 工作空间（Workspace） & ACL  

| 字段 | 类型 | 说明 |
|------|------|------|
| `workspace_id` | UUID (PK) |
| `name` | TEXT |
| `owner_id` | UUID |
| `created_at` | TIMESTAMP |
| `updated_at` | TIMESTAMP |

**权限表（ACL）**  

| 字段 | 类型 | 说明 |
|------|------|------|
| `acl_id` | BIGINT (PK) |
| `workspace_id` | UUID |
| `user_id` | UUID |
| `role` | ENUM (READ, WRITE, ADMIN) |
| `expire_at` | TIMESTAMP (optional) |

- 使用 **行级安全（RLS）** 或在业务层做权限检查。  

### 4. 版本历史（Edit Log）  

- **写入**：每一次块的编辑都会产生一条 **增量日志**，存入 **Kafka**，随后由 **Log Consumer** 持久化到 **Postgres** 的 `block_history` 表或 **对象存储**（压缩后）。  
- **表结构**（增量存储）：

| field | type |
|-------|------|
| `history_id` | BIGINT (PK) |
| `block_id` | UUID |
| `version` | BIGINT |
| `diff` | JSONB (OT/CRDT 增量) |
| `author_id` | UUID |
| `timestamp` | TIMESTAMP |

- **回滚**：读取对应 `block_id` 的历史记录，合并增量恢复到目标版本。  

> **为什么使用增量日志而不是完整快照**：  
  - **成本**：完整快照每次都要复制整块内容，随着块大小增长会导致存储激增。  
  - **回滚速度**：增量日志可以通过 **二分查找** 快速定位到目标版本。  

### 5. 搜索索引模型（Elasticsearch）  

- **文档结构**（每个块对应 ES 文档）  

```json
{
  "block_id": "uuid",
  "page_id": "uuid",
  "workspace_id": "uuid",
  "type": "text",
  "content": "plain text of markdown",
  "tags": ["todo", "frontend"],
  "created_at": "2024-01-01T12:00:00Z",
  "updated_at": "2024-01-02T09:30:00Z"
}
```

- **分片/复制**：根据 **workspace_id** hash 进行路由，默认 5 主分片 + 1 副本，后期可以 **按业务热点** 调整。

## 第四步：核心 API 设计  

> **REST + JSON** 为主，部分实时功能采用 **WebSocket / gRPC**。  

### 1. 鉴权  

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/auth/login` | 返回 JWT（包含 `user_id`, `workspace_ids`) |
| `POST` | `/auth/refresh` | 刷新 token |

### 2. 页面 & 块 CRUD  

| 方法 | 路径 | 请求体 | 响应 | 备注 |
|------|------|--------|------|------|
| `GET` | `/workspaces/{wid}/pages/{pid}` | - | 页面元数据 + 块树（嵌套） | 支持 **分页**（子块） |
| `POST` | `/workspaces/{wid}/pages` | `{title, parent_page_id}` | `page_id` | 创建页面 |
| `PATCH` | `/pages/{pid}` | `{title}` | - | 重命名 |
| `DELETE` | `/pages/{pid}` | - | - | 软删 |

| 方法 | 路径 | 请求体 | 响应 | 备注 |
|------|------|--------|------|------|
| `POST` | `/pages/{pid}/blocks` | `{type, content, position, parent_block_id}` | `block_id` | 新建块 |
| `GET` | `/blocks/{bid}` | - | 块完整信息 |  |
| `PATCH` | `/blocks/{bid}` | `{content, position}` | - | **乐观锁**：需带 `version` |
| `DELETE` | `/blocks/{bid}` | - | - | 软删 |

> **乐观锁实现**：客户端在 PATCH 时带上 `If-Match: <version>`，服务端比对不一致返回 `409 Conflict`，前端弹出“内容已被他人修改”。  

### 3. 实时协同（WebSocket）  

- **连接**：`ws://api.example.com/collab?token=JWT&workspace_id=...`  
- **消息协议（JSON）**  

```json
{
  "type": "join",
  "page_id": "uuid"
}
```

```json
{
  "type": "op",
  "block_id": "uuid",
  "op": { "retain": 5, "insert": "hello" },
  "version": 123,
  "author_id": "uuid"
}
```

- **服务器**：收到 `op` → **验证版本** → **写入 Kafka** → **广播**给同页面的其它客户端 → **异步落库**。  

### 4. 搜索 API  

| 方法 | 路径 | 请求体 | 响应 |
|------|------|--------|------|
| `GET` | `/search` | `q=keyword&workspace_id=...&tags=...&page=1&size=20` | `hits[]`（高亮、分页） |

### 5. 离线同步（REST + 本地缓存）  

| 方法 | 路径 | 请求体 | 响应 |
|------|------|--------|------|
| `POST` | `/sync` | `{client_id, last_synced_version, ops[]}` | `{ack_version, conflicts[]}` |

- **客户端**：在恢复网络后，将本地 **未同步的增量 ops**（同实时协同的格式）一次性发送。  
- **服务端**：对每条 op 进行 **冲突检测**（基于 CRDT 合并），返回成功的 **ack_version**，若有冲突返回 **冲突列表** 让前端弹框。

## 第五步：详细组件设计  

### 1. 块服务（Block Service）  

| 子组件 | 职责 | 技术选型 |
|--------|------|----------|
| **API Handler** | 接收 HTTP 请求、参数校验、返回统一错误结构 | Spring Boot / Express |
| **业务层** | 块的创建、移动、删除、版本号生成、权限校验 | Java Service / Node Service |
| **持久化层** | 与 PostgreSQL 交互，使用 **JPA / MyBatis**，开启 **Read‑Write 分离** | PG 主从 |
| **缓存层** | 读取热点块、页面树，使用 **Redis**（TTL 5 min） | Redis LRU |
| **事件发布** | 把块的增量写入 **Kafka**（topic: `block-change`) | Kafka Producer (async) |

> **不使用**：直接写入 DB 而不产出事件，会导致搜索、协同、审计等子系统无法实时感知变更。  

### 2. 实时协同服务（Collab Service）  

- **核心算法**：**CRDT（RGA/Logoot）** + **OT** 备选（文字块采用 OT，结构化块采用 CRDT）。  
- **实现思路**：  
  1. **客户端** 按 **操作序列**（Insert/Delete）生成 **CRDT op**。  
  2. **发送**到 Collab Service via **WebSocket**。  
  3. **服务端** 将 op 写入 **Kafka**（确保可靠性）并 **广播**给同页面的其他 WebSocket 连接。  
  4. **状态机**（基于 **Redis**）维护每个页面的 **最新版本向量时钟**，帮助快速冲突检测。  

- **数据持久化**：  
  - **增量日志** → Kafka → **Log Consumer** → 写入 **Postgres block_history**（备份）以及 **对象存储**（gzip）用于长期归档。  

- **扩展**：  
  - **分片**：每个工作空间或页面拥有独立的 **Collab 实例**，通过 **Consistent Hash** 将流量路由。  

### 3. 搜索服务（Search Service）  

- **索引流程**：  
  1. **Block Service** 更新块后，**发送** `block-change` 事件。  
  2. **Search Consumer**（Kafka consumer）**解析**增量（全文内容、标签）并 **同步**到 **Elasticsearch**。  
  3. **实时增量**：使用 **bulk API** 每 100 条或 1 s 批量提交，保证 **延迟 < 500 ms**。  

- **分片策略**：  
  - **基于 workspace_id 哈希**（确保同工作空间的文档落在同一分片，查询时路由更快）。  
  - **副本数**：默认 1，热点工作空间可提升到 2。  

- **容灾**：  
  - **快照**：每日对 ES 集群做快照到对象存储（S3），支持跨地域恢复。  

### 4. 离线同步模块  

| 子系统 | 作用 | 技术 |
|--------|------|------|
| **本地缓存** | 在浏览器使用 **IndexedDB** 保存块的本地副本和未同步的 ops | IndexedDB + Service Worker |
| **同步调度** | 网络恢复后批量推送 `ops[]` 到 `/sync` 接口 | exponential backoff + jitter |
| **冲突合并** | 服务器端采用 **CRDT 合并**，返回冲突列表 | Collab Service + Conflict Resolver |
| **回滚/恢复** | 若冲突无法自动合并，前端展示 **冲突解决 UI**（接受服务器、保留本地） | React/Redux |

> **为什么要有离线缓存**：如果仅依赖即时网络，用户在网络抖动时会出现编辑丢失或卡顿，极大影响体验。  

### 5. 权限与 ACL  

- **在 API 层**：拦截器（Spring `HandlerInterceptor`）读取 JWT 中的 `workspace_ids`，再查询 **ACL** 表确认 `role`。  
- **在 DB**：使用 **行级安全（RLS）**（PostgreSQL）或在业务层做过滤，防止 **越权访问**。  

### 6. 监控、日志、追踪  

| 维度 | 工具 |
|------|------|
| **日志** | ELK (Filebeat → Logstash → Kibana) |
| **指标** | Prometheus (exporters for JVM/Node, Kafka, Redis) |
| **分布式追踪** | OpenTelemetry + Jaeger |
| **报警** | Alertmanager + PagerDuty |

> **不做监控**：在面试官眼里，你的系统只能“跑”，但一旦出故障，你无法定位根因，显得不完整。

## 第六步：扩展性与高可用设计  

### 1. 读写分离 & 缓存  

- **读**：前端页面渲染时，优先走 **Redis Cache** → **Postgres read replica** → **Primary**。  
- **写**：所有写请求走 **Primary**，写成功后 **异步刷新**到 **Read Replica**（Postgres streaming replication）。  

> **如果不做读写分离**，单主库在高峰期会因大量页面读取而导致写入延迟，影响编辑体验。  

### 2. 多可用区（Multi‑AZ）部署  

- **每个 AZ** 部署完整的 **API Gateway → Service Mesh → 微服务集群**。  
- **数据库**：使用 **PostgreSQL 多主（Patroni）** 或 **Aurora Global**，实现跨 AZ 自动故障转移。  
- **Kafka**：跨 AZ **复制因子**（replication factor = 3），确保任意 AZ 故障后仍能提供写入。  

### 3. 自动伸缩（Autoscaling）  

- **水平伸缩**：Kubernetes HPA 根据 **CPU、QPS、Latency** 自动扩容微服务。  
- **Kafka 分区**：页面热点会导致某个分区流量激增，使用 **分区再均衡** 或 **按工作空间划分**。  

### 4. 数据备份 & 灾难恢复  

| 数据 | 备份方式 | 恢复目标 (RPO / RTO) |
|------|----------|---------------------|
| PostgreSQL | PITR (WAL) + 每日快照 | RPO < 5 min, RTO < 30 min |
| Elasticsearch | 快照到对象存储（每日） | RPO 12 h, RTO 1 h |
| Kafka | MirrorMaker 跨集群复制 | RPO 0, RTO 5 min |
| 对象存储 | 多 AZ 冗余（S3‑style） | RPO 0, RTO 5 min |

### 5. 灰度发布 & 回滚  

- **Canary Deploy**：使用 **Istio** 的流量分配，把新版本流量限制在 5% 用户。  
- **Rollback**：若监控出现异常，立即将流量切回旧版本。  

### 6. 安全性  

- **传输层**：全站 TLS（HTTPS、WSS）。  
- **身份验证**：OAuth2 + JWT，短期 Access Token + Refresh Token。  
- **审计日志**（虽不在本题范围，但简要提及）：写入 **Kafka** → **审计服务** → **Object Store**，满足合规需求。

## 第七步：常见面试追问与回答  

### 1. OT 与 CRDT 的优缺点，选哪一个？  

| 特性 | OT (Operational Transformation) | CRDT (Conflict‑free Replicated Data Type) |
|------|--------------------------------|--------------------------------------------|
| **核心思路** | 把后到来的操作 **转化** 为基于当前状态的操作 | 所有副本 **自然收敛**，不需要转化 |
| **实现复杂度** | 需要 **中心化的转化函数**，对不同数据结构（表格、代码块）实现难度大 | 每种数据结构需要设计对应的 **CRDT**，但一旦实现，可完全 **去中心化** |
| **延迟** | 必须等到服务器确认后才能转发，**单向延迟** 较低 | 客户端本地直接 **执行**，只需要 **异步广播**，延迟更低 |
| **冲突解决** | 依赖 **转化函数**，对复杂结构易出错 | 基于 **数学可交换性**，冲突自动消除 |
| **存储成本** | 需要 **全局序列号**，日志可能增长快 | 需要 **元数据**（如向量时钟），但通常更轻量 |
| **适用场景** | 纯文本编辑（Google Docs 早期） | 多种 **混合结构**（Notion 块、表格、看板） |

**在 Notion 场景下倾向使用 CRDT**：  
- Notion 的块不仅是富文本，还包含 **表格、看板、嵌入** 等多种结构。  
- CRDT 能让每种块自行实现自己的合并逻辑，**无需中心化转化**，更容易在 **离线后再同步**（客户端本地直接执行）。  
- 若对 **文字块** 要求极致低延迟，可在文字块内部继续使用 **OT**（如 Google Docs 那样的混合），外层统一为 CRDT，保持整体一致性。  

### 2. 如何实现无限制的编辑历史和快速回滚？  

1. **增量日志**：每次编辑只记录 **diff**（OT/CRDT 增量）而不是完整快照。  
2. **压缩归档**：每 1000 条或 1 天的增量日志压缩后写入 **对象存储**（Parquet），作为长期归档。  
3. **回滚流程**：  
   - **获取目标版本号** → **读取**对应 **增量日志**（从对象存储或历史表） → **按时间顺序累计**到目标版本 → **写回最新块**（新版本号）。  
   - 为提升回滚速度，可在 **每 1000 版本** 做一次 **快照**（完整块），回滚时先恢复最近的快照再累计增量。  

> **不做快照**：回滚需要遍历全部增量，时间线性增长，用户体验极差。  

### 3. 搜索索引的分片、复制与失效恢复方案？  

- **分片**：基于 `workspace_id` 哈希分片，确保同一工作空间的文档在同一分片，查询时只路由到相关分片，降低跨分片查询开销。  
- **复制**：每个主分片至少 1 个副本（跨 AZ），实现 **读写分离** 与 **容灾**。  
- **失效恢复**：  
  1. **节点失效** → ES 自动将副本升为主分片，重新分配缺失的副本。  
  2. **分片不均衡** → 使用 **Shard Allocation Awareness** 根据 AZ 标签均衡分配。  
  3 **索引恢复** → 通过 **快照**（每日）在新节点恢复索引数据。  

- **实时增量**：采用 **Kafka → Bulk API**，每批次大小 100 条或 1 s，**保证 99%** 的写入延迟 < 300 ms。  

### 4. 离线编辑冲突如何合并？  

- **CRDT 合并**：每条本地操作都有唯一 **site_id** + **counter**，在服务器端使用 **LWW‑Element‑Set** 或 **RGA** 合并，冲突自动解决。  
- **人工干预**：若冲突涉及 **结构化块**（如表格同一单元格被两个用户编辑），服务器返回 **冲突列表**，前端弹出 **冲突解决 UI**（保留哪一方、手动合并）。  

### 5. 业务层面的权限检查放在哪？  

- **网关层**：**API Gateway** 完成 **身份验证**（JWT 解析）和 **基本权限**（是否属于工作空间）。  
- **服务层**：**业务 Service** 再次校验 **细粒度 ACL**（页面/块的 read/write 权限），防止 **水平越权**。  
- **数据库层**（可选）：使用 **PostgreSQL RLS**，在查询语句中自动过滤掉无权限的数据，形成 **防御性编程**。

## 心得与反思  

### 1. 本题最难的 1‑2 个设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **协同算法的选择**（OT vs CRDT） | Notion 的块种类繁多，文字块、表格块、看板块都有不同的编辑冲突特性。| 先列举每种块的 **冲突模型**，评估 OT 需要为每种块写专门的转化函数，维护成本高；CRDT 虽然需要为每种块实现对应的 CRDT，但一次实现后天然支持离线、去中心化。最终决定 **混合方案**：文字块内部 OT，结构化块 CRDT。|
| **无限历史的存储模型** | 需要兼顾 **写入性能**、**存储成本**、**回滚速度**。| 从“全快照” → “全增量日志” → “增量+定期快照”逐步逼近。采用 **增量日志 + 定期快照 + 对象存储归档**，在满足**快速回滚**（只需加载最近快照 + 少量增量）和**低成本**（增量日志压缩后占用极少空间）之间取得平衡。|

### 2. 新手最容易犯的错误（至少 2 条）  

| 错误 | 影响 | 正确做法 |
|------|------|----------|
| **把所有业务都塞进单体服务**（缺乏拆分） | 随着 QPS 增长，单体难以水平扩展，部署风险大。| 按 **业务边界**（块、协同、搜索、权限）拆分为 **微服务**，每个服务独立伸缩。 |
| **只考虑功能实现，不做容量估算** | 设计的规模与真实需求不匹配，面试官会质疑系统可行性。| 在需求分析阶段 **量化 DAU、QPS、存储**，用公式展示每层的 **吞吐量** 与 **容量**。 |
| **忽视缓存和读写分离** | 高并发读取会压垮主库，导致编辑延迟、页面渲染卡顿。| 引入 **Redis Cache** 与 **Postgres Read Replica**，实现 **读写分离**。 |
| **实时协同只用轮询** | 网络开销大、延迟高，无法实现秒级协同。| 使用 **WebSocket**（或 gRPC streaming）实现 **推送**，并配合 **Kafka** 做可靠传输。 |

### 3. 学习建议和可延伸的方向  

1. **系统设计基础**：  
   - 《Designing Data‑Intensive Applications》——理解 CAP、事务模型、流式处理。  
   - 《系统架构师面试宝典》——掌握常见架构模式（CQRS、Event‑Sourcing）。  

2. **协同编辑技术**：  
   - 阅读 **Google Docs OT**、**Microsoft Fluid Framework**（CRDT）源码。  
   - 实践：自己实现一个简易的 **文字块 OT** 或 **列表 CRDT**，体会冲突合并的细节。  

3. **分布式消息系统**：  
   - 深入学习 **Kafka**（分区、复制、消费组），在本地搭建集群实验。  

4. **搜索引擎**：  
   - 学习 **Elasticsearch** 的倒排索引、分片、路由策略。  
   - 练习 **实时增量索引**（使用 Logstash/Beats 或自研 Producer）。  

5. **高可用与容灾**：  
   - 实践 **Kubernetes** HPA、PodDisruptionBudget、StatefulSet。  
   - 了解 **Patroni**、**AWS Aurora** 多 AZ 架构。  

6. **前端协同**：  
   - 熟悉 **Yjs**、**Automerge** 等 CRDT 库的使用方式。  

> **核心思路**：先把 **业务流程** 用时序图/状态图画清楚，再决定 **数据模型**、**技术选型**，最后围绕 **容量**、**可用性** 进行 **横向扩展** 与 **容错** 设计。
