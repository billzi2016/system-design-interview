# 第 83 天：设计 多人在线游戏匹配系统（类王者荣耀）

> 生成日期：2026-03-04

---

## 1️⃣ 题目背景  
多人在线游戏匹配系统负责在类似《王者荣耀》这类 5v5（或 3v3）对战类手游中，将玩家实时快速地匹配到合适的对局，保证游戏体验的公平性、流畅性和高可用性。

## 2️⃣ 面试场景设定  
> **面试官**：  
> “假设我们要为一款即将上线的 5v5 MOBA 手游（类似王者荣耀）设计玩家匹配服务。请你从零开始设计一个高并发、低延迟、能够支撑千万级玩家的匹配系统。我们先从核心功能和关键指标入手，你会怎么规划？”

## 3️⃣ 功能性需求（Core Functional Requirements）  

| 编号 | 需求描述 |
|------|----------|
| **F1** | **实时匹配**：玩家在进入匹配队列后，系统在 **≤ 5 秒** 内完成匹配并返回房间号。 |
| **F2** | **匹配规则**：基于玩家段位（MMR）、胜率、英雄池、队伍人数、地理位置（IP）等多维度进行综合评分，确保对局公平。 |
| **F3** | **召回/重排**：当玩家长时间未匹配成功时，系统可适当放宽约束（如扩大 MMR 区间），并向玩家展示“匹配中，请稍候”。 |
| **F4** | **房间创建 & 分配**：匹配成功后自动创建游戏房间，分配到对应的游戏服务器实例（Region、负载均衡），返回房间入口信息给客户端。 |
| **F5** | **匹配日志 & 监控**：记录每次匹配的输入参数、匹配时长、匹配成功率等，用于后续数据分析和模型迭代。 |
| **F6** | **玩家取消/掉线处理**：玩家在匹配过程中主动退出或掉线，系统需立即从队列中剔除，并触发重新匹配（若已形成房间，则通知房主并做容错处理）。 |

## 4️⃣ 非功能性需求（Non‑Functional Requirements）  

| 编号 | 指标 | 估算值 | 说明 |
|------|------|--------|------|
| **N1** | **DAU（活跃用户数）** | 30 Million | 峰值时约占全体玩家的 10%。 |
| **N2** | **QPS（匹配请求峰值）** | 120 k QPS | 假设 5% 的 DAU 同时发起匹配请求，且每人平均 2 次匹配/小时。 |
| **N3** | **匹配延迟** | ≤ 5 秒（99% 匹配成功） | 包括排队、算法计算、房间创建全部耗时。 |
| **N4** | **可用性** | 99.99%（每月累计不可用时间 ≤ 4.38 分钟） | 必须做到跨机房容灾、自动故障转移。 |
| **N5** | **存储量** | 约 5 TB/日 | 匹配日志（≈ 200 KB/场）× 2 M 场/日 + 元数据、玩家排队快照等。 |
| **N6**（可选）| **扩展性** | 支持水平扩容至 10× 当前峰值 | 通过无状态服务、分片队列实现。 |

## 5️⃣ 系统边界（Scope）  

**本题需要设计并讨论的内容**  
- 匹配入口 API、请求流转、排队模型（基于 Redis/Zookeeper/消息队列等）  
- 匹配算法（MMR、分层、动态放宽策略）以及其可水平扩展实现  
- 房间/服务器分配机制、跨机房负载均衡  
- 高可用设计（多活部署、故障检测、回滚）  
- 监控、日志、数据持久化方案  

**本题不需要考虑的内容**  
- 游戏本身的业务逻辑（如战斗、结算、道具系统）  
- 客户端 UI 细节与渲染层实现  
- 第三方支付、社交（好友、聊天）功能  
- 具体的英雄平衡或游戏策划层面的规则（只需抽象为匹配维度）

## ## 解题思路总览  

1. **先把需求拆解成「功能」+「非功能」**，明确每个需求的关键指标（如 QPS、时延、可用性）。  
2. **估算规模**（玩家数、并发请求、存储量），把抽象数字转化为具体的系统容量需求。  
3. **画出最小可用（MVP）** 的高层框架：前端入口 → 匹配入口 API → 匹配队列 → 匹配算法服务 → 房间创建服务 → 游戏服务器。  
4. **逐块细化**：  
   - 数据模型（玩家、排队、房间、日志）  
   - API（请求/响应、错误码）  
   - 关键组件（排队层、匹配算法、房间调度）  
5. **在 MVP 基础上**，一步步加入 **水平扩容、容灾、监控、热更新** 等高级特性，使系统满足 **N1‑N6** 的非功能需求。  
6. **准备面试追问**：高 QPS、跨机房容灾、模型热更新的实现思路。  
7. **回顾与反思**：最难决策、常见错误、进一步学习方向。

## ## 第一步：理解需求与规模估算  

### 1️⃣ 功能需求要点（F 系列）  

| 编号 | 关键点 | 为何重要 |
|------|--------|----------|
| **F1** | **≤5 秒** 匹配完成 | 玩家期望即时进入战斗，时延过长直接导致流失。 |
| **F2** | 多维度匹配（MMR、胜率、英雄池、IP） | 保证对局公平、降低卡顿、提升玩家体验。 |
| **F3** | **召回/放宽** 机制 | 防止长尾玩家卡在队列里，提高整体匹配成功率。 |
| **F4** | 自动创建房间并分配服务器 | 匹配完后立即提供可连通的游戏实例。 |
| **F5** | 详细日志 + 监控 | 运营分析、模型迭代、故障定位必备。 |
| **F6** | 取消/掉线处理 | 防止「僵尸玩家」影响匹配质量。 |

### 2️⃣ 非功能需求要点（N 系列）  

| 编号 | 指标 | 估算值 | 关键影响点 |
|------|------|--------|------------|
| **N1** | DAU | 30 M | 决定 **用户画像**、后端容量基准。 |
| **N2** | 峰值 QPS | 120 k | 直接决定 **入口层、排队层** 的水平扩容需求。 |
| **N3** | 匹配延迟 ≤5 s（99%） | 需要 **低时延队列 + 并行匹配**。 |
| **N4** | 可用性 99.99% | 必须 **多活、跨机房、自动故障转移**。 |
| **N5** | 存储 5 TB/日 | 日志写入、归档、冷热分层。 |
| **N6** | 扩展到 10×峰值 | **无状态服务 + 分片** 设计。 |

### 3️⃣ 粗略容量计算  

| 项目 | 计算方式 | 结果 |
|------|----------|------|
| **匹配请求峰值** | 30 M DAU × 5% 同时匹配 × 2 次/小时 ÷ 3600 s | ≈ 120 k QPS |
| **并发排队玩家** | QPS × 匹配时长（上限 5 s） | ≤ 600 k 玩家 |
| **每场日志大小** | 200 KB | – |
| **每日对局数** | 2 M 场（≈ 30 M × 5% × 2 / 60） | – |
| **日志总量** | 200 KB × 2 M ≈ 400 GB + 归档压缩 ≈ 5 TB/日 | – |

> **小结**：系统需要支撑 **约 600k 并发玩家**，**120k QPS** 的入口流量，并且 **日志写入量巨大**，因此必须采用 **分布式、无状态、水平可扩** 的架构。

## ## 第二步：高层架构设计  

### 1️⃣ MVP（最小可用系统）  

```
+-------------------+       +-------------------+       +-------------------+
|   客户端 (App)    | --->  |   API Gateway     | --->  |   匹配入口 Service|
+-------------------+       +-------------------+       +-------------------+
                                            |
                                            v
                                      +-----------+
                                      |   排队层   | (Redis Sorted Set)
                                      +-----------+
                                            |
                                            v
                                   +-------------------+
                                   |   匹配算法 Worker|  (多实例并行)
                                   +-------------------+
                                            |
                                            v
                                   +-------------------+
                                   |   房间创建 Service|
                                   +-------------------+
                                            |
                                            v
                                   +-------------------+
                                   |   游戏服务器池   |
                                   +-------------------+
```

**解释**  
- **API Gateway**：统一入口，做流量控制、鉴权、限流。  
- **匹配入口 Service**：无状态，接收玩家请求，写入排队层。  
- **排队层（Redis）**：提供 **低时延、原子性** 的 `ZADD/ZRANGE` 操作，用来维护 **按分数（MMR）排序的等待队列**。  
- **匹配算法 Worker**：周期性拉取队列（或基于事件），执行 **分层/二分匹配**，生成房间。  
- **房间创建 Service**：负责分配 **游戏服务器实例**（Region + 负载均衡），返回房间号/入口给玩家。  

> **如果不使用 Redis 排队**，直接在数据库里查询会产生 **高延迟、锁竞争**，无法满足 5 秒时限。  

### 2️⃣ 关键设计点  

| 设计点 | MVP 方案 | 为什么要这么做 |
|--------|----------|----------------|
| **入口层伸缩** | API Gateway + Nginx/Envoy + 自动弹性伸缩 | 通过 **水平扩容** 抵御突发流量，且能统一做限流/熔断。 |
| **排队持久化** | Redis 主从+持久化（RDB+AOF） | **内存快**、支持 **有序集合**；持久化防止掉电丢队列。 |
| **匹配算法** | 单机多线程（Worker） | 简单实现，满足 120k QPS（每秒约 2000 匹配任务）时**并行**处理。 |
| **房间调度** | 简单的 **Round‑Robin** + 区域标签 | 初期实现快速，后续可替换为 **负载感知** 调度。 |
| **日志存储** | Kafka → HDFS / OSS | 高吞吐、可离线批处理，满足 5 TB/日。 |
| **监控** | Prometheus + Grafana + Alertmanager | 开源且易集成，覆盖 QPS、延迟、错误率等指标。 |

## ## 第三步：数据库设计  

### 1️⃣ 业务数据（持久化）  

| 表名 | 说明 | 关键字段 | 索引 |
|------|------|----------|------|
| **users** | 玩家基础信息 | `user_id PK`, `mmr`, `win_rate`, `region`, `hero_pool (json)` | `idx_mmr`, `idx_region` |
| **match_queue_snapshot** | 队列快照（用于召回、分析） | `snapshot_id PK`, `user_id FK`, `queue_ts`, `score` | `idx_user_ts` |
| **rooms** | 已创建房间元数据 | `room_id PK`, `status (waiting|running|closed)`, `region`, `server_ip`, `created_ts` | `idx_status_region` |
| **room_players** | 房间‑玩家映射 | `room_id PK+FK`, `user_id PK+FK`, `team (0/1)`, `joined_ts` | `idx_room` |
| **match_logs** | 每局匹配日志（写入 Kafka） | `log_id PK`, `room_id`, `user_id`, `mmr_before`, `mmr_after`, `match_time_ms`, `match_params (json)` | `idx_room_time` |

> **为什么不把排队信息放在关系库？**  
> 排队是 **高频写/读**，且要求 **毫秒级** 响应，使用 **内存 KV（Redis）** 更合适。业务元数据（玩家档案、房间状态）则需要 **强一致**，放在关系库（MySQL/PostgreSQL）或 NewSQL（TiDB）中。

### 2️⃣ 排队层（Redis）结构  

- **Sorted Set**：`queue:{region}:{mode}`（例如 `queue:cn:5v5`）  
  - **member**：`user_id`  
  - **score**：`mmr_normalized + timestamp_factor`（用于 FIFO 同分时）  

- **Hash**：`player:{user_id}` 保存 **召回计数、最近匹配时间、放宽系数** 等临时状态。  

> **原子性**：使用 Lua 脚本一次性完成 `ZADD` + `HSET`，防止并发写入导致不一致。  

### 3️⃣ 日志链路  

1. 匹配成功后，**Match Worker** 把一条结构化日志写入 **Kafka topic** `match_events`.  
2. Kafka 持久化后，**Flume/Spark Streaming** 把数据落盘到 **HDFS** 或 **对象存储**（OSS）。  
3. 离线作业每日聚合生成 **匹配成功率、MMR 变化分布** 等报表。

## ## 第四步：核心 API 设计  

> **原则**：RESTful + JSON，统一错误码，支持 **幂等** 与 **幂等 token**（防止网络重试导致重复入队）。

| 方法 | 路径 | 功能 | 请求体 | 响应体 | 关键字段 |
|------|------|------|--------|--------|----------|
| `POST` | `/match/v1/enter` | 玩家进入匹配队列 | `{ "user_id":123, "mode":"5v5", "client_version":"1.2.3" }` | `{ "queue_id":"q_20240526_001", "estimated_wait_sec":3 }` | `queue_id` 用于后续查询/取消 |
| `GET` | `/match/v1/status/{queue_id}` | 查询匹配进度 | - | `{ "status":"matching|matched|timeout", "room_id":null, "progress":45 }` | `status` 可直接映射 UI |
| `POST` | `/match/v1/cancel` | 取消匹配 | `{ "queue_id":"q_...", "user_id":123 }` | `{ "result":"ok" }` | 幂等 |
| `POST` | `/match/v1/heartbeat` | 客户端心跳（防掉线） | `{ "queue_id":"...", "user_id":123 }` | `{ "result":"alive" }` | 服务器可根据未收到心跳剔除 |
| `GET` | `/match/v1/room/{room_id}` | 获取房间入口信息 | - | `{ "room_id":"r_...", "server_ip":"10.1.2.3", "port":9000, "token":"abc123" }` | 供客户端直接连接游戏服务器 |

### 错误码示例  

| Code | 含义 | 场景 |
|------|------|------|
| `2000` | SUCCESS | 正常 |
| `4001` | 参数错误 | 缺少 `user_id` |
| `4002` | 已在队列中 | 幂等请求 |
| `4030` | 匹配被取消（玩家主动） | `cancel` |
| `5001` | 内部服务异常 | 匹配 Worker 死亡 |
| `5040` | 匹配超时（>5 s） | 需要放宽或提示玩家 |

## ## 第五步：详细组件设计  

下面逐个拆解 **匹配入口 Service → 排队层 → 匹配 Worker → 房间创建 Service → 游戏服务器**，并说明内部实现细节、容错与伸缩方式。

### 1️⃣ 匹配入口 Service  

- **技术栈**：Go/Java（高并发、低 GC），部署在 **容器（K8s）** 中。  
- **职责**：  
  1. **鉴权**（Token → user_id）  
  2. **查询玩家画像**（MMR、地区、英雄池） → 写入 **Redis** 排队。  
  3. **返回 queue_id**，并写入 **match_queue_snapshot**（持久化）供召回。  
- **实现要点**：  
  - **幂等**：同一 `user_id+mode` 只能在同一地区的队列里出现一次，使用 `SETNX` + TTL 防止重复入队。  
  - **限流**：基于 **IP/用户** 的 **令牌桶**，防止刷请求。  

### 2️⃣ 排队层（Redis）  

| 操作 | Lua 脚本（示例） | 解释 |
|------|------------------|------|
| **入队** | `redis.call('ZADD', key, score, user_id); redis.call('HSET', 'player:'..user_id, 'queue_ts', now, 'relax_factor', 0);` | 原子写入有序集合 + 玩家状态。 |
| **取队列** | `redis.call('ZRANGEBYSCORE', key, minScore, maxScore, 'LIMIT', 0, batchSize)` | 匹配 Worker 拉取同分段玩家，避免全表扫描。 |
| **剔除** | `redis.call('ZREM', key, user_id); redis.call('DEL', 'player:'..user_id)` | 取消或掉线时调用。 |

- **分区**：为每个 **Region+Mode** 建立独立 **Sorted Set**，天然实现 **地理分区**，降低跨机房网络延迟。  
- **失效 TTL**：`player:{uid}` 设置 30 s TTL，Worker 每次读取后 `EXPIRE` 重置，防止僵尸玩家残留。  

### 3️⃣ 匹配算法 Worker  

#### 3.1 基本流程  

1. **轮询**（或 **Redis Pub/Sub**）获取待匹配玩家列表（批量 100‑200 条）。  
2. **分层**：先按 **MMR 区间**（如 ±50）分组；若组内人数不足，则 **放宽**（±100、±200 …）**递增**。  
3. **组合**：从同一分层中挑选 **5 名玩家**（或 3 名）构成 **Team A**，再挑选 **另一组** 形成 **Team B**。  
4. **约束检查**：  
   - **英雄冲突**（同英雄池冲突）  
   - **同地区**（IP）不超过阈值  
   - **队伍人数**（5 人）  
5. **生成房间**：调用 **房间创建 Service**，获取 `room_id` 与服务器信息。  
6. **推送结果**：写入 **Redis** `room:{room_id}`，并通过 **推送（WebSocket / Push）** 或 **轮询** 通知玩家。  

#### 3.2 并行化  

- **Worker 集群**：每台机器跑 **N** 个 Goroutine/线程（依据 CPU 核数），共享同一 Redis 队列。  
- **竞争控制**：使用 **Redis Lua 脚本** 实现 **抢占式取走**（`ZPOP`）玩家，防止两台 Worker 抢同一个玩家。  

#### 3.3 放宽策略（召回）  

| 时间已等待 | 放宽系数 | MMR 区间 | 说明 |
|------------|----------|----------|------|
| 0‑2 s | 1.0 | ±50 | 严格匹配 |
| 2‑4 s | 1.5 | ±100 | 放宽 1.5 倍 |
| 4‑5 s | 2.0 | ±200 | 最终放宽，保证 5 s内出局 |

- **实现**：在 `player:{uid}` 中记录 `wait_seconds`，Worker 根据该字段动态计算 `score`（`mmr * factor + timestamp`）。  

#### 3.4 关键指标监控  

- **匹配耗时**：`match_start_ts` → `match_end_ts`（Prometheus histogram）  
- **每秒匹配对局数**：`match_success_qps`  
- **放宽次数**：`relax_counter`  

### 4️⃣ 房间创建 Service  

- **技术栈**：Java SpringBoot / Go，提供 **REST** 接口。  
- **职责**：  
  1. **选择游戏服务器实例**：基于 **Region**、**当前负载（CPU/内存/玩家数）**、**机房健康**。  
  2. **创建房间元数据**：写入 `rooms` 表，状态 `waiting`。  
  3. **分配房间号**：使用 **Snowflake** 或 **UUID**（全局唯一）。  
  4. **返回** `room_id、server_ip、port、token` 给匹配 Worker。  

- **负载感知调度**（后续可升级）：  
  - **实时指标**（Prometheus） → **调度器**（基于最小连接数或最小 CPU 使用率）  
  - **灰度发布**：新服务器先接收少量房间，监控后逐步放量。  

### 5️⃣ 游戏服务器池  

- **容器化**（Docker）+ **Kubernetes** 自动伸缩。  
- **Region**：每个大区（亚太、欧美）部署独立集群，**网络延迟**最小化。  
- **弹性伸缩**：基于 **房间创建速率** 或 **CPU 使用率** 自动扩容。  

### 6️⃣ 监控、告警、日志  

| 监控维度 | 采集方式 | 报警阈值 |
|----------|----------|----------|
| **入口 QPS** | Envoy/Prometheus `envoy_http_downstream_rq_total` | > 150k (5 min) |
| **匹配时延** | Worker `match_latency_histogram` | 99th percentile > 4.5 s |
| **房间创建成功率** | `room_create_success_total` / `room_create_total` | < 99% |
| **Redis 延迟** | `redis_latency_seconds` | > 2 ms |
| **服务异常** | `process_cpu_seconds_total`、`process_memory_usage_bytes` | > 80% CPU |
| **日志积压** | Kafka consumer lag | > 30 s |

- **日志**：所有业务日志统一走 **ELK**（Filebeat → Logstash → Elasticsearch）+ **Kibana** 可视化。

## ## 第六步：扩展性与高可用设计  

### 1️⃣ 多活部署（跨机房）  

| 组件 | 多活方式 | 关键点 |
|------|----------|--------|
| **API Gateway** | **全局 DNS + Anycast**，每个机房都有同名入口 IP。 | 客户端就近路由，降低入口时延。 |
| **匹配入口 Service** | **K8s Deploy** + **Horizontal Pod Autoscaler**，每个机房独立一套。 | 同步 **用户画像缓存**（MySQL 主从或 TiDB 分布式） |
| **Redis 排队** | **Redis Cluster** 跨机房 **Active‑Active（CRDT）**（如 **Dragonfly**）或 **双写 + 双读**。 | 必须保证 **一致性** 与 **低时延**，如果跨机房延迟 > 5 ms，可采用 **分区（Region+Mode）**，每个地区独立排队。 |
| **匹配 Worker** | **无状态**，可以在任意机房部署，**同一地区的玩家只能由本地区 Worker 处理**。 | 防止跨机房网络抖动影响匹配时延。 |
| **房间创建 Service** | **Leader Election**（使用 **etcd/Zookeeper**）保证全局唯一的房间 ID，故障时快速切换。 | 采用 **Snowflake** + **机房 ID**，即使出现双写也能保证唯一。 |
| **游戏服务器** | **独立机房**（Region）部署，**不跨机房**。 | 匹配后直接把玩家路由到同地区服务器，降低 RTT。 |

#### 故障转移流程（示例）  

1. **监控发现** 某机房 `cn-1` 的 **API Gateway** 心跳失效。  
2. **DNS TTL** 设置为 **30 s**，自动切换到 `cn-2` 的 Anycast IP。  
3. **Redis Sentinel** 检测到 `cn-1` 主节点挂掉，自动把 `cn-2` 的从节点提升为新主。  
4. **匹配 Worker** 在 `cn-2` 检测到 **queue 长度激增**，触发 **水平扩容**（HPA）。  
5. **玩家** 若在 `cn-1` 正在匹配，**客户端** 会自动重新请求入口（5 s 超时后重试），最终落到 `cn-2`。  

> **如果不做跨机房容灾**，单点故障会直接导致 **99.99% 可用性** 目标无法达成。  

### 2️⃣ 热更新 MMR / 匹配模型  

- **模型存放**：使用 **Consul/etcd** 保存 **JSON/YAML** 配置（如分层阈值、放宽系数）。  
- **加载方式**：Worker 启动时读取配置，并 **订阅** 配置变更（Watcher）。  
- **热更新流程**：  
  1. 运维在 **etcd** 中上传新模型文件（版本号）。  
  2. 所有 Worker 收到 **watch 事件**，在 **下一个批次** 读取新模型（无重启）。  
  3. 若新模型出现异常，**回滚**只需恢复旧版本即可。  

- **安全措施**：  
  - **灰度发布**：先在 **测试机房**（10% 流量）启用新模型，监控关键指标（匹配成功率、MMR 分布）后再全量放开。  
  - **模型验证**：使用 **离线模拟**（Spark）对新模型进行 **AB Test**，确保不破坏公平性。  

### 3️⃣ 水平扩容（10×）  

| 维度 | 扩容方式 |
|------|----------|
| **入口层** | 增加 **API Gateway** 实例 → **负载均衡**（L4） |
| **匹配入口 Service** | **K8s Deployment** + **HPA**（CPU/请求数） |
| **Redis** | **Cluster** → 增加 **分片数**（每片 1‑2 GB） |
| **Worker** | **无状态** → 按机器数线性扩容；使用 **Kafka** 作为 **队列**（替代 Redis）可以更好水平扩展。 |
| **房间创建 Service** | **微服务化** → 每个 Region 单独部署，使用 **服务网格**（Istio）实现流量控制。 |
| **游戏服务器** | **容器化 + 自动伸缩**（基于房间创建速率） |

> **关键点**：所有服务都保持 **无状态**，状态只保存在 **外部持久化**（Redis、MySQL、Kafka），这样才能做到 **随时增删节点** 而不影响业务。

## ## 第七步：常见面试追问与回答  

### 1️⃣ “如果我们要在 1 秒内完成 5 万 QPS 的匹配请求，你会如何改造排队层和算法实现？”  

| 改造点 | 方案 | 解释 |
|--------|------|------|
| **入口层** | **Nginx+Lua + Rate Limiting**，预先做 **流量削峰**（Token Bucket） | 防止突发流量直接压垮后端。 |
| **排队层** | **Kafka** 代替 Redis，使用 **Topic 分区**（每个 Region+Mode 一个分区） | Kafka 天然 **高吞吐**（百万 QPS），且提供 **持久化**，避免因 Redis GC 产生的卡顿。 |
| **匹配算法** | **离线批处理 + 在线增量**：<br>1) 每秒将最新的玩家快照写入 **Kafka**，<br>2) **流式计算**（Flink）在 **窗口**（1 s）内完成 **近似匹配**，生成房间事件。 | 将 **匹配计算并行化** 到 **Flink 任务**，每个任务处理 1‑2 k 条玩家，整体 5 万 QPS 可在 1 s 内完成。 |
| **房间创建** | **异步回调**：匹配 Worker 只负责 **生成匹配事件**，真正的 **房间创建** 交给 **独立的微服务**，使用 **消息队列**（Kafka）实现 **解耦**。 | 减少匹配路径的阻塞时间。 |
| **缓存层** | **玩家画像** 放入 **本地缓存（Caffeine）+热点刷新**，降低每次请求的 DB 读次数。 | 缓存命中率提升后，整体时延可降至 **<200 ms**。 |

> **为什么不直接把 Redis 直接扩容到 5 万 QPS？**  
> Redis 单实例最大 QPS 大约在 **100‑200 k**（取决于操作），但在 **高并发写入（ZADD）+ 脚本** 场景会出现 **CPU 瓶颈**。使用 **Kafka + Flink** 能把计算拆分成 **数百个并行任务**，更符合 “秒级批处理” 的需求。  

---

### 2️⃣ “请说明你会如何做跨机房的容灾切换，保证匹配服务的 99.99% 可用性。”  

1. **多活 DNS + Anycast**  
   - 每个机房部署 **相同的 Anycast IP**，客户端 DNS TTL 设置为 **30 s**。当某机房不可用，BGP 自动收敛到其它机房的 IP。  

2. **全链路健康检查**  
   - **L7**（Gateway）→ **L4**（Redis）→ **L3**（Worker）→ **L2**（Room Service）使用 **Prometheus + Alertmanager**。  
   - 若某层异常连续 **N** 次（如 3 次 5xx），立即触发 **自动切换**。  

3. **数据同步**  
   - **MySQL/TiDB** 使用 **双活复制**（Galera/ TiDB），确保玩家画像、房间元数据在各机房实时同步。  
   - **Redis** 使用 **Active‑Active（CRDT）** 或 **双写**（写入本地+远端）保证排队信息一致。  

4. **自动故障转移流程**（示例）  
   - **检测**：监控发现 `cn-1` 的 Redis 主节点失联。  
   - **切换**：Sentinel 自动把 `cn-2` 从节点提升为主，`cn-1` 的写请求自动转到 `cn-2`（通过客户端重连或 Proxy）。  
   - **通知**：`cn-1` 的 API Gateway 返回 **503**，客户端立刻重新解析 DNS，路由到 `cn-2`。  

5. **灾难恢复演练**  
   - 每月进行 **Chaos Monkey** 测试：随机关闭某机房的服务实例，验证系统在 **<30 s** 内完成切换并保持 **匹配成功率 > 98%**。  

> **不做跨机房容灾**，单点故障会导致 **整小时不可用**，远超 99.99% SLA（每月不可用时间 ≤ 4.38 min）。  

---

### 3️⃣ “匹配系统的 MMR 计算模型会随时间更新，你打算如何实现模型的热更新而不影响在线匹配？”  

| 步骤 | 具体实现 | 关键点 |
|------|----------|--------|
| **模型存储** | **etcd/Consul** 中保存 **JSON**（如 `mmr_rule_v20240526.json`） | 支持 **版本化**、**watch** 机制。 |
| **加载方式** | Worker 在启动时读取最新模型并 **缓存** 在内存（Map）。 | 读取一次，避免每次匹配都去磁盘。 |
| **热更新** | 使用 **etcd Watcher**：模型文件更新时推送 **事件** 给所有 Worker。<br>Worker 收到事件后 **原子切换** 内存中的模型对象（使用 `volatile`/`AtomicReference`）。 | **无需重启**，只影响新来的匹配请求。 |
| **灰度发布** | 在 **测试机房** 部署新模型，监控 **匹配成功率、MMR 分布**，确认无异常后在 **生产机房** 批量推送。 | 防止模型 bug 直接影响全体玩家。 |
| **回滚** | 若新模型异常，直接在 etcd 中恢复旧版本，Worker 再次收到事件切回旧模型。 | 回滚时间 **秒级**。 |
| **安全校验** | 推送前执行 **Schema 校验**（JSON Schema）和 **单元测试**（匹配模拟）。 | 防止错误的配置导致系统崩溃。 |

> **如果直接在代码里写死模型**，每次改动都需要 **全量部署**，停机时间长且风险大。使用 **配置中心 + 动态加载** 能在 **毫秒级** 完成模型切换，满足业务快速迭代需求。

## ## 心得与反思  

### 1️⃣ 本题最难的 1‑2 个设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **排队层选型（Redis vs Kafka vs 自研 MQ）** | 需要兼顾 **低时延**、**高吞吐**、**数据一致性** 与 **跨机房容灾**。 | - 初始想到 Redis 因为 ZSET 很适合按分数排序。<br>- 随着 QPS 目标升到 **5 万 QPS**，发现单实例 Redis 的写入（ZADD）会成为瓶颈。<br>- 再评估 Kafka：天然持久化、分区扩展、对消费侧（匹配 Worker）解耦。<br>- 最终方案：**低峰期使用 Redis**，**高峰/大规模**切换到 **Kafka+Flink**，实现弹性。 |
| **跨机房容灾的实现细节** | 必须保证 **99.99% SLA**，但跨机房网络延迟、数据复制冲突、故障切换时间都是难点。 | - 先划分系统为 **入口层、排队层、匹配层、房间层**，每层独立做 **多活**。<br>- 选用 **Anycast + 短 TTL DNS** 解决入口路由问题。<br>- 对状态（玩家排队）采用 **Active‑Active Redis** 或 **双写**，避免单点。<br>- 通过 **监控 + 自动化脚本** 完成 **快速故障转移**，并在每次演练中验证切换时长 < 30 s。 |

### 2️⃣ 新手最容易犯的错误（至少 2 条）  

| 错误 | 说明 | 正确做法 |
|------|------|----------|
| **把匹配队列放在关系数据库** | DB 的事务与磁盘 I/O 使得 **写入/读取延迟** 在 10‑100 ms，难以满足 **≤5 秒** 的整体时延。 | 使用 **内存 KV（Redis）** 或 **消息队列（Kafka）**，提供 **O(1)** 的写入/读取并支持有序集合。 |
| **把所有业务写在单体服务里** | 单体服务 **不可水平扩展**，一旦流量激增，整个进程会挂掉，恢复时间长。 | 将系统拆分为 **无状态微服务**（入口、匹配、房间创建），所有状态外部化（Redis、DB），通过 **K8s HPA** 实现弹性伸缩。 |
| **忽视玩家掉线/取消的幂等性** | 若玩家因网络抖动重复发送 “进入匹配” 请求，会导致 **同一玩家多次入队**，匹配结果错误。 | 在 **入队 API** 中使用 **幂等键**（`user_id+mode`），用 `SETNX` 或 **Redis Lua** 保证原子性。 |
| **只关注匹配成功率，忽视时延分布** | 只看整体成功率可能掩盖 **99th percentile** 延迟超标，玩家体验差。 | 在监控中使用 **Histogram** 记录 **匹配时延分布**，并设置 **SLA（99% ≤ 5 s）** 进行报警。 |

### 3️⃣ 学习建议和可延伸的方向  

1. **深入了解分布式队列**：  
   - **Redis ZSET** 的原子脚本、持久化机制。  
   - **Kafka** 的分区、消费者组、事务语义。  
   - **RabbitMQ / Pulsar** 作为备选方案。  

2. **学习流式计算框架**（Flink、Spark Structured Streaming）：  
   - 如何在 **窗口** 中做 **近实时匹配**，实现 **秒级批处理**。  

3. **掌握高可用架构**：  
   - **Consul/etcd** 的服务发现、配置中心、Leader Election。  
   - **Kubernetes** 的 StatefulSet、PodDisruptionBudget、PodAntiAffinity。  

4. **算法层面的匹配模型**：  
   - **Elo / TrueSkill** 等 MMR 计算公式。  
   - **分层 + 双向搜索**、**遗传算法**、**机器学习（RL）** 进行动态放宽。  

5. **性能压测**：  
   - 用 **locust / JMeter / k6** 对 API、Redis、Kafka 进行 **压测**，找出瓶颈并调优。  

6. **系统可观测性**：  
   - **OpenTelemetry** + **Prometheus** 的全链路追踪。  
   - **Grafana Loki** / **ELK** 日志聚合。  

> **总结**：系统设计的核心不是「写出一张图」而是「把需求 → 约束 → 方案 → 取舍」的思考过程。面试时把每一步的 **动机**、**权衡**、**备选** 说清楚，就能展示出系统设计的成熟度。
