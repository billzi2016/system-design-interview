## 题目背景
Twitch 是全球领先的实时互动视频流媒体平台，主要面向游戏玩家、内容创作者以及观众，提供直播、弹幕聊天、打赏、频道订阅等功能。面试官希望你从零开始设计一个能够支撑大规模并发直播的系统。

**面试官**：

## 功能性需求
| 编号 | 功能描述 |
|------|----------|
| 1 | **主播推流**：主播通过 RTMP/RTMPS 将实时视频推送到平台，支持 1080p 60fps、720p 30fps 等多种码率。 |
| 2 | **观众观看**：观众通过 HLS/DASH 或 WebRTC 拉流观看，平台需实现 **自适应码率**（ABR）以适配不同网络条件。 |
| 3 | **实时弹幕/聊天**：观众可以在观看时发送弹幕/聊天消息，消息需要在几百毫秒内送达所有在线观众。 |
| 4 | **互动打赏与订阅**：观众可以给主播送礼物、订阅频道，系统需实时更新主播的收入统计并推送通知。 |
| 5 | **频道推荐**：根据用户兴趣、历史观看记录等，为观众推荐正在直播或即将开播的频道。 |
| 6 | **回放与剪辑**：直播结束后，平台自动保存视频并提供回放、剪辑功能，供观众点播。 |

## 非功能性需求（估算指标）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **DAU（活跃用户数）** | 30 Million | 包括主播、观众、聊天用户。 |
| **并发观看峰值** | 15 Million 观看流 | 高峰时段（美国周五晚上）预计 1.5 M 并发直播，每场平均 10 k 观看人数。 |
| **推流 QPS** | 2 Million req/s | 包括 RTMP 推流入口、转码请求、弹幕写入等。 |
| **观看 QPS** | 30 Million req/s | HLS/WEBRTC 分片请求、清单请求等。 |
| **聊天消息延迟** | ≤ 300 ms | 从用户发送到所有观众收到的端到端延迟。 |
| **视频转码延迟** | ≤ 5 s（从推流到第一个分片可播放） | 需要实时转码成多码率。 |
| **可用性** | 99.99%（每月累计停机时间 ≤ 4.38 h） | 包括直播、聊天、回放等核心服务。 |
| **存储容量** | 200 PB | 假设每小时直播产生 3 TB 原始流，保存 30 天回放及剪辑。 |
| **成本控制** | 单月运营成本 ≤ \$10 M | 通过 CDN、冷热存储、弹性伸缩等手段控制。 |

# 题解

# 设计一个支撑大规模并发直播的系统（类似 Twitch）

> **温馨提示**：下面的每一步都把 **“为什么要这么做”** 说清楚，帮助你从“要怎么做”走到“为什么这么做”。如果你是第一次做系统设计，请把每一段都当成课堂笔记，逐字阅读、思考、记笔记。  

---

## ## 解题思路总览  

1. **从业务出发**：先把功能需求（推流、观看、弹幕、打赏、推荐、回放）拆成 **业务流程**，再把每个流程映射到系统组件。  
2. **估算规模**：用给出的 DAU、并发数、QPS 等指标，算出对 **网络、计算、存储** 的粗略需求。  
3. **最小可用系统（MVP）**：先画出只有单机/单点的最简架构，验证功能完整性。  
4. **逐层演进**：针对 **高并发、低延迟、高可用、可扩展** 的非功能需求，逐层加入 **负载均衡、缓存、分布式存储、消息队列、容灾** 等。  
5. **细化每个核心组件**：包括 **推流入口、转码集群、CDN、播放器、弹幕系统、计费系统、推荐系统、回放存储**。  
6. **接口（API）设计**：明确请求/响应、鉴权、限流等细节。  
7. **高可用与弹性**：从 **故障检测 → 自动恢复 → 多活部署** 完整闭环。  
8. **面试追问准备**：弹幕高并发实现、转码弹性伸缩、全球 CDN 选址等。  

下面按照 **“需求 → 高层 → 细节 → 可用性 → 面试追问”** 的顺序，逐章展开。

## ## 第一步：理解需求与规模估算  

| 需求 | 关键点 | 规模估算（峰值） | 影响的系统资源 |
|------|--------|----------------|----------------|
| 主播推流 | RTMP/RTMPS、1080p60、720p30、转码 | **2 M req/s**（入口）<br>每条流≈5 Mbps（720p）~15 Mbps（1080p） | 网络入口宽带、Ingress 服务器、转码 CPU/GPU |
| 观众观看 | HLS/DASH 或 WebRTC、ABR | **30 M req/s**（分片/manifest）<br>平均并发 15 M 观看流，每流 3 Mbps（平均） | CDN 带宽、边缘缓存、负载均衡 |
| 弹幕/聊天 | 低延迟 ≤300 ms、200 k 条/秒 | **200 k msg/s** | 消息队列、WebSocket/QUIC 连接、内存缓存 |
| 打赏/订阅 | 实时计费、推送通知 | 估计 0.5% 观看用户产生打赏 → ~75 k 事件/秒 | 事务数据库、消息系统、推送服务 |
| 推荐 | 基于兴趣、历史、热点 | 30 M DAU 每日查询 2–3 次 → ~60 M 查询/天 | 实时/离线特征库、召回服务、缓存 |
| 回放/剪辑 | 保存 30 天、点播 | 200 PB 存储（约 3 TB/h） | 对象存储、冷/热分层、转码产出 |
| 高可用 | 99.99% SLA | - | 多活、故障转移、监控告警 |

**计算示例**（帮助你理解为何需要这么多资源）  

- **观看带宽**：15 M 并发 × 3 Mbps ≈ **45 Tbps**（约 5.6 PB/s）。显然我们不能让所有流直接走核心网络，必须 **CDN 边缘缓存** 把流量拉到离用户最近的节点。  
- **转码需求**：假设 1 M 同时推流（峰值），每流需要 2 套转码（720p + 480p），每套转码 CPU≈2 vCPU，GPU≈1。则约 **4 M vCPU + 2 M GPU**，需要 **弹性伸缩**。  

> **为什么要做这些估算？**  
> - 能帮助你判断是“单机能否撑住”还是“必须分布式”。  
> - 能在后面的架构图里明确 **瓶颈点**（入口、转码、CDN、弹幕）。  
> - 面试官常会问“如果流量翻倍怎么办”，有数据支撑的答案更有说服力。

## ## 第二步：高层架构设计  

### 2.1 整体框图（从左到右）

```
+----------------+      +----------------+      +----------------+      +-------------------+
|   客户端 (Web/ | <--->|   CDN Edge     |<---->|   CDN PoP /    |<---->|   核心数据中心   |
|   Mobile App) |      | (HLS/DASH/WEB |      |  Origin Cache  |      | (转码、存储、DB) |
+----------------+      |  RTC)          |      +----------------+      +-------------------+
        ^                     ^   ^                 ^   ^                     ^
        |   WebSocket/QUIC   |   |  HTTP(s)        |   |  HTTP(s)            |
        |   (弹幕、打赏)     |   |  (分片、清单)   |   |  (转码入口)         |
        |                     |   |                 |   |                     |
+----------------+   +----------------+   +----------------+   +---------------------+
|   推流入口    |   |   转码集群     |   |   业务服务层   |   |   持久化存储 (OSS) |
| (RTMP/RTMPS)  |   | (FFmpeg/GPU)   |   | (用户、频道、   |   | (对象存储、Cold)   |
+----------------+   +----------------+   | 计费、推荐)   |   +---------------------+
                                          +----------------+
```

> **解释每块为什么要这么划分**  

| 模块 | 目的 | 不这么划分会有什么问题 |
|------|------|------------------------|
| **客户端** | 发起推流、拉流、弹幕、打赏等交互 | 直接连到核心中心会导致跨国高延迟、带宽浪费 |
| **CDN Edge** | 最近的网络节点缓存分片、提供低延迟播放 | 没有 CDN，所有流量都要回源，核心网络瞬间崩溃 |
| **CDN PoP / Origin Cache** | 跨地域缓存、热点直播的边缘热点复制 | 只靠单一中心，热点直播时会产生热点热点的网络热点，导致卡顿 |
| **转码集群** | 把主播上传的原始码流实时转成多码率、封装 HLS/DASH | 不做转码只能单码率，无法适配不同网络，用户体验差 |
| **业务服务层** | 用户鉴权、频道管理、弹幕/打赏、推荐等业务逻辑 | 把业务逻辑混在转码/存储里会导致耦合、难扩展 |
| **持久化存储** | 长期保存录像、剪辑、封面等 | 如果直接写到本地磁盘，容量不足且不可弹性伸缩 |

### 2.2 核心数据流

1. **推流**：主播 → **RTMP Ingress** → **转码入口** → **转码集群** → **对象存储 (原始 & 多码率)** → **CDN 拉取**  
2. **观看**：观众 → **CDN Edge** → **获取 Manifest** → **分段拉取** → **播放器 (ABR)**  
3. **弹幕**：观众 → **WebSocket/QUIC** → **弹幕写入队列** → **弹幕分发服务** → **推送到所有在线观众**  
4. **打赏**：观众 → **打赏 API** → **事务 DB + 消息队列** → **实时计费服务** → **推送通知**  
5. **推荐**：离线特征 → **特征库** → **召回服务** → **缓存** → **前端调用**

## ## 第三步：数据库设计  

### 3.1 选型原则  

| 数据类型 | 读写特征 | 推荐存储 | 说明 |
|----------|----------|----------|------|
| **用户 / 频道元数据** | 读多写少（更新头像、粉丝数） | **关系型 DB（MySQL / Aurora）** + **读写分离** | 强事务、一致性要求 |
| **直播状态（正在推流、观众数）** | 高并发读写（每秒上千次） | **分布式 KV（Redis / DynamoDB）** | 超低延迟、TTL 支持 |
| **弹幕历史** | 写多读少（回放时需要） | **时序 DB（ClickHouse / Apache Druid）** 或 **分区表的 MySQL** | 支持时间范围查询 |
| **打赏/计费流水** | 强事务、审计 | **关系型 DB + 写入审计表** | 必须保证不丢失 |
| **推荐特征** | 大批量写入、批量读取 | **离线特征库（Hadoop / Spark）** + **在线 KV（Redis）** | 离线特征每日刷新 |
| **视频文件元数据** | 读多写少 | **对象存储 + Metadata DB（MySQL）** | 存放 URL、时长、分辨率等 |
| **日志/监控** | 超大写入 | **ELK / Loki + ClickHouse** | 用于运营分析、异常定位 |

### 3.2 关键表（简化示例）  

```sql
-- users
CREATE TABLE users (
    user_id BIGINT PRIMARY KEY,
    username VARCHAR(64) UNIQUE,
    email VARCHAR(128),
    password_hash VARBINARY(256),
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    role ENUM('streamer','viewer','admin')
);

-- channels
CREATE TABLE channels (
    channel_id BIGINT PRIMARY KEY,
    owner_id BIGINT REFERENCES users(user_id),
    title VARCHAR(256),
    category VARCHAR(64),
    status ENUM('offline','live','postprocessing') DEFAULT 'offline',
    live_start_ts BIGINT,          -- epoch ms, null if offline
    current_viewers INT DEFAULT 0,
    thumbnail_url VARCHAR(512),
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);

-- live_viewers (Redis Hash)
-- key: "live:{channel_id}"
-- fields: "viewers" => int, "last_ts" => epoch ms

-- danmu (弹幕) - ClickHouse partitioned by day
CREATE TABLE danmu (
    channel_id UInt64,
    user_id UInt64,
    content String,
    ts UInt64,                -- epoch ms
    color UInt32,
    size UInt8
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(toDateTime(ts/1000))
ORDER BY (channel_id, ts);
```

> **为什么要把弹幕放在 ClickHouse 而不是 MySQL？**  
> - 弹幕是 **写放大**（200k 条/秒），且查询大多是 **时间范围**（回放时 5 min 前后），ClickHouse 对 **列式存储+压缩+并行查询** 天生友好，能在秒级返回数万条弹幕。  

### 3.3 数据一致性策略  

| 场景 | 一致性要求 | 实现方式 |
|------|------------|----------|
| **用户登录/账户信息** | 强一致 | 主库读写，读写分离后使用 **读后写**（读写分离的副本延迟 < 100 ms） |
| **直播状态** | 最终一致 | 使用 **Redis** 计数，定时同步到 MySQL（每 5 s） |
| **弹幕** | 最终一致 + 不丢失 | **Kafka** 作为持久化日志，消费者写入 ClickHouse；消费者故障时可从 Kafka 重放 |
| **打赏/计费** | 强事务 | **Two‑Phase Commit** 或 **数据库事务** + **消息队列（事务性 Kafka）** 保证“写库+推送”原子性 |

## ## 第四步：核心 API 设计  

> **原则**：RESTful + gRPC（内部） + WebSocket（弹幕/打赏推送）  

### 4.1 鉴权  

- **OAuth2 + JWT**（Access Token 有效期 1h，Refresh Token 30d）  
- 所有业务 API 必须在 **Gateway** 层校验 JWT，避免业务服务直接处理鉴权。  

### 4.2 推流相关 API  

| 方法 | 路径 | 说明 | 参数 | 返回 |
|------|------|------|------|------|
| `POST` | `/api/v1/streams/{channel_id}/start` | 主播申请推流地址（返回 RTMP URL + 流密钥） | `channel_id` (path) <br> `auth_token` (header) | `{ "rtmp_url": "...", "stream_key": "..." }` |
| `POST` | `/api/v1/streams/{channel_id}/stop` | 主播结束推流 | 同上 | `{ "status":"ok" }` |
| `GET` | `/api/v1/streams/{channel_id}/manifest.m3u8` | 观众获取 HLS manifest（CDN 直接返回） | `channel_id` (path) | HLS Manifest |

> **为什么要让主播先请求一次**？  
> - 可以在 **业务层** 记录 **直播开始时间、状态**，防止无效推流占用资源。  
> - 若主播使用 **RTMPS**（加密），需要把 **token** 嵌入 URL，服务端在入口校验后才放行。  

### 4.3 观看相关 API  

| 方法 | 路径 | 说明 | 参数 | 返回 |
|------|------|------|------|------|
| `GET` | `/api/v1/channels/{channel_id}/info` | 返回频道基本信息、当前码率列表、CDN Edge 域名 | `channel_id` | JSON (title, status, bitrate_options, cdn_domain) |
| `GET` | `/api/v1/channels/{channel_id}/stats` | 实时观众数（从 Redis） | `channel_id` | `{ "viewers": 12345 }` |
| `GET` | `/api/v1/recommendations?uid=xxx&limit=20` | 频道推荐列表 | `uid` (query) | `[ {channel_id, title, thumbnail, live_status} ]` |

### 4.4 弹幕（WebSocket）  

- **连接地址**：`wss://chat.twitch-like.com/ws?channel_id=123&token=JWT`  
- **消息结构（JSON）**  

```json
{
  "type": "danmu",
  "payload": {
    "user_id": 987654,
    "content": "好刺激！",
    "color": 16777215,
    "size": 1,
    "timestamp": 1685401234567
  }
}
```

- **服务器响应**：广播给同频道的所有在线用户，**先写入 Kafka**，再 **从 Kafka 消费写入 ClickHouse**，保证 **不丢失**。  

### 4.5 打赏/订阅  

| 方法 | 路径 | 说明 | 参数 | 返回 |
|------|------|------|------|------|
| `POST` | `/api/v1/payments/donate` | 观众向主播打赏 | `{ "channel_id":123, "amount":5.00, "currency":"USD" }` | `{ "payment_id":"xyz", "status":"processing" }` |
| `POST` | `/api/v1/payments/subscribe` | 订阅频道（周期性） | 同上 | `{ "subscription_id":"abc", "status":"active" }` |

- **实现要点**：调用 **第三方支付网关** → **支付回调** → **事务写入 DB + 发送 Kafka 事件** → **计费服务** 更新 **主播收入** → **推送服务**（WebSocket）给观众弹出 “打赏成功”。  

### 4.6 统一错误码  

| Code | 含义 |
|------|------|
| 0 | 成功 |
| 1001 | 参数错误 |
| 1002 | 未登录 / token 失效 |
| 1003 | 权限不足 |
| 2001 | 推流入口不可用 |
| 3001 | 弹幕频率超限（防刷） |
| 4001 | 支付网关错误 |
| 5000 | 系统内部错误 |

## ## 第五步：详细组件设计  

下面把 **每个核心组件** 拆成子模块，说明 **内部实现、技术选型、伸缩方式、容错**。  

### 5.1 推流入口（RTMP/RTMPS Ingress）  

| 子模块 | 功能 | 技术选型 | 伸缩/容灾 |
|--------|------|----------|-----------|
| **负载均衡** | 把主播的 RTMP 连接均匀分配到多台 Ingress 机器 | **L4 TCP Load Balancer**（NGINX+IPVS、AWS NLB） | 多 AZ 部署，Health Check 失效自动剔除 |
| **Ingress Server** | 接收 RTMP，做 **TLS 终止**、**鉴权**、**流量统计** | **NGINX-RTMP** 或 **SRS（Simple RTMP Server）**（开源、支持集群） | 水平扩展，单实例故障不影响其它流 |
| **流控 & 防刷** | 限制每个 IP/账号的推流并发数、码率上限 | NGINX 限流模块 + Redis 计数 | 采用 **令牌桶**，异常时自动降级到 **HLS-only** |
| **写入转码队列** | 将收到的原始 TS/FLV 推送到转码集群 | **Kafka (分区 = channel_id)**，每条流对应一个分区 | Kafka 多副本 (replication factor 3)，保证不丢流 |
| **监控/日志** | 实时记录推流质量（丢帧率、延迟） | **Prometheus Exporter** + **Grafana** | 多副本、全局聚合 |

> **为什么要使用 Kafka 而不是直接调用转码 API？**  
> - **解耦**：推流入口故障不影响转码；转码侧可以 **弹性伸缩**，消费积压的流。  
> - **持久化**：Kafka 记录 7 天（可调）的视频流片段元数据，防止瞬时网络抖动导致丢失。  

### 5.2 转码集群  

| 子模块 | 功能 | 技术选型 | 伸缩/容灾 |
|--------|------|----------|-----------|
| **调度器** | 根据 Kafka 分区分配转码任务（每个 channel 一个任务） | **Kubernetes + custom Operator**（基于 CRD） | 按需扩容 Pod，Pod 自动恢复 |
| **转码引擎** | 实时将原始流转为 HLS/DASH 多码率、WebRTC 低延迟流 | **FFmpeg + NVENC（GPU）** 或 **Intel Quick Sync**；支持 **AV1**（未来） | GPU 节点弹性伸缩；使用 **Horizontal Pod Autoscaler (HPA)** 监控 CPU/GPU 使用率 |
| **分片产出** | 把转码后的视频切片（2 s）写入对象存储 | **S3/MinIO**（分区目录 `channel_id/date/segment_x.ts`） | 多 AZ 多副本，写入时使用 **多线程上传** |
| **清单生成** | 自动生成 HLS `.m3u8`、DASH `.mpd`，并写入 **CDN Origin** | **nginx-rtmp-module** 自带或自行脚本 | CDN 自动轮询 Origin，支持 **Cache‑Invalidation** |
| **转码监控** | 统计 **第一帧延迟**、**CPU/GPU 使用率**、**错误码** | **Prometheus + Alertmanager** | 超时报警 → 自动 **scale‑out** 或 **fallback**（仅保留原始流） |
| **容错** | 任务重试、节点故障自动迁移 | **K8s StatefulSet** + **PodDisruptionBudget** | 失效节点的任务会被重新调度到其他节点，保证 **≤5 s 延迟** |

> **弹性伸缩细节**  
> - **预测式伸缩**：使用 **历史并发曲线 + Prophet** 预估未来 5 min 需要的转码实例数，提前启动。  
> - **实时伸缩**：监控 **Kafka lag**（积压分片数），如果 lag > 30 s，立刻 **scale‑out**。  

### 5.3 CDN 与边缘分发  

| 子模块 | 功能 | 技术选型 | 关键指标 |
|--------|------|----------|----------|
| **边缘节点** | 缓存 HLS/DASH 分片、清单、封面 | **Akamai / CloudFront / GCore**（自建也可） | **首帧 < 2 s**（热点直播） |
| **层级缓存** | 1️⃣ Edge（用户最近） 2️⃣ Regional POP 3️⃣ Origin（对象存储） | **多级 TTL**（分片 30 s，清单 5 s） | **热点刷新**：热点直播时把分片 TTL 降到 5 s，热点节点采用 **主动预取** |
| **实时流（WebRTC）** | 低延迟（< 500 ms） | **WebRTC SFU**（Janus、mediasoup）部署在热点城市 | **流转发**：SFU 直接拉取转码后 2 s 分片，做 **延迟拼接** |
| **负载均衡** | 把用户请求路由到最近节点 | **Anycast DNS + GeoIP** | 99% 请求在 30 ms 内路由完成 |
| **缓存失效** | 直播结束后立即 **purge** 旧分片，防止观众看到 “黑屏” | CDN API 自动调用 | **Purge latency < 200 ms** |

> **为什么需要多级缓存？**  
> - **单级 Edge**：热点直播时同一分片会被 **数十万** 观众请求，单节点容量受限，易出现 **缓存击穿**。  
> - **Regional POP**：把热点分片在更大范围复制，减轻单点压力。  
> - **Origin**：保证 **持久化**，所有分片最终落在对象存储，供回放使用。  

### 5.4 实时弹幕/聊天系统  

1. **入口层**：WebSocket/QUIC 服务器（**nginx + lua** 或 **C++ 高性能框架**）  
2. **消息写入**：每条弹幕写入 **Kafka Topic = `danmu_{channel_id}`**（分区 = channel_id）  
3. **消费层**：  
   - **实时分发**：**RocketMQ / Pulsar**（低延迟）消费后直接推送给同频道在线用户（通过 **WebSocket**）  
   - **持久化**：**ClickHouse** 消费者写入分区表，供回放查询  
4. **防刷**：在入口层使用 **令牌桶**（每用户每秒 ≤ 5 条），超限返回错误码 3001。  

#### 容错与高可用  

| 场景 | 方案 |
|------|------|
| **入口节点宕机** | 采用 **N+1** 部署，DNS 轮询或 **Consul** 注册中心自动剔除失效节点 |
| **Kafka 分区 Leader 失效** | 自动 **ISR**（In‑Sync Replicas）切换，生产者 **重试** |
| **消费端崩溃** | 消费位点保存在 **Kafka offset**，重启后可 **从上次位置恢复** |
| **消息丢失** | 所有弹幕必须 **持久化**（Kafka）+ **幂等写入 ClickHouse**（INSERT … ON DUPLICATE KEY） |

> **为什么弹幕既走 Kafka 又写 ClickHouse？**  
> - **Kafka** 保证 **低延迟**（几毫秒）和 **不丢失**，适合作为 **实时分发** 的背后队列。  
> - **ClickHouse** 适合 **大规模查询**（回放时一次性拉取 5 min 弹幕），且 **列式压缩** 节省存储。两者组合取长补短。  

### 5.5 打赏/订阅计费系统  

1. **支付网关**（Stripe、PayPal） → **Webhook** 回调 → **Kafka Topic `payment_events`**  
2. **计费服务**（微服务）消费 `payment_events`，**事务写入 MySQL**（`payments`、`user_balance`）  
3. **实时通知**：计费服务向 **Redis Pub/Sub** 发布 `user_{uid}_notify`，WebSocket 服务器推送给前端  
4. **收入统计**：每小时 **Spark** Job 从 MySQL 汇总，写入 **Redis** 缓存供主播 Dashboard 实时查看  

#### 防止重复扣款 & 幂等  

- **唯一业务 ID**（`payment_id`）在 **Kafka** 中做 **幂等**（`Exactly-Once`）处理。  
- **数据库唯一约束**（`payment_id` 唯一）防止因回调重放产生重复记录。  

### 5.6 频道管理 & 推荐系统  

| 功能 | 方案 | 关键技术 |
|------|------|----------|
| **频道元数据 CRUD** | RESTful 微服务 | Spring Boot + MySQL |
| **热点检测** | 统计 `live_viewers`（Redis）> 10k → 标记为热点 | Redis Sorted Set + 定时任务 |
| **推荐召回** | 简单 **协同过滤**（用户最近观看的同类频道） + **热点加权** | Spark/Scala 离线计算 → 结果写入 **Redis**（`uid:recommend`） |
| **实时推荐** | 读取 **Redis** 缓存 + **热点列表**（每分钟更新） | API 返回前端渲染 |
| **缓存失效** | 当频道状态变化（上/下线） → **Cache Invalidation** 通过 **Kafka** 通知所有节点 | Kafka + Consistent Hashing |

> **不使用复杂的深度学习模型的原因**：在面试中，**思路**比**实现细节**更重要；而且离线协同过滤已经能满足 80% 场景，后续可以逐步替换为 **深度召回**。

| 场景 | CAP 需求 | 选型 |
|------|----------|------|
| **用户登录/账户** | **强一致 (C + A)** | MySQL 主从 + 读写分离 |

| **弹幕** | **可用性 + 分区容错**（不需要强全局一致） | Kafka + ClickHouse（最终一致） |
| **打赏/计费** | **强一致**（金钱不可错） | MySQL 事务 + 幂等 Kafka |
| **回放视频** | **高可用 + 最终一致** | 对象存储多副本 |

## ## 第七步：常见面试追问与回答  

### 7.1 弹幕系统的高并发实现  

**问题**：如果每秒有 200 k 条弹幕消息，如何保证低延迟且不丢失？  

**回答要点**：

1. **入口层**：  
   - 使用 **WebSocket/QUIC** 长连接，负载均衡（L4）后分配到 **弹幕网关**（基于 **nginx + lua** 或 **C++ 高性能框架**）。  
   - **限流**：每用户令牌桶（5 条/s），防止刷屏导致后端崩溃。  

2. **持久化队列**：  
   - **Kafka Topic = `danmu_{channel_id}`**，分区数 = 1（每频道独占），**replication factor = 3**。  
   - 生产者 **同步写入**（acks=all），确保不丢失。  

3. **实时分发**：  
   - 消费者 **RocketMQ/Pulsar**（低延迟）读取消息后直接 **push** 到对应频道的 **WebSocket** 客户端。  
   - 通过 **HashRing** 将同一频道的所有用户映射到同一分发节点，避免跨节点的广播成本。  

4. **持久化存储**：  
   - 另一个消费者把弹幕写入 **ClickHouse**（分区按天），实现回放查询。  
   - ClickHouse **INSERT … ON DUPLICATE KEY** 保证幂等。  

5. **容错**：  
   - **Kafka** 自动 Leader 选举，写入磁盘保证不丢失。  
   - 消费者崩溃后 **offset** 保存在 Kafka，重启后 **从上次位置恢复**。  
   - **弹幕网关** 使用 **N+1** 部署，健康检查失效自动剔除。  

6. **延迟估算**：  
   - **WebSocket → Kafka**：< 5 ms（网卡 + 线程池）  
   - **Kafka → 分发**：< 10 ms（内部网络）  
   - **分发 → 客户端**：< 150 ms（互联网 RTT）  
   - **总计**：< 200 ms，满足 ≤ 300 ms 要求。  

### 7.2 转码与多码率分发的弹性伸缩  

**问题**：在峰值时段，转码集群需要如何弹性伸缩？如何保证转码延迟 ≤ 5 秒？  

**回答要点**：

1. **任务模型**：每个频道对应 **一个转码任务**，消费 Kafka 中该频道的流分片。  

2. **调度器**：基于 **Kubernetes Operator**（自定义 CRD `LiveStream`），监控：  
   - **Kafka lag**（未转码分片数）  
   - **GPU/CPU 使用率**  
   - **当前观众数**（热点判断）  

3. **伸缩规则**：  
   - **预测式**：使用 **Prophet** 对历史并发曲线做 5‑minute 前瞻，提前 **预热** 10% 额外转码 Pod。  
   - **实时**：若 **lag > 30 s**，HPA 将 **Replica** 数量乘以 1.5；若 **lag < 5 s** 且 CPU < 40%，则 **scale‑in**。  

4. **转码延迟控制**：  
   - **分片长度**：2 s（FFmpeg `-segment_time 2`），每个分片完成后立即上传对象存储。  
   - **并行流水线**：每个转码实例 **同时处理 3‑5 条分片**（pipeline），利用 **GPU 异步编码**。  
   - **监控**：Prometheus 报警 `first_segment_latency > 5s` → **自动弹性扩容** 并 **fallback**（只保留 720p）。  

5. **资源配额**：  
   - **GPU 节点**（NVIDIA T4）每台可支撑约 **50 条 1080p** 流（经验值），使用 **节点池** 动态伸缩（云原生）  
   - **CPU-only** 节点用于 **720p/480p** 的低码率转码。  

6. **容错**：  
   - **Pod Disruption Budget** 保证在升级/维护时至少保留 **80%** 实例。  
   - **转码失败**（FFmpeg 错误码）会 **自动重试 3 次**，若仍失败，将该频道标记为 **“转码异常”**，后台人工介入。  

### 7.3 跨地域 CDN 与观众体验  

**问题**：针对全球观众，如何设计 CDN 选址与缓存层级，确保 99% 观众在 2 秒内获取首帧？  

**回答要点**：

1. **CDN 选址原则**  
   - **覆盖度**：选取 **全球 120+ PoP** 的大型 CDN（Akamai、CloudFront、GCore）  
   - **热点区域**：美国、欧洲、东亚（日本、韩国）是主要流量来源，预置 **专属转码节点**（Edge‑Transcoder）在这些地区。  

2. **多级缓存层级**  
   - **Edge Cache**（距用户 ≤ 50 ms）缓存 **最新分片**（TTL 5 s）  
   - **Regional POP**（距用户 ≤ 200 ms）缓存 **分片集合**（TTL 30 s）  
   - **Origin**（对象存储）持久化 **所有分片**（无 TTL）  

3. **热点直播的缓存刷新**  
   - 当 **观众数 > 5k**，系统自动把该频道标记为 **热点**，调用 CDN API **预热**：  
     - **主动拉取** 最新 30 s 分片到所有 Edge 节点（并行）  
     - **降低 TTL** 为 **5 s**（默认 30 s）  
   - **分片分发**：采用 **Cache‑Control: max‑age=5**，确保 Edge 在 5 s 后自动向 POP 拉取最新分片。  

4. **首帧秒级获取**  
   - **Manifest（m3u8）** TTL 2 s，使用 **HTTP/2 Server Push** 把 **first segment** 直接随清单返回。  
   - **WebRTC**：SFU 直接把 **I‑frame**（关键帧）推送给用户，降低首帧延迟到 **< 500 ms**。  

5. **故障切换**  
   - 若某 Edge 节点失效，**Anycast DNS** 自动把请求路由到最近可用节点。  
   - **CDN 同步**（跨 POP）采用 **分层同步**，保证热点分片在 1 s 内复制到 3 个最近 POP。  

6. **成本控制**  
   - **热点缓存** 使用 **计费型 CDN**（按请求计费），对非热点流量采用 **默认 TTL 30 s**，降低缓存刷新频率。  
   - **对象存储** 采用 **分层存储**（热/冷），30 天后自动转入 **冷存**（Glacier），节约 70% 成本。

## ## 心得与反思  

### 1️⃣ 本题最难的 1‑2 个设计决策  

| 决策 | 难点 | 思考过程 |
|------|------|----------|
| **实时转码与延迟控制** | 同时要满足 **5 s 首帧**、**多码率**、**弹性伸缩**，涉及 GPU、Kafka、K8s、监控等多技术栈的耦合。 | - 先把 **流‑分片** (2 s) 定义为最小单位，保证每个分片可以独立转码并上传。<br>- 再把 **转码任务** 抽象为 **Kafka 消费**，自然实现 **背压**（lag）监控。<br>- 用 **K8s Operator** 把业务规则（lag、CPU）映射到 **Replica**，实现自动伸缩。<br>- 最后加入 **预测模型**（Prophet）避免突发峰值导致瞬时卡顿。 |
| **弹幕的低延迟+持久化** | 需要 **200k 条/秒** 的写入、**毫秒级分发**、**回放查询**，三者往往相互制约。 | - 把 **实时** 与 **离线** 完全分离：实时走 **Kafka → 分发**，离线走 **ClickHouse**。<br>- 通过 **分区 = channel_id** 消除跨频道竞争，保证 **单频道顺序**。<br>- 采用 **幂等写入** 与 **消息持久化** 双保险，防止网络抖动导致消息丢失。 |

### 2️⃣ 新手最容易犯的错误（至少 2 条）  

| 错误 | 结果 | 正确做法 |
|------|------|----------|
| **把所有业务放在单个 monolith**（如把推流、转码、聊天、推荐全部写在同一个服务） | 难以水平扩展，任意一块故障都会导致整个平台不可用。 | 按 **业务边界** 拆分为 **微服务**（推流、转码、聊天、计费、推荐），每个服务独立伸缩、部署。 |
| **忽视背压 & 流控**（直接把所有 RTMP 推流写入转码机器） | 高峰时转码节点瞬时崩溃，导致 **丢帧、延迟爆炸**。 | 使用 **消息队列（Kafka）** 作为 **缓冲层**，监控 **lag** 并基于此做 **弹性伸缩**。 |
| **把弹幕直接写入数据库**（MySQL） | 每秒 200k 写入导致 **写热点**，数据库崩溃，弹幕卡顿。 | 采用 **Kafka + ClickHouse**（列式、分区），写入 ClickHouse 通过批量导入提升吞吐。 |
| **只考虑单机磁盘存储回放** | 30 PB 的视频文件根本放不下，恢复时间长。 | 使用 **对象存储（S3/MinIO）** + **冷热分层**，自动复制、跨地域容灾。 |

### 3️⃣ 学习建议和可延伸的方向  

1. **系统设计的思考框架**  
   - 需求 → 规模估算 → MVP → 高可用 → 细化组件 → 监控/运维 → 成本评估。  
   - 每次设计都先画 **时序图**，帮助明确 **请求路径** 与 **依赖关系**。  

2. **重点技术栈**  
   - **实时流媒体**：FFmpeg、GStreamer、SRT、WebRTC、HLS/DASH。  
   - **容器编排**：Kubernetes、Helm、Operator。  
   - **消息系统**：Kafka、Pulsar、RocketMQ，了解 **Exactly‑Once**、**事务**。  
   - **时序/列式数据库**：ClickHouse、Druid，用于弹幕、日志、监控。  
   - **CDN 与边缘计算**：Anycast DNS、边缘缓存策略、热点预热。  

3. **实战练习**  
   - 搭建一个 **RTMP → HLS** 的小 demo（使用 NGINX‑RTMP + FFmpeg），体会 **转码延迟** 与 **分片**。  
   - 用 **Kafka + ClickHouse** 实现 **弹幕写入 + 实时推送**，感受 **背压** 与 **幂等**。  
   - 在本地 Kubernetes 上部署 **HPA**、**Custom Metrics**，练习 **自动伸缩**。  

4. **阅读材料**  
   - 《Designing Data‑Intensive Applications》——了解 **CAP、事务、流处理**。  
   - 《Live Video Streaming at Scale》——Netflix、YouTube、Twitch 的真实案例。  
   - 官方文档：FFmpeg、Kafka、Kubernetes Autoscaling、AWS CloudFront / Azure CDN。  

5. **面试技巧**  
   - **先说业务**：先把用户流程说清楚，再映射到系统组件。  
   - **量化**：所有关键指标（QPS、延迟、容量）都要给出 **数字**，显示你能做估算。  
   - **权衡**：主动提出 **CAP**、**成本 vs 性能** 的取舍，展示思考深度。
