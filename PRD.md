# PRD：系统设计面试题生成器 + 题解

## 一、项目概述

本项目通过本地部署的大模型（`gpt-oss:120b` via Ollama）自动生成系统设计面试题及配套详细题解，并利用 Git 提交历史模拟"每日刷题"记录，向过去倒推 100 天，每天一道题一次提交。

---

## 二、目标与价值

| 目标 | 说明 |
|------|------|
| 内容生成 | 自动产出高质量系统设计题目与面向新手的完整题解 |
| 刷题历史模拟 | 用 Git 提交历史呈现持续学习轨迹，每天一题，时间随机 |
| 自主运行 | 全程无需人工干预，运行一次即完成 100 天内容生成与提交 |

## 四、双轮 API 调用流程

### 第一轮：生成题目

**Prompt 模板（传给 gpt-oss:120b）：**

```
你是一位资深系统设计面试官，请为以下系统出一道完整的系统设计面试题：

系统：{system_name}

要求：
1. 题目背景：简要介绍该系统是什么、用途是什么（2-3句话）
2. 面试场景设定：模拟真实面试开场白，面试官提出问题
3. 功能性需求：列出 4-6 个核心功能需求（用户能做什么）
4. 非功能性需求：列出 4-5 个关键指标（如 DAU、QPS、延迟、可用性、存储量）
5. 系统边界：明确说明哪些在本题范围内，哪些不考虑
6. 提示与追问：给出 2-3 个面试官可能追问的问题

请用 Markdown 格式输出，语言为中文，结构清晰，适合贴入 GitHub 的 .md 文件。
```

**输出**：题目 Markdown 内容（约 300-500 字）

---

### 第二轮：生成题解

**Prompt 模板（将第一轮结果嵌入）：**

```
你是一位耐心的系统设计导师，正在辅导一位刚入行的后端初学者。
请针对以下系统设计题目，给出一份极其详细、由浅入深、完整的解题过程。

---题目开始---
{question_markdown}
---题目结束---

解答要求：
1. **面向读者**：假设读者是完全没做过系统设计的新手，不要跳步骤
2. **结构层次**：从最小可用系统出发，逐步演进到高可用分布式架构
3. **每一步都要解释为什么**：不只说做什么，要说为什么这样做
4. **必须包含以下章节**（Markdown 二级标题）：
   - ## 解题思路总览
   - ## 第一步：理解需求与规模估算
   - ## 第二步：高层架构设计（画出核心组件）
   - ## 第三步：数据库设计（表结构 / Schema）
   - ## 第四步：核心 API 设计
   - ## 第五步：详细组件设计（逐模块展开）
   - ## 第六步：扩展性与高可用设计
   - ## 第七步：常见面试追问与回答
   - ## 心得与反思
5. **心得与反思章节**要求：
   - 总结本题最难的 1-2 个设计决策
   - 指出新手最容易犯的错误
   - 给出学习建议和延伸阅读方向
6. 语言：全中文，语气亲切，像在手把手教学
7. 格式：完整 Markdown，可直接保存为 .md 文件

请开始解答：
```

**输出**：题解 Markdown 内容（约 1500-3000 字）

```markdown

{第二轮输出：完整解答}
```

## 七、各模块职责详述

### `config.py`
```
OLLAMA_HOST = "http://10.54.79.119:11434"
MODEL = "gpt-oss:120b"
QUESTIONS_DIR = "questions"
DAYS = 100
```

### `systems.py`
- 维护系统名称列表（恰好 100 个，与 DAYS 一一对应）
- 提供 `get_system_for_day(day_index)` 方法，直接按索引取值，全程无重复

### `generator.py`
- `generate_question(system_name) -> str`：第一轮调用，返回题目 md
- `generate_solution(question_md) -> str`：第二轮调用，返回题解 md
- `generate_full(system_name, day_index) -> str`：组合完整 md

### `file_writer.py`
- `write_day_file(day_index, system_name, content) -> filepath`
- 文件名格式：`day_{day_index:03d}_{system_name_slug}.md`

### `commiter.py`
- 从今天（`2026-05-25`）向过去倒推 100 天
- 第 1 天 = 今天，第 100 天 = 2026-02-15
- 每天随机一个提交时间：`09:00 ~ 23:30` 之间随机
- 使用 `GIT_AUTHOR_DATE` 和 `GIT_COMMITTER_DATE` 环境变量设置历史时间
- 提交信息格式：`day {N}: 系统设计 - {系统名}`
- 每次只提交对应当天的 md 文件（`git add questions/day_XXX_*.md`）

### `main.py`
执行流程：
```
1. 初始化：确保 questions/ 目录存在，git init（如未初始化）
2. 循环 day = 100 → 1（从最远的过去到今天）：
   a. 获取该天对应系统名
   b. 调用 generator 生成完整 md（含两轮 API 调用）
   c. 写入文件
   d. 调用 commiter 以历史时间提交
3. 完成提示
```

## 八、`commiter.py` 关键逻辑说明

```
今天 = 2026-05-25
day 1  → 提交日期 = 2026-05-25（今天）
day 2  → 提交日期 = 2026-05-24（昨天）
...
day 100 → 提交日期 = 2026-02-15

每天提交时间 = 随机(hour=9~23, minute=0~59, second=0~59)

git commit 命令通过设置环境变量实现历史时间：
  GIT_AUTHOR_DATE="2026-03-01T14:23:07"
  GIT_COMMITTER_DATE="2026-03-01T14:23:07"
```

## 九、非功能性要求

| 项目 | 要求 |
|------|------|
| 语言 | 全中文输出 |
| 模型超时 | 单次请求超时设为 300s（120b 模型推理慢） |
| 错误重试 | API 调用失败自动重试 2 次，仍失败则写入占位文件并继续 |
| 进度展示 | 每完成一天打印进度：`[42/100] 生成完成：YouTube` |
| 无外部依赖 | 只依赖 `requests`、`subprocess`、标准库 |

## 十、开发顺序建议

1. `config.py` + `systems.py` — 最简单，先建好基础
2. `generator.py` — 核心，先单独测试两轮 API 调用
3. `file_writer.py` — 简单 IO
4. `commiter.py` — 先验证 git 历史时间戳能否正确写入
5. `main.py` — 串联所有模块，跑通 1 天再跑 100 天
